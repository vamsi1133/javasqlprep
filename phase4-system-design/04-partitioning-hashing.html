04 — Partitioning & Consistent Hashing

How you split data across machines — and the one algorithm that shows up in half the case studies.

Why this matters in the interview

Replication (tutorial 03) gives you more copies of the same data. Partitioning gives you more data than one machine can hold, and more write throughput than one machine can absorb. Every design that outgrows a single box eventually needs both, and the moment you say "I'll shard by user ID" the interviewer's follow-ups are already loaded: what happens when you add a node? what if one user is 400× bigger than the others? how do you run a query that spans shards? Those three questions are the entire content of this page, and they are asked in almost every distributed round. Candidates who have only memorized diagrams answer the first one with "rehash everything," which is the wrong answer, and the interviewer stops learning anything from them.

Consistent hashing is the single most reusable idea in this phase. It is the core of the distributed cache (08), it decides which node owns a rate-limit counter (09), it's how a sticky load balancer routes sessions (02), and it's the partitioning scheme inside Cassandra, DynamoDB, Riak, Memcached clients and Envoy. You will be asked to explain it at a whiteboard, from memory, with a diagram. So we are not going to state it — we are going to derive it. Start from the naive scheme, watch it fail with real numbers, and the ring will feel inevitable rather than magical. If you can draw the ring, place the keys, walk clockwise, and then say "and virtual nodes fix the lumpiness and the failure hot-spot," you have covered what 90% of interviewers want and you've done it in ninety seconds.

1. Why partition at all — and why it's a last resort

A single Postgres box in 2026 is enormous. You can rent a machine with 128 vCPUs, 1 TB of RAM and 30 TB of NVMe. That will comfortably serve tens of thousands of QPS and hold billions of rows. Vertical scaling — buy a bigger box — is the correct first answer, and saying so is a senior signal. It has zero engineering cost and it keeps every query, every join, every transaction working exactly as written.

But it has a ceiling and a curve:

  • The ceiling is physical. There is a largest instance type. Once you're on it, there is no next step.
  • The curve is superlinear. The top instance type costs far more than 2× the one below it. You pay a premium for the last increment of headroom.
  • It doesn't help availability. One box is one blast radius. A bigger box is a bigger blast radius.
  • Writes are the real wall. Read replicas scale reads almost for free. Nothing scales writes on a single leader — every write still lands on one machine's WAL.

So you partition (shard): split the data set into disjoint subsets, each owned by a different node. Now you have more total RAM, more total IOPS, more total write throughput, and a smaller blast radius per failure. That's the win. Here's the bill:

What sharding costs youWhy it hurts
Cross-shard joinsA JOIN that used to be one index lookup becomes an application-level fan-out to N nodes plus a merge in your own code. Postgres can't help you anymore.

Say this out loud before you shard anything: "Before I partition, I'd exhaust the cheaper options — vertical scaling, read replicas for the read load, a cache in front of the hot 5% of keys, and archiving cold rows out of the hot table. Sharding is the last thing I'd reach for, because it costs me joins and transactions permanently. I'd shard when writes exceed what one leader can take, or when the working set no longer fits in RAM." That single paragraph is worth more than a perfectly drawn ring diagram, because it shows you know sharding is a cost, not an achievement.
The most common self-inflicted wound in these interviews: a candidate hears "10 million users" in minute three and immediately says "so I'll shard the users table by user_id." Ten million user rows is about 2 GB. It fits in RAM on a laptop. You just took on distributed transactions to solve a problem you don't have, and a good interviewer will push back hard — not because sharding is wrong, but because you didn't do the arithmetic first. Estimate, then decide.
The vocabulary is muddled and you should just be explicit. Partitioning is the general term for splitting a dataset. Sharding usually means partitioning across separate machines. Partitioning inside Postgres (declarative table partitioning) splits one table across files on one machine — helpful for pruning and for dropping old data cheaply, but it does not add throughput. Kafka calls its units partitions; Cassandra calls them partitions too but places them by token; DynamoDB says partitions; MongoDB says chunks in shards. Same idea, different nouns.

2. Vertical vs horizontal partitioning

Two different words that sound like variants of one idea, and they solve unrelated problems. Interviewers occasionally ask you to distinguish them just to check you're not parroting.

Vertical partitioning — split by column / by concern

You take a wide table and cut it lengthways. The users table has id, email, name (read on every request, 200 bytes) and bio, avatar_blob, preferences_json (read almost never, 40 KB). Put the hot narrow columns in one table and the cold fat ones in another. Now the hot table's rows are tiny, so far more of them fit per page, so far more of the table fits in the buffer cache, so your common query gets faster without any new machines.

Scaled up, vertical partitioning is service decomposition: users live in the user service's database, orders in the order service's database. That's why microservices lose joins — you vertically partitioned your schema across network boundaries.

What it buys: cache efficiency, smaller I/O per query, isolation of concerns. What it doesn't buy: write throughput or capacity past one machine. Every row still lives on one box.

Horizontal partitioning (sharding) — split by row

Same columns everywhere; the rows are divided. Users 1–1M on shard A, 1M–2M on shard B. Each shard is a complete, independent database with the identical schema. This is the one that actually scales, and it's what the rest of this page is about.

Choosing the partition key is the whole game. Everything else on this page is downstream of that one decision, and it is essentially irreversible once you have data. The key you pick should (a) appear in the vast majority of your queries, so most requests hit exactly one shard, and (b) have high cardinality and even access, so no single value can dominate. For a chat app, channel_id beats message_id, because you always read messages by channel. For a multi-tenant SaaS, tenant_id keeps every tenant's data co-located — one query, one shard — at the price of the "one whale tenant" problem in §8.

The sentence that earns the point: "I'd pick the partition key by looking at the dominant access pattern, not at the data. If 95% of my reads are 'give me the last 50 messages in channel X', then channel_id is the key — that makes the common query single-shard. Any key that makes my hottest query a scatter-gather is the wrong key, no matter how evenly it distributes."

3. Sharding strategies

Three ways to map a key to a shard. Each is genuinely used in production; each fails in a characteristic way, and the interviewer is listening for whether you can name the failure without being prompted.

3a. Range-based partitioning

Sort the key space and cut it into contiguous ranges. Users A–F → shard 0, G–M → shard 1, and so on. HBase, Bigtable, and Postgres declarative range partitioning work this way; so does MongoDB's default. 

key space:  [ A ────── F ][ G ────── M ][ N ────── S ][ T ────── Z ]
                shard 0       shard 1       shard 2       shard 3

Good at: range scans. "Give me every event between 09:00 and 10:00" or "every user whose name starts with S" touches exactly one shard, because the data you want is physically adjacent. This is the only strategy that preserves ordering, and if your dominant query is a range query, it's the right answer.

Fails at: sequential keys. And your keys are almost always sequential, because the most natural key in the world is a timestamp or an auto-increment ID. Partition events by created_at and 100% of today's writes go to the newest shard. You have bought four machines and you are writing to one of them. The other three hold cold history and idle. This is the classic time-series hot-spot, and it's the single most common way range partitioning blows up.

The fix, when you must have ranges: compound the key. Partition by (sensor_id, timestamp) rather than timestamp alone. Now writes from a thousand sensors spread across shards, and "the last hour of readings for one sensor" is still a single-shard range scan. You lose "the last hour across all sensors" as a cheap query — that becomes scatter-gather. That trade is usually worth naming out loud.

Range partitioning by an auto-increment primary key has exactly the same disease as partitioning by timestamp — an auto-increment ID is a timestamp with extra steps. Every new row goes to the highest range. If you catch yourself saying "I'll shard by ID range," check whether the ID is monotonic. If it is, you've built a one-node write path with three-node hosting costs.

3b. Hash-based partitioning

Run the key through a hash function first, then map the hash to a shard. Because a good hash is uniform and avalanching, user_1 and user_2 land in completely unrelated places, so writes spread evenly no matter how sequential the input was. Cassandra (token ring), DynamoDB, Kafka (hash(key) % partitions), and Memcached all do this. 

  "user_8842"  ─hash─▶  0x9F2A…  ─map─▶  shard 2
  "user_8843"  ─hash─▶  0x0C71…  ─map─▶  shard 0   ← neighbours in input,
  "user_8844"  ─hash─▶  0x7B3D…  ─map─▶  shard 1      strangers on disk

Good at: even distribution, with no thought required. Point lookups by the partition key are a single hop: hash it, go to that node, done. This is the default, and if you don't have a reason to do something else, do this.

Fails at: range scans. Ordering is destroyed by design — that was the point — so "all events between 09:00 and 10:00" now has to ask every shard and merge the results. That's a scatter-gather, and §10 explains why it's expensive in a way that averages hide.

Also fails at: naive resizing. hash(key) % N is the obvious way to map a hash to a shard, and it is a trap. That's §4, and it's why consistent hashing exists.

Cassandra's answer to "even writes but ordered reads" is worth stealing for interviews: it hashes the partition key (spreads data across the ring) but stores rows sorted by a separate clustering key within each partition. So PRIMARY KEY ((sensor_id), timestamp) gives you uniform placement across nodes and an ordered, seekable range scan inside one node. Hash between partitions, sort within them. If you can say that sentence you've said something most candidates can't.

3c. Directory / lookup-based partitioning

Stop computing the location and just look it up. Keep a lookup service — a table, a config in ZooKeeper/etcd — that maps key (or key-range, or tenant) → shard. The router asks the directory, then goes to the shard. 

   client ──▶ ┌────────────────────┐
              │ directory service  │   tenant_42  → shard 3
              │  (ZooKeeper/etcd/  │   tenant_43  → shard 1
              │   a Postgres table)│   tenant_44  → shard 3
              └─────────┬──────────┘   tenant_45  → shard 0   ← moved here
                        │                                        yesterday

                  go to shard 3

Good at: flexibility. You want to move one noisy tenant onto its own dedicated shard? Change one row in the directory. You want to place a European customer's data on EU hardware for GDPR? One row. Rebalancing becomes metadata editing rather than a re-derivation of a hash function, and you can make placement decisions no formula could express ("these three whales each get their own box"). This is exactly why it's common in multi-tenant B2B systems and in fintech, where per-customer isolation and data-residency rules are real requirements. If your interview is at a SaaS/enterprise shop, this is a strong, unfashionable, correct answer.

Fails at: being a single point of failure and a chokepoint. Every single request now consults the directory first. If it's down, your database is down — even though the database is fine. If it's slow, everything is slow, and you've added a network hop to every read. The mitigations are the obvious ones and you should name them unprompted: replicate the directory, cache the map aggressively in every client (it changes very rarely), and version it so a stale client can be told "your map is old, refresh". In practice the map is small — a few thousand entries — so caching it in process memory makes the hop disappear on the happy path.

4. The hash(key) % N disaster

You've chosen hash-based partitioning. You have four cache nodes. The obvious mapping is the one every engineer writes first: 

int node = Math.floorMod(key.hashCode(), N);   // N = 4 nodes

This is correct, uniform, O(1), and requires no coordination. Every client computes the same answer independently. It is also, the moment N changes, one of the great foot-guns in distributed systems — and the interviewer wants to see you find that out for yourself.

Do the arithmetic

Take ten keys with these hash values, and map them with N = 4 and then with N = 5 — you just added a fifth cache node because the working set grew.

Seven of ten keys moved. That isn't bad luck in a small sample — it's the expected behaviour. A key stays put only if h % 4 == h % 5, and in general a key survives a change from N to N+1 with probability roughly 1/(N+1). Going from 4 nodes to 5, about 1 in 5 keys keeps its home and ~80% of your keys remap. Going from 100 to 101, ~99% remap. The scheme doesn't degrade gracefully; adding one node to a hundred is nearly as destructive as adding one to four.

The reason is structural: the modulus is part of the mapping for every key. Change the modulus and you haven't adjusted the function, you've replaced it. There is no relationship between the old assignment and the new one.

Why that's a catastrophe, not an inconvenience

Now attach it to a real system. Say those N nodes are a Memcached/Redis tier fronting Postgres, and Postgres is only alive because the cache absorbs 95% of reads.

steady state          N = 4, hit rate 95%
                      DB sees   5% of 200k QPS  =  10k QPS   ← DB is fine

you add node 5        ~80% of keys now hash to a node that has never seen them
                      hit rate collapses toward 20%
                      DB sees  80% of 200k QPS  = 160k QPS   ← DB is dead

                      … and every timed-out request retries, so it's worse

That's a cache miss storm: a scaling action — the thing you did to improve capacity — instantly becomes an outage. The cache didn't lose the data; every node still holds its entries. But nobody asks for them anymore, because the mapping moved out from under the keys. The data is present and unreachable, which is the most annoying kind of unavailable.

And node removal is not opt-in. A cache node crashing is exactly the same event: N drops from 4 to 3, and ~75% of keys remap. So % N means any single cache node failure triggers a near-total cache flush and a thundering herd onto your database. You cannot operate this. It is a system that punishes you for both growth and failure.

Candidates often try to patch % N rather than replace it: "I'll warm the new node first," or "I'll add nodes in powers of two." Warming doesn't help — the problem is that keys move between existing nodes, not just onto the new one; k1 above moved from node 1 to node 2, and neither of those is the new node. Doubling (4 → 8) does have a nice property (h % 8 is either h % 4 or h % 4 + 4, so exactly half the keys move and each stays "near" its old home), but you can only ever double, you always move 50%, and it does nothing for the case that actually matters — a node dying, which is not a doubling. Note the reasoning, then discard it. The real fix is a different mapping.

State the requirement precisely, because it is what the next section is designed to satisfy:

When a node joins or leaves, I want to remap only the keys that node is responsible for — roughly K/N keys — and leave every other key exactly where it was. Adding a 5th node to 4 should move ~20% of keys, not 80%. And I want that with no central coordinator: every client should be able to compute the owner of a key on its own.

In the interview, do not jump to "consistent hashing" the instant someone says "distributed cache." Spend twenty seconds building the problem first: "The naive thing is hash mod N. That works until N changes — and if I go from 4 nodes to 5, about 80% of keys map to a different node, so my cache hit rate collapses and the whole read load lands on the database at once. Same thing happens when a node just dies. So I need a mapping where adding or removing a node only disturbs that node's share of the keyspace. That's consistent hashing." Deriving the requirement before naming the solution is the difference between someone who has memorized the term and someone who understands it — and interviewers can tell instantly which one you are.

5. Consistent hashing: the ring

The trick that makes it all work is one sentence, and it is worth memorizing verbatim:

Hash the nodes onto the same ring you hash the keys onto, and let each key be owned by the first node it meets walking clockwise.

That's it. Every consequence below falls out of that. Note what disappeared: N is no longer in the formula. The key's position on the ring depends only on the key. Nothing about the key's hash changes when nodes come and go — only the question of which node is next clockwise from it can change, and that only for keys in one arc.

Construction, step by step

  1. Take a hash function with a big output space — say MD5 or MurmurHash, giving you numbers in [0, 2³²). Now bend that number line into a circle by declaring that 2³² − 1 is adjacent to 0. That circle is the ring. It is just the hash output space with wraparound; it is not a physical thing and nothing "goes around" it.
  2. Place the nodes. For each server, hash something stable about it — its name or IP, hash("10.0.0.7") — and that number is its position on the ring. Nodes are now scattered around the circle at pseudo-random positions.
  3. Place the keys. Hash each key with the same function into the same space. hash("user:8842") is a point on the same circle.
  4. Assign ownership. Starting from the key's position, walk clockwise (i.e. toward increasing hash values, wrapping past the top back to 0) until you hit a node. That node owns the key. Equivalently: each node owns the arc from the previous node's position up to its own.

What happens when a node joins

Add node E and it hashes to some position on the ring, say between A and C. Which keys change owner? Only the keys that lie in the arc between A and E — those used to walk clockwise past E's (previously empty) position and land on C; now they stop at E. Every other key on the ring is completely undisturbed. Keys before A still go to A. Keys after E still go to C. Keys owned by B and D never even noticed.

The informal proof — say this out loud

With N nodes at pseudo-random ring positions and K keys hashed uniformly onto the same ring, each node owns an arc that is on average 1/N of the circumference, and therefore roughly K/N keys. When you add the (N+1)th node, it lands in exactly one existing node's arc and splits it. The only keys whose clockwise walk now terminates somewhere new are the keys in the portion of that arc before the new node's position. So the number of keys that move is at most the keys of one node — on average K/N, i.e. 1/N of the data. Every key belonging to the other N−1 nodes is provably untouched, because its clockwise walk never passes the new node's position at all.

Removal is the mirror image: when a node dies, its arc merges into its clockwise successor, so only that dead node's K/N keys move, and they all move to one place. (That last clause is a problem — hold onto it, it's what §6 fixes.)

It was invented at Akamai in 1997 (Karger et al.) for exactly the caching problem in §4 — distributing web objects across a fleet of edge caches that constantly join and leave. The 60-second whiteboard version, which is what you will actually be asked to produce: "I take a hash function with a 32-bit output and treat that output space as a circle. I hash every server's ID onto the circle, and I hash every key onto the same circle. A key belongs to the first server you meet walking clockwise from the key's position. So each server owns the arc behind it. When I add a server, it lands inside exactly one existing arc and takes over the front half of it — only that slice of keys, about K over N, changes owner; everything else on the ring is untouched. When a server dies, its arc merges into its clockwise neighbour. That's the whole idea. It has one real problem, which is that with only five servers the arcs are very uneven — and virtual nodes fix that." Then draw the ring, and keep talking while you draw.

6. Virtual nodes — the fix for the ring's two real problems

The plain ring solves the remapping problem and immediately introduces two new ones. Both come from the same root cause: with a small number of nodes, "pseudo-random positions" are lumpy. Random points on a circle do not spread out politely; they clump. Say this out loud, because interviewers specifically listen for whether you know the basic ring is not the finished answer.

Problem 1 — the arcs are uneven, so the load is uneven

Four nodes on a ring do not carve it into four neat quarters. They land wherever their hashes put them, and it is entirely normal for one node to end up owning 45% of the ring and another 8%. That node with 45% gets 45% of your keys and 45% of your traffic while its peer idles — you're paying for four machines and getting the capacity of about two and a half. With a large N the law of large numbers rescues you, but you don't have a large N; you have six cache nodes. 

PLAIN RING — 4 nodes, lumpy by accident (this is typical, not unlucky)

     0 ────────── A ─── B ──────────────────────── C ─── D ──── 2³²
        ▲             ▲                          ▲
        │ D's arc     │ A's tiny arc             │ B's HUGE arc
        │ (wraps)     │ (8% of keys)             │ (45% of keys)  ← melts

Problem 2 — when a node dies, its entire range dumps on one neighbour

This one is worse, and it's the one candidates miss. On a plain ring, node B's arc is a single contiguous stretch of the circle, and its only clockwise successor is C. So when B dies, 100% of B's load transfers to C — one machine. C was already running at its own steady-state load; now it's asked to serve roughly double. So C falls over. And when C falls over, its arc (now B's + C's) dumps entirely onto D. You have built a cascading failure: a single node death takes down the cluster, one machine at a time, around the ring.

What you actually want when a node dies is for its load to be spread across all the surviving nodes, so each one absorbs an extra 1/(N−1) and nobody notices. The plain ring gives you the opposite.

The fix: each physical node gets many positions

Don't hash the node once. Hash it V times under different labels — hash("A#0"), hash("A#1"), … hash("A#149") — and put a marker at every one of those positions. Each marker is a virtual node (a vnode, or in Cassandra/Dynamo language, a token). All V markers for a given label route to the same physical machine. Typical V is 100–200 per physical node; Cassandra's default is 256 tokens per node.

Nothing else about the algorithm changes. You still hash the key, still walk clockwise, still take the first marker you find. You just look up which physical box that marker belongs to.

Problem 2 is really the headline. Even load is nice; not cascading when a machine dies is the reason this is non-optional in production. Same reason a joining node is painless: it picks up 150 small arcs from many different existing machines, so the migration work is shared and no single donor gets hammered while it streams data out.

Bonus: weighting heterogeneous hardware

Virtual nodes give you a free knob that the plain ring can't express. Ownership is proportional to the number of markers, so give a bigger machine more markers. Your fleet has old 32 GB boxes and new 64 GB boxes? Give the old ones 100 vnodes and the new ones 200, and the new ones will naturally own about twice the keyspace and take about twice the traffic. No special-casing, no separate config — capacity becomes a single integer per node.

This matters more than it sounds. Real clusters are never homogeneous: you're mid-migration, or one AZ has different instance types available, or you're bin-packing on shared hardware. Mention the weighting in the interview — it's a one-liner and very few candidates have it.

Two things people get wrong here. (1) Virtual nodes are not extra processes or extra machines. They are extra entries in a sorted map. Nothing is deployed. If you say "so we'd spin up 150 virtual nodes per server," the interviewer will hear that you don't know what the mechanism is. Say "150 positions on the ring, all pointing at the same box." (2) Virtual nodes do not fix hot keys. They fix uneven key distribution. If one key is being read a million times a second, it hashes to exactly one position, which belongs to exactly one machine, and that machine burns — no number of vnodes changes that. That's §8, and conflating the two is the most common way this topic goes wrong.
The line that shows you've operated one of these, not just read about one: "I'd use virtual nodes — 150 or so per physical node. The even-load argument is the one people give, but the real reason is failure behaviour: with one position per node, when a node dies its entire range lands on a single neighbour, which doubles that neighbour's load and takes it down too — you cascade around the ring. With vnodes, a dead node's ranges are scattered, so its load is picked up by every survivor in small slices. It also lets me weight bigger boxes just by giving them more positions."

7. A consistent-hash ring in Java

Occasionally an interviewer will hand you this as a coding exercise — it's a favourite at the boundary between LLD and system design, and it's a natural pairing with your Phase 1 Java. The whole thing is a TreeMap and one method call. The insight that makes it collapse to nothing:

"Walk clockwise to the first node" is exactly TreeMap.ceilingKey(hash) — the smallest ring position ≥ the key's position. And "wrap past the top back to zero" is exactly TreeMap.firstKey() when ceilingKey returns null.

Say that sentence and then write the code; it makes the implementation self-evident.

import java.security.MessageDigest;
import java.util.*;

public class ConsistentHashRing {

    // The ring: ring position (0 .. 2^32) → the PHYSICAL node at that position.
    // TreeMap because we need "smallest key >= x" — that IS the clockwise walk.
    private final TreeMap<Long, String> ring = new TreeMap<>();

    // How many ring positions each physical node occupies. 100-200 is typical.
    // This is the virtual-node count: it flattens load and spreads a node's
    // keys over ALL survivors when it dies, instead of dumping on one neighbour.
    private final int vnodes;

    public ConsistentHashRing(int vnodes, Collection<String> nodes) {
        this.vnodes = vnodes;
        for (String n : nodes) add(n);
    }

    // Adding a node: drop `vnodes` markers for it around the ring.
    // Each marker steals only the small arc behind it from ONE existing marker,
    // so total keys moved ~= K/N, taken in slices from every other node.
    public void add(String node) {
        for (int i = 0; i < vnodes; i++) {
            ring.put(hash(node + "#" + i), node);   // "10.0.0.7#37" → some position
        }
    }

    // Removing a node (crash or scale-down): delete its markers. Each of its arcs
    // silently merges into whatever marker is clockwise of it — different physical
    // nodes for different arcs. No other key in the ring is affected.
    public void remove(String node) {
        for (int i = 0; i < vnodes; i++) ring.remove(hash(node + "#" + i));
    }

    // THE LOOKUP. Hash the key onto the same ring, walk clockwise to the first
    // marker. ceilingKey = smallest position >= h. If there is none, we ran off
    // the top of the ring — wrap around to firstKey(). That null check IS the ring.
    public String getNode(String key) {
        if (ring.isEmpty()) return null;
        Long pos = ring.ceilingKey(hash(key));
        return ring.get(pos != null ? pos : ring.firstKey());   // O(log N·V)
    }

    // Any well-distributed hash works; MD5/Murmur are the usual choices.
    // String.hashCode() is a poor ring hash — it clusters badly on similar inputs.
    private long hash(String s) {
        try {
            byte[] d = MessageDigest.getInstance("MD5").digest(s.getBytes());
            return ((long) (d[0] & 0xFF) << 24) | ((long) (d[1] & 0xFF) << 16)
                 | ((long) (d[2] & 0xFF) << 8)  |  (long) (d[3] & 0xFF);   // 32-bit ring
        } catch (Exception e) { throw new RuntimeException(e); }
    }
}

Points to volunteer while you write it, in rough order of value:

  • ceilingKey + the firstKey fallback is the entire algorithm. Everything else is bookkeeping. If you're short on time, write getNode first and say the rest is obvious.
  • Lookup is O(log(N·V)) — a binary search over a few thousand entries, i.e. ~12 comparisons. Nobody cares; it's nanoseconds and it's local. There's no network call to find the owner, which is exactly the property that makes this scale: every client independently computes the same answer from the same node list.
  • Don't use String.hashCode(). It's fast but poorly distributed for similar strings, and "similar strings" is precisely what your vnode labels are (node#0, node#1, …). Clustered vnodes defeat the purpose of having them. MD5 or Murmur3.
  • Replication is a two-line extension. To store each key on 3 nodes (as Dynamo/Cassandra do), don't stop at the first marker clockwise — keep walking and collect the next 3 distinct physical nodes. That "distinct physical" filter matters: with vnodes, the next two markers might belong to the same box you already picked, and you'd have three copies on one machine. Mentioning that filter unprompted is a real signal — it's the bug everyone writes first.
  • Thread safety: in production the ring is read constantly and mutated rarely (only on membership change). So don't lock reads — build a new immutable ring on change and swap a volatile reference. Copy-on-write, the same pattern you'd use for a React context value that changes once an hour.
The ring is also stateless derived data — it's a pure function of the node list. So the real distributed-systems problem isn't the ring, it's agreeing on the membership list: who is alive right now? Cassandra and Dynamo answer with a gossip protocol; a Redis Cluster client answers with the cluster's slot map; a Kafka client asks the controller. That's tutorial 05 (coordination) and tutorial 08 (the distributed cache). Point at it and move on — "and I'd need a way to agree on the live node set, either gossip or a coordination service like etcd" — is exactly the right amount to say here.

8. Hot keys & celebrity partitions

Here is the sentence that this entire section exists to install in your head, because it is the thing candidates freeze on:

Hashing gives you an even distribution of keys. It does not give you an even distribution of load.

Consistent hashing, virtual nodes, the ring — all of it assumes every key is worth about the same amount of traffic. Real workloads are Zipfian: a tiny minority of keys take a huge majority of the requests. The distribution is not a bell curve, it's a cliff.

  • The celebrity. A user with 200 million followers. Every one of their posts fans out to a fifth of the product. Their user row is read constantly.
  • The viral item. One tweet, one video, one product on Black Friday. Yesterday it had 3 reads/sec; right now it has 800,000/sec.
  • The whale tenant. You sharded a B2B SaaS by tenant_id. It works beautifully, and then you sign an enterprise customer who is bigger than your other 40,000 customers combined. They are one key. They live on one shard. That shard is now your product's capacity ceiling.
  • The default value. The dull one nobody mentions and everybody ships: 60% of rows have country = 'US' or status = NULL, and you partitioned on that column. Low cardinality is a hot key wearing a disguise.

In all four cases the hash function does its job perfectly. The key hashes to one position, that position belongs to one node, and that node is now serving a hundred times what its peers serve. It saturates its CPU or its NIC and starts timing out — and because a single shard being down usually looks like the whole product being broken, you have an outage on one-sixteenth of your fleet. No amount of virtual nodes helps. A single key cannot be split by a hash function. 

  hashing worked perfectly:  keys are spread evenly across shards
  load did not care:

  shard 0  ████                       2k QPS
  shard 1  ███                        1.5k QPS
  shard 2  ██████████████████████████████████████████  190k QPS  ◀── @celebrity
  shard 3  ████                       2k QPS

  Every shard holds ~25% of the KEYS. One shard takes ~95% of the LOAD.

The fixes — have four of these ready

This is a very common follow-up ("what if one user is much more popular than the rest?"), and the good answer is not one technique, it's a ladder from cheap to expensive. Walk it in order.

1. Cache the hot key — the first and usually sufficient answer

Hot keys are hot precisely because they're read over and over, which makes them the most cacheable objects in your entire system. A celebrity's profile has a 99.999% hit rate on any cache you put in front of it. So put one there: an in-process LRU in every application server, or a small Redis tier in front of the shard. Now the hot key is served from memory on the app tier and the hot shard never sees the traffic.

Say it like this: "Hot keys are the easiest thing in the world to cache — that's what makes them hot. A tiny local cache on each app server with a one-second TTL turns a million reads per second on one shard into one read per second per app server. I'd reach for that before I did anything structural." A one-second TTL sounds useless and is devastatingly effective: at a million QPS it eliminates 99.9999% of the load, and one second of staleness on a follower count is fine.

2. Request coalescing (single-flight)

When the hot key's cache entry expires, a thousand concurrent requests all miss at the same instant and all stampede the shard — you've re-created the problem at the TTL boundary. Fix it by letting one request go fetch and making the other 999 wait on that same in-flight promise. In Java that's a ConcurrentHashMap<Key, CompletableFuture<V>>; you already know this pattern from React Query / SWR deduplicating identical fetches in flight. Related: the thundering herd and cache stampede (tutorial 06).

3. Key salting / splitting — split the key across shards

If the key is write-hot rather than read-hot (a like counter on a viral post, an ad-impression counter), caching can't save you — you can't cache a write. So split the key into N sub-keys and reassemble on read. 

  HOT WRITE KEY                     SALTED
  likes:post_9931                   likes:post_9931:0   ──▶ shard 1
     │                              likes:post_9931:1   ──▶ shard 3
  all increments →                  likes:post_9931:2   ──▶ shard 0
  ONE shard, melts                  …
                                    likes:post_9931:9   ──▶ shard 2

  WRITE: pick a random salt 0..9, increment that counter    → load ÷ 10
  READ:  read all 10 and SUM them                           → 10 reads, 1 result

You've traded a cheap read for a cheap write, which is the right trade when writes are the problem. Note the salt count is a knob: 10 sub-keys divides the write load by 10 and multiplies the read cost by 10. And you don't need to salt everything — salt only the keys you've detected as hot, keeping a small set of "currently hot keys" that readers consult. Same technique, different name, in the ad-click aggregator (tutorial 18).

4. Replicate the hot shard / add read replicas

If the load is reads and you can tolerate replica lag, give the hot shard extra read replicas and fan reads across them. Straightforward and it composes with everything else. It doesn't help writes at all, and it's a per-shard-level fix — you end up with an asymmetric cluster, which is operationally annoying but entirely legitimate.

5. Special-case it — put the celebrity on its own shard

This is where the directory strategy from §3c earns its rent. Detect the whale, and pin it: give the biggest tenant its own dedicated shard, or its own dedicated cluster. Formulas can't express "this one is different"; a lookup table can. Every mature multi-tenant system ends up with a handful of hand-placed customers, and saying so is a mark of someone who's run one.

6. Change the algorithm entirely

Sometimes the right response isn't to spread the hot key but to stop touching it. The canonical example is the news feed: fan-out-on-write pushes a post into every follower's feed, which is fine for you and catastrophic for a celebrity with 200M followers. The real answer is a hybrid — fan out on write for normal users, and for celebrities don't fan out at all; pull their posts in at read time and merge. That's not a partitioning fix; it's a design change driven by exactly this problem. Forward reference: tutorial 10 (News Feed) is where the celebrity problem gets its full treatment, and it's the highest-frequency version of this question in real loops.

You also have to detect the hot key

All of the above assumes you know which key is hot, and hot keys appear without warning — that's what "viral" means. So you need per-key traffic accounting, which you can't do exactly at a million QPS. Sample requests, or keep an approximate heavy-hitters structure (a count-min sketch — fixed memory, small overcount, never undercounts) at the proxy or client, and emit "top-K keys by request rate" every few seconds. Then a hot key can trigger salting or promotion into a local cache automatically. Naming count-min sketch here is a nice, cheap flex; you don't need to derive it.

The trap answer is "I'd use consistent hashing with virtual nodes, so the load is balanced." That answers a different question. Virtual nodes balance the keyspace; a hot key is a single point in the keyspace. If you offer this, the interviewer will simply ask "but all million requests are for the same key — where do they go?" and you'll have burned your credibility on the topic. Get ahead of it: say "note that vnodes don't help here — one key is one position on the ring, so it's one node no matter what. Hot keys need a different set of tools."
The compressed answer to "your shard is hot — what do you do?", which is worth rehearsing until it's fluent: "First I'd find out whether it's hot on reads or on writes, because the fixes are disjoint. If it's reads: cache the key — hot keys are trivially cacheable, and a one-second TTL in a local app-server cache kills nearly all of it — plus request coalescing so I don't stampede on expiry, and read replicas of that shard if I still need more. If it's writes: I can't cache a write, so I'd salt the key — split the counter into ten sub-keys on ten shards, increment a random one, sum on read. And if the hotness is structural rather than transient — a whale tenant — I'd stop trying to be clever and just pin it to its own shard through a directory layer. What I would not say is 'add virtual nodes': a single key is a single point on the ring, so that changes nothing."

9. Rebalancing without a full reshuffle

Rebalancing is what happens after the ring diagram: nodes get added because you grew, removed because you shrank, and replaced because they died. The interviewer's question is "you added a node — now what physically happens to the data, and is the system up while it happens?" Consistent hashing tells you which keys must move. It says nothing about how they move, and that's the operational half of the answer.

Fixed partition count — the pragmatic answer, and the one to lead with

The move used by Riak, Kafka, Elasticsearch and (in effect) Cassandra's vnodes: decouple the number of partitions from the number of nodes. At day one, create far more partitions than you have machines — say 1,024 — and never change that number. Then assign partitions to nodes, many per node. 

  1024 fixed partitions, 4 nodes:      add a 5th node:

  node A: partitions   0-255           node A: 0-204     (gave away ~51)
  node B: partitions 256-511    ──▶    node B: 256-460   (gave away ~51)
  node C: partitions 512-767           node C: 512-716   (gave away ~51)
  node D: partitions 768-1023          node D: 768-972   (gave away ~51)
                                       node E: ~205 partitions, taken in
                                               whole chunks from A,B,C,D

  hash(key) → partition   NEVER CHANGES.   ← the key point
  partition → node        is the only mapping that moves.

The key never moves between partitions. Only whole partitions move between nodes. That's the whole idea, and it's a beautiful simplification, because it turns rebalancing from "recompute where 200 billion keys live" into "move 51 files and update a small metadata table." The key → partition map is a pure, fixed function that every client can compute offline. The partition → node map is a tiny piece of cluster metadata — 1,024 rows — that lives in ZooKeeper/etcd or is gossiped, and it's the only thing that changes.

Choosing the partition count is a real trade-off and worth naming:

Too few partitionsToo many partitions
Coarse-grained — you can't balance well across many nodes, and each partition is huge, so moving one is a big slow transferPer-partition overhead: files, memory, open handles, metadata, replication streams. Kafka in particular degrades with tens of thousands of partitions.
Caps your maximum cluster size at (partitions) nodesWasteful at small scale, and the metadata map gets unwieldy

The rule of thumb: pick a number comfortably larger than the biggest cluster you can imagine — 10× your current nodes is a common heuristic — and accept it as permanent, because changing it later means rehashing everything, which is the problem you were avoiding. (Kafka lets you add partitions, but doing so breaks per-key ordering for existing keys because hash(key) % partitions changes. Same disease as §4, in a different costume. Over-provision the partition count up front.)

Lead with this rather than with the ring when the question is about rebalancing specifically: "I'd fix the partition count up front — say 1,024 — much larger than the node count, and never change it. Keys hash to partitions, and that mapping is permanent. Rebalancing is then just reassigning whole partitions to nodes, which is a metadata change plus a bulk file copy — I never rehash a single key. Consistent hashing and fixed partitions are complementary: the ring tells me who owns what without a coordinator; the fixed partition count makes the unit of movement coarse and cheap."

How the data actually migrates, with the system up

You can't take the write path offline while you copy gigabytes. The standard dance, and it is basically the same in every system:

  1. Assign, but don't cut over. The new node is told it will own partitions 205–255. The old owner keeps serving all reads and writes for them. Nothing has changed for clients yet — this is the crucial property.
  2. Bulk copy. Stream the existing data for those partitions from the old owner to the new one, throttled so the migration doesn't starve live traffic of I/O. This may take hours. Everything still works.
  3. Catch up on the delta. While you copied, writes kept arriving. So the old owner also forwards its ongoing writes for those partitions (or the new node tails the replication log / commit log) until the new node is within a few milliseconds of live.
  4. Flip the pointer. Atomically update partition 205 → node E in the metadata store. Traffic starts arriving at E. This step must be atomic and ordered, which is exactly why a coordination service (ZooKeeper, etcd) is used for the map rather than a config file — see tutorial 05.
  5. Handle in-flight stragglers. Clients cache the map, so some will briefly still ask the old owner. Two standard answers: the old owner redirects them (Redis Cluster does this literally, with a MOVED response carrying the new address), or it proxies the request through for a grace period. Either way clients refresh their map and converge. This is also where a version number on the map earns its keep.
  6. Delete on the old node, after a delay. Not immediately — keep it until you're confident, so a bad cutover can be rolled back by flipping the pointer again.
Rebalancing should be manual, or at least human-approved — say this. Automatic rebalancing plus automatic failure detection is a famous way to destroy a cluster: a node gets slow (GC pause, saturated NIC), the system decides it's dead, and starts migrating its data away — which consumes network and makes the node actually unresponsive, so more nodes are declared dead, so more data moves. The cluster rebalances itself to death under load. Both the Dynamo paper and Designing Data-Intensive Applications come down on the side of "automate the mechanism, but keep a human in the decision." Offering that unprompted is a strong operational-maturity signal, and it costs one sentence.

Removal and failure are different events — don't conflate them

  • Planned removal (scale-down, host replacement): the node drains first — it copies its partitions to their new owners while still serving, then leaves. No availability impact.
  • Unplanned failure (crash): its data is gone from that box, so its replicas must serve while a new node backfills from them. This is why replication and partitioning are two different axes and you need both: a partition with no replica is a partition you lose. In Dynamo-style systems the write that arrives during the gap gets hinted handoff — a neighbour temporarily accepts and stores it with a note saying "this belongs to the dead node, deliver it when it returns." That's tutorial 08.

10. The cross-shard problem

Everything above assumed you know the partition key at request time. When you do, sharding is nearly free: one hop, one node, one index lookup. The bill arrives for every query that doesn't name the partition key, and for every write that touches two of them. This is the permanent tax, and a senior candidate quantifies it rather than waving at it.

10a. Cross-shard joins → scatter-gather

JOIN across shards does not exist. Your database can't do it, because the two tables are on different machines that don't know about each other. Your options are all worse than a join:

  • Co-locate. Choose a partition key that keeps things that are joined together on the same shard. Shard orders and order_items both by customer_id, and the join is local — a normal Postgres join on one box. This is the best answer and you should always reach for it first. It's why picking the partition key is really picking your join boundary.
  • Denormalize. Copy the fields you need into the row that needs them. Store author_name on the post instead of joining to users. You now own the consistency problem (rename → update N rows, asynchronously) and you accept it, because a stale display name for thirty seconds is not a real problem.
  • Application-side join. Fetch from shard A, collect the IDs, fetch from shard B, stitch in your code. This is what you actually do, and it's fine at small fan-out. You've moved the join into the app tier, where it has no indexes and no query planner.
  • Scatter-gather. Ask every shard, merge the answers.
  SCATTER-GATHER — "find all users named 'Chen'" (shard key is user_id)

The tail-latency trap — this is why P99 matters

Here is the point most candidates miss, and it is a genuinely strong thing to say. A scatter-gather is as slow as its slowest shard. You must wait for all of them, so your response time is the maximum of 16 samples, not the average. And that maximum is drawn from the tail of the latency distribution.

Do the arithmetic, because the number is shocking:

  • Each shard is fast: P99 = 100 ms means only 1% of requests are slow. Sounds great.
  • You query 16 shards in parallel. The chance all are fast is 0.99¹⁶ ≈ 0.85.
  • So ~15% of your user-facing requests take 100 ms or more — even though each individual shard hits its P99 SLA perfectly. Your service's P85 is now your shards' P99.
  • Go to 100 shards: 0.99¹⁰⁰ ≈ 0.37. 63% of requests hit a slow shard. Your median is now your shards' tail.

This is Jeff Dean's "tail at scale," and it's the reason latency is measured at P99/P99.9 and not as an average: at fan-out, the tail of your components becomes the body of your service. Averages are a lie for exactly this reason. The mitigations are worth knowing: hedged requests (after P95 elapses, send a duplicate request to a replica and take whichever answers first — costs ~5% extra load, cuts the tail dramatically), tolerating partial results (return 15 of 16 shards' results and degrade rather than block — a search page can do this; a bank balance cannot), and most importantly designing the access pattern so you don't scatter-gather at all.

The sentence that will make an interviewer sit up: "The real cost of scatter-gather isn't the extra work, it's the tail. I have to wait for the slowest shard, so my latency is the max of N samples, not the mean. If each shard has a 100ms P99 and I fan out to 16, then 1 minus 0.99 to the 16th — about 15% of my requests — hit at least one slow shard. My service's P85 is my shards' P99. That's why I'd design the partition key so the hot path never fans out, and why I'd use hedged requests if it has to."

"Transfer $100 from account A to account B" is an ACID one-liner on a single Postgres node — you did this in Phase 1. Put A and B on different shards and there is no single node that can commit or roll back both. Your options, all covered in tutorial 05:

The trade is read cost vs write cost, and consistency is the tiebreaker. Local (a.k.a. document-partitioned) indexes make writes fast and consistent, reads a fan-out — this is Cassandra's and Elasticsearch's model, and it's fine when reads are already scoped to a shard or when fan-out is affordable. Global (term-partitioned) indexes make reads a single hop and writes a distributed, usually asynchronous, eventually-consistent problem — DynamoDB's GSIs.

For the specific "login by email" case there is a much simpler answer that you should give first, because it's what you'd actually ship: keep a separate small email → user_id lookup table, sharded by email. It's a hand-rolled global index over one column, it's a single extra hop, and it's easy to reason about. Then add: "which is the same shape as a global secondary index — I'm just building it explicitly instead of asking the database to."

Unique constraints die when you shard, and almost nobody mentions it. UNIQUE(email) is enforced by one node inspecting one index. Across 16 shards, two concurrent signups with the same email can land on two different shards and both succeed — neither node can see the other. The fixes: (a) route the uniqueness check through a single shard by sharding the uniqueness table by the unique column (the email lookup table above, with a unique constraint on it, written first — a "claim" record), or (b) accept it and reconcile after the fact. Bringing this up unprompted is a strong signal, especially in a fintech loop where correctness is the thing being graded.

Interview Q&A

"Walk me through consistent hashing." ★ (the one you will actually be asked)

Structure your answer in four beats. Do not skip beat 1 — it's what makes the rest land.

1. The problem. "The naive way to spread keys over N nodes is hash(key) % N. That's uniform and needs no coordination, but the modulus is baked into every key's mapping. So the moment N changes, the whole mapping changes. Going from 4 nodes to 5, about 80% of keys land on a different node. If those nodes are caches, my hit rate collapses and the full read load lands on the database at once — a scaling action becomes an outage. And a node dying is the same event, so I can't even avoid it by not scaling."

2. The construction. "So I want a mapping where adding or removing a node only disturbs that node's share. Take a hash with a 32-bit output and treat the output space as a circle. Hash each server's ID onto the circle. Hash each key onto the same circle. A key is owned by the first server you hit walking clockwise. Equivalently, each server owns the arc behind it." (Draw it. Four dots for servers, a few for keys, arrows clockwise.)

3. The property. "Now add a server. It lands inside exactly one existing arc and takes over the part of it before its position. Only those keys — about K/N — change owner. Every other key's clockwise walk never passes the new position, so it's provably untouched. Remove a server and its arc merges into its clockwise neighbour; only its own keys move. That's the guarantee: O(K/N) remapping instead of O(K)."

4. The problem with the basic version, and the fix. "With only a handful of servers the arcs are lumpy — one node can easily own 40% of the ring. Worse, when a node dies its entire range dumps on one neighbour, which doubles that neighbour's load and can take it down too — you cascade around the ring. So each physical node gets ~150 virtual positions instead of one: hash node#0, node#1, … node#149. Ownership becomes many small interleaved arcs, load evens out, and a dead node's ranges are picked up by all the survivors in small slices. It also lets me weight bigger machines by giving them more positions."

Then stop and let them ask. If they want implementation: "it's a TreeMap from ring position to node, and ceilingKey is the clockwise walk, falling back to firstKey to wrap around."

"Your shard is hot — what do you do?" ★

First, diagnose. The fixes for read-hot and write-hot are completely different, and asking which it is is part of the answer.

Read-hot (a celebrity profile, a viral post):

  1. Cache it. Hot keys are the most cacheable objects you have — that's what makes them hot. An in-process LRU on every app server with even a 1-second TTL turns a million reads/sec on one shard into one read/sec per app server. Do this before anything structural.
  2. Coalesce requests so the TTL expiry doesn't stampede: one in-flight fetch, everyone else waits on it.
  3. Read replicas of that shard if you still need more, accepting replica lag.

Write-hot (a like counter on a viral post, ad impressions):

  1. Salt the key. You can't cache a write, so split it: likes:post_9931:0..9, increment a random one, sum the ten on read. Write load ÷ 10, read cost × 10 — the right trade when writes are the problem.
  2. Or buffer and batch — aggregate increments in memory / in a stream processor and flush periodically. That's the ad-click aggregator's answer.

Structurally hot (a whale tenant, permanently 100× the others): stop being clever and pin it — give it its own dedicated shard via a directory/lookup layer. Formulas can't say "this one is different."

And name the trap: "Note that virtual nodes don't help here. A single key hashes to a single point on the ring, which is a single machine, no matter how many virtual positions that machine has. Hot keys are a load problem, not a keyspace problem."

"Why not just use hash(key) % N? Show me why it breaks."

Do it with numbers — it's far more convincing than the general argument. "Take hash 17 with 4 nodes: 17 % 4 = 1. Add a node: 17 % 5 = 2. It moved. A key only stays put if h % 4 == h % 5, which happens for roughly 1 in 5 keys. So going from 4 to 5 nodes remaps about 80% of keys. Going from 100 to 101 remaps about 99%."

"The reason is structural: the modulus is part of the mapping for every key, so changing N doesn't adjust the function, it replaces it — there's no relationship between the old assignment and the new one."

"Concretely, if those are cache nodes, adding one node collapses the hit rate from 95% to about 20%, so my database instantly sees 16× its normal read load and falls over. The same thing happens when a cache node just crashes. That's a system that punishes me for both growth and failure, so I need a mapping where a membership change only disturbs that node's slice — consistent hashing."

Follow-up they may push: "What about doubling — 4 to 8?" "That's better: h % 8 is either h % 4 or h % 4 + 4, so exactly half the keys move. But I can only ever double, I still move 50%, and it does nothing for the case that matters — a node dying isn't a doubling."

"When would you NOT shard?"

"Almost always, until the numbers force me. In order, I'd exhaust: vertical scaling (a modern box holds 30 TB and 1 TB of RAM — most 'we need to shard' problems fit on one machine), read replicas (nearly free, and most workloads are read-heavy), caching the hot working set, archiving cold rows out of the hot table, and indexing / query fixes — a surprising number of 'we're out of capacity' problems are one missing index."

"I'd shard when writes exceed what a single leader can absorb, or when the working set no longer fits in RAM and I'm I/O bound, or when the data simply doesn't fit on the biggest available disk. Those are physical limits. Everything before that is a preference."

"The reason I'm cautious is that sharding costs me joins and single-node transactions permanently, and it's very hard to undo. It's a one-way door."

"How do you pick the partition key?"

"By looking at the dominant access pattern, not at the data. Three criteria:"

  • It should appear in most queries, so the common request is single-shard. If 95% of my reads are 'the last 50 messages in channel X', then channel_id is the key. Any key that turns my hottest query into a scatter-gather is the wrong key, however evenly it distributes.
  • High cardinality and even access. country or status is a hot key wearing a disguise — 60% of rows will share one value.
  • It should co-locate things that are joined or transacted together. Picking a partition key is really picking your join boundary and your transaction boundary. Shard orders and order_items both by customer_id and the join stays local.

"And I'd say out loud that this decision is effectively irreversible once there's data on disk, so it's worth spending real time on — it's the one choice that's expensive to get wrong."

"You've added a node to the ring. What actually happens to the data — is the system up?"

"The ring tells me which keys move; it says nothing about how. The migration is a five-step dance and the system stays up throughout:"

  1. Assign but don't cut over — the new node is told it will own these ranges; the old owner keeps serving them. Nothing changed for clients.
  2. Bulk copy the existing data, throttled so we don't starve live traffic of I/O. Can take hours; everything still works.
  3. Catch up the delta — the old owner forwards ongoing writes for those ranges (or the new node tails the commit log) until it's within milliseconds of live.
  4. Flip the pointer atomically in the metadata store. This is why the partition map lives in something like etcd/ZooKeeper rather than a config file — the flip has to be atomic and ordered.
  5. Handle stragglers — clients cache the map, so some still hit the old owner briefly. It either redirects them (Redis Cluster literally returns MOVED with the new address) or proxies for a grace period, and clients refresh. Then delete the old copy, after a delay, so a bad cutover can be rolled back.

"And I'd keep rebalancing human-approved, not fully automatic: automatic rebalancing plus automatic failure detection is how clusters kill themselves — a node GC-pauses, gets declared dead, its data starts migrating, the migration saturates the network, more nodes look dead, and it cascades."

"Explain virtual nodes. Why not just one position per node?"

"Two reasons, and the second is the important one."

1. Load balance. "Random points on a circle clump. With four nodes it's entirely normal for one to own 45% of the ring and another 8% — you pay for four machines and get the capacity of about two and a half. Giving each node ~150 positions averages 150 random arcs per node, which crushes the variance and converges ownership on 1/N."

2. Failure behaviour — the real reason. "With one position per node, a node's range is one contiguous arc, and its only clockwise successor is one neighbour. So when it dies, 100% of its load lands on that single machine, roughly doubling it. That neighbour falls over, and its combined range dumps on the next one — you cascade around the ring and lose the cluster from one node death. With virtual nodes, the dead node's 150 arcs each merge into whatever marker is clockwise, and those belong to different physical machines — so its load is picked up by every survivor, about 1/(N−1) extra each. Nobody notices. The same is true when a node joins: it takes small slices from everyone, so the migration is parallel instead of hammering one donor."

3. Free bonus. "Ownership is proportional to marker count, so I can weight heterogeneous hardware just by giving the bigger boxes more positions — 200 vs 100. Real fleets are never homogeneous."

And be precise about what they are: "virtual nodes are entries in a sorted map, not processes. Nothing gets deployed."

"Range-based or hash-based sharding for a time-series workload?"

"This is the trap question, so I'd name the trap. Pure range partitioning by timestamp is the worst possible choice even though it looks natural: all of today's writes fall into the newest range, so 100% of write traffic goes to one shard while the others sit idle holding cold history. I've bought N machines and I'm writing to one."

"Pure hash partitioning fixes the writes and destroys the reads — 'give me the last hour for sensor 7' now has to ask every shard and merge, because ordering was deliberately destroyed."

"The answer is a compound key. Partition by (sensor_id, timestamp): hash on sensor_id so writes from thousands of sensors spread evenly across the cluster, and store rows sorted by timestamp within each partition so 'the last hour for one sensor' is still a local ordered range scan. That's exactly Cassandra's model — PRIMARY KEY ((sensor_id), timestamp): hash between partitions, sort within them."

"What I give up is 'the last hour across all sensors' as a cheap query — that becomes a scatter-gather. If that's actually the dominant query I'd handle it with a separate rollup/aggregation pipeline rather than by re-partitioning."

"How do you query by a non-shard-key column? E.g. find the user by email when you sharded by user_id."

"Three options, in the order I'd offer them."

1. A dedicated lookup table — the one I'd actually ship. A small email → user_id table, sharded by email. Login becomes two hops: hash the email, get the user_id, hash the user_id, get the user. It's a hand-rolled global index over one column, it's cheap, and it's easy to reason about."

2. A global (term-partitioned) secondary index — the same shape, done by the database. Reads are one hop. Writes get expensive: writing a user now touches its data shard and a different index shard, so it's a cross-shard write, which in practice is done asynchronously — meaning the index is eventually consistent. DynamoDB GSIs behave exactly like this and it's a documented gotcha: you can write a row and briefly not find it by the indexed attribute."

3. A local (document-partitioned) secondary index — each shard indexes only its own rows. Writes are cheap and consistent (one local transaction), but reads must ask every shard and merge — scatter-gather. This is Cassandra's and Elasticsearch's model, fine when reads are already shard-scoped or fan-out is affordable."

"So it's read cost vs write cost, with consistency as the tiebreaker. For login-by-email specifically, reads are constant and writes are rare, so I want the global shape — I'd just build it explicitly."

"What breaks when you scatter-gather? Why do people care so much about P99?"

"Because a scatter-gather is as slow as its slowest shard — I have to wait for all of them, so my latency is the maximum of N samples, not the mean. And the max is drawn from the tail."

"Do the numbers. Say each shard has a P99 of 100ms — only 1% of its requests are slow, which sounds excellent. I fan out to 16 shards. The chance that all sixteen are fast is 0.99¹⁶ ≈ 0.85. So 15% of my user-facing requests take 100ms or more, even though every shard is meeting its SLA perfectly. My service's P85 is my shards' P99. At 100 shards, 0.99¹⁰⁰ ≈ 0.37 — 63% of requests hit a slow shard."

"That's 'the tail at scale', and it's exactly why we measure P99 and not averages: at fan-out, the tail of your components becomes the body of your service. An average would hide all of this."

"Mitigations: hedged requests — once P95 elapses, fire a duplicate at a replica and take whichever returns first; costs ~5% extra load and cuts the tail dramatically. Tolerate partial results — return 15 of 16 shards and degrade, which a search page can do and a bank balance cannot. And above all, choose the partition key so the hot path never fans out in the first place."

"You sharded the users table. What happened to UNIQUE(email)?"

"It quietly stopped working, and this is the kind of thing that gets shipped and discovered in production."

"A unique constraint is enforced by one node checking one index. Once users is spread over 16 shards, two concurrent signups with the same email hash to two different shards, each node checks its own local index, sees nothing, and both succeed. Neither node can see the other. The constraint is now a local constraint pretending to be a global one."

"Two fixes. The clean one: make the uniqueness check itself single-shard by putting the unique column in a claim table sharded by that column — an emails table partitioned by email, with a real unique constraint, written first as a claim. Whoever wins that insert gets to create the user; the loser gets a duplicate-email error. That's the same lookup table I'd already want for login-by-email, so it costs nothing extra."

"The pragmatic one: accept the tiny race and reconcile — detect duplicates asynchronously and merge or reject. Fine for a username; not fine for anything financial."

"How many partitions would you create, and why can't you change it later?"

"I'd fix the partition count up front at a number much larger than my node count — 1,024 is a common choice — and treat it as permanent. Keys hash to partitions, and that mapping never changes; only the partition → node assignment moves. So rebalancing is 'move 50 whole partitions and update a small metadata table', never 'rehash 200 billion keys'. Riak, Elasticsearch and Kafka all work this way."

Too few: "I cap my maximum cluster size at the partition count, and each partition is huge, so moving one is a big slow transfer and balancing is coarse."

Too many: "Per-partition overhead — files, memory, open handles, metadata, replication streams. Kafka degrades badly with tens of thousands of partitions."

Why it's permanent: "Changing the count changes hash(key) % partitions for existing keys — which is precisely the mod-N disaster again, one level up. Kafka does let you add partitions, and the price is that per-key ordering breaks for existing keys, because a key can now map to a different partition than its history. So: over-provision on day one and never touch it."

"Is consistent hashing always the right answer for partitioning?" (a good candidate says no)

"No, and I'd be suspicious of a design that reaches for it reflexively."

"Consistent hashing solves one specific problem: minimizing remapping when membership changes, with no central coordinator. It shines when nodes join and leave often and clients must route independently — caches, Dynamo-style stores, sticky load balancing."

"If membership is stable and you already have a coordination service, a directory / explicit partition map is often better: it's easier to reason about, easier to debug ('where does tenant 42 live?' is a lookup, not a hash computation), and it lets you place data deliberately — pin a whale tenant, satisfy data-residency rules, isolate a noisy neighbour. A formula can't say 'this one is different', and that one always exists."

"If range queries dominate and the key isn't monotonic, range partitioning beats both, because it's the only scheme that preserves ordering."

"In practice the best real designs combine them: hash-based placement as the default, a fixed large partition count so movement is coarse, and a thin directory layer so exceptions can be hand-placed. Formula for the 99%, table for the 1%."

What to remember

  • Sharding is a last resort, not an opening move. Vertical scaling → read replicas → caching → archiving → then shard. Shard when writes exceed one leader, or the working set won't fit in RAM. Saying this unprompted is worth more than a perfect ring diagram.
  • The permanent bill for sharding: no cross-shard joins, no single-node transactions, no global unique constraints, rebalancing, hot shards. It's a one-way door.
  • Vertical partitioning = split columns by concern (cache efficiency; it's what microservices do to a schema). Horizontal partitioning / sharding = split rows (the one that scales).
  • Picking the partition key is the whole design. Choose it from the dominant access pattern: it must appear in most queries, have high cardinality, and co-locate what you join and transact together. It's effectively irreversible.
  • Range = great range scans, hot-spots on sequential keys (timestamps, auto-increment IDs — 100% of today's writes on one shard). Hash = even distribution, kills range scans. Directory = maximum flexibility and per-tenant placement, but it's a SPOF and an extra hop. Compound keys (hash between partitions, sort within — Cassandra) get you both.
  • hash(key) % N is a trap. 4 nodes → 5 remaps ~80% of keys; 100 → 101 remaps ~99%. For a cache that's a total miss storm and the database dies. A node crashing is the same event, so you can't avoid it by not scaling. Derive this before naming consistent hashing.
  • Consistent hashing, in one sentence: hash nodes and keys onto the same circular space; a key is owned by the first node clockwise from it. Adding or removing a node only remaps that node's arc — ~K/N keys, not ~K. No coordinator needed: every client computes the same ring from the same node list.
  • Virtual nodes (~150 positions per physical node): the arcs stop being lumpy, and — the real reason — a dead node's load spreads across all survivors instead of dumping entirely on one neighbour and cascading. Bonus: weight heterogeneous hardware by giving bigger boxes more positions. They are map entries, not processes.
  • Implementation: a TreeMap<Long, Node>. ceilingKey(hash(key)) is the clockwise walk; firstKey() on null is the wraparound. For replication, keep walking to the next R distinct physical nodes.
  • Hashing evens out keys, not load. Hot keys — celebrities, viral posts, whale tenants, low-cardinality defaults — melt one shard no matter how many vnodes you have. Read-hot: cache it (a 1s TTL kills 99.9999% of load), coalesce requests, add read replicas. Write-hot: salt the key into N sub-keys and sum on read; or buffer and batch. Structurally hot: pin it to its own shard via a directory.
  • Rebalancing: fix a large partition count up front (1,024) and never change it. Keys→partitions is permanent; only partitions→nodes moves. Migrate live: assign → bulk copy (throttled) → catch up the delta → atomically flip the pointer → redirect stragglers → delete late. Keep the decision human — auto-rebalance plus auto-failure-detection is how clusters kill themselves.
  • Scatter-gather's real cost is the tail. You wait for the slowest shard, so latency is the max of N samples. 16 shards each with a 100ms P99 ⇒ ~15% of your requests are slow. Your service's P85 is your shards' P99. That's why P99 matters and averages lie. Mitigate with hedged requests, partial results, or — best — a partition key that never fans out.
  • Secondary indexes: local/document-partitioned = cheap consistent writes, scatter-gather reads (Cassandra, Elasticsearch). Global/term-partitioned = one-hop reads, cross-shard async writes ⇒ eventually consistent (DynamoDB GSIs). For login-by-email, just build an email → user_id table sharded by email, and say it's a hand-rolled GSI.
  • Sharding silently breaks UNIQUE. Two concurrent signups on two shards both win. Fix with a claim table sharded by the unique column, written first.
  • Where this reappears: the distributed cache (08) is consistent hashing in anger; the rate limiter (09) uses it to place counters; the news feed (10) is the celebrity problem at full size; the ad-click aggregator (18) is key-salting at scale. Learn it once here.
If you remember one thing from this page, remember to derive before you name. Anyone can say "consistent hashing." Only a senior candidate says: "mod N breaks because changing N replaces the mapping for every key — 80% of them move — so I need a scheme where a membership change only disturbs one node's slice; that's the ring; and the ring's own failure mode is that a dead node dumps everything on one neighbour, so I need virtual nodes."
