22 — Design a Component Library / Design System

The only frontend round where the "users" are other engineers, and the product is an API you can never take back.

Why this matters in the interview

Every other page in this block designs a screen — a feed, a typeahead, a collaborative editor. This one designs a tool that other teams build screens with, and that single shift changes what's being graded. There is no scroll to virtualize, no 3G race to win, barely any server at all. Instead the interviewer is probing a different kind of seniority: can you design an API that dozens of teams will depend on, that has to stay accessible and on-brand no matter who consumes it, and that you have to evolve for years without breaking every app in the company on a Tuesday. This is the round that separates people who use component libraries from people who could own one — and at 7 years in React, owning one is exactly the level you're interviewing at. If you've ever fought a third-party component that was almost-but-not-quite right, you already have the instincts; this page turns them into a structured answer.

The method is still RADIO (page 19), but it leans hard toward the A and the I. A design system has almost no data flow, so the "D — Data model" step becomes your token architecture — the tiered variables that are, quite literally, the data model of your visual language. The "I — Interface & API" step becomes the heart of the whole interview, because the component's props are the product: compound components, headless hooks, controlled-vs-uncontrolled, and — the highest-signal idea on the page — accessibility as a contract the library owns so consumers can't get it wrong. And the "O" step grows a second half that no screen-design round has: the operational concerns of shipping a library — bundle size and tree-shaking, semantic versioning, breaking changes and codemods, documentation and adoption. Get those right and you're not describing a component library; you're describing a platform.

The framing to open with, and it reframes the entire round: "The thing to hold onto here is that my users are other engineers, and my product is an API contract. So I'll optimize for three things a screen design doesn't care about: that the API is hard to misuse — especially around accessibility, which the library should own so consumers can't get it wrong; that I can evolve it for years without breaking every consuming app; and that a team importing one button doesn't ship my entire library in their bundle. Everything else follows from treating this as API design, not UI design." Say that in the first ninety seconds and you've told the interviewer you've maintained a design system, not just installed one.

1. The prompt

The interviewer gives you the vaguest possible version, on purpose:

"Design a component library — a design system — that all the product teams at our company will use to build their UIs."

Notice what's not here: no user count, no latency target, no data to model. That's not an oversight — it's the tell that this is the one frontend round where scale and performance-of-a-screen are almost irrelevant, and the graded axes are entirely different. The load-bearing questions are all about consumers: how many teams, on what frameworks, how much can you constrain them versus how much do they need to customize, and how do you ship changes to a hundred apps you don't control.

The mid-level move is to start listing components — "okay, a Button, an Input, a Modal…" — and design each one's props. The senior move is to spend the first minutes establishing that a design system is three products stacked on top of each other: a token layer (the visual language as data), a component layer (the React API teams import), and an operational layer (versioning, docs, bundle strategy, adoption). Name those three and you've given the interview its structure before you've drawn a single button.

A real interviewer often narrows this live: "focus on how you'd design the components' APIs," or "walk me through theming," or "how would you ship a breaking change." Treat the prompt as a menu. The sections below are the full 45-minute sweep; in the room you'll go deep on one or two and gesture at the rest. Hand the interviewer the menu — "this breaks into tokens, component API, accessibility, and the shipping story; where do you want me deepest?" — the same menu-then-dive move from the RADIO Optimizations step.

R — Requirements

The R step is the same first move as every RADIO round — pin down the surface before you build — but for a library the clarifying questions are about your consumers and your constraints, not about end-user interactions. You're not asking "what can the user do on this screen," you're asking "who imports this, how much do I let them bend it, and how do I ship them changes." Ask three or four of these, state your assumptions, and move — the point is to prove you know the axes exist, not to fill a form.

Functional requirements — what the system provides to consuming teams

Not "components on a screen," but capabilities the platform exposes: a set of accessible, themeable components (buttons, inputs, selects, modals, tooltips, tables); a token layer so brand and theme changes propagate without touching component code; escape hatches so a team with a genuinely unusual need isn't blocked; and a documentation surface teams can learn from without reading source. Enumerate those and you've enumerated the three layers again — tokens, components, operations.

Non-functional requirements — where a library round announces itself

This table is the equivalent of the "devices / perf budget / a11y" list from the RADIO page, re-pointed at a library. Reciting a few of these unprompted is one of the fastest ways to signal you've maintained a design system and not just consumed one.

Don't skip straight to enumerating components and their props. The classic junior answer is a parade of Button / Input / Modal prop tables with no architecture underneath — which misses that the interesting decisions (how do all these components share a look, stay accessible, and evolve together) live in the layers below and around the components, not in any single one. One well-designed <Select> that demonstrates tokens, headless composition, and the a11y contract is worth more than ten prop tables.
Surface the central tension out loud, because it's the theme the interviewer will keep pressing on: "The core trade-off in a design system is consistency versus flexibility. If I lock everything down, teams with legitimate edge cases will fork the component or drop to raw HTML and I lose all consistency anyway. If I let everyone override everything, the system stops guaranteeing brand or accessibility. So my strategy is: make the common path effortless and on-rails through tokens and variants, and provide deliberate, documented escape hatches — a headless version, a className/style pass-through, a Slot — for the genuine 10%. The escape hatch is what keeps teams inside the system instead of leaving it." Naming that tension first, then showing every later decision resolving it, is the spine of a strong answer.

A — Architecture: the layered system

What it is: the RADIO "Architecture" step, except your boxes aren't screen components — they're the layers of the system and the packages you ship. This is where you prove you see a design system as a stack, not a bag of components. The single most important diagram you can draw is the dependency direction: everything flows up from tokens, and every layer depends only on the ones below it. 

  ┌─────────────────────────────────────────────────────────┐
  │  Patterns / recipes   Form, PageLayout, DataTable...     │  composed from ↓
  ├─────────────────────────────────────────────────────────┤
  │  Components           Button, Input, Select, Modal...    │  styled by tokens ↓
  ├─────────────────────────────────────────────────────────┤
  │  Headless behavior    useSelect, useDisclosure, Focus... │  a11y + interaction
  ├─────────────────────────────────────────────────────────┤
  │  Design tokens        color, space, type, radius...      │  the visual language
  └─────────────────────────────────────────────────────────┘
        every layer depends only on the layers BELOW it

Walk it bottom-up, because that's the order the value flows. Tokens are the visual language expressed as data — the single source of truth for every color, space, font size, radius, and shadow. Headless behavior is the interaction and accessibility logic with no markup — the keyboard handling, focus management, and ARIA wiring that a Select needs, extracted so it can be reused and tested independently of how it looks. Components marry the two: headless behavior for correctness, tokens for appearance, plus the actual DOM. Patterns are opinionated compositions of components — a Form that wires labels, inputs, validation, and error messaging together so a team doesn't reassemble it (and get the a11y wrong) on every screen. The rule that keeps this maintainable: dependencies point down only. A token never imports a component; a component never imports a pattern. Violate that and you get cycles that make the whole thing impossible to version or tree-shake.

How you ship it — the package structure

Architecture on disk matters here because it directly enables (or destroys) tree-shaking and independent versioning. The standard shape is a monorepo (pnpm/npm workspaces + a tool like Turborepo or Nx) with separate publishable packages, so the token layer can be consumed by a non-React app and so each concern versions on its own cadence. 

  packages/
    tokens/        // @acme/tokens — framework-agnostic: CSS vars + JS/JSON
    react/         // @acme/react — the React components (depends on tokens)
    icons/         // @acme/icons — tree-shakeable icon set, one export per icon
    codemods/      // @acme/codemods — migration scripts shipped WITH breaking changes
  apps/
    docs/          // the documentation site (Storybook / a docs app) — dogfoods the library
    playground/    // a sandbox app for developing components in isolation

Two decisions to defend. First, tokens are their own package and deliberately framework-agnostic — that's what lets a future Vue or native app share the exact same visual language, and it's the answer to the "multi-framework?" requirement even if today everyone's on React. Second, one React package, not one package per component. Publishing @acme/button, @acme/input, and forty more is a versioning nightmare (which combination of versions is compatible?) and users hate it; a single well-structured package with proper ESM entry points gives you per-component imports and tree-shaking without per-component packages. You get the bundle benefit in the O step without the release-coordination cost.

This is the frontend mirror of the backend "monolith vs microservices" decision, and the reasoning rhymes. Splitting every component into its own package is microservices-for-a-UI-library: independent deploy in theory, distributed- versioning hell in practice. A single versioned package with clean internal module boundaries is the "modular monolith" — one release, one compatible version, and you still tree-shake at the import level. Say that comparison out loud; it lands with backend-leaning interviewers.
The line that shows you've felt the pain: "I'd keep the whole React library as one versioned package rather than one-package-per-component. Per-component packages sound cleaner but they create a version-matrix problem — consumers end up with Button@2 and Modal@1 that share an incompatible token version, and nobody can tell which combination works. One package means one number to reason about. I still get per-component imports and tree-shaking from ESM entry points, so the bundle argument for splitting doesn't actually require splitting the release." Distinguishing the import boundary from the release boundary is a genuinely senior distinction most candidates miss.

D — Design tokens & theming deep dive

In a screen-design round, "D" is the client store — normalized entities. A design system has no such store, so the tokens are the data model: the structured, single-source-of-truth data that every component reads from, and the layer that makes theming a solved problem instead of a find-and-replace nightmare. This is one of the two deep dives; spend real time here.

What a design token is: a named design decision stored as data rather than baked into code. Instead of color: #1a56db scattered across fifty components, you have a token --color-action-primary that fifty components reference. Change the token, and everything that uses it updates — the exact same single-source-of-truth argument as normalizing a client store, applied to visual design.

The three token tiers — the idea that makes theming tractable

The mistake juniors make is a flat token list: one giant map of blue500, gray100, spacing8. It's better than raw hex codes, but it can't theme — because a component referencing blue500 directly still breaks when the dark theme wants that same "primary action" to be a lighter blue. The fix is three tiers, each referencing the one below, so that swapping a theme means swapping one middle layer while components never change.

TierWhat it isExampleWho references it
1. Primitive (a.k.a. base / global)The raw palette — every literal value the brand owns, named by what it is, not what it's for. Rarely used directly.--blue-500: #1a56db
--gray-900: #111827
--space-4: 1rem		Only semantic tokens. Never a component.
2. Semantic (a.k.a. alias / system)Names a design decision by its role/intent, pointing at a primitive. This is the layer a theme swaps.--color-action-primary: var(--blue-500)
--color-text-default: var(--gray-900)
--color-surface: var(--white)		Component tokens, and components directly.
3. Component (optional)Tokens scoped to one component, pointing at semantics. Lets you retune one component without touching the global system.--button-bg: var(--color-action-primary)
--button-text: var(--color-text-on-action)		Only that component's styles.

The chain to say out loud: --button-bg → --color-action-primary → --blue-500 → #1a56db. The Button only ever knows about --button-bg. Switch to dark mode and you only redefine the semantic tier (--color-action-primary now points at a different primitive); the component tier and the component code never change. That indirection is the entire payoff — it's why a well-tiered system themes in one place and a flat one has to be edited everywhere.

Why CSS custom properties, and not JS constants or Sass variables

The implementation choice matters and interviewers push on it. Sass variables and JS-object themes are resolved at build time — they're gone by the time the browser runs, so switching theme at runtime means shipping and swapping entirely separate stylesheets. CSS custom properties live in the cascade at runtime: they inherit through the DOM, they can be redefined on any subtree, and changing one repaints instantly with no re-render. That's what makes runtime theme switching, per-section theming, and per-user brands cheap. 

/* Tier 1 — primitives: the raw palette, defined once at the root */
:root {
  --blue-500: #1a56db;
  --blue-300: #7aa7f0;
  --gray-900: #111827;
  --gray-0:   #ffffff;
}

/* Tier 2 — semantic: role-based, and THE layer a theme overrides */
:root, [data-theme="light"] {
  --color-action-primary: var(--blue-500);
  --color-text-default:   var(--gray-900);
  --color-surface:        var(--gray-0);
}
[data-theme="dark"] {                 /* swap ONLY the semantic tier */
  --color-action-primary: var(--blue-300);  // lighter blue reads better on dark
  --color-text-default:   var(--gray-0);
  --color-surface:        var(--gray-900);
}

/* The component reads component/semantic tokens and never a raw hex */
.Button {
  --button-bg: var(--color-action-primary);
  background: var(--button-bg);
  color: var(--color-surface);
}

Switching the whole app to dark mode is now document.documentElement.dataset.theme = "dark" — one attribute, no re-render, every component follows because they all resolve through the cascade. And because tokens inherit, you can scope a theme to a subtree (a dark-themed sidebar inside a light app) by setting data-theme="dark" on just that container.

Letting components reference primitive tokens directly (color: var(--blue-500)) is the mistake that quietly destroys themeability. It works in light mode, so it passes review — then dark mode ships and that one component is still hard-blue on a dark surface because it never went through the semantic tier that the theme swapped. The rule to state and enforce (with a lint rule): components reference semantic or component tokens only; primitives are private to the semantic layer.
The framing that shows you understand why the tiers exist, not just that they do: "The tiers are an indirection layer, and they buy the same thing indirection always buys — the freedom to change one side without touching the other. Primitives answer 'what colors does the brand own,' semantics answer 'what does this color mean,' and components answer 'what does this specific part use.' A theme is just a different mapping at the semantic tier, so re-theming touches one file and zero components. Skip the tiers and every theme change is a codebase-wide find-and-replace — which is exactly the drift problem tokens existed to kill."

I — Interface & API: compound components & headless patterns deep dive

This is the heart of the round. In a design system the component's API is the product — the props are the thing every consuming team touches every day, and a badly shaped API is a mistake you'll be apologizing for and can't easily undo. The whole skill here is designing APIs that are flexible without being footguns, and the patterns below are the vocabulary for exactly that. Anchor everything on a running example — a <Select> — because it's complex enough to show every pattern.

The problem: the "props explosion"

The naive API is a single component with a prop for everything: <Select options={...} renderOption={...} label=... searchable clearable placeholder=... optionClassName=... />. It works for the first three use cases, then team four wants a divider between groups, team five wants an icon in one option, team six wants a footer, and you're adding a prop a week until the component has forty of them and is impossible to reason about. This is the props explosion, and it's the failure mode compound components and headless patterns exist to solve — by handing structure back to the consumer instead of trying to anticipate it with configuration.

Pattern 1 — Compound components: composition instead of configuration

A compound component is a set of components designed to work together and share implicit state via Context, exposed as a namespace: <Select>, <Select.Option>, <Select.Group>. The parent owns the state (which option is selected, whether the menu is open); the children read it from Context without the consumer wiring anything up. The consumer composes the structure they want in JSX — the way native <select>/<option> already work — instead of describing it through a renderOption prop. 

// CONSUMER writes structure directly — no options array, no renderOption prop.
// A divider, an icon, a group heading are just... markup the consumer controls.
<Select value={fruit} onChange={setFruit} label="Fruit">
  <Select.Group label="Citrus">
    <Select.Option value="orange">🍊 Orange</Select.Option>
    <Select.Option value="lemon">🍋 Lemon</Select.Option>
  </Select.Group>
  <Select.Option value="apple" disabled>🍎 Apple (out of stock)</Select.Option>
</Select>

// LIBRARY: the parent holds state and hands it down via Context.
const SelectContext = createContext<SelectCtx | null>(null);

function Select({ value, onChange, children, label }: SelectProps) {
  const ctx = { value, onChange };            // shared implicitly with all Options
  return (
    <SelectContext.Provider value={ctx}>
      { /* trigger + listbox rendered here; children are the Options */ }
      {children}
    </SelectContext.Provider>
  );
}
Select.Option = function Option({ value, children, disabled }: OptionProps) {
  const ctx = useContext(SelectContext);
  if (!ctx) throw new Error("Select.Option must be used inside <Select>");
  const selected = ctx.value === value;
  return <li role="option" aria-selected={selected} ...>{children}</li>;
};

Note the throw when an Option is used outside a Select — a compound API should fail loudly and early when misused, because "hard to use wrong" is the entire selling point.

Pattern 2 — Headless: behavior without markup

Compound components still ship your markup and styles. Sometimes a team needs the exact behavior — all the fiddly keyboard navigation, focus management, ARIA, and open/close logic — but with completely different markup (a command palette, a custom combobox). The headless pattern extracts the behavior into a hook that returns state and prop-getters, and renders nothing. The consumer owns 100% of the markup; the library owns 100% of the correctness. This is what libraries like Radix, Headless UI, React Aria, and Downshift are. 

// The hook owns state, keyboard nav, focus, and — critically — ARIA wiring.
// It renders NOTHING. It returns state + prop-getters that spread the right
// roles/aria-*/handlers onto whatever elements the consumer chooses.
function useSelect<T>({ value, onChange, options }: UseSelectArgs<T>) {
  const [open, setOpen] = useState(false);
  const [active, setActive] = useState(0);   // highlighted index for keyboard nav

  function onKeyDown(e: KeyboardEvent) {
    if (e.key === "ArrowDown") setActive(i => Math.min(i + 1, options.length - 1));
    if (e.key === "ArrowUp")   setActive(i => Math.max(i - 1, 0));
    if (e.key === "Enter")     { onChange(options[active]); setOpen(false); }
    if (e.key === "Escape")    setOpen(false);
  }
  return {
    isOpen: open,
    // prop-getters: spread these and the a11y is correct by construction
    getTriggerProps: () => ({ role: "combobox", "aria-expanded": open,
                             "aria-haspopup": "listbox", onClick: () => setOpen(o => !o) }),
    getListboxProps: () => ({ role: "listbox", onKeyDown }),
    getOptionProps: (i: number) => ({ role: "option",
                             "aria-selected": options[i] === value,
                             onMouseEnter: () => setActive(i) }),
  };
}

// Your styled <Select> is now a THIN skin over useSelect. So is a consumer's
// bespoke command palette — same behavior + a11y, totally different markup.

The senior insight: your styled compound <Select> should itself be built on useSelect. That way there's one implementation of the hard part (keyboard + ARIA), the default component is just the common skin over it, and the escape hatch (the raw hook) is the same code path, not a parallel one that drifts. This is how you resolve the consistency-vs-flexibility tension from Requirements: 90% of teams use the batteries-included component; the 10% with a genuinely custom UI drop to the hook and still inherit your accessibility.

Controlled vs uncontrolled — support both

The last API decision every stateful component faces. A controlled component takes its value as a prop and reports changes via onChange — the consumer owns the state (React's <input value onChange>). An uncontrolled component owns its own state internally and only exposes a defaultValue and an onChange notification. A good library component supports both, because they serve different needs: controlled when the parent must drive or sync the value (validation, dependent fields), uncontrolled when the consumer just wants it to work without managing state. 

// The standard "dual" pattern: controlled IF `value` is passed, else internal.
function useControllableState<T>(controlled: T | undefined, defaultValue: T) {
  const [internal, setInternal] = useState(defaultValue);
  const isControlled = controlled !== undefined;
  const value = isControlled ? controlled : internal;   // prop wins if present
  const setValue = (next: T) => {
    if (!isControlled) setInternal(next);               // only self-manage when uncontrolled
    /* always fire onChange so the consumer hears about it either way */
  };
  return [value, setValue] as const;
}
The subtle bug that bites every dual component: silently switching between controlled and uncontrolled at runtime (value going from undefined to a value, or back). React itself warns about this on inputs for a reason — the component doesn't know who owns the state anymore. Decide ownership once at mount based on whether value was provided, and if a consumer flips it, warn in dev. Getting this contract crisp is a mark of someone who's shipped form components.
Tie the patterns to the central tension explicitly: "I'd default rich components to a compound API built on top of a headless hook, and support controlled and uncontrolled use. That gives me three tiers of flexibility from one implementation: the styled component for the 90%, token and variant props for light customization, and the raw hook as the escape hatch for the 10% who need bespoke markup — and crucially, every tier inherits the same accessibility, because they all run through the same hook. I never have to choose between 'consistent' and 'flexible'; the layering gives me both."

Accessibility as a contract deep dive

This is the single highest-signal idea on the page, and it's a genuine reframe of what accessibility is in a design system. On a screen, a11y is a property you have to get right on that one screen. In a component library, a11y becomes a contract the library owns on behalf of every consumer. If your <Modal> correctly traps focus, restores focus on close, sets role="dialog"/aria-modal, and closes on Escape, then every one of the hundred apps that imports it is accessible for that pattern — for free, forever, whether or not the consuming developer knows what a focus trap is. That's the leverage: you get accessibility right once, in the library, and it multiplies across the entire company.

Say this out loud — it's the sentence that most distinguishes a senior answer in this round: "The most valuable thing a design system does isn't consistency of pixels, it's that it makes the accessible path the default path. Individual product teams get a11y wrong constantly — not out of malice, but because focus management and ARIA are hard and easy to forget. If the library owns keyboard interaction, focus, and ARIA inside each component, then consumers literally can't ship an inaccessible Select even if they don't know what a listbox role is. I'm moving accessibility from something every team re-earns to something the platform guarantees. That's a compliance and legal win, not just a UX one."

What "owning the contract" concretely means

It's the reason the useSelect hook returned prop-getters rather than letting consumers wire ARIA themselves. The library's job is to make correct behavior the thing that happens when you do the obvious thing, and to make incorrect behavior require going out of your way.

a11y responsibilityWhat the library owns so the consumer can't get it wrong
Semantic elements<Button> renders a real <button>, not a <div onClick> — so keyboard, focus, and role come free. Most a11y is "the library picked the right element."

Live announcementsAn aria-live region baked into components that change asynchronously (a Toast, a combobox's result count) so screen-reader users hear updates.
Contrast & motionSemantic color tokens are chosen to meet WCAG AA contrast by construction, and animations respect prefers-reduced-motion — so a consumer using tokens is contrast-compliant without checking.

The consumer's remaining job shrinks to the things only they can know: providing an accessible name (the label text, the alt text) and not deliberately breaking the semantics with an escape hatch. Everything structural, the library guarantees. And you enforce it: axe-core in CI, a Storybook a11y addon, and screen-reader testing on the components themselves, so a regression in the library is caught before it multiplies to a hundred apps.

The anti-pattern that breaks the contract is the unstyled, unsemantic escape hatch used carelessly — a team drops the headless hook onto a <div> and forgets to spread getTriggerProps(), or overrides your <button> with a <div role="button"> and loses keyboard support. Escape hatches are necessary, but they're where a11y leaks out of the system. Mitigate by making the prop-getters do the heavy lifting (spreading them is easier than not), documenting the a11y requirements at the escape hatch, and — for the truly critical patterns — not offering a footgun-shaped API in the first place.

O — Bundle size & tree-shaking

Now the RADIO "Optimizations" step, in its library-specific form. The performance question here isn't "how fast is the screen" — it's "when a team imports one Button, how much of my library ends up in their bundle?" The answer must be "just the Button and what it depends on," and achieving that is a packaging problem, not a runtime one. Get it wrong and teams either bloat their apps or refuse to adopt you.

The three things that make a library tree-shakeable

  • Ship ESM, not just CommonJS. Tree-shaking relies on the static structure of ES modules (import/export) — a bundler can see at build time that DataTable is never imported and drop it. CommonJS (require) is dynamic and can't be statically analyzed, so a CJS-only library ships whole. Publish an ESM build and point the package's "module"/"exports" fields at it.
  • Declare "sideEffects": false. This is the flag that unlocks aggressive dead-code elimination. It's a promise to the bundler: "importing any module in this package has no side effects, so if you don't use an export, it's safe to drop the whole module." Forget it and bundlers conservatively keep everything, in case an unused import was doing something. The one exception: if a file does have a side effect (a CSS import, a polyfill), list those files so they're preserved.
  • Provide granular entry points. A single barrel index.js that re-exports everything is fine if ESM + sideEffects are set (the bundler prunes the unused re-exports), but per- component entry points via the "exports" map are the belt-and-suspenders version and protect consumers whose bundler config is weaker.
// package.json — the fields that make @acme/react tree-shakeable
{
  "name": "@acme/react",
  "sideEffects": ["*.css"],          // false everywhere EXCEPT css imports
  "type": "module",
  "exports": {
    ".":         { "import": "./dist/index.js",        "types": "./dist/index.d.ts" },
    "./button":  { "import": "./dist/button/index.js", "types": "./dist/button/index.d.ts" },
    "./select":  { "import": "./dist/select/index.js", "types": "./dist/select/index.d.ts" }
  }
}

// Consumer: with the config above, BOTH of these ship only Button + deps.
import { Button } from "@acme/react";          // barrel — pruned by tree-shaking
import { Button } from "@acme/react/button";   // explicit — pruned regardless of config

Two more levers that matter

  • Externalize peer dependencies. react and react-dom are peerDependencies, never bundled — otherwise every consumer ships a second copy of React and hooks break across the boundary. Same logic for a shared icon or date library.
  • Watch what your styling approach costs at runtime. This is where the CSS-custom-properties decision from the tokens step pays off again: static CSS (or zero-runtime CSS-in-JS like vanilla-extract) adds nothing to the JS bundle, whereas a runtime CSS-in-JS library (styled-components/Emotion) ships a styling engine to every consumer and does work on every render. For a library used by bundle-sensitive apps, prefer the zero-runtime route.
Verify the claim, don't just assert it. In the interview, say you'd measure — a bundle analyzer, and a CI check that fails the build if importing a single component pulls in more than a threshold (a "size-limit" gate). "I'd set a per-component size budget in CI so a stray import that accidentally pulls in the whole library is caught at review time" is the senior version of caring about bundle size — it's the same "measure, don't guess" discipline as the RADIO Optimizations step and the backend bottleneck step.
The distinction that shows you actually understand tree-shaking: "Tree-shaking isn't automatic — it's a contract between my package and the consumer's bundler, and I hold up three ends of it: ESM so the imports are statically analyzable, sideEffects: false so the bundler is allowed to drop unused modules, and externalized peer deps so React isn't duplicated. If any one of those is missing, a team importing one button ships the whole library, and they'll notice. It's the packaging equivalent of the tiered tokens — a little discipline at the boundary so consumers get the good behavior for free."

Versioning, breaking changes & codemods

Here's the concern that has no analogue in any screen-design round and is the truest test of design-system seniority: your API is public and you have to evolve it for years without breaking the hundred apps that depend on it. Every prop you ship is a promise. The whole discipline is about making change cheap and safe for consumers you don't control and can't force to upgrade.

Semver, but understand it as an API contract

Semantic versioning (MAJOR.MINOR.PATCH) is the communication protocol. The key move is recognizing what counts as "breaking" for a component API — it's broader than backend engineers expect, because the surface includes visual and behavioral contracts, not just function signatures.

BumpMeansExamples for a component library
PATCH (1.4.2 → 1.4.3)Backwards-compatible bug fixFix a focus bug; correct a token value that was wrong; a11y fix that doesn't change the API
MINOR (1.4.2 → 1.5.0)Backwards-compatible additionA new component; a new optional prop; a new variant. Existing code keeps working.
MAJOR (1.4.2 → 2.0.0)Breaking change — consumers must actRemoving/renaming a prop; changing a prop's default; renaming a component; a visual change big enough to break layouts; removing a deprecated API

The subtlety worth voicing: in a design system, a significant visual change can be a breaking change even though no code changed. If you alter Button's default padding, every consumer's carefully-tuned layout might shift. Purists ship that as a major; pragmatically you gate large visual changes behind an opt-in or a new variant. Knowing that "breaking" includes the rendered output, not just the TypeScript types, is a senior tell.

The deprecation lifecycle — never break without a runway

Because you can't force upgrades, you never remove something abruptly. The lifecycle: (1) introduce the new API alongside the old; (2) mark the old one deprecated — a @deprecated JSDoc tag (so it strikes through in every consumer's editor) plus a console.warn in dev pointing at the replacement; (3) let both coexist for a defined window (a couple of majors, or a time period); (4) remove the old API in a major, with a codemod. The warning does the socialization work so that by the time you remove it, most consumers have already moved. 

// A deprecated prop: still works, but tells every consumer's IDE and console.
interface ButtonProps {
  variant?: "primary" | "secondary" | "ghost";
  /** @deprecated Use `variant="primary"` instead. Removed in v3. */
  primary?: boolean;
}
function Button({ variant, primary, ...rest }: ButtonProps) {
  if (primary !== undefined && process.env.NODE_ENV !== "production") {
    console.warn("[acme] Button `primary` is deprecated; use variant=\"primary\". See migration guide.");
  }
  const resolved = variant ?? (primary ? "primary" : "secondary");  // old prop still honored
  ...
}

Codemods — the thing that makes a breaking change actually adoptable

A codemod is an automated migration script — typically a jscodeshift transform that parses each consumer's code into an AST, rewrites the old API to the new one, and prints it back. This is the highest-leverage idea in the shipping story: a breaking change you ship with a codemod costs each consumer one command; the same change without one costs every team a manual, error-prone find-and-replace across their codebase — which is why libraries without codemods have zombie majors nobody upgrades to. 

// A jscodeshift transform: rewrite <Button primary> → <Button variant="primary">
export default function transform(file, api) {
  const j = api.jscodeshift;
  const root = j(file.source);
  root.findJSXElements("Button").forEach(el => {
    const attrs = el.node.openingElement.attributes;
    const primary = attrs.find(a => a.name?.name === "primary");
    if (primary) {
      // remove `primary`, add variant="primary"
      el.node.openingElement.attributes = attrs.filter(a => a !== primary);
      el.node.openingElement.attributes.push(
        j.jsxAttribute(j.jsxIdentifier("variant"), j.literal("primary")));
    }
  });
  return root.toSource();
}
// Consumer runs: npx @acme/codemods button-variant ./src   → their whole app migrated.
The end-to-end story to narrate, because it's what "owning" a library really means: "I never ship a breaking change bare. The lifecycle is: add the new API, deprecate the old one with a JSDoc tag and a dev-only console warning so it surfaces in every consumer's editor, keep both alive for a deprecation window, then remove it in a major with a codemod that migrates their code automatically. The codemod is the part that makes the major actually adoptable — it turns 'every team spends a day' into 'every team runs one command.' A migration guide plus a codemod is the difference between a version people upgrade to and a version that splits your user base."

Docs & adoption

The last part of the operational layer, and the one candidates skip because it feels non-technical — which is exactly why saying it scores. A design system that nobody adopts is a failed design system, no matter how elegant the token architecture. If teams find it easier to hand-roll a button than to learn yours, they will, and your consistency guarantee evaporates. Adoption is a product problem, and you should treat the consuming engineers as users whose path of least resistance has to lead through your library.

Docs are the primary interface, not an afterthought

  • Interactive, not static. A component library documented with prose alone is unusable. You want a living docs site — Storybook or a docs app — where every component has runnable examples, a live props playground (change a prop, see it render), and copy-pasteable code. The docs dogfood the library, so if a component is awkward to document, that's a signal the API is awkward.
  • Every state and variant visible. Show the loading, error, disabled, and empty states, all variants, and the accessibility notes, so a consumer doesn't discover the disabled style by shipping it.
  • Guidance, not just API. "Use this vs that," do/don't examples, and — critically — the migration guides that pair with each major version's codemod. Docs are where the versioning story gets socialized.
  • Types are docs. Excellent TypeScript types are the documentation developers actually read, because they show up in autocomplete. A well-typed prop with a JSDoc comment teaches the API at the call site with zero context-switch.

Adoption is a rollout strategy, not a launch

  • Dogfood and seed. Build a real product surface on the system first to prove it, and provide starter templates so a new app is on the system from line one.
  • Lower the switching cost. Codemods aren't only for breaking changes — an adoption codemod that migrates a team off their bespoke buttons onto yours removes the biggest barrier to trying it.
  • Contribution model. Decide early: fully centralized (one team owns everything — high consistency, becomes a bottleneck) or federated (product teams contribute components back under review — scales, needs governance). Naming this trade-off shows you're thinking about the system as an organization, not just code.
  • Measure adoption. Track which apps and versions are on the system (registry download data, a usage-telemetry ping). You can't manage rollout or plan deprecations if you can't see who's on what.
The line that shows you think about a library as a product with users: "I'd treat adoption as a first-class goal, because a system nobody uses guarantees nothing. That means the docs are the real product surface — interactive examples, a props playground, migration guides — and it means making the on-ramp cheaper than rolling your own: starter templates, adoption codemods, and excellent types so the API teaches itself in autocomplete. And I'd measure it — which apps on which versions — because you can't run a deprecation or prove the system's value without knowing who actually depends on you."

Interview Q&A

1. How is designing a component library different from designing a screen?

Your users are other engineers and your product is an API contract, so the graded axes shift entirely. There's almost no data flow and no scale-of-a-screen; instead the interview is about API design that's hard to misuse, accessibility the library owns on behalf of consumers, evolving a public API for years without breaking dependents, and bundle discipline so importing one component doesn't ship the whole library.

The frame to open with: "a design system is three products stacked — a token layer (visual language as data), a component layer (the API teams import), and an operational layer (versioning, docs, bundle strategy, adoption)." Naming those three gives the whole answer its structure.

2. What are design tokens, and why the three tiers?

A token is a named design decision stored as data — --color-action-primary instead of a raw #1a56db scattered everywhere — so one change propagates. The tiers are an indirection layer: primitive (the raw palette, named by what it is: --blue-500), semantic (named by role/intent, pointing at a primitive: --color-action-primary), and optionally component (scoped to one component: --button-bg).

The payoff: a theme is just a different mapping at the semantic tier. Dark mode redefines --color-action-primary to point at a lighter blue; components and their code never change. A flat token list can't do that — components reference the palette directly, so a theme change is a codebase-wide find-and-replace, which is the exact drift problem tokens exist to kill.

3. Why CSS custom properties for theming instead of Sass variables or a JS theme object?

Custom properties are resolved at runtime in the cascade — they inherit through the DOM, can be redefined on any subtree, and changing one repaints instantly with no React re-render. That's what makes runtime theme switching, per-section theming, and per-user brands cheap: dark mode is one attribute, document.documentElement.dataset.theme = "dark".

Sass variables are build-time — gone before the browser runs, so switching themes means shipping and swapping whole stylesheets. A JS theme object through Context works at runtime but couples theming to React and re-renders every consumer on switch. Custom properties let the cascade do the work and keep React uninvolved.

4. What's a compound component and when would you use one?

A set of components designed to work together sharing implicit state via Context, exposed as a namespace: <Select>, <Select.Option>, <Select.Group>. The parent owns state; children read it from Context so the consumer wires nothing. It's the answer to the props explosion — instead of adding a prop every time someone wants a divider or an icon or a group heading, you hand structure back to the consumer, who composes it in JSX like native <select>/<option>.

Use it as the default for rich components with variable internal structure. Skip it for genuinely simple, closed components (a Badge) where a plain prop API is simpler.

5. What does "headless" mean, and how does it relate to compound components?

Headless means the component's behavior — keyboard navigation, focus management, ARIA wiring, open/close logic — is extracted into a hook (e.g. useSelect) that returns state and prop-getters and renders nothing. The consumer owns 100% of the markup and styling; the library owns 100% of the correctness. Radix, Headless UI, React Aria, and Downshift are this pattern.

The senior move is that your styled compound <Select> is itself built on the headless hook. Then there's one implementation of the hard part (keyboard + ARIA), the default component is a thin skin over it, and the escape hatch for teams needing bespoke markup is the same hook — not a parallel code path that drifts. That's how you satisfy both consistency (everyone inherits the same a11y) and flexibility (the 10% can go custom).

6. Why is "accessibility as a contract" the highest-signal idea here?

Because it converts a11y from something every team re-earns per screen into something the platform guarantees once. If the library's <Modal> traps focus, restores it on close, sets the right ARIA, and closes on Escape, then every app importing it is accessible for that pattern — for free, even if the consuming dev doesn't know what a focus trap is. You make the accessible path the default path.

Concretely, the library owns semantic elements, keyboard interaction, focus management, ARIA wiring (spread by prop-getters, not hand-written), live-region announcements, and contrast-compliant tokens. The consumer's job shrinks to providing an accessible name. Enforce it with axe-core in CI so a regression is caught before it multiplies to a hundred apps.

7. What counts as a breaking change for a component library?

Broader than backend engineers expect. Removing or renaming a prop, changing a prop's default, renaming a component, or removing a deprecated API — all clearly major. But also, a significant visual change can be breaking even though no code changed: alter Button's default padding and every consumer's tuned layout may shift. The rendered output is part of the contract, not just the TypeScript types.

So minor = a new component or new optional prop (existing code keeps working); patch = a bug or a11y fix with no API change; major = anything that forces consumers to act, including large visual changes — which pragmatically you'd gate behind an opt-in variant rather than shipping bare.

8. You need to remove a widely-used prop. Walk me through it.

Never abruptly, because you can't force upgrades on independently-deployed apps. The lifecycle: (1) add the new API alongside the old; (2) deprecate the old one — a @deprecated JSDoc tag so it strikes through in every consumer's editor, plus a dev-only console.warn pointing at the replacement; (3) let both coexist for a defined window; (4) remove it in a major shipped with a codemod.

The codemod is the part that makes the major adoptable — a jscodeshift transform that rewrites the old usage to the new one across each consumer's codebase, turning "every team spends a day" into "every team runs one command." A migration guide plus a codemod is the difference between a version people upgrade to and a zombie major that splits your user base.

9. A team imports one Button but ships the whole library. Why, and how do you fix it?

Tree-shaking failed, and it's a packaging problem with three usual causes. ESM: if you only ship CommonJS, imports are dynamic and can't be statically analyzed, so nothing is dropped — publish an ESM build. sideEffects: without "sideEffects": false (or a precise list) the bundler conservatively keeps every module in case an unused import had a side effect. Peer deps: React must be a peerDependency, never bundled, or every consumer duplicates it.

Set all three, optionally expose per-component entry points via the exports map for weaker bundler configs, and add a CI size-limit gate so a stray import that pulls in the whole library fails at review. Measure it with a bundle analyzer rather than asserting it works.

10. Controlled vs uncontrolled — should a library component pick one?

Support both. Controlled (value + onChange, consumer owns state) is needed when the parent must drive or sync the value — validation, dependent fields. Uncontrolled (defaultValue, component owns its own state) is what a consumer wants when they just need it to work without managing state. The standard "dual" pattern: the component is controlled if a value prop is present, otherwise it self-manages, and it always fires onChange so the consumer hears changes either way.

The bug to call out: don't let a component silently switch modes at runtime (value going from undefined to defined). React warns about exactly this on inputs. Decide ownership once at mount and warn in dev if a consumer flips it.

11. How do you handle multiple frameworks — some teams are on Vue or native?

Push the shared layer down to what's framework-agnostic. The token package — CSS custom properties plus JSON/JS — is consumable by anything, so brand and theme stay unified across React, Vue, and native even though the components can't be shared. That's why tokens are their own package from day one; it's cheap insurance even if everyone's on React today.

Above tokens you either build per-framework component packages that all read the same tokens (most common), or, if you truly need one component implementation everywhere, standardize on web components as the primitive layer and wrap them thinly per framework — at the cost of some ergonomic friction. Name the trade-off; don't pretend one library can be idiomatic in every framework for free.

12. How do you drive adoption, and why is that a design concern?

Because a system nobody adopts guarantees nothing — if hand-rolling a button is easier than learning yours, teams will, and your consistency and a11y guarantees evaporate. So you treat consuming engineers as users whose path of least resistance must run through the library. Make docs the real product surface: interactive examples, a live props playground, migration guides, and excellent TypeScript types that teach the API in autocomplete.

Then lower the switching cost — starter templates, adoption codemods that migrate teams off bespoke components — decide the contribution model (centralized vs federated) explicitly, and measure which apps are on which versions so you can run deprecations and prove the system's value. Adoption is a rollout strategy, not a launch event.

What to remember

The round

  • Your users are engineers; your product is an API contract. Grade yourself on API-hard-to-misuse, accessibility the library owns, evolving without breaking dependents, and bundle discipline — not scale or screen-perf.
  • A design system is three products stacked: a token layer (visual language as data), a component layer (the React API), and an operational layer (versioning, docs, bundle, adoption). Name them to structure the answer.
  • The spine is consistency vs flexibility. On-rails common path (tokens + variants), deliberate escape hatches (headless hook, className pass-through) for the genuine 10%. The escape hatch is what keeps teams inside the system.

RADIO, adapted

  • R — Requirements. Ask about consumers and constraints: how many teams / which frameworks, consistency vs customization, the a11y bar, theming (multi-brand? runtime?), bundle sensitivity, and independent vs coordinated deploys. That last one is why versioning is real, not an afterthought.
  • A — Architecture. The layered stack — tokens → headless behavior → components → patterns — with dependencies pointing down only. Ship a monorepo with a framework-agnostic token package and one versioned React package (per-component imports, not per-component releases).
  • D — Design tokens. Tokens are the data model. Three tiers (primitive → semantic → component); a theme swaps only the semantic tier. CSS custom properties for runtime theming; components reference semantic/component tokens, never raw primitives.
  • I — Interface & API. The props are the product. Compound components over a headless hook beats a mega-prop component; support controlled and uncontrolled; the styled component and the escape-hatch hook share one implementation.
  • a11y as a contract. The highest-signal idea: the library owns keyboard, focus, and ARIA (via prop-getters) so consumers can't ship it inaccessible. Get it right once, multiply across every app. Enforce with axe in CI.
  • O — Bundle. Tree-shaking is a contract: ESM + sideEffects:false + externalized peer deps. Prefer zero-runtime CSS. Gate it with a CI size budget.

The operational layer

  • Versioning. Semver where "breaking" includes big visual changes, not just types. Never break bare: add → deprecate (JSDoc + dev warn) → coexist → remove in a major with a codemod. The codemod is what makes the major adoptable.
  • Docs & adoption. A system nobody adopts guarantees nothing. Interactive docs + playground + migration guides + great types; lower the on-ramp with templates and adoption codemods; measure who's on what.
The one line that ties the page together: a design system is API design plus lifecycle management wearing a UI costume. The visuals — tokens, themes, components — are the easy half. The senior half is everything about being depended on: an API teams can't misuse, accessibility they inherit for free, changes you can ship for years without breaking a hundred apps, and a bundle that stays lean. Design the contracts and the lifecycle, and the buttons draw themselves.
