25 — LLD Case Studies

Parking lot, elevator, BookMyShow — worked three ways with one recipe. The round where you write real Java on a whiteboard and get graded on the classes, not the boxes.

Why this matters in the interview

The low-level design round is a different animal from the distributed one, and it catches strong backend candidates off guard because it rewards a skill they've stopped practising: turning a vague English sentence into a clean set of Java classes, in real time, out loud, without freezing. Enterprise loops — Optum, JPMC, Goldman, and most product companies with a mature interview process — run this as a distinct 45-to-60-minute session. You will be asked to "design a parking lot" or "design an elevator," and what's actually being measured is whether your object model is coherent: do responsibilities sit in the right place, are the seams where requirements change guarded by interfaces, and can you talk about concurrency without hand-waving. It is the round where your seven years either show or don't, because a mid-level candidate produces classes that work and a senior candidate produces classes that absorb the next requirement the interviewer is about to throw at them.

There are only three or four problems that come up over and over — parking lot, elevator, a ticket/seat booker, occasionally a vending machine or a rate limiter — and they are all the same problem wearing different costumes, which is exactly why page 24's recipe is worth more than memorising any one of them. This page works the three canonical ones end to end in Java, and for each it does the same five things in the same order: clarify the scope, extract the classes, sketch the core design, name the one design pattern that earns its place, and — the part most candidates skip and every good interviewer probes — say precisely where the concurrency bug lives and how the object model contains it. Do not read this page as three problems to memorise. Read it as the recipe applied three times, so that on the fourth problem you've never seen, you already know the moves.

The framing line to open this round with, before you write a single class: "I'm going to spend the first few minutes on requirements and nouns before I write any code, because in LLD the object model is the answer and a wrong model is expensive to unwind on a whiteboard. I'll start with the interfaces at the seams where I expect requirements to change — pricing, scheduling, allocation — and keep the concrete classes thin. And I'll flag the concurrency-sensitive operations as I go rather than bolting thread-safety on at the end." That sentence tells the interviewer you've done this before, and it buys you the two quiet minutes you need to actually think.

The shared recipe

This is page 24's method in three lines, restated here because you are about to apply it three times and it should be muscle memory by the end of the page:

In the prompt……becomes in the modelWhy
Nouns — the things ("a car parks in a spot on a floor") Classes / entities — Vehicle, ParkingSpot, Floor Each noun that has state or identity is a class. Nouns that are just values (a colour, a size) become enums or fields.
Verbs — the actions ("park", "assign", "pay", "book") Methods — park(vehicle), assignSpot(), book(seats) A verb tells you the method and which class owns it: the class that holds the state the verb changes.
Variation points — "the fee depends on…", "we might support…" Interfaces — PricingStrategy, SpotAssignmentStrategy Anywhere the requirement can change is a seam. Put an interface there so the change is a new class, not an edit to an old one (Open/Closed).

The verbs test is the one that separates good models from bad. "The system calculates the fee" is a trap — which object calculates it? The one holding the data the calculation needs. Misplace a verb and you get an anemic domain model: dumb data classes with all the logic marooned in one god-service. The interviewer sees that instantly.

And the six patterns from page 24 map onto these problems so cleanly it's almost a lookup table — you don't reach for a pattern because you want to show one off, you reach for it because a variation point demands it:

PatternUse it when the prompt says…Shows up in
Strategy"…the X depends on / could be one of several…"Parking fee, seat pricing, elevator scheduling
State"…the thing moves through phases and behaves differently in each…"Elevator (moving/idle), booking (held/confirmed), spot (free/occupied)
Factory"…create the right kind of X from some input…"Vehicle from ticket type, spot from vehicle size
Observer"…when X happens, notify / update Y…"Display boards, seat-availability push, elevator panel lights
Singleton (carefully)"…there is one central coordinator…"ParkingLot, ElevatorController — but say why, and prefer injection
Builder"…an object with many optional attributes…"A Show or Booking with lots of fields
You will not use all six in any one problem, and using too many is itself a red flag — it reads as pattern-fishing. The rule from page 24 holds: introduce a pattern only when you can name the specific requirement change it protects against. "I'll make pricing a strategy so hourly, flat-rate, and weekend pricing are three classes instead of three branches in one method" is a reason. "I'll add a factory because factories are good" is not.

1. Parking Lot

The canonical LLD warm-up. It looks trivial and it is not — the trap is that it's so familiar you start writing classes before you've scoped it, and then the interviewer adds "oh, and motorcycles can park in car spots but not vice versa" and your model buckles. Slow down. Scope first.

Clarify the scope

Ask these before you draw anything. Each answer removes or adds a class, so getting them up front is what keeps the model from thrashing:

QuestionWhy it mattersAssume
"Multiple floors, or one flat lot?"Multiple floors adds a Floor class and makes "find a spot" a two-level search.Multiple floors — it's the more interesting model and barely more code.
"Different vehicle sizes and matching spot sizes?"This is the heart of the allocation logic. Motorcycle / car / truck → compact / regular / large, with fit rules.Yes, three sizes. A smaller vehicle may use a larger spot; not vice versa.
"How is the fee calculated?"Flagged as the variation point → a strategy. Hourly? Flat? Different by vehicle type? Weekend surge?Hourly, but pluggable. Ask for it — it's where the pattern lives.

Functional: a vehicle enters, the system finds and assigns a fitting free spot and issues a ticket; a vehicle exits, the system computes the fee from the parked duration, takes payment, and frees the spot; the system can report free-spot counts per floor per size. Non-functional: the assign-a-spot operation must be correct under concurrent entries (two cars, one last spot), and lookups should be O(1) not a scan of every spot.

Nouns → classes, verbs → methods

Underline the nouns in the requirements: parking lot, floor, spot, vehicle, ticket, fee, payment. Underline the verbs: park/enter, assign, find (a spot), exit, calculate (fee), free. That underlining is the class list and the method list — say that out loud, it's the recipe made visible.

ClassResponsibility (single, ideally)Key methods

Point at the two interfaces as you draw them: "Notice I've made pricing and spot-selection interfaces even though I only have one implementation of each right now. That's deliberate — those are the two places the requirements will move. The interviewer is very likely to say 'now add weekend surge pricing' or 'now prefer spots near the elevator,' and when they do, my answer is 'that's a new strategy class, nothing else changes' instead of me editing a method I already wrote. I'm paying one interface up front to avoid a rewrite live on the whiteboard." Anticipating the follow-up in the model itself is the single most senior move in this round.

The core design in Java

enum VehicleSize { MOTORCYCLE, COMPACT, LARGE }   // ordinal encodes "fits in"

abstract class Vehicle {
    private final String plate;
    private final VehicleSize size;
    // ctor, getSize()...
}
class Motorcycle extends Vehicle { /* size = MOTORCYCLE */ }
class Car        extends Vehicle { /* size = COMPACT   */ }
class Truck      extends Vehicle { /* size = LARGE     */ }

class ParkingSpot {
    private final String id;
    private final VehicleSize size;
    private Vehicle parked;              // null == free

    // A COMPACT vehicle fits a COMPACT or LARGE spot, never a smaller one.
    boolean canFit(Vehicle v) { return v.getSize().ordinal() <= size.ordinal(); }

    // The guarded operation. See the concurrency section — this is THE race.
    synchronized boolean tryAssign(Vehicle v) {
        if (parked != null || !canFit(v)) return false;
        parked = v;
        return true;
    }
    synchronized void release() { parked = null; }
    boolean isFree() { return parked == null; }
}

interface PricingStrategy {          // ← the variation point, made a seam
    Money price(Ticket t, Instant exit);
}
class HourlyPricing implements PricingStrategy {
    public Money price(Ticket t, Instant exit) {
        long hrs = Math.max(1, Duration.between(t.entryTime(), exit).toHours());
        return ratePerHour.times(hrs);
    }
}

class ParkingLot {
    private final List<ParkingFloor> floors;
    private final PricingStrategy pricing;            // injected, not new'd
    private final SpotAssignmentStrategy assignment;   // injected

    public Ticket park(Vehicle v) {
        ParkingSpot spot = assignment.pickAndAssign(floors, v);  // atomic; see below
        if (spot == null) throw new LotFullException(v.getSize());
        return new Ticket(v, spot, Instant.now());
    }

    public Money unpark(Ticket t) {
        Money fee = pricing.price(t, Instant.now());
        t.spot().release();                              // free it AFTER pricing
        return fee;
    }
}

Notice what isn't here: no if (vehicle instanceof Truck) ladders, no giant calculateFee() with a switch on vehicle type, no balance-style mutable count that can drift from reality. Free-count is derived or maintained atomically alongside assignment, never as a second source of truth you update by hand.

Where a pattern earns its place

Strategy, twice. PricingStrategy is the obvious one and the interviewer is fishing for it: the moment they say "weekend rates" or "trucks pay double," you add a class, and the ParkingLot never changes. SpotAssignmentStrategy is the subtler, higher-value one — "assign the nearest spot to the entrance" vs "spread load across floors" vs "first-fit" is a policy that will change, so make it a strategy too. A light touch of Factory (a VehicleFactory turning a gate scan into the right Vehicle subclass) is defensible; a full Observer setup for the "spots available" display boards is a nice mention if you have time, over-engineering if you build it out. Singleton for ParkingLot is a classic trap: yes, there's one lot, but say "conceptually one instance, but I'd inject it rather than use a static getInstance(), because a hard singleton is untestable and I've been burned by that."

The concurrency concern: the last-spot race

Here is the bug the interviewer is really waiting for, and it's the same shape as the seat-booking race in BookMyShow and the overselling problem in 14 — Ticket Booking. Two cars arrive at the same instant and there is exactly one compact spot left. Naive code: 

// BROKEN — check-then-act across two operations.
ParkingSpot spot = floor.findFreeSpot(size);  // both threads see the SAME free spot
if (spot != null) spot.assign(vehicle);      // both assign it → one car is overwritten

Both threads call findFreeSpot, both get the same spot, both assign — one vehicle silently loses its spot, or two tickets point at one physical space and someone gets towed. The fix is to make find-and-assign a single atomic operation, never two steps with a gap. Three ways, in ascending scope, and you should be able to name all three and pick per context:

  1. In-memory, single JVM: the tryAssign() above is synchronized and does the check and the mutation atomically, so exactly one thread can flip a free spot to occupied — the same compare-and-set discipline as an idempotency key's INSERT … ON CONFLICT. The assignment strategy iterates candidate spots and stops at the first tryAssign() that returns true. To avoid a coarse lock over the whole lot, hold the free spots per floor per size in a concurrent structure and only lock the individual spot. Say "the lock is on the spot, not the lot — I don't want every entry serialised behind one mutex."
  2. Multi-node / persisted: the JVM lock is worthless across servers. Push the invariant into the database exactly as in 13 and 14: a conditional update — UPDATE spots SET vehicle_id = ? WHERE id = ? AND vehicle_id IS NULL — and check the affected-row count. One row updated, you won; zero rows, someone beat you, try the next spot. The database's row lock is the atomic compare-and-set.
  3. Optimistic version: a @Version column on the spot; two writers collide and the loser gets an OptimisticLockException and retries against a different spot. Good when contention is low (most spots are free most of the time), which for a parking lot is usually true.
The mistake that reads as inexperience is maintaining a separate int freeCount that you decrement on park and increment on unpark, as a source of truth. Under concurrency it drifts from the actual spot states — a lost update on the counter — and now your display board says "3 free" when the lot is full. Either derive the count from the spots, or maintain it with the same atomic operation that flips the spot, never as an independent second write. It's the mutable-balance-column mistake from 13 in a different costume.
Say the boundary out loud: "If this is one process, a lock on the spot object is enough and I wouldn't over-build it. The moment there are multiple app servers, in-memory locks are a lie and the only correct place for the invariant is the database — a conditional update or a version column. I'd ask which one we are before choosing, because designing for a distributed lot when it's a single JVM is wasted complexity, and designing for a single JVM when it's distributed is a correctness bug." Naming that fork — and refusing to pick blindly — is exactly the judgment this round grades.

2. Elevator System

The elevator flips the parking lot's emphasis. Parking is about allocation under contention; the elevator is about a state machine and a scheduling policy. There is no double-booking bug to catch here — the concurrency is a producer/consumer request queue, not a race for a scarce resource. Recognising that the two problems stress different muscles is itself a signal that you understand what LLD is testing.

Clarify the scope

QuestionWhy it mattersAssume
"One elevator car, or a bank of them?"A bank adds a dispatcher that picks which car serves a hall call — a second scheduling layer.Start with one car, then extend to a bank in follow-up. Say you're doing that on purpose.
"Two request types — buttons inside vs hall buttons outside?"Yes, and they differ: inside requests name a floor; hall requests name a floor and a direction.Both. Model them as distinct request types.
"What scheduling policy? FCFS, or the elevator (SCAN) algorithm?"The variation point → a strategy. FCFS is naive and jerky; SCAN sweeps in one direction serving everything.SCAN by default, pluggable. Ask — it's where the pattern lives.
"Capacity / weight limits, door timings, emergency/maintenance modes?"Real but bottomless. Model states for them; don't simulate physics.Model OUT_OF_SERVICE and door states; skip weight sensors.

Functional: a rider presses a hall button (floor + direction) or a car button (destination); the system schedules the car to serve all pending requests efficiently; the car moves floor by floor, opens doors when it arrives at a requested floor, and idles when there's nothing to do. Non-functional: requests arrive concurrently from many riders and must be enqueued safely; the car's behaviour must be a well-defined state machine so "what happens if a request comes in while moving down" has one unambiguous answer.

Nouns → classes, verbs → methods

Nouns: elevator (car), controller/dispatcher, request, floor, button, door, direction, state. Verbs: request, schedule/dispatch, move, stop, open/close (doors), idle. The word state appearing as a noun is your cue: the car is a state machine, and that's the pattern.

ClassResponsibilityKey methods
ElevatorControllerReceives all requests; runs the scheduling strategy; drives the car(s). The coordinator.submit(Request), step()

SchedulingStrategy (interface)The variation point: given car state + pending requests, what's the next floor to serve?nextStop(car, pending)

The core design in Java

enum Direction { UP, DOWN, IDLE }

interface Request {
    int targetFloor();
    Direction direction();     // IDLE for a car (inside) request
}
record HallRequest(int floor, Direction direction) implements Request {
    public int targetFloor() { return floor; }
}
record CarRequest(int floor) implements Request {
    public int targetFloor() { return floor; }
    public Direction direction() { return Direction.IDLE; }
}

// STATE PATTERN: each state knows what the car may legally do next.
interface ElevatorState {
    ElevatorState next(ElevatorCar car);   // advance one tick; may return a new state
    boolean canAccept(Request r);
}

The State pattern is what keeps this clean. Instead of one move() method with a nest of if (direction == UP && !pending.isEmpty() && ...), each state is a small class that answers "what do I do now, and what state do I become?" The illegal transitions (open the doors while moving) are simply not expressible, because MovingUp has no openDoors path — it can only become DoorsOpen by first arriving: 

class MovingUp implements ElevatorState {
    public ElevatorState next(ElevatorCar car) {
        Integer stop = car.scheduler().nextStop(car);
        if (stop == null)            return new Idle();          // nothing pending
        if (stop == car.floor())      return new DoorsOpen(car);   // arrived → serve
        car.moveOneFloor(Direction.UP);
        return this;                                              // keep climbing
    }
    public boolean canAccept(Request r) { return true; }         // queue it; scheduler orders it
}

class ElevatorController {
    private final ElevatorCar car;
    // thread-safe queue: rider threads produce, the control loop consumes
    private final BlockingQueue<Request> inbox = new LinkedBlockingQueue<>();

    public void submit(Request r) { inbox.offer(r); }   // called from ANY thread

    // The single control loop — the ONLY thread that mutates car state.
    public void run() {
        while (running) {
            inbox.drainTo(car.pending());     // pull all new requests in
            car.tick();                       // state.next(car): one step of the machine
        }
    }
}

Where a pattern earns its place

State for the car's lifecycle (Idle/MovingUp/MovingDown/DoorsOpen/OutOfService) — this is the textbook case and the interviewer wants to see it, because the alternative is a giant conditional that becomes unmaintainable the instant you add a fifth state. Strategy for the scheduling policy: FCFS vs SCAN vs "look ahead / LOOK" is exactly a swappable algorithm, and when the interviewer says "make it prioritise the direction of travel," you hand them a new ScanScheduler and change nothing else. Observer genuinely fits here for the panel lights and floor displays — "when the car reaches a floor, update the indicator on every floor" is textbook publish/subscribe, and it's one of the rare problems where mentioning Observer isn't over-reach. For a bank of elevators, add a top-level dispatcher whose job is "assign this hall call to the best car" — itself a strategy (nearest car, least-loaded, same-direction).

The line that shows you've thought past the single car: "For a bank of N elevators I'd split the two decisions. The dispatcher answers 'which car serves this hall call' — that's a strategy over car states. Each car independently answers 'what's my next stop given my own queue' — that's the per-car scheduler. Keeping those two policies separate means I can change how cars are assigned without touching how a car sequences its own stops. Conflating them into one scheduler is the design that gets ugly at the first follow-up."

The concurrency concern: one writer, many producers

This problem's concurrency is not a race for a scarce resource — it's the classic producer/consumer shape, and the senior move is to design the race out of existence rather than lock your way through it. Many rider threads press buttons concurrently (the producers); the car's physical state is mutated by exactly one control-loop thread (the single consumer). The only shared mutable point is the request queue, so:

  • The inbox is a BlockingQueue — thread-safe by construction, so button presses from any thread just offer() into it with no lock you write yourself. Producers never touch car state directly.
  • Exactly one thread mutates the car. Because only the control loop advances the state machine and reads the pending set, the car's fields need no synchronization at all — there is no second writer to race. This is the same insight as an actor or a single-writer event loop (and, not coincidentally, how a browser's single JS thread avoids UI races). Say "I'd rather have one owner of the mutable state than many threads sharing it behind locks — it's simpler and it's correct by design."
  • Deduping requests (pressing "5" twice shouldn't enqueue two stops) is a Set of pending floors, updated only on the control loop — no lock needed for the same reason.
Candidates over-engineer this one. They reach for a synchronized on every car method, or a lock around the whole controller, because "concurrency means locks." That's backwards. The elegant answer is a single-writer design where the only synchronized handoff is a standard thread-safe queue, and everything downstream of it is single-threaded and therefore trivially correct. Recognising that you can eliminate the race instead of guarding it is the more senior instinct — and it contrasts sharply with parking and BookMyShow, where you genuinely are racing for a scarce resource and cannot design the contention away.

3. BookMyShow (movie ticket booking)

This is the LLD sibling of 14 — Ticket Booking, and the two are worth studying together: page 14 is the distributed version (scale, seat holds across a cluster, the flash-sale thundering herd); this is the object model version (the classes, and the double-booking race at the level of a Seat). The concurrency concern here is the whole point of the problem — this is the one where the race is unavoidable and correctness is the grade. It's the parking-lot last-spot race with money and disappointed fans attached.

Clarify the scope

QuestionWhy it mattersAssume
"Seat-level booking, or just a count of tickets?"Seat-level is the interesting design — it's where the double-booking race lives. A pure count is a different, easier problem.Seat-level — specific seats in a specific hall.
"Do we hold seats while the user pays, with an expiry?"This is the crux. A hold is a temporary reservation; if payment doesn't complete, the seat must free itself.Yes — a 5-minute hold, then auto-release. Ask for it; it's the heart of the design.

Functional: browse shows for a movie in a city; see the seat map for a show with live availability; select seats and hold them; pay within the hold window to confirm; if the window lapses, seats release automatically. Nouns → classes, verbs → methods

Nouns: movie, theatre, screen, show, seat, booking, hold, user, payment, price. Verbs: search, select/hold, book/confirm, pay, cancel, release (expiry). The two verbs that carry the design are hold and confirm — everything hard is in the gap between them.

Call out the Seat vs ShowSeat split deliberately, because getting it wrong quietly breaks everything: "The physical seat A12 exists once, but its bookability is per-show — A12 can be free for the 6pm and sold for the 9pm. So the thing I lock and book is a ShowSeat, the intersection of a seat and a show, not the physical Seat. Modelling the physical seat as the bookable unit is the mistake that makes two shows fight over the same row." That one distinction is a senior tell — juniors almost always conflate the two.

The core design in Java

enum SeatStatus { AVAILABLE, HELD, BOOKED }

class ShowSeat {                       // seat ∩ show — THE unit of contention
    private final String id;             // (showId, seatNo)
    private SeatStatus status = SeatStatus.AVAILABLE;
    private String heldBy;               // user/booking holding it
    private Instant holdExpiry;

    // The guarded transition. Atomic check-and-set — this is THE race.
    synchronized boolean tryHold(String user, Duration window) {
        if (status == SeatStatus.BOOKED) return false;
        if (status == SeatStatus.HELD && Instant.now().isBefore(holdExpiry))
            return false;                  // someone else holds it, unexpired
        status = SeatStatus.HELD;         // free, OR an expired hold we may steal
        heldBy = user;
        holdExpiry = Instant.now().plus(window);
        return true;
    }
    synchronized boolean confirm(String user) {
        if (status != SeatStatus.HELD || !user.equals(heldBy)) return false;
        status = SeatStatus.BOOKED;       // point of no return
        return true;
    }
    synchronized void releaseIfExpired() {
        if (status == SeatStatus.HELD && Instant.now().isAfter(holdExpiry)) {
            status = SeatStatus.AVAILABLE; heldBy = null;
        }
    }
}

class BookingService {
    private final PricingStrategy pricing;
    private final PaymentGateway gateway;

    // Hold MANY seats all-or-nothing: if any seat fails, roll back the ones we got.
    public Booking hold(Show show, List<String> seatIds, String user) {
        List<ShowSeat> got = new ArrayList<>();
        // Lock seats in a CONSISTENT ORDER (by id) to prevent deadlock (see below)
        for (String sid : seatIds.stream().sorted().toList()) {
            ShowSeat seat = show.seat(sid);
            if (seat.tryHold(user, HOLD_WINDOW)) got.add(seat);
            else { got.forEach(ShowSeat::release); // roll back partial holds
                   throw new SeatUnavailableException(sid); }
        }
        return Booking.held(show, got, user);
    }

    public Booking confirm(Booking b, PaymentMethod pm) {
        gateway.charge(b, pm);                       // idempotency-key'd (see 13)
        b.seats().forEach(s -> s.confirm(b.user()));  // held → booked
        return b.confirmed();
    }
}

Where a pattern earns its place

State on the ShowSeat (AVAILABLE → HELD → BOOKED, with the timed HELD → AVAILABLE release) and on the Booking — the transitions are the correctness rules, so encoding them as an explicit machine means an illegal jump (BOOKED back to AVAILABLE) is simply unreachable. Strategy for pricing, exactly as in parking. Observer for the live seat map — "when a seat is held or booked, push the update to everyone watching this show's map" is real publish/subscribe and worth a sentence. Factory/Builder for assembling a Show or a Booking with its many fields is a fine mention. And the hold-with-expiry pattern itself — reserve, then confirm-or-auto-release — is the reusable idea the interviewer wants you to name; it's identical to the seat-hold in 14 and cousin to the authorize/capture split in 13.

The concurrency concern: double-booking, the whole ballgame

Two users click seat A12 for the 9pm show at the same instant. Both must not get it. This is the overselling problem at object granularity, and it is the thing being tested — do not let the interviewer have to ask. The naive version is the same check-then-act bug as the parking lot: 

// BROKEN — a gap between the check and the write.
if (seat.isAvailable())        // both threads see AVAILABLE
    seat.setStatus(HELD);      // both set HELD → seat sold twice

The fix is the same recipe as parking and payments — push the check-and-set into one atomic operation — but say all four layers, because a strong answer distinguishes the single-JVM toy from the real system:

  1. In-memory: tryHold() is synchronized on the ShowSeat, so the read of status and the write happen with no gap — exactly one thread flips AVAILABLE→HELD, the other gets false. The lock is per seat, so unrelated seats never contend.
  2. Persisted, the real answer — three flavours, pick and defend one:
    • Optimistic (usually best here): a @Version column on the seat row. UPDATE show_seats SET status='HELD', held_by=?, version=version+1 WHERE id=? AND status='AVAILABLE'. One row affected = you won; zero = someone beat you. No lock held across the user's think-time. Contention per seat is low, so optimistic wins.
    • Pessimistic: SELECT … FOR UPDATE on the seat rows inside the transaction. Simple and correct, but holds a DB lock — fine for the brief hold operation, bad if you hold it across payment.
    • Unique constraint: a seat_locks(show_seat_id PK) table; INSERT wins or throws a duplicate-key. The insert is the compare-and-set, same primitive as the idempotency key in 13.
  3. Never hold a DB lock across the payment. Payment takes seconds-to-minutes; a row lock held that long serialises the whole show and melts under a popular release. This is why the design is hold-then-pay: the hold is a soft reservation (a status + expiry, no lock held), the user pays against the hold, and confirm flips HELD→BOOKED. Say "I decouple the hold from the payment precisely so I'm never holding a lock while a human types their card number."
  4. Expiry needs a reaper, and it must itself be race-safe. A background sweep (or a lazy check-on-read) releases holds past holdExpiry. The subtle bug: a hold expires at the same moment its owner pays. Guard it — the confirm and the release both go through the seat's atomic transition, and the DB version check means only one of "steal the expired seat" vs "confirm it" can win. Mention 14's note that a TTL on a Redis hold key is the distributed version of this reaper.
Two deadlock-and-drift traps interviewers plant. First: booking multiple seats by locking them in request order — user A locks {12,15}, user B locks {15,12}, and they deadlock holding one seat each. Fix: always acquire seats in a consistent global order (sort by id), as the hold() code does. Second: an all-or-nothing multi-seat hold that grabs three of four seats and then finds the fourth taken — you must roll back the three you got, or you've stranded seats in a held state nobody owns. Both bugs are invisible in a single-seat happy path and both are exactly what a good interviewer pushes into.
The synthesis line that ties all three case studies together — say it and you've shown the interviewer the pattern behind the patterns: "The last parking spot, the last seat, the money in a ledger — these are the same problem. A scarce resource, two writers, and a check-then-act gap that loses one of them. The answer is always to collapse the check and the write into one atomic operation: a synchronized transition in one JVM, a conditional update or a version column or a unique constraint across machines. I never validate in application code and then write, because the gap between them is where the double-booking lives."

4. What "good" looks like under time pressure

You have 45 minutes, a whiteboard, and an interviewer watching you think. You will not finish a "complete" design and you are not meant to — the round is scored on how you work, not on lines of code. Here is what separates the candidates who pass from the ones who wrote more:

MoveWhat the strong candidate doesWhat sinks the weak one
Start with requirements, not classes Spends 5 minutes clarifying scope and stating what's cut. Writes the noun list and verb list on the board first. Starts drawing class Car in minute one, then re-does the model when a requirement lands.
Interfaces at the seams first Declares PricingStrategy, SchedulingStrategy as interfaces before any concrete class, and names the requirement each protects. Writes concrete classes with logic hard-coded, then can't absorb "now add weekend pricing" without edits.
Breadth before depth Gets a thin, complete skeleton of every core class up first, then deepens whichever one the interviewer probes. Perfects equals()/hashCode() and getters on one class while the design has no shape.
Don't gold-plate Says "I'll stub payment behind an interface and not build it — it's not what's being tested." Leaves it and moves on. Builds a full payment state machine and runs out of time before the seat-booking race — the actual grade.
Verbalise trade-offs Says the alternative and why they didn't pick it: "optimistic over pessimistic locking here, because seat contention is low." Presents one design as if it were the only option; can't answer "why not X?"
Flag concurrency proactively Marks the guarded operation as they write it: "this tryHold is the race — I'll come back to how it's enforced." Writes single-threaded code and has to be asked "what about two users at once?"

A concrete 45-minute budget, because a plan stops you from rat-holing: 

  0–5   min   Clarify scope. Functional + non-functional. State what you cut.
  5–10  min   Nouns → classes, verbs → methods. Write both lists on the board.
 10–15  min   Interfaces at the seams (strategy/state). Draw the class skeleton.
 15–30  min   Flesh out the 2–3 core classes. Enums, key method signatures.
 30–40  min   THE deep dive: concurrency. The guarded op + how it's enforced.
 40–45  min   Trade-offs, extensions ("bank of elevators", "distributed"), Q&A.

Two more things worth internalising. First: method signatures over method bodies. On a whiteboard, boolean tryHold(String user, Duration window) with a one-line comment about what it guarantees is worth more than ten lines of body — the signature communicates the design; the body is just typing. Write bodies only for the one or two methods where the logic is the point (the atomic transition, the SCAN step). Second: narrate continuously. Silence is the enemy — the interviewer is grading your reasoning, and reasoning they can't hear scores nothing. Say what you're about to do, do it, say why. A quiet candidate with a perfect model loses to a talking candidate with a good-enough one.

The meta-move that works on every LLD problem, including ones you've never seen: "I don't try to produce the final design in one pass. I get a rough, complete skeleton — every core class present but thin — and then I ask the interviewer which part they want me to go deep on, because that tells me where the points are. And I state my assumptions loudly as I go, so that if I've mis-scoped, they can correct me in minute five instead of minute forty." That turns the round into a collaboration you're steering rather than a test you're failing quietly — which is precisely the seniority signal the interviewer is trying to detect.

Interview Q&A

"Walk me through how you go from the prompt 'design a parking lot' to classes, in the first five minutes."

I don't write a class in the first five minutes — I clarify and extract. I ask the scope questions first: multiple floors, multiple vehicle sizes with fit rules, how the fee is computed, whether we track a specific car for exit. Each answer changes the class list, so I want them before I commit ink.

Then I underline the nouns and verbs in the requirements out loud. Nouns — lot, floor, spot, vehicle, ticket, fee — become classes; nouns that are just values, like size or colour, become enums. Verbs — park, assign, find, exit, calculate — become methods, and each verb tells me which class owns it: the one holding the state the verb changes. assign lives on ParkingSpot because the spot owns its occupied state.

Last, I look for the phrases that signal change — "the fee depends on," "we might assign the nearest spot" — and I put an interface at each: PricingStrategy, SpotAssignmentStrategy. Now the skeleton is: a coordinator (ParkingLot), an entity tree (floor/spot), a value object (Ticket), and two strategy seams. That's the recipe — nouns to classes, verbs to methods, variation points to interfaces — and it's the same recipe for elevator and BookMyShow.

"Two cars arrive at once and there's one spot left. How do you make sure only one gets it?"

This is a check-then-act race, and the fix is to collapse the check and the act into one atomic operation instead of two steps with a gap. The broken version does findFreeSpot() then assign() — both threads find the same spot and both assign it.

In a single JVM, I make tryAssign() on the spot synchronized: it verifies the spot is free and flips it to occupied without releasing the monitor, so exactly one thread wins and the other gets false and tries the next spot. I lock the spot, not the whole lot, so unrelated entries don't serialise.

Across multiple servers, the JVM lock is meaningless, so I push the invariant into the database: a conditional update — UPDATE spots SET vehicle=? WHERE id=? AND vehicle IS NULL — and I check the affected-row count. One row means I won; zero means someone beat me. Or an optimistic @Version column and retry on conflict, which suits a parking lot because contention is low — most spots are free. It's the exact same primitive as an idempotency key's INSERT … ON CONFLICT, and the exact same problem as not overselling a seat.

"Why is the elevator a State pattern and not just a few if-statements?"

Because the car behaves differently depending on what it's doing, and that "depends on the current mode" is the exact smell that State addresses. With if-statements, every method grows a branch for every state — move() needs "am I going up, down, idle, doors open, out of service?" — and adding a sixth state means editing every one of those methods and hoping you covered every combination. That's an Open/Closed violation and it's where the bugs breed.

With State, each mode is a small class that answers two questions: what do I do on the next tick, and what state do I become? MovingUp.next() either keeps climbing, arrives and becomes DoorsOpen, or runs out of work and becomes Idle. The illegal transitions — open the doors mid-flight — aren't expressible, because MovingUp has no path to DoorsOpen that skips arriving. Adding OutOfService is a new class, not a sweep through existing methods. The transition rules become the type structure instead of scattered conditionals.

"How does the elevator handle many people pressing buttons at once — where are the locks?"

There basically aren't any I write by hand, and that's the point. This isn't a race for a scarce resource like the parking spot — it's producer/consumer. Many rider threads press buttons (producers); exactly one control-loop thread mutates the car's state and advances the machine (the single consumer).

The only shared mutable point is the request inbox, and I make that a BlockingQueue, which is thread-safe by construction — button presses just offer() into it. Everything downstream of the queue runs on the single control thread: the pending-stops set, the current floor, the state — none of it needs synchronization because there is no second writer to race. It's the single-writer / actor model, the same reason a browser's one JS thread doesn't have UI data races.

The senior framing is that I designed the race out of existence rather than guarding it with locks. Reaching for synchronized on every car method — which many candidates do — is both unnecessary and a sign you haven't seen that single-ownership of mutable state is simpler and safer than shared-plus-locked.

"In BookMyShow, why do you distinguish a Seat from a ShowSeat?"

Because bookability is per-show, not per-seat. Physical seat A12 exists once in the hall, but it can be free for the 6pm showing and sold for the 9pm. If I model the physical Seat as the thing I lock and book, then booking A12 for the 6pm would appear to book it for the 9pm too — two shows fighting over one object.

So the bookable unit is ShowSeat: the intersection of a seat and a show, carrying its own AVAILABLE/HELD/BOOKED status and hold expiry. The physical Seat is static layout data; the ShowSeat is the per-showing state I contend over. Conflating them is one of the most common LLD mistakes on this problem, and getting it right is a clear senior signal.

"Why hold seats and confirm separately, instead of just booking on click?"

Because payment takes time — seconds if it's a saved card, minutes if the user is typing one — and I refuse to hold a database lock across a human's think-time. If "book on click" took a row lock and kept it through payment, one slow payer would serialise the entire show, and under a popular release the whole thing melts.

So the hold is a soft reservation: a status flip to HELD with an expiry timestamp, no lock held. The user pays against that hold, and confirm flips HELD→BOOKED. The atomic guarded operation is brief — just the status transition — and the slow part (payment) happens with nothing locked. If payment doesn't complete in the window, a reaper releases the seat. It's structurally the same as authorize-then-capture in a payment system: reserve cheaply now, commit expensively later, and make the reservation expire on its own.

"A user holds four seats, three succeed, the fourth is taken. What happens?"

All-or-nothing: I roll back the three I already held. If I don't, those three sit in a HELD state that no booking owns — stranded until their expiry, invisible to the next user, which is a real availability bug that looks like "the site says these seats are taken but nobody bought them." So the hold() method accumulates the seats it acquires, and on the first failure it releases everything it got before throwing.

The subtler trap in the same code is deadlock. If user A locks seats in the order {12, 15} and user B locks {15, 12}, they can each grab one and wait forever for the other. The fix is to always acquire seats in a consistent global order — sort the seat ids before locking — so a cycle can't form. Both bugs are invisible in the single-seat happy path and both are exactly what a good interviewer probes.

"Optimistic or pessimistic locking for the seat booking — which and why?"

Optimistic, for most theatres, because per-seat contention is low — any given seat is usually contended by zero or one users, not a crowd. A @Version column with a conditional update (… WHERE id=? AND status='AVAILABLE') lets writers proceed without holding a lock; the rare collision makes one writer's update affect zero rows, and they retry against another seat. No lock is held across the hold window.

I'd flip to pessimistic (SELECT … FOR UPDATE) only for a genuine flash-sale where a single hot show has thousands of users fighting for the same front-row block in the same second — there, optimistic retries thrash and a short pessimistic lock (held only for the brief hold transition, never across payment) can be calmer. The honest answer is "optimistic by default, and I'd measure contention before adding the complexity of pessimistic or a queue." That's also the page-14 answer at scale — a waiting room / virtual queue in front of the truly hot events.

"When would you NOT use a design pattern here?"

When I can't name the specific requirement change it protects against. A pattern is insurance against a variation, and if there's no variation, it's just indirection that makes the code harder to read. If the fee is a flat rate that will never change, I don't need a PricingStrategy interface with one implementation — that's ceremony. I add the seam when the interviewer signals the variation ("what if trucks pay double?") or when the domain obviously has one.

The failure mode I actively avoid is pattern-fishing: sprinkling Factory, Singleton, and Observer to look sophisticated. It reads as the opposite — someone who reaches for machinery before understanding the problem. The strong move is to introduce each pattern with its justification ("pricing is a strategy so surge pricing is a new class, not an edit") and to explicitly decline the ones that don't earn their place. Restraint is a seniority signal too.

"You're running low on time and only have the class skeleton. What do you do?"

I stop adding classes and go deep on the one thing that's actually graded: the concurrency-sensitive operation. For any of these problems that's the atomic guarded transition — tryAssign, tryHold — and how it's enforced across machines. A complete-looking skeleton plus a correct, well-explained double-booking fix beats a half-built payment subsystem every time, because the interviewer is testing whether I can find and contain the race, not whether I can type getters.

Then I verbalise the extensions I'd make with more time — "bank of elevators splits into a dispatcher plus per-car scheduler," "distributed version pushes the lock into the DB" — so the interviewer knows I see the rest of the design even though I didn't draw it. Naming what you'd do next is almost as good as doing it, and it's far better than silently running out of time mid-getter.

What to remember

Three problems, one recipe, one race. If you remember nothing else, remember that the object model is the answer and the guarded transition is the grade.

  • The recipe, every time: nouns → classes, verbs → methods, variation points → interfaces. A verb belongs to the class that owns the state it changes — misplace it and you get an anemic model with the logic marooned in a god-service. This is the whole of page 24, applied.
  • Interfaces at the seams, before concrete classes. Pricing, scheduling, spot-assignment are the places requirements move. Put an interface there and the follow-up ("add weekend pricing," "prefer nearest spot") is a new class, not a live rewrite. Anticipating the follow-up in the model is the top senior move in this round.
  • The three problems stress different muscles. Parking = allocation under contention (the last-spot race). Elevator = state machine + scheduling, with producer/consumer concurrency you design away. BookMyShow = the double-booking race, unavoidable and the whole grade. Knowing which is which is itself a signal.
  • One race, three costumes. Last parking spot, last seat, money in a ledger — all check-then-act on a scarce resource with two writers. The fix is always the same: collapse check-and-set into one atomic operation. Never validate in application code and then write; the gap between them is where the bug lives.
  • Name all four concurrency layers. (1) In-JVM: a synchronized transition on the contended object, locked per-object not globally. (2) Persisted-optimistic: a @Version column / conditional update, best when contention is low (usually). (3) Persisted-pessimistic: SELECT … FOR UPDATE, for genuine hot spots, held briefly. (4) Unique constraint: the INSERT … ON CONFLICT as compare-and-set. Pick per context and say why — a distributed design on a single JVM is wasted; a single-JVM lock across machines is a correctness bug.
  • Never hold a lock across a human or a payment. That's why booking is hold-then-confirm: a soft reservation (status + expiry, no lock) now, commit later, auto-release on timeout via a race-safe reaper. Same shape as authorize/capture in 13 and the seat-hold in 14.
  • Model the intersection, not the physical thing. ShowSeat (seat ∩ show), not Seat, is the bookable unit — bookability is per-show. Conflating them is the classic BookMyShow error.
  • Patterns map to prompts: "depends on" → Strategy (pricing, scheduling); "moves through phases" → State (elevator, seat/booking lifecycle); "notify when X" → Observer (display boards, live seat map); "one coordinator" → Singleton, but inject it, don't getInstance(). Introduce a pattern only when you can name the change it protects against — pattern-fishing reads as junior.
  • Single-writer beats shared-plus-locked. The elevant's elegant answer is one control thread owning all mutable car state, fed by a thread-safe queue — the race is designed out, not guarded. Recognising when you can eliminate contention rather than lock it is a senior instinct.
  • Deadlock & drift, the planted traps: acquire multiple resources in a consistent global order (sort seat ids) or two users deadlock; roll back partial multi-seat holds or you strand seats; never keep a freeCount as an independent source of truth — derive it or update it atomically with the resource, or it drifts (the mutable-balance-column mistake again).
  • Under time pressure: requirements before classes, breadth before depth, signatures over bodies, don't gold-plate, narrate continuously, and get a thin complete skeleton up before deepening whatever the interviewer probes. A complete skeleton plus a correct race fix beats a half-built payment module. The round grades how you think, so let them hear it.
One last calibration for this round specifically: the interviewer is deciding whether you can be handed a fuzzy problem and trusted to produce a model a team can live in. That's not "knows the most patterns" — it's "puts responsibilities in the right place, guards the seams that will move, finds the concurrency bug before being asked, and says the trade-off out loud." Show restraint where restraint is right, reach for the pattern only when it earns its place, and treat the whiteboard as a conversation you're steering. That's the hire.
