09 — Design a Rate Limiter

The small question that goes deep fast. Archetype: quotas & shared counters under concurrency.

Why this matters in the interview

Interviewers love this one precisely because it looks trivial. "Allow 100 requests per minute per user" is a sentence a junior engineer will answer in ninety seconds with a hash map and a counter — and then spend the remaining forty-three minutes being taken apart. Every layer of the onion is a real distributed-systems problem: the counter has a boundary bug, the fix costs memory, the memory fix costs accuracy, the whole thing has to be shared across N app servers so now you have a read-then-write race, the shared store is now in the hot path of every single request in your system so latency is life-or-death, and then the interviewer asks the question that actually decides the round: "what happens when Redis is down?" There is no diagram to memorise here. There is only judgment, which is exactly what a 7-YOE candidate is graded on.

And the archetype transfers further than almost any other in this phase. Any "at most N per unit of time, counted across a fleet" problem is this problem. Rate limiting an API. Enforcing a plan quota (10,000 API calls a month). A billing meter that counts metered usage. Ad-budget pacing — don't spend more than $500/day of this advertiser's money across 200 ad servers. Concurrent-session limits — this account may stream on 4 devices. Coupon redemption caps. Login-attempt throttling. Every one of them is: a shared counter, incremented concurrently by many machines, checked against a limit, where the cost of being slightly wrong is the design's central negotiation. Learn the shape once and you can walk into any of those prompts and reuse this entire page. Say that out loud in the interview, too — recognising that "this is a shared-counter problem" is itself a senior signal.

1. The prompt

As the interviewer will actually say it — vague on purpose, because the vagueness is the test:

"Design a rate limiter for our public API."

That's it. Nothing about who's limited, what "our public API" is, whether the limit is for security or for money, or how exact it needs to be. If you start drawing boxes, you have already lost points. There are three things unspecified here that each lead to a different design, and finding them is step one.

The single most common failure on this question is answering the algorithm question and stopping. Candidates spend twenty minutes lovingly comparing token bucket and leaky bucket, feel good about it, and never get to the part the interviewer actually cares about — that there are fifty app servers and one counter, and the counter is now a distributed-consensus problem in a trench coat. Budget your time: the algorithms are ten minutes. Distribution, latency, and failure are thirty.

2. Requirements

The clarifying questions to ask first

AskWhy it mattersAnswer you'll assume
Is this for abuse prevention, or for fairness/billing? Abuse prevention tolerates slop — if a scraper gets 105 requests instead of 100, nobody cares. Billing does not: if you sell "10,000 calls/month" and let someone have 10,400, you gave away product. This decides whether you may cache state locally. Abuse prevention and fairness for a public API. Billing quotas are a separate, harder system — I'll call that out.
Is a small overshoot acceptable, or must the limit be exact? This is the question that changes the design. "Approximately 100/min" gives you local counters, batched sync, and sub-millisecond checks. "Exactly 100/min, globally, always" forces every request through a single serialisation point, and that point is now your throughput ceiling and your SPOF. Small overshoot is fine (a few percent). I'll defend that explicitly.
What's the limit key — user, IP, API key, endpoint? Different keys have wildly different cardinality and hot-key risk. An IP limit behind a corporate NAT punishes a whole office. A global limit (one key for everyone) can't be sharded at all. Primarily API key (authenticated) with IP as the fallback for unauthenticated traffic.
Do rules differ per plan / per endpoint? Decides whether "the rules" are a config constant or a small distributed configuration system with its own cache and propagation story. Yes — free/pro/enterprise tiers, and some expensive endpoints have their own tighter limit.
Do we control the clients? If yes, you can push some limiting client-side and rely on well-behaved backoff. If it's a public API, assume clients are hostile, buggy, or both. Public API. Assume nothing about the client.
Open with: "Before I design anything — is this limiter protecting us from abuse, or is it enforcing a paid quota? Because if a user going slightly over the limit costs us nothing, I can keep counters local to each server and sync them lazily, and the check costs microseconds. If it has to be exact, every request has to touch one shared counter, and that counter becomes both my latency budget and my single point of failure. Those are two different systems and I'd rather build the right one." That sentence, in the first ninety seconds, tells the interviewer you have built one of these.

Functional requirements

  • Limit by identity — user ID, API key, or IP address; the limiter accepts a key and doesn't care what produced it.
  • Configurable rules — "100 requests per minute", "1,000 per hour", "10 per second on POST /search". Rules are data, not code, and must be changeable without a deploy.
  • Multiple simultaneous rules — a request can be checked against a per-second and a per-day limit; it passes only if it passes all of them.
  • Reject with HTTP 429 Too Many Requests, with a Retry-After header so a well-behaved client knows exactly when to come back.
  • Expose state proactively via X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset on every response, not just the rejections — so clients can self-pace instead of discovering the wall by hitting it.
  • Explicitly out of scope: DDoS mitigation (that's the CDN/edge, and it is not the same thing — see the follow-ups), WAF rules, authentication itself.

Non-functional requirements — these are the real ones

  • Negligible added latency. This runs on the hot path of every request in the entire system. If the check costs 20 ms, you have added 20 ms to your P50 for every endpoint you own, forever, in exchange for a feature that does nothing on 99.9% of requests. Budget: ≤ 1–2 ms at P99, and be prepared to defend it. This constraint alone kills several otherwise reasonable designs.
  • Accurate enough — bounded, understood overshoot. Not "exact" unless someone pays for exact.
  • Fail gracefully. The limiter is a dependency of everything. If it's down, it must not take the API down with it. A rate limiter whose outage becomes a total outage is a self-inflicted wound and interviewers will hunt for whether you've thought about it.
  • Distributed — enforced consistently across N app servers / gateway instances, which is where all the difficulty lives.
  • Cheap — memory per key must be small and bounded, because there are millions of keys.
Notice how lopsided this is: the functional requirements are almost boring, and every single interesting decision on the page comes from the non-functional ones. That's characteristic of infrastructure problems, and it's why the method in tutorial 01 insists you split them. Say which side is driving your design.

3. Estimation

Round hard, say you're rounding, and get to the one number that matters.

  Users                       1,000,000 registered
  Peak API traffic            100,000 requests/sec  (given / assumed)

  Rate-limit checks needed    1 per request (at least)
  ────────────────────────────────────────────────────────────
  ⇒ The limiter itself must sustain 100,000 QPS.

  With two rules per request (per-second AND per-day), and a
  read+write per check, that's ~200k-400k Redis ops/sec.

  Latency budget
  ──────────────
  Typical API endpoint P99            ~200 ms
  Acceptable limiter overhead         ~1%  ⇒ ~2 ms
  Redis GET/SET, same AZ              ~0.2-0.5 ms round trip   ✅
  Postgres SELECT + UPDATE, same AZ   ~2-10 ms, and it's DISK  ❌
  Cross-region Redis call             ~80 ms                   ❌❌

  Memory
  ──────
  Token bucket per key: ~20 bytes (tokens: float, lastRefill: long)
  1M active keys × 20 B  ≈ 20 MB          → trivial. One Redis node.
  1M keys × 2 rules      ≈ 40 MB          → still trivial.

  Sliding-window LOG, same 1M keys, 100 req/min each:
  100 timestamps × ~16 B × 1M keys ≈ 1.6 GB, and it's proportional
  to the LIMIT, so a 10,000/hr tier is 100× worse.

The number that drives the design is 100k QPS at sub-millisecond latency on a shared counter. Say it out loud. That single line does three things for you at once: it rules out "just put a counter in Postgres" (you'd be doing 100k disk-backed read-modify-writes per second on the same rows — the row locks alone would melt it, and even if they didn't, the latency is 10× your entire budget); it rules out any design where the limiter is a network hop to another service rather than a hop to an in-memory store; and it tells you the state has to be tiny and O(1) per key, which is what will kill the sliding-window log later.

Say: "The 100k QPS figure isn't really about capacity — Redis does that on one box. It's about the fact that this check is now the most frequently executed piece of code in my entire company. Anything I add here, I add to every request. That's why I'm going to be aggressive about O(1) state and about keeping the check to a single round trip, and why I'll eventually argue for doing the check locally with periodic sync."

One more number to keep in your pocket: the storage is a non-issue and you should say so and move on. A rate limiter's data is a few tens of megabytes of ephemeral counters. Candidates waste minutes estimating storage here out of habit. The scarce resources on this problem are latency and coordination, not bytes.

4. API & the headers you owe the client

The rate limiter mostly has no public API — it's middleware, and its "interface" is what it does to other people's responses. But there are two contracts worth writing down, and the headers one is the part candidates forget and interviewers notice.

4.1 The internal check API

The whole limiter, as far as the calling code is concerned, is one function:

public interface RateLimiter {
    // key: "user:9f31c2", "ip:203.0.113.7", "apikey:abc:endpoint:/search"
    Decision check(String key, Rule rule);
}

public record Decision(
    boolean allowed,
    long    limit,        // e.g. 100
    long    remaining,    // tokens/quota left after this request
    long    resetEpochSec,// when the bucket is full / window rolls
    long    retryAfterSec // 0 when allowed
) {}

Keep it that small. A rate limiter that returns a boolean is a rate limiter that cannot produce good headers, and you will regret it. Returning remaining and reset costs nothing — the algorithm computed them anyway — and it's the difference between clients that self-pace and clients that hammer you until they get a 429.

4.2 The HTTP contract — what the client sees

  --- Allowed ---
  HTTP/1.1 200 OK
  X-RateLimit-Limit: 100
  X-RateLimit-Remaining: 42
  X-RateLimit-Reset: 1752505260        # epoch seconds

  --- Rejected ---
  HTTP/1.1 429 Too Many Requests
  Retry-After: 18                      # seconds (RFC 9110 — also accepts an HTTP-date)
  X-RateLimit-Limit: 100
  X-RateLimit-Remaining: 0
  X-RateLimit-Reset: 1752505260
  Content-Type: application/json

  {"error":"rate_limit_exceeded",
   "message":"Rate limit of 100 req/min exceeded for API key.",
   "retry_after":18}

Three details that separate a considered answer from a recited one:

  • 429, not 503, and never 200-with-an-error-body. 429 is the specific status; HTTP clients, SDKs and proxies have retry logic keyed on it. 503 says "I'm broken", which invites retries you don't want. And a 200 carrying an error breaks every generic client on earth.
  • Retry-After is the load-bearing header. It's the only thing that turns a rejection into a cooperative rejection. Without it, a client's only strategy is to poll, and a rate-limited client polling is precisely the retry storm you were trying to prevent (see follow-ups).
  • Send the X-RateLimit-* headers on successes too. As a React developer you already know the client-side value of this: with Remaining in hand, a well-written SDK can throttle itself, show a progress bar, or queue work — before it hits the wall. (The IETF is standardising this as RateLimit-Limit/-Remaining/-Reset without the X-; mentioning that you know the X- forms are the de-facto convention and the un-prefixed forms are the emerging standard is a nice, cheap detail.)
Don't emit the headers by re-querying the limiter. Some candidates describe a design where the middleware calls the limiter to decide, then calls it again to fetch remaining/reset for the headers. That's two round trips per request on the hottest path in the system, and the second read is racy anyway. The check returns everything the headers need, in one call, atomically.

5. Data model — rules and counter state

There are two completely different kinds of data here, with opposite characteristics, and conflating them is a mistake.

5.1 The rules — small, read-mostly, config-shaped

Rules change a few times a day at most and are read on every request. That is a caching problem, not a database problem. Store the source of truth wherever it's convenient (Postgres, or a YAML file in a config repo — genuinely, both are defensible), and have every limiter instance hold the entire rule set in local memory, refreshed every ~10–30 seconds or pushed via a config service. The rule set is kilobytes. It never touches the hot path over the network. 

-- source of truth; changed by humans, read by a background refresher
CREATE TABLE rate_limit_rules (
    id             BIGSERIAL PRIMARY KEY,
    scope          TEXT NOT NULL,   -- 'api_key' | 'ip' | 'user' | 'global'
    plan           TEXT,                -- 'free' | 'pro' | 'enterprise' | NULL = any
    endpoint       TEXT,                -- '/v1/search' | NULL = all endpoints
    algorithm      TEXT NOT NULL,   -- 'token_bucket' | 'sliding_counter' | ...
    capacity       INT  NOT NULL,   -- burst size (token bucket) or limit (window)
    refill_per_sec NUMERIC,             -- steady-state rate
    window_secs    INT,                 -- for window algorithms
    UNIQUE (scope, plan, endpoint)
);

Resolution order at request time, all in local memory: most specific rule wins (plan + endpoint → plan → scope default). A request can match several rules — a per-second one and a per-day one — and it must satisfy all of them.

5.2 The counter state — tiny, ephemeral, brutally hot

This is the opposite in every dimension: written on every request, read on every request, worthless five minutes after it's written, and it must never touch a disk. It lives in Redis, keyed by the limit key, with a TTL so that idle keys evict themselves and you never need a cleanup job. 

  # Token bucket — one Redis HASH per (key, rule), ~20 bytes of payload
  KEY   rl:tb:{apikey:abc123}:{search_per_sec}
  TYPE  hash
  VALUE { tokens: 7.5, last_refill_ms: 1752505243871 }
  TTL   ~ 2 × (capacity / refill_rate)   # idle bucket refills to full anyway,
                                         # so forgetting it is FREE and CORRECT

  # Fixed / sliding window counter — one STRING per (key, window)
  KEY   rl:fw:{apikey:abc123}:1752505200   # bucket = floor(now / window)
  TYPE  string (integer)
  TTL   2 × window_secs                  # keep the PREVIOUS window alive too —
                                         # the sliding counter needs to read it

  # Sliding window LOG — one ZSET per key; score = timestamp = member
  KEY   rl:sl:{apikey:abc123}
  TYPE  zset
  TTL   window_secs
The TTL is doing real design work here, and saying so scores. "Expiry isn't housekeeping, it's the correctness argument. A token bucket that's been idle longer than its refill time is, by definition, full — so a bucket that disappears from Redis and gets recreated full is not a bug, it's exactly right. That means I need no eviction job, no cleanup cron, no unbounded key growth: the state is self-cleaning because forgetting is the correct behaviour." Compare it to the eviction problem in tutorial 08 — here you get it for free.
Those {curly braces} in the Redis keys aren't decoration — they're Redis Cluster hash tags. Only the text inside the braces is hashed to pick a slot, which lets you force all the rules for one API key onto one shard so a multi-rule check (per-second + per-day) can run in a single Lua script. A Lua script in Redis Cluster may only touch keys in one slot; if you forget the hash tag, your beautiful atomic script fails with CROSSSLOT the day you scale out. That is a genuinely hard-won detail and it costs you one sentence.

6. High-level design

The architecture is unremarkable and you should present it fast — all the interesting content is inside two of these boxes. Get here in ten minutes and spend the rest of the time on sections 7 and 8.

   Client ──request──▶  │      API Gateway         │
                        │  ┌────────────────────┐  │
                        │  │ Rate-limit         │  │   1. resolve key + rules
                        │  │ middleware         │──┼── 2. check counters

                        ┌───────────────────────────────┐

The walkthrough — one request, end to end

  1. Identify. The gateway has already authenticated the request, so it has an API key (or, for anonymous traffic, the client IP — taken from the trusted proxy's X-Forwarded-For, not from a header the client can forge). The limit key is apikey:abc123.
  2. Resolve rules. Look up the rules for (plan=pro, endpoint=/v1/search) in the local in-memory map. Two rules match: 10/sec burst 20, and 100,000/day. Zero network calls. Zero milliseconds.
  3. Check counters. One round trip to Redis running a Lua script that evaluates both rules atomically and returns allow/deny plus remaining and reset for each. ~0.3 ms. This is the only network hop the limiter adds, and section 8 is entirely about making it correct and then making it disappear.
  4. Deny path. If any rule fails, respond 429 immediately with Retry-After computed from the tightest failing rule. Increment a rate_limit_rejections{key,rule} counter and move on. The request never reaches the service, never opens a DB connection, never costs you anything — that's the entire point of doing this at the gateway.
  5. Allow path. Attach the X-RateLimit-* headers (from the values the script already returned) and forward to the service.
Note what is not in this diagram: a "rate limiter service." Making the limiter a separate network service that the gateway calls adds a full RPC hop — serialise, cross the network, deserialise, and now you've built a second thing that can be down — to a decision that is fundamentally one Redis command. The limiter is a library plus a data store, not a service. If an interviewer draws a "Rate Limiter Service" box and hands you the marker, ask why it isn't in-process; it's a legitimate question and the honest answers (polyglot fleet, central policy, a team that owns it) are about org structure, not latency.

7. Deep dive 1 — the algorithms

This is where the interview really happens — the first half of it, anyway. There are five algorithms worth knowing, and they form a chain: each one exists because the previous one has a specific, nameable flaw. Present them that way. A candidate who lists five algorithms has memorised a blog post; a candidate who says "the simplest thing has this exact bug, so you fix it like this, and that fix costs you memory, so you make this trade" is doing engineering out loud, and that's the thing being graded.

7.1 Fixed window counter — the simplest thing, and its famous bug

Chop time into fixed windows (say, each calendar minute). Keep one counter per (key, window). Increment on each request; reject when the counter exceeds the limit; let the key expire when the window rolls. 

  bucket = floor(now / window)          # 1752505243 / 60 → 29208420
  key    = "rl:fw:" + userKey + ":" + bucket

  count = INCR key
  if count == 1: EXPIRE key (2 × window)     # first hit sets the TTL
  if count > limit: reject

State: one integer per key per window. Memory: O(1), a few bytes — the cheapest thing on this page. Cost: one INCR, which is atomic in Redis, so it's almost race-free for free (almost — see 8.1). It's genuinely tempting.

And then here is the bug, which you should draw, because drawing it is more persuasive than describing it:

  Limit: 100 requests per minute.

   window 12:00-12:01          window 12:01-12:02
  ├───────────────────────────┼───────────────────────────┤
  │                    ██████ │ ██████                    │
  │                    100 req│ 100 req                   │
  │                  at 12:00:59│ at 12:01:00              │
  └───────────────────────────┴───────────────────────────┘
                       └──────┬──────┘
                    200 requests in ~2 seconds.
                    Both windows are "within limit".
                    Your backend saw 2× the limit, instantly.

  Worst case is always 2× the limit, and it can happen in an
  arbitrarily short instant straddling the boundary.

That's the boundary burst, and it is the single most important thing to be able to explain on this page, because every other algorithm in this section exists to fix it. It's not a theoretical concern: a client that simply retries on a schedule aligned to the minute will discover it by accident, and an attacker will discover it on purpose. If your limit exists because 100/min is genuinely all your backend can take, a fixed window quietly permits 200/min at the worst possible moment.

Say: "Fixed window is one INCR and a TTL, and I'd happily ship it for something forgiving — a per-day quota, say, where a burst at the midnight boundary is harmless. But it allows 2× the limit across a window boundary, and if the limit exists to protect a fragile downstream, 2× is exactly the number that hurts. So the question I'd ask is: does this limit exist for tidiness, or is it load-bearing?" Naming the flaw and then saying when you'd ship it anyway is worth more than reciting a better algorithm.

7.2 Sliding window log — exact, and expensive in the wrong dimension

Forget windows. Store a timestamp for every request in a sorted set, keyed by user. On each request: drop everything older than now − window, count what's left, and if that's under the limit, add this request's timestamp. 

  ZREMRANGEBYSCORE rl:sl:user123  0  (now - 60_000)   # evict stale
  count = ZCARD rl:sl:user123
  if count < limit:
      ZADD rl:sl:user123  now  now:randomSuffix       # suffix = uniqueness
      EXPIRE rl:sl:user123 60
      allow
  else: reject

  The window truly slides — it's the last 60 seconds, continuously:

   t-60s                                            now
     ├──────────── the window, always ───────────────┤
     · · ·  ×  ×  ×      ×   ×  ×  × ×     ×  ×  ×  ×
   evicted  └─────────── counted (must be < 100) ────┘

This is exact. No boundary bug, ever — the window is always precisely the trailing 60 seconds. It also gives you a perfect Retry-After: it's the oldest surviving timestamp plus the window, to the millisecond.

And it's O(number of requests in the window) memory, per key. That's the wrong thing to be proportional to. Your heaviest users — the ones you most want to limit — are exactly the ones who cost you the most memory. A user allowed 10,000 requests/hour costs you 10,000 timestamps, ~160 KB, and a million such users is 160 GB. Worse, in the naive version you store timestamps even for rejected requests if you're not careful, which means an attacker who's being rejected can still make you allocate memory. (Fix: only ZADD on allow — as above. Note the cost of that fix: a rejected request no longer extends the penalty, which is usually what you want anyway.)

The sliding log's cost is proportional to the limit, not to the number of users. Candidates say "O(N) memory" and wave at it. Be specific: "memory per key is proportional to the configured limit, so raising a customer's rate limit raises my memory bill linearly — my most expensive keys are my best customers. That coupling is what makes me reject it for a general-purpose limiter, even though it's the only exact one."

When you would use it: small limits where exactness matters and the volume is low. "5 failed logins per 15 minutes" — five timestamps per key, exactly correct, no boundary exploit on a security-sensitive limit. That's a genuinely good fit and naming it shows you're choosing rather than ranking.

7.3 Sliding window counter — the pragmatic compromise

You want the fixed window's O(1) memory and the sliding log's lack of a boundary cliff. You can have ~95% of both, and the trick is one line of arithmetic: keep two fixed-window counters (current and previous) and weight the previous window's count by how much of it still overlaps the trailing window. 

  elapsed = fraction of the current window that has passed  (0.0 → 1.0)

  estimate = current_count + previous_count × (1 − elapsed)

  ────────────────────────────────────────────────────────────────
  Limit 100/min. It is 12:01:15, so elapsed = 15/60 = 0.25.
  Previous window (12:00-12:01) saw 80 requests.
  Current window  (12:01-12:02) has seen 12 so far.

  estimate = 12 + 80 × (1 − 0.25)
           = 12 + 60
           = 72        → 72 < 100, allow. Remaining ≈ 28.

  Intuition: 75% of the trailing 60-second window still lies inside
  the previous calendar minute, so we count 75% of its traffic.

   12:00              12:01              12:02
     ├──────────────────┼──────────────────┤
     │      80 req      │   12 req so far  │
     └────────┬─────────┴──────┬───────────┘
        count 75% of it   count 100% of it
              ▲                ▲
              └──── trailing 60s window ────┘  (as of 12:01:15)

State: two integers per key. Memory: O(1) — about 20 bytes, same as fixed window. Boundary bug: gone, or near enough. Take the 12:00:59 + 12:01:00 attack from 7.1: the second burst arrives at elapsed ≈ 0, so the previous window's 100 requests are counted at ~100% weight, the estimate is immediately ~100, and the burst is rejected. Failure mode: it assumes traffic in the previous window was evenly distributed. It wasn't — it was a spike at the end. So the estimate can be slightly wrong in both directions: it can allow a few extra requests, or reject a few that a perfect limiter would have allowed. Cloudflare famously measured this against exact counting on real traffic and found the error to be a fraction of a percent, with roughly 0.003% of requests wrongly rejected. That is a rounding error, and you are trading it for a 99% memory reduction.

Say: "Sliding window counter is my default recommendation when the semantics are 'N per window', because it costs the same as a fixed window — two integers — and eliminates the boundary burst by interpolating the previous window's count. The approximation it makes is that traffic within the previous window was uniform, which is false, so it's off by well under a percent in practice. I'll take that error to avoid storing a timestamp per request." Then give the formula. Producing current + previous × (1 − elapsed) from memory, unprompted, is a strong signal — most candidates know the name of this algorithm and not the arithmetic.

7.4 Token bucket — what real systems actually use

Different model entirely, and a better one for APIs. A bucket holds up to B tokens (the capacity) and is refilled at R tokens per second. Every request removes one token. No token, no service. 

              refill: R tokens/sec (continuous)
                        │
                        ▼
                 ┌─────────────┐
                 │ ● ● ● ● ●   │  capacity B = 10
                 │ ● ● ●       │  tokens now = 8
                 └──────┬──────┘
                        │ each request takes 1 token
                        ▼
                   request allowed   (tokens ≥ 1)
                   request 429'd     (tokens < 1)

  Behaviour: an idle client accumulates up to B tokens, so it may
  BURST up to B requests instantly — then it's throttled down to the
  steady rate R, because that's all the refill can sustain.

That burst behaviour is the whole point, and it's why Stripe, AWS, GitHub and most serious APIs use token bucket. Real clients are bursty: a page loads and fires eight parallel requests; a nightly job wakes up and syncs. Punishing that is user-hostile, and a limiter that forbids all bursts (leaky bucket, 7.5) makes normal applications feel broken. Token bucket says: burst all you like, up to B, as long as your long-run average stays under R. B and R are two independent knobs — burst tolerance and sustained rate — and having two knobs is a feature. The window algorithms only give you one.

The implementation trick: lazy refill

The naive reading of "refilled at R tokens per second" is a timer that walks every bucket and tops it up. With ten million buckets that is an absurd amount of work to do continuously, and 99.99% of it is wasted on buckets nobody is looking at. Don't run a timer. Compute the refill on read, from elapsed time. A bucket's token count is a pure function of (tokens at last touch, time since last touch), so you can reconstruct it lazily at the exact moment someone asks — and if nobody ever asks, you did no work at all and the key just expires. 

public final class TokenBucket {
    private final double capacity;      // B — max burst
    private final double refillPerSec;  // R — sustained rate

    private double tokens;               // fractional on purpose
    private long   lastRefillNanos;

    public TokenBucket(double capacity, double refillPerSec) {
        this.capacity     = capacity;
        this.refillPerSec = refillPerSec;
        this.tokens          = capacity;      // start full
        this.lastRefillNanos = System.nanoTime();
    }

    /* The whole algorithm. No timer, no background thread, no scheduler. */
    public synchronized boolean tryAcquire(int permits) {
        refill();
        if (tokens >= permits) {
            tokens -= permits;
            return true;
        }
        return false;
    }

    private void refill() {
        long now     = System.nanoTime();
        double secs  = (now - lastRefillNanos) / 1_000_000_000.0;
        tokens          = Math.min(capacity, tokens + secs * refillPerSec);
        lastRefillNanos = now;
    }

    /* For Retry-After / X-RateLimit-Reset: when will 1 token exist? */
    public synchronized long retryAfterMillis() {
        refill();
        if (tokens >= 1) return 0;
        double deficit = 1 - tokens;
        return (long) Math.ceil(deficit / refillPerSec * 1000);
    }
}

Four things in that code are worth narrating out loud, because each is a small piece of evidence that you've written one:

  • Tokens are a double. If they were integers you couldn't express "refill 0.5 tokens" and a rate of 30/minute would round to nothing on a fast check. Fractional tokens are how sub-1/sec rates work at all.
  • Math.min(capacity, …) is the burst bound. Remove that clamp and a client idle for a week comes back with 600,000 tokens and can hammer you at will. The clamp is the burst policy.
  • nanoTime(), not currentTimeMillis(), for the local version — it's monotonic, so an NTP correction or a leap second can't hand a client free tokens or freeze their bucket. (In Redis you'll use the server's clock instead, precisely so that all app servers agree — see 8.2.)
  • tryAcquire(permits), not tryAcquire(). One line of generality that buys you weighted limiting for free: an expensive endpoint can cost 10 tokens and a cheap one 1. That's how you rate-limit by cost rather than by request count, and it's a follow-up you can now answer in one sentence.
If this looks familiar, it should: this is exactly what Guava's RateLimiter and Bucket4j do, and it's what Spring Cloud Gateway's RequestRateLimiter filter runs (as a Lua script, against Redis) under the names replenishRate = R and burstCapacity = B. Naming the Spring filter is a cheap, concrete win in an enterprise loop — it says you'd reach for a battle-tested implementation rather than hand-rolling one in production, which is the correct instinct and the opposite of what the interview is asking you to do on the whiteboard.

7.5 Leaky bucket — for when the downstream is fragile

A queue drained at a constant rate. Requests arrive whenever they like and join the queue; the queue is served at exactly R per second; if the queue is full, new requests are dropped (or rejected). The output is perfectly smooth regardless of how spiky the input is. 

  bursty in                     smooth out
  ▶▶▶▶      ▶     ▶▶▶▶▶▶  ──▶ ┌──────────┐ ──▶  ▶  ▶  ▶  ▶  ▶  ▶
                              │ ░░░░░░░░ │      exactly R/sec
                              │ queue    │      forever
                              └────┬─────┘
                                   │ overflow → 429 (or drop)
                                   ▼
  Token bucket lets a burst THROUGH (up to B) and then throttles.
  Leaky bucket NEVER lets a burst through — it flattens it,
  by making it WAIT. That waiting is added latency.

Note the inversion: token bucket smooths the average and permits bursts; leaky bucket smooths the instantaneous rate and forbids them. Use leaky bucket when the thing behind you genuinely cannot absorb a spike — a legacy mainframe with a fixed thread pool, a third-party API that charges per call and rate-limits you, a printer queue, an SMS gateway. It's traffic shaping, not just admission control. The cost is real and you must name it: queuing adds latency, and a request that sits in a bucket for 3 seconds may be a request whose client gave up 2 seconds ago (which is the deadline problem from tutorial 06 — don't do work for a client that's gone). For a synchronous public API, users usually prefer a fast 429 to a slow 200.

"Leaky bucket" is used to mean two different things in the literature and candidates get burned by it. There's the queue version above (shapes traffic, adds latency, what most people mean) and the meter version — a bucket that leaks at a constant rate and that each request adds to, rejecting when it would overflow — which is mathematically a mirror image of the token bucket and adds no latency at all. If the interviewer says "leaky bucket", it costs you five seconds to say "the queue-based one that shapes output, yes?" and it prevents an entire conversation at cross purposes.

8. Deep dive 2 — making it distributed

This is the other half of where the interview happens, and it's the reason the question gets asked at all. Everything in section 7 works beautifully on one machine. You do not have one machine.

8.1 The two problems, in order

Start by naming both, because candidates usually spot the first and get ambushed by the second. 

  PROBLEM 1 — the counter must be SHARED.

    50 app servers, each with its own in-memory token bucket
    configured at 100/min.

    Real enforced limit = 50 × 100 = 5000/min.

    Your "100 per minute" limit is a 5000 per minute limit, and it
    silently gets looser every time you autoscale. A limit that
    changes when you add capacity is not a limit.

  PROBLEM 2 — once it's shared, it's CONCURRENT.

    Two servers, same user, same instant, limit = 100:

      Server A: GET count → 99          ─┐  both read 99
      Server B: GET count → 99          ─┘  before either writes
      Server A: 99 < 100 → allow, SET 100
      Server B: 99 < 100 → allow, SET 100

    Two requests allowed. Counter says 100. Reality: 101 served.
    Classic lost update / read-then-write race.

    With 50 servers hammering the same key, you don't overshoot
    by one — you overshoot by roughly the number of concurrent
    checks in flight.

Problem 1's answer is immediate: move the counter to a shared store, and the store is Redis, for the reasons the estimation section already forced — sub-millisecond, in-memory, atomic primitives, and the exact data structures this problem needs (see tutorial 06). Postgres is out on latency and on row contention: 100k concurrent UPDATEs on the same row is a lock convoy, not a rate limiter.

Problem 2 is the interesting one, and there are three answers with different quality.

8.2 Fixing the race — three options, ranked

Option A — INCR alone (atomic, but not sufficient on its own)

Redis's INCR is atomic and returns the new value, which means for the fixed window algorithm you can collapse the read and the write into one operation and the race disappears entirely: 

  count = INCR key            # atomic; returns post-increment value
  if count == 1: EXPIRE key ttl
  if count > limit: reject    # decide AFTER incrementing, on the returned value

This is genuinely race-free for the allow/deny decision, and it's why fixed-window-with-INCR is the world's most deployed rate limiter. But notice what it costs. You increment before you know whether you'll allow, so rejected requests still count against the user — a client being throttled digs its own hole deeper (arguably fine, arguably not; it's a policy choice you should name). And it only works because fixed-window's state update is a single commutative operation. The moment your algorithm needs read → compute → conditionally write — which token bucket, sliding log and sliding counter all do — INCR can't save you. There's also a subtle bug lurking: the INCR and the EXPIRE are two commands, and a crash between them leaves a key with no TTL that never resets. That user is now banned forever. (You'd never find this in testing. You'd find it in production, once.)

Option B — WATCH/MULTI/EXEC (optimistic transactions — works, don't use it)

Redis's transactions with WATCH give you optimistic concurrency control: watch the key, read it, queue your writes, and EXEC fails if anyone touched the key meanwhile. Correct, and exactly the optimistic-locking pattern you know from Postgres version columns. But under contention on a hot key — which is precisely the situation a rate limiter exists to handle — it degenerates: everyone's EXEC fails, everyone retries, and you have a retry storm inside your rate limiter, plus two-to-three round trips per attempt on your hottest path. It's the right idea in the wrong place. Mention it, dismiss it, and say why: "optimistic concurrency is a bad fit when contention is the norm rather than the exception."

Option C — a Lua script (the standard answer)

Redis executes a Lua script atomically on its single thread: no other command from any other client runs in the middle of it. So you ship the entire read-compute-write decision into the server, and the race cannot exist — not because you locked anything, but because there is no interleaving to lose an update in. One round trip, one atomic decision, all the state and all the arithmetic on the same box.

Here is the token bucket with lazy refill, as an atomic Redis script. This is the artifact worth being able to reproduce:

-- rate_limit.lua  ·  KEYS[1] = "rl:tb:{apikey:abc}:search"
-- ARGV: capacity, refillPerSec, nowMillis, requestedTokens
-- Returns: { allowed(0|1), remaining, retryAfterMillis }

local key        = KEYS[1]
local capacity   = tonumber(ARGV[1])
local refillRate = tonumber(ARGV[2])   -- tokens per second
local now        = tonumber(ARGV[3])   -- ms, passed IN by the caller
local requested  = tonumber(ARGV[4])

local state  = redis.call('HMGET', key, 'tokens', 'ts')
local tokens = tonumber(state[1])
local ts     = tonumber(state[2])

if tokens == nil then                     -- first sighting, or the key expired:
  tokens = capacity                        -- an absent bucket is a FULL bucket
  ts     = now
end

-- LAZY REFILL: reconstruct the tokens that "accrued" while we weren't looking
local elapsed = math.max(0, now - ts) / 1000.0
tokens = math.min(capacity, tokens + elapsed * refillRate)

local allowed = 0
local retryAfter = 0

if tokens >= requested then
  tokens  = tokens - requested
  allowed = 1
else
  local deficit = requested - tokens
  retryAfter = math.ceil(deficit / refillRate * 1000)
end

redis.call('HSET', key, 'tokens', tokens, 'ts', now)

-- self-cleaning: a bucket idle for (capacity/rate) seconds is full anyway,
-- so letting it vanish is not just safe, it's exactly equivalent.
redis.call('PEXPIRE', key, math.ceil(capacity / refillRate * 1000) * 2)

return { allowed, math.floor(tokens), retryAfter }

Calling it from Spring, with the script pre-loaded and invoked by SHA so you're not shipping the source every request:

@Component
public class RedisTokenBucketLimiter implements RateLimiter {

    private final StringRedisTemplate redis;
    private final RedisScript<List> script =
        RedisScript.of(new ClassPathResource("rate_limit.lua"), List.class);

    @Override
    public Decision check(String key, Rule rule) {
        try {
            List<Long> r = redis.execute(script,
                List.of("rl:tb:{" + key + "}:" + rule.name()),
                String.valueOf(rule.capacity()),
                String.valueOf(rule.refillPerSec()),
                String.valueOf(Instant.now().toEpochMilli()),
                "1");
            return Decision.of(r);
        } catch (RedisConnectionFailureException | QueryTimeoutException e) {
            // THE decision of this page — see section 10.
            return fallback.check(key, rule);   // local bucket, fail-open-ish
        }
    }
}
Say the reason Lua works, not just that it works: "Redis is single-threaded, so a Lua script is a critical section I get for free — while it runs, no other client's command can interleave with it. That's what makes this clean: I'm not taking a distributed lock, I'm exploiting the fact that Redis already serialises everything. The whole read-modify-write happens where the data is, in one round trip, and the lost-update race is structurally impossible rather than defended against." The phrase "structurally impossible rather than defended against" is the difference between a 3-YOE and a 7-YOE answer.
Pass the timestamp in; don't call TIME inside the script — and don't use each app server's clock either. Both halves matter. Historically Redis forbade non-deterministic commands like TIME before a write because scripts were replicated verbatim to replicas (effects replication since Redis 5 makes this safe, but the habit is right and interviewers of a certain vintage will ask). Meanwhile, using each app server's own clock means 50 slightly-skewed clocks writing to one bucket, and a server whose clock runs 2 seconds fast hands its users free tokens. Pick one clock and be explicit about which: the app server's (simple, tolerates skew badly) or Redis's own via a single TIME call (one authoritative clock, at the cost of that determinism caveat). Naming clock skew unprompted on a rate-limiter question is a real signal — almost nobody does.

8.3 The second problem: you just put Redis in every request's hot path

Congratulations, the counter is correct. Now look at what you've built: every single request in your entire system now makes a network call to one Redis cluster before it does anything else. Roughly 0.3–0.5 ms in the same AZ if all is well — which is fine, and honestly you could stop here and defend it. But be aware of what you've signed up for:

  • You've added a hard dependency to everything. Redis's availability is now an upper bound on your API's availability (until section 10 fixes that).
  • P99 is not P50. Redis's P99 under load, plus a connection-pool wait, plus a GC pause on your side, is not 0.3 ms. It's occasionally 20 ms, and that 20 ms is now on the P99 of every endpoint you own.
  • Cross-AZ or cross-region, this is 2 ms or 80 ms and the design is simply dead. The limiter's Redis must be local to the servers it serves. Which raises the multi-region question — see section 11.

8.4 The optimisation: local buckets with periodic sync

Here's the move that shows you've operated one of these. Stop asking Redis for permission on every request. Ask it for a budget, occasionally, and enforce locally. 

  Each app server keeps a LOCAL token bucket per hot key.
  Redis holds the authoritative global budget.

  Every ~100 ms (or when a local bucket runs dry), the server
  reconciles:  "I consumed 37 tokens for user X since we last spoke.
                Deduct them, and lease me 40 more."

   App server 1 ──┐   local bucket: 40 tokens  ─┐
   App server 2 ──┼──▶ local bucket: 40 tokens  ─┼─▶ Redis: global
   App server 3 ──┘   local bucket: 40 tokens  ─┘   counter, synced
                                                    every 100 ms
   Hot path cost:  a few NANOSECONDS (an in-memory CAS).
   Redis traffic:  100k QPS → ~10-30 calls/sec/server. A 1000× cut.
   Cost of the trade: for up to 100 ms, servers act on stale
   information, so the fleet can collectively overshoot by roughly
   (leased tokens still held) in the worst case.

This is not exotic — it's how high-volume limiters actually work, and it's the same lease/batch idea as the ID-range allocation in the URL shortener (grab a block of 1,000 IDs from the counter, hand them out locally, come back for more). Same pattern, same trade: batch the coordination, accept bounded staleness, get orders of magnitude. Note it also degrades beautifully — if Redis blips, each server still has a local bucket with a lease in it and keeps enforcing something reasonable while it reconnects.

The knobs, and how to describe the trade honestly:

This is the trade-off to make explicitly, out loud, tied back to the clarifying question you asked in section 2: "For abuse prevention I'd absolutely take local buckets with a 100 ms sync — the limiter stops being a network call at all, and the cost is that a determined client might get 105 requests instead of 100 during a burst. Nobody is harmed by that. For billing I would not: if I'm selling 10,000 API calls and a fleet-wide race gives away 400 of them, that's revenue, and multiplied across every customer it's a real number. Metered billing goes through the atomic Redis path, or it gets reconciled asynchronously against a durable event log rather than enforced by the limiter at all." That paragraph is the single highest-value thing on this page. It is the same technical mechanism being accepted and rejected on business grounds, which is exactly what senior judgment looks like.

Four candidate locations. The interviewer wants to hear you pick one and say why the others lose — and, crucially, to hear that "client-side" is never enforcement.

Being able to name both the rule and its exception is what "awareness" looks like.

10. When Redis is down — fail open or fail closed?

This is the best follow-up on the page, and interviewers use it as the senior filter. You have just made Redis a hard dependency of literally every request. Redis is now down — a failover, a network partition, a cluster upgrade gone wrong, an OOM. What does the limiter do? 

  FAIL OPEN                         FAIL CLOSED
  (allow the request)               (reject with 429/503)

  "I can't tell if you're over       "I can't tell if you're over
   the limit, so I'll let you         the limit, so I'll assume the
   through."                          worst and reject you."

  Risk: unlimited traffic hits       Risk: your entire API is down
        the backend during the             because a CACHE is down.
        limiter outage.

  Availability: preserved.           Availability: destroyed.
  Protection:   gone.                Protection:   absolute.

The default is fail open, and here's the argument to make: a rate limiter is a protective control, not a correctness control. Its job is to stop a minority of abusive traffic from harming the majority. If failing closed means that a Redis blip converts "some users are being throttled" into "every user gets a 429", you have taken an outage in a component that only matters for 0.1% of requests and turned it into a 100% outage of your product. A rate limiter whose failure mode is a total outage is a self-own — you've built a system that is strictly less available than the one you were protecting. Say exactly that.

But fail open is not "do nothing", and this is where most candidates stop and lose the point. Fail open, degraded, with a local fallback:

  • Local in-memory bucket per instance. If Redis is unreachable, each server enforces the limit on its own. With 50 servers, the effective limit is 50× — much too loose, and yet infinitely better than unlimited. A single abusive client hitting one server still gets throttled. You've traded exactness for availability, temporarily, on purpose.
  • Tighten the local limit during the outage. If you know traffic is roughly evenly balanced across 50 servers, set the fallback bucket to limit/50 × a fudge factor. Now the fleet-wide enforcement is approximately right even with no shared state at all. This is a genuinely clever answer and it costs one sentence.
  • Short-circuit, don't hang. Wrap the Redis call in a circuit breaker with a tight timeout (5–10 ms — remember your latency budget; a call that takes 500 ms has already failed even if it eventually succeeds). Without a breaker, "Redis is slow" means every request waits 10 ms on a dead socket and your thread pools fill — the cascading failure from tutorial 06, triggered by your own safety feature. The timeout on the limiter's Redis call is the most important config value in this design.
  • Alarm loudly. "Limiter degraded" is a page, not a log line. You are unprotected and you should know it.

When you fail closed instead

Flip the default when the thing behind the limiter will actually fall over, or will actually cost money, if you let the traffic through:

  • A third-party quota you must not exceed. Your payment provider allows 500 TPS and starts charging (or blocking you) above it. Unlimited traffic there doesn't degrade you — it gets your merchant account throttled. Reject. Fail closed.
  • Metered/billed usage. If you can't count it, you can't charge for it. Better to reject than to serve for free.
  • A fragile downstream with no headroom — the legacy mainframe with 40 connections. Letting the flood through kills it, and it takes an hour to bring back. A 429 for five minutes is cheaper than a one-hour outage.
  • Security-critical limits — login attempts, OTP sends. If you can't count failed logins, you are running an open credential-stuffing target. Fail closed (or fall back to a conservative local limit and accept the false positives).
Don't answer "it depends" — make the choice and defend it, which is the whole skill being tested: "Fail open, with a local per-instance fallback bucket, a circuit breaker with a 10 ms timeout, and a page. My reasoning is that the limiter protects against a minority of abusive traffic, and if its failure mode is 'reject everything', I've built something whose outage is worse than the problem it solves. But I'd invert that for any limit protecting a hard external quota — a payment provider's TPS cap, say — because there, letting the traffic through doesn't degrade my system, it breaks a contract. So: fail open by default, fail closed where the limit is load-bearing, and make that a per-rule flag rather than a global policy." That last clause — per-rule, not global — is the one that gets remembered.
The trap here is answering "fail open" reflexively because you read it somewhere, and then folding the instant the interviewer pushes ("so an attacker just DDoSes your Redis and then gets unlimited access to your API?"). Have the comeback ready: "Yes — which is why the fallback is a local bucket rather than nothing, why the limiter's Redis isn't reachable from the internet, and why volumetric attacks are handled at the edge, not here. A rate limiter is not a DDoS defence, and I wouldn't claim it as one."

11. Scaling & bottlenecks

What breaks first at 10×, in the order it breaks.

11.1 The hot key — the first thing that melts

Rate-limit keys are sharded by user/API key, so load spreads evenly across the Redis cluster — right up until it doesn't. Two ways it doesn't:

  • One hammering client. A single API key sending 50k RPS maps to one key, which maps to one slot, which lives on one shard. Adding shards does not help: consistent hashing distributes keys, and you have one key. That shard is now single-threaded (Redis!) and doing 50k round trips a second while the rest of the cluster idles. This is the celebrity-partition problem from tutorial 04, and it is delicious irony: the abusive traffic your rate limiter exists to stop is the traffic that kills your rate limiter.
  • A global limit. "The whole system does at most 10,000 writes/sec to the payment provider" is one counter, by definition. There is no key to shard on. This is the genuinely unavoidable case and you should say so rather than pretending sharding solves it.

The fixes, in the order you'd apply them:

  1. Local buckets with periodic sync (8.4) — this is the real answer, and note that it solves the hot key for free. If each server only talks to Redis 10 times a second per key, a hammering client generates 50 servers × 10/sec = 500 Redis ops/sec instead of 50,000. The hot key evaporates. Say this explicitly: "batching the coordination isn't just a latency optimisation, it's my hot-key mitigation — it's the same fix wearing a different hat."
  2. Shard the counter (key splitting). For a genuinely global limit, split it into N sub-counters — global:writes:0…9, each with limit/10 — and have each app server deterministically use one. You've turned one hot key into ten warm ones. The cost: if load isn't perfectly even across the shards, one sub-limit fills while others have headroom, and you under-admit. Classic, and worth naming: you traded a hot key for a utilisation loss. The refinement is to let a server steal budget from a neighbouring shard when its own runs dry, which recovers most of the loss and costs you a second round trip on the unhappy path.
  3. Reject earlier. A client that is already being throttled doesn't need a fresh Redis check on every request. Cache the "this key is blocked until T" decision locally for a second or two. A blocked abuser then costs you a hash-map lookup, not a network call — which is exactly what you want, since blocked abusers are the ones sending the most traffic.

11.2 Redis itself

  • Vertical first, and it's usually enough. One Redis node does ~100k+ ops/sec with pipelining, and your state is tens of megabytes. Say that out loud — it's an honest, calibrated statement and it prevents you from over-architecting a cluster you don't need. "At 100k QPS with local buckets I'm doing maybe 2k Redis ops/sec. One node, one replica, done. I'd only shard when I can show I need to."
  • Then Redis Cluster, sharded by limit key (with the hash tags from section 5.2 so one key's multiple rules stay in one slot and the Lua script stays legal).
  • Replicas don't help with correctness. Redis replication is asynchronous — reading a counter from a replica gives you a stale count, and a limiter that reads stale counts is a limiter that over-admits. Replicas are for failover, not for scaling the limiter's reads. And note the sharp edge: on failover, unreplicated writes are lost, so a few buckets snap back to a more permissive state. For a rate limiter that's completely acceptable — a momentary blip of leniency — and saying "I'm fine losing counter state on failover; this is ephemeral data and the blast radius is a few seconds of leniency" shows you've thought about durability and correctly decided you don't need it.

11.3 Multi-region

Don't share the counter across regions. A cross-region Redis call is ~80 ms, and adding 80 ms to every request in your system to enforce a limit precisely is an obviously bad trade. Run an independent limiter per region and accept that a client hitting two regions gets up to 2× the limit. If that's unacceptable (it usually isn't — clients are pinned to a region by DNS/GeoDNS anyway), split the budget: each region enforces limit/num_regions, optionally with slow async rebalancing of unused budget between them. If someone insists on an exact global limit across regions, tell them the truth: "that requires cross-region coordination on every request, which costs 80 ms and a shared failure domain. I'd push back and ask what the exactness is actually worth."

11.4 The other bottlenecks worth naming

  • Rules distribution. With per-plan and per-endpoint rules, the rule set is real config. Don't fetch it per request. Poll into local memory every ~10 s and accept that a rule change takes 10 seconds to take effect fleet-wide — which is fine, and which you should state rather than leave implicit.
  • Key cardinality. Per-IP limiting on IPv6 is a cardinality trap: an attacker has a /64 with 18 quintillion addresses in it and gets a fresh bucket for each one. Limit on the /64 prefix, not the address. This is a lovely, specific detail that almost nobody mentions.
  • Observability. Emit rejection rate per rule and per key. The two alarms that matter: rejections spiking (you're under attack, or you shipped a bad limit and are now throttling your own biggest customer), and rejections going to zero (your limiter is silently failing open and you'd never otherwise know).

13. Follow-ups you should expect

"The user has a per-second limit AND a per-day limit. How?"

Multiple rules, evaluated together, and the request passes only if all pass. Two important details. First, evaluate them in one round trip — one Lua script, taking all the applicable rules, checking each, and consuming from all of them only if all would allow. If you check them sequentially you'll consume a token from the per-second bucket and then get rejected by the daily one, leaking budget from the first. Second, the Retry-After comes from the rule that will free up last — no point telling someone to retry in 1 second when their daily quota resets in 6 hours. And put the tightest/cheapest rule first so the common rejection short-circuits fast. The hash-tag trick from 5.2 is what lets all those keys live in one slot so a single script can touch them.

"Different limits per plan tier?"

That's the rules table from section 5.1 doing its job: rules are data, keyed by (scope, plan, endpoint), resolved most-specific-first, cached in local memory, refreshed every 10 seconds. The limiter itself doesn't change at all — it takes a key and a rule. Worth saying: "a plan upgrade should take effect within seconds, not on the next deploy, which is why the rules are config and not code." The one wrinkle: the key must include the plan or the rule name, otherwise upgrading a user from free to pro leaves them holding a bucket sized for free.

"Clients ignore your 429 and hammer you. Now what?"

This is a retry storm, and it's the tutorial 06 material paying off. A client that retries immediately on a 429 turns a rate limit into a busy-wait, and a thousand such clients retrying in lockstep is a self-inflicted DDoS. Three layers of response:

  • Cooperate first: send Retry-After, and ship an SDK that honours it with exponential backoff plus jitter — jitter especially, because synchronized retries are the whole problem and randomness is what desynchronizes them.
  • Make rejection cheap for you: cache "blocked until T" locally (11.1) so a hammering client costs a map lookup, not a Redis call. Rejecting must always be cheaper than serving, or the abuse still wins.
  • Escalate: a client that ignores 429s for sustained periods gets a longer penalty (a "block" bucket with an increasing TTL), then a hard block at the edge/WAF. At some point this stops being rate limiting and starts being abuse mitigation — and you should say so, because knowing where your system's responsibility ends is itself a signal.

"Is this how you'd stop a DDoS?"

No, and this is a trap question — take the bait deliberately. A rate limiter is an application-layer, per-identity control. It assumes the request reached your gateway, which means it already cost you a TCP handshake, a TLS handshake, and a slot in a connection pool. A volumetric DDoS is 500 Gbps from a botnet of 100,000 IPs — each one individually under your per-IP limit, and collectively saturating your network before your limiter ever executes a line of code. "You can't rate-limit your way out of a saturated uplink." DDoS is handled upstream, at the edge: CDN/scrubbing (Cloudflare, Shield), anycast to spread the load, SYN cookies, IP reputation, challenges. The rate limiter is for the legitimate-but-excessive client — the buggy integration, the scraper, the customer who wrote a while-loop. Different threat, different layer, different tool. Being able to say clearly what your system is not for is a senior signal, and this is one of the cleanest opportunities to show it.

"How would you rate-limit by cost rather than by request count?"

Already free from the design: tryAcquire(permits). A cheap GET costs 1 token, a heavy aggregation costs 50. This is exactly how GitHub's GraphQL API and most cloud provider APIs work, because with GraphQL "one request" is a meaningless unit of load — the client chooses how expensive their query is. Compute the cost up front (query complexity analysis) and charge the bucket accordingly. If cost is only known after execution, charge an estimate up front and reconcile afterwards by deducting the difference — which permits some overshoot on a single surprising request, and is fine.

"Should a rejected request count against the limit?"

A policy question with no universal answer, which is why it's a good interview question. Counting rejections means a client in a hot loop digs itself deeper and can never recover — punitive, arguably correct for abuse, and it's what the naive INCR implementation does by accident. Not counting them (the token bucket above, which only decrements on allow) means the client recovers exactly at the refill rate, which is more predictable and friendlier to a client that's merely buggy. Default: don't count rejections, but do separately track rejection rate per key so a client that ignores 429s can be escalated deliberately, as a policy, rather than by accident of implementation.

Interview Q&A

1. What's the boundary problem with a fixed window counter, and what does it actually cost you?

With a 100/min limit and calendar-minute windows, a client can send 100 requests at 12:00:59 and 100 more at 12:01:00. Both windows are individually within limit; your backend just took 200 requests in about one second. The worst case is always 2× the configured limit, delivered in an arbitrarily short instant straddling the boundary.

What it costs depends entirely on why the limit exists. For a daily quota, nothing — a burst at midnight is harmless. But if the limit is sized to what a downstream can survive, 2× at the worst possible moment is precisely the thing you were preventing, and a client whose retry schedule happens to align to the minute will hit it by accident. The fix that keeps the O(1) memory is the sliding window counter.

2. Give me the sliding window counter formula and explain what approximation it makes.

estimate = current_window_count + previous_window_count × (1 − elapsed_fraction_of_current_window)

At 12:01:15 with 60-second windows, 25% of the current window has elapsed, so 75% of the trailing 60 seconds still lies inside the previous minute — weight the previous window's count by 0.75. If the previous minute saw 80 requests and this one has seen 12: 12 + 80 × 0.75 = 72.

The approximation: it assumes traffic in the previous window was uniformly distributed. It wasn't — it might all have been in the last two seconds. So the estimate can be a little high or a little low, and you can both over-admit and over-reject by a small margin. Measured against exact counting on real traffic, the error is a fraction of a percent. You pay that to store two integers instead of a timestamp per request. For anything that isn't billing, it's an obviously good trade.

3. Why is the sliding window log rarely the right answer despite being exact?

Its memory is O(the limit) per key — a timestamp per request in the window. That's proportional to the wrong thing: your most permissive limits (your best-paying customers) become your most expensive keys. A 10,000/hour tier costs 10,000 timestamps (~160 KB) per active key; a million such keys is ~160 GB of RAM for a rate limiter.

It also costs more Redis ops per check (ZREMRANGEBYSCORE + ZCARD + ZADD) than the O(1) algorithms.

When it is right: small limits where exactness is the product. "5 failed logins per 15 minutes" — five timestamps, exactly correct, no boundary exploit on a security-sensitive limit, and it gives you a precise Retry-After for free (oldest surviving timestamp + window).

4. Explain token bucket's lazy refill, and why you don't run a timer.

A bucket's token count is a pure function of (tokens at last touch, time elapsed since). So you never proactively refill anything — on each check you compute tokens = min(capacity, tokens + elapsed_seconds × refill_rate), and then decide. No scheduler, no background thread, no cron sweeping ten million buckets.

That matters at scale for two reasons. A timer would do continuous work proportional to the total number of buckets, when the work you actually need is proportional to the number of active ones — and 99.99% of buckets are idle at any instant. And a bucket that's been idle longer than capacity/rate is by definition full, so a key that quietly expires out of Redis and gets recreated full is not a bug — it's the correct value. That's why the state is self-cleaning: forgetting an idle bucket is free and correct.

Two details in the implementation that matter: tokens must be fractional (a double) or sub-1/sec rates round to zero, and the min(capacity, …) clamp is the burst policy — remove it and a client idle for a week returns with hundreds of thousands of tokens.

5. Token bucket vs leaky bucket — when would you actually pick leaky?

They're opposites in what they smooth. Token bucket bounds the average rate and explicitly permits bursts up to the bucket capacity — which is what you want for a public API, because real clients are bursty (a page loads and fires eight parallel requests) and punishing that makes normal apps feel broken. Leaky bucket (the queue version) bounds the instantaneous rate: requests queue and drain at exactly R/sec, so the output is perfectly flat and no burst ever reaches the downstream.

Pick leaky when the thing behind you genuinely cannot absorb a spike: a legacy system with a fixed thread pool, an SMS gateway, a third-party API that rate-limits you. It's traffic shaping, not admission control.

Its cost, which you must name: queuing adds latency, and a request that sits in the bucket for three seconds may belong to a client that timed out two seconds ago — pure waste. For a synchronous public API, users generally prefer a fast 429 to a slow 200.

6. You have 50 app servers each with an in-memory limiter set to 100/min. What's the actual limit?

5,000/min — and worse, it changes every time you autoscale. A limit that gets looser when you add capacity isn't a limit. (It's also not even evenly 5,000: it depends entirely on how the load balancer happens to distribute that user's requests.)

So the counter must be shared, which means Redis. And that immediately creates the second problem: with a shared counter, two servers can both read 99, both conclude "99 < 100, allow", and both write 100. Two requests allowed on the last token. That's a classic read-then-write race / lost update, and with 50 servers you don't overshoot by one — you overshoot by roughly the number of checks in flight.

7. How do you make the distributed check atomic? Why is Lua the standard answer?

Ship the entire read-compute-write into a Lua script executed by Redis. Redis is single-threaded, and a script runs to completion with no other client's command interleaved — so the script is a critical section, for free. The lost-update race isn't defended against, it's structurally impossible. One round trip, atomic decision, arithmetic executed where the data lives.

The alternatives and why they lose:

  • INCR alone is atomic and genuinely sufficient — but only for fixed window, where the state update is a single commutative op. Token bucket, sliding log and sliding counter all need read → compute → conditionally write, and INCR can't express that. It also counts rejected requests against the user, and the separate EXPIRE can be lost to a crash, leaving a key with no TTL that never resets.
  • WATCH/MULTI/EXEC is correct optimistic concurrency, but it degenerates under contention — and contention is the normal case for a rate limiter. Everyone's EXEC fails, everyone retries, and you've built a retry storm inside your rate limiter.
  • A distributed lock is ~10× the cost and solves a problem Redis's single thread already solved. Don't.
8. The Redis round trip is now on every request in the company. What do you do about it?

Local token buckets with periodic sync. Each app server keeps an in-memory bucket per hot key, enforces locally at nanosecond cost, and reconciles with Redis every ~100 ms (or when its local budget runs dry): "I consumed 37 tokens for user X, deduct them and lease me 40 more."

The effect: the hot-path check becomes an in-memory operation instead of a network call, and Redis traffic drops by roughly 1000× (100k QPS becomes tens of calls per second per server). It also incidentally solves the hot-key problem — the hammering client that would have pinned one Redis shard now generates a handful of ops per second.

The cost: for up to one sync interval, servers act on stale information, so the fleet can collectively overshoot by roughly the sum of the outstanding leases. Bounded and computable, but real. Take this trade for abuse prevention; refuse it for billing, where the overshoot is revenue you gave away.

9. Redis is down. Do you fail open or fail closed?

Fail open by default — allow the traffic — with a local fallback bucket, a circuit breaker with a ~10 ms timeout, and a page. The reasoning: a rate limiter is a protective control, not a correctness control. It exists to stop a minority of abusive traffic from harming the majority. If failing closed turns a Redis blip into "every user gets a 429", you've taken an outage in a component that matters for 0.1% of requests and converted it into a 100% product outage. A limiter whose failure mode is a total outage is a self-own.

But fail open is not "do nothing". Fall back to a per-instance in-memory bucket — with 50 servers the effective limit is 50×, which is far too loose and infinitely better than unlimited; tighten the fallback to limit/N and it's approximately right. And the circuit breaker is not optional: without it, "Redis is slow" means every request blocks on a dead socket and your thread pools fill — your safety feature causes the cascading failure.

Fail closed when the limit is load-bearing: a hard third-party quota (a payment provider's TPS cap — exceeding it breaks a contract, it doesn't just degrade you), metered billing you can't charge for if you can't count it, a fragile downstream with no headroom, or a security limit like login attempts. Best answer: make it a per-rule flag, not a global policy.

10. Where do you put the rate limiter, and is client-side ever acceptable?

API gateway for the coarse limits (per-IP, per-API-key, per-endpoint): traffic dies at the front door before it costs you a thread, an auth check or a DB connection, it's configured in one place, and it protects services you haven't written yet. Plus a thin layer inside the service for limits needing business context the gateway can't see — "five password resets per account per hour". Defence in depth, because the layers have different information and therefore catch different things.

Client-side is a courtesy, never enforcement. Throttling in your SDK genuinely reduces load and gives instant feedback (grey out the button) — but anyone can curl your API, and a limit you can disable in devtools is not a limit. Do it as well, never instead. A candidate who proposes client-side limiting as the mechanism has failed the question.

11. A single API key sends 50k RPS. What happens to your Redis cluster?

One key → one hash slot → one shard, which is single-threaded, now doing 50k ops/sec while the rest of the cluster idles. Adding shards does not help: consistent hashing distributes keys, and there is one key. This is the celebrity-partition problem from tutorial 04, with a nasty irony — the abusive traffic your limiter exists to stop is the traffic that kills your limiter.

Fixes, in order: local buckets with periodic sync (the hot key evaporates — 50 servers × 10 syncs/sec = 500 ops/sec instead of 50,000); cache the "blocked until T" decision locally so an already-throttled client costs a map lookup rather than a network call; and for a genuinely global limit — where there is no key to shard on by definition — split the counter into N sub-counters of limit/N each. That last one trades a hot key for a utilisation loss (one sub-limit fills while others have headroom), which you can partly recover by letting a server borrow budget from a neighbouring shard.

12. Is a rate limiter a DDoS defence?

No, and confusing the two is a real red flag. A rate limiter is an application-layer, per-identity control that runs after the request has reached your gateway — which means it has already cost you a TCP handshake, a TLS handshake and a connection-pool slot. A volumetric DDoS is 500 Gbps from 100,000 IPs, each one individually under your per-IP limit, saturating your uplink before your limiter executes a single line of code. You cannot rate-limit your way out of a saturated network link.

DDoS is handled upstream at the edge: CDN/scrubbing providers, anycast, SYN cookies, IP reputation, challenge pages. The rate limiter's job is the legitimate-but-excessive client — the buggy integration, the scraper, the customer who wrote a while-loop. Different threat, different layer, different tool. Saying clearly what your system is not for is one of the cheapest senior signals available.

What to remember

  • The archetype: quotas & shared counters under concurrency. Rate limiting, plan quotas, billing meters, ad-budget pacing, concurrent-session caps, coupon limits — all the same problem: N per unit time, counted across a fleet. Recognise the shape and reuse this entire page.
  • The clarifying question that changes the design: abuse prevention or billing? Overshoot-tolerant → local counters, lazy sync, microsecond checks. Exact → every request through one serialisation point, which is now your latency ceiling and your SPOF. Ask it in the first ninety seconds.
  • The number: 100k QPS, sub-millisecond, on a shared counter. It rules out Postgres (latency + row contention) and any design where the check is an RPC to another service. Storage is a non-issue — say so and move on.
  • Fixed window is 2× at the boundary. 100 at 12:00:59 + 100 at 12:01:00 = 200 in a second. This flaw is why every other algorithm exists.
  • Sliding window counter: current + previous × (1 − elapsed). O(1) memory, no boundary cliff, sub-1% error (it assumes the previous window was uniform). The default for "N per window" semantics.
  • Sliding window log is exact but O(limit) memory — your best customers become your most expensive keys. Use it only for small, security-sensitive limits ("5 failed logins / 15 min").
  • Token bucket is what real systems use (Stripe, AWS, GitHub). Capacity B = burst, refill R = sustained rate — two independent knobs, and bursts are normal client behaviour. Lazy refill: compute tokens from elapsed time on read; never run a timer; an absent bucket is a full bucket, so the state is self-cleaning.
  • Leaky bucket = a queue drained at a fixed rate. No bursts, perfectly smooth output, adds latency. For shaping traffic into something fragile — not for a synchronous public API.
  • N servers with local counters = N × the limit. Share the counter → Redis → now you have a read-then-write race (both read 99, both allow, you served 101).
  • Fix the race with a Lua script. Redis is single-threaded, so a script is a critical section for free: the race is structurally impossible, not defended against. One round trip. INCR alone only suffices for fixed window; WATCH/MULTI degenerates under exactly the contention you're built to handle.
  • Then get Redis off the hot path: local buckets + ~100 ms sync. 1000× less Redis traffic, nanosecond checks, and it kills the hot key too. The price is bounded overshoot — right for abuse prevention, wrong for billing.
  • Gateway is where it lives (+ in-service for business-context limits). Client-side is a courtesy, never enforcement.
  • Redis down → fail open, with a local fallback bucket, a circuit breaker on a ~10 ms timeout, and a page. A limiter whose outage becomes a total outage is a self-own. Fail closed when the limit is load-bearing (third-party quota, metered billing, fragile downstream, login attempts) — and make it a per-rule flag.
  • Hot keys: one hammering client = one Redis shard on fire; a global limit has no key to shard on at all. Local buckets fix the first; splitting the counter into N sub-counters fixes the second, at the cost of utilisation.
  • A rate limiter is not a DDoS defence. You cannot rate-limit your way out of a saturated uplink — that's the CDN/edge's job. Knowing where your system's responsibility ends is a senior signal.
