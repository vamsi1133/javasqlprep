23 — Design a Real-Time Collaborative UI

Two people typing in the same document at the same time. The whole round is: whose edit wins, and how does nobody notice the fight.

Why this matters in the interview

This is the frontend round with the highest ceiling and the deepest trap door. "Design Google Docs" (or Figma, or Notion, or a shared whiteboard) sounds like it's about editing text, and a weak candidate spends forty minutes on a toolbar and a contenteditable div. The strong candidate knows within the first minute that the entire interview is about one hard problem wearing a friendly costume: two clients mutate the same state concurrently, over an unreliable network, and must converge to the same result without a server referee blocking every keystroke. Everything else — the cursors, the avatars, the toolbar — is set dressing around that problem. Name it out loud in the first ninety seconds and you've already separated yourself from the field.

Here's the good news and the honest calibration in one breath: the deep theory here — Operational Transformation and CRDTs — is awareness-level for a frontend round, and you should say so. Almost no interviewer will ask a React engineer to implement OT's transform function or a CRDT's merge on a whiteboard; if one does, they're running a specialist round and you'd know from the job description. What they will grade is whether you know the three families of conflict resolution exist, can explain the intuition and the trade-off of each, know which real products use which, and can reason about the parts a frontend engineer actually owns: the transport, the optimistic local apply, presence and cursors, and — the thing that quietly separates senior from mid — what happens when the network drops mid-edit and comes back. This page walks the whole thing through RADIO (tutorial 19), stays honest about what's awareness-level versus what you must own, and hands you the sentences to say out loud.

The opening frame, memorised: "Before I draw anything — the core of this problem is concurrent editing: multiple people mutating one shared document at once, over a lossy network, with no time for a server to lock and referee each keystroke. So the spine of my design is (1) a real-time transport, (2) an optimistic local apply so my own typing is never laggy, and (3) a conflict-resolution strategy so everyone converges to the same document. I'll design the client in depth and treat the sync server as a contract. The convergence algorithm — last-write-wins vs OT vs CRDT — I'll cover at the level of trade-offs and which one I'd reach for, and flag that actually implementing OT or a CRDT is its own specialism; stop me if you want to go deeper there than awareness." That sentence does the interviewer's scoping for them and tells them you know exactly where the depth lives.

1. The prompt

The interviewer says: "Design a collaborative document editor — like Google Docs. Multiple users can edit the same document at the same time and see each other's changes live." That's it. As always the prompt is deliberately underspecified, and as always the first move is to not start drawing. But before you clarify, do the reframe out loud, because it's the highest-value sentence in the round and it steers everything the interviewer asks next.

The reframe: this is not a text-editor problem, it's a distributed-state-convergence problem that happens to render as text. Strip the document away and the shape is: N clients each hold a replica of one logical object; each applies local mutations instantly (or the app feels broken); those mutations propagate to the others; and every replica must end up identical no matter what order the mutations arrived in. That is the same shape as a multiplayer game's world state, a shared spreadsheet, a Figma canvas, or a Trello board. Recognising the archetype tells the interviewer you've seen the pattern, not just the product.

Scope check to do immediately, because it changes the difficulty by an order of magnitude: plain text vs rich text, and how many concurrent editors per document. A shared to-do list where fields rarely collide is a different (much easier) build from a rich-text editor where two people bold overlapping ranges of the same sentence. And "3 collaborators" (a doc) is a different scale from "500 cursors on one Figma file." Pin both before you commit to a conflict strategy — the answer to "which algorithm" depends entirely on them.

R — Requirements

Same first move as every RADIO round: pin the surface before you build it. Split functional from non-functional, scope ruthlessly, and say what you're cutting. For a collaborative editor the non-functionals aren't optional colour — they are the design, because latency tolerance and offline expectation are what decide your conflict strategy.

Functional — the user-visible behaviours

  • Multiple users edit one document simultaneously and see each other's edits appear live (target: sub-200 ms).
  • My own typing is never laggy — local edits apply instantly, before any server round trip.
  • I can see who else is here (presence: avatars) and where they are (remote cursors and selections).
  • The document survives reload and everyone who opens it sees the same content (persistence + convergence).
  • Edits made while briefly offline aren't lost — they sync when the connection returns.

What I'd cut and say so: rich-text formatting complexity (I'll design for plain text or a simple character model and note where rich text makes conflict resolution harder), comments/suggestions mode, permissions and sharing, version history UI, and the document list. "I'll go deep on the concurrent-editing core for a plain-ish text model and treat formatting, comments, and access control as extensions I can speak to if you want." Naming the cut is the senior move — it shows you know the core is the whole interview.

Non-functional — this list is the round

Then state your assumptions and move: "I'll assume a Google-Docs-style rich-ish text doc, a handful of concurrent editors (say up to ~20, not 500), sub-200 ms perceived sync, my own edits always instant, graceful survival of short disconnects with a stretch goal of true offline, and byte-identical convergence as the correctness bar. Good?" That single paragraph is the whole non-functional spec, and it already telegraphs that you'll need optimistic apply plus a real conflict-resolution algorithm.

Say the number that drives everything, the way you'd state the read:write ratio in a backend round: "The parameter that actually picks my algorithm is conflict granularity crossed with concurrency. If edits are coarse — separate cells, separate list items — collisions are rare and I can get away with field-level last-write-wins. If it's free-flowing text where two people insert into the same paragraph, positions shift under each other and I need OT or a CRDT. So before I pick, I'm pinning: how fine-grained is a conflict, and how many people hit the same spot at once." That's the senior instinct — derive the algorithm from the requirement instead of name-dropping CRDT because it's trendy.

A — Architecture & components

The architecture insight that carries this whole design: separate the editor UI from the document state from the sync engine. Most candidates fuse them and drown. The React tree that renders text is one concern; the in-memory model of the document is a second; the thing that ships local changes out and merges remote changes in is a third — and that third box, the sync engine, is where the interview actually lives. Drawing this three-layer split first is the single strongest structural signal you can give.

The component / module tree

  <DocumentPage>                     ← route container; owns the session lifecycle
    ├─ <PresenceBar>                  ← avatars of who's here (from presence channel)
    ├─ <Toolbar>                      ← bold/italic/etc → dispatches ops, mostly local UI state
    ├─ <Editor>                       ← the contenteditable / textarea surface
    │    ├─ <RemoteCursorsLayer>      ← overlay: other people's carets + selections
    │    └─ <RemoteSelectionsLayer>   ← overlay: highlighted ranges, colour-per-user
    └─ <ConnectionStatus>             ← "Saved" / "Reconnecting…" / "Offline" indicator

  ── not components, but the layers under them (the real design) ──
  DocumentModel     ← the local source of truth (text + a conflict-friendly structure)
  SyncEngine        ← applies local ops optimistically, sends them, receives + merges remote ops
  Transport         ← the WebSocket wrapper: connect, heartbeat, reconnect, buffer

The reason to split Editor from DocumentModel from SyncEngine: the editor should be a thin, dumb view over the model — it renders the current document and emits intents ("insert 'x' at position 12"), nothing more. It does not know the network exists. The SyncEngine turns those intents into operations, applies them to the model immediately (that's the no-lag requirement), and independently reconciles with the server. That boundary is what lets you swap the whole conflict strategy — LWW, OT, a CRDT library — without touching a line of the editor.

The core loop — one local edit and one remote edit, end to end

  LOCAL edit (I type):
    keystroke ─▶ Editor emits intent ─▶ SyncEngine makes an op
       │
       ├─▶ apply to DocumentModel NOW (optimistic) ─▶ UI updates instantly
       └─▶ Transport.send(op) ─▶ server ─▶ broadcast to peers

  REMOTE edit (you type):
    server ─▶ Transport.onMessage(op) ─▶ SyncEngine
       │
       ├─▶ transform/merge against my un-acked local ops (the hard part)
       └─▶ apply to DocumentModel ─▶ UI updates + remote cursor moves

Two things to say out loud while you draw this. First, the local apply and the send are independent branches — my keystroke hits the screen without waiting for the network, which is the entire reason the app feels native. Second, the inbound path is where conflicts get resolved: a remote op can't just be applied blindly, because I may have local edits the server hasn't seen yet, so positions have shifted. Resolving that is the deep dive (conflict resolution).

The classic wrong turn is making the editor talk to the network directly — onKeyDown fires a fetch / socket send and also mutates the DOM, with no model in between. It works in the demo and collapses the moment two edits race, because there's no place to hold un-acked ops and nothing to transform against. The model-in-the-middle isn't ceremony; it's the only place conflict resolution can happen. If you skip it you've built a design that literally cannot converge.
The sentence that shows you've thought about this as state replication, not UI: "I'm treating the document as a replicated data structure with a local replica I mutate optimistically, not as server data I fetch. That reframes the whole problem: my job on the client is to keep a local replica, apply my ops instantly, and run a merge function on inbound ops so all replicas converge. The editor is just a projection of that replica — which means I can change my convergence algorithm without touching the UI." That's the line that tells the interviewer you understand why this round is hard.

D — Data model

Three data models matter here, and keeping them distinct is itself a scoring move because candidates routinely blur them: (1) the document — the shared, durable, must-converge state; (2) the operation — the unit of change that travels over the wire; and (3) presence — the transient who's-here-and-where channel that must stay out of the document.

1. The document — an op log, not a blob

Do not model the document as "a big string the server owns and clients PUT." You cannot merge two concurrent PUTs of a string without losing one. Instead the durable truth is a log of operations; the current text is a projection of replaying that log (identical to the double-entry ledger idea from tutorial 13 — the balance is derived from immutable entries, never stored and mutated). The server keeps the log, periodically snapshots it so new joiners don't replay from the beginning, and hands a new client snapshot + ops-since-snapshot. 

// An operation — the atomic unit of change. Each is intention-preserving:
// it says WHAT the user did, so it can be transformed against concurrent ops.
type Op =
  | { kind: "insert"; pos: number; text: string }
  | { kind: "delete"; pos: number; len: number };

interface Change {
  op: Op;
  docId: string;
  actor: string;        // which client authored it (client/site id)
  baseVersion: number;   // the server version this op was created against
}

That baseVersion is load-bearing and worth calling out: it tells the server "I made this edit believing the document was at version N." If the server has since advanced to N+3 because other people edited, the server (or client) must transform my op against the 3 ops I hadn't seen before applying it. That single field is what makes ordered convergence possible — it's the frontend cousin of an optimistic-concurrency version column in Postgres.

2. The client replica

On the client you hold the current text (as a projection) plus the machinery to reconcile: the last server version you've acknowledged, and the buffer of local ops you've applied optimistically but the server hasn't confirmed yet. 

interface ClientDoc {
  text: string;              // current projection shown in the editor
  ackedVersion: number;      // highest server version I've merged in
  pending: Change[];        // applied locally, NOT yet acked — the reconnect/transform buffer
}

pending is the most important field in the whole design. It's why my typing is instant (I apply then queue), it's what inbound remote ops get transformed against, and it's exactly what I replay after a reconnect. Empty it too eagerly and you lose edits; never empty it and you double-apply. Managing it correctly is the correctness of the client.

3. Presence — deliberately separate, deliberately lossy

Cursor positions, selections, and "who's online" are a different kind of data and must ride a different channel. They are high-frequency (a cursor moves on every keystroke and mouse move), disposable (a cursor position from 400 ms ago is worthless — you only ever want the latest), and must not be persisted — you never want yesterday's cursor positions in the document history.

Document opsPresence (cursors, selections, online)
DurabilityPersisted forever (the op log)Ephemeral — gone when you disconnect
Delivery guaranteeMust not lose a single op (reliable, ordered)Lossy is fine — only the latest matters
FrequencyPer meaningful editVery high — throttle hard (e.g. 20–50 ms)
Conflict handlingTransform / merge, must convergeLast-write-wins per user — no merge needed
On reconnectReplay the buffer, lose nothingJust re-broadcast current state; drop the gap
Routing cursor movements through the durable document op log is a subtle but serious mistake — you'll persist thousands of cursor ops, bloat the log, slow snapshot replay, and pay reliable-delivery cost for data that's worthless a frame later. Presence gets its own channel with last-write-wins-per-user semantics and no persistence. Interviewers who've built this notice immediately whether you separate the two; it's a fast senior tell.
The framing: "I model the document as an append-only op log, and the visible text is a projection of that log — same discipline as a ledger, where you never store a mutable balance. That gives me history, undo, and a well-defined merge for free. Presence — cursors and selections — rides a completely separate, ephemeral, lossy channel with last-write-wins per user, because it's high-frequency and worthless the instant it's stale. Conflating those two is the most common structural mistake here."

I — Interface & API

Two interfaces, as always in RADIO — but here the network interface is unusually important, because the transport choice is a real interview question and the message protocol is the sync contract. I'll spend most of this step on the transport (WebSockets vs SSE from the client's point of view) and the message protocol, and keep the component-props interface brief.

Transport: WebSockets vs SSE — the decision you must be able to defend

The question hiding inside "how do the edits get there in real time?" is which push transport you use. There are three real options — WebSockets, Server-Sent Events (SSE), and long polling — and for a collaborative editor the answer is WebSockets, but the whole point is being able to say why, and to know when you'd pick SSE instead. (The backend mechanics of these live in tutorial 02 — Networking & Delivery — and tutorial 11's chat system; here we care about the client's side of the choice.)

The deciding axis is directionality. A collaborative editor is intensely bidirectional: the client pushes ops up constantly (every edit) and the server pushes ops down constantly (everyone else's edits). That's the exact shape WebSockets exist for — one long-lived, full-duplex TCP connection where both ends send whenever they like. SSE is one-way: server-to-client only. You'd still need a second channel (plain fetch/POST) for the client-to-server ops, so with SSE you're maintaining two half-connections to simulate what one WebSocket gives you natively.

So the one-liner: "I'll use WebSockets, because collaborative editing is bidirectional and high-frequency — both sides push constantly, and a WebSocket is one full-duplex connection built for exactly that. SSE is a great fit when push is one-way — notifications, a live feed, streaming LLM tokens — but here I'd have to bolt a separate POST channel on for the client's own edits, which is two things to keep in sync instead of one. Long polling is my fallback only if corporate proxies block WebSockets." Note the SSE positive: it's genuinely better when you only push down, and its free auto-reconnect-with-Last-Event-ID is a real feature — say that, so you don't sound like you're just defaulting to WebSockets.

SSE has one underrated superpower this design has to reimplement by hand on WebSockets: resumable streams. EventSource automatically reconnects and sends a Last-Event-ID header so the server can replay everything you missed. On a WebSocket you own that: you track the last server version you saw and, on reconnect, tell the server "catch me up from version N." You're rebuilding SSE's resume feature — which is a good argument for a thin protocol on top of the socket (below), and a good thing to acknowledge out loud.

The message protocol — the real sync contract

Define the handful of message types over the socket. This is the API for this system, more than any REST endpoint. Version everything so reconnect and catch-up work. 

// ── client → server ──
{ t: "op",       change: Change }                  // I made an edit (carries baseVersion)
{ t: "cursor",   pos: number, sel: [number,number] }   // presence — throttled, fire-and-forget
{ t: "resume",   fromVersion: number }            // after reconnect: "catch me up from N"

// ── server → client ──
{ t: "ack",      version: number }               // your op is committed at this server version
{ t: "op",       change: Change, version: number } // someone else's edit (already server-ordered)
{ t: "presence", actor: string, pos, sel, color }  // a peer's cursor moved / they joined
{ t: "sync",     snapshot: string, version: number } // initial load or post-reconnect resync

Two REST endpoints still exist around the edges — GET /docs/:id to bootstrap the snapshot + version before the socket opens, and auth to get a token for the wss:// upgrade — but the live editing all flows over the socket. The senior detail is the server assigns the authoritative version/order: clients propose ops with a baseVersion, the server is the single sequencer that decides the final order and stamps a monotonic version, and the ack is how a client learns its optimistic op is now canonical. That central sequencer is what makes OT tractable (below).

The component interface — brief, because it's the easy half

The editor stays presentational over the model: it takes the current text and remote cursors as props and emits intents. It knows nothing about sockets or versions. 

interface EditorProps {
  text: string;
  remoteCursors: RemoteCursor[];              // { actor, pos, sel, color, name }
  onIntent: (i: EditIntent) => void;      // "insert x at 12" — SyncEngine turns this into an Op
  onCursor: (pos: number, sel: [number,number]) => void;
}
Tie the transport back to the requirement: "The transport isn't a detail — it's dictated by the access pattern. Bidirectional and high-frequency means WebSockets; if this were a read-only live view — say, watching a doc without editing — I'd downgrade to SSE and get simpler infra and free reconnect. I'd also put a thin versioned protocol on top of the socket so I can resume from a known version after a drop, because that resume behaviour is the one thing SSE gives you for free and a raw WebSocket doesn't."

O · Deep dive: conflict resolution — LWW vs OT vs CRDT

This is where the interview really happens. Every other section supports this one. And here is the single most important calibration on the page, so say it out loud early: for a frontend round this is awareness-level. You are expected to know the three families exist, explain the intuition and trade-off of each, and say which you'd reach for and why. You are almost never expected to derive OT's transform functions or hand-write a CRDT merge on the whiteboard — that's a specialist / library-author skill. State that boundary explicitly; it reads as confidence, not evasion.

Why there's a conflict at all — the one example to draw

Draw this, because it makes the whole problem concrete in ten seconds. The document is "HAT". Two users act at the same instant, each against version "HAT": 

  Start:  "HAT"
  User A: insert "C" at pos 0   → intends "CHAT"
  User B: delete 1 char at pos 2 (the "T") → intends "HA"

  Naive apply of BOTH ops as-written, in the order they arrive:
    apply A to "HAT": insert C@0  → "CHAT"
    apply B to "CHAT": delete@2   → "CHT"   ✗  (B meant to remove the T, deleted the A)

  The bug: B's op said "pos 2", but A inserted a char BEFORE pos 2,
  so every position after the insert shifted right by one. B's index is now stale.

That positional drift — indexes computed against one version of the document being applied to a different version — is the entire problem. The three families are three different answers to "what do we do about it."

Family 1 — Last-write-wins (LWW): the honest simple option

Intuition: attach a timestamp (or a version) to each write; when two writes conflict, the later one wins and the earlier is discarded. No merging — one edit is simply thrown away.

Where it's right: coarse-grained, rarely-colliding state. A document's title. A cell in a spreadsheet where two people rarely type in the same cell. A shape's colour on a canvas. A user-profile field. If your "document" is really a bag of independent fields and collisions on the same field are rare, LWW is not a cop-out — it's the correct, cheap answer, and reaching for OT here would be over-engineering.

Where it's catastrophic: free text. Applying LWW to a shared paragraph means the instant two people type near each other, one person's whole edit vanishes — visibly, mid-sentence. It also needs synchronized clocks to even define "later," which is its own distributed-systems trap (a version counter or a logical clock is safer than wall-clock time). LWW throws away data by design; that's fine for a colour, fatal for prose.

LWW is what most "collaborative" apps that aren't editors actually use, and admitting that is a senior move. Trello moving a card, a form where two admins edit different fields, a settings panel — field-level last-write-wins with a version check is plenty. Don't reach past it unless the state is fine-grained text. Match the algorithm to the collision rate, not to the fashion.

Family 2 — Operational Transformation (OT): rewrite the incoming op

Intuition: keep both edits, but before you apply an op that was created against an older version, transform it so its positions account for the ops it didn't know about. In the HAT example, OT takes B's "delete at pos 2", notices A inserted one character before pos 2, and rewrites B's op to "delete at pos 3." Now both apply and everyone lands on "CHA". The magic is a transform(opX, opY) function that adjusts one op to account for a concurrent one. 

  transform(B, A):  B was "delete@2", A was "insert 1 char @0"
    A inserted before B's position → shift B right by A's length
    B' = "delete@3"
  Result on both replicas: "CHA"   ✓ converged, nobody's edit lost

The trade-off: OT is simple to understand and brutal to implement correctly. You need a transform function for every pair of operation types (insert vs insert, insert vs delete, delete vs delete…), it must satisfy hairy mathematical properties (transform in either order must converge — "TP1/TP2"), and rich text multiplies the op types. It also essentially requires a central server to sequence ops into one authoritative order, which is fine for Google Docs (which uses OT) but rules out true peer-to-peer.

Who uses it: Google Docs, Etherpad, older Office online. It's the battle-tested choice for server-mediated text editing. Say clearly it's awareness-level: "OT is what Google Docs uses — the idea is you transform an incoming op's positions against the concurrent ops it didn't see. Implementing the transform functions correctly is famously hard, which is why in practice you'd use a proven library rather than roll it, and why I'd only hand-derive it if this were a specialist round."

Family 3 — CRDTs: make conflicts impossible by construction

Intuition: design the data structure so that concurrent operations commute — applying them in any order yields the same result — which means you never need a transform and never need a central sequencer. For text, the trick is to stop using integer positions (which shift) and instead give every character a unique, stable, densely-orderable identifier that never changes. "Insert after character-id X" is unambiguous no matter what else happened, because X's identity doesn't move even as its integer index does. Deletes become tombstones (mark-not-remove) so a delete and a concurrent insert can't disagree.

The trade-off: conflicts become impossible by construction and it works peer-to-peer with no central server and offline — the properties OT can't give you. The costs: metadata overhead (every character carries an id; tombstones accumulate, though modern CRDTs like Yjs compress this aggressively), and the mental model is less intuitive than OT. You also inherit "true" merges that can be semantically surprising (two people typing in the same spot interleave rather than one winning).

Who uses it: Figma, Apple Notes, Linear, and anything local-first; libraries are Yjs and Automerge. CRDTs are the modern default when you want offline and/or peer-to-peer. Awareness-level line: "CRDTs give every character a stable id so inserts commute — order-independent by construction, so they converge without a server. That's why they're the local-first and P2P choice. I'd reach for Yjs or Automerge rather than build one; writing a correct CRDT is a research-grade exercise, not a whiteboard one."

The answer that wins this whole section: "For a server-mediated doc editor I'd reach for a CRDT via a library like Yjs — it gives me convergence, offline support, and undo without me hand-rolling transform functions, and it's where the industry has landed for new builds. Historically Google Docs solved this with OT, which also works but is notoriously hard to implement correctly and needs a central sequencer. And if the state were coarse-grained — cells, shapes, separate fields — I wouldn't use either; field-level last-write-wins is the right, cheap answer. I want to be upfront that implementing OT or a CRDT from scratch is a specialist exercise; at design level I'm choosing the family and the library, and I'd only go deeper if you want me to." That hits the trade-off, the real-world anchors, the "match the tool to the collision granularity" judgment, and the honest scope of what you'd implement.

O · Deep dive: presence & remote cursors

Presence — the coloured avatars up top and the little name-tagged carets floating in the text — is the highest perceived-value-per-effort feature in the whole product. It's what makes collaboration feel alive. And because it rides its own ephemeral channel (from the D step), it's mercifully independent of the hard convergence problem. Two sub-features: who's here (roster) and where they are (cursors and selections).

Cursors: the position-mapping problem

A remote cursor is "user B's caret is at position 42." But there's a subtle trap that connects straight back to conflict resolution: position 42 in whose version of the document? If I have local un-acked edits that B hasn't seen, B's "42" points at the wrong character on my screen. So a remote cursor position must be transformed through the same op machinery as edits — when I apply a local insert before a remote cursor, I nudge that cursor right too, exactly as I'd transform an op. (CRDT libraries hand you this for free via "relative positions" / "sticky indexes" that survive edits — another reason to lean on a library.)

Rendering cursors without wrecking the editor

You do not inject other people's carets into the editable content — that would corrupt the document and the user's own selection. You render them in an absolutely-positioned overlay layer on top of the text: compute the pixel rect for a character offset (browsers give you this via Range/getClientRects), and paint a 2px coloured bar plus a name label there. Selections are the same idea over a range of rects. This is why <RemoteCursorsLayer> was a sibling overlay in the component tree, not a child of the text. 

// Presence state is a simple last-write-wins map, keyed by user.
// No merge, no history — the latest position per actor is all that matters.
type Presence = Map<string /*actor*/, {
  name: string;
  color: string;          // stable per user for the session → recognisable caret
  pos: number;
  sel: [number, number];
  lastSeen: number;       // timestamp → drive "idle" fade + disconnect cleanup
}>;

Three things that make presence feel good — and cheap

  • Throttle hard. A cursor can move on every keystroke and mousemove — dozens per second. Send at most every ~30–50 ms (throttle, not debounce: you want steady updates during motion, not one at the end). It's lossy data, so dropping intermediate positions is not just acceptable, it's correct.
  • Stable colour per user. Assign each actor a colour on join and keep it for the session so a caret is recognisably "Ada's." Hash the user id to a palette so it's consistent without coordination.
  • Clean up the dead. When a socket closes, broadcast a "left" so peers remove that avatar and caret immediately; back it with a heartbeat timeout so a silently-dropped client's ghost cursor fades after a few seconds instead of haunting the document. lastSeen drives both the idle fade and the sweep.
Forgetting to transform remote cursor positions against your local un-acked edits is the bug that makes cursors "jitter" or sit one character off while you type — it looks janky and it's the exact same positional-drift problem as the HAT example, just applied to a caret instead of an op. If you're using a CRDT library, use its relative-position API for cursors; if you're hand-rolling, run cursors through the same transform path as ops. Don't treat the cursor integer as absolute.
"Presence is a separate ephemeral channel with last-write-wins per user, throttled to ~30ms and never persisted. The one gotcha is that a remote cursor position has to be transformed against my own pending edits — otherwise it drifts a character off as I type — so I map it through the same machinery as ops, or use the CRDT's relative-position API. And I render carets in an overlay layer, never inside the editable text, so I don't corrupt the doc or the user's selection." Naming the cursor-transform gotcha unprompted is a strong signal you've actually built this.

O · Deep dive: offline edits & reconnect

This is the deep dive that quietly separates senior from mid, because it's the part nobody demos and everybody hits in production. The network will drop — a tunnel, a lift, a flaky café WiFi — and the question is what happens to the edits made in the gap. The answer flows directly out of the pending buffer you already designed (D step): keep applying locally, keep buffering, and replay on reconnect. A collaborative editor that freezes or loses your paragraph the moment WiFi blips is a broken product.

The reconnect protocol — buffer, resume, rebase

  WHILE OFFLINE:
    • editor keeps working — ops apply to the local replica optimistically
    • each op appended to `pending` (and persisted to IndexedDB for true offline)
    • the "Offline / Reconnecting…" indicator shows; presence for me goes stale to peers

  ON RECONNECT:
    1. open the socket, send  { t:"resume", fromVersion: ackedVersion }
    2. server replies with every op I missed (versions ackedVersion+1 … latest)
    3. TRANSFORM my `pending` ops against those missed ops   ← the rebase
    4. send my rebased pending ops; server acks them at new versions
    5. `pending` drains to empty → indicator flips back to "Saved"

Step 3 is the crux and worth naming precisely: it's a rebase, exactly like git rebase. My local edits were built on version N; the world has moved to N+k; before my edits can land I replay them on top of the newer base, transforming their positions against everything that happened while I was gone. If you're on a CRDT this is nearly free — ops commute, so you just exchange the missing updates and both sides converge with no explicit transform. If you're on OT, you run each pending op through the transform against each missed op. Saying the word "rebase" out loud lands the concept instantly with anyone who uses git.

The git analogy is worth making explicit because it's exact and your interviewer lives in git: your pending buffer is your unpushed local commits; the server's op log is origin/main; a reconnect is git pull --rebase — fetch what you missed, replay your local work on top, resolve the position shifts. And a CRDT is the world where the rebase can never conflict because the merge is defined to always succeed. That framing turns an exotic-sounding problem into one you solve every day.

Levels of offline — scope it honestly

LevelWhat it meansWhat it costs you
Blip toleranceSurvive a few seconds' disconnect without losing editsCheap — just the in-memory pending buffer + resume protocol. Almost free once you have the buffer.
Reload survivalRefresh the tab mid-edit, keep unsynced workPersist pending (and a snapshot) to IndexedDB; replay on load.
True offlineEdit on a plane for an hour, sync on landingExpensive with OT (huge rebase, server must retain history); this is where CRDTs earn their place — arbitrary offline divergence merges cleanly.

Say which level you're targeting rather than hand-waving "it works offline." For most doc editors, blip tolerance plus reload survival is the pragmatic bar and it's cheap. True multi-hour offline is a product decision that pulls you firmly toward a CRDT — which is one more reason CRDTs win new builds, and a clean way to connect this deep dive back to the conflict-resolution one.

Reconnect hygiene — the details that show experience

  • Exponential backoff with jitter on reconnect attempts (straight from tutorial 06) — don't hammer a struggling server, and don't let a thundering herd of reconnecting clients synchronise into a spike.
  • Idempotent op application. After a messy reconnect you might receive an op you already applied, or the server might re-ack. Ops carry ids and versions so the client can dedupe — apply-once semantics, the same idempotency discipline as tutorial 13's payments.
  • Never block the user's typing on connection state. Offline is a background status, not a modal. The editor stays live the entire time; only the little indicator changes.
"Reconnect is a rebase. My local un-acked edits are unpushed commits; on reconnect I fetch the ops I missed, replay mine on top transforming positions, and drain my buffer — git pull --rebase, essentially. For blip tolerance that's just the in-memory buffer; for reload survival I persist pending ops to IndexedDB; for true multi-hour offline I'd use a CRDT so the merge can't fail. And I back reconnect with exponential backoff plus jitter and idempotent, versioned op application so a retried or duplicated op applies exactly once." That paragraph covers buffering, replay, rebase, the offline spectrum, and resilience hygiene in one breath.

Local-first thinking

Everything above quietly assumes one philosophy, and naming it explicitly is a senior flourish that ties the whole answer together: local-first. The idea is that the local replica is the primary copy and the network is a synchronisation detail, not a gate. The user reads and writes their own local data instantly, always; sync happens in the background and can fail, retry, or wait without the user ever feeling it. It's the inversion of the traditional web app, where the server is the source of truth and the client is a thin, network-gated view.

You've actually been designing local-first the whole page without the label: optimistic local apply (writes hit the local replica first), the pending buffer (local work that outlives the connection), reconnect-as-rebase (sync as a background reconcile), and CRDTs (the data structure that makes offline-primary correct). Local-first is just the name for that stance, and it maps cleanly onto things you already know from tutorial 19: server state as a cache you revalidate, optimistic updates, offline resilience. Here it's turned up to the point where the client can run indefinitely without the server.

The closing frame for the whole design: "Underneath, this is a local-first architecture — the local replica is the source of truth and the network is a background sync detail, which is exactly why my own edits are never laggy and why offline is a graceful degradation rather than a failure. That's the same instinct as optimistic UI and server-state-as-a-cache from the feed round, pushed to its limit: the client can operate entirely on its own replica, and the server's job is convergence, not gatekeeping."

Interview Q&A

1. What's the single most important thing to say in the first two minutes?

Reframe the problem: this isn't a text-editor question, it's a distributed-state-convergence question that happens to render as text. N clients each hold a replica of one document, apply local edits instantly (or the app feels broken), propagate them, and must all converge to the identical result regardless of arrival order. Everything else — toolbar, cursors, avatars — is set dressing around that.

Then scope: plain vs rich text, and how many concurrent editors per doc, because those two answers decide your conflict-resolution family. Naming the core and the scoping questions up front is what separates you from the candidate who starts drawing a toolbar.

2. WebSockets or SSE for this — and why?

WebSockets, because collaborative editing is bidirectional and high-frequency: the client pushes its own edits up constantly and the server pushes everyone else's edits down constantly. A WebSocket is one full-duplex connection built for exactly that. SSE is server-to-client only, so I'd need a separate POST channel for the client's own edits — two half-connections doing one socket's job.

But I'd give SSE its due: it's the better choice when push is one-way — notifications, a live dashboard, streaming LLM tokens, or a read-only viewer of the doc — and its EventSource gives you auto-reconnect with Last-Event-ID replay for free, which on a WebSocket I have to build myself. Long polling is a fallback only if proxies block WS.

3. Explain LWW, OT, and CRDT — and which would you pick?

LWW: timestamp each write, later wins, the earlier is discarded — no merge. Right for coarse, rarely-colliding state (a title, a cell, a shape's colour); catastrophic for free text because it silently drops someone's edit. OT: keep both edits but transform an incoming op's positions against the concurrent ops it didn't see (in "HAT", an insert at 0 shifts a later delete's index by one). Preserves everything, needs a central sequencer, famously hard to implement correctly — Google Docs uses it. CRDT: design the structure so ops commute — every character gets a stable id so "insert after X" is unambiguous regardless of order — so it converges with no central server and works offline/P2P; Figma and Yjs/Automerge apps use it, at the cost of metadata overhead.

I'd pick a CRDT via a library like Yjs for a new build (convergence + offline + undo without hand-rolling transforms), note OT as the proven server-mediated alternative, and use plain field-level LWW if the state is coarse. And I'd flag that actually implementing OT or a CRDT is specialist work — at design level I'm choosing the family and the library.

4. Isn't picking a library a cop-out? Shouldn't I implement the algorithm?

For a frontend system design round, choosing the right family and a proven library is the senior answer — it shows judgment about build-vs-buy and honesty about risk. Correct OT transform functions and correct CRDT merges are research-grade; teams that rolled their own have shipped convergence bugs that corrupt documents. Reaching for Yjs/Automerge is what an experienced engineer actually does.

The one exception is an explicit specialist round (the JD says "distributed systems" or "collaborative infrastructure"), where they may want you to derive a transform on the board. That's rare for a React role, and you'd know it from the posting. In the normal round, over-implementing here means you've spent your best minutes proving something they didn't ask for.

5. Why can't I just have clients PUT the whole document string?

Because you can't merge two concurrent PUTs of a blob without losing one — whoever writes last clobbers the other person's edit entirely (that's LWW on the worst possible granularity). The document has to be modelled as an append-only log of operations, and the visible text is a projection of replaying that log. It's the same discipline as a double-entry ledger: you never store and mutate a balance, you derive it from immutable entries.

The op log also buys you history, undo, and a well-defined merge point. The server periodically snapshots so new joiners don't replay from the beginning, and hands them snapshot + ops-since.

6. How do I make my own typing feel instant while still syncing?

Optimistic local apply. A keystroke applies to the local replica immediately and updates the UI, on one branch; on an independent branch the op is sent to the server. You never gate your own typing on a server ack — that would add a round trip to every character and feel broken. The applied-but-not-yet-acked ops sit in a pending buffer.

That buffer does triple duty: it makes typing instant, it's what inbound remote ops get transformed against (since my local edits have shifted positions the server hasn't seen), and it's exactly what I replay after a reconnect. Managing pending correctly — never dropping, never double-applying — is the core of client correctness.

7. Someone edits offline for a while, then reconnects. What happens?

It's a rebase — literally git pull --rebase. While offline the editor keeps working: ops apply to the local replica and pile up in the pending buffer (persisted to IndexedDB for true offline). On reconnect I send resume(fromVersion), the server streams back the ops I missed, I transform my pending ops on top of those (rebasing their positions onto the newer base), send them, get acks, and the buffer drains.

On a CRDT this is nearly free because ops commute — you just exchange missing updates and both converge. On OT you run each pending op through the transform against each missed op. Multi-hour offline with big divergence is where CRDTs decisively win; for a few-second blip the in-memory buffer alone is enough.

8. How do remote cursors work, and what's the gotcha?

Presence rides a separate ephemeral channel — last-write-wins per user, throttled to ~30ms, never persisted — because a cursor position is high-frequency and worthless a frame later. You render carets and selections in an absolutely-positioned overlay on top of the text (compute pixel rects from character offsets via Range), never inside the editable content, so you don't corrupt the document.

The gotcha: a remote cursor's position is an index into their version of the doc, so if I have local un-acked edits it points at the wrong character on my screen and jitters as I type. You must transform remote cursor positions through the same op machinery as edits — or use a CRDT's relative-position / sticky-index API, which handles it for you.

9. Why keep presence separate from the document?

Because they're opposite kinds of data. Document ops must be persisted forever, delivered reliably and in order, and merged to converge. Presence — cursors, selections, who's online — is ephemeral, high-frequency, lossy-tolerant (only the latest matters), and must not be persisted. Routing cursor movements through the durable op log would bloat it with thousands of worthless entries, slow snapshot replay, and pay reliable-delivery cost for data that's stale instantly.

So presence gets its own channel with LWW-per-user semantics and no durability, while the document gets the reliable, ordered, converging op log. Separating them cleanly is a fast senior tell.

10. How does collaborative undo work — and why is it hard?

It's harder than single-player undo because "undo my last change" must undo your op specifically, not whatever happened most recently globally. If I type, you type, and I hit undo, I should remove my insert, not yours. So undo is per-user: you keep a per-actor stack of your own ops and generate an inverse op (an undo of an insert is a delete), which then flows through the same transform/merge pipeline as any edit so it lands correctly against everyone else's concurrent work.

This is exactly why an op log (not a blob) matters, and it's another reason to lean on a CRDT library — Yjs ships an UndoManager that scopes undo to an origin. Awareness-level is fine here: know that collaborative undo is per-user and inverse-op-based, and that libraries solve it.

11. How does the backend scale this — connections and fan-out?

This is the frontend round so I'd keep it brief and defer to the chat-system page (tutorial 11), but the shape: WebSocket connections are stateful, so you need many gateway servers each holding a subset of sockets, and all editors of one document must be routed to the same place — typically shard by docId so one document's collaborators share a server (or coordinate through Redis pub/sub). That server sequences the doc's ops, appends to the log, and fans out to the room.

The load is modest per document (a handful of editors), so the scaling challenge is number of documents and total connections, not ops-per-doc — very different from a feed. I'd mention sticky routing by doc, a snapshot strategy so replay stays bounded, and horizontal gateways, then steer back to the client, which is what this round grades.

12. What does the editor surface itself use — contenteditable?

For plain text a <textarea> is fine and I'd say so. For rich text you need contenteditable, but raw contenteditable is notoriously inconsistent across browsers, so in practice you build on an editor framework — ProseMirror, Slate, Lexical, TipTap — that gives you a controlled document model and a stable API over contenteditable. Crucially, those frameworks expose a document model with positions, which is exactly what plugs into the op/CRDT layer (Yjs has bindings for ProseMirror, Lexical, etc.).

The point to make: I keep the editor a thin projection over my document model and let the framework own contenteditable's quirks, so the interesting concurrency logic lives in the sync engine, not tangled into DOM event handling.

What to remember

If you revise one thing before this round, revise the opening reframe and the three-family table. The rest is you being calm about what's awareness-level.

  • It's not a text editor, it's distributed-state convergence. N replicas of one document, each applying local edits instantly, all converging to the identical result regardless of arrival order. Say that first; it reframes everything and tells the interviewer you've seen the archetype.
  • Separate the three layers: editor UI · document model · sync engine. The editor is a thin projection that emits intents; the model is the local replica; the sync engine applies optimistically, sends, and merges inbound. That boundary is what lets you swap the conflict algorithm without touching the UI.
  • WebSockets, because editing is bidirectional and high-frequency. Both sides push constantly over one full-duplex connection. SSE is the right call only when push is one-way (feeds, notifications, LLM streams, read-only viewers) — and its free auto-reconnect-with-Last-Event-ID is a genuine advantage you'd rebuild by hand on a socket. Long polling is a proxy-blocked fallback.
  • LWW vs OT vs CRDT — know the intuition, the trade-off, and who uses each. LWW: later wins, drops data, fine for coarse fields, fatal for text. OT: transform incoming ops' positions against concurrent ones, needs a central sequencer, hard to implement — Google Docs. CRDT: stable per-char ids so ops commute, converges with no server, offline/P2P — Figma, Yjs/Automerge.
  • This is awareness-level and you should say so. You pick the family and the library; you don't hand-derive transforms on the board unless it's a specialist round. Choosing Yjs is the senior answer, not a dodge.
  • Match the algorithm to collision granularity. Coarse, rarely-colliding fields → field-level LWW, and reaching for a CRDT there is over-engineering. Free-flowing shared text → OT or CRDT. Derive the choice from the requirement, don't name-drop the trendy option.
  • Model the doc as an op log; the text is a projection. Never PUT a blob — two concurrent blob writes can't merge. Same discipline as a ledger: derive, don't store-and-mutate. Snapshot periodically so replay stays bounded.
  • Optimistic local apply + a pending buffer is the core of the client. Apply your keystroke now, queue the op, send independently. That buffer makes typing instant, is what remote ops transform against, and is what you replay on reconnect. Never drop it, never double-apply it.
  • Presence is a separate, ephemeral, lossy channel. Cursors and selections ride LWW-per-user, throttled to ~30ms, never persisted, rendered in an overlay — never through the durable op log. And transform remote cursor positions against your local edits or they jitter a character off as you type.

One last calibration, because it's the meta-skill this page teaches: the winning move here is confident scoping of your own depth. You demonstrate that you understand OT and CRDTs deeply enough to choose between them and name their trade-offs, and in the same breath you say implementing them is a library's job — so you spend your minutes on the parts a frontend engineer owns: transport, optimistic apply, presence, and reconnect. That combination — real understanding plus honest scope — is exactly what a 7-YOE candidate sounds like and a 3-YOE one doesn't.
