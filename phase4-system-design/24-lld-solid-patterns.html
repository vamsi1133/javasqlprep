24 — LLD in Java: SOLID & the Patterns That Show Up

The low-level design round: turn a vague prompt into clean Java classes in ten minutes.

Why this matters in the interview

At FAANG-grade loops the design round is usually the distributed one you've spent the last twenty pages on. At enterprise and fintech loops — Optum, JPMC, Goldman, and most of the mid-size product companies you'll actually interview at — there is a second, separate round: low-level design, also called object-oriented design. The prompt is smaller ("design a parking lot", "design an elevator", "design a vending machine", "model a deck of cards and a blackjack game") and the whiteboard is code, not boxes. You're not asked how many servers you need; you're asked to produce classes, interfaces, and method signatures that a teammate could read and extend without cursing your name. It is the round where your Phase 1 Java pays off directly, and it is the round most senior candidates under-prepare because they assume "I write OO code every day" is the same as "I can design it out loud in 45 minutes." It isn't.

Here's the thing to internalize: this round is not a trivia quiz on the Gang of Four. Nobody good will ask you to recite the definition of the visitor pattern. What they're grading is judgment — can you spot the one axis along which this system will change, and put a seam there, without wrapping every class in three layers of abstraction it will never need? SOLID and the design patterns are the vocabulary for that judgment, but only if you treat them as concrete refactoring moves rather than slogans. "Single responsibility" is not a mantra; it's a specific instruction: when a class has two reasons to change, split it. "Strategy pattern" is not a diagram to memorize; it's the answer to a specific smell: a growing if/else on a type code. This page teaches each principle and each pattern as a move — the smell that triggers it, the Java that applies it, and, just as important, when applying it would be over-engineering. At 7 YOE, knowing when not to reach for a pattern is worth more than knowing the pattern.

Part 1 — What the LLD round actually is

The high-level design round asks "how do these services talk?" The low-level design round asks "how do these classes talk?" Same skill — decomposition and drawing seams at the right places — one zoom level down. You'll get a single, deliberately under-specified prompt and roughly 45 minutes to: ask a few clarifying questions, identify the core entities, sketch the class relationships, write the key method signatures and enough method bodies to show the interactions, and then respond to "now add feature X" — where X is chosen specifically to test whether your design has a seam in the right place or whether adding X means rewriting three classes.

What's actually being graded, in rough priority order:

  • Decomposition. Did you find the right objects? Are responsibilities cleanly separated, or did everything end up in a 40-method ParkingLotManager god class?
  • Extensibility at the right axis. When the interviewer says "now support motorcycles" or "now add a new payment method", do you change one line, or edit a giant switch? The follow-up is the test. Your whole design exists to make that follow-up a one-liner.
  • Restraint. Did you use abstraction where the system genuinely varies, or did you gold-plate a parking lot with a dependency-injection framework and six patterns it doesn't need? Over-engineering is graded down. A staff engineer reviewing your code wants to see the simplest thing that has the seams in the right place.
  • Fluency. Can you write clean, idiomatic, compiling Java quickly? Enums, interfaces, Optional, immutability, a sensible equals/hashCode when it matters. This is where Phase 1 shows up.
Open the round by scoping, exactly like the HLD round: "Before I model anything — what's in scope? For a parking lot, do we care about multiple floors, different vehicle sizes, real-time availability, payment and pricing, or is this just park-and-retrieve? I'll design the core cleanly and leave obvious seams for the rest." This does two things a junior candidate skips: it stops you from modeling the wrong thing for twenty minutes, and it signals that you know a design is defined as much by what it excludes as by what it includes. Then name your guiding constraint out loud: "I'll optimize for the change I think is most likely — new vehicle types and new pricing rules — and I'll keep everything else concrete until it needs to flex."
The two rounds reward opposite instincts about the same word — "scale". In HLD, scale means machines and you reason about it constantly. In LLD, "will it scale to a million cars?" is almost never the point; the point is "will it scale to a new requirement without a rewrite?" Read the room. If you start estimating QPS in an LLD round you've misread the question.

Part 2 — SOLID as five refactor moves

SOLID is five principles, and every candidate can list them. The ones who score can tell you, for each, the specific code smell it names and the specific move it forces. That's the reframing: don't memorize the definition, memorize the trigger and the fix. Here's the whole table you should carry into the room, then each one with the Java that makes it concrete.

PrincipleThe smell that triggers itThe move it forces
S — Single Responsibility A class changes for two unrelated reasons; its name has an "and" in it; it mixes business logic with I/O. Split the class along the axis of change. One reason to change per class.
O — Open/Closed Adding a new variant means editing an existing switch/if-else on a type code. Extract an interface; add new behavior as a new implementation, not a new branch.
L — Liskov Substitution A subclass throws on a method it "doesn't support", or narrows what the parent promised. Fix the hierarchy — the subtype must honor the supertype's contract, or it isn't one.
I — Interface Segregation Implementers are forced to stub methods they don't need; one fat interface serves several clients. Split the interface into role-specific ones. Depend on the narrowest thing you use.
D — Dependency Inversion A high-level class news a concrete low-level class directly and is welded to it. Depend on an interface, inject the concrete type. Both sides point at the abstraction.

2.1 S — Single Responsibility: split when there are two reasons to change

The definition ("a class should have one reason to change") is uselessly abstract until you translate "reason to change" into "who files the bug." A class that formats an invoice and emails it changes when the finance team wants a new layout, and also when infra swaps SMTP providers — two teams, two reasons, one class. The move is to split it. 

// SMELL: one class, three reasons to change (calculation, formatting, delivery)
class Invoice {
  BigDecimal total() { /* tax + line-item logic */ }
  String toPdf()   { /* layout, fonts, rendering */ }
  void email(String to) { /* SMTP, retries, MIME */ }
}

// MOVE: one responsibility each. Now the layout team and the infra team
// never touch the same file, and each part is unit-testable in isolation.
class Invoice          { BigDecimal total() { ... } }        // pure calculation
class InvoiceRenderer  { byte[] toPdf(Invoice i) { ... } }  // presentation
class InvoiceMailer    { void send(Invoice i, String to) { ... } } // delivery
SRP is the principle most often taken too far. "Split when there are two reasons to change" does not mean "one method per class." A class with five methods that all serve one coherent job has one responsibility. If you can't name two distinct actors who would request a change, don't split. Over-decomposition — twelve anemic one-method classes and a maze of wiring — reads as junior, not clean. The test is the actor, not the line count.

2.2 O — Open/Closed: extract an interface instead of editing a switch

Open for extension, closed for modification: you should be able to add new behavior without touching code that already works and is already tested. The trigger is unmistakable — a switch on a type enum that you have to edit every time a new type appears. Each edit risks the existing cases; each edit re-opens tested code. The move is to make the type an interface and each case an implementation. 

// SMELL: every new shipping method re-opens this method and its tests
BigDecimal cost(Order o, String method) {
  switch (method) {
    case "STANDARD": return flat(o);
    case "EXPRESS":   return flat(o).multiply(TWO);
    // add "OVERNIGHT" here → edit tested code, risk the cases above
  }
}

// MOVE: the seam is an interface. New method = new class, zero edits above.
interface ShippingStrategy { BigDecimal cost(Order o); }
class Standard  implements ShippingStrategy { public BigDecimal cost(Order o){...} }
class Express   implements ShippingStrategy { public BigDecimal cost(Order o){...} }
class Overnight implements ShippingStrategy { public BigDecimal cost(Order o){...} } // just added, nothing else touched

Notice that the fix for the O smell is the strategy pattern — this is not a coincidence. Most of the patterns in Part 3 are just Open/Closed applied to a particular kind of variation. When you name OCP in an interview, follow it immediately with the seam: "so I'll extract this switch into an interface, which is the strategy pattern, and each new type is a new class the rest of the system never has to know about."

2.3 L — Liskov Substitution: a subtype must keep the supertype's promises

If code works with a Bird, it must keep working when you hand it any subclass of Bird — no surprises, no exceptions the caller didn't expect. The classic violation is the Square extends Rectangle trap and its cousin, the subclass that throws on an inherited method: 

// SMELL: Penguin IS-A Bird, but breaks every caller that relies on fly()
class Bird    { void fly() { ... } }
class Penguin extends Bird {
  public void fly() { throw new UnsupportedOperationException(); } // LSP violation
}

// MOVE: the hierarchy was wrong. "Can fly" is a capability, not the base type.
interface Bird    { void eat(); }
interface Flying  { void fly(); }
class Sparrow implements Bird, Flying { ... }
class Penguin implements Bird           { ... } // simply doesn't claim Flying
The practical LSP tell in an LLD round is any throw new UnsupportedOperationException() in an override, or an override that tightens what the parent accepted (needs more, promises less). Say it out loud when you catch yourself: "That override would throw, which means Penguin isn't really substitutable for Bird — the inheritance is modeling 'is-a' when the real relationship is a capability. I'll pull fly into its own interface rather than fake it in the hierarchy." Catching your own LSP violation and refactoring the hierarchy live is one of the strongest signals you can give.

2.4 I — Interface Segregation: don't force stub methods

When one fat interface forces implementers to write { /* not supported */ } bodies, split it. The canonical example is a Machine interface with print, scan, and fax: a simple printer that can't scan or fax is now forced to stub two methods it will never honor — which is also an LSP violation waiting to happen. 

// SMELL: the old printer must stub scan() and fax() it can't do
interface Machine { void print(Doc d); void scan(Doc d); void fax(Doc d); }

// MOVE: role interfaces. Each device implements only what it truly does.
interface Printer { void print(Doc d); }
interface Scanner { void scan(Doc d); }
interface Fax     { void fax(Doc d); }
class SimplePrinter implements Printer { ... }         // no stubs
class OfficeMFP     implements Printer, Scanner, Fax { ... }

ISP and SRP rhyme: SRP is about a class doing one job, ISP is about a client depending on one role. The payoff is that PrintQueue can take a Printer and be completely unaffected by changes to scanning. Depend on the narrowest interface that covers what you actually call.

2.5 D — Dependency Inversion: depend on the interface, inject the concrete

High-level policy should not be nailed to low-level detail. The smell is a business class that constructs its own collaborator with new — new MySqlUserRepository(), new SmtpEmailSender() — because now the policy can't be tested without a real database and can't be pointed at a new implementation without editing it. The move: the high-level class declares an interface it needs, and someone else (a constructor argument, a factory, Spring) supplies the concrete type. This is exactly the pattern you saw all over Phase 1's Spring code — constructor injection is DIP. 

// SMELL: OrderService is welded to Postgres and real email. Untestable in isolation.
class OrderService {
  private final PostgresOrderRepo repo = new PostgresOrderRepo();  // concrete, hard-wired
  private final SmtpMailer mailer      = new SmtpMailer();
}

// MOVE: depend on abstractions; inject the implementations via the constructor.
interface OrderRepo { void save(Order o); }
interface Mailer    { void send(Email e); }

class OrderService {
  private final OrderRepo repo;
  private final Mailer mailer;
  OrderService(OrderRepo repo, Mailer mailer) {   // injected: a real one in prod,
    this.repo = repo; this.mailer = mailer;    // a fake one in tests
  }
}
This is the SOLID letter with the highest interview leverage because it links straight to your Spring phase and to testability. When you inject an interface, "write a unit test" becomes "pass a stub" — no database, no SMTP server, no mocking framework gymnastics. If an interviewer asks "how would you test this?", a design that already inverts its dependencies makes the answer trivial, and that's not an accident — testable and well-factored are the same property viewed from two sides.

Part 3 — The six patterns that show up

There are 23 Gang-of-Four patterns and you will be asked about maybe six of them, ever. The rest are either niche, obsolete (looking at you, singleton), or so absorbed into frameworks that you use them without naming them. These six earn their place in LLD interviews because each answers a smell that naturally arises as you design a parking lot or an order system on a whiteboard. Learn them as smell → fix pairs, and — the part that separates senior from mid — learn the over-engineering boundary for each: the case where reaching for the pattern is worse than not.

PatternProblem it solvesThe smell that calls for itCategory
Strategy Swap one algorithm for another at runtime. A switch/if-else on a type code that keeps growing. Behavioral
Factory Centralize and hide object creation. new ConcreteType() scattered across callers; a creation switch. Creational
Observer Notify many interested parties when something happens. One class calling five others "and also tell X, and Y, and Z" after an event. Behavioral
State Behavior that changes with an object's lifecycle stage. if (status == …) repeated in every method; illegal transitions slip through. Behavioral
Builder Construct a complex/immutable object step by step. A telescoping constructor with eight args, half of them optional. Creational
Decorator Add responsibilities to an object without subclassing. A combinatorial explosion of subclasses (CarWithGpsAndSunroof). Structural

3.1 Strategy — swap the algorithm behind an interface

What it is: a family of interchangeable algorithms, each in its own class behind a common interface, chosen and swapped at runtime. Why it exists: to kill the growing type-code switch — it's Open/Closed made physical. The smell: a method that branches on a mode or type and gains a new branch every time the product does. You already saw the shape in §2.2; here it is as a first-class object the caller holds and can change on the fly. 

interface PricingStrategy { BigDecimal price(Ride r); }

class StandardPricing implements PricingStrategy {
  public BigDecimal price(Ride r) { return base(r); }
}
class SurgePricing implements PricingStrategy {
  private final double multiplier;
  SurgePricing(double m) { this.multiplier = m; }
  public BigDecimal price(Ride r) { return base(r).multiply(valueOf(multiplier)); }
}

class FareCalculator {
  private PricingStrategy strategy;                 // held, not hard-coded
  FareCalculator(PricingStrategy s) { this.strategy = s; }
  void setStrategy(PricingStrategy s) { this.strategy = s; } // swap at runtime
  BigDecimal quote(Ride r) { return strategy.price(r); }
}
In Java a strategy with a single method is just a functional interface — PricingStrategy could be a Function<Ride, BigDecimal> and each implementation a lambda. Say that out loud: "since this is a one-method interface, in modern Java each strategy can just be a lambda, and the pattern collapses into passing a function." It shows you understand the pattern is about the seam, not the ceremony.
When NOT to use it: if there are exactly two cases and there will only ever be two — a boolean and an if is clearer than two classes and an interface. Strategy earns its keep when the set of algorithms is open (you expect more) or when you need to swap them at runtime. A fixed, closed pair of behaviors wrapped in the full pattern is ceremony that reviewers read as cargo-culting.

3.2 Factory — centralize the new

What it is: a single place that decides which concrete class to instantiate, so callers ask for what they want, not how it's built. Why it exists: to stop object-creation logic (and the switch that picks the type) from leaking into every caller — a Dependency-Inversion enabler, since callers now depend on the interface and the factory owns the concrete choice. The smell: the same if (type == …) new X() else new Y() duplicated wherever an object is created, so adding a type means hunting down every site. 

interface Vehicle { int spotsNeeded(); }
class Car        implements Vehicle { public int spotsNeeded() { return 1; } }
class Motorcycle implements Vehicle { public int spotsNeeded() { return 1; } }
class Bus        implements Vehicle { public int spotsNeeded() { return 5; } }

class VehicleFactory {
  static Vehicle create(VehicleType type) {   // ONE switch, in ONE place
    return switch (type) {
      case CAR        -> new Car();
      case MOTORCYCLE -> new Motorcycle();
      case BUS        -> new Bus();
    };
  }
}

Note what the factory does and doesn't buy you. It centralizes the creation switch so there's one place to edit, and it lets every caller work against the Vehicle interface. It does not make creation Open/Closed by itself — you still edit the switch to add Truck. That's usually fine and honest to admit: "the factory still has one switch, but it's the only switch — the twenty callers don't change." If truly open extensibility matters, you register creators in a map (Map<VehicleType, Supplier<Vehicle>>) so new types self-register rather than editing the switch — but say that only if the requirement justifies it.

When NOT to use it: don't build a factory for an object with one implementation and a simple constructor. new EmailValidator() does not need a EmailValidatorFactory. The pattern pays off when creation involves a real choice between types, non-trivial setup, or you want to hide the concrete class from callers. A factory that only ever returns one type is pure indirection — a reviewer will ask "why isn't this just a constructor?" and they'll be right.

3.3 Observer — publish an event, let subscribers react

What it is: a subject maintains a list of observers and notifies them all when its state changes, without knowing who they are or what they do. Why it exists: to decouple "something happened" from "everything that must react to it" — the subject stays closed to new reactions. The smell: a method that, after doing its real work, has a growing tail of "and also notify inventory, and email the customer, and update analytics, and ping the fraud service" — each new consumer edits the same method and welds it to one more collaborator. This is the in-process cousin of the pub/sub and Kafka fan-out you met in tutorial 06. 

interface OrderObserver { void onOrderPlaced(Order o); }

class OrderService {
  private final List<OrderObserver> observers = new ArrayList<>();
  void subscribe(OrderObserver o) { observers.add(o); }   // add reactions without editing place()

  void place(Order o) {
    repo.save(o);                                          // the real work
    for (OrderObserver obs : observers) obs.onOrderPlaced(o); // fan out, subject stays ignorant
  }
}

class EmailReceipt   implements OrderObserver { public void onOrderPlaced(Order o){...} }
class InventoryHold implements OrderObserver { public void onOrderPlaced(Order o){...} }
class AnalyticsTap  implements OrderObserver { public void onOrderPlaced(Order o){...} }
Name the two decisions the pattern forces, because interviewers push on exactly these: ordering and failure isolation. Say: "observers run in registration order, and I have to decide whether one observer throwing should block the others — usually not, so I'd wrap each call in a try/catch and log, or hand off to an async executor so a slow analytics sink can't delay the customer's response." That synchronous-vs-async and one-failure-shouldn't-fail-all judgment is the whole reason to prefer explicit observers over a hard-coded call list, and it echoes the "does this need to happen before I respond?" question from tutorial 06.
When NOT to use it: if there's exactly one reaction and there will only ever be one, a direct method call is clearer than an observer list — the indirection hides control flow ("what actually runs when I place an order?" becomes a runtime question instead of a readable one). Observer also makes debugging harder and can create surprise cycles (A observes B observes A). Reach for it when there are genuinely several independent reactions, or you expect new ones.

3.4 State — behavior that depends on lifecycle stage

What it is: each state of an object becomes its own class that knows how to behave and which state comes next; the object delegates to its current state. Why it exists: to replace the status flag that gets re-checked in every method, and to make illegal transitions impossible rather than merely discouraged. The smell: the same if (status == SHIPPED) … else if (status == …) ladder copy-pasted across cancel(), ship(), refund(), and a bug where an order got shipped twice because one method forgot to check the flag. Vending machines, order lifecycles, and document-approval flows are the classic LLD prompts where this pattern is the intended answer. 

interface OrderState {
  OrderState ship();     // each method returns the NEXT state
  OrderState cancel();   // illegal actions throw or return this
}

class Placed implements OrderState {
  public OrderState ship()   { return new Shipped(); }    // legal transition
  public OrderState cancel() { return new Cancelled(); }
}
class Shipped implements OrderState {
  public OrderState ship()   { throw new IllegalStateException("already shipped"); }
  public OrderState cancel() { throw new IllegalStateException("cannot cancel a shipment"); }
}

class Order {
  private OrderState state = new Placed();
  void ship()   { state = state.ship(); }    // no status switch anywhere
  void cancel() { state = state.cancel(); }  // the state object enforces legality
}
State and strategy are the same shape — an object delegating to a swappable interface — and candidates conflate them. The distinction to state cleanly: "strategy is chosen by the caller and doesn't change itself; state is chosen by the object's own transitions — each state decides the next one. Same structure, opposite driver." A finite-state machine drawn on the whiteboard (nodes = states, edges = legal transitions) is the artifact interviewers love here; the code is just that diagram typed out.
When NOT to use it: two or three states with simple, rarely-changing rules don't need a class per state — an enum with a canTransitionTo method, or a transition table (Map<State, Set<State>>), is lighter and often clearer. The full pattern pays off when each state carries genuinely different behavior across several methods, not just a different label. Spinning up five classes to model a two-state on/off toggle is over-engineering.

3.5 Builder — assemble a complex or immutable object step by step

What it is: a separate object that accumulates parameters through chained calls and produces the final, ideally immutable, object in one build(). Why it exists: to replace the "telescoping constructor" — the four overloaded constructors you write because half the fields are optional — and to let you build immutable objects without a twelve-argument call where nobody can tell which true is which. The smell: new Pizza(true, false, true, false, 2, null, "L") — unreadable, error-prone, and a nightmare the moment two booleans are adjacent. 

class Pizza {
  private final Size size;          // required
  private final boolean cheese, olives; // optional, default false

  private Pizza(Builder b) {         // private ctor: only the builder constructs it
    this.size = b.size; this.cheese = b.cheese; this.olives = b.olives;
  }

  static class Builder {
    private final Size size;      // required → constructor arg of the builder
    private boolean cheese, olives;
    Builder(Size size) { this.size = size; }
    Builder cheese() { this.cheese = true; return this; } // return this → chaining
    Builder olives() { this.olives = true; return this; }
    Pizza build() { return new Pizza(this); }        // validate here if needed
  }
}

// Reads like a sentence; required arg enforced by the type system:
Pizza p = new Pizza.Builder(Size.LARGE).cheese().olives().build();
In an interview you rarely hand-write the whole builder — you say "I'd use a builder here; in practice Lombok's @Builder or a record generates it" and move on. Know the hand-written version so you can produce it if pushed, but the point you're scoring is why: required fields as builder constructor args, optionals as chained setters, immutability via a private constructor and final fields, and a single build() where cross-field validation lives.
When NOT to use it: an object with two or three fields, all required, doesn't need a builder — a plain constructor or a Java record is shorter and clearer. Builder earns its place at roughly four-plus fields, or when several are optional, or when you want immutability with readable construction. Writing a builder for a two-field value class is boilerplate for its own sake.

3.6 Decorator — add behavior by wrapping, not subclassing

What it is: a wrapper that implements the same interface as the thing it wraps, adds its own behavior, and delegates the rest — so you can stack wrappers to compose features at runtime. Why it exists: to escape the subclass explosion you get when features combine. The smell: you need Coffee, CoffeeWithMilk, CoffeeWithMilkAndSugar, CoffeeWithSugar, CoffeeWithMilkAndSugarAndCream… every combination of N optional features wants its own subclass, which is 2ᴺ classes. Decorator makes each feature one wrapper and lets you stack them. This is the exact pattern behind java.io — new BufferedReader(new InputStreamReader(new FileInputStream(f))) is three decorators wrapping a stream. 

interface Coffee { BigDecimal cost(); String desc(); }

class Espresso implements Coffee {          // the base component
  public BigDecimal cost() { return valueOf(2.00); }
  public String desc()     { return "espresso"; }
}

abstract class AddOn implements Coffee {    // wraps a Coffee, IS a Coffee
  protected final Coffee inner;
  AddOn(Coffee c) { this.inner = c; }
}
class Milk extends AddOn {
  Milk(Coffee c) { super(c); }
  public BigDecimal cost() { return inner.cost().add(valueOf(0.50)); } // add, then delegate
  public String desc()     { return inner.desc() + " + milk"; }
}

// Compose features at runtime by stacking wrappers — no new subclass per combo:
Coffee order = new Sugar(new Milk(new Espresso()));  // espresso + milk + sugar
The one-liner that shows you know this from real code, not a textbook: "decorator is how java.io works — BufferedReader wrapping an InputStreamReader wrapping a FileInputStream — each layer adds one capability and forwards the rest. It's also how HTTP middleware and servlet filters chain. So when I need optional, stackable, order-sensitive features on one object, I wrap instead of subclass." Naming a standard-library instance of a pattern is worth three textbook definitions.
When NOT to use it: if features don't combine — a coffee is either small, medium, or large and never "small and large" — that's a strategy or a plain field, not a decorator; you don't have a combinatorial explosion to escape. Decorator also produces many small classes and deep wrapping that can be hard to debug (a stack trace ten wrappers deep), and the order of wrapping can matter in non-obvious ways. Use it specifically for independent, optional, composable responsibilities.

Part 4 — From prompt to classes in ~10 minutes

The patterns are useless if you freeze at "design a parking lot" and stare at the whiteboard. What you need is a repeatable procedure that turns any prompt into a class diagram in the first ten minutes, so the remaining thirty-five are spent on the interesting bits — the deep dive and the follow-ups. Here it is. It's the same recipe page 25 applies to full case studies; learn it here on the mechanics.

4.1 The recipe: nouns → classes, verbs → methods, variation points → interfaces

  1. Clarify and scope (2 min). Ask the two or three questions that halve the problem, then state what you're building and what you're cutting. "Multiple floors? Payment? I'll model spots, vehicles, and assignment cleanly, and stub pricing." Write the in-scope list where you can see it.
  2. Nouns → classes / entities (2 min). Underline the nouns in the requirements. Parking lot, floor, spot, vehicle, ticket, payment, gate. Each meaningful noun is a candidate class. Distinguish entities (have identity and lifecycle: Ticket, Vehicle) from value objects (defined by their fields, immutable: Money, TimeRange). Enums for closed sets: VehicleType, SpotType, SpotStatus.
  3. Verbs → methods, and find their owner (2 min). The verbs are the behavior — park, unpark, assign a spot, issue a ticket, calculate a fee, pay. Each verb becomes a method, and the harder question is which class owns it. "Assign a spot" belongs to a ParkingLot or a dedicated SpotAssigner, not to Vehicle. Put behavior next to the data it operates on; if a method reaches into another object's fields to do its job, it's probably on the wrong class.
  4. Variation points → interfaces (2 min). This is the senior step. Ask: where will this change? The spot-assignment algorithm (nearest? cheapest? random?) → SpotAssigner interface (strategy). The fee calculation (flat? hourly? tiered?) → PricingStrategy interface. Vehicle creation → factory. The payment method → interface. Each answer to "where will this change?" is a seam, and each seam is one of the patterns from Part 3. Everything that won't vary stays a plain concrete class.
  5. Relationships and a diagram (1 min). Draw has-a (composition: a Lot has Floors has Spots) and is-a (only where Liskov holds). Then write the key method signatures and one or two method bodies that show the main interaction end to end.
  6. Walk one scenario, then invite the follow-up. Trace "a car arrives and parks" through your classes out loud. Then you're positioned for the extension the interviewer will throw — and because you put the seams in at step 4, it lands on an interface, not a rewrite.
The single sentence that frames the whole exercise, say it near the start: "I'll model the nouns as classes and the verbs as methods, and then the only real design decision is where I expect change — those points become interfaces, everything else stays concrete until it needs to flex." That tells the interviewer you know a design is defined by its seams, and that you won't over-abstract the parts that don't move. It's the thesis of this entire page compressed to one line.

4.2 Worked micro-example: the parking lot, in one pass

Requirements: "A parking lot has floors, each with numbered spots of different sizes. Vehicles of different types park in the smallest spot that fits. Issue a ticket on entry, charge by the hour on exit." Run the recipe: 

  NOUNS  → ParkingLot, Floor, Spot, Vehicle, Ticket        (classes)
          VehicleType{CAR,BIKE,BUS}, SpotType, SpotStatus  (enums)
          Money, Duration                                  (value objects)

  VERBS  → park(vehicle) / unpark(ticket)   → ParkingLot
          findSpot(vehicle)                  → SpotAssigner  (varies!)
          price(ticket)                      → PricingStrategy (varies!)

  SEAMS  → SpotAssigner   interface  (nearest vs smallest-fit vs cheapest)
          PricingStrategy interface  (hourly vs flat vs weekend rate)
          PaymentMethod   interface  (card vs cash vs wallet)

interface SpotAssigner   { Optional<Spot> assign(Vehicle v, List<Spot> free); }
interface PricingStrategy { Money price(Ticket t); }

class ParkingLot {
  private final List<Floor> floors;
  private final SpotAssigner assigner;      // injected seam
  private final PricingStrategy pricing;    // injected seam

  Ticket park(Vehicle v) {
    Spot spot = assigner.assign(v, freeSpots())
        .orElseThrow(() -> new LotFullException());  // Optional forces the caller to handle "full"
    spot.occupy(v);
    return new Ticket(v, spot, now());
  }

  Money unpark(Ticket t) {
    t.spot().vacate();
    return pricing.price(t);            // "add weekend rates" = new strategy, this method unchanged
  }
}

That's a complete, extensible core in one pass. When the interviewer says "now add electric-vehicle charging spots", you add a SpotType and a matching SpotAssigner rule — no edit to park. "Now add surge pricing on weekends" is a new PricingStrategy — no edit to unpark. That's the payoff of doing step 4 up front, and it's exactly the muscle page 25 drills across the full parking-lot, elevator, and BookMyShow case studies.

Two failure modes bracket this round. Under-design: everything lands in one ParkingLotManager god class with a 30-branch switch — no seams, and every follow-up is a rewrite. Over-design: you introduce an abstract factory, a visitor, and a dependency-injection container for a parking lot, and spend your forty minutes wiring instead of designing. The target is the narrow middle: seams exactly where you named a variation point in step 4, plain concrete classes everywhere else. When in doubt, start concrete and say "I'd extract an interface here the moment a second implementation appears." Naming the seam you're choosing not to build yet is itself the senior signal.

Interview Q&A

1. You keep saying "SOLID as moves, not slogans." Give me one, applied.

Open/Closed. The slogan is "open for extension, closed for modification," which means nothing until you tie it to a smell. The smell is a switch on a type code that I have to edit every time a new type appears — each edit re-opens code that's already tested and risks the existing branches. The move is: extract the switch into an interface, make each case an implementation, and route the choice through a factory or strategy. Now a new type is a new class the other twenty callers never see.

I'd say it exactly like that in the room — "I see a growing type switch, so I'll apply Open/Closed by pulling it into an interface" — because it shows I recognized a concrete problem and reached for the principle as a tool, not that I memorized a definition.

2. What's the difference between strategy and state? They look identical.

Structurally they are identical — an object delegating to a swappable interface. The difference is who drives the swap. In strategy, the caller picks the algorithm and it doesn't change itself: a FareCalculator is handed SurgePricing and keeps using it. In state, the object's own transitions pick the next behavior: Placed.ship() returns Shipped, and the object moves itself along.

The tell in a prompt: if it's a lifecycle — order, document approval, vending machine, TCP connection — it's state, and I'd draw the finite-state machine first. If it's "compute this N different ways" — pricing, routing, compression — it's strategy. Same code shape, opposite driver.

3. When would you deliberately NOT use a design pattern?

Whenever the variation the pattern abstracts doesn't actually exist yet. A strategy for two fixed cases that will only ever be two — use a boolean. A factory for a single implementation with a plain constructor — use new. A builder for a two-field record — use the constructor. A state machine for an on/off toggle — use an enum.

The reason this matters at senior level is that over-engineering is graded down. Every premature abstraction is indirection a future reader has to hold in their head for a flexibility nobody needed. My rule: start concrete, and extract the interface the moment a second real implementation appears — and I'll say that out loud, because naming the seam I'm choosing not to build yet is the judgment they're testing.

4. Walk me from "design a vending machine" to classes in a couple of minutes.

Nouns first: VendingMachine, Product, Slot/Inventory, Coin/Payment. Enums for the closed sets. Verbs: select product, insert money, dispense, refund. Those are methods, mostly on VendingMachine.

Then the seams. A vending machine is a lifecycle — idle → collecting money → dispensing → out of stock — so the behavior of "insert coin" depends on which state I'm in. That's the state pattern, and it's the intended answer here: each state is a class that knows the legal next transition, which kills the if (status == …) ladder and makes "dispense before payment" structurally impossible. Payment method (coin vs card) is a strategy or a small interface. I'd draw the state machine, then implement two states to show the interaction.

5. What's a Liskov violation and how do you catch one live?

A subtype that doesn't honor the supertype's contract — so code written against the parent breaks when handed the child. The two field tells: an override that throws UnsupportedOperationException ("this subclass doesn't really do that method"), and an override that strengthens preconditions (needs more than the parent, or returns less than the parent promised). The classic is Square extends Rectangle: set width, and a Square silently changes height too, breaking any code that assumed they're independent.

I catch it by noticing I'm about to write a throwing override, and I say: "that means this isn't really an is-a relationship — the shared thing is a capability, so I'll pull it into its own interface rather than fake it in the hierarchy." Refactoring an UnsupportedOperationException into a capability interface, live, is one of the strongest moves in this round.

6. Isn't the singleton pattern one of "the ones that show up"?

It shows up in questions mostly as a trap. Singletons are global mutable state wearing a pattern costume: they're hard to test (you can't substitute them), they hide dependencies (a class using a singleton doesn't declare it), they create hidden coupling, and the naive double-checked-locking version is a concurrency footgun. If I need exactly one of something, I'd rather create it once at composition root and inject it — which is what Spring does when it makes a bean singleton-scoped. That gives me "one instance" without the global-variable pathology, and it stays testable because I can inject a different one in a test. So if asked, I explain the pattern and then explain why I'd reach for dependency injection instead.

7. My design has one giant class with all the logic. What principle names the fix, and how do I split it?

Single Responsibility, and the operative test is the actor: how many distinct people or teams would request a change to this class? A ParkingLotManager that assigns spots, calculates fees, processes payments, and prints tickets has four reasons to change. I split along those axes: SpotAssigner, PricingStrategy, PaymentProcessor, TicketPrinter, with a thin ParkingLot orchestrating them.

The nuance I'd add so I don't over-correct: SRP does not mean one method per class. Five methods serving one coherent job is one responsibility. I split when I can name two distinct actors, not to hit a line count — twelve anemic one-method classes are as much a smell as one god class, just the opposite one.

8. Why do you keep injecting interfaces through constructors? Justify the ceremony.

It's Dependency Inversion, and it buys two things that matter in every follow-up. Testability: a class that takes an OrderRepo interface can be tested by passing a stub — no database, no mocking framework — so when they ask "how would you test this?" the answer is trivial. Extensibility: swapping Postgres for an in-memory repo, or a real mailer for a fake, is a wiring change, not a code change to the business class.

It's the same thing your Spring code does — constructor injection is DIP — so it's not extra ceremony, it's the ceremony you already pay, made explicit. The class declares what it needs as an abstraction; the composition root decides the concrete type. High-level policy and low-level detail both point at the interface in the middle.

9. An interviewer says "add a new notification channel." What did your original design need for that to be a one-liner?

It needed a seam at the notification axis, put in during step 4 of the recipe. If I'd hard-coded sendEmail() and sendSms() as a branch in my order code, "add push" means editing tested code and threading a new case through. If instead I have a NotificationChannel interface with Email and Sms implementations, "add push" is one new class plus registering it — strategy plus a dash of observer if multiple channels fire on one event.

That's the whole game: the follow-up is chosen to test whether the seam is in the right place. So when I design up front I ask "where will this change?" — and notification channels, payment methods, pricing rules, and entity types are the usual answers — and I put interfaces exactly there and nowhere else.

10. You've got 45 minutes and a vague prompt. How do you not freeze?

I run a fixed procedure so I never face a blank page. Two minutes clarifying scope. Two minutes underlining nouns → those are my classes and enums. Two minutes on verbs → methods, and deciding which class owns each (put behavior next to its data). Two minutes on the one senior question — where will this change? — and each answer becomes an interface. One minute drawing has-a/is-a relationships. Then I write signatures and walk one scenario end to end.

That's the first ten minutes, deterministic, so the remaining thirty-five go to the deep dive and the follow-ups — which is where judgment actually gets graded. The recipe isn't the impressive part; it's the thing that buys me room to be impressive. Concurrency, edge cases, and the "now add X" extensions all land better when I'm not still panicking about what the classes are.

What to remember

The round

  • LLD/OOD is a separate round at enterprise and fintech loops — Optum, JPMC, Goldman and most mid-size product companies. The whiteboard is code, not boxes. It's HLD zoomed one level down: same decomposition skill, applied to classes.
  • Graded on decomposition, extensibility at the right axis, restraint, and Java fluency — in that order. The "now add X" follow-up is the real test; your whole design exists to make it a one-liner. Over-engineering is graded down.

SOLID as five moves

  • S — two reasons to change (two actors) → split the class. Not one method per class.
  • O — a growing type switch → extract an interface, new variant = new class. (This is literally strategy/factory.)
  • L — an override that throws or narrows the contract → fix the hierarchy; pull the capability into its own interface. Watch for UnsupportedOperationException.
  • I — implementers forced to stub unused methods → split the interface into roles. Depend on the narrowest thing you call.
  • D — a class that news its own collaborators → depend on an interface, inject the concrete. Constructor injection is DIP; it makes testing trivial.

The six patterns — smell → fix, and when not to

  • Strategy — swap an algorithm at runtime; fixes a growing type switch. Not for a fixed pair of cases (use a boolean). One-method strategy = a lambda.
  • Factory — centralize the new; fixes scattered creation switches. It's one switch, not zero. Not for a single implementation with a plain constructor.
  • Observer — fan out an event to many reactors; fixes the "and also notify X, Y, Z" tail. Decide ordering and failure isolation (sync vs async). Not for a single fixed reaction.
  • State — behavior driven by lifecycle; fixes the repeated status ladder and makes illegal transitions impossible. Same shape as strategy, but the object's transitions drive the swap. Not for two trivial states (use an enum).
  • Builder — step-by-step immutable construction; fixes the telescoping constructor. Required fields as builder ctor args, optionals chained, validate in build(). Not for two-field objects (use a record).
  • Decorator — stack optional features by wrapping; fixes the subclass explosion. It's how java.io and HTTP middleware work. Not for features that don't combine.

Prompt → classes in 10 minutes

  • Scope (2m) → nouns become classes/enums/value objects (2m) → verbs become methods, owned by the class holding their data (2m) → variation points become interfaces (2m) → relationships + signatures (1m) → walk one scenario (1m).
  • The one senior question that generates the whole design: "where will this change?" Every answer is a seam and a pattern; everything else stays a plain concrete class until it needs to flex.
  • Avoid both failure modes: the god class with a 30-branch switch (under-design) and the abstract-factory-visitor-DI cathedral for a parking lot (over-design). Aim for the narrow middle.
The through-line, and the sentence to land near the end of any LLD round: a design is defined by its seams — the places you deliberately made it easy to change — and by the places you deliberately left concrete. SOLID tells you which smell needs a seam; the patterns are the seams pre-shaped for common variations; the recipe finds where the seams go. Put an interface exactly where you expect change and nowhere else, say why for each one, and name the seams you're choosing not to build yet. That last habit — restraint stated out loud — is what separates a 7-YOE answer from a 3-YOE one, and it's exactly the muscle the next page drills on full case studies.
