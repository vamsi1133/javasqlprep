20 — Design a News Feed UI

The frontend round's most common prompt, walked end to end as RADIO. The server is one API contract; the browser is the whole interview.

Why this matters in the interview

"Design a news feed" is to the frontend round what "design a URL shortener" is to the backend round: the canonical warm-up that every interviewer has in their back pocket, and the one a strong candidate is expected to make look easy. You already designed the backend of this exact product in tutorial 10 — fan-out on write versus read, the celebrity problem, timeline caching. Forget almost all of it. This is the other interview. The interviewer has drawn the client as the whole subject and hand-waved the server to a box that returns pages of posts, and every minute you spend on how the feed service shards is a minute spent proving you can do the round you weren't asked to do. What they actually want to watch is how you build the component: how it renders ten thousand posts without melting the browser, how it paginates without the scroll jumping, how a like feels instant, how images load without the layout lurching. That is a home game for you — seven years in the browser is exactly the experience this round rewards — so the whole task here is to turn instinct into a narrated, sequenced answer.

We walk it as RADIO from tutorial 19 — Requirements, Architecture and components, Data model, Interface and API, Optimizations — because that structure keeps you honest and back-loads your time onto the deep-dive where the points live. Four techniques are the marquee of a feed answer and the interviewer is fishing for each: list virtualization (windowing — you render twenty DOM nodes for a ten-thousand-item list), cursor pagination (and being able to say precisely why offset drifts when new posts arrive), optimistic updates with rollback (the like that feels instant), and a normalized client cache (one copy of each post, so the feed and the detail view never disagree). Hit those four with the reasoning and not just the library names, treat the backend as a one-line contract, and this becomes the round you're the favourite to win. Everything below is the say-it-out-loud version of that answer.

Open by naming the round and locking scope, exactly as tutorial 19 taught: "I'm reading this as a frontend system design problem, so I'll design the client — component architecture, where state lives, data flow, and then performance, network and accessibility — and treat the feed backend as an API contract I'll define but not scale. I know how I'd fan out and cache timelines server-side, but I'll assume that exists behind a paginated endpoint unless you want me to design it too." Thirty seconds, and you've told the interviewer you know there are two rounds and you're deliberately in this one.

The prompt

The interviewer gives you the whole thing in one breath:

"Design the news feed for a social app — think Twitter or Instagram. Focus on the frontend."

"Focus on the frontend" is doing a lot of work in that sentence, and sometimes the interviewer doesn't even say it — you're expected to infer it from the fact that you're in a frontend loop. Either way, the load-bearing decisions are all unstated. Is this a text feed (Twitter) or a media-first grid (Instagram)? Does it update live as new posts arrive, or only when you pull to refresh? Mobile web, desktop, or both? How long is the list — hundreds of items or effectively infinite? Are you building the viewer, the composer, or both?

As in every case study, the vagueness is the first thing graded, and the senior move is to make the problem smaller and sharper before you draw a single component — and to say what you're cutting so the interviewer knows the omission is a choice, not a gap. The one difference from the backend rounds: the clarifying questions here are about the experience, not the scale. Nobody is going to hand you a DAU figure, and if you invent one so you can perform the estimation ritual, you've announced you're in the wrong interview.

If the interviewer does start pushing on how the timeline is assembled server-side — fan-out, ranking, caching — that's your cue that they want a full-stack answer, and tutorial 10 is your script for that half. But let them pull you there. The default reading of "design a news feed" in a frontend loop is the client, and you should commit to the client until told otherwise.

R — Requirements

Pin down the surface before you build it — same first move as every round, but the questions are about behaviour and experience, not throughput. Ask three or four, state your assumptions, and move; you're demonstrating that you know the axes, not filling in a form.

Functional requirements — for a UI, these are user-visible interactions

Enumerate what the user can see and do, because that list is your component list. For a feed:

  • View a scrollable list of posts — author, text, media, timestamp, like/comment counts.
  • Infinite scroll — more posts load automatically as you approach the bottom.
  • Like / unlike a post — and see the count change immediately.
  • See new posts that arrive while you're reading — at minimum a "12 new posts" pill, ideally with pull-to-refresh.
  • Open a post to a detail view (comments, full thread) — a navigation target.

Now cut, out loud: "I'll build viewing, infinite scroll, liking, and the new-posts affordance in depth. I'll treat composing a post, the comment thread, and the profile page as navigation targets rather than designing them — unless you'd rather I go deep on one of those instead." Naming the cut is the senior move; it's the frontend equivalent of saying which subsystems you're descoping in a backend design.

Non-functional requirements — this is where the round announces itself

The backend non-functionals (availability, latency, consistency, durability) don't apply. The frontend list is a different one, and reciting a few of them unprompted is one of the fastest ways to signal which interview you know you're in.

Non-functionalThe question to askWhy it changes the design

Good?" That single sentence is the entire non-functional spec, and it took fifteen seconds.

The one estimation that earns its place

There's no DAU-to-QPS arithmetic here, and grinding one out wastes minutes you need for the deep-dive. The estimate that does change a decision is DOM weight and payload weight: 

A feed page = 20 posts.
  JSON per post   ≈ 1–2 KB  → ~30 KB metadata per page   (cheap)
  Image per post  ≈ 100–300 KB → 2–6 MB of media per page (DOMINATES)

An "infinite" scroll after 10 minutes = ~500 posts loaded.
  500 posts as raw DOM: each post ≈ 30–50 DOM nodes
    → 15,000–25,000 nodes → layout/paint jank, memory bloat, GC pauses
  500 posts virtualized: ~20 visible + overscan ≈ 30 nodes on the page. Flat.

Two numbers, two decisions: images are the performance problem (so image loading gets a deep-dive), and DOM node count grows without bound unless you window the list (so virtualization is mandatory). That's estimation used to justify a choice — the only kind that belongs in this round.

Don't turn Requirements into a numbers exercise out of backend habit, and don't skip it out of eagerness to draw components. The failure at both ends looks junior: the candidate who computes fan-out QPS for a frontend prompt has misread the room, and the candidate who starts sketching a component tree before scoping the surface is coding before reading the ticket. Ask a few questions, estimate only DOM/payload weight, state assumptions, move.
Name accessibility as a requirement before anyone asks, and flag the one that's genuinely hard here: "I'll target WCAG AA and fold it into each component. The two non-obvious pieces in a feed are that a virtualized list must stay keyboard- and screen-reader-navigable even though most rows aren't in the DOM, and that new posts arriving should be announced through an aria-live region rather than silently appearing. I'll come back to both in optimizations." Most candidates forget a11y entirely; naming the specific feed-shaped a11y risks up front is a clear senior tell.

A — Architecture & components

The frontend high-level design: your boxes are components, your arrows are data flow and events. Produce three things — the component tree, a statement of where each piece of state lives, and the direction data flows. This is the step that most directly tests "can you decompose a UI," which is a top-scored signal.

The component tree — draw this first

  <FeedPage>                         ← route container; owns the feed query + mutations
    ├─ <NewPostsPill>                 ← "12 new posts" — appears when live items arrive
    ├─ <VirtualFeedList>              ← the windowed scroll container (the hard part)
    │    ├─ <FeedItem>                ← one post, repeated ×20-ish (only visible ones mount)
    │    │    ├─ <PostHeader>         ← avatar, name, timestamp, overflow menu
    │    │    ├─ <PostBody>           ← text + <PostMedia> (lazy-loaded image)
    │    │    └─ <PostActions>        ← like / comment / share — owns optimistic like state
    │    └─ <LoadMoreSentinel>        ← IntersectionObserver target; triggers next page
    └─ <Toast>                        ← rollback errors / undo affordances

How you justify the seams. Each component has one nameable responsibility, and the two seams interviewers reward are reuse and re-render isolation. FeedItem and PostActions are the same on the feed and on the post-detail page, so they're factored out — reuse. And PostActions is its own leaf specifically because the like toggle changes often; keeping that volatile state in a small leaf means a like on post 5 re-renders one button, not the whole list. VirtualFeedList is separate from FeedPage because windowing is a self-contained concern with its own scroll math — the page shouldn't know how the list is rendered, only what data to feed it.

Where does state live? — the highest-value question in this step

For every piece of state, answer: how local can it be? Say the principle out loud — lift state to the lowest common ancestor that needs it, and no higher. Too global re-renders half the tree on every change; too local forces awkward threading when a sibling needs it.

StateWhere it livesWhy
The list of posts + pagination cursorServer-cache library (React Query / SWR), read by FeedPageIt's server state — a cached copy of data the server owns. The library gives caching, dedup, background revalidation, and the infinite-query cursor for free.
"Is this post liked" + like countThe normalized cache entry, toggled optimistically via PostActionsOne source of truth so the feed and detail view agree; the optimistic write is isolated to one entry so post 5's like doesn't touch post 6.
Scroll position / which rows are windowedLocal to VirtualFeedListPure view state the virtualizer derives from scrollTop; nothing above the list needs it.
"12 new posts buffered, not yet shown"FeedPage (or the query cache, held aside)Buffered live items live above the list but below the app; showing them is a user action, so you don't inject mid-scroll.
Logged-in user / themeApp-global (Context)Read in many places, changes rarely — the exact profile Context is good at.

Notice that almost nothing is truly global. That observation is itself a signal: candidates who reach for a single Redux store to hold the feed, the scroll position, and the like states have misunderstood where each kind of state wants to live. Most of this feed's "state" is a server cache; the rest is local view state.

Data flow — data down, events up

Draw the arrows and trace one interaction end to end. Data flows down as props; events flow up as callbacks. A like doesn't mutate anything in place — it fires an intent upward, the mutation layer updates the cache, and the new data flows back down. 

  server ──fetch page──▶ FeedPage ──posts (props)──▶ VirtualFeedList ──post──▶ FeedItem
                            ▲                                                      │
                            └──────────── onLike(postId) (event bubbles up) ──────┘
                            │
                     useLikeMutation writes the cache → new data flows back DOWN
  (data DOWN as props · events UP as callbacks · one direction · cache is the pivot)

Container vs presentational — the vocabulary for the seam

Say it plainly: "FeedPage is the container — it owns the feed query and the mutation hooks. Everything below it is presentational, driven by props, which keeps the leaves pure, trivially testable, and reusable on the detail page." Hooks blur the hard rule (any component can be "smart"), but the vocabulary is still the clearest way to name the boundary in one breath.

Don't decompose by visual boxes — decompose by responsibility, state, and re-render rate. The shallow-answer tell is a tree that mirrors the CSS layout (Header, Feed, Sidebar) with no thought to what owns which state or what re-renders when. The reason PostActions is its own component is not that it looks like a distinct region — it's that its state changes on every tap and you want that change to stop at one leaf. Carve along the axes of change and reuse, and justify each seam that way.
The line that shows you design for render cost structurally: "I'm isolating the like state in PostActions and virtualizing the list on purpose — both are about bounding the blast radius of a change. A like re-renders one button, not the feed; a scroll only mounts and unmounts a handful of rows, not thousands. That's the same instinct as picking a shard key so one hot entity doesn't melt one node — I'm deciding, up front, what a single change is allowed to touch." Connecting a render-cost decision to a backend blast-radius decision reads as someone who thinks about performance the same way in both halves of the stack.

D — Data model

The shape of the data as the client holds it — not database tables. Two decisions dominate this step: splitting server state from client state, and normalizing the store so the same post never exists in two places that can drift.

Server state vs client state

State the split explicitly — it instantly separates you from candidates who think "state management" means "put everything in Redux." The feed is server state: a cached copy of data the server owns, whose hard problems are all cache problems — staleness, revalidation, loading/error states, request dedup, pagination. The right tool is a server-cache library (React Query, SWR, RTK Query, Apollo), not a hand-rolled useEffect fetch and not Redux. The composer draft, which post's menu is open, the scroll position — those are client state, and the client is their source of truth.

The senior framing: server state is not really state, it's a cache, and everything from tutorial 06 applies in the browser. A server-cache library is doing cache-aside with stale-while-revalidate for you: serve the cached page instantly, revalidate in the background, dedupe concurrent requests for the same key (ten FeedItems asking for user:42 collapse to one request — the single-flight fix). Say "the feed is a cache, not state" and you've named the defining problem of modern frontend data.

Normalize the store — the client-side "third normal form"

The same post appears in the feed, in a "liked posts" list, in the detail view, in search results. Do not store copies. Store each entity once, keyed by id, and hold lists as arrays of ids that reference it. This is database normalization moved into the client, for the same reason: one source of truth, so an update in one place is seen everywhere and nothing drifts. 

// DENORMALIZED — the like lives inside each list copy. Like a post in the feed
// and the detail view still shows the old count. Two truths for one post. It WILL drift.
const bad = {
  feed:   [ { id: "p1", likeCount: 12, likedByMe: false, author: { id: "u9", name: "Ada" } } ],
  detail: { id: "p1", likeCount: 12, likedByMe: false, author: { id: "u9", name: "Ada" } },
};

// NORMALIZED — one post entity, one user entity; lists hold ids and the ORDER.
// Toggle posts["p1"].likedByMe in ONE place and every view re-renders correctly.
const good = {
  posts: {
    p1: { id: "p1", authorId: "u9", text: "...", mediaUrl: "...",
          likeCount: 12, likedByMe: false, commentCount: 3, createdAt: "..." },
  },
  users: { u9: { id: "u9", name: "Ada", avatarUrl: "..." } },
  feed:  { ids: ["p1", "p2", "p3"], nextCursor: "c_3" },  // ORDER + cursor, separate from entities
};

The split that makes this work: feed.ids holds the order and the pagination cursor, while posts holds the entities. That separation is what lets you append the next page (push ids onto the array) without touching any stored post, prepend a new live post by unshifting one id, dedupe a post that appears in two lists, and update one post's like-count in exactly one place. It's the same reason the backend keeps an index separate from the rows — and, as you'll see in the pagination section, it's also what makes cursor pagination clean and offset pagination a mess.

React Query / Apollo give you two flavours of this. Apollo (and RTK Query with normalizedComparator-style setups) normalizes by __typename:id under the hood, so writing one entity updates every query that references it automatically. React Query is document-cache by default — each query key holds its own copy — so if you want cross-list consistency you either normalize yourself or write to every relevant cache key on mutation. Knowing which your library does, and therefore how a like propagates, is a strong "I've shipped this" detail.

Derived state — compute, don't store

If a value can be computed from existing state, don't store it as its own state — derive it during render (useMemo if it's expensive). "Total visible posts," "has a next page" (!!nextCursor), "is the new-posts pill showing" (bufferedIds.length > 0) are all derived. Storing them creates a second source of truth you now have to keep in sync — the exact drift problem normalization exists to prevent.

Putting the feed in Redux is the most common data-model mistake here, and it's costly to defend. You inherit every cache problem by hand: loading flags, error flags, "is it stale," refetch-on-focus, dedup, the infinite-scroll cursor merge, cache eviction — reimplementing React Query badly, in reducers. If an interviewer sees you reach for Redux to hold fetched posts, the follow-up is "how do you invalidate that when it goes stale, and how do you dedupe two components fetching the same page?" — and the honest answer is "I wouldn't put it there." Client stores are for client state.
Open this step with the split as one paragraph: "I'll separate server state from client state. The feed, posts, users and comments are server state — I'll model them as a normalized, id-keyed cache managed by React Query or Apollo, with lists holding ids and the cursor separately from the entities, so I get dedup, background revalidation, and single-source-of-truth updates for free. Client state — the composer draft, an open menu, scroll position — stays local. Almost nothing is truly global." That paragraph makes the whole Optimizations step — optimistic updates, cache invalidation, prepending live posts — fall out naturally instead of feeling bolted on.

I — Interface & API

Two contracts, and strong candidates address both: the component interface (props in, callbacks out — how components talk to each other) and the network interface (endpoints and shapes — how the client talks to the server). Backend-leaning candidates design only the second and leave half the score on the table, because they don't think of component props as an API at all.

The component interface — data in as props, intent out as callbacks

// PostActions: small, reusable, presentational. Data in; the only thing it emits
// is an intent (onLike). It has no idea where the data comes from or what a like DOES.
interface PostActionsProps {
  postId: string;
  likeCount: number;
  likedByMe: boolean;
  onLike: (postId: string) => void;      // event UP — parent owns the meaning of "like"
  onOpenComments: (postId: string) => void;
}

Two judgment calls to voice. Keep the leaf presentational: PostActions takes likedByMe as a prop rather than reading the cache, so you can drop it on the feed and the detail page and unit-test it by rendering with props and asserting the callback fired. Emit intent, not implementation: it calls onLike(postId), not "optimistically increment and POST to this URL." The container owns what a like means. That's the props-API equivalent of a clean service boundary.

The network interface — you're the consumer; ask for exactly what the UI renders

// GET the feed — cursor-paginated (see below for exactly why cursor, not offset)
GET /api/feed?cursor=c_3&limit=20
// → {
     posts: [ { id, authorId, text, mediaUrl, likeCount, likedByMe, commentCount, createdAt } ],
     users: [ { id, name, avatarUrl } ],   // side-loaded → drop straight into the normalized store
     nextCursor: "c_4"                   // null when there are no more pages
   }

// Like a post — the mutation behind onLike. Idempotent, so a double-tap is safe.
PUT /api/posts/:id/like        // PUT not POST: "liked" set twice == liked once
// → { likeCount: 31, likedByMe: true }   // server returns the authoritative count

Three decisions to point at. The response side-loads users separately from posts — that's the normalized store shape from the D step reaching back into the API, so the client drops entities straight into their tables without unpacking nested duplicates. PUT for the like makes the mutation idempotent, so the double-tap race (or a retry) can't produce a count of 2 — the same idempotency reasoning as tutorial 06, applied to a button. And you request only rendered fields — asking for a 40-field user object to show an avatar and a name is the client-side SELECT *, and on mobile it's paid for in bytes over a slow radio.

Cursor vs offset pagination — and exactly why offset drifts

This comes up in nearly every frontend design and it's the single most probed line of a feed answer, so have the reasoning cold. For an infinite-scrolling, frequently-changing list, use cursor-based pagination (?cursor=c_3), never offset-based (?offset=40 / ?page=3).

Offset means "skip N rows, take the next 20." That's a position counted from the top of the list — and the top of a feed is the least stable point in the whole system, because new posts are prepended there constantly. Walk the drift concretely. The user loads page 1 (offset 0, posts A–T) and scrolls. While they read, three new posts are prepended at the top. Now they hit the bottom and the client asks for page 2 — ?offset=20&limit=20: 

TIME 0 — user loads page 1:            server list = [ A B C D … T | U V W X … ]
         offset 0, limit 20  → returns   A B C D … T   (positions 0–19)
         user has now seen:              A B C D … T

TIME 1 — 3 new posts arrive at the top: server list = [ N1 N2 N3 A B C … R | S T U … ]
                                                          ↑ everything shifted DOWN by 3

TIME 2 — user scrolls to bottom, asks for page 2:
         offset 20, limit 20  → "skip the first 20, take 20"
         but positions 20–39 are now      R S T U V …
                                          ↑↑↑ R, S, T were positions 17-19 before —
                                              the user ALREADY SAW them. DUPLICATES.
         and if posts had been DELETED instead, positions would shift the other way
         and the user would SKIP posts they never saw. GAPS.

That's the drift: offset addresses items by position from the top, and inserts/deletes above the read window change every position below them, so the second page overlaps (duplicate posts) or jumps (missing posts). On an infinite feed with live prepends, it's not a rare edge case — it's the steady state.

A cursor addresses items by a stable pointer instead of a count. The cursor c_3 encodes "the position after post T" — usually T's (createdAt, id), encoded — and "give me 20 after T" returns the same 20 posts no matter how many items were inserted above T. The pointer is anchored to an item, and items don't move relative to themselves. New posts prepended at the top simply don't affect a query that says "after T." That's why the normalized store keeps nextCursor next to feed.ids: the cursor is the "where am I" bookmark, and it survives anything happening above it.

Offset (?offset=40&limit=20)Cursor (?cursor=c_3&limit=20)
Addresses items byPosition counted from the topA stable pointer to a specific item
New item prepended mid-scrollDrifts — positions shift, next page shows duplicates (or gaps on delete)Stable — "after T" is unaffected by inserts above T
Deep-page DB costSlow — the DB scans and discards the skipped rowsFast — an indexed seek straight to the cursor
Jump to an arbitrary pageTrivial (page 7 = offset 120)Not supported — only next / prev
Best forFixed, page-numbered tables (admin lists, search results with page numbers)Infinite scroll, feeds, any live or reordering list

The one-liner to say: "Offset pagination addresses items by position from the top, and the top of a feed is where new posts land — so by the time the user scrolls to page 2, every position has shifted and they see duplicates, or gaps if something was deleted. A cursor points at a stable item — 'give me 20 after this post' — so it's immune to inserts above it, and it's an indexed seek instead of scan-and-discard at deep pages. I'd only use offset for a fixed, page-numbered table where jumping to page 7 is a real feature and the data isn't shifting underneath."

Two subtler cursor gotchas worth pre-empting, because they're the interviewer's next question. First: the cursor must be built on a stable, unique sort key — (createdAt, id), not createdAt alone, or two posts with the same timestamp can straddle the page boundary and one gets skipped. Second: new posts still don't appear in an already-loaded feed on their own — cursor pagination only fixes drift while paging forward; surfacing live items at the top is a separate mechanism (the "12 new posts" pill that prepends buffered ids), which is exactly why we modelled it as its own state in the Architecture step.
Tie the two interfaces together: "There are two contracts. The network API I shape to the component: side-load users so I can normalize, cursor-paginate so the feed doesn't drift as new posts arrive, and make the like idempotent so a double-tap is safe. The component API I keep presentational — PostActions takes data as props and emits onLike, knowing nothing about fetching — so it's reusable on the detail page and testable in isolation. And I'd point out the network shape and the store shape are the same decision: the API side-loads because the client normalizes." Naming both interfaces and connecting the wire format back to the cache is the move that reads as a senior frontend engineer rather than a backend engineer visiting.

O — Optimizations

This is where the interview really happens. Optimizations is the deep-dive and the biggest scoring bucket, and it's where seven years of shipping frontend pays off, because these are problems you've actually hit. You will not have time to cover everything, so the skill is to menu the axes, then go deep on the one or two the interviewer picks. Say the menu out loud:

AxisFor a feed, that meansTriggered by the follow-up
Runtime / renderingList virtualization (the marquee), memoization, stable keys, colocated state"It's an infinite list — how does it stay smooth at 5,000 posts?"
Network & perceived perfOptimistic likes, SWR caching, prefetch the next page, cancel stale requests"What happens on slow 3G / when the like request fails?"
Load performanceCode-split routes, SSR/stream first content, image priority, Core Web Vitals"Why is first content slow / the bundle large?"
Images & layout stabilityLazy loading, reserved dimensions (no CLS), responsive srcset, skeletons"The layout jumps as images load — and images are your payload"
AccessibilitySemantic <article>/<button>, keyboard, aria-live for new posts, focus in a virtualized list"How does this work with a screen reader / keyboard only?"

For a feed the interviewer almost always steers to virtualization or optimistic updates — so those get their own deep-dive sections below, followed by image loading and perceived performance because images are the feed's payload. But name the menu first: it proves you know the whole surface even when you only demo part of it, and it hands the interviewer the wheel.

Don't list all five axes exhaustively — you'll run out of clock and cover everything shallowly. And don't optimize speculatively. The failure mode is a candidate who sprinkles useMemo everywhere and adds a Web Worker nobody needs. Virtualization only matters past a few hundred rows; memoizing a cheap component can cost more than it saves. Menu the axes, let the interviewer pick, and go deep exactly there.
The framing that shows you optimize from data, not vibes: "I wouldn't add any of this blind — I'd measure first with the React Profiler and Web Vitals, and in a feed the bottleneck is almost always images and bundle size long before it's render logic. So my first two fixes are lazy-loading images with reserved dimensions and code-splitting; virtualization comes in the moment the list is long enough that DOM node count dominates. That ordering — cheapest, highest-impact fix first, verified by measurement — is the same 'the bottleneck is rarely where you guess' instinct as the backend round." Hearing that in a frontend round tells the interviewer you've optimized real apps, not memorized a technique list.

Deep dive: infinite scroll & list virtualization

The instant you say "infinite scroll," the interviewer is waiting to hear "virtualization." These are two separate mechanisms and strong candidates keep them straight: infinite scroll is about the network (fetch the next page as the user nears the bottom), and virtualization is about the DOM (render only the rows on screen). You need both, and conflating them is a common tell.

Infinite scroll — the sentinel, not a scroll handler

To load more, you need to know when the user is near the bottom. The naive answer is a scroll listener computing scrollHeight − scrollTop − clientHeight on every scroll event — which fires dozens of times a second and thrashes layout. The modern answer is an IntersectionObserver sentinel: an empty <div> just past the last row that the browser tells you about when it enters the viewport, off the main thread, with no scroll math. 

// LoadMoreSentinel — fires fetchNextPage when it scrolls into view.
function LoadMoreSentinel({ onReach, hasNextPage }) {
  const ref = useRef(null);
  useEffect(() => {
    if (!hasNextPage) return;
    const io = new IntersectionObserver(
      ([entry]) => { if (entry.isIntersecting) onReach(); },
      { rootMargin: "600px" },   // fire BEFORE it's visible → next page is ready by
    );                              // the time the user gets there. Prefetch, not react.
    io.observe(ref.current);
    return () => io.disconnect();
  }, [onReach, hasNextPage]);
  return <div ref={ref} aria-hidden="true" />;
}

The rootMargin: "600px" is the senior touch — you trigger the fetch a screen before the sentinel is actually visible, so the next page is loading (or loaded, from cache) by the time the user scrolls to it and the feed never shows a spinner. Pair it with React Query's useInfiniteQuery, which manages the pages array and threads nextCursor through for you.

Virtualization (windowing) — render 20 nodes for a 10,000-item list

Here is the core insight, and you should say it in exactly these terms: it's the number of DOM nodes, not the length of your data array, that kills the browser. The browser must lay out, paint, and keep in memory every node on the page; a post is 30–50 nodes, so 500 posts is ~20,000 nodes and the main thread chokes on layout and garbage collection. Virtualization decouples how many items exist from how many are rendered.

You keep the full data array in memory (it's cheap — a few hundred KB of JSON), but you only mount the ~20 rows currently in the viewport, plus a small overscan buffer above and below so fast scrolling doesn't flash blank. As the user scrolls, you unmount rows leaving the viewport and mount rows entering it — recycling a constant handful of DOM nodes. To keep the scrollbar honest, a single tall spacer element (or top/bottom padding) reserves the full scroll height as if all rows were present. 

  data array: 10,000 posts in memory (cheap)     what's actually in the DOM:

     ┌─ post 0                                    ┌──────────────────────┐
     │  post 1        (not rendered)              │  (spacer: height of  │
     │  …                                         │   rows 0…N, ~0 nodes)│
     │  post 4,981  ┐                              ├──────────────────────┤
     │  post 4,982  │ overscan (buffer)            │  post 4,981  overscan│
     ├─ post 4,983  ┐                              │  post 4,982          │
     │  post 4,984  │                              │  post 4,983 ┐        │  ← ~20 real
     │  …           │ VIEWPORT (visible ~20 rows)  │  …          │ visible │    FeedItem
     │  post 5,002  ┘                              │  post 5,002 ┘        │    nodes
     │  post 5,003  ┐ overscan (buffer)            │  post 5,003  overscan│
     │  post 5,004  ┘                              ├──────────────────────┤
     │  …                                          │  (spacer: height of  │
     └─ post 9,999   (not rendered)                │   rows below)        │
                                                   └──────────────────────┘
   DOM node count is CONSTANT (~30 rows) no matter how long the data grows.

You don't hand-roll this in an interview — name react-window / @tanstack/react-virtual and describe what they do. But you must be able to explain the one genuinely hard part, because it's the follow-up: variable row heights. A fixed-height list is easy — row i sits at i × rowHeight. A feed isn't fixed-height: a post with three lines of text and no image is short; one with a photo is tall. So the virtualizer must measure each row after it renders and cache its height, estimating unmeasured rows and correcting the total scroll height as real measurements come in. react-virtual's measureElement does exactly this. Saying "variable heights are the hard part, and I'd let the virtualizer measure-and-cache rather than guess" is the detail that separates someone who's used windowing from someone who's only read about it.

Virtualization quietly breaks three things you must address or the interviewer will. (1) Ctrl-F / in-page find and SEO — off-screen rows aren't in the DOM, so browser find and crawlers can't see them; fine for an authed feed, a real problem for public content (mitigate with SSR of the first page). (2) Accessibility & focus — a screen reader and the Tab key can only reach mounted rows, and if focus is inside a row that unmounts as you scroll, focus is lost; you manage this with the virtualizer's a11y support and by not unmounting a focused row. (3) Anchoring / scroll jump — when a row's measured height differs from its estimate, or an image loads and grows a row, everything below shifts; the virtualizer must anchor to the current top row and compensate, or the feed janks under the user.
Say the decoupling out loud, then bound it: "The thing that kills a long list is DOM node count, not array length, so I virtualize — keep all the data in memory but only mount the ~20 visible rows plus overscan, and recycle nodes as they scroll. The hard part for a feed specifically is variable row heights, so I'd use a virtualizer that measures each row and caches it rather than assuming a fixed height. And I wouldn't reach for it at all under a few hundred rows — below that the DOM is fine and virtualization just adds scroll-jump and a11y complexity for nothing. It's a tool I turn on when node count actually dominates, not a default." Naming when not to virtualize is what makes it read as judgment rather than a memorized keyword.

Deep dive: optimistic updates & rollback

A like that waits for a server round trip before the heart fills feels broken, especially on a slow network. The fix is the optimistic update: update the UI immediately from the client as if the request had already succeeded, fire it in the background, and roll back if it fails. This is a top-three thing to raise for a feed, and it's why the like state was modelled as an isolated, normalized cache entry back in the Architecture and Data steps — so the toggle is cheap, local, and trivially reversible.

The pattern: snapshot → mutate → settle

Every correct optimistic update is the same three-beat shape, and the middle beat — snapshotting the previous value so you can restore it — is the one candidates forget. You can't roll back to a value you didn't save. 

// Optimistic like with React Query. The shape is: onMutate (snapshot + apply),
// onError (restore the snapshot), onSettled (re-sync with the server's truth).
const likeMutation = useMutation({
  mutationFn: (postId) => api.put(`/posts/${postId}/like`),

  onMutate: async (postId) => {
    // 1. Cancel in-flight refetches so they can't overwrite our optimistic write.
    await queryClient.cancelQueries({ queryKey: ["post", postId] });

    // 2. SNAPSHOT the current value — this is the rollback token. Don't skip it.
    const previous = queryClient.getQueryData(["post", postId]);

    // 3. Apply the optimistic change to the ONE normalized entry.
    queryClient.setQueryData(["post", postId], (p) => ({
      ...p, likedByMe: true, likeCount: p.likeCount + 1,
    }));

    return { previous };   // hand the snapshot to onError
  },

  onError: (_err, postId, ctx) => {
    // 4. ROLL BACK to the snapshot, then toast so the user knows it didn't stick.
    queryClient.setQueryData(["post", postId], ctx.previous);
    toast("Couldn't like that — try again");
  },

  onSettled: (_data, _err, postId) => {
    // 5. Reconcile with the server's authoritative count (someone else may have
    //    liked it too). Invalidate → background refetch → cache holds the truth.
    queryClient.invalidateQueries({ queryKey: ["post", postId] });
  },
});

Because the store is normalized, the optimistic write targets one entry — post:p1 — and every view referencing it (the feed row, the detail page, a "liked posts" list) updates together. If you'd denormalized, you'd have to find and patch the same post in every list, and miss one, and it would drift. This is the payoff of the D step made concrete: optimistic updates are easy on a normalized cache and painful on a denormalized one.

Why cancel, and why re-sync

  • Cancel in-flight queries (onMutate). If a background feed refetch is already in flight when you like, it could land after your optimistic write and clobber it with stale pre-like data. Cancelling first closes that race.
  • Re-sync on settle (onSettled). Your optimistic +1 assumed nothing else changed, but another user may have liked the same post in the same second. Invalidating after the mutation lets a background refetch replace your guess with the server's authoritative count — the client's +1 is a placeholder, the server's number is the truth.

When NOT to be optimistic

Optimism is a bet that the request will succeed, so use it where that bet is safe and a rare rollback is harmless — likes, follows, bookmarks, reordering. Don't use it, or make the pending state explicit, when a wrong-then-corrected value would mislead or cost something: anything money-shaped (tutorial 13's ledger is the opposite philosophy — there you'd never show success you can't guarantee), a post that might be rejected by content moderation, or a payment. The rule of thumb to say: "optimistic where the failure is rare and the rollback is cheap; pessimistic where being briefly wrong is worse than being briefly slow."

The two failures interviewers probe. Forgetting the snapshot — candidates apply the change and set up an onError that "reverts" by subtracting 1, which is wrong if the request that failed was itself a double-fire or the value changed underneath; always roll back to the captured previous value, not by inverting the operation. And the flicker on re-sync — if onSettled refetches and the server returns the same number your optimistic write already showed, a naive implementation flashes a spinner or re-renders a jump; invalidate quietly in the background (stale-while-revalidate) so the already-correct UI doesn't visibly churn.
The line that shows you've shipped this: "The optimistic update is three beats — snapshot the current value, apply the change to the one normalized entry, and on error roll back to the snapshot with a toast; on settle I invalidate to reconcile with the server's authoritative count, because my +1 was a placeholder, not the truth. The reason it's clean is the normalized cache: I patch post:p1 once and every view updates. And I'd only do it where the bet is safe — a like, not a payment. For money I want the opposite: never show success I can't guarantee." Naming the snapshot, the reconcile, and the where-not-to in one breath is a complete, senior answer.

Deep dive: image loading & perceived performance

The estimation in the Requirements step already told you where the bytes are: metadata is ~30 KB a page, images are 2–6 MB. Images are the feed's payload, so image loading is the feed's real performance problem — and it's also the biggest source of the janky, lurching experience that makes a feed feel cheap. This section is about two things: getting images on screen efficiently, and making the whole feed feel fast even when the network isn't.

Layout stability — reserve the space before the image loads

The single worst feed bug, and the one with its own Core Web Vitals metric (CLS — Cumulative Layout Shift), is content jumping as images load. A post renders with a zero-height image, text flows up to fill the gap, then the image arrives, pushes everything down, and the user — who was reading — loses their place or taps the wrong thing. The fix is to reserve the image's box before it loads by giving the browser the aspect ratio up front: 

// The server sends width/height (or an aspect ratio) with each post's media.
// The browser reserves the exact box, so text never reflows when the image arrives.
<img
  src={post.mediaUrl}
  width={post.mediaWidth} height={post.mediaHeight}   // intrinsic size → reserved box, CLS = 0
  loading="lazy"                                     // don't fetch until near the viewport
  decoding="async"                                    // decode off the main thread
  srcSet={`${post.mediaUrl}?w=400 400w, ${post.mediaUrl}?w=800 800w`}
  sizes="(max-width: 600px) 400px, 800px"           // send phones the small variant
  alt={post.altText}                                  // a11y — real alt text, not ""
/>

Every attribute there is a decision worth naming. width/height (or a CSS aspect-ratio) reserve the box and kill CLS. loading="lazy" means below-the-fold images don't compete for bandwidth with the ones the user can actually see. srcset/sizes ship a phone a 400px-wide image instead of the 800px desktop one — often the biggest single byte saving on mobile, where bytes are latency. And modern formats (AVIF/WebP with a fallback) cut image weight 30–50% for free.

Perceived performance — feeling fast beats being fast

Perceived performance is the discipline of managing the user's perception of speed, which is often a bigger lever than raw milliseconds. The techniques:

  • Skeletons over spinners. A gray placeholder in the shape of a post tells the user "content is coming and here's roughly what," which feels faster and more stable than a spinner — and because the skeleton has the post's dimensions, it doubles as CLS insurance. A spinner says "wait"; a skeleton says "almost there."
  • Blur-up / LQIP placeholders. Show a tiny blurred thumbnail (a few hundred bytes, often inlined as a data URI in the post JSON) instantly, then swap in the full image when it loads. The user sees something real in the image's box immediately instead of a blank rectangle.
  • Prefetch the next page. The rootMargin on the sentinel already loads the next page of data before the user reaches it; do the same for images — the feed should feel bottomless because the next screen is always already there.
  • Optimistic interactions. The instant like from the previous section is perceived performance — it removes the round trip from the user's experience of the tap.
  • Stable, jank-free scroll. Reserved image boxes + virtualization anchoring mean the content under the user's thumb never jumps. A feed that doesn't shift feels higher quality than one that's technically faster but lurches.

The through-line: the user doesn't have a stopwatch, they have a feeling. A feed that shows a skeleton in 100 ms, a blurred image immediately, never shifts layout, and responds to a tap instantly feels faster than one that's a bit quicker on raw metrics but flashes blank rectangles and jumps around. Optimize the perception, backed by the real numbers where they matter (images, bundle).

Two image mistakes that read as inexperience. Lazy-loading the above-the-fold images too — loading="lazy" on the first two or three posts delays your LCP (Largest Contentful Paint) because the browser waits to fetch the very image that is the hero content; eager-load or fetchpriority="high" the first screen, lazy-load the rest. And not sizing images at all — shipping a 2000px original to a 400px-wide phone slot wastes 90% of the bytes and the decode time, then downscales it anyway. Serve responsive variants; the CDN or an image service (or Next/Image) generates them.
Close the whole optimizations arc by connecting it back to the estimate: "I'd anchor image work in the numbers — a feed page is ~30 KB of JSON and several MB of images, so images are my payload and my perceived-speed problem in one. So: reserve every image box to keep CLS at zero, lazy-load below the fold but eager-load the first screen so I don't hurt LCP, ship responsive AVIF/WebP variants so a phone gets phone-sized bytes, and cover the loading gap with a skeleton and a blur-up placeholder. The user judges the feed on whether it flashes and jumps, not on a Lighthouse number — so I optimize the perception, backed by the byte math where it counts." Grounding perceived-performance polish in the payload estimate is the move that ties the case study together.

Interview Q&A

1. Walk me through how you'd design a news feed UI.

Name the round first: "I'll treat this as frontend system design — client architecture, state, data flow, performance — and assume the feed backend exists behind a paginated API." Then RADIO. R: view + infinite scroll + like + new-posts affordance; cut compose/comments to nav targets; estimate DOM and payload weight (images dominate, node count grows unboundedly). A: component tree with a virtualized list and an isolated PostActions leaf; state at the lowest ancestor that needs it. D: feed is server state in a normalized, id-keyed cache, order/cursor separate from entities. I: props API (data in, onLike out) plus a cursor-paginated, side-loaded network API. O: menu the axes, then deep-dive virtualization and optimistic likes, plus image loading.

The whole answer is back-loaded onto O deliberately — that's where the points are — and every step names a trade-off as it's made.

2. Cursor vs offset pagination — which, and why exactly does offset drift?

Cursor, always, for a feed. Offset addresses items by position counted from the top ("skip 40, take 20"), and the top of a feed is where new posts are constantly prepended. So between loading page 1 and scrolling to page 2, inserts above the read window shift every position below them: offset=20 now points at rows the user already saw (duplicates), or — if items were deleted — past rows they never saw (gaps).

A cursor addresses items by a stable pointer — "give me 20 after post T," where the cursor encodes T's (createdAt, id). Items don't move relative to themselves, so inserts above T don't affect the query; it's also an indexed seek instead of scan-and-discard at deep pages. Offset only wins for fixed, page-numbered tables where "jump to page 7" is a real feature and the data isn't shifting underneath.

3. There are 10,000 posts in the feed — what do you do?

Virtualize (window) the list. The killer isn't the array length — it's the DOM node count; a post is 30–50 nodes, so 10,000 posts is hundreds of thousands of nodes and the browser chokes on layout, paint, and GC. Keep all the data in memory but mount only the ~20 rows in the viewport plus a small overscan buffer, recycling nodes as the user scrolls, with a spacer reserving the full scroll height. Name react-window / @tanstack/react-virtual.

The follow-up is variable row heights (a text post vs a photo post), which is the genuinely hard part — the virtualizer must measure each row after render and cache it, correcting the total height as measurements arrive. And virtualization breaks Ctrl-F, SEO, and focus for off-screen rows, so I'd mention SSR-ing the first page and managing focus deliberately.

4. Infinite scroll and virtualization — aren't those the same thing?

No, and keeping them straight is a signal. Infinite scroll is a network concern: fetch the next page as the user nears the bottom — I'd use an IntersectionObserver sentinel with a rootMargin so the page prefetches before the user arrives, not a scroll handler that thrashes layout. Virtualization is a DOM concern: render only the visible rows regardless of how many are loaded.

You need both. Infinite scroll without virtualization loads 5,000 posts into 5,000 DOM nodes and janks; virtualization without infinite scroll windows a list you can't grow. Together: fetch pages lazily via the sentinel, keep them in a normalized cache, and window the render.

5. How does an optimistic like work, and how do you roll it back?

Three beats. Snapshot the current cache value for that post (the rollback token — the beat people forget), apply the optimistic change to the one normalized entry (likedByMe=true, likeCount+1) so the UI updates instantly, and on error roll back to the snapshot with a toast. Also cancel in-flight refetches in onMutate so a background fetch can't clobber the optimistic write, and on settle invalidate to reconcile with the server's authoritative count (someone else may have liked it too — my +1 was a placeholder).

It's clean specifically because the cache is normalized: I patch post:p1 once and the feed row, the detail page, and any other list referencing it all update together. And I'd only be optimistic where the failure is rare and rollback is cheap — a like, not a payment.

6. Why normalize the client cache, and what breaks if you don't?

So each post has one source of truth. If the same post is copied into the feed list, the detail view, and a "liked" list, then liking it updates one copy and the others drift — the feed shows 13 likes, the detail page shows 12. Store each entity once keyed by id, keep lists as arrays of ids, and an update in one place is seen everywhere. Keep the order and cursor (feed.ids, nextCursor) separate from the entities (posts) so you can append a page, prepend a live post, or dedupe without touching stored objects.

The concrete payoff: optimistic updates and cache invalidation become one-line writes on a normalized store and a find-and-patch-every-list nightmare on a denormalized one. Apollo normalizes by id automatically; React Query is document-cache by default, so you either normalize yourself or write to each relevant key.

7. New posts arrive while the user is scrolling. What do you do with them?

Don't inject them mid-scroll — that shifts content under the user's thumb and they lose their place (and it's the exact drift offset pagination causes, self-inflicted). Buffer the new posts' ids aside and surface a "12 new posts" pill; when the user taps it, prepend the buffered ids to feed.ids and scroll to top. That keeps the reading position stable and makes showing new content a user choice.

How the new posts arrive is a separate decision: WebSocket or SSE for true real-time, or polling on an interval for "good enough" freshness. For a feed, near-real-time via poll or SSE is usually plenty, and I'd say so rather than reaching for a socket reflexively. For a screen reader I'd announce "12 new posts available" via an aria-live="polite" region so it isn't a silent visual-only change.

8. Where should each piece of the feed's state live?

Lowest common ancestor that needs it, and no higher. The posts + cursor are server state — a server-cache library (React Query/SWR), not the component and not Redux. The like toggle lives in the normalized cache entry so all views agree and the optimistic write is isolated. Scroll/window position is local to the virtualized list. The buffered "new posts" sit in the page or aside in the cache. User/theme is app-global Context.

The observation to voice: almost nothing here is truly global. Candidates who put the whole feed in one Redux store have misread where each kind of state wants to live — most of a feed's "state" is a server cache, and the rest is local view state.

9. Images make the feed feel slow and janky. How do you fix it?

Images are the payload — a page is ~30 KB of JSON and several MB of images — so this is the real performance problem. Kill layout shift (CLS) by reserving each image's box with its intrinsic width/height or an aspect ratio before it loads, so text never reflows when the image arrives. Lazy-load below the fold (loading="lazy") but eager-load the first screen so you don't hurt LCP. Ship responsive srcset variants in modern formats (AVIF/WebP) so a phone gets phone-sized bytes.

Then cover the gap with perceived-performance tricks: a skeleton in the post's shape (which is also CLS insurance) and a blur-up LQIP so the user sees something real in the box immediately. The user judges the feed on whether it flashes and jumps, not on a raw metric.

10. The interviewer says "make it faster." Where do you start?

Ask what "faster" means — load or runtime — then measure before touching anything. For a feed the honest answer is that the bottleneck is almost always images and bundle size long before render logic, so I'd instrument with the React Profiler and Web Vitals rather than sprinkle useMemo. Load fixes: code-split routes, lazy-load below-the-fold images/assets, SSR the first screen for fast first content. Runtime fixes: virtualize the long list, memoize genuinely expensive subtrees, keep high-frequency handlers off the main thread.

The senior move is refusing to optimize speculatively — virtualization is pointless under a few hundred rows, memoizing a cheap component costs more than it saves. Cheapest high-impact fix first, verified by measurement. Same instinct as the backend round: the bottleneck is rarely where you'd guess.

11. How do you keep a virtualized, infinite feed accessible?

Start with semantics: each post is an <article>, actions are real <button>s (keyboard operability and the right role for free). The feed-specific hazards are two. Virtualization hides rows from the accessibility tree and from Tab — off-screen rows aren't in the DOM, so a screen reader and keyboard can only reach mounted rows, and if focus is inside a row that unmounts on scroll, focus is lost; use the virtualizer's a11y support and avoid unmounting a focused row. New posts arriving is a silent visual change — announce it through an aria-live="polite" region ("12 new posts available") so it isn't invisible to a screen reader.

Then the baseline: visible focus rings, sufficient contrast, real alt text on images, prefers-reduced-motion for any transition. Naming the virtualization-vs-a11y tension unprompted is a strong senior signal — most candidates don't realize windowing breaks the accessibility tree.

12. This is a frontend round — should you ever talk about the backend at all?

Briefly, as a contract, and only to justify a client decision. You should define the feed endpoint's shape (cursor-paginated, side-loaded users, idempotent like) because those choices are what make your client design work — that's the "I — Interface" step. What you don't do is scale it: no fan-out, no timeline caching, no sharding, unless the interviewer explicitly asks (then tutorial 10 is your script).

The tell you've got the balance right: you can say "I'd want the API to cursor-paginate and side-load users, and I know how I'd build that service, but I'll treat it as given and spend my time on the client." That shows range without spending your best minutes on the wrong round.

What to remember

The frame

  • It's the frontend round. Name it in your first sentence and treat the feed backend as a one-line API contract. You built the fan-out backend in tutorial 10 — don't rebuild it here; the client is the whole subject.
  • Clarify the experience, not the scale. No DAU-to-QPS. The only estimation that earns its place is DOM weight and payload weight — which tells you images are the payload and node count grows unbounded.

RADIO for a feed

  • R. Functional = view, infinite scroll, like, new-posts pill (they enumerate your components). Non-functional = devices, list size, real-time, network, perf budget, a11y. Scope and cut out loud.
  • A. Component tree carved by responsibility and re-render rate; isolate the like in a small leaf. State at the lowest ancestor that needs it. Data down as props, events up as callbacks, cache as the pivot.
  • D. The feed is server state = a cache (React Query/SWR), not Redux. Normalize: one entity per id, lists hold ids, order/cursor kept separate from entities. Derive, don't store.
  • I. Two contracts — the props API (data in, intent out) and the network API (side-load users, idempotent like, and cursor pagination).
  • O. Menu the axes, then deep-dive where they point.

The four techniques the interviewer is fishing for

  • List virtualization. DOM node count, not array length, kills the browser. Mount ~20 visible rows + overscan, recycle on scroll, reserve full scroll height. Variable row heights = measure-and-cache. Don't virtualize under a few hundred rows.
  • Cursor pagination. Offset addresses items by position from the top, which shifts as posts are prepended → duplicates (or gaps on delete). A cursor points at a stable item — "20 after post T" — immune to inserts above it, and an indexed seek at depth.
  • Optimistic updates. Snapshot → apply to the one normalized entry → roll back to the snapshot on error; cancel in-flight fetches, reconcile on settle. Clean because the cache is normalized. Likes yes, money no.
  • Normalized cache. One copy per entity so the feed and detail view never drift; it's what makes optimistic updates and invalidation one-line writes.

Images & perceived performance

  • Reserve every image box (CLS = 0), lazy-load below the fold but eager-load the first screen (LCP), ship responsive AVIF/WebP variants. Skeletons and blur-up placeholders cover the loading gap. The user judges the feed on flashing and jumping, not on a metric.
  • Measure before optimizing — the bottleneck is images and bundle size long before render logic. Same instinct as the backend round.
The one line that ties the page together: a news feed is four techniques and one contract. Virtualize the render, cursor-paginate the fetch, make the like optimistic, normalize the cache — and put the server behind a paginated endpoint you define in thirty seconds. Everything else is narrating the trade-offs you already make by instinct. Hit those four with the why and not just the library name, name what each costs, and you've given the answer this round is graded on.
