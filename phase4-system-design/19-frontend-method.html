19 — The Frontend System Design Method (RADIO)

A different round, graded on different things. Draw a load balancer here and you fail it.

Why this matters in the interview

Everything in Block B taught you to answer "how do we serve ten million users?" — estimation, sharding, replication, consensus. Then a recruiter schedules you for a frontend system design round, you walk in, draw a load balancer and three database replicas, talk about QPS for fifteen minutes, and you fail. Not because the answer was wrong, but because you answered a different question than the one being asked. The frontend round is not the distributed round with React sprinkled on top. It is a genuinely separate interview with its own rubric, its own vocabulary, and its own failure modes — and the single most common way strong backend-leaning candidates lose it is by treating it as the round they studied for. This page exists so that never happens to you. As a React dev with seven years in the browser, you are the favourite to win this round; you just have to recognize that you're in it.

The good news is that the frontend round has a method exactly as repeatable as the six-step backend method from tutorial 01, and it has a name candidates actually use: RADIO — Requirements, Architecture and components, Data model, Interface and API, Optimizations. It is the frontend analogue of "requirements → estimation → API → data model → high-level design → bottlenecks," and it plays the same role: it stops you freezing at the whiteboard, it makes sure you cover what's graded, and it lets you spend your energy on judgment instead of on remembering what comes next. Pages 20–23 — the feed UI, the typeahead, the design system, the collaborative editor — are all worked transcripts that walk this exact loop. Learn RADIO here, and those four pages become variations on a shape you already own. The rest of this tutorial teaches each letter as a concept: what it is, why the interviewer put it there, how to actually do it, and the one sentence to say out loud that tells them you've done this before.

The round is different — and so is the grading

Start with the thesis, because it reframes everything that follows: the frontend system design round grades the browser half of the system that the backend round deliberately hand-waves. When a backend interviewer draws a box labelled "client," that box is a rounding error to them — a thing that issues requests. The frontend interviewer zooms all the way into that box and asks: what components is it made of, where does its state live, how does data flow through it, what happens on a slow network, what happens with a screen reader, what happens when the user clicks the like button twice before the first request returns. None of those questions have a QPS number in them. That is the whole shift.

What each round is actually about

The two rounds share a skeleton — clarify, design, deep-dive, discuss trade-offs — but almost nothing else. Where the backend round spends its scoring budget on scale and correctness under distribution, the frontend round spends it on component design, state management, data flow, rendering performance, and accessibility. Sharding, consensus, and read-replica lag simply do not appear. If you catch yourself reaching for them, you've drifted into the wrong interview.

RADIO is the frontend six-step method

You already internalized a five-to-six-step loop for the backend round in tutorial 01. RADIO is the same idea, re-pointed at the browser. Map them side by side and the round stops feeling foreign — you're doing the same motions (clarify the problem, sketch the structure, define the data, define the contracts, then make it good), just grading different artifacts at each step.

The single most common way to fail this round is to answer the backend round instead. The tell is when a candidate designs a news feed UI and spends ten minutes on how the feed service shards posts, fans out to followers, and caches timelines in Redis. All true, all irrelevant — the interviewer wanted to know how the feed component virtualizes a 10,000-item list, how it paginates without the scroll jumping, and how it shows an optimistic like. You've spent your best minutes proving you can do the other interview. If the prompt is a screen — "design Twitter's feed," "design an autocomplete," "design a photo grid" — assume it's the frontend round until told otherwise, and put the server behind a one-line API contract you can define in thirty seconds and move on.
You already have the instincts this round rewards; you're just not used to narrating them. Every day you decide whether a piece of state belongs in a component, in context, or in a server-cache library; whether to debounce an input; whether a list needs virtualizing; whether a control is keyboard-reachable. The frontend round is that daily judgment, spoken out loud and defended. The work of this page is teaching you the vocabulary and the order (RADIO) so the judgment you already have comes out as a structured 45-minute answer instead of a pile of correct-but-unsequenced observations.
Open the round by naming it: "Before I design anything — I'm reading this as a frontend system design problem, so I'll focus on the client architecture, state, data flow, performance and accessibility, and treat the backend as an API contract I'll define but not scale. Shout if you actually want me to design the service too." This does three things a junior candidate skips: it proves you know there are two different rounds, it locks scope so you don't waste minutes on the wrong half, and it hands the interviewer an explicit chance to redirect you. Thirty seconds, and you've already signalled that you've been in this room before.

R — Requirements

What it is: the same first move as the backend round — pin down what you're building before you build it — but the questions you ask are about the experience, not the scale. You still split functional from non-functional. You still scope ruthlessly and say what you're cutting. What changes is the content of both lists.

Why it exists: the prompt is deliberately vague ("design a news feed"), and a feed for Twitter is a different build from a feed for a photo-sharing app or an enterprise activity stream. The interviewer is watching whether you define the surface before you draw components. Skipping this is the frontend equivalent of coding before reading the ticket, and it reads exactly as junior here as it does in the backend round.

Functional requirements — for a UI, these are user-visible behaviours

Not endpoints — interactions. What can the user see and do on this screen? For a feed: view a scrollable list of posts, load more as they scroll, like/unlike a post, compose a new post, open a post's comments. Enumerate the interactions and you've enumerated the components you'll need to design. This is also where you cut: "I'll build viewing, infinite scroll, and liking; I'll stub compose and comments as navigation targets rather than designing them, unless you'd rather I go deep on one of those." Naming the cut is the senior move — same as backend.

Non-functional requirements — this is where the round announces itself

The backend round's non-functionals are availability, latency, consistency, durability. The frontend round's are a completely different list, and reciting it unprompted is one of the fastest ways to signal you know which interview you're in.

Non-functionalThe question to askWhy it changes the design

How you do it: ask three or four of these, don't recite all seven — you're demonstrating that you know the axes, not filling a form. Then state your assumptions and move: "I'll assume responsive desktop and mobile web, WCAG AA, works on 3G with skeletons, English-only for now, fetch-on-load with pull-to-refresh rather than live push. Good?" That single sentence is the non-functional spec, and it took fifteen seconds.

Do not turn Requirements into a numbers exercise out of backend habit. There is usually no DAU-to-QPS arithmetic here, and grinding one out wastes the minutes you need for Architecture and Optimizations. The one estimation that does earn its place is list size and payload weight — "if a feed page is 20 posts at ~2 KB of JSON each plus images, that's a 40 KB metadata payload and the images dominate, so image loading is my real performance problem" — because that number actually drives a design decision. Estimate only what changes what you'll build.
Say the accessibility bar out loud in Requirements, before anyone asks: "I'll target WCAG AA — full keyboard operability and correct semantics for a screen reader — and I'll fold that into each component as I design it rather than bolting it on at the end." Most candidates either forget a11y entirely or treat it as an afterthought in a corner of the whiteboard. Naming it as a requirement up front, and then actually honouring it when you get to the components, is one of the clearest senior signals available in this round — and it's a home-game for you, because you've shipped this before.

A — Architecture & components

What it is: the frontend round's version of the high-level box diagram, except your boxes are components and your arrows are data flow and events. You produce three things here — a component tree, a statement of where each piece of state lives, and the direction data flows — and this is the step that most directly tests "can this person decompose a UI," which is a top-scored dimension.

Why it exists: component decomposition is to a frontend engineer what service decomposition is to a backend engineer — it's the structural judgment the whole role rests on. Break the UI into the wrong pieces and every later decision (where state lives, what re-renders, what you can reuse) gets harder. The interviewer is watching whether you carve along the right seams: by responsibility and reuse, not by "boxes I can see on the screen."

The component tree — draw this first

Sketch the UI as a tree, top-down, the same way you'd draw services in a backend HLD. Naming the tree forces the decomposition to be explicit and gives the interviewer something concrete to push on. 

  <FeedPage>                        ← route-level container; owns data fetching
    ├─ <Composer>                    ← new-post input; local form state
    ├─ <FeedList>                    ← the virtualized scroll container
    │    ├─ <FeedItem>               ← one post (repeated ×N, virtualized)
    │    │    ├─ <PostHeader>        ← avatar, name, timestamp
    │    │    ├─ <PostBody>          ← text + media
    │    │    └─ <PostActions>       ← like / comment / share buttons
    │    └─ <LoadMoreSentinel>       ← IntersectionObserver trigger for next page
    └─ <Toast>                       ← transient errors / undo affordances

How you decide the seams. A component should have one responsibility, and you should be able to name it. FeedItem renders one post; PostActions owns the interaction buttons; the split exists because actions have their own local state (is-liking, optimistic count) and you don't want that state forcing the whole item to re-render. The two seams interviewers reward you for finding: reuse (PostActions is the same on the feed and the detail page — factor it out) and re-render isolation (state that changes often lives in a small leaf so the change doesn't ripple upward).

Where does state live? — the highest-value question in this step

This is the frontend analogue of "where does this data live" in the backend round, and it's graded just as hard. For every piece of state, you answer: how local can it be? The guiding principle you should say out loud is lift state to the lowest common ancestor of the components that need it, and no higher. State that's too global re-renders half the tree on every keystroke; state that's too local has to be awkwardly threaded back up when a sibling needs it.

StateWhere it livesWhy
Composer draft textLocal to Composer (useState)Nothing else needs it; keeping it local means keystrokes don't re-render the feed.
"Is this post liked" + countLocal to PostActions, hydrated from server dataOptimistic toggle is isolated to one leaf; a like on post 5 doesn't touch post 6.
The list of posts + pagination cursorContainer (FeedPage) or a server-cache libraryShared by the whole list; it's server state (see the D step), so ideally React Query / SWR owns it.
Current theme / logged-in userApp-global (Context / store)Read in many places, changes rarely — the exact profile Context is good at.
"Which post's comment sheet is open"Container or URLUI state that outlives a leaf; putting it in the URL makes it shareable and back-button-friendly.

Data flow — data down, events up

Draw the arrows. In a React-shaped architecture, data flows down through props and events flow up through callbacks (or dispatches). The container fetches the feed and passes each post down; a like click travels up as an onLike(postId) callback the container (or a mutation hook) handles, then the new data flows back down. Saying "unidirectional data flow" and then actually tracing one interaction end-to-end on your diagram is worth more than any amount of box-drawing. 

   server ──fetch──▶ FeedPage ──posts (props)──▶ FeedList ──post──▶ FeedItem
                        ▲                                              │
                        └──────── onLike(postId) (event bubbles up) ──┘
   (data flows DOWN as props · events flow UP as callbacks · one direction)

Container vs presentational — a useful vocabulary, not a rule

The old container/presentational split — "smart" components that fetch and own state, "dumb" components that only take props and render — is less of a hard rule since hooks (a hook lets any component be "smart"), but the vocabulary is still the clearest way to talk about the seam in an interview. Say: "FeedPage is the container — it owns data fetching and the mutation handlers; everything below it is presentational and driven by props, which keeps the leaves pure, trivially testable, and reusable on the detail page." That sentence communicates your whole architecture in one breath.

Don't decompose by visual boxes — decompose by responsibility and state. The giveaway of a shallow answer is a tree that mirrors the CSS layout (Header, Sidebar, MainColumn) with no thought to what owns which state or what re-renders when. A layout is not an architecture. Two visually identical regions can be one reusable component; one visual box can be three components because three different things change inside it at three different rates. Carve along the axes of change and reuse, and justify each seam that way.
The line that shows you think about performance structurally, before anyone says the word "slow": "I'm keeping the like state inside PostActions rather than in the container on purpose — if the liked-count lived in the container's post array, toggling one like would re-render the entire list. Colocating the volatile state in the leaf means a like re-renders exactly one button." That's the same instinct as picking a shard key so one hot entity doesn't melt one node — you're bounding the blast radius of a change — and stating it unprompted tells the interviewer you design for render cost the way a backend engineer designs for load.

D — Data model

What it is: the shape of the data as the client holds it. Not database tables — that's the backend round's data model. This is the client-side store: what entities you keep in memory, how they're keyed and related, and — the decision that dominates this step — which of your state is server state and which is client state.

Why it exists: most "frontend bugs" in a design are really data-model bugs. Duplicated data that drifts out of sync, a list and a detail view showing different like-counts for the same post, a cache you can't invalidate — all of these come from modelling the client's data badly. Getting this right is what makes the Optimizations step (caching, optimistic updates) possible instead of painful.

The split that organizes everything: server state vs client state

This is the single most important distinction in modern frontend architecture, and stating it explicitly instantly separates you from candidates who think "state management" means "put everything in Redux." The two kinds of state have opposite properties and want opposite tools.

Server state (server-cache state)Client state (UI state)
What it isData that lives on the server; the client holds a cached copyData that only ever exists in the browser
ExamplesThe feed, a user profile, comments, search resultsIs a modal open, form draft text, selected tab, theme toggle
Owned byThe server — the client's copy can go staleThe client — it is the source of truth
Key problemsCaching, revalidation, staleness, loading/error states, deduping requests, refetchJust… setting and reading it. No network, no staleness.
Right toolReact Query / SWR / RTK Query / Apollo — a server-cache libraryuseState, useReducer, Context, or a small store (Zustand/Redux)
Wrong toolHand-rolled useEffect + useState fetching, or dumping it in ReduxA server-cache library (pointless overhead)

The senior framing: server state is not really state, it's a cache — and everything you learned about caching in tutorial 06 applies directly in the browser. It goes stale. It needs a revalidation strategy (stale-while-revalidate is the default a library like SWR is literally named after). It needs loading and error states. It needs request deduplication so ten components asking for the same user don't fire ten requests. Treating server data as a cache — rather than as state you manually keep in sync with useEffect — is the whole reason React Query exists, and saying so is a strong signal.

This maps cleanly onto tutorial 06's cache patterns. A server-cache library is doing cache-aside with stale-while-revalidate for you: it serves the cached value instantly (the hit), fires a background refetch (revalidate), and swaps in fresh data when it arrives. Request deduplication is the same single-flight / request coalescing fix from the stampede section — ten components mounting at once and all wanting user:42 collapse to one network request. You already know this machinery from the backend page; the browser is just another tier of the same cache stack.

Normalize shared entities — the client-side "third normal form"

When the same entity appears in multiple places — a user who authored ten posts, a post that shows in the feed and the detail view — do not store copies. Store each entity once, keyed by id, and reference it by id elsewhere. This is exactly database normalization, moved into the client, and it exists for the same reason: to keep a single source of truth so an update in one place is seen everywhere and nothing drifts. 

// DENORMALIZED — the same author is copied into every post.
// Update the author's name and you must find and fix N copies. It WILL drift.
const bad = {
  feed: [
    { id: "p1", text: "...", author: { id: "u9", name: "Ada" } },
    { id: "p2", text: "...", author: { id: "u9", name: "Ada" } },
  ],
};

// NORMALIZED — entities stored once by id; lists hold ids and reference them.
// Update users["u9"].name in ONE place and every post re-renders correctly.
const good = {
  posts: { p1: { id: "p1", text: "...", authorId: "u9", likedByMe: false, likeCount: 12 },
           p2: { id: "p2", text: "...", authorId: "u9", likedByMe: true,  likeCount: 30 } },
  users: { u9: { id: "u9", name: "Ada", avatarUrl: "..." } },
  feed:  { ids: ["p1", "p2"], nextCursor: "c_2" },  // the ORDER lives here, separate from the entities
};

Note the separation in the normalized shape: feed.ids holds the order and the pagination cursor, while posts holds the entities. That split is what lets you append the next page (push ids) without touching stored posts, dedupe a post that appears in two lists, and update one post's like-count in exactly one place. It's the same reason the backend round keeps an index separate from the rows.

Derived state — compute, don't store

The last rule of the client data model: if a value can be computed from existing state, do not store it as its own state — derive it during render (memoized if it's expensive). "Number of unread items," "is the form valid," "the filtered visible list" are all derived. Storing them creates a second source of truth you now have to keep in sync, which is the exact drift problem normalization exists to prevent. Say "that's derived, I'll compute it with a useMemo rather than store it" and you've closed off a whole class of bugs before they're raised.

Putting server state in Redux (or any client store) is the most common data-model mistake, and it's a costly one to defend. You inherit every hard cache problem by hand: you write the loading flags, the error flags, the "is it stale," the refetch-on-focus, the dedup, the cache eviction — reimplementing React Query badly, in reducers. If an interviewer sees you reach for Redux to hold fetched API data, expect the follow-up "how do you invalidate that when it goes stale?" — and the honest answer is "I wouldn't put it there." Client stores are for client state: cross-cutting UI concerns like theme, auth session, a multi-step wizard. Server data belongs in a server-cache library.
Say the split as your opening line for this step: "I'll separate server state from client state. The feed, profiles and comments are server state — I'll model them as a normalized, id-keyed cache managed by React Query, so I get deduping, background revalidation and cache invalidation for free. Client state — open modals, the composer draft, selected tab — stays local or in a small store. Almost nothing needs to be truly global." That one paragraph tells the interviewer you understand the defining problem of modern frontend data, and it makes the Optimizations step (optimistic updates, cache invalidation) fall out naturally instead of feeling bolted on.

I — Interface & API

What it is: the contracts. And the thing to internalize — the reason this letter is worth its own step rather than being folded into Architecture — is that in a frontend design there are two interfaces, and strong candidates address both:

  • The component interface (props API) — how your components talk to each other. The props a component accepts, the callbacks it emits. This is API design pointed inward, and it's unique to the frontend round.
  • The network API — how the client talks to the server. Endpoints, request/response shapes, pagination. This is the same API-design skill as the backend round, but you're the consumer defining what you need, not the producer designing the whole service.

Why it exists: the props API is where component reusability is won or lost — a badly shaped prop interface makes a component impossible to reuse and impossible to test. The network API is where client-side performance is won or lost, because pagination shape, over-fetching, and round-trip count are decided here. Both are graded; skipping the props API is the more common miss because backend-leaning candidates don't think of "component props" as an API at all.

The component interface — props down, callbacks up

Design each reusable component's props like a public API: minimal, well-typed, and hard to misuse. The pattern is the data-flow direction from the Architecture step made concrete — data comes in as props, events go out as callbacks. 

// PostActions: a small, reusable, presentational component.
// Data in as props; the only thing it emits is an intent (onLike).
interface PostActionsProps {
  postId: string;
  likeCount: number;
  likedByMe: boolean;
  onLike: (postId: string) => void;   // event UP — parent decides what "like" means
  onComment: (postId: string) => void;
}

// It renders from props and calls back on interaction. No fetching, no store
// access — which is exactly what makes it reusable on the feed AND the detail page,
// and trivially testable: render with props, click, assert the callback fired.

Two judgment calls to voice here. First, keep the leaf presentational: PostActions takes likedByMe as a prop rather than reading a store, so it has no idea where its data comes from — that's what lets you drop it anywhere. Second, emit intent, not implementation: it calls onLike(postId), not "increment this counter and POST to this URL." The parent owns the meaning of a like. That separation is the props-API equivalent of a clean service boundary.

The network API — you're the consumer, so ask for exactly what the UI needs

Define the handful of endpoints the screen needs, with their shapes. You're not designing the whole service; you're specifying the contract you'd want your backend team to build. The senior instinct is to shape the API around the component's needs — request the fields you render and no more, and get the round-trip count down. 

// GET the feed — cursor-paginated (see below for why cursor, not offset)
GET /api/feed?cursor=c_2&limit=20
// → {
     posts: [ { id, authorId, text, mediaUrl, likeCount, likedByMe, createdAt } ],
     users: [ { id, name, avatarUrl } ],   // side-loaded → normalize straight into the store
     nextCursor: "c_3"
   }

// Like a post — the mutation behind onLike. Idempotent so a double-tap is safe.
PUT /api/posts/:id/like      // PUT not POST: setting "liked" twice == liked once
// → { likeCount: 31, likedByMe: true }

Notice the response side-loads users separately from posts — that's the normalized store shape from the D step reaching back into the API design, so the client can drop entities straight into their tables without unpacking nested duplicates. And notice PUT for the like: making the mutation idempotent means the double-tap race (user taps like twice, or the request retries) can't produce a like-count of 2. That's the same idempotency reasoning as tutorial 06, applied to a button.

Pagination: cursor, not offset — and know why

For any infinite-scrolling or frequently-changing list, use cursor-based pagination (?cursor=c_2), not offset-based (?page=3 / ?offset=40). This comes up in nearly every frontend design and the reasoning is worth having cold.

Offset (?offset=40&limit=20)Cursor (?cursor=c_2&limit=20)
How it works"Skip 40 rows, take 20""Give me 20 after this stable pointer"
New item inserted at top mid-scrollDrift — everything shifts down by one, so row 40 is now what was row 39; you re-see a post or skip oneStable — the cursor points at a specific item, unaffected by inserts above it
Deep pages performanceSlow — the DB still scans and discards the skipped rowsFast — an indexed seek to the cursor
Jump to an arbitrary pageTrivial (page 7 = offset 120)Not supported — you can only go next/prev
Best forFixed, page-numbered tables (admin lists, search results with page numbers)Infinite scroll, feeds, any real-time list

The one-liner: "Offset pagination drifts when items are inserted or deleted while the user scrolls — they see duplicates or gaps — so for a feed I'll use a cursor, which points at a stable position and is immune to inserts above it. I'd only use offset for a fixed, page-numbered table where jumping to page 7 matters." Tutorial 20 builds the whole feed around exactly this decision.

Don't skip the props API and only design the network endpoints — that's the backend-habit version of this step, and it leaves half the score on the table. The props API is where the interviewer sees whether you can build a reusable component versus a one-off. Equally, don't over-fetch: requesting a fat /user object with 40 fields to render an avatar and a name is the client-side version of a SELECT *, and on mobile it's paid for in bytes over a slow radio. Shape the request to the render.
Tie the two interfaces together out loud: "There are two contracts here. The network API I'll shape to the component's needs — side-load users so I can normalize, cursor-paginate the feed so it doesn't drift, and make the like idempotent so a double-tap is safe. The component API I'll keep presentational — PostActions takes data as props and emits onLike, knowing nothing about fetching — because that's what lets me reuse it on the detail page and unit-test it in isolation." Naming both interfaces, and connecting the network shape back to the normalized data model, is the move that reads as a senior frontend engineer rather than a backend engineer visiting.

O — Optimizations

What it is: the deep-dive, and the biggest scoring bucket in the round. This is the frontend equivalent of the backend "bottlenecks & scale" step — where the interviewer stops nodding along and starts pushing. But instead of "what breaks at 10× traffic," the pressure is "what happens on a slow network, with a huge list, with a screen reader, when a request fails." You will not have time to cover everything; the skill is to name the four axes, then go deep on the one or two the interviewer cares about.

Why it exists: a UI that's correct but slow, janky, or unusable with a keyboard is a UI that shipped and got complained about. This step is where seven years of actually-shipping-frontend pays off, because these are the problems you've genuinely hit. There are four axes, and you should say them as a menu before diving: performance (load), performance (runtime/rendering), network efficiency, and accessibility.

Axis 1 — Load performance: get to first content fast

  • Code-splitting & lazy loading. Don't ship the comment sheet, the composer's rich-text editor, and the settings modal in the initial bundle. Route-level and interaction-level React.lazy / dynamic import() so the first paint downloads only what the first screen needs. This is the frontend version of "don't do work on the critical path you don't have to."
  • SSR / streaming vs client-render. If time-to-first-content is the priority (it usually is for content sites), render on the server so the user sees posts before the JS boots, then hydrate. If it's an app-behind-a-login where interactivity matters more, CSR is fine. Name the trade-off; don't default blindly.
  • Critical assets & fonts. Preload the hero image, avoid layout-shifting web fonts (CLS), inline critical CSS. These are the Core Web Vitals levers — LCP, CLS, INP — and naming them by acronym signals you've been measured on them.

Axis 2 — Runtime/rendering: keep it smooth after it loads

  • List virtualization (windowing) — the marquee technique of this round. A feed of 10,000 posts must not put 10,000 DOM nodes on the page; the browser can't lay out or paint that, and memory balloons. Render only the ~20 rows in the viewport (plus a small overscan buffer) and recycle nodes as the user scrolls. This is mandatory for any long list and is the answer the interviewer is fishing for the instant you say "infinite scroll." react-window / react-virtual are the names to drop.
  • Avoiding needless re-renders. The colocated-state decision from the Architecture step is the first line of defense. Beyond it: memoize expensive subtrees (React.memo), memoize expensive computations (useMemo) and stable callbacks (useCallback) so children don't re-render on identity churn, and use stable keys in lists (never the array index for a reorderable list — it breaks reconciliation and state gets attached to the wrong row).
  • Keep the main thread free. Debounce/throttle high-frequency handlers (scroll, resize, keystroke); move genuinely heavy work (parsing, diffing) off to a Web Worker; batch DOM reads and writes to avoid layout thrash. The frame budget is ~16 ms at 60 fps — that's the "latency number" of this round.

Axis 3 — Network efficiency & perceived performance

  • Optimistic updates. When the user likes a post, update the UI immediately from the client and fire the request in the background; if it fails, roll back and show a toast. The interaction feels instant instead of gated on a round trip. This is a top-three thing to mention, and it's why the like-count lived as isolated leaf state in the Architecture step — so the optimistic toggle is cheap and local.
  • Caching & revalidation. The server-cache library (D step) serves cached data instantly and revalidates in the background — stale-while-revalidate. Dedupe concurrent requests for the same key. Prefetch the likely-next data (the next page as the sentinel approaches; a post's detail on hover).
  • Right-sizing payloads. Paginate (don't fetch 10,000 posts), request only rendered fields, compress, and lazy-load images below the fold with loading="lazy" and correctly-sized srcset variants. On mobile, bytes are latency.
  • Handle the race. For anything type-ahead-driven, out-of-order responses are a real bug: request for "re" returns after request for "reac" and clobbers the newer results. Cancel stale requests (AbortController) or ignore responses that don't match the current query. Tutorial 21 is built around this race.

Axis 4 — Accessibility: a scored dimension, not a bonus

  • Semantic HTML first. A real <button>, not a <div onClick> — you get keyboard operability, focus, and the correct screen-reader role for free. Most a11y wins are "use the right element," and saying that is worth more than reciting ARIA attributes.
  • Keyboard operability. Everything interactive must be reachable and operable by keyboard, with a visible focus ring and a sensible tab order. For composite widgets (a typeahead, a menu) manage focus and roving tabindex deliberately.
  • Screen readers & ARIA — but only where semantics fall short. Use the ARIA patterns (a combobox for autocomplete, a dialog for a modal), label controls, and announce dynamic changes with an aria-live region so a screen-reader user knows new posts loaded or a like registered.
  • The rest of the inclusive baseline: sufficient colour contrast, respect prefers-reduced-motion, don't convey meaning by colour alone, trap focus inside modals and restore it on close.

Putting RADIO on the clock

A method you can't fit in 45 minutes is a method you'll abandon under pressure. Here's how RADIO spends the minutes, and — the part that matters most — where the score actually is. Unlike the backend round, which spreads its points fairly evenly, the frontend round is heavily back-loaded: Architecture and Optimizations are where you win or lose, and the first two letters are setup you should move through briskly.

Interview Q&A

1. How is a frontend system design round different from a backend one?

Same skeleton — clarify, design, deep-dive, trade-offs — but the graded artifacts are completely different. The backend round scores estimation, sharding, consistency, and scale ("what breaks at 10× traffic?"). The frontend round scores component decomposition, where state lives, data flow, rendering/network performance, and accessibility ("what happens on slow 3G, with a huge list, with a screen reader?").

The tell for which one you're in: if the prompt is a screen — a feed, an autocomplete, a photo grid — it's frontend, and you should treat the server as a one-line API contract rather than something to scale. The fastest way to fail is to answer the backend round by reflex: drawing load balancers and talking QPS when the interviewer wanted to know how the feed component virtualizes 10,000 rows.

2. What does RADIO stand for, and how does it map to the backend method?

Requirements, Architecture and components, Data model, Interface and API, Optimizations. It's the frontend analogue of the six-step backend loop: Requirements maps to requirements, Architecture to the high-level box diagram (but the boxes are components), Data model to the data model (but it's the client store, not DB tables), Interface to API design (but there are two interfaces — component props and the network API), and Optimizations to bottlenecks/scale. Estimation folds in rather than being a headline step.

The value of the acronym is the same as the backend method's: it stops you freezing, guarantees you cover what's graded, and lets you spend energy on judgment instead of remembering what comes next.

3. Why do backend-style answers fail this round?

Because they answer a question nobody asked. The frontend interviewer draws the client as the whole subject and hand-waves the server; a candidate who spends ten minutes sharding the feed service, fanning out to followers, and caching timelines in Redis has produced a correct answer to the other interview and spent their best minutes on it. It's not that the backend content is wrong — it's off-topic, and off-topic reads as "doesn't understand what this round is."

The fix is the opening sentence: name the round, say you'll treat the backend as an API contract, and invite the interviewer to redirect if they actually want the service designed too.

4. What's the difference between server state and client state, and why does it matter so much?

Server state is a cached copy of data the server owns — the feed, profiles, comments. Its hard problems are all cache problems: staleness, revalidation, loading/error states, request deduplication. The right tool is a server-cache library (React Query, SWR, RTK Query, Apollo) that solves those for you. Client state is data that only exists in the browser — is a modal open, draft text, selected tab — and the client is the source of truth; useState/Context/a small store is enough.

It matters because the most common data-model mistake is putting server state in Redux, which means hand-rolling every cache concern (loading flags, staleness, dedup, invalidation) badly in reducers. Saying "server state is a cache, not state" and picking tools accordingly is one of the strongest signals in the round.

5. Where should a given piece of state live?

The principle: lift state to the lowest common ancestor of the components that need it, and no higher. Composer draft text → local to the composer (nothing else needs it, and keeping it local means keystrokes don't re-render the feed). A post's like toggle → local to that post's actions leaf (so a like on post 5 doesn't re-render post 6). The list of posts → the container or, better, a server-cache library, since it's server state. Theme/auth → app-global Context (read widely, changes rarely).

Too global and you re-render half the tree on every change; too local and you thread state awkwardly back up when a sibling needs it. Colocating volatile state in a small leaf is also your first performance optimization — it bounds what re-renders, the same way a good shard key bounds what one write touches.

6. Why normalize the client-side store?

So each entity has one source of truth. If the same author is copied into every post, updating their name means finding and fixing N copies, and one you miss drifts — the feed shows the old name, the detail view shows the new one. Store each entity once keyed by id, keep lists as arrays of ids that reference them, and an update in one place is seen everywhere. It's database normalization moved into the browser, for the same reason.

Keep the order (and pagination cursor) separate from the entities — feed.ids holds the sequence, posts holds the objects. That lets you append a page, dedupe a post that appears in two lists, and update one post's like-count in exactly one place.

7. Cursor vs offset pagination — which and why?

Cursor for any feed, infinite scroll, or frequently-changing list. Offset (?offset=40) means "skip 40, take 20," so when a new item is inserted at the top while the user scrolls, everything shifts and they see a duplicate or skip a post — the list drifts. A cursor points at a stable item and is immune to inserts above it; it's also faster at deep pages (an indexed seek instead of scanning and discarding skipped rows).

The trade-off: cursors can't jump to an arbitrary page. So offset still wins for a fixed, page-numbered table (admin lists, search results with page numbers) where "go to page 7" is a real feature and the data isn't shifting underneath.

8. What is an optimistic update and when would you use one?

Update the UI immediately from the client as if the request already succeeded, fire it in the background, and roll back (with a toast) if it fails. Liking a post, sending a chat message, checking a to-do — the interaction feels instant instead of gated on a round trip, which is a big perceived-performance win, especially on slow networks.

Use it when the operation almost always succeeds and a rare rollback is acceptable (a like). Avoid it, or make the pending state explicit, when a wrong-then-corrected value would be alarming or costly — anything money-shaped, or where showing success that later reverses would mislead. This is why the like-count is isolated leaf state in the architecture: it makes the optimistic toggle cheap, local, and easy to roll back.

9. There are 10,000 items in the list — what do you do?

Virtualize (window) the list. Never put 10,000 DOM nodes on the page — the browser can't lay out or paint that and memory balloons. Render only the ~20 rows in the viewport plus a small overscan buffer, and recycle nodes as the user scrolls (react-window / react-virtual). Combine it with cursor-paginated fetching so you're not loading 10,000 items over the network either, and an IntersectionObserver sentinel to trigger the next page.

Say the reasoning, not just the library: the DOM node count, not the array length, is what kills scroll performance, so the fix is to decouple "how many items exist" from "how many are rendered." That's the marquee technique of this round and the interviewer is usually fishing for it the moment you say "infinite scroll."

10. How do you handle accessibility in a design round?

Treat it as a scored requirement, named up front, and fold it into each component rather than bolting it on. The baseline, in priority order: semantic HTML first (a real <button> gives you keyboard operability, focus, and the right screen-reader role for free — most a11y wins are "use the right element"); full keyboard operability with a visible focus ring and sensible tab order; ARIA patterns where semantics fall short (combobox for autocomplete, dialog for a modal), labelled controls, and an aria-live region to announce dynamic changes; then contrast, prefers-reduced-motion, and focus trapping in modals.

Naming a11y as a requirement before anyone asks, and then actually honouring it component by component, is one of the clearest senior signals in the round — most candidates forget it entirely.

11. How do you avoid unnecessary re-renders?

Structure first, memoize second. The structural fix is state colocation — keep volatile state (a like toggle) in a small leaf so a change doesn't ripple up through the whole tree. Then, where profiling shows it's needed: React.memo to skip re-rendering a subtree whose props didn't change, useMemo/useCallback to keep expensive values and callback identities stable so memoized children aren't re-rendered by reference churn, and stable list keys (never the array index for a reorderable list — it misattaches state to the wrong row during reconciliation).

Say "measure first" out loud: memoization isn't free, and wrapping cheap components in React.memo can cost more than it saves. Reach for the Profiler, find the component that actually re-renders too often, and fix that one.

12. The interviewer says "make it faster." Where do you start?

Ask what "faster" means — load or runtime — because they have different fixes, then measure before touching anything. If it's load: check bundle size and code-split, lazy-load below-the-fold assets and routes, consider SSR/streaming for first content, and look at images (almost always the heaviest thing). If it's runtime: profile for re-renders and long tasks, virtualize long lists, debounce high-frequency handlers, move heavy work off the main thread.

The senior move is refusing to optimize speculatively: "I'd instrument with the Profiler and Web Vitals and fix the actual bottleneck — which is usually images and bundle size long before it's render logic — rather than sprinkling useMemo everywhere." Same instinct as the backend round: the bottleneck is rarely where you'd guess, so measure.

What to remember

The round

  • It's a different interview, graded differently. Component decomposition, state placement, data flow, rendering/network performance, and accessibility — not QPS, sharding, or consensus. The server is a one-line API contract you assume exists.
  • Answering the backend round is the #1 way to fail. If the prompt is a screen, it's the frontend round. Draw a load balancer here and you've spent your best minutes on the wrong interview.
  • Name the round in your first sentence — "I'll treat this as frontend system design and the backend as an API contract; stop me if you want the service too." It locks scope and signals you've done this.

RADIO

  • R — Requirements. Functional = user-visible interactions (they enumerate your components). Non-functional = devices, performance budget, network, accessibility, i18n, real-time, list size. Estimate only list size and payload weight — the numbers that change a decision. Scope and cut out loud.
  • A — Architecture. Draw the component tree; carve by responsibility and reuse, not by visual boxes. State lives at the lowest common ancestor that needs it, and no higher. Data down as props, events up as callbacks. "Container vs presentational" is the vocabulary for the seam.
  • D — Data model. Split server state (a cache — React Query/SWR) from client state (useState/store). Server state is not state, it's a cache: stale-while-revalidate, dedup, invalidate. Normalize shared entities by id; keep order separate from entities; derive, don't store. Don't put server data in Redux.
  • I — Interface. Two contracts. The props API (data in, intent out — keeps leaves reusable and testable) and the network API (side-load for normalization, cursor pagination so lists don't drift, idempotent mutations so a double-tap is safe). Don't over-fetch.
  • O — Optimizations. The deep-dive and the biggest bucket. Menu the four axes, then dive where they point: load perf (code-split, SSR, Core Web Vitals), runtime (virtualize long lists, memoize, stable keys, off-main-thread), network (optimistic updates, SWR cache, prefetch, cancel stale requests), accessibility (semantic HTML, keyboard, ARIA patterns, live regions). Measure before optimizing.

The clock

  • ~5 R · ~10 A · ~7 D · ~8 I · ~13 O. The round is back-loaded — Architecture and Optimizations are where the points are. Timebox R and D so you don't starve the deep-dive; let the interviewer pull you into O.
  • Trade-offs score throughout, not just at the end — name the cost of every choice as you make it, the same discipline as every other page in this phase.
The one line that ties the whole page together: the frontend round zooms into the box the backend round hand-waves. Everything you already do by instinct — deciding where a piece of state belongs, whether to debounce, whether a list needs virtualizing, whether a control is keyboard-reachable — is exactly what's being graded. RADIO isn't new knowledge for you; it's a structure that turns seven years of shipping React into a sequenced, narrated, 45-minute answer. Learn the order, name the trade-offs, and this becomes the round you're the favourite to win.
