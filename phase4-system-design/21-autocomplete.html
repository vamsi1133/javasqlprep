21 — Design an Autocomplete / Typeahead

A tiny input box with a surprisingly deep interview inside it — and one race condition that separates the people who've shipped this from the people who've read about it.

Why this matters in the interview

Autocomplete is the frontend round's favourite prompt because it looks trivial and isn't. Anyone can wire an input to a fetch and render a list; the interview is entirely in the second-order problems that a naive version gets wrong. Every keystroke wants to fire a request — so you need debouncing. Fast typing fires those requests faster than they return, and they come back out of order — so you need request cancellation and a stale-response guard, which is the single bug that trips up the most candidates and the one this page is built around. The user navigates the results with the keyboard, not the mouse — so you need a real keyboard model. And a screen-reader user has to be told the listbox opened and what's highlighted — so you need the ARIA combobox pattern. None of these has a QPS number in it. All of them are judgment, and all of them are things you've hit for real in seven years of shipping React.

So this is a frontend system design round, and you drive it with RADIO — Requirements, Architecture, Data model, Interface, Optimizations — exactly as tutorial 19 laid out. The shape is the same as the feed in tutorial 20, but the centre of gravity moves: for a feed, Optimizations is dominated by virtualization and optimistic updates; for a typeahead it's dominated by the request lifecycle — when you fire, when you cancel, which response you're allowed to trust — and by the accessibility contract of a widget that a keyboard-and-screen-reader user has to operate without ever touching a mouse. The backend that actually serves the suggestions (a trie or a search index) is a one-line API contract to you here; we'll touch it at the end so you can speak to it, but the round is about the browser. Get the out-of-order race right, narrate the ARIA pattern like someone who has implemented it, and you've won this round.

Open by naming the round and the real problem in one breath: "I'll treat this as a frontend design — I'll assume the backend gives me a suggestions endpoint and design the client against it. The interesting parts of a typeahead aren't the happy path, they're the request lifecycle — debouncing keystrokes, cancelling stale requests, and guarding against out-of-order responses so a slow earlier request can't overwrite a newer one — plus full keyboard and screen-reader support via the ARIA combobox pattern. I'll structure it as RADIO and spend most of the time there." That sentence tells the interviewer you already know where the depth is, before you've drawn a single box.

The prompt

The interviewer gives you one line:

"Design an autocomplete component — like the search box on Google or Amazon. The user types, and they see suggestions."

That's it, and it's vague on purpose. Before designing anything, do two things. First, name the round so scope is locked: this is a frontend problem, the backend is an API contract, and you'll say so out loud (the senior line above). Second, separate the two things people call "autocomplete," because which one you're building changes the whole design:

Search autocomplete (suggestions)Autocomplete select (from a known set)
ExampleGoogle search box, Amazon product searchA country picker, an @-mention menu, a tag selector
Source of optionsA server that ranks and returns suggestions per queryA finite list, often already on the client, filtered locally
What "select" doesFills the query / navigates to a results page — free text is allowedCommits one canonical value — free text usually isn't valid
The hard partThe async request lifecycle — debounce, cancel, ordering, cacheFast local filtering + the same keyboard/a11y model

Say which you're building and why: "I'll design the search-suggestions flavour — server-backed, free-text allowed — because that's where the interesting async problems live. The keyboard and accessibility model is identical for both, so it transfers to the local-filter case; I'd just swap the network fetch for an in-memory filter." Now the async lifecycle — the meat — is explicitly in scope, and you've shown you know there's more than one shape here.

Reading the round right is worth stating: nobody is asking you how the suggestion service shards its index or handles a million QPS. If you start estimating query volume and sketching a trie sharded by prefix, you're answering the backend round (tutorial 17's territory) and burning the minutes that should go to the request lifecycle and accessibility. The whole server collapses to one endpoint: GET /suggest?q=…. Draw that box, put a one-line contract on it, and zoom into the browser.

R — Requirements

Same first move as every design: pin the surface before you build. For a widget this small the functional list is short, but the non-functional list is where the round announces itself — a typeahead lives or dies on latency, keyboard support, and behaviour on a bad network.

Functional — the interactions, not endpoints

  • As the user types, show a ranked list of suggestions matching the current query.
  • Show suggestions only after a minimum length (say 1–2 characters) — don't fire on the empty string.
  • Navigate the list with the keyboard (up/down arrows), and with the mouse (hover/click).
  • Select a suggestion with Enter or click — which fills the input and fires an onSelect. Close the list with Escape.
  • Highlight the matching substring in each suggestion (a small but expected touch).
  • Show loading, empty ("no results"), and error states — not just the happy path.

What I'll cut, out loud: "I'll skip recent-searches history, multi-select/tag-input, and grouped/categorized results unless you want one — they're additive and don't change the core lifecycle. I'll design the single-select search box well." Naming the cut is the senior move; it shows you scoped deliberately rather than forgot.

Non-functional — this is the real spec

Non-functionalThe question to askWhy it changes the design
Latency / responsiveness"How fast must suggestions feel? Is p95 under ~100 ms the bar?"Drives debounce timing, client caching, and prefetch. Autocomplete that lags behind the keystrokes is worse than none.
Network conditions"Should this hold up on slow/flaky 3G?"This is where debounce, request cancellation, and the out-of-order guard earn their place — a slow network is exactly when responses arrive scrambled.
Accessibility"WCAG AA — full keyboard and screen-reader support via the combobox pattern?"A typeahead is a composite widget; a11y here is real work (roles, aria-activedescendant, live regions), not one attribute. First-class scored dimension.
Request volume / cost"Is a request per keystroke acceptable, or is the backend cost-sensitive?"The frontend's version of estimation: it decides how aggressively you debounce and cache to collapse keystrokes into requests.
Result count & freshness"How many suggestions — ~10? Do they change fast, or can I cache per query?"Small, stable result sets are cacheable on the client; that turns backspacing and re-typing into zero network.
Devices"Desktop, mobile, or both? On-screen keyboard?"Mobile has no hover and a soft keyboard that eats screen space; touch targets and virtual-keyboard behaviour matter.

Then assume and move: "I'll assume p95 suggestions under ~100 ms perceived, must work on 3G, WCAG AA with full keyboard and screen-reader support, request-per-keystroke is not acceptable so I'll debounce and cache, ~10 results per query, desktop and mobile. Good?" Fifteen seconds, and that sentence is the spec — and notice it already commits you to the three techniques (debounce, cache, cancel) that the round is about.

The one estimation that earns its place here is the keystroke-to-request ratio, and it's worth saying out loud because it motivates everything downstream: "If a user types a 10-character query at ~200 ms per keystroke and I fire on every one, that's 10 requests for one search, 9 of them instantly stale. Debouncing at ~200–300 ms collapses a burst of typing into roughly one request at the pause, and a client cache makes backspacing free. So before I optimize rendering, I optimize how often I talk to the network — that's the dominant cost and the dominant source of bugs in this widget." That reframes debounce/cache/cancel not as tricks but as the core of the design.

A — Architecture & components

The tree is small, which is a trap: the temptation is to build it as one giant component with a dozen useStates. Resist it. Decompose so the reusable, dumb piece (the input + listbox rendering) is separate from the smart piece that owns the async lifecycle, because that seam is exactly what makes the widget testable and lets the same UI work for server search and local filtering.

The component tree

  <Autocomplete>                    ← the smart container; owns query, results, request lifecycle
    ├─ <ComboboxInput>               ← the <input>: controlled value, keydown handler, ARIA wiring
    ├─ <SuggestionList>              ← the popup listbox (role="listbox"); rendered only when open
    │    └─ <SuggestionItem>         ← one option (role="option"), ×N; highlights the match substring
    ├─ <StatusRegion>                ← visually-hidden aria-live region: "5 results available" / "no results"
    └─ (a data hook)  useSuggest(query) ← debounce + fetch + cancel + cache; returns {results, status}

The important seam is that the async machinery lives in a hook, not in a component's JSX. useSuggest(query) is where debouncing, the AbortController, the stale-response guard, and the cache all live; the components stay presentational. That split is what you'd say out loud: "I'll pull the whole request lifecycle into a useSuggest hook so the components are dumb — the input renders a value and emits keystrokes, the list renders options and emits selection. The hook is where the hard part lives, and it's unit-testable in isolation without rendering anything."

Where does state live?

Everything here is fairly local — a typeahead is a self-contained widget, so almost nothing needs to be global. The principle from tutorial 19 still applies: keep each piece at the lowest owner that needs it.

StateWhere it livesWhy
The input's text (query)Autocomplete container (controlled input)Both the input and the hook need it; it's the single source of truth that drives fetching.
The debounced queryInside useSuggestDerived from query on a timer; only the hook cares. Keeps the input responsive while the network lags.
Suggestions + status (loading/error)Inside useSuggest (it's server state)A cached copy of server data — loading/error/stale are cache concerns, so the hook owns them.
activeIndex (highlighted option)Autocomplete containerPure UI state driven by arrow keys; the input needs it (for aria-activedescendant) and the list needs it (to style the highlight).

Data flow — controlled input, one direction

The input is controlled: its value is React state, every keystroke updates that state, the state drives the hook, and results flow back down. Draw it and trace one keystroke end to end — that's worth more than any box-drawing. 

  keystroke ─▶ setQuery(v) ─▶ query ─▶ useSuggest(query)
                                 │             │ (debounce → fetch → cancel → guard → cache)
                                 ▼             ▼
                          ComboboxInput   {results, status}
                                                │
                                                ▼
                                          SuggestionList ─(hover/click / Enter)─▶ onSelect(item)
   (data flows DOWN · keystrokes && selection flow UP · the input value is always React-owned)
The classic shallow answer builds the whole thing as one component with the fetch inlined in a useEffect right next to the JSX. It works in a demo and collapses the moment the interviewer pushes: "where do you cancel? how do you test the debounce? how do you reuse this for a local filter?" — because the lifecycle is tangled into rendering. Pull it into a hook first and every follow-up has a clean home. A layout is not an architecture, and for this widget the architecture is the hook.
The structural line that shows you're thinking about the input's responsiveness: "I keep the raw query and the debouncedQuery as two separate values on purpose. The input is bound to the raw query so typing is always instant and never gated on the network; the debounced query is what triggers fetches. Conflating them — debouncing the input value itself — is how you get an input that feels laggy, because now the character you typed doesn't appear until the timer fires." That distinction is subtle, extremely common to get wrong, and instantly reads as someone who has built this.

D — Data model

The client's data model for a typeahead is small but has one subtlety worth naming: it splits cleanly into server state (the suggestions, which are a cache of what the server returned) and client/UI state (the query text, which option is highlighted, whether the popup is open). Getting that split right is what makes caching and the stale-guard fall out naturally instead of feeling bolted on.

The shape the hook holds

// UI / client state — lives in the container, the client is the source of truth
interface ComboboxUIState {
  query: string;          // raw input text, updated every keystroke
  isOpen: boolean;        // is the listbox visible
  activeIndex: number;     // -1 = none highlighted; else index into results
}

// Server state — a CACHE of what /suggest returned; the hook owns it
interface Suggestion {
  id: string;             // stable key — never use the array index as the React key
  text: string;           // what to display && put in the input on select
  type?: string;           // optional: "product" | "category" — for grouping/icons
}

interface SuggestState {
  results: Suggestion[];
  status: "idle" | "loading" | "success" | "error";
}

Two things to point out on this shape. First, each suggestion carries a stable id — you'll use it as the React key and as the id of the role="option" element that aria-activedescendant points at, so it has to be stable across renders, not the array index. Second, activeIndex is an index, not a copy of the highlighted suggestion — deriving the active item as results[activeIndex] keeps one source of truth, so a new result set can't leave you pointing at a suggestion that no longer exists (you just reset the index to -1).

The cache: keyed by query string

The highest-leverage piece of the data model is the response cache: a map from query string to the results for that query. It's what makes backspacing, re-typing, and revisiting a prefix cost zero network requests. 

// query string → the results the server gave for it. Lives in a ref (survives renders),
// or use a server-cache library keyed by ["suggest", query] to get this for free.
const cache: Map<string, Suggestion[]> = new Map();

// On a new debounced query: cache hit → render instantly, no fetch. Miss → fetch, then store.
if (cache.has(q)) return cache.get(q);       // zero-latency path
This is the same machinery as tutorial 19's "server state is a cache" point, and it maps straight onto tutorial 06's cache patterns: the query cache is cache-aside, and if you use React Query / SWR keyed by ["suggest", query] you also get request deduplication (two components asking for the same query collapse to one request) and stale-while-revalidate for free. In an interview it's fine to hand-roll a Map to show you understand the mechanism, then say "in production I'd let React Query own this so I get dedup, TTL eviction, and revalidation without writing it."
Name the cache's limits before you're asked — bounded and short-lived: "I'd cap the cache (an LRU of, say, the last 50 queries) and give entries a short TTL, because suggestions can go stale — trending queries change, a product goes out of stock. For a search box I'm happy serving a slightly stale cached list instantly and letting it refresh in the background; for something where correctness matters I'd shorten the TTL. The point is the cache is an optimization with an eviction and freshness policy, not an unbounded memory leak that shows yesterday's results."

I — Interface & API

Two contracts, as always in a frontend round: the props API of the component (how a consumer uses your combobox) and the network API (how the client talks to the suggestion service). The props API is where reusability is won — a good one lets the exact same component do server search or local filtering.

The component (props) API — make it source-agnostic

interface AutocompleteProps<T> {
  // How to GET options for a query. Return a promise; the component owns debounce/cancel/cache.
  // This one prop is the whole seam: pass a fetch for server search, or a local filter fn.
  getSuggestions: (query: string, signal: AbortSignal) => Promise<T[]>;

  onSelect: (item: T) => void;        // event UP — parent decides what selecting means
  onQueryChange?: (query: string) => void;

  renderItem?: (item: T, active: boolean) => ReactNode;  // customise a row
  getItemLabel: (item: T) => string;  // text to show in the input on select

  minChars?: number;      // default 1 — don't fire on empty
  debounceMs?: number;    // default ~250
  placeholder?: string;
}

The load-bearing decision is getSuggestions(query, signal). By making the data source a prop that returns a promise — and threading the AbortSignal through it — the component knows nothing about the network. Pass an implementation that fetches for the search case; pass one that filters an in-memory array for the country-picker case. Same keyboard model, same ARIA, same rendering. That's the sentence to say: "the component owns the interaction and lifecycle; the caller owns where data comes from, through a single async prop. That's what makes it reusable across server search and local filter." Note also the parent gets onSelect(item) — the component emits intent, not "navigate to this URL"; the parent decides what selection means.

The network API — you're the consumer

// One endpoint. You specify what you need; you're not designing the service.
GET /api/suggest?q=reac&limit=10
// → {
     query: "reac",                // ECHO the query back — critical for the stale-response guard (see below)
     suggestions: [
       { id: "s1", text: "react",          type: "query" },
       { id: "s2", text: "react native",   type: "query" },
       { id: "s3", text: "react router",   type: "query" }
     ]
   }

Three consumer-side asks worth voicing. Keep the payload tiny — id, text, maybe a type for an icon; this response is fetched on nearly every keystroke-pause, so every field is paid for repeatedly over a slow radio. Cap the count server-side (limit=10) — nobody keyboard-navigates 500 options. And the one that matters most: echo the query back in the response. That single field lets the client verify a response still matches the current input before rendering it — the cheapest possible defence against the out-of-order race we're about to dig into. Ask for it explicitly: "I'd have the endpoint echo the query it answered, so the client can drop responses that no longer match what's in the box."

Don't over-fetch here out of backend habit. Returning a fat object per suggestion (descriptions, images, prices) so you "have it ready" is a SELECT * on the hottest path in the app — it's requested on every pause in typing. Fetch the label and an id; fetch the rich detail only when the user actually selects or hovers a specific row. Shape the request to what the dropdown renders, nothing more.
Tie the two interfaces together: "The network contract I keep minimal and I ask it to echo the query, because that echo is what makes the client-side ordering guard trivial. The component contract I keep source-agnostic — one getSuggestions(query, signal) prop — so the same widget serves a server search box and a local dropdown, and threading the AbortSignal through that prop means cancellation works no matter who supplies the data." Connecting the echoed-query in the network shape to the guard in the component is the move that reads as a frontend engineer who's designed this contract before, not just consumed one.

O — Optimizations: debounce vs throttle, caching & prefetch

This is the deep-dive, and for a typeahead it's almost entirely about the request lifecycle: how often you fire, which responses you trust, and how much you can avoid the network altogether. Menu it for the interviewer — "there are four things I'd optimize: how often I fire requests (debounce), which response I'm allowed to render (the ordering guard), avoiding the network (cache + prefetch), and the interaction itself (keyboard + a11y). The ordering guard is the interesting one; want me to start there?" — then go where they point. Nine times out of ten they pick the race. We'll build up to it: first debounce vs throttle, then caching and prefetch, then the race gets its own section because it deserves it.

Debounce vs throttle — know exactly when each

Both limit how often a handler runs; they differ in which calls survive, and picking the wrong one is a real mistake. Debounce waits for a pause: it fires once, after the events stop for N ms — perfect for "the user finished typing." Throttle fires at a steady rate: at most once per N ms while events keep coming — perfect for a stream you want to sample, like scroll or mouse-move.

DebounceThrottle
FiresOnce, N ms after the last event (the pause)At most once per N ms during a burst
Mental model"Wait until they stop""Sample at a fixed rate"
If events never stopNever fires until they pauseKeeps firing, one per interval
Right forTypeahead input, resize-then-recompute, save-on-idle, validating a fieldScroll position, mouse-move, drag, resize-while-dragging, rate-limiting a firehose
Wrong forA continuous stream you must sample (a throttle job) — debounce would starve itAn input that should fire once when typing stops — throttle fires mid-word on stale prefixes

For the search box the answer is debounce, and you should say why not throttle: "I want one request when they pause, not a request every 250 ms while they're still typing — throttling a search input fires on half-typed prefixes like 'rea' and 'reac' and wastes requests I'm about to invalidate anyway. Debounce waits for the pause." Here's the debounce, written as a hook so it's testable: 

// Returns `value` only after it has stopped changing for `delay` ms.
// The input stays bound to the RAW value (instant); this debounced copy drives fetching.
function useDebounced<T>(value: T, delay = 250): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id);   // each new keystroke clears the pending timer → only the pause survives
  }, [value, delay]);
  return debounced;
}

The whole debounce is in that clearTimeout cleanup: every keystroke cancels the previous pending timer, so only a genuine pause of delay ms lets the update through. Around 200–300 ms is the sweet spot — short enough to feel live, long enough to swallow a burst of fast typing. Mention the tradeoff: too short and you're barely debouncing; too long and suggestions feel laggy behind the keystrokes.

Caching — turn backspace into zero network

The query cache from the D step is the second lever. Before firing a request for a debounced query, check the cache; a hit renders instantly with no network at all. This is enormous in practice because of how people actually type: they overshoot and backspace, or type a prefix, pause, and extend it — all of which revisit queries you've already answered. 

async function load(q: string, signal: AbortSignal) {
  if (cache.has(q)) return cache.get(q)!;        // HIT → instant, no request
  const res = await fetch(`/api/suggest?q=` + encodeURIComponent(q), { signal });
  const data = await res.json();
  cache.set(q, data.suggestions);                // store for next time (bounded LRU + TTL, per the D step)
  return data.suggestions;
}

Prefetch — hide latency before the keystroke

The third lever spends a little network to make the common case feel instant:

  • Prefetch on focus. When the input is focused but empty, fetch the default/trending suggestions so the dropdown is populated the instant they type the first character (or even before).
  • Prefetch the likely next query. This is a judgment call — occasionally warm the cache for a one-character extension of what they've typed if the backend is cheap. Usually not worth it; mention it as an option, not a default.
  • Prefetch the result destination. The higher-value prefetch: when a suggestion is highlighted, you can warm the actual search-results page or the selected entity's detail, so pressing Enter feels instant. Same idea as hovering a link and prefetching its route.
The measure-first discipline applies here too: "I wouldn't add prefetch speculatively — it trades backend load and bandwidth for perceived latency, and on a metered mobile connection that trade can be wrong. I'd ship debounce + cache first, measure the real p95 with those in place, and only add prefetch if there's still a perceptible gap. The cache usually buys most of the win for free, because typing is so repetitive." Naming the cost of an optimization before adding it is the same instinct as the backend "measure before you optimize."

The out-of-order response race — the bug everyone ships

This is where the interview really happens. Everything so far is table stakes; this is the problem that separates candidates who have shipped a typeahead from those who have read a tutorial about one. It's subtle, it's a genuine production bug, and volunteering it (rather than waiting to be asked) is one of the strongest signals you can send in this round.

The bug: newer request, older answer

Debouncing reduces requests but doesn't eliminate concurrency — a pause after "reac" fires request A, then the user types "t", pauses again, and fires request B for "react". Two requests are now in flight. HTTP makes no promise about which returns first: B might come back before A. If your code just renders whatever response arrives, the sequence is a disaster: 

  t0  user types "reac"   → fire request A (for "reac")
  t1  user types "react"  → fire request B (for "react")
  t2  response B arrives  → render "react" suggestions        ✅ correct, newest query
  t3  response A arrives  → render "reac" suggestions          ❌ STALE — clobbers the newer results!
      (the input says "react" but the dropdown now shows suggestions for "reac")

The user is staring at an input that reads "react" and a suggestion list for "reac". It's not a crash, it flickers by, and it's maddeningly intermittent because it only happens when A is slower than B — which is exactly what a flaky network makes common. A naive useEffect-per-keystroke that calls setResults on whatever resolves has this bug, and most autocompletes written in an interview under time pressure have it too. Name it out loud: "the trap here is that responses can arrive out of order, and the last one to arrive wins even if it's for an older query. I need to make the newest query win, not the newest response."

The two fixes — and use both

Cancel stale requests (AbortController)Guard stale responses (sequence / query check)
What it doesAborts the in-flight A when B starts, so A's response never arrivesLets A arrive but ignores it because it no longer matches the current query
MechanismAbortController + signal passed to fetch; abort on cleanupCompare the response's query (or a monotonic request id) against the latest before rendering
Bonus winFrees the socket/backend work for the abandoned request — saves bandwidthWorks even for things you can't cancel (a cache hit, a non-abortable API, a resolved promise)
LimitationA request already delivered can't be un-delivered — abort has a race window of its ownDoesn't save the wasted network work; the request still completes, you just drop it
VerdictPrimary defence — cancel what you canBackstop — the guarantee that even an uncancelled straggler can't clobber you

Belt and braces is the senior answer: cancel to save work and close most of the window, and guard as the hard guarantee that a response for a stale query can never render. Here's both in one hook. 

function useSuggest(rawQuery: string) {
  const query = useDebounced(rawQuery, 250);
  const [state, setState] = useState<SuggestState>({ results: [], status: "idle" });
  const latest = useRef(0);   // monotonic request id — the sequence guard

  useEffect(() => {
    if (query.length < 1) { setState({ results: [], status: "idle" }); return; }

    const seq = ++latest.current;          // this request's ticket number
    const controller = new AbortController();  // FIX 1: cancellation
    setState((s) => ({ ...s, status: "loading" }));

    load(query, controller.signal)
      .then((results) => {
        // FIX 2: the guard — only the NEWEST request is allowed to write state.
        // A stale straggler has seq < latest.current, so it's dropped here.
        if (seq === latest.current) setState({ results, status: "success" });
      })
      .catch((err) => {
        if (err.name === "AbortError") return;   // aborted on purpose — not an error
        if (seq === latest.current) setState((s) => ({ ...s, status: "error" }));
      });

    return () => controller.abort();      // new query (or unmount) → abort the previous in-flight request
  }, [query]);

  return state;
}

Walk the interviewer through the two lines that matter. The cleanup () => controller.abort() runs every time query changes, so starting request B aborts request A — A's promise rejects with an AbortError we swallow, and its socket work is freed. The if (seq === latest.current) check is the backstop: even if A somehow already resolved before we aborted (the abort race), its seq is now less than latest.current, so it's forbidden from calling setState. The invariant you state out loud: "only the most recent request is ever allowed to write results."

The query-echo from the network API is an equivalent guard and sometimes cleaner: instead of a sequence counter, compare response.query === currentQuery and drop the response if they differ. The monotonic seq is slightly more robust because two identical queries in quick succession (type, backspace, retype the same thing) would pass a string-equality check but still want the newer one to win. Either is a fine answer; knowing why you'd pick the counter is the senior detail.
A subtle wrong "fix" is to only cancel with AbortController and skip the sequence guard. Abort is not instantaneous — there's a window where the request has already resolved on the network but your abort hasn't taken effect, and in that window the stale .then still runs and clobbers your results. It's rarer, so it survives a quick manual test and ships. Worse, cancellation does nothing for the cache-hit path — a synchronous cached result for an old query can resolve and render out of order too. The sequence guard covers both, which is exactly why you keep it even after adding cancellation. "Cancel to be efficient, guard to be correct" is the line.
The framing that lands this whole section: "The mental bug is thinking the newest response should win. It shouldn't — the newest request should win, and those are different things the instant the network reorders them. So I make the request the source of truth: I tag each one, I abort the previous when a new one starts, and I refuse to apply any response that isn't from the latest request. That combination means a slow 3G connection can scramble my responses however it likes and the dropdown always matches what's in the box." That's the sentence that tells the interviewer you've been burned by this in production and fixed it properly.

Keyboard navigation & the ARIA combobox

The other half of the deep-dive, and a home game for you. A typeahead that only works with a mouse is broken for keyboard users and invisible to screen-reader users — and accessibility is a scored dimension in this round, not a bonus. There's an established recipe here (the WAI-ARIA combobox pattern), and knowing it by name signals you've done this for real.

Keyboard model — the keys and exactly what each does

The rule that makes the keyboard model clean: focus stays in the <input> the entire time. The arrow keys do not move DOM focus into the list; they move a "virtual" highlight (activeIndex) while the real focus never leaves the input. This is called the aria-activedescendant pattern, and it's why the user can keep typing while navigating options.

KeyWhen list is openNotes
ArrowDownMove highlight to next option; from none, highlight the first; wrap to top at the endIf closed, open the list and highlight the first option
ArrowUpMove highlight to previous option; wrap to bottom from the topWrapping is a judgment call — say it either way, just be deliberate
EnterSelect the highlighted option (fill input, fire onSelect, close)If nothing highlighted, submit the raw query as a free-text search
EscapeClose the list and clear the highlight; keep the typed textA second Escape (or a pattern choice) can clear the input
Home / EndJump highlight to first / last optionNice-to-have; mention it, don't dwell
TabAccept highlight (or just leave) and move focus onward — do not trap TabThe list is not a focus trap; Tab must escape the widget normally
 
function onKeyDown(e: KeyboardEvent) {
  switch (e.key) {
    case "ArrowDown":
      e.preventDefault();                          // stop the caret from jumping to end of input
      setActiveIndex((i) => (i + 1) % results.length);   // wrap to top
      setOpen(true);
      break;
    case "ArrowUp":
      e.preventDefault();
      setActiveIndex((i) => (i <= 0 ? results.length - 1 : i - 1));
      break;
    case "Enter":
      if (activeIndex >= 0) select(results[activeIndex]);
      else submitFreeText(query);
      break;
    case "Escape":
      setOpen(false); setActiveIndex(-1);
      break;
  }
}

Two details worth calling out: the e.preventDefault() on the arrows stops the browser's default "move the text caret" behaviour so ArrowUp doesn't jump your cursor to the start of the input, and selection is keyboard and mouse sharing one path — a click and an Enter both call the same select(item), so behaviour can't drift between them.

The ARIA combobox pattern — what a screen reader needs

Semantic HTML gets you a real <input> and real <li>s, but a suggestion popup has no native element, so this is the one place ARIA genuinely earns its keep. The combobox pattern is a specific set of roles and relationships; wire them exactly: 

// The input announces itself as a combobox that controls a listbox popup.
<input
  role="combobox"
  aria-expanded="true"              // is the listbox open? toggles with isOpen
  aria-controls="suggest-listbox"    // id of the listbox it owns
  aria-autocomplete="list"          // suggestions appear in a list (not inline-completed)
  aria-activedescendant="opt-s2"    // id of the CURRENTLY highlighted option (or absent)
/>

// The popup is a listbox; each row is an option with a stable id and selected state.
<ul id="suggest-listbox" role="listbox">
  <li id="opt-s1" role="option" aria-selected="false">react</li>
  <li id="opt-s2" role="option" aria-selected="true" >react native</li>
</ul>

// A visually-hidden live region announces result-count changes without moving focus.
<div aria-live="polite" class="sr-only">5 suggestions available</div>

The mechanism to explain: focus never leaves the input, so how does a screen reader announce the highlighted row? Through aria-activedescendant — it points at the id of the active <li>, and when you update that attribute as the user arrows down, the screen reader reads out the newly-referenced option even though DOM focus is still on the input. That's the whole trick, and it's why each option needs the stable id from your data model. aria-expanded tells the user the popup opened; aria-selected marks the current option; the aria-live="polite" region announces "5 suggestions available" / "no results" so a non-visual user knows the list changed after they typed.

Do not build the popup out of <div onClick>s and call it done, and do not put tabindex on the options to move focus into the list. Both break the pattern: divs give the screen reader no role to announce, and moving focus into the list means the user can't keep typing to refine the query. The correct model is counter-intuitive but firm — focus stays in the input, the highlight is virtual via aria-activedescendant. Getting this specific detail right is a strong senior signal precisely because it's the part most people get wrong.
Narrate the pattern by name and by mechanism: "For accessibility I'd implement the ARIA combobox pattern — the input is role=combobox with aria-expanded and aria-controls pointing at a role=listbox popup of role=option rows. Crucially, focus stays in the input and I move a virtual highlight with aria-activedescendant, so a screen-reader user hears each option as they arrow through it while still being able to type. And I'd announce the result count through an aria-live region, because a change that's obvious visually is silent otherwise." Saying "aria-activedescendant" correctly and explaining why focus stays put is one of the highest-value fifteen seconds available in this round.

The backend, briefly

This is a frontend round, so the server is a one-line contract — but you should be able to speak to what's behind it in thirty seconds if asked, because it shows range. The classic data structure for prefix suggestions is a trie (prefix tree): each node is a character, a path from the root spells a prefix, and each node caches its top-k completions ranked by popularity so a lookup is "walk to the prefix node, return its precomputed top-k." That's why suggestions come back in a few milliseconds — the ranking is precomputed, not calculated per request.

The one-paragraph version to say and then stop: "Behind /suggest is typically a trie or a search index like Elasticsearch. A trie stores completions by prefix with each node caching its top-k ranked results, so a query is an O(length-of-prefix) walk plus returning a precomputed list. It's built offline from query logs, ranked by frequency (with recency and personalization layered on), sharded by prefix at scale, and cached hard because it's read-heavy and rarely changes minute to minute. But that's the backend round — tutorial 17's search-index territory — and here I'd treat it as a fast, cacheable endpoint and stay focused on the client."

Knowing where to stop is the signal. Mentioning the trie proves you understand the full system; diving into how to shard it, rebalance it, or rebuild it from Kafka streams proves you've forgotten which round you're in. One paragraph, then hand it back: "happy to go deeper on the index if you want, but I'd assumed the client is the focus here."

Interview Q&A

1. Debounce or throttle a search input — and why?

Debounce. You want one request when the user pauses, not a steady stream while they're still typing. Debounce waits for a gap of N ms after the last keystroke and fires once; throttle fires at a fixed rate during the burst, which for an input means requests for half-typed prefixes ("rea", "reac") that you're about to invalidate anyway.

Throttle is the right tool for a continuous stream you want to sample — scroll position, mouse-move, drag. The one-liner: "debounce = wait until they stop; throttle = sample at a fixed rate. A search box wants the pause, so debounce, around 200–300 ms."

2. Two requests are in flight and the older one returns last. What happens, and how do you fix it?

That's the out-of-order response race — the marquee bug. Request A (for "reac") and request B (for "react") are both in flight; HTTP doesn't guarantee order, so A can arrive after B. Naive code renders whatever resolves last, so the dropdown shows "reac" suggestions while the input says "react". It's intermittent and network-dependent, which is why it survives testing and ships.

Fix it with both mechanisms: cancel the previous request with AbortController when a new one starts (saves work, closes most of the window), and guard with a monotonic sequence id — tag each request, and only let a response write state if its id is still the latest. The invariant: "the newest request wins, not the newest response." Cancel for efficiency, guard for correctness — the guard also covers cache hits and non-abortable calls that cancellation can't.

3. Why keep both AbortController and the sequence guard? Isn't one enough?

They cover different gaps. AbortController stops wasted network work and closes most of the race, but abort isn't instantaneous — a request can already have resolved before the abort lands, and that stale .then still runs. Cancellation also does nothing for a synchronous cache hit, which can resolve out of order too.

The sequence guard (if (seq === latest.current)) is the hard guarantee: no matter how a response arrived — slow straggler, cache hit, un-cancellable API — it's forbidden from writing state unless it's from the most recent request. So: cancel to be efficient, guard to be correct. Dropping the guard is the subtle wrong answer.

4. How does the client avoid hitting the network on every query?

A query-keyed cache: a map from query string to the results the server returned. Before fetching a debounced query, check the cache — a hit renders instantly with zero network. This wins big because typing is repetitive: people overshoot and backspace, or type a prefix, pause, and extend it, all revisiting queries already answered.

Keep it bounded (an LRU of the last ~50 queries) with a short TTL, since suggestions go stale. In production I'd let React Query / SWR own it keyed by ["suggest", query], which also gives request deduplication and stale-while-revalidate for free. In an interview, hand-rolling a Map shows you understand the mechanism.

5. Walk me through the keyboard interaction.

Focus stays in the input the whole time; the arrows move a virtual highlight (activeIndex), not DOM focus. ArrowDown/ArrowUp move the highlight (wrapping is a deliberate choice); Enter selects the highlighted option, or submits the raw text as a free-text search if nothing's highlighted; Escape closes the list and keeps the typed text; Tab leaves the widget normally (never trap it).

Two details: preventDefault() on the arrows so the text caret doesn't jump, and click and Enter share one select(item) path so mouse and keyboard behaviour can't drift apart. Keeping focus in the input is what lets the user keep refining the query while navigating results.

6. How does a screen-reader user experience this? Which ARIA do you use?

The WAI-ARIA combobox pattern. The input is role="combobox" with aria-expanded (is the popup open), aria-controls (the listbox id), and aria-autocomplete="list". The popup is role="listbox" and each row is role="option" with a stable id and aria-selected.

The key mechanism is aria-activedescendant on the input, pointing at the id of the highlighted option. Because focus stays in the input, that attribute is how the screen reader knows which option to announce as the user arrows through — no focus ever moves into the list. A visually-hidden aria-live="polite" region announces "5 suggestions available" / "no results" so a change that's obvious visually isn't silent. Getting the focus-stays-in-input detail right is the strongest a11y signal here.

7. Why is the input value separate from the debounced query?

So typing is always instant. The input is a controlled component bound to the raw query, updated on every keystroke, so characters appear immediately. The debounced query is a separate value derived on a timer, and it's what triggers fetches. If you debounce the input value itself, the character you typed doesn't render until the timer fires and the input feels laggy — a very common mistake.

Two values, two jobs: raw query drives the UI (instant), debounced query drives the network (throttled).

8. What's your component decomposition, and where does the async logic live?

A smart Autocomplete container, a presentational ComboboxInput and SuggestionList/SuggestionItem, a visually-hidden live region — and, most importantly, a useSuggest(query) hook that owns the entire request lifecycle: debounce, fetch, AbortController, the sequence guard, and the cache.

Putting the lifecycle in a hook (not inline in JSX) makes it unit-testable without rendering, keeps the components dumb and reusable, and gives every hard follow-up ("where do you cancel? how do you test debounce?") a clean home. The data source is a single getSuggestions(query, signal) prop, so the same widget does server search or local filtering.

9. The user backspaces quickly from "react" to "rea". Anything special?

Two things fall out of the design. First, the cache almost certainly already has "rea" (they typed through it on the way up), so it renders instantly with no request. Second, backspacing fires new debounced queries, so the same out-of-order guard applies — an in-flight request for "react" must not clobber the now-current "rea" results, which the sequence guard handles. This is exactly why the guard has to cover the cache-hit path too: a synchronous cached "rea" and a slow network "react" can resolve out of order.

10. What if the backend is untouched and returns everything unranked, unlimited?

Then I push what I can to the client and name the cost. I'd cap what I render (top ~10), filter/rank on the client if the payload is small enough, and cache aggressively so I re-request rarely. But I'd flag that a big unranked payload on every keystroke is expensive on mobile and that the right fix is server-side ranking and a limit — the client can paper over it but shouldn't have to. If the full option set is small and static (a country list), just fetch it once and filter locally; that's the local-filter flavour and the network problems disappear entirely.

11. How would you test this component?

Split it by the seam. The useSuggest hook is tested in isolation with fake timers and a mocked fetch: assert it debounces (N keystrokes → 1 request), that it aborts the previous request on a new query, and — the important one — that an out-of-order resolution is dropped (resolve request A after B and assert B's results remain). The components are tested with props: render, fire keydowns, assert the highlight moves and onSelect fires. For a11y, assert the roles and that aria-activedescendant tracks the highlighted option's id.

That out-of-order test is the one to call out — it's the bug that regresses silently, so it's the test most worth writing.

12. When would prefetch actually be worth it?

Prefetch-on-focus is usually worth it: fetch default/trending suggestions when the empty input is focused so the dropdown is ready before the first keystroke — cheap and high-impact. Prefetching the result destination (warming the search-results page or a highlighted entity's detail so Enter feels instant) is worth it when that navigation is the common next step. Prefetching the likely next query is usually not worth it — it spends backend load and bandwidth speculatively.

The discipline: don't add any of it speculatively. Ship debounce + cache, measure the real p95, and add prefetch only if a perceptible gap remains — on metered mobile the trade can be actively wrong.

What to remember

The round

  • It's a frontend round; the server is a one-line contract (GET /suggest?q=…). The interest is in the browser: the request lifecycle and accessibility. Drive it with RADIO.
  • Say which autocomplete you're building — server-backed search (async lifecycle is the meat) vs local filter from a known set (same keyboard/a11y, no network). Pick the search flavour; the model transfers.

The request lifecycle (RADIO's O — where the round is won)

  • Debounce, don't throttle a search input (~200–300 ms). Debounce = fire on the pause; throttle = sample a stream. Throttle is for scroll/mouse-move, not typing.
  • Keep the raw query and the debounced query separate. Input bound to raw (instant typing); debounced value drives fetching. Debouncing the input value itself makes the input feel laggy.
  • The out-of-order race is the marquee bug. Two in-flight requests can return in the wrong order, so a stale response clobbers newer results. Fix with both: AbortController to cancel the previous request, and a monotonic sequence guard so only the latest request may write state. "The newest request wins, not the newest response." Guard also covers cache hits and non-abortable calls.
  • Cache by query string (bounded LRU + short TTL) so backspacing and re-typing cost zero network; in production let React Query/SWR own it for dedup + revalidation. Prefetch on focus; measure before prefetching anything more.

Interaction & accessibility (also O)

  • Keyboard: focus stays in the input; arrows move a virtual highlight (activeIndex). Enter selects (or submits free text), Escape closes, Tab escapes normally. preventDefault() the arrows; click and Enter share one select() path.
  • ARIA combobox pattern: input is role="combobox" + aria-expanded + aria-controls; popup is role="listbox" of role="option". The key is aria-activedescendant pointing at the highlighted option's id — that's how a screen reader announces the highlight without moving focus. An aria-live region announces the result count.

Structure & the backend

  • Put the whole lifecycle in a useSuggest hook, not inline in JSX — testable in isolation, components stay dumb. Data source is one getSuggestions(query, signal) prop, so the same widget does server search or local filter.
  • The backend (trie / search index) is one paragraph. Prefix tree caching top-k ranked completions per node, built offline from query logs, sharded by prefix, cached hard. Mention it, then stop — going deeper is answering the backend round (tutorial 17).
If you remember one sentence from this page: a typeahead is an exercise in making the newest request win. Debounce decides when you ask, cancellation and the sequence guard decide which answer you're allowed to trust, the cache decides when you don't have to ask at all, and the combobox pattern decides how everyone — including a screen-reader user — drives it. Volunteer the out-of-order race before you're asked, wire aria-activedescendant correctly, and you've demonstrated the two things that separate someone who has shipped this from someone who has read about it.
