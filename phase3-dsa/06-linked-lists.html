06 — Linked Lists

Pointer surgery: three atoms — reverse, fast/slow, dummy head — that compose into every problem on this page.

Why interviewers ask this

Linked lists are the cheapest way for an interviewer to find out whether you can hold a mutable structure in your head without a debugger. There's no clever algorithm here — no dynamic programming, no graph theory. There is exactly one skill: rewiring pointers in the right order so you never lose the reference to the rest of the list. That's why the pass/fail line is so sharp. A candidate who writes curr.next = prev before saving curr.next has just orphaned every node after curr, and the list is gone — not corrupted, gone, unreachable, garbage-collected. There is no recovering from it in the trace. Interviewers at the enterprise tier (Optum, JPMC, Goldman and peers) use these problems precisely because the failure is binary and visible: either you drew the three-pointer dance and got it right, or you didn't draw anything and hand-waved.

The second reason is that linked-list problems are composable in a way almost nothing else on LeetCode is. There are about three atoms on this page — in-place reversal, fast/slow pointers, and the dummy head — and the medium problems are literally those atoms glued end to end. Palindrome Linked List is "find the middle" plus "reverse a list" plus "walk two lists in lockstep." Reorder List is the same three, in a different order. Once you see that, the mediums stop being separate problems and become one problem you've already solved. And then there's LRU Cache, which is not really a linked-list puzzle at all — it's a systems-design question wearing a data-structure costume, and it is the single most-asked design problem at JPMC and Goldman Sachs, because every trading system, session store, and quote cache in the building is an LRU. It gets the most space on this page for that reason.

1. Recognizing the pattern

You don't get to choose a linked list the way you choose a stack — the problem hands you one. So recognition here means something different: given a list, which of the three atoms does this problem want? Read the statement for these signals.

  • "Reverse", "in reverse order", "swap adjacent", "rotate" → in-place reversal (§2b). Anything that changes the direction of links is the prev/curr/next dance, possibly applied to a sub-range.
  • "Detect a cycle", "find the middle", "the Nth node from the end", "is it a palindrome" → two pointers moving at different speeds or with a fixed gap (§2c). The tell is that you need to know something about the list's length or shape but you're forbidden (or too proud) to make two passes.
  • "Merge", "remove", "insert", "delete the head" → you will be modifying the front of the list, which means you want a dummy head (§2a). The instant a problem lets you delete or replace the first node, the dummy head stops being a nicety and becomes the difference between clean code and a nest of if (head === null) branches.
  • "O(1) extra space" on a list problem → that phrase is almost always the interviewer forbidding the obvious answer (copy the values into an array, solve it there). It's a nudge toward reversal or fast/slow. Copy List with Random Pointer's follow-up and Palindrome's follow-up are both exactly this.
  • "O(1) get and put", "evict the least-recently-used" → hashmap + doubly-linked list (§6). Any time a design problem needs both random access by key and ordering that you mutate, those two requirements are what force the two structures together.
The recognition sentence that separates you from a junior candidate on this topic isn't about pointers at all — it's about decomposition. Say it out loud: "This is find-the-middle, then reverse the second half, then compare — I've written both of those pieces already, so let me write them as helpers and compose them." Interviewers at 7-YOE loops explicitly grade for whether you factor the problem into named sub-steps or write one 40-line function with four pointers named p1 through p4. The second one might even pass; it won't score.
A brute force that always works on linked lists: dump the nodes into a JS array (while (curr) { arr.push(curr); curr = curr.next; }), solve the problem with array indexing where life is easy, and re-link at the end. It's O(n) space. Say this out loud, then say why you're not doing it — "the array version is O(n) space; the point of the pointer version is O(1), so let me do that." Naming the brute force and rejecting it for a stated reason is worth real points, and it also gives you a fallback if the pointer version falls apart with five minutes left.

2. The templates

2a. The ListNode shape, and the dummy head

JavaScript has no built-in linked list. There is no LinkedList class, no standard-library helper, nothing. What LeetCode hands you is a plain object graph — nodes are just objects with two fields: 

// This is the entire definition. LeetCode pre-defines it; you don't write it.
function ListNode(val, next) {
  this.val  = (val === undefined ? 0 : val);
  this.next = (next === undefined ? null : next);
}

// So a list is literally: { val: 1, next: { val: 2, next: { val: 3, next: null } } }
// `head` is a reference to the first object. If you lose that reference, the list is gone.

That last line is the whole subject. The entire skill on this page is not losing the reference to the rest of the list. A node is reachable only through the next chain from something you're still holding. The moment you overwrite a next pointer without having first saved what it pointed at, the tail of your list becomes unreachable — JavaScript's GC will happily collect it, and no amount of staring at your code will bring it back. Every pointer-rewiring bug in every problem on this page reduces to that one mistake.

The dummy head. A recurring headache: many operations (delete a node, merge two lists, build a new list) have to handle "what if the node I'm changing is the head?" as a special case, because the head has no predecessor whose next you can rewire. The fix is to manufacture a predecessor:

// The dummy-head idiom — memorize the shape, it appears in half the problems here.
const dummy = new ListNode(0, head);   // a fake node sitting BEFORE the real head
let prev = dummy;

// ... do your work; prev.next is always a legal thing to overwrite,
//     even when the "node" being removed/replaced is the original head ...

return dummy.next;                      // the (possibly NEW) head — never return `head`!

Exactly why this eliminates the special case, in one worked example. Suppose you're deleting a node x. The deletion is one line: prev.next = x.next — you need a prev. Now:

  • Without the dummy: if x === head, there is no prev. So you need a branch: if (x === head) head = head.next; else prev.next = x.next; — and now you also have to remember to return the updated head, and every subsequent line that assumed head hasn't moved is suspect. This is where candidates lose five minutes and their composure.
  • With the dummy: dummy is the head's predecessor. Deleting the real head is dummy.next = head.next — structurally identical to deleting any other node. The special case stopped existing. You return dummy.next, which is correct whether the head changed or not.
Frame the dummy head as a sentinel, and connect it to what you already know: it's the same design move as the zero-height bar in Largest Rectangle (§04), a null terminator in C strings, or a <Fragment> wrapper so a component always has one root. "I'll add a sentinel node before the head so that the 'delete the first node' case isn't a special case." Say that sentence and you've told the interviewer you understand why the idiom exists, not just that it's a trick you saw once.

2b. In-place reversal — the prev/curr/next dance

This is the atom. Reverse Linked List is nominally an EASY, but the code below reappears — verbatim, as a subroutine — inside Palindrome Linked List, Reorder List, Reverse Nodes in k-Group, and Reverse Linked List II. Learn it once, properly, and four other problems get shorter.

You're walking the list with a curr pointer and flipping each node's next to point backwards. Three pointers, because you need three facts at once: where you came from (prev, the new target of curr.next), where you are (curr), and where you were about to go (next) — because the instant you flip curr.next, that information is destroyed. 

let prev = null;   // the reversed portion, built up behind us. Starts empty (null).
let curr = head;   // the node we're currently flipping

while (curr) {
  const next = curr.next;  // (1) SAVE the rest of the list — BEFORE we destroy the link
  curr.next = prev;        // (2) FLIP: curr now points backwards
  prev = curr;             // (3) ADVANCE prev — curr is now the head of the reversed portion
  curr = next;             // (4) ADVANCE curr — using the reference we saved in step (1)
}

return prev;  // curr is null; prev is the LAST node we flipped = the new head

The order of those four lines is the entire problem. Read them as a sentence: save, flip, advance, advance. The two things that must be true:

  • next = curr.next must come BEFORE curr.next = prev. Swap those two lines and line 1 reads back the value line 2 just wrote — next becomes prev, you walk backwards into the part you already reversed, and the rest of the list (everything after curr) is now unreachable. Nothing points at it. It's gone. This single ordering bug fails more candidates on this topic than every other mistake combined.
  • prev = curr must come BEFORE curr = next. If you advance curr first, curr no longer holds the node you wanted to assign to prev. Same class of bug: you clobbered a value you still needed.
  • Return prev, not curr. The loop exits when curr === null (walked off the end). prev is holding the last node processed — the original tail — which is the new head.
  • prev starts at null, and that's not an accident. The original head must end up with next === null, because it becomes the new tail. Initializing prev = null makes that happen on the very first iteration, for free.
Draw it. Literally, on the whiteboard or the shared doc, before you type: three boxes labelled 1 → 2 → 3, with arrows for prev, curr, next, and step through two iterations by hand. It takes ninety seconds and it is the single highest-ROI ninety seconds in the entire linked-list section of your prep. Candidates who type first and debug later spend ten minutes; candidates who draw first spend three. Interviewers notice which one you are.

2c. Fast & slow pointers (Floyd's)

Two pointers walking the same list at different speeds — slow takes one step, fast takes two. That's it. Two completely different questions fall out of it depending on where you stop: 

let slow = head, fast = head;

// Guard BOTH: fast might be the last node (fast.next === null), or already past the end.
while (fast && fast.next) {
  slow = slow.next;        // 1 step
  fast = fast.next.next;   // 2 steps

  // USE A: cycle detection — if there's a loop, fast eventually laps slow.
  if (slow === fast) return true;
}

// USE B: middle of the list — fast fell off the end, so slow is at the midpoint.
return slow;

Why the middle works: pure arithmetic. When fast has travelled 2k nodes, slow has travelled k. fast stops after roughly n nodes, so slow is at n/2. For an even-length list this loop lands slow on the second of the two middles (in 1→2→3→4, slow ends at 3); starting fast = head.next instead lands it on the first. Which one you want is problem-dependent — Reorder List and Palindrome want the first-middle behaviour so the two halves split cleanly. Know that the knob exists; you will get it wrong once and then never again.

Why Floyd's cycle detection works — and this is the part interviewers actually ask you to justify, so internalize the argument rather than the conclusion:

  • If there's no cycle, fast runs off the end and the loop terminates. Trivially correct.
  • If there is a cycle, both pointers eventually enter it and stay in it forever — you can't leave a cycle, every node in it points to another node in it. So from that moment on, both are circling the same loop.
  • Now measure the gap: the distance from fast forward around the loop to slow. Each step, fast gains exactly one node on slow (it moves 2, slow moves 1: net +1). So the gap shrinks by exactly one, every single step. A quantity that is a non-negative integer and strictly decreases by 1 each step must hit zero — and gap zero means slow === fast. They collide. They can never jump over each other, precisely because the step is 1, not 2 or more. That's the whole proof.
  • Time bound: the gap starts at most the cycle length C, so collision happens within C steps of both being inside the loop → overall O(n).
Say the gap argument out loud, unprompted: "Fast gains one node on slow every step, so the gap between them decreases by exactly one each iteration and can never skip past zero — they must collide inside the cycle." That is the exact sentence an interviewer is listening for when they ask "but why does that terminate?" It's also the mechanism that makes Find the Duplicate Number (§4) work, which is why that problem is on this page instead of the arrays page.

3. Reversal & the dummy head

Reverse Linked List (LC #206) EASY ★

Given the head of a singly linked list, reverse the list and return the head of the reversed list. Reverse it in place — you may not allocate new nodes, and you may not copy the values into an array and rebuild. The nodes must be the same objects; only their next pointers change. An empty list reverses to an empty list.

Example 1: Input: head = [1,2,3,4,5] → Output: [5,4,3,2,1] — the original tail 5 is the new head, and the original head 1 now has next = null.
Example 2: Input: head = [] → Output: [] — the loop body never runs and you return prev, which is still null. The template handles this without a special case, which is a nice thing to point out.
Constraints: 0 ≤ number of nodes ≤ 5000; −5000 ≤ Node.val ≤ 5000.

Solve on LeetCode ↗

Thinking process. The brute force is honest and worth stating: walk the list pushing every node into an array, then walk the array backwards re-linking arr[i].next = arr[i-1]. O(n) time, O(n) space, and the interviewer will immediately ask for O(1). So think about what "reverse in place" actually means at the node level: for every node, next must be reassigned to point at its predecessor instead of its successor. But a singly linked node doesn't know its predecessor — so carry it with you. That's prev. And the instant you write curr.next = prev, you have destroyed the only reference to the rest of the list — so save it first. That's next. Three pointers, and each one exists because of a specific thing that would otherwise be unrecoverable. That's the derivation; you should be able to reconstruct it live rather than recite it. Then the loop terminates when curr walks off the end, at which point prev is sitting on the original tail — the new head. Return prev.

Solution (JS) — O(n) time · O(1) space 
var reverseList = function(head) {
  let prev = null;   // head of the reversed portion built so far; null so the old head becomes the new tail
  let curr = head;   // node we're flipping right now

  while (curr) {
    const next = curr.next;  // SAVE first. After the next line this reference is unrecoverable.
    curr.next = prev;        // FLIP: point backwards
    prev = curr;             // curr joins the reversed portion, becoming its head
    curr = next;             // step forward using the saved reference
  }

  return prev;  // curr === null, so prev is the last node flipped = the new head
};

Trace on [1,2,3]:
Start: prev=null, curr=1.
Iter 1: next=2; 1.next=null; prev=1; curr=2. List so far: 1→null.
Iter 2: next=3; 2.next=1; prev=2; curr=3. Reversed so far: 2→1→null.
Iter 3: next=null; 3.next=2; prev=3; curr=null. Reversed: 3→2→1→null.
Loop exits (curr is null). Return prev = 3. ✓

Complexity: O(n) time — each node is visited exactly once and does O(1) pointer work. O(1) space — three local references regardless of list size. No allocation at all: the same n node objects come out, rewired.

Follow-ups you should expect:

  • "Do it recursively." Recurse to the end, then unwind flipping links: const newHead = reverseList(head.next); head.next.next = head; head.next = null; return newHead; with a base case of if (!head || !head.next) return head;. The line to say afterwards is the honest one: "it's O(n) space because of the call stack — the iterative version is strictly better, and on a 10⁵-node list the recursion would blow the stack." Volunteering that trade-off is the senior move.
  • "Reverse only nodes between positions left and right" (LC 92, Reverse Linked List II) — same loop, but you walk to position left−1 first, run the dance for right−left+1 iterations, then stitch the three segments back together. A dummy head makes this bearable because left can be 1.
  • "Reverse in groups of k" (LC 25, Reverse Nodes in k-Group) — HARD, and it's this exact loop in a wrapper that checks there are k nodes remaining before committing. Rare at the enterprise tier; common at FAANG.
The bug, stated one more time because it is the bug: writing curr.next = prev before const next = curr.next. Your code will then read back prev as "the next node," walk backwards into the already-reversed part, and either loop forever or return a two-element list. If your reversed list mysteriously has the wrong length, this is why — every single time. Check the order of those two lines before you check anything else.

Merge Two Sorted Lists (LC #21) EASY ★

You're given the heads of two linked lists, list1 and list2, each already sorted in non-decreasing order. Splice them together into one sorted list and return its head. The result must be built by re-linking the existing nodes — don't allocate new nodes for the values. Either list may be empty.

Example 1: Input: list1 = [1,2,4], list2 = [1,3,4] → Output: [1,1,2,3,4,4] — note both 1s survive; this is a merge, not a union.
Example 2: Input: list1 = [], list2 = [0] → Output: [0] — one list being empty must not require a special branch; the tail-attach step at the end handles it.
Constraints: 0 ≤ length of each list ≤ 50; −100 ≤ Node.val ≤ 100; both lists are sorted ascending.

Solve on LeetCode ↗

Thinking process. This is the merge step of merge sort, and you already know the algorithm: compare the two front elements, take the smaller, advance that list, repeat. The linked-list difficulty isn't the algorithm — it's the bookkeeping. You need a tail pointer to append to, and on the very first append you don't have one yet, because you don't know which list's head wins until you compare. So you'd need if (!head) { head = winner; tail = winner; } else { tail.next = winner; tail = winner; } inside the loop — a branch that fires exactly once and pollutes every iteration. Manufacture the tail. Start with const dummy = new ListNode(0) and let tail = dummy. Now tail is never null, tail.next = winner is always legal, and at the end the real head is dummy.next. Then the last insight: when one list runs out, the other is already sorted and already linked — so you don't loop through it, you attach it in one pointer assignment. That's the line that also silently handles Example 2.

Solution (JS) — O(n + m) time · O(1) space 
var mergeTwoLists = function(list1, list2) {
  const dummy = new ListNode(0);  // sentinel: gives us a tail to append to on iteration 1
  let tail = dummy;                  // last node of the merged list so far

  while (list1 && list2) {
    // <= (not <) keeps the merge STABLE: equal values keep list1 first.
    // Doesn't change correctness here, but it's the right habit and interviewers ask.
    if (list1.val <= list2.val) {
      tail.next = list1;               // splice the existing node in — no allocation
      list1 = list1.next;              // advance the list we consumed from
    } else {
      tail.next = list2;
      list2 = list2.next;
    }
    tail = tail.next;                  // the node we just appended is the new tail
  }

  // One list is exhausted. The other is already sorted AND already linked,
  // so attach the whole remainder in ONE assignment. This also covers the
  // case where an input list was empty from the start.
  tail.next = list1 || list2;

  return dummy.next;  // the real head — dummy itself is discarded
};

Trace on [1,2,4] and [1,3,4]: 1 <= 1 → take list1's 1, tail=1. 2 > 1 → take list2's 1, tail=1(b). 2 <= 3 → take 2. 4 > 3 → take 3. 4 <= 4 → take list1's 4; list1 is now empty, loop exits. tail.next = list2 attaches the remaining [4]. Result: 1→1→2→3→4→4. ✓

Complexity: O(n + m) time — every node from both lists is examined and appended exactly once, and the remainder-attach is O(1) rather than a loop. O(1) space — one sentinel node and one pointer; the output reuses the input nodes, so nothing scales with n. (If you count the dummy node, it's O(1) anyway: a single node.)

Follow-ups you should expect:

  • "Merge k sorted lists" (LC 23, HARD) — the natural escalation and a genuinely common one. Two answers: (a) a min-heap of the k current heads, O(N log k); (b) merge them pairwise, tournament-style, calling this exact function log k times, also O(N log k) and requiring no heap. Answer (b) is the one to lead with here because it reuses the function you just wrote. It's covered properly in §08 (Heaps).
  • "Recursive version?" Elegant — if (l1.val <= l2.val) { l1.next = merge(l1.next, l2); return l1; } — but O(n+m) stack space. Mention it, then note the iterative one is what you'd ship.
  • "What if they're sorted descending?" Flip the comparator. Nothing else changes. This is the interviewer checking that you wrote a merge and not a memorized snippet.
The dummy node is thrown away — you return dummy.next, and dummy itself is garbage after the function returns. That feels wasteful and isn't: one node, O(1), and it buys you the removal of a branch from the hot loop. This is exactly the trade in React when you wrap children in a <Fragment> so a component always has one root — you pay for a node that renders to nothing, and in exchange every consumer stops special-casing.

Remove Nth Node From End of List (LC #19) MEDIUM ★

Given the head of a linked list and an integer n, remove the n-th node from the end of the list and return the head. It's guaranteed that n is valid (1 ≤ n ≤ length). The interesting constraint is the follow-up, which interviewers always ask up front: can you do it in one pass? — meaning you may only walk the list once, so you cannot first count the nodes and then walk again to position length − n.

Example 1: Input: head = [1,2,3,4,5], n = 2 → Output: [1,2,3,5] — the 2nd from the end is 4.
Example 2: Input: head = [1], n = 1 → Output: [] — you removed the only node, so the head itself changed and the list is now empty. Any solution that returns head at the end is already wrong here.
Example 3: Input: head = [1,2], n = 2 → Output: [2] — you removed the head. This is the case the dummy node exists for, and it's the case that breaks naive solutions.
Constraints: 1 ≤ size ≤ 30; 0 ≤ Node.val ≤ 100; 1 ≤ n ≤ size.

Solve on LeetCode ↗

Thinking process. The two-pass answer is obvious: count the length L, then walk L − n − 1 steps to reach the node before the target and unlink. State it, then kill it — one pass is requested. The insight is that "n-th from the end" is a statement about a gap, not a position. If I hold two pointers exactly n nodes apart and I advance both at the same speed, then at the moment the leading pointer reaches the end, the trailing pointer is by definition n nodes from the end. The gap is invariant under simultaneous advancement — that's the whole trick, and it's worth saying in exactly those words. So: send fast ahead n steps, then walk fast and slow together until fast falls off. To delete the node you need its predecessor, so arrange for slow to land one before the target — start slow at a dummy node instead of at head, which shifts it back by one and makes removing the actual head (Example 3) require no special case whatsoever. Two birds, one sentinel.

Solution (JS) — O(L) time, one pass · O(1) space 
var removeNthFromEnd = function(head, n) {
  const dummy = new ListNode(0, head);  // sentinel BEFORE head → deleting the head needs no special case
  let slow = dummy;                       // will end up ONE BEFORE the node to delete
  let fast = head;                        // starts n ahead of slow (dummy → head is already 1 step)

  // Open a gap of exactly n nodes between slow and fast.
  for (let i = 0; i < n; i++) {
    fast = fast.next;
  }

  // Advance BOTH at the same speed. The gap is invariant, so when fast
  // hits null (one past the tail), slow is exactly n+1 from the end —
  // i.e. the node just before the one we want gone.
  while (fast) {
    fast = fast.next;
    slow = slow.next;
  }

  slow.next = slow.next.next;  // unlink the target; GC reclaims it

  return dummy.next;           // NOT `head` — head may be the node we just deleted
};

Trace on [1,2,3,4,5], n = 2: slow = dummy, fast = 1. Gap loop: fast → 2 → 3. Now walk together: (slow=1, fast=4), (2, 5), (3, null) — loop exits. slow is at 3; slow.next is 4, the 2nd from the end. ✓ 3.next = 5 → [1,2,3,5].
Trace on [1,2], n = 2: gap loop moves fast from 1 → 2 → null. The while never runs, so slow is still dummy. dummy.next = 1.next = 2. Return dummy.next = 2 → [2]. ✓ The head was removed and not one line of the code knew or cared. That's the sentinel earning its keep.

Complexity: O(L) time — fast traverses the list exactly once (n steps then L−n steps = L), and slow trails it. Technically it's "one pass" in the sense that matters: each node is visited a constant number of times and you never restart from head. O(1) space — two pointers and a sentinel.

Follow-ups you should expect:

  • "What if n is bigger than the list?" Guaranteed not to happen here, but say what you'd do in production: guard the gap loop with if (!fast) return head.next; or validate up front. Interviewers like that you thought about the contract.
  • "Now remove the n-th from the start." Trivially easier — walk n−1 steps from the dummy and unlink. The point of the question is to check you understand why the from-the-end version needed the gap at all.
  • "Delete a node given only a pointer to it, no head" (LC 237) — the cute one. You can't unlink it (no predecessor), so you impersonate the next node: node.val = node.next.val; node.next = node.next.next;. Works for any node but the tail.
The gap technique has a name you should use: "two pointers with a fixed offset." It's the same family as fast/slow (§2c) — both are "two pointers, one relationship held invariant while you sweep." Fast/slow holds a speed ratio invariant; this holds a distance invariant. Naming the family, rather than treating each as a separate trick, is what lets you derive the next one you've never seen.

Add Two Numbers (LC #2) MEDIUM

You're given two non-empty linked lists representing two non-negative integers. The digits are stored in reverse order — the head is the ones digit — and each node holds a single digit. Add the two numbers and return the sum as a linked list, also in reverse order. Neither number has leading zeros, except the number 0 itself. You may not convert the lists to JS numbers and add them (the lists can be 100 digits long — far past Number.MAX_SAFE_INTEGER).

Example 1: Input: l1 = [2,4,3], l2 = [5,6,4] → Output: [7,0,8] — that's 342 + 465 = 807, reversed.
Example 2: Input: l1 = [9,9,9,9], l2 = [9,9] → Output: [8,9,0,0,1] — 9999 + 99 = 10098. Two distinct failure modes at once: the lists have different lengths, and there's a final carry that creates a new most-significant digit. If your loop condition is while (l1 && l2) you fail this; if you forget the trailing carry you fail this.
Constraints: 1 ≤ length of each list ≤ 100; 0 ≤ Node.val ≤ 9.

Solve on LeetCode ↗

Thinking process. Notice first why the digits being reversed is a gift, not an obstacle: addition carries from least-significant to most-significant, so reverse order means the head is exactly where you want to start and you walk both lists forward in lockstep. (If they were in forward order — that's LC 445 — you'd have to reverse them first or use stacks. Say that; it shows you understand why the constraint is phrased that way.) Then it's grade-school column addition: at each column, sum = d1 + d2 + carry; the digit you emit is sum % 10 and the carry you propagate is Math.floor(sum / 10) (which is always 0 or 1, since 9 + 9 + 1 = 19). The three things that trip people, all visible in Example 2: (1) the lists can be different lengths, so treat a missing digit as 0 rather than terminating; (2) the loop must keep running while either list has nodes; (3) when both lists are done, a leftover carry of 1 is a real digit and needs its own node. Fold all three into one loop condition — while (l1 || l2 || carry) — and the special cases evaporate. Build the output with a dummy head, exactly as in Merge Two Sorted Lists.

Solution (JS) — O(max(n, m)) time · O(max(n, m)) space for the output 
var addTwoNumbers = function(l1, l2) {
  const dummy = new ListNode(0);  // sentinel so we always have a tail to append to
  let tail = dummy;
  let carry = 0;

  // `|| carry` is the whole trick: it keeps the loop alive for one extra
  // iteration when both lists are exhausted but a carry still needs a digit.
  while (l1 || l2 || carry) {
    const d1 = l1 ? l1.val : 0;   // a spent list contributes 0 — handles unequal lengths
    const d2 = l2 ? l2.val : 0;

    const sum = d1 + d2 + carry;    // max 9 + 9 + 1 = 19, so carry is always 0 or 1
    carry = Math.floor(sum / 10);
    tail.next = new ListNode(sum % 10);  // emit one digit
    tail = tail.next;

    if (l1) l1 = l1.next;          // only advance a list that still has nodes
    if (l2) l2 = l2.next;
  }

  return dummy.next;
};

Trace on [9,9,9,9] + [9,9]:
col 0: 9+9+0 = 18 → emit 8, carry 1.
col 1: 9+9+1 = 19 → emit 9, carry 1.
col 2: 9+0+1 = 10 → emit 0, carry 1. (l2 is spent; it contributes 0.)
col 3: 9+0+1 = 10 → emit 0, carry 1. (l1 spent after this.)
col 4: both lists null, but carry === 1 so the loop runs again: 0+0+1 = 1 → emit 1, carry 0.
Loop exits. Output [8,9,0,0,1] = 10098. ✓ The final digit exists only because of || carry.

Complexity: O(max(n, m)) time — one iteration per output digit, and the output has at most max(n, m) + 1 digits. Space O(max(n, m)) for the newly allocated result nodes; if you don't count the required output, the auxiliary space is O(1). Make that distinction out loud — "the output isn't auxiliary space" is a small, precise thing that senior candidates say and juniors don't.

Follow-ups you should expect:

  • "What if the digits were stored in forward order?" (LC 445, Add Two Numbers II.) Three options: reverse both lists, add, reverse the result; or push both onto stacks and pop (which is the elegant one, and ties straight back to §04); or compute the lengths and pad. The stack answer is the one to lead with.
  • "Can't you just parse the lists into numbers and add?" No — 100 digits blows past Number.MAX_SAFE_INTEGER (2⁵³−1, ~16 digits) and you'd silently lose precision. In JS you could reach for BigInt, and it'd actually pass — mention it as a language-specific escape hatch, then solve it properly anyway, because the interviewer is testing carry logic, not your knowledge of BigInt.
  • "Add three lists / add in base k." Same loop; carry can exceed 1 and you swap 10 for k. The structure doesn't change, which is the point.

4. Fast & slow pointers

Linked List Cycle (LC #141) EASY ★

Given the head of a linked list, determine whether it contains a cycle — that is, whether some node's next pointer leads back to a node earlier in the list, so that following next forever never reaches null. Return true if there's a cycle, false otherwise. The follow-up, which is really the whole problem: solve it using O(1) memory.

Example 1: Input: head = [3,2,0,-4] where the tail's next points back to index 1 (the node 2) → Output: true — walking from the head you cycle 2 → 0 → -4 → 2 → … forever.
Example 2: Input: head = [1,2], no back-pointer → Output: false.
Example 3: Input: head = [1] with 1.next = 1 (points at itself) → Output: true — the degenerate one-node cycle. Make sure your loop guard doesn't miss it.
Constraints: 0 ≤ number of nodes ≤ 10⁴; −10⁵ ≤ Node.val ≤ 10⁵.

Solve on LeetCode ↗

Thinking process. The immediate answer is a Set: walk the list, add each node object to the set, and if you ever see a node you've already stored, that's a cycle. It's correct, it's O(n) time, it's easy to write, and it's O(n) space — which is what the follow-up forbids. (Note carefully: you must store the node references, not the values. Values repeat harmlessly in a list with no cycle; only identity proves a revisit. JS Set compares objects by reference, so this Just Works — but say it, because a candidate who stores node.val has written a wrong solution that passes the sample tests.) To get to O(1), stop trying to remember where you've been and instead exploit the geometry: a cycle is a track, and on a circular track a faster runner always laps a slower one. Send slow one step at a time and fast two. No cycle → fast hits null and you're done. Cycle → both are eventually trapped inside it, and since fast closes the gap by exactly one node per step (it gains 2 − 1 = 1), the gap must reach zero. They meet. They cannot leapfrog past each other, precisely because the gap decreases by exactly 1 and not more — that's the argument you say out loud when the interviewer pushes on "but how do you know it terminates?"

Solution (JS) — O(n) time · O(1) space 
var hasCycle = function(head) {
  let slow = head;
  let fast = head;

  // Guard BOTH fast and fast.next: fast takes two steps, so it can fall off
  // the end in the middle of a hop. Checking only `fast` throws on fast.next.next.
  while (fast && fast.next) {
    slow = slow.next;        // 1 step
    fast = fast.next.next;   // 2 steps → fast gains exactly 1 node on slow per iteration

    // Compare REFERENCES (===), not values. Two different nodes can hold the same val.
    if (slow === fast) return true;
  }

  return false;  // fast reached the end → the list terminates → no cycle
};

Trace on 3 → 2 → 0 → -4 → (back to 2): both start at 3.
Iter 1: slow=2, fast=0. Not equal.
Iter 2: slow=0, fast=2 (fast went -4 → wrapped to 2). Not equal.
Iter 3: slow=-4, fast=-4 (fast went 0 → -4). Equal → true. ✓
On the self-loop [1] with 1.next = 1: fast and fast.next are both truthy, so the loop runs; slow=1, fast=1, they're equal → true. ✓ The degenerate case needs no special handling.

Complexity: O(n) time. If there's no cycle, fast reaches the end in n/2 iterations. If there is one, both pointers enter the cycle within O(n) steps, and then the gap — at most the cycle length C — closes at one node per iteration, so they meet within another C ≤ n steps. Total O(n). Space O(1): two references, no matter how long the list is. That O(1) is the entire reason this problem exists; the Set version is not the answer they want, it's the answer they want you to reject.

Follow-ups you should expect:

  • "Now return the node where the cycle begins" (LC 142, Linked List Cycle II) — this follow-up is asked more often than not, so know it. After the collision, reset one pointer to head, leave the other at the meeting point, and advance both one step at a time; they meet exactly at the cycle's entrance. The proof: let F = distance from head to the cycle entrance, and let the meeting point be a distance a into the cycle. When they met, slow had walked F + a and fast had walked 2(F + a); the difference, F + a, must be a whole number of loops. So walking F more steps from the meeting point also lands on the entrance — and walking F steps from the head lands there too. Hence they converge there. You do not need to reproduce that proof under pressure; you need to know the procedure and to be able to say "the algebra works out because fast has walked exactly twice slow's distance."
  • "Return the length of the cycle." Once they collide, hold one pointer still and walk the other around until it comes back — that count is the cycle length.
  • "Where does this matter in real code?" Cycle detection in a dependency graph, an infinite-redirect detector, or JSON.stringify blowing up on circular references. Good, brief, real answer if asked.
The loop guard is while (fast && fast.next), never while (fast). With only fast checked, a list of even length lands fast on the last node, and fast.next.next throws TypeError: Cannot read properties of null. Conversely, checking only fast.next throws when fast is already null. You need both, in that order, and short-circuit && is what makes the order matter.

Middle of the Linked List (LC #876) EASY

Given the head of a singly linked list, return the middle node. If there are two middle nodes (an even-length list), return the second one. You should do it in one pass — no counting the length first and then re-walking.

Example 1: Input: head = [1,2,3,4,5] → Output: the node 3 (and therefore the list [3,4,5], since LeetCode prints from the returned node onward). Odd length, single middle.
Example 2: Input: head = [1,2,3,4,5,6] → Output: the node 4, i.e. [4,5,6] — not 3. The problem explicitly wants the second middle, and this is the only thing you can actually get wrong here.
Constraints: 1 ≤ number of nodes ≤ 100; 1 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. Two passes is trivial: count L, then walk Math.floor(L / 2) steps. Fine, but you can do it in one, and you already have the machinery. Ratio, not counting: if fast moves at twice slow's speed and starts at the same place, then whenever fast has covered the whole list, slow has covered exactly half of it. Run fast until it falls off the end, and read off slow. Everything interesting is in the termination condition, which decides which of the two middles you land on for an even-length list — and that's genuinely the only difficulty, so make sure you understand the knob rather than memorizing one form:

  • while (fast && fast.next), both starting at head → lands slow on the second middle. This is what LC 876 asks for.
  • while (fast.next && fast.next.next), or equivalently starting fast = head.next → lands slow on the first middle, i.e. the last node of the first half. This is the one you want in Palindrome and Reorder List, because there you need to cut the list in two and you need a handle on the node just before the cut.
Solution (JS) — O(n) time · O(1) space 
var middleNode = function(head) {
  let slow = head;
  let fast = head;

  // fast travels 2x. When fast can no longer take a full 2-step hop,
  // slow has covered exactly half the distance fast did.
  while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
  }

  // Odd length: fast === null-ish landing exactly on the tail's next; slow is the single middle.
  // Even length: fast === null; slow is the SECOND middle — which is what the problem wants.
  return slow;
};

Trace on [1,2,3,4,5] (odd): start slow=1, fast=1. → slow=2, fast=3. → slow=3, fast=5. Now fast.next is null → loop exits. Return 3. ✓
Trace on [1,2,3,4,5,6] (even): slow=1,fast=1 → slow=2,fast=3 → slow=3,fast=5 → slow=4, fast=null. Loop exits. Return 4 — the second middle. ✓ Now flip the start to fast = head.next and re-run: slow=1,fast=2 → slow=2,fast=4 → slow=3,fast=6 → fast.next null, exit. Return 3 — the first middle. That single character is the whole knob.

Complexity: O(n) time — fast traverses the list once (n/2 iterations, each O(1)). O(1) space — two pointers.

Follow-ups you should expect:

  • "Return the first middle instead." Change the start or the guard, as above. Being able to produce both variants on demand — and to explain why they differ, rather than trying both until the tests pass — is the actual signal being measured by this otherwise-trivial problem.
  • "Split the list into two halves." Get the first middle, then const second = slow.next; slow.next = null;. That two-line cut is the setup step for both composites in §5 — you'll write it twice more on this page.
  • "What if it's a doubly linked list?" Same answer; the pointers don't care. Doubly-linked only helps if you can walk from both ends, which lets you converge two pointers instead — also O(n), no better.
Fast/slow is the linked-list cousin of something you already do: it's a ratio of traversal speeds, the same way a debounce and a throttle are ratios of event rate to handler rate. The reason it feels alien isn't the idea — it's that arrays let you compute the middle with arr[n >> 1] and a list denies you random access. Every fast/slow trick on this page is a workaround for not having an index.

Find the Duplicate Number (LC #287) MEDIUM ★

You're given an array nums of n + 1 integers, where every integer is in the range [1, n] inclusive. By the pigeonhole principle at least one value must repeat. There is exactly one repeated number (though it may appear more than twice). Return it — subject to two brutal constraints that are the entire problem: you must not modify the array, and you must use only O(1) extra space. (So: no sorting, no frequency map, no marking elements negative.)

Example 1: Input: nums = [1,3,4,2,2] → Output: 2. (n = 4, array length 5, values all within 1..4.)
Example 2: Input: nums = [3,1,3,4,2] → Output: 3.
Example 3: Input: nums = [2,2,2,2,2] → Output: 2 — the duplicate appears five times, not twice. Any solution that assumes exactly two occurrences (like XOR tricks) dies here.
Constraints: 1 ≤ n ≤ 10⁵; nums.length == n + 1; 1 ≤ nums[i] ≤ n; exactly one value is repeated.

Solve on LeetCode ↗

Thinking process — and this is the best "aha" on the page, so slow down and enjoy it. Every easy answer is banned by a constraint, and that's not cruelty, it's a signpost. A Set or frequency map is O(n) space — banned. Sorting and scanning for adjacent equals is O(n log n) and modifies the array — banned. Marking nums[Math.abs(x)] *= -1 as you go is the classic O(1)-space array trick — banned, because it modifies the array. So: what structure do you have that is read-only, O(1) space, and can find a repeat?

Look at the constraints again, and take them as a hint about shape: the array has n + 1 slots (indices 0…n) and every value is in [1, n] — which means every value is a valid index into the array. That is a very deliberate gift. So define a function: from index i, jump to index nums[i]. That's a next pointer. The array is a linked list: node i's successor is node nums[i]. Start at index 0 and follow the jumps.

Now, what does a duplicate mean in that list? If two different indices i and j hold the same value v, then two different nodes both point at node v — node v has two incoming edges. A node with two parents in a chain of "each node has exactly one outgoing edge" is precisely the entry point of a cycle. And the sequence can never leave the array (values are always in range, so it never runs off the end) — it must eventually revisit something, i.e. loop forever. The duplicate number IS the entrance to the cycle. Index 0 is guaranteed not to be inside the cycle, because no value can equal 0 (values start at 1), so nothing points at index 0 — making it a legitimate "head" outside the loop, exactly the setup Floyd's needs.

So the problem is Linked List Cycle II (LC 142) in a costume: run Floyd's fast/slow to find a collision inside the cycle, then reset one pointer to the start and walk both one step at a time — where they meet is the cycle entrance, which is the duplicate value. O(n) time, O(1) space, array untouched. The reason this problem is on the linked-lists page and not the arrays page is exactly this: recognizing that an array of in-range values is a linked list is the single insight being tested. If you say the sentence "every value is a valid index, so this array defines a linked list, and a duplicate value is a node with two incoming edges — that's a cycle entrance", you have already passed this question, whether or not you finish the code.

Solution (JS) — O(n) time · O(1) space · array unmodified 
var findDuplicate = function(nums) {
  // Treat the array as a linked list: node i --> node nums[i].
  // Legal because every value is in [1, n] and indices run 0..n.
  // Index 0 is never pointed AT (no value is 0), so it's outside the cycle: a valid head.

  // ---- Phase 1: Floyd's — find a meeting point INSIDE the cycle. ----
  let slow = nums[0];        // one hop from the head
  let fast = nums[nums[0]];  // two hops from the head

  while (slow !== fast) {
    slow = nums[slow];             // 1 step
    fast = nums[nums[fast]];       // 2 steps
  }
  // No null checks needed: the walk can never leave the array, so it MUST cycle.

  // ---- Phase 2: find the cycle ENTRANCE = the duplicate value. ----
  // Reset one pointer to the head (index 0) and advance BOTH one step at a time.
  // They converge exactly at the entrance (see the LC 142 argument).
  slow = 0;
  while (slow !== fast) {
    slow = nums[slow];
    fast = nums[fast];
  }

  return slow;  // the entrance node's INDEX is the duplicated VALUE
};

Trace on nums = [1,3,4,2,2]: the implied list from index 0 is 0 → 1 → 3 → 2 → 4 → 2 → 4 → 2 → … (index 0 holds 1, index 1 holds 3, index 3 holds 2, index 2 holds 4, index 4 holds 2 — and now it loops between 2 and 4). The cycle is 2 → 4 → 2, and its entrance is node 2.
Phase 1: slow = nums[0] = 1; fast = nums[nums[0]] = nums[1] = 3. Not equal.
→ slow = nums[1] = 3; fast = nums[nums[3]] = nums[2] = 4. Not equal.
→ slow = nums[3] = 2; fast = nums[nums[4]] = nums[2] = 4. Not equal.
→ slow = nums[2] = 4; fast = nums[nums[4]] = nums[2] = 4. Equal at 4. Meeting point found.
Phase 2: slow = 0, fast = 4. → slow = nums[0] = 1, fast = nums[4] = 2. → slow = nums[1] = 3, fast = nums[2] = 4. → slow = nums[3] = 2, fast = nums[4] = 2. Equal at 2 → return 2. ✓ Which is indeed the duplicate.

Complexity: O(n) time — Phase 1 collides within O(n) steps (the pointers enter the cycle in at most n steps and then the gap closes one per step), and Phase 2 walks at most n more. O(1) space — two integer variables. The array is read but never written, satisfying both constraints exactly. This is the only commonly known algorithm that hits both constraints at O(n) time; a binary-search-on-value-range solution (count how many elements are ≤ mid; pigeonhole tells you which half holds the duplicate) also satisfies both but costs O(n log n) — worth naming as your backup if Floyd's escapes you under pressure.

Follow-ups you should expect:

  • "Prove Phase 2 works." Same algebra as LC 142: at the collision, slow has walked F + a and fast has walked 2(F + a), where F is the head-to-entrance distance. The extra F + a that fast walked must be a whole number of loops, so from the collision point, F more steps lands on the entrance — and F steps from the head lands there too.
  • "What if there could be more than one duplicate value?" Floyd's breaks — the "exactly one repeated number" guarantee is what makes the cycle structure unique. Then you're back to a Set, or sorting.
  • "What if you WERE allowed to modify the array?" Then it's easy and you should say so: index-as-marker (negate nums[Math.abs(x)], and the first index you find already negative is the duplicate) or cyclic sort, both O(n)/O(1). The interviewer added "read-only" precisely to close those doors — recognizing which door each constraint closes is the skill.
  • "Give me the binary search version." Search the value space 1..n: for a candidate mid, count how many array elements are ≤ mid. If that count exceeds mid, the duplicate is in the lower half. O(n log n) time, O(1) space. It's a nice cross-reference to §05 (search on the answer) and a good thing to have in your pocket.
This problem is the reason you learn patterns and not problems. Nothing in the statement mentions a linked list. It's an array question, filed under arrays on most lists, and it is secretly Floyd's cycle detection. The transferable move — "an array where every value is a valid index is a functional graph, and repeats mean cycles" — shows up again in Missing Number, First Missing Positive, and Circular Array Loop. When you meet a problem whose constraints seem gratuitously cruel ("O(1) space AND read-only AND O(n)"), that's not the interviewer being mean; that's the interviewer telling you which algorithm they want. Read the constraints as a hint, not a hurdle.

5. Composites

Here is the payoff for the last two sections, and the single most important idea on this page. The medium linked-list problems are not new problems. They are the easy problems, glued together. You have three atoms — find the middle (§4), reverse a list (§3), and walk two lists in lockstep merging them (§3) — and the two canonical mediums below are literally those three atoms called in sequence. Your job in the interview is not to invent anything; it's to notice the decomposition and say it out loud, then write three small helpers you've already written. Candidates who don't see the decomposition write one enormous function with six pointers and get lost. Candidates who do see it are done in eight minutes and sound like engineers.

Palindrome Linked List (LC #234) EASY

Given the head of a singly linked list, return true if the sequence of values reads the same forwards and backwards. The follow-up is the real question, and you should assume it's coming: do it in O(n) time and O(1) space. A singly linked list gives you no way to walk backwards, which is exactly what makes this awkward — and exactly what forces the interesting solution.

Example 1: Input: head = [1,2,2,1] → Output: true. Even length; the two halves are mirror images.
Example 2: Input: head = [1,2] → Output: false.
Example 3: Input: head = [1,2,3,2,1] → Output: true. Odd length — the middle element 3 has no partner and must simply be ignored, not compared. Your comparison loop has to tolerate the two halves being off by one, which is why it's driven by the (shorter) reversed half.
Constraints: 1 ≤ number of nodes ≤ 10⁵; 0 ≤ Node.val ≤ 9.

Solve on LeetCode ↗

Thinking process. Start with the honest brute force and say it: copy every val into an array, then two-pointer from both ends inward — five lines, O(n) time, O(n) space, and it passes. Offer it as your baseline, then immediately attack the space. The obstacle is that a singly linked list only goes forward, so you cannot compare "the i-th from the front" with "the i-th from the back"… unless you make the back half go forward too. And you know how to make a list run in the other direction: you reverse it (§2b).

So the algorithm assembles itself out of parts you already own: (1) find the middle with fast/slow — you wrote that in §4; (2) reverse the second half in place — you wrote that in §3; (3) walk the first half and the reversed second half forward together, comparing values — that's a lockstep walk, the same shape as the merge loop. That's it. Three functions you have already written, called in order. Nothing here is new, and that is the lesson — say it to the interviewer verbatim: "This is find-the-middle plus reverse-a-list plus compare-in-lockstep; I've got all three, let me compose them." The odd-length case handles itself: if you stop when the reversed (second) half runs out, the unpaired middle element is never compared, which is correct — a single middle character never affects palindromicity.

Solution (JS) — O(n) time · O(1) space 
var isPalindrome = function(head) {
  if (!head || !head.next) return true;  // 0 or 1 nodes: trivially a palindrome

  // ---- ATOM 1 (from §4): find the middle with fast/slow. ----
  // Both start at head with guard (fast && fast.next), so for an EVEN list
  // slow lands on the FIRST node of the second half — exactly the cut point.
  let slow = head, fast = head;
  while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
  }
  // [1,2,2,1] → slow = the 3rd node (2). [1,2,3,2,1] → slow = the 3rd node (3),
  // i.e. the odd middle is swallowed into the second half. Harmless: it just
  // makes the second half one node longer, and we stop when it runs out.

  // ---- ATOM 2 (from §3): reverse the second half, starting at slow. ----
  let prev = null, curr = slow;
  while (curr) {
    const next = curr.next;   // SAVE before you destroy — the one rule
    curr.next = prev;
    prev = curr;
    curr = next;
  }
  // `prev` is now the head of the reversed second half (the ORIGINAL TAIL).

  // ---- ATOM 3: walk both halves forward in lockstep and compare. ----
  let left = head;    // first half, walking forward from the original head
  let right = prev;   // second half, walking forward through the reversed nodes

  // Drive the loop off `right` (the shorter-or-equal side). For odd lengths the
  // unpaired middle simply never gets compared — which is exactly what we want.
  while (right) {
    if (left.val !== right.val) return false;
    left = left.next;
    right = right.next;
  }

  return true;
};

Trace on [1,2,2,1]: fast/slow → slow stops at the 3rd node (2). Reverse from there: the second half 2→1 becomes 1→2, so prev = the node 1 (the original tail). Now left walks 1, 2 from the head and right walks 1, 2 from the reversed tail. Compare 1==1 ✓, 2==2 ✓. right becomes null (the reversal set the old middle node's next to null) → return true. ✓
Trace on [1,2,3,2,1]: slow lands on 3. Reversing from 3 gives 1→2→3. left walks 1,2,3; right walks 1,2,3. All three match — including the middle 3, which happens to compare against itself. Return true. ✓
Trace on [1,2]: slow lands on 2; reversed second half is just 2. left = 1, right = 2. 1 !== 2 → false. ✓

Complexity: O(n) time — three sequential O(n) passes (find middle ≈ n/2 hops of fast, reverse n/2 nodes, compare n/2 pairs). Three passes is still O(n); do not let an interviewer talk you into believing that "one pass" is required, and do not contort the code to fuse them. O(1) space — every pointer is a fixed local; no array, no recursion.

Follow-ups you should expect:

  • "You destroyed the input list. Is that acceptable?" This is the follow-up, and it is asked constantly. The honest answer: as written, the first half still points forward and the second half now points backward, so the list is mangled. In a real system that's a bug — a caller holding head now sees a truncated list. The fix is two lines: after comparing, reverse the second half again to restore it, and re-link. Say: "I'd restore the list before returning — in production, a read-only-looking predicate that mutates its argument is a landmine." That sentence alone is a senior signal; most candidates never think of it.
  • "What about thread safety / concurrent readers?" The in-place version is fundamentally unsafe if anyone else can read the list while you're halfway through. The O(n)-space array copy is safe. That's a genuine engineering trade-off and a great thing to name at a bank, where "someone else is reading this structure" is the default assumption.
  • "Recursive solution?" There's a cute one that recurses to the tail and unwinds while a shared front pointer advances. It's O(n) stack space and harder to explain — mention it exists, don't write it.
Write the three atoms as separate steps with a comment header on each, exactly as above, even though you could inline them. It costs nothing and it makes the structure legible to the interviewer, who is watching for whether you decomposed the problem or just started typing pointers. If you have time, actually extract them as named helper functions (findMiddle, reverse) — then Reorder List, next, is four lines.

Reorder List (LC #143) MEDIUM ★

You're given a singly linked list L₀ → L₁ → … → Lₙ₋₁ → Lₙ. Reorder it, in place, to be L₀ → Lₙ → L₁ → Lₙ₋₁ → L₂ → Lₙ₋₂ → … — that is, interleave the list with itself reversed, taking one from the front, one from the back, one from the front, and so on until every node is placed exactly once. You may not modify the node values — only the next pointers. Return nothing; the list is modified in place.

Example 1: Input: [1,2,3,4] → Output: [1,4,2,3] — front 1, back 4, front 2, back 3.
Example 2: Input: [1,2,3,4,5] → Output: [1,5,2,4,3] — odd length; the middle element 3 ends up last and alone, with next = null. If your final list has a cycle or drops a node, it's almost always this case.
Constraints: 1 ≤ number of nodes ≤ 5 × 10⁴; 1 ≤ Node.val ≤ 1000.

Solve on LeetCode ↗

Thinking process. Look at the target order and ask what it is: you're alternately taking from the front of the list and from the back. The problem, as always, is that a singly linked list has no "back" — you can't walk backwards to grab Lₙ, then Lₙ₋₁. The brute force (array of nodes, then two-pointer from both ends re-linking) is O(n) space and perfectly correct — state it, then beat it.

To beat it, notice the rewritten target: L₀ → Lₙ → L₁ → Lₙ₋₁ → … is the first half of the list interleaved with the reversed second half. And now you have your recipe, which is the same three atoms as Palindrome, in the same order, with the last one changed from "compare" to "interleave":

  1. Split. Find the middle (fast/slow) and cut the list into two halves. Here you want the first middle — the last node of the first half — so you can set its next = null and genuinely sever the two halves. Use the while (fast.next && fast.next.next) variant (§4).
  2. Reverse the second half in place (§2b).
  3. Merge the two halves by alternating one node from each — this is a lockstep weave, structurally the same loop as Merge Two Sorted Lists (§3) minus the comparison.

Say this decomposition out loud before you write a single line. It takes fifteen seconds and it converts a problem that intimidates candidates into three things you can each do in your sleep. This is precisely the "composite" lesson from Palindrome, and Reorder List is where interviewers check whether you actually learned it or just memorized one solution. The one subtlety to be careful about in step 3: after the split, the first half is equal to or one longer than the second half, so the weave runs out of second-half nodes first — drive the loop off the second half and let the first half's last node keep its natural null tail.

Solution (JS) — O(n) time · O(1) space 
var reorderList = function(head) {
  if (!head || !head.next) return;  // 0 or 1 nodes: already reordered

  // ===== STEP 1: split. Find the LAST NODE OF THE FIRST HALF. =====
  // Guard on fast.next && fast.next.next → slow stops on the FIRST middle,
  // which is what we need in order to cut cleanly.
  let slow = head, fast = head;
  while (fast.next && fast.next.next) {
    slow = slow.next;
    fast = fast.next.next;
  }

  let second = slow.next;  // head of the second half
  slow.next = null;        // SEVER. Without this line the merged list ends in a cycle.

  // [1,2,3,4]   → first = 1→2,     second = 3→4
  // [1,2,3,4,5] → first = 1→2→3,  second = 4→5   (first half is the longer one)

  // ===== STEP 2: reverse the second half (the §2b atom, verbatim). =====
  let prev = null;
  while (second) {
    const next = second.next;  // save, flip, advance, advance
    second.next = prev;
    prev = second;
    second = next;
  }
  second = prev;  // second now points at the REVERSED second half's head (the old tail)

  // [1,2,3,4]   → second = 4→3
  // [1,2,3,4,5] → second = 5→4

  // ===== STEP 3: weave the two halves, alternating. =====
  let first = head;
  while (second) {          // second is the shorter-or-equal half → it runs out first
    // Save BOTH successors before we clobber either .next — same rule as reversal.
    const f = first.next;
    const s = second.next;

    first.next = second;    // splice a back-node in after a front-node
    second.next = f;        // and point it at the next front-node

    first = f;              // advance both, using the saved references
    second = s;
  }
  // When second runs out, first's chain already terminates in null. Done — in place.
};

Trace on [1,2,3,4,5]:
Step 1: fast/slow with the fast.next && fast.next.next guard → slow stops at 3. second = 4; 3.next = null. First half: 1→2→3. Second half: 4→5.
Step 2: reverse 4→5 → 5→4. second = 5.
Step 3: first=1, second=5. Save f=2, s=4. 1.next=5, 5.next=2. Now 1→5→2→3 with 4 still dangling. Advance: first=2, second=4.
Next iteration: save f=3, s=null. 2.next=4, 4.next=3. Now 1→5→2→4→3. Advance: first=3, second=null → loop exits.
3.next is still null from the severing in step 1, so the list terminates correctly. Result: [1,5,2,4,3]. ✓

Complexity: O(n) time — three linear passes (find middle, reverse half, weave), so ~2n pointer operations total. O(1) space — a handful of local references; no array of nodes, no recursion. Note that the O(n)-space array approach is also O(n) time; the entire reason to do the pointer version is the space, so say that when you choose it.

Follow-ups you should expect:

  • "What happens if you forget slow.next = null?" Excellent question and you should pre-empt it: the first half's tail still points into the second half, so after the weave you get a cycle, and LeetCode hangs or reports a cyclic list. Severing the halves isn't hygiene — it's load-bearing. Point at that line when you write it.
  • "Why the first middle and not the second?" Because you cut after slow. If slow landed on the second middle, the first half would be too short and the second half too long, and the weave would run out of front-nodes before back-nodes. Being able to answer this rather than guessing at the guard is the difference between owning §4 and having memorized it.
  • "Now do the inverse — un-reorder it." Rare, but it's the same three atoms run backwards (unweave into two lists, reverse the second, concatenate). Naming that is enough.
In the weave, you must save both first.next and second.next before you overwrite either one. Overwrite first.next first and you've lost the next front-node; overwrite second.next first and you've lost the next back-node. It's the exact same "save before you destroy" rule as the reversal atom, just with two lists in play instead of one — which is why the reversal drill pays for itself here.

Copy List with Random Pointer (LC #138) MEDIUM ★

You're given a linked list where every node has the usual next pointer and an extra random pointer, which can point to any node in the list, or to null. Construct a deep copy of the list: allocate n brand-new nodes, where each new node's val matches its original, and each new node's next and random point to the corresponding new nodes, not to any node in the original list. None of the pointers in the copied list may point back into the original list. Return the head of the copy.

Example 1: Input: [[7,null],[13,0],[11,4],[10,2],[1,0]] — each pair is [val, randomIndex]. So node 13's random points at index 0 (the node 7), node 11's points at index 4 (the node 1), and so on. Output: the same structure, built from entirely new node objects.
Example 2: Input: [[1,1],[2,1]] — node 1's random points at index 1 (itself's successor) and node 2's random points at index 1 (itself). Self-references and forward references both have to work.
Example 3: Input: [] → Output: [].
Constraints: 0 ≤ n ≤ 1000; random is null or points to some node in the list.

Solve on LeetCode ↗

Thinking process. First, understand precisely why this isn't trivial. If there were only next pointers, you'd walk the list once, cloning as you go — easy. The random pointer breaks that because it can point forward, to a node you haven't cloned yet. When you're cloning node 0 and its random points at node 4, there is no clone of node 4 yet. So you can't do it in a single naive pass; you need some way to answer the question "given an original node, what is its clone?" — and that's a mapping problem.

The straightforward answer: a hashmap, two passes. Pass 1: walk the list and create a clone of every node, storing map.set(original, clone). Pass 2: walk again, and for each original, wire up map.get(orig).next = map.get(orig.next) and map.get(orig).random = map.get(orig.random). A JS Map keys on object identity, so this is direct and clean. O(n) time, O(n) space for the map. This is a completely acceptable answer and you should write it first. Don't skip it to look clever — write the obvious correct thing, then volunteer the optimization.

The follow-up: O(1) extra space (not counting the output). And the trick is genuinely delightful. The map exists only to answer "where is the clone of node X?" — so store the answer on node X. You can't add a field, but you can abuse next: interleave each clone directly behind its original, so the list temporarily becomes A → A' → B → B' → C → C' → …. Now "the clone of X" is just X.next — a pointer dereference instead of a map lookup. The random wiring becomes a one-liner with a beautiful symmetry: X.next.random = X.random.next — read it as "X's clone's random is X's random's clone." Then a third pass un-zips the two lists, restoring the original's next pointers and extracting the copy. Three passes, O(n) time, O(1) auxiliary space. The general move — "the hashmap's only job is a lookup; can I encode that lookup into the structure itself?" — is the transferable idea, and it's what the interviewer is actually fishing for.

Solution 1 (JS) — hashmap, O(n) time · O(n) space — write this one first 
var copyRandomList = function(head) {
  if (!head) return null;

  // Map keys on OBJECT IDENTITY, which is exactly what we need:
  // original node object → its clone.
  const map = new Map();

  // Pass 1: clone every node's value. Pointers left null for now, because
  // random can point FORWARD to a node we haven't cloned yet.
  let curr = head;
  while (curr) {
    map.set(curr, new Node(curr.val));
    curr = curr.next;
  }

  // Pass 2: now EVERY clone exists, so every lookup succeeds. Wire the pointers.
  curr = head;
  while (curr) {
    const clone = map.get(curr);
    // map.get(null) returns undefined, so coerce to null explicitly.
    clone.next   = map.get(curr.next)   || null;
    clone.random = map.get(curr.random) || null;
    curr = curr.next;
  }

  return map.get(head);
};

Why two passes and not one: because random can point forward. On a single pass, cloning node 0 whose random points at node 4, map.get(node4) is undefined — the clone doesn't exist yet. Separating "create all the objects" from "wire all the pointers" is the fix, and it's a pattern you'll see again (it's the same reason a two-phase commit exists, and the same reason React builds the whole element tree before committing it).

Complexity: O(n) time (two linear passes, O(1) map ops). O(n) space for the map — n entries. This is the answer that gets you full marks unless the interviewer explicitly asks for constant space.

Solution 2 (JS) — interleaved nodes, O(n) time · O(1) space — the follow-up 
var copyRandomList = function(head) {
  if (!head) return null;

  // ===== PASS 1: weave each clone in DIRECTLY BEHIND its original. =====
  // A → B → C   becomes   A → A' → B → B' → C → C'
  // Now "the clone of X" is simply X.next. The list IS the hashmap.
  let curr = head;
  while (curr) {
    const next = curr.next;              // save before we splice — the eternal rule
    curr.next = new Node(curr.val, next); // clone points onward to the original's successor
    curr = next;                          // skip OVER the clone we just inserted
  }

  // ===== PASS 2: wire the clones' random pointers. =====
  curr = head;
  while (curr) {
    // Read it aloud: "X's clone's random  =  X's random's clone."
    // Guard the null random — null.next would throw.
    curr.next.random = curr.random ? curr.random.next : null;
    curr = curr.next.next;                // hop original → original
  }

  // ===== PASS 3: un-zip. Restore the originals AND extract the copy. =====
  // Restoring the input is not optional — you were asked to COPY it, not eat it.
  const copyHead = head.next;
  curr = head;
  while (curr) {
    const clone = curr.next;
    curr.next  = clone.next;                        // original re-links to the next original
    clone.next = clone.next ? clone.next.next : null;  // clone links to the next clone
    curr = curr.next;
  }

  return copyHead;
};

Trace on A(random→C) → B(random→A) → C(random→null):
Pass 1 weaves: A → A' → B → B' → C → C'. Every clone's next currently points at the original that followed its source, and every clone's random is null.
Pass 2: at A, A.random is C, so A.next.random = C.next = C' — i.e. A'.random = C'. ✓ At B, B.random is A, so B'.random = A.next = A'. ✓ At C, C.random is null → C'.random = null. ✓ Every random pointer landed on a clone, never on an original — that's the property the interleaving buys you.
Pass 3 un-zips into A → B → C (restored) and A' → B' → C' (the answer). Return A'. ✓

Complexity: O(n) time — three linear passes. O(1) auxiliary space: the only allocation is the n output nodes, which are the required output and by convention don't count. Contrast that with the map version's n extra map entries. Make the "the output isn't auxiliary space" distinction explicitly — interviewers check for it.

Follow-ups you should expect:

  • "Why does curr.random.next give you the clone, rather than something else?" Because of the interleaving invariant: after pass 1, every original node's next is its own clone, without exception. So anyOriginal.next is always "the clone of anyOriginal." Being able to state the invariant, not just the line, is what makes this look derived instead of memorized.
  • "Do you have to restore the original list?" LeetCode won't fail you if you don't, and plenty of accepted solutions skip pass 3. Do it anyway and say why: a function called copy that destroys its input is a production bug. This is exactly the kind of thing a bank's interviewer notices.
  • "Which would you actually ship?" The Map version, almost certainly — it's five lines shorter, obviously correct, and O(n) space on a list you already hold in memory is nothing. The interleaved version is the one you write when memory is genuinely the constraint. Saying that out loud ("the clever one isn't automatically the right one") reads as seniority, not as dodging.
  • "Generalize it: deep-clone an arbitrary object graph with cycles." Same problem, and now the hashmap version is the only viable one — that's structuredClone, and it's how any cycle-safe serializer works. Good connection to make; you've probably debugged a circular-reference crash in JSON.stringify.
Careful with the class name: this problem's node is called Node, not ListNode, and its constructor is Node(val, next, random). Using new ListNode(...) here is an instant runtime error and a slightly embarrassing thirty seconds. Read the given signature before you type — a small thing, and interviewers do register it.

6. Design: LRU Cache

This is the finale, and it deserves the space. LRU Cache is the most-asked design problem in the enterprise tier — JPMC and Goldman Sachs ask it constantly, and so do Optum, Capital One, Bloomberg, and essentially every large product company. The reason is not that it's a cute puzzle. It's that every one of those companies runs LRU caches in production: a market-data quote cache, a session store, a compiled-query cache, a CDN edge. When they ask this, they're asking a question they actually have to answer at work, and they're grading whether you can derive a data structure from a set of performance requirements rather than recall one. That's the senior skill, and it's why this problem is worth an hour of your prep on its own.

LRU Cache (LC #146) MEDIUM ★

Design a data structure that behaves like a fixed-size cache with a least-recently-used eviction policy. Implement the class LRUCache:

  • LRUCache(capacity) — initialize the cache with a positive capacity.
  • get(key) — return the value for key if it exists, otherwise -1. A successful get counts as a use and makes that key the most-recently-used.
  • put(key, value) — insert or overwrite the value for key, and mark it most-recently-used. If this insert pushes the cache over capacity, evict the least-recently-used key.

The requirement that is the entire problem: get and put must each run in average O(1) time. Drop that and this is a five-line wrapper around an array. Keep it, and every naive approach dies.

Example 1: With capacity = 2:
put(1, 1) → cache is {1=1}, order (LRU→MRU): [1]
put(2, 2) → {1=1, 2=2}, order [1, 2]
get(1) → returns 1, and reorders: order is now [2, 1] — a read is a use.
put(3, 3) → over capacity, evict the LRU, which is 2 (not 1, because get(1) refreshed it). Cache: {1=1, 3=3}, order [1, 3]
get(2) → -1 (evicted)
put(4, 4) → evicts 1. Cache: {3=3, 4=4}
get(1) → -1; get(3) → 3; get(4) → 4.
Example 2 (the one that catches people): capacity = 2; put(1,1); put(2,2); put(1,10) → the cache holds {1=10, 2=2} and the order is [2, 1]. Overwriting an existing key must also refresh its recency and must NOT evict anything — the size didn't grow. A put on an existing key is an update, not an insert.
Constraints: 1 ≤ capacity ≤ 3000; 0 ≤ key ≤ 10⁴; 0 ≤ value ≤ 10⁵; up to 2 × 10⁵ calls to get/put.

Solve on LeetCode ↗

Deriving the structure (do this out loud — it's most of the score)

Thinking process. Don't reach for the answer. Build it from the requirements, one dead end at a time — this narration is the interview, and an interviewer who watches you derive hashmap-plus-DLL will rate you far above one who watches you type it from memory.

There are exactly two things this structure must do, and they pull in opposite directions:

  1. Look up a value by key in O(1). That's a hash map. There is no other option and no debate. ✔
  2. Know which key is least-recently-used, and update recency on every access, in O(1). A hash map cannot do this — it has no order. So you need a second structure that maintains an ordering you can mutate cheaply.

Now walk the candidates for #2 and let each one fail for a stated reason:

  • An array, most-recent at the end. On access, remove the key from wherever it is and push it to the end. Finding it is O(n) (unless the map stores its index — but then every removal shifts the array and invalidates every stored index, so the removal itself is O(n)). ✗ O(n) per access.
  • A timestamp on each entry, plus a scan to find the minimum on eviction. get is O(1), but eviction is O(n) — you scan every entry to find the oldest. ✗ (A priority queue fixes the scan but makes it O(log n), and the problem said O(1).)
  • A singly linked list, most-recent at the head. Moving a node to the head is O(1) if you already hold the node… but to unlink a node you need its predecessor, and a singly linked list can't give you one without walking from the head. ✗ O(n) to unlink.

And that last failure names the fix precisely. You need a node you can splice out in O(1) while holding only the node itself — which means the node must know both its neighbours. That is the definition of a doubly linked list. With node.prev and node.next, removing a node from anywhere in the list is two assignments: 

node.prev.next = node.next;   // my predecessor now skips me…
node.next.prev = node.prev;   // …and my successor points back past me
// O(1). No traversal. This is the ONLY reason the list is doubly linked.

So: hash map from key → the DLL node (giving O(1) "where is this key's node?"), and a doubly linked list ordered by recency (giving O(1) "move this node to the front" and O(1) "who's at the back?"). The map answers identity; the list answers order. Neither can do the other's job, which is exactly why the answer is both. Say that sentence in the interview.

The derivation above is the deliverable. Interviewers at JPMC and Goldman routinely tell candidates afterwards that they scored the "why a doubly linked list and not a singly linked list" answer more heavily than the code. The answer is one sentence and you should have it ready cold: "Because I need to unlink an arbitrary node in O(1) given only the node, and to unlink a node you need its predecessor — a singly linked list makes me walk the list to find it, a doubly linked list stores it." Nothing else about the structure requires the back-pointers.

The sentinel head and tail — and why you want two

A doubly linked list is fiddly at the boundaries: removing the first node means updating the list's head; removing the last means updating tail; an empty list means both are null; a one-node list means the node is both. That's four special cases in every single operation, and they are where the bugs live.

Kill all of them with two permanent sentinel nodes — a fake head and a fake tail that never hold data and are never removed: 

head <-> [MRU] <-> ... <-> [LRU] <-> tail
 ^                                     ^
 fake, never holds data                fake, never holds data

With sentinels in place, every real node is guaranteed to have both a prev and a next. That single invariant means node.prev.next = node.next can never throw, no matter where the node is — first, last, or only. The empty-list case, the single-node case, the remove-the-head case, the remove-the-tail case: all four stop existing. This is the §2a dummy-head idea, applied twice, and it is the difference between a 25-line class and a 60-line one full of null checks. Pick a convention and hold it: I'll use "right after head = most recently used, right before tail = least recently used." Then tail.prev is always the eviction victim.

Solution (JS) — hashmap + doubly linked list · O(1) get · O(1) put · O(capacity) space 
// A doubly linked node. `prev` is the entire reason this class exists:
// it's what lets us unlink an arbitrary node in O(1) without traversing.
class DListNode {
  constructor(key, value) {
    this.key = key;      // we store the KEY on the node too — see the eviction note below
    this.value = value;
    this.prev = null;
    this.next = null;
  }
}

class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();   // key → DListNode. Answers "where is this key's node?" in O(1).

    // Two sentinels. They never hold data and are never removed, which
    // guarantees every REAL node has both a prev and a next → zero null checks.
    this.head = new DListNode(0, 0);  // head.next  === most recently used
    this.tail = new DListNode(0, 0);  // tail.prev  === least recently used  ← eviction victim
    this.head.next = this.tail;
    this.tail.prev = this.head;
  }

  // ---- Two private DLL primitives. Every public method is built from these. ----

  // Splice a node OUT. O(1) — no traversal, because the node knows both neighbours.
  _remove(node) {
    node.prev.next = node.next;
    node.next.prev = node.prev;
  }

  // Splice a node in right after the head sentinel → it becomes most-recently-used.
  _addToFront(node) {
    node.next = this.head.next;   // wire the NEW node's pointers first…
    node.prev = this.head;
    this.head.next.prev = node;   // …then rewire the neighbours to point at it
    this.head.next = node;
  }

  // ---- Public API ----

  get(key) {
    if (!this.map.has(key)) return -1;

    const node = this.map.get(key);

    // A READ IS A USE. Refresh recency: unlink, then re-insert at the front.
    // Forgetting this is the #1 wrong answer — the code passes small tests
    // and then evicts the wrong key.
    this._remove(node);
    this._addToFront(node);

    return node.value;
  }

  put(key, value) {
    if (this.map.has(key)) {
      // UPDATE an existing key: overwrite the value, refresh recency, size unchanged.
      // Crucially: do NOT evict here. The cache didn't grow.
      const node = this.map.get(key);
      node.value = value;
      this._remove(node);
      this._addToFront(node);
      return;
    }

    // INSERT a new key.
    const node = new DListNode(key, value);
    this.map.set(key, node);
    this._addToFront(node);

    // Over capacity? Evict the least-recently-used = the node just before the tail sentinel.
    if (this.map.size > this.capacity) {
      const lru = this.tail.prev;
      this._remove(lru);
      // THIS is why the node stores its key: we found the victim via the LIST,
      // but we must also delete it from the MAP — and the map is keyed by key.
      // Without node.key we'd have to scan the map to find the entry. O(n). Dead.
      this.map.delete(lru.key);
    }
  }
}

Trace on Example 1 (capacity = 2). Writing the list as head ⇄ … ⇄ tail, MRU on the left:

  • put(1,1) → new node, add to front. List: head ⇄ 1 ⇄ tail. size 1 ≤ 2, no eviction.
  • put(2,2) → List: head ⇄ 2 ⇄ 1 ⇄ tail. size 2, no eviction.
  • get(1) → found. _remove(1) then _addToFront(1). List: head ⇄ 1 ⇄ 2 ⇄ tail. Returns 1. Note that 2 is now the LRU (tail.prev), purely because of a read.
  • put(3,3) → new node, add to front → head ⇄ 3 ⇄ 1 ⇄ 2 ⇄ tail. size 3 > 2 → evict tail.prev, which is node 2. Unlink it, and map.delete(2) using lru.key. List: head ⇄ 3 ⇄ 1 ⇄ tail. ✓ (A solution that forgot to refresh on get would have evicted 1 here — the wrong answer, and the exact thing this test case is built to catch.)
  • get(2) → not in map → -1. ✓
  • put(4,4) → add to front → head ⇄ 4 ⇄ 3 ⇄ 1 ⇄ tail, size 3 > 2 → evict tail.prev = 1. List: head ⇄ 4 ⇄ 3 ⇄ tail. ✓
  • get(1) → -1 ✓; get(3) → 3 (and moves 3 to front); get(4) → 4. ✓

Trace on Example 2: put(1,1); put(2,2) → head ⇄ 2 ⇄ 1 ⇄ tail. put(1,10) → key exists, so we take the update branch: set value = 10, unlink node 1, re-add to front → head ⇄ 1 ⇄ 2 ⇄ tail, size still 2, nothing evicted. ✓ A solution that treats every put as an insert would have evicted here, and also left a duplicate node in the list. That's the second-most-common bug in this problem.

Complexity — spell all three out:

  • get: O(1). One map lookup (O(1) average), one _remove (two pointer assignments), one _addToFront (four pointer assignments). No loops anywhere in the class — literally not one for or while, which is a nice thing to point at when you claim O(1).
  • put: O(1). Same, plus an O(1) map.set/map.delete. Eviction is O(1) because tail.prev hands you the victim directly — no scan.
  • Space: O(capacity). The map holds at most capacity entries and the list at most capacity nodes plus two sentinels. Bounded by construction, which is the entire point of a cache.
  • Be precise about "average": hash-map operations are O(1) average, not worst case (adversarial key collisions could degrade them). The problem says "average O(1)" for exactly this reason. Saying "average, because the hashmap is average O(1) — the linked-list half is worst-case O(1)" is a genuinely senior distinction and takes four seconds.
Three bugs, in the order they actually happen: (1) Forgetting that get refreshes recency. It's a read, so it doesn't feel like a mutation, but LRU counts reads as uses. Miss it and you evict the wrong key — and, cruelly, small tests still pass. (2) Treating put on an existing key as an insert. You then push a second node for the same key and evict something you shouldn't have. Always branch on map.has(key) first. (3) Not storing the key on the node. You find the eviction victim through the list (tail.prev), but you must delete it from the map, which is keyed by key. Without node.key you'd have to scan the map for the entry whose value is that node — O(n), and your whole O(1) claim collapses. That one line in the constructor is load-bearing and it's worth pointing at while you write it.

The JS-specific follow-up: "can't you just use a Map?"

Yes. And you should raise it yourself, because it demonstrates that you know your language deeply — but you should also know exactly why the interviewer will make you do it the long way anyway.

JavaScript's Map is not a plain hash table: it is specified to preserve insertion order, and it iterates in that order. That means map.keys().next().value gives you the oldest-inserted key — which is exactly the LRU, provided you keep the insertion order in sync with the recency order. And you can: map.delete(key) followed by map.set(key, value) moves the key to the end of the iteration order, because a re-inserted key is treated as a fresh insertion. So a Map gives you, for free, exactly the "move to most-recent" and "find the least-recent" operations you built a doubly linked list to get.

Solution (JS) — the Map shortcut, ~15 lines · O(1) get · O(1) put 
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    // JS Map preserves INSERTION ORDER. We maintain the invariant:
    // iteration order == recency order, oldest first.
    this.map = new Map();
  }

  get(key) {
    if (!this.map.has(key)) return -1;

    const value = this.map.get(key);

    // delete + re-set moves the key to the END of the iteration order.
    // A re-inserted key counts as a FRESH insertion — that's the whole trick.
    this.map.delete(key);
    this.map.set(key, value);

    return value;
  }

  put(key, value) {
    // Delete first so that an UPDATE also moves the key to the back (most recent).
    // If the key is absent, delete() is a harmless no-op — one branch handles both cases.
    this.map.delete(key);
    this.map.set(key, value);

    if (this.map.size > this.capacity) {
      // The FIRST key in iteration order is the oldest = the LRU.
      const lruKey = this.map.keys().next().value;
      this.map.delete(lruKey);
    }
  }
}

This is correct, it passes LeetCode, and it is genuinely O(1) per operation — Map.prototype.delete, set, and has are all O(1) average, and map.keys().next() creates an iterator and takes one step, which is O(1), not O(n). (Be sure you can defend that last point: it does not iterate the whole map; it advances the iterator exactly once.)

And now the honest part. Here is what to say, unprompted, the moment you finish writing it:

"This works because the spec guarantees Map's insertion order and delete-then-set re-inserts at the back. But I'd expect you'd want the hashmap-plus-doubly-linked-list version, because this one is leaning on a language feature to do the part of the problem you're actually asking about — I'm outsourcing the ordered structure to V8 instead of building it. Let me write the DLL version."

Why the interviewer will make you do it the real way, and why they're right:

  • The problem is the ordered structure. The whole question is "how do you maintain O(1) recency ordering?" The Map version answers it with "someone else already did." That may be the correct engineering call in production, but it isn't an answer to the interview question, and a good interviewer will say so politely and then ask you to implement it from primitives.
  • It doesn't generalize. The instant they change the eviction policy — LFU (LC 460), TTL-based, a two-tier ARC — the Map trick evaporates and you need real pointer manipulation. Interviewers know this, and "now make it LFU" is a common escalation at JPMC and Goldman. If you only know the Map trick you have nothing to build on.
  • It's language-specific. Java's LinkedHashMap has an accessOrder flag that does the same thing, and Python's OrderedDict has move_to_end — every language has its cheat, and every interviewer has seen it. Reaching for it is not clever; knowing it exists and knowing why it's not the answer is.
  • The DLL version is what's under the hood anyway. V8's Map is a hash table over an insertion-ordered entry array; LinkedHashMap is literally a hash map plus a doubly linked list. When you write the DLL version you're writing the thing the shortcut is built on — and being able to say that is a strong signal.

The optimal play in the room: write the DLL version as your primary answer, then say "as a footnote — in JS specifically you can lean on Map's insertion order and do this in fifteen lines; here's how, and here's why I didn't lead with it." You get credit for both the data structure and the language depth, in the right order. Leading with the Map trick gets you asked to do it again properly, and now you're doing the hard part with less time on the clock.

The best thing you can add to this problem, and almost nobody does: connect it to production. One or two sentences, at the end, unprompted. "In a real service I'd think about thread safety — this structure is not safe under concurrent access, and a real cache would need a lock around the list mutations or a sharded design to avoid contention. I'd also want a TTL alongside the LRU, because a stale quote is worse than a cache miss." At a bank, that is not showing off; that is the actual job, and it moves you from "solved the LeetCode problem" to "has built a cache." If they ask a follow-up you can also mention that LFU (LC 460) is the natural next question — it's the same hashmap-plus-list idea, but you keep one doubly linked list per frequency and a pointer to the minimum frequency. Naming it is enough; you don't have to write it.

Interview Q&A

Why must next = curr.next come before curr.next = prev? Explain it as if I'm skeptical.

Because curr.next is the only reference to the rest of the list. Nothing else in the program points at the node after curr — the previous nodes have already been flipped to point backwards, and prev points behind you. So the instant you execute curr.next = prev, that reference is overwritten and unrecoverable: the entire tail of the list becomes unreachable from any variable you hold, and the garbage collector is free to take it. There is no way to get it back.

If you swap the two lines, the code doesn't crash — which is what makes it insidious. const next = curr.next now reads back the value prev that was just written, so you set curr = prev and walk backwards into the part you already reversed. You'll return a list of length 1 or 2, or loop forever.

The general rule, and it applies to every problem on this page: save every pointer you're about to clobber, before you clobber it. In Reorder List's weave you save two of them. In Copy List with Random Pointer's interleave you save one. It's always the same rule.

What exactly does a dummy head buy you, and when should you NOT bother with one?

It buys you one thing: a guaranteed predecessor for the head node. Most list operations (delete, insert, splice) are expressed as prev.next = something, and the head is the one node with no prev. Manufacturing a fake one collapses "what if it's the head?" from a branch into a non-event. It also gives you a stable handle to return: dummy.next is the correct head whether or not the real head changed, so you never have to track a moving head variable.

Use one when: the head might be deleted or replaced (Remove Nth From End), or you're building a new list and need somewhere to append the very first node (Merge Two Sorted Lists, Add Two Numbers).

Skip it when: the head is guaranteed to stay put and you're not building an output list — Reverse Linked List doesn't need one (prev = null already plays that role), and neither do the pure fast/slow problems, which only read.

And the tell that you needed one and didn't use one: your function has an if (node === head) branch in it. That branch is the dummy head asking to be born.

Why does Floyd's fast/slow provably terminate on a cycle? Why can't fast jump over slow?

Once a cycle exists, both pointers eventually enter it and can never leave — every node in a cycle points to another node in the cycle. So from some step onward, both are circling the same loop.

Now define the gap: the number of steps from fast forward around the loop to slow. Each iteration fast moves 2 and slow moves 1, so fast gains exactly one node per iteration and the gap shrinks by exactly one, every step. A non-negative integer that strictly decreases by 1 each step must reach 0 — and gap 0 means the pointers are on the same node. That's the termination proof.

And that's also why fast can't jump over slow: a step size of +1 on the gap cannot skip a value. If fast moved three steps to slow's one, the gap would shrink by 2 per iteration and could go from 1 straight to −1, i.e. leapfrog — and you'd only be guaranteed to meet if the gap happened to have the right parity relative to the cycle length. The 2:1 ratio is the smallest one that guarantees a hit, which is exactly why it's the ratio everyone uses.

Bound: the gap is at most the cycle length C, so they collide within C steps of both being inside → O(n) overall.

Fast/slow lands on the "second middle" for even-length lists. How do you get the first middle, and when does it matter?

Two equivalent ways, and you should be able to produce either:

  • while (fast && fast.next) with both starting at head → second middle. ([1,2,3,4] → 3.) This is what LC 876 wants.
  • while (fast.next && fast.next.next), or equivalently starting fast = head.next → first middle. ([1,2,3,4] → 2.)

It matters whenever you need to cut the list. To sever the halves you write slow.next = null, which requires slow to be the last node of the first half — the first middle. That's Reorder List, and it's why that problem uses the second form. Palindrome tolerates either (a slightly longer second half just means the unpaired middle element compares against itself, harmlessly), which is a nice thing to notice and say.

Don't memorize which guard goes with which problem. Ask yourself the one question — "do I need the node before the cut, or the node after it?" — and the guard follows.

Find the Duplicate Number is an array problem. Why is it in a linked-list tutorial?

Because it is a linked-list problem in disguise, and spotting that is the entire question.

The constraints hand it to you: the array has n+1 slots (indices 0…n) and every value is in [1, n]. So every value is a valid index — which means you can define next(i) = nums[i] and the array becomes a linked list. Following the jumps from index 0, you can never run off the end (values are always in range), so the walk must eventually revisit a node: it cycles.

Now, what is a duplicate? If indices i and j both hold value v, then two nodes point at node v — node v has two incoming edges, which in a "one outgoing edge per node" structure is exactly the entrance to the cycle. So the duplicate value is the cycle's entry node, and the problem is Linked List Cycle II (LC 142) verbatim: Floyd's to find a collision, then reset one pointer to the head and walk both one step at a time until they meet.

(Index 0 is guaranteed outside the cycle, because no value equals 0, so nothing points at it — which is what makes it a valid "head.")

The transferable lesson: when the constraints look gratuitously harsh (O(n) time AND O(1) space AND read-only), they're not a hurdle — they're a hint telling you which algorithm is intended. Read them that way.

Palindrome and Reorder List both feel like new problems. Convince me they aren't.

They're the same three atoms, in the same order, with one step swapped:

StepPalindrome (LC 234)Reorder List (LC 143)
1Find the middle (fast/slow)Find the middle (fast/slow) and cut
2Reverse the second halfReverse the second half
3Walk both halves forward, compareWalk both halves forward, interleave

That's the whole difference: step 3 is if (a.val !== b.val) return false in one and a.next = b; b.next = aNext in the other. If you can write Reverse Linked List and Middle of the Linked List, you can already write both of these — the only new skill is noticing the decomposition.

So the move in the interview is to say it before you write anything: "This is find-the-middle, then reverse, then a lockstep pass — three things I can each do in three lines." Then write three commented blocks. Candidates who don't decompose write one 40-line function with pointers named p1…p4, lose track, and run out of time.

Why does LRU Cache need a doubly linked list? Isn't a singly linked list enough?

No, and this is the highest-value question on the page — interviewers weight it heavily.

On every access you must move an arbitrary node to the front of the recency order, in O(1). The hashmap gives you the node itself in O(1). But to unlink a node you need to rewire its predecessor's next pointer — and a singly linked node does not know its predecessor. Finding it means walking from the head: O(n). Your O(1) guarantee is dead.

A doubly linked node stores prev, so unlinking is two assignments with no traversal at all:

node.prev.next = node.next;
node.next.prev = node.prev;

That is the only reason for the back-pointers. Nothing else in the design needs them. Say exactly that: "I need the back-pointer so I can splice a node out in O(1) while holding only the node."

Bonus: the same reasoning tells you why tail.prev is the eviction victim in O(1). Without back-pointers, finding the node before the tail would also be an O(n) walk.

In LRU Cache, why store the key inside the node when the map is already keyed by key?

Because eviction crosses from one structure into the other, in the direction the map can't handle.

You find the victim through the list: it's tail.prev. You now hold a node. But you also have to remove that entry from the map, and the map is keyed by key, not by node. Without node.key, your only option is to iterate the map looking for the entry whose value is that node — O(n), and your entire O(1) claim collapses on eviction.

One field in the constructor, and the eviction becomes this.map.delete(lru.key) — O(1). It's the smallest line in the class and the one most likely to be missing from a candidate's first draft. Point at it when you write it; interviewers notice that you knew why it was there.

You can implement LRU in 15 lines with a JS Map. Should you?

Know it, mention it, don't lead with it.

It genuinely works: JS Map is specified to preserve insertion order, a re-inserted key moves to the back (so delete + set = "mark as most recent"), and map.keys().next().value gives you the oldest key in O(1) — it advances an iterator once, it does not scan. All operations are O(1). It passes LeetCode.

But the interviewer is asking "how do you maintain an O(1)-updatable ordering?", and the Map answer is "V8 already did." That's not an answer to the question. Three concrete reasons they'll push back:

  • It doesn't generalize. "Now make it LFU" (LC 460) — a common escalation at JPMC/Goldman — and the trick evaporates; you need real pointer work (one DLL per frequency).
  • Every language has this cheat. Java's LinkedHashMap(accessOrder=true), Python's OrderedDict.move_to_end. The interviewer has seen it. Reaching for it isn't clever; knowing why it isn't the answer is.
  • The cheat is literally the thing you were asked to build. LinkedHashMap is a hash map plus a doubly linked list. You'd be handing back the answer wrapped in a library.

So: write the DLL version, then add "as a footnote, in JS you can lean on Map's insertion order to do this in fifteen lines — here's how, and here's why I didn't lead with it." You get credit for the structure and the language depth, in the right order.

When is the O(n)-space "dump it into an array" approach actually the right call?

More often than prep culture admits, and being able to say when is a seniority signal.

Use it when: the list is small and bounded; the code needs to be obviously correct and maintainable; other threads may read the list while you work (the in-place reversal in Palindrome temporarily leaves the structure in a broken state — genuinely unsafe under concurrency); or you're not allowed to mutate the input at all.

Reject it when: the problem explicitly says O(1) space, or the interviewer asks for a follow-up — at which point the O(1) pointer version is what they're buying.

The move in the room is always the same: state the array solution first, in one sentence, with its complexity — then say why you're not doing it. "The array version is O(n) time and O(n) space and takes four lines; since you've asked for constant space, let me do the pointer version." You've now demonstrated you saw the easy answer, understood the constraint, and made a deliberate choice — instead of looking like the pointer version is the only thing you know.

How do you debug a linked-list solution when you can't console.log the whole list?

You can, actually — and you should build the habit now, because it costs seconds in an interview and saves minutes:

const show = (h) => { const out = []; let c = h, n = 0;
  while (c && n++ < 20) { out.push(c.val); c = c.next; }  // the cap saves you from cycles!
  return out.join(' -> '); };

The n++ < 20 cap is the important bit: if your rewiring accidentally created a cycle — the classic Reorder List bug where you forgot slow.next = null — an uncapped print loops forever and hangs your session. A capped print immediately shows you a repeating pattern, which diagnoses the cycle.

Beyond that: draw the boxes. Three or four boxes with labelled arrows, and step through two iterations by hand before you type. Every experienced engineer who is good at these problems does this, and every candidate who is bad at them skips it. It's not a crutch — it's the method.
