04 — Stacks & Monotonic Stacks

LIFO matching/undo, and monotonic stacks for "next greater/smaller" questions.

Why interviewers ask this

Stacks are the highest-frequency data-structure topic at enterprise screens, and Valid Parentheses may be the single most-asked problem across the enterprise tier — banks, fintech, healthcare, and large non-FAANG product companies (Optum, JPMC, Goldman Sachs and their peers) — because it fits in 15 minutes, has a clean canonical solution, and instantly separates candidates who reach for the right structure from candidates who try to regex their way through nesting. The design flavor (Min Stack, Queue via Stacks) shows up constantly in fintech loops because these companies live on order books, transaction logs, and undo/replay semantics — all fundamentally LIFO or FIFO structures with extra invariants bolted on. As a frontend engineer you already use stacks daily: the call stack, browser history, undo in every editor, and React's reconciliation before Fiber flattened it. Interviewers at your level expect you to connect those dots out loud.

The second half of this tutorial — monotonic stacks — is where the medium questions live. "Next greater element," "days until a warmer temperature," "span of stock prices" — these look like O(n²) scan-ahead problems, and the whole interview is watching whether you can turn the nested loop into a single pass by keeping a stack of unresolved candidates. It's one idea, one template, reused across a dozen LeetCode problems. Nail the invariant ("my stack is always decreasing; anything I'd push that breaks that resolves the elements it beats") and you can derive Daily Temperatures, Next Greater Element, and even Largest Rectangle in Histogram from scratch under pressure instead of recalling memorized code — which is exactly the senior signal a 7-YOE loop is grading for.

1. Recognizing the pattern

Reach for a plain stack when the problem statement has any of these signals:

  • Matching / nesting — brackets, HTML tags, nested function calls, "is this string well-formed?" Every opener must be closed by the most recent unclosed opener. "Most recent first" is the literal definition of LIFO.
  • "The most recent thing wins / gets processed first" — undo history, back button, collapsing adjacent duplicates ("remove all adjacent pairs"), simplifying a Unix path (.. cancels the most recent directory).
  • Deferred computation in postfix / left-to-right evaluation — Reverse Polish Notation, basic calculators. You stash operands until an operator arrives to consume them.
  • Design problems that bend LIFO — "build a queue out of stacks," "a stack that also returns its minimum in O(1)." These test whether you understand the structure well enough to compose it.

Reach for a monotonic stack when you see:

  • "Next greater / next smaller / previous greater / previous smaller element" — verbatim, this phrasing is the pattern's name tag.
  • "How long until X happens?" scanning forward — days until a warmer temperature, stock span, "first building taller than this one." Each element is waiting for a future element to resolve it.
  • "Largest area / widest range bounded by a minimum or maximum" — Largest Rectangle in Histogram, Trapping Rain Water (stack variant), Maximal Rectangle. The boundary of a region is defined by the nearest smaller element on each side.
  • An O(n²) brute force where the inner loop scans forward/backward for the first element beating the current one. If the inner scan stops at the first qualifying element, a monotonic stack almost always collapses it to O(n).
Say the recognition sentence out loud before you code: "Each element needs to find the first later element that's bigger — that's a next-greater query, so I'll keep a decreasing stack of indices that are still waiting for their answer." Enterprise-tier interviewers (JPMC, Goldman, and similar loops) explicitly score "identified the pattern and named the invariant" — it's often worth more than finishing the last edge case.
In JavaScript, a plain array is the stack. push() and pop() operate on the tail and are amortized O(1). Never use shift()/unshift() to simulate a stack or queue — they're O(n) because every remaining element re-indexes. This exact mistake is why the two-stack queue problem exists.

2. The templates

2a. Array-as-stack in JavaScript

No class needed. Four idioms cover everything:

const stack = [];

stack.push(x);                          // push — O(1) amortized
const top = stack[stack.length - 1];    // peek — O(1)  (or stack.at(-1))
const popped = stack.pop();             // pop — O(1); returns undefined if empty
const isEmpty = stack.length === 0;     // empty check
Always guard the empty stack. [].pop() silently returns undefined instead of throwing, so a missing stack.length check doesn't crash — it produces a wrong answer three lines later, which is much harder to debug at a whiteboard. Make while (stack.length && …) muscle memory.

2b. The monotonic-stack template

A monotonic stack is an ordinary stack with one rule enforced at push time: the values (from bottom to top) always stay sorted — decreasing for next-greater queries, increasing for next-smaller queries. Elements sit on the stack while they're unresolved: they haven't yet met the element that answers their query. When a new element arrives that would break the ordering, it is the answer for everything it beats — so you pop those, record their answers, then push the newcomer. 

// Template: "next greater element" for every index (decreasing stack)
function nextGreaterIndexes(nums) {
  const res = new Array(nums.length).fill(-1);  // -1 = no greater element exists
  const stack = [];  // holds INDICES whose answer is unknown; nums[…] strictly decreasing bottom→top

  for (let i = 0; i < nums.length; i++) {
    // nums[i] resolves every waiting element it beats
    while (stack.length && nums[stack[stack.length - 1]] < nums[i]) {
      const j = stack.pop();   // j's next greater element is i
      res[j] = i;              // store i, i - j (distance), or nums[i] — whatever the problem asks
    }
    stack.push(i);             // i now waits for ITS next greater element
  }
  // anything still on the stack keeps res = -1: nothing to its right ever beat it
  return res;
}

Line by line:

  • stack holds indices, not values. Indices let you compute distances (i - j) and recover values (nums[j]); values alone can't recover positions. Store raw values only when positions are irrelevant (see NGE I below).
  • The while is the engine. The top of the stack is the smallest unresolved element (stack is decreasing), so if the newcomer beats the top, keep popping — it may beat several. The moment it doesn't beat the top, stop: everything deeper is even bigger.
  • The push restores the invariant. After the pops, everything left on the stack is ≥ the newcomer, so pushing it keeps the stack decreasing. The invariant holds by induction — that's what makes the code correct without ever re-checking deeper elements.
  • Leftovers are the "no answer" cases. After the loop, elements still stacked never met a greater element; their pre-filled -1/0 stands.

Why this is O(n), not O(n²): ignore the nesting and count stack operations. Each index is pushed exactly once. Each index can be popped at most once — popping removes it forever. So across the entire run there are at most n pushes and n pops, ≤ 2n stack operations total. The inner while isn't "extra work per iteration"; it's spending pops from a budget of n. Some iterations pop many, most pop none, and the total is bounded. This is amortized analysis, and saying it unprompted — "each element is pushed and popped at most once, so it's O(n) total" — is a scoring moment in the interview.

3. Stack classics

Valid Parentheses (LC #20) EASY ★

You're given a string containing only the six bracket characters ( ) [ ] { }. Decide whether it's well-formed: every opening bracket must be closed by the same type of bracket, and brackets must close in the correct order — an inner pair must fully close before its enclosing pair does. An empty stack of leftovers at the end is required too: a dangling opener like "(" is invalid.

Example 1: Input: s = "([{}])" → Output: true — each closer matches the most recent unclosed opener.
Example 2: Input: s = "([)]" → Output: false — the ) arrives while [ is the most recent open bracket; types don't match.
Example 3: Input: s = "((" → Output: false — nothing is mismatched, but two openers are never closed.
Constraints: 1 ≤ s.length ≤ 10⁴; s consists only of ()[]{}.

Solve on LeetCode ↗

Thinking process. Try to solve it with a counter and you'll discover why a stack is necessary: a counter can verify balance for one bracket type, but with three types it can't remember which opener is currently awaiting closure — "([)]" has balanced counts for every type and is still invalid because of ordering. The rule "a closer must match the most recent unclosed opener" is a LIFO rule, so store openers on a stack. Walk the string once: on an opener, push it; on a closer, the top of the stack must be its partner — pop and compare, and if the stack is empty or the types differ, fail immediately. At the end, the stack must be empty (no abandoned openers). A tiny lookup map from each closer to its expected opener keeps the code branch-free and shows polish. Note the two distinct failure modes the examples exercise: wrong type / early closer (Ex 2) and leftover opener (Ex 3) — interviewers deliberately probe both.

Solution (JS) — O(n) time · O(n) space 
var isValid = function(s) {
  const pairs = { ')': '(', ']': '[', '}': '{' };  // closer → required opener
  const stack = [];

  for (const ch of s) {
    if (ch in pairs) {
      // ch is a closer: the most recent unclosed opener MUST be its partner.
      // pop() on an empty stack returns undefined, which !== pairs[ch],
      // so the "closer with nothing open" case fails here too — no extra branch.
      if (stack.pop() !== pairs[ch]) return false;
    } else {
      stack.push(ch);        // ch is an opener: it's now the most recent unclosed one
    }
  }

  return stack.length === 0;  // leftover openers ⇒ invalid
};

Trace on "([{}])": push (, push [, push {; } pops { ✓; ] pops [ ✓; ) pops ( ✓; stack empty → true. On "([)]": push (, push [; ) pops [ ≠ ( → false. ✓

Complexity: O(n) time — one pass, O(1) per character. O(n) space worst case (all openers, e.g. "(((((").

Follow-ups you should expect:

  • Early exit: if s.length is odd, return false before the loop — a one-liner that reads as attention to detail.
  • "Now the string contains other characters too (a code file)." Same algorithm — just continue past non-bracket characters. Say it; don't rewrite.
  • "Now * can be (, ), or empty" — that's LC 678 Valid Parenthesis String; the clean answer tracks a range [lo, hi] of possible open counts, O(1) space. Knowing the escalation path exists is enough at this tier.
Do not try to repeatedly delete "()"/"[]"/"{}" substrings with String.replace in a loop until the string stops changing. It's correct but O(n²) time (each pass copies the string) and interviewers at this tier will ask you to do better — you'll end up deriving the stack under pressure anyway. Start with the stack.

Implement Queue using Stacks (LC #232) EASY

Build a FIFO queue using only standard stack operations (push-to-top, peek-at-top, pop-from-top, empty). Implement the class MyQueue with push(x) (enqueue at the back), pop() (dequeue from the front and return it), peek() (return the front without removing), and empty(). Follow-up baked into the problem: achieve amortized O(1) per operation.

Example:
push(1); push(2); peek() → 1 (front is the oldest element); pop() → 1; empty() → false.
Constraints: 1 ≤ x ≤ 9; ≤ 100 calls; all pop/peek calls are valid (queue non-empty).

Solve on LeetCode ↗

Thinking process. A stack reverses order; the trick is that two reversals restore it. Keep an inbox stack that receives every push, and an outbox stack that serves every pop/peek. When the outbox is empty and you need the front, tip the entire inbox over into the outbox — popping each element and pushing it — which reverses it, so the oldest element lands on top of the outbox. The subtle point candidates miss: only transfer when the outbox is empty. If you transfer while the outbox still has elements, newer elements land on top of older ones and order breaks. The naive alternative — making push O(n) by shuffling everything through a helper stack on every insert — is correct but strictly worse; the two-stack version is the one that demonstrates you understand amortized cost.

Solution (JS) — amortized O(1) time per op · O(n) space 
class MyQueue {
  constructor() {
    this.inbox = [];   // receives pushes (newest on top)
    this.outbox = [];  // serves pops (oldest on top, because transfer reverses)
  }

  push(x) {
    this.inbox.push(x);                 // O(1) always
  }

  // Ensure outbox has the front element on top. Transfer ONLY when outbox is empty.
  _refill() {
    if (this.outbox.length === 0) {
      while (this.inbox.length) {
        this.outbox.push(this.inbox.pop());  // reverse inbox into outbox
      }
    }
  }

  pop() {
    this._refill();
    return this.outbox.pop();           // oldest element
  }

  peek() {
    this._refill();
    return this.outbox[this.outbox.length - 1];
  }

  empty() {
    return this.inbox.length === 0 && this.outbox.length === 0;
  }
}

Trace: push(1) → inbox [1]. push(2) → inbox [1,2]. peek() → outbox empty, transfer: pop 2 → outbox [2], pop 1 → outbox [2,1]; top is 1 ✓. pop() → outbox non-empty, no transfer, returns 1 ✓. empty() → inbox [] but outbox [2] → false ✓.

Complexity — the amortized argument (this is the actual interview content): a single pop() can cost O(n) when it triggers a transfer. But follow any one element through its lifetime: it is pushed onto inbox once, moved to outbox once, and popped once — at most 3 stack operations ever, regardless of interleaving. So m operations cost O(m) total, i.e. amortized O(1) each. Phrase it as "I charge the cost of the transfer to the elements being moved, and each element is only moved once." Space O(n) for n stored elements.

Follow-ups:

  • "Now implement a Stack using Queues" (LC 225): the mirror problem — on push, enqueue then rotate the queue n−1 times so the newest sits at the front; push O(n), pop O(1).
  • "Worst-case O(1) for every operation, not amortized?" Not possible with two plain stacks; real-time queues need incremental transfer schemes — fine to say "amortized is the standard answer; hard-real-time variants exist but aren't expected here."
  • Where this shows up for real: this is exactly how persistent/functional queues (e.g. in immutable Redux-style code or OCaml's Queue) are built from two immutable lists.
The two-stack queue is the reason shift() is a trap. A "queue" built as arr.push() + arr.shift() looks fine and passes small tests, but shift() re-indexes every remaining element, so draining n items is O(n²). At 10⁵ elements that's the difference between 1 ms and several seconds — and it's a real production bug pattern in JS job runners, not just an interview footnote.

Min Stack (LC #155) MEDIUM ★

Design a stack that supports push(val), pop(), top() and getMin() — retrieving the minimum element currently in the stack — with every one of those four operations running in O(1) time. Not amortized O(1), not O(log n): constant. The whole problem is that last requirement; drop it and this is a one-line wrapper around an array. Implement the class MinStack with a no-argument constructor.

Example 1: push(-2); push(0); push(-3); getMin() → -3; pop(); top() → 0; getMin() → -2 — popping -3 must restore the previous minimum, not recompute it from scratch.
Example 2: push(5); push(5); getMin() → 5; pop(); getMin() → 5 — duplicates of the minimum must survive one pop. This is the edge case that breaks the naive "keep a single min variable" answer.
Constraints: −2³¹ ≤ val ≤ 2³¹ − 1; pop/top/getMin are only called on a non-empty stack; up to 3 × 10⁴ calls.

Solve on LeetCode ↗

Thinking process. Start from the two answers that don't work and let them push you to the one that does. (1) Scan for the min in getMin(): O(n) per call — fails the requirement. (2) Cache a single min field: push updates it in O(1), but pop is the killer — pop the current minimum and you have no idea what the previous minimum was, so you'd have to rescan. That failure is the entire hint: the minimum is not a property of the stack, it's a property of each prefix of the stack — and a stack is nothing but a history of prefixes. So store the answer with the element: alongside each value, record "the minimum of everything at or below me." That number is computed once at push time in O(1) (Math.min(val, currentMin)) and it's immutable — nothing below you can ever change while you're on the stack, because the only way to reach those elements is to pop you first. getMin() is then just reading the top's second field, and pop() restores the old minimum for free by simply discarding the pair. Duplicates (Example 2) fall out automatically: each copy of 5 carries its own min = 5. The general principle — precompute per-element what an expensive query would recompute — is exactly useMemo: trade O(n) space for O(1) reads.

Solution (JS) — O(1) time per op · O(n) space 
class MinStack {
  constructor() {
    // One stack of PAIRS: [value, minOfEverythingAtOrBelowThisEntry].
    // The second field is frozen at push time and can never go stale, because
    // the only way to change what's below an entry is to pop the entry first.
    this.stack = [];
  }

  push(val) {
    const prevMin = this.stack.length
      ? this.stack[this.stack.length - 1][1]
      : Infinity;                       // empty stack ⇒ nothing beats val
    this.stack.push([val, Math.min(val, prevMin)]);
  }

  pop() {
    this.stack.pop();                  // discarding the pair automatically restores the old min
  }

  top() {
    return this.stack[this.stack.length - 1][0];
  }

  getMin() {
    return this.stack[this.stack.length - 1][1];  // O(1): just read it
  }
}

Trace on Example 1: push(-2) → stack [[-2,-2]]. push(0) → min(0, −2) = −2 → [[-2,-2],[0,-2]]. push(-3) → min(−3, −2) = −3 → [[-2,-2],[0,-2],[-3,-3]]. getMin() reads top's second field → -3 ✓. pop() drops [-3,-3]; top() → 0 ✓; getMin() → top is [0,-2] → -2 ✓ — restored without a single comparison.

Complexity: every operation is a constant number of array reads/writes ⇒ O(1) worst case, not amortized. Space is O(n): two numbers per element instead of one — a 2× constant factor, still linear. State the tradeoff explicitly: "I'm buying O(1) getMin with one extra number per element."

Follow-ups you should expect:

  • "Can you use less space?" Yes — keep a second stack of minima and only push onto it when val <= getMin(), popping it only when the value you pop equals its top. Same O(1) bounds, less memory when minima are rare. The <= (not <) is mandatory: with <, Example 2's second 5 never gets recorded and the first pop() wrongly removes the min. Interviewers plant duplicates specifically to catch this.
  • "Now do getMax too." Same pair trick with a third field — say it in one sentence, don't rewrite the class.
  • "O(1) space overhead?" There's an encoding trick storing 2*val - min deltas; it overflows in most languages and is a party trick, not a real answer. Mention that you know it exists and that you'd reject it in a code review — that's the senior response.
  • Why fintech loves this one: it's the interview version of a running-minimum on a price/risk stream — JPMC and Goldman ask it, then ask "now make it a sliding window minimum," which is LC 239 (monotonic deque — see the sliding-window tutorial). The bridge between those two problems is the same invariant.
The single-min-variable solution passes LeetCode's first sample and then fails, which is the worst possible outcome in a live interview — you've committed to an approach out loud. Kill it before you write it by asking yourself the pop question: "when I pop the minimum, where does the previous minimum come from?" If your design has no answer, it's wrong.

Evaluate Reverse Polish Notation (LC #150) MEDIUM

You're given an array of string tokens in Reverse Polish (postfix) notation — operands come first, and each operator applies to the two tokens immediately preceding it, e.g. ["2","1","+"] means 2 + 1. Evaluate the expression and return the integer result. Valid operators are + - * /. Division between two integers truncates toward zero (so 6 / -132 is 0, not -1). The input is always a valid expression, and operands may be negative.

Example 1: Input: ["2","1","+","3","*"] → Output: 9 — that's ((2 + 1) * 3).
Example 2: Input: ["4","13","5","/","+"] → Output: 6 — that's (4 + (13 / 5)) = 4 + 2: the / consumes the two tokens nearest it (13 and 5), not the first two.
Example 3: Input: ["5","-3","/"] → Output: -1 — truncation toward zero (-1.66… → -1). Math.floor would give -2 and fail. This is the test case that fails most submissions.
Constraints: 1 ≤ tokens.length ≤ 10⁴; each token is an operator or an integer in [−200, 200]; the expression is always valid.

Solve on LeetCode ↗

Thinking process. Postfix exists precisely because it's stack-evaluable: it has no parentheses and no precedence rules, since the order is already baked into the token sequence. That means there's no parsing to do — there's only deferred computation, which is trigger #3 from §1. Read tokens left to right. A number is an operand whose consumer hasn't arrived yet, so push it and wait. An operator is the consumer: it needs the two most recently-pushed operands — LIFO again — so pop two, apply, and push the single result back, because that result is itself an operand for some later operator. Two details decide whether you pass: pop order (the first pop is the right operand — it matters for - and /, and getting it backwards is the #1 bug here) and truncation toward zero (Math.trunc, never Math.floor). At the end, exactly one value remains on the stack: the answer. If you can articulate "postfix is a serialized post-order traversal of the expression tree, and the stack is reconstructing that tree bottom-up," you've said the thing that separates you from a bootcamp grad.

Solution (JS) — O(n) time · O(n) space 
var evalRPN = function(tokens) {
  // Operator table beats an if/else chain: adding operators later is a one-line change.
  const ops = {
    '+': (a, b) => a + b,
    '-': (a, b) => a - b,
    '*': (a, b) => a * b,
    '/': (a, b) => Math.trunc(a / b),   // trunc toward ZERO: -1.6 → -1 (Math.floor would give -2)
  };

  const stack = [];   // operands still waiting for an operator to consume them

  for (const t of tokens) {
    if (Object.hasOwn(ops, t)) {
      // ORDER MATTERS: the LAST pushed operand is the RIGHT-hand side.
      const b = stack.pop();          // right operand
      const a = stack.pop();          // left operand
      stack.push(ops[t](a, b));      // the result is itself an operand for a later operator
    } else {
      stack.push(Number(t));           // tokens are strings; "-3" parses fine, "+"/"-" are caught above
    }
  }

  return stack.pop();   // a valid expression leaves exactly one value behind
};

Trace on Example 2 ["4","13","5","/","+"]: push 4 → [4]; push 13 → [4,13]; push 5 → [4,13,5]; / → b = 5, a = 13, trunc(13/5) = 2 → [4,2]; + → b = 2, a = 4 → [6]; return 6 ✓. Note how / grabbed 13 and 5 — the two nearest operands — with no precedence logic anywhere in the code.

Complexity: O(n) time — each token is handled once with O(1) work (a push, or two pops and a push). O(n) space for the stack; the worst case is an expression that front-loads its operands (["1","2","3","4","+","+","+"] holds 4 at once). The stack depth equals the maximum operand nesting, i.e. the height of the expression tree.

Follow-ups you should expect:

  • "Now evaluate infix: "3+2*2"" — LC 227 Basic Calculator II. Same stack, but now precedence exists: push numbers, and when you meet a *// apply it immediately against the stack top; for - push the negated number; sum the stack at the end. Say "I'd handle precedence by resolving high-precedence operators eagerly" — you don't need to write it.
  • "Add parentheses" — LC 224. Now you push the running state onto a stack at ( and restore it at ). Or convert infix → postfix with the shunting-yard algorithm and reuse the code above. Naming shunting-yard is worth a point.
  • "What if the expression is invalid?" Guard: an operator with fewer than 2 operands on the stack, or a final stack length ≠ 1, means malformed input. Volunteering the validation contract is a production instinct that reads well at 7 YOE.
  • Division by zero — LeetCode guarantees it won't happen; in real code Math.trunc(1/0) is Infinity, not a throw. Mention it once, move on.
JavaScript specifics that bite here: Number("-3") → -3 ✓, but parseInt would also happily eat garbage suffixes. And t in ops works, but it walks the prototype chain — a token literally named "constructor" or "toString" would be treated as an operator. Object.hasOwn(ops, t) (or a Map) is the safe form. It's not exploitable in this problem's constraints, but noticing it is exactly the code-review instinct enterprise interviewers listen for.

Next Greater Element I (LC #496) EASY

You're given two arrays, nums1 and nums2, where nums1 is a subset of nums2 and all values in both arrays are distinct. For each value x in nums1, find it inside nums2 and return the first element to its right in nums2 that is greater than x. If no such element exists, the answer for that x is -1. Return the answers as an array aligned with nums1.

Example 1: Input: nums1 = [4,1,2], nums2 = [1,3,4,2] → Output: [-1,3,-1] — for 4, only 2 lies to its right and it's smaller → -1; for 1, the next element 3 is greater → 3; for 2, nothing lies to its right → -1.
Example 2: Input: nums1 = [2,4], nums2 = [1,2,3,4] → Output: [3,-1] — 2's next greater is 3; 4 is the last element, so -1.
Constraints: 1 ≤ nums1.length ≤ nums2.length ≤ 1000; 0 ≤ values ≤ 10⁴; all integers are unique; every element of nums1 also appears in nums2.

Solve on LeetCode ↗

Thinking process. The brute force writes itself: for each x, scan nums2 to find it, then scan forward to the first bigger value — O(n·m), and at these constraints it passes. Do not submit that; the interviewer is asking this to see the template. Notice the shape of the inner loop: scan forward for the first element beating the current one — trigger #4 from §1, verbatim. So flip the work around. Instead of asking "what comes after x?" for each query, do one left-to-right pass over nums2 and let each arriving element answer the elements it beats: keep a decreasing stack of values still waiting for a bigger neighbour, and when a bigger value shows up, pop every waiter it beats and record it as their answer in a Map. Then nums1 is just m map lookups. The knobs: direction = next, comparison = greater ⇒ decreasing stack, strictness moot (values are distinct), and — the one thing that's different from the template as written in §2b — you store values, not indices, because the output is a value and the positions in nums2 are never needed. That's knob #4, not a different algorithm. Say that out loud: "positions don't appear in the output, so I'll simplify the stack to hold values directly."

Solution (JS) — O(n + m) time · O(n) space 
var nextGreaterElement = function(nums1, nums2) {
  // value → its next greater value in nums2. Values are unique, so a value is a valid key.
  const answer = new Map();

  // The monotonic stack. Holds VALUES (not indices) — the template's knob #4:
  // the output is a value and no distance is ever needed, so indices buy us nothing.
  // Invariant: strictly DECREASING from bottom to top (next-GREATER query).
  const stack = [];

  for (const num of nums2) {
    // num resolves every waiter it beats — this is the whole algorithm.
    while (stack.length && stack[stack.length - 1] < num) {
      answer.set(stack.pop(), num);
    }
    stack.push(num);   // num now waits for ITS next greater element
  }
  // Anything still stacked never met a bigger element ⇒ absent from the map ⇒ -1 below.

  return nums1.map(x => answer.get(x) ?? -1);
};

Trace on Example 1 (nums2 = [1,3,4,2]): 1 → stack empty → push → [1]. 3 → beats top 1 → pop, map{1:3}; stack empty → push → [3]. 4 → beats 3 → pop, map{1:3, 3:4}; push → [4]. 2 → doesn't beat 4 → push → [4,2]. Leftovers 4 and 2 never got an answer. Now map nums1 = [4,1,2]: 4 not in map → -1; 1 → 3; 2 not in map → -1. Result [-1,3,-1] ✓.

Complexity: O(n + m) time — the pass over nums2 pushes and pops each element at most once (n pushes, ≤ n pops ⇒ O(n) amortized), then m O(1) map lookups. O(n) space for the stack and map. Compare out loud with the O(n·m) brute force; the interviewer is waiting for that sentence.

Follow-ups you should expect:

  • "What if values aren't distinct?" The Map keyed by value collapses duplicates — you'd have to key by index instead, which is why the §2b template stores indices by default. Being able to explain why the general template uses indices and this specialization doesn't is the real test here.
  • "Next Greater Element II — the array is circular" (LC 503): run the same loop over 2n iterations using i % n, and only push during the first lap. Two lines of change, same invariant.
  • "Next Greater Element III" (LC 556) is a different beast (next permutation of the digits) — don't be fooled by the shared name.
  • Previous greater element: same loop, but record the answer at push time (the surviving stack top is your previous-greater) instead of at pop time. That's knob #1 flipped.
Notice what just happened: the Map lookup at the end isn't part of the pattern at all — it's an indexing step glued onto a monotonic stack. Enterprise interviewers love composite problems like this because they check that you can decompose ("precompute all the answers once, then answer queries in O(1)") rather than pattern-match a single trick. Narrate the decomposition before you write a line: "I'll precompute next-greater for every element of nums2 in one pass, then answer nums1 by lookup."

Daily Temperatures (LC #739) MEDIUM ★

Given an array temperatures where temperatures[i] is the temperature on day i, return an array answer such that answer[i] is the number of days you have to wait after day i to get a warmer temperature. If no future day is warmer, answer[i] = 0. "Warmer" is strict: an equal temperature does not count.

Example 1: Input: [73,74,75,71,69,72,76,73] → Output: [1,1,4,2,1,1,0,0] — day 2 (75°) waits until day 6 (76°), a gap of 4; the last two days never see anything warmer.
Example 2: Input: [30,40,50,60] → Output: [1,1,1,0] — a strictly increasing run: every day is resolved immediately by the next one, so the stack never holds more than one element.
Example 3: Input: [30,30,30] → Output: [0,0,0] — equal is not warmer. This is the strictness knob, and it's the case that silently breaks a <= comparison.
Constraints: 1 ≤ temperatures.length ≤ 10⁵; 30 ≤ temperatures[i] ≤ 100. Note the length: 10⁵ means an O(n²) solution (10¹⁰ operations) is guaranteed to TLE — the constraint is telling you to find the linear pass.

Solve on LeetCode ↗

Thinking process. This is the canonical monotonic stack; if you can only rederive one problem in this tutorial under pressure, make it this one. Brute force: for each day, scan forward until you find a warmer day — O(n²), and the 10⁵ constraint kills it. Where's the waste? Consider [75, 71, 69, 72]. When day 3 (72) arrives, it resolves day 2 (69) and day 1 (71) in one shot — but the brute force rediscovers 72 separately from each of them. Worse, day 1 (71) is useless to day 0 (75): any future day that's warmer than 75 is automatically warmer than 71, so once 71 sits behind a bigger 75, 71 can never be the answer to anything 75 is still waiting for. Cooler, later elements shadow nothing and are answered first. That's the invariant, and it's exactly a decreasing stack: bottom-to-top the waiting days get cooler, because any day cooler than a previous day gets resolved earlier than it. So: walk left→right holding a stack of indices of days still waiting for a warmer day; when today beats the day on top, pop it and its answer is i - j (knob #4 = distance, not value); repeat while today keeps beating the top; then push today. Days left on the stack at the end keep their pre-filled 0. Knob #3, strictness: the problem says strictly warmer, so pop only on < — using <= would resolve equal-temperature days and produce a wrong answer on Example 3.

Solution (JS) — O(n) time · O(n) space 
var dailyTemperatures = function(temperatures) {
  const n = temperatures.length;
  const answer = new Array(n).fill(0);   // 0 = "never gets warmer" — the leftovers keep this

  // INDICES of days still waiting for a warmer day.
  // Invariant: temperatures[stack[…]] is strictly DECREASING bottom → top.
  const stack = [];

  for (let i = 0; i < n; i++) {
    // Today resolves every waiting day it is STRICTLY warmer than.
    // '<' not '<=': equal temperature is not "warmer", so equal days must keep waiting.
    while (stack.length && temperatures[stack[stack.length - 1]] < temperatures[i]) {
      const j = stack.pop();
      answer[j] = i - j;      // knob #4: record the DISTANCE, which is why we stored indices
    }
    stack.push(i);            // today now waits for ITS warmer day; invariant restored
  }

  return answer;
};

Trace on Example 1 [73,74,75,71,69,72,76,73]:

  • i=0 (73): stack empty → push. stack [0].
  • i=1 (74): 73 < 74 → pop 0, answer[0] = 1 - 0 = 1. Push 1. stack [1].
  • i=2 (75): 74 < 75 → pop 1, answer[1] = 1. Push 2. stack [2].
  • i=3 (71): 75 < 71? No → push 3. stack [2,3] (temps 75, 71 — decreasing ✓).
  • i=4 (69): 71 < 69? No → push 4. stack [2,3,4] (75, 71, 69 ✓).
  • i=5 (72): 69 < 72 → pop 4, answer[4] = 1. 71 < 72 → pop 3, answer[3] = 5 - 3 = 2. 75 < 72? No → stop. Push 5. stack [2,5]. One element resolved two waiters — that's the O(n²) work collapsing.
  • i=6 (76): 72 < 76 → pop 5, answer[5] = 1. 75 < 76 → pop 2, answer[2] = 6 - 2 = 4. Push 6. stack [6].
  • i=7 (73): 76 < 73? No → push 7. stack [6,7].
  • Leftovers 6 and 7 keep answer = 0. Result: [1,1,4,2,1,1,0,0] ✓.

Complexity: O(n) time. The nested while is not a second factor of n: each index is pushed exactly once and popped at most once, so total inner-loop iterations across the whole run are ≤ n. Say the sentence: "amortized — n pushes and at most n pops, so ≤ 2n stack operations total." Space O(n) for the stack, worst case a strictly decreasing input ([100,99,98,…]) where nothing ever resolves. The output array doesn't count against auxiliary space.

Follow-ups you should expect:

  • "Can you do it in O(1) extra space?" Yes, actually — scan right to left and reuse the answer array to jump: from day i, hop j = i + 1, then j += answer[j] while temperatures[j] <= temperatures[i], skipping over whole resolved blocks. It's O(1) auxiliary and still near-linear. Offering this after shipping the stack version is a strong close; leading with it is showing off.
  • "Now do it right-to-left." Equivalent formulation of the same problem — some interviewers ask for it to check you actually understand the invariant instead of having memorized one direction.
  • "Same question, but 'previous warmer day'" — flip knob #1: iterate the same way, but read your answer off the stack top before pushing.
  • Stock Span (LC 901) is this exact algorithm wearing a suit — "how many consecutive days back was the price ≤ today's?" It's a common fintech phrasing at JPMC/Goldman, and it's previous greater with the distance recorded. If you can say "that's Daily Temperatures with knob #1 flipped," you've won the round.
Don't store temperatures on the stack here. It feels cleaner (like NGE I above), but the answer is a distance, and you cannot recover a distance from a value — you'd need positions you threw away. Match the stack's payload to knob #4, the thing you record: value out ⇒ store values; distance or area out ⇒ store indices. Deciding this before you type prevents the most common mid-solution rewrite.

Car Fleet (LC #853) MEDIUM

n cars are driving toward the same destination, target miles away, on a single-lane road (no passing). Car i starts at position[i] and drives at a constant speed[i] mph. If a faster car catches up to a slower one, it is stuck behind it and they travel on as a single unit — a car fleet — at the slower car's speed. A single car is also a fleet. Cars that meet exactly at the target still count as having merged. Return how many fleets arrive at the target.

Example 1: Input: target = 12, position = [10,8,0,5,3], speed = [2,4,1,1,3] → Output: 3. The car at 10 (speed 2) arrives at t = 1. The car at 8 (speed 4) would arrive at t = 1 too — it catches the car ahead exactly at the target, so they merge into one fleet. The cars at 5 and 3 (times 7 and 3) — the car at 3 is faster and catches the car at 5 before the target, so they form a fleet arriving at t = 7. The car at 0 (speed 1) arrives at t = 12 alone. Three fleets.
Example 2: Input: target = 10, position = [3], speed = [3] → Output: 1 — one car is one fleet.
Example 3: Input: target = 100, position = [0,2,4], speed = [4,2,1] → Output: 1 — arrival times from the back are 25, 49, 96: each car is faster than the one ahead of it, so everything piles up behind the slowest car at position 4 and arrives as a single fleet.
Constraints: 1 ≤ n ≤ 10⁵; 0 < target ≤ 10⁶; 0 ≤ position[i] < target; all positions are unique; 0 < speed[i] ≤ 10⁶.

Solve on LeetCode ↗

Thinking process. Simulating the road (advance time, detect collisions, merge) is a trap — it's fiddly, floating-point-y, and O(n²) at best. Reframe. The only thing that matters about a car is when it would reach the target if nothing were in its way: time = (target − position) / speed. Now the physics collapses to one rule. Take two cars, A ahead of B (A has the bigger position). B catches A before the target iff timeB <= timeA — B "wants" to arrive no later than A but can't pass, so B is absorbed and the merged fleet arrives at timeA, the time of the car ahead. And crucially, the fleet's time is the maximum of its members' times, so a merged fleet is indistinguishable from its lead car.

That gives you the algorithm: sort cars by position, closest to the target first, and walk them. Maintain a stack of the arrival times of the fleet leaders you've committed to so far. For each car, compare its time to the top of the stack (the leader immediately ahead of it): if time <= top, it gets swallowed — don't push it, and note that the fleet's time is unchanged (still top). Otherwise it's slower than everything ahead, it will never catch anyone, and it starts a brand-new fleet — push it. The answer is the stack's height.

Map it to the knobs: this is a next-style left-to-right sweep after sorting (knob #1), the stack is kept increasing in arrival time bottom→top (knob #2), ties merge so the comparison is <= (knob #3), and what you "record" is nothing at all — just the surviving count (knob #4). The twist worth naming out loud: here the newcomer is the one that disappears, not the stack top, because a car behind can never overtake — the leader's time always wins. Same invariant, inverted absorption. Say that and the interviewer knows you derived it rather than recalled it.

Solution (JS) — O(n log n) time · O(n) space 
var carFleet = function(target, position, speed) {
  // 1. Pair each car with its speed, then sort by position DESCENDING:
  //    process the car closest to the target first, because it can only be blocked
  //    by cars ahead of it — and there are none.
  const cars = position
    .map((p, i) => [p, speed[i]])
    .sort((a, b) => b[0] - a[0]);

  // 2. Stack of ARRIVAL TIMES of committed fleet leaders.
  //    Invariant: strictly INCREASING bottom → top (each new fleet is slower than the one ahead).
  const stack = [];

  for (const [p, s] of cars) {
    const time = (target - p) / s;    // hours to reach target IF the road were empty

    // If this car would arrive no later than the fleet directly ahead, it catches it
    // (at or before the target) and is absorbed. The fleet's arrival time is unchanged —
    // it's the MAX of its members' times, which is already on top of the stack.
    // '<=' not '<': meeting exactly AT the target still counts as a merge.
    if (stack.length && time <= stack[stack.length - 1]) continue;

    // Otherwise it's slower than everything ahead: it never catches up, so it leads
    // a new fleet. Pushing it keeps the stack increasing.
    stack.push(time);
  }

  return stack.length;   // one entry per surviving fleet
};

Trace on Example 1 (target = 12, positions [10,8,0,5,3], speeds [2,4,1,1,3]). Sorted by position descending, with times (12 − p) / s:

  • p=10, s=2 → time 1. Stack empty → push. Stack [1].
  • p=8, s=4 → time 1. 1 <= 1 → merges (catches it exactly at the target). Skip. Stack [1].
  • p=5, s=1 → time 7. 7 <= 1? No → new fleet. Stack [1, 7].
  • p=3, s=3 → time 3. 3 <= 7 → it catches the fleet ahead → absorbed. Skip. Stack [1, 7] — note the fleet still arrives at 7, the slower leader's time.
  • p=0, s=1 → time 12. 12 <= 7? No → new fleet. Stack [1, 7, 12].

Stack height 3 ✓ — and notice the stack is strictly increasing, exactly as the invariant promises.

Complexity: O(n log n) time, dominated by the sort — the stack sweep itself is O(n) (each car is examined once and pushed at most once). Say that explicitly; a candidate who reports "O(n) because monotonic stack" has stopped thinking. O(n) space for the paired/sorted array and the stack.

Follow-ups you should expect:

  • "Do you even need the stack?" No — since you only ever compare against the current maximum time, a single let maxTime = 0; if (time > maxTime) { fleets++; maxTime = time; } gives O(1) space. Ship the stack version, then offer this. The stack is the derivation; the running max is the optimization. Interviewers love watching a candidate simplify their own correct solution.
  • "Floating point?" Division makes times non-integers. With these constraints doubles are exactly precise enough, but the bulletproof alternative is comparing (target − p₁) · s₂ against (target − p₂) · s₁ — cross-multiplication, integers only. In fintech interviews (JPMC, Goldman) volunteering "I'd avoid floating-point comparison in production money/timing code" lands well.
  • "Car Fleet II" (LC 1776, hard) asks when each car collides — that one does need a genuine pop-based monotonic stack processed right-to-left. Naming it is enough at this tier.
  • "What if positions can repeat?" The constraints forbid it; ask what the tie means physically (two cars in the same spot?) rather than guessing. Asking beats assuming.
The sort direction is the single highest-leverage decision in this problem. Sorting ascending by position (the reflex) forces you to reason backwards about who catches whom, and most candidates who do it that way never recover. Sort descending — nearest to the target first — and the rule becomes local and one-directional: "can I catch the thing immediately in front of me?" When a problem has an obvious one-way dependency (a car can only be blocked by cars ahead), order your traversal so that the thing you depend on has already been computed. That's the same instinct as topological ordering, and as memoized recursion.

Hard problems are rarely asked at the enterprise tier — an Optum or JPMC loop is far more likely to give you Valid Parentheses and Daily Temperatures and grade your communication than to hand you a hard. Do this one last, and do it because it's the final exam for the §2b template: every knob gets turned, and the width computation forces you to actually understand what the stack means. If you can derive Largest Rectangle from the template, you own monotonic stacks permanently.

Largest Rectangle in Histogram (LC #84) HARD ★

You're given an array heights where heights[i] is the height of the i-th bar of a histogram; every bar has width 1 and the bars are adjacent, sitting on a common baseline. Return the area of the largest rectangle that can be drawn inside the histogram. The rectangle must be axis-aligned and contiguous: it spans a range of consecutive bars, and its height is capped by the shortest bar in that range. So a rectangle over bars [l, r] has area (r − l + 1) × min(heights[l..r]).

Example 1: Input: [2,1,5,6,2,3] → Output: 10 — the winner is the rectangle of height 5 spanning bars 2 and 3 (heights 5 and 6): 5 × 2 = 10. Bar 3 alone gives only 6; spanning all six bars gives 1 × 6 = 6, because the height-1 bar caps everything.
Example 2: Input: [2,4] → Output: 4 — the taller single bar beats the 2-wide rectangle of height 2.
Example 3: Input: [2,2,2] → Output: 6 — all equal: the rectangle is the whole histogram. Equal heights are where the strictness knob and the "under-computed width" worry live; the solution below handles them for a reason worth understanding.
Constraints: 1 ≤ heights.length ≤ 10⁵; 0 ≤ heights[i] ≤ 10⁴. Again 10⁵ ⇒ O(n²) will TLE, and note heights can be 0.

Solve on LeetCode ↗

Thinking process. Enumerating all (l, r) ranges is O(n²) (or O(n³) if you recompute the min). Kill that by changing what you enumerate. Every maximal rectangle is capped by some bar — so let each bar i be the shortest bar of its rectangle, and ask: how far can that rectangle stretch? Answer: as far left and as far right as the bars stay ≥ heights[i]. It stops at the first strictly shorter bar on each side. So the rectangle for bar i is heights[i] × (rightSmaller[i] − leftSmaller[i] − 1), and the answer is the max over all i. Read that back: you need the next smaller element and the previous smaller element for every bar — and that is literally the §2b template with knob #2 flipped from greater to smaller, which flips the stack from decreasing to increasing.

Now the elegant part, and the thing to say out loud: you don't need two passes — one increasing stack gives you both boundaries at once. When bar i arrives and forces you to pop bar j, that pop tells you two things simultaneously:

  • The right boundary of j is i. You're popping j because heights[i] is smaller — i is by definition the first bar to the right of j that's shorter. The rectangle can't include i, so it ends at i − 1.
  • The left boundary of j is whatever is underneath j on the stack. The stack is increasing, so the element below j is the nearest bar to the left that is shorter than j — everything between them was already popped, which means everything between them was ≥ heights[j]. The rectangle can't include that lower bar either, so it starts at left + 1. If the stack is empty after the pop, nothing to the left is shorter: the rectangle extends all the way to index 0, which you model as a virtual boundary at left = −1.

Hence width = i − left − 1: the span is the open interval (left, i) — both boundary bars are excluded, which is where the two subtractions come from. This one line is where everyone breaks, so don't trust the algebra from memory — plug in real numbers on the whiteboard, every time. Two checks that take five seconds: (a) heights = [5] — the lone bar is popped by the sentinel at i = 1 with an empty stack beneath it ⇒ width = 1 − (−1) − 1 = 1 ✓, one bar wide, as it must be. (b) heights = [2,2,2] — the last bar (index 2) is popped at i = 3 with an empty stack beneath ⇒ width = 3 − (−1) − 1 = 3 ⇒ area 6 ✓, the whole histogram.

Last piece: the sentinel. When the loop ends, bars are still on the stack — the increasing tail of the histogram, which nothing ever popped, so their areas were never computed. Two fixes: drain the stack in a second loop after the main one (using i = n as the right boundary), or — cleaner, and the version to write — run the loop to i = n inclusive and treat the phantom bar at index n as having height 0. Zero is ≤ every possible height, so it pops the entire stack and computes every remaining area with the correct right boundary of n. One virtual bar, and the special case disappears entirely. Naming that trick — "I'll append a zero-height sentinel so the flush is just another iteration" — is a strong senior signal, and it's the same instinct as a null terminator or a dummy head node in a linked list.

Solution (JS) — O(n) time · O(n) space 
var largestRectangleArea = function(heights) {
  const n = heights.length;
  let best = 0;

  // INDICES of bars whose rectangle is still unresolved (we don't yet know how far right it goes).
  // Invariant: heights[stack[…]] is non-decreasing bottom → top — knob #2 flipped to
  // "next SMALLER", so the stack is INCREASING (the mirror of the next-greater template).
  const stack = [];

  // i runs to n INCLUSIVE: index n is a virtual sentinel bar of height 0.
  // Height 0 is ≤ everything, so it flushes the whole stack with right boundary n.
  for (let i = 0; i <= n; i++) {
    const h = i === n ? 0 : heights[i];

    // Bar i is shorter than the top ⇒ the top's rectangle CANNOT extend past i.
    // So i is the top's right boundary, and we can finally compute its area.
    while (stack.length && heights[stack[stack.length - 1]] >= h) {
      const height = heights[stack.pop()];

      // After popping, the new top is the nearest bar to the LEFT that is shorter
      // than the popped bar (everything between was taller, so it got popped earlier).
      // Empty stack ⇒ nothing shorter to the left ⇒ virtual boundary at -1.
      const left = stack.length ? stack[stack.length - 1] : -1;

      // Span is the OPEN interval (left, i): exclude both boundary bars ⇒ two -1's collapse to one.
      const width = i - left - 1;

      best = Math.max(best, height * width);
    }

    stack.push(i);   // bar i now waits to find ITS next smaller bar; invariant restored
  }

  return best;
};

Trace on Example 1 [2,1,5,6,2,3] (n = 6, so the sentinel fires at i = 6):

  • i=0 (h=2): stack empty → push. stack [0].
  • i=1 (h=1): heights[0]=2 ≥ 1 → pop 0. height 2; stack now empty ⇒ left = −1; width = 1 − (−1) − 1 = 1; area 2. best = 2. Push 1. stack [1].
  • i=2 (h=5): heights[1]=1 ≥ 5? No → push. stack [1,2] (heights 1, 5 — increasing ✓).
  • i=3 (h=6): heights[2]=5 ≥ 6? No → push. stack [1,2,3] (1, 5, 6 ✓).
  • i=4 (h=2): heights[3]=6 ≥ 2 → pop 3. height 6; new top is 2 ⇒ left = 2; width = 4 − 2 − 1 = 1; area 6. best = 6.
    Still: heights[2]=5 ≥ 2 → pop 2. height 5; new top is 1 ⇒ left = 1; width = 4 − 1 − 1 = 2; area 10. best = 10. ← the answer, and look where the width came from: right boundary i=4 (the bar of height 2 that stopped it), left boundary index 1 (the bar of height 1 beneath it), both excluded.
    heights[1]=1 ≥ 2? No → stop. Push 4. stack [1,4].
  • i=5 (h=3): heights[4]=2 ≥ 3? No → push. stack [1,4,5].
  • i=6 (sentinel, h=0): heights[5]=3 ≥ 0 → pop 5. left = 4; width = 6 − 4 − 1 = 1; area 3.
    heights[4]=2 ≥ 0 → pop 4. left = 1; width = 6 − 1 − 1 = 4; area 8 (the height-2 rectangle spanning bars 2..5 — it clears bars of height 5, 6, 2, 3 ✓).
    heights[1]=1 ≥ 0 → pop 1. stack empty ⇒ left = −1; width = 6 − (−1) − 1 = 6; area 6 (the full-width height-1 rectangle ✓).
    Stack empty. Every bar's rectangle has now been measured.
  • Return best = 10 ✓.

Why >= and not > (the duplicate-heights question): with equal heights, popping on >= means an earlier equal bar is popped by a later equal bar, and its width gets cut short. That looks like a bug — and it isn't, because the later equal bar survives, and when it is eventually popped it sees a left boundary that reaches back past the earlier one (which is no longer on the stack), so it computes the full width. The maximum is therefore still found; some non-maximal areas are just under-counted, which is harmless since you're taking a Math.max. Verify on [2,2,2]: bar 0 pops at i=1 with width 1 (area 2), bar 1 pops at i=2 with width 2 (area 4), bar 2 pops at the sentinel with left = −1 and width 3 (area 6) ✓. > also works and is arguably cleaner to justify — but >= keeps the stack strictly increasing, which is easier to reason about. Either way, have the answer ready: "what about duplicate heights?" is the #1 follow-up on this problem.

Complexity: O(n) time — the outer loop runs n + 1 times and each index is pushed exactly once and popped at most once, so the inner while does ≤ n total iterations across the whole run (the same amortized argument as every other problem in §4). O(n) space for the stack; the worst case is a strictly increasing histogram ([1,2,3,…,n]) where nothing gets popped until the sentinel.

Follow-ups you should expect:

  • "Maximal Rectangle" (LC 85) — a binary matrix; find the largest all-1s rectangle. The move: for each row, build a histogram where heights[c] = the number of consecutive 1s ending at that row in column c, then run this exact function on each row's histogram. O(rows × cols). Recognizing "that's just LC 84 called m times" is the whole problem; interviewers who ask 85 are really asking whether you own 84.
  • "Trapping Rain Water with a stack" (LC 42) — the same machinery with the comparison inverted (a decreasing stack, popping to compute the water trapped between two walls). Two pointers is the better answer there; knowing both is the flex.
  • "Do it without the sentinel." Then you need an explicit drain loop after the main loop, with i = n as the right boundary. Same code, duplicated. State that the sentinel exists purely to eliminate that duplication — that's a design argument, not a trick.
  • "Can you do it with divide and conquer?" Yes — recurse on the minimum bar: the answer is the best of (full width × min height), left half, right half. It's O(n log n) average but O(n²) on a sorted histogram, so it's strictly worse. Say so and stay with the stack.
The reason this problem is a rite of passage isn't the code — it's nine lines — it's that you cannot fake understanding of width = i − left − 1. If you can stand at a whiteboard, point at the popped bar, point at the bar that stopped it on the right, point at the bar beneath it on the stack, and say "the rectangle lives strictly between these two, so its width is the gap minus the two boundary bars", you have demonstrated the single skill this entire tutorial exists to build: the stack isn't storage, it's a record of unresolved obligations, and every pop is one obligation being settled with exactly the information needed to settle it.

Interview Q&A

Your monotonic-stack solution has a loop inside a loop. Justify O(n) to a skeptical interviewer.

Don't analyze the worst case of a single iteration and multiply — analyze the total work. Count stack operations, not loop iterations: each index is pushed exactly once, and once popped it is gone forever, so it can be popped at most once. Across the entire run there are ≤ n pushes and ≤ n pops — at most 2n stack operations, each O(1). The inner while isn't extra work per outer iteration; it's spending pops from a global budget of n. One iteration might pop 50 elements, but then 50 other iterations pop nothing. That's amortized analysis — the same reasoning that makes Array.prototype.push O(1) despite occasional reallocation, and the same reasoning behind the two-stack queue in §3. Say the sentence unprompted: "each element is pushed and popped at most once, so it's O(n) total."

Should the stack hold indices or values? How do you decide before you start typing?

Look at knob #4 — what you record at pop time — and let it dictate the payload:

  • Output is a value and positions never appear in the answer → values are fine (Next Greater Element I).
  • Output is a distance (i - j) → you need indices (Daily Temperatures).
  • Output is an area/width computed from both boundaries → you definitely need indices (Largest Rectangle).
  • The values might contain duplicates and you're keying a Map by them → indices, always.

The safe default is indices: you can always recover a value with nums[i], but you can never recover a position from a value. Deciding this in the first fifteen seconds prevents the most common mid-solution rewrite in the whole pattern.

Decreasing stack or increasing stack — how do you pick, without memorizing?

Derive it from the query, don't recall it. The stack holds elements that are still waiting. An element leaves the stack the moment the newcomer answers it.

  • Query is "next greater" → a bigger newcomer resolves the waiter → you pop when newcomer > top → therefore whatever survives is ≥ the newcomer → the stack is decreasing bottom→top.
  • Query is "next smaller" → mirror everything → the stack is increasing.

One-line check that never fails: the stack is sorted in the opposite direction of the thing you're hunting. Hunting bigger ⇒ decreasing stack. Largest Rectangle hunts the next smaller bar (that's what caps a rectangle), which is why its stack is the increasing one.

Where do duplicates break a monotonic stack, and how does the strictness knob fix it?

Knob #3 (< vs <=) is entirely about equal values, and it's usually the only thing separating a correct submission from a wrong one:

  • Daily Temperatures demands strictly warmer, so pop only on <. Using <= would resolve an equal-temperature day and return 1 instead of 0 on [30,30,30].
  • Car Fleet says a car that arrives at exactly the same time still merges, so the comparison is <=.
  • Min Stack's two-stack variant must push onto the min-stack on <=, not < — otherwise a duplicate minimum isn't recorded and the first pop() destroys it.
  • Largest Rectangle tolerates either, uniquely, because the later duplicate re-derives the full width once the earlier one is off the stack — and you're taking a max, so an under-counted intermediate is harmless.

The move in the interview: ask. "Do equal values count?" is a free point, and the answer sets your operator.

Next greater vs. previous greater — what actually changes in the code?

Almost nothing, and that's the point. Iterate left→right either way; what changes is when you read the answer:

  • Next greater: the answer for element j is discovered at pop time — the newcomer that evicts j is j's answer.
  • Previous greater: the answer for element i is available at push time — after you've popped everything i beats, whatever is left on top is, by the invariant, the nearest larger element to i's left (empty stack ⇒ none exists).

Same loop, same invariant, one extra read before the push. That's knob #1. (Stock Span, LC 901, is exactly this: previous-greater with the distance recorded — and it's a frequent fintech phrasing.) The alternative — iterate right→left for "previous" — also works and is a fine thing to mention, but it's not necessary.

What's the sentinel trick, and where else does it show up?

A sentinel is a fake element appended (or prepended) to the input so that the boundary case stops being a special case. In Largest Rectangle, a virtual bar of height 0 at index n is smaller than every real bar, so it pops the whole stack — which means the "drain the leftovers after the loop" code disappears; the flush is just the last iteration of the normal loop. Same idea, different clothes: the -1 virtual left boundary when the stack empties; a dummy head node in linked-list problems; a null terminator; a <Fragment> wrapper. Frame it as a design choice, not a trick: "I'm adding one impossible element so that the loop body is total — no branch for the last case, less code to get wrong." That framing reads as an experienced engineer; "it's a trick I remembered" doesn't.

Why can't you just use shift() to build a queue in JavaScript?

Because Array.prototype.shift() removes from the front, which forces every remaining element to move down one slot — O(n) per call, so draining n items is O(n²). (Engines optimize some cases, but you cannot rely on it and you certainly can't claim it in a complexity analysis.) push/pop operate on the tail and are amortized O(1), which is why an array is a fine stack but a poor queue. Real options: keep a head index that you increment instead of shifting (O(1), at the cost of memory that isn't reclaimed until you compact), use a linked list, or use the two-stack construction from §3 — which is precisely why LC 232 exists. Volunteering this in a BFS/graph problem ("I'll use an index-based queue rather than shift so the traversal stays O(V + E)") is cheap, real senior signal.

Connect this to something you already use as a frontend engineer.

Three genuine connections worth having ready:

  • The call stack. Recursion is a stack — the engine pushes a frame per call and pops on return. Any recursive algorithm can be rewritten with an explicit stack, which is exactly how you avoid a RangeError: Maximum call stack size exceeded on a deeply skewed tree. Saying "if the input could be deep I'd convert this recursion to an explicit stack" is a two-second sentence with real weight.
  • Undo/redo and browser history. Two stacks, and the redo stack gets cleared on a new action — which is the same "most recent first" LIFO discipline as bracket matching.
  • Min Stack ≈ useMemo. You precompute per-element what an expensive query would otherwise recompute, spending O(n) space to buy O(1) reads. Same trade, different scale.

Do not force an analogy where it doesn't fit — a strained "monotonic stacks are like the virtual DOM" costs you more than saying nothing.

You're 20 minutes in and the monotonic solution isn't coming. What do you do?

Write the brute force, out loud, immediately — "let me get a correct O(n²) on the board first, then optimize." At the enterprise tier a working brute force with a clear optimization plan often scores better than a half-finished clever solution, because you're graded on speed, clean code, and narration. Then attack the brute force's inner loop with one question: "is my inner scan looking for the first element that beats the current one?" If yes, it's a monotonic stack — state the invariant ("I'll keep a decreasing stack of indices still waiting for their answer"), and start writing the template from §2b. Even if you run out of time, you've said the sentence the rubric is looking for. Silence is the only unrecoverable failure mode.
