02 — Two Pointers

Two indices walking an array so that every step permanently eliminates a possibility — collapsing O(n²) into O(n).

Why interviewers ask this

Two Pointers is the cheapest test an interviewer has for "does this candidate think, or does this candidate enumerate?" Almost every problem on this page has an obvious nested-loop solution that any bootcamp graduate can write in three minutes. The entire interview happens in the next three minutes: can you find the structural property — sortedness, symmetry, or an in-place compaction goal — that lets you throw away half the search space with each move? At the enterprise tier (Optum, JPMC, Goldman Sachs and their peers) Valid Palindrome and Move Zeroes are stock phone-screen warm-ups, and 3Sum and Container With Most Water are the most-asked onsite mediums after Two Sum itself. They're asked precisely because the brute force is so tempting: the interviewer wants to watch you reject it and justify the replacement.

There's a second reason, and it's the one that trips up frontend engineers specifically. Half of this pattern — the reader/writer family — is about mutating an array in place with O(1) extra space. As a React developer you have spent seven years being told never to mutate, to always filter and map into fresh arrays, because immutability is what makes reference-equality diffing work. That instinct is correct in React and actively wrong here. When an interviewer says "do it in place, O(1) extra space," they are asking you to hand-write Array.prototype.filter with no output buffer — and the candidates who fail are the ones who reach for a new array on reflex and then can't get rid of it. Section 2b makes that translation explicit. Learn to switch modes on the word "in-place," and say the word "in-place" back to them, so they know you heard it.

1. Recognizing the pattern

There are two completely different algorithms hiding under the one name "two pointers," and candidates conflate them constantly. They share only the fact that two indices are in flight. Everything else — where the pointers start, which way they move, what invariant they maintain, what the problem is even asking for — is different. Learn to name which family you're in before you write a line.

Reach for converging pointers (left = 0, right = n - 1, move inward) when:

  • The input is sorted — or you're allowed to sort it. This is the loudest signal in all of DSA. "Given a sorted array…" in a problem statement is the interviewer telling you the answer. Sortedness is what makes "the sum is too small, so move left up" a valid deduction rather than a guess.
  • You're searching for a pair (or triple) that satisfies a target condition — sums to k, differs by k, multiplies to k. Two Sum II, 3Sum, 4Sum, Two Sum Less Than K.
  • The problem is about symmetry — palindromes, reversing, "compare the string with itself backwards." Here the two ends are compared rather than summed, but the motion is identical.
  • You're maximizing something computed from a pair of endpoints, and the quantity has a monotone component you can exploit — Container With Most Water is the archetype.

Reach for reader/writer pointers (both start at the left, move at different speeds) when:

  • The words "in place" or "O(1) extra space" appear. Verbatim. If the problem forbids an output array, it is telling you to keep a write cursor.
  • You're removing, keeping, or partitioning elements by a predicate — "remove all instances of val," "move zeroes to the end," "remove duplicates from a sorted array," "partition into evens then odds."
  • The return value is a length, not an array — "return k, the number of elements remaining; what you leave beyond index k doesn't matter." That return signature is a reader/writer fingerprint.
  • Three-way partitioning — Dutch National Flag / Sort Colors uses three cursors, but it's the same family: writers at both ends, a reader sweeping between them.
Say the recognition sentence out loud before you code. For converging: "The array is sorted, so I can start at both ends and let the comparison tell me which pointer to move — each move eliminates a whole set of pairs permanently, which is why this is O(n) instead of O(n²)." For reader/writer: "This is an in-place filter — I'll keep a write index for where the next kept element goes, and a read index sweeping ahead of it." Enterprise-tier rubrics score "named the pattern and stated the invariant" as its own line item; it often carries more weight than nailing the last edge case.
A third family, fast/slow pointers on linked lists (Floyd's cycle detection, find-the-middle), is sometimes filed under "two pointers." It's a genuinely different idea — the pointers move at different rates through a structure with no random access — and it lives in 06 — Linked Lists. Don't confuse the reader/writer "slow/fast" naming (same speed conceptually, one just skips writes) with the linked-list "slow/fast" (literally 1× and 2× speed).

2. The templates

2a. Converging pointers

Two indices start at opposite ends and walk toward each other. The loop runs while they haven't crossed, and on each iteration you evaluate the pair, then move exactly one pointer inward — the one whose movement is justified by the ordering. 

// Template: find a pair in a SORTED array summing to target
function convergingPair(nums, target) {
  let left = 0;
  let right = nums.length - 1;

  while (left < right) {          // strict <: a pointer must never pair with itself
    const sum = nums[left] + nums[right];

    if (sum === target) {
      return [left, right];        // found it
    } else if (sum < target) {
      left++;                       // need a BIGGER sum. nums[left] is the smallest value
                                    // still in play, so it can't help — discard it forever.
    } else {
      right--;                      // need a SMALLER sum. nums[right] is the largest value
                                    // still in play — discard it forever.
    }
  }
  return [-1, -1];                // pointers crossed: no pair exists
}

Line by line:

  • left < right, not left <= right. Equality means both pointers sit on the same element, and "a pair" almost always means two distinct indices. Use <= only when a single element is a legal answer in its own right (rare — e.g. some binary-search-flavored scans). Getting this wrong is the single most common two-pointer bug.
  • Exactly one pointer moves per iteration. If you ever move both in the same branch, you've skipped pairs and the algorithm is wrong. (The one legitimate exception is after recording a hit, when the problem wants all distinct pairs — then you advance both, because you've already consumed this pair. 3Sum does exactly that.)
  • The comparison tells you which pointer to move — you never choose. That's the whole pattern. sum < target means the current left is too small to ever work with any remaining partner, because every remaining partner is ≤ nums[right], and nums[left] + nums[right] is already short. So nums[left] is dead. Kill it and never look back.
  • Loop exit is a proof of absence. When the pointers cross, every pair has been either tested or provably eliminated — so returning "not found" is sound, not a guess.

Why this is O(n), not O(n²) — and the sentence to say out loud: the brute force tests all n(n−1)/2 pairs. This loop runs at most n times, because left and right only ever move toward each other and the gap between them shrinks by one every iteration. The reason we're allowed to skip the other pairs is the load-bearing part: every step eliminates a possibility permanently. When you do left++, you are not merely "moving on" — you are asserting that nums[left] cannot be half of the answer with any partner still in range, and that assertion is licensed by sortedness. Say that sentence in the interview verbatim: "Each move eliminates a possibility permanently, so I visit O(n) pairs instead of O(n²) — and sortedness is what makes the elimination valid." That second clause is what separates a 7-YOE answer from a memorized one.

Converging pointers on an unsorted array is nonsense, and interviewers plant this trap. If the array isn't sorted, sum < target tells you nothing — a bigger value could be anywhere. Either sort first (accepting O(n log n) and the loss of original indices) or use a hash map (plain Two Sum, LC 1, covered in 01 — Arrays & Hashing). Choosing between "sort + two pointers" and "hash map" is itself an interview question: hash map wins on time (O(n)) and preserves indices; sort + two pointers wins on space (O(1) beyond the sort) and is the only one that generalizes to 3Sum.

2b. Reader/writer pointers

Both pointers start at the left. The reader (read, sometimes called fast) sweeps every element exactly once. The writer (write, or slow) marks the slot where the next kept element belongs, and only advances when something is actually kept. The writer therefore lags behind the reader by exactly the number of elements discarded so far. 

// Template: in-place filter. Keep every element satisfying keep(x).
// Returns k = the number of kept elements; nums[0..k-1] are them, in order.
function inPlaceFilter(nums, keep) {
  let write = 0;               // next slot to write a KEPT element into

  for (let read = 0; read < nums.length; read++) {
    if (keep(nums[read])) {
      nums[write] = nums[read];  // compact it leftward into the kept region
      write++;                   // writer only advances on a KEEP — this is the whole trick
    }
    // if we don't keep it, write stays put and the next kept element overwrites this slot
  }

  return write;                // == count of kept elements == length of the compacted prefix
}

Line by line:

  • The invariant is everything: at the top of every iteration, nums[0 .. write-1] is the finished answer so far, and nums[write .. read-1] is garbage you're free to overwrite. State that invariant out loud — it makes the correctness self-evident and it's exactly what the interviewer is listening for.
  • write <= read, always. The writer starts equal to the reader and can only fall behind (it skips advances; the reader never does). So you are never writing to a slot the reader hasn't already read. This is why overwriting in place is safe and why you don't need a backup copy — and it is the one-sentence proof of correctness.
  • No else branch. Discarding an element is the absence of work. Candidates who add an else { /* skip */ } or try to swap discarded elements to the back are overcomplicating it — unless the problem cares about what's left in the tail (see Move Zeroes, where it does).
  • The return value is a length. LeetCode's in-place problems grade nums[0..k-1] and ignore the tail. Returning write — not the array — is the correct signature and shows you read the spec.
The JS connection you already know. That template is Array.prototype.filter. Really: const kept = nums.filter(keep) does the same job in one line. The only difference is that filter allocates a fresh output array — O(n) extra space — and reader/writer reuses the input buffer, giving O(1). That is precisely why interviewers forbid the extra array: they are testing whether you can express filter as a loop with a write cursor, which is what every language's standard library does under the hood. When the interviewer says "in place, O(1) space," the mental translation is "write filter without the output buffer." Say that. It reframes a puzzle into a chore you already know how to do.

Why this is O(n) time and O(1) space: the reader touches each of the n elements exactly once and does O(1) work per element; the writer never moves backward, so total writer movement is also bounded by n. That's ≤ 2n index movements, so O(n) — and unlike converging pointers, there's no subtlety here; it's a single for loop. Space is O(1): two integer cursors, no allocation. Note the contrast with converging pointers, where O(n) needed an argument (the elimination lemma). Here it's obvious. The hard part of reader/writer is never the complexity — it's resisting the urge to allocate.

3. Converging pointers

Valid Palindrome (LC #125) EASY ★

Given a string s, decide whether it is a palindrome after converting all uppercase letters to lowercase and removing every character that is not a letter or a digit. Punctuation, spaces, and symbols are simply invisible for the purposes of the check. By this definition the empty string counts as a palindrome, since it reads the same forward and backward.

Example 1: Input: s = "A man, a plan, a canal: Panama" → Output: true — after cleaning, the string is "amanaplanacanalpanama", which reads identically in reverse.
Example 2: Input: s = "race a car" → Output: false — cleans to "raceacar"; the first character r and the last r match, but a vs a… then c vs c… and finally e vs a disagree.
Example 3: Input: s = " " → Output: true — every character is stripped, leaving the empty string. This is the edge case interviewers actually check.
Constraints: 1 ≤ s.length ≤ 2×10⁵; s consists only of printable ASCII.

Solve on LeetCode ↗

Thinking process. The one-liner is irresistible and you should mention it: clean with a regex, lowercase, then compare against the reversed string — const t = s.toLowerCase().replace(/[^a-z0-9]/g, ''); return t === [...t].reverse().join(''). It's correct, and it's O(n) time. But it's O(n) space: you built a cleaned copy, then a character array, then a reversed array, then a joined string. The interviewer's follow-up is guaranteed and it is always the same: "can you do it in O(1) space?" The insight is that a palindrome is defined by symmetry, and symmetry is checkable from the outside in: put one pointer at each end, and repeatedly compare. The cleaning step doesn't need a new string either — instead of removing junk characters up front, each pointer just skips over junk characters as it walks. When both pointers have landed on alphanumerics, compare them (case-folded); if they differ, you're done — return false immediately. If they match, they've done their job: step both inward and forget them. That's the whole algorithm, and it never allocates anything bigger than a single character.

Solution (JS) — O(n) time · O(1) space 
var isPalindrome = function(s) {
  // Helper: is this character a letter or a digit? Regex on a single char is fine
  // and reads clearly; a charCode range check is faster but noisier.
  const isAlnum = (c) => /[a-z0-9]/i.test(c);

  let left = 0;
  let right = s.length - 1;

  while (left < right) {
    // Advance past junk. The `left < right` guard inside each skip loop is
    // essential: a string of pure punctuation would otherwise run off the end.
    while (left < right && !isAlnum(s[left]))  left++;
    while (left < right && !isAlnum(s[right])) right--;

    // Both pointers now sit on alphanumerics (or have met). Case-fold and compare.
    if (s[left].toLowerCase() !== s[right].toLowerCase()) return false;

    // Matched pair: it can never invalidate the palindrome, so retire both.
    left++;
    right--;
  }

  // Pointers crossed or met without a mismatch ⇒ every symmetric pair agreed.
  return true;
};

Trace on "race a car": left=0(r), right=9(r) → match, move to 1, 8. a vs a → match, move to 2, 7. c vs c → match, move to 3, 6. left=3 is e; right=6 is   (space) → the right skip loop runs, right=5 (a). Now e vs a → mismatch → false. ✓ On " ": left=0, right=0, the while (left < right) guard is immediately false, so we never enter the loop → true. ✓

Complexity: O(n) time — each pointer only moves inward, so together they traverse the string at most once (n total moves, whether the move is a skip or a match). O(1) space — two integers and one transient lowercased character. Contrast with the regex version's O(n) space, and volunteer that contrast before you're asked.

Follow-ups you should expect:

  • "Now you're allowed to delete at most one character" — LC 680, Valid Palindrome II. On the first mismatch, the answer is a palindrome iff s[left+1..right] or s[left..right-1] is one. Recurse into a helper that checks a plain range. Still O(n): the branch happens at most once.
  • "Why not String.prototype.reverse?" — it doesn't exist; strings are immutable in JS, which is exactly why the naive approach must allocate. Knowing that is a small credibility win.
  • "Unicode?" — s[i] indexes UTF-16 code units, so an emoji or an astral-plane character is two "characters" and would be split. For real code you'd iterate [...s] or use Intl.Segmenter. LeetCode constrains to ASCII, so you note the caveat and move on.
Forgetting the left < right guard inside the two skip loops. Write while (!isAlnum(s[left])) left++; and the input ",.;" walks left straight off the end of the string, where s[left] is undefined and isAlnum(undefined) throws (or, worse, silently returns false and loops forever). Every skip loop in a converging-pointer solution needs the crossing guard duplicated inside it. This is the #1 bug in this problem and interviewers feed you " " specifically to catch it.

Two Sum II — Input Array Is Sorted (LC #167) MEDIUM

You're given a 1-indexed array numbers that is already sorted in non-decreasing order, and a target. Find the two numbers that add up to target and return their 1-based indices as [index1, index2] with index1 < index2. Exactly one solution is guaranteed to exist, and you may not use the same element twice. The catch that makes it a Medium rather than an Easy: you must solve it using only constant extra space — so the hash map you'd reach for in plain Two Sum is off the table.

Example 1: Input: numbers = [2,7,11,15], target = 9 → Output: [1,2] — 2 + 7 = 9, at 1-based positions 1 and 2.
Example 2: Input: numbers = [2,3,4], target = 6 → Output: [1,3] — the pointers must pass over the middle element 3 entirely; the answer is the two ends.
Example 3: Input: numbers = [-1,0], target = -1 → Output: [1,2] — negatives are allowed; nothing about the logic changes, which is worth noticing since it confirms the method depends on ordering, not on sign.
Constraints: 2 ≤ numbers.length ≤ 3×10⁴; −1000 ≤ numbers[i] ≤ 1000; numbers is sorted non-decreasing; −1000 ≤ target ≤ 1000; exactly one solution exists.

Solve on LeetCode ↗

Thinking process. This is the problem that shows why sortedness is what makes converging pointers work, so derive it properly rather than pattern-matching. Brute force: try every pair, O(n²). Hash map (the plain-Two-Sum move): one pass, O(n) time — but O(n) space, which the problem explicitly bans. So you need something that exploits the one thing this problem gives you that Two Sum doesn't: the array is sorted. Here's the exploitation. Put left at the smallest element and right at the largest, and look at sum = numbers[left] + numbers[right]. Three cases:

  • sum === target → done.
  • sum < target → the sum is too small. Now ask: could numbers[left] be part of the answer at all? Its best possible partner is the biggest remaining number, which is numbers[right] — and even that pairing falls short. Every other candidate partner is ≤ numbers[right], so it would fall even shorter. Therefore numbers[left] is in no valid pair. Discard it: left++. That is a proof, not a heuristic.
  • sum > target → mirror image. numbers[right]'s best partner is the smallest remaining number, numbers[left], and that pairing already overshoots. So numbers[right] is in no valid pair. right--.

Each iteration permanently eliminates one element — and with it, all O(n) pairs that element could have formed. That's how you go from n²/2 pairs to at most n iterations. And notice exactly where sortedness entered: it's the only reason you could say "its best possible partner is numbers[right]." Shuffle the array and that sentence becomes false, the elimination becomes invalid, and the algorithm silently returns wrong answers. Say this out loud in the interview; it is the entire point of the problem.

Solution (JS) — O(n) time · O(1) space 
var twoSum = function(numbers, target) {
  let left = 0;
  let right = numbers.length - 1;

  while (left < right) {
    const sum = numbers[left] + numbers[right];

    if (sum === target) {
      // Problem is 1-INDEXED. Forgetting this is the most common wrong answer.
      return [left + 1, right + 1];
    }

    if (sum < target) {
      // numbers[left] is the smallest value left in play, and even paired with
      // the largest (numbers[right]) it undershoots ⇒ it's in no valid pair.
      left++;
    } else {
      // Symmetric: numbers[right] overshoots even with the smallest partner.
      right--;
    }
  }

  return [-1, -1];  // unreachable given the guarantee, but never leave a loop unclosed
};

Trace on [2,7,11,15], target 9: left=0(2), right=3(15) → sum 17 > 9, so 15 can't work with anything → right=2(11) → sum 13 > 9, so 11 can't work with anything → right=1(7) → sum 9 === target → return [0+1, 1+1] = [1,2]. ✓ Note the algorithm eliminated 15 and 11 without ever testing them against 7 — those pairs were provably dead.

Complexity: O(n) time — the gap right - left strictly decreases every iteration and starts at n-1, so there are at most n-1 iterations, each O(1). O(1) space — two integers, no allocation. That's the constraint the problem was built around.

Follow-ups you should expect:

  • "How does this differ from Two Sum (LC 1)?" — the killer question, and the answer is the whole tutorial. LC 1's array is unsorted and asks for original indices, so sorting would destroy the answer; you use a hash map, O(n) time / O(n) space. Here the array is pre-sorted and space is constrained, so you use converging pointers, O(n) time / O(1) space. Different inputs, different tools. Naming the tradeoff unprompted is a strong signal.
  • "What if the array weren't sorted and you still needed O(1) space?" — sort it yourself: O(n log n) time, O(1) extra space (in-place sort), then run this. Slower, but it meets the space bound. This is exactly the move 3Sum makes.
  • "Could you binary search instead?" — yes: for each i, binary search for target - numbers[i] in the suffix. O(n log n), O(1) space. Correct but strictly worse than the O(n) two-pointer scan; worth mentioning to show range, then discarding.
  • "What if multiple pairs exist and you want them all?" — on a hit, record it and move both pointers (skipping duplicate values on each side), then keep going. That's precisely the inner loop of 3Sum, coming up next.
Interviewers at JPMC and Goldman often ask LC 1 and LC 167 back to back, and the point is not to see you solve both — it's to see whether you notice they are different problems. A candidate who reflexively writes a hash map for LC 167 has pattern-matched on the words "two sum" instead of reading the constraints. Read the constraints out loud. "Sorted" and "constant extra space" are not decoration; they are the problem.

3Sum (LC #15) MEDIUM ★

Given an integer array nums, return all triplets [nums[i], nums[j], nums[k]] with i, j, k all distinct, such that nums[i] + nums[j] + nums[k] === 0. The catch — and the entire interview — is that the result set must not contain duplicate triplets. Two triplets count as duplicates if they contain the same multiset of values, regardless of the order they appear in or which indices produced them. The output triplets may be returned in any order.

Example 1: Input: nums = [-1,0,1,2,-1,-4] → Output: [[-1,-1,2],[-1,0,1]] — note there are two different index-triples that produce [-1,0,1] (using either of the two -1s), and only one of them may appear in the output. That is the trap.
Example 2: Input: nums = [0,1,1] → Output: [] — no triplet sums to zero; a correct solution must return the empty list, not crash on the short array.
Example 3: Input: nums = [0,0,0,0] → Output: [[0,0,0]] — four zeros admit four distinct index-triples, all producing the same value-triplet. Exactly one goes in the output. This is the case that breaks naive duplicate handling.
Constraints: 3 ≤ nums.length ≤ 3000; −10⁵ ≤ nums[i] ≤ 10⁵.

Solve on LeetCode ↗

Thinking process. Brute force is three nested loops, O(n³) — at n = 3000 that's 2.7×10¹⁰ operations, dead on arrival. State it, price it, discard it. Now the reduction that makes this problem famous: fix one element and the rest is Two Sum. If nums[i] is in the triplet, then the other two must sum to -nums[i]. So loop i over the array (O(n)) and, for each, solve Two Sum on the remainder for target -nums[i]. If you solve that inner Two Sum with a hash map you get O(n²) time and O(n) space — acceptable, but deduplication becomes miserable. If instead you sort the array first, the inner Two Sum becomes the converging-pointer scan from LC 167: O(n) per i, O(1) space, and — critically — sorting puts equal values adjacent, which turns duplicate elimination from a set-of-serialized-keys hack into two clean skip loops. Total: O(n log n) for the sort + O(n²) for the scans = O(n²).

Now the part that is actually the interview: duplicate skipping. There are exactly two places duplicates can enter, they are independent, and candidates almost always find one and miss the other. Be explicit about both:

  • Skip 1 — duplicate anchors (the outer i). If nums[i] === nums[i-1], then i is a repeat of an anchor you already fully processed. Every triplet it could produce was already produced by i-1, whose scan searched a superset of the range (it started one slot earlier). So continue. The comparison must be against i-1 (the element behind you), not i+1. Comparing forward would skip the first occurrence of a repeated value and keep the last — which is wrong for [0,0,0,0] in a subtle way and wrong outright when the anchor needs its own duplicates as partners: in [-1,-1,2] the anchor -1 pairs with the other -1. Guard with i > 0 so index -1 is never read.
  • Skip 2 — duplicate partners (after recording a hit). When left + right completes a valid triplet, you push it and then must advance. Advancing only left is a bug: if nums[left+1] === nums[left] you'd immediately re-find the identical triplet. So advance both pointers past the hit, and then run two skip loops: while (left < right && nums[left] === nums[left-1]) left++; and the mirror on the right. Both guarded by left < right, because [0,0,0,0] will otherwise walk them through each other.

A crucial detail about Skip 2: only skip duplicates after a successful hit, not on every iteration. If the sum is too small you just left++ — no skip needed, because you haven't emitted anything, so landing on a duplicate value is harmless (it'll produce the same sum and get rejected the same way; correctness is untouched, and adding the skip there is a common over-correction that costs you nothing but reads as confused). The final polish: after sorting, if nums[i] > 0 you can break outright — the array is ascending, so the anchor and everything after it are positive and can never sum to zero. Small, free, and interviewers notice.

Solution (JS) — O(n²) time · O(1) extra space (excluding output & sort) 
var threeSum = function(nums) {
  // Sort so that (a) converging pointers become valid, and (b) equal values sit
  // adjacent, which makes duplicate-skipping a simple neighbour comparison.
  nums.sort((a, b) => a - b);   // numeric comparator — NOT the default lexicographic one!

  const res = [];
  const n = nums.length;

  for (let i = 0; i < n - 2; i++) {
    // Early exit: array is ascending, so once the anchor is positive the two
    // larger elements after it are positive too — sum can never reach 0.
    if (nums[i] > 0) break;

    // SKIP 1 — duplicate ANCHOR. Compare BACKWARD (i-1), never forward.
    // i-1 already scanned a superset of this range, so i can only re-find its triplets.
    if (i > 0 && nums[i] === nums[i - 1]) continue;

    // Inner problem: find pairs in nums[i+1 .. n-1] summing to -nums[i].
    // This is verbatim Two Sum II on a sorted subarray.
    let left = i + 1;
    let right = n - 1;

    while (left < right) {
      const sum = nums[i] + nums[left] + nums[right];

      if (sum < 0) {
        left++;             // too small: nums[left] is dead (see Two Sum II proof)
      } else if (sum > 0) {
        right--;            // too big: nums[right] is dead
      } else {
        res.push([nums[i], nums[left], nums[right]]);

        // Consumed this pair, so move BOTH pointers — advancing only one would
        // re-find the same triplet if the neighbour is a duplicate.
        left++;
        right--;

        // SKIP 2 — duplicate PARTNERS, only after a hit. Both guards are needed:
        // [0,0,0,0] would otherwise march the pointers straight through each other.
        while (left < right && nums[left] === nums[left - 1])  left++;
        while (left < right && nums[right] === nums[right + 1]) right--;
      }
    }
  }

  return res;
};

  • i=0 (-4): not > 0, no predecessor. left=1(-1), right=5(2) → sum -3 < 0 → left=2(-1) → sum -3 < 0 → left=3(0) → sum -2 < 0 → left=4(1) → sum -1 < 0 → left=5, crossed. Nothing found — correct, -4 is in no zero triplet.

  • i=2 (-1): Skip 1 fires — nums[2] === nums[1] → continue. This is what prevents the second [-1,0,1] from being emitted. Without this line, the output has a duplicate and the submission fails.

Trace on [0,0,0,0] (the Skip-2 stress test): i=0, left=1, right=3 → sum 0 → push [0,0,0]. Both move: left=2, right=2. Complexity: Time O(n²) — the sort is O(n log n), then the outer loop runs n times and each inner converging scan is O(n) (the pointers cover the subarray once), giving O(n²), which dominates. Space: O(1) auxiliary if you don't count the output list, plus whatever the sort uses — be honest and say "O(log n) to O(n) depending on the engine's sort implementation; V8's TimSort is O(n)." The output itself can hold O(n²) triplets in the worst case, but output space is conventionally excluded.

Follow-ups you should expect:

  • "Where exactly can duplicates enter, and how does each skip prevent them?" — this is the question. Rehearse the two-skip answer until it's fluent: duplicate anchors (outer, compare to i-1) and duplicate partners (inner, only after a hit, compare to the slot you just left). If you can only articulate one, you have not learned this problem.
  • "Could you dedupe with a Set instead?" — yes: serialize each sorted triplet as a string key and dedupe at the end. It works and it's O(n²) still, but it costs O(n²) space in the worst case and reads as a workaround for not understanding where the duplicates come from. Mention it, then explain why you prefer the skips.
  • "Now 4Sum" (LC 18) — same idea, one more nested anchor loop with its own backward-looking duplicate skip: O(n³). The generalization is kSum: recurse, peeling one anchor per level with a dedupe skip, until k = 2, where you drop into this two-pointer scan. Being able to describe kSum in two sentences is a strong close.
  • "3Sum Closest" (LC 16) — identical scaffold, but instead of testing sum === 0 you track Math.abs(sum - target) against the best seen, and there's no dedupe at all (you return a single number). It's the easiest possible variant once you own this one.
nums.sort() with no comparator sorts lexicographically by string in JavaScript, so [10, 9, -1] becomes [-1, 10, 9]. Every converging-pointer deduction is then garbage and you get a wrong answer that looks plausible on small inputs. Always write nums.sort((a, b) => a - b). This is a JS-specific trap that costs candidates real interviews, and an interviewer who has seen it before will be watching your fingers when you type that line.

Container With Most Water (LC #11) MEDIUM ★

You're given an integer array height of length n, where height[i] is the height of a vertical line drawn at x-coordinate i. Pick two of these lines so that they, together with the x-axis, form a container. Return the maximum amount of water such a container can hold. The water level is capped by the shorter of the two chosen lines (water spills over the short side), and the width is the horizontal distance between them — so for lines at indices i and j, the area is min(height[i], height[j]) × (j − i). You may not slant the container, and the lines between your two choices do not block anything — ignore them entirely.

Example 1: Input: height = [1,8,6,2,5,4,8,3,7] → Output: 49 — pick the lines at indices 1 (height 8) and 8 (height 7): min(8,7) × (8−1) = 7 × 7 = 49. Notice the winner uses neither the tallest pair nor the widest pair; it's the best product.
Example 2: Input: height = [1,1] → Output: 1 — the only pair: min(1,1) × 1 = 1. The minimum legal input.
Example 3: Input: height = [1,2,4,3] → Output: 4 — indices 2 and 3 give min(4,3) × 1 = 3; indices 1 and 3 give min(2,3) × 2 = 4. Width can beat height.
Constraints: n == height.length; 2 ≤ n ≤ 10⁵; 0 ≤ height[i] ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. Brute force: every pair, O(n²). At n = 10⁵ that's 5×10⁹ — too slow, so say so and move on. Now the key observation: the array is not sorted and you can't sort it (sorting would destroy the x-coordinates, which are the width). So what justifies converging pointers here? Not sortedness — a greedy exchange argument, and this is the thing interviewers actually probe. Start with left = 0 and right = n - 1: the widest possible container. Every other container is narrower, so the only way any of them beats this one is by being taller. Compute its area, then ask which pointer to move — and here's the crux, which you must be able to prove, not assert.

The proof that moving the shorter wall is safe. This is the whole interview at Goldman and JPMC-tier loops; they will ask "why is that correct?" and a candidate who says "because it's the bottleneck" without an argument gets marked down. Deliver it like this:

Suppose height[left] < height[right], so left is the shorter wall. I claim no container using left can ever beat the one I just measured — so I can retire left forever.

Consider any other container that still uses left. Its partner is some j with left < j < right (everything outside [left, right] is already retired). Two things happen to it, and both are bad:

  • Width strictly shrinks: j − left < right − left, because j < right.
  • Height cannot grow: the height is min(height[left], height[j]), and that is ≤ height[left] no matter what height[j] is. But the container I just measured already had height exactly height[left] (since left was the shorter wall, min(height[left], height[right]) = height[left]). So the new height is ≤ the old height.

Strictly smaller width × no-bigger height ⇒ strictly smaller area. Every remaining container involving left is worse than the one I already recorded. Therefore discarding left loses nothing, and I move left++. Symmetric argument if right is the shorter wall. If they're equal, both are capped by that height and you can move either (or both) — the same argument applies to each.

The punchline is the sentence from §2: each step eliminates a possibility permanently. Every left++ throws away all O(n) containers that used that wall — with proof — which is exactly why n²/2 pairs collapse to n iterations.

With the proof in hand the code is six lines. Note what you're not doing: you're not tracking the max height, not sorting, not caring about the lines in between. Track only the best area seen and let the exchange argument drive the pointers.

Solution (JS) — O(n) time · O(1) space 
var maxArea = function(height) {
  let left = 0;
  let right = height.length - 1;
  let best = 0;

  while (left < right) {
    // Water spills over the SHORTER wall, so it caps the level.
    const h = Math.min(height[left], height[right]);
    const w = right - left;
    best = Math.max(best, h * w);

    // Retire the shorter wall. Proof: any other container using it would be
    // strictly narrower AND capped at a height no greater than this one's,
    // so it cannot beat the area we just recorded. Nothing is lost.
    if (height[left] < height[right]) {
      left++;
    } else {
      right--;   // covers the tie: both walls are capped at the same height,
                 // so retiring either is safe — no special case needed.
    }
  }

  return best;
};

  • left=0(1), right=8(7) → area min(1,7) × 8 = 8. best = 8. Left is shorter → left=1.
  • left=1(8), right=8(7) → area min(8,7) × 7 = 49. best = 49. Right is shorter (7 < 8) → right=7.
  • left=1(8), right=7(3) → area 3 × 6 = 18. Right shorter → right=6.
  • left=1(8), right=6(8) → area 8 × 5 = 40. Tie → right=5.
  • left=1(8), right=5(4) → area 4 × 4 = 16 → right=4.
  • left=1(8), right=4(5) → area 5 × 3 = 15 → right=3.
  • left=1(8), right=3(2) → area 2 × 2 = 4 → right=2.
  • left=1(8), right=2(6) → area 6 × 1 = 6 → right=1, crossed. Return 49. ✓

Notice the algorithm never evaluated the pair (0, 4) or (2, 8) or dozens of others — the exchange argument killed them off, each one provably.

Complexity: O(n) time — exactly one pointer moves per iteration and the gap shrinks by one each time, so at most n − 1 iterations, O(1) work each. O(1) space — three integers. This is the full O(n²) → O(n) collapse, and the only reason it's sound is the exchange argument; without it, this code is an unjustified heuristic that happens to pass.

Follow-ups you should expect:

  • "Prove that moving the shorter wall can't skip the optimum." — the exchange argument above, verbatim. If you get asked this and freeze, the rest of the interview goes badly regardless of whether your code passed. Rehearse it as prose.
  • "What if the walls were equal — could you move both at once?" — yes, and it's a valid micro-optimization: on a tie, both walls are capped at the same height, so each is individually retirable by the same argument. Moving both saves one iteration. Correct, but don't lead with it; lead with the proof.
  • "How is this different from Trapping Rain Water?" — the classic pairing, and they're genuinely different. Here the bars have zero width and you pick exactly two of them, ignoring everything between; there, the bars have width, they block water, and you sum the water over every position. Same converging-pointer skeleton, different quantity. See the Stretch section.
  • "Could you do better than O(n)?" — no. You must at minimum read every height; any unread bar could be the tall one that makes the answer. Ω(n) lower bound. Saying this crisply closes the problem.
A tempting-but-wrong instinct here is to move the pointer at the taller wall, reasoning "the tall wall is already good, let's find it a better partner." That's backwards: the tall wall's potential is preserved by keeping it (it can still be the cap for a future pair), whereas the short wall's potential is already maximized right now, at the maximum available width. Keep what still has upside; retire what doesn't. That framing also generalizes — it's the same instinct behind Trapping Rain Water's pointer choice.

Remove Duplicates from Sorted Array (LC #26) EASY

Given an integer array nums sorted in non-decreasing order, remove the duplicates in place so that each unique element appears exactly once, preserving the relative order. Return k, the number of unique elements. Your solution is graded on the first k slots: after you return, nums[0..k-1] must hold the unique elements in their original order, and whatever remains in nums[k..] is ignored entirely — it does not matter what you leave there. You must use O(1) extra memory: allocating a second array and copying back is explicitly disallowed.

Example 1: Input: nums = [1,1,2] → Output: k = 2, with nums = [1,2,_] — the underscore is any value; the grader stops at index k−1.
Example 2: Input: nums = [0,0,1,1,1,2,2,3,3,4] → Output: k = 5, with nums = [0,1,2,3,4,_,_,_,_,_] — runs of length 3 exercise the "skip several in a row" path.
Example 3: Input: nums = [7] → Output: k = 1 — the single-element case, where a naive nums[read] !== nums[read - 1] check would read index −1.
Constraints: 1 ≤ nums.length ≤ 3×10⁴; −100 ≤ nums[i] ≤ 100; nums is sorted non-decreasing.

Solve on LeetCode ↗

Thinking process. Your JS reflex is [...new Set(nums)] — one line, correct, and it allocates two O(n) structures, so it fails the space bound. The interviewer will not accept it and you should preempt them: "that's the idiomatic answer but it's O(n) space; the problem wants in-place, so I need a write cursor." Now, the array is sorted, which means duplicates are adjacent — an element is a duplicate if and only if it equals its immediate predecessor. That's the whole insight; the sortedness turns a "have I seen this before?" question (which needs a Set) into a "is this the same as the last one?" question (which needs one comparison).

So: keep(x) is "x differs from the last element I kept." Set write = 1 (index 0 is trivially unique — a run of one, nothing to compare it to) and read from index 1. For each read, compare nums[read] against nums[write - 1] — the last element actually written, not nums[read - 1]. Both happen to work here (because in a sorted array the last written element always equals the previous distinct value), but nums[write - 1] is the version that generalizes: it's the one that still works for the "allow at most two duplicates" follow-up, where you compare against nums[write - 2]. Reach for the general form now and the follow-up is a one-character edit.

Solution (JS) — O(n) time · O(1) space 
var removeDuplicates = function(nums) {
  if (nums.length === 0) return 0;   // constraints forbid it, but cost-free rigor

  // nums[0] is always unique (it's the first element of the first run),
  // so it is already "written". Start reading from index 1.
  let write = 1;

  for (let read = 1; read < nums.length; read++) {
    // Sorted ⇒ duplicates are adjacent ⇒ "seen before?" reduces to
    // "equal to the last thing I KEPT?". Compare to nums[write - 1],
    // not nums[read - 1] — the write-relative form generalizes (see follow-up).
    if (nums[read] !== nums[write - 1]) {
      nums[write] = nums[read];   // compact this new distinct value leftward
      write++;                    // writer advances ONLY on a keep
    }
    // duplicate ⇒ do nothing; write stays put and this slot gets overwritten later
  }

  return write;   // == count of distinct values == length of the valid prefix
};

Trace on [0,0,1,1,1,2,2,3,3,4]: start write=1. read=1: nums[1]=0 === nums[0]=0 → skip. read=2: 1 !== 0 → nums[1]=1, write=2. Array is now [0,1,1,1,1,2,2,3,3,4] — note index 1 was overwritten, and that's fine, because the reader is already past it (write ≤ read, always). read=3,4: both 1 === nums[1]=1 → skip. read=5: 2 !== 1 → nums[2]=2, write=3. read=6: skip. read=7: 3 !== 2 → nums[3]=3, write=4. read=8: skip. read=9: 4 !== 3 → nums[4]=4, write=5. Return 5, with nums[0..4] = [0,1,2,3,4]. ✓ On [7]: the loop never runs (read=1 ≥ length) → return 1. ✓

Complexity: O(n) time — one pass, one comparison and at most one assignment per element. O(1) space — two integer cursors, zero allocation. Note the invariant that makes the in-place overwrite safe: write ≤ read at all times, because the writer starts equal and only ever falls behind. You are never clobbering a value you haven't already read.

Follow-ups you should expect:

  • "Allow each element at most twice" (LC 80) — the reason to have written nums[write - 1]. Start write = 2, read from 2, and keep when nums[read] !== nums[write - 2]. That single change generalizes to "at most k times." If you'd hardcoded nums[read - 1] you'd have to redesign; this way you edit one digit. Interviewers love this because it tests whether your first solution was principled or lucky.
  • "Remove Element" (LC 27) — the same skeleton with keep = nums[read] !== val; no sortedness needed at all, since the predicate doesn't depend on neighbours. The purest instance of the template.
  • "What if the array weren't sorted?" — then adjacency no longer implies duplication and you genuinely need a Set, so O(n) space becomes unavoidable (or you sort first, at O(n log n), if order may be destroyed). Naming what sortedness bought you is the senior answer.
Do not try to splice() duplicates out of the array as you go. It looks in-place — no new array literal appears in your code — but every splice shifts all subsequent elements, making it O(n) per removal and O(n²) overall, and it mutates the array underneath your loop index so you also start skipping elements. The same objection applies to shift() and unshift(). "In place" means overwrite slots with a write cursor, not call array methods that internally re-index.

Move Zeroes (LC #283) EASY

Given an integer array nums, move all the 0s to the end of it while maintaining the relative order of the non-zero elements. You must do this in place, without making a copy of the array. Unlike Remove Duplicates, the tail is not ignored here — the grader checks the entire array, so the zeroes must actually be present at the end, in the right quantity.

Example 1: Input: nums = [0,1,0,3,12] → Output: [1,3,12,0,0] — the non-zeroes keep their order (1 before 3 before 12) and both zeroes land at the back.
Example 2: Input: nums = [0] → Output: [0] — no non-zeroes at all; the write cursor never advances and the whole array is zero-filled (i.e. left alone).
Example 3: Input: nums = [1,2,3] → Output: [1,2,3] — no zeroes at all; every element is kept, the writer never falls behind the reader, and every write is a self-assignment. Worth noting because it's the case a swap-based solution has to handle gracefully.
Constraints: 1 ≤ nums.length ≤ 10⁴; −2³¹ ≤ nums[i] ≤ 2³¹ − 1. Follow-up: minimize the total number of operations.

Solve on LeetCode ↗

Thinking process. Reframe it and it becomes a problem you've already solved: "move the zeroes to the end" is the same as "keep the non-zeroes, compacted to the front, then fill the rest with zeroes." That's the §2b in-place filter with keep(x) = x !== 0, followed by a cleanup loop. So:

  • Pass 1 — reader/writer. For each element, if it's non-zero, write it at write and advance the writer. At the end, nums[0..write-1] holds every non-zero value in original order, and write is how many there were.
  • Pass 2 — from write to the end, set every slot to 0. Exactly the right number of zeroes, because you kept exactly the non-zeroes.

Two passes, still O(n). Now the follow-up the problem literally prints — "minimize the total number of operations" — points at a slicker single-pass version: instead of copying then zero-filling, swap nums[read] and nums[write] whenever you keep. Since everything strictly between write and read is known to be zero (that's the invariant: the writer only lags on zeroes), swapping pushes a zero into the slot the non-zero vacated — the zeroes bubble backward for free, and no second pass is needed. The tiny extra polish: skip the swap when write === read, since swapping an element with itself is a wasted write; on an array with no zeroes at all (Example 3) that turns n pointless writes into zero writes. That guard is the "minimize operations" answer, and mentioning it unprompted is the whole reason this Easy is worth asking.

Solution (JS) — O(n) time · O(1) space 
// Single-pass swap version. Invariant: everything in nums[write .. read-1] is 0,
// so swapping a non-zero at `read` back to `write` sends a zero forward for free.
var moveZeroes = function(nums) {
  let write = 0;   // next slot for a non-zero; also == count of non-zeroes so far

  for (let read = 0; read < nums.length; read++) {
    if (nums[read] !== 0) {
      // Skip the self-swap. When there are no zeroes yet, write === read and
      // swapping would be n wasted writes — this guard IS the "minimize
      // operations" follow-up the problem statement asks for.
      if (write !== read) {
        [nums[write], nums[read]] = [nums[read], nums[write]];  // destructuring swap
      }
      write++;   // writer advances ONLY on a keep
    }
    // zero ⇒ leave the writer parked on it; the next non-zero will swap it forward
  }
  // No return value: LeetCode grades the mutated array in place.
};

Trace on [0,1,0,3,12]:

  • read=0: value 0 → skip. write stays 0 (parked on the zero).
  • read=1: value 1 ≠ 0, and write(0) !== read(1) → swap indices 0 and 1 → [1,0,0,3,12]. write=1. The zero got pushed to index 1 for free.
  • read=2: value 0 → skip.
  • read=3: value 3, write(1) !== read(3) → swap indices 1 and 3 → [1,3,0,0,12]. write=2.
  • read=4: value 12, swap indices 2 and 4 → [1,3,12,0,0]. write=3.
  • Final: [1,3,12,0,0]. ✓ Order of non-zeroes preserved, zeroes at the back, zero allocations.

On [1,2,3]: every iteration has write === read, so the guard fires every time and not a single write occurs — the array is already correct and we do zero work. That's the operation-minimizing behaviour.

Complexity: O(n) time — one pass, O(1) per element. O(1) space — one cursor, and the destructuring swap allocates a transient two-element array that engines optimize away (mention it, then say you'd use a temp variable if the interviewer is being strict about allocation). Writes: exactly one swap (two writes) per non-zero that is out of position, versus 2n writes for the naive copy-then-fill.

Follow-ups you should expect:

  • "Why not filter then pad?" — nums.filter(x => x !== 0) is the same algorithm with an output buffer, so it's O(n) space and the problem forbids it. This is the exact filter-in-place framing from §2b; saying "this is filter without the output array" lands well.
  • "Does order of the zeroes matter?" — zeroes are indistinguishable, so no. But order of the non-zeroes does, and that's what rules out the tempting "swap every zero with the last element" trick, which is O(n) but scrambles the non-zeroes. State that you considered and rejected it.
  • "Move zeroes to the front instead." — mirror the whole thing: iterate from the right with a write cursor at n-1. Same template, reversed. If you built it as a template you'll do this in 30 seconds; if you memorized it you'll flounder.
  • "Now partition into evens-then-odds" (LC 905) — identical skeleton, keep = x % 2 === 0, and if relative order doesn't matter you can even use converging pointers from both ends. Recognizing that the order requirement is what forces reader/writer over converging is the insight.
Notice the difference between this and Remove Duplicates: there, the tail was garbage and the return value was a length; here, the tail must be zero-filled and there's no return value at all. Same engine, different contract. Read the return signature — it tells you what the grader checks. A void return means "I'm grading the whole mutated array"; an int return means "I'm grading only the prefix."

Sort Colors / Dutch National Flag (LC #75) MEDIUM

You're given an array nums containing only the integers 0, 1, and 2, representing the colors red, white, and blue. Sort the array in place so that all the 0s come first, then all the 1s, then all the 2s. You must do it without using any library sort function, and the follow-up asks for a one-pass algorithm using only constant extra space — which rules out the obvious counting solution too.

Example 1: Input: nums = [2,0,2,1,1,0] → Output: [0,0,1,1,2,2].
Example 2: Input: nums = [2,0,1] → Output: [0,1,2] — tiny, but it exercises the case where a swap brings a value that itself needs relocating.
Example 3: Input: nums = [2,2,2] → Output: [2,2,2] — all elements belong at the far end. This is the case that catches the classic bug: if you advance the reader after swapping with the high side, you'll leave 2s stranded.
Constraints: n == nums.length; 1 ≤ n ≤ 300; nums[i] is 0, 1, or 2.

Solve on LeetCode ↗

Thinking process. Say the two easy solutions out loud, then discard them, because the interviewer is asking this for the third one. (1) Library sort: forbidden, and O(n log n) anyway. (2) Counting sort: count the 0s, 1s and 2s in one pass, then overwrite the array in a second pass. This is O(n) time, O(1) space, and it is completely correct — say so, because it proves you can solve it — but it's two passes, and the follow-up explicitly wants one. (3) The one-pass answer is Dijkstra's Dutch National Flag partition, and it's a three-pointer generalization of the reader/writer family.

Three cursors carve the array into four regions, and the invariant is the entire algorithm — memorize the invariant, not the code:

  • nums[0 .. low-1] — settled 0s.
  • nums[low .. mid-1] — settled 1s.
  • nums[mid .. high] — unknown / unprocessed. This is the shrinking window the reader sweeps.
  • nums[high+1 .. n-1] — settled 2s.

mid is the reader. Loop while mid <= high (note: <=, not < — high is inclusive in the unknown region, so its element must still be examined). Three cases:

  • nums[mid] === 0 → swap it down to low, then low++ and mid++. Why is it safe to advance mid here? Because whatever came back from low is already known: low is inside the settled-1s region (or equals mid), so the incoming value is a 1 (or the 0 itself) — either way it belongs at mid−1 and needs no further inspection.
  • nums[mid] === 1 → it's already in the right region. Just mid++. No swap.
  • nums[mid] === 2 → swap it up to high, then high-- — and do not advance mid. This is the one line candidates get wrong and it's the whole point of the problem. The value that just came back from high came from the unknown region; you have never looked at it. It could be a 0, which would then be stranded in the middle forever. So you must re-examine the same mid on the next iteration. Advancing mid here is the Dutch-flag bug, and [2,2,2] or [2,0,1] will expose it instantly.

Termination is guaranteed even though mid sometimes doesn't move: every iteration either advances mid or decrements high, so the unknown region [mid, high] shrinks by at least one every single time. That shrinking-gap argument is the O(n) proof, and it's the same shape as the converging-pointer argument in §2a.

Solution (JS) — O(n) time · O(1) space · one pass 
// Dutch National Flag (Dijkstra). Invariant at every step:
//   [0 .. low-1]    = 0s (settled)
//   [low .. mid-1]  = 1s (settled)
//   [mid .. high]   = UNKNOWN  ← shrinks by ≥1 every iteration
//   [high+1 .. n-1] = 2s (settled)
var sortColors = function(nums) {
  let low  = 0;
  let mid  = 0;
  let high = nums.length - 1;

  const swap = (i, j) => { [nums[i], nums[j]] = [nums[j], nums[i]]; };

  // <= because `high` is INSIDE the unknown region and must be examined.
  while (mid <= high) {
    if (nums[mid] === 0) {
      swap(low, mid);
      low++;
      mid++;   // SAFE to advance: the value swapped in came from [low..mid-1],
               // which is the settled-1s region, so it's already inspected.
    } else if (nums[mid] === 2) {
      swap(mid, high);
      high--;
      // DO NOT advance mid! The value swapped in came from the UNKNOWN region
      // and has never been examined — it could be a 0 that must move to the front.
      // This single omission is the classic Dutch-flag bug. Test with [2,0,1].
    } else {
      mid++;   // nums[mid] === 1: already in its final region, nothing to do
    }
  }
  // No return: sorted in place.
};

Trace on [2,0,2,1,1,0]: low=0, mid=0, high=5.

  • nums[0]=2 → swap(0,5) → [0,0,2,1,1,2], high=4. mid stays 0.
  • nums[0]=0 → swap(0,0) (self, harmless) → low=1, mid=1.
  • nums[1]=0 → swap(1,1) → low=2, mid=2.
  • nums[2]=2 → swap(2,4) → [0,0,1,1,2,2], high=3. mid stays 2.
  • nums[2]=1 → mid=3.
  • nums[3]=1 → mid=4. Now mid(4) > high(3) → loop ends.
  • Result: [0,0,1,1,2,2]. ✓ Note the very first step is exactly the case where advancing mid would have been fatal: the incoming 0 would have been skipped forever.

Trace on [2,0,1] (the bug-exposer): nums[0]=2 → swap(0,2) → [1,0,2], high=1, mid stays 0. nums[0]=1 → mid=1. nums[1]=0 → swap(0,1) → [0,1,2], low=1, mid=2. Now mid(2) > high(1) → done: [0,1,2]. ✓ Had we advanced mid after the 2-swap, we'd have skipped the 1 at index 0 and returned [1,0,2].

Complexity: O(n) time — the unknown region [mid, high] has size high - mid + 1, starts at n, and strictly shrinks by at least one every iteration (either mid goes up or high goes down), so there are at most n iterations, each O(1). One pass. O(1) space — three integer cursors. Note that although mid sometimes stalls, the algorithm still touches each element a bounded number of times, so the O(n) bound is exact, not amortized hand-waving.

Follow-ups you should expect:

  • "Why can't you advance mid after swapping with high?" — the question. Answer with the provenance argument: the value from low is known (it's from a settled region), the value from high is unknown (it's from the unprocessed region). Asymmetric provenance ⇒ asymmetric pointer advance. That sentence is the senior answer.
  • "What if there were k colors, not 3?" — Dutch flag doesn't generalize cleanly past 3 (you'd need k−1 boundaries and the swaps get tangled). For k colors, counting sort is the right answer: O(n + k) time, O(k) space, two passes. Knowing where the pattern stops is as valuable as knowing the pattern.
  • "Where does this show up outside interviews?" — it's the partition step in 3-way quicksort, which is how you make quicksort robust to arrays with many duplicate keys. That connection is a genuine systems answer and it lands well at fintech shops that care about sort performance on low-cardinality data.
  • "Is your solution stable?" — no. Swapping with high hurls elements across the array, destroying relative order. It doesn't matter here (0s, 1s and 2s are indistinguishable within a color), but noticing it unprompted signals you understand what you gave up for one-pass O(1).
Lead with counting sort, then upgrade yourself. "The two-pass counting solution is O(n) time and O(1) space and I could write it in 30 seconds — but the follow-up asks for one pass, so let me do the Dutch National Flag partition instead." That sequence shows you can hit the bar and then exceed it, which is precisely the narration enterprise-tier rubrics reward. Jumping straight to the clever answer without acknowledging the simple one sometimes reads as a memorized solution rather than a derived one.

5. Stretch — Trapping Rain Water

This is a Hard, and it is rarely asked at the enterprise tier. Banks, fintech and healthcare loops overwhelmingly draw from the easy/medium pool you just worked through; Trapping Rain Water shows up mostly at FAANG and at the occasional "let's see how far he goes" final round. Do it last, and only once the seven problems above are automatic. Its value here is that it is the cleanest possible demonstration that converging pointers are driven by an invariant you can prove, not by a rule you memorized — and if the Container With Most Water proof is fresh, this one will click.

Trapping Rain Water (LC #42) HARD ★

Given n non-negative integers representing an elevation map where the width of each bar is 1, compute how much water it can trap after raining. Water settles in the valleys between bars. Concretely: the water sitting above position i is bounded by the tallest bar anywhere to its left and the tallest bar anywhere to its right — water at i rises to min(maxLeft, maxRight) and then spills. Subtract the bar itself, and the water at i is min(maxLeft, maxRight) − height[i] (or 0 if that's negative). Sum over every position.

Example 1: Input: height = [0,1,0,2,1,0,1,3,2,1,2,1] → Output: 6 — water pools in several separate basins; e.g. index 2 (height 0) is walled by a 1 on the left and a 3 on the right, so it holds min(1,3) − 0 = 1 unit.
Example 2: Input: height = [4,2,0,3,2,5] → Output: 9 — a single wide basin between the 4 and the 5. Index 2 (height 0) holds min(4,5) − 0 = 4; the interior bars of height 2 and 3 each hold a little too. This example is the one that breaks solutions that only look at adjacent walls.
Example 3: Input: height = [3,2,1] → Output: 0 — strictly descending, no right wall anywhere, nothing is trapped. Every position has maxRight ≤ height[i].
Constraints: n == height.length; 1 ≤ n ≤ 2×10⁴; 0 ≤ height[i] ≤ 10⁵.

Solve on LeetCode ↗

Thinking process. The single most important move is to stop thinking about basins and start thinking column by column. Don't try to identify valleys, find their walls, and compute their areas — that's a bookkeeping nightmare with nested basins, and it's where most candidates drown. Instead ask, for one position i in isolation: how much water sits on top of it? Answer: min(maxLeft[i], maxRight[i]) − height[i], clamped at zero. That's it. The total is the sum over all i. Once you say "I'll compute the water above each column independently and sum," the problem transforms from a Hard into a bookkeeping exercise — and saying that sentence is the interview.

That reframing gives you a ladder of three solutions, and you should walk the interviewer up it:

  • Brute force, O(n²) / O(1): for each i, scan left for the max and right for the max. Correct, obviously too slow at n = 2×10⁴. Ten seconds to state.
  • Prefix/suffix arrays, O(n) / O(n): the maxes are recomputed n times, so precompute them. One left-to-right pass builds maxLeft[], one right-to-left pass builds maxRight[], one final pass sums. This is the solution to reach for first in a real interview: it's easy to get right under pressure, it's O(n), and it's a completely acceptable answer. Write this if you're short on time.
  • Two pointers, O(n) / O(1): the optimization the interviewer wants next — kill the two arrays.

The two-pointer insight, and why it's correct. You don't actually need maxLeft[i] and maxRight[i] — you only need min(maxLeft, maxRight). And often you can know that minimum without knowing both terms. Run left from 0 and right from n−1, tracking leftMax (the tallest bar seen from the left so far) and rightMax (tallest seen from the right so far). Now:

Suppose leftMax < rightMax. Then for position left, the true maxRight[left] is at least rightMax — because rightMax is a bar that genuinely exists somewhere to the right of left, and the true maximum to the right can only be larger. So min(maxLeft[left], maxRight[left]) = min(leftMax, something ≥ rightMax), and since leftMax < rightMax ≤ something, that minimum is exactly leftMax. The left side is the binding constraint, and you know it for certain even though you have not seen the whole right side. So you can finalize the water at left right now — leftMax − height[left] — and advance. Symmetrically when rightMax ≤ leftMax, the right side binds and you finalize right.

That is the same shape as the Container proof: a comparison between the two ends licenses you to permanently resolve and discard one of them. Each step eliminates a possibility permanently. Two pointers, one pass, no arrays.

Solution (JS) — O(n) time · O(1) space (plus the O(n)-space version to lead with)

Step 1 — the O(n) space version. Write this first in a real interview. It's near-impossible to get wrong and it earns the O(n) time credit immediately:

var trapWithArrays = function(height) {
  const n = height.length;
  if (n === 0) return 0;

  const maxLeft  = new Array(n);   // maxLeft[i]  = tallest bar in height[0..i]
  const maxRight = new Array(n);   // maxRight[i] = tallest bar in height[i..n-1]

  maxLeft[0] = height[0];
  for (let i = 1; i < n; i++) maxLeft[i] = Math.max(maxLeft[i - 1], height[i]);

  maxRight[n - 1] = height[n - 1];
  for (let i = n - 2; i >= 0; i--) maxRight[i] = Math.max(maxRight[i + 1], height[i]);

  let total = 0;
  for (let i = 0; i < n; i++) {
    // Water above column i is capped by the shorter of its two tallest walls.
    // Both maxes INCLUDE height[i], so this is never negative — no clamp needed.
    total += Math.min(maxLeft[i], maxRight[i]) - height[i];
  }
  return total;
};

Step 2 — the O(1) space two-pointer version. This is what they're fishing for:

var trap = function(height) {
  let left = 0;
  let right = height.length - 1;
  let leftMax = 0;    // tallest bar seen in height[0 .. left]
  let rightMax = 0;   // tallest bar seen in height[right .. n-1]
  let total = 0;

  while (left < right) {
    if (height[left] < height[right]) {
      // The RIGHT side is guaranteed to have a bar at least as tall as
      // height[right], and height[right] > height[left]. So whatever the true
      // maxRight[left] is, it is >= height[right] > height[left]. That means
      // min(maxLeft, maxRight) for column `left` is decided entirely by the LEFT.
      // We can finalize this column now, without ever seeing the full right side.
      leftMax = Math.max(leftMax, height[left]);
      total += leftMax - height[left];   // never negative: leftMax >= height[left]
      left++;
    } else {
      // Mirror image: the left side has a bar >= height[left] >= height[right],
      // so column `right` is bounded by the right side. Finalize and advance.
      rightMax = Math.max(rightMax, height[right]);
      total += rightMax - height[right];
      right--;
    }
  }

  return total;
};

Trace on [4,2,0,3,2,5] (expected 9): left=0, right=5, leftMax=0, rightMax=0, total=0.

  • height[0]=4 < height[5]=5 → left branch. leftMax = max(0,4) = 4. Water: 4 − 4 = 0. total = 0. left=1.
  • height[1]=2 < 5 → left. leftMax stays 4. Water: 4 − 2 = 2. total = 2. left=2.
  • height[2]=0 < 5 → left. Water: 4 − 0 = 4. total = 6. left=3.
  • height[3]=3 < 5 → left. leftMax stays 4. Water: 4 − 3 = 1. total = 7. left=4.
  • height[4]=2 < 5 → left. Water: 4 − 2 = 2. total = 9. left=5.
  • left(5) < right(5) is false → exit. Return 9. ✓

Observe: the algorithm never once needed to know that the tallest bar on the right was 5 specifically — it only needed to know it was at least as tall as the left wall. That's the whole trick, and it's why one pass with two scalars suffices.

Complexity: O(n) time — exactly one pointer moves per iteration and they converge, so at most n iterations, O(1) each. O(1) space — four integers, no arrays. The array version is also O(n) time but O(n) space; the two-pointer version's entire contribution is eliminating those two arrays by noticing you only ever need the minimum of the two maxes, and the comparison at the ends tells you which one that is.

Follow-ups you should expect:

  • "Solve it with a monotonic stack." — a genuinely different O(n) solution that computes water basin-by-basin (horizontally) rather than column-by-column (vertically): keep a decreasing stack of indices; when a taller bar arrives it forms the right wall of a basin, so pop and settle. O(n) time, O(n) space. It's the bridge to 04 — Stacks and to Largest Rectangle in Histogram. Being able to say "there's a stack solution too, and it's the same machinery as Largest Rectangle" is a strong close even if you don't write it.
  • "How does this relate to Container With Most Water?" — the pairing they love. Both use converging pointers driven by "move the shorter side," but Container picks two zero-width lines and takes a max, while Trapping has unit-width bars that block water and sums over every column. Different quantity, same elimination logic. If you can articulate the difference you've demonstrated you understand the pattern rather than the problems.
  • "Trapping Rain Water II" (LC 407, 2-D grid) — genuinely hard, and two pointers is useless there: water escapes in any direction, so you need a min-heap flood-fill from the border inward. Correct answer at this tier: name the approach in one sentence and don't attempt it unless asked.
  • "Why doesn't total += leftMax - height[left] ever go negative?" — because you update leftMax = max(leftMax, height[left]) before the subtraction, so leftMax ≥ height[left] by construction. The clamp everyone reaches for is unnecessary. A small point, but interviewers ask it to see whether you actually reasoned about your own code.
The two-pointer version's condition is height[left] < height[right] — comparing the current bars, not leftMax < rightMax. Both formulations can be made to work, but they need different update orders and mixing them up produces a solution that's off by a few units on inputs like [4,2,0,3,2,5] and passes on the sample. Pick one form, understand exactly why it's correct, and write only that one. If you find yourself "adjusting signs until the tests pass," you have not understood the invariant — go back to the proof.

Interview Q&A

What single property of the input makes converging pointers valid, and what breaks without it?

Sortedness (or, in the palindrome case, the fact that the comparison is symmetric; or, in Container/Trapping, a provable greedy exchange argument). The pattern works because a comparison at the two ends lets you deduce that one endpoint cannot participate in any remaining solution — and that deduction requires knowing something about all the elements you haven't examined. Sortedness is what gives you that knowledge: "everything between left and right lies between nums[left] and nums[right]."

Remove the ordering and the deduction evaporates. In an unsorted array, sum < target tells you nothing about whether nums[left] could pair with some element in the middle, so left++ silently discards valid answers. The code still runs, still returns something, and is wrong. That failure mode — plausible wrong output — is why interviewers plant unsorted arrays as a trap.

Explain the O(n²) → O(n) collapse in one sentence.

"Each pointer move eliminates a possibility permanently — so instead of testing all n²/2 pairs, I test at most n, because every step provably rules out every pair involving the endpoint I just discarded."

That's the sentence. It has two halves and both matter: the counting half (the pointers only move toward each other, so at most n iterations) and the justification half (why skipping the untested pairs is sound). Candidates who give only the first half are describing the code; candidates who give both are explaining the algorithm. Interviewers can tell the difference instantly.

Two Sum (LC 1) vs Two Sum II (LC 167) — when do you use a hash map and when converging pointers?
  • Hash map when the array is unsorted and you need the original indices. O(n) time, O(n) space. Sorting is not an option because it destroys the indices you're asked to return.
  • Converging pointers when the array is already sorted, or when you're allowed to sort (because you only need the values, not the original positions), and especially when space is constrained. O(n) time on a pre-sorted array, O(1) space.

The decision procedure: "Do I need original indices? Then hash map. Am I given sortedness for free, or is O(1) space demanded? Then two pointers." And note the generalization asymmetry — the hash-map trick does not extend gracefully to 3Sum (deduplication becomes horrible), while the two-pointer form extends to 3Sum, 4Sum and kSum by just adding anchor loops. That's the real reason 167 is on the syllabus.

In 3Sum, exactly where can duplicate triplets enter, and how does each skip prevent them?

Two places, independently, and you must be able to name both:

  1. Duplicate anchors. If nums[i] === nums[i-1], the outer index i is a repeat of an anchor already processed, and its inner scan covers a subset of the range that i-1's scan covered. Every triplet it can find was already found. Fix: if (i > 0 && nums[i] === nums[i-1]) continue; — compare backward, never forward. Comparing forward would drop the first occurrence, breaking triplets like [-1,-1,2] where the anchor needs its own duplicate as a partner.
  2. Duplicate partners. After pushing a valid triplet, if you advance only left and nums[left] is unchanged, you'll re-find the identical triplet. Fix: advance both pointers past the hit, then run while (left < right && nums[left] === nums[left-1]) left++; and the mirror on right. Both guards are load-bearing: [0,0,0,0] would otherwise walk the pointers through each other.

And a precision point that separates the fluent from the memorizers: the partner skip belongs only in the hit branch, not on the sum < 0 / sum > 0 branches. Landing on a duplicate value when you haven't emitted anything is harmless — it produces the same sum and gets rejected identically. Adding a skip there isn't a bug, but it reads as not understanding why the skip exists.

Prove that moving the shorter wall in Container With Most Water can't skip the optimal answer.

Say height[left] < height[right], and you've just recorded the area height[left] × (right − left). Claim: no container still using left can beat it.

Take any surviving partner j with left < j < right. Its width is j − left, which is strictly less than right − left. Its height is min(height[left], height[j]), which is at most height[left] — the exact height of the container just recorded. Strictly smaller width times a no-greater height gives a strictly smaller area. So left is in no better container, and retiring it costs nothing. Symmetric if right is shorter. On a tie, both walls are individually retirable by the same argument, so moving either (or both) is safe.

The intuition to state alongside the proof: "the short wall's potential is already fully realized right now, at maximum width; the tall wall still has upside as a cap for future pairs. Retire what has no upside."

What does "in place, O(1) extra space" actually forbid — and what does it permit?

Forbidden: any allocation that grows with n. A second array, a Set, a Map, a joined string, the result of filter/map/ slice/spread. Also forbidden in spirit: splice()/shift() in a loop — they don't allocate a visible array but they re-index the underlying buffer, making the algorithm O(n²).

Permitted: a constant number of scalars (cursors, running maxes, a temp for swapping), and overwriting the input array itself. That's the point — the input buffer is your workspace.

The mental translation for a JS developer: "in place, O(1) space" means "write Array.prototype.filter without the output buffer." The reader/writer template is exactly that: a read cursor that sweeps every element, and a write cursor that only advances when the predicate says keep. Every library filter is implemented this way internally — it just happens to write into a fresh buffer instead of the source one.

Why is it safe to overwrite the input array while you're still reading it?

Because of the invariant write ≤ read, always. Both cursors start at 0; the reader advances on every iteration, the writer advances only on a keep. So the writer can only fall behind — it can never overtake. Therefore every slot you write to has already been read, and the value you clobber is either garbage (a discarded element) or the very element you're writing back (a self-assignment).

Say this invariant out loud when you write the loop. It converts "isn't that dangerous?" — which is the interviewer's actual question — into a one-sentence proof of safety, and it's precisely the sort of reasoning a 7-YOE candidate is expected to volunteer rather than be prompted for.

In Dutch National Flag, why do you advance mid after swapping with low, but not after swapping with high?

Provenance. The two swaps pull values from regions you know different amounts about:

  • Swapping with low pulls in a value from [low .. mid-1] — the settled 1s region. You have already examined it. It's a 1 (or, when low === mid, it's the 0 you just placed). Either way it needs no further inspection, so mid++ is safe.
  • Swapping with high pulls in a value from [mid .. high] — the unknown region. You have never looked at it. It could be a 0 that must travel all the way to the front. Advancing mid would strand it. So you re-examine the same mid next iteration.

Asymmetric knowledge ⇒ asymmetric pointer advance. Termination is still guaranteed because every iteration either advances mid or decrements high, shrinking the unknown region by at least one — so at most n iterations. Test the bug with [2,0,1]: advancing mid after the high-swap returns [1,0,2].

You've sorted the array to enable two pointers. What did that cost, and when is it a mistake?

Costs: (1) time floor of O(n log n) — so if a hash-map solution is O(n), you've made the problem asymptotically slower; (2) the original indices are destroyed; (3) the input is mutated, which matters if the caller owns it.

It's a mistake when the problem asks for indices (plain Two Sum: sort and you can no longer return the answer without an O(n) index map, at which point you should have used the hash map) or when relative order is part of the contract (Move Zeroes: sorting would obviously destroy the required ordering of the non-zeroes).

It's the right call when you need values not positions, and duplicates must be handled — 3Sum is the canonical case: sorting costs O(n log n), which is dominated by the O(n²) scan anyway, so it's free, and it buys you adjacency-based deduplication, which is the difference between a clean solution and a Set-of-stringified-triplets hack. "The sort is free because the scan is O(n²) regardless" is a great line to deliver.

How do you decide between converging pointers, reader/writer, and a sliding window?
  • Converging — you're evaluating a pair of endpoints and the input is sorted or symmetric, or you have a greedy exchange argument. Pointers start apart and move toward each other. Output is typically a pair, a boolean, or a max/min over pairs.
  • Reader/writer — you're rewriting the array itself under a keep-predicate, in place. Pointers start together and move in the same direction at different rates. Output is typically a length, or a mutated array.
  • Sliding window (see 03) — you care about a contiguous subarray and an aggregate over it (sum, distinct count, frequency). Both pointers move rightward, and [left, right] is a window whose contents you maintain incrementally. Output is typically a longest/shortest/count-of windows.

The tell: does the answer live in a contiguous range? → sliding window. Does the answer involve two elements from opposite ends of an ordering? → converging. Am I being asked to mutate in place? → reader/writer. Reader/writer and sliding window look identical on the page (two indices, both moving right), which is why people confuse them — the difference is that a window maintains state about its contents, while a writer just marks a destination slot.

Can two pointers ever be worse than O(n)? When does the pattern fail entirely?

The scan itself is always O(n) — the pointers converge and the gap shrinks monotonically, full stop. But the surrounding work isn't free: sorting first makes the whole thing O(n log n); nesting it inside an anchor loop (3Sum) makes it O(n²); two anchor loops (4Sum) makes it O(n³). The pointer scan is a cheap inner engine, not a magic O(n) wrapper around any problem.

The pattern fails entirely when you cannot justify eliminating an endpoint. Concretely: an unsorted array with no exchange argument (you'd be guessing); a problem where the answer might use two elements that are both in the middle after you've discarded the ends (which is exactly what an invalid elimination causes); or 2-D structures where "the two ends" isn't even well defined (Trapping Rain Water II — water escapes in any direction, so you need a heap-based flood fill). Knowing where the pattern stops is worth as much as knowing where it applies, and volunteering a limit unprompted reads as genuine understanding rather than pattern-matching.
