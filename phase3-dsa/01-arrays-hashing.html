01 — Arrays & Hashing

Spend memory to buy speed: a hash map turns an O(n) inner scan into an O(1) lookup. That one move is most of the easy–medium canon.

Why interviewers ask this

This is the entry point to every DSA loop on earth, and not because arrays are interesting. It's because hashing is the single highest-leverage idea in interview algorithms: a huge fraction of easy and medium problems are an O(n²) brute force whose inner loop is a lookup — "have I seen this before?", "does the complement exist?", "how many times did this appear?" — and every one of those collapses to O(n) the moment you pay O(n) memory for a hash map. Interviewers at the enterprise tier (Optum, JPMC, Goldman Sachs and their peers) open with Two Sum or Contains Duplicate on purpose: they take five minutes, and the whole signal is in whether you reach for the map immediately and say why, or whether you write the nested loop and have to be nudged.

The second half of this tutorial — prefix sums — is the same trade applied to ranges instead of values. "Sum of every subarray," "product of everything except me," "how many subarrays sum to k" all look quadratic and all collapse to a single pass once you precompute cumulative aggregates and let a hash map answer the "have I seen this running total before?" question. Subarray Sum Equals K is the payoff problem where both ideas fire at once, and it is a genuine medium that separates candidates. As a React developer you have already internalized this exact trade: useMemo and a normalized byId lookup table are literally "spend space so the hot path stops recomputing." You are not learning a new instinct here — you are learning to name one you already have, and to reach for it under a 25-minute clock.

1. Recognizing the pattern

Reach for a hash map / hash set the instant a problem statement contains any of these:

  • "Duplicate", "unique", "distinct", "seen before", "appears twice" — membership questions. A Set answers them in O(1).
  • "Count", "frequency", "how many times", "most common", "top k" — a frequency map, always. Map<value, count> is the entire first half of the solution.
  • "Anagram", "permutation of", "same characters" — two things are anagrams iff their frequency maps are identical. That's the definition, not a trick.
  • "Group / bucket things that share a property" — Map<canonicalKey, list>. The whole problem becomes "what's the canonical key?"
  • "Find a pair / complement that sums to target" — as you scan, ask the map for target - x. You're converting "search the rest of the array" into "look it up."
  • An O(n²) brute force whose inner loop is a search. This is the master trigger. If the inner loop asks "does X exist somewhere else in this array?", a hash map deletes it.

Reach for a prefix aggregate (running sum/product, possibly from both directions) when you see:

  • "Subarray" or "contiguous range" plus a sum/product condition — the range [i..j] aggregate is derivable from two prefixes, so you never re-add the middle.
  • "…except itself" / "excluding index i" — everything-to-my-left combined with everything-to-my-right. Two sweeps, no division.
  • Repeated range queries on a static array — precompute once in O(n), answer each query in O(1).
  • "How many subarrays satisfy X" — and if the array can contain negatives or zeros, sliding window is out and prefix-sum + hash map is in. (That distinction is the entire point of Subarray Sum Equals K.)
Say the recognition sentence out loud before you write anything: "My brute force is O(n²) because the inner loop is searching the array for a value. I can pre-index those values in a hash map and get the search down to O(1), so the whole thing becomes one pass — O(n) time, O(n) space." That sentence, unprompted, is the difference between "solved it" and "solved it like someone with seven years of experience." Enterprise rubrics literally have a line for stated the time/space tradeoff.

2. The templates

2a. The core idea the whole course rests on: trading space for time

Almost every array problem starts the same way. You need to know something about element x that depends on the other elements — does its complement exist, has it appeared before, how many times, what came before it. The naive way to answer that is to go look: for each x, walk the array. That's an O(n) question asked n times ⇒ O(n²).

The fix is always the same shape. Instead of searching for the answer when you need it, build a structure that already knows it. A hash map is a device that converts "search a collection" (O(n)) into "read a key" (O(1) average) at the cost of storing the collection a second time (O(n) space). Drop that into the middle of the nested loop and the inner loop disappears: 

// The brute force. The inner loop is a SEARCH — that's the tell.
for (let i = 0; i < n; i++) {
  for (let j = 0; j < n; j++) {      // ← O(n) work to answer one question about nums[i]
    if (nums[j] === somethingAbout(nums[i])) { /* … */ }
  }
}

// The same algorithm with the search pre-indexed. The inner loop is now a lookup.
const seen = new Map();          // O(n) space — this is the price
for (let i = 0; i < n; i++) {
  if (seen.has(somethingAbout(nums[i]))) { /* … */ }  // ← O(1) average. The n² is gone.
  seen.set(nums[i], i);            // build the index as you go — one pass, not two
}

That is the whole move. n × O(n) becomes n × O(1). O(n²) → O(n) time, O(1) → O(n) space. You are buying time with memory, and in an interview memory is nearly always the cheaper currency — the constraints (n ≤ 10⁵) are there to tell you that O(n²) will time out and O(n) memory is fine.

You already do this every day. A normalized Redux store (byId) exists so components stop doing items.find(i => i.id === id) on every render. useMemo exists so an expensive derivation isn't recomputed on the hot path. Both are "precompute once, read in O(1), pay memory." Hashing in an interview is the same instinct with a stopwatch on it. Feel that click now, because tutorials 02–14 are variations on it: two pointers trades a hash map for sortedness, sliding window trades it for a monotone condition, DP trades it for overlapping subproblems. All of them are "don't recompute what you can remember."

2b. Template — the frequency map

Three lines you will write in a dozen interviews. Burn them in:

// Build: value → how many times it occurs. Works for numbers, chars, words, anything.
const freq = new Map();
for (const x of arr) {
  freq.set(x, (freq.get(x) ?? 0) + 1);   // ?? 0 handles "first time seen" without a branch
}

// Read:
freq.get(x) ?? 0        // count of x (0 if absent — never undefined-propagate into arithmetic)
freq.has(x)             // membership — O(1) average
freq.size               // number of DISTINCT values
[...freq.entries()]     // [[value, count], …] — in INSERTION order, which Map guarantees

Line by line, because each line is a decision you'll have to defend:

  • new Map(), not {}. A plain object coerces every key to a string, so 1 and "1" become the same bucket. A Map keeps keys at their real type, preserves insertion order for all key types, and has no prototype to collide with. See the callout below — this is the JS-specific thing interviewers probe.
  • (freq.get(x) ?? 0) + 1. get returns undefined for a missing key, and undefined + 1 is NaN — a silent wrong answer, not a crash. The ?? 0 is the entire "initialize on first sight" branch, collapsed into an operator. Use ?? and not ||: || also swallows a legitimate count of 0.
  • One pass builds it. Never build the map in one loop and then re-scan the array to "fix it up" — if you find yourself doing that, the map is holding the wrong key.
  • freq.size is free. Number of distinct values, O(1). Half of "are these anagrams / is there a duplicate" problems are one size comparison away from done.

Why this is O(n) and not O(n²): the loop runs n times, and each iteration does a constant number of hash operations — get and set each hash the key (O(1) for a number or a short string) and index into a bucket array. Hash collisions can in theory degrade a lookup to O(n) if every key lands in the same bucket, which is why the honest claim is O(1) average, not O(1) worst case. Say "average" out loud once; it costs you nothing and it's the kind of precision that gets noticed. Total: O(n) time, O(k) space where k is the number of distinct values (≤ n, and often a constant — 26 for lowercase letters, which is worth saying when the alphabet is bounded).

Map vs plain object vs Set — the JS specifics you must have straight.
  • Object keys are coerced to strings. const o = {}; o[1] = 'a'; o['1'] = 'b'; leaves one key — o[1] is now 'b'. In a frequency map over mixed types, that's a wrong answer you'll never spot at a whiteboard. Map uses SameValueZero: 1 and '1' stay distinct, and objects/arrays can be keys by identity.
  • Objects have a prototype. 'constructor' in {} is true, and {}['toString'] is a function, not undefined. If your keys are user-supplied strings (words, chars) this can genuinely bite. Fixes: Object.create(null), Object.hasOwn(obj, k), or just use a Map.
  • Map preserves insertion order for every key type. Object key order is only guaranteed for string keys — and integer-like keys ("1", "2") get sorted numerically first, ahead of the string keys. If order matters, that's a Map.
  • Set is the membership half of a Map: add, has, delete, size, all O(1) average. When you only care whether you've seen something and not how many times, use a Set and say so — it signals you picked the minimal structure.
  • All of Map/Set's core ops are O(1) average — get, set, has, delete, add. Iterating is O(n). Map is also measurably faster than an object for frequent adds/deletes, and unlike an object it won't be silently deoptimized into dictionary mode.
One legitimate exception: when keys are guaranteed to be lowercase a–z, a fixed new Array(26).fill(0) indexed by ch.charCodeAt(0) - 97 is faster and O(1) space. Reach for it only when you say why: "the alphabet is bounded, so I'll use a 26-slot array instead of a map."

2c. Template — prefix sums

A prefix sum array answers "what's the sum of nums[i..j]?" in O(1) instead of O(j−i). Define prefix[k] = nums[0] + … + nums[k-1] — the sum of the first k elements, with prefix[0] = 0. That leading zero is not decoration; it's what makes the formula total, with no special case for i === 0. 

// Build: prefix[k] = sum of the first k elements. prefix[0] = 0 (the empty prefix).
const prefix = [0];
for (let i = 0; i < nums.length; i++) {
  prefix.push(prefix[i] + nums[i]);   // prefix has length n+1
}

// Query: sum of nums[i..j] INCLUSIVE — one subtraction, O(1).
const rangeSum = (i, j) => prefix[j + 1] - prefix[i];

// The hash-map variant (the one that actually shows up): "how many subarrays sum to k?"
// A subarray (i..j) sums to k  ⟺  prefix[j+1] - prefix[i] === k  ⟺  prefix[i] === running - k.
// So you never build the array at all — you stream the running sum and ASK THE MAP.
const counts = new Map([[0, 1]]);   // the empty prefix (sum 0) has been seen once — CRITICAL
let running = 0, total = 0;
for (const x of nums) {
  running += x;
  total += counts.get(running - k) ?? 0;   // every earlier prefix equal to running-k ends a valid subarray HERE
  counts.set(running, (counts.get(running) ?? 0) + 1);
}

Line by line:

  • The algebra is the algorithm. sum(i..j) = prefix[j+1] − prefix[i]. Rearranged for a target: prefix[i] = prefix[j+1] − k. Both forms say the same thing; the second one is a lookup, which is exactly the trigger from §1. Derive it on the whiteboard — don't recall it.
  • counts maps a prefix-sum value → how many times it has occurred. Not whether — how many times, because several different starting points can produce the same prefix and each one is a distinct valid subarray. Using a Set here is the classic wrong answer.
  • The [[0, 1]] seed is the whole edge case. It represents the empty prefix, and it's what lets a subarray that starts at index 0 be counted (its "earlier prefix" is the empty one). Forget it and you undercount every subarray anchored at the start — a bug that passes half the tests, which is the worst kind.
  • Look up before you insert. If you insert running first, then a k === 0 query would match the prefix you just added and count an empty subarray. Order matters.

Why this is O(n) and not O(n²): the brute force enumerates every subarray — n(n+1)/2 of them — and sums each one. Even with the prefix array making each sum O(1), enumerating all pairs (i, j) is still O(n²). The map version never enumerates pairs at all: it fixes the right endpoint (the loop variable) and asks the map, in O(1), how many valid left endpoints exist — collapsing an entire inner loop into a lookup. One pass, n iterations, O(1) average work each ⇒ O(n) time, O(n) space (the map can hold n distinct prefix sums). Same trade as §2a, applied to ranges.

3. Hashing classics

Contains Duplicate (LC #217) EASY

Given an integer array nums, return true if any value appears at least twice, and false if every element is distinct. Nothing about positions, nothing about which value — a single boolean.

Example 1: Input: nums = [1,2,3,1] → Output: true — 1 appears at index 0 and again at index 3.
Example 2: Input: nums = [1,2,3,4] → Output: false — all four values are distinct.
Example 3: Input: nums = [1,1,1,3,3,4,3,2,4,2] → Output: true — you should return on the second 1, at index 1, without looking at the rest. Early exit matters.
Constraints: 1 ≤ nums.length ≤ 10⁵; −10⁹ ≤ nums[i] ≤ 10⁹.

Solve on LeetCode ↗

Thinking process. Brute force: for every i, scan every j > i and compare — O(n²), and at 10⁵ that's 10¹⁰ comparisons, an instant TLE. Look at what the inner loop is doing: it's asking "does this value exist anywhere else?" That's a membership query, trigger #1 from §1. So keep a Set of everything seen so far and ask it directly: if the current value is already in the set, you're done; otherwise add it and move on. One pass, O(1) per element. There is a second correct answer worth naming out loud — sort, then check adjacent pairs — which is O(n log n) time but O(1) extra space if the sort is in-place. Say both, then pick the set: "sorting gives me O(1) space but costs a log factor; the set is O(n) time at O(n) space, and with n up to 10⁵ time is the scarcer resource here." Choosing and justifying is the whole point of an easy problem. And there's a one-liner — new Set(nums).size !== nums.length — which is beautiful but always consumes the full array; the explicit loop returns early on the first repeat, which matters on Example 3.

Solution (JS) — O(n) time · O(n) space 
var containsDuplicate = function(nums) {
  const seen = new Set();   // membership only — no counts needed, so a Set, not a Map

  for (const x of nums) {
    // O(1) average lookup replaces the O(n) inner scan of the brute force
    if (seen.has(x)) return true;   // early exit: stop the moment we know the answer
    seen.add(x);
  }

  return false;   // walked the whole array without a repeat ⇒ all distinct
};

// The one-liner, for when the interviewer says "can you do it in one line?":
// return new Set(nums).size !== nums.length;   // elegant, but no early exit

Trace on Example 3 [1,1,1,3,3,4,3,2,4,2]: x=1 → not in set → add → {1}. x=1 → seen.has(1) is true → return true immediately, after two iterations. The other eight elements are never touched. ✓

Complexity: O(n) time — one pass, and Set.has/Set.add are O(1) average (hash collisions are the theoretical worst case, not the expected one). O(n) space worst case, when every element is distinct and the set grows to n. Best case the space is O(1) and the time is O(1), when the duplicate is at index 1.

Follow-ups you should expect:

  • "Do it with O(1) extra space." Sort in place, then scan adjacent pairs — O(n log n) time, O(1) space (well, O(log n) for the sort's stack). Name the tradeoff explicitly; that's the answer they want.
  • "Contains Duplicate II" (LC 219): duplicates only count if they're within k indices of each other. Same set, but you delete the element that just fell out of the window — that's a sliding window over a set (tutorial 03).
  • "What if nums doesn't fit in memory?" External sort, or a Bloom filter for a probabilistic answer. Naming Bloom filters at a bank interview (they're used in dedup pipelines and cache layers) is a genuine senior signal — just be honest that it can false-positive.
  • "The values are guaranteed to be in [1, n]" — then it's Find the Duplicate (LC 287) and you can do it with cycle detection in O(1) space. Different problem, worth recognizing (tutorial 06).
Do not use nums.indexOf(x) !== nums.lastIndexOf(x) inside a loop. It reads clean and it is O(n²) — indexOf is a linear scan, so you've rebuilt the brute force in library clothing. Any array method that searches (includes, indexOf, find, some) inside a loop is an O(n²) red flag. That habit is the single most common way JS candidates fail the complexity question after writing "correct" code.

Valid Anagram (LC #242) EASY

Given two strings s and t, return true if t is an anagram of s — that is, if t uses exactly the same characters with exactly the same multiplicities, just rearranged. Same letters, same counts, order irrelevant.

Example 1: Input: s = "anagram", t = "nagaram" → Output: true — both are {a:3, n:1, g:1, r:1, m:1}.
Example 2: Input: s = "rat", t = "car" → Output: false — same length, but t to c is a mismatch in counts.
Example 3: Input: s = "aacc", t = "ccac" → Output: false — same length, same set of characters, different counts (a:2,c:2 vs a:1,c:3). This is why a Set is not enough and you need a frequency map.
Constraints: 1 ≤ s.length, t.length ≤ 5 × 10⁴; both consist of lowercase English letters.

Solve on LeetCode ↗

Thinking process. Read the definition literally: two strings are anagrams if and only if their character-frequency maps are identical. That's not a trick — it is the definition, so the algorithm is just "compute both maps and compare." First, a free O(1) rejection: if the lengths differ they cannot be anagrams, return early. Then the elegant version uses one map instead of two: increment for every character of s, decrement for every character of t. If they're anagrams, every count cancels to zero. You can even bail out the instant a count goes negative — that means t has a character s doesn't have enough of. The obvious alternative — sort both strings and compare — is O(n log n) and one line; mention it, then say "but counting is O(n) and the alphabet is bounded, so I'll count." Since the constraints promise lowercase letters only, the map can be a fixed 26-slot array, which makes the space genuinely O(1) — say that out loud, it's the detail that turns a correct answer into a good one.

Solution (JS) — O(n) time · O(1) space (bounded alphabet) 
var isAnagram = function(s, t) {
  // Free O(1) rejection — different lengths can never be anagrams.
  if (s.length !== t.length) return false;

  // Alphabet is bounded (a–z), so a fixed 26-slot array beats a Map:
  // O(1) space, no hashing, and index = charCode - 97 ('a' is 97).
  const counts = new Array(26).fill(0);

  for (let i = 0; i < s.length; i++) {
    counts[s.charCodeAt(i) - 97]++;   // s adds
    counts[t.charCodeAt(i) - 97]--;   // t removes — same loop, lengths are equal
  }

  // Anagrams ⇒ every letter added by s was removed by t ⇒ all counts are 0.
  return counts.every(c => c === 0);
};

// General version (Unicode, any alphabet) — same idea with a Map:
var isAnagramMap = function(s, t) {
  if (s.length !== t.length) return false;
  const freq = new Map();
  for (const ch of s) freq.set(ch, (freq.get(ch) ?? 0) + 1);
  for (const ch of t) {
    const c = freq.get(ch) ?? 0;
    if (c === 0) return false;   // t has a char s ran out of — early exit
    freq.set(ch, c - 1);
  }
  return true;   // lengths matched and nothing went negative ⇒ perfect cancellation
};

Trace on Example 3 (s = "aacc", t = "ccac"): lengths match (4 = 4). i=0: a++ , c-- → a:1, c:−1. i=1: a++, c-- → a:2, c:−2. i=2: c++, a-- → a:1, c:−1. i=3: c++, c-- → a:1, c:−1. Final: a = 1 ≠ 0 → false ✓. The leftover +1 on a and −1 on c are exactly the mismatch.

Complexity: O(n) time — one pass over both strings, O(1) work per character, plus a fixed 26-element scan at the end. Space is O(1): 26 slots regardless of input size. For the general Map version, space is O(k) where k is the number of distinct characters — still bounded by the alphabet, not by n. Compare out loud with the sort-and-compare solution: O(n log n) time, and in JS [...s].sort().join('') also allocates O(n).

Follow-ups you should expect:

  • "What if the input contains Unicode?" The 26-slot array breaks. Switch to a Map, and iterate with for…of (which walks code points) rather than by index (which walks UTF-16 code units and splits emoji/surrogate pairs in half). This is a real JS trap and interviewers who know JS will ask.
  • "Case-insensitive? Ignore spaces and punctuation?" Normalize first (s.toLowerCase().replace(/[^a-z]/g, '')), then run the same algorithm. Say "I'd normalize, then count" — don't rewrite the core.
  • "Now group all the anagrams in a list of words." That's Group Anagrams, two problems down — and this frequency map becomes its key. Recognize the escalation.
  • "Why not just compare sorted strings?" You can. It's O(n log n) vs O(n), and it's a perfectly acceptable first answer if you name the cost — but you should offer the counting version as the optimization.
Both of these easies are really one question in disguise: "do you reach for a hash structure without being asked?" The interviewer is not testing anagram theory. They are calibrating you in the first five minutes so they know how hard to push in the next thirty. Solve them fast, state the complexity unprompted, and offer the alternative — then you get the interesting problem instead of another easy.

Two Sum (LC #1) EASY ★

Given an array of integers nums and an integer target, return the indices of the two numbers that add up to target. You may assume that exactly one valid answer exists, and you may not use the same element twice. The answer may be returned in any order. Note carefully: the output is indices, not values — that single word dictates the shape of your hash map.

Example 1: Input: nums = [2,7,11,15], target = 9 → Output: [0,1] — nums[0] + nums[1] = 2 + 7 = 9.
Example 2: Input: nums = [3,2,4], target = 6 → Output: [1,2] — note 3 + 3 = 6 is not allowed, because that would reuse index 0.
Example 3: Input: nums = [3,3], target = 6 → Output: [0,1] — duplicate values at different indices are fine. This is the case that breaks a naive "build the whole map first, then look up" implementation, because the second 3 overwrites the first one's index.
Constraints: 2 ≤ nums.length ≤ 10⁴; −10⁹ ≤ nums[i], target ≤ 10⁹; exactly one valid answer exists. The array is not sorted.

Solve on LeetCode ↗

Thinking process. This is the most-asked problem in the industry, and it is asked precisely because it is the smallest possible demonstration of the space-for-time trade from §2a. Brute force: two nested loops testing every pair — O(n²). Now interrogate the inner loop. For a fixed nums[i], you are not really "trying every j" — you are searching for one specific number, the complement target − nums[i]. There is exactly one value that can complete the pair, and you're scanning the whole array to find it. Searching for a known value is what a hash map deletes. So: keep a Map from value → index of everything you've already passed. For each new element, compute its complement and ask the map for it — O(1). If it's there, you have both indices and you're done. If not, record the current value → index and move on.

Two details decide whether it's clean. First, one pass, not two: check the map before inserting the current element. That single ordering choice makes "don't reuse the same element" fall out for free — the current element isn't in the map yet, so it can't match itself when target = 2 × nums[i]. It also handles Example 3 correctly: the first 3 is stored, the second 3 finds it, and neither needs the map to hold two indices for one value. Second, value is the key, index is the payload, because you look up by value and return indices. Get that backwards — index as key — and the map buys you nothing at all. Say it out loud before you type: "I'm keying by value because value is what I'm searching for."

Solution (JS) — O(n) time · O(n) space 
var twoSum = function(nums, target) {
  // value → index, for every element we've ALREADY passed.
  // Key is the value because the value is what we search for; payload is the index
  // because the index is what we return. Getting this backwards is the classic error.
  const seen = new Map();

  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];   // the ONE value that completes this pair

    // LOOK UP BEFORE INSERTING. Two consequences, both free:
    //   1. nums[i] can't pair with itself (it isn't in the map yet)
    //   2. duplicate values work: [3,3] target 6 → i=1 finds the 3 stored at i=0
    if (seen.has(complement)) {
      return [seen.get(complement), i];   // earlier index first — order doesn't matter anyway
    }

    seen.set(nums[i], i);   // nums[i] is now a candidate complement for everything to its right
  }

  return [];   // unreachable given the "exactly one solution" guarantee, but never leave a fn without a return
};

Trace on Example 1 ([2,7,11,15], target 9): i=0, nums[0]=2 → complement 7 → map is empty, no hit → store {2 → 0}. i=1, nums[1]=7 → complement 2 → seen.has(2) ✓ → return [seen.get(2), 1] = [0, 1] ✓. Two iterations, no inner loop.
Trace on Example 3 ([3,3], target 6): i=0 → complement 3 → map empty → store {3 → 0}. i=1 → complement 3 → hit → return [0, 1] ✓. Note the second 3 never overwrites the first, because we return before the set.

Complexity: O(n) time — a single pass with O(1) average has/get/ set. O(n) space for the map in the worst case (no pair found until the last element). Versus the brute force's O(n²) time / O(1) space. State the trade in one sentence: "I'm spending O(n) memory to turn the inner O(n) search into an O(1) lookup." That is the sentence the interviewer is listening for, and it is why this problem exists.

Follow-ups you should expect:

  • "What if the array is sorted?" Then you don't need the map at all — converging two pointers from both ends gives O(n) time and O(1) space (LC 167, tutorial 02). Recognizing that sortedness replaces the hash map is a real insight, and it's the bridge into the next tutorial.
  • "What if there are multiple valid pairs and you must return all of them?" You can't return early; the map now needs value → list of indices, and you must be careful not to emit the same pair twice.
  • "Now do 3Sum." (LC 15) — sort, fix one element, and two-pointer the rest: O(n²). The hash-map version exists but the duplicate handling is miserable. Tutorial 02.
  • "What if you can't use extra space?" Sort a copy (O(n log n)) and two-pointer — but sorting destroys the original indices, so you'd have to sort [value, index] pairs. Naming that gotcha unprompted is the senior move.
  • "Two Sum IV — the input is a BST." In-order traversal gives you a sorted array; same two-pointer. Or just use a set during traversal.
The two-pass version — build the whole map first, then loop again looking for complements — is the buggiest common answer. With nums = [3,3], target 6, the second 3 overwrites the first's index, so the map holds {3 → 1} and you return [1, 1]: the same element used twice. Candidates then bolt on an if (seen.get(c) !== i) guard, which fixes that but is a patch on a design mistake. The one-pass, look-up-before-you-insert ordering makes both edge cases structurally impossible. Design them out; don't guard them out.

Group Anagrams (LC #49) MEDIUM ★

Given an array of strings strs, group together all the strings that are anagrams of one another. Return the groups as an array of arrays. The groups may be returned in any order, and the strings within a group may be in any order — but every string must appear in exactly one group, and a string with no anagram partner forms a group of size one.

Example 1: Input: strs = ["eat","tea","tan","ate","nat","bat"] → Output: [["eat","tea","ate"],["tan","nat"],["bat"]] — three groups: the aet family, the ant family, and bat alone.
Example 2: Input: strs = [""] → Output: [[""]] — the empty string is its own group. Your canonical key must survive an empty input.
Example 3: Input: strs = ["a"] → Output: [["a"]] — a lone string still gets a group.
Constraints: 1 ≤ strs.length ≤ 10⁴; 0 ≤ strs[i].length ≤ 100; strs[i] consists of lowercase English letters. Note the shape: n = number of strings (up to 10⁴), k = max string length (only 100). Two different variables — your complexity must mention both.

Solve on LeetCode ↗

Thinking process. Brute force: compare every string to every other string with the Valid Anagram check — O(n² · k). Too slow, and more importantly it's the wrong shape: grouping is not a pairwise problem. The reframe that solves it: anagrams are exactly the strings that share a canonical form. If you can map every string to a fingerprint such that anagrams (and only anagrams) collide, then grouping is just Map<fingerprint, list> — one pass, done. So the entire problem reduces to one design question: what's the key?

Two good answers, and you should offer both. (a) The sorted string. "eat", "tea", "ate" all sort to "aet". Dead simple, costs O(k log k) per string ⇒ O(n · k log k) overall. (b) The character-count signature. Build the 26-slot count array from Valid Anagram and serialize it (e.g. "1#0#0#…"). That's O(k) per string ⇒ O(n · k) overall, strictly better asymptotically. In practice, with k ≤ 100, sorting is often faster because the constant factor of building and joining a 26-element array beats a tiny sort — so the honest senior answer is: "I'll sort the key because it's clearer; if k were large I'd switch to a count signature to drop the log factor." Naming the tradeoff and picking deliberately beats silently picking the theoretically-optimal one.

One more design note that matters more than either: the map's value is a list, not a count. This is knob #2 from §2c. The moment a problem says "group," the payload becomes an array and your loop body becomes "get-or-create the bucket, then push."

Solution (JS) — O(n · k log k) time · O(n · k) space 
var groupAnagrams = function(strs) {
  // canonical key → the list of original strings that share it.
  // Map (not {}) so the buckets come out in insertion order and no prototype key can collide.
  const groups = new Map();

  for (const s of strs) {
    // The fingerprint: anagrams — and ONLY anagrams — sort to the same string.
    // [...s] spreads by code point; .sort() is lexicographic, which is what we want for a–z.
    const key = [...s].sort().join('');   // "eat" → "aet", "tea" → "aet"

    // get-or-create the bucket, then push. This is THE grouping idiom — learn it once.
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key).push(s);   // push the ORIGINAL string, not the key
  }

  return [...groups.values()];   // the buckets are the answer
};

// Variant (b): O(n·k) — the count signature. Faster in theory; use it when k is large.
var groupAnagramsCount = function(strs) {
  const groups = new Map();
  for (const s of strs) {
    const counts = new Array(26).fill(0);
    for (let i = 0; i < s.length; i++) counts[s.charCodeAt(i) - 97]++;
    // The '#' separator is NOT optional: without it, [1,11] and [11,1] both stringify
    // to "111" and two different words would collide into one group.
    const key = counts.join('#');
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key).push(s);
  }
  return [...groups.values()];
};

Trace on Example 1: "eat" → key "aet" → new bucket → {aet: ["eat"]}. "tea" → key "aet" → existing bucket → {aet: ["eat","tea"]}. "tan" → key "ant" → new → {aet:[…], ant:["tan"]}. "ate" → "aet" → {aet:["eat","tea","ate"]}. "nat" → "ant" → {ant:["tan","nat"]}. "bat" → "abt" → new. Return the three buckets ✓. Every string was touched exactly once, and no string was ever compared to another string.

Complexity: with n strings of max length k: O(n · k log k) time — the sort of each key dominates (n keys × k log k each), and every map operation is O(k) at worst because hashing a length-k string reads all k characters. Space O(n · k): the map holds every string once plus its key. The count-signature variant is O(n · k) time — it removes the log factor by never sorting. Say both numbers with the variables named; "O(n log n)" is wrong here and interviewers notice, because the log factor is over the string length, not the array length.

Follow-ups you should expect:

  • "Can you avoid the sort?" Yes — the 26-count signature above, O(k) per string. Have it ready; this is the standard follow-up and the reason to know both.
  • "Why the # separator in the count key?" Because [1,11].join('') and [11,1].join('') are both "111". Delimiter-free serialization is ambiguous. This is a small question with a big signal — it's a serialization bug, and enterprise interviewers love catching it.
  • "Now the strings can be Unicode." The 26-array dies; use a Map-based signature with sorted entries, or fall back to sorting code points. Mention that [...s] (spread) handles surrogate pairs and s.split('') does not.
  • "Do it in parallel / at scale (10⁹ strings)." This is map-reduce, literally: the canonical key is the shuffle key. Saying that at a bank or healthcare shop — where the real system is a Spark job — lands well.
The transferable idea here is canonicalization, and it's worth naming as such: "I need a fingerprint that's identical for all members of a group and different for everything else." That single sentence solves this problem, "group by shifted alphabet" (LC 249), "find duplicate files" (hash the contents), and React's key prop. Interviewers grade the reframe, not the sort call — once you say "canonical key," the code is a formality.

Top K Frequent Elements (LC #347) MEDIUM ★

Given an integer array nums and an integer k, return the k most frequent elements. You may return the answer in any order, and the problem guarantees the answer is unique (no ties at the cut-off that would force an arbitrary choice). The follow-up printed on the problem itself is the real question: your algorithm must run in better than O(n log n) — which is the interviewer's way of saying "don't just sort by frequency."

Example 1: Input: nums = [1,1,1,2,2,3], k = 2 → Output: [1,2] — frequencies are {1:3, 2:2, 3:1}; the top two are 1 and 2.
Example 2: Input: nums = [1], k = 1 → Output: [1].
Example 3: Input: nums = [4,1,-1,2,-1,2,3], k = 2 → Output: [-1,2] — both appear twice, everything else once. Negative values are legal, which quietly kills any "use the value as an array index" idea.
Constraints: 1 ≤ nums.length ≤ 10⁵; k is in the range [1, number of distinct elements]; the answer is guaranteed unique.

Solve on LeetCode ↗

Thinking process. Step one is never in doubt: build the frequency map (§2b), O(n). The whole problem is step two — given {value → count}, extract the top k by count without sorting.

The obvious move is to sort the map's entries by count descending and take k. That's O(d log d) where d is the number of distinct values — correct, dead simple, and it violates the stated follow-up. The classical answer is a min-heap of size k: push each (value, count), pop when the heap exceeds k, and what's left is the top k in O(d log k). That's tutorial 08 — JavaScript has no built-in heap, so you'd have to write one, and in a 25-minute slot that's expensive. Here is the better answer, and the one you should lead with: bucket sort.

The insight is a constraint nobody points at: a frequency can never exceed n. Counts aren't arbitrary numbers — they're bounded small integers in [1, n]. Any time your sort keys are bounded integers, sorting is overkill; you can index by them. So make an array buckets of length n + 1, where buckets[c] is the list of every value that occurs exactly c times. Filling it is O(d). Then walk the buckets from the high end downward, collecting values until you have k. Total: O(n) — no sort, no heap, no log factor anywhere. Say the sentence out loud: "the counts are bounded by n, so instead of sorting by count I can bucket by count — that's a counting sort, and it makes the whole thing linear." That is the answer this problem is actually testing for.

Solution (JS) — O(n) time · O(n) space (bucket sort) 
var topKFrequent = function(nums, k) {
  // ── Step 1: the frequency map. §2b, verbatim. O(n).
  const freq = new Map();          // value → count
  for (const x of nums) freq.set(x, (freq.get(x) ?? 0) + 1);

  // ── Step 2: bucket by count. THE KEY INSIGHT: a count is at most nums.length,
  // so counts are bounded small integers ⇒ use them as ARRAY INDICES instead of sort keys.
  // buckets[c] = every value that appears exactly c times. Index 0 is unused (no value
  // occurs 0 times), which is why the array is length n+1.
  const buckets = Array.from({ length: nums.length + 1 }, () => []);
  for (const [value, count] of freq) {
    buckets[count].push(value);
  }

  // ── Step 3: walk buckets from the HIGHEST count downward, taking values until we have k.
  const res = [];
  for (let c = buckets.length - 1; c >= 1 && res.length < k; c--) {
    for (const value of buckets[c]) {
      res.push(value);
      if (res.length === k) return res;   // exact-k early exit; guarantees we never over-collect
    }
  }

  return res;
};

Trace on Example 1 ([1,1,1,2,2,3], k = 2): freq = {1→3, 2→2, 3→1}. n = 6, so buckets has 7 slots. Fill: buckets[3] = [1], buckets[2] = [2], buckets[1] = [3]; the rest empty. Walk down from c = 6: empty, empty, empty (c = 6,5,4) → c = 3 → push 1, res = [1] → c = 2 → push 2, res.length === 2 === k → return [1,2] ✓. The 3 in buckets[1] is never even visited.

Complexity: O(n) time. Building the map is O(n). Filling the buckets is O(d) where d ≤ n distinct values. The downward walk visits at most n + 1 bucket slots and pushes at most d values total, so it's O(n) even though it looks like a nested loop — the inner loop's total work across all iterations is bounded by the number of distinct values, not multiplied by it. (Same amortized argument as a monotonic stack: count the items, not the loops.) O(n) space for the map plus the bucket array. Beat that against the sort (O(d log d)) and the heap (O(d log k)) out loud — the interviewer asked for "better than O(n log n)" and you delivered strictly linear.

Follow-ups you should expect:

  • "Do it with a heap instead." Min-heap of size k over the map entries, comparing by count: push, and pop when size > k. O(d log k) time, O(k) space — less space than bucketing, which is the heap's real advantage and the honest reason to prefer it when d is huge and k is tiny. JavaScript has no built-in heap; you write one (tutorial 08). Know the tradeoff even if you code the buckets.
  • "What if the data is a stream and you can't hold it all?" Then it's a streaming top-k: a bounded min-heap, or approximate structures like Count–Min Sketch / Space-Saving. Naming Count–Min Sketch at a fintech shop (they use it for real, in fraud and telemetry pipelines) is a strong signal.
  • "What if there are ties at the k-th position?" The problem guarantees uniqueness; if it didn't, you'd need a documented tie-break rule. Ask — clarifying an ambiguous spec is graded.
  • "Quickselect?" Yes — partition the entries around the k-th largest count: O(d) average, O(d²) worst. It's the theoretically cute answer, but bucket sort is O(n) worst case here and much easier to get right under pressure. Say exactly that.
Bucket sort works here for one reason and you must be able to state it: the sort keys are bounded integers. Frequencies live in [1, n], so they can index an array. The moment the keys are unbounded, or floats, or need a custom comparator, bucketing dies and you're back to a heap or a sort. This is the same reason counting sort beats comparison sort on small-range integers — and it's why "sort the frequencies" is a code smell, not a solution. Don't reach for it reflexively; reach for it because the keys are bounded, and say so.

Majority Element (LC #169) EASY

Given an array nums of size n, return the majority element — the element that appears more than ⌊n/2⌋ times. You may assume the majority element always exists in the array. That guarantee is not a convenience; it is the hinge the clever solution swings on, so read it twice.

Example 1: Input: nums = [3,2,3] → Output: 3 — 3 appears twice out of 3, and 2 > ⌊3/2⌋ = 1.
Example 2: Input: nums = [2,2,1,1,1,2,2] → Output: 2 — 2 appears 4 times out of 7 (> 3). Note it is not a contiguous run, and note that 1 briefly "leads" in the middle of the array — any solution that commits early is wrong.
Constraints: n = nums.length; 1 ≤ n ≤ 5 × 10⁴; −10⁹ ≤ nums[i] ≤ 10⁹. Follow-up printed on the problem: solve it in O(n) time and O(1) space.

Solve on LeetCode ↗

Thinking process. The frequency map answers it in five lines: count everything, return the key whose count exceeds n / 2. O(n) time, O(n) space. Write that first — it's correct, it's fast, and having a working solution on the board before you get clever is exactly the enterprise-tier play. Then the interviewer asks the follow-up printed on the problem: O(1) space. That's your cue for Boyer–Moore voting.

Derive it, don't recall it. Imagine every element is a voter holding a token. Take one candidate and let every disagreeing element cancel one of its tokens, one-for-one. Because the majority element appears more than half the time, it has strictly more tokens than everything else combined — so no matter how the cancellations pair up, it is mathematically impossible for it to be fully cancelled. Everything else can, at best, annihilate itself against it. That gives an algorithm with two variables: a current candidate and a count. Walk the array. If count === 0, adopt the current element as the new candidate. Then increment count if the element agrees with the candidate, decrement if it disagrees. Whatever candidate you're holding at the end is the majority element.

The subtle part — and the thing to say out loud, because it's where candidates get uneasy — is what the algorithm does not claim. count is not the frequency of the candidate. It's the running margin: how far the current candidate is ahead of everything else in this stretch of the array. And the guarantee only holds because a majority exists. If it might not exist, this returns garbage — so you'd add a second verification pass counting the candidate's true occurrences. Volunteering that unprompted is the difference between "I memorized Boyer–Moore" and "I understand Boyer–Moore."

Solution (JS) — O(n) time · O(1) space (Boyer–Moore voting) 
// ── Answer 1: the frequency map. Write this FIRST — it's correct and it takes 30 seconds.
var majorityElementMap = function(nums) {
  const freq = new Map();
  for (const x of nums) {
    const c = (freq.get(x) ?? 0) + 1;
    if (c > nums.length / 2) return x;   // check as you go — no second pass needed
    freq.set(x, c);
  }
};

// ── Answer 2: Boyer–Moore voting. O(1) space — the printed follow-up.
var majorityElement = function(nums) {
  let candidate = null;
  let count = 0;   // NOT the candidate's frequency — its MARGIN over everything else so far

  for (const x of nums) {
    // Margin hit zero: the previous candidate was fully cancelled out in this stretch,
    // so it's dead to us. Adopt x and start a fresh contest.
    if (count === 0) candidate = x;

    // x either supports the candidate (+1) or cancels one of its tokens (−1).
    count += (x === candidate) ? 1 : -1;
  }

  // The true majority has MORE tokens than everything else COMBINED, so it can never
  // be fully cancelled — it must be the one left standing.
  return candidate;

  // If majority were NOT guaranteed, you'd need a verification pass:
  //   let c = 0; for (const x of nums) if (x === candidate) c++;
  //   return c > nums.length / 2 ? candidate : -1;
};

  • x=2: count 0 → candidate = 2. 2 === 2 → count 1.
  • x=2: agrees → count 2.
  • x=1: disagrees → count 1.
  • x=1: disagrees → count 0. Candidate 2 is fully cancelled — but it isn't eliminated, it's just no longer protected.
  • x=1: count is 0 → candidate = 1. Agrees → count 1.
  • x=2: disagrees → count 0.
  • x=2: count is 0 → candidate = 2. Agrees → count 1.
  • Return 2 ✓. Notice how 1 genuinely took over as candidate in the middle and still lost — the cancellations can't save a non-majority element over the whole array.

Complexity: O(n) time — one pass, two comparisons per element. O(1) space — two scalars, regardless of n. That's the entire reason this algorithm exists; the map version is equally fast in time but pays O(n) memory. State it as a trade: "same time, but I've dropped the O(n) map for two variables."

Follow-ups you should expect:

  • "What if the majority element might not exist?" Add the verification pass (commented above). Still O(n) time, O(1) space — you just do two passes instead of one. Volunteer this before they ask.
  • "Majority Element II — find all elements appearing more than ⌊n/3⌋ times." (LC 229) Generalized Boyer–Moore: there can be at most two such elements (three would need > n total), so you track two candidates and two counters, then verify both. Being able to say "at most ⌊n/k⌋ candidates, so I keep k−1 counters" shows you understood the mechanism, not the code.
  • "Sorting solution?" Sort and return nums[Math.floor(n/2)] — the middle element must be the majority, because an element occupying more than half the slots always covers the midpoint. O(n log n), one line, and a genuinely elegant thing to mention.
  • "Randomized?" Pick a random element and verify; it succeeds with probability > 1/2 per attempt, so it's O(n) expected. A fun aside; don't lead with it.
  • Where it's real: Boyer–Moore is a streaming algorithm — it needs O(1) memory and one pass, so it runs on data you can't store. That's why it shows up in log/telemetry pipelines and in "find the dominant ticker in this trade feed" style fintech questions.
Notice the shape of this problem, because it recurs all over the enterprise tier: the hash map is the expected answer, and the O(1)-space algorithm is the bonus. The winning play is not to open with Boyer–Moore to look clever — it's to write the map solution in 60 seconds, state its O(n) space cost yourself, and then say "if you want O(1) space, there's a voting algorithm — the majority element can't be fully cancelled by everything else combined." You've now demonstrated speed, correctness, self-critique, and depth, in that order. That sequence is what a 7-YOE rubric is built to reward.

Same trade as §2a — precompute so you stop recomputing — but now the thing you precompute is a running aggregate instead of a lookup table. Two problems, and they are the two halves of the idea: Product Except Self is prefix aggregates without a hash map (two directional sweeps), and Subarray Sum Equals K is prefix aggregates with one (the payoff). Product of Array Except Self (LC #238) MEDIUM ★

Given an integer array nums, return an array answer such that answer[i] is equal to the product of all the elements of nums except nums[i]. Two hard requirements make this a real problem instead of a one-liner: you must run in O(n) time, and you may not use the division operator. (The problem also guarantees every product fits in a 32-bit integer, so you never have to worry about overflow.)

Example 1: Input: nums = [1,2,3,4] → Output: [24,12,8,6] — e.g. answer[0] = 2·3·4 = 24, answer[2] = 1·2·4 = 8.
Example 2: Input: nums = [-1,1,0,-3,3] → Output: [0,0,9,0,0] — the single zero makes every product zero except at the zero's own index, where the answer is the product of the others: (−1)·1·(−3)·3 = 9. This is the example that kills the division approach — you'd be dividing the total product (which is 0) by 0.
Example 3: Input: nums = [0,0,2] → Output: [0,0,0] — two zeros means every answer is zero, including at each zero's index. A "count the zeros and special-case it" solution has to handle 0, 1, and ≥2 zeros separately. The clean solution handles all three without knowing zeros exist.
Constraints: 2 ≤ nums.length ≤ 10⁵; −30 ≤ nums[i] ≤ 30; the product of any prefix or suffix fits in a 32-bit integer. Follow-up: can you do it in O(1) extra space (the output array doesn't count)?

Solve on LeetCode ↗

Thinking process. Brute force: for each i, multiply the other n−1 elements — O(n²). The seductive "optimization" is to compute the total product once and set answer[i] = total / nums[i] — O(n), but it's banned, and the ban is not arbitrary spite: Example 2 shows exactly why it's fragile. A single zero makes total zero and destroys all the information; two zeros make it worse. Division is the wrong tool even when it's allowed.

So what is answer[i], really? It's (everything to the left of i) × (everything to the right of i). That's the whole insight, and it's just the "…except itself" trigger from §1. Both of those factors are prefix aggregates, and both can be built in a single directional sweep:

  • left[i] = product of nums[0 … i−1] — build it left to right: left[i] = left[i−1] × nums[i−1], with left[0] = 1 (the empty product).
  • right[i] = product of nums[i+1 … n−1] — build it right to left, symmetrically, with right[n−1] = 1.
  • answer[i] = left[i] × right[i]. Done. Three passes, O(n) time, O(n) space.

Then the follow-up: O(1) extra space. Notice that left can be written directly into the output array (it's not "extra" — the output never counts), and then right doesn't need an array at all, because you consume it immediately: sweep back through the output multiplying by a single running suffix product held in one variable. Two passes, one accumulator, no extra array. That's the version to write. And notice what never appears in the code: any mention of zero. The zeros in Examples 2 and 3 are handled by ordinary multiplication — you never special-case them. Say that out loud; it's the sign you found the right decomposition rather than a patched-up one.

Solution (JS) — O(n) time · O(1) extra space 
var productExceptSelf = function(nums) {
  const n = nums.length;
  const answer = new Array(n);

  // ── Pass 1 (left → right): answer[i] = product of EVERYTHING LEFT OF i.
  // We build the prefix products straight into the output array, so they cost no extra space.
  let prefix = 1;                 // the empty product — nothing is left of index 0
  for (let i = 0; i < n; i++) {
    answer[i] = prefix;           // write BEFORE multiplying: nums[i] must be excluded from its own answer
    prefix *= nums[i];            // now prefix covers 0..i, ready for i+1
  }

  // ── Pass 2 (right → left): multiply in the product of EVERYTHING RIGHT OF i.
  // The suffix product is a single scalar — we consume it as we go, so no second array.
  let suffix = 1;                 // nothing is right of index n-1
  for (let i = n - 1; i >= 0; i--) {
    answer[i] *= suffix;          // left-of-i (already there) × right-of-i = the answer
    suffix *= nums[i];            // now suffix covers i..n-1, ready for i-1
  }

  // No division, and no special case for zeros — they just multiply like any other number.
  return answer;
};

Trace on Example 1 [1,2,3,4]:

  • Pass 1: i=0 → answer[0] = 1, prefix = 1. i=1 → answer[1] = 1, prefix = 2. i=2 → answer[2] = 2, prefix = 6. i=3 → answer[3] = 6, prefix = 24. Now answer = [1, 1, 2, 6] — each slot holds the product of everything to its left.
  • Pass 2: i=3 → answer[3] = 6 × 1 = 6, suffix = 4. i=2 → answer[2] = 2 × 4 = 8, suffix = 12. i=1 → answer[1] = 1 × 12 = 12, suffix = 24. i=0 → answer[0] = 1 × 24 = 24, suffix = 24.
  • Result [24,12,8,6] ✓.

Trace on Example 2 [-1,1,0,-3,3] (the zero case): Pass 1 leaves answer = [1, -1, -1, 0, 0] (once the prefix hits the 0 at index 2, every later prefix is 0). Pass 2 sweeps back with suffix = 1 → answer[4] = 0·1 = 0, suffix = 3; answer[3] = 0·3 = 0, suffix = −9; answer[2] = −1 · −9 = 9 ✓; suffix = 0 (multiplied by nums[2] = 0); answer[1] = −1 · 0 = 0; answer[0] = 1 · 0 = 0. Result [0,0,9,0,0] ✓. The 9 appears at exactly the index whose own zero was excluded — and nowhere in the code did we test for zero.

Complexity: O(n) time — two linear passes, one multiply each. O(1) extra space: two scalar accumulators (prefix, suffix). The output array is required by the problem and by convention does not count against auxiliary space — say that convention out loud, because an interviewer who hears "O(1) space" while looking at an n-element array will otherwise assume you're confused.

Follow-ups you should expect:

  • "What if division were allowed?" Then: count the zeros. Zero zeros → total / nums[i]. Exactly one zero → every answer is 0 except at the zero's index, which gets the product of the non-zero elements. Two or more zeros → all zeros. It's O(n) and it works, but it's three branches instead of none, and the prefix/suffix version is both cleaner and safer against floating-point drift. Say precisely that; it shows you evaluated the alternative rather than reciting a rule.
  • "Now make it sums instead of products." Identical structure: answer[i] = totalSum − nums[i], or the same two sweeps. Sums have an inverse that's always safe, so subtraction is fine — which is exactly why the problem chose products: multiplication's inverse breaks at zero.
  • "Range product / range sum queries on a static array." That's the prefix array from §2c, answering any query in O(1). Products need care with zeros; sums don't.
  • "Overflow?" The constraints promise 32-bit safety. In real JS, numbers past 2⁵³ silently lose precision — you'd reach for BigInt. Mention it once; it's a production instinct, not an interview requirement.
The ordering inside pass 1 is where this problem is won or lost. It must be answer[i] = prefix then prefix *= nums[i] — never the reverse. Multiply first and prefix already includes nums[i], so answer[i] contains the very element it's supposed to exclude, and the whole thing is silently wrong (it becomes the full product at every index). It's a one-line swap and it produces plausible-looking output, which makes it brutal to debug at a whiteboard. The mental check: "at the moment I write answer[i], has prefix seen nums[i] yet? It must not have."

Subarray Sum Equals K (LC #560) MEDIUM

Given an array of integers nums and an integer k, return the total number of contiguous subarrays whose elements sum to exactly k. Not the longest one, not whether one exists — the count. Subarrays are contiguous and non-empty, and two subarrays are different if they start or end at different indices, even if they contain the same values. Crucially: nums may contain negative numbers and zeros.

Example 1: Input: nums = [1,1,1], k = 2 → Output: 2 — the subarrays [0..1] and [1..2]. They overlap; both count.
Example 2: Input: nums = [1,2,3], k = 3 → Output: 2 — [1,2] and [3]. A single element is a valid subarray.
Example 3: Input: nums = [1,-1,0], k = 0 → Output: 3 — [1,-1], [-1,0]… wait, that sums to −1. The three are [1,-1], [0], and [1,-1,0]. This is the example that proves sliding window cannot work here: with negatives, growing the window can decrease the sum, so there's no monotonic condition to shrink on.
Constraints: 1 ≤ nums.length ≤ 2 × 10⁴; −1000 ≤ nums[i] ≤ 1000; −10⁷ ≤ k ≤ 10⁷.

Solve on LeetCode ↗

Thinking process. This is the payoff problem of the tutorial: both templates fire at once, and if you can derive it cold you own the pattern. Start with the brute force — for every start i, extend to every end j, accumulating the sum: O(n²), roughly 4 × 10⁸ operations at the constraints. Too slow, and, more importantly, wasteful in a specific way: you're re-adding the same middle elements over and over.

Kill that waste with the prefix identity from §2c: sum(i..j) = prefix[j+1] − prefix[i]. So a subarray ending at j sums to k exactly when prefix[i] = prefix[j+1] − k. Read that again, because it is the entire problem: you have converted "search for a subarray" into "search for a previously-seen prefix value." And searching for a known value is precisely what §2a says a hash map deletes.

So: sweep left to right holding a running sum. At each index, the number of valid subarrays ending here is the number of earlier prefixes equal to running − k — which a Map<prefixValue, howManyTimesSeen> answers in O(1). Add that to the total, then record the current prefix. Three things decide correctness, and you should name all three:

  • Counts, not membership. The map's value must be a count, because several different start indices can produce the same prefix sum, and each is a distinct subarray. A Set here undercounts — it's the single most common bug in this problem.
  • Seed the map with {0 → 1}. That's the empty prefix. Without it, any subarray starting at index 0 (whose "earlier prefix" is the empty one) is never counted. On Example 2 you'd return 1 instead of 2.
  • Look up before you insert. Otherwise, when k = 0, the prefix you just wrote matches itself and you count a phantom empty subarray.

And the sentence that earns the most credit here has nothing to do with hash maps: "I can't use a sliding window, because the array contains negatives — expanding the window doesn't monotonically increase the sum, so there's no valid shrink condition. That's what forces prefix-sum plus a hash map." Interviewers ask this problem specifically to see whether you'll reflexively reach for a window. Say why you didn't.

Solution (JS) — O(n) time · O(n) space 
var subarraySum = function(nums, k) {
  // prefixSumValue → HOW MANY TIMES that running sum has occurred so far.
  // A COUNT, not a boolean: several different start indices can share a prefix sum,
  // and each one is a separate valid subarray. A Set here silently undercounts.
  //
  // Seed with (0 → 1): the EMPTY prefix has sum 0 and has "occurred once".
  // This is what lets a subarray that starts at index 0 be counted. Forget it and
  // you undercount every answer anchored at the start of the array.
  const counts = new Map([[0, 1]]);

  let running = 0;   // prefix sum of nums[0..i]
  let total = 0;     // how many valid subarrays we've found

  for (const x of nums) {
    running += x;

    // sum(i..j) === k  ⟺  running - prefix[i] === k  ⟺  prefix[i] === running - k.
    // So EVERY earlier prefix equal to (running - k) is the start of a subarray ending here.
    // LOOK UP BEFORE INSERTING, or a k of 0 would match the prefix we're about to add.
    total += counts.get(running - k) ?? 0;

    // Register the current prefix as a possible start-boundary for future indices.
    counts.set(running, (counts.get(running) ?? 0) + 1);
  }

  return total;
};

Trace on Example 1 ([1,1,1], k = 2):

  • Start: counts = {0:1}, running = 0, total = 0.
  • x=1 → running = 1. Look up 1 − 2 = −1 → absent → total stays 0. Record: {0:1, 1:1}.
  • x=1 → running = 2. Look up 2 − 2 = 0 → count 1 → total = 1 (that's the subarray [1,1] at indices 0–1, whose start boundary is the empty prefix — this is the seed earning its keep). Record: {0:1, 1:1, 2:1}.
  • x=1 → running = 3. Look up 3 − 2 = 1 → count 1 → total = 2 (the subarray [1,1] at indices 1–2). Record: {0:1, 1:1, 2:1, 3:1}.
  • Return 2 ✓. Note we never enumerated a single pair of indices.

Trace on Example 3 ([1,-1,0], k = 0): counts {0:1}. x=1 → running 1, look up 1 → absent → total 0; counts {0:1, 1:1}. x=−1 → running 0, look up 0 → count 1 → total 1 (subarray [1,-1]); counts {0:2, 1:1} — note 0 now has count 2, which is exactly why the map stores counts. x=0 → running 0, look up 0 → count 2 → total 3 (adding [0] and [1,-1,0]); return 3 ✓. A Set would have returned 2 here.

Complexity: O(n) time — one pass, one get and one set per element, each O(1) average. O(n) space — worst case every prefix sum is distinct and the map holds n + 1 entries. Compare out loud with the O(n²) brute force and, more importantly, explain why the O(1)-space sliding window is unavailable: negatives break the monotonicity a window depends on.

Follow-ups you should expect:

  • "What if all numbers are positive?" Then a sliding window works and gives O(1) space — expanding always increases the sum, so you can shrink from the left when you overshoot. Being able to state exactly which precondition unlocks the window is the highest-value thing you can say on this problem.
  • "Return the longest such subarray instead of the count." (LC 325) Change the map's value from a count to the earliest index at which each prefix occurred, and only record a prefix the first time you see it — earliest start ⇒ longest subarray. Same skeleton, one knob turned.
  • "Subarray sums divisible by K." (LC 974) Key the map by running % k instead of running — two prefixes with the same remainder bracket a divisible subarray. Watch JS's %: it returns negative results for negative operands, so normalize with ((r % k) + k) % k. That gotcha is a frequent silent failure.
  • "Contiguous Array — equal 0s and 1s." (LC 525) Map every 0 to −1, then it's "longest subarray summing to 0." Same template, a relabeling trick on top.
  • "Binary Subarrays With Sum / count of subarrays with sum ≤ k" — for non-negative inputs, "exactly k" = "at most k" − "at most k−1", each computed with a sliding window. Different pattern; know it exists.
This problem is a family, not a one-off, and the family is worth naming out loud: "count/length of subarrays satisfying a condition on their aggregate" — LC 560, 325, 974, 525, 523 are the same eight lines with the key or the payload changed. The moment you hear "subarray" plus "sum," write prefix[j] − prefix[i] on the board before you write any code, and then ask yourself the two questions that finish it: "what do I key the map by?" and "what do I store — a count, or an index?" Deriving the answer from the algebra in front of the interviewer beats reciting a memorized solution by a wide margin, because they can then throw any variant at you and watch you handle it.

Reverse Words in a String (LC #151) MEDIUM

Given an input string s, reverse the order of the words. A word is a maximal run of non-space characters. The input may contain leading spaces, trailing spaces, and multiple consecutive spaces between words; the returned string must have the words in reverse order joined by a single space, with no leading or trailing space. The letters within each word are not reversed — only the word order.

Example 1: Input: s = "the sky is blue" → Output: "blue is sky the" — the clean case.
Example 2: Input: s = " hello world " → Output: "world hello" — leading and trailing spaces must be stripped from the output entirely.
Example 3: Input: s = "a good example" → Output: "example good a" — the three spaces between "good" and "example" collapse to one. This is the case a naive split(' ').reverse().join(' ') fails, because splitting on a single-space literal produces empty-string "words".
Constraints: 1 ≤ s.length ≤ 10⁴; s contains English letters, digits, and spaces; there is at least one word. Follow-up: if the string were a mutable character array, could you do it in O(1) extra space?

Solve on LeetCode ↗

Thinking process. The algorithm is trivial — tokenize, reverse, join — so the entire interview is the tokenizer. Before you type a character, say the three edge cases out loud: leading spaces, trailing spaces, repeated interior spaces. Naming them unprompted is worth more here than the code, because the interviewer's rubric is literally a list of them.

In JavaScript there are two honest ways to tokenize. (a) s.trim().split(/\s+/) — trim kills the leading/trailing spaces, and splitting on the regex \s+ (one-or-more whitespace) collapses the interior runs. Both edge cases die in one line. (b) A manual two-pointer scan from the right, skipping spaces and collecting each word — more code, but it's what you'd write in a language without regex, and it's the honest answer if the interviewer says "don't use split." Lead with (a), then offer (b) if they push. Do not use split(' ') — that splits on a single literal space, so "a b" yields ["a", "", "b"] and the empty string becomes a phantom word that reappears in your output as a double space. That's Example 3, and it's the trap.

The O(1)-space follow-up is worth having ready as a sentence, not as code: reverse the entire character array in place, then reverse each individual word in place. Two reversals compose back to "word order flipped, letters intact" — the same "two reversals restore order" trick that makes the two-stack queue work. In JS, strings are immutable, so O(1) is impossible without a char array; say that explicitly rather than pretending.

Solution (JS) — O(n) time · O(n) space 
var reverseWords = function(s) {
  return s
    .trim()          // kills LEADING and TRAILING spaces (edge cases 1 and 2)
    .split(/\s+/)    // \s+ = one-or-more whitespace ⇒ COLLAPSES interior runs (edge case 3).
                     // split(' ') would produce empty-string "words" here — that's the classic bug.
    .reverse()       // the actual "algorithm", such as it is
    .join(' ');     // exactly one space between words, per the spec
};

// Manual tokenizer — the answer when the interviewer says "no split/regex".
// Scan from the RIGHT so words come out in reverse order naturally.
var reverseWordsManual = function(s) {
  const words = [];
  let i = s.length - 1;

  while (i >= 0) {
    while (i >= 0 && s[i] === ' ') i--;          // skip the run of spaces (handles ALL three cases)
    if (i < 0) break;                             // only spaces were left

    const end = i;                                // last char of this word
    while (i >= 0 && s[i] !== ' ') i--;          // walk left to the word's start
    words.push(s.slice(i + 1, end + 1));           // i now sits on the space BEFORE the word
  }

  return words.join(' ');   // already in reverse order — we scanned right to left
};

Trace on Example 3 "a good example": trim() → unchanged (no outer spaces). split(/\s+/) → ["a", "good", "example"] — the three-space run is a single delimiter, so no empty strings appear. reverse() → ["example", "good", "a"]. join(' ') → "example good a" ✓. Contrast split(' '), which would give ["a","good","","","example"] → joined back as "example good a" ✗.
Trace on Example 2 " hello world ": trim() → "hello world"; split → ["hello","world"]; reverse → ["world","hello"]; join → "world hello" ✓.

Complexity: O(n) time — trim, split, reverse and join are each a linear pass, and four linear passes is still O(n). O(n) space for the token array and the output string. In JS you cannot do better than O(n) space, because strings are immutable — every "modification" allocates. Say that; it's the correct answer to the O(1)-space follow-up, not a dodge.

Follow-ups you should expect:

  • "Do it in O(1) extra space." Only possible on a mutable char array (C++/Java char[]): reverse the whole array, then reverse each word in place. Describe it in one sentence and note the JS immutability constraint — that's the full-credit answer.
  • "Don't use split or regex." The manual right-to-left scanner above. Have it ready; this is asked more often than you'd think, precisely because split(/\s+/) makes the problem trivial.
  • "Reverse the characters of each word but keep word order" (LC 557) — the mirror problem. s.split(' ').map(w => [...w].reverse().join('')).join(' '). Note that here split(' ') is correct, because that problem's spec preserves the original spacing exactly.
  • "What counts as whitespace?" Tabs? Newlines? \s covers them; a literal ' ' doesn't. Asking this is a free point.
The whole problem is split(' ') vs split(/\s+/), and it is silently wrong — the single-space version passes Example 1, passes most hand-written tests, and then produces "example good a" on the one input that has a double space. If you take one habit from this problem, take this: whenever you tokenize a string, immediately ask "what happens with zero-length tokens?" Every real-world CSV/log-parsing bug you have ever debugged is a version of this question, and interviewers at data-heavy shops know it.

Roman to Integer (LC #13) EASY

Roman numerals are written with seven symbols — I=1, V=5, X=10, L=50, C=100, D=500, M=1000 — normally in descending order, and the value is the sum of the symbols. But there are six subtractive cases, where a smaller symbol placed before a larger one means "subtract it": IV=4, IX=9, XL=40, XC=90, CD=400, CM=900. Given a valid Roman numeral string, return its integer value.

Example 1: Input: s = "III" → Output: 3 — pure addition, 1 + 1 + 1.
Example 2: Input: s = "LVIII" → Output: 58 — L(50) + V(5) + I + I + I. No subtraction anywhere; symbols are already descending.
Example 3: Input: s = "MCMXCIV" → Output: 1994 — M(1000) + CM(900) + XC(90) + IV(4). Three subtractive pairs in one string; this is the test case that separates the correct solutions from the almost-correct ones.
Constraints: 1 ≤ s.length ≤ 15; s contains only the seven Roman characters; the input is guaranteed to be a valid Roman numeral in the range [1, 3999].

Solve on LeetCode ↗

Thinking process. The instinct that produces bad code is to hard-code the six subtractive pairs as a second lookup table and scan two characters at a time, juggling an index that sometimes advances by 1 and sometimes by 2. That works, but it's fiddly and it's twelve map entries you have to recall correctly under pressure.

The elegant reframe collapses all six special cases into one local rule. Look at what every subtractive pair has in common: the symbol is smaller than the symbol immediately after it. And notice that IV = 5 − 1 is the same as (−1) + 5. So you don't need pairs at all — you need a sign: if a symbol's value is less than the value of the symbol to its right, subtract it; otherwise add it. That single rule generates IV, IX, XL, XC, CD, CM and every additive case, with one lookup table of seven entries and no pair handling. Walk the string once, compare each symbol to its neighbor, add or subtract. The last character has no right-hand neighbor, so it is always added — which falls out naturally if you scan right-to-left and track the largest value seen so far, or just guard the index if you scan left-to-right.

Say the reframe out loud: "instead of special-casing six pairs, I'll note that a subtractive symbol is exactly one that's smaller than its right neighbor — so I just need a sign, not a pair table." On an easy problem, the reasoning is the only thing being graded; anyone can produce 1994 eventually.

Solution (JS) — O(n) time · O(1) space 
var romanToInt = function(s) {
  // Seven entries. NOT twelve — the six subtractive pairs are handled by the sign rule below.
  const value = { I: 1, V: 5, X: 10, L: 50, C: 100, D: 500, M: 1000 };

  let total = 0;

  for (let i = 0; i < s.length; i++) {
    const curr = value[s[i]];
    const next = value[s[i + 1]];   // undefined past the end — see the comparison below

    // THE ONE RULE: a symbol smaller than its right neighbour is subtractive (IV, IX, XL, …).
    // On the last character, next is undefined, and (curr < undefined) is FALSE
    // (NaN comparisons are always false) — so the last symbol is always ADDED. Convenient,
    // but relying on NaN semantics is fragile; the explicit form is written below.
    if (next !== undefined && curr < next) {
      total -= curr;   // e.g. the I in "IV": contributes -1, then V contributes +5 ⇒ 4
    } else {
      total += curr;
    }
  }

  return total;
};

Trace on Example 3 "MCMXCIV":

  • i=0 M(1000), next C(100): 1000 ≥ 100 → add → total = 1000.
  • i=1 C(100), next M(1000): 100 < 1000 → subtract → total = 900.
  • i=2 M(1000), next X(10): add → total = 1900. (The CM pair has now contributed −100 + 1000 = 900. ✓)
  • i=3 X(10), next C(100): 10 < 100 → subtract → total = 1890.
  • i=4 C(100), next I(1): add → total = 1990. (XC = 90 ✓)
  • i=5 I(1), next V(5): subtract → total = 1989.
  • i=6 V(5), no next → add → total = 1994 ✓. (IV = 4 ✓)

Complexity: O(n) time where n = s.length ≤ 15 — one pass, two O(1) lookups per character. O(1) space: the value table has exactly 7 entries regardless of input, so it's a constant, not a function of n. (Being precise about "a fixed-size table is O(1), not O(k)" is a small thing that gets noticed.)

Follow-ups you should expect:

  • "Now do Integer to Roman" (LC 12) — the reverse, and it's asked as the immediate follow-up more often than not. Greedy: keep a table of value/symbol pairs including the six subtractive ones ([1000,"M"], [900,"CM"], [500,"D"], [400,"CD"], …), sorted descending, and repeatedly subtract the largest value that fits, appending its symbol. Interestingly the reverse direction is where the pair table earns its keep — have this ready.
  • "What if the input is invalid — "IIII", or "IC"?" LeetCode guarantees validity. In production you'd validate with a regex or check that no symbol repeats more than three times and that only the six legal subtractive pairs occur. Offer the validation contract unprompted — at a bank, "what does this do with bad input?" is the question you'll actually be asked on the job.
  • "Why is a plain object OK here when §2 said use a Map?" Because the keys are seven known, fixed, single-character strings — no coercion risk, no user-supplied keys, no prototype collision. Knowing when the rule doesn't apply is stronger than following it blindly.
The pattern under this problem is "replace a table of special cases with a local invariant." Six subtractive pairs become one comparison against the right neighbour. You'll meet the same move in Valid Parentheses (a closer/opener map instead of an if-chain), in Basic Calculator (precedence resolved eagerly instead of parsed), and in every reducer you've ever refactored from a switch-of-twelve-cases into three rules. When your solution has a growing list of special cases, stop and hunt for the invariant that generates them — that hunt is senior engineering, and interviewers can hear you doing it.

Longest Consecutive Sequence (LC #128) MEDIUM ★

Given an unsorted array of integers nums, return the length of the longest sequence of consecutive integers present in the array — that is, the longest run of values x, x+1, x+2, … all of which appear somewhere in nums. The elements do not need to be adjacent in the array, and their order in the array is irrelevant; only their values matter. Duplicates in the input do not extend a sequence. The catch: your algorithm must run in O(n) time.

Example 1: Input: nums = [100,4,200,1,3,2] → Output: 4 — the sequence [1,2,3,4] is present (scattered across the array); 100 and 200 are isolated.
Example 2: Input: nums = [0,3,7,2,5,8,4,6,0,1] → Output: 9 — the values 0 through 8 are all present. Note the duplicate 0: it must not make the answer 10.
Example 3: Input: nums = [] → Output: 0 — the empty case, and the reason your loop must not assume there's a first element.
Constraints: 0 ≤ nums.length ≤ 10⁵; −10⁹ ≤ nums[i] ≤ 10⁹. That value range is enormous, which kills any "index into an array of size max−min" idea. And the O(n) requirement explicitly forbids the obvious sort.

Solve on LeetCode ↗

Thinking process. The reflex answer is to sort and scan for the longest consecutive run, dedupe as you go. That is correct, it's about eight lines, and it's O(n log n) — write it if you're stuck, because a working solution beats a stalled one. But the problem explicitly demands O(n), which is the interviewer telling you: get rid of the sort.

Here's the reframe. Sorting exists to answer "what comes next?" But you don't need the whole ordering — you only need to answer "is x + 1 in the array?", which is a membership query, which is a Set, which is O(1). So dump every value into a Set (which also deduplicates for free, handling Example 2) and then, for each value, walk upward: x, x+1, x+2… while the set still contains them, counting as you go.

But that's still O(n²) as written — for [1,2,3,…,n], starting from 1 you'd walk n steps, from 2 another n−1, and so on. The fix is the entire problem, and it is one if statement: only start walking from a value that is the beginning of its sequence. And a value x begins a sequence exactly when x − 1 is not in the set — again, an O(1) membership check. With that guard, every sequence is walked from its head exactly once, and no value is ever visited by a walk that didn't start at its own head.

Now the complexity argument, and this is what the problem is really testing. The code has a loop inside a loop and it is nevertheless O(n): across the entire run, each value is visited by an inner walk at most once, because a value is only ever reached by the walk that starts at the head of its own sequence, and there is exactly one such head. Total inner-loop steps across all iterations ≤ n. Say it out loud — "each element is touched at most twice: once by the outer loop, once by the walk over its own sequence, so it's O(n) total" — because an interviewer who sees a nested loop and doesn't hear that sentence will assume you wrote an O(n²) solution by accident. This is the same amortized-counting move as the monotonic stack in tutorial 04: count the work per element, not the loops on the page.

Solution (JS) — O(n) time · O(n) space 
var longestConsecutive = function(nums) {
  // A Set gives O(1) membership — which is the ONLY question we ever need to ask.
  // It also deduplicates for free, so a repeated value can't inflate a run (Example 2).
  const values = new Set(nums);

  let longest = 0;   // 0 is also the correct answer for an empty input (Example 3)

  for (const x of values) {   // iterate the SET, not nums — duplicates are already gone

    // ── THE WHOLE ALGORITHM IS THIS GUARD ──
    // Only walk upward from the HEAD of a sequence. x is a head iff x-1 is absent.
    // Without this line the code is O(n^2); with it, every sequence is walked exactly once.
    if (values.has(x - 1)) continue;   // x sits in the MIDDLE of a run — someone else will cover it

    // x is a head. Walk up as far as the set lets us.
    let current = x;
    let length = 1;
    while (values.has(current + 1)) {
      current++;
      length++;
    }

    longest = Math.max(longest, length);
  }

  return longest;
};

Trace on Example 1 [100,4,200,1,3,2]:

  • Set = {100, 4, 200, 1, 3, 2}.
  • x = 100: is 99 in the set? No → 100 is a head. Walk: is 101 present? No. Length 1. longest = 1.
  • x = 4: is 3 in the set? Yes → 4 is mid-sequence → continue. No walk happens. This is the guard doing its job.
  • x = 200: 199 absent → head. No 201. Length 1.
  • x = 1: is 0 in the set? No → head. Walk: 2 ✓ → 3 ✓ → 4 ✓ → 5 ✗. Length 4. longest = 4.
  • x = 3: 2 present → skip. x = 2: 1 present → skip.
  • Return 4 ✓. Count the inner-loop steps across the whole run: 0 + 0 + 0 + 3 + 0 + 0 = 3. Not n × n — bounded by n, exactly as the amortized argument promises.

Complexity: O(n) time. Building the set is O(n). The outer loop runs over ≤ n distinct values, doing O(1) work each. The inner while looks like a second factor of n but isn't: a value is only stepped over by the walk that begins at the head of its own sequence, and each value belongs to exactly one sequence with exactly one head — so the total number of inner steps across the entire run is at most n. Each element is therefore touched O(1) times amortized. O(n) space for the set. Beat this against the O(n log n) sort explicitly: "sorting gives me a full ordering, but I only ever need to ask 'is x+1 present?' — that's a membership query, so I can pay O(n) space and drop the log factor."

Follow-ups you should expect:

  • "Prove it's O(n) — there's a nested loop right there." This will be asked; it's the whole reason the problem is a medium. The answer is the amortized argument above: the guard ensures each sequence is walked exactly once, from its head, so total inner iterations ≤ n. Have the sentence ready verbatim.
  • "What if you remove the values.has(x - 1) guard?" Still correct, but degrades to O(n²) on a fully consecutive input like [1..n]. Being able to name the exact input that breaks it is a strong signal.
  • "Do it with union-find." You can — union each x with x+1 when both exist, and the answer is the largest component size. It's O(n·α(n)) and it's a legitimate answer, but it's more machinery for no gain. Mention it, then say you'd ship the set version because it's simpler to review (tutorial 10 covers union-find).
  • "Return the sequence itself, not just its length." Track the head value alongside the max length, then reconstruct at the end. Two extra lines; don't rewrite.
  • "Now the numbers arrive as a stream." That's genuinely harder — you'd maintain a map from boundary values to interval lengths and merge intervals as new values arrive (LC 128's streaming cousin). Say it's an interval-merge problem and move on; nobody expects the code here.
This is the problem to reach for when an interviewer asks "tell me about a time you optimized something." The move it demonstrates is the one that generalizes: you were about to buy a total ordering (a sort) when all you actually needed was a single yes/no question (membership). Sorting is O(n log n) because it answers every comparison question about the data; a hash set is O(1) because it answers exactly one. Overpaying for generality you don't need is the most common performance bug in real systems, not just in interviews — and phrasing it that way, at a bank, in a room where someone maintains a service that sorts a million rows to find one, will land harder than any code you write.

Interview Q&A

Why is a hash map lookup O(1)? And when is it not?

A hash map stores entries in an array of buckets. To find a key, it runs the key through a hash function to get a bucket index, then jumps straight there — one arithmetic step plus one array index, independent of how many entries exist. That's the O(1).

But it is O(1) average, not O(1) worst case, and the honest engineer says "average":

  • Collisions. Two keys can hash to the same bucket. With a bad hash function (or adversarial input), every key lands in one bucket and lookup degrades to a linear scan — O(n). Real engines mitigate this; you can't assume it away in a complexity claim.
  • Resizing. When the map gets too full it reallocates and rehashes everything — O(n) for that one insert. Amortized over all inserts it's still O(1), the same amortization argument as Array.push.
  • Hashing the key isn't free. Hashing a number is O(1); hashing a length-k string reads all k characters, so it's O(k). This matters in Group Anagrams, where the complexity is O(n·k), not O(n).

Saying "O(1) average, assuming a decent hash and no adversarial keys" costs you four seconds and marks you as someone who's read past the tutorial layer.

Map vs plain object in JavaScript — which do you use in an interview, and why?

Default to Map. Four concrete reasons, and you should be able to give at least two on demand:

  • Key coercion. Object keys are always strings. o[1] and o["1"] are the same slot — so a frequency map over mixed types silently merges buckets. Map uses SameValueZero and keeps 1 and "1" distinct. It can also key by object identity, which an object literally cannot do (it would stringify to "[object Object]" — every object becoming one key).
  • Prototype pollution. 'toString' in {} is true. If your keys come from the input (words, characters, user data), an innocuous if (map[key]) can be true for a key you never inserted. Map has no prototype chain on its data.
  • Order. Map preserves insertion order for every key type. Objects only guarantee it for string keys, and integer-like keys are hoisted to the front in ascending numeric order — a real surprise if you're iterating.
  • Ergonomics and perf. .size is O(1) (vs Object.keys(o).length, which is O(n) and allocates). Map is directly iterable. And engines don't deopt a Map into dictionary mode on heavy add/delete the way they can with an object.

The legitimate exceptions: a plain object when the keys are a small fixed set of known strings (Roman numerals, bracket pairs) — clearer to read, no downside; and a fixed-size array when the keys are bounded small integers (26 letters, ASCII), which turns O(k) space into O(1) and skips hashing entirely. Choosing deliberately and saying why is the point.

When do you use a Set instead of a Map?

When you need membership only and never a payload. "Have I seen this?" → Set. "How many times / at what index / grouped with what?" → Map. Contains Duplicate and Longest Consecutive Sequence are Set problems; Two Sum (needs the index), Valid Anagram (needs the count) and Group Anagrams (needs the list) are Map problems.

It's a small choice but it's a visible one: reaching for a Map and then never using the values tells the interviewer you're pattern-matching rather than designing. And the reverse mistake is worse — using a Set where you needed counts is exactly the bug that breaks Subarray Sum Equals K, because two different start indices can produce the same prefix sum and each one is a distinct subarray.

The interviewer says "reduce the space complexity." What are your options?

There is a standard menu, and knowing it means you're never stuck:

  • Sort first. Sorting often replaces a hash map entirely — duplicates become adjacent, pairs become a two-pointer sweep. Cost: O(n log n) time, but O(1) extra space if the sort is in-place. This is the trade behind Contains Duplicate and Two Sum II.
  • Two pointers. If the data is (or can be) sorted, converging pointers answer pair/triple questions in O(1) space. That's the whole of tutorial 02.
  • Write into the output array. The output doesn't count as auxiliary space — Product Except Self exploits this to get "O(1) extra" while returning an n-element array.
  • A counting invariant. Boyer–Moore replaces an entire frequency map with two scalars by exploiting a mathematical property of the answer. Rare, but devastating when it applies.
  • Bit manipulation. XOR-based tricks (Single Number, Missing Number) replace a set with one integer. Tutorial 14.
  • Bounded-alphabet arrays. 26 slots instead of a hash map ⇒ genuinely O(1) space.

The meta-move: ask which resource is actually constrained before optimizing. "Is memory the bottleneck here, or latency?" is a legitimate question, and at an enterprise shop it's the question the real system forced someone to answer last quarter.

What do the constraints tell you before you've written a line of code?

They are the interviewer telling you the intended complexity, and reading them is free signal. Rough calibration (assume ~10⁸ simple operations per second):

  • n ≤ 10–20 → exponential is fine. Backtracking, subsets, permutations (2ⁿ or n!).
  • n ≤ 500–1000 → O(n²) or even O(n³) passes. A nested loop is expected, so don't burn twenty minutes hunting for a linear solution that doesn't exist.
  • n ≤ 10⁵–10⁶ → you need O(n) or O(n log n). This is the range where a hash map or a sort is being asked for. Every problem in this tutorial lives here.
  • n ≥ 10⁸ → O(log n) or O(1). Binary search, or a closed-form formula.

The value range matters too: Longest Consecutive's −10⁹ ≤ nums[i] ≤ 10⁹ is what forbids an index-by-value array and forces a hash set. Saying "n is 10⁵, so O(n²) is 10¹⁰ operations and will TLE — I need linear or n log n" in your first thirty seconds tells the interviewer you've done this before.

Prefix sums vs sliding window — how do you tell which one the problem wants?

One question decides it: can the array contain negative numbers?

  • All non-negative → a sliding window works, because growing the window monotonically increases the sum, so there's a valid rule for when to shrink. Window gives you O(1) space.
  • Negatives possible → the window is dead. Adding an element can decrease the sum, so "the sum is too big, shrink from the left" is no longer sound. You need prefix sums + a hash map, at O(n) space. This is exactly why Subarray Sum Equals K (which allows negatives) is a hash problem and Minimum Size Subarray Sum (positives only) is a window problem.

Second discriminator: "count all subarrays" vs "find the best subarray." Counting almost always wants the prefix-sum map, because the map can report how many valid left boundaries exist in O(1). Finding a single best one is more often a window or a Kadane-style scan.

Say the decision out loud — "the array has negatives, so I can't slide a window; I'll use prefix sums with a hash map" — because interviewers set this trap deliberately.

You wrote an O(n) solution but it has a nested loop. How do you defend it?

With an amortized counting argument: stop counting loops and start counting work per element. The template sentence is "each element is processed at most once by the inner loop, so the total inner work across the entire run is bounded by n — not multiplied by it."

Three places in your prep where you'll need exactly this:

  • Longest Consecutive Sequence — the x-1 guard means each sequence is walked once, from its head. Total walk steps ≤ n.
  • Top K Frequent (bucket sort) — the bucket walk looks nested, but the inner loop's total pushes across all buckets is bounded by the number of distinct values.
  • Monotonic stacks (tutorial 04) — each index is pushed once and popped at most once, so ≤ 2n stack operations total.

An interviewer who sees a nested loop and doesn't hear this argument will write "claimed O(n), actually O(n²)" on the feedback form. Volunteer it; never wait to be challenged.

Connect hashing to what you already do as a React engineer.

Three genuine connections — use them, they're not stretches:

  • Normalized state (byId). Every Redux/RTK guide tells you to store entities as { [id]: entity } instead of an array. Why? Because items.find(i => i.id === id) is O(n) on every render and byId[id] is O(1). That is Two Sum's insight, in production, and you've been shipping it for years.
  • useMemo / memoization. Precompute an expensive derivation once, keep it in memory, read it in O(1) on the hot path. Every problem in this tutorial is the same trade with the clock running.
  • The key prop. React's reconciler builds a keyed map of the previous children so it can match old and new nodes in O(n) instead of O(n²). That's canonicalization plus a hash map — the exact Group Anagrams move. It's also why a missing key degrades diffing, and why an index-as-key is a bug: the key isn't stable, so the map matches the wrong things.

Do not force it further ("prefix sums are like the virtual DOM") — a strained analogy costs more than silence. But these three are real, and a 7-YOE candidate who grounds an algorithm in a system they've actually shipped sounds unmistakably different from one reciting a textbook.

You're 10 minutes in and stuck. What's the recovery procedure?

Four steps, in this order, out loud:

  1. State the brute force and its complexity. "The naive approach is two nested loops, O(n²)." You now have a correct algorithm on the board, and at the enterprise tier a working brute force with a clear optimization plan often outscores a half-finished clever solution.
  2. Interrogate the inner loop. Ask literally: "what is my inner loop doing?" If the answer is "searching for a value" or "recomputing a sum I already computed," you have your optimization — a hash map, or a prefix aggregate. That question resolves the majority of easy/medium array problems.
  3. Say the trade. "I'll pre-index that search in a hash map — O(n) space to make the lookup O(1) — which makes the whole thing one pass."
  4. Write it, narrating. Say what each line is for as you type. If you run out of time mid-solution, you have still demonstrated recognition, derivation, and a correct complexity analysis — which is most of the rubric.

Silence is the only unrecoverable failure. An interviewer cannot give partial credit for thinking they can't hear.
