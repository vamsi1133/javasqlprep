07 — Trees & BST

DFS/BFS traversal templates, subtree recursion, and BST ordering invariants — the most-asked data structure after arrays.

Why interviewers ask this

Trees are the single most reliable topic at enterprise-tier loops — banks, fintech, healthcare, and large non-FAANG product companies (Optum, JPMC, Goldman Sachs and their peers). Ask around and the same titles keep surfacing: Invert Binary Tree, Maximum Depth, Level Order Traversal, Validate BST — and Goldman interviewers in particular are known to reach for Zigzag Level Order (LC 103). The reason is practical: tree questions are quick to state, need no long setup, and give the interviewer a crisp read on the one skill arrays can't test — recursion. Can you define a base case, trust the recursive call to handle the subtree, and combine child answers without hand-simulating the whole call stack? At 7 YOE you're expected to write a clean five-line recursive solution in two minutes and then spend the saved time on the follow-ups: complexity on a skewed tree, converting to iteration, and the BFS variant.

For a frontend candidate, trees are also the topic where your day job genuinely helps — say so. The DOM is a tree; React's virtual DOM diff walks two trees the way Same Tree (LC 100) does; a file-explorer or nested-comments component is level-order rendering; and every "walk this config/AST/menu structure" utility you've written was a DFS. Interviewers grade narration as much as code, and "this is the same shape as diffing two component trees" is cheap, honest senior signal. This tutorial gives you three templates (recursive DFS, BFS with a level loop, iterative inorder) and then makes you spend them across seventeen problems — by the end, every tree question you meet should decompose into "which template, plus what per-node work?"

1. Recognizing the pattern

The node, and how LeetCode draws trees

Every problem on this page uses the same node shape. LeetCode predefines it; in an on-site whiteboard round you may be asked to define it yourself, so know it cold: 

function TreeNode(val, left, right) {
  this.val   = (val   === undefined ? 0    : val);
  this.left  = (left  === undefined ? null : left);
  this.right = (right === undefined ? null : right);
}

Problem statements describe trees in array form: a level-order listing where null marks a missing child. So [3,9,20,null,null,15,7] means: 

        3
       / \
      9    20
          /  \
        15    7

Index 0 is the root, then each level left-to-right, with null holding the place of an absent child (and absent children of a null are simply not listed). You'll never parse this format in code — LeetCode builds the tree for you — but you must read it fluently to trace examples.

DFS or BFS? The decision in one question

Ask: does the answer care about levels?

ChooseWhen the problem says…MechanismExamples here
DFS (recursion) "depth", "path", "subtree", "same/mirror", "validate", "sum along a route" — anything where a node's answer is built from its children's answers. Recursive function; the call stack is your traversal state. Space O(h). Invert, Max Depth, Diameter, Validate BST, Max Path Sum
BFS (queue) "level by level", "level averages", "zigzag", "right side view", "minimum depth" (first leaf found wins), "connect nodes at same level". Queue + a loop that freezes the current level size. Space O(w), the widest level. Level Order, Zigzag, Right Side View

Almost every DFS problem can be done with BFS and vice versa — the table is about which is natural. Right Side View is the classic "either works, defend your pick" question, and interviewers like hearing the tradeoff: DFS costs stack depth O(h), BFS costs queue width O(w).

The "answer at every node" idiom

A large family of tree problems — Diameter, Balanced, Count Good Nodes, Max Path Sum — share one shape that trips up even strong candidates: the value your recursion returns is not the value the problem asks for. Diameter asks for the longest path anywhere in the tree, but the useful thing a subtree can report upward is its height. The idiom: run a postorder DFS that returns the "building block" (height, contribution, max-so-far), and update a variable in the enclosing scope with the real answer at every node as you pass through. One traversal, two jobs. Once you've internalized this from Diameter (LC 543), the hard version — Max Path Sum (LC 124) — is the same code with different arithmetic.

The BST invariant — the whole subtree, not just the child

A binary search tree promises: every node in the left subtree < node < every node in the right subtree — recursively, all the way down. Two consequences do all the work in interviews:

  • Inorder traversal visits values in sorted ascending order. This single fact solves Kth Smallest (LC 230), gives the elegant Validate BST alternative, and answers "find the two swapped nodes" follow-ups.
  • One comparison at a node discards an entire subtree. Searching, inserting, and LCA (LC 235) all run in O(h) because at each node you know which side the target(s) must be on.
The invariant constrains the entire subtree, not just the immediate children. The tree [5,1,7,null,null,3,8] has every parent bigger than its left child and smaller than its right child — and it is not a valid BST, because 3 sits in 5's right subtree while being smaller than 5. "Check each node against its children" is the most common wrong answer to LC 98, and interviewers plant this exact trap. The fix — pass down (min, max) bounds — is problem 13 below.
Trees are where you should mention recursion depth limits unprompted. LeetCode constraints usually cap nodes at 104; a fully skewed tree (every node has one child — effectively a linked list) makes recursion depth equal to n, which is fine at 104 but worth flagging: "if the tree could be huge and skewed I'd convert to an explicit stack to avoid blowing the call stack." That one sentence, plus writing O(h) instead of O(log n) for space — and noting h ranges from log n (balanced) to n (skewed) — is most of the senior-level complexity discussion for this whole topic.

2. The templates

2a. Recursive DFS — the shape of every tree recursion

Three lines, always the same three lines. Write them before you know what the problem wants, then fill in the blanks: 

var dfs = function(node) {
  // 1. BASE CASE — the empty subtree. What is the answer for "nothing"?
  if (node === null) return IDENTITY;

  // 2. RECURSE — trust the calls. Do NOT trace them in your head.
  const left  = dfs(node.left);
  const right = dfs(node.right);

  // 3. COMBINE — build this node's answer from its children's answers.
  return combine(node.val, left, right);
};

Line by line, and why each line is where the marks are:

  • The base case is the identity element of the combine step. Summing? Return 0. Measuring height? Return 0. Checking "are all nodes valid"? Return true — an empty tree is vacuously valid. Ask "what value makes combine a no-op?" and the base case writes itself. Getting this wrong (returning 1 for height, say) is the single biggest source of off-by-one bugs in tree problems.
  • The recursion is a leap of faith, and that's the skill. Assume dfs(node.left) already returns the correct answer for the entire left subtree. You do not verify it; you use it. Candidates who try to mentally simulate the call stack stall out and run out of time. This is the same trust you already extend to a recursive React render — you don't simulate the whole child tree to reason about one component.
  • Combine is where the problem actually lives. Max Depth combines with 1 + max(l, r). Invert combines by swapping. Balanced combines by checking |l − r| ≤ 1. Everything else on this page is boilerplate.
  • Order matters when you mutate. Preorder (work before recursing), inorder (between), postorder (after). If your combine step needs child answers — and it usually does — you are postorder by definition.

Why this is O(n) time and O(h) space: the recursion visits each node exactly once and does O(1) work per node (a comparison, an addition, a swap), so time is O(n) — n nodes, not n², because nothing re-walks a subtree it already walked. Space is the depth of the call stack, which is the height h of the tree: O(log n) if balanced, O(n) if skewed. Always write O(h) and then say the range out loud. The exception to watch for is a recursion that calls another recursion on the same subtree (Subtree of Another Tree, and the naive Balanced) — that's how you accidentally get O(n²), and spotting it is a graded moment.

2b. BFS — the frozen-level-size loop

One detail turns a flat traversal into a level-aware one: capture queue.length into a variable before the inner loop. That snapshot is exactly the current level; anything you enqueue inside the loop belongs to the next level and must not be consumed by this one. 

var bfs = function(root) {
  const res = [];
  if (!root) return res;              // guard: empty tree ⇒ empty answer

  let queue = [root];

  while (queue.length) {
    const size = queue.length;         // FREEZE the level boundary — the whole trick
    const level = [];
    const next = [];                   // children collected here, consumed next round

    for (let i = 0; i < size; i++) {
      const node = queue[i];           // index instead of shift() — see the Note below
      level.push(node.val);            // ← per-node work goes here
      if (node.left)  next.push(node.left);
      if (node.right) next.push(node.right);
    }

    res.push(level);                   // ← per-LEVEL work goes here
    queue = next;                      // swap in the next level, O(1)
  }
  return res;
};
In JavaScript, Array.prototype.shift() is O(n) — every remaining element re-indexes — so the textbook while (q.length) { const node = q.shift(); … } silently degrades BFS to O(n²) on a wide tree. Two clean fixes: iterate the array by index and swap in a fresh next array each level (above), or keep one array plus a moving head pointer. Say this out loud — it's a JS-specific insight most candidates don't have, and at 7 YOE it reads as knowing your runtime, not just your algorithms.

2c. Iterative inorder — the one iterative traversal worth memorizing

For BSTs, inorder emits sorted order, and the iterative form lets you stop early — which is the whole point of Kth Smallest. Push the entire left spine, pop (that's the next-smallest node), then pivot right and repeat. 

var inorder = function(root) {
  const stack = [];
  let curr = root;
  const out = [];

  while (curr || stack.length) {
    while (curr) {          // dive left, stacking the path (the "left spine")
      stack.push(curr);
      curr = curr.left;
    }
    curr = stack.pop();      // nothing left of curr ⇒ curr is the next value in sorted order
    out.push(curr.val);      // ← VISIT. In Kth Smallest, you return here when k hits 0.
    curr = curr.right;       // its right subtree is the next-smallest region
  }
  return out;
};

This is just the recursive version with the call stack made explicit — the stack array holds exactly the frames the runtime would have held. That equivalence is the answer to "can you do it iteratively?", which you will be asked.

3. Recursion warm-ups

Seven problems, one template. Do them in order — each one adds exactly one new idea to the previous, and by number 7 you'll have every DFS variation an enterprise loop can throw at you. Time-box each of the first four to ten minutes; they are literally screen-warm-up questions and you're expected to be fluent, not clever.

1. Invert Binary Tree (LC #226) EASY ★

Given the root of a binary tree, produce its mirror image: at every node, the left and right subtrees swap places — recursively, all the way down. Return the root of the inverted tree. (You may mutate the tree in place; LeetCode accepts the same root back.) This is the problem Max Howell of Homebrew fame famously failed at Google, which made it permanently famous — expect it as a warm-up specifically because everyone has heard of it.

Example 1: Input: root = [4,2,7,1,3,6,9] → Output: [4,7,2,9,6,3,1] — every level is reversed because every node swapped its two children.
Example 2: Input: root = [2,1,3] → Output: [2,3,1] — one swap at the root.
Example 3: Input: root = [] → Output: [] — the empty tree is its own mirror; your code must not crash on null.
Constraints: 0 ≤ nodes ≤ 100; −100 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. There's no brute force to beat here — the interest is entirely in whether you reach for recursion cleanly. Define what "inverted" means for one node: its left child becomes its right child, and both of those children are themselves inverted. That sentence is the algorithm: recurse into both subtrees, then swap the pointers. The base case is the empty subtree — inverting nothing gives you nothing, so return null. Note that the swap and the recursion commute: you can swap first and then recurse into the (now moved) children, or recurse first and then swap. Both are correct because each node's swap is independent of its descendants' swaps. Say that out loud — it shows you understand why it works rather than that you memorized a shape.

Solution (JS) — O(n) time · O(h) space 
var invertTree = function(root) {
  // Base case: mirroring an empty subtree yields an empty subtree.
  if (root === null) return null;

  // Trust the recursion: each call returns a FULLY inverted subtree.
  const left  = invertTree(root.left);
  const right = invertTree(root.right);

  // Combine: hang them on the opposite sides. Destructuring avoids a temp var.
  root.left  = right;
  root.right = left;

  return root;   // same node object, children now mirrored
};

// Iterative variant (BFS) — say this exists; you'll be asked.
var invertTreeIter = function(root) {
  if (!root) return null;
  const stack = [root];
  while (stack.length) {
    const node = stack.pop();
    [node.left, node.right] = [node.right, node.left];  // swap in place
    if (node.left)  stack.push(node.left);
    if (node.right) stack.push(node.right);
  }
  return root;
};

Trace on [4,2,7,1,3,6,9]: at node 1, 3, 6, 9 (leaves) both recursive calls return null, swap is a no-op. At node 2: left = invert(1) = 1, right = invert(3) = 3, then 2.left = 3, 2.right = 1 → subtree [2,3,1]. Symmetrically node 7 becomes [7,9,6]. At the root: 4.left = 7's subtree, 4.right = 2's subtree → [4,7,2,9,6,3,1]. ✓

Complexity: O(n) time — every node is visited exactly once and does a constant-time pointer swap. O(h) space for the call stack: O(log n) on a balanced tree, O(n) on a fully skewed one. The iterative version trades the call stack for an explicit stack — same asymptotic space, but immune to stack overflow.

Follow-ups you should expect:

  • "Do it without recursion." The BFS/DFS-with-a-stack version above. This is the #1 follow-up on every easy tree problem, so have it ready for all of them.
  • "Don't mutate the input — return a new tree." Same recursion, but build fresh nodes: return new TreeNode(root.val, invert(root.right), invert(root.left)). Note the swap now happens in the argument order. This is the immutable/persistent version — worth naming the connection to how React treats props.
  • "What if the tree had k children per node?" Reverse the children array, recurse into each. Generalizes trivially.

2. Maximum Depth of Binary Tree (LC #104) EASY ★

Return the maximum depth of a binary tree: the number of nodes along the longest path from the root down to the farthest leaf. Read that definition carefully — it counts nodes, not edges, so a single-node tree has depth 1 and the empty tree has depth 0. (Some problems define depth in edges; always confirm which one the interviewer means.)

Example 1: Input: root = [3,9,20,null,null,15,7] → Output: 3 — the longest root-to-leaf path is 3 → 20 → 15, which is three nodes.
Example 2: Input: root = [1,null,2] → Output: 2 — a right-skewed chain of two.
Constraints: 0 ≤ nodes ≤ 10⁴; −100 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. The recursive definition falls straight out of the English: the depth of a tree is one (for the root itself) plus the depth of its deeper subtree. Empty tree → 0, which is exactly the identity that makes 1 + max(0, 0) = 1 come out right for a leaf. That's the whole solution, and it should take you under a minute. The reason this problem is still worth your time is the follow-up: "now do minimum depth." Candidates reflexively write 1 + Math.min(l, r) and get it wrong, because a null child returns 0 and a node with only one child would then report depth 1 — but that node isn't a leaf. Minimum depth needs an explicit "if one child is missing, don't count it" branch (or a BFS that stops at the first leaf). Being able to explain why max is safe and min is not is the actual senior signal on this problem.

Solution (JS) — O(n) time · O(h) space 
var maxDepth = function(root) {
  // Identity for max(): an empty subtree contributes depth 0.
  if (root === null) return 0;

  // The deeper of the two subtrees decides the answer; +1 counts THIS node.
  return 1 + Math.max(maxDepth(root.left), maxDepth(root.right));
};

// ---- The follow-up they actually ask: MINIMUM depth (LC 111). ----
// Naive `1 + Math.min(l, r)` is WRONG: for [1,2] (left child only) it returns
// 1 + min(1, 0) = 1, but node 1 is not a leaf — the answer is 2.
var minDepth = function(root) {
  if (root === null) return 0;
  const l = minDepth(root.left);
  const r = minDepth(root.right);
  // If either side is missing, the shortest path MUST go through the other side.
  if (l === 0 || r === 0) return 1 + l + r;   // one of them is 0, so this is 1 + the real one
  return 1 + Math.min(l, r);
};

Trace on [3,9,20,null,null,15,7]: maxDepth(9) = 1 + max(0,0) = 1. maxDepth(15) = 1, maxDepth(7) = 1. maxDepth(20) = 1 + max(1,1) = 2. maxDepth(3) = 1 + max(1, 2) = 3. ✓

Complexity: O(n) time — each node contributes one comparison and one addition; no node is revisited. O(h) space for the recursion stack, worst case O(n) on a skewed tree. Note there is no way to beat O(n): you cannot know the depth without looking at every node, so the lower bound is the runtime.

Follow-ups you should expect:

  • "Minimum depth?" The trap above. If you'd rather sidestep it entirely, BFS and return the level of the first node you meet with no children — that's also strictly faster in practice, since it short-circuits instead of exploring the whole tree.
  • "Iteratively?" BFS with the frozen-level loop, counting levels; or DFS with an explicit stack of [node, depth] pairs.
  • "Now count the nodes / the leaves / the nodes at depth k." Same recursion, different combine: 1 + l + r, or l + r with a leaf base case. Notice you're now just turning the one knob.
maxDepth and "height" are the same number computed the same way — the two words differ only in perspective (depth measures down from the root, height measures up from the leaves), and for the root they coincide. Half the problems on this page are secretly this five-line function with something extra bolted on: Balanced compares the two heights, Diameter sums them. Learn it once.

3. Same Tree (LC #100) EASY

Given the roots of two binary trees p and q, decide whether they are identical: the same structure and the same values at every corresponding position. Two empty trees are identical. A tree and an empty tree are not.

Example 1: Input: p = [1,2,3], q = [1,2,3] → Output: true.
Example 2: Input: p = [1,2], q = [1,null,2] → Output: false — same values, same node count, different shape: 2 is a left child in one and a right child in the other. Structure counts.
Example 3: Input: p = [1,2,1], q = [1,1,2] → Output: false — same shape, same multiset of values, but position-by-position they differ.
Constraints: 0 ≤ nodes ≤ 100; −10⁴ ≤ Node.val ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. This is the first problem where the recursion takes two nodes instead of one — the mental shift you need for Symmetric, Subtree, and LCA later. You're not walking a tree; you're walking two trees in lockstep. Every recursive call receives one node from each. Enumerate the cases for a pair: both null → identical (true); exactly one null → shapes differ (false); values differ → false; otherwise recurse on (p.left, q.left) and (p.right, q.right) and require both to be true. There's no clever optimization to find — the value here is stating those four cases crisply and in the right order, because checking p.val before the null checks is an instant null-dereference crash. This is literally what React's reconciler does when it diffs two element trees: same type and same position → keep and recurse; different → bail out. Say that; it's free, honest senior signal.

Solution (JS) — O(n) time · O(h) space 
var isSameTree = function(p, q) {
  // 1. Both empty — identical. This also terminates every branch.
  if (p === null && q === null) return true;

  // 2. Exactly one empty — structures differ. MUST come before touching .val.
  if (p === null || q === null) return false;

  // 3. Both exist but disagree here — done, no need to look deeper.
  if (p.val !== q.val) return false;

  // 4. This node matches — the trees are the same iff BOTH subtree pairs match.
  //    && short-circuits, so a mismatch on the left skips the right entirely.
  return isSameTree(p.left, q.left) && isSameTree(p.right, q.right);
};

Trace on Example 2 (p = [1,2], q = [1,null,2]): roots both exist, 1 === 1 ✓. Recurse left: p.left = node 2, q.left = null → case 2 fires → false. The && short-circuits and the right pair is never examined. Result: false. ✓

Complexity: O(n) time where n is the size of the smaller tree — you stop the moment the structures diverge, so it's O(min(n, m)) strictly, and O(n) in the matching case. O(h) space for the call stack. The short-circuit means a mismatch at the root is O(1), which is worth mentioning.

Follow-ups you should expect:

  • "What if the trees are equal but the children may be swapped at any node?" That's "flip equivalent" trees (LC 951): at each node, either both pairs match straight, or both match crossed — (a.left~b.left && a.right~b.right) || (a.left~b.right && a.right~b.left).
  • "Iteratively?" One stack (or queue) holding pairs [p, q]; pop a pair, run the same four cases, push the two child pairs.
  • "Now check if q is a subtree of p." That's problem 7 below — and it uses this function as its inner check.

4. Symmetric Tree (LC #101) EASY

Given the root of a binary tree, decide whether the tree is a mirror image of itself — i.e. whether it's symmetric about a vertical line drawn through the root. Both structure and values must mirror.

Example 1: Input: root = [1,2,2,3,4,4,3] → Output: true — the left subtree [2,3,4] is the mirror of the right subtree [2,4,3].
Example 2: Input: root = [1,2,2,null,3,null,3] → Output: false — both 2s have a right child named 3, so the values agree but the positions don't mirror; a true mirror would need the left 2's right child to face the right 2's left child.
Constraints: 1 ≤ nodes ≤ 1000; −100 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. Candidates get stuck because they try to write isSymmetric(node) — a single-node recursion — and there's simply nothing useful to return. Symmetry isn't a property of one node; it's a property of a pair of nodes facing each other across the mirror. So change the function signature: isMirror(a, b) asks "are these two subtrees mirror images of each other?" Now the recursion writes itself, and it is Same Tree with one line changed: instead of comparing a.left with b.left, you compare a.left with b.right and a.right with b.left — the outer pair and the inner pair. Then isSymmetric(root) is just isMirror(root.left, root.right). "Change the signature so the state you need is in the arguments" is a habit that pays off across this whole page — it's the same move that fixes Validate BST.

Solution (JS) — O(n) time · O(h) space 
var isSymmetric = function(root) {
  if (!root) return true;
  // The tree is symmetric iff its two subtrees mirror EACH OTHER.
  return isMirror(root.left, root.right);
};

// Are subtrees a and b mirror images? Note the two nodes face each other
// across the axis — a's LEFT lines up with b's RIGHT.
function isMirror(a, b) {
  if (a === null && b === null) return true;   // both absent — mirrors
  if (a === null || b === null) return false;  // one absent — asymmetric shape
  if (a.val !== b.val) return false;

  return isMirror(a.left, b.right)    // OUTER pair: a's far side vs b's far side
      && isMirror(a.right, b.left);   // INNER pair: the two nodes nearest the axis
}

Trace on [1,2,2,3,4,4,3]: isMirror(2ᴸ, 2ᴿ): values match. Outer pair: isMirror(3, 3) → both leaves, match ✓. Inner pair: isMirror(4, 4) → match ✓. Both true → true. Now Example 2, [1,2,2,null,3,null,3]: isMirror(2ᴸ, 2ᴿ), values match. Outer pair is isMirror(2ᴸ.left = null, 2ᴿ.right = 3) → one null, one not → false. ✓ The crossed comparison is exactly what catches this; a naive isSameTree(root.left, root.right) would wrongly return true.

Complexity: O(n) time — each node participates in exactly one pair comparison. O(h) space for the call stack.

Follow-ups you should expect:

  • "Iteratively?" Push pairs onto a queue: start with [root.left, root.right], and for each popped pair enqueue [a.left, b.right] and [a.right, b.left]. Same crossed pattern, no recursion.
  • "Could you invert the left subtree and compare with Same Tree?" Yes — isSameTree(invert(root.left), root.right) is correct and cute, but it mutates the input and costs an extra pass. Mention it, don't lead with it.
  • "Is the tree symmetric in structure only, ignoring values?" Drop the value check. Trivial, but they're testing whether you actually understand what each line enforces.
Do not write isSymmetric(root.left) && isSymmetric(root.right). It's the natural-looking recursion and it's wrong: it checks that each half is internally symmetric, not that the halves mirror one another. The tree [1,2,2,3,null,null,4] has two internally-fine subtrees and is not symmetric. When a property relates two structures, the recursion must take both of them as arguments — that's the lesson, and it's the same lesson in Subtree and in LCA.

5. Diameter of Binary Tree (LC #543) EASY ★

The diameter of a binary tree is the length of the longest path between any two nodes in the tree — the path need not pass through the root. Length is measured in edges, not nodes. Given the root, return the diameter. It's tagged Easy; it is not, and interviewers know it. This is the highest-value problem in this section.

Example 1: Input: root = [1,2,3,4,5] → Output: 3 — the longest path is 4 → 2 → 5, or equivalently 4 → 2 → 1 → 3, which spans 3 edges. Note it goes through node 2, not through the root, in the first case.
Example 2: Input: root = [1,2] → Output: 1 — one edge.
Example 3: Input: a tree where the root's left subtree is a long skewed chain of 10 nodes and the right child is a single leaf → the diameter lives entirely inside the left subtree and never touches the root. This is the case that kills the naive answer.
Constraints: 1 ≤ nodes ≤ 10⁴; −100 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. Start with the obvious observation: any path in a tree has a single highest node — its "peak" — and the path goes down into that peak's left subtree and down into its right subtree. So for a fixed peak node, the longest path peaking there is height(left) + height(right) edges. The diameter is therefore max of that quantity over every node, since every path peaks somewhere. That immediately gives an O(n²) brute force: for each node, compute both subtree heights (O(n) each) and take the max — too slow, and wasteful, because computing the height of the root already computes the heights of everything below it.

The collapse to O(n) is the pattern you must own: return one thing, track another. Run a single postorder DFS whose return value is the height — because that's what a parent needs from its child — and, at every node as you pass through, update a variable in the enclosing scope with left + right, the candidate diameter peaking there. One traversal computes every height, and en route it evaluates every possible peak. The reason this trips people up is that the function's return value (1 + max(l, r), a height) is not the answer to the problem (max(l + r), a diameter) — those are two different quantities produced by the same walk. Say it explicitly in the interview: "My recursion returns height, but I record the best diameter in a closure variable as a side effect at each node." Once this clicks, Max Path Sum (problem 16) is the same code with different arithmetic, and Balanced (next) is the same code with a different side condition.

Solution (JS) — O(n) time · O(h) space 
var diameterOfBinaryTree = function(root) {
  let best = 0;  // the ANSWER lives here, not in the return values

  // Returns the HEIGHT of this subtree in EDGES (empty = -1, leaf = 0).
  // Using -1 for empty makes leaf height come out as 1 + max(-1,-1) = 0,
  // which keeps everything in edge units and avoids off-by-one mixing.
  function height(node) {
    if (node === null) return -1;

    const l = height(node.left);
    const r = height(node.right);

    // SIDE EFFECT: the longest path PEAKING at this node runs down both sides.
    // (l + 1) edges to reach the left subtree's deepest node, (r + 1) to the right.
    best = Math.max(best, (l + 1) + (r + 1));

    // RETURN: what a PARENT needs is only how deep I go — one side, the deeper one.
    return 1 + Math.max(l, r);
  }

  height(root);
  return best;
};

Trace on [1,2,3,4,5]: height(4): l = r = −1, best = max(0, 0+0) = 0, returns 0. height(5): same, returns 0. height(2): l = 0, r = 0 → best = max(0, 1 + 1) = 2 (the path 4→2→5), returns 1 + max(0,0) = 1. height(3): returns 0. height(1): l = 1, r = 0 → best = max(2, 2 + 1) = 3 (the path 4→2→1→3), returns 2. Answer 3. ✓ Notice best was updated at node 2 and at node 1 — every node gets its shot at being the peak.

Complexity: O(n) time — one postorder pass, O(1) work per node. Contrast the brute force, which recomputes heights from scratch at every node for O(n²) (O(n·h) precisely, degenerating to O(n²) on a skewed tree). Space O(h) for the call stack. The whole trick is that heights are computed bottom-up once and reused.

Follow-ups you should expect:

  • "Return the path itself, not just its length." Track the peak node alongside best, then walk down the deepest chain on each side from that node. Doable, but say up front it changes the return type and costs an extra descent.
  • "What if edges have weights?" Replace 1 + with weight + in both places. Same shape — and that's exactly the step up to Max Path Sum (LC 124).
  • "Why not just 2 × maxDepth?" Because the diameter needn't pass through the root — Example 3 above is the counterexample. Have that ready; interviewers plant it.
  • "Avoid the closure variable." Return a pair [height, bestSoFar] from each call and combine. Functionally pure, slightly noisier code. Both are accepted; know that the pair version exists.
The closure variable is not a hack, and you should defend it as a design choice: "The recursion returns the value my parent needs; the answer I actually want is a maximum over all nodes, which is naturally an accumulator. Threading it through the return type would mean returning a tuple at every level for no benefit." That framing — what does my caller need vs. what am I accumulating — is what distinguishes a 7-YOE answer from a 2-YOE one, and it generalizes to Balanced, Count Good Nodes, House Robber III, and Max Path Sum.

6. Balanced Binary Tree (LC #110) EASY

A binary tree is height-balanced when, at every node, the heights of its two subtrees differ by at most 1. Given the root, return whether the tree is balanced. The quantifier is the whole problem: it isn't enough for the root to be balanced — every single node must be.

Example 1: Input: root = [3,9,20,null,null,15,7] → Output: true — heights differ by at most 1 everywhere.
Example 2: Input: root = [1,2,2,3,3,null,null,4,4] → Output: false — node 2 (the left child) has a left subtree of height 2 and a right subtree of height 0. The root looks fine in isolation; the violation is deeper.
Example 3: Input: root = [] → Output: true — an empty tree is balanced.
Constraints: 0 ≤ nodes ≤ 5000; −10⁴ ≤ Node.val ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. The direct translation is: for each node, compute height(left) and height(right), check the difference, and recurse. That's correct but it's the same O(n²) trap as Diameter's brute force — height re-walks the entire subtree, and you call it at every node. The fix is the same idiom: one bottom-up pass that computes heights, detecting imbalance on the way up, at the exact moment the two child heights are in hand. The elegant encoding is a sentinel: let the height function return −1 to mean "this subtree is already unbalanced". Then any parent that sees −1 from a child immediately propagates −1 without doing more work, and the top-level answer is simply "did we get −1?". Interviewers like this because it fuses "compute" and "validate" into one traversal — but you should still say out loud that a boolean-plus-height pair would be clearer production code, and that you're using the sentinel because it's compact. Knowing the difference between interview-compact and production-clear is itself senior signal.

Solution (JS) — O(n) time · O(h) space 
var isBalanced = function(root) {
  // Returns the height of the subtree, OR -1 as a sentinel meaning
  // "somewhere below me the balance property is already broken".
  function check(node) {
    if (node === null) return 0;   // empty subtree: height 0, trivially balanced

    const l = check(node.left);
    if (l === -1) return -1;         // short-circuit: no point measuring the right side

    const r = check(node.right);
    if (r === -1) return -1;

    // Both subtrees are individually balanced. Is THIS node balanced?
    if (Math.abs(l - r) > 1) return -1;

    return 1 + Math.max(l, r);   // balanced so far — report my real height upward
  }

  return check(root) !== -1;
};

Trace on Example 2, [1,2,2,3,3,null,null,4,4]: the two 4s are leaves → height 1 each. The left 3 has children 4 and 4 → |1 − 1| = 0 ✓, height 2. The right 3 is a leaf → height 1. The left 2 sees l = 2, r = 1 → |2 − 1| = 1 ✓, height 3. The right 2 is a leaf → height 1. The root sees l = 3, r = 1 → |3 − 1| = 2 > 1 → returns −1 → answer false. ✓ (Notice the imbalance was caught at the root here; on other inputs the sentinel fires deep and propagates straight up untouched.)

Complexity: O(n) time — each node's height is computed exactly once, bottom-up, and the sentinel short-circuits everything above a violation. The naive "call height() at every node" version is O(n·h) → O(n²) on a skewed tree; being able to state why that version is quadratic (heights recomputed from scratch for every ancestor) is the point of this problem. Space O(h).

Follow-ups you should expect:

  • "Show me the O(n²) version and tell me its cost." Expect this — they want the contrast, not the code.
  • "Avoid the magic −1." Return { height, balanced } (or [h, ok]). Same algorithm, self-documenting. Offer it as the version you'd merge.
  • "Why do real trees care?" Because an unbalanced BST degrades to a linked list and its O(log n) operations become O(n) — which is exactly why AVL and red-black trees exist and rebalance on insert. Naming that connection lands well in a bank/fintech loop, where indexes are B-trees.

7. Subtree of Another Tree (LC #572) EASY

Given the roots of two binary trees root and subRoot, return true if there exists a node in root such that the subtree rooted at that node is identical to subRoot. "Subtree of a node" means that node and all of its descendants — you can't match a partial fragment.

Example 1: Input: root = [3,4,5,1,2], subRoot = [4,1,2] → Output: true — the node 4 in root heads a subtree identical to subRoot.
Example 2: Input: root = [3,4,5,1,2,null,null,null,null,0], subRoot = [4,1,2] → Output: false — node 4's subtree now has an extra node 0 hanging under the 2. It contains subRoot's shape but isn't identical to it, and identity is what's required.
Constraints: 1 ≤ root nodes ≤ 2000; 1 ≤ subRoot nodes ≤ 1000; −10⁴ ≤ Node.val ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. Decompose it into two questions you already answered: "is subtree-of" means "is there a node where Same Tree returns true". So walk every node of root, and at each one run the Same Tree check against subRoot. That's the whole solution and it's completely acceptable — but be the candidate who states its complexity honestly and unprompted: O(n × m), because for each of n nodes you may compare up to m nodes. Also nail the base cases, which is where this problem actually bites: if root is null, the answer is false (a non-empty subRoot can't live in nothing) — but note that LeetCode's constraints guarantee subRoot is non-empty, so you don't need to handle "the empty tree is a subtree of everything". Example 2 exists specifically to catch people who wrote a "contains" check instead of an "identical" check: once you find a matching root value, you must compare the entire subtree, extra nodes and all.

Solution (JS) — O(n · m) time · O(h) space 
var isSubtree = function(root, subRoot) {
  // Ran out of tree to search — subRoot (non-empty per constraints) isn't here.
  if (root === null) return false;

  // Does the match start HERE? (full identity check, not a value check)
  if (isSameTree(root, subRoot)) return true;

  // Otherwise the match, if any, is strictly inside one of the children.
  return isSubtree(root.left, subRoot) || isSubtree(root.right, subRoot);
};

// Problem 3, verbatim — reuse it, don't rewrite it.
function isSameTree(p, q) {
  if (p === null && q === null) return true;
  if (p === null || q === null) return false;
  if (p.val !== q.val) return false;
  return isSameTree(p.left, q.left) && isSameTree(p.right, q.right);
}

Trace on Example 2: at root 3, isSameTree(3-subtree, [4,1,2]) → values 3 ≠ 4 → false. Recurse left to node 4: isSameTree compares 4 vs 4 ✓, 1 vs 1 ✓, then 2 vs 2 ✓ — but node 2 in root has a right child 0 while subRoot's 2 has none, so the pair (0, null) hits the "exactly one null" case → false. Recursion continues into 1, 2, 0, 5; none match → false. ✓ This is precisely why the inner check must be identity, not "contains".

Complexity: O(n × m) time in the worst case (n = nodes in root, m = nodes in subRoot) — isSameTree is O(m) and you may call it at all n nodes. In practice it's much faster because isSameTree bails on the first value mismatch. Space O(h_root + h_sub) for the nested call stacks — call it O(n) worst case.

Follow-ups you should expect:

  • "Can you beat O(n·m)?" Yes, and this is the intended senior escalation: serialize both trees to strings with explicit null markers (e.g. "^4,^1,#,#,^2,#,#") and ask whether serialize(subRoot) is a substring of serialize(root). That's O(n + m) with KMP, O(n·m) worst case with naive includes. The null markers and the per-node delimiter are load-bearing — without them, value 12 matches inside 112, and a straight preorder without nulls doesn't uniquely determine a tree.
  • "Merkle hashing?" Hash each subtree bottom-up (hash(node) = H(val, hash(left), hash(right))) and compare root hashes — O(n + m), the same idea a version-control system uses to diff trees. Naming this is a strong, cheap flex.
  • "What if the empty tree counts as a subtree?" Then subRoot === null returns true immediately. Ask which convention they want rather than assuming.
Notice what just happened across problems 3, 4, and 7: three different questions, one function. Same Tree is the engine; Symmetric is Same Tree with the recursive calls crossed; Subtree is Same Tree called at every node. When you meet a new tree problem, ask "which of my existing pieces is this made of?" before you invent anything. Composing known pieces under time pressure is exactly what shipping features looks like, and interviewers read it that way.

8. Binary Tree Level Order Traversal (LC #102) MEDIUM ★

Given the root of a binary tree, return its values grouped by level, top to bottom, and left to right within each level — i.e. an array of arrays, one inner array per depth. The empty tree returns [].

Example 1: Input: root = [3,9,20,null,null,15,7] → Output: [[3],[9,20],[15,7]] — three levels; note 15 and 7 are on the same level despite having different parents… actually they share parent 20, but the grouping is by depth, not by parent, which is the distinction the next example makes.
Example 2: Input: root = [1,2,3,4,null,null,5] → Output: [[1],[2,3],[4,5]] — 4 and 5 land in the same group even though they have different parents (2 and 3). Depth is the only thing that groups.
Example 3: Input: root = [] → Output: [].
Constraints: 0 ≤ nodes ≤ 2000; −1000 ≤ Node.val ≤ 1000.

Solve on LeetCode ↗

Thinking process. A plain queue-based BFS visits nodes in exactly the right order — but it hands you one flat list, and you've lost the level boundaries. Where did they go? They were never recorded. So the entire problem is: how do I know when one level ends? Two answers exist and you should know both. (1) Freeze the queue length at the top of each round: whatever is in the queue right now is the current level, exactly, because the previous round enqueued precisely its children and nothing else. Process that many nodes, and everything you enqueue meanwhile belongs to the next round. (2) Push [node, depth] pairs and bucket by depth. The first is the standard and reads better; the second generalizes to DFS. It's also worth saying that DFS solves this too — carry a depth parameter down and res[depth].push(val), creating the row on first visit. Being able to offer both, and to say why you'd pick BFS here (the answer is inherently level-shaped, so let the traversal match the output), is a stronger answer than either solution alone.

Solution (JS) — O(n) time · O(w) space 
var levelOrder = function(root) {
  const res = [];
  if (!root) return res;               // empty tree ⇒ [], not [[]]

  let queue = [root];

  while (queue.length) {
    const size = queue.length;          // FREEZE: exactly the nodes of the current level
    const level = [];
    const next = [];

    for (let i = 0; i < size; i++) {
      const node = queue[i];            // index, not shift() — shift() is O(n)
      level.push(node.val);

      // Children are the NEXT level. Skipping nulls keeps the queue clean;
      // the alternative (enqueue nulls, filter later) is a common source of bugs.
      if (node.left)  next.push(node.left);
      if (node.right) next.push(node.right);
    }

    res.push(level);                    // per-LEVEL work: emit the finished row
    queue = next;                       // advance one level, O(1)
  }
  return res;
};

// The DFS alternative — same output, different traversal. Worth knowing.
var levelOrderDFS = function(root) {
  const res = [];
  function dfs(node, depth) {
    if (!node) return;
    if (res.length === depth) res.push([]);   // first node ever seen at this depth
    res[depth].push(node.val);
    dfs(node.left,  depth + 1);              // left first ⇒ rows stay left-to-right
    dfs(node.right, depth + 1);
  }
  dfs(root, 0);
  return res;
};

Trace on [3,9,20,null,null,15,7]: queue [3], size 1 → level [3], next [9, 20]. Round 2: size 2 → level [9, 20]; 9 has no children, 20 contributes 15 and 7 → next [15, 7]. Round 3: size 2 → level [15, 7], next []. Queue empty → stop. Result [[3],[9,20],[15,7]]. ✓ The size snapshot is what stopped round 2 from also swallowing 15 and 7.

Complexity: O(n) time — every node is enqueued once and dequeued once, O(1) work each (this is why avoiding shift() matters: with shift() each dequeue is O(n) and the whole thing becomes O(n²) on a wide tree). Space O(w) where w is the widest level — for a complete tree that's about n/2 leaves, so O(n) worst case. Note the contrast worth saying aloud: BFS costs width, DFS costs height. On a wide, shallow tree DFS is cheaper; on a deep, skinny one BFS is.

Follow-ups you should expect:

  • "Bottom-up level order" (LC 107): identical code, then res.reverse() — or res.unshift(level), which is O(n) per insert and therefore the worse answer. Say why you chose reverse.
  • "Average of each level" (LC 637): swap res.push(level) for res.push(sum / size). You already have size — that's the tell that the template is doing the work.
  • "Largest value in each level" / "level with the most nodes" — same loop, different per-level reduction. Once you see it, every "…per level" problem is this one.
  • "Do it with DFS" — the second function above. They ask this to check you understand that BFS isn't required, just natural.
The frontend parallel is exact and worth one sentence: this is how you render a nested comment thread or a file tree by depth, and it's how virtualized tree components decide what's on screen. More usefully, BFS is what you reach for when you need the shallowest answer — shortest path, minimum depth, "first level where X happens" — because BFS finds it before it explores anything deeper. That's the same reason BFS (not DFS) is the tool for shortest path on an unweighted graph in tutorial 10. State the principle, not just the code.

9. Binary Tree Zigzag Level Order Traversal (LC #103) MEDIUM

Same as Level Order, but the direction alternates: level 0 (the root) is read left-to-right, level 1 right-to-left, level 2 left-to-right again, and so on — a boustrophedon or "snake" order. Return the values grouped by level in that zigzag order. This one shows up repeatedly in Goldman Sachs loops, usually as the medium after an easy warm-up, so it deserves the extra ten minutes.

Example 1: Input: root = [3,9,20,null,null,15,7] → Output: [[3],[20,9],[15,7]] — level 0 forward, level 1 reversed (20 before 9), level 2 forward again.
Example 2: Input: root = [1,2,3,4,5,6,7] → Output: [[1],[3,2],[4,5,6,7]] — note level 2 is forward, because it's an even-indexed level. Getting the parity right is half the problem.
Example 3: Input: root = [1] → Output: [[1]].
Constraints: 0 ≤ nodes ≤ 2000; −100 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. Resist the urge to invent a new traversal. The traversal is unchanged — you still visit each level left-to-right, and you still enqueue children left-to-right. Only the presentation of each finished row flips. So take Level Order verbatim and add one boolean: leftToRight, toggled at the end of every round; when it's false, reverse the row before pushing it. That's a two-line diff, and saying "this is Level Order with the row reversed on odd levels" before you write anything is the answer the interviewer is listening for. Two refinements worth mentioning: (a) instead of level.reverse() you can unshift each value — but unshift is O(k) per call, making the row O(k²), so reverse() (a single O(k) pass) is strictly better; (b) the genuinely "clever" solution people reach for — two stacks, alternating which child you push first — works, is O(n), and is harder to get right under pressure. Take the boolean. Cleverness that costs you five minutes of debugging is a net negative at this tier.

Solution (JS) — O(n) time · O(w) space 
var zigzagLevelOrder = function(root) {
  const res = [];
  if (!root) return res;

  let queue = [root];
  let leftToRight = true;   // level 0 reads forward

  while (queue.length) {
    const size = queue.length;
    const level = [];
    const next = [];

    // The TRAVERSAL never changes — always left-to-right, always left child first.
    for (let i = 0; i < size; i++) {
      const node = queue[i];
      level.push(node.val);
      if (node.left)  next.push(node.left);
      if (node.right) next.push(node.right);
    }

    // Only the OUTPUT flips. reverse() is one O(k) pass; unshift() would be O(k²).
    res.push(leftToRight ? level : level.reverse());
    leftToRight = !leftToRight;
    queue = next;
  }
  return res;
};

Trace on [3,9,20,null,null,15,7]: Round 1: size 1, level [3], leftToRight = true → push [3]; flip to false; next [9, 20]. Round 2: size 2, level collected in traversal order [9, 20], leftToRight = false → reverse → push [20, 9]; flip to true; next [15, 7]. Round 3: level [15, 7], forward → push as-is. Result [[3],[20,9],[15,7]]. ✓ Critically, node 20's children were still enqueued left-then-right, which is why level 2 comes out correctly ordered — the flip is cosmetic, applied per row.

Complexity: O(n) time — one enqueue/dequeue per node, plus the reverses, which together touch each node at most once more (∑ level sizes = n), so still O(n). Space O(w) for the queue plus the current row.

Follow-ups you should expect:

  • "Do it without reversing." Write into the row at a computed index: const idx = leftToRight ? i : size - 1 - i; level[idx] = node.val; — fills the array in place, no second pass. Clean, and it shows you can manipulate indices deliberately.
  • "Do it with two stacks." Pop from stack A pushing children left→right onto B; pop from B pushing children right→left onto A. Correct and O(n); mention it, but explain you'd ship the boolean because it's a smaller diff from a template you've already proven.
  • "Do it with DFS." Same depth-indexed DFS as before, then reverse the odd-indexed rows at the end. Works, but the reversal is now a post-pass over the whole result.
  • "Why is unshift bad here?" Because JS arrays are contiguous: unshift re-indexes every element, so building a k-element row costs O(k²). This is the same reason §2b avoids shift(). Enterprise interviewers do probe JS runtime knowledge on frontend candidates.

10. Binary Tree Right Side View (LC #199) MEDIUM ★

Imagine standing to the right of the tree and looking at it horizontally. Return the values of the nodes you can see, ordered top to bottom. Concretely: the rightmost node of every level — which is not the same as "walk down the right spine", because a level's rightmost visible node may live in the left subtree if the right subtree ended earlier.

Example 1: Input: root = [1,2,3,null,5,null,4] → Output: [1,3,4] — level 0 shows 1, level 1's rightmost is 3, level 2's rightmost is 4 (node 5 is hidden behind it).
Example 2: Input: root = [1,2,3,4] → Output: [1,3,4] — this is the example that matters. At level 2, the only node is 4, and it hangs off the left subtree. Naively following right pointers from the root gives [1,3] and misses it entirely.
Example 3: Input: root = [] → Output: [].
Constraints: 0 ≤ nodes ≤ 100; −100 ≤ Node.val ≤ 100.

Solve on LeetCode ↗

Thinking process. Reframe the picture into a precise rule — that's the whole exercise. "What I can see from the right" means "for each depth, the node that is furthest right", which means: one node per level, the last one in left-to-right order. Once phrased that way, the BFS solution is Level Order with the row thrown away: keep only the value at index size − 1. Done in six lines.

The DFS solution is more interesting and is what the interviewer usually wants to see next. Traverse right child first, carrying the depth. The first node you reach at any given depth is, by construction, the rightmost one at that depth — so if res.length === depth, this node is the first arrival at a new depth and you record it; otherwise you've already seen that level's winner and you skip. Right-first DFS is a genuinely elegant trick and it's the classic "either works, defend your pick" moment: BFS costs O(w) queue space and reads obviously; DFS costs O(h) stack space and is the right call if the tree is wide but shallow. Say the tradeoff out loud — this problem is on the page precisely to give you a place to say it.

Solution (JS) — O(n) time · O(w) space (BFS) or O(h) (DFS) 
// ---------- BFS: take the last node of every level ----------
var rightSideView = function(root) {
  const res = [];
  if (!root) return res;

  let queue = [root];
  while (queue.length) {
    const size = queue.length;
    const next = [];

    for (let i = 0; i < size; i++) {
      const node = queue[i];
      // Last node of this level ⇒ the one visible from the right.
      if (i === size - 1) res.push(node.val);
      if (node.left)  next.push(node.left);
      if (node.right) next.push(node.right);
    }
    queue = next;
  }
  return res;
};

// ---------- DFS: go RIGHT first; first arrival at a depth wins ----------
var rightSideViewDFS = function(root) {
  const res = [];

  function dfs(node, depth) {
    if (!node) return;

    // res.length === depth means no node from this depth has been recorded yet.
    // Because we always descend RIGHT before LEFT, the first arrival at a
    // given depth is necessarily its rightmost node.
    if (res.length === depth) res.push(node.val);

    dfs(node.right, depth + 1);   // RIGHT FIRST — this is the whole trick
    dfs(node.left,  depth + 1);
  }

  dfs(root, 0);
  return res;
};

Trace the DFS on Example 2, [1,2,3,4] (root 1; left child 2 with left child 4; right child 3): dfs(1, 0) — res.length (0) === 0 → record 1, res = [1]. Go right: dfs(3, 1) — res.length (1) === 1 → record 3, res = [1,3]; node 3 has no children. Back at the root, go left: dfs(2, 1) — res.length (2) !== 1 → skip (level 1 already has its winner, 3). Descend: dfs(4, 2) — res.length (2) === 2 → record 4 → res = [1,3,4]. ✓ Node 4 is captured even though it sits in the left subtree — which is exactly what a naive right-spine walk misses.

Complexity: both are O(n) time — every node is visited once, and the per-node work is O(1) (the res.length === depth test is why DFS needs no auxiliary map). BFS space is O(w), the widest level; DFS space is O(h), the height. Pick by tree shape, and say so: a wide, shallow tree favors DFS; a deep, skinny one favors BFS. That sentence is the entire senior signal on this problem.

Follow-ups you should expect:

  • "Left side view." BFS: take i === 0. DFS: descend left first. One character each — and if you built it as a template, you can say that instead of retyping it.
  • "Bottom view / top view." Now you need horizontal distance, not depth: BFS carrying col (left = col−1, right = col+1), keeping the first (top view) or last (bottom view) node per column. Same machinery, one more coordinate.
  • "Vertical order traversal" (LC 987) — the hard cousin: group by column, tie-break by row then value. Recognize it, and note that it needs a sort, so it's O(n log n).
  • "Why does right-first DFS work?" Be ready to prove it in one sentence: within a depth, the traversal reaches nodes strictly right-to-left, so the first one seen is the rightmost. Interviewers ask because candidates often reproduce the trick without understanding it.
Two traps in this section, both common. First: writing the right side view by chasing node.right down the spine. It passes Example 1 and fails Example 2, and it will be the interviewer's first test input. Second: forgetting the if (!root) return [] guard. All three problems here start by enqueuing the root; enqueue a null and your first iteration dereferences it. LeetCode's constraints allow 0 nodes on all three — the empty-tree test case is in the suite, and it is the most common "so close" failure on an otherwise perfect submission.

5. BST problems

Everything here is one of the two consequences from §1: inorder gives you sorted order, and one comparison discards a whole subtree. Problems 11 and 12 are deliberately paired — the same question asked of a BST and of a general tree — because the contrast is the interview: it shows whether you actually exploit structure when you're given it, or just fall back on a generic traversal.

11. Lowest Common Ancestor of a BST (LC #235) EASY ★

Given a binary search tree and two nodes p and q that are guaranteed to exist in it, return their lowest common ancestor: the deepest node that has both p and q as descendants. Per the standard definition, a node is a descendant of itself — so if p is an ancestor of q, the answer is p. All values are unique.

Example 1: Input: root = [6,2,8,0,4,7,9,null,null,3,5], p = 2, q = 8 → Output: 6 — they're on opposite sides of the root, so the root is where their paths split.
Example 2: Input: same tree, p = 2, q = 4 → Output: 2 — 4 lives inside 2's subtree, and a node counts as its own descendant. This is the case people get wrong.
Constraints: 2 ≤ nodes ≤ 10⁵; all Node.val unique; p ≠ q; both exist in the tree.

Solve on LeetCode ↗

Thinking process. Forget trees for a second and think about what "lowest common ancestor" means in a BST: it's the node where the two search paths — the path you'd walk to find p, and the path you'd walk to find q — diverge. Before that point they're identical; after it they never meet again. So walk down from the root and ask one question at each node: are p and q both smaller than me? Then the split hasn't happened yet — go left. Both bigger? Go right. Anything else — one on each side, or one of them is me — means I'm the split point, and I'm the answer. That last clause is the whole problem: you don't need a special case for "p is an ancestor of q", because if p.val === node.val then neither "both smaller" nor "both bigger" holds, and you correctly return node. And notice you never look at a subtree that can't contain the answer — that's why this is O(h), not O(n), and why it's the BST version rather than the general version (problem 12). No recursion is even needed; a while loop gives you O(1) space.

Solution (JS) — O(h) time · O(1) space 
var lowestCommonAncestor = function(root, p, q) {
  let node = root;

  while (node) {
    if (p.val < node.val && q.val < node.val) {
      // Both targets are strictly in the LEFT subtree ⇒ the split is deeper.
      // Discarding the right subtree here is what buys us O(h).
      node = node.left;
    } else if (p.val > node.val && q.val > node.val) {
      // Both strictly right ⇒ still haven't split.
      node = node.right;
    } else {
      // Either they straddle this node, or one of them IS this node.
      // Both mean: this is the deepest node containing both. Done.
      return node;
    }
  }
  return null;   // unreachable given the constraints, but don't leave a dangling path
};

Trace on Example 2 (tree rooted at 6, p = 2, q = 4): at node 6 — 2 < 6 and 4 < 6, both smaller → go left to node 2. At node 2 — is 2 < 2? No. Is 2 > 2? No. So neither branch fires → return node 2. ✓ The "ancestor of itself" case is handled by the structure of the conditions, with no special-casing. On Example 1 (p = 2, q = 8): at node 6, 2 < 6 but 8 > 6 — not both smaller, not both bigger → return 6 immediately. ✓

Complexity: O(h) time — you descend one level per iteration and never backtrack; h is log n for a balanced BST and n for a skewed one. O(1) space with the loop, since no stack is kept. The recursive version is the same code and O(h) space; prefer the loop and say why: "it's tail-recursive, so I unrolled it — constant space." That's a nice, cheap thing to have noticed.

Follow-ups you should expect:

  • "What if p or q might NOT be in the tree?" Then you must verify: after finding the candidate split node, search for both values below it. The clean answer is "the current code assumes existence, as the constraints state; if that's dropped I'd add a containment check, which is another O(h) each."
  • "Now do it on a general binary tree." That's exactly problem 12 — and the whole point is that you lose the invariant, so you lose the O(h).
  • "Distance between two nodes in a BST?" Find the LCA, then measure the depth from the LCA down to each node: d(lca,p) + d(lca,q). LCA is the standard building block for every "relationship between two nodes" question.
When a problem hands you a BST, the interviewer is testing one thing: do you use the invariant, or do you write a generic tree traversal that ignores it? A correct O(n) DFS on this problem is a miss, even though it passes all the tests. Say it explicitly before you start: "It's a BST, so I can compare against the node value and throw away half the tree at every step — that gives O(h) instead of O(n)." This is the highest-leverage sentence on this entire page.

12. Lowest Common Ancestor of a Binary Tree (LC #236) MEDIUM

Same question, no BST guarantee: given the root of an ordinary binary tree and two nodes p and q that both exist in it, return their lowest common ancestor. Again, a node is a descendant of itself. Values are unique, but they tell you nothing about where a node lives — so you cannot navigate; you must search.

Example 1: Input: root = [3,5,1,6,2,0,8,null,null,7,4], p = 5, q = 1 → Output: 3 — one target in each subtree of the root.
Example 2: Input: same tree, p = 5, q = 4 → Output: 5 — node 4 is a descendant of 5 (via 2), and 5 is its own descendant, so 5 is the LCA. The self-ancestor case again — it's the one interviewers probe.
Constraints: 2 ≤ nodes ≤ 10⁵; all values unique; p ≠ q; both exist in the tree.

Solve on LeetCode ↗

Thinking process. With no ordering, you can't choose a direction — you have to look everywhere, so O(n) is the floor. The brute force people reach for first is: find the root-to-p path and the root-to-q path as arrays, then walk both from the front and take the last node they agree on. That works, is O(n) time and O(n) space, and is perfectly explainable. Offer it, then improve on it.

The elegant version collapses "find the paths" and "compare the paths" into one postorder pass by returning a cleverly chosen value: lca(node) returns a node if this subtree contains p, q, or the LCA itself, and null if it contains none of them. Base cases: an empty subtree returns null; if node is p or q, return it immediately (don't descend — this is exactly what makes the self-ancestor case fall out for free, because if the other target is below, this node is still the LCA). Otherwise recurse both ways. If both sides come back non-null, the two targets were found in different subtrees, so this node is the split point → return it. If only one side is non-null, pass that result upward unchanged — it's either a found target or an already-determined LCA, and either way it's the best news this subtree has. The subtlety candidates miss: the function's return value means two different things depending on context, and that's fine, because the caller only ever needs "did this side contain anything relevant?" Be ready to say that clearly.

Solution (JS) — O(n) time · O(h) space 
var lowestCommonAncestor = function(root, p, q) {
  // Base 1: nothing here.
  if (root === null) return null;

  // Base 2: found a target. Return it WITHOUT descending.
  // If the other target lives below me, I'm still the LCA (self-descendant rule),
  // and my parent will see exactly one non-null child and propagate me up.
  if (root === p || root === q) return root;

  const left  = lowestCommonAncestor(root.left,  p, q);
  const right = lowestCommonAncestor(root.right, p, q);

  // Targets found on BOTH sides ⇒ their paths split HERE ⇒ I am the LCA.
  if (left && right) return root;

  // Exactly one side (or neither) reported something. Pass it up unchanged:
  // it is either a found target or an LCA already decided deeper down.
  return left !== null ? left : right;
};

Trace on Example 2 (root 3; left child 5 with children 6 and 2; 2 has children 7 and 4; right child 1 with children 0 and 8. p = 5, q = 4): call on 3 → not a target, recurse. Left: call on 5 → root === p → return 5 immediately (we never even look at 4 down there — we don't need to). Right: call on 1 → recurse into 0 (null result) and 8 (null result) → both null → returns null. Back at 3: left = 5, right = null → not both → return left = 5. ✓ The answer is 5, and the early return is precisely what makes the ancestor-of-itself case correct.

Complexity: O(n) time — worst case every node is visited once; no ordering means no pruning, so you cannot do better in the general case. O(h) space for the recursion stack. Contrast with problem 11's O(h) time and say the reason out loud: "the BST invariant buys pruning; without it, I have to search the whole tree."

Follow-ups you should expect:

  • "What if p or q might not exist in the tree?" The code above would happily return the one that does exist, which is wrong. Fix: also return a found-count, or do a full traversal that verifies both were seen. Flag this proactively — it's the most common trapdoor on LC 236, and noticing an unstated assumption in your own solution is a strong senior move.
  • "Nodes have parent pointers." Then it's the linked-list intersection problem: walk up from each node to build depths, lift the deeper one, then advance both until they meet — O(h) time, O(1) space. Recognizing the reduction to tutorial 06 is the win.
  • "Many LCA queries on the same tree." Preprocess: binary lifting / Euler tour + sparse table gives O(log n) or O(1) per query after O(n log n) setup. You will not be asked to implement this at this tier — knowing the name is sufficient and impressive.

13. Validate Binary Search Tree (LC #98) MEDIUM ★

Given the root of a binary tree, determine whether it is a valid binary search tree. Valid means, at every node: every value in its left subtree is strictly less than the node, every value in its right subtree is strictly greater, and both subtrees are themselves valid BSTs. Duplicates are not allowed. This is the trap promised in §1 — and one of the most frequently asked mediums in the entire enterprise tier.

Example 1: Input: root = [2,1,3] → Output: true.
Example 2: Input: root = [5,1,4,null,null,3,6] → Output: false — node 4 is in 5's right subtree and has left child 3. Locally, 3 < 4 looks fine; globally, 3 sits to the right of 5 while being smaller than 5. Invalid.
Example 3: Input: root = [5,4,6,null,null,3,7] → Output: false — same shape of failure, deeper. Every parent–child pair is individually correct. Any solution that only compares a node to its immediate children returns true here and is wrong.
Constraints: 1 ≤ nodes ≤ 10⁴; −2³¹ ≤ Node.val ≤ 2³¹ − 1 (note: the values can hit the 32-bit extremes — that detail is deliberate).

Solve on LeetCode ↗

Thinking process. The wrong answer writes itself in fifteen seconds: check left.val < node.val < right.val at each node and recurse. Example 3 kills it. The reason it fails is that the BST property is a constraint from ancestors, not from parents: when you descend left from node 5, every node down there — not just 5's child — inherits an upper bound of 5. When you then descend right from 4, that node inherits a lower bound of 4 while still carrying the upper bound of 5 from above. So the missing information is context from above, and the fix is the move you learned in Symmetric Tree: change the signature so the state you need is in the arguments. Carry a running (min, max) window down the recursion. Every node must land strictly inside its window; descending left tightens the max to the current value, descending right tightens the min. Start the root with an unbounded window, and — because the constraints allow node values at the exact 32-bit extremes — use -Infinity / Infinity rather than Number.MIN_SAFE_INTEGER, or you will fail the test case containing 2147483647. That specific test case exists on LeetCode purely to catch this.

There's a second, equally valid solution worth knowing because it generalizes: inorder traversal must be strictly increasing. Walk the tree inorder, keep only the previously-visited value, and fail the moment curr ≤ prev. It's O(n) time and O(h) space, needs no bounds, and it's the tool for the follow-up "two nodes were swapped — recover the tree" (LC 99). Present the bounds solution first (it short-circuits and reads more directly), then mention inorder as the alternative. Having two derivations for the same problem, and a reason to prefer one, is exactly the depth a 7-YOE loop is scoring.

Solution (JS) — O(n) time · O(h) space 
// ---------- Solution A: (min, max) bounds passed DOWN ----------
var isValidBST = function(root) {
  // Node must satisfy:  low < node.val < high  (both bounds EXCLUSIVE — no duplicates).
  // ±Infinity, not MIN/MAX_SAFE_INTEGER: node values may legitimately be ±2^31.
  function valid(node, low, high) {
    if (node === null) return true;   // empty subtree is vacuously a BST

    if (node.val <= low || node.val >= high) return false;

    // Going LEFT: everything below must stay under node.val, but the inherited
    // `low` from ancestors STILL APPLIES — that's the whole point.
    return valid(node.left,  low, node.val)
        && valid(node.right, node.val, high);   // going RIGHT: tighten the lower bound
  }

  return valid(root, -Infinity, Infinity);
};

// ---------- Solution B: inorder must be strictly increasing ----------
var isValidBSTInorder = function(root) {
  let prev = -Infinity;   // last value emitted by the inorder walk
  const stack = [];
  let curr = root;

  while (curr || stack.length) {
    while (curr) { stack.push(curr); curr = curr.left; }
    curr = stack.pop();

    // Strictly increasing: `<=` catches duplicates too.
    if (curr.val <= prev) return false;
    prev = curr.val;

    curr = curr.right;
  }
  return true;
};

Trace Solution A on Example 3, [5,4,6,null,null,3,7] (root 5; left 4; right 6, whose children are 3 and 7): valid(5, −∞, ∞) → 5 is inside ✓. Left: valid(4, −∞, 5) → 4 < 5 ✓, leaf, true. Right: valid(6, 5, ∞) → 6 > 5 ✓. Now 6's left child: valid(3, 5, 6) — the window inherited low = 5 from the root's right turn and high = 6 from this left turn. Is 3 > 5? No → false. ✓ The naive parent-only check would have compared 3 to its parent 4 (or 6), found it smaller, and wrongly said valid. The ancestor bound is what catches it.

Complexity: O(n) time — every node is visited once and does two O(1) comparisons; the && short-circuits so an invalid left subtree skips the right entirely. O(h) space for the call stack. Solution B is the same asymptotics with an explicit stack. Neither can beat O(n): a violation can hide at any node, so you must look at all of them.

Follow-ups you should expect:

  • "Are duplicates allowed?" Ask this before you code. LC 98 says no (strict inequality), but many real BSTs allow duplicates on one side. Changing <= to < is the whole diff, and asking the question is graded.
  • "Exactly two nodes were swapped by mistake — recover the tree" (LC 99). This is why Solution B matters: an inorder walk of a corrupted BST has one or two descents; the first descent's larger element and the last descent's smaller element are the swapped pair. Swap their values back. O(n) / O(h).
  • "Why not just do an inorder traversal into an array and check it's sorted?" Correct, but it's O(n) extra space and it can't short-circuit. Solution B is the same idea with only prev retained — say that; it's the sort of memory-conscious refinement seniors are expected to volunteer.
  • "What breaks if you use Number.MIN_SAFE_INTEGER as the initial bound?" Nothing on most inputs — and then LeetCode's [2147483647] test case fails you. Use ±Infinity. (An alternative that dodges the issue entirely: pass null bounds and check low !== null && node.val <= low.)
The other half of this trap is only tightening one bound. When you go left you must pass (low, node.val) — carrying the inherited low forward — not (-Infinity, node.val). Resetting the untouched bound silently discards the ancestor constraint and re-introduces exactly the bug you were fixing: it will accept Example 2. Each recursive call tightens one bound and inherits the other. Write the two calls side by side, as above, so the asymmetry is visible on the page and you can point at it while you explain.

14. Kth Smallest Element in a BST (LC #230) MEDIUM ★

Given the root of a binary search tree and an integer k, return the k-th smallest value in the tree, 1-indexed (so k = 1 means the minimum). It is guaranteed that 1 ≤ k ≤ n.

Example 1: Input: root = [3,1,4,null,2], k = 1 → Output: 1 — the inorder sequence is 1, 2, 3, 4, so the smallest is 1.
Example 2: Input: root = [5,3,6,2,4,null,null,1], k = 3 → Output: 3 — inorder is 1, 2, 3, 4, 5, 6; the third is 3. Note the answer happens to be the root's left child, but that's a coincidence — there is no positional shortcut.
Constraints: 1 ≤ k ≤ n ≤ 10⁴; 0 ≤ Node.val ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. One line of derivation: inorder traversal of a BST emits values in sorted ascending order, so the k-th value it emits is the answer. The naive implementation collects the whole traversal into an array and returns arr[k-1] — correct, O(n) time and O(n) space. Now improve it in the two ways the interviewer wants. First, you don't need the array: just count as you visit, and stop the moment the counter hits k. Second — and this is why §2c exists — the recursive version can't easily stop; you'd have to thread a flag through every frame or throw. The iterative inorder can just return in the middle of the loop. That makes it O(h + k) rather than O(n): you push the left spine (O(h)) and then pop exactly k times. If k = 1 on a million-node tree you touch only the left spine. Say that number out loud — "iterative lets me short-circuit, so it's O(h + k), not O(n)" — because it's the difference between a passing answer and a strong one.

Solution (JS) — O(h + k) time · O(h) space 
var kthSmallest = function(root, k) {
  const stack = [];
  let curr = root;

  while (curr || stack.length) {
    // Dive as far left as possible: the deepest-left node is the smallest
    // value not yet emitted. The stack holds the ancestors we owe a visit to.
    while (curr) {
      stack.push(curr);
      curr = curr.left;
    }

    curr = stack.pop();       // next value in ASCENDING order
    if (--k === 0) return curr.val;   // SHORT-CIRCUIT — the reason we went iterative

    curr = curr.right;        // everything > curr but < its parent lives here
  }

  return -1;   // unreachable: constraints guarantee k <= n
};

Trace on Example 2 (root = [5,3,6,2,4,null,null,1], k = 3): dive left from 5 → stack [5, 3, 2, 1], curr = null. Pop 1 → k becomes 2, not zero; go right from 1 → null. Pop 2 → k becomes 1; go right from 2 → null. Pop 3 → --k === 0 → return 3. ✓ Note we never touched nodes 4, 5, or 6 — the short-circuit paid off on a tiny tree, and it pays off enormously on a large one.

Complexity: O(h + k) time — the initial dive down the left spine costs O(h), then each of the k pops does O(1) work plus, amortized, at most one push per node visited. In the worst case (k = n) that's O(n), but the k-dependence is the honest bound and it's better than O(n) whenever k is small. Space O(h) for the stack.

Follow-ups you should expect:

  • "The BST is modified often (insert/delete) and you need the k-th smallest frequently. Optimize." This is the real follow-up and LeetCode states it explicitly. Augment each node with count = size of its subtree. Then finding the k-th smallest is a descent: if leftCount + 1 === k the node is the answer; if k ≤ leftCount go left; else go right with k -= leftCount + 1. That's O(h) per query, and inserts/deletes update counts along their O(h) path. Being able to describe this augmentation is the difference between "solved it" and "designed it" — and it's exactly the kind of order-statistics index that a bank's matching engine actually runs.
  • "Kth LARGEST?" Mirror it: reverse inorder (right, node, left) with the same counter. Don't collect and reverse.
  • "Recursively, with the early exit?" Possible with a counter in a closure and an early return guard at the top of the function, but it still unwinds the whole stack. Mention that iterative expresses the intent better here.
Remember what problems 11–14 have in common: none of them needed a clever algorithm. Each one needed you to notice which invariant applies and then write ten obvious lines. That's what BST questions test, and it's why they're so popular in enterprise screens — they're impossible to fake and quick to grade.

15. Construct Binary Tree from Preorder and Inorder Traversal (LC #105) MEDIUM ★

You're given two integer arrays: preorder, the preorder traversal of a binary tree (node, then left subtree, then right subtree), and inorder, the inorder traversal of the same tree (left subtree, then node, then right subtree). All values are unique. Reconstruct the tree and return its root.

Example 1: Input: preorder = [3,9,20,15,7], inorder = [9,3,15,20,7] → Output: [3,9,20,null,null,15,7] — preorder[0] = 3 is the root; in inorder, 3 sits at index 1, so [9] is its entire left subtree and [15,20,7] its entire right subtree.
Example 2: Input: preorder = [-1], inorder = [-1] → Output: [-1] — a single node; your recursion must terminate cleanly on a one-element range.
Example 3: Input: preorder = [1,2,3], inorder = [3,2,1] → Output: a left-skewed chain 1 → 2 → 3. The root 1 is last in inorder, so its right subtree is empty — the case that breaks sloppy index arithmetic.
Constraints: 1 ≤ preorder.length ≤ 3000; inorder.length === preorder.length; all values unique; inorder is guaranteed to be a permutation of preorder.

Solve on LeetCode ↗

Thinking process. Two facts, and the algorithm is forced. Fact one: preorder tells you who the root is — it's the first element, always, because preorder visits the node before anything below it. Fact two: inorder tells you how big each side is — find the root's value in inorder; everything to its left is exactly the left subtree, everything to its right is exactly the right subtree, because inorder emits a node's entire left subtree before the node itself. Neither array alone is enough (preorder alone doesn't tell you where the left subtree stops), but together they pin the tree down uniquely.

So: take preorder[0] as the root, locate it in inorder at index i, and now you know the left subtree has k = i - inStart nodes. That count is the bridge back to preorder: the next k entries of preorder are the left subtree's preorder, and everything after them is the right subtree's preorder. Recurse on both halves. The naive implementation does inorder.indexOf(rootVal) — an O(n) scan inside an O(n) recursion, giving O(n²). The one-line fix, and the thing the interviewer is waiting for: pre-build a Map from value → index in inorder so the lookup is O(1) and the whole thing is O(n). Values are guaranteed unique, which is precisely what makes that map well-defined — say that; it shows you read the constraints and understood why they're there. The cleanest way to manage the preorder side is a single moving pointer (a closure variable) that you advance once per node created, which sidesteps all the index arithmetic that makes this problem error-prone.

Solution (JS) — O(n) time · O(n) space 
var buildTree = function(preorder, inorder) {
  // value → index in inorder. Values are unique (per constraints), so this is
  // well-defined and turns an O(n) scan into an O(1) lookup: O(n^2) → O(n).
  const idx = new Map();
  for (let i = 0; i < inorder.length; i++) idx.set(inorder[i], i);

  // A single moving cursor into preorder. Because we always build the ROOT,
  // then the ENTIRE left subtree, then the entire right subtree — which is
  // exactly preorder — consuming one element per node created keeps it aligned.
  let p = 0;

  // Build the subtree whose inorder slice is inorder[lo..hi] (inclusive).
  function build(lo, hi) {
    if (lo > hi) return null;   // empty range ⇒ no node (this is Example 3's right side)

    const rootVal = preorder[p++];        // FACT 1: preorder's next value is this subtree's root
    const root = new TreeNode(rootVal);

    const mid = idx.get(rootVal);         // FACT 2: inorder splits the range around the root

    // Order matters: the left subtree must consume its preorder elements FIRST,
    // because in preorder they come before the right subtree's. Swapping these
    // two lines silently builds the mirror of the wrong tree.
    root.left  = build(lo, mid - 1);
    root.right = build(mid + 1, hi);

    return root;
  }

  return build(0, inorder.length - 1);
};

Trace on Example 1 (preorder = [3,9,20,15,7], inorder = [9,3,15,20,7]; map = {9:0, 3:1, 15:2, 20:3, 7:4}): build(0, 4) takes preorder[0] = 3, p = 1, mid = 1. Left = build(0, 0): takes preorder[1] = 9, p = 2, mid = 0; its children are build(0, −1) and build(1, 0) — both have lo > hi → null. Node 9 done. Right = build(2, 4): takes preorder[2] = 20, p = 3, mid = 3. Its left is build(2, 2) → takes preorder[3] = 15; its right is build(4, 4) → takes preorder[4] = 7. Result: [3,9,20,null,null,15,7]. ✓ Notice the cursor p marched 0→4 in exactly preorder sequence and never needed to be reset — that's the invariant that makes the cursor version safe.

Complexity: O(n) time — the map costs one O(n) pass, then build is called once per node (plus once per null child) and does O(1) work each. O(n) space: the map is O(n), the call stack is O(h), and the output tree is O(n). Explicitly contrast the O(n²) version that scans inorder with indexOf at every node — on a skewed tree that's a million operations instead of a thousand, and stating the improvement unprompted is the graded moment here.

Follow-ups you should expect:

  • "Now from inorder + postorder" (LC 106). Postorder ends with the root, so consume it from the back and build the right subtree first — the mirror image of this code. If you built it with a cursor, the diff is three lines. Interviewers love asking this immediately after 105 to see if you understood or memorized.
  • "Preorder + postorder?" Does not uniquely determine a binary tree — you can't tell whether a lone child is a left or right child. It does work for full binary trees (LC 889). Knowing which pairs are sufficient is a genuinely strong answer.
  • "What if values can repeat?" The map breaks, and so does the whole approach — the reconstruction isn't unique any more. Say this immediately; it's the assumption the constraints are quietly protecting.
  • "Build a BST from a sorted array" (LC 108) — much easier and worth mentioning as the sibling: the middle element is the root, recurse on halves, and it's automatically height-balanced.
The framing to say out loud: "Preorder gives me the root; inorder gives me the split. Together they're enough to reconstruct uniquely — and I'll index inorder in a hash map up front so each lookup is O(1) instead of a linear scan." You've just stated the correctness argument and the optimization in one breath, before writing a line. That's what senior narration sounds like, and it's the same reflex as reaching for a lookup map in Two Sum.

16. Binary Tree Maximum Path Sum (LC #124) HARD

A path is any sequence of nodes where each consecutive pair is connected by an edge, and no node appears twice. A path does not need to pass through the root, and it does not need to start or end at a leaf. The path sum is the sum of the values of its nodes. Given the root, return the maximum path sum over all possible paths. Node values may be negative — which is the entire difficulty.

Example 1: Input: root = [1,2,3] → Output: 6 — the path 2 → 1 → 3 uses all three nodes.
Example 2: Input: root = [-10,9,20,null,null,15,7] → Output: 42 — the best path is 15 → 20 → 7 (= 42). Going up through the root would cost you −10, so the optimal path lives entirely inside the right subtree and never touches the root.
Example 3: Input: root = [-3] → Output: -3 — a path must contain at least one node, so you cannot return 0 by "taking nothing". Any solution initialized to 0 instead of -Infinity fails here.
Constraints: 1 ≤ nodes ≤ 3·10⁴; −1000 ≤ Node.val ≤ 1000.

Solve on LeetCode ↗

Thinking process. You already solved the structure of this problem — it's Diameter (problem 5) with sums instead of edge counts. Same observation: every path has a single highest node, its peak. For a fixed peak, the best path through it is node.val + bestDownwardPath(left) + bestDownwardPath(right). So run one postorder DFS, return the best downward path from each node (that's what a parent can use), and track the best peaked path in a closure variable at every node. Identical skeleton to Diameter — say that out loud, it converts a "hard" into a "medium I've already done".

Two things change, and both come from negatives. First: you can decline a subtree. If the best downward path from the left child is negative, taking it makes your path worse — so clamp it to zero: Math.max(0, gain(left)). That single Math.max(0, …) is the whole difference from Diameter, and it encodes "a path may stop at this node". Second: the return value must be a straight line, not a peak. What you hand your parent is node.val + max(leftGain, rightGain) — one branch only, because your parent is going to extend that path upward, and a path can't fork. Returning val + left + right would let the parent build a Y shape, which is not a path. Candidates conflate "the answer at this node" with "what I return", and this is where they lose the problem. Finally, initialize the tracker to -Infinity, not 0 — Example 3 is the all-negative test case that punishes the shortcut.

Solution (JS) — O(n) time · O(h) space 
var maxPathSum = function(root) {
  // -Infinity, not 0: a path must contain ≥ 1 node, so an all-negative
  // tree like [-3] must answer -3. Initializing to 0 silently fails that.
  let best = -Infinity;

  // gain(node) = the best sum of a path that STARTS at node and goes strictly
  // DOWNWARD (a straight line, never forking). That's the only thing a parent
  // can splice onto its own path.
  function gain(node) {
    if (node === null) return 0;   // contributing nothing is worth 0

    // Clamp at 0: a negative branch is one we simply DON'T take.
    // This is the only line that differs in spirit from Diameter.
    const left  = Math.max(0, gain(node.left));
    const right = Math.max(0, gain(node.right));

    // The best path PEAKING here forks both ways. This is a candidate ANSWER,
    // but it is NOT what we return — a parent can't extend a forked path.
    best = Math.max(best, node.val + left + right);

    // Return the straight line: this node plus AT MOST ONE branch.
    return node.val + Math.max(left, right);
  }

  gain(root);
  return best;
};

Trace on Example 2, [-10,9,20,null,null,15,7]: gain(9): children 0 and 0 → best = max(−∞, 9) = 9; returns 9. gain(15): best = 15; returns 15. gain(7): best stays 15; returns 7. gain(20): left = max(0, 15) = 15, right = max(0, 7) = 7 → best = max(15, 20 + 15 + 7 = 42) = 42; returns 20 + max(15, 7) = 35 — note it returns 35, the straight line 20→15, not 42. gain(−10): left = max(0, 9) = 9, right = max(0, 35) = 35 → candidate −10 + 9 + 35 = 34, which does not beat 42. Answer 42. ✓ The peak really did live below the root, exactly as the problem's example insists.

Complexity: O(n) time — one postorder pass, O(1) work per node. O(h) space for the call stack. Same bounds as Diameter, because it is Diameter.

Follow-ups you should expect:

  • "Return the actual path, not the sum." Record the peak node when you update best, then re-descend from it choosing the higher-gain child each step (you'd want to memoize the gains to avoid recomputing). Say the shape; don't burn ten minutes coding it unless asked.
  • "What if the path must go leaf-to-leaf?" Then you can't clamp negatives away, and a node with only one child can't be a peak. Different problem, and it's worth naming why the clamp is now illegal — that's the check that you understood the clamp rather than copied it.
  • "Why return only one branch?" Have the one-sentence answer loaded: "Because my parent will extend whatever I return upward, and a path can't fork — if I returned both branches, the parent would build a Y, which isn't a path." This is the question that separates people who solved it from people who remembered it.
  • "House Robber III" (LC 337) is the same closure/postorder machinery with a different combine rule — worth knowing they're relatives.

17. Serialize and Deserialize Binary Tree (LC #297) HARD

Design an algorithm to turn a binary tree into a string (serialize) and to reconstruct the identical tree from that string (deserialize). The format is entirely up to you — the only requirement is that deserialize(serialize(root)) returns a tree structurally and value-identical to the original. The tree is not a BST; node values may be negative; the tree may be empty.

Example 1: Input: root = [1,2,3,null,null,4,5] → Output: the same tree back. Your intermediate string might be "1,2,#,#,3,4,#,#,5,#,#" — the shape is yours to choose.
Example 2: Input: root = [] → Output: []. The empty tree must round-trip too; it's a real test case.
Example 3: Input: root = [1,-2,3] → negative values must survive the round trip, which rules out any single-character-per-node encoding and forces a real delimiter.
Constraints: 0 ≤ nodes ≤ 10⁴; −1000 ≤ Node.val ≤ 1000.

Solve on LeetCode ↗

Thinking process. Start from what problem 15 taught you: a preorder traversal alone does not determine a tree — you can't tell where one subtree ends and the next begins. LC 105 fixed that by supplying a second array. Here you get to design the format, so fix it a cheaper way: write the nulls down. A preorder walk that emits a sentinel (say #) for every absent child is unambiguous, because every node now has exactly two recorded children and the recursion has a definite stopping rule. That's the entire insight, and it's the same insight behind the O(n+m) follow-up to Subtree of Another Tree.

Deserialization then mirrors serialization exactly: read tokens left to right with a single cursor; a # returns null; anything else becomes a node whose left subtree you build first (consuming however many tokens it needs) and whose right subtree you build from whatever's left. The cursor never rewinds. Two implementation details are where people actually lose this problem: (1) use a comma delimiter and split(',') — with no delimiter, 1 then 2 is indistinguishable from 12, and negatives are hopeless; (2) convert back with Number(token), since split gives you strings, and a tree of "5" is not a tree of 5. A BFS/level-order format works equally well and is what LeetCode's own display uses — pick preorder, because the recursion is four lines and BFS bookkeeping is not.

Solution (JS) — O(n) time · O(n) space (both directions) 
// Format: preorder with '#' marking every absent child, comma-delimited.
// The null markers are what make preorder ALONE sufficient — without them,
// the same token stream describes many different trees (see LC 105).

var serialize = function(root) {
  const out = [];

  function dfs(node) {
    if (node === null) {
      out.push('#');     // record the ABSENCE — this is the whole trick
      return;
    }
    out.push(String(node.val));   // node ...
    dfs(node.left);              // ... then its entire left subtree ...
    dfs(node.right);             // ... then its entire right subtree  (= preorder)
  }

  dfs(root);
  return out.join(',');   // delimiter is mandatory: multi-digit and negative values
};

var deserialize = function(data) {
  const tokens = data.split(',');
  let i = 0;   // single forward cursor; it never rewinds

  function build() {
    const token = tokens[i++];       // consume exactly one token per call
    if (token === '#') return null;   // absence: the recursion's base case

    // Number(), not the raw string — a node of "5" is not a node of 5.
    const node = new TreeNode(Number(token));

    // Read in the SAME order we wrote: left subtree fully, then right.
    node.left  = build();
    node.right = build();
    return node;
  }

  return build();
};

Trace on [1,2,3,null,null,4,5] (root 1; left child 2, a leaf; right child 3 with children 4 and 5): serialize emits 1, then into 2 → 2, #, #; then into 3 → 3, into 4 → 4, #, #, into 5 → 5, #, #. String: "1,2,#,#,3,4,#,#,5,#,#". Deserialize: token 1 → node 1; build its left → token 2 → node 2; build 2's left → # → null; 2's right → # → null; node 2 complete and returns. Back at node 1, build its right → token 3 → node 3, whose left consumes 4,#,# and whose right consumes 5,#,#. Cursor lands exactly at the end. ✓ Note how the two functions are structurally the same recursion — one writing, one reading — which is why the cursor stays in sync without any index arithmetic.

Complexity: O(n) time in both directions — each node emits (or consumes) a constant number of tokens, and there are n nodes plus n+1 nulls, so the string has 2n+1 tokens. O(n) space for the string, plus O(h) for the recursion stack. Worth stating: the null markers cost you n+1 extra tokens and buy you unambiguity — that's the trade, and it's a good one.

Follow-ups you should expect:

  • "Make it more compact." A leaf currently costs three tokens (5,#,#). You can drop trailing nulls with a level-order format (what LeetCode's own display does), or encode structure as a bitstring separate from values. Only go here if asked — say the trade-off and move on.
  • "Now do it for a BST" (LC 449). You don't need null markers at all: a plain preorder is sufficient, because the BST invariant tells you where each subtree ends — rebuild using the same (min, max) bounds idea from problem 13. This is a lovely callback and interviewers do ask it as the follow-up.
  • "Do it iteratively / for a huge tree." Recursion depth is O(h); a 10⁴-node skewed tree is fine in JS but say you'd switch to an explicit stack (or the BFS format) if depth were a real concern.
  • "Where does this show up for real?" Everywhere: it's exactly what happens when a React component tree is serialized for SSR hydration, what a DOM snapshot in a testing library does, and what any AST cache writes to disk. That framing lands especially well when the interviewer knows you're coming from frontend.
Look at what closed here. Serialize/Deserialize is Construction (problem 15) with the ambiguity fixed by null markers, and the null-marker trick is the same one that gave Subtree of Another Tree (problem 7) its O(n+m) solution. Diameter (5), Balanced (6) and Max Path Sum (16) are one postorder skeleton with three different combine rules. Level Order (8), Zigzag (9) and Right Side View (10) are one loop with three different per-level lines. Seventeen problems, four ideas. Interview Q&A

These are the conceptual questions that get asked around the code — while you're typing, or in the five minutes after you finish. They're worth as many points as the solution. Answer each one out loud before you open it.

Why is tree recursion O(h) space and not O(1)? And what is h, exactly?

Every recursive call pushes a frame onto the call stack, and a frame isn't popped until its call returns. In a postorder DFS, the deepest chain of simultaneously-open frames is one per node on the current root-to-node path — so the peak stack depth is the height of the tree, h. That's O(h) auxiliary space even though you allocated nothing yourself.

h ranges from ⌊log₂ n⌋ (perfectly balanced — each level doubles, so you need only log n levels to hold n nodes) to n − 1 (fully skewed — every node has one child, and the tree is a linked list wearing a tree's clothes). This is why you should always write O(h) and then immediately say the range aloud, rather than writing O(log n): O(log n) is a claim about balance that the problem usually hasn't given you. The practical consequence — and the sentence that earns the point — is: "on a skewed tree of 10⁵ nodes, recursion would risk a stack overflow, so I'd convert to an explicit stack."

DFS or BFS — how do you actually decide, in one sentence?

Does the answer care about levels? If yes — "level by level", "per-level average", "zigzag", "right side view", "minimum depth", "shortest path" — use BFS, because the level structure is exactly what the frozen-queue-length loop hands you for free. If the answer at a node is built from its children's answers — depth, diameter, balance, validity, sums along paths — use DFS, because the call stack already gives you the postorder combine.

The cost tradeoff is the follow-up, so lead with it: DFS costs O(h) stack, BFS costs O(w) queue (w = the widest level, which is about n/2 for a complete tree). A deep skinny tree favors BFS; a wide shallow tree favors DFS. And note that BFS short-circuits for shallowest-answer questions — Minimum Depth via BFS stops at the first leaf and may never touch most of the tree, whereas DFS must explore everything.

Explain the "return one thing, track another" pattern. Which problems use it and why is it necessary?

It's needed whenever the quantity a parent needs from a child is different from the quantity the problem asks for. In Diameter, a parent needs the child's height (to compute its own height), but the answer is the maximum over all nodes of leftHeight + rightHeight — a different quantity entirely. So the recursion returns the height and accumulates the answer in a closure variable, updated at every node on the way up. One traversal, two jobs, O(n).

Without it you get the O(n²) trap: computing the answer at each node by calling a helper that re-walks that node's entire subtree, when the postorder pass already computed those values on its way up and threw them away. The family: Diameter (543), Balanced (110), Max Path Sum (124), Count Good Nodes (1448), House Robber III (337). The purist alternative is to return a tuple [buildingBlock, bestSoFar] and combine it at each level — same complexity, more ceremony. Either is accepted; know both and have a reason for your pick.

Why does checking left.val < node.val < right.val fail to validate a BST?

Because the BST property is a constraint imposed by all ancestors, not just the parent. The tree [5,4,6,null,null,3,7] passes every parent–child check — 4 < 5, 6 > 5, 3 < 6, 7 > 6 — and is not a BST, because 3 lives in 5's right subtree while being less than 5. Every node in a right subtree must exceed the ancestor you turned right at, however far above it that ancestor is.

The fix is to pass the accumulated constraint down: a running (low, high) window, where turning left tightens high to the current node's value and turning right tightens low — while the other bound is inherited unchanged. Resetting the inherited bound to ±Infinity re-introduces the exact bug. The equivalent formulation, worth knowing because it powers LC 99: an inorder traversal of a valid BST is strictly increasing, so you can just compare each value to the previously visited one.

Why is queue.shift() a problem in a JavaScript BFS?

Because JS arrays are index-contiguous: shift() removes the head and then re-indexes every remaining element, so it is O(n), not O(1). A BFS that dequeues with shift() therefore costs O(n) per dequeue and O(n²) overall on a wide tree — quietly, with a passing submission on small tests. The same applies to unshift() when you're building a row front-to-back (Zigzag): O(k) per insert, O(k²) per level.

Fixes: iterate the current level by index and swap in a fresh next array each round (what §2b does); or keep one array with a moving head pointer and never mutate the front. Volunteering this is disproportionately valuable for a frontend candidate — it shows you reason about the runtime you actually work in, not just about Big-O in the abstract.

A node counts as its own descendant. Why does that convention matter so much in LCA?

Because it decides the answer whenever one target is an ancestor of the other. Under the standard convention, LCA(5, 4) where 4 lives under 5 is 5, not 5's parent. Both LCA solutions on this page get this right structurally rather than by special-casing, and you should be able to point at the exact line:

  • BST (LC 235): when p.val === node.val, neither "both smaller" nor "both bigger" holds, so control falls into the else and returns node. The convention is free.
  • General (LC 236): if (root === p || root === q) return root; — you return without descending. If the other target is below, this node is still the LCA, and the parent will see exactly one non-null child and propagate it up unchanged.

Always confirm the convention with the interviewer before coding — it's a one-second question that prevents a wrong answer.

Which pairs of traversals uniquely determine a binary tree — and why?

Preorder + inorder ✓ and postorder + inorder ✓. In both cases one array identifies the root (preorder's first element, postorder's last) and inorder tells you how the remaining nodes split into left and right subtrees. Recurse.

Preorder + postorder ✗ — not in general. Given a node with exactly one child, neither traversal reveals whether that child is a left or a right child, so multiple trees produce the same pair. (It does work for full binary trees, where every node has 0 or 2 children — that's LC 889.) And any single traversal ✗, which is precisely why Serialize/Deserialize (297) must write down the nulls: a preorder walk with explicit null markers is sufficient on its own, because the markers restore the missing "where does this subtree end" information. Being able to state which pairs are sufficient — and why — is a strong, compact display of understanding.

You're given a BST and you write a correct O(n) traversal. Why might that still be a miss?

Because the interviewer handed you an invariant and you didn't use it. When a problem says "search tree", the thing being tested is whether you prune: at each node, one comparison tells you which side the answer must be on, so you throw the other subtree away and the cost drops from O(n) to O(h). LCA of a BST (235) is exactly this: an O(n) generic DFS passes every test case and still reads as a weaker answer than the six-line O(h) descent.

The habit to build: the moment you read "binary search tree", say "so inorder gives me sorted order, and one comparison discards half the tree." Then check whether either fact solves the problem — usually one of them does. Kth Smallest uses the first; LCA, search, insert, and Range Sum use the second.

How would you handle a tree so large that recursion overflows the stack?

Convert the recursion to an explicit stack. The call stack was only ever storing (node, phase) pairs, so you can store them yourself in a heap-allocated array — no depth limit beyond memory. Preorder is the easy one (push right, then left, so left pops first); inorder is §2c; postorder is the fiddly one (either push each node twice with a visited flag, or do a reversed preorder and reverse the output).

Say the numbers: JS engines typically blow up around 10⁴–10⁵ frames, so a balanced tree of a million nodes (h ≈ 20) is completely safe, while a skewed tree of 10⁵ nodes is not. That's the real risk criterion — height, not node count — and stating it that way is the senior version of the answer. For BFS the analogous concern is memory, not stack: the queue holds a whole level, which for a complete tree is ~n/2 nodes.

Where does any of this show up in real frontend work?

Directly, and you should say so — it's honest and it differentiates you. The DOM is a tree, and querySelectorAll, event bubbling (a walk up the ancestor chain), and every "find the closest matching parent" utility are tree traversals. React's reconciler diffs two element trees the way Same Tree (100) compares two trees — matching by type and position, bailing early on mismatch. Rendering a comment thread or a file explorer by depth is level-order traversal. SSR hydration is Serialize/Deserialize (297). Every AST tool you've used — Babel, ESLint, Prettier — is a DFS with a visitor at each node.

The reason to have this loaded isn't trivia: enterprise interviewers explicitly grade "connects the problem to real engineering experience", and a frontend candidate who can say "this is the same shape as reconciling two component trees" converts an abstract puzzle into evidence of seniority at zero cost.
