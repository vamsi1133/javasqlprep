08 — Heaps & Top-K

Priority access without full sorting — JS has no built-in heap, so you write one once and reuse it everywhere.

Why interviewers ask this

Heap problems are a staple of enterprise-tier loops — banks, fintech, healthcare, and large non-FAANG product companies (Optum, JPMC, Goldman Sachs and their peers) — because the domain is drowning in priority queues: order books matching the best bid first, job schedulers draining the highest-priority task, alerting systems surfacing the K most critical incidents, rate limiters and hospital triage queues. When a JPMC or Optum interviewer asks "find the K closest points" or "kth largest element," they're asking a sanitized version of a system they run in production, and they're grading one specific instinct: do you reach for an O(n log n) full sort when only K elements matter, or do you know the cheaper tool? At 7 YOE, "sort it and slice" is a yellow flag on a medium — not because it's wrong, but because it signals you don't know why it's wasteful.

There's a second, JavaScript-specific reason this topic earns its own tutorial: JS has no standard-library heap. Java has PriorityQueue, Python has heapq, C++ has std::priority_queue — a frontend candidate has nothing. Interviewers know this, and it quietly raises the bar: you either write a correct heap from scratch under pressure, or you confidently say "assume a MinHeap class with push/pop/peek — I'll implement it if you want" and keep moving. Both are fine; fumbling between them is not. This tutorial makes you write the class once, properly, so that in the room it's either muscle memory or a one-sentence assumption. And learn the recognition tell now: the phrase "top K" (or "K closest," "K most frequent," "kth largest") in a problem statement is a dead giveaway — the interviewer chose those words because they want a heap (or quickselect) and are waiting to see if you hear it.

1. Recognizing the pattern

Reach for a heap when the problem statement has any of these signals:

  • "Top K / smallest K / K closest / K most frequent" — the flagship phrasing. You need the K best elements by some score, and the other n−K elements' relative order is irrelevant. Paying O(n log n) to sort everything is answering a question nobody asked.
  • "Kth largest / kth smallest" — you need exactly one rank statistic. Heap of size K, or quickselect if the whole array is in memory and you may mutate it.
  • "Streaming" / "design a class with an add() method" — data arrives one element at a time and you must answer queries between arrivals. Sorting is off the table (you'd re-sort per element); a heap gives O(log n) per arrival. Kth Largest in a Stream and Find Median from a Data Stream are both this signal verbatim.
  • "Schedule / process by priority" / "always take the largest (or smallest) remaining" — repeated extract-max or extract-min with interleaved inserts. Task Scheduler, Last Stone Weight, meeting-room allocation, CPU scheduling. Any greedy algorithm whose greedy step is "grab the current extreme" runs on a heap.
  • "Merge K sorted things" — K sorted lists/arrays/iterators, and at each step you want the smallest of the K current heads. That's a K-element min-heap.

Sort vs. heap vs. quickselect — the decision table

Three tools answer "give me the extreme elements," and interviewers explicitly reward choosing among them out loud. The question to ask is: how much order do I actually need, and is the data all here or still arriving?

The sentence to say before writing any code: "I only need the K largest, not full order, so instead of sorting at O(n log n) I'll keep a min-heap capped at size K — O(n log K) time, O(K) space — and if you'd like I can compare it against quickselect." Thirty seconds, and you've demonstrated the entire decision table. At the enterprise tier this framing is frequently worth more than a perfect implementation, because most candidates jump straight to .sort().
"Top K" does not mean "use a size-K heap of the K candidates you like" — it means the heap's job is to evict. To keep the K largest you keep a min-heap (the weakest of your K champions sits on top, ready to be knocked out); to keep the K smallest you keep a max-heap. Candidates reflexively pair "largest" with "max-heap" and end up needing O(n) space. The inversion is counterintuitive exactly once — internalize it in §4 and it's yours forever.

2. Heap mechanics

A binary heap is a complete binary tree — every level full except possibly the last, which fills left to right — obeying one rule, the heap property: every parent is ≤ both of its children (min-heap), or ≥ both (max-heap). That's it. Note what it does not promise: siblings are unordered, the second-smallest element could be either child of the root, and an inorder walk gives you garbage. A heap is partially ordered, and that laziness is exactly why it's cheap: maintaining "the minimum is on top" costs O(log n), while maintaining "everything is in order" costs O(n log n).

2a. The array trick — there is no tree

Because the tree is complete, it has no holes, so you can flatten it into an array in level order and compute the links arithmetically instead of storing pointers. This is the whole implementation secret: a heap is a plain JavaScript array plus three index formulas. 

// For a node at index i (0-based):
parent(i) = (i - 1) >> 1   // integer divide by 2 — (i-1)/2 floored
left(i)   = 2 * i + 1
right(i)  = 2 * i + 2

//        1              index: 0  1  2  3  4  5
//      /   \            array: [1, 3, 2, 7, 4, 9]
//     3     2           parent(4) = (4-1)>>1 = 1  -> value 3  OK
//    / \   /            left(1)   = 3            -> value 7  OK
//   7   4 9
Use (i - 1) >> 1, not Math.floor((i - 1) / 2). Both are correct, but the shift is shorter, faster, and signals you know the array layout is powers-of-two arithmetic — same reason you write 2 * i + 1 inline rather than calling a helper. The one JS caveat: >> coerces to a 32-bit int, which is irrelevant here (array indices never approach 2³¹) but worth knowing before you reach for it elsewhere.

2b. Sift-up (push) vs sift-down (pop)

Every heap operation is "break the invariant in exactly one spot, then walk that violation along a single root-to-leaf path until it's fixed." There are only two directions to walk, and knowing which belongs to which operation is the mechanic:

  • Sift-up (bubble-up / heapify-up) — used by push. Append the new value at the end of the array — the only position that keeps the tree complete. It may outrank its parent, so compare with the parent and swap upward, repeating until it loses to its parent or it reaches index 0. You compare against exactly one node per level (the parent), so it costs at most height = ⌊log₂ n⌋ swaps.
  • Sift-down (bubble-down / heapify-down) — used by pop. The root is the answer, so you save it — but you can't leave a hole at index 0. Move the last element into the root (again: the only removal that preserves completeness) and let it sink: compare against both children, swap with the better one (the smaller, for a min-heap), repeat. Two comparisons per level, at most log n levels.
In _siftDown you must swap with the smaller of the two children, not with the first child that happens to beat the parent. Swap with the larger child in a min-heap and you've just made that child the parent of a smaller sibling — the heap property is still broken, silently, and your pop() starts returning wrong answers ten operations later. This is the #1 bug in hand-rolled heaps, and it's why the template computes best across both children before it swaps once.

2c. Why building a heap is O(n) but n pushes is O(n log n)

This is the most-asked heap theory question, and most candidates fumble it by hand-waving "n elements × log n each." Both statements are true — they are different algorithms, and the difference is which direction you sift:

  • n pushes = O(n log n). Every push appends at a leaf and sifts up. The path from a leaf to the root is the full height, log n. So the worst case genuinely is n × log n — push 1, 2, 3, … n into a min-heap in decreasing order and every single element travels all the way to the root.
  • Floyd's build-heap = O(n). Take the raw array as-is and sift-down every node from index ⌊n/2⌋−1 (the last node that even has a child) back to index 0. Now each node's cost is bounded by its height above the leaves, not the height of the whole tree — and a heap is overwhelmingly leaves. Half the nodes are leaves (height 0 → zero work). A quarter have height 1. An eighth have height 2. Total work ≈ Σh (n / 2h+1) × h, a series that converges to 2n. The one-liner to say: "almost all the nodes are near the bottom and barely move, so the cheap majority dominates and the sum telescopes to O(n)."

The practical upshot: if you're handed the whole array up front, heapify it in O(n). If elements arrive one at a time (a stream), you have no choice but to push and you pay O(log n) each. Either way, the pops still cost O(log n) apiece — so draining all n elements is O(n log n) no matter how you built it. That's heapsort, and it's exactly why "just heapify it" never beats sorting when you actually need everything in order.

"A heap is not a search structure" is a sentence that buys credibility. When the interviewer probes — "could you also delete an arbitrary element? get the 2nd smallest in O(1)? look up by id?" — the answer for a plain heap is no: deletion by value is O(n) because you must scan to find the index first. The real fixes are worth naming: keep a Map from value → index and sift both ways after the swap-out, or use lazy deletion (mark it dead in a Set, discard it when it surfaces at the top) — which is what production job schedulers and Dijkstra implementations actually do. Candidates who quietly assume a heap is a sorted array walk straight into this.

3. Build your MinHeap once

This is the centerpiece of the page. Java hands you PriorityQueue, Python hands you heapq, C++ hands you std::priority_queue — JavaScript hands you nothing, and no Node release has changed that. So you write this class once, drill it until it's typing rather than thinking, and paste it above every heap solution for the rest of your career. Forty lines, three minutes to type, and it is the single highest-leverage block of code a JavaScript candidate can commit to memory. 

class MinHeap {
  // compare(a, b) < 0  =>  a comes out FIRST. Default: numeric ascending (a true min-heap).
  constructor(compare = (a, b) => a - b) {
    this.data = [];             // the complete tree, flattened in level order
    this.compare = compare;
  }

  get size() { return this.data.length; }
  peek()     { return this.data[0]; }   // O(1) — the extreme always sits at index 0

  push(val) {
    this.data.push(val);                 // the end is the only slot that keeps the tree complete
    this._siftUp(this.data.length - 1);
  }

  pop() {
    if (this.data.length === 0) return undefined;
    const top  = this.data[0];          // this is the answer we hand back
    const last = this.data.pop();        // removing the LAST leaf keeps the tree complete
    if (this.data.length) {             // if it's empty now, `last` WAS the top — nothing to fix
      this.data[0] = last;              // plug the hole at the root...
      this._siftDown(0);               // ...and let the misfit sink to its rightful level
    }
    return top;
  }

  // Walk UP while this node outranks its parent. At most log n swaps (one compare per level).
  _siftUp(i) {
    while (i > 0) {
      const p = (i - 1) >> 1;                                  // parent index
      if (this.compare(this.data[i], this.data[p]) >= 0) break;  // parent still wins => done
      [this.data[i], this.data[p]] = [this.data[p], this.data[i]];
      i = p;
    }
  }

  // Walk DOWN, always swapping with the BETTER of the two children. At most log n swaps.
  _siftDown(i) {
    const n = this.data.length;
    while (true) {
      const l = 2 * i + 1, r = 2 * i + 2;
      let best = i;
      if (l < n && this.compare(this.data[l], this.data[best]) < 0) best = l;
      if (r < n && this.compare(this.data[r], this.data[best]) < 0) best = r;
      if (best === i) break;               // parent beats both kids => heap property restored
      [this.data[i], this.data[best]] = [this.data[best], this.data[i]];
      i = best;
    }
  }

  // O(n) bulk build (Floyd): sift down every NON-leaf, bottom-up. The leaf half needs no work.
  static heapify(arr, compare) {
    const h = new MinHeap(compare);
    h.data = arr;                                    // take ownership of the array — no copy, no n pushes
    for (let i = (arr.length >> 1) - 1; i >= 0; i--) h._siftDown(i);
    return h;
  }
}

The four things to notice, because each one is a question an interviewer can ask:

  • push appends then sifts up; pop swaps in the last leaf then sifts down. Both mutate the array only at positions that keep the tree complete — that's the invariant that makes the index arithmetic legal in the first place.
  • pop guards two cases: empty (return undefined) and size-1 (after data.pop() the array is empty, so there's nothing to sift). Get the second one wrong and you write undefined into data[0] and resurrect a ghost element. Interviewers do test pop on a one-element heap.
  • The comparator is injected, not hardcoded. That's what makes this one class serve every problem below.
  • heapify is static and O(n) — it exists so you can say "I'll build in O(n) rather than n pushes" and actually have the code to back it.

3a. The comparator trick: one class, both heaps

Notice there is no MaxHeap class anywhere on this page, and there never will be. A max-heap is a min-heap that lies about ordering. Flip the comparator and the same forty lines run backwards: 

const minHeap = new MinHeap();                          // smallest number pops first
const maxHeap = new MinHeap((a, b) => b - a);            // LARGEST number pops first

// By a field on an object (K Closest Points):
const byDist  = new MinHeap((a, b) => a.dist - b.dist);

// By a field in a [score, payload] tuple (Design Twitter — newest tweet first):
const byTime  = new MinHeap((a, b) => b[0] - a[0]);

// Multi-key: highest count first, alphabetical tiebreak (K Most Frequent Words)
const multi   = new MinHeap((a, b) => b.count - a.count || a.word.localeCompare(b.word));

Burn in the rule: compare(a, b) < 0 means "a pops before b." It is the exact contract you already know from Array.prototype.sort — sort's comparator decides which element lands earlier in the array; the heap's decides which element comes out earlier. Same mental model, same a - b / b - a muscle memory, same || chaining for tiebreaks. If you can write the sort comparator, you can write the heap comparator, and you never need a second class.

3b. Now the inversion — a min-heap for the K largest (as promised in §1)

Derive it; don't memorize it. You're the bouncer at a club with exactly K spots, and only the strongest K get in. A newcomer walks up. The only question you ever need to answer is: "is this newcomer stronger than the weakest person currently inside?" So the one person you must be able to see instantly — the one standing by the door, on top of the heap — is the weakest of your current champions. Weakest on top means min-heap. If the newcomer beats them, the weakling is thrown out and the newcomer takes the spot; if not, the newcomer walks away and the heap is untouched. 

// Keep the K LARGEST elements of a stream  =>  MIN-heap capped at K.
const heap = new MinHeap();          // top = weakest champion = the eviction candidate
for (const x of nums) {
  heap.push(x);
  if (heap.size > k) heap.pop();    // evict the weakest: it can never be in the top K again
}
// heap now holds exactly the K largest, and heap.peek() IS the Kth largest.

And the mirror image: to keep the K smallest, keep a max-heap, whose top is the worst (largest) of your K survivors — the one to evict the moment something smaller arrives. One sentence covers both directions forever: the top of the heap is always the element you are most willing to throw away. Say that out loud in the room and you will never re-derive the inversion under pressure again.

The cap check must be > k, not >= k, and it must come after the push. Push-then-evict lets the heap hold K+1 elements for one instant, which is what makes the newcomer compete fairly against the sitting weakling — the loser of that comparison is whatever ends up on top, and it gets popped. Check >= k before pushing and you evict a champion to make room for a newcomer that may be worse than everyone. (The "clever" variant — if (heap.size < k) heap.push(x); else if (x > heap.peek()) { heap.pop(); heap.push(x); } — is also correct and slightly faster, since it skips the push for hopeless elements. Write the three-line version, then mention the optimization.)
Why this is O(n log K) and not O(n log n): the heap never exceeds K elements, so every push and pop walks a tree of height log K, not log n. With K = 10 and n = 10⁹ — a stream you could not store even if you wanted to — log K ≈ 3.3 and your memory footprint is ten slots. That contrast ("this is the only approach that survives an input which doesn't fit in RAM") is exactly the follow-up an Optum or JPMC interviewer is holding in reserve, and it's the reason they asked a Top-K question in the first place. Volunteer it.

4. Warm-ups

Two problems, twenty minutes, and the two comparator directions are yours. Do not skip them because they're rated easy — the size-K min-heap in the first one is the exact skeleton you'll reuse in K Closest Points and Kth Largest in an Array, and the second one is the shortest possible drill for "max-heap = flipped comparator."

Kth Largest Element in a Stream (LC #703) EASY

Design a class KthLargest that tracks the k-th largest element in a stream of numbers. The constructor takes k and an initial array nums. Each call to add(val) appends val to the stream and returns the k-th largest element seen so far — k-th largest by value with duplicates counted separately, i.e. the k-th element in sorted-descending order, not the k-th distinct value. The problem guarantees there are always at least k elements by the time you're asked.

Example 1: k = 3, nums = [4, 5, 8, 2]. Then add(3) → 4 (stream sorted desc: 8,5,4,3,2 — third is 4); add(5) → 5 (8,5,5,4,3,2 — third is 5); add(10) → 5 (10,8,5,5,4,3,2 — third is 5); add(9) → 8; add(4) → 8.
Example 2: k = 1, nums = []. add(-3) → -3; add(-2) → -2 — with k = 1 this degenerates to "running maximum," which is a useful sanity check on your code and a hint that the top of the heap is the answer.
Constraints: 1 ≤ k ≤ 10⁴; 0 ≤ nums.length ≤ 10⁴; −10⁴ ≤ val ≤ 10⁴; at most 10⁴ calls to add; the stream always has ≥ k elements when add returns.

Solve on LeetCode ↗

Thinking process. The brute force is honest and instructive: keep every number in an array, and on each add, sort descending and return index k−1. That's O(n log n) per call — with 10⁴ calls you're re-sorting a growing array ten thousand times, which is O(n² log n) overall. Now ask the question that collapses it: which of those numbers can ever be the answer? Only the top k. The 4000th largest number you ever saw can never become the 3rd largest — new arrivals only push elements down the ranking, never up. So everything below rank k is dead weight and you may discard it permanently. Keep exactly k elements, and make the one you can see in O(1) be the smallest of them — because the smallest of your top-k is the k-th largest, and it's also the element you'd evict when something better arrives. Min-heap capped at k. Each add is one push plus maybe one pop: O(log k). This is the club-bouncer inversion from §3b, and it is why this "easy" problem is worth doing carefully — it's the template for three mediums.

Solution (JS) — O(log k) per add · O(k) space 
// (assumes the MinHeap class from §3 is in scope)
class KthLargest {
  constructor(k, nums) {
    this.k = k;
    this.heap = new MinHeap();      // MIN-heap: its top is the WEAKEST of our k champions
    for (const n of nums) this.add(n);  // reuse add() — no duplicated logic, and it self-trims
  }

  add(val) {
    this.heap.push(val);                     // let the newcomer in unconditionally...
    if (this.heap.size > this.k) {
      this.heap.pop();                       // ...then throw out whoever is now weakest
    }
    // The heap holds the k largest; its minimum IS the kth largest.
    return this.heap.peek();
  }
}

Trace (k = 3, nums = [4, 5, 8, 2]): the constructor calls add four times — heap grows [4], [4,5], [4,5,8] (size 3, no evictions yet), then add(2) pushes 2 → size 4 → pop evicts 2 → heap = {4, 5, 8}, top = 4. Now add(3): push 3, size 4, pop evicts 3 (it's the weakest), heap = {4,5,8}, peek → 4 ✓. add(5): push 5, size 4, pop evicts 4, heap = {5,5,8}, peek → 5 ✓. add(10): push 10, pop evicts 5, heap = {5,8,10}, peek → 5 ✓. add(9): push 9, pop evicts 5, heap = {8,9,10}, peek → 8 ✓. add(4): push 4, it's immediately the weakest, pop evicts it, heap unchanged, peek → 8 ✓.

Complexity: add does at most one push and one pop on a heap of size ≤ k+1 → O(log k) time. The constructor calls add n times → O(n log k). Space is O(k) — not O(n), and that's the entire point: the class can run forever on an infinite stream with a fixed memory footprint. Say the space bound out loud; it's the reason the problem says "stream."

Follow-ups you should expect:

  • "What if k is close to n?" Then log k ≈ log n and the heap's advantage evaporates; a sorted structure is fine. The heap wins when k ≪ n — say so rather than pretending it's always optimal.
  • "Now give me the k-th smallest in a stream." Same code, one character: new MinHeap((a,b) => b - a). Keep the k smallest in a max-heap whose top is the worst survivor.
  • "Now the running median." That's LC 295 — two heaps. It's the last problem on this page, and this follow-up is exactly how interviewers escalate from easy to hard inside one 45-minute slot.
  • "Could you avoid the push when the value is hopeless?" Yes: if (size < k) push; else if (val > peek()) { pop(); push(val); }. Same big-O, fewer operations.
Reusing this.add(n) inside the constructor isn't just tidy — it's the version that survives an empty nums. Candidates who write a separate "bulk load then trim" path frequently break on nums = [] with k = 1. Reuse the method that already handles one element correctly. (If you want the O(n) build instead: MinHeap.heapify(nums.slice()), then pop until size === k. Faster asymptotically, more code, more places to slip — mention it, don't lead with it.)

Last Stone Weight (LC #1046) EASY

You have an array of stone weights. Each turn, take the two heaviest stones, x and y with x ≤ y, and smash them together: if they weigh the same, both are destroyed; if not, the lighter one is destroyed and the heavier one is left with weight y − x (and goes back in the pile). Repeat until at most one stone remains. Return the weight of the last stone, or 0 if none is left.

Example 1: Input stones = [2,7,4,1,8,1] → Output 1. Smash 8 and 7 → 1 remains, pile = [2,4,1,1,1]. Smash 4 and 2 → 2 remains, pile = [2,1,1,1]. Smash 2 and 1 → 1, pile = [1,1,1]. Smash 1 and 1 → both destroyed, pile = [1]. One stone left: 1.
Example 2: Input stones = [1] → Output 1 — no smash ever happens; the loop must not run. This is the edge case that breaks naive while (true) code.
Example 3: Input stones = [2,2] → Output 0 — equal stones annihilate and the pile empties, so you must handle "the heap is empty" without indexing into undefined.
Constraints: 1 ≤ stones.length ≤ 30; 1 ≤ stones[i] ≤ 1000.

Solve on LeetCode ↗

Thinking process. The simulation is forced on you by the problem — there is no clever closed form (the "can I partition the stones into two near-equal halves?" reformulation is a different problem, LC 1049, and it's DP). So the only question is: what structure answers "give me the current maximum, and let me insert a new value" fastest? Sorting the array every turn is O(n log n) per turn. Scanning for the max twice is O(n) per turn. Repeated extract-max with interleaved inserts is the literal job description of a max-heap: pop, pop, maybe push, repeat, at O(log n) a pop. With n ≤ 30 every approach passes, which is exactly why this problem is a signal question — the interviewer isn't testing your throughput, they're testing whether you name the right structure when the constraint doesn't force it. Do it with a heap and say why.

Solution (JS) — O(n log n) time · O(n) space 
var lastStoneWeight = function(stones) {
  // MAX-heap = MinHeap with an inverted comparator. No second class, ever.
  const heap = MinHeap.heapify(stones.slice(), (a, b) => b - a);   // O(n) build; slice() = don't mutate input

  // Need TWO stones to smash. size <= 1 ends the game — this guards stones = [1].
  while (heap.size > 1) {
    const y = heap.pop();    // heaviest
    const x = heap.pop();    // second heaviest, so x <= y automatically
    if (y !== x) {
      heap.push(y - x);     // survivor rejoins the pile; equal stones just vanish
    }
  }

  // pop() on an empty heap returns undefined, so fall back to 0 explicitly.
  return heap.size === 1 ? heap.peek() : 0;
};

Trace on [2,7,4,1,8,1]: heapify → max-heap with 8 on top. Pop 8, pop 7, unequal → push 1. Heap now {4,2,1,1,1}. Pop 4, pop 2 → push 2. Heap {2,1,1,1}. Pop 2, pop 1 → push 1. Heap {1,1,1}. Pop 1, pop 1 → equal, nothing pushed. Heap {1}, size 1 → loop exits → return 1 ✓. On [2,2]: pop 2, pop 2, equal, nothing pushed; heap is empty → return 0 ✓. On [1]: the loop body never runs → return 1 ✓.

Complexity: the O(n) heapify is free next to the loop. Each iteration removes at least one stone permanently (two pops, at most one push), so there are at most n iterations, each costing O(log n) → O(n log n) time. Space O(n) for the heap (O(1) extra if you're allowed to heapify the input array in place, which MinHeap.heapify does by design — drop the .slice() and say so).

Follow-ups you should expect:

  • "Why not just sort each round?" Because you only need the top two, not full order, and you re-insert after every smash — that's the exact access pattern (extract-max + insert) a heap is built for. Sorting per round is O(n² log n) overall.
  • "n ≤ 30 — wouldn't a sorted array be simpler?" Yes, and you should acknowledge it: at n = 30, sort() + pop() from the tail is genuinely fine and shorter. The heap is the answer that scales, and naming both is the senior move.
  • "What if the stones smashed were the two lightest?" One character: the default comparator. That's Huffman coding's inner loop, and it's worth saying — Huffman is exactly "repeatedly merge the two smallest," a min-heap.
  • "Minimize the last stone's weight by choosing which stones to smash." Different problem entirely (LC 1049, Last Stone Weight II) — it's a subset-sum DP, not greedy. Recognizing that the greedy heap answer is wrong there is a strong signal.
This problem is where you should point out that greedy simulation runs on heaps. Whenever a problem says "repeatedly take the largest/smallest remaining, do something, put the result back," the algorithm is a heap loop — Huffman encoding, CPU scheduling, merging files with minimum cost, IPO/capital problems, Task Scheduler later on this page. You're not learning a stone puzzle; you're learning the shape of every greedy-with-reinsertion problem you'll ever see.

K Closest and Kth Largest are the two most common heap mediums in existence; Task Scheduler is the greedy one that separates people who simulate from people who reason; Design Twitter is what happens when a heap shows up inside a design round, which is the shape a lot of enterprise loops prefer over pure algorithm puzzles.

K Closest Points to Origin (LC #973) MEDIUM ★

You're given an array points where points[i] = [xᵢ, yᵢ] represents a point on the X-Y plane, and an integer k. Return the k points closest to the origin (0, 0), measured by ordinary Euclidean distance, √(x² + y²). You may return the answer in any order, and the answer is guaranteed to be unique except for the order of the points themselves.

Example 1: Input points = [[1,3],[-2,2]], k = 1 → Output [[-2,2]]. Distances are √10 ≈ 3.16 and √8 ≈ 2.83, so (−2,2) is closer.
Example 2: Input points = [[3,3],[5,-1],[-2,4]], k = 2 → Output [[3,3],[-2,4]] (or [[-2,4],[3,3]] — order is free). Squared distances are 18, 26, 20; the two smallest are 18 and 20.
Constraints: 1 ≤ k ≤ points.length ≤ 10⁴; −10⁴ ≤ xᵢ, yᵢ ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. Brute force: compute every distance, sort by it, take the first k. That's correct and O(n log n), and if you stop there on a medium at 7 YOE you've left points on the table — you sorted 10⁴ elements to answer a question about k of them. The collapse is the mirror of §3b: you want the k smallest, so you keep a max-heap capped at k, whose top is the farthest of your current k finalists — the one to evict the moment something closer shows up. Every point costs one push and maybe one pop on a heap of size ≤ k+1 → O(n log k) time, O(k) space, and it works on a stream of points that never fits in memory.

Then there's the detail this problem exists to test: you never need the square root. Distance is √(x² + y²), but √ is monotonically increasing — if d₁ < d₂ then √d₁ < √d₂ and vice versa. Since you only ever compare distances and never report one, ranking by x² + y² gives an identical ordering. Dropping Math.sqrt removes n floating-point operations and, more importantly, removes floating-point error from your comparator — with integer inputs, x² + y² is an exact integer, so ties are exact and there's no ε-comparison anywhere. Saying "I'll compare squared distances, since sqrt is monotonic and I never need the actual magnitude" is a free senior signal, and it generalizes: any time you sort or compare by a monotonic transform, apply the comparison before the transform.

Solution (JS) — O(n log k) time · O(k) space 
var kClosest = function(points, k) {
  // We want the k SMALLEST distances => MAX-heap, so the top is the FARTHEST
  // of our current finalists — i.e. the one we're most willing to evict.
  // Heap items are [sqDist, point]; comparator: bigger sqDist pops first.
  const heap = new MinHeap((a, b) => b[0] - a[0]);

  for (const [x, y] of points) {
    const d = x * x + y * y;      // NO Math.sqrt — sqrt is monotonic, so it can't change the ranking
    heap.push([d, [x, y]]);
    if (heap.size > k) heap.pop();  // evict the farthest: it can never be in the closest k now
  }

  // Whatever survived IS the answer; order is unspecified, so just drain it.
  return heap.data.map(([, p]) => p);
};

Trace on [[3,3],[5,-1],[-2,4]], k = 2: push [18,(3,3)] → heap {18}. Push [26,(5,−1)] → heap top is 26 (max-heap), size 2, no eviction. Push [20,(−2,4)] → size 3 > 2 → pop removes the top, which is the largest squared distance, 26 → (5,−1) is evicted. Heap holds {18 → (3,3), 20 → (−2,4)} → return [[3,3],[-2,4]] ✓. Notice the eviction worked precisely because the top was the worst finalist.

Complexity: n iterations, each doing O(log k) heap work → O(n log k) time. The heap never holds more than k+1 items → O(k) space. Compare against the sort baseline: O(n log n) time, O(n) space. With n = 10⁴ and k = 5, log k ≈ 2.3 versus log n ≈ 13 — roughly 5× fewer comparisons, and 2000× less memory.

Follow-ups you should expect:

  • "Can you do better than O(n log k)?" Yes — quickselect on squared distance gives O(n) average: partition around a pivot until the k-th smallest lands at index k−1, then return points.slice(0, k). Worst case O(n²), needs the whole array in memory. Full treatment in the next problem.
  • "The points arrive as a stream of a billion — what changes?" Nothing. That's the entire selling point of the size-k heap: O(k) memory, one pass, no random access. Quickselect and sorting both die here.
  • "Now closest to an arbitrary point (a, b), not the origin." Change one line: d = (x-a)**2 + (y-b)**2. Still no sqrt.
  • "Return them sorted by distance." Drain the heap and reverse (a max-heap drains farthest-first), or sort the k survivors — O(k log k) on top, which is negligible.
Do not store the point and recompute x*x + y*y inside the comparator. It looks tidy and it re-does the multiplication on every comparison — O(n log k) recomputations instead of n. Compute the score once, store [score, payload], and compare on [0]. This "decorate, then compare" habit (the Schwartzian transform) is the same reason you precompute sort keys in a React list rather than deriving them in the comparator on every render.

Kth Largest Element in an Array (LC #215) MEDIUM ★

Given an unsorted integer array nums and an integer k, return the k-th largest element in the array. This is the k-th largest in sorted order, not the k-th distinct element — duplicates each occupy their own rank. The problem explicitly asks: can you solve it without sorting?

Example 1: Input nums = [3,2,1,5,6,4], k = 2 → Output 5. Sorted descending: 6,5,4,3,2,1 — the 2nd is 5.
Example 2: Input nums = [3,2,3,1,2,4,5,5,6], k = 4 → Output 4. Sorted descending: 6,5,5,4,3,3,2,2,1 — the 4th is 4. Note the duplicate 5s each consume a rank; a "distinct values" reading would wrongly give 3.
Constraints: 1 ≤ k ≤ nums.length ≤ 10⁵; −10⁴ ≤ nums[i] ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. Baseline: nums.sort((a,b) => b - a)[k-1] — one line, O(n log n), and completely correct. Say it out loud as your baseline (never pretend you don't see it), then immediately say why it's more than the problem needs: it produces a total order when you asked for a single rank. Two better tools exist, and the interview is watching you choose between them and know their costs:

  • Size-k min-heap — the club bouncer from §3b, verbatim. O(n log k) time, O(k) space, works on streams, deterministic, never mutates the input. This is the answer you give first.
  • Quickselect — quicksort that only recurses into the half that contains the answer. O(n) average time, O(1) extra space, but O(n²) worst case, and it destroys the input's order.

The quickselect insight is worth deriving properly, because it's the same partition step you already know from quicksort. Pick a pivot and partition the array so everything ≥ pivot is on the left and everything < pivot is on the right. Now the pivot lands at some final index p — and that index is its true rank, forever, regardless of how unsorted both halves still are. If p === k - 1, the pivot is your answer and you're done. If p > k - 1, the k-th largest is somewhere in the left half; recurse there and throw the right half away without ever looking at it. That discard is where the speedup comes from: quicksort recurses into both halves (n + n/2·2 + … = n log n), quickselect recurses into one, so the work is n + n/2 + n/4 + … = 2n = O(n) on average. Beautiful — and fragile.

Be honest about quickselect's worst case, out loud, before the interviewer raises it: if your pivot is always the extreme element (which happens on an already sorted array if you always pick the last element), each partition shrinks the range by exactly one and you get n + (n−1) + (n−2) + … = O(n²). The mitigation is a random pivot, which makes the O(n²) case require adversarial luck rather than merely sorted input — and you must actually write the randomization, not just mention it. The bulletproof fix, median-of-medians, guarantees O(n) worst case but has a constant so large it's slower in practice; naming it and rejecting it is exactly the right level of depth here. The heap answer is the safe one to give first; offer quickselect as the optimization, and let the interviewer ask for it.
Solution 1 (JS) — heap · O(n log k) time · O(k) space 
var findKthLargest = function(nums, k) {
  // k LARGEST => MIN-heap of size k. Its top is the weakest champion,
  // and once we've seen everything, the weakest champion IS the kth largest.
  const heap = new MinHeap();

  for (const x of nums) {
    heap.push(x);
    if (heap.size > k) heap.pop();   // evict the weakest — it's out of the running for good
  }

  return heap.peek();
};

Trace on [3,2,1,5,6,4], k = 2: push 3 → {3}. Push 2 → {2,3}, size 2, keep. Push 1 → size 3, pop evicts 1 → {2,3}. Push 5 → size 3, pop evicts 2 → {3,5}. Push 6 → pop evicts 3 → {5,6}. Push 4 → pop evicts 4 (it's weaker than 5) → {5,6}. peek() → 5 ✓ — the smallest of the two largest is the 2nd largest.

Complexity: n pushes and up to n pops on a heap of size ≤ k+1 → O(n log k). Space O(k). Note that when k = n this degrades to O(n log n) — no worse than sorting, and the answer is then just the array minimum, which is worth flagging as a sanity check on your reasoning.

Solution 2 (JS) — quickselect · O(n) average, O(n²) worst · O(1) space 
var findKthLargest = function(nums, k) {
  // The kth LARGEST sits at index k-1 once the array is sorted DESCENDING.
  // So partition descending (big on the left) and hunt for final index k-1.
  const target = k - 1;
  let lo = 0, hi = nums.length - 1;

  // Iterative, not recursive: no call-stack growth, so space is truly O(1).
  while (true) {
    const p = partition(nums, lo, hi);
    if (p === target) return nums[p];   // pivot landed on its FINAL rank — that's the answer
    if (p < target) lo = p + 1;         // answer is to the RIGHT; discard everything left of p
    else            hi = p - 1;         // answer is to the LEFT;  discard everything right of p
  }
};

// Lomuto partition, DESCENDING: everything >= pivot ends up left of the pivot.
// Returns the pivot's final index, which is its true rank in the array.
function partition(nums, lo, hi) {
  // RANDOM pivot. Without this, a sorted input degrades to O(n^2) — the classic trap.
  const r = lo + Math.floor(Math.random() * (hi - lo + 1));
  [nums[r], nums[hi]] = [nums[hi], nums[r]];      // park the pivot at the end, out of the way

  const pivot = nums[hi];
  let store = lo;                                // boundary: nums[lo..store-1] are all >= pivot

  for (let i = lo; i < hi; i++) {
    if (nums[i] > pivot) {                      // strict > keeps duplicates from all piling onto one side
      [nums[i], nums[store]] = [nums[store], nums[i]];
      store++;
    }
  }

  [nums[store], nums[hi]] = [nums[hi], nums[store]];  // pivot slots into its final home
  return store;
}

Trace on [3,2,1,5,6,4], k = 2 (target index 1): suppose the random pivot picks 4. Partition descending → everything > 4 goes left: [5,6,4,3,2,1]-ish, with 4 landing at index 2. Is 2 === 1? No, and 2 > 1, so the answer is to the left: set hi = 1 and never look at indices 3–5 again — that discard is the whole algorithm. Partition [5,6]: say the pivot is 5; 6 > 5 so 6 moves left, 5 lands at index 1. Is 1 === 1? Yes → return nums[1] = 5 ✓.

Complexity: each partition is O(range). On average a random pivot splits the range roughly in half, so the total work is n + n/2 + n/4 + … = 2n = O(n) average. Worst case — every pivot is the extreme — gives n + (n−1) + … + 1 = O(n²); the random pivot makes that vanishingly unlikely rather than impossible. Space is O(1): the loop is iterative and the partition is in place. The costs you must state: it mutates the caller's array, it needs the whole array in memory (no streaming), and its runtime is randomized.

Follow-ups you should expect:

  • "Which would you ship?" The honest answer, and the one that scores: "heap, unless profiling says otherwise — O(n log k) with a tiny constant, no worst case, no mutation, and it survives a stream. Quickselect is faster on average but I'd only take a randomized O(n²) tail risk in a hot path I've measured."
  • "Guarantee O(n) worst case." Median-of-medians pivot selection (the BFPRT algorithm) — provably O(n), but the constant is bad enough that it loses to the randomized version in practice. Name it, explain the tradeoff, move on. Nobody expects you to code it.
  • "Now the k-th smallest." Flip the partition comparison to <, or just call the same code with k = n - k + 1. Do the substitution out loud; off-by-ones here are the most common failure.
  • "nums has 10⁹ elements across 100 machines." Now neither works as written: this is a distributed top-k — each machine computes its local top-k with a heap, ships k elements to a coordinator, and the coordinator merges 100 sorted lists of k. That's Merge K Sorted Lists, later on this page. Enterprise interviewers love this escalation because it's a real system.

Task Scheduler (LC #621) MEDIUM ★

You're given an array tasks of CPU tasks, each labeled with an uppercase letter, and an integer n — the cooldown: two identical tasks must be separated by at least n intervals. Each interval, the CPU either runs one task or sits idle. Tasks may be run in any order. Return the minimum number of intervals needed to finish every task.

Example 1: Input tasks = ["A","A","A","B","B","B"], n = 2 → Output 8. A valid schedule: A → B → idle → A → B → idle → A → B. Each A is 3 apart (≥ n+1 = 3) ✓. There aren't enough distinct tasks to fill the gaps, so the CPU idles twice.
Example 2: Input tasks = ["A","C","A","B","D","B"], n = 1 → Output 6 — A → B → C → D → A → B. Zero idles: there was enough variety to pack the gaps, so the answer is simply tasks.length. This is the case that kills naive formulas.
Example 3: Input tasks = ["A","A","A","B","B","B"], n = 0 → Output 6 — no cooldown at all, so nothing idles.
Constraints: 1 ≤ tasks.length ≤ 10⁴; tasks[i] is an uppercase English letter; 0 ≤ n ≤ 100.

Solve on LeetCode ↗

Thinking process. First, notice the labels are irrelevant — only the counts matter. Swap every "A" for every "B" and the answer doesn't change. So the input is really a multiset of frequencies.

Now the greedy. At each interval you want to run the task with the highest remaining count that isn't on cooldown. Why is greedy correct? Because the bottleneck is the most frequent task: it's the one that will still be pending when everything else is exhausted, so every interval you spend not reducing it is an interval you may have to pay for as an idle later. Draining the biggest first keeps the frequency distribution as flat as possible, and a flat distribution is exactly one with no forced idles. "Always take the largest remaining" + "put it back with a smaller count" = max-heap with re-insertion, the shape from Last Stone Weight.

The implementation trick that makes it clean: process in cycles of n+1 intervals. In one cycle you pop up to n+1 distinct tasks off the max-heap, run one of each, and only after the whole cycle do you push the still-nonzero ones back. Holding them out of the heap for the duration of the cycle is the cooldown — you've made it structurally impossible to schedule the same task twice within n+1 slots. If the heap runs dry mid-cycle, the remaining slots are idles, and you pay for them only if tasks still remain (a partial final cycle costs just the tasks you ran, not the padding).

Solution 1 (JS) — greedy max-heap · O(n_tasks · log 26) ≈ O(n_tasks) time · O(1) space 
var leastInterval = function(tasks, n) {
  // Labels don't matter — only counts do. At most 26 of them.
  const counts = new Map();
  for (const t of tasks) counts.set(t, (counts.get(t) ?? 0) + 1);

  // MAX-heap of remaining counts: always run the most-pending task first.
  const heap = MinHeap.heapify([...counts.values()], (a, b) => b - a);

  let time = 0;

  while (heap.size > 0) {
    // One CYCLE = n+1 slots. Running each task at most once per cycle IS the cooldown.
    const ranThisCycle = [];

    for (let slot = 0; slot <= n; slot++) {   // slot = 0..n  =>  exactly n+1 slots
      if (heap.size > 0) {
        const c = heap.pop() - 1;              // run one instance of the most-pending task
        if (c > 0) ranThisCycle.push(c);      // held OUT of the heap until the cycle ends
        time++;                                // a real task ran
      } else if (ranThisCycle.length > 0) {
        time++;                                // heap is dry but work remains => forced IDLE
      }
      // If the heap is dry AND nothing is coming back, we're done — don't pad the tail.
    }

    for (const c of ranThisCycle) heap.push(c);  // cooldown served; they're eligible again
  }

  return time;
};

Trace on ["A","A","A","B","B","B"], n = 2: counts = {A:3, B:3}; max-heap {3,3}. Cycle 1 (3 slots): pop 3 (an A) → 2, held; time = 1. Pop 3 (a B) → 2, held; time = 2. Slot 3: heap is empty but ranThisCycle has entries → idle, time = 3. Push back {2,2}. Cycle 2: same shape → time = 6, push back {1,1}. Cycle 3: pop 1 → 0, not held, time = 7; pop 1 → 0, time = 8; slot 3: heap empty and ranThisCycle is empty → no idle charged. Loop exits. Return 8 ✓. On Example 2 the heap always has a distinct task ready, no slot ever idles, and the answer is exactly 6 = tasks.length ✓.

Complexity: every heap operation is on ≤ 26 elements, so each is O(log 26) = O(1) for practical purposes. Total heap operations are bounded by the number of task instances plus idles → the loop runs O(answer) times, and the answer is O(tasks.length + idles). Treat it as O(N · log 26) ≈ O(N) time and O(1) space (at most 26 counts, regardless of input size). Stating "the alphabet is bounded at 26, so the heap is effectively O(1)" is the observation that turns a good answer into a strong one.

The follow-up: the O(1) math formula. The simulation is what you write first because it's obviously correct. But the interviewer will very often ask "can you compute it without simulating?" — and there's a beautiful closed form. Think of the schedule as a grid. Take the most frequent task, with count maxFreq. It must appear maxFreq times, and consecutive appearances must be ≥ n+1 apart, which carves the timeline into maxFreq − 1 full blocks of width n + 1, plus a final partial row: 

// tasks = AAAABBBBCCD, n = 2   =>  maxFreq = 4 (A and B), countOfMax = 2

//   |<-- n+1 = 3 -->|
//     A   B   C          <-- block 1  \
//     A   B   C          <-- block 2   |  (maxFreq - 1) = 3 blocks of width (n+1)
//     A   B   D          <-- block 3  /
//     A   B               <-- the final row: one slot per task tied at maxFreq

// length = (maxFreq - 1) * (n + 1) + countOfMax = 3 * 3 + 2 = 11

Every block is n + 1 wide (either filled with other tasks or left idle), there are maxFreq − 1 of them, and then the final row holds one slot for each task tied at maxFreq. So the schedule length is (maxFreq − 1) × (n + 1) + countOfMax. The one catch: if there are so many distinct tasks that they overflow the blocks, there are no idles at all and the answer is simply tasks.length. The formula can under-count in that case, so you take the max of the two:

Solution 2 (JS) — the O(1) formula · O(N) time · O(1) space 
var leastInterval = function(tasks, n) {
  const counts = new Array(26).fill(0);
  for (const t of tasks) counts[t.charCodeAt(0) - 65]++;   // 65 = 'A'

  const maxFreq    = Math.max(...counts);
  const countOfMax = counts.filter(c => c === maxFreq).length;  // how many tasks TIE for most frequent

  // The frame built by the bottleneck task: (maxFreq-1) blocks of width (n+1),
  // plus one final row holding every task tied at maxFreq.
  const framed = (maxFreq - 1) * (n + 1) + countOfMax;

  // If there's enough VARIETY to fill every gap, no idle ever happens and the
  // answer is just the task count. The frame can undercount, so take the max.
  return Math.max(framed, tasks.length);
};

Trace on Example 1 (AAABBB, n = 2): maxFreq = 3, countOfMax = 2 (A and B tie), framed = (3−1)×3 + 2 = 8; tasks.length = 6 → max = 8 ✓. Example 2 (ACABDB, n = 1): maxFreq = 2, countOfMax = 2 (A and B), framed = 1×2 + 2 = 4; tasks.length = 6 → max(4, 6) = 6 ✓ — this is precisely the case the Math.max exists to rescue, and it's the line candidates omit. Example 3 (n = 0): framed = 2×1 + 2 = 4, tasks.length = 6 → 6 ✓.

Complexity: one pass to count, one pass over 26 buckets → O(N) time, O(1) space, and — unlike the heap — no simulation, so the constant is tiny.

Follow-ups you should expect:

  • "Prove the greedy is correct." The most frequent task is the bottleneck: it alone forces (maxFreq−1)×(n+1) + countOfMax intervals no matter what else you do, so that's a hard lower bound. The greedy achieves it (it never idles when a distinct task is available), so it's optimal. Lower bound + achievability = proof; that structure is what interviewers want when they say "prove it."
  • "Now output the actual schedule, not just the length." The formula can't — you need the heap simulation, and you record which task you popped in each slot. This is exactly why you write the heap version first: it generalizes, the formula doesn't.
  • "Tasks now have different durations / priorities / deadlines." The closed form dies immediately; the heap survives with a richer comparator. Real schedulers (and this is the JPMC/Optum batch-job flavor of the question) are always the heap.
  • "Why Math.max(framed, tasks.length)?" Because with enough distinct tasks the blocks overflow and there are zero idles — the frame is a lower bound on length, not the length itself.
Give both solutions, in this order: heap first (obviously correct, generalizes, produces a real schedule), then the formula ("and if you only need the length, there's an O(1) counting argument…"). Leading with the formula is a trap — it looks like memorization, and the moment the interviewer perturbs the problem ("now print the schedule", "now tasks have durations") you have nothing. Deriving the simulation and then collapsing it to arithmetic is the exact narrative a 7-YOE candidate is being graded on.

Design Twitter (LC #355) MEDIUM

Design a simplified Twitter. Users post tweets, follow and unfollow each other, and can see the 10 most recent tweets in their news feed. Implement:

  • postTweet(userId, tweetId) — compose a new tweet.
  • getNewsFeed(userId) — return the IDs of the 10 most recent tweets in this user's feed, most recent first. The feed contains tweets posted by the user and by everyone they follow.
  • follow(followerId, followeeId) · unfollow(followerId, followeeId).

Example: postTweet(1, 5); getNewsFeed(1) → [5] (your own tweets are always in your feed). follow(1, 2); postTweet(2, 6); getNewsFeed(1) → [6, 5] — tweet 6 is newer, so it comes first. unfollow(1, 2); getNewsFeed(1) → [5].
Example 2 (the trap): follow(1, 1) then postTweet(1, 5) then getNewsFeed(1) → [5], not [5, 5] — a user following themselves must not see their tweets twice. And unfollow(1, 1) must not remove your own tweets from your feed.
Constraints: 1 ≤ userId, tweetId ≤ 10⁴; at most 3×10⁴ calls total.

Solve on LeetCode ↗

Thinking process. Start with the data model, out loud, before any algorithm: a Map from userId → an append-ordered array of that user's tweets (so each user's own list is already sorted newest-last for free), a Map from userId → a Set of followees, and a single global monotonically increasing counter that timestamps every tweet. The counter is the crux — the only way to order tweets across different users is a global clock, and candidates who store only tweetIds discover in the middle of getNewsFeed that they have no way to interleave two users' lists.

Now the feed. Brute force: concatenate every followee's entire tweet list, sort by timestamp descending, take 10. That's O(T log T) where T is every tweet ever posted by everyone you follow — a user following 500 accounts with 10⁵ tweets between them re-sorts 10⁵ items to return ten. The collapse: each user's list is already sorted. You have k sorted lists and want the 10 largest across them — that is k-way merge, exactly Merge K Sorted Lists. Seed a max-heap with only the newest tweet from each followee (k entries), pop the global newest, and push that user's next-newest back in as a replacement. Ten pops, and each pop touches one heap of size ≤ k. O(k + 10 log k) instead of O(T log T), and you never look at the ancient tweets at all.

Solution (JS) — post O(1) · feed O(k + 10 log k) · follow/unfollow O(1) 
class Twitter {
  constructor() {
    this.time    = 0;         // GLOBAL clock — the only way to order tweets ACROSS users
    this.tweets  = new Map();   // userId -> [[time, tweetId], ...] appended in order => already sorted
    this.follows = new Map();   // userId -> Set of followeeIds
  }

  postTweet(userId, tweetId) {
    if (!this.tweets.has(userId)) this.tweets.set(userId, []);
    this.tweets.get(userId).push([this.time++, tweetId]);   // O(1); list stays sorted by construction
  }

  follow(followerId, followeeId) {
    if (!this.follows.has(followerId)) this.follows.set(followerId, new Set());
    this.follows.get(followerId).add(followeeId);
  }

  unfollow(followerId, followeeId) {
    // Set.delete on a missing key is a harmless no-op — no guard needed.
    this.follows.get(followerId)?.delete(followeeId);
  }

  getNewsFeed(userId) {
    // A Set unions "me" with my followees, so self-follow can't duplicate me.
    const sources = new Set(this.follows.get(userId) ?? []);
    sources.add(userId);                                   // you always see your OWN tweets

    // MAX-heap by timestamp. Each entry: [time, tweetId, ownerId, indexInOwnersList].
    // Seed it with only the NEWEST tweet per source — k entries, not T.
    const heap = new MinHeap((a, b) => b[0] - a[0]);

    for (const uid of sources) {
      const list = this.tweets.get(uid);
      if (!list || list.length === 0) continue;
      const i = list.length - 1;                          // last = newest, since we only ever append
      heap.push([list[i][0], list[i][1], uid, i]);
    }

    const feed = [];
    while (feed.length < 10 && heap.size > 0) {
      const [, tweetId, uid, i] = heap.pop();               // globally newest unseen tweet
      feed.push(tweetId);
      if (i > 0) {                                        // that user has an older tweet behind it
        const list = this.tweets.get(uid);
        heap.push([list[i - 1][0], list[i - 1][1], uid, i - 1]);  // advance THAT list by one
      }
    }
    return feed;
  }
}

Trace: postTweet(1,5) → tweets{1: [[0,5]]}, clock = 1. getNewsFeed(1) → sources = {1}; heap seeded with [0,5,1,0]; pop → feed = [5] ✓. follow(1,2); postTweet(2,6) → tweets{2: [[1,6]]}, clock = 2. getNewsFeed(1) → sources = {2, 1}; heap gets [1,6,2,0] and [0,5,1,0]; the max-heap pops timestamp 1 first → 6, then timestamp 0 → 5 → [6, 5] ✓ — the global clock, not the tweetId, is what ordered them. unfollow(1,2) → sources = {1} → [5] ✓.

Complexity: postTweet, follow, unfollow are all O(1). getNewsFeed seeds k entries (k = people you follow) at O(k log k) — or O(k) if you heapify — then does at most 10 pops and 10 pushes at O(log k) each → O(k + 10 log k), independent of how many tweets exist. Space is O(total tweets + total follow edges) for storage, plus O(k) for the heap during a feed call.

Follow-ups you should expect:

  • "A user follows themselves." Handled: sources is a Set, so add(userId) is idempotent — and unfollowing yourself can't hide your own tweets, because add(userId) runs after the followee set is copied. Point this out unprompted; it's the bug most submissions have.
  • "Cap each user's stored tweets at their most recent 10." Legal, since the feed is only ever 10 long, and it bounds memory. A nice thing to volunteer — it shows you're thinking about retention, not just correctness.
  • "Real Twitter — fan-out on write or fan-out on read?" This is the actual system-design question hiding inside the LeetCode problem. What you just wrote is fan-out on read (merge at query time): cheap writes, expensive reads. Fan-out on write pushes each new tweet into every follower's precomputed feed: expensive writes, O(1) reads — great until Taylor Swift tweets to 100M followers. Real systems do both, fanning out on write for normal users and merging on read for celebrities. Saying this sentence in an enterprise loop is worth more than the code.
  • "Why store [time, tweetId] instead of just tweetId?" Because tweetIds aren't ordered — LeetCode hands you arbitrary integers. Cross-user ordering requires a monotonic clock. (Real systems solve this with Snowflake IDs: a timestamp baked into the high bits, so the id is the clock.)
This is the shape of heap questions in enterprise design rounds: the heap isn't the problem, it's one component of the problem, and the interviewer is watching whether you reach for it at the right moment. Notice how much of the score here lives outside the algorithm — the data model, the self-follow edge case, the fan-out discussion. Talk through the Map/Set/clock choices before you write getNewsFeed, exactly as you'd talk through state shape before writing a React reducer.

Merge k Sorted Lists (LC #23) HARD ★

You're given an array lists of k linked lists, each one already sorted in ascending order. Merge them into a single sorted linked list and return its head. Individual lists may be empty, and the array itself may be empty.

Example 1: Input lists = [[1,4,5],[1,3,4],[2,6]] → Output [1,1,2,3,4,4,5,6]. Merging all the nodes into one sorted sequence.
Example 2: Input lists = [] → Output []. And lists = [[]] → Output [] — an array containing one empty list. These two are separate failure modes and interviewers test both; a solution that seeds the heap with lists[i] unconditionally pushes null and crashes in the comparator.
Constraints: 0 ≤ k ≤ 10⁴; 0 ≤ lists[i].length ≤ 500; −10⁴ ≤ node.val ≤ 10⁴; each list is sorted ascending; the total node count is ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. Brute force: collect all N node values into an array, sort, rebuild a list. O(N log N), O(N) space, and it throws away the fact that the input was already sorted — which is the entire gift the problem is handing you. Interviewers accept it as a baseline and then ask for better.

Here's the insight, and it's tiny: at every step of the merged output, the next node must be the smallest of the k current heads. Nothing deeper in any list can beat its own head — each list is sorted. So you never need to look at more than k nodes at a time. Finding the minimum of k values by scanning is O(k) per node, giving O(N·k) overall; finding it with a min-heap of the k heads is O(log k) per node, giving O(N log k). The heap holds nodes, not values — that's the trick that lets you advance the right list after each pop: you pop a node, append it to the output, and push node.next as its replacement, so the heap always contains exactly one candidate per still-alive list.

Two things worth saying out loud. First, use a dummy head and a tail pointer to build the output — the same technique from tutorial 06, and for the same reason: it kills the "is this the first node?" special case entirely. Second, the alternative solution is divide and conquer — pair the lists up and merge two at a time (using Merge Two Sorted Lists from tutorial 06), halving k on each round. It's also O(N log k), it needs no heap at all, and if the interviewer's language has no priority queue it's arguably the better answer. Naming both, and knowing they hit the same complexity by different routes, is the senior move here.

Solution (JS) — heap of heads · O(N log k) time · O(k) space 
// Definition for singly-linked list:
// function ListNode(val, next) { this.val = val; this.next = next; }

var mergeKLists = function(lists) {
  // MIN-heap ordered by node VALUE, but it stores the NODES themselves —
  // that's what lets us advance the correct list after every pop.
  const heap = new MinHeap((a, b) => a.val - b.val);

  // Seed with the head of each NON-EMPTY list. The null guard is the edge case
  // that kills lists = [] and lists = [[]] — never push null into a heap.
  for (const node of lists) {
    if (node) heap.push(node);
  }

  // Dummy head: the standard linked-list trick (see tutorial 06) that removes
  // the "is this the first node?" branch from the loop entirely.
  const dummy = { val: 0, next: null };
  let tail = dummy;

  while (heap.size > 0) {
    const node = heap.pop();    // smallest of the k current heads — nothing deeper can beat it
    tail.next = node;          // splice it onto the output (we RE-USE nodes; no allocation)
    tail = node;

    if (node.next) {
      heap.push(node.next);    // that list's new head takes its place — heap stays <= k
    }
  }

  tail.next = null;            // sever any stale pointer trailing off the last node
  return dummy.next;         // null automatically when every list was empty
};

Trace on [[1,4,5],[1,3,4],[2,6]]: seed the heap with the three heads → {1ᴬ, 1ᴮ, 2ᶜ}. Pop 1ᴬ → output [1]; push its next, 4ᴬ → heap {1ᴮ, 2ᶜ, 4ᴬ}. Pop 1ᴮ → output [1,1]; push 3ᴮ → heap {2ᶜ, 3ᴮ, 4ᴬ}. Pop 2ᶜ → output [1,1,2]; push 6ᶜ → {3ᴮ, 4ᴬ, 6ᶜ}. Pop 3ᴮ → push 4ᴮ → {4ᴬ, 4ᴮ, 6ᶜ}. Pop 4ᴬ → push 5ᴬ. Pop 4ᴮ (list B is exhausted, nothing pushed) → {5ᴬ, 6ᶜ}. Pop 5ᴬ, pop 6ᶜ → heap empty. Output [1,1,2,3,4,4,5,6] ✓. Notice the heap never held more than 3 = k nodes.

Complexity: every one of the N nodes is pushed exactly once and popped exactly once, and the heap never exceeds k elements → O(N log k) time. Space is O(k) for the heap — and O(1) extra beyond that, because you're relinking existing nodes rather than allocating new ones. Contrast the baseline (collect-and-sort): O(N log N) time and O(N) space. When k = 4 and N = 10⁶, log k = 2 versus log N = 20 — a 10× win, and the whole reason this problem is asked.

Follow-ups you should expect:

  • "No priority queue available." Divide and conquer: merge lists pairwise (lists[0]+lists[1], lists[2]+lists[3], …), halving k each round using the two-list merge from tutorial 06. Same O(N log k) — each of the log k rounds touches all N nodes — with O(1) space if the merge is iterative. It's the answer a lot of interviewers actually prefer.
  • "Merge k sorted arrays instead." Identical algorithm; the heap stores [value, arrayIndex, elementIndex] tuples instead of nodes, because arrays have no .next pointer to follow. This variant is the one that shows up in distributed systems — see the next bullet.
  • "You have 100 machines each holding a sorted shard — give me the global top 10." This is the problem. Each machine ships its local top 10; the coordinator runs a k-way merge over 100 sorted streams and stops after 10 pops. That's the answer to the LC 215 distributed follow-up, and it's the shape of every "merge shards" question in a fintech loop.
  • "Stability / duplicates?" Equal values pop in arbitrary order (a heap is not stable). If the problem needs a deterministic tiebreak, put the list index in the comparator: (a,b) => a.val - b.val || a.listIdx - b.listIdx.
Never push null into the heap. If lists contains an empty list, seeding with heap.push(lists[i]) unconditionally puts null in data, and the very next comparator call evaluates null.val and throws. The same guard applies to the replacement push: only push node.next if it exists. Two one-line guards, and they are exactly the two test cases ([] and [[]]) that LeetCode leads with.

Find Median from a Data Stream (LC #295) HARD ★

The median of a list of numbers is the middle value once they're sorted — and if the count is even, it's the average of the two middle values. Design a class MedianFinder that supports a stream: addNum(num) adds an integer to the data structure, and findMedian() returns the median of everything added so far. Both are called repeatedly and interleaved — the numbers never stop arriving, and you're asked for the median at arbitrary moments in between.

Example 1: addNum(1); addNum(2); findMedian() → 1.5 (even count → average of 1 and 2). addNum(3); findMedian() → 2.0 (odd count → the true middle).
Example 2: addNum(-1); findMedian() → -1.0; addNum(-2); findMedian() → -1.5; addNum(-3); findMedian() → -2.0. Negatives and a strictly decreasing stream — the case that breaks lazy rebalancing code, because every new element belongs on the low side.
Constraints: −10⁵ ≤ num ≤ 10⁵; findMedian() is only called after at least one addNum; up to 5×10⁴ calls. Follow-ups baked into the problem: what if all numbers are in [0, 100]? what if 99% are in [0, 100]?

Solve on LeetCode ↗

Thinking process — derive it, because this one really is derivable. Start with the brute force and let it push you to the answer. Keep everything in an array and sort on every findMedian(): O(n log n) per query, hopeless. Improve it: keep the array always sorted, inserting each new number in its correct position via binary search. Now findMedian() is O(1) — just index the middle — but addNum is O(n), because binary search finds the slot in O(log n) and then the splice shifts every later element. With 5×10⁴ calls that's 10⁹ element moves. So close, and yet the shifting kills it.

Now ask the question that unlocks the whole problem: how much of that sorted order did I actually use? To compute a median you need the two elements at the middle boundary and nothing else. You do not care how the bottom half is arranged internally. You do not care how the top half is arranged internally. You only care about the largest element of the bottom half and the smallest element of the top half. And "give me the max of this pile in O(1)" plus "give me the min of that pile in O(1)" is precisely two heaps standing back to back: 

//        lo: MAX-heap                        hi: MIN-heap
//   (the smaller half)                  (the larger half)
//
//     1   2   3   [4]        |        [5]   6   7   8
//                  ^                    ^
//            lo.peek() = 4        hi.peek() = 5
//            (biggest of the        (smallest of the
//             small half)            big half)
//
//   The median is hiding at that seam, and BOTH sides of it are O(1) to read.
//   Even count  -> (4 + 5) / 2 = 4.5
//   Odd count   -> whichever heap is allowed to be bigger; here we let `lo` win.

That's the entire idea: split the stream at the median and store each half in the heap that puts the median boundary on top. The bottom half needs its maximum exposed → max-heap. The top half needs its minimum exposed → min-heap. Everything else is bookkeeping.

The two invariants (this is what you actually have to get right)

Every addNum must leave both of these true. State them out loud before you write the code:

  1. Ordering: max(lo) ≤ min(hi) — every element in the bottom half is ≤ every element in the top half. Without this, the two "middle" values aren't middle values at all and the median is garbage.
  2. Balance: the sizes differ by at most 1. Conventionally, let lo be the one allowed to hold the extra element, so lo.size === hi.size or lo.size === hi.size + 1. Then the median read is trivial: if the sizes are equal, average the two tops; otherwise, it's lo.peek().

The rebalance that maintains both is a two-step dance, and the order matters. Step 1 — push into lo unconditionally, then immediately move lo's top into hi. That forced hand-off is what enforces the ordering invariant: whatever the new number was, after it passes through lo, the element that gets promoted to hi is the largest of the bottom half — which is exactly the element that belongs on the boundary. You never have to write an if (num < lo.peek()) comparison at all; the heaps sort it out for you. Step 2 — if hi is now bigger than lo, push its top back into lo. That restores the balance invariant. Three lines, no branching on the value, provably correct.

Solution (JS) — addNum O(log n) · findMedian O(1) · O(n) space 
class MedianFinder {
  constructor() {
    // lo = the SMALLER half, in a MAX-heap  => lo.peek() is the biggest of the small half
    // hi = the LARGER  half, in a MIN-heap  => hi.peek() is the smallest of the big half
    // The median lives at the seam between those two tops.
    this.lo = new MinHeap((a, b) => b - a);   // max-heap (inverted comparator)
    this.hi = new MinHeap();                  // min-heap (default)
    // INVARIANTS:  (1) max(lo) <= min(hi)   (2) lo.size === hi.size  ||  lo.size === hi.size + 1
  }

  addNum(num) {
    // STEP 1 — ordering. Push into lo, then hand lo's top to hi.
    // The forced hand-off means we never compare num against anything ourselves:
    // whatever ends up on top of lo IS the right element to promote to hi.
    this.lo.push(num);
    this.hi.push(this.lo.pop());

    // STEP 2 — balance. That hand-off may have made hi too big; pull one back.
    // (lo is allowed to be exactly one LARGER than hi — that's our convention.)
    if (this.hi.size > this.lo.size) {
      this.lo.push(this.hi.pop());
    }
  }

  findMedian() {
    // Odd count: lo holds the extra element, so its top IS the median.
    if (this.lo.size > this.hi.size) return this.lo.peek();

    // Even count: the two middle values are the two tops. Average them.
    // Divide by 2 (not | 0, not Math.floor) — the median of [1,2] is 1.5, not 1.
    return (this.lo.peek() + this.hi.peek()) / 2;
  }
}

Trace, step by step (this is the part to rehearse):

  • addNum(1): push 1 into lo → lo{1}. Hand off: pop 1 from lo, push into hi → lo{}, hi{1}. Balance: hi.size (1) > lo.size (0) → pull back → lo{1}, hi{}. findMedian() → lo is bigger → 1.
  • addNum(2): push 2 into lo → lo{2, 1} (max-heap, top = 2). Hand off: pop 2, push into hi → lo{1}, hi{2}. Balance: sizes are 1 and 1, equal → no pull-back. State: lo{1}, hi{2} — and look: 1 ≤ 2, invariant (1) holds automatically. findMedian() → sizes equal → (1 + 2)/2 = 1.5 ✓.
  • addNum(3): push 3 into lo → lo{3, 1} (top = 3). Hand off: pop 3 → hi{2, 3} (min-heap, top = 2). Now lo{1}, hi{2,3}. Balance: hi.size (2) > lo.size (1) → pull back hi's top (2) into lo → lo{2, 1}, hi{3}. findMedian() → lo bigger → lo.peek() = 2 ✓.
  • The decreasing stream (Example 2), which is where the hand-off earns its keep: addNum(-1) → lo{−1}, hi{}. addNum(-2): push −2 into lo → lo{−1, −2} (max-heap, top = −1). Hand off pops −1, not the number we just added → lo{−2}, hi{−1}. Sizes equal. Median = (−2 + −1)/2 = −1.5 ✓. Notice the algorithm silently routed the new number to the low side and promoted the old one — with zero explicit comparisons. That's the elegance, and it's why the "push into lo, then hand off" order is not negotiable.

Complexity: addNum does a bounded number of heap operations — one push, one pop, one push, and at most one more pop/push pair — each O(log n) on a heap of size ≤ n/2 + 1 → O(log n). findMedian reads at most two tops → O(1), which is the point of the whole design: medians are queried far more often than the stream is added to. Space is O(n) — you genuinely must retain every element, because any future arrival can shift the median boundary. Contrast the sorted-array approach: O(n) per addNum from the splice. You've traded an O(n) insert for an O(log n) one and kept the O(1) read.

Follow-ups you should expect — and the first two are printed in the problem statement, so they will be asked:

  • "What if all numbers are in the range [0, 100]?" Drop the heaps entirely: keep a counting array of 101 buckets. addNum is O(1) (counts[num]++), and findMedian walks the 101 buckets accumulating a running total until it crosses n/2 → O(101) = O(1). A bounded universe always beats a comparison structure. This is the same instinct as counting sort, and it's a strong answer.
  • "What if 99% are in [0, 100] but 1% are outliers?" Hybrid: counting buckets for the dense range, plus two small heaps (or two sorted lists) for the values below 0 and above 100, tracking how many fell outside. The median is almost always inside the bucketed range, so you handle the common case in O(1) and only consult the outlier structures for the counts. Saying "I'd bucket the hot path and keep the tails in overflow structures" is the answer they're fishing for.
  • "Now give me the median of the last 1000 elements only" (a rolling median — this is the actual fintech version: a moving median over a price feed). Two heaps alone can't do it, because you must remove an arbitrary old element and a heap has no O(log n) delete-by-value. The fixes: lazy deletion (mark the expired element in a Map, discard it if it ever surfaces at a top, and track "virtual" sizes) — or reach for a balanced BST / order-statistic tree, which is what a production system uses. Knowing why the two-heap solution breaks here is worth more than the fix itself.
  • "Why not a balanced BST from the start?" A BST with subtree sizes gives O(log n) insert and O(log n) select-by-rank, so it solves this and the rolling variant. But it's ~150 lines to implement in JS versus ~15 with two heaps. Right tool, wrong budget — say exactly that.
  • "What if findMedian is called far more often than addNum?" You already win: the read is O(1). Point out that you designed for that skew deliberately — the sorted-array approach makes the opposite trade, and knowing which side of the trade you're on is the whole discussion.
Three ways this goes wrong under pressure. (1) Skipping the hand-off and writing if (num <= lo.peek()) lo.push(num) else hi.push(num) — it can work, but you must then handle the empty-lo case (peek() is undefined, and every comparison against undefined is false) and rebalance in both directions. The push-then-hand-off version has none of those branches. (2) Integer-dividing the even case — the median of [1, 2] is 1.5; LeetCode compares floats and | 0 silently fails half the tests. (3) Letting the balance drift by 2 because you only rebalance in one direction. Pick the convention (lo may hold the extra), write both invariants in a comment above the class, and check them against your trace.
Two-heaps is a pattern, not a trick for one problem, and naming it as such is the senior signal: whenever you need to track a boundary inside a dynamic set — the median, the k-th element, "the top half versus the bottom half" — split the set in two and store each side in the heap that exposes the boundary element. It generalizes immediately: IPO (LC 502) runs a min-heap on capital and a max-heap on profit; Sliding Window Median (LC 480) is this exact class plus lazy deletion; and the rolling median over a market data feed is why a Goldman or JPMC interviewer has this problem in their back pocket in the first place. If you get asked Kth Largest in a Stream early in a loop, expect this as the escalation — and now you have the derivation, not just the code.

Interview Q&A

These are the conceptual questions that get asked around the coding problem — while you're typing, or in the last five minutes. Answer them in two or three sentences. Rambling here is how strong coders lose points.

Why is heapify O(n) and not O(n log n)?

Because Floyd's build-heap sifts down, and the cost of sifting a node down is bounded by its height above the leaves — not by the height of the whole tree. In a complete binary tree, roughly half the nodes are leaves (height 0, zero work), a quarter have height 1, an eighth have height 2, and so on. Summing the work gives Σh=0..log n (n / 2h+1) × h, and that series converges to 2n. The cheap majority dominates.

The "obvious" O(n log n) bound comes from the other construction: n separate pushes, each of which sifts up from a leaf. That path really is the full height every time, so n pushes genuinely is O(n log n). Same data structure, two build strategies, different costs — and the difference is entirely which direction you sift. Say it in one line: "sift-down from the bottom is cheap because most nodes are near the bottom; sift-up from the bottom is expensive because the root is far away."

When do you use a heap instead of just sorting?

Two situations, and only two. Name them both:

  • Streaming / unbounded input. The data arrives one element at a time and you must answer queries in between, or it's simply too large to hold. You cannot sort what you cannot hold, and re-sorting on every arrival is O(n log n) per element. A heap absorbs each arrival in O(log n) and answers instantly. Kth Largest in a Stream and Find Median from a Data Stream exist to test exactly this.
  • k ≪ n. You need only the k best of n, so a size-k heap runs in O(n log k) time and — the part that actually matters — O(k) space. With n = 10⁹ and k = 10 that's the difference between ten slots and an out-of-memory error.

The flip side, which you should volunteer, is when a heap is not worth it: when you need the full sorted order anyway, when k ≈ n (log k ≈ log n, so the heap buys nothing), or when n is small enough that .sort() is simpler and the constant factor dominates. "I'd sort — unless it's a stream or k is much smaller than n" is the whole decision rule in one sentence.

To keep the k largest elements, why do you use a min-heap?

Because the only element you ever need to see is the one you're about to throw away — the weakest of your current k champions — and the top of a min-heap is exactly that. When a new element arrives, the single question is "is it better than my current worst?" If yes, evict the worst and admit the newcomer; if no, discard the newcomer. Both cases need the minimum of the retained set in O(1), so the retained set lives in a min-heap.

The mirror follows for free: to keep the k smallest, keep a max-heap. And the general rule that makes you never confuse them again: the top of the heap is always the element you are most willing to throw away. As a bonus, that min-heap's top is also the k-th largest element itself — which is why LC 215 and LC 703 are one line apart.

How do you make a max-heap in JavaScript when you've only written a MinHeap?

You invert the comparator — new MinHeap((a, b) => b - a) — and you're done. There is no separate class, ever. The comparator contract is identical to Array.prototype.sort's: compare(a, b) < 0 means "a comes out first," so a - b gives you a min-heap and b - a gives you a max-heap. The same trick handles objects ((a,b) => a.dist - b.dist), tuples ((a,b) => b[0] - a[0]) and multi-key tiebreaks (a.freq - b.freq || …).

The hack you'll see in other people's code — negating every value on the way in and negating again on the way out (heap.push(-x)) — works for plain numbers and breaks the moment you store objects, so don't build the habit. The negation trick exists because Python's heapq has no comparator parameter; JavaScript doesn't have that limitation, because you wrote the class yourself.

Heap vs. quickselect for "find the k-th largest" — which do you actually give?

Give the heap first: O(n log k) time, O(k) space, deterministic, doesn't mutate the input, and it survives a stream. Then offer quickselect as the optimization and let the interviewer pull it out of you: O(n) average — partition, and recurse only into the half that contains the answer, so the work is n + n/2 + n/4 + … = 2n.

Be honest about quickselect's costs in the same breath, because that honesty is the senior signal: the worst case is O(n²) (an adversarial or already-sorted input with a bad pivot rule — mitigate with a random pivot, which you must actually write), it mutates the caller's array, it needs the whole input in memory, and it can't be used on a stream. Median-of-medians guarantees O(n) worst case but its constant is bad enough that it loses in practice. "Heap unless I've measured that this is a hot path" is the correct engineering answer, and it's what a 7-YOE candidate is expected to say.

Can you delete an arbitrary element from a heap? Find an arbitrary element?

Not efficiently, and you should say so before the interviewer traps you with it. A heap is not a search structure — the heap property only orders parents against children, so searching for a value is a linear O(n) scan, and deleting one costs O(n) just to find it (the removal itself, once you know the index, is O(log n): swap with the last element, pop, then sift the replacement both ways — up if it's too small, down if it's too big).

The two production fixes are worth naming, because real systems use them constantly:

  • An index map: maintain a Map from value → index and update it on every swap. Now delete-by-value is O(log n). This is how you build an indexed priority queue, and it's what a proper Dijkstra with decrease-key needs.
  • Lazy deletion: don't remove anything. Mark the element dead in a Set, and when it surfaces at the top, pop and discard it instead of returning it. Simpler, and what most schedulers and Sliding Window Median implementations actually do — at the cost of the heap holding garbage for a while.
Why is a heap stored in a flat array with no pointers?

Because a heap is a complete binary tree — every level is full except possibly the last, which fills strictly left to right. A complete tree has no gaps, so laying it out in level order produces a dense array, and the parent/child links become arithmetic: parent(i) = (i - 1) >> 1, left(i) = 2i + 1, right(i) = 2i + 2. No node objects, no pointer chasing, and the whole structure is contiguous in memory, which is cache-friendly and fast.

This is also why push appends at the end and pop removes the last leaf before sifting: those are the only mutations that preserve completeness. Break completeness and the index arithmetic stops being valid, and the entire implementation collapses. Completeness isn't a nice property of a heap — it's a load-bearing one.

Explain the two-heaps technique in one breath.

"Split the data at the boundary you care about, and store each side in the heap that puts the boundary element on top." For a running median: the lower half goes in a max-heap (so its maximum — the left edge of the median — is at peek()), the upper half goes in a min-heap (so its minimum — the right edge — is at peek()). Maintain two invariants: ordering (max(lo) ≤ min(hi)) and balance (sizes differ by at most 1). Then the median is either the top of the bigger heap, or the average of the two tops.

The rebalance is three lines and needs no comparisons: push into lo, hand lo's top over to hi (that enforces ordering), then if hi got bigger, hand its top back (that enforces balance). O(log n) per insert, O(1) per query. The pattern generalizes beyond medians — IPO (LC 502) and Sliding Window Median (LC 480) are the same two-heaps skeleton.

Is a heap sorted? What does heap.data actually look like?

No. A heap is only partially ordered: every parent beats its children, and that's the entire promise. Siblings are unordered relative to each other, the second-smallest element could be at index 1 or index 2, and printing heap.data gives you something that looks almost-but-not-quite sorted — [1, 3, 2, 7, 4, 9] is a perfectly valid min-heap. Trying to read the k-th element off data[k-1] is a real bug candidates ship.

The only reliable way to get sorted output is to pop repeatedly, which costs O(n log n) — that is heapsort, and it's exactly why a heap never beats sorting when you need the whole order. The one exception on this page: in K Closest Points the answer may be returned in any order, so dumping heap.data straight out is legal and saves you the drain.

Why does JavaScript have no built-in priority queue, and what do you say about it in the interview?

There's no deep reason — the standard library simply never grew one, and no Node release has added it. Java has PriorityQueue, Python has heapq, C++ has std::priority_queue; JavaScript has Array.prototype.sort and your own two hands.

What to say in the room, at the moment you recognize a heap problem: "JavaScript has no built-in heap, so I'll write a small MinHeap class — push, pop, peek, with an injected comparator — unless you'd rather I assume one exists and focus on the algorithm." Then do whichever they pick, without hesitating. Most interviewers will let you assume it, and the ones who don't are handing you a chance to write forty lines of clean, well-named, correctly-guarded code — which is a gift, not a punishment, precisely because most candidates fumble it. The fumbling is the thing being measured; the class itself is easy.

Re-solve drills

Reading heap code is worthless. The only thing that transfers to the interview room is typing the MinHeap class from a blank file until it comes out in one pass with no compile errors. Everything below is built around that.

The daily 5-minute drill (do this every single day until the loop)

Open a blank file. Type MinHeap from memory — constructor with the comparator, size, peek, push, pop, _siftUp, _siftDown. Then check yourself against §3 on these five points, which are where it always breaks:

  • (i - 1) >> 1 for the parent; 2i + 1 / 2i + 2 for the children.
  • _siftDown swaps with the better of the two children, chosen before any swap happens.
  • pop handles the empty heap and the one-element heap (after data.pop() the array is empty — don't write undefined into data[0]).
  • The comparator is injected, and compare(a,b) < 0 means "a pops first."
  • heapify starts at (n >> 1) - 1 and walks down to 0.

Five minutes. It should end at zero errors within a week.
