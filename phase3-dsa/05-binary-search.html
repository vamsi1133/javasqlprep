05 — Binary Search

The boundary template that kills off-by-ones, rotated arrays, and search-on-answer.

Why interviewers ask this

Binary search is the highest-signal-per-line topic in enterprise-tier loops. Banks, fintech and healthcare companies (JPMC, Goldman, Optum and their peers) love it because the whole exercise fits in fifteen minutes: the idea is one sentence, but the implementation exposes exactly the thing they're grading a 7-YOE engineer on — precision under pressure. Everyone "knows" binary search; a surprising fraction of senior candidates still ship an infinite loop, return mid instead of a boundary, or fumble < vs <= when the interviewer nudges the problem from "find the target" to "find the first occurrence". Off-by-one bugs in a 12-line function are a louder red flag at senior level than not knowing an obscure algorithm.

It's also the gateway to a genuinely common medium family. Rotated-array search (LC 33/153) is a fintech staple, Time-Based Key-Value Store (LC 981) is a design-flavored favorite because it looks like a real versioned cache, and "search-on-answer" problems like Koko Eating Bananas test whether you can spot a hidden monotonic structure — the same instinct behind capacity planning and rate-limit tuning questions. Master one rigorous template with a stated invariant and every problem on this page becomes a five-minute variation. That's the pitch of this tutorial: not nine tricks, one template applied nine times.

1. Recognizing the pattern

Binary search applies far beyond "the array is sorted". The real trigger is any of these signals in the problem statement:

  • "Sorted" — sorted array, sorted rows, "non-decreasing", timestamps that are "strictly increasing". The obvious case.
  • "O(log n) required" — when the statement demands logarithmic time on array data, there is essentially one tool. Rotated-array problems say this explicitly.
  • "Minimum X such that…" / "maximum X such that…" — even when nothing is sorted. "Minimum eating speed so she finishes in h hours", "smallest capacity to ship within d days", "least divisor". You binary-search the answer, not the input. This is the search-on-answer sub-pattern and it's the one candidates miss.
  • "Latest value at or before time t" — versioned lookups, snapshots, "most recent ≤". That's an upper-bound search over stored timestamps.

The unifying idea: binary search doesn't need a sorted array — it needs a monotonic predicate. Take some yes/no question p(i) over an index or value range. If the answers, read left to right, look like F F F F T T T (all falses, then all trues, never flipping back), binary search finds the F→T boundary in O(log n) probes. A sorted array is just the special case p(i) = "nums[i] >= target". Koko is the case p(k) = "at speed k she finishes in time" — eating faster never hurts, so the predicate is monotonic.

When you spot the pattern out loud, say the predicate, not the mechanics. "Feasibility is monotonic in speed — if she can finish at speed k, she can finish at any speed above k — so I'll binary-search the smallest feasible speed" is a senior sentence. "I'll use two pointers and cut in half" is a junior one. Naming the monotonic structure is what proves you'd recognize this in a novel problem.
Don't reach for binary search just because the input contains numbers. If the predicate isn't monotonic (e.g. "is there a pair summing to X near index i"), halving the range discards valid answers. State why monotonicity holds in one sentence before you code — interviewers at fintech shops ask for exactly this justification.

2. The template

There are many binary-search skeletons in the wild (closed lo <= hi, half-open lo < hi, "leave two elements" lo + 1 < hi…). Pick one, know its invariant cold, and use it everywhere. This course standardizes on the half-open boundary template: the search space is [lo, hi) — lo inclusive, hi exclusive — and the loop finds the first index where a monotonic predicate turns true. 

// THE canonical template: first index i in [0, n) where p(i) is true.
// Requires: p is monotonic — false...false true...true.
// Returns n if p is never true.
function firstTrue(n, p) {
  let lo = 0, hi = n;              // search space is the half-open [lo, hi)
  while (lo < hi) {                // loop while the space is non-empty
    const mid = lo + ((hi - lo) >> 1); // overflow-proof midpoint, floors toward lo
    if (p(mid)) hi = mid;          // mid is true → answer is mid or earlier → keep mid
    else        lo = mid + 1;      // mid is false → answer is strictly after mid
  }
  return lo;                       // lo === hi: the boundary
}

Line by line, with the invariant. The invariant that holds before and after every iteration:

Everything left of lo is known-false. Everything at or right of hi is known-true (or out of range). The answer — the first true index — is always inside [lo, hi].
  • lo = 0, hi = n — initially nothing is known, so the candidate range is everything, plus the virtual index n meaning "never true". That's why the template returns a value you must check, not a guaranteed hit.
  • while (lo < hi) — a half-open interval [lo, hi) is empty exactly when lo === hi. No <=, ever, in this template.
  • mid = lo + ((hi - lo) >> 1) — the midpoint, written so the sum never overflows (matters in Java/C++; in JS it also keeps you inside fast int32 math — see Q&A #1). Because lo < hi, we get lo <= mid < hi: mid is always a valid, untested index.
  • p(mid) → hi = mid — mid answered true, so the first true is at mid or earlier. Setting hi = mid keeps mid in the candidate set (as the exclusive bound, it's the "known-true" frontier). We never throw away a possible answer.
  • else → lo = mid + 1 — mid answered false, so the boundary is strictly right of mid. mid itself is safely discarded.

Why it can't infinite-loop. Each iteration either sets hi = mid or lo = mid + 1. Since lo <= mid < hi:

  • hi = mid strictly decreases hi (because mid < hi).
  • lo = mid + 1 strictly increases lo (because mid >= lo).

The interval length hi - lo strictly shrinks every single iteration, so the loop terminates in ⌈log₂ n⌉ + 1 iterations, full stop. Compare with the classic bug: in a closed-interval loop, writing hi = mid (instead of mid - 1) with while (lo <= hi) spins forever when lo === hi === mid. The half-open template makes that bug unrepresentable — that's why we standardize on it.

Lower bound and upper bound

Two predicates you'll instantiate constantly on a sorted array nums:

// lowerBound: first index with nums[i] >= target.
// (insertion point; equals index of first occurrence if target exists)
function lowerBound(nums, target) {
  let lo = 0, hi = nums.length;
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (nums[mid] >= target) hi = mid;
    else lo = mid + 1;
  }
  return lo;
}

// upperBound: first index with nums[i] > target (strictly greater).
// upperBound - 1 = index of the LAST occurrence of target (if present).
// Handy identity: upperBound(nums, t) === lowerBound(nums, t + 1) for integers.
function upperBound(nums, target) {
  let lo = 0, hi = nums.length;
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (nums[mid] > target) hi = mid;
    else lo = mid + 1;
  }
  return lo;
}

Notice they are the same function with predicates nums[i] >= t vs nums[i] > t. Everything on this page — first/last position, rotated minimum, Koko, TimeMap — is one of these two with a different predicate. "Does target exist?" is lowerBound(nums, t) < n && nums[lowerBound(nums, t)] === t. "Count of target" is upperBound - lowerBound. "Latest timestamp ≤ t" is upperBound(ts, t) - 1.

The single most common interview bug: returning lo from a boundary search and forgetting the two failure checks — lo === n (predicate never true / target bigger than everything) and, for exact search, nums[lo] !== target. Say "and now I validate the boundary" out loud; it reads as rigor.

The search-on-answer generalization

Nothing in the template mentions an array. Replace "index range [0, n)" with "answer range [minAnswer, maxAnswer]" and "array lookup" with "feasibility check", and you get the pattern behind Koko, Ship Packages, Split Array — a huge slice of enterprise mediums: 

// Smallest feasible answer in [minA, maxA], given monotonic feasible().
// (feasible false...false true...true as the candidate answer grows)
function minFeasible(minA, maxA, feasible) {
  let lo = minA, hi = maxA + 1;      // half-open [minA, maxA + 1)
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (feasible(mid)) hi = mid;
    else lo = mid + 1;
  }
  return lo;  // caller guarantees maxA is feasible, so no n-check needed
}

Cost: O(log(range) × cost(feasible)) — the check is usually a linear scan, giving O(n log range) overall. The three-step recipe to narrate:

  1. Answer range — what's the smallest and largest the answer could possibly be? (Koko: 1 banana/hr to max-pile bananas/hr.)
  2. Feasibility check — given a candidate answer, can I verify it in O(n)?
  3. Monotonicity argument — one sentence for why feasible answers form a suffix (or prefix) of the range.
One more shape you'll see below: exact-match search (LC 704, 33, 74) adds an early if (nums[mid] === target) return mid; inside the same half-open loop. It's an optimization, not a different template — the boundary machinery is unchanged, and falling out of the loop still means "not found".

3. Exact match

The warm-up family: "is target in here, and where?" You still run the half-open loop; you just add an early return mid when you happen to land on the target. If the interval drains, the target isn't there.

Binary Search (LC #704) EASY ★

You're given an array nums sorted in strictly ascending order (all values distinct) and an integer target. Return the index of target in nums, or -1 if it isn't present. The problem explicitly demands O(log n) runtime — a linear scan is a rejection even though it would pass every sample case.

Example 1: Input: nums = [-1,0,3,5,9,12], target = 9 → Output: 4 — nums[4] === 9.
Example 2: Input: nums = [-1,0,3,5,9,12], target = 2 → Output: -1 — 2 would sit between 0 and 3, but it isn't stored, so there's no index to return.
Example 3: Input: nums = [5], target = 5 → Output: 0 — the single-element case is what smokes out infinite loops. Make sure your loop probes it and then terminates.
Constraints: 1 ≤ nums.length ≤ 10⁴; −10⁴ ≤ nums[i], target ≤ 10⁴; all values unique, sorted ascending.

Solve on LeetCode ↗

Thinking process. The brute force is a linear scan — O(n), and the statement bans it. The insight is the one thing sortedness actually buys you: a single comparison at any index tells you which side the target must be on, so one probe eliminates half the remaining candidates instead of one candidate. That's log₂(10⁴) ≈ 14 probes instead of 10,000. Drop it into the template: probe mid; if it equals the target you're done; if nums[mid] < target the target is strictly right, so lo = mid + 1; otherwise it's at mid or left, so hi = mid. The equality branch already consumed the "at mid" case, so hi = mid is safe and the interval still shrinks every iteration. There's a second, equally correct framing worth knowing because every other problem on this page uses it: run lowerBound, then validate the boundary — i = lowerBound(nums, target), answer i < n && nums[i] === target ? i : -1. Same complexity, no special case, and it generalizes to duplicates for free.

Solution (JS) — O(log n) time · O(1) space 
var search = function(nums, target) {
  let lo = 0, hi = nums.length;      // half-open [lo, hi): hi starts PAST the end

  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);  // lo <= mid < hi, so mid is always a real index

    if (nums[mid] === target) return mid;  // exact-match shortcut (values are distinct)

    if (nums[mid] < target) lo = mid + 1;   // mid too small → answer strictly right of mid
    else                    hi = mid;       // mid too big → answer strictly left; mid is ruled out
  }

  return -1;                            // interval emptied without a hit
};

// The template-purist version — no special case, and it survives duplicates.
// This is the one to write if the interviewer might escalate to "first occurrence".
var search2 = function(nums, target) {
  let lo = 0, hi = nums.length;
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (nums[mid] >= target) hi = mid;   // p(i) = nums[i] >= target — monotonic F..F T..T
    else lo = mid + 1;
  }
  // lo is the boundary. Now VALIDATE it — both checks matter:
  //   lo === nums.length  → target is larger than everything
  //   nums[lo] !== target → target would slot in here, but isn't stored
  return (lo < nums.length && nums[lo] === target) ? lo : -1;
};

Trace on nums = [-1,0,3,5,9,12], target = 9: lo=0, hi=6 → mid=3, nums[3]=5 < 9 → lo=4. lo=4, hi=6 → mid=5, nums[5]=12 > 9 → hi=5. lo=4, hi=5 → mid=4, nums[4]=9 === target → return 4 ✓. Three probes over six elements. Now the miss, target = 2: lo=0,hi=6 → mid=3 (5 > 2) → hi=3; lo=0,hi=3 → mid=1 (0 < 2) → lo=2; lo=2,hi=3 → mid=2 (3 > 2) → hi=2; now lo === hi === 2, loop ends → -1 ✓. Note where it stopped: index 2 is exactly where a 2 would be inserted. That's the next problem, for free.

Complexity: O(log n) time — hi - lo is at least halved every iteration, so it hits 0 in ⌈log₂ n⌉ + 1 steps. O(1) space — two integers, no recursion. Writing it recursively costs O(log n) stack frames, which is a strictly worse answer to volunteer.

Follow-ups you should expect:

  • "Now there are duplicates and I want the first occurrence." That's search2, unchanged — the boundary version already returns the leftmost match. Say so; it's the whole reason to prefer it.
  • "Write it recursively." Trivial, but volunteer that iterative is preferred: no stack frames, and JS engines don't guarantee tail-call elimination, so deep recursion is a real risk in a language you'd actually ship.
  • "Why lo + ((hi - lo) >> 1) and not (lo + hi) / 2?" See Q&A #1 — a near-certain question in a Java shop.
The classic infinite loop is while (lo <= hi) paired with hi = mid. When lo === hi === mid and the predicate is true, hi = mid changes nothing and you spin forever. It happens when you half-remember the closed-interval skeleton and half-remember this one. Commit to lo < hi with hi = mid / lo = mid + 1 and never mix the two.

Search a 2D Matrix (LC #74) MEDIUM

You're given an m × n matrix with two guarantees: every row is sorted ascending left-to-right, and the first integer of each row is greater than the last integer of the row above it. Given a target, return true if it appears anywhere in the matrix. O(log(m·n)) is required.

Example 1: Input: matrix = [[1,3,5,7],[10,11,16,20],[23,30,34,60]], target = 3 → Output: true — it's at row 0, column 1.
Example 2: Input: same matrix, target = 13 → Output: false — 13 belongs between 11 and 16 in row 1, but that slot doesn't exist.
Constraints: 1 ≤ m, n ≤ 100; −10⁴ ≤ matrix[i][j], target ≤ 10⁴; rows sorted, and the rows themselves ordered (row i's last element < row i+1's first).

Solve on LeetCode ↗

Thinking process. Read the two guarantees together and say the sentence that solves the problem: "this isn't a matrix, it's a single sorted array of length m·n that someone wrapped every n elements." Concatenating the rows in row-major order gives a globally ascending sequence — that's exactly what the second guarantee buys. So run the ordinary 1-D template over the virtual index range [0, m·n) and translate each probe back into coordinates: row = Math.floor(idx / n), col = idx % n. Nothing else changes. The alternative — binary-search the rows to find the candidate row, then binary-search inside it — is also O(log m + log n) = O(log(m·n)) and perfectly acceptable, but it's twice the code and twice the bounds to fumble. Two judgment calls matter: the divisor is n, the number of columns (candidates reflexively reach for matrix.length, which is rows), and the flattening works only because of the cross-row guarantee. LC 240 ("Search a 2D Matrix II") drops that guarantee, and there flattening is simply wrong. Interviewers escalate from one to the other constantly, so know which guarantee you're standing on.

Solution (JS) — O(log(m·n)) time · O(1) space 
var searchMatrix = function(matrix, target) {
  const m = matrix.length, n = matrix[0].length;

  // Treat the matrix as the sorted array [0, m*n) that it really is.
  let lo = 0, hi = m * n;

  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);

    // Unflatten. n is the COLUMN count: divide by columns, mod by columns.
    const val = matrix[Math.floor(mid / n)][mid % n];

    if (val === target) return true;
    if (val < target) lo = mid + 1;
    else             hi = mid;
  }

  return false;
};

Trace on Example 1 (m=3, n=4; the virtual array is [1,3,5,7,10,11,16,20,23,30,34,60], target 3): lo=0, hi=12 → mid=6 → matrix[1][2] = 16 > 3 → hi=6. lo=0, hi=6 → mid=3 → matrix[0][3] = 7 > 3 → hi=3. lo=0, hi=3 → mid=1 → matrix[0][1] = 3 === target → true ✓. For target 13 the probes converge on lo = hi = 6 (11 and 16 straddle it), the interval empties → false ✓.

Complexity: O(log(m·n)) = O(log m + log n) time — one binary search over m·n virtual cells, with O(1) arithmetic per probe. O(1) space: you never materialize the flattened array. (Actually calling matrix.flat() first is O(m·n) time and space — it makes the code shorter and the complexity worse, and an interviewer will catch it.)

Follow-ups you should expect:

  • "Now rows and columns are each sorted, but row i+1 may start below row i's end." That's LC 240 and flattening breaks. Start at the top-right cell: too big → move left, too small → move down. Each step eliminates an entire row or column, so it's O(m + n). Naming the right tool for the changed guarantee is the whole point of the follow-up.
  • "Return the coordinates, not a boolean." Return [Math.floor(mid / n), mid % n] at the hit — the flattening is a genuine bijection, so nothing else changes.
  • "What if the rows have different lengths?" Index arithmetic dies. You'd build a prefix-sum of row lengths and binary-search that to find the owning row, then search within it. Worth naming out loud even if you don't code it.
The reusable idea is bigger than this one problem: binary search operates on an index space, not on a data structure. Anything you can address with an integer i in O(1), whose values are monotonic in i, is binary-searchable — a flattened matrix, a virtual concatenation, even a function whose values you never store. That reframing is the bridge to Section 6.

4. Boundary search (first / last)

Now the answer is the boundary — there's no "found it, return mid" shortcut, because the thing you're looking for is a position, not a value. This is where the template earns its keep and where sloppy binary search dies.

Search Insert Position (LC #35) EASY

Given a sorted array of distinct integers and a target, return the index of the target if it's present. If it isn't, return the index where it would be inserted to keep the array sorted. Must run in O(log n).

Example 1: Input: nums = [1,3,5,6], target = 5 → Output: 2 — present, at index 2.
Example 2: Input: nums = [1,3,5,6], target = 2 → Output: 1 — 2 belongs between 1 and 3, so it would be inserted at index 1.
Example 3: Input: nums = [1,3,5,6], target = 7 → Output: 4 — bigger than everything, so it appends at the end. This is the case that makes the answer n, an index that doesn't exist in the array, and it's exactly why the template's return value ranges over [0, n] rather than [0, n).
Constraints: 1 ≤ nums.length ≤ 10⁴; −10⁴ ≤ nums[i], target ≤ 10⁴; sorted ascending, all distinct.

Solve on LeetCode ↗

Thinking process. Don't think of this as "search, and if you miss, do something." Think about what the insertion point is: the first index whose value is ≥ the target. If the target exists, that index holds it (distinct values, so the first ≥ is the equal one). If it doesn't, that index holds the first element bigger than the target — precisely where it slots in. If nothing is ≥ the target, the answer is n. That is lowerBound, verbatim, with no post-processing and no validation, because the boundary itself is the answer. This problem is on the page for exactly that reason: it's the purest expression of the template, and once you see that LC 704 and LC 35 differ only in "do you validate the boundary or return it raw," you've internalized the pattern. Say out loud: "the predicate is nums[i] >= target, it's monotonic because the array is sorted, and the first true index is the insertion point."

Solution (JS) — O(log n) time · O(1) space 
var searchInsert = function(nums, target) {
  let lo = 0, hi = nums.length;   // hi = n: the "append at the end" answer must be reachable

  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);

    // p(mid) = "nums[mid] is big enough to sit at or after the insertion point"
    if (nums[mid] >= target) hi = mid;  // keep mid: it may BE the answer
    else lo = mid + 1;                  // mid is too small; the answer is past it
  }

  return lo;  // no validation, no -1 — the boundary IS the answer (0..n inclusive)
};

Trace on nums = [1,3,5,6], target = 2: lo=0, hi=4 → mid=2, nums[2]=5 ≥ 2 → hi=2. lo=0, hi=2 → mid=1, nums[1]=3 ≥ 2 → hi=1. lo=0, hi=1 → mid=0, nums[0]=1 < 2 → lo=1. lo === hi === 1 → return 1 ✓. And on target = 7: every probe is too small, so lo ratchets 0 → 2 → 3 → 4 and the answer is 4 = n. The template returned an index one past the end without a single special case — that's the design working.

Complexity: O(log n) time, O(1) space. Identical to LC 704; only the return line differs.

Follow-ups you should expect:

  • "Now there are duplicates — where do you insert?" lowerBound inserts before all equal elements, upperBound inserts after them. For a stable insert (preserving arrival order) you want upperBound. Knowing both by name is the answer.
  • "Use it to maintain a sorted list under repeated inserts." Binary search is O(log n) but the array splice is O(n) — so inserts are O(n) regardless. If that's the real workload, you want a balanced BST / skip list / heap, not an array. That trade-off is a genuinely senior thing to raise unprompted.
  • "Now count how many elements are strictly less than the target." That's this exact return value. Boundary searches are counting queries in disguise.
In your day job this is Array.prototype.findIndex(v => v >= target) — same answer, O(n). The interview is asking you to do the O(log n) version by hand. Worth knowing that JS ships no built-in binary search (no bisect like Python, no Arrays.binarySearch like Java); that absence is precisely why the JS version of this question gets asked.

First Bad Version (LC #278) EASY

You ship versions 1..n. At some point one version broke, and every version after a bad one is also bad. You're given an API isBadVersion(version) that returns a boolean, and you must find the first bad version while minimizing the number of API calls (the call is expensive — imagine it kicks off a full regression suite).

Example 1: Input: n = 5, first bad = 4 → Output: 4. Calls: isBadVersion(3) → false, isBadVersion(5) → true, isBadVersion(4) → true. Three calls, not five.
Example 2: Input: n = 1, first bad = 1 → Output: 1 — with only one candidate there's nothing to search; a correct implementation makes zero API calls here.
Constraints: 1 ≤ bad ≤ n ≤ 2³¹ − 1. A bad version is guaranteed to exist.

Solve on LeetCode ↗

Thinking process. There's no array here at all — and that's the point of putting this problem on the page. What you have is a black-box predicate over the range 1..n, and the problem statement hands you the monotonicity guarantee on a plate: "every version after a bad one is also bad" is literally F F F T T T. So this is firstTrue with p = isBadVersion and nothing else. The brute force is calling the API on 1, 2, 3, … until it returns true — O(n) expensive calls; binary search makes it ⌈log₂ n⌉ ≈ 31 calls for n = 2³¹. Two details separate a clean answer from a merely-passing one. First: the version range is 1-based, so initialize lo = 1, and set hi = n — not n + 1 — because the answer is guaranteed to exist, so version n is a known-true frontier you never have to test. The half-open interval [1, n) can still return n, so correctness is preserved and you've saved an API call, which is the stated optimization target. Say that out loud; it's the whole point of the "minimize calls" clause. Second: this is git bisect. Mention it — it lands, because it's true.

Solution (JS) — O(log n) API calls · O(1) space 
// The API is injected by LeetCode:  isBadVersion(version) -> boolean
var solution = function(isBadVersion) {
  return function(n) {
    // Versions are 1-based. hi = n (not n + 1) because a bad version is
    // GUARANTEED to exist — so n is known-true and never needs testing.
    // The half-open interval [lo, hi) can still return n itself.
    let lo = 1, hi = n;

    while (lo < hi) {
      const mid = lo + ((hi - lo) >> 1);   // n can be 2^31 - 1: (lo + hi) would overflow int32 here

      if (isBadVersion(mid)) hi = mid;      // mid is bad → first bad is mid or earlier
      else lo = mid + 1;                    // mid is good → first bad is strictly after mid
    }

    return lo;  // lo === hi: the first bad version
  };
};

Trace on n = 5, first bad = 4: lo=1, hi=5 → mid = 1 + (4 >> 1) = 3 → isBadVersion(3) = false → lo=4. lo=4, hi=5 → mid = 4 → isBadVersion(4) = true → hi=4. lo === hi === 4 → return 4 ✓, in two API calls. On n = 1: lo = hi = 1, the loop body never runs, return 1 — zero API calls. ✓

Complexity: O(log n) API calls — that's the real cost metric here, and you should say so explicitly rather than reciting "O(log n) time." The arithmetic is O(1) per call and space is O(1).

Follow-ups you should expect:

  • "Where have you seen this in real life?" git bisect — commits are versions, the build/test is isBadVersion, monotonicity is the assumption that the bug persists once introduced. Bisecting 1,000 commits takes 10 builds, not 1,000. Also: canary rollouts, and finding the first failing row in a batch import.
  • "What if the API is flaky — it sometimes lies?" Monotonicity breaks and binary search is unsound. You'd need repeated sampling / voting per probe, or fall back to a linear scan. This is a genuinely good senior answer: notice the assumption, then say what fails without it.
  • "What if versions aren't ordered — a dependency graph, not a line?" That's exactly why git bisect gets hard with merges; it linearizes the DAG first. Mentioning it signals you've actually used the tool.
Notice you have now written the identical loop three times with three different predicates: nums[i] === target-with-sides, nums[i] >= target, and isBadVersion(i). When an interviewer asks "how would you approach this?", the sentence that scores is "this is a first-true boundary search over a monotone predicate; the only thing I need to decide is what the predicate is." The code is then dictation.

Find First and Last Position of Element in Sorted Array (LC #34) MEDIUM ★

Given an array of integers sorted in non-decreasing order (duplicates allowed) and a target, return [firstIndex, lastIndex] — the starting and ending positions of the target's run. If the target isn't present, return [-1, -1]. O(log n) is required, so "find one occurrence, then walk outward" is explicitly ruled out.

Example 1: Input: nums = [5,7,7,8,8,10], target = 8 → Output: [3,4] — the 8s occupy indices 3 and 4.
Example 2: Input: nums = [5,7,7,8,8,10], target = 6 → Output: [-1,-1] — 6 is absent; the insertion point exists but no element does.
Example 3: Input: nums = [2,2,2,2], target = 2 → Output: [0,3] — the run is the entire array. This is the case that punishes "find any match, then expand left and right," which degrades to O(n) here.
Constraints: 0 ≤ nums.length ≤ 10⁵; −10⁹ ≤ nums[i], target ≤ 10⁹; nums is sorted non-decreasing. Note nums may be empty.

Solve on LeetCode ↗

Thinking process. The trap is trying to be clever: find any occurrence with an ordinary binary search, then scan outward to the run's edges. That's correct but O(n) in the worst case — Example 3 is 4 elements of pure identical target, and a real test case will be 10⁵ of them. Interviewers include that case deliberately. The insight is to stop looking for one clever pass and instead run two boundary searches, because the two edges of the run are two different predicates: the first index is lowerBound(nums, target) — first i with nums[i] >= target — and the index just past the last is upperBound(nums, target) — first i with nums[i] > target. So the run is the half-open interval [lowerBound, upperBound) and the last index is upperBound - 1. Two O(log n) searches is still O(log n). Then validate exactly once: if lowerBound === n or nums[lowerBound] !== target, the target isn't there — and notice you only need to check the lower bound, because if the target is absent, the two bounds are equal and the interval is empty. Free bonus you should offer unprompted: upperBound − lowerBound is the count of occurrences, which is the standard follow-up.

Solution (JS) — O(log n) time · O(1) space 
var searchRange = function(nums, target) {
  // One helper, one knob: `strict` picks > instead of >=.
  // strict=false → lowerBound: first i with nums[i] >= target
  // strict=true  → upperBound: first i with nums[i] >  target
  const bound = (strict) => {
    let lo = 0, hi = nums.length;
    while (lo < hi) {
      const mid = lo + ((hi - lo) >> 1);
      const p = strict ? nums[mid] > target : nums[mid] >= target;
      if (p) hi = mid;
      else lo = mid + 1;
    }
    return lo;
  };

  const first = bound(false);   // where the run STARTS

  // Validate ONCE, on the lower bound. Two failure modes, both handled here:
  //   first === nums.length → target exceeds every element (also covers nums = [])
  //   nums[first] !== target → target would slot in at `first` but isn't stored
  if (first === nums.length || nums[first] !== target) return [-1, -1];

  const last = bound(true) - 1;  // upperBound is one PAST the run → step back one
  return [first, last];

  // Bonus, free: bound(true) - bound(false) === how many times target occurs.
};

Trace on nums = [5,7,7,8,8,10], target = 8 (n = 6):
lowerBound (p: nums[i] >= 8): lo=0,hi=6 → mid=3, nums[3]=8 ≥ 8 → hi=3. lo=0,hi=3 → mid=1, nums[1]=7 < 8 → lo=2. lo=2,hi=3 → mid=2, nums[2]=7 < 8 → lo=3. lo===hi=3 → first = 3.
Validate: 3 < 6 and nums[3] === 8 ✓.
upperBound (p: nums[i] > 8): lo=0,hi=6 → mid=3, 8 > 8? no → lo=4. lo=4,hi=6 → mid=5, nums[5]=10 > 8 → hi=5. lo=4,hi=5 → mid=4, 8 > 8? no → lo=5. lo===hi=5 → upperBound = 5, so last = 4.
Return [3, 4] ✓. Count = 5 − 3 = 2, which is right. On Example 3 ([2,2,2,2], target 2): lowerBound = 0, upperBound = 4 → [0, 3] in 2·log₂4 = 4 probes, not the 4 linear steps a scan-outward solution would take (and not the 100,000 steps it would take at full input size).

Complexity: O(log n) time — two independent binary searches, 2·⌈log₂ n⌉ probes, and constants don't change the class. O(1) space. Contrast with find-then-expand: O(log n + k) where k is the run length, which is O(n) when the array is all target. Saying that comparison unprompted is what marks this as a MEDIUM you actually understand rather than one you memorized.

Follow-ups you should expect:

  • "How many times does the target occur?" upperBound − lowerBound. No new code. Offer it before they ask.
  • "Can you do it with one search instead of two?" You can search for target and target + 0.5 (or use the integer identity upperBound(t) === lowerBound(t + 1)) — same two searches wearing a hat. There is no genuine one-pass version, and confidently saying "no, and here's why: the two edges are two different predicates" is a better answer than inventing one.
  • "Now the array is rotated as well." Compose: find the pivot (LC 153), then run both bounds on the correct sorted segment. That's the next section.
Writing the second search by copy-pasting the first and "adjusting a sign" is where people lose this problem. The only difference between the two is >= versus >. Everything else — hi = nums.length, while (lo < hi), hi = mid / lo = mid + 1, return lo — is byte-for-byte identical. If you find yourself also changing hi = mid - 1 or the loop condition, you've broken it. Factor it into one helper with a flag, like the solution does, and the bug becomes unwritable.

5. Rotated arrays

A sorted array that's been rotated around an unknown pivot — [0,1,2,4,5,6,7] becomes [4,5,6,7,0,1,2] — is the classic fintech medium, and it's the first place the array itself stops being monotonic. The array isn't sorted anymore, so you cannot binary-search it with nums[mid] < target. The fix is to find a predicate that is still monotonic, or to exploit the one structural fact that always holds: split a rotated array anywhere and at least one of the two halves is a plain sorted array.

Find Minimum in Rotated Sorted Array (LC #153) MEDIUM ★

An array of unique integers, originally sorted ascending, has been rotated between 1 and n times (rotating n times returns it to the original, so "not rotated at all" is a legal input). Find the minimum element in O(log n). You are not told the rotation amount.

Example 1: Input: nums = [3,4,5,1,2] → Output: 1 — the original [1,2,3,4,5] rotated 3 times; the minimum sits at the seam.
Example 2: Input: nums = [11,13,15,17] → Output: 11 — rotated a full 4 times, i.e. not rotated at all. The minimum is at index 0 and there is no seam. Every wrong solution to this problem breaks on this case.
Example 3: Input: nums = [4,5,6,7,0,1,2] → Output: 0.
Constraints: 1 ≤ n ≤ 5000; −5000 ≤ nums[i] ≤ 5000; all values unique; the array is a rotation of a sorted array.

Solve on LeetCode ↗

Thinking process. Math.min(...nums) is O(n) and the statement bans it, so you need a monotone predicate over the index. Here is the derivation that makes this problem trivial and that almost nobody volunteers: anchor on the last element. Let anchor = nums[n - 1]. Picture the rotated array as two ascending runs — a "high run" (everything from the original array that got moved to the front) followed by a "low run" (everything from the minimum to the original end). Every element in the high run is strictly greater than the anchor; every element in the low run is less than or equal to the anchor (the anchor is itself in the low run). So the predicate p(i) = nums[i] <= anchor reads F F F T T T T across the array — perfectly monotonic — and the first true index is the minimum. That's firstTrue, unmodified. And it handles Example 2 for free: if the array was never rotated, the high run is empty, p(0) is already true, and the search returns index 0. No special case, no "check if nums[0] < nums[n-1] and bail," no seam detection. Set hi = n - 1 rather than n, because p(n - 1) is trivially true (the anchor is ≤ itself), so it's a known-true frontier you never need to probe.

The version of this you'll see everywhere compares nums[mid] against nums[hi] — a moving right endpoint — instead of a fixed anchor. It's correct, but only in the closed-interval skeleton, and it's fragile: change one bound and it silently breaks. Freezing anchor = nums[n - 1] before the loop keeps the predicate a genuine function of i alone, which is what "monotone predicate" actually requires. Do it that way and you can reason about it; do it the other way and you're pattern-matching.
Solution (JS) — O(log n) time · O(1) space 
var findMin = function(nums) {
  const n = nums.length;
  const anchor = nums[n - 1];   // FIXED. Never changes inside the loop.

  // p(i) = nums[i] <= anchor.  Across a rotated array this reads F..F T..T:
  //   the pre-seam "high run" is all > anchor; the post-seam "low run" is all <= anchor.
  // The first true index is exactly the minimum.
  let lo = 0, hi = n - 1;    // hi = n-1: p(n-1) is trivially true, so never probe it

  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (nums[mid] <= anchor) hi = mid;  // mid is in the low run → min is at mid or earlier
    else lo = mid + 1;                  // mid is in the high run → min is strictly after mid
  }

  return nums[lo];  // lo === hi === index of the minimum (0 if never rotated)
};

Trace on [4,5,6,7,0,1,2] (n=7, anchor = nums[6] = 2): lo=0, hi=6 → mid=3, nums[3]=7 > 2 → high run → lo=4. lo=4, hi=6 → mid=5, nums[5]=1 ≤ 2 → low run → hi=5. lo=4, hi=5 → mid=4, nums[4]=0 ≤ 2 → hi=4. lo === hi === 4 → nums[4] = 0 ✓.
Trace on [11,13,15,17] (anchor = 17): lo=0,hi=3 → mid=1, 13 ≤ 17 → hi=1; lo=0,hi=1 → mid=0, 11 ≤ 17 → hi=0; lo===hi===0 → 11 ✓. The un-rotated case falls out of the same predicate with zero special-casing.
Single element [5]: lo=0, hi=0, loop never runs → nums[0] = 5 ✓.

Complexity: O(log n) time — the interval shrinks every iteration exactly as in the base template; nothing about rotation changes the halving argument, only the predicate. O(1) space.

Follow-ups you should expect:

  • "Now duplicates are allowed" (LC 154). Monotonicity breaks: with [3,3,1,3] and anchor 3, p(0) is true but index 0 isn't the minimum, so the F..F T..T structure is gone. The standard fix is: when nums[mid] === anchor you cannot tell which run you're in, so shrink the range by one and retry. That makes the worst case O(n) (all elements equal), and you should say the worst case is O(n) — proving that no O(log n) algorithm exists here is the actual point of the follow-up.
  • "Return the index of the minimum, not the value." Return lo. That index is the rotation count, and it's what the next problem needs.
  • "Find the maximum." The maximum is at lo - 1, wrapping to n - 1 when lo === 0. One line, and thinking of it as "the element just before the seam" is the correct mental model.

Search in Rotated Sorted Array (LC #33) MEDIUM ★

Same setup — an ascending array of distinct integers, rotated at an unknown pivot — but now find the index of a given target, or return -1 if it isn't present. O(log n) required. This is one of the most-asked mediums in the enterprise tier: JPMC, Goldman and their peers use it precisely because it separates "I memorized binary search" from "I can reason about an invariant under a twist."

Example 1: Input: nums = [4,5,6,7,0,1,2], target = 0 → Output: 4.
Example 2: Input: nums = [4,5,6,7,0,1,2], target = 3 → Output: -1 — 3 is in neither run. It's the miss case, and it's the one that exposes a wrong side-selection: a broken solution often still finds present targets and only fails on absent ones.
Example 3: Input: nums = [1], target = 0 → Output: -1.
Constraints: 1 ≤ n ≤ 5000; −10⁴ ≤ nums[i], target ≤ 10⁴; all values distinct; nums is a rotation of an ascending array.

Solve on LeetCode ↗

Thinking process. There are two legitimate routes, and you should know both — but lead with the second, because it's the one they want.

Route A — decompose. Run LC 153 to find the pivot index k (the minimum). Now you have two sorted segments, [0, k) and [k, n). Compare the target to nums[0] to decide which segment it could live in, then run a plain lowerBound there and validate. Two binary searches, O(log n), and it reuses code you already wrote. It's a perfectly good answer and a fine fallback if you blank — say it out loud even if you code Route B, because it demonstrates you can compose solutions.

Route B — one pass, "one half is always sorted." Here's the key observation, and it's the sentence to say: "however I cut a rotated array at mid, at most one seam exists, so at least one of the two halves contains no seam — that half is plainly sorted, and in a plainly sorted half I can decide membership with two comparisons." So each iteration: (1) figure out which half is sorted by comparing nums[lo] to nums[mid] — if nums[lo] <= nums[mid], the left half is clean; (2) check whether the target falls inside that clean half's range; (3) if yes, recurse into it; if no, the target must be in the other half, so go there. That's it. Notice what you're doing: you can't test the target against mid directly (the array isn't monotone), so you test it against the sorted half's endpoints, where the ordinary comparison is valid again. Every iteration still halves the interval, so O(log n) holds. The distinctness constraint is what makes nums[lo] <= nums[mid] a reliable "left half is sorted" test — with duplicates it lies, and that's LC 81.

Solution (JS) — O(log n) time · O(1) space 
// Route B: one pass. Same half-open skeleton, smarter side-selection.
var search = function(nums, target) {
  let lo = 0, hi = nums.length;   // half-open [lo, hi); nums[hi - 1] is the last live element

  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (nums[mid] === target) return mid;

    if (nums[lo] <= nums[mid]) {
      // LEFT half [lo, mid] has no seam → it is plainly sorted.
      // Only inside a sorted range is a range-membership test valid.
      if (nums[lo] <= target && target < nums[mid]) hi = mid;       // target lives in the clean left
      else lo = mid + 1;                                        // so it must be in the messy right
    } else {
      // RIGHT half [mid, hi) has no seam → it is plainly sorted.
      if (nums[mid] < target && target <= nums[hi - 1]) lo = mid + 1; // target lives in the clean right
      else hi = mid;                                              // so it must be in the messy left
    }
  }

  return -1;
};

// Route A: decompose — find the seam, then run a normal search on one segment.
// Longer, but it's two things you already know, composed. A fine interview answer.
var searchViaPivot = function(nums, target) {
  const n = nums.length, anchor = nums[n - 1];

  let lo = 0, hi = n - 1;                    // LC 153, verbatim
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (nums[mid] <= anchor) hi = mid; else lo = mid + 1;
  }
  const k = lo;                                // index of the minimum = the seam

  // Pick the segment the target could possibly be in, then plain lowerBound it.
  let a, b;
  if (k > 0 && target >= nums[0] && target <= nums[k - 1]) { a = 0; b = k; }
  else { a = k; b = n; }

  while (a < b) {
    const mid = a + ((b - a) >> 1);
    if (nums[mid] >= target) b = mid; else a = mid + 1;
  }
  return (a < n && nums[a] === target) ? a : -1;
};

Complexity: O(log n) time — every branch either sets hi = mid or lo = mid + 1, exactly as in the base template, so hi - lo strictly shrinks and the halving argument is untouched. Follow-ups you should expect:

  • "Now duplicates are allowed" (LC 81). nums[lo] === nums[mid] no longer tells you which half is sorted — [1,1,1,0,1] defeats it. You add a branch: when nums[lo] === nums[mid], do lo++ and retry. Worst case becomes O(n) (all equal), and again, naming the worst case is the answer they want.
  • "Why nums[hi - 1] and not nums[hi]?" Because hi is exclusive. If you slip and read nums[hi] you get undefined, every comparison with it is false, and you get a subtly wrong answer rather than a crash. This is the JS-specific trap in this problem.
  • "Find the rotation count." That's LC 153's return index k — offer it; it shows you see the two problems as one.
If you freeze in the interview, say Route A out loud and code it. "Find the pivot, then binary-search the right segment" is two things you already know composed together, it is the same O(log n), and shipping a correct decomposed solution beats stalling on the elegant one. Enterprise interviewers grade a working answer plus clear narration well above a clever answer you didn't finish. Then, if there's time, say "the one-pass version avoids the second search — want me to write it?"

6. Search on the answer space

This is the highest-leverage idea on the page, and the one that turns binary search from a party trick into a general-purpose tool. Everything so far searched an array. Now the array is not the thing you search at all. You search the space of possible answers, and the array only shows up inside the check that tells you whether a candidate answer works.

The tell in the problem statement is always some flavor of "find the minimum X such that some condition holds" (or maximum). When you see that, run the three-step recipe from Section 2:

  1. What is the answer range? Name the smallest and largest values the answer could conceivably take. This is usually trivially derivable and candidates skip it, then flail.
  2. Given a candidate answer, can I check it in one linear pass? If yes, you have feasible(x).
  3. Is feasible monotonic? One sentence: "if x works, anything bigger also works." If that sentence is true, binary-search the range for the smallest working x. If it's false, you don't have this pattern.
Nothing here requires the input to be sorted. Koko's piles array is in arbitrary order, and it stays in arbitrary order — sorting it wouldn't help. What's monotonic is the predicate over candidate answers, not the data. Internalizing that "binary search needs monotonicity, not sortedness" is the single most useful sentence in this tutorial.

Koko Eating Bananas (LC #875) MEDIUM ★

Koko has n piles of bananas; piles[i] is the number of bananas in the i-th pile. The guards are away for h hours. Koko picks an integer eating speed k (bananas per hour). Each hour she chooses one pile and eats up to k bananas from it — if the pile has fewer than k bananas left, she eats the whole pile and then rests for the remainder of that hour (she does not start another pile in the same hour). She wants to finish all the bananas before the guards return. Return the minimum integer k such that she can eat all the bananas within h hours.

Example 1: Input: piles = [3,6,7,11], h = 8 → Output: 4. At k=4 the hours are ⌈3/4⌉ + ⌈6/4⌉ + ⌈7/4⌉ + ⌈11/4⌉ = 1 + 2 + 2 + 3 = 8 ≤ 8 ✓. At k=3 it's 1 + 2 + 3 + 4 = 10 > 8 ✗. So 4 is the smallest speed that fits.
Example 2: Input: piles = [30,11,23,4,20], h = 5 → Output: 30. There are 5 piles and exactly 5 hours, so she gets exactly one hour per pile — the speed must clear the biggest pile in a single hour. This is the case that pins the upper end of the answer range.
Example 3: Input: piles = [30,11,23,4,20], h = 6 → Output: 23. One spare hour buys her the ability to split the 30-pile into two hours (⌈30/23⌉ = 2), so the constraint drops to the second-largest pile.
Constraints: 1 ≤ piles.length ≤ 10⁴; piles.length ≤ h ≤ 10⁹; 1 ≤ piles[i] ≤ 10⁹. Note h ≥ piles.length is guaranteed — an answer always exists.

Solve on LeetCode ↗

Thinking process. Start where you actually would: brute force. Try k = 1, check if she finishes in time; if not try k = 2; keep going. The check itself is easy — at speed k, pile p costs ⌈p / k⌉ hours (the ceiling is the "she rests the rest of the hour" rule, and it is the entire subtlety of the problem statement) — so total hours is Σ ⌈piles[i] / k⌉, an O(n) sum. The brute force is therefore O(n · maxPile), and with maxPile up to 10⁹ that is hopeless. But look at what the brute force is: it's a linear scan over candidate answers k = 1, 2, 3, …, testing a yes/no predicate at each one and stopping at the first "yes." A linear scan for the first true value of a predicate is the thing binary search replaces. That's the whole leap.

All that's left is to justify the halving. Is feasible(k) = "she finishes within h hours at speed k" monotonic? Yes, and here is the one sentence to say out loud: "eating faster never costs more time — every ⌈p / k⌉ is non-increasing in k — so if she can finish at speed k, she can finish at every speed above k. Feasibility is false…false true…true as k grows, so I'll binary-search the smallest feasible k." Then the range: the slowest conceivable speed is 1 (she must eat something), and the fastest speed that could ever be needed is max(piles) — beyond that she still takes one hour per pile, so extra speed buys nothing, and max(piles) is guaranteed feasible since h ≥ piles.length. So search [1, max(piles)] and you're done.

Solution (JS) — O(n log(max piles)) time · O(1) space 
var minEatingSpeed = function(piles, h) {
  // STEP 1 — the answer range.
  //   slowest useful speed: 1 (she has to eat something)
  //   fastest useful speed: max(piles) — past that, she still burns 1 hour per pile,
  //   so more speed buys nothing. Guaranteed feasible because h >= piles.length.
  let lo = 1, hi = Math.max(...piles);   // [1, max] — max is a known-true frontier, never probed

  // STEP 2 — the feasibility check, O(n).
  // At speed k, pile p costs ceil(p / k) hours: she may not start a second pile
  // in the same hour, so a leftover of even 1 banana still costs a whole hour.
  const feasible = (k) => {
    let hours = 0;
    for (const p of piles) {
      hours += Math.ceil(p / k);
      if (hours > h) return false;   // early exit; also keeps `hours` small
    }
    return hours <= h;
  };

  // STEP 3 — feasible is monotonic in k (faster never hurts), so it reads F..F T..T.
  // This is firstTrue() from Section 2, with an answer range instead of an index range.
  while (lo < hi) {
    const mid = lo + ((hi - lo) >> 1);
    if (feasible(mid)) hi = mid;      // mid works → the answer is mid or slower
    else lo = mid + 1;                // mid is too slow → the answer is strictly faster
  }

  return lo;  // smallest feasible speed
};

Trace on piles = [3,6,7,11], h = 8: range is [1, 11] → lo=1, hi=11.
mid = 1 + (10 >> 1) = 6 → hours = ⌈3/6⌉+⌈6/6⌉+⌈7/6⌉+⌈11/6⌉ = 1+1+2+2 = 6 ≤ 8 → feasible → hi=6.
lo=1, hi=6 → mid=3 → hours = 1+2+3+4 = 10 > 8 → infeasible → lo=4.
lo=4, hi=6 → mid=5 → hours = 1+2+2+3 = 8 ≤ 8 → feasible → hi=5.
lo=4, hi=5 → mid=4 → hours = 1+2+2+3 = 8 ≤ 8 → feasible → hi=4.
lo === hi === 4 → return 4 ✓. Four feasibility checks instead of scanning k = 1…11.

Complexity: O(n · log(max piles)) time. The binary search runs ⌈log₂(10⁹)⌉ ≈ 30 iterations, and each iteration pays O(n) for the feasibility sum — so roughly 30 × 10⁴ = 3×10⁵ operations, trivially fast. Compare the brute force: O(n · max piles) ≈ 10⁴ × 10⁹ = 10¹³. O(1) space. State the complexity as "log of the answer range times the cost of one check" rather than "O(log n)" — being precise that the log is over the value range and not the array length is a senior tell, and it's the thing a sloppy candidate gets wrong.

Follow-ups you should expect:

  • "Prove the monotonicity." ⌈p/k⌉ is non-increasing in k for every pile independently, so the total is non-increasing, so once the total drops to ≤ h it stays ≤ h. Have this ready — fintech interviewers ask for the proof, not just the claim.
  • "Same problem, different clothes." LC 1011 Capacity To Ship Packages Within D Days (range = [max weight, sum of weights]), LC 410 Split Array Largest Sum, LC 1482 Minimum Days to Make Bouquets, LC 1283 Smallest Divisor. Recognizing that these are one problem is the entire payoff of this section; if you see any of them, the three-step recipe is your whole answer.
  • "Why not just sort the piles?" Sorting is a red herring — the feasibility sum is order-independent. Saying "sorting buys nothing here; the monotonicity I need is over k, not over the array" is exactly the recognition being tested.
  • "What if the speed could be fractional?" Then you binary-search the reals: loop while (hi - lo > 1e-6) with mid = (lo + hi) / 2 and no +1 anywhere. Same idea, different termination.
Two ways this goes wrong, both avoidable. (1) Wrong upper bound. Setting hi = h or hi = sum(piles) "to be safe" isn't incorrect, but if you set hi too low — say hi = Math.max(...piles) / 2 because you're optimizing prematurely — the answer falls outside the range and the search silently returns garbage. Derive the bound, don't guess it. (2) Integer division. In JS, p / k is a float, so Math.ceil(p / k) just works; but if you reflexively write the Java-ism (p + k - 1) / k you'll get a float back and break the sum. Write Math.ceil and move on.
The reason this section is worth more than the rest of the page combined: search-on-answer is hidden. Nothing in "Koko Eating Bananas" says "sorted," "O(log n)," or "binary search" — the words that trigger everyone else's pattern-matching are simply absent. What's present is "minimum k such that…", and that phrase alone is the trigger. Enterprise loops love these because they look like scheduling and capacity-planning problems, which is what those companies actually do: minimum number of servers to handle the load, smallest batch size that meets the SLA, lowest rate limit that keeps the queue drained. Say that connection out loud — it reframes you from "candidate solving a puzzle" to "engineer who's seen this shape in production."

7. Binary search inside a design problem

Enterprise loops love this shape: the question looks like "design a small system," and the binary search is buried in one method. Interviewers use it to check that you reach for the right algorithm when nobody has told you an algorithm is needed — which is, of course, how it works at work.

Time Based Key-Value Store (LC #981) MEDIUM

Design a key-value store that keeps every historical value of a key and can answer point-in-time queries. Implement TimeMap with two operations. set(key, value, timestamp) stores the value for the key at the given timestamp. get(key, timestamp) returns the value that was set for that key at the largest timestamp ≤ the queried timestamp; if no such value exists (or the key was never set at all), return the empty string "". Crucially, the problem guarantees that all set calls for a given key arrive with strictly increasing timestamps.

Example 1: set("foo", "bar", 1); get("foo", 1) → "bar" (exact hit); get("foo", 3) → "bar" (nothing was set at 3, so we fall back to the most recent earlier write, at t=1).
Example 2: continuing: set("foo", "bar2", 4); get("foo", 4) → "bar2"; get("foo", 5) → "bar2".
Example 3: get("foo", 0) → "" — the query predates every write for that key. This is the boundary-hits-zero case, and it's the one people drop.
Constraints: 1 ≤ key.length, value.length ≤ 100; 1 ≤ timestamp ≤ 10⁷; at most 2·10⁵ calls total to set and get combined; timestamps for a given key are strictly increasing across set calls.

Solve on LeetCode ↗

Thinking process. Read the guarantee — "set calls for a key have strictly increasing timestamps" — and say what it means: if I append each write to a per-key array, that array is already sorted by timestamp, for free, forever. That single sentence turns a design problem into a binary search problem, and a lot of candidates never say it and end up sorting on every read. So the structure is a Map from key → a pair of parallel arrays (times[] and values[], index-aligned). set is two pushes, O(1).

Now get. The brute force is scanning the key's history backwards for the first timestamp ≤ the query — O(m) per read, and with 2·10⁵ calls all hammering one hot key that's 10¹⁰ operations. But "the largest timestamp ≤ t" in a sorted array is exactly upperBound(times, t) − 1: upperBound gives you the first index strictly after t, so the element just before it is the last one at or before t. Then the only edge case is upperBound === 0, which means every stored timestamp is greater than the query — nothing was written yet — so return "". That's Example 3, and it's the single validation this problem needs. Notice this is the "validate the boundary" discipline from Section 2, showing up in a problem that never mentions binary search.

Solution (JS) — set: O(1) · get: O(log m) · O(total writes) space 
class TimeMap {
  constructor() {
    // key → { times: [ascending], values: [index-aligned with times] }
    // Parallel arrays beat an array of objects here: the binary search only
    // ever touches `times`, so it stays cache-friendly and the code stays flat.
    this.store = new Map();
  }

  set(key, value, timestamp) {
    if (!this.store.has(key)) this.store.set(key, { times: [], values: [] });
    const e = this.store.get(key);

    // The problem guarantees strictly increasing timestamps per key, so a plain
    // push keeps `times` sorted. No sorting, no insertion search. O(1).
    e.times.push(timestamp);
    e.values.push(value);
  }

  get(key, timestamp) {
    const e = this.store.get(key);
    if (!e) return "";   // key never written

    // upperBound: first index whose timestamp is STRICTLY GREATER than the query.
    let lo = 0, hi = e.times.length;
    while (lo < hi) {
      const mid = lo + ((hi - lo) >> 1);
      if (e.times[mid] > timestamp) hi = mid;
      else lo = mid + 1;
    }

    // The element just BEFORE that boundary is the latest write at or before `timestamp`.
    // lo === 0 means every stored write is in the future → nothing to return.
    return lo === 0 ? "" : e.values[lo - 1];
  }
}

Trace: set("foo","bar",1) → times [1], values ["bar"]. get("foo",1): upperBound over [1] for t=1 → lo=0,hi=1, mid=0, 1 > 1? no → lo=1; boundary 1 → return values[0] = "bar" ✓. get("foo",3): 1 > 3? no → boundary 1 → values[0] = "bar" ✓ (the fall-back-to-earlier semantics come out of the boundary, not out of an if). set("foo","bar2",4) → times [1,4]. get("foo",5): lo=0,hi=2 → mid=1, 4 > 5? no → lo=2; boundary 2 → values[1] = "bar2" ✓. get("foo",0): lo=0,hi=2 → mid=1, 4 > 0? yes → hi=1; mid=0, 1 > 0? yes → hi=0; boundary 0 → "" ✓.

Complexity: set is O(1) — a Map lookup plus two amortized-O(1) pushes. get is O(log m) where m is the number of writes for that key (not the total number of writes — be precise about this; it's the kind of distinction that gets noticed). Space is O(total writes), since every version is retained by design. Note what the increasing-timestamp guarantee bought: without it, set would need an insertion at the right position — an O(m) splice — or a different structure entirely.

Follow-ups you should expect:

  • "What if timestamps arrive out of order?" Then set must insert in sorted position: binary-search the insertion point (LC 35, exactly) and splice — O(m) per write because of the array shift, though only O(log m) comparisons. If writes are hot, switch to a balanced BST / skip list, or buffer and sort lazily on first read. Naming the trade-off is the answer.
  • "How would you bound memory?" This is an append-only version store — it grows forever. In production you'd cap retention (keep the last k versions per key, or evict by age) and, on the read path, note that you can prune everything older than the oldest live snapshot. This question is really "have you ever operated a system?" — it's why fintech shops like this problem.
  • "What is this, really?" Multi-version concurrency control. Postgres MVCC, an S3 versioned bucket, a Git object store, or the snapshot semantics behind a React useSyncExternalStore. Saying "this is MVCC" out loud is worth real points.
  • "Return the value at the largest timestamp ≥ t instead." That's lowerBound, and the answer is values[lo] with the validation lo === times.length → "". One knob, flipped.
A JS-specific trap: don't reach for an object literal {} as the outer map. Keys here are arbitrary strings, and an object inherits "constructor", "toString", "__proto__" and friends — store["constructor"] is truthy on an empty object, so your "key never written" check silently lies. new Map() has no prototype keys, is faster for frequent insertion, and preserves any key type. Use Map.

Median of Two Sorted Arrays (LC #4) HARD

Given two sorted arrays nums1 (length m) and nums2 (length n), return the median of the two arrays merged. The required runtime is O(log(m + n)) — which specifically rules out the obvious answer of merging them.

Example 1: Input: nums1 = [1,3], nums2 = [2] → Output: 2.0 — the merged array is [1,2,3], odd length, so the median is the middle element.
Example 2: Input: nums1 = [1,2], nums2 = [3,4] → Output: 2.5 — merged is [1,2,3,4], even length, so the median is the mean of the two middle elements.
Example 3: Input: nums1 = [], nums2 = [1] → Output: 1.0 — one array may be empty, which is the case that breaks every solution that indexes without guarding.
Constraints: 0 ≤ m, n ≤ 1000; 1 ≤ m + n ≤ 2000; −10⁶ ≤ nums[i] ≤ 10⁶; both arrays sorted ascending.

Solve on LeetCode ↗

Thinking process. The brute force is to merge and index the middle: O(m + n) time, O(m + n) space, about six lines, and you should write it first and say "this is O(m+n); the statement wants O(log(m+n)), so let me improve it." A correct brute force on the board is worth more than a broken clever solution, and at an enterprise loop it may well be enough.

Now the real idea. Stop thinking about finding the median and start thinking about partitioning. The median is defined by a cut: split the combined 2000 elements into a left half and a right half of (almost) equal size, such that every element on the left is ≤ every element on the right. Once you have that cut, the median is read straight off the seam — max(left) if the total is odd, or the average of max(left) and min(right) if it's even. And here's the collapse: because each array is already sorted, a cut in the combined array is completely determined by how many elements you take from nums1. Take i from nums1, and you're forced to take j = half − i from nums2 to fill the left side. One unknown. One dimension. Binary search it.

What makes i correct? Two cross-conditions: nums1[i-1] ≤ nums2[j] and nums2[j-1] ≤ nums1[i] (the within-array conditions are free — each array is sorted). And here's the monotonicity: as i grows, nums1[i] grows while j = half − i shrinks so nums2[j-1] shrinks. So the predicate p(i) = "nums1[i] ≥ nums2[j-1]" — "I've taken enough from nums1" — is false…false true…true in i. That's firstTrue again, over the range [0, m], and the smallest i satisfying it is the correct cut (the other cross-condition then holds automatically). Search the smaller array so the range is [0, min(m,n)], which both bounds the runtime by the smaller array and, more importantly, guarantees j stays in [0, n]. Use ±Infinity as the sentinel for the out-of-range edges — that's what makes the empty-array and cut-at-the-end cases evaporate instead of needing four ifs.

Solution (JS) — O(log(min(m, n))) time · O(1) space 
var findMedianSortedArrays = function(nums1, nums2) {
  // Always binary-search the SMALLER array: keeps the range tight AND
  // guarantees j = half - i never falls outside [0, n].
  if (nums1.length > nums2.length) return findMedianSortedArrays(nums2, nums1);

  const m = nums1.length, n = nums2.length;
  const total = m + n;

  // Size of the LEFT partition. The +1 makes the left side hold the extra
  // element when total is odd, so the median is simply max(left).
  const half = (total + 1) >> 1;

  // Sentinels: an index off the left edge is -Infinity, off the right edge is +Infinity.
  // This is what kills every edge case (empty array, cut at position 0, cut at the end).
  const at = (arr, i) =>
    i < 0 ? -Infinity : (i >= arr.length ? Infinity : arr[i]);

  // i = how many elements we take from nums1 into the LEFT partition, in [0, m].
  // p(i) = "nums1[i] >= nums2[half - i - 1]" = "I've taken ENOUGH from nums1".
  // Monotonic: as i grows, nums1[i] grows and nums2[half-i-1] shrinks. F..F T..T.
  let lo = 0, hi = m;   // p(m) is true (nums1[m] is +Infinity), so the answer exists in [0, m]

  while (lo < hi) {
    const i = lo + ((hi - lo) >> 1);
    if (at(nums1, i) >= at(nums2, half - i - 1)) hi = i;   // enough → maybe fewer would do
    else lo = i + 1;                                    // too few from nums1 → take more
  }

  const i = lo, j = half - i;   // the cut: i from nums1, j from nums2

  // Read the median straight off the seam.
  const maxLeft = Math.max(at(nums1, i - 1), at(nums2, j - 1));
  if (total % 2 === 1) return maxLeft;   // odd: left holds the extra element

  const minRight = Math.min(at(nums1, i), at(nums2, j));
  return (maxLeft + minRight) / 2;
};

Trace on Example 1 (nums1 = [1,3], nums2 = [2]): nums1 is longer, so we swap → nums1 = [2] (m=1), nums2 = [1,3] (n=2). total = 3, half = (3+1) >> 1 = 2.
lo=0, hi=1 → i=0: is at(nums1, 0) = 2 ≥ at(nums2, 2−0−1 = 1) = 3? No — we took too few from nums1 → lo=1.
lo === hi === 1 → i=1, j = 2 − 1 = 1. Left partition = {nums1[0]=2} ∪ {nums2[0]=1} = {1,2}; right = {nums2[1]=3}. Check the cut: every left element ≤ every right element ✓. maxLeft = max(at(nums1,0)=2, at(nums2,0)=1) = 2. total is odd → return 2 ✓.

Trace on Example 2 ([1,2], [3,4], no swap needed): m=n=2, total=4, half=2.
lo=0, hi=2 → i=1: is at(nums1,1) = 2 ≥ at(nums2, 2−1−1 = 0) = 3? No → lo=2. lo === hi === 2 → i=2, j=0.
Left = {1,2}, right = {3,4}. maxLeft = max(nums1[1]=2, at(nums2,−1)=−Infinity) = 2. Even → minRight = min(at(nums1,2)=Infinity, nums2[0]=3) = 3 → (2+3)/2 = 2.5 ✓. Note both sentinels fired, and neither needed an if.

Trace on Example 3 ([], [1]): m=0, n=1, total=1, half=1. lo=hi=0, the loop never runs → i=0, j=1. maxLeft = max(at([], −1) = −Infinity, at([1], 0) = 1) = 1. Odd → 1 ✓. The empty array required zero special handling — that's the sentinel doing its job.

Complexity: O(log(min(m, n))) time — we binary-search only the smaller array's cut position, and each probe is O(1) arithmetic. That's better than the required O(log(m+n)), and saying so is a nice close. O(1) space (the one recursive call to swap the arguments is a tail call that runs at most once — inline it with a destructuring swap if the interviewer squints at it).

Follow-ups you should expect:

  • "Generalize to the k-th smallest of two sorted arrays." Same cut idea with half replaced by k. This is the more common and more reasonable question — LC 4 is the special case k = (m+n+1)/2.
  • "Now there are K sorted arrays." The partition trick doesn't generalize cleanly; you'd merge with a min-heap (O(N log K)) or binary-search the answer value and count how many elements fall below it — which is Section 6's pattern wearing a different hat.
  • "Just merge and index." Offer this first, unprompted, with its O(m+n) cost stated. At an enterprise loop it is frequently accepted, and volunteering the simple correct solution before the clever one is a senior habit, not a concession.
Do not attempt LC 4 from memory in a live interview. The partition solution has four index expressions that must all be right simultaneously, and a single off-by-one produces a plausible-but-wrong median that you will not spot by inspection. Either derive it out loud from the cut (the version above is derivable — the sentinels and the (total + 1) >> 1 both have reasons you can state), or write the O(m+n) merge, say why it's O(m+n), and offer the partition idea in words. Confidently shipping the linear solution while describing the logarithmic one beats fumbling the logarithmic one for fifteen minutes.

Interview Q&A

1. Why mid = lo + ((hi - lo) >> 1) instead of (lo + hi) / 2?

The famous answer: integer overflow. In Java, C++, C#, Go — anywhere int is 32-bit — if lo and hi are both large (say both near 2³¹−1, which is exactly the range in First Bad Version, where n ≤ 2³¹ − 1), then lo + hi wraps around to a negative number, mid becomes negative, and you index out of bounds. This bug lived in the JDK's Arrays.binarySearch for nine years and was written up by Joshua Bloch in 2006 as "Nearly All Binary Searches and Mergesorts Are Broken." lo + ((hi − lo) >> 1) is algebraically identical but never forms the large sum, so it cannot overflow.

Now the honest part, because you're a JS developer and the interviewer may be a Java developer. In JavaScript, numbers are IEEE-754 doubles with exact integers up to 2⁵³, so (lo + hi) / 2 genuinely cannot overflow for any array you could ever allocate (JS arrays cap at 2³²−1 elements). The overflow bug is not real in JS and you should be able to say that. But you must still write the safe form and still name the reason, for three reasons: (1) it is a scripted question at Java/C++ shops — JPMC, Goldman and their peers will ask it, and "in JS it doesn't matter, but here's the bug it prevents in Java" is the answer that scores; (2) (lo + hi) / 2 in JS produces a float for odd sums, so you'd need Math.floor anyway; (3) the >> version floors for free.

One real JS trap, though. >> coerces its operand to a signed 32-bit integer. For array indices that's always fine. But in search-on-answer problems the range is over values, and if the range can exceed 2³¹−1 (≈ 2.1×10⁹) — a sum of weights, a byte count, a nanosecond timestamp — then >> silently truncates and your search goes haywire. In that case write lo + Math.floor((hi - lo) / 2). Koko's range tops out at 10⁹ so >> is safe there; know when it isn't.

2. Why the half-open [lo, hi) template rather than the closed lo <= hi one?

Both are correct if written correctly. The half-open version is chosen because the two most common bugs become unrepresentable in it:

  • Infinite loops. Every iteration either strictly decreases hi (hi = mid, and mid < hi always) or strictly increases lo (lo = mid + 1). The interval always shrinks. The closed template's hi = mid with lo <= hi spins forever when lo === hi === mid; you have to remember hi = mid - 1, and under pressure people don't.
  • Off-by-ones at the ends. hi = n means "the answer might be nowhere," and the return value naturally ranges over [0, n] — which is exactly what Search Insert Position needs (an insertion point can be n) and what a not-found result needs. The closed template has to bolt those on.
  • It matches the rest of programming. [start, end) is how slice, Python ranges, C++ iterators and every loop you've ever written are indexed. hi - lo is the length. Consistency is a real defense against error.

The meta-answer, and the one to give: "I use one template for every binary search, I know its invariant, and I never mix skeletons." Which skeleton matters far less than that sentence being true.

3. State the loop invariant. Why is the algorithm correct?

Invariant: everything at an index < lo is known-false; everything at an index ≥ hi is known-true (or off the end); therefore the first true index always lies in [lo, hi].

Initialization: lo = 0, hi = n — nothing is known, and the claim is vacuously true. Maintenance: if p(mid) is true, then by monotonicity every index ≥ mid is also true, so setting hi = mid preserves the invariant. If p(mid) is false, every index ≤ mid is false, so lo = mid + 1 preserves it. Termination: the interval strictly shrinks each pass, so the loop ends with lo === hi; combined with the invariant, that single index is the boundary. That's a complete correctness proof in four sentences, and being able to give it — rather than "trust me, this is the template" — is what distinguishes a 7-YOE answer at a bank.

4. When does binary search not apply, even though the input is sorted?

Sortedness is neither necessary nor sufficient — monotonicity of the predicate is the real requirement. Cases where a sorted array still doesn't help:

  • The predicate isn't monotonic. "Is there a pair summing to X in [i, n)?" isn't a false-then-true function of i, so halving discards real answers. (That problem wants two pointers.)
  • No O(1) random access. Binary search on a linked list is O(n) to reach mid, so the whole thing is O(n log n) — worse than a linear scan. Same for a generator or a stream. Random access is a hard prerequisite, and it's why B-trees exist instead of "binary search the disk."
  • The comparison is expensive or noisy. If p can lie (a flaky test in a git bisect), the halving decision may be wrong and unrecoverable — you have discarded the answer forever.
  • n is tiny. Below roughly 30–60 elements, a linear scan is often faster in practice: it's branch-predictable and cache-friendly, while binary search jumps around memory. Real implementations (including sort routines' insertion-sort cutoffs) exploit this. Don't lead with it in an interview, but know it.

And conversely: Koko's piles is unsorted and binary search works perfectly, because the monotonic thing is the predicate over k. Say "binary search needs monotonicity, not sortedness" and you'll have said the most useful sentence on this page.

5. lowerBound vs upperBound — what's the difference and what do you build with them?
  • lowerBound(a, t) = first index with a[i] >= t. With duplicates, it's the first occurrence; if t is absent, it's the insertion point.
  • upperBound(a, t) = first index with a[i] > t. It's one past the last occurrence.

Everything else is composition, and you should be able to rattle these off:

  • Exists? i = lowerBound(a,t); i < n && a[i] === t
  • First / last index (LC 34): lowerBound(a,t) and upperBound(a,t) - 1
  • Count of t: upperBound(a,t) - lowerBound(a,t)
  • Count strictly less than t: lowerBound(a,t)
  • Largest element ≤ t (LC 981, the "floor" query): a[upperBound(a,t) - 1], with the === 0 guard
  • Smallest element ≥ t (the "ceiling" query): a[lowerBound(a,t)], with the === n guard
  • Integer identity: upperBound(a, t) === lowerBound(a, t + 1) — occasionally lets you write one function instead of two

These are C++'s std::lower_bound/std::upper_bound and Python's bisect_left/bisect_right. JavaScript ships neither, which is exactly why you're being asked to write them.

6. How do you binary-search a continuous (floating-point) answer space?

Drop the +1 and change the termination condition — you can't reach an "empty interval" on the reals, so you iterate until the interval is smaller than the required precision: 

let lo = 0, hi = 1e9;
for (let iter = 0; iter < 100; iter++) {   // fixed iteration count beats an epsilon loop:
  const mid = (lo + hi) / 2;            // no infinite loop when the epsilon is unreachable
  if (feasible(mid)) hi = mid;
  else lo = mid;                        // note: NOT mid + 1 — there is no "next" real
}
return lo;

100 iterations of halving shrinks any range by a factor of 2¹⁰⁰, which is far past double precision — so it's exact and it cannot hang. The while (hi - lo > 1e-6) form is more common but can loop forever when the values are large enough that hi - lo stops shrinking in floating point. This shows up in LC 644 (Maximum Average Subarray II) and in "minimum time / maximum rate" problems with real-valued answers.

7. Why is O(log n) optimal for search? Can you do better?

For comparison-based search on a sorted array, no. Each comparison yields one bit, and you need to distinguish n + 1 outcomes (n positions plus "absent"), so any decision tree has depth ≥ log₂(n+1). Binary search hits that bound, so it's optimal.

You can beat it only by using information beyond comparisons. Hashing gives O(1) expected — but destroys order, so no range or floor/ceiling queries (this trade-off is the actual answer to "why not just use a hash map for LC 981?": you'd get exact-timestamp lookups in O(1) and the "most recent ≤ t" query, which is the whole problem, not at all). Interpolation search guesses where the value should be given a uniform distribution and averages O(log log n) — but degrades to O(n) on skewed data and is never worth proposing in an interview. And a B-tree doesn't beat log n asymptotically; it wins by making the branching factor huge so each disk/cache-line fetch buys many bits — which is why databases index with B-trees rather than binary search trees. Any of these is a strong 30-second answer.

8. Where does binary search actually show up in a frontend/full-stack codebase?
  • Virtualized lists (react-window, TanStack Virtual, ag-Grid). With variable row heights you keep a prefix-sum array of offsets and binary-search it on every scroll event to find the first visible row. That's upperBound, running at 60fps. If you've ever used a windowed table, you've shipped this code.
  • git bisect — LC 278 with commits as versions.
  • Database indexes — B-tree lookups, and the range scan WHERE ts <= $1 ORDER BY ts DESC LIMIT 1 is literally the LC 981 get.
  • Rate limiters and capacity planning — "smallest worker pool that keeps p99 under 200ms" is Koko with different nouns; you binary-search the config and load-test the predicate.
  • Timeline / waveform scrubbing — mapping an x-coordinate to an event index in a sorted event log.
  • Debug bisection in general — halving a 5,000-line diff to find the breaking change, or binary-searching a feature-flag combination. Same idea, applied to your afternoon.

Having one of these ready — especially the virtualized-list one, which is genuinely yours — turns "I know binary search" into "I've used binary search," and that difference is audible.

9. Duplicates keep coming up. Summarize what breaks and what it costs.

Duplicates are the standard escalation in every rotated-array question, and the answer is the same shape each time — the pattern is worth memorizing as a pattern:

  • Plain sorted array with duplicates: nothing breaks. lowerBound/upperBound handle it by construction; that's the point of boundary search. Still O(log n).
  • Rotated + duplicates (LC 81, LC 154): the "which half is sorted?" test nums[lo] <= nums[mid] becomes ambiguous when they're equal — [1,1,1,0,1] looks identical from lo and mid whether the seam is left or right. The fix is a fallback: when nums[lo] === nums[mid], you learn nothing, so shrink the range by one (lo++) and retry.
  • The cost: worst case O(n), when every element is equal. And this isn't a weakness of the fix — no algorithm can do better, because the array [1,1,…,1,0,1,…,1] hides the 0 anywhere, and you cannot avoid looking at essentially every element.

Say the last point out loud. "The fix degrades to O(n), and that's optimal — you can prove no sublinear algorithm exists here" is the answer being fished for; "I'd add a check for duplicates" is not.

10. Give me the 30-second recognition drill.

When a problem lands, run this in your head, in order:

  1. Is the input sorted, or does it demand O(log n)? → boundary search on the index. Decide: exact match (validate the boundary) or the boundary itself (return it raw).
  2. Does it say "minimum/maximum X such that…"? → search on the answer space. Name the range, the O(n) feasibility check, and the monotonicity sentence, in that order, out loud.
  3. Is it sorted-but-rotated? → either anchor on nums[n-1] to find the seam, or use "one half is always sorted."
  4. Is it a design problem with timestamps or versions? → per-key sorted array plus upperBound − 1.
  5. None of the above but you keep wanting to halve something? → write the predicate down explicitly and check it's F…F T…T. If it isn't, this isn't binary search — go look at two pointers or a heap.

Then, whatever you found: state the predicate, state why it's monotonic, write the ten-line template, and validate the boundary. Every problem on this page is that sequence.
