03 — Sliding Window

Fixed and variable windows over contiguous ranges — grow with the right pointer, shrink with the left, never restart.

Why interviewers ask this

Sliding window is arguably the highest-ROI pattern in this entire course for the enterprise tier — banks, fintech, healthcare, and large non-FAANG product companies (Optum, JPMC, Goldman Sachs and their peers). Screens at that tier lean heavily on string and array mediums, and Longest Substring Without Repeating Characters (LC 3) is among the most frequently reported questions across the whole tier — it's short to state, has an obvious brute force, and cleanly separates candidates who know the pattern (10 minutes, clean O(n)) from those who reinvent it under pressure (O(n²) with bugs). The Buy/Sell Stock pair shows up constantly too, because fintech interviewers can't resist a prices array. If you only had one week, this tutorial plus 01-Arrays would cover a startling fraction of what you'll actually be asked.

The pattern also produces strong senior signal when you narrate it well. The core insight — a nested loop re-examines work the previous iteration already did; a window keeps that work and adjusts it incrementally — is exactly the instinct interviewers want to see at 7 YOE, because it's the same instinct behind memoizing a React selector or debouncing a scroll handler instead of recomputing on every event. Interviewers at enterprise loops are grading how you get to O(n): did you name the brute force, identify the redundant work, and then state the invariant your window maintains? That three-beat narration ("brute force is O(n²) because…, the redundancy is…, so I'll maintain a window where…") is rehearsable, and this tutorial drills it on every problem.

1. Recognizing the pattern

Sliding window applies when the answer is about a contiguous range of an array or string, and you can evaluate a candidate range using information you maintain incrementally as the range's endpoints move. The trigger words in a problem statement are remarkably consistent:

  • "Longest / shortest substring…" or "longest / smallest subarray…" — the classic tells. Substring and subarray both mean contiguous; subsequence does not (subsequence problems are usually DP, tutorial 11/13).
  • "…with at most K" — at most K distinct characters, at most K zeros flipped, at most K replacements. The "budget" phrasing almost always means a variable window that grows until the budget is exceeded, then shrinks.
  • "…containing all characters of…" — a coverage condition (Minimum Window Substring). Grow until covered, shrink while still covered.
  • "Every window of size K" or "any substring of length K" — a fixed window. Slide one step at a time: add the entering element, remove the leaving one.
  • "Maximum sum / average of a subarray of length K" — fixed window again; the running sum updates in O(1) per slide.

Window vs. two pointers vs. prefix sum — the decision guide

These three get confused because all involve indices moving across an array. The distinction that keeps you honest in an interview:

TechniqueWhat the indices meanUse whenCanonical problem
Sliding window Two pointers moving the same direction; everything between them is the current candidate answer. Contiguous range + a condition you can maintain incrementally (counts, sums, distinct chars) + monotonic repair: when the window is invalid, shrinking from the left moves it toward valid. LC 3, LC 76, LC 1004
Two pointers (converging) Pointers starting at opposite ends moving toward each other; the pair (or gap) is what's evaluated, not the contents between. Sorted data or a symmetric structure where one comparison tells you which pointer is safe to discard. Two Sum II, Container With Most Water (tutorial 02)
Prefix sum + hash map No moving window — for each index, look up whether a complementary prefix was seen before. Subarray with an exact target sum when elements can be negative — shrinking a window no longer monotonically fixes anything, so the window breaks down. Subarray Sum Equals K (LC 560, tutorial 01)
The most common misapplication: "subarray summing to exactly K" with negative numbers is NOT a sliding window problem. The window technique depends on a monotonic relationship — grow the window, the metric moves one way; shrink it, the metric moves the other way. Negative numbers destroy that (adding an element can decrease the sum), so you can never decide whether to grow or shrink. That's prefix-sum territory. Interviewers at enterprise-tier companies (Goldman among them) are fond of this exact "when does the window fail?" follow-up — it's Q&A #8 below.
When you spot the pattern, say the invariant out loud before coding: "I'll maintain a window that always satisfies <condition>, expand right one step at a time, and shrink left only when the condition breaks." Naming the invariant first is the difference between a senior narration and someone pattern-matching from memorized code — and it makes your own debugging trivial, because every bug is a violated invariant.

2. The templates

Two templates cover every problem on this page. Learn them as shapes, not as code to paste — in the interview you re-derive them in ten seconds from the invariant.

Template A — variable window (grow right, shrink left)

function variableWindow(s) {
  let left = 0, best = 0;
  // ...state that describes the current window (a Map of counts, a sum, a zero-counter)

  for (let right = 0; right < s.length; right++) {
    // 1. EXPAND: bring s[right] into the window state
    //    e.g. count.set(s[right], (count.get(s[right]) || 0) + 1)

    while (/* window [left..right] violates the condition */) {
      // 2. SHRINK: evict s[left] from the state, then advance left
      left++;
    }

    // 3. RECORD: window [left..right] is now valid — its length is right - left + 1
    best = Math.max(best, right - left + 1);
  }
  return best;
}

Line by line, the parts that matter:

  • The for drives right, unconditionally. Every element enters the window exactly once. You never touch right inside the loop body.
  • The while drives left, conditionally. It runs only when the window is broken, and each iteration permanently evicts one element. left only ever moves rightward — it never resets, never backtracks.
  • right - left + 1 is the window length. The + 1 is because both ends are inclusive — forgetting it is the #1 off-by-one in this pattern.
  • Where you record the answer depends on the question. Maximizing (longest valid window)? Record after the shrink loop, when the window is valid again. Minimizing (shortest window satisfying a condition)? Record inside the shrink loop, while the window is still valid and you're testing how small it can get (see LC 76).

Why this is O(n), not O(n²)

A nested while inside a for looks quadratic. It isn't, and being able to argue this crisply is a guaranteed follow-up. The argument is amortized analysis by counting pointer movement: right moves forward exactly n times total. left only moves forward, never backward, so across the entire run of the algorithm it also moves at most n times — no matter how those moves are distributed among iterations. Total pointer movements ≤ 2n, and each movement does O(1) work (with an array/Map counter), so the whole thing is O(n). Don't analyze the worst case of one iteration and multiply — analyze the total work. Each element is added to the window once and removed at most once. Two touches per element. Done.

Template B — fixed window (slide a length-K frame)

function fixedWindow(nums, k) {
  // 1. Build the state for the first window: indices 0 .. k-1
  let windowSum = 0;
  for (let i = 0; i < k; i++) windowSum += nums[i];

  let best = windowSum;

  // 2. Slide: each step, one element enters (index right) and one leaves (index right - k)
  for (let right = k; right < nums.length; right++) {
    windowSum += nums[right];       // entering element
    windowSum -= nums[right - k];   // leaving element — the frame stays exactly k wide
    best = Math.max(best, windowSum);
  }
  return best;
}
  • No shrink loop at all — the window size is fixed by the problem, so every slide is exactly one add and one remove. If you find yourself writing a while in a fixed-window problem, you've misread the problem (or it's actually variable).
  • The leaving index is right - k (equivalently, keep an explicit left and do left++ per slide). Off-by-one check: when right = k, the element leaving is index 0. ✓
  • The state can be richer than a sum — LC 567 slides a 26-slot frequency array and a match counter.
Quick self-test for which template you need: does the problem statement fix the window size? "Of length K" / "every window of size K" → fixed (Template B). "Longest…" / "shortest…" / "at most K <something>" → variable (Template A). LC 567 below is the interesting hybrid: it reads like a search problem but the permutation length pins the window size, making it fixed.
In JavaScript specifically: prefer a plain 26-slot Array over a Map when the alphabet is known to be a–z or A–Z (constraints will say). It's faster, and — more interview-relevant — it makes the O(1) space claim obvious. When the character set is unbounded (unicode, "any printable character"), reach for Map and say that space is O(min(n, alphabet)). Mentioning that tradeoff unprompted is cheap senior signal.

3. Warm-ups — the stock problems

Best Time to Buy and Sell Stock (LC #121) EASY ★

You're given an array prices where prices[i] is the price of a stock on day i. You may choose one day to buy and a later day to sell. Return the maximum profit you can make. If no pair of days produces a profit, return 0 — you're allowed to simply not trade. The "later" is the whole problem: you cannot sell before you buy, so this is not just max - min of the array.

Example 1: Input: prices = [7,1,5,3,6,4] → Output: 5 — buy on day 1 (price 1), sell on day 4 (price 6). Note the global maximum (7, on day 0) is useless: it happens before the minimum.
Example 2: Input: prices = [7,6,4,3,1] → Output: 0 — prices only fall, so every sale loses money; you decline to trade.
Constraints: 1 ≤ prices.length ≤ 10⁵; 0 ≤ prices[i] ≤ 10⁴.

Solve on LeetCode ↗

Thinking process. Brute force: for every buy day i, scan every sell day j > i and track the best prices[j] - prices[i]. That's O(n²), and at n = 10⁵ it's 5 billion comparisons — dead. Now find the redundant work, which is the whole exercise: when you move the sell day forward by one, you re-scan every earlier buy day, even though the only buy day that ever matters is the cheapest one seen so far. Any other buy day is dominated — a lower buy price is at least as good for every possible sell day. So you don't need the history, you need one number. Sweep once, keep minPrice = the cheapest price to the left of today, and at each day ask "if I sold today, what would I make?" — prices[i] - minPrice. Take the max of those. This is a window, and it's worth framing it that way even though the code doesn't look like Template A: left is the buy day, right is the sell day, and "shrinking" is the moment you find a cheaper price and jump left all the way to right. It's the minimal, degenerate case of the pattern — which is exactly why it's the right place to start.

Solution (JS) — O(n) time · O(1) space 
var maxProfit = function(prices) {
  let minPrice = Infinity;  // cheapest buy day seen so far — the window's left edge
  let best = 0;             // 0, not -Infinity: "don't trade" is always legal

  for (const price of prices) {
    if (price < minPrice) {
      // A cheaper buy day dominates every earlier one for ALL future sells,
      // so the old left edge is worthless — collapse the window onto today.
      minPrice = price;
    } else {
      // Selling today against the best buy so far. The else matters:
      // on the day we set minPrice, profit would be 0 anyway — nothing to gain.
      best = Math.max(best, price - minPrice);
    }
  }

  return best;
};

Trace on [7,1,5,3,6,4]: 7 → minPrice 7, best 0. 1 → cheaper, minPrice 1, best 0. 5 → best = max(0, 5−1) = 4. 3 → best = max(4, 3−1) = 4. 6 → best = max(4, 6−1) = 5. 4 → best = max(5, 4−1) = 5. Return 5 ✓ — and notice the algorithm never once considered selling at 7, because 7 had no cheaper day before it.

Complexity: O(n) time — a single pass, O(1) work per day. O(1) space — two scalars, no matter how long the price history is. That O(1) space is the part to say out loud; the brute force didn't need extra space either, so your improvement is purely in time, and you should name that precisely rather than hand-waving "it's faster."

Follow-ups you should expect:

  • "Return the buy and sell days, not just the profit." Track minIndex alongside minPrice, and when you improve best, snapshot [minIndex, i]. Same pass; just remember the bounds. (This "remember where, not just how much" upgrade is exactly what LC 76 demands later — it's the same muscle.)
  • "Now you can buy and sell as many times as you like" — that's LC 122, and the window vanishes entirely: sum every positive daily delta (if (prices[i] > prices[i-1]) profit += prices[i] - prices[i-1]). Recognizing that the constraint change turns a window problem into a greedy one-liner is good signal; you don't need to code it unless asked.
  • "What about at most two transactions / at most k?" LC 123 / LC 188 — that's DP, not sliding window (tutorial 11). Say so, and say why: with multiple transactions the state is no longer "one window," it's "which transaction am I on and am I holding?" Naming the boundary of a pattern is more valuable than attempting the DP badly.
  • "Would this work on a live price stream?" Yes, unchanged — it's a single pass with O(1) state, so it's already an online/streaming algorithm. Fintech interviewers (JPMC, Goldman) like this question; the answer is free, so take the point.
Do not let the interviewer see you reach for Math.min(...prices). Beyond hiding an O(n) scan inside a one-liner, it's wrong here — the global minimum may occur on the last day, where it can never be sold against. The trap in this problem isn't complexity, it's the ordering constraint, and candidates who blurt "it's just max minus min" have failed the actual test. Say the constraint out loud first: "buy must strictly precede sell, so I only ever compare today's price against the minimum to its left."

4. Variable-window classics

These four are the working core of the pattern, and the first two are the highest-frequency sliding-window questions in the entire enterprise tier. Do them in order — each one turns exactly one knob from the table above.

Longest Substring Without Repeating Characters (LC #3) MEDIUM ★

Given a string s, find the length of the longest substring that contains no repeated characters. Substring means contiguous — you cannot skip characters. Return just the length, not the substring itself.

Example 1: Input: s = "abcabcbb" → Output: 3 — "abc" is the longest run with all-distinct characters; extending it to "abca" repeats a.
Example 2: Input: s = "bbbbb" → Output: 1 — every window of length ≥ 2 repeats, so the best you can do is a single character.
Example 3: Input: s = "pwwkew" → Output: 3 — the answer is "wke". This example exists to catch a specific mistake: "pwke" has four distinct characters, but it is a subsequence, not a substring. If your algorithm returns 4 here, you're not maintaining contiguity.
Constraints: 0 ≤ s.length ≤ 5·10⁴; s consists of English letters, digits, symbols and spaces (so the alphabet is not just a–z — reach for a Map).

Solve on LeetCode ↗

Thinking process. Brute force: for every start i, extend j forward, keeping a Set of what you've seen, and stop when you hit a duplicate. That's O(n²) time (and it's worth writing on the whiteboard — it earns you the redundancy observation). Where's the waste? Suppose starting at i you got all the way to j before hitting a duplicate. The brute force now restarts at i + 1 and re-examines i+1 … j from scratch — but you already know those characters are pairwise distinct. That's the redundant work, and killing it is the whole pattern. Instead of restarting, maintain a window [left..right] whose invariant is "all characters inside are distinct." Push right forward one character at a time. If the new character is already in the window, the invariant is broken and there is exactly one way to repair it: advance left past the previous occurrence of that character — because any window still containing that earlier copy is dead. Every character enters once and leaves at most once, so it's O(n). The only real decision left is how you shrink: a while loop evicting one character at a time from a Set (pure Template A), or a Map of last-seen indices that lets you jump left in one step. Both are O(n); the Map version is the one to write, and the trap inside it is worth more than the problem.

Solution (JS) — O(n) time · O(min(n, alphabet)) space 
var lengthOfLongestSubstring = function(s) {
  const lastSeen = new Map();  // char → the most recent index where we saw it
  let left = 0, best = 0;

  for (let right = 0; right < s.length; right++) {
    const ch = s[right];

    // The duplicate only matters if its last occurrence is INSIDE the current
    // window (index >= left). A stale entry from a character we already slid
    // past is not a conflict — and forgetting this check makes `left` move
    // BACKWARD, which silently corrupts the answer. This is THE bug here.
    if (lastSeen.has(ch) && lastSeen.get(ch) >= left) {
      left = lastSeen.get(ch) + 1;  // jump past the old copy in one step
    }

    lastSeen.set(ch, right);  // record/overwrite — must happen AFTER the check

    // Window [left..right] is valid again, so measure it. Maximizing ⇒ record after the repair.
    best = Math.max(best, right - left + 1);
  }

  return best;
};

Trace on "pwwkew":

  • right=0 'p' — unseen. lastSeen{p:0}. window [0..0] = "p", best 1.
  • right=1 'w' — unseen. lastSeen{p:0,w:1}. window [0..1] = "pw", best 2.
  • right=2 'w' — seen at 1, and 1 ≥ left(0) ⇒ conflict. left = 2. lastSeen{w:2}. window [2..2] = "w", best stays 2.
  • right=3 'k' — unseen. window [2..3] = "wk", best 2.
  • right=4 'e' — unseen. window [2..4] = "wke", best = 3.
  • right=5 'w' — seen at 2, and 2 ≥ left(2) ⇒ conflict. left = 3. window [3..5] = "kew", length 3, best stays 3.

Returns 3 ✓, and never at any point considered the non-contiguous "pwke".

Complexity: O(n) time — right advances n times, left only ever moves forward (the jump is a single assignment, and it can only assign a larger value), and Map get/set are O(1). Space is O(min(n, alphabet)): the Map holds at most one entry per distinct character. Say it that way rather than "O(n)" — with the a–z constraint it collapses to O(1), and knowing which bound binds is the senior answer.

Follow-ups you should expect:

  • "Return the substring itself, not the length." Snapshot bestLeft = left whenever you improve best, then s.slice(bestLeft, bestLeft + best). Same pass.
  • "At most K distinct characters instead of zero repeats" — LC 340. Same skeleton, different invariant: keep a Map of counts, and while (map.size > k) evict from the left, deleting the key when its count hits 0. If you can state that transformation without writing it, you've demonstrated the pattern rather than the problem.
  • "Why a Map and not a 26-slot array?" Because the constraints say digits, symbols and spaces — the alphabet isn't a–z. Quote the constraint back at them; interviewers plant it deliberately.
  • "Can you do it with a Set and no jumping?" Yes — while (set.has(ch)) { set.delete(s[left]); left++; } then set.add(ch). It's the purer Template A and is still O(n) (each char is deleted at most once). It's the version to fall back on if you blank on the staleness check.
lastSeen.get(ch) >= left is not defensive padding — omit it and you get wrong answers on inputs like "abba". Walk it: at right=3 ('a'), lastSeen.get('a') is 0, but left has already advanced to 2. Without the guard you'd set left = 0 + 1 = 1 — dragging left backward, re-admitting the 'b' at index 2 into the window, and returning 3 instead of 2. The invariant to hold in your head, and to say out loud: left never decreases. If a line of your code can decrease left, that line is the bug.

Longest Repeating Character Replacement (LC #424) MEDIUM ★

You're given a string s of uppercase English letters and an integer k. You may choose any k positions in the string and replace each with any uppercase letter you like (they don't all have to be the same letter, and you may use fewer than k). After doing so, return the length of the longest substring consisting of a single repeated character.

Example 1: Input: s = "ABAB", k = 2 → Output: 4 — replace both As with B (or both Bs with A) and the whole string becomes one repeated character.
Example 2: Input: s = "AABABBA", k = 1 → Output: 4 — replace the A at index 3, turning the substring "BABB" (indices 2–5) into "BBBB". Note the answer is not the whole string, and the window that wins does not start at index 0 — this is the example that punishes a greedy "extend from the front" approach.
Example 3: Input: s = "AAAA", k = 0 → Output: 4 — no budget needed; the window is already uniform.
Constraints: 1 ≤ s.length ≤ 10⁵; s consists of uppercase English letters only; 0 ≤ k ≤ s.length.

Solve on LeetCode ↗

Thinking process. The problem statement is doing its best to hide the window, so reframe it — this reframing is the interview. Don't think about which letters to replace. Instead ask: for a given window, is it achievable? If you're going to collapse a window into one repeated character, you would obviously keep the letter that already appears most often inside it and replace everything else. So the cost of a window is windowLength − countOfMostFrequentCharacter, and the window is valid iff that cost ≤ k. That single sentence converts a vague "choose replacements" problem into the purest possible instance of Template A: grow right, and while the cost exceeds k, shrink from the left. The brute force — check all O(n²) substrings, each costing O(n) or O(26) to evaluate — collapses to O(n) for exactly the usual reason: the frequency counts update incrementally as the endpoints move, so you never recount a window from scratch. Now the part that gets argued about: what is maxCount, the most-frequent count, when the window shrinks? The naive instinct is to recompute it. You don't have to — and the reason why is the best thing you can say on this problem. See the note below the solution.

Solution (JS) — O(n) time · O(1) space (26 slots) 
var characterReplacement = function(s, k) {
  const count = new Array(26).fill(0);  // counts INSIDE the current window; A–Z ⇒ 26 slots, O(1) space
  let left = 0, maxCount = 0, best = 0;

  for (let right = 0; right < s.length; right++) {
    // EXPAND: s[right] joins the window. 65 is 'A'.
    const r = s.charCodeAt(right) - 65;
    count[r]++;
    maxCount = Math.max(maxCount, count[r]);  // only ever grows — see the note below

    // The cost of making this window uniform = keep the most frequent letter,
    // replace all the others. Invalid when that exceeds our budget k.
    while ((right - left + 1) - maxCount > k) {
      // SHRINK: evict s[left], then advance. left never moves backward.
      count[s.charCodeAt(left) - 65]--;
      left++;
    }

    // RECORD: window is valid again. Maximizing ⇒ record after the repair.
    best = Math.max(best, right - left + 1);
  }

  return best;
};

Trace on s = "AABABBA", k = 1:

  • right=0 'A' — count{A:1}, maxCount 1. Cost = 1−1 = 0 ≤ 1 ✓. best 1.
  • right=1 'A' — count{A:2}, maxCount 2. Cost = 2−2 = 0 ✓. best 2.
  • right=2 'B' — count{A:2,B:1}, maxCount 2. Cost = 3−2 = 1 ≤ 1 ✓ (replace that one B). best 3.
  • right=3 'A' — count{A:3,B:1}, maxCount 3. Cost = 4−3 = 1 ✓. best 4 (window "AABA" → replace the B).
  • right=4 'B' — count{A:3,B:2}, maxCount 3. Cost = 5−3 = 2 > 1 ✗. Shrink: evict s[0]='A' → count{A:2,B:2}, left=1. Cost = 4−3 = 1 ✓ (maxCount still 3 — stale, and harmless). best stays 4.
  • right=5 'B' — count{A:2,B:3}, maxCount 3. Cost = 5−3 = 2 > 1 ✗. Shrink: evict s[1]='A' → count{A:1,B:3}, left=2. Cost = 4−3 = 1 ✓. best stays 4 — window is "BABB", the one the problem statement names.
  • right=6 'A' — count{A:2,B:3}, maxCount 3. Cost = 5−3 = 2 > 1 ✗. Shrink: evict s[2]='B' → count{B:2}, left=3. Cost = 4−3 = 1 ✓. best stays 4.

Returns 4 ✓.

Complexity: O(n) time — right makes n moves, left makes at most n total, and every window update is O(1) (an array increment, not a scan). O(1) space: 26 fixed slots regardless of input length. If the interviewer asks for the O(26) recompute of maxCount, note it would still be O(26n) = O(n) — a constant factor, not a complexity change. Being able to say "that's a constant-factor argument, not a complexity argument" is a small but reliable senior tell.

Follow-ups you should expect:

  • "Why don't you decrement maxCount when shrinking?" — the money question. Answer in the note below; rehearse it verbatim, because it comes up almost every time this problem is asked.
  • "Change while to if — does it still work?" Yes, surprisingly. Since maxCount never shrinks, the window never shrinks either; it just slides. One eviction per expansion is enough to hold the size. It's a well-known micro-optimization, but write the while — it's the honest template, it's obviously correct, and it survives the follow-up "now return the actual substring."
  • "Now return the substring, not the length." Careful — this is where the stale maxCount finally bites, because a window that merely looks valid under a stale count isn't guaranteed to be genuinely uniform-able. You'd track bestLeft only when you strictly improve best, which is still sound (see the note).
  • "Longest subarray with at most k zeros flipped" — LC 1004, Max Consecutive Ones III. Identical skeleton, with the state reduced to a single zeroCount. It's the easiest problem in this family; if you can see it as the same problem, you've generalized correctly.
Why never decrementing maxCount is correct — say this out loud, don't just believe it. A stale (too-large) maxCount can only make the cost look smaller, so it can only cause you to accept a window you shouldn't. But a wrongly-accepted window is never longer than the best genuinely-valid window you already recorded: to record a new maximum you'd need the window to grow, and a window only grows when right advances past the point where maxCount was legitimately achieved — which would update maxCount for real. Formally: best is only ever updated to a value that some earlier, honestly-valid window actually attained. So the stale count lets junk windows drift by unpunished, but they can never win. The one-liner for the interview: "maxCount is a high-water mark, not the current maximum — a stale value can only let a bad window slide through, never let it beat the best valid one, so the answer is unaffected."

Permutation in String (LC #567) MEDIUM

Given two strings s1 and s2, return true if s2 contains a permutation of s1 as a substring — that is, if some contiguous slice of s2 is a rearrangement of s1's characters, using each exactly as many times as s1 does. Return false otherwise.

Example 1: Input: s1 = "ab", s2 = "eidbaooo" → Output: true — the slice "ba" at index 3 is a permutation of "ab".
Example 2: Input: s1 = "ab", s2 = "eidboaoo" → Output: false — both a and b appear, but never adjacently. Contiguity is the whole constraint.
Example 3: Input: s1 = "adc", s2 = "dcda" → Output: true — the slice "dca" at index 1. Included because it kills the tempting shortcut of "does s2 contain all of s1's letters somewhere?" — that test would also pass on s2 = "dxcxa", which is false.
Constraints: 1 ≤ s1.length, s2.length ≤ 10⁴; both consist of lowercase English letters.

Solve on LeetCode ↗

Thinking process. The key observation arrives before any code: every permutation of s1 has exactly s1.length characters. So you are not searching for a window of unknown size — the size is pinned. This is Template B, a fixed frame, even though the statement reads like a search problem. Second observation: two strings are permutations of each other iff their character-frequency vectors are identical. So the question becomes: does any length-s1.length window of s2 have the same 26-slot frequency vector as s1? Brute force: slide the frame and compare the two vectors at every position — O(26n), which honestly already passes. But the O(1)-per-slide version is short, and it's what demonstrates you understand incremental state. Maintain a matches counter: how many of the 26 letters currently have have[c] === need[c]. When it hits 26, every letter agrees — you've found a permutation. Each slide changes exactly two letters' counts (one in, one out), so matches can only change by ±1 per letter touched, and you fix it up in O(1). The counter is the trick worth internalizing; it reappears in LC 76, where it's called formed.

Solution (JS) — O(n) time · O(1) space 
var checkInclusion = function(s1, s2) {
  // A window of size s1.length can't even exist — bail before touching an index.
  if (s1.length > s2.length) return false;

  const A = 97;  // 'a'
  const need = new Array(26).fill(0);  // target frequency vector (s1)
  const have = new Array(26).fill(0);  // frequency vector of the current window of s2

  // Build BOTH vectors over the first s1.length characters in one loop.
  for (let i = 0; i < s1.length; i++) {
    need[s1.charCodeAt(i) - A]++;
    have[s2.charCodeAt(i) - A]++;
  }

  // matches = how many of the 26 letters already agree exactly.
  // Letters absent from both count as agreeing (0 === 0) — that's why the
  // target is 26 and not "the number of distinct letters in s1".
  let matches = 0;
  for (let c = 0; c < 26; c++) if (need[c] === have[c]) matches++;

  // Slide the fixed frame. `right` is the entering index; `right - s1.length` is the leaving one.
  for (let right = s1.length; right < s2.length; right++) {
    if (matches === 26) return true;  // checks the window ENDING at right-1

    // One character enters. Its count crossing need[] can only flip matches by 1.
    const inC = s2.charCodeAt(right) - A;
    have[inC]++;
    if (have[inC] === need[inC]) matches++;             // just became exact
    else if (have[inC] === need[inC] + 1) matches--;    // was exact, now overshot

    // One character leaves — symmetric logic, opposite direction.
    const outC = s2.charCodeAt(right - s1.length) - A;
    have[outC]--;
    if (have[outC] === need[outC]) matches++;             // just became exact
    else if (have[outC] === need[outC] - 1) matches--;    // was exact, now undershot
  }

  // The loop checks windows ending at right-1, so the FINAL window is checked here.
  return matches === 26;
};

Trace on s1 = "ab", s2 = "eidbaooo": Initial frame is "ei". need = {a:1, b:1}; have = {e:1, i:1}. Letters a, b, e, i disagree ⇒ matches = 22. Slide to "id", "db", "ba"… When the frame reaches "ba" (indices 3–4): a enters (have[a] 0→1, now equals need[a]=1 ⇒ matches++), and earlier b had entered and d/i left. At that point every one of the 26 letters agrees — a:1=1, b:1=1, and all 24 others 0=0 ⇒ matches = 26. The check at the top of the next iteration fires and returns true ✓.

Complexity: O(n) time where n = s2.length — one O(26) setup (a constant) plus one pass doing O(1) work per slide. Note it is not O(26n): the matches counter is exactly what removes the 26 from the inner loop. O(1) space — two 26-slot arrays, fixed regardless of input size. The constraints promise lowercase-only, which is what licenses the array over a Map; quote that when you justify O(1).

Follow-ups you should expect:

  • "Return all the starting indices, not just true/false" — that's LC 438, Find All Anagrams in a String, and it's the same code: instead of returning true, push the window's start index into a result array and keep sliding. These two problems are genuinely one problem; solve 567 and you get 438 free.
  • "Why is the target 26 rather than the number of distinct letters in s1?" Because you're comparing full frequency vectors, and a letter absent from both strings trivially agrees. If you'd rather track only s1's distinct letters, you must also ensure no foreign letter sneaks in — which the fixed window size already guarantees. Either framing is fine; know which one your code uses.
  • "What if the alphabet is unicode?" Swap the arrays for Maps and the target becomes need.size; space becomes O(k) for k distinct characters in s1. Same algorithm.
  • "Could you just sort each window and compare?" O(n · m log m). Correct, and worth naming as the brute force before you discard it — but say why you're discarding it.
The two else if branches are where this problem is lost. It is tempting to write if (have[inC] <= need[inC]) matches++; — wrong, because matches counts letters that are exactly equal, and a count moving from 3 to 4 when it needs 2 changes nothing. Each count change can cross the equality line in exactly one of three ways: into equality (matches++), out of equality (matches−−), or neither (no change). Enumerate those three cases explicitly, and the code writes itself. Get this wrong and you'll be debugging an off-by-one in a counter you can't see, on a whiteboard, with someone watching.

Minimum Size Subarray Sum (LC #209) MEDIUM

Given an array of positive integers nums and a positive integer target, return the minimal length of a contiguous subarray whose sum is greater than or equal to target. If no such subarray exists, return 0. Note carefully: it's ≥ target, not = target, and the array is all-positive — both facts are load-bearing.

Example 1: Input: target = 7, nums = [2,3,1,2,4,3] → Output: 2 — the subarray [4,3] sums to 7 and has the minimal length. (Longer qualifying subarrays exist, e.g. [2,3,1,2] sums to 8 — but we want the shortest, not the first.)
Example 2: Input: target = 4, nums = [1,4,4] → Output: 1 — a single element can be the answer.
Example 3: Input: target = 11, nums = [1,1,1,1,1,1,1,1] → Output: 0 — the entire array sums to 8, so nothing qualifies. Your Infinity sentinel has to survive this and come back as 0.
Constraints: 1 ≤ target ≤ 10⁹; 1 ≤ nums.length ≤ 10⁵; 1 ≤ nums[i] ≤ 10⁴ (strictly positive — remember this line, it's the one that makes the window legal at all).

Solve on LeetCode ↗

Thinking process. This is the mirror image of everything above, and that's exactly why it's here: the previous problems all maximized a window, so they recorded the answer after the shrink loop restored validity. Here you minimize, so you record inside the shrink loop — while the window is still valid and you are actively testing how much smaller it can get before it breaks. Flipping that one line is the entire lesson. The derivation: brute force is O(n²) (or O(n³) if you re-sum each candidate) — for every start, extend until the sum reaches target. The redundancy is the usual one: when the start advances, you re-add elements you already summed. So maintain a running sum instead. Grow right, adding as you go. The moment sum >= target, the window qualifies — record its length, then immediately try to make it shorter by evicting from the left, and keep evicting for as long as it still qualifies. Why is that sound? Because all elements are positive: removing an element strictly decreases the sum, so shrinking moves you monotonically toward invalid, and growing moves you monotonically toward valid. That monotonicity is the entire license to use a window here — and it's precisely what evaporates when negatives are allowed (Q&A #8).

Solution (JS) — O(n) time · O(1) space 
var minSubArrayLen = function(target, nums) {
  let left = 0, sum = 0;
  let best = Infinity;  // sentinel: "no qualifying subarray found yet"

  for (let right = 0; right < nums.length; right++) {
    sum += nums[right];  // EXPAND

    // Note this is a while, not an if: after one eviction the window may STILL
    // qualify (e.g. a huge element just entered), and we want the smallest one.
    while (sum >= target) {
      // RECORD FIRST — the window is valid RIGHT NOW. This is the flip vs. LC 3/424:
      // minimizing means measuring while valid, then trying to break it.
      best = Math.min(best, right - left + 1);

      // SHRINK: all nums are positive, so this strictly lowers the sum —
      // guaranteed progress toward invalid, so the loop always terminates.
      sum -= nums[left];
      left++;
    }
  }

  return best === Infinity ? 0 : best;  // translate the sentinel to the required 0
};

Trace on target = 7, nums = [2,3,1,2,4,3]:

  • right=0: sum 2. 2 < 7, no shrink.
  • right=1: sum 5. < 7.
  • right=2: sum 6. < 7.
  • right=3: sum 8 ≥ 7 ✓. Record length 3−0+1 = 4 → best 4. Evict nums[0]=2 → sum 6, left 1. 6 < 7, stop.
  • right=4: sum 10 ≥ 7 ✓. Record 4−1+1 = 4 → best stays 4. Evict nums[1]=3 → sum 7, left 2. Still ≥ 7 ✓ — this is why it's a while. Record 4−2+1 = 3 → best 3. Evict nums[2]=1 → sum 6, left 3. Stop.
  • right=5: sum 9 ≥ 7 ✓. Record 5−3+1 = 3 → best stays 3. Evict nums[3]=2 → sum 7, still ≥ 7 ✓, left 4. Record 5−4+1 = 2 → best 2. Evict nums[4]=4 → sum 3, left 5. Stop.

Returns 2 ✓ — the subarray [4,3], exactly as the problem states.

Complexity: O(n) time — the usual amortized argument: right moves n times, left moves at most n times across the whole run, O(1) work per move. O(1) space — a running sum and two indices; no prefix array needed, which is the improvement over the O(n)-space prefix-sum + binary-search solution (see follow-ups).

Follow-ups you should expect:

  • "The problem statement asks for an O(n log n) solution as a follow-up — what is it?" Prefix sums plus binary search: build prefix[] (strictly increasing, because all nums are positive), then for each i binary-search the smallest j with prefix[j] − prefix[i] ≥ target. It's O(n log n) time and O(n) space — strictly worse than your window on both axes. Say that. LeetCode lists it as the harder follow-up, and knowing that your O(n)/O(1) answer already dominates it is a confident, correct thing to point out.
  • "Now the array can contain negatives." The window breaks outright — see Q&A #8. For "exactly K" with negatives you'd use prefix sums + a hash map (LC 560); for "at least K" with negatives you need a monotonic deque over prefix sums (LC 862, Hard). Naming the escalation is plenty at this tier.
  • "Return the subarray itself." Snapshot bestLeft = left whenever you improve best, then slice. Same pass — the same "remember the bounds" upgrade as LC 121 and LC 76.
  • "What if target is huge and nothing qualifies?" Example 3 covers it: best stays Infinity and you return 0. Interviewers love watching whether you initialized best to 0 by reflex — which would make Math.min return 0 forever and silently break every case.
The single most transferable sentence on this page: "Maximizing a window means recording after the shrink loop, when the window has just become valid again. Minimizing means recording inside the shrink loop, while the window is still valid and I'm testing how much I can remove." Every sliding-window problem you will ever see is one of those two, and candidates who haven't articulated the distinction reliably put the Math.min in the wrong place and then can't explain why their answer is off by one. Say it before you code; it also tells the interviewer you've generalized rather than memorized.

Minimum Window Substring (LC #76) HARD ★

Given two strings s and t, return the shortest substring of s that contains every character of t, including duplicates. If no such substring exists, return the empty string "". The window may contain extra characters not in t — that's allowed; it just has to cover t. Multiplicity is the part everyone under-reads: if t = "AABC", a window containing only one A does not qualify. The answer is guaranteed unique if it exists.

Example 1: Input: s = "ADOBECODEBANC", t = "ABC" → Output: "BANC" — it contains A, B and C. Note that "ADOBEC" also covers t and appears earlier, but it's length 6; "BANC" is length 4. You want the shortest, not the first.
Example 2: Input: s = "a", t = "a" → Output: "a".
Example 3: Input: s = "a", t = "aa" → Output: "" — t needs two as and s only supplies one. This is the example that fails any solution using a Set instead of a count map, which is the most common wrong turn here.
Constraints: 1 ≤ s.length, t.length ≤ 10⁵; both consist of uppercase and lowercase English letters. Follow-up posed by the problem itself: can you do it in O(n)?

Solve on LeetCode ↗

Thinking process. Brute force: every substring (O(n²) of them), each checked for coverage in O(n or 26) — O(n³)/O(n²), obviously dead. The window formulation almost writes itself once you name the invariant, and it's the "shrink to minimum" shape you just learned in LC 209, with a coverage condition instead of a sum: grow right until the window covers t; then shrink left as far as you can while it still covers t, recording the length each step; when coverage breaks, resume growing. Two implementation problems remain, and they're the only real content. First: how do you check coverage in O(1) rather than rescanning 26 counters? Do what LC 567 did — maintain a counter, here called formed, of how many distinct characters of t currently have their full required multiplicity present in the window. Coverage holds exactly when formed === required, where required is the number of distinct characters in t. Increment formed only at the precise moment a character's window count reaches its needed count (===, never >= — going from 3 to 4 when you need 2 changes nothing), and decrement it only when a count drops below what's needed. Second: you must return the substring, not its length, so recording bestLen is not enough — you have to snapshot bestLeft at the same instant. That's the same "remember the bounds" upgrade the follow-ups to LC 121 and LC 209 kept hinting at; this problem is where it becomes mandatory.

Solution (JS) — O(s + t) time · O(s + t) space 
var minWindow = function(s, t) {
  if (t.length > s.length || t.length === 0) return "";

  // need: how many of each char t requires (duplicates matter — hence a Map, not a Set)
  const need = new Map();
  for (const ch of t) need.set(ch, (need.get(ch) || 0) + 1);

  const required = need.size;   // DISTINCT chars we must fully satisfy
  let formed = 0;             // how many of those are currently satisfied IN FULL
  const window = new Map(); // counts of relevant chars inside [left..right]

  let left = 0;
  let bestLen = Infinity, bestLeft = 0;  // must remember WHERE, not just how long

  for (let right = 0; right < s.length; right++) {
    const ch = s[right];

    // EXPAND. Characters not in t are simply ignored — they're allowed in the
    // window, they just never affect coverage (only its length).
    if (need.has(ch)) {
      window.set(ch, (window.get(ch) || 0) + 1);
      // === not >=: formed counts chars that JUST became fully satisfied.
      // Going from 3 to 4 copies when we need 2 must not increment anything.
      if (window.get(ch) === need.get(ch)) formed++;
    }

    // SHRINK while the window is STILL valid — minimizing, so we record inside.
    while (formed === required) {
      if (right - left + 1 < bestLen) {
        bestLen = right - left + 1;
        bestLeft = left;              // snapshot the bounds at the same instant
      }

      const lch = s[left];
      if (need.has(lch)) {
        window.set(lch, window.get(lch) - 1);
        // Only a DROP BELOW the requirement breaks coverage. Dropping from
        // 4 copies to 3 when we need 2 keeps the window valid — keep shrinking.
        if (window.get(lch) < need.get(lch)) formed--;
      }
      left++;  // left never moves backward — the invariant that makes this O(n)
    }
  }

  return bestLen === Infinity ? "" : s.slice(bestLeft, bestLeft + bestLen);
};

Trace on s = "ADOBECODEBANC", t = "ABC": need = {A:1, B:1, C:1}, required = 3.

  • right walks A D O B E C. At A (idx 0) window{A:1} ⇒ formed 1. At B (idx 3) ⇒ formed 2. At C (idx 5) ⇒ formed 3 = required.
  • Shrink loop fires. Window is [0..5] = "ADOBEC", length 6 → bestLen 6, bestLeft 0. Evict s[0]='A': window{A:0} < need{A:1} ⇒ formed drops to 2, left = 1. Loop exits.
  • Growing resumes: O D E B. At B (idx 9) window has B:2 — but need{B} is 1, so window.get('B') === 2 !== 1, no increment. Correct: B was already satisfied. formed still 2.
  • At A (idx 10) ⇒ window{A:1} === need{A:1} ⇒ formed 3. Shrink: window [1..10] = "DOBECODEBA", length 10 — not better than 6, so no record. Evict from the left: D, O, B (B goes 2→1, still ≥ need 1 ⇒ formed stays 3, keep shrinking), E, C — evicting C (idx 5) drops C to 0 < 1 ⇒ formed 2, left = 6. Loop exits.
  • Growing resumes to C (idx 12) ⇒ formed 3. Shrink: window [9..12] = "BANC", length 4 < 6 → bestLen 4, bestLeft 9. Evict B ⇒ formed 2, left 10. Loop exits. String ends.

Returns s.slice(9, 13) = "BANC" ✓. Note the algorithm found "ADOBEC" first and then beat it — which is precisely why you record the minimum rather than returning the first hit.

Complexity: O(s + t) time. Building need is O(t). The main loop is the standard amortized argument: right advances |s| times, left advances at most |s| times over the entire run, and every step is O(1) Map work. Space is O(s + t) in the worst case for the two Maps, though it's really O(distinct chars) — bounded by 52 given the constraints, so you can fairly call it O(1) space and say why. The s.slice at the end is O(bestLen), dominated by the pass.

Follow-ups you should expect:

  • "Why === in the expand and < in the shrink, rather than >= and <=?" The single most-asked question about this code. Because formed counts state transitions across the "fully satisfied" line, not the current state. A count going 3 → 4 when you need 2 crosses nothing, so it must change nothing. If you use >=, formed will race past required and the window will report coverage it doesn't have.
  • "Optimize by only visiting characters that are in t." The known refinement: prefilter s into an array of [index, char] pairs for characters that appear in t, and run the window over that. It's a large constant-factor win when t is tiny and s is huge, and it changes nothing asymptotically. Mention it; only implement it if asked.
  • "What if you needed the longest covering window instead?" Trick question — it's the whole string if it covers at all. A good moment to show you can spot a degenerate problem instead of coding one.
  • "Substring with exactly the characters of t, no extras?" That's LC 438/567 (fixed-size window) — you've already solved it. Point back at Permutation in String.
Two wrong turns cost most candidates this problem. (1) Using a Set for t. It silently ignores duplicates and fails Example 3 (t = "aa"). If t can repeat a character — and it can — you need counts. (2) Checking coverage by comparing whole maps inside the loop ([...need].every(([c,n]) => (window.get(c)||0) >= n)). It's correct but it's an O(26) scan on every single step, and worse, it obscures the actual insight. The formed counter is the answer to "how do I check a set-coverage condition in O(1) as the set changes incrementally" — that's the reusable idea, and it's what the interviewer is watching for.

Sliding Window Maximum (LC #239) HARD

You're given an array nums and an integer k. A window of size k slides from the far left of the array to the far right, moving one position at a time. Return an array containing the maximum value in the window at each of its positions. For an array of length n there are exactly n − k + 1 window positions, so that's the length of your output.

Example 1: Input: nums = [1,3,-1,-3,5,3,6,7], k = 3 → Output: [3,3,5,5,6,7]. Position by position: [1,3,-1]→3, [3,-1,-3]→3, [-1,-3,5]→5, [-3,5,3]→5, [5,3,6]→6, [3,6,7]→7. Watch the third window: the max (3) left the window, and the new max came from an element that was previously being dominated. That's the entire difficulty of the problem.
Example 2: Input: nums = [1], k = 1 → Output: [1].
Example 3: Input: nums = [7,2,4], k = 2 → Output: [7,4] — the max can be the oldest element in the window, which is why you can't just track "the biggest thing I've seen recently."
Constraints: 1 ≤ nums.length ≤ 10⁵; −10⁴ ≤ nums[i] ≤ 10⁴; 1 ≤ k ≤ nums.length.

Solve on LeetCode ↗

Thinking process. The brute force is trivial — for each of the n−k+1 windows, scan its k elements — giving O(nk), which at n = 10⁵ and k = 10⁵ is 10¹⁰. Dead. So why can't you just keep a running maximum, the way LC 121 kept a running minimum? Because elements expire. A running max works when elements only ever enter; here, the element that was the max slides out of the window and you need the next-best — but you threw it away. A max-heap fixes that (O(n log k), with lazy deletion of stale indices) and is a perfectly respectable answer to give. The O(n) answer requires one more observation, and it's the good one: if nums[i] >= nums[j] and i > j, then j can never be the maximum again — for as long as j is inside any future window, the newer and bigger i is inside it too. So j is permanently dominated and can be discarded on the spot. What survives, then, is exactly a decreasing sequence of indices: the current max at the front, then the element that would take over if the front expired, then the one after that. That is a monotonic deque.

The connection to tutorial 04 — say this out loud. This is the same invariant as the monotonic stack ("the structure holds elements still waiting for their turn; a new element evicts everyone it dominates"), with exactly one addition: elements can also die of old age. A monotonic stack only ever evicts from one end, because its elements are only ever invalidated by value. Here elements are invalidated by value at the back (a bigger newcomer dominates them) and by age at the front (they slid out of the window). Two eviction reasons ⇒ two ends ⇒ deque instead of stack. That sentence — "it's a monotonic stack that also needs to expire from the front, so it becomes a deque" — is the whole derivation, and it's how you get to this solution under pressure without having memorized it.

Solution (JS) — O(n) time · O(k) space 
var maxSlidingWindow = function(nums, k) {
  const res = [];
  const dq = [];    // INDICES; nums[dq] is strictly decreasing from front to back
  let head = 0;  // manual front pointer — see the note: never use shift()

  for (let right = 0; right < nums.length; right++) {
    // 1. EVICT FROM THE BACK — by VALUE.
    // Anyone smaller than the newcomer is dominated: nums[right] is both bigger
    // AND newer, so it outlives them in every future window. They can never win.
    while (dq.length > head && nums[dq[dq.length - 1]] <= nums[right]) {
      dq.pop();
    }
    dq.push(right);  // the newcomer now waits its turn at the back

    // 2. EVICT FROM THE FRONT — by AGE.
    // The leader may have slid out of the window. Only ONE element leaves per
    // step, so an `if` suffices here — no while needed.
    if (dq[head] <= right - k) head++;

    // 3. RECORD. Once the window is full (right is at index k-1 or later),
    // the front of the deque is BY CONSTRUCTION the max of [right-k+1 .. right].
    if (right >= k - 1) res.push(nums[dq[head]]);
  }

  return res;
};

Trace on nums = [1,3,-1,-3,5,3,6,7], k = 3 (showing the deque as values for readability):

  • right=0 (1) — deque empty, push. dq = [1]. Window not full.
  • right=1 (3) — 3 ≥ 1, so pop the 1 (dominated: newer AND bigger). Push 3. dq = [3]. Not full.
  • right=2 (−1) — −1 < 3, nothing to evict. Push. dq = [3, −1]. Window [1,3,−1] is full → record front = 3 ✓.
  • right=3 (−3) — −3 < −1. Push. dq = [3, −1, −3]. Front index is 1, not ≤ 3−3=0, so it stays. Record 3 ✓.
  • right=4 (5) — 5 dominates −3, then −1, then 3 — all popped. dq = [5]. Record 5 ✓. (Note the front eviction by age wasn't even needed: value-domination had already cleared index 1.)
  • right=5 (3) — 3 < 5. Push. dq = [5, 3]. Record front 5 ✓.
  • right=6 (6) — 6 dominates 3, then 5. dq = [6]. Record 6 ✓.
  • right=7 (7) — 7 dominates 6. dq = [7]. Record 7 ✓.

Output [3,3,5,5,6,7] ✓. Notice the deque is never longer than k, and every index is pushed once and popped once.

Complexity: O(n) time — and give the amortized argument, exactly as with the monotonic stack: each index is pushed to the deque exactly once and removed at most once (either popped from the back by a bigger newcomer, or skipped past at the front by age). That's ≤ 2n deque operations across the entire run, so the inner while is spending from a budget of n, not doing O(k) work per iteration. O(k) space — the deque holds at most one index per window position, since it's strictly decreasing and all its indices are in-window.

Follow-ups you should expect:

  • "Can you do it with a heap?" Yes: push [value, index], and before reading the top, lazily pop anything whose index has expired. O(n log k) time, O(n) space. Worth naming immediately — it's a legitimate answer and it buys you goodwill while you work toward the deque. (You'd need to hand-roll the heap in JS; there's no built-in — see tutorial 08.)
  • "Why <= and not < when evicting from the back?" With duplicates, an equal-valued older element is also useless (the newer copy survives at least as long and is just as big), so evicting it keeps the deque smaller. Using < is still correct — it just keeps redundant equal elements around. This is the "strictness" knob from tutorial 04's monotonic-stack table.
  • "Sliding window minimum?" Flip the comparison and keep an increasing deque. Same code. This symmetry is the tell that you've learned the pattern, not the problem.
  • "Where does this show up in real systems?" Streaming metrics — a rolling max/min over the last k events, or over a time window, in monitoring and trading systems. It's genuinely used, which is why fintech interviewers occasionally reach for it despite it being a Hard.
JavaScript has no deque, and the naive fix will silently destroy your complexity. Array.shift() removes from the front in O(n) — it re-indexes every remaining element — so a "deque" built from push/pop/shift is quietly O(n·k) in the worst case, and you will have confidently claimed O(n) to an interviewer while writing an O(n²) algorithm. The fix is the one used above: keep a plain array and a manual head index, and "pop the front" by doing head++. Guard the emptiness check as dq.length > head rather than dq.length. (Memory grows to O(n) rather than O(k), since you never reclaim the skipped prefix — an entirely fair trade, and worth saying out loud: "I'm trading O(n) memory for true O(1) front removal, because JS arrays don't give me an O(1) shift." That one sentence is exactly the kind of language-aware detail that separates a 7-YOE candidate from a 2-YOE one.)

Interview Q&A

Conceptual questions that actually get asked about this pattern — in screens, and as the "so why does that work?" follow-up after you've passed the tests. Answer each one out loud before you open it. If you can't say it in two sentences, you don't have it yet.

1. Your loop has a while nested inside a for. Justify O(n).

Don't analyze one iteration and multiply — count total pointer movement across the whole run. right advances exactly n times. left only ever moves forward and never resets, so over the entire execution it advances at most n times, regardless of how those moves are distributed among iterations. Total movements ≤ 2n, each doing O(1) work, so O(n). Equivalently: every element enters the window exactly once and leaves at most once — two touches per element.

This is amortized analysis, and naming it is the point. The while isn't extra work per iteration; it's spending from a fixed budget of n evictions. One iteration might evict twenty elements, but then twenty later iterations evict none. It's the identical argument to the monotonic stack in tutorial 04 ("each element is pushed once and popped once") and to the two-stack queue's amortized O(1) pop — three different problems, one argument. Learn the argument once.

2. Do you record the answer inside the shrink loop or after it? How do you decide?

It depends entirely on whether you're maximizing or minimizing, and it's the single most common place to put a correct algorithm's one wrong line:

  • Maximizing ("longest window satisfying X"): record after the shrink loop. While the loop is running the window is invalid — measuring it would be meaningless. You shrink until it becomes valid again, and only then is right − left + 1 a legitimate candidate. LC 3, LC 424.
  • Minimizing ("shortest window satisfying X"): record inside the shrink loop. The loop condition is validity — you're shrinking precisely because the window still qualifies and you want to know how much smaller it can get before it stops qualifying. Every iteration is a valid, shorter candidate. LC 209, LC 76.

The tell that you got it backwards: your minimizing answer is always the first qualifying window rather than the smallest, or your maximizing answer is off by one. Note the shrink condition also inverts — maximizing shrinks while (invalid), minimizing shrinks while (valid) — which is really the same fact stated twice.

3. Fixed window or variable window — how do you tell from the problem statement, in one read?

Ask one question: does anything in the problem pin the window's size? If yes — "of length k", "every window of size k", or something that implies a size — it's fixed (Template B): no shrink loop, just one add and one remove per slide, with the departing index at right − k. If the size is what you're solving for — "longest…", "shortest…", "at most k distinct…" — it's variable (Template A): grow with right, shrink with a conditional while on left.

The word "implies" is where candidates get caught, and it's why LC 567 (Permutation in String) is on this page. Nothing in that statement says "window of size k" — but every permutation of s1 has exactly s1.length characters, so the size is pinned by the problem's semantics rather than by its wording. Spotting that turns a search problem into a fixed-frame slide. Same for LC 438 (Find All Anagrams).

4. In Longest Repeating Character Replacement, why is it safe to never decrease maxCount?

Because maxCount is a high-water mark, not the current maximum. A stale (too-large) value makes the computed replacement cost look smaller than it really is, so the only error it can cause is accepting a window it shouldn't. But a wrongly-accepted window can never beat the answer: to set a new maximum, the window has to grow, and it can only grow past a size that was legitimately achieved when maxCount was honestly that high. So best is only ever assigned a value that some genuinely-valid window actually attained. Junk windows drift through; they never win.

One-liner for the interview: "A stale maxCount can let a bad window slide by, but it can't let one beat the best valid window — so the answer is unaffected, and I save an O(26) rescan on every shrink." This is the most-asked follow-up on the problem; rehearse it. (If the interviewer wants no cleverness, recomputing maxCount on each shrink is O(26) = O(1) anyway — the complexity is identical, so you lose nothing by conceding it.)

5. Map, plain object, or a 26-slot array for the window's counts? What does the choice do to your complexity claim?
  • 26-slot Array — use it when the constraints promise a–z or A–Z (LC 424, LC 567). Index with charCodeAt(i) − 97. It's the fastest option, and it makes the O(1) space claim self-evident: 26 slots regardless of input size.
  • Map — use it when the alphabet is unbounded: unicode, digits, symbols, spaces (LC 3's constraints explicitly say so; LC 76's say upper and lower case). Space is O(min(n, alphabet)) — say it that way, not just "O(n)", because it shows you know which bound actually binds.
  • Plain object {} — works, but avoid it: keys are coerced to strings, Object.keys().length is O(n) (so tracking "how many distinct" gets expensive, whereas map.size is O(1)), and prototype keys are a real footgun (obj["constructor"] is truthy on a fresh object). Map costs nothing and has none of these problems.

The senior move is to quote the constraint line back at them when you justify the choice — "constraints say lowercase English letters, so I'll use a 26-slot array and this is O(1) space." Interviewers plant that line deliberately; noticing it is free credit.

6. In LC 3, why does the "jump left" version need the lastSeen.get(ch) >= left check?

Because the Map remembers every character ever seen, including ones the window has already slid past. Those entries are stale: they record a position outside [left..right], so they are not conflicts at all. Without the guard you'd "repair" a collision that isn't one and drag left backward.

Concretely, on "abba": at right = 3 ('a'), lastSeen.get('a') is 0, but left is already 2 (it jumped there when the second 'b' collided). Unguarded, you'd set left = 0 + 1 = 1, re-admitting the 'b' at index 2 and returning 3 instead of the correct 2.

The invariant to state and defend: left never decreases. Any line that could decrease it is a bug. (An equivalent, guard-free formulation is left = Math.max(left, lastSeen.get(ch) + 1), which enforces monotonicity structurally — some people find that version harder to get wrong, and it's fine to write it instead.)

7. Sliding window vs. two pointers vs. prefix sum — pick the right one in one sentence each.
  • Sliding window — two pointers moving the same direction, where the contents between them are the candidate answer, and shrinking from the left monotonically repairs an invalid window. Contiguous ranges with an incrementally maintainable condition.
  • Two pointers (converging) — pointers starting at opposite ends moving toward each other, where the pair or the gap is what's evaluated, not the contents. Requires sorted data or a symmetry that lets one comparison prove which pointer is safe to discard (Two Sum II, Container With Most Water).
  • Prefix sum + hash map — no window at all; for each index you look up whether a complementary prefix was seen before. This is the fallback for exact-sum subarray questions, especially when negatives make the window illegal (Subarray Sum Equals K).

The discriminating question, asked in this order: Is the answer a contiguous range? (if no → not a window at all) → Can I maintain its validity incrementally? → Does shrinking from the left monotonically move an invalid window toward valid? If all three are yes, it's a sliding window. If the third fails, it's prefix sums.

8. When does the sliding window break? Give a problem it cannot solve.

The window requires monotonicity: growing must move the metric one way and shrinking must move it the other. Break that and the technique is unusable, because you can no longer decide whether to grow or shrink — which is the algorithm's only decision.

The canonical breaker is negative numbers with a sum condition. Take "find a subarray summing to exactly K" on [3, 4, −7, 1, 2]. Suppose the window's sum is below K. Do you grow? Adding the next element might decrease the sum (it could be −7). Suppose the sum is above K. Do you shrink? Removing an element might increase the sum. Neither pointer has a defensible move, so there is no correct window algorithm. That's why LC 209 (Minimum Size Subarray Sum) states "positive integers" in its constraints — that line is not decoration, it is the licence to use a window. And it's why LC 560 (Subarray Sum Equals K), which permits negatives, is a prefix-sum + hash-map problem (tutorial 01) and not a window problem, despite looking almost identical.

The second, subtler breaker: conditions that aren't monotone in the window at all — a condition that can be false for a window, true for a sub-window and true again for a super-window has no "shrink until it's valid" behavior to exploit.

Say this when you spot it: "Sliding window needs the condition to be monotone under shrinking. With negative values that fails, so I'd switch to prefix sums with a hash map." Enterprise-tier interviewers (Goldman is fond of this) ask "when does your approach fail?" specifically to see whether you understand the precondition or just pattern-matched the shape of the problem. Knowing the boundary of a technique is worth more than one extra solved problem.

9. Why does Sliding Window Maximum need a deque and not a running max, a heap, or a monotonic stack?
  • Not a running max, because elements expire. A running max (as in LC 121) works only when elements exclusively enter. Here the current max eventually slides out of the window, and you need the next-best — which you discarded.
  • A heap works, at O(n log k): store [value, index] and lazily discard the top while its index has fallen out of the window. This is a legitimate answer — offer it immediately, then improve on it.
  • Not a monotonic stack, because a stack has one end. In tutorial 04's monotonic stack, elements are only ever invalidated by value (a newcomer dominates them) — one reason, one end. Here elements are invalidated by value at the back (a bigger, newer element dominates them) and by age at the front (they slid out of the window). Two independent eviction reasons ⇒ you must remove from two ends ⇒ deque.

The invariant is otherwise identical to the monotonic stack's: the structure holds the elements still capable of being the answer, in decreasing order; a newcomer evicts everyone it dominates, then queues up behind them. Same amortized O(n) argument, too — each index is pushed once and removed once.

And the JS-specific trap worth volunteering: there is no deque in JavaScript, and Array.shift() is O(n). Use a plain array with a manual head index and advance it instead of shifting, or your "O(n)" solution is secretly O(n·k).

10. What's the actual space complexity of a sliding window, and when is it not O(1)?

It's the size of the window state, never the window itself — you don't materialize the substring, you only track two indices into it. So:

  • O(1) when the state is a scalar (LC 121's minPrice, LC 209's sum) or a fixed-size frequency array (LC 424, LC 567 — 26 slots).
  • O(min(n, alphabet)) when the state is a Map keyed by character (LC 3, LC 76). Bounded by the alphabet, which the constraints usually cap — for LC 76's upper+lowercase letters that's 52, so it's honestly O(1), and saying "O(min(n, σ)), which the constraints cap at 52" is a strictly better answer than "O(n)".
  • O(k) when the state is a structure that must hold candidates (LC 239's deque) — or O(n) in JS if you use the head-pointer trick and never reclaim the skipped prefix, which is a deliberate time-for-memory trade you should name rather than hide.

The mistake to avoid: claiming O(n) space because you're "looking at substrings." You aren't storing them. If you ever find yourself calling s.slice() or s.substring() inside the loop, you've made the algorithm O(n²) in time and introduced space you don't need — slice exactly once, at the end, from the bestLeft/bestLen you recorded (LC 76).

The 60-second self-test

Before you call this tutorial done, close it and answer these out loud. Any hesitation is a re-read, not a pass:

  • Write Template A from memory. Where does best get updated for a longest problem, and where for a shortest one — and why are they different?
  • Justify O(n) for a while nested in a for, in two sentences, without saying the word "usually."
  • Name three things a window's state can be, and give the space complexity of each.
  • Give a problem that looks exactly like a sliding window and isn't — and say what breaks.
  • What's the departing index in a fixed window of size k when the entering index is right? (If you hesitated: right − k.)
  • Why is right − left + 1 the window length, and what happens to your answer if you forget the + 1?
One last framing to carry into the interview. Every problem on this page is the same three-beat narration, and you should say all three beats before you write a line of code: "The brute force is O(n²) because it re-examines the range from every start. The redundant work is that when the start moves, the elements in between are re-scanned even though I already know what's in them. So I'll maintain a window with the invariant <X>, expand right, and shrink left only when <X> breaks." Fill in <X> — "all characters distinct", "at most k replacements needed", "sum ≥ target", "covers every character of t" — and you have derived the solution out loud, in front of the interviewer, without having memorized anything. That derivation, not the code, is what a 7-YOE loop is grading.
