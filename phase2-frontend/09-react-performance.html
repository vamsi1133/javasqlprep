09 — React Performance & Optimization

The systematic answer to "how do you optimize a slow React app?" — measure first, classify the problem, apply the matching fix, verify with numbers.

Why interviewers ask this

"How would you optimize a slow React app?" is a guaranteed question at the senior level, and it is graded on process, not vocabulary. A mid-level candidate answers with a bag of tricks: "memo, useMemo, useCallback, code splitting, virtualization." A senior candidate answers with a runbook: reproduce the problem, measure it, classify it (startup cost vs. interaction cost vs. re-render storm), apply the fix that matches the classification, verify the fix with numbers, and then put a guardrail in place so it doesn't regress.

The single most important signal you can send is "I measure before I optimize." Interviewers have all watched engineers sprinkle React.memo across a codebase and make it slower and harder to read. If your first sentence is "I'd open the React DevTools Profiler and record the slow interaction," you're already ahead of most candidates. The second most important signal is knowing the trade-offs: memoization has a cost, virtualization breaks Ctrl+F and accessibility if done carelessly, code splitting adds loading states you must design for.

The graded skill: this tutorial builds a rehearsable narrative — a diagnostic tree you can walk through out loud in 3–4 minutes, with one worked story containing real-looking before/after numbers. Interviewers remember stories with numbers; they forget lists of APIs.

1. The performance mental model Tier 1

Before any technique, get the model right, because interviewers probe it directly: re-renders are usually cheap. A render is just calling your component functions and diffing the resulting element trees. React was designed so that re-rendering is the default, normal thing that happens. The goal of optimization is not zero re-renders — it is avoiding expensive renders (large subtrees, heavy computation inside render) and long tasks that block the main thread past a frame budget.

Two distinct budgets, and every performance problem falls into one of them:

BudgetWhat it coversKey metricsTypical fixes
Startup Getting the app on screen: download, parse, execute, first render, hydration Bundle size, LCP, TTI/TBT, hydration cost Code splitting, tree shaking, asset optimization, SSR/RSC, lazy hydration
Interaction Responding to input: typing, clicking, scrolling INP (Interaction to Next Paint), commit durations in the Profiler, long tasks > 50ms Re-render control, virtualization, deferring/debouncing, moving work off the main thread

The frame budget is roughly 16ms at 60fps (~8ms at 120Hz). The browser considers any task over 50ms a "long task" — that's the threshold where an interaction starts to feel laggy, and it's what INP punishes. A React commit that takes 200ms means the user's keystroke took 200ms+ to paint. That's the enemy — not the fact that 40 components re-rendered in 3ms. 

// The render pipeline, and where cost lives:
//
//   event → state update → RENDER PHASE → COMMIT PHASE → browser layout/paint
//
// Render phase (interruptible in concurrent React):
//   - call component functions, build new element tree, diff ("reconciliation")
//   - cost ∝ number of components rendered × work inside each
//
// Commit phase (synchronous, cannot be interrupted):
//   - apply DOM mutations, run layout effects
//   - cost ∝ number of DOM mutations
//
// Browser work after commit:
//   - style recalc, layout, paint, composite
//   - cost ∝ number of DOM nodes affected → why 10k-row tables hurt
//     even when React itself is fast
Pitfall: equating "component re-rendered" with "DOM updated." If a component re-renders and produces the same elements, React diffs them and touches nothing in the DOM. Unnecessary re-renders waste CPU in the render phase, which only matters at scale. Saying "every re-render repaints the DOM" in an interview is a red flag.
Senior framing: "I optimize the expensive path, not every path. First I find out whether we're paying at startup or at interaction time, because the fixes barely overlap. And I never optimize without a profile — my intuition about what's slow has been wrong too many times."

2. Measure first: profiling tools Tier 1

The measure-first doctrine needs concrete tools attached to it, or it sounds like a platitude. Know these four and what each tells you:

2.1 React DevTools Profiler

The primary tool for render-phase problems. Record while performing the slow interaction, then read the flamegraph: each bar is a component, width ∝ render time, colored bars rendered in this commit, gray bars did not. The commit timeline at the top shows every commit in the recording with its duration — a "re-render storm" looks like dozens of commits per keystroke, or single commits that are hundreds of milliseconds wide.

Enable "Record why each component rendered while profiling" in the profiler settings. Then clicking any component tells you: props changed (and which ones), state changed (which hook), parent rendered, or context changed. This one checkbox answers 80% of "why is this re-rendering?" questions.

Note: the Profiler measures React's work. A commit can be 5ms while the browser then spends 300ms in layout because you mutated 10,000 DOM nodes. If the flamegraph looks fine but the app still janks, the problem is downstream of React — go to the Chrome Performance panel.

2.2 The <Profiler> API (programmatic)

For measuring in code — CI perf tests, production sampling, or logging while you iterate: 

import { Profiler } from 'react';

function onRender(id, phase, actualDuration, baseDuration, startTime, commitTime) {
  // id:             the "id" prop of the Profiler tree that committed
  // phase:          "mount" | "update" | "nested-update"
  // actualDuration: time spent rendering this commit (with memoization applied)
  // baseDuration:   estimated time to re-render the ENTIRE subtree without memoization
  //                 → actualDuration much smaller than baseDuration = memoization is working
  if (actualDuration > 16) {
    reportToAnalytics({ id, phase, actualDuration });
  }
}

<Profiler id="ProductTable" onRender={onRender}>
  <ProductTable rows={rows} />
</Profiler>

The actualDuration vs baseDuration comparison is the interview gem: it quantifies how much your memoization is actually saving. If they're nearly equal, your memo calls aren't bailing anything out.

2.3 Chrome Performance panel

For everything outside React: long tasks (red-flagged blocks > 50ms), layout/style recalculation cost, paint cost, script parse/eval at startup, and third-party script cost. Use the Interactions track (Chrome's INP tooling) to trace a specific click/keystroke from input event to next paint. Always profile with 4–6× CPU throttling — your M-series laptop hides problems that every mid-range Android phone will surface.

2.4 Field data (RUM)

Lab profiles reproduce; field data prioritizes. web-vitals reporting INP/LCP/CLS from real users tells you which pages and interactions are actually slow at p75, so you profile the right thing. 

import { onINP, onLCP, onCLS } from 'web-vitals';

onINP((metric) => sendToAnalytics('INP', metric.value, metric.attribution));
onLCP((metric) => sendToAnalytics('LCP', metric.value));
onCLS((metric) => sendToAnalytics('CLS', metric.value));
// Targets at p75: INP < 200ms, LCP < 2.5s, CLS < 0.1
Senior framing: "React DevTools Profiler for render cost and 'why did this render', Chrome Performance panel for main-thread and browser-side cost, the Profiler API when I need numbers in CI, and RUM to decide what's worth profiling in the first place."

3. Re-render control: memo, useMemo, useCallback Tier 1

3.1 Why children cascade (30-second recap)

When a component's state updates, React re-renders it and, by default, its entire subtree — every child it returns, regardless of whether their props changed. React doesn't do a props diff before rendering a child; rendering is how it finds out what changed. Props changing is not the trigger for a re-render — the parent rendering is. This default is fine (renders are cheap) until one of those children is a 5,000-row table.

3.2 React.memo — the bailout

memo wraps a component so that when its parent re-renders, React does a shallow comparison of old props vs new props (Object.is per key) and skips the render if all are equal. 

const ProductRow = memo(function ProductRow({ product, onSelect }) {
  return (
    <tr onClick={() => onSelect(product.id)}>
      <td>{product.name}</td>
      <td>{formatPrice(product.price)}</td>
    </tr>
  );
});

When it pays: the wrapped subtree is expensive (many nodes or heavy work) and its props are referentially stable across the parent's re-renders. Both conditions, or it's dead weight.

When it's useless — the classic failure modes, all of which pass a new reference every render:

// ALL THREE of these silently defeat memo — the comparison never passes:

<MemoRow
  onSelect={(id) => select(id)}          // 1. new function identity every render
  style={{ color: 'red' }}                // 2. new object literal every render
>
  <Icon />                                 {/* 3. children = new element object
                                              every render — memo around a component
                                              that receives JSX children rarely helps */}
</MemoRow>
Pitfall — the custom comparator: memo(Component, arePropsEqual) lets you write your own comparison, and it's how people introduce the worst class of bug: the comparator returns true ("equal, skip") when something the component reads actually changed, and the UI goes stale — it silently shows old data. Deep-equality comparators are also a perf trap: on large props they can cost more than the render they're skipping. Reach for a custom comparator last, and never compare functions "by ignoring them."

3.3 useMemo / useCallback — referential stability

These hooks have two distinct jobs, and interviewers like candidates who separate them:

  1. Skip expensive recomputation (useMemo around a genuinely heavy calc — see §6).
  2. Preserve reference identity so that things downstream that compare by reference — memo'd children, effect dependency arrays, context values — see "unchanged."

Job 2 is why memo and useCallback are a pairing. One without the other is broken:

// BROKEN pair #1: memo'd child, unstable callback → memo never bails out
function Parent() {
  const [query, setQuery] = useState('');
  const handleSelect = (id) => setSelected(id);   // new identity every render
  return <MemoizedTable onSelect={handleSelect} />; // props "changed" every time
}

// BROKEN pair #2: stable callback, child not memo'd → useCallback saves nothing
function Parent() {
  const handleSelect = useCallback((id) => setSelected(id), []);
  return <Table onSelect={handleSelect} />;  // Table re-renders anyway:
}                                              // its parent rendered, no memo to bail

// WORKING pair: stability + a consumer of stability
const Table = memo(TableImpl);
function Parent() {
  const handleSelect = useCallback((id) => setSelected(id), []);
  const columns = useMemo(() => buildColumns(config), [config]);
  return <Table onSelect={handleSelect} columns={columns} />;
}
Note: useCallback(fn, deps) is literally useMemo(() => fn, deps). And stability chains: a useMemo'd object that includes an un-memoized function is still unstable. Audit the whole chain of references feeding a memo'd component.

3.4 When memoization HURTS Tier 1

This is the senior differentiator. Memoization is not free:

  • Comparison cost: every parent render still runs the shallow compare on every memo'd child. For cheap components that would have re-rendered in 0.05ms, you pay the compare and usually the render anyway (because some prop did change).
  • Memory: useMemo/useCallback retain the previous value and deps array per hook per component instance. Thousands of instances × several hooks adds up.
  • Complexity tax: dependency arrays are a maintenance liability — a missed dep is a stale-closure bug, an over-broad dep silently defeats the memo. Wrapped code is harder to read.
  • False confidence: teams that memoize everything stop profiling, and the actual problem (one 300ms computation, one 10k-node list) hides behind a wall of green memos.
Senior take on premature memoization culture: "Default to not memoizing. Memoize at the boundaries the profiler points at — typically a handful of expensive subtrees like tables, charts, and editors — and make the memo actually work by stabilizing exactly the props that cross that boundary. A codebase where every component is wrapped in memo and every value in useMemo is a codebase where nobody profiled."

3.5 The React Compiler changes this calculus

The React Compiler (stable as of React 19 era tooling) auto-memoizes components and values at build time — it does, correctly and exhaustively, what hand-written memo/useMemo/useCallback do partially and error-prone-ly. On compiler-enabled codebases, hand memoization becomes mostly obsolete; your job shifts to keeping components compliant with the Rules of React so the compiler can do its work. Everything else in this tutorial — virtualization, code splitting, context architecture, main-thread management — the compiler does not solve. Full treatment in tutorial 10.

4. Composition as free memoization Tier 1

The most under-used optimization in React needs zero APIs: restructure the tree so the frequently-updating state doesn't sit above the expensive subtree. Two moves, remember them as a slogan: push state down, lift content up.

4.1 Worked example 1 — push state DOWN (state colocation)

// BEFORE: search text lives at the top → every keystroke re-renders
// the whole page, including <ExpensiveAnalyticsPanel /> (80ms render)
function Dashboard() {
  const [search, setSearch] = useState('');
  return (
    <>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <SearchResults query={search} />
      <ExpensiveAnalyticsPanel />   {/* doesn't read search, pays anyway */}
    </>
  );
}

// AFTER: move the state into the only subtree that needs it
function Dashboard() {
  return (
    <>
      <SearchSection />              {/* state lives here now */}
      <ExpensiveAnalyticsPanel />   {/* never re-renders on keystrokes */}
    </>
  );
}

function SearchSection() {
  const [search, setSearch] = useState('');
  return (
    <>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <SearchResults query={search} />
    </>
  );
}

4.2 Worked example 2 — lift content UP (the children prop)

When the stateful component must wrap the expensive one (theming wrappers, scroll trackers, resizable panes), pass the expensive part as children. The key mechanic: the parent creates the children element, so when only the wrapper's own state changes, the children prop is the same element object as last render — and React skips re-rendering it. 

// BEFORE: scroll position updates re-render the heavy table on every scroll tick
function ScrollTracker() {
  const [scrollY, setScrollY] = useState(0);
  // ...scroll listener sets scrollY...
  return (
    <div className={scrollY > 100 ? 'stuck' : ''}>
      <HeavyTable rows={rows} />   {/* re-created and re-rendered per tick */}
    </div>
  );
}

// AFTER: content lifted up — HeavyTable's element is created by App,
// so ScrollTracker's re-renders reuse the identical children reference
function ScrollTracker({ children }) {
  const [scrollY, setScrollY] = useState(0);
  return <div className={scrollY > 100 ? 'stuck' : ''}>{children}</div>;
}

function App() {
  return (
    <ScrollTracker>
      <HeavyTable rows={rows} />   {/* rendered only when App renders */}
    </ScrollTracker>
  );
}
Senior framing: "Before I reach for memo, I ask whether the state is simply in the wrong place. Colocation and the children pattern eliminate whole classes of re-renders structurally — no dependency arrays to maintain, no comparators to get wrong, and it usually improves the component design too."
Pitfall: the children trick only works when the element is created outside the re-rendering component. If ScrollTracker itself wrote children = <HeavyTable /> inside its body, the reference would be new each render and the bailout disappears.

5. The Context re-render problem Tier 1

The rule: when a provider's value changes identity (Object.is), every consumer of that context re-renders — even consumers that only care about a field that didn't change, and even if a memo sits between provider and consumer (context updates tunnel through memo bailouts by design). Context has no selectors. 

// The classic self-inflicted wound: a new object every provider render
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('dark');
  return (
    // ❌ fresh object identity each render → ALL consumers re-render
    // whenever AppProvider re-renders for ANY reason
    <AppContext.Provider value={{ user, theme, setUser, setTheme }}>
      {children}
    </AppContext.Provider>
  );
}

Fixes, ranked from cheapest to most structural:

  1. Memoize the provider value — stops incidental identity churn (provider re-rendered but data didn't change). Doesn't help when the data really does change often. 
    const value = useMemo(() => ({ user, theme, setUser, setTheme }),
                      [user, theme]);   // setters from useState are stable
  2. Split state and dispatch contexts — components that only trigger changes (buttons, forms) never re-render when the data changes, because the dispatch/setter object is stable forever. 
    const StateContext = createContext(null);
const DispatchContext = createContext(null);

function Provider({ children }) {
  const [state, dispatch] = useReducer(reducer, initialState);
  return (
    <StateContext.Provider value={state}>
      <DispatchContext.Provider value={dispatch}>  {/* dispatch is stable */}
        {children}
      </DispatchContext.Provider>
    </StateContext.Provider>
  );
}
// A toolbar that only dispatches: useContext(DispatchContext) → never
// re-renders on state changes.
  3. Slice contexts by update frequency — ThemeContext (changes rarely), UserContext (changes on login), NotificationsContext (changes constantly) as separate providers, so consumers subscribe only to the churn they need.
  4. Selector layer — libraries like use-context-selector approximate useContextSelector(Ctx, s => s.user.name) so consumers re-render only when their slice changes. Know it exists and why (React shipped no built-in selector); it relies on workarounds, so most teams jump to option 5 instead.
  5. Move hot state to an external store subscribed via useSyncExternalStore — this is exactly how Zustand and Redux avoid the problem. The store lives outside React; components subscribe with a selector; the library re-renders a component only when its selected value changes. Context is then used (if at all) just to pass the store instance, which never changes identity. 
    // Zustand: subscription with a selector — no provider re-render cascade
const useStore = create((set) => ({
  notifications: [], user: null,
  addNotification: (n) => set(s => ({ notifications: [...s.notifications, n] })),
}));

function Badge() {
  // re-renders ONLY when the count changes — not on user changes
  const count = useStore(s => s.notifications.length);
  return <span>{count}</span>;
}
Senior framing: "Context is a dependency-injection mechanism, not a state manager. It's perfect for low-churn values — theme, auth, locale. The moment a context value updates many times per second with many consumers, I split it or move it to a store with selector-based subscriptions."

6. Expensive computation & the main thread Tier 1

6.1 useMemo for genuinely heavy calcs — with proof

function ProductList({ products, filters, sortKey }) {
  // Measured in the Profiler: 120ms for 50k products.
  // Without useMemo this re-ran on EVERY render, including ones
  // triggered by unrelated state (open/close a tooltip → 120ms hitch).
  const visible = useMemo(() => {
    const t0 = performance.now();
    const result = sortProducts(filterProducts(products, filters), sortKey);
    console.debug('filter+sort', performance.now() - t0); // temporary proof
    return result;
  }, [products, filters, sortKey]);

  return <VirtualList items={visible} />;
}

The bar for "genuinely heavy": if the computation doesn't show up as a visible chunk in the flamegraph or a performance.now() delta above ~1–2ms, the useMemo is bookkeeping, not optimization. Say that in the interview.

6.2 Input-driven work: debounce or defer

When the expensive work is driven by typing, the fix isn't making one run faster — it's running it less often, or at lower priority than the keystroke echo: 

// Option A: debounce — delay the work until typing pauses (network-ish work)
const [query, setQuery] = useState('');
const [debounced, setDebounced] = useState('');
useEffect(() => {
  const id = setTimeout(() => setDebounced(query), 250);
  return () => clearTimeout(id);
}, [query]);
// input binds to `query` (instant echo); expensive list reads `debounced`

// Option B: useDeferredValue — React-native prioritization (CPU-bound work)
const deferredQuery = useDeferredValue(query);
const results = useMemo(() => expensiveFilter(items, deferredQuery), [items, deferredQuery]);
// Input stays responsive; the list update renders at background priority
// and can be interrupted by the next keystroke. Deep dive → tutorial 10.

Rule of thumb: debounce for I/O (fewer network requests), defer for CPU (same amount of work, lower priority, interruptible).

6.3 Off the main thread: Web Workers

If a single computation is irreducibly 200ms+ (parsing a 20MB CSV, diffing large documents, crypto), no amount of memoization helps — it must leave the main thread: 

// worker.js
self.onmessage = (e) => {
  const rows = parseCsv(e.data);        // heavy, but off the UI thread
  self.postMessage(rows);
};

// Component
function useCsvParser() {
  const workerRef = useRef(null);
  const [rows, setRows] = useState(null);
  useEffect(() => {
    const worker = new Worker(new URL('./worker.js', import.meta.url));
    worker.onmessage = (e) => setRows(e.data);
    workerRef.current = worker;
    return () => worker.terminate();
  }, []);
  return { rows, parse: (file) => workerRef.current.postMessage(file) };
}

Mention Comlink for ergonomics (turns postMessage plumbing into async function calls) and the caveat: data passed to a worker is structured-cloned (copy cost on large payloads) unless you use Transferables (ArrayBuffer) — cloning a 100MB object can itself jank the main thread.

6.4 Chunking long loops

When work must stay on the main thread, slice it so paint can happen between slices:

async function processInChunks(items, fn, chunkSize = 500) {
  for (let i = 0; i < items.length; i += chunkSize) {
    items.slice(i, i + chunkSize).forEach(fn);
    // Yield to the event loop so input/paint can run.
    // scheduler.yield() where supported; setTimeout(0) as fallback.
    await new Promise(r => setTimeout(r, 0));
  }
}

7. List virtualization Tier 1

7.1 Why 10,000 DOM nodes kill you

Rendering 10k rows isn't primarily a React problem — React can build the element tree in tens of milliseconds. The killers are downstream:

  • Initial commit: creating ~10k × (cells per row) real DOM nodes — often 100k+ nodes — takes seconds.
  • Layout & paint: every style recalculation and layout pass now scales with the full node count. A hover style, a window resize, a font load — each triggers work across 100k nodes.
  • Memory: DOM nodes, fibers, and event bookkeeping per row; tabs OOM on mobile.
  • Every subsequent React commit diffs a huge tree even when one row changes.

7.2 Windowing

Render only what's visible plus a small buffer (overscan): a 10k-row list at ~35px row height in a 700px viewport is 20 visible rows + ~5 overscan each side ≈ 30 DOM rows, absolutely positioned inside a spacer div whose height equals the full list (so the scrollbar is honest). On scroll, recycle: swap which items the 30 rows show.

7.3 TanStack Virtual example

import { useVirtualizer } from '@tanstack/react-virtual';

function VirtualList({ items }) {
  const parentRef = useRef(null);
  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 35,   // fixed-size case: exact; variable: an estimate
    overscan: 5,
  });

  return (
    <div ref={parentRef} style={{ height: 600, overflow: 'auto' }}>
      <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
        {virtualizer.getVirtualItems().map((v) => (
          <div
            key={items[v.index].id}
            // variable-size rows: let the virtualizer measure the real height
            ref={virtualizer.measureElement}
            data-index={v.index}
            style={{
              position: 'absolute', top: 0, left: 0, width: '100%',
              transform: `translateY(${v.start}px)`,
            }}
          >
            <Row item={items[v.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}

Overscan trades DOM count for blank-flash protection during fast scrolls — 5–10 rows is typical. Sticky headers are the classic hard part: the header must live outside the recycled rows (or be pinned via the virtualizer's range) because a header row that scrolls out of the window gets unmounted; group-sticky headers need "which group owns the current scroll offset" logic. react-window is the lighter fixed-focus alternative; TanStack Virtual is headless and handles dynamic sizes, horizontal, and grids.

7.4 Infinite scroll + IntersectionObserver

function useInfiniteScroll(loadMore) {
  const sentinelRef = useRef(null);
  useEffect(() => {
    const el = sentinelRef.current;
    if (!el) return;
    const observer = new IntersectionObserver(
      ([entry]) => { if (entry.isIntersecting) loadMore(); },
      { rootMargin: '400px' }   // start loading before the user hits bottom
    );
    observer.observe(el);
    return () => observer.disconnect();
  }, [loadMore]);   // loadMore must be stable — useCallback in the caller
  return sentinelRef;
}
// <div ref={sentinelRef} /> sits after the last row. Pairs with
// TanStack Query's useInfiniteQuery for pages + a virtualizer for DOM count:
// infinite DATA growth without infinite DOM growth.

7.5 Keys recap + the CSS-only cousin

Dynamic lists need stable, identity-based keys — with index keys, prepending one item makes React see every row's props change (row 0's content is now row 1's), forcing a full-list re-render/DOM update and corrupting row-level state (inputs, focus). With ID keys, prepend = one insert. 

/* content-visibility: the CSS-only cousin of virtualization.
   The browser skips layout/paint for off-screen sections; DOM still
   exists (Ctrl+F and a11y keep working), so it's weaker but nearly free. */
.feed-item {
  content-visibility: auto;
  contain-intrinsic-size: auto 120px;  /* placeholder size → honest scrollbar */
}
Pitfall: virtualization's costs — native Ctrl+F can't find unmounted rows, screen readers need correct aria-rowcount/aria-rowindex, and SEO loses non-rendered content. A senior answer names these trade-offs unprompted and mentions content-visibility as the lighter option when the list is only a few hundred items.

8. Code splitting & lazy loading Tier 1

Startup-budget territory: users pay download + parse + execute for every byte of JS before interactivity. The fix is shipping less up front — split by route first (biggest win, natural boundaries), then by heavy component.

8.1 Route-based splitting — the full pattern

import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// Each lazy() call = a separate chunk, fetched on first render of that route.
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings  = lazy(() => import('./pages/Settings'));
const Reports   = lazy(() => import(
  /* webpackChunkName: "reports" */ './pages/Reports'   // named chunk for debugging
));

function App() {
  return (
    <ErrorBoundary fallback={<ChunkErrorScreen />}>   {/* REQUIRED pairing */}
      <Suspense fallback={<PageSkeleton />}>
        <Routes>
          <Route path="/" element={<Dashboard />} />
          <Route path="/settings" element={<Settings />} />
          <Route path="/reports" element={<Reports />} />
        </Routes>
      </Suspense>
    </ErrorBoundary>
  );
}

Why the error boundary is non-optional: chunk loads fail in the wild — flaky mobile networks, and the classic deploy race where the user's open tab requests a hashed chunk that the new deploy deleted. Handle it: an error boundary that offers "reload to update" (many teams auto-reload once on chunk-load errors).

8.2 Component-level splitting: pay on demand

// A rich-text editor is 250KB and used by 5% of sessions — don't ship it to 100%.
const RichTextEditor = lazy(() => import('./RichTextEditor'));

function CommentBox() {
  const [editing, setEditing] = useState(false);
  return editing ? (
    <Suspense fallback={<EditorSkeleton />}>
      <RichTextEditor />
    </Suspense>
  ) : (
    <button onClick={() => setEditing(true)}>Write a comment…</button>
  );
}

Classic candidates: modals, charts, editors, PDF viewers, admin panels, anything behind a click.

8.3 Preloading on intent — the import() promise trick

Lazy loading trades bundle size for a loading flash at click time. Erase the flash by fetching on intent: import() is just a function returning a promise — call it early and the module lands in the module cache, so the later lazy() resolution is instant. 

const importEditor = () => import('./RichTextEditor');
const RichTextEditor = lazy(importEditor);

<button
  onMouseEnter={importEditor}   // hover-to-click gap ≈ 200–400ms of free download
  onFocus={importEditor}        // keyboard users get the same treatment
  onClick={() => setEditing(true)}
>
  Write a comment…
</button>

// into view or is hovered (framework routers ship this built in).

8.4 Bundle analysis workflow — find the 300KB date library

A rehearsable mini-story interviewers love:

  1. Visualize: source-map-explorer 'dist/assets/*.js' or rollup-plugin-visualizer (Vite) / webpack-bundle-analyzer — a treemap of what's actually in each chunk by source module.
  2. Spot the offender: the treemap shows moment + every locale at ~300KB raw (~70KB gz) in the entry chunk, imported by one "format a timestamp" util.
  3. Fix by weight class: replace with date-fns (per-function imports, tree-shakeable — a few KB for what's used) or zero-dependency Intl.DateTimeFormat; other classics — full lodash import instead of lodash-es per-function, a chart lib in the entry chunk for a page nobody starts on, duplicated dependency versions.
  4. Verify & guard: re-run the analyzer, confirm the entry chunk delta, and add a size budget (size-limit / bundlesize in CI) so the regression can't come back silently.

Also know vendor chunking at awareness level: bundlers split stable dependencies (react, react-dom) into a separate long-cached chunk so app-code deploys don't invalidate the vendor bytes users already have. Tooling mechanics in depth → tutorial 16.

JS isn't the only startup cost. The React-adjacent asset checklist:

<img
  src="/products/chair-800.avif"

  • Fonts: font-display: swap vs optional, preloading the one critical WOFF2, and size-adjusted fallbacks — full strategy in tutorial 17.
  • Third-party scripts are frequently the top main-thread consumer in real traces — analytics, chat widgets, A/B tools. Load them defer/after-interactive, lazy-init on idle or interaction, and audit quarterly; the honest senior answer is "the slowest thing in our app wasn't our code."
  • Don't hydrate static content: marketing sections and docs don't need React attached client-side. React Server Components render zero-JS-shipped parts of the tree — the structural fix — covered in the SSR/RSC tutorial.

10. State management performance Tier 2

10.1 Selector granularity

Redux's useSelector re-render rule: after every store dispatch, each subscribed component re-runs its selector and re-renders iff the selected value changed by reference (strict equality by default). Two failure modes: 

// ❌ Selecting the whole slice: re-renders on ANY change within it
const filters = useSelector(state => state.filters);          // big object

// ❌ Worse: constructing a new reference IN the selector →
// "new value" after every dispatch → re-render on EVERY dispatch app-wide
const activeIds = useSelector(state =>
  state.todos.filter(t => t.active).map(t => t.id));           // fresh array each call

// ✅ Select primitives — reference equality just works
const sortKey = useSelector(state => state.filters.sortKey);

// ✅ Derived data: memoize with createSelector (reselect) — recomputes only
// when its INPUTS change, returns the cached (stable) reference otherwise
const selectActiveIds = createSelector(
  [(state) => state.todos],
  (todos) => todos.filter(t => t.active).map(t => t.id)
);
const activeIds = useSelector(selectActiveIds);

// ✅ Or shallow-compare a small object of picks
const { sortKey, page } = useSelector(
  s => ({ sortKey: s.filters.sortKey, page: s.pagination.page }),
  shallowEqual
);

Zustand is the same model, different spelling: useStore(s => s.count) for primitives, useShallow(s => ({a: s.a, b: s.b})) for multi-picks.

10.2 Normalized state shape

// ❌ Deeply nested tree: updating one comment = deep-cloning the path to it,
// and every selector of the tree sees a "new" ancestor object
{ posts: [{ id, title, comments: [{ id, text, author: {…} }] }] }

// ✅ Normalized entity maps (what RTK's createEntityAdapter gives you):
{
  posts:    { byId: { p1: { id:'p1', title, commentIds:['c1','c2'] } }, allIds:['p1'] },
  comments: { byId: { c1: { id:'c1', text, authorId:'u1' } },           allIds:['c1','c2'] },
  users:    { byId: { u1: { id:'u1', name } } }
}
// Editing comment c1 touches comments.byId.c1 (+ the byId map) — posts and
// users keep their references, so their subscribers don't re-render. O(1)
// lookups, no duplicated author objects drifting out of sync.

10.3 Stop putting server data in global state

The biggest architectural perf win of the last five years: server cache ≠ client state. API data in Redux means hand-rolled loading flags, staleness logic, and every fetch dispatching through one global store (every dispatch = every selector re-runs). TanStack Query (or RTK Query/SWR) treats server data as a cache keyed by query, with per-query subscriptions: 

const { data: products, isPending } = useQuery({
  queryKey: ['products', filters],
  queryFn: () => fetchProducts(filters),
  staleTime: 60_000,        // fresh for 1min → no refetch storm on remount/refocus
});
// Only components using this queryKey re-render on its updates.
// Local UI state (open modals, input drafts) stays in useState — colocated,
// cheap, and gone when the component unmounts.

11. Forms performance Tier 2

A controlled input sets state per keystroke — which re-renders whatever owns that state. Fine for a login form; a disaster when the state lives on a 60-field form component: every keystroke re-renders all 60 fields plus validation. Typical profile: 40–80ms per keystroke on mid-range hardware — visible lag. 

// ❌ Scale problem: one state object at the top of a big form
function BigForm() {
  const [values, setValues] = useState(initialValues);   // 60 fields
  const set = (name) => (e) =>
    setValues(v => ({ ...v, [name]: e.target.value }));  // whole form re-renders
  return <>{fields.map(f => <Field key={f.name} value={values[f.name]} onChange={set(f.name)} />)}</>;
}

// ✅ react-hook-form: uncontrolled inputs + a subscription model
import { useForm } from 'react-hook-form';

function BigForm() {
  const { register, handleSubmit, formState: { errors } } = useForm();
  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      {/* register() attaches a ref — the DOM holds the value.
          Typing causes ZERO React re-renders. */}
      <input {...register('email', { required: true })} />
      <input {...register('company')} />
      <button>Save</button>
    </form>
  );
}

// Need to react to one field? Subscribe narrowly — only THIS component
// re-renders as 'plan' changes, the rest of the form is untouched:
function PlanSummary({ control }) {
  const plan = useWatch({ control, name: 'plan' });
  return <Summary plan={plan} />;
}

Why it's fast: react-hook-form keeps values in the DOM (uncontrolled, via refs) and maintains an internal subscription registry — components opt in to specific fields with useWatch/watch, mirroring the external-store-with-selectors pattern from §5. React only gets involved at validation-error transitions and submit.

Isolating hot inputs without a library: give the laggy input its own state in its own component (state colocation, §4.1) and propagate upward on blur/debounce instead of per keystroke. Controlled inputs aren't "wrong" — they're per-keystroke subscriptions; just keep the subscription scope tiny.

12.1 The worked story (rehearse this)

"Typing lag in a filterable 5,000-row table." Support reported the operations dashboard 'freezing while typing' in the filter box. I reproduced with a production data snapshot and 4× CPU throttle — each keystroke visibly hitched. The DevTools Profiler showed one commit per keystroke at ~450ms, and the flamegraph was wall-to-wall <TableRow>: all 5,000 rows re-rendering per keystroke, with 'why did this render' saying parent rendered — the filter state lived on the component that also rendered the table, and rows weren't memoized. The Chrome trace added a second problem: ~180ms of layout after each commit, because all 5,000 rows were real DOM nodes.

Three targeted changes: (1) virtualized the table with TanStack Virtual — DOM went from 5,000 rows to ~30; (2) memo'd TableRow and stabilized its two callback props with useCallback so unchanged visible rows bail out; (3) wrapped the filter text in useDeferredValue with the filtered array in a useMemo, so the keystroke echo commits instantly and the list update renders at background priority, interruptible by the next keystroke.

Result: keystroke commits went from ~450ms to ~4–8ms, the deferred list commit to ~16ms, layout cost collapsed with the DOM count, and field INP on that page went from ~600ms to ~90ms at p75. Then I added a Profiler onRender assertion in an integration test failing above 50ms on that tree, so it can't quietly regress."

Note the grading rubric hidden in that story: reproduced first, profiled second, two tools, classified before fixing, fixes matched the classification, numbers on both sides, and a guardrail. That structure — not the specific APIs — is what earns the senior signal.

13.2 Avoiding layout thrash

// ❌ Interleaved read → write → read forces synchronous layout each iteration
elements.forEach(el => {
  const h = el.offsetHeight;         // READ (forces layout if styles dirty)
  el.style.height = h * 2 + 'px';    // WRITE (dirties styles)
});                                   // next read re-forces layout: O(n) layouts

// ✅ Batch: all reads, then all writes → one layout
const heights = elements.map(el => el.offsetHeight);
elements.forEach((el, i) => { el.style.height = heights[i] * 2 + 'px'; });

// then set state that changes styles can ping-pong — measure in
// useLayoutEffect once, and prefer CSS (flex/grid/aspect-ratio) over
// JS measurement wherever possible.

Interview Q&A

1. "How do you optimize a slow React app?" — the headline answer

Lead with process: reproduce with realistic data and CPU throttling; profile (React DevTools Profiler for render cost + 'why did this render', Chrome Performance for main-thread/browser cost); classify — startup problem, expensive single render, re-render storm, or browser-side layout/paint; apply the matching fix (code splitting / memo+composition / virtualization / batching DOM work); verify with before/after numbers; guard with CI budgets. Then offer one concrete story with numbers (§12.1). Never lead with "add React.memo."

2. Are re-renders bad? When do they actually matter?

No — re-renders are React's normal operation and usually cost fractions of a millisecond. They matter when the rendered subtree is large or does heavy work, when they happen at high frequency (per keystroke, per scroll tick, per websocket message), or when the resulting commits exceed the interaction budget (~50ms long-task threshold, 16ms frame). A re-render that produces identical elements changes zero DOM. Optimize expensive renders and long tasks, not the render count itself.

3. What exactly does React.memo compare, and when does it fail silently?

A shallow comparison: for each prop key, Object.is(prevProps[k], nextProps[k]). It fails silently whenever any prop is recreated per parent render: inline arrow functions, object/array literals, and JSX children (a new element object every render). The component still works — it just never bails out, so you pay the compare plus the render. Fix by stabilizing the props (useCallback/useMemo/hoisting) or restructuring so the memo isn't needed.

4. When would you NOT use useMemo/useCallback?

When nothing consumes the stability: no memo'd child, no effect dep, no context value downstream — then useCallback is pure overhead. When the computation is trivial (the memo bookkeeping costs as much as recomputing). And when the dep array would be so churny it never hits cache. Memoization adds memory retention and dependency-array maintenance risk; I add it where a profiler shows a payoff or where a stability contract exists, not by default. The React Compiler makes the manual versions largely unnecessary where it's adopted.

5. Explain the children-prop optimization. Why does it work?

Move state into a wrapper and accept the expensive subtree as children. When only the wrapper's state changes, the wrapper re-renders but its children prop is the exact same element object the parent created last time — React sees an identical element reference and skips re-rendering that subtree. It's memoization by structure: no APIs, no comparators, no dependency arrays. The condition: the element must be created outside the re-rendering component.

6. Why do all context consumers re-render together, and how do you fix it?

Context has no selectors: when the provider's value changes identity, every consumer re-renders — and this tunnels through memo bailouts by design. Fixes in order: memoize the provider value (kills incidental identity churn); split state and dispatch contexts (dispatch is stable, so trigger-only components never re-render); slice contexts by update frequency; use-context-selector as a stopgap; or move hot state to an external store (Zustand/Redux) subscribed via useSyncExternalStore with selectors, which is how those libraries avoid the problem entirely.

7. How do Redux/Zustand avoid the re-render problem Context has?

The store lives outside React and never changes identity — nothing re-renders because "the store changed." Components subscribe with selectors via useSyncExternalStore; after each store update, each subscriber's selector runs and the component re-renders only if the selected value's reference changed. So granular selectors = granular re-renders. The failure mode is a selector that constructs a new reference each call (a .filter().map() inline) — that re-renders on every dispatch; fix with createSelector/reselect memoization or shallow equality.

8. A component re-renders and you don't know why. Debug it.

React DevTools Profiler with "record why each component rendered" enabled — it reports props changed (which), state changed (which hook), parent rendered, or context changed. If props "changed" but look identical, it's referential: an object/function recreated upstream. For hooks, changed-hook indices point at the culprit state. Escalations: a temporary useEffect logging prev-vs-next props with Object.is per key, or the why-did-you-render library in dev.

9. Walk me through rendering a 50,000-row table.

Don't render 50,000 DOM rows — initial commit takes seconds, and every layout/paint scales with node count forever after. Virtualize: render visible rows + overscan (~30 rows) absolutely positioned in a full-height spacer so the scrollbar is honest, recycling rows on scroll (TanStack Virtual / react-window). Memoize the row component and stabilize its props so scrolling only mounts the entering rows. Stable ID keys. Fixed row height if the design allows (exact math); otherwise measured dynamic sizes. Call out the trade-offs unprompted: Ctrl+F breaks, screen readers need aria-rowcount/rowindex, sticky group headers need pinning logic. If it's only a few hundred rows, content-visibility: auto may be enough.

10. Fixed-size vs variable-size virtualization — what changes?

Fixed size: offset = index × rowHeight — exact positions, exact scrollbar, cheapest. Variable size: you start from an estimate, measure rows as they render (ResizeObserver), and correct offsets — the scrollbar can shift as estimates are replaced by measurements, and scroll-to-index is approximate until neighbors are measured. Use fixed whenever the design permits; variable for feeds/comments/chat where content dictates height.

11. How do you set up route-based code splitting, and what must accompany it?

React.lazy(() => import('./Page')) per route, a Suspense fallback (skeleton, not spinner-flash), and — non-negotiable — an error boundary: chunk loads fail on flaky networks and on the deploy race where a stale tab requests a deleted hashed chunk, so offer/trigger a reload. Then improve perceived speed by preloading chunks on link hover/viewport, and wrap navigations in a transition so the current page stays interactive while the next one loads.

12. Your bundle is 2MB. What's your workflow?

Analyze before cutting: source-map-explorer or the bundler's visualizer to see the treemap of the entry chunk. Typical findings: a monolithic date library with all locales (~300KB — replace with date-fns per-function imports or Intl), full lodash instead of lodash-es imports, a chart/editor library in the entry chunk used on one deep page (lazy-load it), duplicate dependency versions, and missing route splitting. Verify each change in the treemap, then lock it in with a size-limit CI budget. Named dynamic-import chunks keep the output debuggable.

13. What's the import() preloading trick?

import() is just a promise-returning function, and the module cache dedupes it. Hoist the thunk — const load = () => import('./Editor') — pass it to lazy(load), and also call load() on hover/focus/viewport-entry. The 200–400ms of human intent time downloads the chunk, so by click the lazy resolution is instant: bundle savings without the loading flash.

14. Typing in a search box lags. Diagnose and fix.

Profile first — the near-certain signature is a big commit per keystroke because the input's state lives above an expensive subtree. Fixes by layer: colocate the input state (or isolate the hot input and propagate on debounce); split urgency with useDeferredValue so the keystroke echoes instantly and the filtered list renders interruptibly in the background; useMemo the filter computation keyed on the deferred value; virtualize the results if they're long; debounce if the work is network-bound. Verify: commit per keystroke should drop from hundreds of ms to single digits.

15. Debounce vs useDeferredValue — when each?

Debounce delays the work until input pauses — right for I/O (fewer requests) and when you control the delay. It's a fixed time cost even on fast machines and doesn't help once the work starts (still blocks when it runs). useDeferredValue doesn't reduce work — it deprioritizes it: the urgent render commits immediately, the deferred render happens in the background and is interruptible by the next keystroke, adapting to device speed automatically. CPU-bound rendering → defer; network → debounce; heavy both → debounce the fetch, defer the render.

16. When does work belong in a Web Worker, and what are the gotchas?

When a single computation is irreducibly long (100ms+) and can't be chunked or deferred acceptably — parsing large files, heavy transforms, crypto, image processing. Gotchas: no DOM access; postMessage structured-clones payloads, so a 100MB object copy can itself jank the main thread — use Transferables (ArrayBuffer) or keep data worker-side and send back summaries; worker startup isn't free, so pool/reuse; Comlink removes the messaging boilerplate. React integration: create in an effect, terminate on cleanup, deliver results via setState.

17. React DevTools Profiler vs Chrome Performance panel — what does each tell you?

DevTools Profiler: React's render phase — which components rendered per commit, how long, and why (props/state/context/parent). It won't show browser work after commit. Chrome Performance: the whole main thread — long tasks, style/layout/paint cost, script parse/eval, GC, third-party scripts, with the Interactions track tracing input→paint for INP. A 5ms commit followed by 300ms of layout means React is innocent and the DOM/CSS is the problem — you need both tools to see that.

18. actualDuration vs baseDuration in the Profiler onRender callback?

actualDuration is the time this commit actually spent rendering the subtree, with memoization bailouts applied. baseDuration estimates rendering the entire subtree with no bailouts (sum of each component's most recent render time). actualDuration ≪ baseDuration → memoization is earning its keep; roughly equal → your memos aren't bailing anything out and are candidates for removal. It's the built-in way to quantify whether memoization is working.

19. Why is a controlled 60-field form slow, and how does react-hook-form fix it?

Controlled inputs set state per keystroke; when that state lives on the form component, every keystroke re-renders all 60 fields plus validation — tens of ms per key. react-hook-form keeps values in the DOM via refs (uncontrolled) and runs a subscription registry: typing causes zero React renders, and components that need a live value subscribe narrowly with useWatch, so only they update. It's the selector pattern applied to forms. Without a library: colocate each hot input's state and propagate on blur/debounce.

20. What does the React Compiler change about performance work?

It auto-memoizes components and intermediate values at build time — exhaustively and without dependency-array bugs — making manual memo/useMemo/useCallback mostly redundant where it's enabled; your job becomes keeping code compliant with the Rules of React (purity, no mutation during render) so it can apply. It does NOT fix the structural problems: too many DOM nodes, unsplit bundles, context architecture, main-thread-blocking computation, layout thrash. So the runbook survives; the hand-memoization chapter shrinks.

21. INP is 500ms on your dashboard. Where do you look?

First, RUM attribution: which interaction, which element, and how the 500ms splits across input delay (main thread busy when the event fired — often third-party or long hydration), processing (your handlers + the synchronous React commit), and presentation delay (rendering/paint after handlers). Long processing → profile that interaction: giant commit (memo/virtualize/defer) or heavy handler (chunk it, move it post-paint). Long input delay → find what's hogging the thread at idle: scripts, polling, hydration. Long presentation → DOM size and layout cost. Fix, then confirm p75 moves in the field, not just in the lab.

22. Explain key-based reset and why it beats an effect.

Changing key tells React it's a different instance: unmount, remount, all state fresh from initializers. Compared to "reset state in a useEffect when the prop changes": the effect approach renders once with stale state, then re-renders after the effect — a flash of wrong data plus extra code paths; key-reset does one clean initial render with zero synchronization logic. Use it for "edit form per selected record" cases. Cost to note: full remount means effects re-run and DOM rebuilds — intended here, but not for frequently-changing keys.

23. What is layout thrashing and where does it bite in React apps?

Interleaving DOM reads (offsetHeight, getBoundingClientRect) with writes (style/class changes): each read after a write forces a synchronous layout, so a loop becomes O(n) layout passes. In React it bites in measurement code — useLayoutEffect that reads geometry then sets state that changes styles, repeating; drag/resize handlers reading and writing per mousemove; tooltip/popper positioning. Fix: batch all reads, then all writes; measure once; prefer CSS (aspect-ratio, flex/grid, transforms) over JS measurement; for animations, stick to transform/opacity which skip layout.

24. Server data in Redux vs TanStack Query — the performance argument?

Server data in a global store means: every fetch dispatches through one store (every dispatch runs every subscribed selector), hand-rolled caching/staleness/dedupe, and components coupled to global shape. Query libraries treat server data as a cache keyed by query with per-key subscriptions — only components on that key re-render on its updates — plus staleTime to stop refetch storms, request dedupe, background revalidation, and pagination/infinite primitives. Global stores shrink to genuinely client-global state, which shrinks the re-render blast radius of everything.

25. How do you stop performance regressions from coming back?

Budgets enforced in CI: bundle size per entry chunk (size-limit fails the build over threshold), Lighthouse CI on key pages, and a Profiler-API assertion in integration tests on the hottest trees (fail if commit exceeds N ms with production-scale fixture data). In the field, continuous web-vitals RUM with alerting on p75 INP/LCP regressions per release. The senior point: an unguarded optimization is a temporary one — the next feature will eat it silently.

            item={item}                       // pass the original reference
            done={done.has(item.id)}          // spread state as a primitive prop
            onToggle={handleToggle}
            badge={badgeEl}
          />
        ))}
      </ul>
    </>
  );
}
// Adjust Row to accept `done` separately. Now typing in the input
// re-renders TodoList, but every Row bails: item (same ref),
// done (same boolean), onToggle (stable), badge (hoisted).
// Toggling re-renders ONLY rows whose `done` boolean flipped.

Exercise 2 — Composition, no memo allowed

The mouse-position tracker below re-renders <ExpensiveChart /> (measured 60ms) on every mousemove. Eliminate the chart re-renders using only restructuring — no memo, useMemo, or useCallback.

function Dashboard() {

  return (

Solution 
// Lift content up: the tracker owns the hot state and receives the
// expensive subtree as children. Dashboard creates the chart element
// once per ITS render, so tracker re-renders reuse the same reference
// and React skips the chart subtree.
function MouseTracker({ children }) {

  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      <Crosshair x={pos.x} y={pos.y} />
      {children}                     {/* same element object as last render */}
    </div>
  );
}

    </MouseTracker>
  );
}

Exercise 3 — Context architecture surgery

One AppContext holds { user, theme, notifications, addNotification, toggleTheme }. Notifications arrive several times per second; the profiler shows the theme toggle button, the user avatar, and 40 other consumers re-rendering with every notification. Redesign the context layer.

Solution 
// 1. Slice by update frequency + split actions from state.
const UserContext          = createContext(null);  // changes on login
const ThemeContext         = createContext(null);  // changes rarely
const NotificationsContext = createContext(null);  // hot — changes constantly
const ActionsContext       = createContext(null);  // NEVER changes

function AppProviders({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('dark');
  const [notifications, dispatch] = useReducer(notificationsReducer, []);

  // stable forever: setters are stable, dispatch is stable, deps = []
  const actions = useMemo(() => ({
    toggleTheme: () => setTheme(t => t === 'dark' ? 'light' : 'dark'),
    addNotification: (n) => dispatch({ type: 'add', n }),
  }), []);

  return (
    <ActionsContext.Provider value={actions}>
      <UserContext.Provider value={user}>
        <ThemeContext.Provider value={theme}>
          <NotificationsContext.Provider value={notifications}>
            {children}
          </NotificationsContext.Provider>
        </ThemeContext.Provider>
      </UserContext.Provider>
    </ActionsContext.Provider>
  );
}
// Result: notification churn re-renders ONLY NotificationsContext
// consumers. The theme button consumes ActionsContext (+ ThemeContext
// for its icon) — untouched by notifications. Avatar consumes
// UserContext — untouched.
//
// 2. If notification consumers are still too broad (e.g. one badge
// needs only the count while a panel needs the list), that's the cue
// to move notifications to an external store (Zustand) and subscribe
// with selectors: useStore(s => s.notifications.length).

Exercise 4 — Virtualize a log viewer

Build a viewer for 100,000 log lines (fixed 28px height) with TanStack Virtual, plus: auto-scroll-to-bottom on new logs (tail mode) that disengages when the user scrolls up, and a "jump to latest" button. Sketch the solution.

Solution 
function LogViewer({ logs }) {
  const parentRef = useRef(null);
  const [tailing, setTailing] = useState(true);

  const v = useVirtualizer({
    count: logs.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 28,       // fixed height → exact math, no measuring
    overscan: 10,
  });

  // Tail mode: when new logs arrive and tailing is on, pin to the end.
  useEffect(() => {
    if (tailing && logs.length > 0) {
      v.scrollToIndex(logs.length - 1, { align: 'end' });
    }
  }, [logs.length, tailing]);      // v is stable per instance

  // Disengage when the user scrolls away from the bottom.
  const onScroll = () => {
    const el = parentRef.current;
    const atBottom =
      el.scrollHeight - el.scrollTop - el.clientHeight < 40;
    setTailing(atBottom);          // scroll up → false; return to bottom → true
  };

  return (
    <div style={{ position: 'relative' }}>
      <div ref={parentRef} onScroll={onScroll}
           style={{ height: 500, overflow: 'auto', fontFamily: 'monospace' }}>
        <div style={{ height: v.getTotalSize(), position: 'relative' }}>
          {v.getVirtualItems().map(item => (
            <div key={item.key}
                 style={{ position: 'absolute', top: 0, left: 0, width: '100%',
                          height: 28, transform: `translateY(${item.start}px)` }}>
              <LogLine line={logs[item.index]} />
            </div>
          ))}
        </div>
      </div>
      {!tailing && (
        <button className="jump-latest"
                onClick={() => { setTailing(true);
                                 v.scrollToIndex(logs.length - 1, { align: 'end' }); }}>
          Jump to latest ↓
        </button>
      )}
    </div>
  );
}
// Key decisions to narrate: ~28 DOM rows instead of 100k; index keys are
// acceptable ONLY if logs are append-only (prepends/deletions → stable ids);
// memo(LogLine) so scroll commits only mount entering rows; if logs stream
// faster than 60/s, buffer and flush appends with requestAnimationFrame
// batching so you commit once per frame, not once per message.

Exercise 5 — Run the runbook on a cold-start problem

PM report: "the app takes 9 seconds to become usable on mid-range Android." Lab confirms: LCP 6.2s, TBT 3.8s, entry bundle 1.9MB gzipped. Write your step-by-step plan with expected impact per step.

Solution

Classify: startup budget, twin problems — bytes (1.9MB → download+parse) and main-thread execution (TBT 3.8s). Plan, ordered by impact-per-effort:

  1. Bundle treemap (source-map-explorer). Suppose it shows: charting lib 420KB used on /analytics only; moment+locales 310KB; a PDF lib 380KB behind an "export" button; full lodash 90KB.
  2. Route-split so the start route ships only its own code — charts and analytics move to their chunks. Expected: entry −500KB+.
  3. Component-split the PDF exporter behind its click, preloading on hover. Expected: −380KB from every non-exporting session.
  4. Replace/tree-shake: moment → date-fns or Intl (−~300KB raw), lodash → per-function lodash-es (−~70KB). Ballpark entry after 2–4: ~500–600KB gz.
  5. TBT pass: Performance trace of remaining long tasks — typically hydration of below-the-fold sections and third-party scripts. Defer third-party to post-interactive/idle; if on a framework, stream with Suspense boundaries so hydration is selective; consider RSC for static sections.
  6. LCP pass: ensure the hero image is preloaded/fetchpriority=high, not lazy-loaded, dimensions set; check server TTFB.
  7. Verify: re-run Lighthouse with Moto-G-class throttling — target LCP < 2.5s, TBT < 300ms — and confirm p75 field LCP/INP over the next release.
  8. Guard: size-limit CI budget on the entry chunk (e.g. fail > 250KB gz), Lighthouse CI on the landing route.

The grading points: treemap before touching code, ordered fixes with expected numbers, both budgets addressed (bytes AND main thread), lab + field verification, and a CI guard.
