Box model deep dive T1

content-box vs border-box

Every box has four areas, inside-out: content → padding → border → margin. The box-sizing property controls what the width/height properties refer to:

  • content-box (initial value, per CSS 2.1 legacy): width sets the content area only. Rendered width = width + padding + border.
  • border-box: width sets content + padding + border. The content area shrinks to make room. Margin is never included in either model.
/* Same declared width, very different rendered widths */
.a { box-sizing: content-box; width: 200px; padding: 20px; border: 5px solid; }
/* rendered: 200 + 20*2 + 5*2 = 250px wide */

.b { box-sizing: border-box; width: 200px; padding: 20px; border: 5px solid; }
/* rendered: exactly 200px; content area = 200 - 40 - 10 = 150px */

Why the universal border-box reset exists: with content-box, width: 50%; padding: 1rem overflows its parent — the two 50% halves plus padding exceed 100%. border-box makes percentage layouts and form controls composable: "the box is this big, period." The canonical reset:

*, *::before, *::after { box-sizing: border-box; }

/* Alternative "inheritable" form — lets a component opt back out
   for a whole subtree by setting box-sizing on its root: */
html { box-sizing: border-box; }
*, *::before, *::after { box-sizing: inherit; }
Form controls (input, select, textarea, button) already default to border-box in the UA stylesheet — one reason mixing reset-less custom elements with native inputs produces misaligned widths.

Margin collapsing — the actual rules

This is a classic gotcha because it only applies in a very specific situation, and interviewers love asking you to enumerate the cases. Vertical margins of block-level boxes in the same block formatting context, in normal flow, collapse when they are adjoining — meaning nothing separates them (no border, no padding, no clearance, no inline content, no line box). The collapsed margin equals the larger of the two (for two positives), the more-negative of two negatives, or the sum when signs mix.

Case 1 — adjacent siblings:

<p class="first">Paragraph one</p>
<p class="second">Paragraph two</p>

.first  { margin-bottom: 30px; }
.second { margin-top: 20px; }
/* Gap between them = 30px, NOT 50px. max(30, 20). */

Case 2 — parent and first/last child (the one that actually bites in production):

<div class="card">
  <h3>Title</h3>   <!-- h3 has UA margin-top ≈ 1em -->
</div>

.card { background: lavender; margin-top: 0; }
/* Visual result: the h3's margin-top "escapes" THROUGH the card.
   The lavender box gets pushed down and the h3 sits flush at its top —
   the margin appears OUTSIDE the parent, not as space inside it. */

This happens because there is no border, padding, or inline content between the parent's top edge and the child's top edge, so the two margin-tops are adjoining and collapse into one margin that now belongs, effectively, to the parent. Fixes — anything that breaks "adjoining":

  • padding-top: 0.05px or a real border-top on the parent
  • Make the parent establish a new BFC: display: flow-root (the modern, no-side-effect fix), overflow: hidden/auto, or make it a flex/grid container (flex/grid item margins never collapse)
  • padding-based spacing plus gap instead of child margins entirely

Case 3 — empty blocks: if a block has no border, padding, height, or content, its own margin-top and margin-bottom collapse with each other (and then with neighbors). An "empty spacer div" with margin: 40px 0 contributes 40px total, not 80px.

Margins never collapse: horizontally; across a BFC boundary; on floats; on absolutely positioned elements; on inline-block boxes; between flex or grid items; or when the root element is involved. If an interviewer asks "how do you stop margin collapse," the crisp answer is: break adjacency (border/padding) or move one box into a different formatting context (flow-root/flex/grid/float/abs-pos).

Negative margins (yes, they're spec'd behavior)

Negative margins are legitimate layout tools, not hacks, and they behave asymmetrically — a detail that impresses when you can state it precisely:

  • Negative top/left margin pulls the element itself in that direction (it moves).
  • Negative bottom/right margin leaves the element in place and pulls following content toward/over it — it effectively shrinks the space the element "reports" to the layout after itself.
  • On width: auto block boxes, negative horizontal margins widen the box (the used width solves the equation margin + border + padding + width = containing block).
/* Full-bleed child inside a padded prose column — the classic use: */
.prose { max-width: 65ch; padding-inline: 1.5rem; margin-inline: auto; }
.prose .bleed { margin-inline: -1.5rem; }
/* The image cancels the parent's padding and spans the full column.
   Because width is auto, negative margins GROW the box — no width math. */

/* Pre-gap-era gutter grid (know it to read old codebases): */
.row { display: flex; flex-wrap: wrap; margin: -8px; }
.row > * { margin: 8px; }
/* The negative outer margin swallows the items' edge margins. */
Negative margins also participate in margin collapsing: when a positive and negative margin collapse, the result is their sum (e.g. 20px and −30px → −10px). Content overlapped by negative margins still receives pointer events per paint order — overlap bugs from negative margins are really stacking questions.

Percentage padding is relative to WIDTH

Both padding-top: 10% and padding-left: 10% resolve against the inline size (width) of the containing block — never the height. Same for percentage margins. The spec rationale: height is often not known until layout finishes (content-driven), so resolving vertical padding against height would create a circular dependency. This quirk powered the classic aspect-ratio hack:

/* Legacy 16:9 box (pre aspect-ratio property) */
.ratio { position: relative; padding-top: 56.25%; } /* 9/16 of the WIDTH */
.ratio > iframe { position: absolute; inset: 0; width: 100%; height: 100%; }

/* Modern equivalent */
.ratio-modern { aspect-ratio: 16 / 9; }

min-/max-width interactions

The constraint resolution order the browser follows: compute the tentative used width from width (or auto), then clamp by max-width, then clamp by min-width. Crucially, min-width wins over max-width when they conflict:

.box { width: 100px; max-width: 150px; min-width: 200px; }
/* used width = 200px. min beats max beats width. */
Remember min-width's initial value is auto, not 0 — and in flex and grid layouts auto resolves to the content-based minimum size. That single spec line is the root cause of the two most famous modern layout bugs: unshrinkable flex items (§ flexbox) and grid tracks blown out by a long word or wide <pre>. In non-flex/grid flow, min-width: auto behaves as 0.

box-sizing edge cases worth naming out loud in an interview:

  • min-width/max-width respect box-sizing too — with border-box they constrain the border box.
  • box-sizing is not inherited; that's why the reset targets * (or uses the inherit pattern).
  • Percentage width resolves against the containing block's content box width regardless of the child's box-sizing.
  • Replaced elements with intrinsic ratios (img) interact with box-sizing at the border-box level: border-box + border means the drawn image area shrinks.

Specificity & the cascade — internals T1

The cascade order (what actually decides the winner)

When multiple declarations target the same property on the same element, the browser sorts them by these criteria in order — a later criterion is only consulted to break ties in the earlier ones:

  1. Origin & importance: transition declarations → !important UA styles → !important user styles → !important author styles → animations → normal author styles → normal user styles → normal UA styles. Note !important reverses the origin order — that is why UA !important rules are untouchable.
  2. Cascade layers: within author styles, @layer order. Normal declarations: later-declared layers beat earlier ones; unlayered styles beat all layers. !important flips this too — earlier layers' important declarations win.
  3. Specificity of the selector.
  4. Order of appearance: last declaration wins.
Scope proximity (from @scope) slots between specificity and source order in the current cascade spec. Also: element-attached styles (the style="" attribute) are modeled as a pseudo-"specificity beyond triples" — they beat any selector-based specificity but are still author-origin, so an !important stylesheet rule beats a normal inline style.

Cascade layers in practice — the architecture pattern to name-drop:

/* Declare order up front — THIS line is the whole architecture.
   Later layers beat earlier ones (for normal declarations): */
@layer reset, vendor, tokens, components, utilities;

@layer reset { * { margin: 0; box-sizing: border-box; } }

/* Third-party CSS goes into a low layer at import time: */
@import url("datepicker.css") layer(vendor);

@layer components {
  .btn { padding: .5rem 1rem; background: var(--accent); }
}

@layer utilities {
  /* A single-class utility now beats ANY selector in components,
     regardless of specificity — no more .btn.btn.btn hacks: */
  .hidden { display: none; }
}

/* Unlayered styles beat all layers → app-level escape hatch. */

Why interviewers care: layers solve the exact problem utility-class and BEM conventions solved socially — specificity ordering — but in the language itself. Being able to say "specificity only matters within a layer" shows you've internalized the cascade's sort order rather than memorized specificity folklore.

Specificity as (a, b, c) triples

Specificity is a three-component tuple compared lexicographically — not a base-10 number. No amount of classes ever beats one ID.

                              but takes the specificity of its most
                              specific argument (.ext → one class) */

/* (0,10,0) does NOT beat (1,0,0): compare a first, 1 > 0, done. */

:is() vs :where()

Both accept a forgiving selector list and match identically. The difference is purely specificity:

  • :is(...) — takes the specificity of its most specific argument, even for arguments that didn't actually match. :is(#hero, p) span is (1,0,1) when matching a span inside a plain p.
  • :where(...) — specificity is always zero, regardless of arguments. This is the tool for writing overridable defaults in design systems and resets.
/* Design-system default any consumer can override with a bare class */
:where(.ds-button) { padding: .5rem 1rem; background: navy; }

.my-button { background: teal; }   /* (0,1,0) beats (0,0,0) — wins */

/* Had we used .ds-button directly, consumer needs equal specificity
   PLUS later source order — brittle. */

Inline styles and !important

Priority ladder for author styles, low → high: selector rules → inline style="" → !important selector rules → !important inline styles. To override a third-party !important you must beat it within the important lane:

  1. Same importance, higher specificity: #x.y!important beats .y!important.
  2. Same importance and specificity, later source order.
  3. Cascade layers: put the vendor CSS in a layer; your unlayered normal styles beat their layered normal styles, and for important declarations an earlier layer wins — so layering strategy can neutralize vendor !important.
  4. Last resorts: style.setProperty('color', 'red', 'important') from JS, or a completely different property that wins the visual fight (e.g., outline vs their border).

Inheritance and the four reset keywords

Inherited by default: mostly text-ish properties — color, font-*, line-height, letter-spacing, text-align, visibility, cursor, list-style, white-space, direction. Not inherited: box/layout properties — margin, padding, border, background, display, position, width, overflow.

Formatting contexts T1

Block formatting context (BFC)

A BFC is an isolated mini-layout: an independent region in which block boxes are laid out and which does not interact with the outside for floats and margin collapsing. What establishes one (know the list):

  • The root element (<html>)
  • Floats and absolutely/fixed positioned elements
  • display: flow-root — the purpose-built, zero-side-effect trigger
  • overflow anything other than visible/clip on a block box (hidden, auto, scroll)
  • display: inline-block, table cells/captions
  • contain: layout | content | paint, and multicol containers
  • Flex and grid items (their children lay out in a fresh context; the containers themselves establish flex/grid formatting contexts, not BFCs — same isolating effects in practice)

What creating a BFC fixes — the two interview classics:

<!-- 1. Float containment: parent collapses to zero height -->
<div class="clearfix-victim">
  <img class="float-left" src="pic.jpg" alt="">
</div>

.float-left { float: left; }
.clearfix-victim { display: flow-root; }
/* A BFC's height includes its floats. Before flow-root, the fix was
   the ::after clearfix or overflow:hidden (which clips). */

/* 2. Margin-collapse barrier (parent/child case from §box-model) */
.card { display: flow-root; }  /* child margin-top now stays inside */

/* 3. Bonus: a BFC does not wrap under an adjacent float —
   the classic media-object left-image / non-wrapping-text layout: */
.media-body { display: flow-root; }

Inline formatting context quirks

Whitespace: inline-level boxes are separated by the literal whitespace in your HTML — collapsed to a single space, but a space nonetheless. The infamous "4px gap between inline-blocks" comes from the newline between tags:

<div class="pill-list">
  <span class="pill">One</span>
  <span class="pill">Two</span>    <!-- the newline+indent collapses to
                                       ONE space ≈ 4px at 16px font -->
</div>

.pill { display: inline-block; background: #eef; padding: .2rem .6rem; }

/* Visual result: pills have a mystery gap that margin: 0 can't remove —
   it isn't margin, it's a text space between inline boxes. Fixes: */
.pill-list { display: flex; gap: .25rem; } /* modern: whitespace-only text
                                              between flex items is dropped */
/* legacy: parent font-size: 0; or comment-joining the tags in HTML. */

Anonymous boxes: when block and inline children mix (<div>text <p>para</p></div>), the engine wraps the loose text in an anonymous block box so the BFC only contains blocks. You can't select anonymous boxes — a reason "why can't I style this text without a wrapper" has no CSS answer, and a nice depth signal when explaining the box tree vs the DOM tree.

The image baseline gap — a genuine classic. An <img> inside a div shows a few pixels of mystery space below it:

<div class="thumb"><img src="cat.jpg" alt="Cat"></div>

/* Why: img is inline (a replaced inline box). It sits on the line box's
   BASELINE, and the space below the baseline is reserved for descenders
   (g, y, p) of the strut — the invisible zero-width text the line box
   sizes itself around. The "gap" is descender space, derived from the
   parent's font metrics and line-height. */

/* Fixes (any one): */
.thumb img { display: block; }        /* no line box, no baseline */
.thumb img { vertical-align: bottom; } /* or middle/top */
.thumb { line-height: 0; }            /* collapse the strut */

vertical-align only applies to inline-level boxes and table cells — a top-3 misconception is trying it on a block child to center it. It aligns a box within its line box relative to the baseline (baseline | middle | top | text-bottom | 20% | -3px...). middle aligns to baseline + half the parent's x-height — which is why icons aligned with middle still look slightly off next to caps-height text.

Flexbox deep dive T1

The axis model

Flexbox is a one-dimensional layout: content is distributed along the main axis (set by flex-direction), and aligned on the perpendicular cross axis. Every "which property do I use" question resolves by asking which axis am I on:

Main axisCross axis
Distribute spacejustify-contentalign-content (multi-line only)
Align items— (no justify-items in flexbox)align-items / per-item align-self
Item sizingflex-basis, flex-grow, flex-shrinkalign-self: stretch + height/width

With flex-direction: column, everything rotates: justify-content is now vertical. This is also writing-mode relative — in vertical writing modes row runs vertically.

flex-basis vs width — resolution order

The used main size of a flex item is resolved as: flex-basis → falls back to width/height → falls back to content size, then flexing (grow/shrink) adjusts from that hypothetical size, then min/max clamping applies. So flex-basis beats width when both are set (any value except auto; flex-basis: auto means "look at width").

.item { flex-basis: 200px; width: 500px; }  /* basis wins: starts at 200px */
.item { flex-basis: auto;  width: 500px; }  /* auto defers → 500px */
.item { flex-basis: auto; }  /* no width either → content size */
.item { flex-basis: 0; }     /* start from zero — all space is "free" */

Know the shorthands cold — they change more than one value:

Shorthandgrow shrink basisBehavior
flex: initial (default)0 1 autoWon't grow; shrinks from content size
flex: auto1 1 autoGrows/shrinks from content size — big content gets proportionally more
flex: 11 1 0%Basis zero → all space distributed by grow → equal columns regardless of content
flex: none0 0 autoFully rigid
"Why does flex: 1 give equal columns but flex: auto doesn't?" — because with basis 0 the free space is the entire container, split purely by grow ratios; with basis auto, only the leftover after content sizes is split, so bigger content keeps its head start. That one sentence is a strong interview answer.

The grow/shrink math (worked examples)

Growing. Free space = container − Σ(hypothetical sizes). Each item gets free × (its grow ÷ Σ grows).

<div class="row"><div class="a"></div><div class="b"></div><div class="c"></div></div>

.row { display: flex; width: 800px; }
.a { flex: 1 1 100px; }   /* grow 1 */
.b { flex: 2 1 100px; }   /* grow 2 */
.c { flex: 1 1 200px; }   /* grow 1 */

/* free = 800 - (100+100+200) = 400.  Σgrow = 4.  unit = 100.
   a = 100 + 1×100 = 200
   b = 100 + 2×100 = 300
   c = 200 + 1×100 = 300   ✓ totals 800 */

Shrinking is the part candidates get wrong: shrink is weighted by shrink factor × flex-basis, not by shrink factor alone. Bigger items absorb more of the deficit — this prevents small items from shrinking to nothing first.

.row { display: flex; width: 600px; }
.a { flex: 0 1 400px; }   /* shrink 1, basis 400 */
.b { flex: 0 2 200px; }   /* shrink 2, basis 200 */
.c { flex: 0 1 200px; }   /* shrink 1, basis 200 */

/* overflow = (400+200+200) - 600 = 200 to remove.
   scaled factors: a = 1×400 = 400
                   b = 2×200 = 400
                   c = 1×200 = 200      Σ = 1000
   a loses 200 × 400/1000 = 80  → 320
   b loses 200 × 400/1000 = 80  → 120
   c loses 200 × 200/1000 = 40  → 160   ✓ totals 600 */
If clamping (min-width, content minimums) freezes an item during grow/shrink, the algorithm re-runs distribution among the remaining unfrozen items — that's why results sometimes differ from the naive one-pass math.

The min-width:auto overflow gotcha

The #1 real-world flexbox bug: "my flex item won't shrink / text-overflow: ellipsis doesn't work / the layout blows out." Cause: on flex items, min-width: auto (the initial value) computes to the item's automatic minimum size ≈ its min-content size. So an item containing a long unbreakable string (URL, token, filename) refuses to shrink below that string's width, no matter what flex-shrink says.

<div class="chat-row">
  <img class="avatar" src="a.png" alt="">
  <div class="msg">
    <p class="ellipsis">https://example.com/very/long/unbroken/url...</p>
  </div>
</div>

.chat-row { display: flex; gap: .5rem; }
.msg { flex: 1; min-width: 0; }      /* ← THE FIX. (or overflow: hidden) */
.ellipsis { white-space: nowrap; overflow: hidden; text-overflow: ellipsis; }

/* Without min-width: 0, .msg's min size = the URL's min-content width →
   it can't shrink → ellipsis never triggers and the row overflows.
   overflow: hidden also works because it changes the automatic minimum
   size to zero per spec. Column direction: same story with min-height. */

align-items vs align-content vs align-self

  • align-items — default cross-axis alignment of items within their line. Initial: stretch (why flex children are equal height by default).
  • align-self — the per-item override of align-items.
  • align-content — distributes the lines themselves when there is more than one (needs flex-wrap: wrap and spare cross space). On a single-line flex container it does nothing — a favorite trick question. (It now also applies to block containers for easy vertical centering, a nice modern-CSS mention.)

gap, wrapping, order

.tags { display: flex; flex-wrap: wrap; gap: .5rem 1rem; } /* row-gap column-gap */
  • gap only creates space between items — no first/last edge hacks or negative-margin grids needed. It also counts as used space in the shrink math.
  • flex-wrap: wrap creates new lines when hypothetical sizes overflow; each line runs grow/shrink independently — which is why the last wrapped row of "equal" cards grows wider than the rest (fix with grid, or a max-width on items).
  • order reorders visual painting only. DOM order still drives tab sequence and screen-reader order.
order (and flex-direction: row-reverse) create a mismatch between visual order and focus order — a WCAG 2.4.3 (Focus Order) failure when the content order is meaningful. Reorder the DOM when the order matters; reserve order for purely presentational shuffles (e.g., responsive hero image position).

Wrapping subtleties & absolute children

/* The "last row stretches" problem: wrapped equal cards, but the
   final line has 2 items that grow to fill the whole row: */
.cards { display: flex; flex-wrap: wrap; gap: 1rem; }
.card  { flex: 1 1 250px; }
/* Each LINE runs grow independently — a 2-item last line splits the
   row between 2 cards. Flexbox lines don't know about each other.
   Fixes: .card { max-width: 340px; } to cap the stretch, or switch to
   grid repeat(auto-fill, minmax(250px, 1fr)) where tracks are global. */
  • An absolutely positioned child of a flex container does not participate in flex layout at all — it takes no space and ignores grow/shrink — but its static position is resolved using the container's justify-content/align-items, a rarely known spec detail useful for badges and corner affordances.
  • Margins on flex items never collapse, and float/vertical-align have no effect on flex items — blockification: children of flex/grid containers have their display blockified (inline → block, inline-flex → flex).
  • Anonymous flex items: raw text directly inside a flex container gets wrapped in an anonymous item — whitespace-only runs are ignored, which is why the inline-block 4px gap disappears under flex.

Canonical patterns

/* Nav bar: logo left, actions right — the auto-margin trick */
.nav { display: flex; align-items: center; gap: 1rem; }
.nav .actions { margin-left: auto; }   /* auto margins eat free space
                                          BEFORE justify-content runs */

/* Media object: fixed image, flexible non-wrapping body */
.media { display: flex; gap: 1rem; align-items: flex-start; }
.media > img { flex: none; }
.media > .body { flex: 1; min-width: 0; }

/* Equal-height cards with footer pinned to bottom */
.cards { display: flex; gap: 1rem; }
.card  { flex: 1; display: flex; flex-direction: column; }
.card .cta { margin-top: auto; }   /* pushes itself to the bottom */

Grid deep dive T1

Explicit vs implicit grid

grid-template-columns/rows/areas define the explicit grid. Anything placed outside it (more children than cells, or placement like grid-row: 10) forces the browser to generate implicit tracks, sized by grid-auto-rows/grid-auto-columns (default: auto). The classic symptom: "my extra cards are squished rows with no height" — because implicit rows defaulted to auto and you only styled the template rows.

.grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  grid-auto-rows: minmax(120px, auto);  /* size the OVERFLOW rows too */
  gap: 1rem;
}

fr and minmax()

fr distributes leftover space after fixed tracks, gaps, and content minimums are satisfied — it's a fraction of free space, not of the container. Crucially, 1fr is really minmax(auto, 1fr): the track can't shrink below its content's min-content size. That's the grid twin of the flexbox min-width:auto bug:

/* Track blowout: a <pre> or long URL in a 1fr column makes it wider
   than its "fair share" and can overflow the page. */
grid-template-columns: repeat(3, 1fr);        /* can blow out */
grid-template-columns: repeat(3, minmax(0, 1fr));  /* truly equal, clampable */

minmax(min, max) defines a sizing range; if max < min, max is ignored. minmax(200px, 1fr) — "at least 200px, then share leftovers" — is the backbone of responsive card grids.

auto-fill vs auto-fit — the actual difference

.a { grid-template-columns: repeat(auto-fill, minmax(200px, 1fr)); }
.b { grid-template-columns: repeat(auto-fit,  minmax(200px, 1fr)); }

Both create as many ≥200px tracks as fit. They differ only when there are fewer items than tracks:

  • auto-fill keeps the empty tracks in place. Two cards in a 5-track 1000px grid stay ~200px each; 3 empty tracks hold the rest. Items keep a consistent size — good for dashboards where cards shouldn't balloon.
  • auto-fit collapses empty tracks to 0, so the surviving tracks' 1fr maxes absorb the freed space: two cards stretch to ~500px each. Good for "fill the row gracefully" galleries.

With enough items to fill every track, they render identically — say that explicitly in interviews; it shows you know it's an edge-case distinction.

grid-template-areas & line-based placement

/* App shell via named areas — self-documenting */

   of the EXPLICIT grid: -1 is the last line. */
.full-bleed { grid-column: 1 / -1; }        /* span all columns */
.feature    { grid-row: 1 / span 2; grid-column: 2 / 4; }
A grid with N explicit column tracks has N+1 lines; -1 refers to the last explicit line, so in the presence of implicit tracks 1 / -1 may not span everything — a subtle correctness point that lands well when mentioned.

Named lines — the third placement vocabulary, and the cleanest way to build the popular "content column with full-bleed breakouts" page layout:

.page {
  display: grid;
  grid-template-columns:
    [full-start] minmax(1rem, 1fr)
    [content-start] min(65ch, 100% - 2rem) [content-end]
    minmax(1rem, 1fr) [full-end];
}
.page > *          { grid-column: content; } /* name-pair shorthand:
                                                content-start/content-end */
.page > .full-bleed { grid-column: full; }

/* Visual result: prose is centered at a readable measure; any element
   tagged .full-bleed (hero image, code block) spans edge-to-edge —
   no negative-margin hacks, works at every viewport. Also note
   grid-template-areas auto-generates names: area "main" creates
   main-start/main-end lines on both axes. */

Auto-placement and dense

The auto-placement algorithm walks the grid cursor forward, placing each item in the next slot it fits. Positioning & stacking contexts T1

This is the single highest-yield senior CSS topic. Expect a live-debugging question ("this modal renders under the header — why?") and be ready to narrate the mechanism.

Why sticky "doesn't work" — the checklist

.toolbar { position: sticky; top: 0; }
  1. No offset: without top/bottom/left/right, sticky never engages.
  2. An ancestor with overflow: hidden/auto/scroll/clip: the sticky element sticks relative to its nearest scroll container — if that ancestor doesn't actually scroll (e.g. overflow: hidden added for a clearfix), the element sticks relative to a scrollport that never moves → appears dead. Debug by walking up the tree checking getComputedStyle(el).overflow.
  3. Parent height: sticky travel is confined to its containing block (the parent). If the parent is exactly as tall as the sticky element — sticky is the only child, or the element itself stretched to fill the parent — there is zero travel range. The reliable rule: the element sticks only while its parent's box is passing through the scrollport, so the parent must be taller than the sticky child. Classic instance: a sticky sidebar that's a flex item. align-items: stretch makes the sidebar itself as tall as the row, so it can never move within it — fix with align-self: flex-start, which lets the sidebar keep its content height inside the tall flex line and travel within it.
  4. Sticky table headers: put sticky on th, not thead (per-engine quirks), and beware borders with border-collapse: collapse not traveling with the stuck cells.

Containing blocks — the transform trap

The containing block for absolute is the nearest ancestor with position ≠ static. For fixed it's normally the viewport. But — per the CSS Transforms and Filter Effects specs — an ancestor with any of the following becomes the containing block for absolute and fixed descendants:

  • transform or translate/rotate/scale (any non-none value), perspective
  • filter or backdrop-filter (non-none)
  • will-change: transform | filter | perspective
  • contain: layout | paint | strict | content, and container-type: size | inline-size (container queries!)
<!-- THE interview favorite: -->
<div class="animated-panel">      <!-- transform: translateX(0) from an animation -->
  <div class="modal">...</div>    <!-- position: fixed; inset: 0 -->
</div>

/* Expected: modal covers the viewport.
   Actual: modal is trapped inside .animated-panel, scrolls with it,
   sized to the panel. Because the transformed ancestor is now the
   containing block, "fixed" behaves like "absolute" inside it. */

Fixes: render the overlay outside the transformed subtree (React portal into document.body — this is the reason portals exist), remove the leftover transform after animation (beware will-change left on "for performance"), or use the top-layer primitives (<dialog>.showModal(), Popover API) which escape ancestor clipping and stacking entirely.

Stacking contexts — the full model

Painting order is governed by a tree of stacking contexts. Two rules carry every debugging session:

  1. z-index only applies to positioned elements and flex/grid items. On a plain static block it's ignored.
  2. An element's z-index competes only within its nearest ancestor stacking context. Stacking contexts are atomic: the entire subtree paints as one unit at the parent context's level. A child's z-index: 999999 can never escape a parent context that sits at z-index: 1.

What creates a stacking context — the list to recite:

  • Root element (<html>)
  • position: relative/absolute with z-index ≠ auto; position: fixed or sticky (always, even with z-index auto)
  • Flex or grid item with z-index ≠ auto
  • opacity < 1
  • transform, translate/rotate/scale, filter, backdrop-filter, perspective, clip-path, mask — any non-none value
  • isolation: isolate (the do-nothing-but-this property)
  • will-change naming any of the above; contain: layout | paint
  • mix-blend-mode ≠ normal

Paint order inside one stacking context (simplified CSS 2.2 Appendix E), bottom to top:

┌─ 7. positive z-index descendants/contexts (ascending; DOM order ties) ─ TOP
│  6. z-index: 0 / auto positioned descendants (+ new contexts at 0)
│  5. inline-level content (text paints above floats!)
│  4. floats (non-positioned)
│  3. in-flow, non-positioned block descendants (backgrounds/borders)
│  2. negative z-index child stacking contexts
└─ 1. this context's own background and borders ──────────────── BOTTOM

Two practical corollaries hide in that list: z-index: -1 puts a child behind its parent's content but still above the parent's background (layer 2 vs layer 1) — the decorative-blob pattern; and a positioned element with z-index: auto already paints above all normal-flow siblings (layer 6 vs 3) — which is why sprinkling position: relative alone changes overlaps.

The "z-index: 9999 doesn't work" walkthrough

<header class="site">...</header>         <!-- position: sticky; z-index: 10 -->
<main>
  <section class="hero">                  <!-- opacity: .98  ← the villain -->
    <div class="dropdown">...</div>       <!-- position: absolute; z-index: 9999 -->
  </section>
</main>

/* Stacking tree:
   root
   ├─ header       → context at z:10   (sticky + z-index)
   └─ .hero        → context at z:auto/0 (opacity < 1 !)
        └─ .dropdown  z:9999 — but only WITHIN .hero

   .hero (level 0) paints below header (level 10). The dropdown is
   sealed inside .hero, so its 9999 never competes with the header.
   It slides UNDER the sticky header. */

Debugging method (narrate this in interviews): walk up from the "losing" element to the root; at each ancestor check the stacking-context triggers (DevTools shows "Creates stacking context" in the Layers/Elements panels; Edge and Chrome have a stacking-context inspector). Find the two sibling contexts that actually compete — here header vs .hero — and adjust their z-indexes, not the deep child's. Fix: remove the opacity, or give .hero z-index: 20… which may cause new overlaps, hence:

/* isolation: isolate — create a stacking context ON PURPOSE, with no
   visual side effects, to stop children leaking out and fighting
   unrelated parts of the page: */
.card { isolation: isolate; }
/* Now the card's internal z-index games (badge over image, etc.)
   can never overlap another card or the app header. Design systems
   should isolate every self-contained composite component. */

The top layer — the modern escape hatch

The top layer is a browser-managed rendering layer above the entire document — above every stacking context, immune to ancestor overflow clipping and transform containing blocks. Elements get promoted into it only via APIs, never via z-index:

<dialog id="confirm">
  <p>Delete this item?</p>
  <form method="dialog"><button>Cancel</button><button value="ok">Delete</button></form>
</dialog>
<script>confirm.showModal(); // top layer + focus trap + inert background</script>

<button popovertarget="menu">Options</button>
<div id="menu" popover>...</div>  <!-- top layer, light-dismiss, no z-index -->

::backdrop { background: rgb(0 0 0 / .5); backdrop-filter: blur(2px); }

Within the top layer, order is promotion order (last promoted paints on top). This is the architectural answer to overlay bugs: no portal, no z-index registry, no stacking-context spelunking. showModal() additionally makes the rest of the document inert — the accessibility work you'd otherwise hand-roll.

Tie it together for interviews: transform creates both a stacking context and a containing block for fixed/absolute descendants — which is why "I added a hover animation and my dropdown broke in two different ways" bugs cluster. And the top layer (dialog, popover, fullscreen) is why new codebases can retire the z-index: 2147483647 folklore entirely — a strong closing point in any stacking answer.

   can't enlarge it. The rem term in the sum keeps it zoomable. */
Interview trap: "what's font-size: 10vw's accessibility problem?" — it's immune to text-only zoom and user font-size preferences, capping effective scaling. Always anchor fluid type with a rem component and sane clamp bounds.

:nth-child vs :nth-of-type — the difference

:nth-child(n) counts all sibling elements, then checks whether the nth one also matches the rest of the selector. :nth-of-type(n) counts only siblings of the same element type.

<section>
  <h2>Title</h2>
  <p>one</p>     <!-- child #2, p-of-type #1 -->
  <p>two</p>     <!-- child #3, p-of-type #2 -->
</section>

p:nth-child(2)   { color: red; }   /* matches "one": it IS the 2nd child AND a p */
p:nth-child(1)   { color: red; }   /* matches NOTHING — 1st child is the h2 */
p:nth-of-type(1) { color: red; }   /* matches "one" — 1st p among p's */

Interview Q&A

Walk me through why * { box-sizing: border-box } is a near-universal reset.

"The initial value, content-box, makes width mean the content area only, so padding and border are added on top — width: 50% + padding overflows a two-column layout. border-box makes declared width equal the visible box, so percentages, padding, and borders compose predictably; it also matches how form controls already behave in UA stylesheets. It's applied with * plus ::before/::after because box-sizing doesn't inherit; a variant sets it on html and inherits it everywhere so components can opt out subtree-wide."

My card's background starts below where I expect, and the child heading's margin appears outside the card. What's happening?

"Parent–child margin collapsing. The child's margin-top is adjoining the parent's margin-top — nothing (border, padding, inline content, clearance) separates their top edges — so per spec they collapse into a single margin that effectively acts on the parent. Fixes: add padding-top or border-top to the parent to break adjacency, or give the parent a new formatting context — display: flow-root is the side-effect-free option; making it flex or grid also works because flex/grid item margins never collapse."

Exactly when do margins collapse, and what's the resulting size?

"Only vertical margins of block-level boxes in normal flow within the same BFC: adjacent siblings, parent with first/last child when no border/padding/content separates them, and an empty block's own top and bottom margins. The collapsed margin is the max of the positives; with negatives involved it's the largest positive plus the most negative (so two negatives give the more negative one). Never across BFC boundaries, never for floats, abs-pos, inline-blocks, or flex/grid items, never horizontally."

What is padding-top: 56.25% doing in old aspect-ratio hacks?

"Percentage padding — including vertical — resolves against the containing block's width, because height may not be known yet and the spec avoids the circular dependency. So padding-top of 56.25% creates a box whose height is 9/16 of its width: a 16:9 frame. Content then goes in an absolutely positioned child filling the box. Today aspect-ratio: 16/9 replaces the whole trick."

Explain the full cascade order, not just specificity.

"The cascade sorts by criteria in order: first origin and importance — important UA beats important user beats important author, then animations, then normal author, user, UA; note !important inverts origin order. Within author styles, cascade layers apply: unlayered beats layered for normal declarations, and later layers beat earlier ones — flipped for important. Only then specificity, and finally source order. Specificity is genuinely the third tiebreaker, which surprises people who treat it as the whole story."

Compute the specificity of #nav ul li.active a:hover::before and compare with [id="nav"] .active a.

"First: one ID (a=1); one class + one pseudo-class (b=2); ul, li, a, and ::before (c=4) → (1,2,4). Second: [id="nav"] is an attribute selector — b, not a! — plus .active, so b=2, plus a c=1 → (0,2,1). (1,2,4) wins on the first component. The attribute-selector-for-ID trick is a real technique to reference an ID at class-level specificity."

When would you use :where() over :is()?

"They match identically, but :is() contributes the specificity of its most specific argument while :where() contributes zero. I use :where() for library/reset/default styles so any consumer selector — even a bare class — overrides them without specificity wars; :is() when I just want grouping ergonomics and normal specificity, e.g. :is(h1,h2,h3) code. One trap: :is(#a, .b) is ID-level specificity even when it matched via .b."

A third-party stylesheet uses !important. Rank your options to override it.

"Within the important lane: my own !important with higher specificity, or with equal specificity and later source order. Cleaner: cascade layers. For normal declarations, unlayered styles beat layered ones — so importing vendor CSS into @layer vendor lets my plain unlayered styles win without any specificity contest. For important declarations the layer order inverts: an earlier layer's important declarations beat a later layer's — so if I declare @layer overrides, vendor; and put my important rules in overrides (declared first), they beat the vendor's importants. Last resorts: el.style.setProperty(prop, val, 'important') from JS. And I'd flag the smell: recurring important-wars mean the codebase needs a layering strategy, not bigger hammers."

What's the difference between initial, unset, and revert?

"initial is the per-property spec default — element-blind, so display: initial on a div is inline. unset means inherit for inherited properties, initial otherwise — what all: unset uses to strip an element. revert rolls back to the previous origin, usually the UA stylesheet, so it restores the browser's element-appropriate default — what people usually actually want. revert-layer does the same one cascade layer up."

What is a BFC, what creates one, and name two bugs it fixes.

"A block formatting context is an isolated layout region: floats inside don't affect outside, margins don't collapse across its boundary, and it doesn't overlap external floats. Created by the root element, floats, absolute positioning, overflow other than visible/clip, inline-block, table cells, contain: layout/paint, and — purpose-built — display: flow-root. It fixes (1) float containment: a parent of floats collapses to zero height unless it's a BFC; (2) parent–child margin collapse: a BFC boundary is a collapse barrier; and (3) it enables the media-object pattern since a BFC won't wrap under an adjacent float."

There's a mysterious gap under an image inside a div. Why, and name three fixes.

"Because img is a replaced inline element, it sits on the text baseline of its line box, and the line box reserves descender space below the baseline based on the parent's font metrics. That descender space is the gap. Fixes: display: block on the image (no line box at all), vertical-align: bottom/middle (leave the baseline), or line-height: 0/font-size: 0 on the parent (collapse the strut). I default to display: block for standalone images."

Compute the final widths: container 600px, items with flex: 1 1 300px, flex: 1 1 200px, flex: 1 1 300px.

"Total basis 800 vs 600 → shrink by 200. Shrink is weighted by shrink-factor × basis: 300, 200, 300, sum 800. Item1 loses 200×300/800 = 75 → 225. Item2 loses 200×200/800 = 50 → 150. Item3 → 225. Check: 225+150+225 = 600. The key sentence: shrink weights by factor times basis, so larger items give up proportionally more — unlike grow, which weights by factor alone. And in reality content minimums might clamp an item, triggering redistribution."

Why doesn't text-overflow: ellipsis work on my flex item, and why does min-width: 0 fix it?

"Flex items have min-width: auto, which resolves to the content-based minimum size — roughly the min-content width. The long unbreakable text sets a floor below which the item cannot shrink, so it never becomes narrower than the text and ellipsis never triggers; the row overflows instead. min-width: 0 removes the floor; overflow: hidden on the item also works because per spec a non-visible overflow zeroes the automatic minimum. Same mechanism vertically with min-height in column direction, and the same idea in grid where 1fr is really minmax(auto, 1fr) — hence minmax(0, 1fr)."

Difference between align-items and align-content in flexbox?

"align-items aligns items within their flex line on the cross axis — default stretch, which is why flex children are equal height. align-content distributes the lines within the container when there are multiple lines and spare cross space — so it does nothing on a single-line (non-wrapping) container, which is the trap in the question. align-self is the per-item override of align-items."

flex: 1 vs flex: auto — when do they render differently?

"flex: 1 = 1 1 0%: basis zero, so the whole container is free space and grow ratios alone decide sizes — equal columns regardless of content. flex: auto = 1 1 auto: items start at content size, only leftover space is shared, so content-heavy items end up bigger. Different whenever contents differ. Also worth saying: bare flex: 1 sets shrink to 1 and basis to 0%, so never set grow via longhand alone and expect the shorthand's defaults."

Explain auto-fill vs auto-fit.

"Both, inside repeat() with a minmax track, create as many tracks as fit the container. They diverge only when items don't fill all tracks: auto-fill keeps empty tracks, so items stay at their track size and empty space remains as ghost columns; auto-fit collapses empty tracks to zero, letting the 1fr maxes of occupied tracks absorb the space, so few items stretch wide. Dashboard cards with stable sizes → auto-fill; gallery that should always fill the row → auto-fit."

Why can a grid column with 1fr still overflow, and what's the fix?

"Because 1fr is spec-defined as minmax(auto, 1fr) — the auto minimum means the track can't shrink below its content's min-content size. A wide <pre>, table, or unbroken string forces the track wider than its fair share, overflowing the container. Fix: minmax(0, 1fr) for the track, or min-width: 0 / overflow: hidden / word-break on the item. This is the grid version of the flexbox min-width:auto gotcha — same spec concept, automatic minimum size."

How do you choose grid vs flexbox?

"Two axes of judgment. Dimensionality: flexbox is 1D — a row or column where wrapped lines are independent; grid is 2D — rows and columns that must align. Control direction: flexbox is content-out — item sizes drive the negotiation; grid is layout-in — I define tracks and content conforms. So: toolbar, media object, tag cloud, centering → flex. Page shell, card grid where row 2 aligns with row 1, forms, anything needing cross-row alignment → grid. And they compose: grid for the shell, flex inside components. Subgrid now covers the 'align internals across cards' case that used to be unsolvable."

My position: fixed modal is trapped inside a panel instead of covering the viewport. Debug it.

"Some ancestor has a transform, filter, backdrop-filter, perspective, will-change: transform, or contain: layout/paint — per the Transforms/Filters specs any of those makes that ancestor the containing block for fixed descendants, so 'fixed' degrades to absolute-within-the-panel. Common culprits: a leftover transform: translateX(0) from a slide-in animation, or will-change left on for performance. I'd walk up the tree in DevTools checking computed styles for those properties. Fixes: portal the modal to document.body — this is literally why React portals exist — remove the transform after the animation ends, or use <dialog>.showModal()/popover, which render in the top layer and are immune to ancestor containing blocks and stacking contexts."

z-index: 999999 isn't working. Give me your debugging monologue.

"First: is the element positioned or a flex/grid item? z-index is ignored on static block elements. Second — the real issue — z-index only competes within the nearest ancestor stacking context, and stacking contexts are atomic. I walk up from my element looking for context creators: positioned + z-index, opacity < 1, transform, filter, will-change, isolation, mix-blend-mode, contain: paint, or fixed/sticky position. Say I find the parent section has opacity: .98 — that seals my element inside a context whose own level is what actually competes with the header's context. The fix happens at the level where the two sibling contexts meet: raise the section's z-index or remove the opacity — not inflate the child's number. And to prevent the class of bug, isolation: isolate on composite components deliberately scopes their internal z-index games. Long-term, top-layer primitives (dialog, popover) end the arms race for overlays."

What creates a stacking context? List as many triggers as you can.

"Root element; positioned (relative/absolute) with non-auto z-index; fixed or sticky positioning unconditionally; flex/grid items with non-auto z-index; opacity below 1; any transform, translate/rotate/scale, filter, backdrop-filter, perspective, clip-path, or mask; isolation: isolate; mix-blend-mode other than normal; will-change naming any such property; contain: layout or paint; and top-layer elements. The pattern: anything that forces the subtree to composite as a unit."

Why might position: sticky silently do nothing? Three distinct causes.

"One: no inset — sticky needs top/bottom/left/right to know its stick threshold. Two: an ancestor with non-visible overflow becomes the scroll container it sticks within; if that ancestor doesn't scroll (overflow: hidden used for clipping), the sticky element never moves relative to it — appears broken. Three: the parent (containing block) is no taller than the sticky element, so there's no travel range — common when the sticky element is the only child, or a stretched flex sibling situation misleads you; the element only sticks while its parent's box intersects the scrollport. Bonus: in tables, apply sticky to th cells, not thead."

em vs rem — and why unitless line-height?

"rem is relative to the root font-size — predictable, respects user font settings, no compounding; my default for type and spacing scales. em is relative to the element's own font-size (parent's, when sizing font-size itself), so nested em font-sizes multiply — 0.9em three levels deep is 0.729× — but that self-reference is exactly right for component-relative padding, like a button whose padding scales with its label. Line-height should be unitless because a unitless value inherits as a ratio recomputed per descendant font-size, whereas 1.5em/150% inherits as a fixed computed length, so larger-text children get cramped, overlapping lines."

Why is height: 100vh broken on mobile and what replaced it?

"Historically vh was defined against the largest viewport — URL bar retracted — so with the bar visible, 100vh is taller than the visible area and bottom content (CTAs, input bars) hides behind browser chrome. The viewport-unit trio fixes it: svh = small viewport (chrome visible, always safe), lvh = large (the old behavior), dvh = dynamic, tracking the chrome live at the cost of layout resizes during scroll. Rule: svh for must-see content, dvh for app shells, and pair with env(safe-area-inset-*) on notched devices."

:nth-child vs :nth-of-type?

"p:nth-child(2) asks: is this element the second child of its parent, counting every element, and is it a p? Both must hold, so a heading occupying slot 1 shifts everything. p:nth-of-type(2) counts only p siblings — second paragraph regardless of what's interleaved. The modern escape hatch is :nth-child(2 of .item), which counts only matching siblings with full selector specificity control."

How do you correctly style focus without harming accessibility?

"Never bare :focus { outline: none } — that blinds keyboard users. Use :focus-visible for the ring: browsers apply it for keyboard and programmatic focus but not most mouse clicks, so keyboard users get a clear indicator and mouse users don't get flashes. Keep a fallback :focus style for older engines, ensure 3:1 contrast for the indicator, and prefer outline + outline-offset over box-shadow since outline survives overflow: hidden clipping better and respects forced-colors mode."

How would you detect a touch device in CSS, and what's wrong with the question?

"The premise is wrong — devices are hybrids, so I detect capabilities, not device classes. @media (hover: none) and (pointer: coarse) targets a primary input that can't hover and is imprecise — style for bigger touch targets and no hover-dependent UI there. (hover: hover) and (pointer: fine) enables hover affordances. any-hover/any-pointer report whether any attached input has the capability, for hybrids like touch laptops. And regardless of the query, hover must never be the only path to functionality."

Hands-on exercises

Exercise 1 — The centering zoo

Center a child div horizontally and vertically inside a 400px-tall parent, in five different ways: (a) grid one-liner, (b) flexbox, (c) absolute + transform, (d) absolute + auto margins, (e) the modern block-layout way with no flex/grid at all. State one limitation of each.

Solution 
/* (a) Grid — tersest */
.parent { display: grid; place-items: center; }
/* limitation: all direct children get centered; grid context may
   not be wanted if children need normal flow */

/* (b) Flexbox */
.parent { display: flex; justify-content: center; align-items: center; }
/* limitation: multiple children line up in a row; anonymous flex
   items around text can surprise */

/* (c) Absolute + transform — works with unknown child size */
.parent { position: relative; }
.child  { position: absolute; top: 50%; left: 50%;
          transform: translate(-50%, -50%); }
/* limitations: out of flow (parent won't size to child);
   transform creates a stacking context AND a containing block
   for fixed descendants; half-pixel blur possible */

/* (d) Absolute + auto margins — needs inset:0 */
.parent { position: relative; }
.child  { position: absolute; inset: 0; margin: auto;
          width: max-content; height: max-content; }
/* limitation: needs a resolvable child size (explicit or max-content) */

/* (e) Modern block layout — no new formatting context needed */
.parent { align-content: center; text-align: center; }
.child  { display: inline-block; }
/* align-content now applies to plain block containers (2024+ engines);
   limitation: newer browsers only — know the support story */

Exercise 2 — Holy grail, twice

Build the holy grail layout — header, footer, 3 columns (nav 200px / fluid main / aside 240px), footer pinned to the bottom on short pages, columns equal height — once with grid and once with flexbox. Then answer: which parts were harder in flexbox, and why does the main column need minmax(0, 1fr) in the grid version?

Solution 
/* GRID — the natural fit */
body {
  display: grid; min-height: 100dvh;
  grid-template: "header header header" auto
                 "nav    main   aside"  1fr
                 "footer footer footer" auto
               / 200px minmax(0, 1fr) 240px;
}
header { grid-area: header; }  nav { grid-area: nav; }
main   { grid-area: main; }    aside { grid-area: aside; }
footer { grid-area: footer; }

@media (max-width: 700px) {
  body { grid-template: "header" "nav" "main" "aside" "footer" / 1fr;
         grid-template-rows: auto auto 1fr auto auto; }
}

/* FLEXBOX — needs nesting */
body { display: flex; flex-direction: column; min-height: 100dvh; }
.middle { display: flex; flex: 1; }
nav   { flex: 0 0 200px; }
main  { flex: 1; min-width: 0; }   /* the classic fix again */
aside { flex: 0 0 240px; }

/* Answers: flexbox needs an extra wrapper (.middle) because it's 1D;
   equal-height columns come free from align-items: stretch, but the
   2D header/footer relationship needs the column-direction outer flex.
   minmax(0,1fr) on main: 1fr = minmax(auto,1fr); wide content (code
   blocks, tables) would otherwise set a min-content floor and push
   the aside off-screen. */

Exercise 3 — Sticky footer + sticky header, no bugs

Page with a header that sticks on scroll, a footer that sits at the viewport bottom on short pages (but flows normally on long pages), and a main area with a code block that must not cause horizontal page overflow. Then: add overflow-x: hidden to body and explain what breaks.

Solution 
body { min-height: 100dvh; display: flex; flex-direction: column; margin: 0; }
header { position: sticky; top: 0; z-index: 10; background: #fff; }
main   { flex: 1; min-width: 0; }        /* sticky-footer core: flex:1 */
main pre { overflow-x: auto; }           /* scroll the code, not the page */
footer { margin-top: auto; }             /* belt-and-suspenders w/ flex:1 */

/* What breaks with body { overflow-x: hidden }:
   1. body becomes a scroll container → position: sticky on header now
      binds to BODY's scrollport instead of the viewport. In practice
      browsers propagate body overflow to the viewport unless html ALSO
      has non-visible overflow — so the trap fully fires with
      html { overflow-x: hidden } too, a very common "fix" for
      horizontal scroll. Sticky dies silently.
   2. overflow-x: hidden forces overflow-y to compute to auto —
      you may get an unexpected vertical scroll container.
   Correct fix for stray horizontal overflow: find the overflowing
   element (min-width floors, 100vw + scrollbar, unbroken strings)
   or use overflow-x: clip, which is NOT a scroll container and
   leaves sticky alone. */

Exercise 4 — Masonry-ish gallery with dense packing

Build a responsive image gallery: columns auto-generate at ≥160px, rows are 100px units, some items span 2 columns and/or 2 rows, and gaps left by spanning items get backfilled. No media queries. Then explain why this is "masonry-ish" and not true masonry, and note the a11y caveat.

Solution 
.gallery {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(160px, 1fr));
  grid-auto-rows: 100px;
  grid-auto-flow: row dense;    /* backfill holes */
  gap: 8px;
}
.gallery > * { overflow: hidden; border-radius: 8px; }
.gallery img { width: 100%; height: 100%; object-fit: cover; display: block; }
.gallery .wide { grid-column: span 2; }
.gallery .tall { grid-row: span 2; }
.gallery .big  { grid-column: span 2; grid-row: span 2; }

/* Why not true masonry: every item still snaps to fixed row tracks —
   items don't pack directly under the item above at arbitrary heights
   the way Pinterest columns do. True masonry needs the proposed
   grid-template-rows: masonry / display: masonry, or JS/columns hacks
   (CSS columns reorder content top-to-bottom per column — bad for
   reading order). A11y caveat: `dense` reorders items visually while
   DOM/tab order is unchanged — fine for non-sequential image grids,
   wrong for content with meaningful order. */

Exercise 5 — The stacking-context escape room

Given this structure, the tooltip must render above the page header. You may not move elements in the DOM and may not remove the animation. Fix it and explain every step.

<header class="app-header">...</header>   <!-- sticky; z-index: 100 -->
<ul class="results">
  <li class="result-card">                <!-- animation leaves transform: none? NO:
                                              card has transform: translateY(0)
                                              and opacity: .999 from a fade-in -->
    <button>Info</button>
    <div class="tooltip">...</div>        <!-- absolute; z-index: 10000 -->
  </li>
</ul>
Solution 
/* Diagnosis: .result-card has transform + opacity < 1 → it IS a
   stacking context at level auto/0. The tooltip's z-index: 10000 only
   ranks it inside the card. The card's context (0) paints below the
   header's context (100). Also: the transform makes the card the
   containing block for any fixed-position tooltip strategy.

   Fix without moving DOM or killing the animation: */

/* 1. End the animation in a state with NO lingering context triggers:
      animate with a keyframes fill-mode that lands on identity, then
      remove residual properties — opacity must end at exactly 1 and
      transform must be `none` after the animation (animation-fill-mode:
      none, or a JS animationend cleanup). While animating, the tooltip
      can't escape — acceptable, tooltips don't open mid-fade-in. */
.result-card { animation: fade-in .3s ease-out; }
@keyframes fade-in {
  from { opacity: 0; transform: translateY(8px); }
  to   { opacity: 1; transform: none; }   /* and no styles OUTSIDE the
                                              animation setting transform/opacity */
}

/* 2. If the card must keep a transform (hover lift), the only
      CSS-side lever left is raising the whole card's context above
      the header while the tooltip is open: */
.result-card:has(.tooltip.open) { z-index: 101; position: relative; }
/* :has() flips the card's context level only when needed, so cards
   don't permanently stack over the sticky header while scrolling. */

/* 3. The robust architectural answer: top layer. Make the tooltip a
      popover — it escapes ALL ancestor stacking contexts and
      containing blocks, no z-index at all: */
<button popovertarget="tip1">Info</button>
<div id="tip1" popover>...</div>
/* (Anchor positioning or JS keeps it attached to the button.)
   In an interview, walking through 1 → 2 → 3 shows mechanism,
   pragmatism, and platform knowledge in that order. */

Exercise 6 — Flex math on paper

No browser allowed. Container: 1000px, gap: 20px, three items: A flex: 2 1 200px, B flex: 1 2 300px, C flex: 1 1 100px; max-width: 220px. Compute the final widths, including the clamp-and-redistribute step.

Solution 
/* Space for items = 1000 − 2 gaps (40) = 960.
   Σ basis = 200+300+100 = 600 → free space = 360 → GROW phase.
   Σ grow = 2+1+1 = 4 → unit = 90.
   Tentative: A = 200+180 = 380, B = 300+90 = 390, C = 100+90 = 190.

   Clamping: C's max-width is 220 — tentative 190 < 220, no violation.
   (Had C's tentative been > 220, we'd freeze C at 220 and re-run:
   remaining free = 360 − 120 = 240 split A:B = 2:1 → A 360, B 380.)

   Final: A = 380px, B = 390px, C = 190px.  380+390+190+40 = 1000 ✓
   Note B's shrink: 2 was irrelevant — we were in the grow phase;
   shrink only matters when Σbasis exceeds the container. Spotting
   that red herring is exactly what the interviewer wants. */
