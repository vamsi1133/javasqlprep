13 — Testing Frontend

What to test, at which level, and what to leave alone — RTL, MSW, Vitest, and Playwright with senior judgment attached.

Why interviewers ask this

Testing questions are unusual: they grade judgment at least as hard as tool knowledge. Anyone can memorize getByRole. What separates a senior candidate is being able to say what deserves a test, at which level of the stack, and — just as importantly — what doesn't. Interviewers have all seen the two failure modes: the team with 4,000 unit tests that all pass while the checkout button is broken in production, and the team whose every refactor turns 80 tests red without a single real bug being caught. Both are testing strategies that optimize for a number instead of for confidence.

At FAANG, testing shows up inside the coding round ("how would you test this component you just built?") and in system/design follow-ups ("what's your CI story for this feature?"). At product startups it's often more direct: "our test suite is slow and flaky, what would you do?" — a question about ROI and prioritization wearing a technical costume. This tutorial covers the modern stack (React Testing Library, user-event v14, MSW v2, Vitest, Playwright) with exactly-correct async and timer examples — the code people copy into real suites — and, threaded through everything, the judgment framing interviewers actually score.

1. Testing philosophy at the senior level T1

Trophy vs pyramid — why frontend inverted the middle

The classic testing pyramid (many unit tests, fewer integration, a handful of e2e) came from a backend world where "unit" meant a pure class with injectable dependencies and integration tests meant slow, shared databases. Frontend broke both assumptions. A React "unit" — one component rendered in isolation with every child and hook mocked — tests almost nothing real: components exist to compose, and their bugs live in the seams (props wiring, state flowing to the DOM, event handlers, data fetching). Meanwhile, jsdom made integration-style tests fast: rendering a whole feature tree with a mocked network runs in milliseconds.

Kent C. Dodds' testing trophy captures the resulting sweet spot: a base of static analysis (TypeScript, ESLint — bugs caught for free on every keystroke), a modest band of unit tests for pure logic, a fat middle of integration tests (component trees + RTL + MSW), and a thin cap of e2e for the journeys that pay the bills. The slogan is "the more your tests resemble the way your software is used, the more confidence they can give you" — with cost rising as you go up, so the trophy is the point of best confidence-per-dollar, not a claim that e2e is bad.

LevelWhat belongs hereWhat it catchesWhat it costs

Integration (RTL + MSW) Component behavior: user does X → sees Y, incl. loading/error/empty states Wiring, state→DOM, handlers, fetch flows, a11y regressions Fast in jsdom; the workhorse — most tests live here
E2E (Playwright) Critical user journeys only: signup, login, checkout, the one flow that makes money Real-browser, real-integration failures nothing else can see Slow, flake-prone, expensive to maintain — ration them

"Test behavior, not implementation" — made concrete

This phrase is repeated so often it has lost meaning. Here is exactly what an implementation-detail test looks like, and why it's a liability: 

// ❌ IMPLEMENTATION-DETAIL TESTS — every assertion here is a refactor landmine

// 1. Asserting internal state values (enzyme-era habit)
expect(wrapper.state('isOpen')).toBe(true);
//    Breaks if you rename isOpen → expanded, move it into a reducer, or lift it
//    into a parent. Catches zero bugs the DOM assertion below wouldn't catch.

// 2. Calling instance methods directly
wrapper.instance().handleToggle();
//    Breaks the moment the class becomes a function component. Tests a function
//    the user can never call — users click things, they don't invoke methods.

// 3. Asserting props passed to a child (shallow rendering's whole business model)
expect(wrapper.find(Modal).prop('visible')).toBe(true);
//    Breaks if Modal's prop is renamed to `open`, or Modal is replaced by a
//    Dialog, or the boolean is derived differently — even though the USER-VISIBLE
//    behavior (a dialog appears) is identical in all cases.

// ✅ BEHAVIOR TEST — same coverage, survives all of the above refactors
const user = userEvent.setup();
render(<Settings />);
await user.click(screen.getByRole('button', { name: /edit profile/i }));
expect(await screen.findByRole('dialog', { name: /edit profile/i })).toBeInTheDocument();

The litmus test: could a user (or a screen reader) observe what you're asserting? If not, the test is coupled to how the code works rather than what it does. Implementation-detail tests have the worst possible failure matrix: they produce false negatives (red on safe refactors, training the team to ignore or mechanically "fix" tests) and false positives (green while the behavior is broken — the state was set, but nothing rendered it). Two exceptions are fine to name in an interview: asserting a callback prop was called with the right arguments (that callback is the component's public contract with its parent), and asserting an outgoing request body (the contract with the server). Contracts at the boundary are behavior; internals are not.

What to test at each level — the routing rule

  • Pure logic → unit tests. If a function takes data and returns data, test it as a function. Don't render a component to test a price formatter. Corollary: extract gnarly logic out of components precisely so it becomes unit-testable.
  • Component behavior → RTL integration. Render the realistic tree (real children, real context providers, mocked network via MSW). One test per user-meaningful behavior: "shows validation error on empty submit", "renders empty state when API returns []".
  • Critical journeys → e2e. The 5–10 flows where breakage equals lost revenue or lost trust. Not "every page renders" — that's what integration tests and smoke checks are for.

Coverage honesty

100% coverage means every line executed, not every behavior verified. A test that renders a component and asserts nothing gets you full line coverage and zero confidence. Coverage is useful as a negative signal — "this payment module is at 12%, that's a risk" — and nearly useless as a positive one. The senior framing: coverage tells you where you definitely have no tests; it never tells you where you have good ones. Target the risky paths: money, auth, data mutation, anything with branching edge cases, anything that's broken before. Let trivial glue code go untested; the type checker has it.

When asked "what's your coverage target?", refuse the number gracefully: "I care about a coverage floor on changed lines in CI (say 70–80%) as a smoke alarm, but I review what the tests assert, not what they execute. I'd rather have 60% coverage of behavior-level tests than 95% of snapshot noise." Then pivot to the ROI conversation: every test has a purchase price (writing it), a carrying cost (maintaining it through refactors, CI minutes, flake triage), and a payoff (bugs caught × cost of those bugs). Tests on stable, risky, high-traffic code have great ROI. Tests pinning implementation details of churning UI have negative ROI — you pay maintenance forever and they never catch anything.

2. React Testing Library — the deep version T1

The guiding principle, operationalized

RTL's design enforces the philosophy above by subtraction: it gives you no access to state, instances, or props — only the rendered DOM and ways to query it like a user would. render() mounts your tree into a jsdom container; screen is a pre-bound query object for the whole document (prefer it over destructuring queries from render — one idiom everywhere, and it plays nicely with portals that render outside the container). 

import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

test('submits the search term', async () => {
  const user = userEvent.setup();          // v14: always setup() first
  const onSearch = vi.fn();
  render(<SearchBar onSearch={onSearch} />);

  await user.type(screen.getByRole('searchbox', { name: /search products/i }), 'ssd');
  await user.click(screen.getByRole('button', { name: /search/i }));

  expect(onSearch).toHaveBeenCalledWith('ssd');
});

Query priority — and why it doubles as an a11y audit

RTL documents an explicit priority order. It is not stylistic — each step down the list is a step away from how real users (and assistive tech) perceive the page:

The senior insight to say out loud: this ordering makes your test suite a continuous accessibility audit for free. If getByRole('button', { name: /save/i }) can't find your save button, it's because it's a <div onClick> with no role, or an icon button with no accessible name — both real defects for keyboard and screen-reader users. Teams that reach for data-testid first forfeit that audit and couple tests to attributes users never see.

user-event v14 — why it replaced fireEvent

fireEvent.click(el) dispatches exactly one synthetic click event. A real click is a sequence: pointerover → pointerdown → focus (if focusable) → pointerup → click, respecting disabled state, pointer-events CSS, and focusability. user-event simulates the full sequence, which is why it catches bugs fireEvent can't: a button covered by an overlay with pointer-events: none, a handler that depends on focus arriving first, a keydown-based shortcut. Similarly user.type() fires per-character keydown/keypress/input/keyup, so your per-keystroke validation and debounce logic actually exercise. 

const user = userEvent.setup();      // v14 pattern — do this per test, BEFORE render

await user.click(button);            // all APIs are async in v14 — always await
await user.type(input, 'hello{Enter}');
await user.keyboard('{Tab}');        // keyboard navigation — great for a11y flows
await user.selectOptions(select, 'US');
await user.upload(fileInput, file);
await user.hover(tooltipTrigger);
await user.clear(input);
setup() exists because v14 tracks a coherent input-device state (which keys are down, pointer position) across calls, and because it's the injection point for fake-timer integration (setup({ advanceTimers: vi.advanceTimersByTime }) — see §3). Direct calls like userEvent.click(el) still work but forfeit both; the setup pattern is the one to show in interviews.

3. Async testing done right T1

Async is where most real-world test pain lives: flaky waits, act() warnings, timers that never fire. The rules below are exact — this is the section to know cold.

findBy* — the default for "it appears later"

test('loads and shows the user', async () => {
  render(<UserProfile id="1" />);

  // findBy = getBy + waitFor: retries the query until it matches or ~1s passes
  expect(await screen.findByRole('heading', { name: 'Ada Lovelace' })).toBeInTheDocument();

  // and loading is gone — absence check, so queryBy
  expect(screen.queryByText(/loading/i)).not.toBeInTheDocument();
});

waitFor — semantics people get wrong

waitFor(callback) runs the callback repeatedly (on an interval and on DOM mutations) until it stops throwing, or times out. Two consequences follow directly from "runs repeatedly":

  • Never put side effects inside. A user.click() inside waitFor may execute five times before the assertion passes — clicking five times, submitting five forms. Do the action outside; wait for the outcome inside.
  • Keep it to a single assertion (or one logical condition). If assertion #1 keeps failing, you wait the full timeout before ever learning whether #2 would pass, and the failure message points at the wrong thing. One condition per waitFor; put follow-up synchronous assertions after it.
// ❌ side effect inside waitFor — click may fire multiple times
await waitFor(async () => {
  await user.click(screen.getByRole('button', { name: /save/i }));
  expect(screen.getByText(/saved/i)).toBeInTheDocument();
});

// ✅ act outside, assert outcome inside — and prefer findBy when it's "element appears"
await user.click(screen.getByRole('button', { name: /save/i }));
expect(await screen.findByText(/saved/i)).toBeInTheDocument();

// waitFor earns its keep for non-element conditions:
await waitFor(() => expect(onSubmit).toHaveBeenCalledTimes(1));

// disappearance has a dedicated helper — resolves when the element is removed:
await waitForElementToBeRemoved(() => screen.queryByText(/loading/i));
waitForElementToBeRemoved throws if the element is already gone when you call it — it needs to observe the removal. If the timing is uncertain, guard with a presence check first, or just assert the positive replacement state with findBy instead (waiting for the data to appear implies loading disappeared, in most UIs).

The act() warning, actually explained

act() is React's test-time contract: "run this work, then flush everything it caused — state updates, re-renders, effects, and (with modern React) the microtasks in between — before returning." It makes the post-act DOM deterministic instead of mid-update. The warning "An update to X inside a test was not wrapped in act(...)" means: a state update happened at a moment no act() scope was active — almost always because an async thing (a resolved fetch, a timer, a transition) landed after your test stopped looking.

Key fact interviewers probe: RTL already wraps render, user-event calls, and its async utilities in act() for you. So manually wrapping those in act(...) is redundant noise. The warning is not asking you to sprinkle act() — it's telling you your test has a synchronization bug: 

// ❌ the warning-producing test: fetch resolves after assertions finish
test('shows user', () => {
  render(<UserProfile id="1" />);
  expect(screen.getByText(/loading/i)).toBeInTheDocument();
  // test ends → fetch resolves → setState fires outside any act() → warning
  // (and worse: the "loaded" behavior is silently untested)
});

// ❌ the cargo-cult "fix" — wrapping in act() just silences the symptom
await act(async () => { render(<UserProfile id="1" />); });

// ✅ the real fix — await the async outcome; the warning disappears BECAUSE
//    the test now observes the update it was previously abandoning
test('shows user', async () => {
  render(<UserProfile id="1" />);
  expect(screen.getByText(/loading/i)).toBeInTheDocument();
  expect(await screen.findByText('Ada Lovelace')).toBeInTheDocument();
});
The rehearsable line: "An act() warning is a symptom, not the disease. It means my test finished — or asserted — before some async update landed. The fix is to await the thing (findBy, waitFor, waitForElementToBeRemoved), not to wrap everything in act(). The only times I write act() by hand are advancing fake timers and driving updates in renderHook tests, because those bypass RTL's pre-wrapped APIs."

Fake timers + user-event: the debounced search test, in full

This is the classic tricky test. Two systems must cooperate: user.type() internally uses a small delay between keystrokes (awaiting real timers), and your debounce uses setTimeout. With fake timers on, user-event would deadlock waiting for time that never passes — unless you tell it how to advance the fake clock. That's what advanceTimers in setup() is for. 

// SearchBox.tsx — debounces onSearch by 500ms
function SearchBox({ onSearch }) {
  const [value, setValue] = useState('');
  useEffect(() => {
    if (!value) return;
    const id = setTimeout(() => onSearch(value), 500);
    return () => clearTimeout(id);
  }, [value, onSearch]);
  return <input role="searchbox" aria-label="Search"
           value={value} onChange={e => setValue(e.target.value)} />;
}

// SearchBox.test.tsx — the exact, correct pattern
import { render, screen, act } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

beforeEach(() => vi.useFakeTimers());
afterEach(() => {
  vi.runOnlyPendingTimers();   // let queued cleanups fire…
  vi.useRealTimers();          // …then restore, or later tests hang mysteriously
});

test('debounces: one search call, 500ms after the last keystroke', async () => {
  const onSearch = vi.fn();
  // THE critical line — user-event advances the fake clock instead of awaiting it:
  const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTime.bind(vi) });
  render(<SearchBox onSearch={onSearch} />);

  await user.type(screen.getByRole('searchbox', { name: /search/i }), 'react');

  // mid-debounce: nothing yet
  expect(onSearch).not.toHaveBeenCalled();

  // advance 499ms — still nothing (each keystroke reset the timer; 500 counts
  // from the LAST one). act() because the timer callback triggers a state-less
  // call here, but in general timer callbacks setState → they need an act scope.
  act(() => vi.advanceTimersByTime(499));
  expect(onSearch).not.toHaveBeenCalled();

  act(() => vi.advanceTimersByTime(1));
  expect(onSearch).toHaveBeenCalledTimes(1);
  expect(onSearch).toHaveBeenCalledWith('react');
});
Three ways this test rots if written casually: (1) forgetting advanceTimers in setup() → user.type times out or hangs; (2) forgetting vi.useRealTimers() in afterEach → later tests using real waits stall; (3) advancing timers without act() → the debounce fires, calls setState, and you get the act warning plus a stale DOM. Interviewers who've maintained real suites recognize instantly whether you've fought this battle.

Flushing promises with fake timers

If the timer callback itself starts async work (a debounced fetch), advancing timers fires the callback but doesn't wait for its promises. Use the async advancing APIs, which yield to the microtask queue between timer runs: 

await act(async () => { await vi.advanceTimersByTimeAsync(500); });
// or run everything scheduled:  await vi.runAllTimersAsync();
expect(await screen.findByText('3 results')).toBeInTheDocument();

4. MSW — network-level API mocking T1

Why the network level beats mocking fetch/axios

The old way — vi.mock('axios') or stubbing global.fetch — replaces your HTTP client with a function returning a canned object. That skips everything between your component and the wire: URL construction, query-param serialization, request bodies, headers, response parsing, HTTP status handling, interceptors, and your data-fetching library's cache/retry behavior. Your test passes while the real request 400s because a param was named wrong.

MSW (Mock Service Worker) intercepts at the network layer — in Node tests by patching the request internals (setupServer), in the browser via an actual Service Worker (setupWorker). Your app runs its entire real data path; only the server on the other end is fake. Three senior-grade consequences:

  • Refactor-proof: swap fetch → axios → TanStack Query and every test still passes untouched — the mock is defined in HTTP terms, not client-library terms.
  • Realistic failure testing: you can return actual 500s, delays, network errors — exercising the same error-handling code paths production hits.
  • One mock, everywhere: the same handlers power tests, local dev without a backend, and Storybook. The mock becomes a living contract, not per-test throwaway.

Setup — MSW v2 syntax (rest → http)

// src/mocks/handlers.ts  (v2: `http` + `HttpResponse`; v1's `rest`/`res(ctx…)` is gone)
import { http, HttpResponse, delay } from 'msw';

export const handlers = [
  http.get('/api/users', () => {
    return HttpResponse.json([
      { id: 1, name: 'Ada Lovelace', email: 'ada@analytical.dev' },
      { id: 2, name: 'Grace Hopper', email: 'grace@navy.mil' },
    ]);
  }),

  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({ id: Number(params.id), name: 'Ada Lovelace' });
  }),

  http.post('/api/users', async ({ request }) => {
    const body = await request.json();          // standard Fetch API Request
    return HttpResponse.json({ id: 3, ...body }, { status: 201 });
  }),
];

// src/mocks/server.ts
import { setupServer } from 'msw/node';
import { handlers } from './handlers';
export const server = setupServer(...handlers);

// vitest.setup.ts
import { server } from './src/mocks/server';
beforeAll(() => server.listen({ onUnhandledRequest: 'error' })); // unmocked call = failure, not silence
afterEach(() => server.resetHandlers());   // per-test overrides don't leak
afterAll(() => server.close());

The full worked example — loading / success / error / empty

This quartet is the canonical "show me you can test a data-fetching component" answer. Note where each state is produced: happy path from the default handlers, error and empty via server.use() per-test overrides — which prepend a one-off handler that resetHandlers() clears. 

import { render, screen } from '@testing-library/react';
import { http, HttpResponse, delay } from 'msw';
import { server } from '../mocks/server';
import { UserList } from './UserList';

test('shows a loading indicator, then the users', async () => {
  render(<UserList />);

  expect(screen.getByRole('status')).toHaveTextContent(/loading/i);

  expect(await screen.findByRole('list', { name: /users/i })).toBeInTheDocument();
  const items = screen.getAllByRole('listitem');
  expect(items).toHaveLength(2);
  expect(items[0]).toHaveTextContent('Ada Lovelace');
});

test('shows an error state on 500', async () => {
  server.use(
    http.get('/api/users', () => new HttpResponse(null, { status: 500 }))
  );
  render(<UserList />);

  expect(await screen.findByRole('alert')).toHaveTextContent(/couldn't load users/i);
  // and a recovery affordance — error states should be actionable
  expect(screen.getByRole('button', { name: /retry/i })).toBeInTheDocument();
});

test('shows an empty state when there are no users', async () => {
  server.use(http.get('/api/users', () => HttpResponse.json([])));
  render(<UserList />);

  expect(await screen.findByText(/no users yet/i)).toBeInTheDocument();
  expect(screen.queryByRole('list')).not.toBeInTheDocument();
});

test('network failure (not an HTTP status) also lands in the error state', async () => {
  server.use(http.get('/api/users', () => HttpResponse.error())); // connection-level failure
  render(<UserList />);
  expect(await screen.findByRole('alert')).toBeInTheDocument();
});
await delay(200) (or delay('infinite')) inside a handler is how you make loading states observable or permanent — far better than racing a fast mock. And if you use TanStack Query, create a fresh QueryClient per test with retry: false, or the error test waits through three retries before failing over.

GraphQL awareness

import { graphql, HttpResponse } from 'msw';

export const handlers = [
  graphql.query('GetUser', ({ variables }) => {
    return HttpResponse.json({
      data: { user: { id: variables.id, name: 'Ada Lovelace' } },
    });
  }),
  graphql.mutation('UpdateUser', () => {
    return HttpResponse.json({ errors: [{ message: 'Forbidden' }] }); // GraphQL errors are 200s!
  }),
];

MSW matches GraphQL operations by name, regardless of endpoint — and it lets you test the GraphQL quirk that errors arrive in a 200 response body, which pure fetch-mocks routinely get wrong.

5. Mocking judgment T2

Module mocking — vi.mock / jest.mock and the hoisting gotcha

vi.mock('./analytics') is hoisted above all imports by the transform — it runs before your module-under-test loads, so the mock is in place when the import resolves. The gotcha: the factory can't close over variables declared later in the file (they don't exist yet at hoist time). Vitest throws a helpful error; Jest fails silently unless the variable starts with mock. 

// ❌ ReferenceError — `track` isn't initialized when the hoisted factory runs
const track = vi.fn();
vi.mock('./analytics', () => ({ track }));

// ✅ vi.hoisted lifts the variable creation along with the mock
const { track } = vi.hoisted(() => ({ track: vi.fn() }));
vi.mock('./analytics', () => ({ track }));

// ✅ partial mock — keep the real module, replace one export
vi.mock('./api', async (importOriginal) => {
  const actual = await importOriginal<typeof import('./api')>();
  return { ...actual, uploadFile: vi.fn().mockResolvedValue({ url: '/f/1' }) };
});

Mocking time and dates

vi.useFakeTimers();
vi.setSystemTime(new Date('2026-07-02T09:00:00Z'));  // Date.now(), new Date() now frozen

expect(formatRelative(new Date('2026-07-01T09:00:00Z'))).toBe('yesterday');

vi.useRealTimers();  // always restore — afterEach, not at the end of the happy path

Mocking browser APIs — the jsdom gaps

jsdom deliberately doesn't implement layout or several observer APIs. Any component using matchMedia, IntersectionObserver, or ResizeObserver crashes in tests until you stub them — in the setup file, once, not per-test: 

// vitest.setup.ts — the standard jsdom-gap stubs
Object.defineProperty(window, 'matchMedia', {
  writable: true,
  value: vi.fn().mockImplementation((query) => ({
    matches: false, media: query, onchange: null,
    addEventListener: vi.fn(), removeEventListener: vi.fn(),
    addListener: vi.fn(), removeListener: vi.fn(),   // legacy API some libs still call
    dispatchEvent: vi.fn(),
  })),
});

class ObserverStub {
  observe = vi.fn(); unobserve = vi.fn(); disconnect = vi.fn();
  takeRecords = () => [];
}
vi.stubGlobal('IntersectionObserver', ObserverStub);
vi.stubGlobal('ResizeObserver', ObserverStub);

// also commonly needed: scrollIntoView, URL.createObjectURL, clipboard
Element.prototype.scrollIntoView = vi.fn();
If a behavior genuinely depends on layout (virtualized lists, "load more when the sentinel is visible"), a stub that never fires tests nothing — that behavior belongs in Playwright, where a real layout engine exists. Knowing which side of that line a test sits on is exactly the judgment being graded.

When NOT to mock

  • Your own components. Mocking <UserCard /> inside a <UserList /> test converts an integration test back into a shallow-render test — the wiring between them (the most bug-prone part) is now unverified. Render the real child. Exception: a genuinely heavy third-party embed (map SDK, video player, chart canvas) — stub the foreign thing, never your own.
  • Your own state management. Don't mock useSelector/useStore. Provide a real store with preloaded state (§7) — otherwise selectors, reducers, and subscription wiring are all outside the test.
  • The thing you're testing. Sounds obvious; happens constantly via transitive over-mocking. Each mock is a bet that the boundary is stable and correct — every one you add shrinks what the test can catch.

Spies on callback props — the legitimate contract assertion

const onAddToCart = vi.fn();
render(<ProductCard product={ssd} onAddToCart={onAddToCart} />);

await user.click(screen.getByRole('button', { name: /add to cart/i }));

expect(onAddToCart).toHaveBeenCalledTimes(1);
expect(onAddToCart).toHaveBeenCalledWith({ productId: ssd.id, quantity: 1 });

Module mocks vs dependency injection — prefer the seam you designed

vi.mock reaches into the module graph and swaps a file — powerful, but it couples the test to a file path and hides the dependency. If a component instead receives its dependencies via props or context (an onSave callback, an ApiProvider, a ClockContext), tests inject fakes through the front door — no hoisting rules, no path coupling, and the seam is visible in the component's API. The senior heuristic: if you keep needing vi.mock for the same module, that's a design signal — the dependency wants to be injectable. Reserve module mocking for third-party edges and true singletons.

Snapshot testing — the honest take

Snapshots assert "output equals whatever it was last time someone pressed u." That's the whole critique in one sentence. For a large component tree, the snapshot is hundreds of lines of markup nobody reads; every legitimate change churns it; reviewers rubber-stamp the diff; and the test degenerates into a change-detector with no opinion about correctness — pure carrying cost.

  • Bad: full-tree component snapshots as your primary component tests.
  • Fine: small, stable, serializable outputs where the whole value is the shape — an error object, a generated className map, a transformed API payload, a CLI help string. Inline snapshots (toMatchInlineSnapshot) are the best form: the expected value lives in the test file, so the diff is reviewed in place and can't silently sprawl.
  • Discipline: a snapshot update in a PR is a claim — "I looked at this diff and the new output is intended." If your team can't honestly say that, delete the snapshot; it's asserting nothing.

7. Testing hooks, stores, and non-visual logic T2

renderHook — when there's no component to render

renderHook (now exported from @testing-library/react itself) mounts a tiny host component that calls your hook and exposes the latest return value on result.current. Updates you trigger imperatively must be wrapped in act() — this is one of the two legitimate manual-act sites: 

import { renderHook, act } from '@testing-library/react';

test('useCounter clamps at max', () => {
  const { result } = renderHook(() => useCounter({ max: 2 }));

  act(() => result.current.increment());
  act(() => result.current.increment());
  act(() => result.current.increment());   // should clamp

  expect(result.current.count).toBe(2);
});

// hooks that read context: supply providers via the wrapper option
const { result } = renderHook(() => useCart(), {
  wrapper: ({ children }) => <CartProvider>{children}</CartProvider>,
});
result.current is a live getter for the latest render's value — but destructuring (const { count } = result.current) freezes a stale copy from before your act() call. Always read through result.current.x at assertion time.

Often better: test the hook through a consuming component

A hook's purpose is to power a component; if it has one obvious consumer, test that — you verify the hook and its integration, via user-visible behavior, with no act() bookkeeping. Reserve renderHook for genuinely reusable library hooks with many consumers, where you want to pin the hook's contract independent of any one UI.

Reducers and selectors — the payoff of extraction

// Pure function → trivial, exhaustive, millisecond tests. No render, no DOM, no async.
test('applying a coupon never produces a negative total', () => {
  const state = cartReducer(cartWith({ total: 500 }), applyCoupon({ amountOff: 900 }));
  expect(selectTotal(state)).toBe(0);
});

This is the concrete argument for extracting logic from components: every branch you move from JSX into a reducer/selector moves from "needs an integration test" to "needs a one-line unit test." Edge-case matrices (empty cart, expired coupon, currency rounding) belong here, not in RTL.

Store-connected components — real store, preloaded state

// Redux Toolkit: a renderWithStore helper beats mocking useSelector every time
function renderWithStore(ui, { preloadedState } = {}) {
  const store = configureStore({ reducer: rootReducer, preloadedState });
  return { store, ...render(<Provider store={store}>{ui}</Provider>) };
}

test('checkout button disabled for an empty cart', () => {
  renderWithStore(<CartSummary />, { preloadedState: { cart: { items: [] } } });
  expect(screen.getByRole('button', { name: /checkout/i })).toBeDisabled();
});

// Zustand: reset the (module-singleton) store between tests, then set state directly
beforeEach(() => useCartStore.setState(initialCartState, true));
test('badge shows item count', () => {
  useCartStore.setState({ items: [item1, item2] });
  render(<CartBadge />);
  expect(screen.getByText('2')).toBeInTheDocument();
});

Why real-store-over-mocked-hook is the senior answer: mocking useSelector hardcodes the derived value, so selectors, reducer transitions, and re-render subscriptions are all unexercised — precisely the layers where store bugs live. With a real store, dispatching an action in the test verifies the whole loop: action → reducer → selector → DOM.

8. Playwright e2e — and Cypress literacy T2

Why Playwright won

  • Real cross-browser engines: Chromium, Firefox, and WebKit (the only practical way to test Safari-ish behavior in CI on Linux) from one API.
  • Auto-waiting, web-first assertions: await expect(locator).toBeVisible() retries until it passes or times out; actions like click() wait for the element to be visible, stable, enabled, and unobscured before acting. This one design choice deletes the sleep(2000) class of flake at the source.
  • Trace viewer: a failed CI run hands you a time-travel recording — DOM snapshots, network, console, per-action screenshots. Debugging remote flake goes from folklore to forensics.
  • Isolation + parallelism: each test gets a fresh browser context (cheap incognito profile), so tests parallelize safely across workers and shard across CI machines.

Locators — the same philosophy as RTL, on purpose

import { test, expect } from '@playwright/test';

test('user can sign in', async ({ page }) => {
  await page.goto('/login');

  await page.getByLabel('Email').fill('ada@lovelace.dev');
  await page.getByLabel('Password').fill('correct-horse-battery');
  await page.getByRole('button', { name: 'Sign in' }).click();

  await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
  await expect(page).toHaveURL(/\/dashboard/);
});

getByRole, getByLabel, getByText, getByTestId — deliberately mirroring Testing Library. A team that standardizes on role-based locators gets one selector language across unit, integration, and e2e, and the a11y-audit side effect at every level. A locator is lazy — it re-queries at action time, which is what makes auto-waiting and retry-ability work.

Fixtures and auth state reuse — stop logging in per test

// global-setup: log in ONCE, persist cookies + localStorage to disk
import { chromium } from '@playwright/test';

export default async function globalSetup() {
  const browser = await chromium.launch();
  const page = await browser.newPage();
  await page.goto('http://localhost:3000/login');
  await page.getByLabel('Email').fill(process.env.E2E_USER!);
  await page.getByLabel('Password').fill(process.env.E2E_PASS!);
  await page.getByRole('button', { name: 'Sign in' }).click();
  await page.waitForURL('**/dashboard');
  await page.context().storageState({ path: 'e2e/.auth/user.json' });
  await browser.close();
}

// playwright.config.ts — every test starts already authenticated
use: { baseURL: 'http://localhost:3000', storageState: 'e2e/.auth/user.json', trace: 'on-first-retry' }

Login is the slowest, flakiest step in most suites, and it multiplied by every test. storageState amortizes it to once per run (or once per role — an admin.json and a member.json, selected per project/fixture). This is a standard "how do you make e2e fast?" answer.

Network interception — hermetic vs true e2e

test('shows the outage banner when the status API is down', async ({ page }) => {
  await page.route('**/api/status', (route) =>
    route.fulfill({ status: 503, contentType: 'application/json',
                    body: JSON.stringify({ error: 'maintenance' }) })
  );
  await page.goto('/dashboard');
  await expect(page.getByRole('alert')).toContainText(/scheduled maintenance/i);
});

The trade-off to articulate: true e2e (real backend, seeded test data) is the only thing that verifies the whole system — but it's slow, needs environment discipline, and fails for reasons outside your code. Hermetic e2e (route.fulfill everything) is fast and deterministic — but it's really a browser-grade integration test; the backend contract is unverified. Sensible split: the handful of revenue-critical journeys run true e2e against a staging environment; broader UI flows run hermetic; forced-failure scenarios (503s, timeouts) are hermetic by necessity — you can't ask staging to fall over on cue.

Flake control — the operational discipline

  • No manual sleeps, ever. waitForTimeout(3000) is either too long (slow suite) or too short (flake). Every wait should be a condition: a web-first assertion, waitForURL, or a waitForResponse.
  • Retries as a detector, not a rug. retries: 2 in CI keeps the pipeline green, but a test that passed on retry is flagged "flaky" in the report — track those, don't ignore them.
  • Quarantine, don't delete or skip-and-forget: move known-flaky tests to a non-blocking job with an owner and a deadline. An ignored red suite trains everyone to ignore red.
  • Root causes are usually: shared mutable test data (fix: per-test unique data / API-seeded fixtures), unawaited navigation, animations (disable in test env), and time (freeze it).
  • CI sharding: --shard=1/4 splits the suite across machines; with fully-parallel isolated tests, wall-clock time divides almost linearly. Merge reports afterward.

Cypress literacy — one honest paragraph

Cypress runs your test code inside the browser alongside the app (an iframe + injected runner), whereas Playwright drives the browser from outside via CDP/driver protocols. Cypress's architecture gives it a famously interactive dev experience (time-traveling command log, live reload) and made it the 2018–2021 default; it also imposes real constraints — historically one browser tab, same-origin friction, limited multi-tab/iframe control, WebKit support still experimental, and a command queue with its own async model instead of plain await. Teams still reasonably pick it for its component-testing runner, existing suites and team familiarity, or its dashboard ecosystem. In an interview, respect it, state the architectural difference accurately, and explain your default is Playwright for cross-browser coverage, parallelism, worker isolation, and standard async/await ergonomics.

9. The senior test-strategy answer T2

"How would you test this feature?" is the most common senior testing question, and it wants a portfolio allocation, not a tool list. Here's a rehearsable shape, using "add a checkout coupon field" as the feature:

  1. Static first: the coupon API response and cart math get real types. A discriminated union for CouponResult makes "forgot the expired-coupon case" a compile error, not a test to write.
  2. Pure logic → unit: extract applyCoupon(cart, coupon) and test the matrix as data: percentage vs fixed, stacking rules, floor-at-zero, currency rounding, expiry. Ten one-line tests, milliseconds each.
  3. Component behavior → RTL + MSW: render the real checkout form. Tests: valid code updates the displayed total; invalid code shows an inline error (role="alert") and doesn't change the total; the field is disabled while validating; a 500 shows a retriable error. All four via MSW handlers + server.use overrides.
  4. One happy-path e2e: Playwright — apply a real seeded coupon on staging, complete checkout, assert the order total. One test, because checkout is revenue-critical; not five, because the variants are already covered a level down.
  5. A11y assertions ride along: the RTL tests already enforce label/role correctness by construction (query priority); add jest-axe / axe-core checks (expect(await axe(container)).toHaveNoViolations()) on the form's key states for rules queries can't catch (contrast, aria misuse).
  6. Visual regression only where pixels are the product: not for this form. For a marketing page, a design system, or a chart library, Chromatic/Percy screenshot diffs earn their review cost — elsewhere they're snapshot noise at higher resolution.
  7. Test data builders: cartWith({ items: 2, total: 500 }) style factories (plain functions or fishery/faker) so tests state only what they care about — the single biggest readability lever in a mature suite.
  8. CI gates: on every PR — typecheck, lint, unit + integration (fast, parallel, must pass); on merge to main — the e2e suite (sharded), preview-deploy smoke test; nightly — full cross-browser matrix and visual diffs. Fast feedback where iteration happens, expensive certainty where it protects the trunk.
Close the answer with the maintenance stance: "And I treat the suite as a product — flaky tests get quarantined with an owner, implementation-detail tests get deleted when they cry wolf, and if a test has never failed for a real reason in a year and slows every CI run, I'll question its ROI. The goal is a suite the team trusts enough to deploy on green." That last sentence — deploy on green — is the whole point of testing, and saying it signals you know that.

10. Frontier awareness T3

  • React Server Components & server actions: unit-testing RSC is still rough — async components and the server/client boundary don't fit jsdom's model cleanly (partial support is emerging, but it's not settled practice). Current pragmatic answer: keep RSC thin (fetch + compose), unit-test the extracted data/transform functions, test client islands with RTL as usual, and cover the composed server-rendered result with Playwright against a running app. Server actions: test the action as a function where possible, plus one e2e for the form round-trip.
  • Storybook interaction tests: a play function in a story runs testing-library-style interactions inside the story; the test-runner (or Vitest addon) executes all stories headlessly in CI. Attractive because the story is the test fixture — one artifact serves docs, visual review, and behavior tests.
  • Contract testing: (Pact et al., or pragmatically: generating MSW handlers + types from the OpenAPI/GraphQL schema) attacks the residual risk of hermetic mocks — "my mock says the API returns userName, the API says user_name." Schema-derived mocks make that drift a build failure instead of a production incident.
  • Mutation testing: (Stryker) flips your operators/conditions and checks whether any test fails — measuring whether tests assert, not just execute. Too slow to run everywhere; excellent as a periodic audit of a critical module, and the sharpest rebuttal to coverage-percentage worship.

Interview Q&A

Q1: Testing pyramid vs testing trophy — which do you follow for frontend, and why?

The trophy, because frontend inverts the pyramid's economics. The pyramid assumed unit tests were the cheapest source of confidence and integration tests were slow. In frontend, an isolated component "unit" (children and hooks mocked) verifies almost nothing — component bugs live in the seams: props wiring, state reaching the DOM, fetch flows. Meanwhile jsdom + MSW made integration tests nearly as fast as unit tests. So: a base of static analysis (free bug-catching), unit tests reserved for pure logic (reducers, formatters, validation), the bulk of tests at the RTL-integration level where user-visible behavior is verified, and a thin e2e layer for revenue-critical journeys. The principle underneath: confidence per dollar — tests that resemble real usage catch more real bugs, and you buy as much of that as each level's cost allows.

Q2: "Test behavior, not implementation" — give me a concrete example of an implementation-detail test and why it's harmful.

Classic enzyme-era example: expect(wrapper.state('isOpen')).toBe(true), or asserting wrapper.find(Modal).prop('visible'). Both break if I rename the state field, convert class to hooks, or swap Modal for Dialog — refactors that change zero user-visible behavior — while passing if the state is set but nothing actually renders. Worst of both worlds: false negatives on safe refactors (training the team to mechanically appease tests) and false positives on real breakage. The behavior version — click the trigger, assert getByRole('dialog') appears — survives every one of those refactors and fails for every real regression. My litmus test: could a user or screen reader observe this assertion? Two boundary exceptions are legitimate: callback props called with correct arguments, and outgoing request payloads — those are the component's public contracts.

Q3: Why does RTL rank getByRole first and getByTestId last?

The order tracks how closely each query resembles real perception. getByRole with the accessible-name option queries the same accessibility tree a screen reader navigates — if getByRole('button', { name: /save/i }) fails, either it's a clickable div with no role or an icon button with no accessible name, both genuine a11y defects. So the query priority doubles as a free, continuous accessibility audit. testid sits last because it corresponds to nothing any user perceives — the test passes regardless of whether the element is labeled, visible to AT, or even the right kind of control. I keep test-ids as an escape hatch for genuinely unnameable containers, and I treat each one as a small signal that the markup might lack semantics.

Q4: Explain getBy vs queryBy vs findBy — and when each is correct.

getBy: synchronous, throws (with a DOM dump) on zero or multiple matches — for elements that must exist right now. queryBy: returns null on zero matches — its only proper job is asserting absence (expect(queryByText(...)).not.toBeInTheDocument()), because getBy would throw before the negated matcher runs. findBy: async — effectively getBy wrapped in waitFor, retrying until match or ~1s timeout — for anything appearing after async work. The All variants return arrays with matching semantics (queryAllBy gives [], so it's the one for asserting a count that might be zero). A common smell: using queryBy for presence checks — it returns null and you get "can't read property of null" instead of RTL's helpful error.

Q5: Why user-event over fireEvent?

fireEvent.click dispatches one bare event; user.click simulates the full interaction sequence — pointerover, pointerdown, focus, pointerup, click — respecting disabled state, pointer-events, and focusability. user.type fires per-character keydown/input/keyup, so keystroke-level logic (debounce, per-char validation, maxlength) actually executes. That fidelity catches real bugs fireEvent silently passes: a button visually covered by an overlay, a handler depending on focus order, keyboard shortcuts. In v14 the pattern is const user = userEvent.setup() per test — setup maintains coherent device state across calls and is where fake-timer integration plugs in (advanceTimers) — and every API is async, so always await. fireEvent remains a fallback for events user-event can't produce, like a raw scroll or programmatic load event.

Q6: You see "not wrapped in act(...)" warnings. What does act() actually do, and what's the correct fix?

act() defines a scope in which React flushes everything the enclosed work caused — state updates, re-renders, effects — before continuing, so assertions see a settled DOM. The warning means a state update fired when no act scope was active — in tests, almost always an async operation (fetch resolution, a timer) landing after the test stopped observing. Crucially, RTL already wraps render, user-event, and its async utilities in act — so the fix is never to sprinkle act() around; act-wrapping just silences the symptom while leaving the post-update behavior untested. The real fix is to await the async outcome: await screen.findBy…, waitFor, or waitForElementToBeRemoved — the warning disappears because the test now observes the update. The two places I write act() manually: advancing fake timers whose callbacks setState, and imperative updates in renderHook tests — both bypass RTL's pre-wrapped surface.

Q7: What are the rules for what goes inside a waitFor callback?

waitFor re-runs the callback — on an interval and on DOM mutations — until it stops throwing or times out. Two rules follow. First, no side effects inside: a user.click in the callback may execute several times before the assertion passes — multiple submissions, multiple mutations. Act outside, assert the outcome inside. Second, one assertion (one logical condition) per waitFor: with several, a failure in the first burns the whole timeout and masks whether the others would pass, and the error points at the wrong condition. Also worth saying: when the condition is "an element appears," findBy is the same mechanism with better ergonomics — I reserve waitFor for non-element conditions like expect(spy).toHaveBeenCalled(), and use waitForElementToBeRemoved for disappearance.

Q8: Why MSW instead of vi.mock-ing your fetch wrapper or axios?

Because mocking the client skips the entire data path: URL and query-string construction, request-body serialization, headers, status-code handling, response parsing, interceptors, and whatever TanStack Query/SWR does with the result. Those layers are where fetch bugs actually live — a test against a mocked axios passes while production 400s on a misnamed param. MSW intercepts at the network level, so the app executes all of its real code and only the server is fake. Three concrete wins: tests survive refactors of the fetching layer (fetch→axios→React Query changes zero tests); error paths are realistic (actual 500 responses, actual network errors via HttpResponse.error(), delays); and the same handlers serve tests, local dev, and Storybook, so the mock becomes a shared, reviewable contract. I run the server with onUnhandledRequest: 'error' so an unexpected request fails loudly, and use server.use() for per-test error/empty overrides that resetHandlers() cleans up.

Q9: Walk me through testing the four states of a data-fetching component.

Default MSW handlers return the happy payload. Loading: render, synchronously assert the spinner/skeleton (by role="status"); if the mock resolves too fast to observe, add await delay(200) in the handler. Success: await screen.findByRole for the loaded content, then assert loading is gone with queryBy. Error: server.use(http.get(url, () => new HttpResponse(null, { status: 500 }))), then findByRole('alert') — and I assert the retry affordance exists, because an error state without recovery is a design bug. Empty: override with HttpResponse.json([]) and assert the empty-state copy, plus absence of the list. One caveat I always mention: with TanStack Query, build a fresh QueryClient per test with retry: false, or the error test silently waits through three retries.

Q10: How do you test a debounced search input? (The fake-timers + user-event question.)

The trap is that user-event awaits real time between keystrokes, so with fake timers enabled it hangs — unless you wire them together: vi.useFakeTimers(), then userEvent.setup({ advanceTimers: vi.advanceTimersByTime }) so user-event advances the fake clock instead of waiting. Then: type the query, assert the callback hasn't fired; advance 499ms inside act() (timer callbacks trigger state updates), assert still nothing — this proves each keystroke reset the timer; advance the final 1ms and assert exactly one call with the final string. Cleanup matters: vi.runOnlyPendingTimers() then vi.useRealTimers() in afterEach, or later tests hang. And if the debounced callback does async work — a fetch — advancing timers isn't enough; await vi.advanceTimersByTimeAsync(500) also yields to the microtask queue so the promise chain flushes.

Q11: When do you choose NOT to mock something?

My default is to mock only at boundaries I don't own: the network (via MSW), time, browser APIs jsdom lacks, and heavy third-party SDKs. I don't mock my own components — mocking UserCard inside a UserList test removes the wiring between them from the test, and wiring is where component bugs live. I don't mock my own state management — a real store with preloaded state exercises selectors, reducers, and subscriptions; mocking useSelector hardcodes the answer. Every mock is a bet that the boundary behaves as the mock claims; each one narrows what the test can catch. And if I find myself repeatedly vi.mock-ing the same internal module, I read that as a design smell — that dependency wants to be injected via props or context, where tests can swap it through the front door.

Q12: What's the vi.mock/jest.mock hoisting gotcha, and how do you do a partial mock?

vi.mock(path, factory) is hoisted above all imports by the transform so the mock exists before the module graph loads. Consequence: the factory runs before the rest of your file's top-level code, so it can't reference variables declared below it — you get a ReferenceError (Vitest at least tells you; Jest historically failed cryptically unless the variable name started with mock). The fix in Vitest is vi.hoisted(() => ...) to lift the shared spy's creation alongside the mock. Partial mocks: give the factory importOriginal — vi.mock('./api', async (importOriginal) => ({ ...await importOriginal(), uploadFile: vi.fn() })) — keeping real exports and replacing one. Jest's equivalent is jest.requireActual spread into the factory return.

Q13: Your team is on Jest; the app is on Vite. Do you migrate to Vitest? What actually changes?

Almost certainly yes, and the honest pitch is that it's a config migration, not a test rewrite. Vitest reuses the Vite config — same aliases, plugins, and transforms as the app — killing the maintain-two- pipelines problem and the whole class of "Jest encountered an unexpected token" ESM failures, since Vitest is ESM/TS-native. The expect API is deliberately Jest-compatible; RTL, jest-dom, and MSW work unchanged; the mechanical work is jest.* → vi.* (or a codemod / globals flag) and moving setup files. Wins: dramatically faster watch mode, threaded parallelism, in-source and type testing. What I'd check before committing: any dependence on Jest-only ecosystem pieces, and jsdom vs happy-dom choice — happy-dom is faster but occasionally misses APIs, and it's switchable per file, so I'd start with jsdom compatibility and optimize later. I would not migrate a React Native codebase — Jest is still the paved road there.

Q14: Give me your honest opinion of snapshot testing.

A snapshot asserts "output equals last time someone pressed u" — it has no opinion about correctness, only about change. For big component trees that's negative-ROI: hundreds of lines nobody reads, churn on every legitimate change, and rubber-stamped updates that would happily lock in a bug. Where I do use them: small, stable, serializable outputs where the shape is the value — transformed API payloads, error objects, generated config — and preferably as inline snapshots, so the expected value sits in the test file and gets genuinely reviewed in the diff. The discipline test I apply: when someone updates a snapshot, can they honestly say "I read this diff and the new output is intended"? If the team can't, the snapshot is asserting nothing and should be replaced with a few targeted assertions on what actually matters.

Q15: How do you test a custom hook?

First question: does it need direct testing at all? If the hook has one consumer, I test through that component — user-visible behavior, no act bookkeeping, and the integration is covered too. For genuinely reusable hooks (a library useDebounce, usePagination), I use renderHook from @testing-library/react: read the latest value via result.current, wrap imperative updates in act() (renderHook is one of the legitimate manual-act sites), and supply context via the wrapper option. Key trap: never destructure from result.current before acting — you capture a stale snapshot; always read result.current.x at assertion time. And if the "hook" is really wrapping pure logic, the best move is extracting that logic into a plain function and unit-testing it exhaustively, leaving the hook as thin React plumbing.

Q16: How do you test components connected to Redux or Zustand?

Real store, preloaded state — never mock the hooks. For Redux Toolkit, a renderWithStore helper builds a fresh configureStore with the real root reducer and a preloadedState argument, wraps the UI in Provider, and returns the store so tests can also dispatch or inspect. For Zustand, the store is a module singleton, so the discipline is resetting it in beforeEach (useStore.setState(initial, true)) and then setting the scenario's state directly. Why this over mocking useSelector: the mock hardcodes derived values, so selectors, reducer transitions, and subscription-driven re-renders — the layers where store bugs actually occur — go unexercised. With a real store, a test can click a button and verify the full loop: action → reducer → selector → DOM. It also means selector/reducer edge cases still get their own cheap pure-function unit tests, separate from the component tests.

Q17: Why has Playwright largely displaced Cypress and Selenium for new projects?

Three structural reasons. First, real cross-browser: Chromium, Firefox, and WebKit from one API — the only practical Safari-adjacent coverage in Linux CI. Second, auto-waiting web-first assertions: expect(locator).toBeVisible() retries until pass-or-timeout and actions wait for actionability (visible, stable, enabled, unobscured), which eliminates the manual-sleep class of flake by design. Third, isolation and scale: cheap per-test browser contexts make full parallelism and CI sharding safe, and the trace viewer turns remote failures into a time-travel recording of DOM, network, and console. Architecturally, Cypress runs the test inside the browser with the app — great interactivity, but historical constraints on multi-tab, origins, and browser support — while Playwright drives from outside over CDP/driver protocols with plain async/await. I'd still respect an existing Cypress suite; the migration cost is real and its component runner and dashboard keep teams productive.

Q18: Your e2e suite is flaky and everyone re-runs it until green. What do you do?

Treat it as an incident, because "re-run until green" means the suite no longer gates anything. Immediate triage: turn on trace-on-retry and use the retry report to identify flaky tests rather than hide them — a pass-on-retry is a signal, not a success. Quarantine the known-flaky set into a non-blocking job with named owners and deadlines, so the blocking suite is trustworthy again this week. Then fix root causes, which are nearly always: manual sleeps (replace with condition waits — web-first assertions, waitForURL, waitForResponse), shared mutable test data (per-test unique data, API-seeded fixtures instead of UI setup), animations (disable in the test env), time (freeze it), and login-per-test (storageState). Structurally, shrink the suite: many "e2e" tests are really integration tests wearing a browser — push them down to RTL+MSW or hermetic route-fulfilled Playwright where determinism is free. The metric I'd track is pass-on-first-try rate, and the goal is cultural: red must mean something again.

Q19: In Playwright, when do you intercept the network (route.fulfill) versus hit a real backend?

It's a confidence-vs-determinism trade. True e2e — real backend, seeded staging data — is the only test that verifies the whole system including the API contract, so the revenue-critical journeys (signup, login, checkout) get that treatment, few in number and heavily maintained. Hermetic e2e — route.fulfill for every call — is deterministic, fast, and runs anywhere, but it's really a browser-grade integration test: if the backend renames a field, it stays green. I use it for breadth (many UI flows) and for forced-failure scenarios — 503s, slow responses, malformed payloads — which you structurally cannot ask a shared staging environment to produce on cue. The residual risk of hermetic mocks drifting from reality is exactly what contract testing / schema-generated mocks address, which is the senior-level connective tissue worth naming.

Q20: A PM asks why you're "spending sprint time on tests" — and separately, leadership wants 100% coverage. Your answer?

Same answer to both, from opposite directions: tests are an economic instrument, and both zero-testing and coverage-worship misprice it. To the PM: the tests I write buy deployment speed — a trusted suite is what lets us ship on green without a manual QA pass, catch regressions in minutes instead of in production, and refactor without fear; I'm spending sprint time to make every future sprint faster. To leadership: coverage measures lines executed, not behavior verified — a suite of assertion-free renders scores 100% and catches nothing, and the last 20% forces tests on trivial glue with negative ROI. What I'll commit to instead: a coverage floor on changed lines as a smoke alarm, behavior-level tests concentrated on the risky paths (money, auth, data mutation), e2e on the critical journeys, and — if they want proof of test quality — a periodic mutation-testing audit on critical modules, which measures whether tests actually fail when the code is broken. That's the number that means something.

Exercise 1 — Test an accordion component (queries + user-event)

Given this component, write tests that: (a) only the first panel's content is hidden initially, (b) clicking a header reveals its panel and sets aria-expanded, (c) clicking again hides it, (d) keyboard users can operate it with Enter. Use role-based queries only — no test-ids.

function Accordion({ items }) {
  const [openId, setOpenId] = useState(null);
  return (
    <div>
      {items.map(item => (
        <section key={item.id}>
          <h3>
            <button aria-expanded={openId === item.id}
                    onClick={() => setOpenId(openId === item.id ? null : item.id)}>
              {item.title}
            </button>
          </h3>
          {openId === item.id && <div role="region" aria-label={item.title}>{item.body}</div>}
        </section>
      ))}
    </div>
  );
}
Solution 
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

const items = [
  { id: 1, title: 'Shipping', body: 'Ships in 2 days' },
  { id: 2, title: 'Returns',  body: '30-day returns' },
];

test('panels are collapsed initially', () => {
  render(<Accordion items={items} />);
  // content absent → queryBy; role=region only exists when open
  expect(screen.queryByRole('region')).not.toBeInTheDocument();
  expect(screen.getByRole('button', { name: 'Shipping' }))
    .toHaveAttribute('aria-expanded', 'false');
});

test('clicking a header reveals its panel', async () => {
  const user = userEvent.setup();
  render(<Accordion items={items} />);

  await user.click(screen.getByRole('button', { name: 'Shipping' }));

  const region = screen.getByRole('region', { name: 'Shipping' });
  expect(region).toHaveTextContent('Ships in 2 days');
  expect(screen.getByRole('button', { name: 'Shipping' }))
    .toHaveAttribute('aria-expanded', 'true');
  // and the OTHER panel stayed closed — behavior, not incidental structure
  expect(screen.queryByRole('region', { name: 'Returns' })).not.toBeInTheDocument();
});

test('clicking an open header collapses it', async () => {
  const user = userEvent.setup();
  render(<Accordion items={items} />);
  const header = screen.getByRole('button', { name: 'Shipping' });

  await user.click(header);
  await user.click(header);

  expect(screen.queryByRole('region')).not.toBeInTheDocument();
  expect(header).toHaveAttribute('aria-expanded', 'false');
});

test('operable via keyboard', async () => {
  const user = userEvent.setup();
  render(<Accordion items={items} />);

  await user.tab();                       // focus lands on first header button
  expect(screen.getByRole('button', { name: 'Shipping' })).toHaveFocus();
  await user.keyboard('{Enter}');
  expect(screen.getByRole('region', { name: 'Shipping' })).toBeInTheDocument();
});

Judgment notes: absence via queryBy (never getBy); asserting aria-expanded is legitimate because it's part of the accessible contract, not an internal; the keyboard test doubles as the a11y check that these are real buttons, not click-divs.

Exercise 2 — Async data component with MSW: all four states

OrderHistory fetches GET /api/orders on mount, shows a role="status" loader, then a list of orders (role="list", accessible name "Order history"), an empty state ("No orders yet"), or a role="alert" error with a Retry button that refetches. Write the four state tests plus a retry-recovers test, using MSW v2 with server.use overrides.

Solution 
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { http, HttpResponse, delay } from 'msw';
import { server } from '../mocks/server';   // listen/reset/close live in vitest.setup.ts
import { OrderHistory } from './OrderHistory';

const orders = [
  { id: 'o-1', total: 4999, item: 'Mechanical keyboard' },
  { id: 'o-2', total: 1299, item: 'USB-C cable' },
];

test('loading → success', async () => {
  server.use(http.get('/api/orders', async () => {
    await delay(100);                       // make the loading state reliably observable
    return HttpResponse.json(orders);
  }));
  render(<OrderHistory />);

  expect(screen.getByRole('status')).toBeInTheDocument();

  const list = await screen.findByRole('list', { name: /order history/i });
  expect(within(list).getAllByRole('listitem')).toHaveLength(2);
  expect(screen.queryByRole('status')).not.toBeInTheDocument();
});

test('empty state', async () => {
  server.use(http.get('/api/orders', () => HttpResponse.json([])));
  render(<OrderHistory />);

  expect(await screen.findByText(/no orders yet/i)).toBeInTheDocument();
  expect(screen.queryByRole('list')).not.toBeInTheDocument();
});

test('error state on 500', async () => {
  server.use(http.get('/api/orders', () => new HttpResponse(null, { status: 500 })));
  render(<OrderHistory />);

  const alert = await screen.findByRole('alert');
  expect(alert).toHaveTextContent(/couldn't load your orders/i);
});

test('retry recovers from a transient failure', async () => {
  const user = userEvent.setup();
  let calls = 0;
  server.use(http.get('/api/orders', () => {
    calls += 1;
    return calls === 1
      ? new HttpResponse(null, { status: 500 })   // first call fails…
      : HttpResponse.json(orders);                    // …retry succeeds
  }));
  render(<OrderHistory />);

  await screen.findByRole('alert');
  await user.click(screen.getByRole('button', { name: /retry/i }));

  expect(await screen.findByRole('list', { name: /order history/i })).toBeInTheDocument();
  expect(screen.queryByRole('alert')).not.toBeInTheDocument();
});

Judgment notes: the stateful handler in the retry test is the standard MSW pattern for "fails then succeeds" — no client-mock could express this as naturally. If the component uses TanStack Query, wrap each render in a fresh QueryClientProvider with retry: false or the 500 test stalls through automatic retries.

Exercise 3 — Debounced autocomplete: fake timers + user-event + MSW together

CityAutocomplete debounces input by 300ms, then fetches GET /api/cities?q=… and renders suggestions in a role="listbox". Write a test proving: no request fires mid-typing; one request fires 300ms after the last keystroke with the correct query param; suggestions render. This combines every async mechanism in §3–4.

Solution 
import { render, screen, act } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { http, HttpResponse } from 'msw';
import { server } from '../mocks/server';
import { CityAutocomplete } from './CityAutocomplete';

beforeEach(() => vi.useFakeTimers());
afterEach(() => {
  vi.runOnlyPendingTimers();
  vi.useRealTimers();
});

test('debounces, sends the right query, renders suggestions', async () => {
  const seenQueries: string[] = [];
  server.use(http.get('/api/cities', ({ request }) => {
    const q = new URL(request.url).searchParams.get('q');
    seenQueries.push(q!);                    // asserting the OUTGOING request = contract test
    return HttpResponse.json([{ id: 1, name: 'Berlin' }, { id: 2, name: 'Bern' }]);
  }));

  const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTime.bind(vi) });
  render(<CityAutocomplete />);

  await user.type(screen.getByRole('combobox', { name: /city/i }), 'ber');

  // mid-debounce: no request has fired
  expect(seenQueries).toHaveLength(0);

  // advance past the debounce; the callback starts a fetch → use the ASYNC advance
  // so promise microtasks flush between timer runs
  await act(async () => { await vi.advanceTimersByTimeAsync(300); });

  // exactly one request, with the final debounced value — not one per keystroke
  expect(seenQueries).toEqual(['ber']);

  const listbox = await screen.findByRole('listbox');
  const options = within(listbox).getAllByRole('option');
  expect(options.map(o => o.textContent)).toEqual(['Berlin', 'Bern']);
});

Why each piece is there: advanceTimers in setup keeps user.type from hanging under fake timers; advanceTimersByTimeAsync (not the sync version) lets the fetch promise chain resolve; the seenQueries capture asserts the outgoing request — the one implementation boundary that IS contract, catching a misnamed query param that a client-side mock would never see.

Exercise 4 — Signup form: full user flow with validation and submit contract

A signup form has Email, Password, a "Terms" checkbox, and a Create account button. Rules: submit with invalid email shows "Enter a valid email" tied to the field; password under 8 chars shows an error; the button is disabled until Terms is checked; a valid submit calls onSignup({ email, password }) exactly once and disables the button while pending. Write the suite.

Solution 
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

function setup(props = {}) {
  const user = userEvent.setup();
  const onSignup = vi.fn().mockResolvedValue(undefined);
  render(<SignupForm onSignup={onSignup} {...props} />);
  return {
    user, onSignup,
    email: () => screen.getByLabelText(/email/i),
    password: () => screen.getByLabelText(/password/i),
    terms: () => screen.getByRole('checkbox', { name: /terms/i }),
    submit: () => screen.getByRole('button', { name: /create account/i }),
  };
}

test('invalid email shows an error associated with the field', async () => {
  const { user, email, password, terms, submit, onSignup } = setup();

  await user.type(email(), 'not-an-email');
  await user.type(password(), 'longenough1');
  await user.click(terms());
  await user.click(submit());

  // the error is programmatically tied to the input → the INPUT's accessible
  // description contains it (aria-describedby); this asserts the a11y wiring too
  expect(email()).toHaveAccessibleDescription(/enter a valid email/i);
  expect(email()).toBeInvalid();                  // aria-invalid="true"
  expect(onSignup).not.toHaveBeenCalled();
});

test('short password shows an error', async () => {
  const { user, email, password, terms, submit, onSignup } = setup();

  await user.type(email(), 'ada@lovelace.dev');
  await user.type(password(), 'short');
  await user.click(terms());
  await user.click(submit());

  expect(screen.getByText(/at least 8 characters/i)).toBeInTheDocument();
  expect(onSignup).not.toHaveBeenCalled();
});

test('submit is disabled until terms are accepted', async () => {
  const { user, terms, submit } = setup();
  expect(submit()).toBeDisabled();
  await user.click(terms());
  expect(submit()).toBeEnabled();
});

test('valid submit calls onSignup once and shows pending state', async () => {
  // a controllable promise lets us observe the in-flight state deterministically
  let resolveSignup!: () => void;
  const onSignup = vi.fn(() => new Promise<void>(res => { resolveSignup = res; }));
  const { user, email, password, terms, submit } = setup({ onSignup });

  await user.type(email(), 'ada@lovelace.dev');
  await user.type(password(), 'longenough1');
  await user.click(terms());
  await user.click(submit());

  // the contract with the parent: right shape, exactly once
  expect(onSignup).toHaveBeenCalledTimes(1);
  expect(onSignup).toHaveBeenCalledWith({ email: 'ada@lovelace.dev', password: 'longenough1' });

  // while pending: disabled (double-submit guard)
  expect(await screen.findByRole('button', { name: /creating/i })).toBeDisabled();

  resolveSignup();
  expect(await screen.findByRole('button', { name: /create account/i })).toBeEnabled();
});

Judgment notes: the tiny setup() helper is a test-data-builder in miniature — each test states only what it varies. The deferred-promise trick is the deterministic way to test pending UI (no setTimeout races). toHaveAccessibleDescription asserts the error is announced, not merely rendered nearby — a validation test that also guards the aria-describedby wiring.

Write (a) a true-e2e happy path: an authenticated user adds a product to the cart and completes checkout against a seeded staging backend; (b) a hermetic failure test: the payment API returns 402 and the user sees an actionable error without losing their cart. Use storageState for auth and no manual waits anywhere.

export default defineConfig({
  globalSetup: './e2e/global-setup',       // logs in once, writes e2e/.auth/user.json

import { test, expect } from '@playwright/test';

test('happy path: add to cart and complete checkout', async ({ page }) => {
  // seeded product on staging — data setup happens via API/fixtures, not UI clicks
  await page.goto('/products/e2e-test-ssd');

  await page.getByRole('button', { name: 'Add to cart' }).click();
  // web-first assertion — retries until the cart badge updates; no sleep
  await expect(page.getByRole('link', { name: 'Cart, 1 item' })).toBeVisible();

  await page.getByRole('link', { name: /cart/i }).click();
  await page.getByRole('button', { name: 'Checkout' }).click();

  await page.getByLabel('Card number').fill('4242 4242 4242 4242');
  await page.getByLabel('Expiry').fill('12/28');
  await page.getByLabel('CVC').fill('123');
  await page.getByRole('button', { name: 'Pay now' }).click();

  await expect(page.getByRole('heading', { name: 'Order confirmed' })).toBeVisible();
  await expect(page).toHaveURL(/\/orders\/[\w-]+/);
});

test('payment declined: actionable error, cart preserved', async ({ page }) => {
  // hermetic override of ONE endpoint — you can't ask staging to decline on cue
  await page.route('**/api/payments', (route) =>
    route.fulfill({
      status: 402,
      contentType: 'application/json',
      body: JSON.stringify({ error: 'card_declined' }),
    })
  );

  await page.goto('/products/e2e-test-ssd');
  await page.getByRole('button', { name: 'Add to cart' }).click();
  await page.getByRole('link', { name: /cart/i }).click();
  await page.getByRole('button', { name: 'Checkout' }).click();
  await page.getByLabel('Card number').fill('4242 4242 4242 4242');
  await page.getByLabel('Expiry').fill('12/28');
  await page.getByLabel('CVC').fill('123');
  await page.getByRole('button', { name: 'Pay now' }).click();

  const alert = page.getByRole('alert');
  await expect(alert).toContainText(/card was declined/i);
  await expect(page.getByRole('button', { name: /try a different card/i })).toBeVisible();
  // the cart survived the failure — the assertion that catches real money bugs
  await expect(page.getByRole('link', { name: 'Cart, 1 item' })).toBeVisible();
});
