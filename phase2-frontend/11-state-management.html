11 — State Management

Classify the state first, then pick the tool — Redux/RTK internals, server-cache libraries, atomic stores, and the decision framework seniors are expected to recite.

Why interviewers ask this

State management is on your resume, which means it will be probed in a very specific way: not "do you know Redux syntax" but "can you justify an architecture?" The junior tell is tool advocacy ("we use Zustand because Redux is boilerplate"). The senior tell is classification before selection: you name the category of state in front of you — server cache, shared client state, local UI state, URL state, form state — and each category maps to a best-fit tool with known trade-offs. Interviewers will push on internals (how does useSelector decide to re-render? what exactly does staleTime gate? how does Immer make a "mutation" immutable?) precisely because internals are unfakeable — you either have the model or you don't.

At FAANG, expect mechanism questions and "design the state layer for X" system-design hybrids. At product startups, expect war-story probing: "your app got slow — walk me through it," "you inherited a 400-reducer Redux app — what now," "why did you pick TanStack Query over RTK Query." Both reward the same thing: a rehearsable decision framework backed by exact mechanics. This tutorial builds both. Everything is current for Redux Toolkit 2.x, TanStack Query v5, Zustand v5, React 18/19; legacy patterns (sagas, Recoil, hand-rolled Redux) are covered as literacy because you'll meet them in real codebases and in "what would you migrate first" questions.

1. The taxonomy — five kinds of state T1

Open every state-management answer with this. It is the single highest-leverage framing in the whole topic, because it converts "which library do you like" into an engineering classification problem. "State" is not one thing. An app has at least five distinct categories, each with a different lifecycle, ownership, and consistency model — and therefore a different best-fit tool:

CategoryExamplesOwned byBest-fit tools
Local UI state is dropdown open, hover, input focus, accordion expanded One component useState / useReducer
Shared client state cart contents, selected workspace, undo stack, feature toggles set client-side The client — the browser is the source of truth Lifted state → Context (low-frequency) → Zustand / Redux Toolkit / Jotai
Server cache user profile, product list, search results, notifications The server. The client only holds a possibly-stale snapshot TanStack Query, RTK Query, SWR — or RSC on the server
URL state current route, filters, page number, active tab, selected item id The address bar / history stack Router search params (useSearchParams), path params
Form state field values, dirty/touched flags, validation errors, submit status The form, until submit react-hook-form, TanStack Form; native uncontrolled + FormData for simple cases

Server cache is not client state — the load-bearing distinction

Remote data has properties that client state never has: it is owned elsewhere (someone else can change it while you're looking at it), it can be stale, it must be fetched asynchronously (loading/error/success lifecycle), it should be deduplicated (two components asking for the same user should trigger one request), it should be refetched on signals (focus, reconnect, interval), and mutations must reconcile the local snapshot with the server's truth (invalidation or optimistic patches). None of that is a state problem — it's a caching problem, and hand-rolling a cache inside Redux means hand-writing request status flags, dedup logic, staleness tracking, and invalidation for every endpoint.

The strongest one-liner in the topic: "Most of the pain that historically drove teams to Redux was misclassified server cache." The 2016-era Redux store was 80% isLoading/error/ data triples and thunks that fetched into them. Move that to TanStack Query or RTK Query and the global client store shrinks dramatically — often to the point where Context or Zustand covers what's left. Saying this, unprompted, is how you signal you've lived through the shift rather than read about it.

The colocation principle

The second framing rule: keep state as close to where it's used as possible; lift it only when a second consumer actually appears. ("Colocation" — Kent C. Dodds popularized the term.) Local state is easier to delete, easier to reason about, resets naturally on unmount, and re-renders the smallest possible subtree. Global state is a liability you take on deliberately: every reader couples to it, it outlives components (so you must manage resetting it), and it widens the re-render blast radius. The failure mode interviewers are testing for is the dev who reaches for the global store by default. 

// Escalation ladder — move DOWN this list only when forced:
// 1. useState in the component that uses it
// 2. Lift to nearest common parent (two siblings need it)
// 3. Composition / children props (avoid prop drilling without any library)
// 4. Context (subtree-wide, low write frequency: theme, auth, locale)
// 5. External store (app-wide, high frequency, or needs access outside React)
//
// Server data never enters this ladder — it goes to a server-cache library.
"We put everything in Redux so it's all in one place" is an anti-signal at senior level. Uniformity is not a virtue when it means dropdown-open booleans live in a global singleton, survive navigation, need manual reset actions, and force devtools noise on every keystroke. Consistency of approach per category is the goal, not one bucket for everything.

2. React built-ins as tools T1

Before any library: React ships a complete toolkit for local and subtree state. Interviewers check that you treat these as first-class tools with known limits, not as the thing you outgrow on day one.

useState vs useReducer

Both hold local state; the difference is where update logic lives. useState spreads logic across event handlers; useReducer centralizes it in a pure function keyed by action. Reach for useReducer when: (a) the next state depends on multiple pieces of the previous state, (b) several handlers perform overlapping transitions, (c) you want to unit-test transitions without rendering, or (d) you're encoding a lite state machine (§10). Bonus mechanic worth citing: dispatch has a stable identity, so passing it deep never invalidates memo'd children — unlike a fresh callback closure. 

type State = { items: CartItem[]; couponCode: string | null };

type Action =
  | { type: 'added'; item: CartItem }
  | { type: 'removed'; id: string }
  | { type: 'couponApplied'; code: string };

function cartReducer(state: State, action: Action): State {
  switch (action.type) {
    case 'added': {
      const existing = state.items.find(i => i.id === action.item.id);
      return existing
        ? { ...state, items: state.items.map(i =>
            i.id === action.item.id ? { ...i, qty: i.qty + 1 } : i) }
        : { ...state, items: [...state.items, action.item] };
    }
    case 'removed':
      return { ...state, items: state.items.filter(i => i.id !== action.id) };
    case 'couponApplied':
      return { ...state, couponCode: action.code };
  }
}

const [state, dispatch] = useReducer(cartReducer, initialState);
// Testable with zero React: expect(cartReducer(s, a)).toEqual(...)

Lifting state & composition

When two siblings need the same value, lift it to the nearest common ancestor and pass it down. When the pain is drilling through layers that don't care, the first fix is composition, not a store: pass the already-built element as children so intermediate layers never see the data. This also has a rendering benefit (children received as props aren't re-created when the wrapper re-renders — see tutorial 07).

Context — transport, not a state manager

Be precise about what Context is: a mechanism for making a value available to a subtree without prop threading. It has no store semantics — no selectors, no update batching strategy, no subscription granularity. Every consumer of a context re-renders whenever the provider's value is a new reference, regardless of which part of the value it reads. That makes Context the right tool for low-write-frequency, app-wide values — theme, locale, current user, feature flags, a dependency-injected service client — and the wrong tool for anything updating per-keystroke or per-frame. 

// The classic footgun: a fresh object every render invalidates ALL consumers
<AuthContext.Provider value={{ user, login, logout }}>   // new {} each render!

// Fixes, in order of preference:
// 1. Memoize the value
const value = useMemo(() => ({ user, login, logout }), [user]);

// 2. Split contexts by update frequency / by concern
<UserContext.Provider value={user}>          // changes rarely
  <AuthActionsContext.Provider value={actions}> // stable forever
    {children}

// 3. Hold state ABOVE the provider so typing in a form inside the
//    provider's subtree doesn't re-render the provider itself.
The stock interview phrasing: "Context is transport, not state management. It solves prop drilling; it does not solve update granularity. If consumers need to subscribe to slices of a frequently-changing value, that's exactly the gap external stores fill with selector-based subscriptions via useSyncExternalStore." (React's experimental use(Context) + context selectors have been discussed for years; as of React 19 there is still no built-in selective subscription.)

3. Redux & Redux Toolkit — deep T1

Redux is on your resume, so expect the deepest probing here: principles → why they exist → RTK's modern shape → internals. Structure your answers the same way.

3.1 Core principles, and the why behind each

  • Single store — one state tree per app. Why: one place to inspect, serialize, persist, and hydrate; devtools can show the entire app's truth at a point in time.
  • State is read-only; changes happen by dispatching actions — plain serializable objects describing what happened. Why: every change has a name and a log entry; you can replay, record, and ship the action log in a bug report.
  • Changes are made by pure reducers — (state, action) => newState, no mutation, no side effects. Why: purity makes transitions deterministic and testable; immutability makes change detection a reference comparison (prev !== next) instead of a deep walk, and enables time-travel because old states are never destroyed.
  • Unidirectional flow — view dispatches → reducer computes → store notifies → view re-reads. Why: no two-way binding spaghetti; you can always answer "what caused this state?"

The umbrella why: predictability as a product feature of the architecture. Time-travel debugging, action replay, and "serialize the store into the bug ticket" all fall out of serializable actions + pure reducers + immutable snapshots. When asked "why Redux over X," lead with the debugging/audit story, not the API.

3.2 Redux Toolkit is the standard — legacy vs createSlice

RTK has been the official recommended way to write Redux since ~2019; hand-rolled Redux (action-type constants, action creators, switch reducers, manual immutable spreads, manual store setup) is legacy you can read, not style you should write. The delta: 

// LEGACY (recognize it, don't write it)
const ADD_TODO = 'todos/ADD_TODO';
const addTodo = (text) => ({ type: ADD_TODO, payload: { id: nanoid(), text } });
function todosReducer(state = [], action) {
  switch (action.type) {
    case ADD_TODO:
      return [...state, { ...action.payload, done: false }];
    default:
      return state;
  }
}

// MODERN — createSlice generates action creators + types from reducer names
import { createSlice, PayloadAction } from '@reduxjs/toolkit';

const todosSlice = createSlice({
  name: 'todos',
  initialState: [] as Todo[],
  reducers: {
    // Looks like mutation. It is NOT — Immer runs inside createSlice.
    todoAdded(state, action: PayloadAction<string>) {
      state.push({ id: nanoid(), text: action.payload, done: false });
    },
    todoToggled(state, action: PayloadAction<string>) {
      const todo = state.find(t => t.id === action.payload);
      if (todo) todo.done = !todo.done;
    },
  },
});

export const { todoAdded, todoToggled } = todosSlice.actions; // 'todos/todoAdded'
export default todosSlice.reducer;

3.3 How Immer works — the follow-up you must nail

Immer wraps your state in a Proxy (the "draft"). Your reducer "mutates" the draft; the proxy traps every get/set. On a write, Immer copies the touched object once (copy-on-write) and records that this node changed. When the reducer returns, Immer finalizes: changed nodes and their ancestors up to the root are replaced with new objects; everything untouched keeps its original reference — structural sharing. That last part is why memoized selectors and React.memo keep working: unchanged branches are reference-identical across updates, exactly as if you'd hand-written the spreads.

Two Immer traps interviewers fish for: (1) Mutate or return, never both. In a case where you return a brand-new value (e.g. return initialState), don't also mutate the draft in the same code path — Immer throws. (2) console.log(state) inside a slice reducer prints a Proxy; use current(state) from RTK/Immer to snapshot it. Also remember Immer only helps inside createSlice/createReducer — mutating in a thunk or component is still a real bug.

3.5 createAsyncThunk — the request lifecycle

For async work that lands in the store (when you're not using RTK Query), createAsyncThunk generates a three-phase action lifecycle: pending → fulfilled | rejected. You handle the phases in extraReducers: 

export const fetchUser = createAsyncThunk(
  'users/fetchById',
  async (userId: string, { rejectWithValue }) => {
    const res = await fetch(`/api/users/${userId}`);
    if (!res.ok) return rejectWithValue(await res.json());
    return (await res.json()) as User;
  }
);

const usersSlice = createSlice({
  name: 'users',
  initialState: { entity: null as User | null,
                  status: 'idle' as 'idle'|'loading'|'succeeded'|'failed',
                  error: null as string | null },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchUser.pending,   (s) => { s.status = 'loading'; s.error = null; })
      .addCase(fetchUser.fulfilled, (s, a) => { s.status = 'succeeded'; s.entity = a.payload; })
      .addCase(fetchUser.rejected,  (s, a) => { s.status = 'failed';
                                                s.error = a.error.message ?? 'failed'; });
  },
});
Then immediately note: this hand-written status/error/entity triple is the boilerplate RTK Query and TanStack Query eliminate — plus the dedup, caching, and refetching that createAsyncThunk never gives you (dispatch the thunk from two components and you get two fetches). Showing you know when NOT to use createAsyncThunk is worth more than the syntax.

3.6 Selectors, useSelector, and memoization — the re-render rule

The exact mechanic: after every dispatched action, react-redux runs your selector against the new state and re-renders the component if the result is not reference-equal (=== by default) to the previous result. Two consequences: 

// ✅ Fine: selecting a primitive or a stored reference
const count = useSelector(s => s.cart.items.length);       // number: === works
const items = useSelector(s => s.cart.items);              // same ref until items change

// ❌ Re-renders on EVERY action, related or not:
const names = useSelector(s => s.users.list.map(u => u.name)); // new array each call
const pair  = useSelector(s => ({ a: s.a, b: s.b }));           // new object each call
// (react-redux warns about these in dev: "Selector returned a different result
//  when called with the same parameters")

Fixes: pass shallowEqual as the second arg for the object-pair case, or — the canonical answer — memoize derived data with createSelector (Reselect, re-exported by RTK): 

import { createSelector } from '@reduxjs/toolkit';

const selectItems  = (s: RootState) => s.cart.items;
const selectCoupon = (s: RootState) => s.cart.couponCode;

export const selectCartTotal = createSelector(
  [selectItems, selectCoupon],           // input selectors
  (items, coupon) => {                   // result fn: runs ONLY when an input ref changes
    const subtotal = items.reduce((sum, i) => sum + i.price * i.qty, 0);
    return coupon ? subtotal * 0.9 : subtotal;
  }
);
// Mechanics: createSelector caches the last inputs + last result (cache size 1 by
// default). If every input selector returns a reference-equal value, it returns
// the CACHED result — same reference — so useSelector sees === and skips the render.

Selector factory — the cache-size-1 gotcha: one memoized selector shared by 50 list rows, each calling it with a different id, thrashes the cache and memoizes nothing. Give each component its own instance: 

const makeSelectTodosByProject = () =>
  createSelector(
    [(s: RootState) => s.todos.entities, (_: RootState, projectId: string) => projectId],
    (entities, projectId) =>
      Object.values(entities).filter(t => t.projectId === projectId)
  );

function ProjectTodos({ projectId }: { projectId: string }) {
  const selectTodos = useMemo(makeSelectTodosByProject, []); // per-instance cache
  const todos = useSelector(s => selectTodos(s, projectId));
  ...
}

// per-argument caching without the factory dance.)

3.7 Normalized state — createEntityAdapter

Store relational data like a database, not like the API response. Why normalize: (1) no deeply nested updates — each entity is one flat record; (2) O(1) lookup by id instead of array scans; (3) a single source of truth per entity — a comment's author isn't duplicated inside five posts, so updating the author touches one record; (4) lists become arrays of ids, so reordering doesn't touch entities. 

import { createEntityAdapter, createSlice } from '@reduxjs/toolkit';

const todosAdapter = createEntityAdapter<Todo>({
  sortComparer: (a, b) => b.createdAt - a.createdAt,
});
// State shape it manages: { ids: string[], entities: Record<string, Todo> }

const todosSlice = createSlice({
  name: 'todos',
  initialState: todosAdapter.getInitialState({ status: 'idle' }), // extra fields ok
  reducers: {
    todoAdded: todosAdapter.addOne,
    todoUpdated: todosAdapter.updateOne,   // { id, changes: {...} }
    todosReceived(state, action: PayloadAction<Todo[]>) {
      todosAdapter.setAll(state, action.payload);
      state.status = 'succeeded';
    },
  },
});

export const {
  selectAll: selectAllTodos,     // memoized: entities → sorted array
  selectById: selectTodoById,    // O(1)
  selectIds: selectTodoIds,
} = todosAdapter.getSelectors((s: RootState) => s.todos);

// List perf pattern: parent subscribes to selectTodoIds (stable when only an
// entity's FIELDS change); each memo'd <TodoRow id={id}/> subscribes to

3.8 Side effects: listener middleware, and saga/observable literacy

Listener middleware (RTK-native) is the modern answer to "reactive side effects": run logic when an action is dispatched or a predicate on state changes — with async, cancellation, and take/condition helpers — without adding a generator runtime. 

import { createListenerMiddleware } from '@reduxjs/toolkit';
export const listenerMiddleware = createListenerMiddleware();

listenerMiddleware.startListening({
  actionCreator: todoAdded,
  effect: async (action, api) => {
    api.cancelActiveListeners();          // debounce-style: kill previous run
    await api.delay(500);
    await analytics.track('todo_added', { id: action.payload.id });
    // api.dispatch / api.getState / api.condition(...) also available
  },
});

redux-saga — two-paragraph literacy for legacy codebases: sagas are long-running processes written as generator functions that yield plain effect descriptions (call, put, take, select, fork) which the saga middleware interprets. Because a saga yields descriptions rather than executing directly, tests can step the generator and assert on yielded effects without mocking I/O. Concurrency operators are the draw: takeLatest (cancel the previous run — free race-condition handling for search-as-you-type), takeLeading, debounce, race, and structured cancellation of forked tasks.

The cost: a second programming model inside your app, weak TypeScript ergonomics, and the realization that ~90% of saga usage was "fetch on action" — which server-cache libraries now do better. redux-observable is the same niche with RxJS: epics are functions (action$) => action$ that map an action stream to a stream of new actions (switchMap ≈ takeLatest). Interview stance: "I can read and maintain sagas; for new code I'd use RTK Query for data and listener middleware for the residual reactive logic, keeping sagas only for genuinely complex long-lived workflows until they're migrated."

3.9 Redux internals — store in ~20 lines, and how React subscribes

// The entire conceptual core of Redux: a closure over state + listeners.
function createStore(reducer, preloadedState) {
  let state = preloadedState;
  let listeners = [];

  function getState() { return state; }

  function dispatch(action) {
    state = reducer(state, action);        // synchronous, one at a time
    listeners.forEach(l => l());           // notify AFTER state is replaced
    return action;
  }

  function subscribe(listener) {
    listeners.push(listener);
    return function unsubscribe() {
      listeners = listeners.filter(l => l !== listener);
    };
  }

  dispatch({ type: '@@redux/INIT' });      // populate initial state
  return { getState, dispatch, subscribe };
}
// Middleware = a wrapper around dispatch:
//   (storeApi) => (next) => (action) => next(action)
// composed right-to-left, so each middleware sees actions before the reducer.
// Thunk is trivially: action typeof 'function' ? action(dispatch, getState) : next(action)

How react-redux connects this to React (v8+): useSelector is built on useSyncExternalStore. It subscribes to the store; on every notify it runs the selector against getState() and compares with the previous result; if unchanged, no re-render. uSES also guarantees consistency with concurrent rendering — if the store changes while React is rendering, React detects the tear and re-renders synchronously rather than committing a frame where two components read different store versions. (Pre-v8 react-redux did this manually with subscriptions + forced updates and a top-down subscription tree to avoid "zombie children" reading fresher state than their parents.)

4. RTK Query — the server-cache half of RTK T1

RTK Query is Redux Toolkit's built-in data-fetching layer (@reduxjs/toolkit/query/react). It stores its cache in the Redux store (a slice you add), generates React hooks per endpoint, and handles dedup, caching, invalidation, and lifecycle for you. If a team already runs Redux, this is the default answer for server cache. 

import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

export const api = createApi({
  reducerPath: 'api',
  baseQuery: fetchBaseQuery({
    baseUrl: '/api',
    prepareHeaders: (headers, { getState }) => {
      const token = (getState() as RootState).auth.token;
      if (token) headers.set('authorization', `Bearer ${token}`);
      return headers;
    },
  }),
  tagTypes: ['Todo'],
  endpoints: (build) => ({
    getTodos: build.query<Todo[], void>({
      query: () => '/todos',
      // Tag the list AND each entity, so both can be invalidated precisely:
      providesTags: (result) =>
        result
          ? [...result.map(t => ({ type: 'Todo' as const, id: t.id })),
             { type: 'Todo', id: 'LIST' }]
          : [{ type: 'Todo', id: 'LIST' }],
    }),
    addTodo: build.mutation<Todo, Partial<Todo>>({
      query: (body) => ({ url: '/todos', method: 'POST', body }),
      invalidatesTags: [{ type: 'Todo', id: 'LIST' }],   // refetch list queries
    }),
    updateTodo: build.mutation<Todo, { id: string; changes: Partial<Todo> }>({
      query: ({ id, changes }) => ({ url: `/todos/${id}`, method: 'PATCH', body: changes }),
      invalidatesTags: (r, e, arg) => [{ type: 'Todo', id: arg.id }],  // just this one
    }),
  }),
});

export const { useGetTodosQuery, useAddTodoMutation, useUpdateTodoMutation } = api;

// Store wiring:
configureStore({
  reducer: { [api.reducerPath]: api.reducer, ... },
  middleware: (gDM) => gDM().concat(api.middleware),
});

Tag invalidation is the core model: queries provide tags; mutations invalidate tags; any active query providing an invalidated tag automatically refetches. The id-plus-LIST convention above gives you both granular ("this todo changed → refetch queries that provided it") and collection-level ("something was added → refetch the list") invalidation.

Cache lifetime: a cache entry is kept alive while any component holds a subscription to it (i.e. a mounted hook). When the last subscriber unmounts, the entry survives for keepUnusedDataFor seconds (default 60) before removal — RTK Query's analogue of TanStack's gcTime. Refetch-on-remount/focus behaviors exist but are opt-in (refetchOnMountOrArgChange, refetchOnFocus + setupListeners) — a real default-behavior difference from TanStack Query, which refetches stale data on those triggers by default.

Optimistic updates — onQueryStarted + patchResult

updateTodo: build.mutation<Todo, { id: string; changes: Partial<Todo> }>({
  query: ({ id, changes }) => ({ url: `/todos/${id}`, method: 'PATCH', body: changes }),
  async onQueryStarted({ id, changes }, { dispatch, queryFulfilled }) {
    // Patch the cached getTodos data immediately; keep the inverse patch:
    const patchResult = dispatch(
      api.util.updateQueryData('getTodos', undefined, (draft) => {
        const todo = draft.find(t => t.id === id);       // draft is an Immer draft
        if (todo) Object.assign(todo, changes);
      })
    );
    try {
      await queryFulfilled;         // server confirmed — keep the patch
    } catch {
      patchResult.undo();           // server rejected — roll back
      // Optionally: dispatch(api.util.invalidateTags([{ type: 'Todo', id }]))
      // to re-sync with server truth instead of trusting the rollback.
    }
  },
}),
The same onQueryStarted hook with await queryFulfilled before patching gives you "pessimistic" cache writes (update cache from the server response without a refetch). And onCacheEntryAdded is where streaming updates live — subscribe to a WebSocket while the cache entry exists, patch the entry on messages, tear down on cacheEntryRemoved.

5. TanStack Query — the server-cache king T1

TanStack Query (formerly React Query; v5 current) is the most widely deployed dedicated server-cache library and the deepest well of follow-up questions. Get the two clocks exactly right — it's the most commonly flubbed mechanic in this entire topic.

5.1 queryKey design — keys are identity AND dependencies

A query is identified by its queryKey (an array, hashed structurally — object key order doesn't matter). Two rules: (1) every variable the queryFn reads must be in the key — the key is the dependency array of data fetching; change the key and you address a different cache entry, which triggers a fetch for it. (2) Design keys hierarchically, generic → specific, because invalidation and cache matching use prefix matching: 

// Conventional key factory (avoids typo'd ad-hoc keys across the app):
export const todoKeys = {
  all:    ['todos'] as const,
  lists:  () => [...todoKeys.all, 'list'] as const,
  list:   (filters: Filters) => [...todoKeys.lists(), filters] as const,
  detail: (id: string) => [...todoKeys.all, 'detail', id] as const,
};

useQuery({ queryKey: todoKeys.list({ status: 'open', page }), queryFn: ... });

// Prefix matching in action:
queryClient.invalidateQueries({ queryKey: todoKeys.all });     // everything todo-ish
queryClient.invalidateQueries({ queryKey: todoKeys.lists() }); // all lists, no details

5.2 staleTime vs gcTime — the two clocks (exactly right)

staleTimegcTime (v4: cacheTime)

The narrative version: mount a query → data is served from cache if present (regardless of staleness — stale-while-revalidate) → if the data is stale, a background refetch fires. The refetch triggers — new observer mounts, window refocus, network reconnect, refetchInterval — all share the same guard: they only refetch stale queries. So staleTime is your one dial for "how chatty is this app": staleTime: 30_000 means "within 30s of a fetch, everyone reads cache and nobody hits the network." Set gcTime longer than staleTime where instant back-navigation matters; gcTime: 0 means "forget the moment nobody's looking" (rarely what you want).

The flub interviewers listen for: "cacheTime/gcTime is how long data is cached before refetching." No — gcTime never causes a refetch; it's garbage collection of inactive entries. Staleness (staleTime) decides refetching; gcTime decides memory. A query on screen all day is unaffected by gcTime. Second flub: "refetchOnWindowFocus spams my API" — it only refires for stale queries; the fix is a sensible staleTime, not refetchOnWindowFocus: false everywhere.

Dedup mechanics: all useQuery observers for the same key attach to one shared Query object in the QueryClient's cache. 5.5 Optimistic updates — the full onMutate/context/rollback ceremony

const toggleTodo = useMutation({
  mutationFn: (todo: Todo) => api.patch(`/todos/${todo.id}`, { done: !todo.done }),

  onMutate: async (todo) => {
    // 1. Cancel outgoing refetches so a slow response can't clobber our patch
    await queryClient.cancelQueries({ queryKey: todoKeys.lists() });

    // 2. Snapshot for rollback
    const previous = queryClient.getQueryData<Todo[]>(todoKeys.list(filters));

    // 3. Optimistically write the cache
    queryClient.setQueryData<Todo[]>(todoKeys.list(filters), (old) =>
      old?.map(t => t.id === todo.id ? { ...t, done: !t.done } : t)
    );

    // 4. Whatever we return becomes `context` for onError/onSettled
    return { previous };
  },

  onError: (_err, _todo, context) => {
    if (context?.previous) {
      queryClient.setQueryData(todoKeys.list(filters), context.previous); // rollback
    }
  },

  onSettled: () => {
    // 5. Success OR failure: re-sync with server truth
    queryClient.invalidateQueries({ queryKey: todoKeys.lists() });
  },
});
Narrate the why of each step when asked: cancelQueries prevents an in-flight refetch (started before the mutation) from resolving after your patch and overwriting it; the snapshot-in-context pattern exists because onError runs in a different closure than onMutate; the final invalidateQueries in onSettled means even a "successful" optimistic path converges to server truth (the server may have computed fields you didn't predict). Also mention the cheap alternative: v5 exposes mutation.variables while pending, so for simple UIs you can just render the pending item inline without touching the cache at all.

6. Zustand — the minimal-store pattern T1

Zustand is a ~1kB external store: a vanilla store (closure over state + a listener set, same shape as the Redux core you saw above) plus a React hook bound via useSyncExternalStore. No provider, no actions/reducers ceremony, no context — components subscribe with selectors. 

import { create } from 'zustand';

interface CartState {
  items: CartItem[];
  couponCode: string | null;
  addItem: (item: CartItem) => void;
  removeItem: (id: string) => void;
  total: () => number;
}

export const useCartStore = create<CartState>()((set, get) => ({
  items: [],
  couponCode: null,
  addItem: (item) =>
    set((state) => ({ items: [...state.items, item] })),   // set MERGES top-level keys
  removeItem: (id) =>
    set((state) => ({ items: state.items.filter(i => i.id !== id) })),
  total: () => get().items.reduce((s, i) => s + i.price * i.qty, 0),
}));

// Usage — the selector is the subscription:
const items   = useCartStore((s) => s.items);      // re-renders only when items ref changes
const addItem = useCartStore((s) => s.addItem);    // stable fn ref: never re-renders
const whole   = useCartStore();                    // ⚠️ no selector = re-render on ANY change

Mechanics worth stating precisely: set performs a shallow merge at the top level (you return only the keys that changed; nested objects must still be updated immutably — or add the immer middleware). Updates are immutable snapshots, so change detection is reference equality — same discipline as Redux, less ceremony. After each set, listeners run; each component hook re-runs its selector and re-renders only if the selected value changed (Object.is by default). For multi-field selections, use shallow comparison: 

import { useShallow } from 'zustand/react/shallow';   // v5 idiom

const { items, couponCode } = useCartStore(
  useShallow((s) => ({ items: s.items, couponCode: s.couponCode }))
);
// Without useShallow this object literal is a new ref every notify → re-render
// on every store change. (v4 legacy: useStore(selector, shallow) second arg.)

Slices pattern, persist, transient updates

// Slices: compose one store from focused creators (Redux-slice ergonomics, no combineReducers)
const createCartSlice  = (set, get) => ({ items: [], addItem: ... });
const createUserSlice  = (set, get) => ({ user: null, setUser: ... });
export const useBoundStore = create()((...a) => ({
  ...createCartSlice(...a),
  ...createUserSlice(...a),
}));

// persist middleware — localStorage cart that survives refresh:

  persist(
    (set) => ({ items: [], addItem: ... }),
    { name: 'cart', partialize: (s) => ({ items: s.items }) }  // persist data, not fns
  )
);

// Transient updates — read without rendering (cursors, canvas, 60fps data):
useEffect(() =>
  useCartStore.subscribe((state) => {
    badgeRef.current!.textContent = String(state.items.length);  // direct DOM write
  }), []);

// Works OUTSIDE React too — interceptors, analytics, game loops:
const token = useAuthStore.getState().token;

Why teams pick it — and what you give up

Pick it for: no provider wrapper (usable outside React — axios interceptors, event handlers, non-React widgets); a full API you can hold in your head (create, set, get, subscribe); render precision by default via selectors; ~1kB. It's the pragmatic middle for shared client state once server cache is extracted. You give up: Redux's enforced conventions (with set callable from anywhere, a large codebase needs self-discipline to keep updates in named store actions); devtools depth (a devtools middleware exists and integrates with Redux DevTools, but there's no serializable-action architecture, weaker time-travel semantics); and ecosystem weight — RTK's entity adapter, listener middleware, RTK Query, established patterns for 100-engineer codebases. Honest summary: Zustand trades guardrails for freedom; whether that's a win depends on team size and discipline.

Module-level store singletons (both Zustand's create at module scope and a Redux export const store) are fine in SPAs but a cross-request state leak on SSR servers — one Node process serves many users; user A's cart can hydrate into user B's HTML. The fix is per-request store creation (§11). Raising this unprompted when Zustand comes up is a strong senior signal.

7. Jotai and the atomic model T2

Zustand and Redux are top-down: one store object, components select slices. Jotai is bottom-up: state is a graph of atoms — independently subscribable units — and derived atoms that recompute from them. The mental model to say out loud: a spreadsheet. Primitive atoms are input cells; derived atoms are formula cells; change one input and exactly the dependent formulas recompute, and exactly the components reading affected cells re-render. Dependency tracking is automatic — whatever your read function gets becomes a dependency. 

import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

// Primitive atoms — config objects, not containers; state lives in the Provider/store
const priceAtom    = atom(100);
const quantityAtom = atom(2);
const taxRateAtom  = atom(0.08);

// Derived (read-only) atom — the "formula cell":
const totalAtom = atom((get) => get(priceAtom) * get(quantityAtom) * (1 + get(taxRateAtom)));

// Writable derived atom — read + custom write:
const discountedAtom = atom(
  (get) => get(priceAtom) * 0.9,
  (get, set, newPrice: number) => set(priceAtom, newPrice / 0.9)
);

function Total() {
  const total = useAtomValue(totalAtom);       // subscribes ONLY to totalAtom's graph
  return <span>{total.toFixed(2)}</span>;
}
function QtyInput() {
  const [qty, setQty] = useAtom(quantityAtom); // re-renders only on quantity changes
  ...
}
// Async derived atoms return promises and integrate with Suspense:
const userAtom = atom(async (get) => fetchUser(get(userIdAtom)));

When atomic wins: many independent interactive values with fine-grained dependencies — design tools (every shape's x/y/selected an atom), dashboards of independent widgets, complex filter panels where each control should re-render alone. In a single-store model you'd either write ever-more-precise selectors or eat wide re-renders; atoms give granularity structurally. When a single store wins: one cohesive domain updated by well-defined events (checkout flow, document editor with undo), where you want the whole state visible in one devtools tree and transitions centralized. Atoms are also naturally code-split — an atom ships with the lazy chunk that uses it, no central store file accreting every feature's keys.

Recoil is the origin of this model (Meta, 2020): atoms/selectors with keys, same spreadsheet idea. It never left "experimental," development stalled after Meta layoffs, and the repo was archived in early 2025 — the interview-relevant fact. Treat Recoil in a codebase as a migration candidate; Jotai is the maintained successor with a near-identical model (minus string keys). MobX literacy: the other classic paradigm — transparent reactivity. State is observable objects (Proxy-based); components wrapped in observer() track exactly which properties they dereference during render and re-render when precisely those mutate. You write straight-line mutable code (todo.done = true inside an action) and granular updates fall out automatically. Trade-offs: superb update precision and terse code, but implicit dependency graphs are harder to trace than an action log, and it fights the immutable-snapshot grain of the React ecosystem (uSES, concurrent-rendering assumptions). You'll meet it in older enterprise apps and OOP-leaning teams — respect it, don't trash it.

8. URL as state T2

The most under-used store in most apps is the address bar — and mentioning it unprompted is a reliable brownie point. State that describes what the user is looking at — filters, sort, page, active tab, selected item, search text — often belongs in the URL, because the URL gives you for free what no client store can: shareable links ("send me what you're seeing"), back/forward navigation as undo for view state, refresh-proofness, bookmarkability, and server-side renderability (the server sees the full view state in the request). 

// React Router (Next.js: useSearchParams + router.replace — same idea)
import { useSearchParams } from 'react-router-dom';

function ProductFilters() {
  const [searchParams, setSearchParams] = useSearchParams();

  const category = searchParams.get('category') ?? 'all';
  const page     = Number(searchParams.get('page') ?? '1');
  const inStock  = searchParams.get('inStock') === 'true';

  function updateFilter(key: string, value: string) {
    setSearchParams(prev => {
      const next = new URLSearchParams(prev);
      value ? next.set(key, value) : next.delete(key);
      if (key !== 'page') next.set('page', '1');   // filter change resets pagination
      return next;
    }, { replace: true });   // replace: don't spam history for every keystroke
  }

  // URL state composes perfectly with server cache — params ARE the queryKey:
  const products = useQuery({
    queryKey: ['products', { category, page, inStock }],
    queryFn: () => fetchProducts({ category, page, inStock }),
    placeholderData: keepPreviousData,   // keep old page visible while next loads
  });
}

Practical rules: URLs hold strings, so you own (de)serialization and validation — parse with defaults and clamp bad values (someone will type ?page=-1); a Zod schema at the parse boundary is the clean pattern, and libraries like nuqs package the typed-param pattern for Next.js/React Router. Use replace for high-frequency updates (typing) and push for discrete navigation-like changes (page, tab) so Back behaves sensibly. Keep URLs human-scale: filters and ids, yes; a 20-field wizard draft, no — that's session/local storage or a store. When URL beats a store: if losing the state on refresh would annoy the user, or if "share this view" is plausible, URL wins. When it doesn't: secrets/PII (URLs leak into logs, history, referrers), ephemeral UI (open dropdowns), and high-churn transient values.

Three worked scenarios — say these out loud

Scenario 1 — e-commerce cart. Classify: is the cart client state or server state? It depends on the business — that's the senior answer. Guest carts on a content-first store: client state — Zustand with persist (survives refresh, works before login, zero backend cost), items keyed by product id, totals as derived selectors. Logged-in carts that must sync across devices and survive to checkout: it's server state — the cart is a resource; use TanStack Query with optimistic addItem mutations (onMutate patch, rollback on error, invalidate on settle). Real systems often hybrid: local for guests, merge-to-server on login. The product detail page and pricing are unambiguously server cache either way.

Scenario 2 — collaborative analytics dashboard filters. Filters (date range, segments, breakdown, tab) describe what the user is viewing and "share this view with a teammate" is a core feature → URL search params are the store, parsed through a schema with defaults. The chart data is server cache keyed by those params — queryKey: ['metrics', parsedFilters] — with staleTime tuned to the data's real refresh cadence (analytics aggregated hourly ⇒ minutes of staleTime, not 0) and placeholderData: keepPreviousData so charts don't unmount between filter changes. If truly "collaborative" (live cursors/presence), that's a WebSocket feeding either the query cache (setQueryData) or a small Zustand store for ephemeral presence. Note what's absent: no Redux — nothing here is shared client state.

Scenario 3 — multi-step wizard (KYC/onboarding, 5 steps). Form state + flow state. Field values: react-hook-form, one form per step or one form with step-scoped validation via a Zod resolver. Flow position: current step in the URL (/onboarding/step-3 or ?step=3) so refresh and Back work — with a route guard redirecting to the first incomplete step (deep-linking to step 4 without steps 1–3 is an invalid state). Accumulated cross-step draft: a small wizard-scoped store (Context+useReducer or Zustand), persisted to sessionStorage so an accidental refresh doesn't wipe ten minutes of typing; alternatively persist each completed step to the server as a draft resource — then it's server state and resumable cross-device. If transitions have real rules (retry limits, conditional branches, doc-verification pending states), model the flow as an explicit state machine (§10). No global store: the wizard's state should die when the wizard does.

Close the framework with the meta-point interviewers actually score: "The library choice is the least important decision. What matters is classifying each piece of state correctly, keeping it as local as possible, and being consistent per category. I'd rather inherit a well-classified Context app than a badly-classified Redux one." Also name the socio-technical factors — team familiarity, existing codebase conventions, hiring pool, devtools/audit requirements — as first-class inputs, not afterthoughts.

10. State machines — XState literacy & the reducer-lite pattern T2

Boolean-flag soup is the disease: isLoading, isError, isSuccess, isRetrying — four booleans is sixteen combinations, of which maybe five are meaningful, and nothing stops isLoading && isError from happening. A state machine says: the system is in exactly one named state; only declared events cause transitions; undeclared events in a given state are ignored. Impossible states become unrepresentable rather than merely untested. 

// XState v5 — payment step of a wizard
import { setup, assign } from 'xstate';

const paymentMachine = setup({
  guards: { hasRetriesLeft: ({ context }) => context.retries < 3 },
}).createMachine({
  id: 'payment',
  initial: 'idle',
  context: { retries: 0, error: null },
  states: {
    idle:       { on: { SUBMIT: 'processing' } },
    processing: { on: { SUCCESS: 'succeeded',
                        FAILURE: [
                          { target: 'retrying', guard: 'hasRetriesLeft',
                            actions: assign({ retries: ({ context }) => context.retries + 1 }) },
                          { target: 'failed' },     // guard false → fall through
                        ] } },
    retrying:   { after: { 2000: 'processing' } },  // delayed transition
    succeeded:  { type: 'final' },
    failed:     { on: { RESET: 'idle' } },
  },
});
// Note what's structurally impossible now: double-submit (SUBMIT isn't handled
// in 'processing'), a 4th retry, success-after-failure. The machine is also a
// diagram — XState's visualizer renders it for PM/design review.

When to reach for it: multi-step flows with rules (checkout, onboarding, doc upload pipelines), media players, drag-and-drop, connection lifecycles — anywhere you catch yourself writing if (isX && !isY && wasZ). The honest caveat: XState is a paradigm buy-in with real learning cost; for most cases the "lite" version — a discriminated union + useReducer — captures 80% of the value with zero dependencies: 

type State =
  | { status: 'idle' }
  | { status: 'processing' }
  | { status: 'succeeded'; receiptId: string }
  | { status: 'failed'; error: string; retries: number };
// TypeScript enforces it: state.receiptId only exists when status === 'succeeded'.

function reducer(state: State, event: Event): State {
  switch (state.status) {                    // key move: switch on STATE first,
    case 'idle':                             // then event — illegal transitions
      return event.type === 'SUBMIT'         // simply return `state` unchanged
        ? { status: 'processing' } : state;
    case 'processing':
      switch (event.type) {
        case 'SUCCESS': return { status: 'succeeded', receiptId: event.receiptId };
        case 'FAILURE': return { status: 'failed', error: event.error, retries: 0 };
        default: return state;               // SUBMIT while processing: ignored. Free!
      }
    ...
  }
}

11. RSC-era state management T3

React Server Components rearrange the taxonomy's biggest category. Server cache largely moves to the server: server components await data directly during render and ship HTML/serialized trees — for read-only views there is no client fetch, no loading spinner, no client cache entry at all. Client stores shrink to the interactive islands: 'use client' components that genuinely need stateful interactivity (cart widget, live filters, editors). The practical outcome: less global client state than ever, with mutations flowing through server actions + revalidate/router refresh, or through a client query cache where you need optimistic UX and background sync.

The per-request store gotcha

// ❌ SPA habit that becomes a bug on the server:
export const store = createStore();          // module scope = ONE store per Node
// process, shared across ALL requests. User A's auth slice can render into
// user B's HTML. Module state also persists across requests (leaks accumulate).

// ✅ Factory + provider: one store per request tree, created once per client mount
'use client';
export function StoreProvider({ children, initialState }) {
  const storeRef = useRef<AppStore | null>(null);
  if (!storeRef.current) {
    storeRef.current = makeStore(initialState);   // makeStore = () => configureStore(...)
  }
  return <Provider store={storeRef.current}>{children}</Provider>;
}
// Same pattern for Zustand: createStore (vanilla) in a ref, expose via a
// context, read with useStore(ctx, selector) — the documented Next.js recipe.
// Rule of thumb: on the server, NOTHING user-scoped may live at module scope.

      <TodoListClient />

Why bother with a client cache when RSC can fetch? Because RSC data is read-time: refreshing it means a server round-trip re-render. If the view needs client-side interactivity on that data — optimistic mutations, polling, focus refetch, infinite scroll — hydrate-into-Query gives server-rendered first paint plus full client cache semantics after. Views without those needs should stay pure RSC. That per-view judgment call is the T3 talking point.

Interview Q&A

Q1. "How do you decide where a piece of state should live?"

Classify first: server cache, shared client state, local UI state, URL state, or form state — each category has a best-fit tool. Server data goes to a server-cache library (TanStack Query, or RTK Query if Redux is already there); view-describing state that should survive refresh and be shareable goes to the URL; forms go to a form library; and for client state I apply colocation — useState in the component, lift to a common parent when a second consumer appears, Context for low-frequency subtree values, and an external store (Zustand/RTK) only for genuinely app-wide, high-frequency client state. The most common architecture mistake I've seen is misclassifying server cache as client state — that's what filled 2016-era Redux stores with loading/error boilerplate.

Q2. "Why is 'server cache' not just client state with extra steps?"

Because the client doesn't own it. Remote data can be changed by others while displayed, so it has properties client state never has: staleness, background revalidation, request deduplication, retry, and mutation reconciliation (invalidate or optimistically patch, then converge to server truth). Those are cache semantics, not state semantics. Modeling it as client state forces you to hand-write status flags, dedup, and invalidation per endpoint — badly. A dedicated cache layer gives stale-while-revalidate, focus/reconnect refetch, and in-flight dedup as defaults.

Q3. "Is React Context a state management tool?"

Context is transport, not state management — it delivers a value to a subtree without prop drilling, but has no store semantics: no selectors, no subscription granularity. Every consumer re-renders when the provider's value reference changes, regardless of which field it reads. So it's right for low-write-frequency app-wide values — theme, auth, locale, injected services — and wrong for per-keystroke data. State still lives in useState/useReducer above the provider; Context just moves it. Notably, react-redux and Jotai use Context only to pass the store/scope reference down — subscriptions happen outside Context, via useSyncExternalStore.

Q4. "Walk me through what happens inside Redux when you dispatch an action."

The store is a closure over state and a listener array. dispatch(action) passes the action through the middleware chain (each middleware is store => next => action, so thunk can intercept functions); the root reducer runs synchronously — combineReducers delegates each slice, and unchanged slices return their existing references; the returned tree replaces state; then all listeners are notified. On the React side, useSelector (built on useSyncExternalStore since react-redux v8) re-runs its selector against the new state and re-renders only if the result isn't reference-equal to the last one. uSES also protects against tearing under concurrent rendering — if the store changes mid-render, React falls back to a synchronous re-render rather than committing inconsistent reads.

Q5. "Why does Redux insist on immutability and serializable actions?"

Immutability makes change detection a reference comparison — react-redux and Reselect can decide "did my slice change" with === instead of deep equality — and it preserves old state snapshots, which is what makes time-travel debugging possible. Serializable actions make the action log recordable, replayable, diffable, and shippable in a bug report. Both constraints buy the same thing: predictability and debuggability as architectural guarantees. Drop either and you keep Redux's ceremony while losing its actual payoff — which is why RTK ships dev-mode middleware that throws on accidental mutation and non-serializable payloads.

Q6. "createSlice lets you 'mutate' state. How is that not breaking Redux's rules?"

Immer. createSlice wraps each case reducer in produce: your "state" parameter is a Proxy draft that traps reads and writes. Writes trigger copy-on-write — the touched node is cloned once, the change is recorded — and on return Immer finalizes a new immutable tree where changed nodes and their ancestors are new objects while every untouched branch keeps its original reference (structural sharing). So downstream reference-equality checks — useSelector, createSelector, React.memo — behave exactly as if I'd hand-written the nested spreads. The rules to remember: mutate the draft or return a new value, never both in one path; and console.log of a draft prints a Proxy — use current(state).

Q7. "When does useSelector cause a re-render, and how do you prevent wasted ones?"

After every dispatch, each useSelector re-runs its selector and re-renders the component iff the result isn't reference-equal (===) to the previous result. So selectors that construct fresh values — .map(), .filter(), object literals — re-render on every action. Fixes, in order: select primitives or stored references; pass shallowEqual for multi-field object selections; memoize derivations with createSelector, which reuses the cached result (same reference) while its input selectors return reference-equal values; and for parameterized selectors shared across many component instances, a per-instance selector factory (or Reselect 5's weakMapMemoize), because default createSelector has a cache size of one.

Q8. "Why normalize Redux state, and what does createEntityAdapter give you?"

Normalization stores entities like database tables: { ids: [], entities: { [id]: entity } }. Three wins: O(1) lookup by id instead of array scans; no deep nested updates — every entity is one flat record; and a single source of truth per entity, so an author embedded in fifty posts is one record updated once. createEntityAdapter generates the CRUD reducers (addOne, updateOne, upsertMany…), optional sortComparer-maintained ordering, and memoized selectors (selectAll, selectById, selectIds). The performance pattern it enables: a list subscribes to selectIds (stable when only fields change) while each memoized row subscribes to selectById — editing one entity re-renders exactly one row.

Q9. "You inherit a large redux-saga codebase. What's your take?"

First, read it fluently: sagas are generators yielding effect descriptions (call/put/take/fork) that the middleware interprets, which makes them testable by stepping the generator, and operators like takeLatest give elegant cancellation. Then triage: in my experience most saga code is "fetch on action" — that migrates endpoint-by-endpoint to RTK Query, which deletes the fetch sagas and their loading/error state. Residual reactive logic ("when X is dispatched, do Y") moves to RTK listener middleware, which covers debounce/cancel/condition without the generator runtime. Genuinely complex long-lived orchestrations stay as sagas until there's a concrete reason to touch them — migration is incremental, per-feature, never big-bang.

Q10. "Explain staleTime vs gcTime precisely."

Two independent clocks. staleTime starts when a fetch resolves and answers "how long is this data fresh?" While fresh, any component using the key is served from cache with no network request, and refetch triggers (mount, window focus, reconnect, interval) do nothing. Once stale, cached data is still shown instantly, but those triggers fire background refetches — stale-while-revalidate. Default 0. gcTime starts when the query becomes inactive — last observer unmounts — and answers "how long does the unused entry stay in memory?" Default 5 minutes. Crucially, gcTime never causes a refetch; it only decides whether the next mount finds instant cached data or a cold loading state. staleTime controls network chatter; gcTime controls memory and back-navigation UX.

Q11. "How does TanStack Query avoid duplicate requests when five components need the same data?"

All observers of a queryKey attach to a single shared Query object in the QueryClient's cache. When a fetch for that key is in flight, any newly mounted observer joins the existing promise instead of firing another request — so five components calling useQuery(['user', id]) in one render pass produce one network call, and all five receive the result. After that, staleTime governs: within it, further mounts read cache silently; past it, a mount triggers one shared background refetch. This is the strongest practical argument for "hooks that fetch" over prop drilling data down — dedup makes colocated data access free.

Q12. "Implement an optimistic update. Why each step?"

In onMutate: (1) await queryClient.cancelQueries(key) — kills in-flight refetches that started before the mutation, which could otherwise resolve after my patch and overwrite it with pre-mutation data; (2) snapshot getQueryData(key); (3) setQueryData with the predicted result; (4) return { previous } — this becomes the context argument, because onError runs in a different closure. In onError: restore the snapshot. In onSettled: invalidateQueries(key) regardless of outcome, so even the success path converges to server truth (server-computed fields, concurrent edits). I'd also mention when not to bother: for a simple pending row, v5's mutation.variables renders the in-flight item without touching the cache.

Q13. "TanStack Query or RTK Query — how do you choose?"

Same architecture — keyed async cache with stale-while-revalidate, invalidation, optimistic updates — so it's an ecosystem decision. Already invested in Redux? RTK Query: cache in the same store, one devtools, one mental model, declarative tag invalidation, endpoints optionally codegen'd from OpenAPI. No Redux? TanStack Query standalone: no reason to adopt a Redux store just to host a cache, and it brings the deeper query feature set — mature infinite queries, Suspense hooks, persistence plugins, dedicated devtools. One behavioral delta worth knowing: TanStack refetches stale data on focus/reconnect/mount by default; RTK Query's equivalents are opt-in. The anti-pattern is running both in one app without a boundary.

Q14. "How does Zustand work under the hood, and why no Provider?"

create builds a vanilla store — a closure over a state object and a Set of listeners, with getState/setState/subscribe (structurally the same core as Redux's store) — and returns a hook bound to it via useSyncExternalStore. The hook takes a selector; on each notify it re-runs the selector and re-renders only if the selected value changed (Object.is, or shallow via useShallow). No Provider is needed because the store is a plain module-scope object, not a React context value — which is also why it works outside React (useStore.getState() in an axios interceptor). The trade-off of that convenience: module-scope singletons are a cross-request leak on SSR, so server-rendered apps use a per-request store factory passed through context instead.

Q15. "Your team wants to replace Redux with Zustand. Steelman both sides."

For Zustand: most of the Redux store is probably misclassified server cache — extract that to a query library first, and the remaining client state rarely justifies Redux's ceremony; Zustand gives selector-precise subscriptions, no provider, ~1kB, and an API a new hire learns in an hour. For Redux: RTK already fixed the boilerplate; the enforced convention — every change is a named, logged, serializable action through a pure reducer — is a governance feature that scales with team size, and devtools time-travel plus the RTK ecosystem (entity adapter, listener middleware, RTK Query) are real assets. My actual position: the migration that matters is server-cache extraction; after that, whether the residue lives in RTK slices or a Zustand store is a minor decision — I'd optimize for team familiarity and not rewrite working code without a driver.

Q16. "What problem does the atomic model (Jotai) solve that single-store libraries don't?"

Update granularity as a structural property instead of a selector discipline. In a single store, every dispatch notifies every subscriber, and precision depends on writing good selectors. With atoms, state is a dependency graph — components subscribe to specific atoms, derived atoms track exactly what they get — so changing one atom recomputes and re-renders only its dependents, like a spreadsheet recalculating affected formula cells. It wins for many independent fine-grained values: design canvases, per-widget dashboards, complex filter panels. A single store still wins for one cohesive domain with centralized transitions, whole-tree devtools visibility, and undo. I'd also note Recoil, the model's origin, is archived — Jotai is the maintained implementation.

Q17. "When should application state live in the URL?"

When the state describes what the user is viewing — filters, sort, page, tab, selected id — and shareability or refresh-survival has value. The URL delivers three things no client store can: shareable links, back/forward as undo for view state, and refresh-proofness; it also makes the view server-renderable since the request carries full view state. Implementation notes: URLs are strings, so parse with validation and defaults (a Zod schema at the boundary, or a lib like nuqs); use replace for high-frequency updates so history isn't spammed; reset page on filter change; and the params flow straight into the queryKey, so URL state and server cache compose naturally. Exclusions: secrets/PII (URLs leak into logs and referrers) and ephemeral UI state.

Q18. "What's a state machine buying you over booleans, and when is XState overkill?"

Booleans multiply: four flags is sixteen combinations, most meaningless, all representable — isLoading && isError can happen and must be defensively coded around. A machine inverts it: exactly one named state, transitions only via declared events, undeclared events ignored — impossible states become unrepresentable, double-submit dies structurally because SUBMIT isn't handled in processing, and the machine doubles as a reviewable diagram. XState adds guards, delayed transitions, nested/parallel states, and actor orchestration — worth it for genuinely complex flows (payments, media players, connection lifecycles). For everything else, the lite version suffices: a discriminated union ({ status: 'succeeded'; receiptId }) with useReducer, switching on state first then event, which gets exhaustiveness-checked by TypeScript at zero dependency cost.

Q19. "How do React Server Components change your state architecture?"

They relocate the biggest category. Server cache moves server-side: server components await data during render, so read-only views need no client fetch, no spinner, no cache entry. Client stores shrink to interactive islands — the 'use client' components that genuinely need state. Mutations can flow through server actions plus revalidation instead of client-side invalidation. Two disciplines follow: no user-scoped module-level store singletons on the server — per-request store factories held in a ref inside a client provider, or you leak one user's state into another's HTML — and for views that need client cache semantics (optimistic updates, polling, infinite scroll), prefetch server-side and hydrate into TanStack Query via dehydrate/HydrationBoundary so you get server-rendered first paint and live cache behavior after.

Q20. "A dashboard re-renders everything on every keystroke in a search box. Diagnose."

The input's state is too global — likely in a context whose provider value is recreated per render (invalidating all consumers), or a store selected without precision (useSelector returning fresh objects, or a store hook with no selector). Fix by classification and colocation: the raw keystroke value is local UI state — useState in the search box; only the debounced term should escape, ideally into the URL (?q=, with replace) feeding a queryKey. If a store must hold it, subscribers select narrowly (primitives, useShallow/shallowEqual, memoized selectors), and non-visual consumers use transient subscriptions (store.subscribe) that skip rendering entirely. The pattern-level answer: high-frequency state demands either locality or selector-precise subscriptions — never broadcast context.

Q21. "How would you test state logic across these tools?"

Push logic into pure, React-free units and test those directly: Redux reducers and Reselect selectors are plain functions (expect(reducer(state, action)).toEqual(...)); Zustand stores test vanilla — store.getState().addItem(x) then assert, resetting initial state between tests; state machines test as transition tables. For server-cache behavior, test through the component with a mocked network (MSW) and a fresh QueryClient per test with retry: false — you're testing your keys/invalidation wiring and UI states, not the library. Integration tests then cover the seams: "mutation settles → list refetches → UI shows the new row." The design takeaway: the more state logic lives in pure functions rather than effects and handlers, the cheaper the whole pyramid gets.

Q22. "What's your take on putting form state in a global store?"

Almost always wrong. Form state is high-frequency (per keystroke), lifecycle-bound (should die with the form), and rich in metadata (dirty, touched, errors, submit status) that a generic store models poorly — and global stores make every keystroke a global event, with devtools noise and re-render risk to match. react-hook-form is the default: uncontrolled inputs via register keep keystrokes out of React state entirely (refs + native events), with isolated re-renders for subscribed fields and schema validation via resolvers. The exceptions where form data legitimately escapes: multi-step wizard drafts (session-persisted wizard store, or server-persisted draft resource) and autosave (debounced mutation — at which point it's becoming server state anyway).

Q23. "How do tags-based invalidation and key-based invalidation differ?"

RTK Query is declarative: queries provide tags ({ type: 'Todo', id } plus a LIST sentinel), mutations invalidate tags, and the middleware refetches any active query whose provided tags intersect — the relationship between writes and reads is data, defined once at the endpoint. TanStack Query is structural: you invalidate by key prefix (invalidateQueries(['todos']) hits lists and details), so precision comes from hierarchical key design and the discipline lives at mutation call-sites (or centralized via mutation defaults/callbacks). Tags shine when many mutations touch overlapping data — the mapping is self-documenting; keys shine in flexibility and no upfront registry. Both converge on the same principle: after a write, mark affected reads stale and let the cache refetch — never manually patch everything and hope.

Q24. "Design the state layer for a Trello-style board. Go."

Classify each piece. Board/lists/cards: server cache — TanStack Query, keys like ['board', id] with normalized-ish structure; card moves are optimistic mutations (onMutate reorders locally, rollback on error, invalidate on settle) because drag-drop cannot wait on a round trip. In-flight drag state (what's dragged, hover position): high-frequency ephemeral client state — local state or a tiny Zustand store with transient subscriptions, never the query cache. Current board/card-modal: URL (/board/123?card=456) — shareable deep links are core to the product. Card-edit form: react-hook-form in the modal. Live collaboration: WebSocket events patch the query cache via setQueryData (or targeted invalidation on conflict). Filters: URL params. Note the absence of a big global store — after classification there's almost no shared client state left, and I'd say exactly that out loud.

Q25. "What is tearing, and how do external stores avoid it?"

Tearing is two components rendering values from different versions of the same store within one committed UI — impossible when rendering was synchronous, possible once concurrent React can pause mid-render: the store mutates during the pause, and components rendered after the pause read newer state than those before. useSyncExternalStore is the contract that fixes it: the store supplies subscribe and getSnapshot; React checks snapshot consistency at the end of the render pass, and if the store changed mid-render, it discards and re-renders synchronously rather than committing a torn frame. This is why react-redux v8+, Zustand, and TanStack Query all sit on uSES, and why immutable snapshots matter — getSnapshot must return a stable, cheaply-comparable value. Trade-off worth naming: uSES updates are de-facto synchronous, so external-store updates don't get the time-slicing benefits of useState-based transitions.

Exercise 2 — Build Zustand's core, then bind it to React

Implement createStore(initialState) returning { getState, setState, subscribe } where setState accepts a partial or an updater fn and shallow-merges. Then write useStore(store, selector) using useSyncExternalStore so components re-render only when their selected value changes. Prove it with two components selecting different keys and a log of renders.

Solution 
function createStore(initialState) {
  let state = initialState;
  const listeners = new Set();
  return {
    getState: () => state,
    setState: (partial) => {
      const next = typeof partial === 'function' ? partial(state) : partial;
      if (next !== state) {
        state = { ...state, ...next };          // shallow merge, new reference
        listeners.forEach((l) => l());
      }
    },
    subscribe: (l) => { listeners.add(l); return () => listeners.delete(l); },
  };
}

function useStore(store, selector) {
  return useSyncExternalStore(
    store.subscribe,
    () => selector(store.getState()),   // snapshot = selected value
    () => selector(store.getState())    // server snapshot (SSR)
  );
}
// Key insight to verify: uSES re-renders when the SNAPSHOT changes by Object.is.
// Because selector runs inside getSnapshot, selecting s.count gives a primitive —
// updating s.name notifies listeners, but count's snapshot is unchanged → no
// re-render. Also see the caveat: a selector returning a fresh object each call
// would loop/thrash — exactly why real Zustand offers useShallow.

Exercise 3 — Reselect mechanics by hand

Implement createSelector(inputSelectors, resultFn) with cache size 1: if every input selector returns a reference-equal value vs last call, return the cached result. Then construct a Redux-like scenario proving (a) an unrelated dispatch does NOT recompute, (b) a relevant change does, and (c) the memoized result keeps reference identity so a === check (as useSelector performs) skips the re-render.

Solution 
function createSelector(inputs, resultFn) {
  let lastArgs = null;
  let lastResult;
  return (state, ...params) => {
    const args = inputs.map((sel) => sel(state, ...params));
    const same = lastArgs && args.length === lastArgs.length &&
                 args.every((a, i) => Object.is(a, lastArgs[i]));
    if (!same) {
      lastResult = resultFn(...args);
      lastArgs = args;
    }
    return lastResult;                 // cached ⇒ SAME reference ⇒ useSelector skips
  };
}

// Proof harness:
const selectTodos  = (s) => s.todos;
const selectOpen   = createSelector([selectTodos],
                       (todos) => todos.filter(t => !t.done));
const s1 = { todos: [{ id: 1, done: false }], theme: 'dark' };
const r1 = selectOpen(s1);
const s2 = { ...s1, theme: 'light' };     // unrelated change, todos same ref
console.assert(selectOpen(s2) === r1);    // (a)+(c): no recompute, same ref
const s3 = { ...s2, todos: [...s2.todos, { id: 2, done: false }] };
console.assert(selectOpen(s3) !== r1);    // (b): recomputed

// watch cache-size-1 thrash — motivating the selector-factory pattern.

Exercise 4 — Optimistic toggle with rollback (TanStack Query)

Against a mock API (e.g. MSW or a delayed in-memory server) where PATCH /todos/:id fails 30% of the time, build a todo list with optimistic toggle: instant checkbox flip, rollback + toast on failure, convergence via invalidation on settle. Add the detail most people miss: verify a slow, pre-mutation refetch can't overwrite your optimistic state (hint: what does cancelQueries exist for?).

Solution sketch 
useMutation({
  mutationFn: (t) => api.patch(`/todos/${t.id}`, { done: !t.done }),
  onMutate: async (t) => {
    await qc.cancelQueries({ queryKey: ['todos'] });          // the missed detail
    const previous = qc.getQueryData(['todos']);
    qc.setQueryData(['todos'], (old) =>
      old.map(x => x.id === t.id ? { ...x, done: !x.done } : x));
    return { previous };
  },
  onError: (_e, _t, ctx) => { qc.setQueryData(['todos'], ctx.previous); toast('Failed'); },
  onSettled: () => qc.invalidateQueries({ queryKey: ['todos'] }),
});

// window (starts a background refetch of pre-toggle data), toggle within 3s.
// Without cancelQueries the stale GET resolves last and un-toggles your checkbox.
// Grading checklist: snapshot returned via context (not a closure variable),
// rollback restores it, onSettled invalidates on BOTH outcomes, and rapid
// double-toggles still converge (invalidation guarantees eventual server truth).

Exercise 5 — Wizard as a lite state machine + URL

Build a 3-step signup wizard (account → profile → confirm) where: the current step is a discriminated-union state driven by useReducer; the step is mirrored in the URL (?step=) with deep-link guarding (landing on step 3 without completing 1–2 redirects to the first incomplete step); accumulated answers persist to sessionStorage so refresh resumes; and SUBMIT during submission is structurally ignored.

Solution sketch 
type WizardState =
  | { step: 'account' }
  | { step: 'profile';  account: AccountData }
  | { step: 'confirm';  account: AccountData; profile: ProfileData }
  | { step: 'submitting'; account: AccountData; profile: ProfileData }
  | { step: 'done' };
// The union IS the guard: you cannot construct 'confirm' without both payloads.

function reducer(state: WizardState, e: Event): WizardState {
  switch (state.step) {
    case 'account':
      return e.type === 'ACCOUNT_DONE' ? { step: 'profile', account: e.data } : state;
    case 'profile':
      if (e.type === 'PROFILE_DONE') return { ...state, step: 'confirm', profile: e.data };
      if (e.type === 'BACK') return { step: 'account' };
      return state;
    case 'confirm':
      return e.type === 'SUBMIT' ? { ...state, step: 'submitting' } : state;
    case 'submitting':
      return e.type === 'SUCCESS' ? { step: 'done' } : state;  // SUBMIT ignored: free
    case 'done': return state;
  }
}

// sessionStorage draft; clamp deeper requests back (the redirect guard).

// useReducer from it. Field editing inside each step: react-hook-form; only
// validated step data enters the wizard state — two layers, cleanly separated.
