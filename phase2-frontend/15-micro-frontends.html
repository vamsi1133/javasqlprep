15 — Micro-Frontends

An organizational architecture wearing a technical costume — when to split, how to compose, and what it really costs.

Why interviewers ask this

Micro-frontends (MFE) is on senior job descriptions everywhere, and it's one of the highest-signal architecture questions an interviewer can ask — because it separates people who can configure Module Federation from people who understand why the architecture exists, what it costs, and when it's the wrong call. The failure mode interviewers are screening against is real: teams of eight engineers adopting MFE because a conference talk made it sound modern, then drowning in duplicated React copies, drifting design systems, and un-debuggable cross-app state.

At FAANG scale, the question is usually a system-design prompt: "we have 12 teams shipping into one dashboard — design the frontend architecture." At product startups it's often the inverse trap: "should we adopt micro-frontends?" where the expected senior answer is probably not, and here's the framework for deciding. Both are testing the same muscle: can you connect team topology to architecture (Conway's law), enumerate honest costs, and defend a default of simplicity? This tutorial builds that judgment first, then goes deep on the mechanics — because once you've correctly decided to split, the mechanics (shared singletons, remoteEntry flow, routing ownership, contract testing) are where senior candidates get probed line by line.

1. Why MFE — and why usually not T1

Start every MFE conversation with this sentence, because it reframes the entire discussion: micro-frontends solve an organizational problem, not a technical one. Nothing about splitting a frontend into independently deployed pieces makes it faster, smaller, or more reliable — in fact it usually makes all three worse. What it buys you is the ability for many teams to ship into one product surface without coordinating deploys.

The actual problem being solved

When one frontend codebase is shared by many teams, three organizational pains compound:

  • Deploy contention. Team A's release is blocked because Team B's flaky test is red on main. Release trains form; lead time balloons; "who broke the build" becomes a full-time conversation. This is the single most common trigger.
  • Coupled tech evolution. Upgrading React, the router, or the build tool requires every team to move at once. The slowest team sets the pace for everyone. MFE lets one team run React 19 while another is still finishing its React 17 migration.
  • Ownership ambiguity. Conway's law says your architecture will mirror your communication structure whether you like it or not. If five teams share one app with no enforced boundaries, you get five teams' worth of conventions smeared across one codebase. MFE makes team boundaries physical: each team owns build, deploy, on-call, and roadmap for its slice.
  THE PROBLEM MFE SOLVES (organizational, not technical)

The honest cost list

A senior answer names the costs unprompted. Every one of these is a recurring tax, not a one-time setup fee:

CostWhat it looks like in practice
Bundle duplication Without disciplined dependency sharing, every MFE ships its own React, router, date library, design system. Users pay in bytes and parse time; two React copies on one page breaks hooks and context outright (§3).
UX consistency drift Independent teams + independent release cadences = buttons that look 90% the same, three subtly different modals, inconsistent spacing. Requires an actively governed design system to counteract — which is itself a team's worth of ongoing work.
Cross-MFE state pain Anything two MFEs both need — auth, cart, feature flags, locale — now crosses a deployment boundary. Every shared value becomes a versioned contract between separately deployed artifacts.
Operational complexity N pipelines, N monitoring dashboards, N rollback procedures, and failure modes that only appear in specific combinations of deployed versions ("checkout 4.2 + shell 1.9 breaks, neither is broken alone").
Local dev orchestration "Run the app locally" now means running 2–8 servers or pointing your local remote at staging versions of everything else. Onboarding time and dev-loop friction go up measurably.
Testing across seams Unit tests stay easy; the integration surface between MFEs is where bugs hide, and no single team's suite covers it. You need contract tests and a centrally owned e2e critical path (§6).
The tell of a senior engineer in this conversation: they describe MFE as buying deployment independence with user-facing and operational currency, and they can say exactly when that trade is worth it. The tell of a mid-level engineer: they lead with Module Federation config and treat the costs as footnotes.

The senior default: modular monolith first

The default answer for most teams is a modular monolith in a monorepo with enforced boundaries: one deployable artifact, but internal packages with explicit public APIs, import rules enforced by tooling (Nx module boundaries, ESLint no-restricted-imports, Turborepo package graphs), and per-package code ownership. You get 80% of the organizational benefit — clear ownership, independent code review, parallel work — with none of the runtime composition tax. Crucially, a well-boundaried monolith is MFE-ready: if team growth later forces the split, the seams already exist. 

  THE MIGRATION LADDER (climb only as far as the org forces you)

  1. Monolith            one repo, tangled imports        ← where most start
  2. Modular monolith    one deploy, enforced package     ← the senior default
                         boundaries + code ownership

Decision framework — the triggers that justify the split

  • Team count and contention: roughly 3–4+ frontend teams with measurable deploy contention (release trains, merge queues backing up, teams blocking each other weekly). Below that, the coordination cost of MFE exceeds the coordination cost it removes.
  • Genuinely independent release cadences: e.g., a regulated payments team that ships monthly with compliance review, next to a growth team shipping ten times a day. Forcing them through one pipeline hurts both.
  • Acquisitions / legacy integration: you bought a company; its Vue app must appear inside your React product next quarter. Runtime composition is often the only realistic option.
  • Gradual migration — the strangler fig: often the strongest justification. You have a legacy app (jQuery, AngularJS, old Rails views) that cannot be rewritten big-bang. An app shell + MFE seams let you replace it route by route, shipping value continuously, with the legacy app itself mounted as just another "remote" until it withers away (§8 has the full story).
"We want micro-frontends for performance" is a red flag answer. MFE almost always costs performance (duplication, extra network hops, runtime composition overhead). If someone cites performance as the motivation, they've confused MFE with code-splitting — which a monolith does fine with import() and route-level chunks.
A useful interview litmus test to cite: "could two of these teams realistically want to deploy at different times and be blocked by each other today?" If the answer is no — because it's one team, or deploys aren't contended — MFE is solving a problem you don't have.

2. Composition approaches — the taxonomy T1

"Micro-frontends" is a family of composition techniques, not one technology. Interviewers expect you to lay out the option space before advocating for one. The two axes that matter: when composition happens (build time → server → client runtime) and what granularity teams own (whole routes vs fragments of a page).

ApproachComposedIsolationDeploy independenceBest for
Build-time (npm packages)At buildNone (one bundle)✗ — host must rebuild/redeploySmall orgs; shared UI libs; "not really MFE" but often correct
Client-side runtime (Module Federation, import maps)In browserJS-level only✓ fullSPA-heavy products, many teams, shared framework
Server-side composition (SSI/ESI, BFF-assembled)On server/edgePer-fragment✓ fullContent/commerce sites where SSR + SEO dominate

2.1 Build-time composition (npm packages)

  BUILD-TIME COMPOSITION

  team-a/  ──publish──▶  @org/feature-a  ──┐
  team-b/  ──publish──▶  @org/feature-b  ──┤──▶  host app: npm install,
  team-c/  ──publish──▶  @org/feature-c  ──┘     ONE build, ONE deploy
                                                 ONE bundle graph (deduped!)
  ✓ perfect dedupe, typechecked seams, trivial local dev
  ✗ shipping feature-a's fix still requires a HOST release

Publishing features as npm packages and composing them in one host build is not micro-frontends by the strict definition — deploys are still coupled through the host. But say this in the interview: it's frequently the right answer anyway. You get versioned contracts, full type safety across seams, and perfect dependency dedupe. Its real weakness is release latency: a package fix reaches users only when the host rebuilds and redeploys — which reintroduces exactly the contention MFE exists to remove. So it fits orgs where deploy contention isn't (yet) the bottleneck.

2.2 Client-side runtime composition

  CLIENT-SIDE RUNTIME COMPOSITION (Module Federation / import maps)

  Browser
  ┌───────────────────────────────────────────────────┐
  │  App Shell (host)  ← loaded from shell.cdn.com    │
  │  ┌─────────────────────────────────────────────┐  │
  │  │ 1. fetch remoteEntry.js from each remote    │  │
  │  │ 2. negotiate shared deps (one React!)       │  │
  │  │ 3. dynamically import exposed modules       │  │
  │  └─────────────────────────────────────────────┘  │
  │   ┌───────────────┐   ┌────────────────────────┐  │
  │   │ checkout MFE  │   │ catalog MFE            │  │
  │   │ checkout.cdn… │   │ catalog.cdn…           │  │
  │   └───────────────┘   └────────────────────────┘  │
  └───────────────────────────────────────────────────┘
  Each remote deployed independently; browser assembles the app.

The dominant flavor today. Two main mechanisms: Module Federation (webpack 5+/Rspack — §3 in depth) where bundles expose and consume modules at runtime with dependency-sharing negotiation; and import maps + native ESM, where the browser itself resolves bare specifiers (import "catalog/App") to versioned URLs — simpler, standards-based, but with cruder dependency sharing (you share exactly what the import map pins, no version negotiation).

2.3 Server-side composition

  SERVER-SIDE COMPOSITION

  Browser ──GET /product/42──▶ ┌──────────────────────────┐
                               │  Composition layer        │
                               │  (edge / BFF / gateway)   │
                               │   ├─▶ header service ─────┼─▶ <header html>
                               │   ├─▶ product service ────┼─▶ <product html>
                               │   └─▶ recs service ───────┼─▶ <recs html>
                               │  stitches fragments into  │
                               │  ONE streamed HTML page   │
                               └──────────────────────────┘

Each team runs a service that renders HTML fragments; a composition layer stitches them per request. Lineage: SSI (Server Side Includes), ESI (Edge Side Includes — Akamai/Fastly-era), then Zalando's Tailor and Finn.no's Podium as the node-era frameworks, and today often just a BFF that fetches fragments and streams. Strengths: real SSR per fragment (SEO, fast first paint), no client framework coupling. Costs: you now operate a composition tier; client-side interactivity across fragments still needs a client contract; and per-fragment latency is summed or streamed carefully to avoid the slowest team's fragment stalling the page.

2.4 iframes — the strongest isolation

  IFRAME COMPOSITION

  ┌─ parent document (shell) ────────────────────────┐
  │   ┌─ iframe: payments.other-corp.com ─────────┐  │
  │   │  separate document, JS realm, CSS world,  │  │
  │   │  cookies, CSP — a different UNIVERSE      │  │
  │   └───────────────▲───────────────────────────┘  │
  │        postMessage│(the ONLY bridge)             │
  └───────────────────┴──────────────────────────────┘

Unfashionable but genuinely correct in one niche: when isolation is the requirement — payment forms (PCI scope containment: Stripe Elements is an iframe MFE), embedding third-party or acquired apps you don't trust, plugin systems running arbitrary code. Nothing else gives you a separate JS realm, style universe, and security boundary. The seams are the price, and interviewers like hearing them named: resize (the iframe doesn't size to content — you need a postMessage height protocol or ResizeObserver relay), focus/keyboard (tab order and focus trapping across documents is painful, screen-reader context resets), routing (inner navigation doesn't update the outer URL without an explicit sync protocol), and postMessage as an untyped, stringly, async-only API — always validate event.origin and wrap it in a typed client.

2.5 Web components as the neutral contract

Custom elements give teams a framework-agnostic interface: the contract is "an element named <checkout-widget> with these attributes/properties and these DOM events," not "a React component with these props." Inside, a team can use React, Vue, Svelte, or vanilla — and swap it later without the host noticing. Shadow DOM adds real style encapsulation (outer CSS can't leak in, inner can't leak out — themable via CSS custom properties, which pierce intentionally). Costs: SSR of shadow DOM (Declarative Shadow DOM helps but is newer), passing rich data (attributes are strings; you need property assignment conventions), and framework event interop (largely solved — React 19 finally handles custom element properties/events natively). In practice web components often serve as the wrapper contract around MFEs that are internally framework-rendered.

2.6 Route-based vs fragment-based — the granularity axis

  ROUTE-BASED (common, simpler)          FRAGMENT-BASED (rare, harder)

  /catalog/*   → Team Catalog MFE        ┌─ one page ──────────────┐
  /checkout/*  → Team Checkout MFE       │ [header — Team Platform]│
  /account/*   → Team Account MFE        │ [search — Team Discovery]│
                                         │ [cart   — Team Checkout]│
  App shell: nav chrome + router +       │ [recs   — Team ML]      │
  auth + which-MFE-owns-which-prefix.    └─────────────────────────┘
  Only ONE team's code runs per page     4 teams' code composing,
  (plus the shell).                      sharing, colliding per page

Route-based split is the simplest and most common real-world deployment: a thin app shell owns chrome, auth, and top-level routing; each team owns entire route subtrees. Cross-team interaction is minimized because only one MFE is active at a time — most of the hard problems in §4 shrink dramatically. Fragment-based (multiple teams' components live on one page) is where MFE gets genuinely hard: shared rendering context, style collisions, event coordination, combinatorial version testing. Recommend route-based by default; treat fragment composition as an escalation you accept only for specific high-value pages (e.g., a commerce product page genuinely assembled by search, pricing, and recommendations teams).

A crisp interview formulation: "composition approach is a spectrum of when you pay the integration cost. Build-time pays it in release coupling, server-side pays it in infrastructure, client-side pays it in runtime complexity and bytes, iframes pay it in UX seams. I pick based on which currency the org can most afford."

3. Module Federation — deep dive T1

The mental model

Module Federation (webpack 5, now also Rspack and others) lets separately built and deployed bundles behave like one module graph at runtime. Each build produces a container that can expose some of its modules and consume modules from other containers. A host is a container that consumes; a remote is one that exposes (any app can be both — federation is bidirectional).

The linchpin is remoteEntry.js: a small manifest file each remote emits. It does not contain the remote's application code — it contains the container's runtime interface: the list of exposed modules, which chunks implement them, and which shared dependencies this remote provides or needs. The host loads remoteEntry, negotiates shared dependencies, then lazily fetches only the chunks for modules it actually imports. 

  MODULE FEDERATION RUNTIME FLOW

  HOST (shell.example.com)                REMOTE (checkout.example.com)
  ────────────────────────                ─────────────────────────────
  1. page load: host bundle runs
  2. sees import("checkout/Cart")
  3. ──GET checkout.example.com/remoteEntry.js──▶  (tiny manifest, ~5–30KB)
  4. container.init(sharedScope)  ◀── host and remote register what they
        │                             PROVIDE and REQUIRE into a shared
        │                             scope object {react: {18.3.1: …}}
  5. version negotiation: "host provides react@18.3.1, remote requires
     ^18.2.0 → satisfied → remote will USE THE HOST'S react, its own
     react chunk is never downloaded"
  6. container.get("./Cart") ──▶ returns a factory
  7. ──GET only the chunks Cart needs (cart.[hash].js …)──▶
  8. host renders <Cart /> — one React instance, one module graph

Configuration — the canonical example

// ── remote: checkout app ── webpack.config.js
const { ModuleFederationPlugin } = require("webpack").container;

module.exports = {
  output: { publicPath: "auto" },       // critical: chunks resolve relative
  plugins: [                             // to wherever remoteEntry was loaded
    new ModuleFederationPlugin({
      name: "checkout",                  // global container name
      filename: "remoteEntry.js",        // the manifest artifact
      exposes: {
        "./Cart": "./src/components/Cart",     // public API of this MFE —
        "./routes": "./src/routes",            // treat like a published pkg
      },
      shared: {
        react:        { singleton: true, requiredVersion: "^18.2.0" },
        "react-dom":  { singleton: true, requiredVersion: "^18.2.0" },
        "react-router-dom": { singleton: true, requiredVersion: "^6.20.0" },
        "@org/design-system": { singleton: true, requiredVersion: "^4.0.0" },
      },
    }),
  ],
};

// ── host: app shell ── webpack.config.js
new ModuleFederationPlugin({
  name: "shell",
  remotes: {
    // "name@URL" — name must match the remote's `name`
    checkout: "checkout@https://checkout.cdn.example.com/remoteEntry.js",
    catalog:  "catalog@https://catalog.cdn.example.com/remoteEntry.js",
  },
  shared: {
    react:       { singleton: true, eager: true, requiredVersion: "^18.2.0" },
    "react-dom": { singleton: true, eager: true, requiredVersion: "^18.2.0" },
  },
});

// ── consuming, in the shell ──
const Cart = React.lazy(() => import("checkout/Cart"));
Detail interviewers like: because the host imports from remotes asynchronously, the host's own entry usually needs to be async too — the classic pattern is a tiny index.js containing only import("./bootstrap"), giving webpack an async boundary in which to negotiate shared modules before any React code runs. eager: true on the host's shared deps is the alternative: bundle them into the entry so they're available synchronously (host pays bytes up front; typical for the shell, wrong for remotes).

Shared dependencies — the crux

This is the part of federation interviews probe hardest, because it's where real-world MFE breaks. The shared config populates a runtime share scope: an in-memory registry, keyed by package name, where every container advertises the versions it can provide and the ranges it requires. On first use of a shared module, the runtime picks a provider — by default the highest available version that satisfies the consumer's requiredVersion range — and hands the same module instance to everyone compatible with it. Each remote still ships its own copy as a fallback chunk; that chunk is simply never downloaded if the scope already satisfies the requirement. That's the dedupe mechanism: not build-time, but a runtime auction.

OptionSemanticsWhen / why
singleton: true At most ONE instance in the share scope, ever. If a container requires an incompatible version, it does NOT load its own copy — it gets the existing one plus a console warning (or a hard error with strictVersion: true). Mandatory for react, react-dom, the router, any context-carrying or global-state library.
requiredVersion The semver range this container needs. Drives negotiation and mismatch warnings. Defaults from your package.json if omitted — pin it explicitly in shared configs to avoid surprises. Always set explicitly for singletons; it's your compatibility contract.
eager: true Bundle the shared module into the entry chunk, available synchronously — no async boundary needed, but no dedupe benefit for that container (it always ships its copy). Host/shell only. On remotes it defeats the purpose and bloats every remote.
strictVersion: true Throw at runtime instead of warning when the negotiated version violates requiredVersion. Prefer failing loudly in staging over silently running a version a team never tested against.
version / fallback What this container offers to the scope, and the local copy used if nothing in scope satisfies the requirement (non-singleton case: multiple versions can coexist, each consumer getting a compatible one). Fine for leaf utilities (lodash, date-fns) where two instances are wasteful but harmless.

Why singleton for React is non-negotiable: two React copies on one page is not a performance bug, it's a correctness bug. Hooks work via a shared internal dispatcher — a component rendered by React-copy-A calling hooks from React-copy-B throws the infamous "Invalid hook call" error. Context is identity-based: a ThemeContext created in copy A is a different object in copy B, so every useContext across the boundary silently returns the default value — auth contexts, theme, router context all break. Two ReactDOMs also mean two event systems and two schedulers fighting over the same document. The same identity argument applies to react-router (router context), and to any shared store whose "sharedness" depends on module-instance identity.

The subtle production failure: everything works in staging, then one team ships a remote requiring react@^19 while the shell still provides 18. With singleton: true and no strictVersion, that remote silently runs on React 18 — a version it never tested — and breaks in ways its own CI could never catch. Mitigations: strictVersion in non-prod, a platform-team-owned policy for coordinated framework upgrades (singletons are the one place MFE teams must move together), and contract tests that assert each remote's requiredVersion against the shell's provided versions at deploy time.

Dynamic remotes — runtime URL resolution

Hardcoding remote URLs in webpack config couples environments to builds. Production setups resolve remotes at runtime: from env config, a manifest service, or per-request (which also enables canarying a remote by serving different remoteEntry URLs to different cohorts — §6). 

// Dynamic remote loading — no `remotes` config needed in the host
async function loadRemote(scope, module, url) {
  await __webpack_init_sharing__("default");       // populate share scope
  await loadScript(url);                            // adds window[scope]
  const container = window[scope];                  // e.g. window.checkout
  await container.init(__webpack_share_scopes__.default); // negotiate deps
  const factory = await container.get(module);      // e.g. "./Cart"
  return factory();
}

// URLs come from config, not from the build:
const manifest = await fetch("/mfe-manifest.json").then(r => r.json());
// { "checkout": "https://checkout.cdn…/v437/remoteEntry.js", … }
const Cart = React.lazy(async () => ({
  default: (await loadRemote("checkout", "./Cart",
                             manifest.checkout)).default ?? …
}));

(Module Federation 2.0's runtime — @module-federation/enhanced — wraps exactly this in a first-class API: init({ remotes }) + loadRemote("checkout/Cart"), plus runtime plugins for retries, fallbacks, and manifest-driven resolution.)

Federation + TypeScript — the types-drift problem

import("checkout/Cart") has no types: the module exists only at runtime, so TypeScript can't see across the seam, and worse, types drift — the remote redeploys with a changed prop contract and the host's stale type declarations still compile green. Approaches, in increasing automation: (1) hand-maintained declare module "checkout/Cart" ambient files — drift guaranteed; (2) a shared types package per seam (@org/checkout-contracts) that both sides depend on — the contract is versioned and reviewed like an API, my usual recommendation because it forces contract thinking, not just type sync; (3) automated type extraction — @module-federation/typescript / the 2.0 dts-plugin, which emits a types archive alongside remoteEntry and syncs it into consumers, keeping tsc honest against the deployed reality. Even with (3), remember types check at build time but remotes change after — contract tests remain necessary (§6).

Ecosystem awareness: MF 2.0, Rspack, Vite

  • Module Federation 2.0 / @module-federation/enhanced: federation extracted from webpack into a standalone spec + runtime — a manifest protocol richer than raw remoteEntry, runtime plugins (retry, fallback, telemetry hooks), first-class dynamic remotes, and the dts plugin for type sync. Backed by ByteDance's production usage; the direction of travel.
  • Rspack: the Rust webpack-compatible bundler ships native Module Federation support — the practical answer to "federation locks me into slow webpack builds."
  • Vite: @originjs/vite-plugin-federation and the newer @module-federation/vite implement the same host/remote model over ESM output. Interop across bundlers works but is the least-paved path — a fully aligned toolchain (or the 2.0 runtime as the common denominator) saves pain.

Failure handling — remotes go down

A remote is now a runtime network dependency: CDN hiccup, bad deploy, corporate proxy — the import can reject. Unhandled, one team's outage becomes a blank page for the whole product. The pattern is error boundary + fallback module, plus retry: 

class RemoteBoundary extends React.Component {
  state = { failed: false };
  static getDerivedStateFromError() { return { failed: true }; }
  componentDidCatch(err) { reportToTeam(this.props.owner, err); }
  render() {
    return this.state.failed ? this.props.fallback : this.props.children;
  }
}

const Cart = React.lazy(() =>
  import("checkout/Cart").catch(() => {
    // remoteEntry unreachable or chunk failed — degrade, don't die
    return import("./fallbacks/CartUnavailable");   // local stub module
  })
);

<RemoteBoundary owner="team-checkout" fallback={<CartUnavailable />}>
  <Suspense fallback={<CartSkeleton />}>
    <Cart />
  </Suspense>
</RemoteBoundary>

Both layers matter: the .catch on the dynamic import handles load failures (network, missing chunk), the boundary handles render failures (the remote loaded but throws). Decide the degraded UX per seam deliberately — a dead recommendations rail can vanish silently; a dead checkout needs an apologetic retry screen and a page.

Framing that lands well: "Module Federation turns a build-time dependency graph into a distributed system. Everything we know about distributed systems applies: version skew, partial failure, contracts, observability per node. Teams that adopt it as 'just webpack config' get distributed-systems problems without distributed-systems discipline."

4. Cross-MFE concerns — the hard parts T2

Loading remote code is the easy 20%. Interviewers dig here because these seams — routing, state, auth, design, CSS — are where MFE projects actually bleed. For each: the options, the recommendation, and the cost.

4.1 Routing

  ROUTING OWNERSHIP MODELS

  A) Shell owns the (only) router          B) Each remote owns its router
  ┌─ shell: BrowserRouter ──────────┐      ┌─ shell: tiny prefix matcher ──┐
  │  /catalog/*  → <CatalogApp/>    │      │ /checkout/* → mount checkout  │
  │  /checkout/* → <CheckoutApp/>   │      │   ┌─ checkout MFE ─────────┐  │
  │  remotes register route trees   │      │   │ own Router(basename=   │  │
  │  or render <Routes> against     │      │   │  "/checkout") on the   │  │
  │  the SHARED router context      │      │   │  same history — or a   │  │
  └─────────────────────────────────┘      │   │  MEMORY router + sync  │  │
  needs router as shared singleton;        │   └────────────────────────┘  │
  one nav model, deep links "just work"    └───────────────────────────────┘
                                           looser coupling; history-sync
                                           protocol needed between levels

Model A (shell owns the router) is the right default for route-based MFE on one framework: the shell creates the single history/router; remotes expose route definitions (an exposed ./routes module) or render their own <Routes> subtree against the shared router context. Requires the router library as a shared singleton (context identity again). One coherent model for links, transitions, guards, scroll restoration. Model B — each remote instantiates its own router, either on the real history with a basename, or a memory router with an explicit two-way sync to the shell's URL — buys framework/version independence (a Vue remote inside a React shell) at the cost of a synchronization protocol you now own, and it's where deep-linking bugs breed.

Deep-linking is the classic demo-vs-production gap: clicking around works, but a cold load of /checkout/payment?step=2 404s or loses state because (a) the CDN/server isn't rewriting all paths to the shell's index.html, (b) the remote's internal router never initialized from the URL (memory-router setups), or (c) the remote treated in-app state as implicit instead of URL-encoded. Test cold-load and refresh on every route depth, not just top-level navigation. Also watch for double history entries when two routers both push for one navigation.

4.2 State & communication — ranked

The governing principle: every piece of shared state is a coupling between independently deployed artifacts — the exact thing MFE was adopted to remove. So: share as little as possible, prefer duplication over coupling for anything cheap to duplicate, and rank mechanisms by how loosely they couple:

  1. The URL (first choice). Selected product, filters, tab, step — if it describes "where the user is," put it in the URL. It's a contract both MFEs already honor, it survives refresh, it deep-links, it's debuggable in the address bar, and it couples nobody to a library.
  2. Custom events / a typed event bus (second choice). For fire-and-forget facts: "item added to cart," "user logged out." Pub/sub over window CustomEvents keeps MFEs unaware of each other — they share only the event contract. Make the contract typed and versioned:
// @org/event-contracts — a versioned package BOTH sides depend on
export interface EventMap {
  "cart:item-added":   { sku: string; qty: number };
  "auth:logged-out":   { reason: "expired" | "user" };
}

export function emit<K extends keyof EventMap>(type: K, detail: EventMap[K]) {
  window.dispatchEvent(new CustomEvent(type, { detail }));
}

export function on<K extends keyof EventMap>(
  type: K, handler: (detail: EventMap[K]) => void
): () => void {
  const fn = (e: Event) => handler((e as CustomEvent).detail);
  window.addEventListener(type, fn);
  return () => window.removeEventListener(type, fn);   // unsubscribe
}

// team-checkout emits…            // team-header listens…
emit("cart:item-added",            const off = on("cart:item-added",
     { sku: "A1", qty: 2 });            d => setBadge(b => b + d.qty));
  1. Shared store singleton (last resort — coupling alarm). A shared Redux/Zustand store exposed by the shell technically works, but now every consumer is coupled to the store's shape, its library version (singleton!), and its update semantics — you've rebuilt the monolith's tangle across deploy boundaries, where a shape change is a breaking cross-team API change. If unavoidable, keep it to a tiny, explicitly versioned slice (auth/user/feature-flags), expose it behind a narrow accessor API rather than raw store shape, and treat changes like public API changes with deprecation windows.
"Duplicate over couple" is worth saying explicitly: if two MFEs both need the user's display name, letting each fetch it (or read it from a token claim) is usually cheaper long-term than building a shared user store both deploy against. Redundant fetches are visible and cheap; hidden coupling is invisible and expensive.

4.3 Auth

  AUTH TOPOLOGIES

  A) Cookie-domain strategy               B) Shell-managed token + bridge
  httpOnly cookie on .example.com         ┌─ shell: owns login, holds     ─┐
     │                                    │  tokens in memory, runs THE   │
     ├─▶ shell        (sends cookie)      │  ONE silent-refresh loop      │
     ├─▶ catalog API  (sends cookie)      │   ├─ getToken(): Promise      │
     └─▶ checkout API (sends cookie)      │   └─ provided to remotes via  │
  MFEs never touch tokens at all —        │      context / injected client│
  the browser is the auth bridge          └───────────────────────────────┘

Strategy A — cookie on a shared parent domain — is the cleanest when all MFEs and their APIs live under one registrable domain: an httpOnly, Secure, SameSite cookie set on .example.com rides along on every request from every MFE. No token ever exists in JS (XSS can't exfiltrate it), and no MFE contains auth logic. Costs: CSRF discipline, and it breaks across unrelated domains (acquisitions again). Strategy B — shell-managed tokens — the shell owns the OAuth/OIDC dance, keeps the access token in memory, and hands remotes either the token via a context bridge or, better, a pre-authenticated fetch client (narrower contract: remotes can make calls but never see or store the token). Either way, the non-negotiable rule: silent refresh runs in exactly one place — the shell. Five MFEs independently refreshing the same session produce token races, refresh-token reuse errors, and random logouts that are miserable to debug.

4.4 Design consistency

Design drift is the most visible MFE failure — users don't see your deploy topology, but they instantly see three different button styles. Countermeasures, in layers:

  • A shared design-system package (@org/design-system) as the primary tool, with an explicit versioning policy: teams may run at most one major behind (allow N-1). Without a policy you get a team frozen on v2 for two years; forcing exact lockstep reintroduces coupled deploys. N-1 with a stated deprecation window is the workable middle. Decide consciously whether it's a shared singleton at runtime (dedupe, but versions must be compatible) or bundled per MFE (duplication, but true independence) — for an N-1 policy, per-MFE bundling with shared tokens is often the honest choice.
  • Design tokens as the lowest common denominator. Colors, spacing, typography, radii as CSS custom properties published from one source. Even a rogue MFE on a different framework version — or a different framework — consumes tokens, so the worst-case drift is component behavior, not the brand palette.
  • Icon/font dedupe: the shell loads brand fonts once (with font-display and preload); icons ship as a shared sprite/package, not re-bundled per MFE — otherwise users download your icon set N times and fonts flash per-MFE.

5. Performance in an MFE world T2

MFE starts performance-negative; the job is clawing it back to parity. Four fronts:

5.1 Framework dedupe

Covered mechanically in §3 — shared singletons are the tool — but own the outcome: audit what actually ships. Run bundle analysis across the composed page, not per repo; per-repo analysis is how three copies of a date library hide. Common leaks: a remote pinning a dependency version outside every other range (negotiation fails → fallback chunk downloads), heavy transitive deps nobody listed in shared, and eager: true cargo-culted onto remotes.

5.2 Caching remoteEntry correctly

  THE remoteEntry CACHING DILEMMA — and its resolution

  Stable URL: /remoteEntry.js            Versioned URL via manifest:
    must be no-cache (it's the            /mfe-manifest.json  (no-cache, tiny)
    deploy pointer!) → a network            └─ "checkout": ".../v437/remoteEntry.js"
    round-trip on EVERY page load                              └─ immutable,
    per remote, before its code                                   cache forever ✓
    can even start downloading           deploy = publish new manifest;
                                         rollback = point back to v436 ✓

The trap: remoteEntry is the mutable pointer to a remote's current deploy, so it can't be long-cached at a stable URL — but fetching it fresh per pageview per remote adds latency exactly on the critical path. Resolution: versioned, immutable remoteEntry URLs (…/v437/remoteEntry.js, Cache-Control: immutable) discovered through one tiny non-cached manifest (or short-TTL edge config). Bonus: the manifest is your deploy, rollback, and canary mechanism — flipping a pointer, not moving files. A remote's inner chunks are already content-hashed and cache-forever as usual.

5.3 The waterfall risk

  NAIVE MFE LOAD — a three-hop waterfall
  t0 ── shell HTML ──▶ t1 ── shell JS runs, discovers route needs
       "checkout" ──▶ t2 ── fetch remoteEntry ──▶ t3 ── negotiate,
       fetch cart chunks ──▶ t4 ── fetch cart's lazy children… ──▶ render

  MITIGATED
  t0 ── shell HTML carries hints for the LIKELY route:
        <link rel="preload" href=".../v437/remoteEntry.js" as="script">
        (server/edge knows the URL path → knows which remote → injects hints)
     ── shell JS + remoteEntry + first chunks download IN PARALLEL
     ── prefetch other remotes' entries on idle / on nav hover

Each hop is discovered only after the previous finishes — shell, then remoteEntry, then chunks — which is how MFE pages lose seconds of LCP versus a monolith whose route chunk was one hop. Mitigations: edge-injected preload hints for the route's remote (the server knows the URL → knows the owning remote → can hint before any JS runs), preconnect to remote CDN origins, idle-time prefetch of remoteEntry for likely next remotes (nav hover is a good trigger), and keeping remotes' own internal code-splitting shallow on the critical path.

5.4 Budgets, attribution, and loading UX

  • Per-MFE performance budgets, enforced in each remote's CI (size-limit / bundlesize on the exposed-module chunks) — with N teams, an unowned global budget degrades by diffusion of responsibility; per-team budgets keep regressions attributable at the PR that caused them.
  • Attribute Web Vitals per team: tag RUM data with the owning MFE and its deployed version (route ownership makes this easy; fragment MFEs need element-level attribution — LoAF script attribution and marking INP targets help). "LCP regressed" is a meeting; "checkout v437 regressed LCP 400ms" is a rollback.
  • Skeleton/fallback UX during remote load: remote loading is real user-visible latency, so design it — route-shaped skeletons in Suspense fallbacks (shell owns them, so they paint instantly), stable layout dimensions to avoid CLS when the remote lands, and a timeout path into the §3 error fallback rather than an infinite skeleton.

6. Delivery & governance T2

MFE's whole payoff is delivery independence — so the delivery system is where the architecture is actually won or lost. Interviewers use this section to check whether you've operated MFE or just built a demo.

Independent CI/CD + contract tests at the seams

  PER-MFE PIPELINE

  team repo ─▶ build ─▶ unit/CT ─▶ CONTRACT TESTS ─▶ publish v438 ─▶ flip
                                   ├ exposed modules still exist     manifest
                                   ├ props/events match contract pkg  pointer
                                   ├ requiredVersions ⊆ shell provides
                                   └ (no other team's suite runs — that's
                                      the point AND the risk being managed)

Each MFE ships from its own pipeline with no other team in the loop — that's the point. The risk it creates (integration breakage nobody's suite covers) is managed by contract tests at the seams: assertions that the remote still exposes the modules the shell imports, that exposed components accept the contracted props and emit the contracted events (the @org/*-contracts packages from §4 make these executable), and that the remote's shared requiredVersion ranges are satisfiable by the shell's provided versions. Cheap, fast, and they catch the majority of cross-team breaks before deploy.

Versioned deploys, rollback, and canarying a single MFE

Because deploys are manifest-pointer flips (§5.2), rollback is instant and per-remote — point checkout back at v437 without touching anyone else. The same mechanism gives you per-MFE canarying: the manifest service (or edge worker) serves v438's remoteEntry URL to 5% of sessions and v437 to the rest, while RUM and error tracking are segmented by remote+version. One subtlety worth naming: with N independently canarying remotes, users run version combinations no one explicitly tested — keep singleton/shared contracts conservative, and log the full remote-version vector with every error report so "checkout 4.38 + shell 1.9" failures are diagnosable.

Ownership boundaries — who owns the shell

The shell (router, auth, shared-dependency policy, manifest service, design tokens, error/RUM plumbing) must not be orphaned or owned by whichever feature team touched it last. The standard answer is a platform team that treats the shell and shared tooling as an internal product: feature teams are its customers, changes to shared contracts go through it, and it owns the singleton-upgrade choreography (the React 18→19 migration across all remotes is their program to run). Without this, shared decisions decay into tragedy-of-the-commons.

Documentation and contracts as artifacts

  • Typed event contracts and exposed-module type packages, versioned and changelogged — the §4 event bus is only as good as the discipline around its contract package.
  • Semver discipline on exposed modules: an exposed module is a published API. Renaming ./Cart or changing its props is a major version; breaking changes ship behind a deprecation window (expose both, announce, remove) because you cannot atomically update your consumers — they deploy independently, that's the whole premise.
  • A living ownership map: which team owns which routes/fragments/events — the first thing an on-call engineer needs at 3am.

E2E across MFEs

Full-matrix e2e across N independently deploying remotes is combinatorially doomed — don't promise it. The workable pattern: a thin, centrally owned critical-path suite (login → browse → add to cart → pay; ~10–20 journeys) running against the integrated environment on every remote's deploy and on a schedule, owned by the platform team with a hard flakiness budget. Teams keep everything else (unit, component, their own e2e) inside their own boundary. Depth belongs to contract tests; the central suite only proves the seams hold on the paths that pay the bills.

The migration story — strangler fig, rehearsed

Also have a migration narrative ready, because "we have a legacy app" is the most common real-world MFE prompt and the strongest justification for the pattern:

  ┌─ legacy app ─┐       ┌─ shell (new) ───────┐  ┌─ shell ─────────────┐

                         └─────────────────────┘  └─────────────────────┘

Name the honest costs too: users pay a double-runtime tax mid-migration, auth/session must bridge both worlds from day one, and the middle phase — where every incident starts with "is it legacy or new?" — must be actively kept short rather than allowed to become the permanent state.

Interview Q&A

Q1. What problem do micro-frontends actually solve?

An organizational one: many teams shipping into one product surface without blocking each other. Concretely — independent deploys (no release trains or shared merge queues), independent tech evolution (teams upgrade frameworks on their own schedule), and physical ownership boundaries that align architecture with team structure (Conway's law). MFE does not solve technical problems — it typically makes performance, consistency, and operational simplicity worse. If a candidate's motivation is "smaller bundles" or "faster pages," they've confused MFE with code-splitting, which a monolith already does with dynamic import().

Q2. When would you advise AGAINST micro-frontends?

Most of the time. Specifically: fewer than ~3–4 frontend teams; no measurable deploy contention; one team that just wants "better modularity" (use a modular monolith — monorepo, enforced package boundaries via Nx/ESLint rules, code ownership); performance as the stated goal; or a team without the platform capacity to own the shell, shared-dependency governance, and contract testing. The costs — bundle duplication, UX drift, cross-MFE state pain, N pipelines, local dev orchestration, seam testing — are a permanent tax; the benefit only exceeds it when inter-team coordination is the actual bottleneck. Strong closing line: "the modular monolith is MFE-ready — I'd build the boundaries now and split only when the org forces it."

Q3. Walk me through what happens at runtime when a Module Federation host renders a remote component.

(1) Host code hits import("checkout/Cart"). (2) The federation runtime fetches the remote's remoteEntry.js — a small manifest exposing the container interface, not the app code. (3) Host calls container.init(sharedScope): both sides register the shared dependencies they provide/require into a common share scope. (4) Version negotiation: for each shared package, the runtime picks a satisfying provider (e.g., remote requires react@^18.2, host provides 18.3.1 → remote uses the host's React; its own React fallback chunk is never downloaded). (5) container.get("./Cart") returns a module factory; only the chunks implementing Cart are fetched. (6) The component renders inside the host's tree with a single React instance. Bonus points: mention the async boundary (import("./bootstrap") pattern) or eager shared deps on the host, and publicPath: "auto" so the remote's chunks resolve relative to where remoteEntry was loaded.

Q4. Why must React be a singleton in the shared config? What actually breaks with two copies?

Correctness, not just bytes. Hooks: React resolves hooks through an internal dispatcher tied to the rendering React instance — a component from copy A rendered by copy B throws "Invalid hook call." Context: context objects are compared by identity, so a provider created in copy A is invisible to a useContext in copy B — auth, theme, and router context silently return defaults, which is worse than crashing because it half-works. Also: two ReactDOM event systems and two schedulers on one document. singleton: true guarantees one instance in the share scope; pair it with an explicit requiredVersion and (in non-prod) strictVersion: true so incompatibilities fail loudly instead of silently running an untested version.

Q5. A remote declares requiredVersion: "^19.0.0" for React but the shell provides 18.3. What happens?

Depends on config. With singleton: true (the correct setting for React) and default options: the remote still receives the shell's React 18.3 — singletons never load a second copy — and the runtime logs a version-mismatch warning. That's dangerous: the remote runs against a major version it never tested. With strictVersion: true it throws instead, which you'd catch with the load-failure fallback. Without singleton, the remote would load its own React 19 fallback — two Reacts, see Q4, worse. The real answer is organizational: framework majors on singletons are the one coordinated migration MFE can't dodge, so the platform team runs it as a program, and deploy-time contract checks assert every remote's ranges against what the shell provides.

Q6. How do you handle a remote that fails to load in production?

Two failure layers, two tools. Load failures (remoteEntry unreachable, chunk 404 after a bad deploy): .catch on the dynamic import, returning a local fallback module — optionally with a retry (once, with backoff) before degrading. Render failures (remote loaded but throws): a React error boundary around the mount point, tagged with the owning team for alerting. UX per seam is a product decision: a recommendations rail can disappear silently; checkout gets an apology + retry screen and pages the owning team. Senior addition: this is why versioned remoteEntry URLs matter — a "remote down" incident is usually a bad deploy, and rollback is repointing the manifest, seconds not minutes.

Q7. Route-based vs fragment-based composition — how do you choose?

Default to route-based: shell owns chrome/auth/top-level routing; each team owns whole route subtrees; only one team's MFE is active per page. This minimizes every hard problem — style collisions, shared state, event coordination, version-combination testing — because the MFEs barely coexist. Fragment-based (multiple teams composing one page at runtime) is an order of magnitude harder and only justified when a single high-value page is genuinely built by multiple teams — e.g., a commerce product page with search, pricing, and recommendations teams. Even then, scope fragment composition to those specific pages rather than making it the default platform capability.

Q8. How should micro-frontends communicate? Rank the options.

Ranked by coupling, ascending: (1) the URL — anything describing "where the user is" (selection, filters, step) belongs there; it's refresh-safe, deep-linkable, and couples nobody. (2) Typed custom events / event bus — fire-and-forget facts ("cart:item-added"), with the event map in a versioned contracts package both sides depend on, so the coupling is to a reviewed contract, not to each other. (3) Shared store singleton — a coupling alarm: every consumer binds to store shape, library version, and update semantics across deploy boundaries. If forced, keep it a tiny slice (auth/user/flags) behind a narrow accessor API. Overarching principle: minimize shared state, and prefer duplicating cheap data over coupling — redundant fetches are visible and cheap; hidden coupling is invisible and expensive.

Q9. How does routing work across MFEs, and what goes wrong?

Two models. (A) Shell owns the single router; remotes expose route definitions or render subtrees against the shared router context — requires the router as a shared singleton, gives one coherent navigation model; the right default for one-framework, route-based setups. (B) Each remote runs its own router — real history with a basename, or a memory router synced to the shell — buying framework independence at the cost of an explicit history-sync protocol. Classic failures: deep links that 404 on cold load (missing catch-all rewrite to the shell's index.html), memory routers that ignore the initial URL, state kept in JS instead of the URL so refresh loses it, and double history entries when two routers both push for one navigation. Test cold-load and refresh at every route depth.

Q10. How do you keep UX consistent across independently shipping teams?

Layered defense. (1) A shared design-system package with an explicit version policy — allow N-1 major with a stated deprecation window; lockstep reintroduces coupled deploys, no policy yields teams frozen years behind. (2) Design tokens (colors/spacing/type as CSS custom properties) as the lowest common denominator — even an MFE on a divergent DS version or different framework consumes tokens, capping worst-case drift at component behavior, not brand. (3) Dedupe fonts and icons through the shell so they load once. (4) Governance: the platform/design-system team tracks adoption versions per MFE and runs upgrade windows. Honest caveat: some drift is the price of independent shipping — you manage it, you don't eliminate it.

Q11. Design auth for a micro-frontend product.

If all MFEs and APIs share a registrable domain: an httpOnly Secure cookie on .example.com — no token in JS at all, every MFE's requests carry it automatically, plus CSRF protections. Otherwise: the shell owns the OIDC flow and holds tokens in memory, exposing either the token via a context bridge or — better — a pre-authenticated fetch client so remotes can call APIs without ever seeing the token. Non-negotiable either way: exactly one silent refresh loop, in the shell. Multiple MFEs refreshing independently cause token races and refresh-token-reuse revocations — random logouts that are brutal to debug. Login/logout state changes propagate as typed events ("auth:logged-out") so remotes react without polling.

Q12. How do you prevent shipping three copies of React and two design systems to users?

Mechanism: Module Federation shared — singletons for react/react-dom/router/DS, version negotiation so remotes reuse the host-provided instance and never download their fallback chunks. Process: audit the composed page, not per-repo bundles (per-repo analysis is where duplicates hide); per-MFE size budgets in CI on exposed chunks; watch for negotiation failures (a remote pinning outside everyone's range silently pulls its fallback) and cargo-culted eager: true on remotes. For non-singleton utilities (lodash, date-fns), sharing is nice-to-have — duplication there is waste, not breakage. And the honest note: perfect dedupe is a monolith property; MFE approaches it asymptotically with ongoing effort.

Q13. How do you cache remoteEntry.js? It seems to need to be both fresh and fast.

That tension is the point of the question. remoteEntry at a stable URL is the mutable pointer to the remote's current deploy → must be no-cache → a blocking network round-trip per remote per page load. Resolution: publish remoteEntry at versioned immutable URLs (/v437/remoteEntry.js, Cache-Control: immutable) and discover them through one tiny non-cached manifest (or short-TTL edge config). Deploy = flip the manifest pointer; rollback = flip it back; canary = serve different pointers per cohort. Inner chunks are content-hashed and cache-forever as usual. This one answer demonstrates you've run MFE in production — it's the kind of thing demos never surface.

Q14. What does eager: true do in shared config, and where is it appropriate?

It bundles the shared module directly into the consuming container's entry chunk so it's available synchronously — removing the need for an async boundary before first use. Cost: that container always ships its own copy of the module, forfeiting dedupe for itself. Appropriate on the host/shell (something must provide React up front, and the shell loads exactly once), often paired with — or instead of — the import("./bootstrap") async-entry pattern. Wrong on remotes: an eager remote ships React inside every remote bundle, exactly the duplication federation exists to prevent. A crisp mnemonic: eager on the provider-of-first-resort, lazy everywhere else.

Q15. Testing strategy for a system of 6 MFEs — what runs where?

Push depth to the edges, keep the center thin. Inside each MFE (team-owned): unit + component tests as usual, plus its own e2e against its routes with seams stubbed. At the seams (the MFE- specific layer): contract tests — exposed modules exist with contracted props/events (executable via shared contract packages), event payloads match the typed event map, shared requiredVersion ranges are satisfiable by the shell. Center (platform-owned): one thin critical-path e2e suite (~10–20 journeys: login → browse → buy) against the integrated env, run on every remote deploy, with a strict flakiness budget. Anti-pattern to call out: attempting full-matrix e2e across independently versioned remotes — combinatorially impossible; that's what contracts and per-remote canaries with good RUM segmentation are for.

Q16. How would you canary just the checkout MFE without touching the rest of the product?

The manifest/edge layer that resolves remote URLs is the control point: serve checkout/v438/remoteEntry.js to 5% of sessions (sticky per session — mixing versions within a session confuses users and telemetry) and v437 to the rest. Segment RUM, errors, and business metrics (conversion!) by remote+version; auto-halt on threshold breach by flipping the pointer back — rollback is instant because old versions remain published at immutable URLs. The subtlety worth volunteering: with several remotes canarying independently, users experience version combinations nobody explicitly tested — log the full remote-version vector on every error so combination-specific failures ("checkout 438 + shell 1.9") are diagnosable, and keep shared/singleton contracts conservative for exactly this reason.

Q17. When are iframes the right micro-frontend answer, and what are the seams?

When isolation is the requirement, not a nice-to-have: payment capture (PCI scope containment — Stripe Elements is an iframe MFE), embedding untrusted/third-party/acquired apps, plugin systems running arbitrary code. Nothing else provides a separate document, JS realm, style universe, and security boundary (plus CSP/sandbox attributes). The seams to name: resize (no content-based sizing — you need a postMessage height protocol), focus/keyboard (tab order and focus trapping across documents, screen-reader context resets), routing (inner navigation doesn't reflect in the outer URL without an explicit sync), and postMessage itself — async-only, untyped, requires origin validation, so wrap it in a typed client. Everywhere else, the UX seams make iframes the wrong default.

Q18. single-spa vs Module Federation — do they compete?

No — different layers, commonly combined. Module Federation is a code delivery and dependency-sharing mechanism: how separately built bundles load each other's modules and negotiate one React. single-spa is a lifecycle orchestrator: apps register bootstrap/mount/unmount functions and it activates/deactivates them by URL, framework-agnostically (React next to Vue next to Angular). A shell can use MF to fetch remotes and single-spa to decide who's mounted when; equally, single-spa can load via import maps/SystemJS with no MF, and MF works with a hand-rolled shell and no single-spa. Distinguishing these two roles cleanly is a reliable literacy check.

Q19. How do you keep TypeScript honest across federation boundaries?

The problem: federated imports exist only at runtime, so tsc can't see them, and any local type declarations drift as the remote redeploys — stale types compile green against a changed contract. Options, increasingly automated: hand-written ambient declare module files (drift guaranteed); a versioned shared contracts package per seam that both sides depend on — my default recommendation, because it forces contract review, not just type sync; or automated extraction — @module-federation/typescript / MF 2.0's dts plugin, which publishes a types archive next to remoteEntry and syncs it into consumers. Key senior caveat: types are checked at build time but remotes change after your build — so types reduce drift, but runtime contract tests at deploy time remain necessary.

Q20. Walk me through migrating a legacy jQuery app to React using micro-frontend techniques.

Strangler fig, route by route. (1) Put a thin shell — or just an edge router — in front; the legacy app becomes the catch-all "remote." Zero user-visible change; the riskiest infrastructure step lands first while stakes are lowest, and it's fully reversible. (2) Bridge cross-cutting concerns once: shared-domain auth cookie so both worlds share the session; a small typed event adapter for the few facts that must cross (e.g., cart count). (3) Pick one route — high value, low coupling — rebuild it as a React remote, point the shell at it; rollback is repointing. (4) Repeat in priority order, deleting legacy code as routes move — progress measured in routes migrated, not "% rewritten." (5) When the catch-all serves nothing, remove it — then decide honestly whether the remotes stay separate (multiple teams?) or fold into one app, because the seams were scaffolding. Costs to volunteer: double-runtime payload mid-migration, and the "is it legacy or new?" incident- triage phase, which you keep deliberately short. Route granularity is what keeps jQuery and React from ever fighting over the same DOM.

Hands-on exercises

Exercise 1 — Design drill: "Should we split?"

You join a company with a single Next.js e-commerce app: 40 frontend engineers in 6 teams (catalog, search, checkout, account, growth, platform). Deploys go out twice a week through a release train; teams regularly slip a full cycle because another team's regression froze the train. Checkout has PCI-driven compliance review on its releases. Leadership asks: "micro-frontends?" Spend 20 minutes producing: your clarifying questions, a go/no-go with reasoning, and — if go — the target architecture with its cost list and guardrails. Then compare.

Model answer

Clarifying questions first: How much of the slippage is frontend-caused vs backend/infra (don't fix the wrong bottleneck)? Do teams own disjoint route surfaces already? Any near-term framework migrations? Does platform have capacity to own a shell as a product?

Verdict: a justified split — this scenario clears the bar. Six teams, chronic deploy contention with quantifiable lead-time cost, and one team (checkout) with a genuinely different release cadence (compliance review) — that's two independent triggers from §1. An 8-engineer/2-team version of this company should get a modular-monolith answer instead; say that contrast out loud.

Shape: route-based split — this is Next.js, so lead with Next.js Multi-Zones (each team an independent Next deployment owning path prefixes, stitched by rewrites) as the lowest-machinery option; Module Federation only if heavy intra-page composition across teams emerges later. Sequence by value: checkout first (its compliance cadence is the sharpest pain and its boundary is naturally clean), then search/growth (highest deploy frequency), then the rest — don't split all six at once. Shared: design-system package with N-1 policy + tokens; auth via cookie on the shared domain; URL-first communication (with hard navigation between zones, cross-MFE runtime state barely exists — a feature, not a bug).

Costs to state: hard navigations between zones; duplicated framework payload per zone (no shared runtime); 6 pipelines and dashboards; local dev orchestration; drift risk. Guardrails: platform team owns the edge router/rewrites map + tokens + the thin critical-path e2e; per-zone Web Vitals attribution and budgets; contract = URL structure + cookie + design tokens, all documented. Exit criteria: if team topology consolidates, zones fold back — route seams make that cheap.

Exercise 2 — Debugging drill: the broken singleton

A federated app: shell (React 18.3) + remotes catalog and reviews. After reviews ships a release, production shows: "Invalid hook call" errors sourced to reviews' components, and — intermittently, only when a user visits reviews first — the catalog MFE renders logged-out UI despite a valid session. The reviews team swears their code is unchanged except "dependency bumps." Network tab shows an extra react.[hash].js chunk loading from the reviews CDN. Diagnose the root cause, explain BOTH symptoms mechanically, and specify the fix plus the guardrails that would have caught it pre-deploy.

Model answer

Root cause: the reviews release bumped React to 19 in its package.json, and its shared config either lost singleton: true (e.g., a webpack config refactor replaced the explicit shared block with a helper that defaults it off) or never pinned requiredVersion — so negotiation failed and the reviews container loaded its own React fallback chunk. The extra react.[hash].js from the reviews CDN is the smoking gun: with a working singleton, that chunk is never downloaded.

Symptom 1 (invalid hook call): components created against reviews' React copy end up rendered in a tree driven by the shell's React (or share elements across copies — e.g., shell-provided children rendered inside reviews' tree). Hooks resolve through the per-instance dispatcher, so crossing copies throws.

Symptom 2 (logged-out catalog, only when reviews loads first): context identity. When reviews initializes first, share-scope resolution can leave parts of the graph bound to reviews' React/module instances; the shell's AuthContext object in copy A is a different identity than what consumers bound to copy B look up, so useContext falls back to the default value — { user: null } — rendering logged-out UI with a valid session. Order-dependence ("only when visited first") is the classic tell of share-scope resolution varying by load order.

Fix: restore singleton: true + explicit requiredVersion on react/react-dom in the reviews config; roll back the remote via manifest pointer immediately (instant, per-remote); coordinate the React 19 move as a platform-run program across all remotes, shell last-or-first per plan — never one remote unilaterally.

Guardrails that would have caught it: deploy-time contract check asserting each remote's shared config (singleton flags + ranges) against shell-provided versions; strictVersion: true in staging so the mismatch throws before prod; a composed-page bundle audit alerting on any second framework chunk; and centralizing the shared-deps block in a platform-owned preset package so individual teams can't silently drop singleton.

Exercise 3 — Migration drill: the acquired Vue app

Your React product (app shell + MF remotes, React 18 singletons, react-router owned by the shell) just acquired a company whose product is a Vue 3 SPA with its own auth (separate domain, its own OIDC client) and its own design language. The business wants it reachable at /portfolio/* inside your product within one quarter, with a "feels integrated" experience within a year. Design the integration in two phases: what ships this quarter, and the path to real integration. Name the routing, auth, styling, and dependency-sharing decisions and their costs at each phase.

Model answer

Phase 1 (this quarter): integrate at the coarsest viable seam — do NOT federate Vue into the React runtime. Two defensible options; pick by auth feasibility. (a) iframe mount: the shell owns /portfolio/* and renders the Vue app in an iframe — strongest isolation, zero style/dependency interaction, fastest to ship; pay the seams with a postMessage protocol for height, inner-route→outer-URL sync, and auth handoff. (b) Route-zone mount (better if domains can converge): serve the Vue app under your domain via edge rewrite at /portfolio/*, hard navigation between worlds — no iframe seams, but requires moving it behind your domain now. Either way, no shared dependencies: React and Vue don't share anything; MF's shared scope is irrelevant across frameworks — say this explicitly, it's a common interview trap ("just federate it" — federation transports modules, it doesn't make Vue components React components). Auth: quarter-1 minimum is SSO at the IdP level (same OIDC provider, token exchange or a session-bridge endpoint) so users log in once, even if the Vue app keeps its own client. Styling: ship design tokens (CSS custom properties) into the Vue app immediately — colors/type/spacing converge for cheap while components stay untouched.

Phase 2 (the year): strangler-fig the Vue app itself, only where justified. First move it fully behind your domain + shared cookie auth, deleting its OIDC client (one refresh loop — the shell's). Then apply the §1 decision framework to each part of the portfolio app: screens the acquired team keeps owning long-term can stay Vue indefinitely behind the route seam (a perfectly good MFE end-state — the interviewer is checking you don't assume "integration = rewrite"); screens moving to your existing teams' roadmaps get rebuilt as React remotes route-by-route, each rebuilt route repointed at the shell. "Feels integrated" is delivered by tokens + shared chrome (shell-owned header/nav around both worlds) + URL/deep-link coherence — not by framework uniformity. Costs to state: double framework payload on cross-world navigation for the whole period; two component libraries maintained; iframe accessibility seams if (a) was chosen — which is a reason to prefer (b) once domains allow; and an explicit end-state decision — leaving "temporary" bridges undecided is how integrations calcify.
