1. Promise semantics, precisely Tier 1

You cannot implement promise-based utilities — or debug production async code — with a fuzzy mental model. These are the exact rules.

States and settling

  • A promise is in exactly one of three states: pending, fulfilled (with a value), or rejected (with a reason).
  • Fulfilled + rejected together are called settled. A promise settles at most once — later calls to resolve/reject inside the executor are silently ignored. This "first settle wins" rule is exactly what makes Promise.race and the timeout pattern work.
  • The executor function passed to new Promise(executor) runs synchronously, immediately. Only the .then callbacks are deferred (to the microtask queue).
// Executor runs synchronously; handlers run as microtasks.
console.log('A');
const p = new Promise((resolve) => {
  console.log('B');          // runs NOW, not later
  resolve(1);
  resolve(2);                // ignored — already settled
  throw new Error('nope');   // ALSO ignored — already settled
});
p.then((v) => console.log('C', v));
console.log('D');
// Output: A B D C 1

The three .then chaining rules

Every .then(onFulfilled, onRejected) call returns a new promise. What settles that new promise is determined entirely by what the handler does:

Handler does…Returned promise…
returns a plain value (incl. undefined)fulfills with that value
throwsrejects with the thrown error
returns a promise/thenableadopts its state — waits for it, then mirrors its fulfillment or rejection
 
Promise.resolve(1)
  .then((v) => v + 1)                        // rule 1: fulfills with 2
  .then((v) => { throw new Error('boom'); }) // rule 2: rejects
  .catch((e) => fetch('/recover'))           // rule 3: adopts the fetch promise
  .then((res) => res.json());                // runs only after fetch settles
A very common quiz: Promise.resolve(1).then(() => {}).then(v => console.log(v)) logs undefined — the arrow body {} is a block, not an object literal, so the handler returns undefined, which fulfills the next promise (rule 1). Rejections only come from throwing or returning a rejected promise — never from returning "falsy" values.

Error propagation and .catch positioning

A rejection propagates down the chain, skipping every onFulfilled handler, until it meets an onRejected handler (a .catch or a second argument to .then). Once handled, the chain is back on the fulfillment track — .catch returns a new promise that fulfills with whatever the catch handler returns (unless it throws again).

Why .then(ok, fail) ≠ .then(ok).catch(fail):

// Version A: fail does NOT see errors thrown inside ok.
// The pair only handles rejections of the UPSTREAM promise.
getUser().then(renderUser, showError);

// Version B: catch sees upstream rejections AND errors thrown in renderUser.
getUser().then(renderUser).catch(showError);
  • Use .then(ok, fail) when you want to handle only the upstream failure and let handler bugs surface separately (this exact distinction matters inside a Promise.all polyfill).
  • Use .then(ok).catch(fail) as the default: one safety net for the whole chain above it.

finally semantics

  • .finally(cb) runs on fulfillment or rejection; cb receives no arguments (it can't know the outcome).
  • It is pass-through: the returned promise settles with the original value/reason, ignoring cb's return value…
  • …with two exceptions: if cb throws (or returns a rejected promise), that error replaces the outcome; and if cb returns a pending promise, settlement is delayed until it resolves.
fetchData()
  .finally(() => setLoading(false))  // runs either way; value passes through
  .then(render)
  .catch(showError);

Microtask timing (30-second recap)

Promise reactions (.then/await continuations) are microtasks. After each macrotask (script, timer, event handler) the engine drains the entire microtask queue — including microtasks queued by other microtasks — before rendering or running the next macrotask. That's why Promise.resolve().then(f) always beats setTimeout(f, 0), and why an infinite microtask loop freezes the page while an infinite setTimeout loop doesn't. Full treatment in tutorial 04.

Narration line that lands well: "A promise doesn't make anything asynchronous by itself — the executor is synchronous. A promise is just a container for an eventual value plus a guarantee that observers are called asynchronously, exactly once, in a consistent order."

2. async/await: desugaring & the parallelism trap Tier 1

What await actually does

async/await is syntax over promises — but knowing the precise desugaring lets you answer any ordering quiz:

  • An async function always returns a promise. return v resolves it with v (adopting v if it's a thenable); an uncaught throw rejects it.
  • The function body runs synchronously until the first await. At each await, the rest of the function is suspended and scheduled as a microtask continuation once the awaited value settles.
  • await x works on any value: non-thenables are wrapped as if by Promise.resolve(x), so await 42 still yields to the microtask queue once.
  • await on a rejected promise throws the rejection reason at that line — which is what makes try/catch work with async code.
async function f() {
  const a = await getA();
  return a * 2;
}
// is morally equivalent to:
function f() {
  return Promise.resolve(getA()).then((a) => a * 2);
}

The sequential-vs-parallel mistake (asked constantly)

// ❌ Sequential: total time = sum of all requests
async function loadDashboardSlow(ids) {
  const results = [];
  for (const id of ids) {
    results.push(await fetchUser(id));  // each await blocks the next fetch from STARTING
  }
  return results;
}

// ✅ Parallel: total time = the slowest request
async function loadDashboardFast(ids) {
  return Promise.all(ids.map((id) => fetchUser(id)));
  // .map starts ALL fetches immediately; Promise.all just waits
}
// Timing diagram, 3 requests × 100ms each:
//
// await-in-loop (300ms total):        Promise.all (100ms total):
//   req1 ██████████                     req1 ██████████
//   req2           ██████████           req2 ██████████
//   req3                     ██████     req3 ██████████
The subtle middle case: const a = fetchA(); const b = fetchB(); return [await a, await b]; — starting both promises before awaiting gives parallelism without Promise.all. But if b rejects while you're still awaiting a, the rejection is unobserved for a tick and can fire an unhandledrejection event. Promise.all is the correct tool; mention this trap and you score points.
Sequential awaiting is sometimes correct — when each call depends on the previous result, or when you must not hammer an API (see the concurrency pool for the middle ground: N at a time).

Error-handling patterns

// Pattern 1: narrow try/catch — only wrap what can fail, keep handling specific
async function save(form) {
  let user;
  try {
    user = await api.createUser(form);
  } catch (err) {
    if (err instanceof ValidationError) return showFieldErrors(err);
    throw err;                    // don't swallow what you can't handle
  }
  await redirectToProfile(user);  // bugs here surface loudly, not as "save failed"
}

// Pattern 2: .catch on the promise — good for fallback VALUES
const config = await fetchConfig().catch(() => DEFAULT_CONFIG);

// Pattern 3: partial failure — don't let one widget kill the dashboard
const [user, orders] = await Promise.allSettled([fetchUser(), fetchOrders()]);
A giant try { ...20 lines... } catch (e) { console.log(e) } is an anti-pattern trifecta: it hides which operation failed, it swallows programmer errors (typos become "network error" toasts), and it converts rejections into silent undefineds. Catch narrowly, handle specifically, rethrow the rest.

Also mention when relevant:

  • Top-level await — legal in ES modules only; the importing module waits for it. Great for one-time config; dangerous in a widely-imported module because it delays the whole graph.
  • for await...of — iterates an async iterable (object with [Symbol.asyncIterator]), awaiting each item: streaming fetch bodies, paginated APIs, async generators. Note it's inherently sequential — that's usually the point.
async function* paginate(url) {
  while (url) {
    const page = await (await fetch(url)).json();
    yield* page.items;
    url = page.next;         // null on last page ends the loop
  }
}
for await (const item of paginate('/api/items')) render(item);

3. Combinators, timeouts & AbortController Tier 1

The four combinators — decision table

CombinatorResolves when…Rejects when…Use for
Promise.allall fulfill → array of values, input order preservedany one rejects (fail-fast, first rejection wins)"I need every result or the operation is meaningless" — parallel fetches for one screen
Promise.allSettledalways → array of {status:'fulfilled', value} / {status:'rejected', reason}neverIndependent operations where partial success is fine — batch saves, dashboard widgets
Promise.racefirst promise to settle fulfillsfirst to settle rejectsTimeouts, "whichever source answers first"
Promise.anyfirst to fulfill (ignores rejections)all reject → AggregateError with .errors arrayRedundant sources: CDN mirrors, fastest-of-N endpoints
Two classic traps: (1) when Promise.all fail-fasts, the other promises keep running — JS promises are not cancellable; you only stop observing them. Pair with AbortController if you need real cancellation. (2) race vs any: race cares about the first settle (a fast rejection wins the race); any cares about the first fulfillment.

Timeout-a-promise with race

function withTimeout(promise, ms) {
  let timerId;
  const timeout = new Promise((_, reject) => {
    timerId = setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms);
  });
  // finally: clear the timer either way so we don't hold the event loop / leak
  return Promise.race([promise, timeout]).finally(() => clearTimeout(timerId));
}
This only stops waiting; the underlying work continues. For real cancellation of a fetch, use AbortSignal.timeout() below — and say so in the interview.

AbortController / AbortSignal — the modern cancellation story

Promises can't be cancelled; AbortController is the standard fix: a controller owns a signal, you pass the signal to any abortable operation, and controller.abort(reason) flips signal.aborted to true and fires its 'abort' event. Every React dev doing data fetching needs this cold.

// 1. Aborting fetch — rejects with DOMException named 'AbortError'
const controller = new AbortController();
fetch('/api/search?q=react', { signal: controller.signal })
  .then((r) => r.json())
  .catch((err) => {
    if (err.name === 'AbortError') return;  // expected — not a real failure
    throw err;
  });
controller.abort();  // e.g. user typed again, or component unmounted

// 2. React effect cleanup — the canonical pattern
useEffect(() => {
  const controller = new AbortController();
  fetchResults(query, controller.signal).then(setResults).catch(ignoreAbort);
  return () => controller.abort();  // cancels on unmount AND on query change
}, [query]);

// 3. addEventListener cleanup — one abort removes MANY listeners
const c = new AbortController();
window.addEventListener('resize', onResize, { signal: c.signal });
window.addEventListener('scroll', onScroll, { signal: c.signal });
c.abort();  // both removed — no need to keep handler references around

// 4. Static helpers
const s1 = AbortSignal.timeout(5000);          // auto-aborts with 'TimeoutError' after 5s
const s2 = AbortSignal.any([s1, userSignal]);  // aborts when EITHER does — compose timeout + manual cancel
fetch(url, { signal: s2 });

Making your own operations abortable — the contract every abortable API follows: (a) if already aborted, fail immediately; (b) listen for 'abort' to stop work and reject with signal.reason; (c) remove the listener when done to avoid leaks.

function abortableDelay(ms, signal) {
  return new Promise((resolve, reject) => {
    if (signal?.aborted) return reject(signal.reason);       // (a) pre-aborted
    const id = setTimeout(() => {
      signal?.removeEventListener('abort', onAbort);          // (c) cleanup
      resolve();
    }, ms);
    const onAbort = () => {
      clearTimeout(id);                                       // (b) actually stop the work
      reject(signal.reason);
    };
    signal?.addEventListener('abort', onAbort, { once: true });
  });
}
Interviewers increasingly probe cancellation because it's where real apps break (stale search results, setState-after-unmount, wasted bandwidth). If you volunteer "I'd thread an AbortSignal through this API" during any async implementation, you're speaking the language of TanStack Query, Axios v1+, and Node's own APIs — all of which standardized on it.

4. Implement: debounce & throttle Tier 1

Problem: Implement debounce(fn, wait) — returns a function that delays invoking fn until wait ms have elapsed since the last call. Then throttle(fn, wait) — invokes fn at most once per wait ms.

Clarifying questions to ask:

  • "Should the debounced function support a leading (immediate) invocation option?"
  • "Do you want a cancel() method? A flush()?"
  • "Should this and arguments be preserved?" (Yes — say it before they ask.)
  • "For throttle: leading edge, trailing edge, or both?"

Naive debounce (say: "I'll start simple, then harden it")

function debounce(fn, wait) {
  let timerId;
  return (...args) => {
    clearTimeout(timerId);           // every call resets the clock
    timerId = setTimeout(() => fn(...args), wait);
  };
}

Problems: loses this (arrow function + fn(...args) — breaks when debouncing a method), no way to cancel a pending call (unmount leak in React), no leading option.

Production debounce

function debounce(fn, wait, { leading = false, trailing = true } = {}) {
  let timerId = null;
  let lastArgs, lastThis;

  const invoke = () => {
    const args = lastArgs, thisArg = lastThis;
    lastArgs = lastThis = undefined;      // clear BEFORE calling (fn might call debounced again)
    fn.apply(thisArg, args);
  };

  function debounced(...args) {          // function keyword: we need dynamic `this`
    lastArgs = args;
    lastThis = this;
    const isLeadingCall = leading && timerId === null;

    clearTimeout(timerId);               // safe even if timerId is null
    timerId = setTimeout(() => {
      timerId = null;
      // lastArgs is undefined if the only call was the leading one —
      // that check prevents a duplicate trailing invocation
      if (trailing && lastArgs !== undefined) invoke();
    }, wait);

    if (isLeadingCall) invoke();
  }

  debounced.cancel = () => {
    clearTimeout(timerId);
    timerId = null;
    lastArgs = lastThis = undefined;
  };

  debounced.flush = () => {              // fire pending call NOW (e.g. on form submit)
    if (timerId !== null && lastArgs !== undefined) {
      clearTimeout(timerId);
      timerId = null;
      invoke();
    }
  };

  return debounced;
}

Edge cases handled (list these out loud):

  • this + args preserved via function keyword and fn.apply — el.addEventListener('input', debounce(handler, 300)) keeps this === el.
  • leading: true fires immediately on the first call of a burst; the lastArgs !== undefined guard stops a single call from firing twice (leading + trailing).
  • cancel() clears both the timer and captured args — no stale closure retained, no post-unmount call.
  • Timer cleanup: we null out timerId when the timer fires so leading detection (timerId === null) stays correct.

Throttle — timestamp + timer hybrid (leading and trailing)

Two naive approaches, each half-right: timestamp-only gives a leading call but drops the final call of a burst (a scroll handler misses the resting position); timer-only gives trailing calls but delays the first reaction. Production throttle combines both:

function throttle(fn, wait, { leading = true, trailing = true } = {}) {
  let lastInvokeTime = 0;
  let timerId = null;
  let lastArgs, lastThis;

  const invoke = (time) => {
    lastInvokeTime = time;
    const args = lastArgs, thisArg = lastThis;
    lastArgs = lastThis = undefined;
    fn.apply(thisArg, args);
  };

  function throttled(...args) {
    const now = Date.now();
    if (lastInvokeTime === 0 && !leading) lastInvokeTime = now;  // suppress leading edge
    lastArgs = args;
    lastThis = this;
    const remaining = wait - (now - lastInvokeTime);

    if (remaining <= 0) {                // window elapsed → invoke immediately (leading)
      if (timerId !== null) { clearTimeout(timerId); timerId = null; }
      invoke(now);
    } else if (trailing && timerId === null) {
      timerId = setTimeout(() => {       // schedule ONE trailing call for end of window
        timerId = null;
        invoke(leading ? Date.now() : 0);
      }, remaining);
    }
    // calls landing mid-window with a timer pending just update lastArgs —
    // the trailing call uses the FRESHEST arguments
  }

  throttled.cancel = () => {
    clearTimeout(timerId);
    timerId = null;
    lastInvokeTime = 0;
    lastArgs = lastThis = undefined;
  };

  return throttled;
}

When to use which

ScenarioToolWhy
Search-as-you-type, autosave, resize-end layoutdebounceYou only care about the final state after the user pauses
Scroll position tracking, drag, mousemove, infinite-scroll checksthrottleYou need periodic updates during continuous activity — debounce would go silent for the whole gesture
Button double-click protectionthrottle (leading, no trailing) or onceAct on the first click, swallow the burst

The React gotcha

// ❌ New debounced fn every render → its private timer is thrown away →
// nothing is actually debounced across renders
function Search() {
  const [q, setQ] = useState('');
  const search = debounce((v) => api.search(v), 300);   // recreated each render!
  ...
}

// ✅ Create once; cancel on unmount
function Search() {
  const search = useMemo(() => debounce((v) => api.search(v), 300), []);
  useEffect(() => () => search.cancel(), [search]);     // this is why cancel() exists
  ...
}
Even the useMemo version has a stale-closure risk: if the debounced callback reads props/state, it sees the values from the render where it was created. Fixes: pass everything as arguments, or keep the latest callback in a ref (useEffectEvent / the "latest ref" pattern). Raising this unprompted is a strong senior signal.

Follow-ups you may get: add maxWait (debounce that can't starve — which is how lodash implements throttle: debounce(fn, wait, { maxWait: wait })); return the result — explain why leading mode can return a value but trailing mode needs a promise.

5. Implement: Promise.all / allSettled / race polyfills Tier 1

Problem: Implement Promise.all(iterable) from scratch: resolves with an array of values in input order when all fulfill; rejects with the first rejection.

Clarifying questions:

  • "Can the iterable contain non-promise values?" (Yes — spec wraps them with Promise.resolve.)
  • "Empty iterable?" (Resolves with [] — synchronously scheduled, still async delivery.)
  • "Should results preserve input order even if promises settle out of order?" (Yes — this is the crux.)

Naive attempt — and why it's wrong

// ❌ Pushing on settle → results appear in COMPLETION order, not input order
promises.forEach((p) => p.then((v) => { results.push(v); ... }));

Fix: write each result to its index, and count completions separately — you can't use results.length as the counter because assigning to index 2 of an empty array makes length 3 with holes.

Production Promise.all

function promiseAll(iterable) {
  return new Promise((resolve, reject) => {
    const items = [...iterable];              // accept any iterable, snapshot it
    const results = new Array(items.length);
    let remaining = items.length;

    if (remaining === 0) return resolve(results);   // edge: empty → resolve []

    items.forEach((item, i) => {
      // Promise.resolve handles plain values AND foreign thenables
      Promise.resolve(item).then(
        (value) => {
          results[i] = value;                 // write by INDEX → input order preserved
          remaining -= 1;                     // count-based resolution
          if (remaining === 0) resolve(results);
        },
        reject                                // first rejection wins; later resolve() calls
      );                                      // are no-ops because promises settle once
    });
  });
}

Say while coding: "I don't need to 'cancel' the other promises on rejection — reject settles the outer promise, and the settle-once rule makes every later resolve call a no-op. And I pass reject as the second .then argument rather than .catch, since I only want upstream rejections."

allSettled and race (short — same skeleton)

function promiseAllSettled(iterable) {
  const items = [...iterable];
  return promiseAll(items.map((item) =>
    Promise.resolve(item).then(
      (value) => ({ status: 'fulfilled', value }),
      (reason) => ({ status: 'rejected', reason })
    )
  ));  // mapping every outcome to fulfillment means promiseAll can never fail-fast
}

function promiseRace(iterable) {
  return new Promise((resolve, reject) => {
    for (const item of iterable) {
      Promise.resolve(item).then(resolve, reject);  // first settle wins — that's it
    }
    // edge: empty iterable → stays pending FOREVER (matches the spec)
  });
}

Follow-ups: Promise.any (inverted counting + AggregateError — solved in drills); "make promiseAll accept a concurrency limit" (that's the pool).

6. Implement: promisify Tier 1

Problem: Convert a Node-style callback function — fn(...args, (err, result) => {}) — into one that returns a promise.

Clarifying questions: "Error-first callback convention? Can the callback receive multiple success values? Should this be preserved (method on an object)?"

function promisify(fn) {
  return function promisified(...args) {        // function keyword → preserves `this`
    return new Promise((resolve, reject) => {
      fn.call(this, ...args, (err, result) => { // append the callback LAST
        if (err != null) reject(err);            // error-first convention
        else resolve(result);
      });
    });
  };
}

// Usage:
const readFileAsync = promisify(fs.readFile);
const data = await readFileAsync('config.json', 'utf8');

Edge cases: this preserved (promisifying db.query must keep db); err != null (loose) treats both null and undefined as success; if fn throws synchronously, the executor is already running, so the throw is caught by the promise machinery and becomes a rejection — free correctness, worth pointing out.

Follow-ups: multi-value callbacks ((err, a, b) → resolve with array — Node's util.promisify uses a symbol for custom behavior); reverse direction, "callbackify".

7. Implement: EventEmitter Tier 1

Problem: Implement a class with on(event, fn), off(event, fn), once(event, fn), emit(event, ...args).

Clarifying questions:

  • "If a handler calls off during an emit, should removal affect the current emit?" (Convention: no — snapshot the list.)
  • "Should on return an unsubscribe function?" (Modern API design — do it.)
  • "Duplicate handler registered twice — called twice?" (Node says yes; simplest and I'll match it.)
  • "Should a throwing handler stop the others?" (Discuss; default: let it throw, or isolate with try/catch.)
class EventEmitter {
  #events = new Map();   // Map<string, Array<fn>> — Map, not {}, so no
                         // prototype-key collisions ('constructor', '__proto__')
  on(event, handler) {
    if (typeof handler !== 'function') throw new TypeError('handler must be a function');
    const list = this.#events.get(event) ?? [];
    list.push(handler);
    this.#events.set(event, list);
    return () => this.off(event, handler);   // unsubscribe fn — the modern contract
  }

  off(event, handler) {
    const list = this.#events.get(event);
    if (!list) return this;
    // match the handler itself OR a once-wrapper wrapping it
    const i = list.findIndex((h) => h === handler || h.original === handler);
    if (i !== -1) list.splice(i, 1);          // remove ONE occurrence (Node semantics)
    if (list.length === 0) this.#events.delete(event);  // don't leak empty arrays
    return this;
  }

  once(event, handler) {
    const wrapper = (...args) => {
      this.off(event, wrapper);               // remove BEFORE calling —
      handler.apply(this, args);              // re-entrant emit inside handler can't double-fire
    };
    wrapper.original = handler;               // so off(event, handler) can find it
    return this.on(event, wrapper);
  }

  emit(event, ...args) {
    const list = this.#events.get(event);
    if (!list || list.length === 0) return false;
    for (const handler of [...list]) {        // SNAPSHOT: on/off during emit can't
      handler.apply(this, args);              // skip or double-call handlers mid-loop
    }
    return true;
  }
}

The two subtle bugs this design avoids — narrate them:

  • Mutation during emit: without the [...list] snapshot, a handler calling off shifts the array under the loop and the next handler gets skipped. This is a real bug class in hand-rolled emitters.
  • once + off interop: users hold a reference to their handler, not your wrapper. Tagging wrapper.original lets off(event, theirHandler) still work.
Memory-leak discussion (expect it as a follow-up): the emitter holds strong references to every handler, and handlers close over their subscribers — a long-lived emitter (app-wide bus) subscribed to by short-lived components keeps those components, their DOM, and their state alive forever. Mitigations: always return/store unsubscribe functions and call them on teardown (React: in the effect cleanup); support { signal } in on like addEventListener does; a removeAllListeners(event) escape hatch; and Node's max-listeners warning exists precisely to surface forgotten unsubscribes.

Follow-ups: wildcard events (on('*', fn) — check a second list in emit); async handlers (emit returning Promise.all of results); priority ordering (solved in drills).

8. Implement: deep clone Tier 1

Problem: deepClone(value) — a fully independent copy: mutating the clone never affects the original.

Clarifying questions:

  • "Which types must I support — just JSON-ish data, or Date, RegExp, Map, Set?"
  • "Can the input contain circular references?" (Ask this one always — it's the trap.)
  • "Functions, DOM nodes, class instances — clone, copy by reference, or throw?"
  • "Can I just use structuredClone?" (Great question — implement anyway, then compare.)

Naive: JSON round-trip — know exactly why it fails

const clone = JSON.parse(JSON.stringify(obj));
// Loses: undefined & functions (dropped), Date (→ string), Map/Set (→ {}),
// NaN/Infinity (→ null), BigInt (throws). Circular refs: THROWS.
// Fine for plain config objects; wrong as a general answer.

Production deepClone

function deepClone(value, seen = new WeakMap()) {
  // 1. Primitives & functions: returned as-is.
  //    (typeof null === 'object' — hence the explicit null check)
  if (value === null || typeof value !== 'object') return value;

  // 2. Cycles: if we've cloned this exact object before, reuse that clone.
  //    WeakMap so the bookkeeping doesn't prevent garbage collection.
  if (seen.has(value)) return seen.get(value);

  // 3. Built-ins with internal state (not copiable via key iteration):
  if (value instanceof Date)   return new Date(value.getTime());
  if (value instanceof RegExp) return new RegExp(value.source, value.flags);

  if (value instanceof Map) {
    const result = new Map();
    seen.set(value, result);               // register BEFORE recursing (cycle safety)
    for (const [k, v] of value) result.set(deepClone(k, seen), deepClone(v, seen));
    return result;
  }

  if (value instanceof Set) {
    const result = new Set();
    seen.set(value, result);
    for (const item of value) result.add(deepClone(item, seen));
    return result;
  }

  if (Array.isArray(value)) {
    const result = [];
    seen.set(value, result);
    for (let i = 0; i < value.length; i++) result[i] = deepClone(value[i], seen);
    return result;
  }

  // 4. Plain objects — preserve the prototype (class instances keep methods)
  const result = Object.create(Object.getPrototypeOf(value));
  seen.set(value, result);
  for (const key of Reflect.ownKeys(value)) {   // string keys + symbols
    result[key] = deepClone(value[key], seen);
  }
  return result;
}

The critical line to narrate: seen.set(value, result) happens before recursing into children. For a self-referencing object (a.self = a), the recursive call finds the half-built clone in seen and wires the cycle correctly instead of recursing forever.

Edge cases handled: primitives, null, functions (by reference — cloning a closure is impossible in general), Date/RegExp/Map/Set, arrays (incl. as Map values), circular and shared references (two properties pointing at the same object stay the same clone — seen gives you this for free), symbol keys, prototypes.

structuredClone — the modern answer

Say this proactively: "In production I'd reach for structuredClone(value) — a native, spec'd deep clone that handles cycles, Date, Map, Set, ArrayBuffer, typed arrays, Blob, and more. It's the same algorithm used for postMessage and IndexedDB." Then show you know its limits:

  • Throws (DataCloneError) on: functions, DOM nodes, class instances with methods you care about — actually it clones own data properties but the prototype is lost (clone of a class User instance is a plain object), plus Error subclass specifics, WeakMap/WeakRef, property getters (invoked, result cloned — accessor-ness lost).
  • So the hand-rolled version still earns its keep for class instances — which is why the interviewer asks anyway.

Follow-ups: depth limit / max-nodes guard; clone with a transform callback; "how would you deep-freeze instead?" (same traversal, Object.freeze, no cycle map needed if you check Object.isFrozen).

function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {         // fn.length = declared parameter count
      return fn.apply(this, args);          // enough args → actually invoke
    }
    // not enough yet → return a collector that accumulates more
    return (...next) => curried.apply(this, [...args, ...next]);
  };
}

Step 3: bounded cache — LRU via Map's ordering trick

Unbounded memoization is a memory leak with good PR. The classic fix is least-recently-used eviction. The elegant JS implementation exploits that Map preserves insertion order: delete + re-insert moves a key to the "back", so the first key is always the least recently used.

class LRUCache {
  #map = new Map();
  constructor(capacity) {
    if (capacity < 1) throw new RangeError('capacity must be ≥ 1');
    this.capacity = capacity;
  }

  get(key) {
    if (!this.#map.has(key)) return undefined;
    const value = this.#map.get(key);
    this.#map.delete(key);          // re-insert → moves key to most-recent position
    this.#map.set(key, value);
    return value;
  }

  set(key, value) {
    if (this.#map.has(key)) {
      this.#map.delete(key);        // updating counts as a "use" too
    } else if (this.#map.size >= this.capacity) {
      const oldest = this.#map.keys().next().value;   // FIRST key = LRU
      this.#map.delete(oldest);
    }
    this.#map.set(key, value);
  }

  has(key) { return this.#map.has(key); }   // design choice: has() does NOT refresh recency
}

// Plug it in: memoize with bounded memory
function memoizeLRU(fn, capacity = 100, keyFn = (...a) => JSON.stringify(a)) {
  const cache = new LRUCache(capacity);
  return function (...args) {
    const key = keyFn(...args);
    if (cache.has(key)) return cache.get(key);
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

Say while coding: "All operations are O(1) — Map is a hash table plus an insertion-order linked list, so I get the doubly-linked-list-plus-hashmap LRU design from the classic LeetCode problem for free." Follow-ups: TTL expiry (store {value, expiresAt}); cache statistics; memoize.clear().

// Recursive with depth — mirrors Array.prototype.flat(depth)
function flatten(arr, depth = 1) {
  const result = [];
  for (const item of arr) {
    if (Array.isArray(item) && depth > 0) {
      result.push(...flatten(item, depth - 1));
    } else {
      result.push(item);
    }
  }
  return result;
}
flatten([1, [2, [3, [4]]]], Infinity);   // [1, 2, 3, 4]

// Iterative with an explicit stack — no recursion depth limit
function flattenIterative(arr) {
  const stack = [...arr];
  const result = [];
  while (stack.length > 0) {
    const item = stack.pop();                // take from the END (that's the twist)
    if (Array.isArray(item)) stack.push(...item);
    else result.push(item);
  }
  return result.reverse();                   // pop() walked right-to-left → reverse once
}
The iterative version's reverse() is where people slip: popping from the stack processes elements right-to-left, so results come out backwards. Alternatives — stack.shift() keeps order but each shift is O(n), or push children reversed. One final reverse() is the clean O(n) answer.

sleep — abortable

const sleep = (ms, { signal } = {}) =>
  new Promise((resolve, reject) => {
    if (signal?.aborted) return reject(signal.reason);
    const id = setTimeout(() => {
      signal?.removeEventListener('abort', onAbort);   // cleanup — no listener leak
      resolve();
    }, ms);
    const onAbort = () => {
      clearTimeout(id);                                 // actually cancel the timer
      reject(signal.reason);
    };

retry — exponential backoff + jitter + AbortSignal (production-grade)

Clarify: "Retry on every error, or only retryable ones (network/5xx/429 — never 400)? Max attempts? Should waits be cancellable?"

async function retry(task, {
  retries = 3,             // retries AFTER the first attempt → up to 4 calls total
  baseDelay = 200,
  maxDelay = 10_000,
  shouldRetry = () => true,
  signal,
  onRetry,                 // observability hook: (err, attempt) => log/metric
} = {}) {
  for (let attempt = 0; ; attempt++) {
    try {
      return await task({ attempt, signal });   // pass signal down so the WORK is abortable too
    } catch (err) {
      const outOfAttempts = attempt >= retries;
      if (outOfAttempts || !shouldRetry(err) || signal?.aborted) throw err;

      // Exponential: 200, 400, 800, ... capped at maxDelay
      const backoff = Math.min(maxDelay, baseDelay * 2 ** attempt);
      // "Equal jitter": half fixed + half random. Without jitter, every client
      // that failed together retries together — a synchronized thundering herd
      // that re-kills the recovering server.
      const delay = backoff / 2 + Math.random() * (backoff / 2);

      onRetry?.(err, attempt + 1);
      await sleep(delay, { signal });    // abortable wait — abort during backoff rejects here
    }
  }
}

// Usage — retry transient failures only, cancellable by the caller:
const data = await retry(
  ({ signal }) => fetchJson('/api/report', { signal }),
  { retries: 4, shouldRetry: (e) => e.status === 429 || e.status >= 500, signal: controller.signal }
);

Edge cases to narrate: abort during the sleep (handled — sleep rejects), abort during the task (handled — signal passed down; and shouldRetry should return false for AbortError), non-retryable errors thrown immediately, delay cap so attempt 10 isn't a 3-minute wait. Bonus: honor a Retry-After header when present.

Concurrency pool — run N at a time

Problem: given 100 task functions and a limit of 5, run at most 5 concurrently; resolve with all results in input order.

Clarify: "Tasks are functions returning promises, right? (If I'm handed promises, the work has already started — nothing to limit.) Fail-fast like Promise.all, or collect all outcomes?"

// The elegant worker-pool formulation — memorize this one.
async function promisePool(tasks, limit) {
  const results = new Array(tasks.length);
  let nextIndex = 0;

  async function worker() {
    // Each worker pulls the next unclaimed task until none remain.
    while (nextIndex < tasks.length) {
      const i = nextIndex++;       // read+increment is safe: JS is single-threaded and
                                   // there's no await between the read and the write
      results[i] = await tasks[i]();
    }
  }

  // Spawn min(limit, tasks.length) workers; each is an async fn = a promise.
  const workers = Array.from({ length: Math.min(limit, tasks.length) }, worker);
  await Promise.all(workers);      // also propagates the first rejection (fail-fast)
  return results;
}

// Usage:
const tasks = urls.map((url) => () => fetchJson(url));   // note: thunks, not promises
const data = await promisePool(tasks, 5);

Why this is correct — the three-sentence proof to give: "Each worker is a sequential loop, so a worker never runs two tasks at once; with K workers, at most K tasks are in flight. Workers claim indices from a shared counter, and because JavaScript is single-threaded with no await between reading and incrementing nextIndex, no two workers can claim the same index. Results are written by index, so order is preserved regardless of completion order."

Follow-up: a reusable pLimit-style limiter (limit shared across independent call sites — how the p-limit package works):

function pLimit(concurrency) {
  const queue = [];
  let active = 0;

  const runNext = () => {
    if (active >= concurrency || queue.length === 0) return;
    active++;
    const { fn, resolve, reject } = queue.shift();
    fn().then(resolve, reject)
        .finally(() => { active--; runNext(); });  // a finishing task admits the next one
  };

  // Returns a function that schedules fn under the shared limit
  return (fn) => new Promise((resolve, reject) => {
    queue.push({ fn, resolve, reject });
    runNext();
  });
}

const limit = pLimit(3);
const results = await Promise.all(urls.map((u) => limit(() => fetchJson(u))));

Edge cases: limit ≥ tasks.length degrades to plain Promise.all; empty task list → resolves []; a rejecting task rejects the pool but already-started tasks keep running (same caveat as Promise.all — combine with AbortSignal to actually stop them); errors in pLimit still trigger finally, so the slot is always released — no starvation.

Five components mount and each fetches /api/me → five identical requests. Deduplication shares one in-flight promise per key — this is the core trick inside TanStack Query and SWR (same key ⇒ same in-flight promise ⇒ one network call, many subscribers).

const inFlight = new Map();

function dedupedFetch(url) {
  const existing = inFlight.get(url);
  if (existing) return existing;           // join the request already in progress

  const promise = fetch(url)
    .then((res) => {
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return res.json();
    })
    .finally(() => inFlight.delete(url));  // CRITICAL: delete on settle — success or failure.

                                            // replaying its rejection to every future caller.

Polling with backoff + visibility pause

function poll(task, { interval = 5000, maxInterval = 60_000, signal } = {}) {
  let delay = interval;
  (async () => {
    while (!signal?.aborted) {
      // Pause while tab is hidden — don't burn battery/server on background tabs
      if (document.visibilityState === 'hidden') {
        await new Promise((resolve) => document.addEventListener(
          'visibilitychange', resolve, { once: true, signal }));
        if (signal?.aborted) break;
      }
      try {
        await task({ signal });
        delay = interval;                        // success → reset backoff
      } catch {
        delay = Math.min(delay * 2, maxInterval); // failure → back off (recovering server)
      }
      try { await sleep(delay, { signal }); } catch { break; }  // abort during sleep exits
    }
  })();
}

15. Interview Q&A

Q1. What are the possible states of a promise, and what does "settled once" guarantee?

Pending, fulfilled, rejected. A promise transitions at most once, permanently; subsequent resolve/reject calls are ignored. This guarantee is load-bearing: Promise.race and the timeout pattern work because the first settle wins and the losers' calls are no-ops. It also means attaching .then after settlement still works — the handler is scheduled as a microtask with the stored value.

Q2. Why is .then(onSuccess, onError) different from .then(onSuccess).catch(onError)?

The two-argument form's onError only handles rejections of the upstream promise — an error thrown inside onSuccess is not caught by its sibling. The chained .catch handles both upstream rejections and errors thrown in onSuccess, because it sits downstream of the promise .then returns. Use the two-arg form when you deliberately don't want the error handler to mask handler bugs (e.g., inside a Promise.all polyfill); use chained .catch as the everyday default.

Q3. What exactly happens at an await expression?

The operand is wrapped as if by Promise.resolve(). The function suspends; the remainder of its body becomes a reaction on that promise, scheduled onto the microtask queue when it settles. Control returns synchronously to the caller. If the promise rejects, resumption throws the reason at the await site. Even await 42 suspends for at least one microtask tick — which is why inserting an await can change observable ordering.

Q4. When is await in a loop correct, and when is it a bug?

It's a bug when iterations are independent — you serialize work that could overlap, turning N×100ms of parallel I/O into sequential. Use Promise.all(items.map(fn)). It's correct when each iteration depends on the previous result (pagination cursors), when order/side-effect sequencing matters (migrations), or when you must limit pressure on a resource — though for "some but not unlimited parallelism" the right tool is a concurrency pool.

Q5. Promise.all vs allSettled vs race vs any — one-line discriminator for each?

all: all-or-nothing, fail-fast on first rejection, ordered results. allSettled: never rejects, gives every outcome as {status, value|reason} — for independent operations. race: first to settle wins, fulfillment or rejection — for timeouts. any: first to fulfill wins, ignores rejections until all fail, then rejects with AggregateError (inspect .errors) — for redundant sources.

Q6. Does Promise.all cancel the remaining promises when one rejects?

No. Promises are not cancellable; Promise.all merely stops waiting — every other operation continues to completion, consuming network/CPU, its result discarded. For real cancellation, pair the combinator with an AbortController: on first failure, call controller.abort() and have each task honor the shared signal.

Q7. How does AbortController work, and how do you make your own function abortable?

A controller owns a signal; abort(reason) sets signal.aborted, stores signal.reason, and fires the 'abort' event. Consumers like fetch reject with an AbortError DOMException. To make your own function abortable, follow the contract: (1) check signal.aborted up front and fail immediately; (2) add an 'abort' listener that stops the underlying work (clear timers, close connections) and rejects with signal.reason; (3) remove the listener on completion. Also know AbortSignal.timeout(ms) and AbortSignal.any([...]) for composing timeout + user cancellation.

Q8. Debounce vs throttle — definitions and one canonical use case each?

Debounce fires after activity stops: it postpones invocation until wait ms pass with no new calls — search-as-you-type, autosave. Throttle fires during activity at a bounded rate: at most once per window — scroll/mousemove handlers where you need periodic updates throughout the gesture. Litmus test: "do I want the final state after a pause (debounce) or a steady stream of samples (throttle)?"

Q9. Why does debounce(fn, 300) inside a React component body silently do nothing, and what's the fix?

Every render creates a new debounced function with a fresh internal timer, so the "previous call" it's supposed to supersede belongs to a different closure — nothing is debounced across renders (and the old timers still fire). Fix: create it once with useMemo(() => debounce(fn, 300), []) or store it in a ref, and call .cancel() in the effect cleanup. Then handle the stale-closure follow-up: pass changing values as arguments or read them from a "latest" ref.

Q10. In a Promise.all polyfill, why can't you use results.push(value) and results.length as the completion check?

Push ordering follows completion order, not input order — a fast third promise lands at index 0. Writing to results[i] fixes order but breaks length as a counter: assigning index 2 of an empty array yields length 3 with holes, so you'd resolve early with sparse results. Hence the two separate mechanisms: index-addressed writes for ordering, a decrementing remaining counter for completion.

Q11. In an EventEmitter, why must emit iterate over a copy of the handler list?

Handlers may call on/off during the emit. Mutating the array mid-iteration causes skipped or double-invoked handlers (removing index 1 while the loop is at index 1 shifts index 2 into the current slot). Snapshotting ([...list]) freezes the invocation set at emit time: removals apply to future emits, and a once wrapper that unsubscribes itself can't corrupt the loop.

Q12. How do event emitters cause memory leaks, and what are the mitigations?

The emitter holds strong references to handlers; handlers close over their subscriber's scope. A long-lived emitter (app singleton) + short-lived subscribers (components) that never unsubscribe = components, their DOM, and captured state stay reachable forever. Mitigations: return unsubscribe functions from on and call them on teardown (React effect cleanup); accept { signal } so one abort detaches everything; once for one-shots; leak canaries like Node's max-listeners warning. WeakRef-based handler storage is possible but rarely worth the complexity.

Q13. What does JSON.parse(JSON.stringify(x)) get wrong as a deep clone?

undefined, functions, and symbols are dropped (or become null in arrays); Date becomes an ISO string; Map/Set/RegExp become {}; NaN/Infinity become null; BigInt throws; getters are evaluated and flattened; prototypes/class identity are lost; and circular references throw TypeError. It's acceptable only for known plain-JSON data — and even then structuredClone is now the better native answer.

Q14. What can structuredClone handle that JSON can't — and what can't it clone?

Handles: circular references, Date, RegExp, Map, Set, ArrayBuffer/typed arrays, Blob/File, Error objects — the structured clone algorithm shared with postMessage and IndexedDB. Can't: functions and DOM nodes (throws DataCloneError), WeakMap/WeakSet, property descriptors/getters (getter results are cloned as plain values), and prototypes — a class instance comes back as a plain object with its data but not its methods. That last one is why hand-rolled clones still exist.

Q15. Why is a WeakMap the right structure for both cycle-tracking in deepClone and object-keyed memoization?

WeakMap keys are held weakly: the map doesn't prevent its keys from being garbage collected, and entries vanish when keys die. In deepClone the seen map lives only for the call, but WeakMap is still semantically right (bookkeeping shouldn't extend lifetimes) — and doubles as the cycle detector: registering the clone before recursing means a self-reference finds the in-progress clone instead of recursing infinitely. In memoization it eliminates the classic leak where a cache keyed by objects pins every object (and its results) in memory forever. Trade-off: no iteration, no size, no clear — by design, since GC timing is unobservable.

Q16. How can a Map implement an O(1) LRU cache?

Map preserves insertion order and is internally a hash table plus an order list, so: on get, delete and re-insert the key — it moves to the most-recent end in O(1). On set at capacity, evict map.keys().next().value — the first key, i.e., least recently used. This is exactly the hashmap + doubly-linked-list LRU design, with the linked list provided by Map's insertion-order guarantee.

Q17. What's the danger in memoizing an async function, and the fix?

You cache the promise, which is great for deduplicating concurrent calls — but if the promise rejects, the rejected promise stays cached and every future call replays the failure without ever retrying. Fix: evict on rejection — cache.set(key, p); p.catch(() => cache.delete(key));. Related design questions: should in-flight and settled entries have different lifetimes (TanStack Query's staleTime vs gcTime), and do you need TTL on successes too?

Q18. Explain the stale-response race in a typeahead and rank the fixes.

Two keystrokes launch two requests; the network doesn't guarantee response order, so the older request can resolve last and overwrite newer results. Fixes, weakest to strongest: (1) ignore-stale closure flag in the effect — simple, but the wasted request completes anyway; (2) monotonic token / takeLatest wrapper — generic, framework-free; (3) AbortController per request, aborting the previous — also cancels network work and composes with fetch natively; (4) a data-fetching layer (TanStack Query) that keys requests, dedupes, cancels, and caches — what you'd actually ship. Bonus: debouncing input reduces the race window but does not eliminate it — it's a complement, not a fix.

Q19. Why does exponential backoff need jitter?

When a server blips, all its clients fail simultaneously. With deterministic backoff they all retry simultaneously too — at 200ms, 400ms, 800ms — arriving as synchronized waves ("thundering herd") that can re-topple the recovering service. Randomizing each client's delay (e.g., equal jitter: backoff/2 + random(backoff/2), or full jitter: random(backoff)) decorrelates the waves and spreads load flat. Also cap the delay, retry only retryable errors (5xx/429/network — never 400), and honor Retry-After.

Q20. In the worker-pool concurrency limiter, why is const i = nextIndex++ safe without locks?

JavaScript executes on a single thread with run-to-completion semantics: code between awaits never interleaves. The read of nextIndex and its increment happen in one synchronous step with no await between them, so no other worker can observe the intermediate state — two workers can't claim the same index. The general lesson: in JS, race conditions live at await boundaries, not between adjacent synchronous statements. If the claim spanned an await (const i = nextIndex; await something; nextIndex = i + 1), it would be a genuine race.

Drill 1 — Implement Promise.any

Resolve with the first fulfillment; if all reject, reject with an AggregateError whose .errors preserves input order. Handle empty iterables and non-promise values.

Solution & notes 
function promiseAny(iterable) {
  return new Promise((resolve, reject) => {
    const items = [...iterable];
    const errors = new Array(items.length);
    let remaining = items.length;

    // Edge: empty input → immediate AggregateError (per spec)
    if (remaining === 0) {
      return reject(new AggregateError([], 'All promises were rejected'));
    }

    items.forEach((item, i) => {
      Promise.resolve(item).then(
        resolve,                            // first fulfillment wins (settle-once)
        (reason) => {
          errors[i] = reason;               // index-addressed → error order = input order
          if (--remaining === 0) {
            reject(new AggregateError(errors, 'All promises were rejected'));
          }
        }
      );
    });
  });
}

Note the symmetry with Promise.all: the counting logic is identical with success and failure roles swapped. Saying that sentence in the interview shows you understand the pattern rather than memorized four polyfills.

Drill 2 — Cancellable fetch wrapper with timeout

Implement fetchWithTimeout(url, { timeout, signal, ...init }): rejects with TimeoutError after timeout ms, still cancellable by the caller's own signal, and throws on non-2xx responses.

Solution & notes 
// Modern version — compose signals declaratively
async function fetchWithTimeout(url, { timeout = 8000, signal, ...init } = {}) {
  const timeoutSignal = AbortSignal.timeout(timeout);      // aborts with 'TimeoutError'
  const combined = signal
    ? AbortSignal.any([signal, timeoutSignal])             // EITHER aborts the fetch
    : timeoutSignal;

  const res = await fetch(url, { ...init, signal: combined });
  if (!res.ok) {
    const err = new Error(`HTTP ${res.status} for ${url}`);
    err.status = res.status;                               // lets retry() decide retryability
    throw err;
  }
  return res;
}

// Manual fallback if AbortSignal.any is off the table — same idea by hand:
async function fetchWithTimeoutManual(url, { timeout = 8000, signal, ...init } = {}) {
  const controller = new AbortController();
  const timerId = setTimeout(
    () => controller.abort(new DOMException('Timed out', 'TimeoutError')), timeout);
  const onCallerAbort = () => controller.abort(signal.reason);
  signal?.addEventListener('abort', onCallerAbort, { once: true });
  try {
    const res = await fetch(url, { ...init, signal: controller.signal });
    if (!res.ok) throw Object.assign(new Error(`HTTP ${res.status}`), { status: res.status });
    return res;
  } finally {
    clearTimeout(timerId);                                 // cleanup on EVERY path
    signal?.removeEventListener('abort', onCallerAbort);
  }
}

Callers distinguish outcomes by err.name: 'TimeoutError' (retry candidate), 'AbortError' (user cancelled — ignore silently), HTTP error with .status (retry only 5xx/429).

Drill 3 — Pipe for async functions

Implement asyncPipe(...fns) where each stage may be sync or async, each receives the previous stage's resolved value, and the first rejection short-circuits.

Solution & notes 
const asyncPipe = (...fns) => (input) =>
  fns.reduce(
    (acc, fn) => acc.then(fn),        // .then adopts fn's result whether it's a value
    Promise.resolve(input)            // or a promise (chaining rule 3) — sync and async
  );                                  // stages need zero special-casing

// Equivalent, arguably clearer for narration:
const asyncPipe2 = (...fns) => async (input) => {
  let value = input;
  for (const fn of fns) value = await fn(value);   // sequential BY DESIGN here
  return value;
};

const loadUserPosts = asyncPipe(
  (id) => fetchJson(`/users/${id}`),
  (user) => user.postsUrl,            // sync stage mixes in seamlessly
  (url) => fetchJson(url)
);

The one-liner works because .then's adoption rule erases the sync/async distinction. Error propagation is free: a rejection skips remaining .thens and surfaces to the caller. Follow-up: add tap/trace stages, or an AbortSignal threaded through each stage.

Drill 4 — Rate limiter: at most N calls per second

Implement rateLimit(fn, limit, interval): the wrapper accepts unlimited calls but executes at most limit per sliding interval, queueing the excess in order. Each call returns a promise for its (eventual) result.

Solution & notes 
function rateLimit(fn, limit, interval = 1000) {
  const queue = [];        // pending invocations, FIFO
  const timestamps = [];   // start times of executions in the current window

  const process = () => {
    const now = Date.now();
    // 1. Slide the window: drop timestamps older than `interval`
    while (timestamps.length > 0 && now - timestamps[0] >= interval) {
      timestamps.shift();
    }
    // 2. Execute while we have both queued calls and window capacity
    while (queue.length > 0 && timestamps.length < limit) {
      const { args, resolve, reject } = queue.shift();
      timestamps.push(Date.now());
      // Promise.resolve normalizes sync returns AND sync throws into the promise
      Promise.resolve().then(() => fn(...args)).then(resolve, reject);
    }
    // 3. Still queued? Wake up exactly when the oldest timestamp expires
    if (queue.length > 0) {
      setTimeout(process, interval - (Date.now() - timestamps[0]));
    }
  };

  return (...args) =>
    new Promise((resolve, reject) => {
      queue.push({ args, resolve, reject });
      process();
    });
}

const limitedSearch = rateLimit(searchApi, 5, 1000);  // ≤ 5 calls/sec, extras queue up

Design notes to narrate: this limits start rate, not concurrency — pair with pLimit if you need both. process() is idempotent (it re-derives everything from state), so an extra timer firing is harmless. Distinguish from throttle: throttle drops intermediate calls; a rate limiter defers them — every call eventually runs.

Drill 5 — Implement once(fn)

The returned function invokes fn at most once; subsequent calls return the first result without calling fn again. Preserve this and arguments. Discuss: what if the first call throws?

Solution & notes 
function once(fn) {
  let called = false;
  let result;
  return function (...args) {
    if (!called) {
      called = true;                   // set BEFORE calling — if fn (re-)enters the
      result = fn.apply(this, args);   // wrapper, it won't recurse infinitely
    }
    return result;
  };
}

const init = once(() => expensiveSetup());
init(); init(); init();               // setup runs exactly once

Design decisions to surface: (1) a boolean flag rather than result === undefined — fn may legitimately return undefined; (2) if the first call throws, this implementation still marks called = true, so the failure is permanent (and later calls return undefined) — for an initializer you often want "throw = didn't count, allow retry": move called = true after the call, or cache and rethrow the error deliberately. Asking the interviewer which semantics they want IS the senior answer. (3) For async fn, the memoize-rejection caveat applies: cache the promise, evict on rejection if retries should be possible.

Drill 6 — Pub-sub with priority

Extend the emitter: on(event, handler, priority = 0) — on emit, higher-priority handlers run first; equal priorities run in subscription order. A handler may call stopPropagation() to prevent lower-priority handlers from running.

Solution & notes 
class PriorityEmitter {
  #events = new Map();   // Map<string, Array<{handler, priority, seq}>>
  #seq = 0;

  on(event, handler, priority = 0) {
    const list = this.#events.get(event) ?? [];
    list.push({ handler, priority, seq: this.#seq++ });
    // Sort: priority DESC, then seq ASC. Array.prototype.sort is stable,
    // but the explicit seq tiebreak makes the guarantee self-documenting.
    list.sort((a, b) => b.priority - a.priority || a.seq - b.seq);
    this.#events.set(event, list);
    return () => this.off(event, handler);
  }

  off(event, handler) {
    const list = this.#events.get(event);
    if (!list) return;
    const i = list.findIndex((e) => e.handler === handler);
    if (i !== -1) list.splice(i, 1);
    if (list.length === 0) this.#events.delete(event);
  }

  emit(event, payload) {
    const list = this.#events.get(event);
    if (!list) return false;
    let stopped = false;
    const stopPropagation = () => { stopped = true; };
    for (const { handler } of [...list]) {       // snapshot, as always
      if (stopped) break;
      handler(payload, { stopPropagation });
    }
    return true;
  }
}

// Usage: validation (10) runs before analytics (0); can veto them.
bus.on('form:submit', validate, 10);
bus.on('form:submit', trackAnalytics, 0);

Trade-off worth stating: sorting on subscribe is O(n log n) per on but keeps emit a straight scan — right for the common emit-heavy workload. (Insertion via binary search would be O(n) per subscribe; either is fine, but say why you chose.) stopPropagation mirrors DOM event semantics, which interviewers appreciate as prior art.

Drill 7 — JSON.stringify for a subset

Support objects, arrays, strings, numbers, booleans, null. Match real semantics: undefined/functions are dropped from objects but become null in arrays; non-finite numbers become null. Bonus: cycle detection.

Solution & notes 
function stringify(value, seen = new WeakSet()) {
  if (value === null) return 'null';

  const t = typeof value;
  if (t === 'number')  return Number.isFinite(value) ? String(value) : 'null';  // NaN/Infinity → null
  if (t === 'boolean') return String(value);
  if (t === 'string')  return quote(value);
  // undefined / function / symbol → the SENTINEL undefined:
  // callers decide (drop key in objects, 'null' in arrays) — real JSON.stringify semantics
  if (t !== 'object')  return undefined;

  if (seen.has(value)) throw new TypeError('Converting circular structure to JSON');
  seen.add(value);

  let json;
  if (Array.isArray(value)) {
    json = `[${value.map((v) => stringify(v, seen) ?? 'null').join(',')}]`;
  } else {
    const entries = [];
    for (const key of Object.keys(value)) {        // own enumerable string keys only
      const s = stringify(value[key], seen);
      if (s !== undefined) entries.push(`${quote(key)}:${s}`);  // drop undefined-valued keys
    }
    json = `{${entries.join(',')}}`;
  }
  seen.delete(value);   // only reject CYCLES; the same object appearing twice as a sibling is legal
  return json;
}

function quote(s) {
  // minimal escaping for the subset: backslash, quote, common control chars
  return `"${s.replace(/[\\"]/g, (c) => '\\' + c)
              .replace(/\n/g, '\\n').replace(/\r/g, '\\r').replace(/\t/g, '\\t')}"`;
}

The interesting 20%: the undefined-as-sentinel protocol (exactly how the real API signals "omit this"), array vs object treatment of unserializable values, and seen.delete after recursion — a WeakSet you never delete from would wrongly reject DAGs (shared but acyclic references), which real JSON.stringify allows. Skipped, and worth listing as known omissions: toJSON(), replacer, indent, unicode edge escaping.

Drill 8 — getByPath / setByPath (lodash get/set)

get(obj, 'a.b[2].c', default?) — walk the path, return default if any hop is missing. set(obj, 'a.b[2].c', value) — walk and create intermediate containers (array when the next key is an index, else object).

Solution & notes 
// Shared: 'a.b[2].c' → ['a','b','2','c'] (also accepts a pre-split array)
const toKeys = (path) =>
  Array.isArray(path)
    ? path
    : path.replace(/\[(\d+)\]/g, '.$1')   // [2] → .2
          .split('.')
          .filter(Boolean);               // drops empties from leading '.' etc.

function get(obj, path, defaultValue) {
  let current = obj;
  for (const key of toKeys(path)) {
    if (current == null) return defaultValue;  // == null catches null AND undefined —
    current = current[key];                     // can't read properties of either
  }
  return current === undefined ? defaultValue : current;
  // note: an explicit null result is returned as-is (lodash semantics) —
  // only undefined triggers the default
}

function set(obj, path, value) {
  const keys = toKeys(path);
  if (keys.length === 0) return obj;
  let current = obj;
  for (let i = 0; i < keys.length - 1; i++) {
    const key = keys[i];
    if (typeof current[key] !== 'object' || current[key] === null) {
      // create the missing container; look AHEAD to pick its type
      current[key] = /^\d+$/.test(keys[i + 1]) ? [] : {};
    }
    current = current[key];
  }
  current[keys.at(-1)] = value;
  return obj;
}

get({ a: { b: [{ c: 3 }] } }, 'a.b[0].c');      // 3
get({}, 'a.b.c', 'fallback');                   // 'fallback'
set({}, 'a.b[1].c', 5);                         // { a: { b: [ <hole>, { c: 5 } ] } }

Edge cases that earn points: current == null mid-path (the whole reason get exists), null-vs-undefined default semantics, the look-ahead array-vs-object decision in set, and paths as arrays (which also sidesteps keys containing dots). Security note for real code: guard set against __proto__/constructor/prototype path segments — unguarded setByPath is the canonical prototype pollution vulnerability, and mentioning it unprompted is a strong senior signal.
