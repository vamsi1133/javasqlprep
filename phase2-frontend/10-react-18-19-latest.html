10 — React 18/19 & the Latest

Concurrent rendering, Suspense, Server Components, Actions, and the Compiler — the currency check that separates "7 years of React" from "7 years of 2019 React".

Why interviewers ask this

At 7 years of experience, nobody doubts you can write a reducer. What interviewers do doubt — because it's true of a large fraction of senior candidates — is whether your mental model has kept up with the two biggest shifts in React's history: concurrent rendering (React 18) and the server/client component split plus Actions (React 19). Many candidates hand-wave "concurrent mode makes things faster" or claim "Server Components are just SSR" — both are wrong, and interviewers know these are the exact traps.

Being crisp here is a genuine differentiator. You want to be the candidate who can say precisely what a transition does under the hood, why 'use client' is a boundary and not a location directive, and what the React Compiler does to your carefully hand-tuned useMemo calls. This tutorial builds those models rigorously, with the exact distinctions interviewers probe.

How this fits the course: Tutorial 09 covered classic performance (memoization, profiling). This one covers the modern scheduling and architecture layer that increasingly makes manual optimization unnecessary. Tutorial 14 goes deep on SSR/streaming infrastructure; here we cover just enough streaming to explain why Suspense boundaries matter architecturally.

1. React 18 foundations Tier 1

createRoot — the switch that turns everything on

React 18's features are gated behind the new root API. If you're still calling ReactDOM.render, you're running React 18 code in a legacy compatibility mode: no automatic batching outside events, no transitions, no concurrent features at all.

// React 17 and earlier (legacy root)
import ReactDOM from 'react-dom';
ReactDOM.render(<App />, document.getElementById('root'));

// React 18+ (concurrent-capable root)
import { createRoot } from 'react-dom/client';
const root = createRoot(document.getElementById('root'));
root.render(<App />);

// Hydration equivalent (SSR apps)
import { hydrateRoot } from 'react-dom/client';
hydrateRoot(document.getElementById('root'), <App />);

What the legacy root lacked, precisely:

  • Synchronous, uninterruptible rendering only. Once a render started it ran to completion on the main thread, no matter how long it took.
  • Batching only inside React event handlers. Two setState calls in a setTimeout caused two renders.
  • No priority lanes. Every update was equally urgent — a keystroke and a giant list re-render competed head-to-head.

Automatic batching everywhere

In React 18, multiple state updates in any context — event handlers, promises, setTimeout, native event listeners — are batched into a single re-render.

function handleClick() {
  // React 17: batched (1 render) — this always worked
  setCount(c => c + 1);
  setFlag(f => !f);
}

async function handleSave() {
  await api.save(data);
  // React 17: TWO renders (outside the event handler's synchronous frame)
  // React 18: ONE render (automatic batching)
  setSaving(false);
  setToast('Saved!');
}

// Escape hatch when you genuinely need the DOM updated between calls:
import { flushSync } from 'react-dom';
flushSync(() => setCount(c => c + 1)); // DOM is updated here
inputRef.current.scrollIntoView();       // sees the new DOM
Pitfall: code that (accidentally) relied on reading the committed DOM between two async state updates broke on React 18 upgrades. The fix is flushSync, used sparingly — it forces a synchronous render and defeats batching. If you find yourself reaching for it often, the design is fighting React.

Concurrent rendering — what it IS, precisely

This is the single most-fumbled definition in senior interviews. Nail it:

The crisp definition: Concurrent rendering means React's render phase is interruptible. React can start rendering an update, pause partway through to let the browser handle urgent work (a keystroke, a click, painting), then resume, restart, or abandon that render entirely. Updates are assigned priority lanes: urgent updates (typing, clicking — things where users expect instant feedback) can interrupt non-urgent ones (transitions — large re-renders where a brief delay is acceptable). Commits are still synchronous and atomic — you never see a half-committed UI.

Mechanically: React renders in small units of work (Fiber's incremental architecture made this possible back in 16), yielding to the main thread between units. When a higher-priority update arrives mid-render, React throws away or shelves the in-progress work-in-progress tree and services the urgent update first, then re-renders the low-priority work from scratch against the latest state.

Concurrent rendering — what it ISN'T

MisconceptionReality
"React renders on multiple threads in parallel."No. Everything is on the one main thread. "Concurrent" means interleaved / time-sliced, like an OS scheduler, not parallel execution.
"Upgrading to createRoot makes my app concurrent."No. Concurrency is opt-in per update. Plain setState is still rendered synchronously-in-effect (urgent lane, uninterrupted in practice). You get interruptible rendering only when you use the features: startTransition, useDeferredValue, Suspense-driven renders, hydration.
"It makes rendering faster."No. The same component work happens (sometimes more, since abandoned renders are wasted work). It makes the app more responsive by keeping the main thread available for urgent input.
"Users can see partially rendered UI."No. Rendering is interruptible; committing is not. The DOM mutation phase is synchronous and atomic.

Why purity rules become load-bearing

Because React may render a component multiple times for one committed update, or render it and throw the result away, render-phase side effects go from "bad practice" to "actual bugs":

function Bad({ items }) {
  // ❌ Mutation during render: in concurrent React this may run
  // 2–3 times per commit, or run for a render that's abandoned.
  analytics.track('rendered');       // fires unpredictably many times
  globalCache.set(items[0].id, ...); // pollutes state from discarded renders
  return <List items={items} />;
}

Rules that concurrency makes non-negotiable: render must be a pure function of props/state/context; side effects live in event handlers and effects; never mutate props, state, or values created outside the current render. StrictMode's double-invocation in dev exists precisely to surface these violations before concurrency does.

Tearing and useSyncExternalStore (recap)

Tearing: when a render is paused mid-way and an external (non-React) store changes underneath it, components rendered before the pause see the old value and components rendered after see the new one — one committed frame showing two versions of the truth. React state doesn't tear (React snapshots it per render); external mutable stores can.

import { useSyncExternalStore } from 'react';

function useStore(store, selector) {
  return useSyncExternalStore(
    store.subscribe,                    // (callback) => unsubscribe
    () => selector(store.getState()),   // getSnapshot (client)
    () => selector(store.getServerState()) // getServerSnapshot (SSR)
  );
}

useSyncExternalStore fixes tearing by making external reads consistent: if the store changes during a concurrent render, React detects the snapshot mismatch and falls back to a synchronous re-render. This is why Redux, Zustand, and friends all migrated to it. The trade-off: updates through it are de-facto synchronous, so external-store updates don't get the benefits of transitions — one reason "keep state in React when you can" is real advice, not dogma.

2. useTransition Tier 1

useTransition lets you mark specific state updates as non-urgent. React renders them in a low-priority, interruptible lane while keeping the current UI on screen and fully interactive.

const [isPending, startTransition] = useTransition();

function handleChange(e) {
  setQuery(e.target.value);          // urgent: the input must echo instantly
  startTransition(() => {
    setSearchResults(filter(e.target.value)); // non-urgent: can lag & be interrupted
  });
}
  • startTransition(fn) — every state update scheduled inside fn is flagged as a transition. (There's also a standalone startTransition export for use outside components, without isPending.)
  • isPending — true while the transition render hasn't committed. Use it for subtle feedback: dim the stale results, show an inline spinner — the old UI stays visible instead of vanishing.

The worked example: a filterable list that never blocks typing

function FilterableList({ allItems }) {
  const [query, setQuery] = useState('');
  const [filtered, setFiltered] = useState(allItems);
  const [isPending, startTransition] = useTransition();

  function handleChange(e) {
    const q = e.target.value;
    setQuery(q); // URGENT — keystroke echo, commits immediately
    startTransition(() => {
      // NON-URGENT — rendering 10,000 rows. If the user types again
      // before this finishes, React ABANDONS this render and restarts
      // with the latest query. Intermediate keystrokes never produce
      // wasted committed frames.
      setFiltered(allItems.filter(i => i.name.includes(q)));
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      <div style={{ opacity: isPending ? 0.6 : 1 }}>
        <HugeList items={filtered} />
      </div>
    </>
  );
}

Under the hood: the transition update is placed in a transition lane. React renders the new tree in memory, yielding to the browser every few milliseconds so input events and paints stay smooth. If an urgent update (the next keystroke) arrives, React interrupts, commits the urgent update, discards the stale in-progress transition, and starts a fresh transition render with the newest state. Only the final one commits.

Transitions vs debouncing — the semantic difference

Debounce / throttleTransition

Best forNetwork-bound cost: don't fire 10 API requests for 10 keystrokesCPU-bound cost: rendering is expensive but data is local

Interview line: "Debouncing rate-limits when work begins — right tool for network cost. Transitions don't delay work at all; they make the rendering of it interruptible — right tool for CPU cost. For a search box hitting an API and rendering a big result list, you might use both: debounce the fetch, wrap the results-state update in a transition."

React 19: async functions in transitions (the bridge to Actions)

React 19 lets you pass an async function to startTransition. isPending stays true through the await, and state updates after the await are still part of the transition. An async function used this way is what React 19 calls an Action (see §6).

const [isPending, startTransition] = useTransition();

function handleSubmit() {
  startTransition(async () => {
    const err = await updateName(name);   // isPending covers the request
    if (err) { setError(err); return; }
    redirect('/profile');
  });
}
Caveat: in React 19, state updates after an await inside a transition must themselves be wrapped in another startTransition call if you need them treated as part of the transition in all cases (the async context isn't automatically propagated across every await in current releases — the team has said this is a known limitation). Form primitives like useActionState handle this for you, which is one reason to prefer them.

3. useDeferredValue Tier 1

useTransition requires that you own the setState call. When you don't — the value arrives as a prop, or comes from a library hook, or from context — you can't wrap its update in startTransition. useDeferredValue solves the same problem from the other end: instead of marking the update low-priority, you take a value and ask for a lagging copy of it.

function SearchResults({ query }) {        // query updates urgently upstream
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery; // built-in staleness signal

  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      <ExpensiveResults query={deferredQuery} />
    </div>
  );
}

How it works: on an urgent update, React first re-renders with the deferred value still holding the old value (fast — the expensive subtree gets identical props), commits that, then schedules a background re-render — effectively a transition — where the deferred value catches up. If new updates keep arriving, the background render is abandoned and restarted, exactly like a transition. It is the transition mechanism, exposed as a value-level API.

The catch everyone misses: memo is required

The trap: useDeferredValue only helps if the expensive consumer is memoized. The optimization is: "the urgent render passes the same deferred value as last time, so the expensive child can bail out." Without memo, the child re-renders during the urgent render anyway — same props, but no bail-out — and you've gained nothing.
// ❌ Useless: ExpensiveChart re-renders on every keystroke regardless
function Panel({ value }) {
  const deferred = useDeferredValue(value);
  return <ExpensiveChart data={compute(deferred)} />;
}

// ✅ Correct: the urgent render sees identical props → skipped
const ExpensiveChart = memo(function ExpensiveChart({ value }) {
  const data = useMemo(() => computePoints(value), [value]); // heavy
  return <svg>{/* thousands of points */}</svg>;
});

function Panel({ value }) {
  const deferred = useDeferredValue(value);
  return <ExpensiveChart value={deferred} />;  // memo + stable prop = bail-out
}

initialValue (React 19)

// First render (including SSR) uses '' — commits fast —
// then a background re-render runs with the real value.
const deferred = useDeferredValue(expensiveInput, '');

Useful for making the very first paint cheap: render a placeholder-priced version immediately, fill in the expensive version right after.

vs debounce/throttle

DebounceThrottleuseDeferredValue
DelayFixed, after last eventFixed intervalNo fixed delay — as soon as the machine keeps up
Adapts to device speedNoNoYes — fast device ≈ zero lag; slow device lags gracefully
Blocks main thread when work runsYes (work is still sync)YesNo — background render is interruptible
Good for network requestsYesYesNo — it doesn't reduce how many values you observe
Choosing between the two hooks: control the setState? → useTransition (more explicit, gives you isPending). Only have the value (prop, context, library state)? → useDeferredValue. They compile down to the same lane mechanics.

4. Suspense beyond lazy loading Tier 1

Most 7-YOE candidates know Suspense only as the wrapper for React.lazy. Post-18, Suspense is a general protocol for declaratively expressing "this subtree isn't ready yet" — for code, data, and hydration.

The mental model: throw a promise

Conceptually, a component "suspends" by throwing a promise during render (in practice today: by calling use(promise) or a Suspense-enabled library hook — you don't hand-write throws). React catches it like an error boundary catches errors: it walks up to the nearest <Suspense>, shows its fallback, and when the promise resolves, re-renders the subtree.

// What Suspense-enabled libraries do internally (simplified):
function read(resource) {
  if (resource.status === 'fulfilled') return resource.value;
  if (resource.status === 'rejected')  throw resource.error;  // → error boundary
  throw resource.promise;                                     // → Suspense boundary
}

// This is why ad-hoc fetching can't suspend: you need a cache that
// survives the retried render. Suspense-ready sources today:
// use(promise), React.lazy, Next.js/RSC async components,
// TanStack Query's useSuspenseQuery, Relay.
Interview trap: "Can I just throw a promise from my own fetch call?" Technically that's the protocol, but it's not a supported public contract for app code — the render will re-run and re-fetch infinitely without an external cache. Say: "Suspense for data is designed to be integrated by frameworks and caching libraries; app code consumes it through use() or library hooks."

Boundary granularity is a design decision

<Suspense fallback={<PageSkeleton />}>        {/* coarse: whole page pops in at once */}
  <Header />
  <Suspense fallback={<FeedSkeleton />}>      {/* nested: feed streams independently */}
    <Feed />
  </Suspense>
  <Suspense fallback={<SidebarSkeleton />}>
    <Sidebar />
  </Suspense>
</Suspense>
  • A suspending component activates the nearest boundary above it; everything inside that boundary is replaced by the fallback — siblings included. Boundary placement therefore decides your loading choreography.
  • Too coarse: one slow widget blanks the whole page. Too fine: popcorn UI — a dozen spinners resolving at different times, layout shift everywhere. Design boundaries around meaningful visual units the designer would skeleton.
  • Pair each boundary with an error boundary at the same altitude: promise rejection surfaces as a thrown error to the nearest error boundary.

Suspense + transitions: avoiding the fallback flash

Key rule: when a state update inside a transition causes an already-visible subtree to suspend, React does NOT show the fallback. It keeps the current UI on screen (with isPending true) until the new content is ready. Non-transition updates that suspend do replace shown content with the fallback — the jarring flash.

function TabContainer() {
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();

  function selectTab(next) {
    startTransition(() => setTab(next)); // old tab stays visible while
  }                                       // the new tab's data suspends

  return (
    <>
      <Tabs onSelect={selectTab} current={tab} dimmed={isPending} />
      <Suspense fallback={<Skeleton />}>   {/* only for INITIAL load */}
        <TabPanel tab={tab} />
      </Suspense>
    </>
  );
}
The pattern to say out loud: "Suspense fallbacks for initial loads; transitions for subsequent navigations/refetches so users keep the old content with a pending indicator instead of a spinner flash. That's the modern replacement for hand-rolled keepPreviousData logic."

SuspenseList (awareness)

<SuspenseList revealOrder="forwards" tail="collapsed"> coordinates multiple sibling boundaries — e.g., reveal feed items top-to-bottom even if lower ones resolve first, preventing content jumping. Know it exists and what problem it solves; it has spent a long time as an experimental API, so flag that: "still not in the stable release channel."

Streaming SSR + selective hydration — the architectural payoff

Suspense boundaries are the unit of streaming and selective hydration (full details in tutorial 14; here's the interview-grade summary):

  1. Streaming HTML: with renderToPipeableStream, the server flushes HTML for the parts that are ready immediately; a suspended boundary's slot ships as the fallback HTML. When the server data resolves, React streams the real HTML plus a tiny inline script that swaps it into place — before any client JS loads. TTFB and first paint no longer wait for the slowest query.
  2. Selective hydration: the client hydrates boundary-by-boundary rather than all-or-nothing; hydration itself is interruptible concurrent work.
  3. Interaction-driven priority: if the user clicks inside a not-yet-hydrated boundary, React records the event, prioritizes hydrating that boundary first, and replays the interaction.
This is why "Suspense is a spinner component" undersells it badly. Suspense boundaries are the architectural seams along which React can stream, hydrate, and prioritize independently.

5. Server Components — the correct mental model Tier 1

This is the highest-leverage section in this tutorial. RSC is where senior candidates most visibly separate into "actually understands it" vs "read a tweet about it." Build the model from the ground up.

The two-environment model

With React Server Components (RSC), your component tree spans two environments, and each component belongs to exactly one:

Server ComponentsClient Components
Where they executeOnly on the server (or at build time). Never shipped to, never run in, the browser.In the browser — and also once on the server if SSR is used (to produce initial HTML).
Bundle costZero bytes of their code in the client bundle (including their imports — heavy markdown parsers, date libs, SDKs stay server-side).Full component + dependency code ships to the client.
CapabilitiesCan be async; can read databases, filesystems, secrets, internal services directly.The React you've written for 7 years: state, effects, refs, event handlers, browser APIs.
Cannot useuseState, useEffect, any hook with client lifetime; event handlers; browser APIs.async/await in the component body; direct server resources.
Re-rendersOnly when the server re-runs them (navigation, revalidation, action). No client re-render, ever.Normal React lifecycle on the client.
How markedDefault in RSC frameworks (e.g., Next.js App Router).Opt-in with the 'use client' directive at a module boundary.
 
// app/note/[id]/page.tsx — a SERVER component (default; note: async!)
import { db } from '@/lib/db';          // server-only import — never bundled
import { NoteEditor } from './NoteEditor';

export default async function NotePage({ params }) {
  const note = await db.notes.find(params.id);   // direct data access,
  return (                                        // no API layer needed
    <article>
      <h1>{note.title}</h1>
      <NoteEditor initialText={note.text} />      {/* client island */}
    </article>
  );
}
// NoteEditor.tsx — a CLIENT component
'use client';
import { useState } from 'react';

export function NoteEditor({ initialText }) {     // props crossed the wire
  const [text, setText] = useState(initialText);  // → must be serializable
  return <textarea value={text} onChange={e => setText(e.target.value)} />;
}

The RSC payload

The server doesn't send HTML for server components (that's SSR's job) — it sends the RSC payload: a compact, streamable serialization of the rendered tree. It interleaves three kinds of things:

  1. Rendered output of server components — their JSX result, already computed, as data (element type + props). The component functions themselves never appear.
  2. References to client components — "mount module NoteEditor from chunk abc123 here" — plus…
  3. The serialized props to pass them.
// Sketch of the wire format (illustrative, not exact):
1: I ["./NoteEditor.js", ["chunk-abc123"], "NoteEditor"]   // client ref
0: ["$", "article", null, { "children": [
     ["$", "h1", null, { "children": "Groceries" }],
     ["$", "$L1", null, { "initialText": "milk, eggs" }]    // ← client slot + props
   ]}]

On the client, React reconciles this payload into the existing tree. Crucially, on navigation the payload can be re-fetched and merged without destroying client state — a client component in an unchanged position keeps its state, unlike a full-page HTML swap.

The rules — where candidates get burned

Rule 1: Server can render client; props must be serializable. Anything a server component passes to a client component crosses a serialization boundary. Plain data, arrays, objects, JSX, promises: fine. Functions, class instances, Dates-with-methods-you-rely-on, Maps/Sets (framework-dependent): not fine. The one exception: Server Actions (functions marked 'use server') can be passed — they serialize as an RPC reference, not as code.

Rule 2: Client components CANNOT import server components. The import would drag server code (and its DB drivers and secrets) into the client bundle — so any module a client module imports is treated as client code. But a client component can receive server-rendered output as children/props. This is the composition pattern, and interviewers specifically test it:

// ❌ WRONG — importing a server component from a client module
'use client';
import { ServerChart } from './ServerChart';  // ServerChart is now forced
                                              // to be a client component
                                              // (breaks if it uses fs/db)
export function Dashboard() {
  return <div><ServerChart /></div>;
}

// ✅ RIGHT — pass server output THROUGH the client component as children
'use client';
export function Dashboard({ chart }) {        // a "slot"
  const [expanded, setExpanded] = useState(false);
  return (
    <div onClick={() => setExpanded(x => !x)}>
      {expanded ? chart : <Preview />}        // already-rendered server JSX
    </div>
  );
}

// In a server component (parent):
<Dashboard chart={<ServerChart />} />         // rendered on the server,
                                              // threaded through as data

The key insight: <ServerChart /> is rendered on the server, and what reaches Dashboard is its serialized output in the RSC payload — data, not code. The client component just places it.

'use client' is a BOUNDARY, not a per-file label

The single most common RSC misunderstanding: 'use client' does not mean "this file runs on the client" in isolation, and you do not put it on every client file. It marks a door in the module graph: "from this import point onward, we're in the client world." Every module transitively imported from a 'use client' module becomes part of the client bundle automatically — no directive needed on them. Conversely, a file with no directive isn't "a server component" by essence; it's shared — it becomes server or client depending on who imports it. Mark modules that must never leak to the client with the server-only package to get a build error instead of a silent leak.
// utils/format.ts — no directive: SHARED code.
// Imported by a server component → runs on server, not bundled.
// Imported by a client component → bundled and runs in the browser.
export const money = (n) => n.toLocaleString('en-US', {style:'currency', currency:'USD'});

// lib/db.ts — must NEVER reach the browser:
import 'server-only';   // build-time error if a client module imports this

Also note: 'use client' components still SSR (they render to HTML on the server for first paint). The directive is about bundling and hydration, not "skips the server."

The one-sentence version to deliver in an interview: "SSR answers 'how do I get HTML for first paint of my client components?' — the code still ships and hydrates. They compose: server components render to the RSC payload, and SSR streams the combined tree as initial HTML."

What RSC buys you, and what it costs

Solves:

  • Bundle size: rendering-only code (markdown → HTML, syntax highlighting, i18n tables, CMS SDKs) contributes zero client bytes.
  • Data-fetching waterfalls: instead of client-side fetch cascades (render → fetch → render child → child fetches…), fetching is colocated in components but executes on the server, close to the data, with parent/child fetches resolved in one round trip to the browser.
  • Direct backend access: query the DB in the component; no bespoke REST/GraphQL endpoint per widget, no over/under-fetching layer to maintain.
  • Automatic code-splitting seams: every 'use client' boundary is a natural split point.

Costs (say these too — it signals judgment):

  • A genuinely new mental model — the whole team must internalize the boundary rules; serialization errors and "you're importing server code into the client" become a new bug class.
  • Framework requirement: RSC needs a bundler + router + server integration (Next.js App Router is the mainstream one; React Router v7/framework mode and others are adopting). You don't sprinkle RSC into a Vite SPA.
  • Ecosystem churn: libraries needed to split into server-safe and client parts; CSS-in-JS runtime libraries were notably incompatible.
  • You need a server (or build-time prerendering) in the serving path — a cost SPAs didn't have.

Async server components & fetch dedup (awareness)

6. Actions & React 19 form primitives Tier 1

React 19 finally gives mutations first-class treatment. Two related ideas share confusingly similar names — keep them distinct:

  • Action (client concept): any async function used in a transition (e.g., passed to <form action> or run via startTransition). React manages its pending state, errors, and optimistic updates.
  • Server Action / Server Function ('use server'): a function that executes on the server but is callable from client code — React's RPC mechanism. A Server Action used in a form is one kind of Action.

'use server' — the RPC mental model

// actions.ts
'use server';                     // every export = a server endpoint
import { db } from '@/lib/db';

export async function createNote(prevState, formData) {
  const title = formData.get('title');
  if (!title || title.length < 3) {
    return { error: 'Title must be at least 3 characters' };
  }
  await db.notes.insert({ title });
  revalidatePath('/notes');       // framework: refresh server-rendered data
  return { error: null };
}

What actually happens: the bundler replaces the function with a reference (an ID) on the client. Calling it issues an HTTP request to the framework's action endpoint; arguments are serialized across, the function body runs on the server, the return value is serialized back. So: never trust its inputs (validate/authorize server-side like any endpoint — anyone can invoke it with crafted payloads), and arguments/returns must be serializable.

Directive confusion check: 'use server' does NOT mark a server component (server components are the default and have no directive). It marks a server-callable function. 'use client' and 'use server' are not opposites-of-each-other for components; one marks a bundle boundary, the other marks an RPC.

<form action={fn}> and progressive enhancement

<form action={createNote}>
  <input name="title" />
  <button type="submit">Add</button>
</form>

React 19 extends <form>/<button formAction> to accept functions. On submit, React calls the function with the FormData, inside a transition, and resets uncontrolled fields on success. With a server action, frameworks wire this up so the form works before hydration — or with JS disabled: a plain HTML POST hits the action endpoint. That's the progressive-enhancement story, and it's a phrase worth saying verbatim.

useActionState — the full mutation loop

'use client';
import { useActionState } from 'react';
import { createNote } from './actions';

export function NewNoteForm() {
  // (action, initialState) → [state, formAction, isPending]
  const [state, formAction, isPending] = useActionState(createNote, { error: null });

  return (
    <form action={formAction}>
      <input name="title" aria-invalid={!!state.error} />
      {state.error && <p role="alert">{state.error}</p>}
      <button disabled={isPending}>
        {isPending ? 'Saving…' : 'Add note'}
      </button>
    </form>
  );
}

The action receives (prevState, formData) — note the extra first argument compared to a bare form action. Return value becomes the new state: perfect for validation errors, success messages, or the created entity. This replaces the classic hand-rolled {loading, error, data} + try/catch + setLoading dance. (Historical note: briefly named useFormState in react-dom during the RC period; the shipped React 19 API is useActionState from react.)

useOptimistic — optimistic UI with automatic rollback

'use client';
import { useOptimistic } from 'react';

function Thread({ messages, sendMessage }) {
  const [optimisticMessages, addOptimistic] = useOptimistic(
    messages,                                  // source of truth (server data)
    (current, newText) => [                    // how to apply an optimistic patch
      ...current,
      { text: newText, pending: true }
    ]
  );

  async function formAction(formData) {
    addOptimistic(formData.get('text'));       // 1. instant UI (inside the action's transition)
    await sendMessage(formData);               // 2. real mutation on server
  }                                            // 3. server data refreshes → `messages`
                                               //    prop updates → optimistic state
                                               //    is DISCARDED and replaced by truth
  return (
    <>
      {optimisticMessages.map((m, i) =>
        <p key={i} style={{ opacity: m.pending ? 0.5 : 1 }}>{m.text}</p>)}
      <form action={formAction}>
        <input name="text" /><button>Send</button>
      </form>
    </>
  );
}

Rollback semantics — the part to articulate: the optimistic state lives only for the duration of the enclosing action/transition. When it settles, React re-renders from the real source value. Success → the refreshed server data includes the item, so the swap is invisible. Failure → the source never changed, so the optimistic item simply vanishes. You never write rollback code; "rollback" is just re-rendering from truth.

Error handling for actions

  • Expected failures (validation, business rules): return them — return { error: '…' } — and render via useActionState. Don't throw for control flow.
  • Unexpected failures: a thrown error in an action surfaces to the nearest error boundary (frameworks add hooks like onCaughtError on the root and route-level error files).
  • Remember thrown errors from server actions may have their messages redacted in production (to avoid leaking server internals) — another reason to return structured errors for anything user-facing.
Tie it together: Actions = async transitions. isPending from useTransition/useActionState/useFormStatus are the same underlying pending flag at different altitudes: "I own the call" / "I own the form" / "I'm a child of the form." Optimistic state is scoped to the action's lifetime. Once you present the family as one system, you sound like you've shipped it.

7. The use() hook Tier 2

use() (React 19) reads a resource during render. Two resources are supported: promises and Context.

use(promise) — Suspense-integrated data reading

// SERVER component: start the fetch, DON'T await — pass the promise down
export default function Page() {
  const commentsPromise = fetchComments();     // kicked off immediately
  return (
    <Suspense fallback={<CommentsSkeleton />}>
      <Comments promise={commentsPromise} />   {/* promises are serializable
                                                   across the RSC boundary! */}
    </Suspense>
  );
}

// CLIENT component: unwrap with use()
'use client';
import { use } from 'react';

function Comments({ promise }) {
  const comments = use(promise);   // suspends until resolved;
  return comments.map(c => <p key={c.id}>{c.text}</p>);
}

This is the recommended streaming pattern: the server starts fetching without blocking its own render, streams the shell, and the client component suspends on the promise until the data streams in. Rejections propagate to the nearest error boundary (or the promise's own .catch-provided fallback via Promise.catch before passing).

use(Context) — the conditional context read

import { use } from 'react';

function Heading({ children }) {
  if (children == null) return null;          // early return BEFORE the read —
  const theme = use(ThemeContext);            // illegal with useContext, fine with use()
  return <h1 className={theme}>{children}</h1>;
}

use is the one hook-shaped API that relaxes the Rules of Hooks: it may be called inside conditionals, loops, and after early returns. (It must still be called during render — not in event handlers or effects.) It otherwise behaves like useContext: nearest provider above, subscribes to updates.

Caveats

Don't create promises during render in client components. use(fetch(url)) inside a client component creates a new promise every render → suspend → re-render → new promise → infinite loop, plus a React warning that promises created in client render aren't supported without a Suspense-compatible cache. Correct sources: promises created in server components and passed down, or promises from a caching layer (TanStack Query, framework loaders, cache()).
use vs await in a server component: await blocks that server component's own render until data arrives; starting the promise and passing it to use() in a child (under Suspense) lets the shell stream first. Choose per data criticality: above-the-fold critical data → await; secondary content → stream with use.

9. The React Compiler Tier 2

What it does

The React Compiler is a build-time optimizer (Babel/SWC plugin) that automatically memoizes your components and hooks. It analyzes your code against the Rules of React and rewrites each component into a version with a fine-grained memo cache: every expression whose inputs haven't changed — computed values, JSX subtrees, callbacks — is reused from the cache instead of being recreated.

// You write:
function List({ items, filter }) {
  const visible = items.filter(i => i.type === filter);   // recomputed every render?
  return <Rows data={visible} onPick={(id) => select(id)} />;
}

// Compiler output (conceptually — real output uses a cache hook, often shown as `c`):
function List({ items, filter }) {
  const $ = _c(4);                       // memo cache slots
  let visible;
  if ($[0] !== items || $[1] !== filter) {
    visible = items.filter(i => i.type === filter);
    $[0] = items; $[1] = filter; $[2] = visible;
  } else { visible = $[2]; }
  // JSX + callback similarly cached per-input …
}

Effectively: as-if you had written perfect useMemo/useCallback/memo everywhere — but finer-grained (per-expression, not per-hook-call), impossible to get dependency arrays wrong, and with zero code noise. It can do this safely because it statically verifies components follow the Rules of React (purity, no mutation of props/state); when it can't prove a component is safe, it skips that component rather than risk changing behavior.

What it means for manual memoization

  • Existing useMemo/useCallback/memo keep working — the compiler respects them (and can use them as hints).
  • Going forward, hand-written memoization is largely obsolete for re-render performance in compiled codebases. Don't rip it out preemptively, but stop adding it for that purpose.
  • Still-legitimate uses of useMemo: guaranteeing referential stability for correctness (e.g., a value in an effect dependency you must not retrigger) and genuinely expensive computations where you want an explicit, auditable cache.

Requirements & adoption status

  • Strict Rules of React compliance. The eslint-plugin-react-hooks (v6+, which absorbed the compiler lint rules — originally eslint-plugin-react-compiler) flags violations. Rule-breaking components are silently skipped, not broken — but they don't get optimized.
  • Status: stable release (1.0 in late 2025) after powering Instagram/Facebook surfaces in production. Opt-in: you add babel-plugin-react-compiler (or the SWC path) to your build; Next.js exposes it via config (reactCompiler: true). Targets React 19 natively; supports 17/18 via a runtime shim package.
  • Not a new language: it compiles plain JavaScript/TypeScript React — no annotations required.
"Does the compiler replace useMemo?" — the nuanced answer: "For its dominant use case — preventing unnecessary re-renders and re-computations — yes, and it does it more thoroughly than humans do. But it's an optimization, not a semantic guarantee: the compiler may skip components it can't verify, and it doesn't promise stability as an API contract. So for correctness-critical identity (an effect dep that must not fire, a subscription key) I'd still be explicit. And it only helps re-render cost — it does nothing for a single genuinely slow computation on first render; that's still useMemo, a worker, or the server."

Interview Q&A

1. What is concurrent rendering in React 18, precisely?

It makes the render phase interruptible. React assigns updates to priority lanes; while rendering a low-priority update, it yields to the main thread regularly, and if an urgent update (typing, clicks) arrives, React pauses or abandons the in-progress render, services the urgent update, and restarts the low-priority render against the latest state. It is time-slicing on one thread — not parallelism, not automatic (you opt in per-update via transitions, deferred values, and Suspense), and not faster in raw work — it improves responsiveness. Commits remain synchronous and atomic, so users never see a half-applied UI.

2. What changed with createRoot vs ReactDOM.render beyond syntax?

createRoot enables the concurrent-capable renderer: automatic batching in all contexts (promises, timeouts, native handlers — not just React events), support for transitions and useDeferredValue, Suspense for data with streaming/selective hydration, and priority-based scheduling. Legacy render ran React 18 in a compatibility mode with synchronous, uninterruptible renders and event-only batching. ReactDOM.render is removed entirely in React 19.

3. When would automatic batching change behavior during a React 18 upgrade, and what's the escape hatch?

Any code that set state multiple times outside a React event handler — after an await, in setTimeout, in a native listener — used to render between the calls and now renders once. Code that read the DOM between those updates (measuring, focusing, scroll math) breaks. The escape hatch is flushSync(() => setState(...)), which forces a synchronous commit before the next line — used sparingly since it defeats batching and hurts performance.

4. Why do the purity rules matter more under concurrent rendering?

Because React may invoke a component's render multiple times for one commit, or render it and throw the work away when an urgent update interrupts. Side effects during render (mutations, logging, cache writes, subscriptions) therefore execute an unpredictable number of times, including for renders that never commit. Pure renders make re-execution and abandonment invisible. StrictMode's dev double-rendering exists specifically to surface these violations early — and Rules-of-React compliance is also what makes the React Compiler's transforms safe.

5. What is tearing, and how does useSyncExternalStore prevent it?

Tearing is one committed frame showing two versions of the same data: a concurrent render pauses, an external mutable store changes, and components rendered after the pause read the new value while earlier ones read the old. React state can't tear (snapshotted per render); external stores can. useSyncExternalStore(subscribe, getSnapshot) prevents it — React compares snapshots and, if the store changed mid-render, discards the torn render and re-renders synchronously. Trade-off: those updates can't be time-sliced, so external-store updates don't benefit from transitions.

6. Explain useTransition and what happens under the hood when a transition is interrupted.

useTransition returns [isPending, startTransition]; updates scheduled inside startTransition go into a low-priority transition lane. React renders them in memory, yielding periodically. If an urgent update arrives (next keystroke), React interrupts, commits the urgent update, discards the now-stale work-in-progress tree, and starts a fresh transition render with the newest state — so intermediate values never commit. The old UI stays visible throughout, with isPending for feedback. In React 19, the function may be async — that's an Action.

7. Transitions vs debouncing — when is each correct?

They rate-limit different resources. Debouncing delays when work starts by a fixed timer — right when the cost is network-bound (don't send 10 requests for 10 keystrokes), but it adds artificial latency on fast machines and still jank when the work eventually runs. Transitions start work immediately but render it interruptibly at low priority — right when the cost is CPU-bound rendering of local data, with zero added latency and device-adaptive behavior. A search box that fetches then renders a huge list may use both: debounce the fetch, transition the results render.

8. When useDeferredValue over useTransition, and what's the memo catch?

Use useDeferredValue when you don't control the setState — the value arrives as a prop, from context, or from a library. It returns a lagging copy: the urgent render commits with the old deferred value, then a background (transition-lane) render catches it up, restartable if new values arrive. The catch: it only helps if the expensive consumer is memoized — the win is that the urgent render passes identical props to the expensive child so memo bails out. Without memo, the child re-renders on the urgent pass anyway and you've optimized nothing.

9. How does Suspense work mechanically — the "throw a promise" model?

A component signals "not ready" by throwing a promise during render (in practice via use() or a Suspense-enabled library — hand-throwing isn't a supported app-level API). React catches it like an error boundary catches errors, unwinds to the nearest <Suspense>, renders its fallback, and subscribes to the promise; on resolution it re-renders the subtree, which now reads the cached value synchronously. That's why data suspending requires a cache outside render — otherwise the retry re-creates the promise forever. Rejections propagate to the nearest error boundary.

10. How do you avoid the Suspense fallback flashing when refetching already-visible content?

Wrap the state update that triggers the refetch in startTransition. Rule: when a transition causes a currently visible subtree to suspend, React does not retreat to the fallback — it keeps the old UI on screen with isPending true until the new content is ready. Non-transition updates that suspend do swap in the fallback. So: fallbacks for initial mounts, transitions for tab switches, pagination, refetches — the declarative replacement for keep-previous-data hacks.

11. What do Suspense boundaries buy you in streaming SSR?

They're the streaming and hydration units. With renderToPipeableStream, the server flushes the shell HTML immediately; a suspended boundary ships its fallback, and when its data resolves the server streams the real HTML with an inline script that swaps it in — before any client JS loads. On the client, hydration proceeds boundary-by-boundary (selective hydration), it's interruptible, and if the user interacts with a not-yet-hydrated boundary, React prioritizes hydrating that boundary and replays the event. First paint no longer waits for the slowest query, and interactivity isn't all-or-nothing.

12. Are React Server Components the same as SSR? Give the precise distinction.

No — the classic trap. SSR renders client components to HTML for first paint; their code still ships to the browser and hydrates, and they re-render client-side thereafter. Server components execute only on the server, ship zero bytes of their code, never hydrate, and never re-render on the client — their output crosses as the RSC payload (serialized tree), not HTML. They compose: in Next.js App Router, server components render to the payload and SSR turns the combined tree — including client components' first frame — into streamed HTML. One sentence: SSR is about where the first render happens; RSC is about where components live permanently.

13. What exactly does 'use client' mean?

It marks a boundary in the module graph, not "this file runs on the client" as an isolated fact. Placed at the top of a module, it says: when the server tree imports this module, switch worlds — this module and everything it transitively imports becomes client-bundle code. You don't annotate every client file; only the entry doors. Undirectived modules are shared — server or client depending on the importer (guard true server code with the server-only package). And 'use client' components still render on the server during SSR for initial HTML — the directive governs bundling and hydration, not "skip the server."

14. A client component needs to render a server component inside it. How?

It can't import it — importing from a client module drags the target into the client bundle, which breaks anything using server-only resources. Instead, use composition: the client component accepts children (or any JSX prop) as a slot, and a server component parent passes the server component into it: <ClientShell><ServerContent /></ClientShell>. The server content is rendered on the server; what flows through the client component is its serialized output in the RSC payload — data, not code. The client component controls placement/visibility; the server retains rendering.

15. What is the RSC payload and why does it matter that it's not HTML?

It's a compact streamable serialization of the rendered tree: server components' rendered output as data, interleaved with references to client components (module/chunk IDs) plus their serialized props. Because it's a component tree, not markup, the client can reconcile it into the live tree on navigation — client components in unchanged positions keep their state (form inputs, video playback), which a full HTML replacement would destroy. It also encodes which chunks to load, giving automatic code-splitting at every client boundary.

16. What problems does RSC actually solve, and what does it cost?

Solves: (1) bundle size — render-only dependencies (markdown, highlighting, SDKs) never ship; (2) client fetch waterfalls — data fetching is colocated in components but executes server-side near the data, resolving nested fetches before one response streams to the browser; (3) direct backend access — no per-widget API endpoints; (4) natural code-splitting at client boundaries. Costs: a new mental model with new bug classes (serialization limits, server-code leaks), a hard framework requirement (bundler + router + server integration — you can't sprinkle it into a plain SPA), ecosystem churn (libraries splitting into server/client halves; runtime CSS-in-JS incompatible), and a server in the serving path.

17. What is a Server Action ('use server'), and what are the security implications?

A function marked 'use server' compiles into an RPC: the client gets an opaque reference; calling it (directly, or via <form action>) sends a serialized request to a framework endpoint where the real function executes, returning a serialized result. Security: every server action is a public endpoint — anyone can invoke it with arbitrary arguments regardless of your UI, so validate and authorize inside the action, never trust hidden fields or client checks. Also note 'use server' marks server-callable functions, not server components (those are the default and take no directive) — a common confusion.

18. Walk through useActionState, useFormStatus, and useOptimistic — how do they divide the work?

useActionState(action, initial) wraps a mutation: returns [state, formAction, isPending], where the action receives (prevState, formData) and its return value becomes the new state — the home for validation errors and results. useFormStatus() (from react-dom) lets a child of a form read the nearest ancestor form's pending state — enabling reusable design-system submit buttons without prop drilling; it returns pending: false in the component that renders the form itself. useOptimistic(source, reducer) gives a temporary optimistic view applied during the action's transition; when the action settles, React re-renders from the real source — rollback is automatic because optimistic state simply expires. All three ride the same async-transition (Action) machinery.

19. How does progressive enhancement work with React 19 forms?

When a server action is passed to <form action>, the framework emits a real HTML form that POSTs to the action's endpoint. If JS hasn't loaded (slow network, mid-hydration) or is disabled, the browser's native submit still executes the mutation server-side and re-renders the page. Once hydrated, React intercepts submission, runs the action as an async transition, and gives you isPending/optimistic UI. So the form is functional from first HTML byte, and JS upgrades the experience rather than gating it.

20. What does use() do that useContext and useEffect-based fetching can't?

Two things. (1) use(promise) reads a promise during render with Suspense integration — the recommended pattern being: create the promise in a server component (don't await), pass it across the RSC boundary (promises serialize), and unwrap in a client component under Suspense, so the shell streams while data loads — no effect/setState round trip, no loading-state boilerplate. (2) use(Context) reads context conditionally — it's exempt from the top-level-only hook rule and can sit inside ifs/loops/after early returns. Caveat: never create the promise during a client render (use(fetch(...))) — new promise each render means an infinite suspend loop; promises must come from the server or a cache.

21. Does the React Compiler make useMemo/useCallback obsolete?

Mostly, for their dominant purpose. The compiler auto-memoizes at build time — per-expression caches for values, JSX, and callbacks, gated on Rules-of-React compliance it verifies statically; components it can't verify are skipped, not broken. Existing manual memoization keeps working, and you should stop adding it for re-render performance in compiled codebases. Nuance to volunteer: memoization-as-optimization is now the compiler's job, but memoization-as-correctness (a referentially stable value that an effect or subscription depends on) is still worth being explicit about, since the compiler treats memoization as an optimization, not a semantic guarantee — and none of this helps a computation that's slow on first render.

22. What's the current status and setup of the React Compiler?

Stable (1.0, late 2025), production-proven on Meta surfaces, and opt-in: add babel-plugin-react-compiler (or the SWC integration) to the build — Next.js exposes it as a config flag. It targets React 19 natively and supports 17/18 through a runtime helper package. Pair it with the compiler-aware lint rules (now folded into eslint-plugin-react-hooks) so Rules-of-React violations — which cause components to be skipped silently — surface in CI.

23. Name the notable removals/deprecations in React 19.

Removed: propTypes and function-component defaultProps (use TypeScript and ES default parameters), legacy context (contextTypes/getChildContext), string refs, ReactDOM.render/hydrate/unmountComponentAtNode/findDOMNode, module-pattern components, UMD builds. Deprecated/sunsetting: forwardRef (ref is a normal prop now), Context.Provider (render the context itself), react-test-renderer. Also: uncaught render errors are reported via root options/window.reportError instead of being re-thrown. These make great rapid-fire currency checks.

24. What are <Activity> and the View Transitions integration?

<Activity mode="visible|hidden"> (React 19.2, evolved from the Offscreen experiment) keeps a hidden subtree alive: state preserved, pre-rendered/updated at background priority, effects disconnected while hidden — ideal for inactive tabs or pre-warming likely-next screens, replacing display:none hacks and state-destroying unmounts. <ViewTransition> integrates the browser's View Transitions API with React: elements marked for transition animate automatically when they change inside transitions, Suspense reveals, or navigations. Both are worth flagging as "recent additions — 19.2 era" to show calibrated currency.

25. Your product team asks: "Should we adopt RSC/App-Router-style architecture?" How do you answer like a senior?

Frame it as fit, not fashion. Strong fit: content/commerce/dashboard apps with lots of read-heavy, data-backed UI — you get smaller bundles, server-side data colocation, streaming-first UX, SEO. Weak fit: highly interactive local-state apps (design tools, editors) where nearly everything would be a client component anyway — you'd pay the mental-model and infra tax for little payoff; a SPA with a good data layer (TanStack Query) stays simpler. Also weigh: team familiarity (boundary bugs are a real onboarding cost), hosting (you now run a server), library compatibility, and migration path (RSC adoption is per-route incremental in Next.js). Concluding line: "RSC moves the default location of rendering to the server and makes the client the opt-in — adopt it where that default matches your app's shape."

Hands-on exercises

Exercise 1 — Fix the janky search (transitions + deferred value)

You're given a component where typing in a filter input stutters because it re-renders a 5,000-row list on every keystroke. First fix it with useTransition; then refactor to a version where the list is a separate component receiving query as a prop, and fix that with useDeferredValue. Add visual staleness feedback to both.

Solution 
// Version A — you own the setState: useTransition
function Search({ rows }) {
  const [text, setText] = useState('');
  const [query, setQuery] = useState('');
  const [isPending, startTransition] = useTransition();

  function onChange(e) {
    setText(e.target.value);                       // urgent echo
    startTransition(() => setQuery(e.target.value)); // low-priority list
  }
  return (
    <>
      <input value={text} onChange={onChange} />
      <div style={{ opacity: isPending ? 0.5 : 1 }}>
        <BigList rows={rows} query={query} />
      </div>
    </>
  );
}

// Version B — query arrives as a prop: useDeferredValue
const BigList = memo(function BigList({ rows, query }) {
  const visible = useMemo(
    () => rows.filter(r => r.name.includes(query)), [rows, query]);
  return visible.map(r => <Row key={r.id} row={r} />);
});

function Results({ rows, query }) {
  const deferredQuery = useDeferredValue(query);
  const stale = query !== deferredQuery;
  return (
    <div style={{ opacity: stale ? 0.5 : 1 }}>
      <BigList rows={rows} query={deferredQuery} />
    </div>
  );
}

Key points: two state variables in version A (urgent text, deferred query); in version B the memo on BigList is mandatory — without it the deferred value provides no benefit, since the urgent render would re-render the list anyway.

Exercise 2 — Design the Suspense boundary layout

A dashboard page has: a header (instant), a KPI strip (fast query, ~50ms), a main chart (slow query, ~2s), and an activity feed (medium, ~400ms). Sketch the JSX with Suspense boundaries so that (a) the header and KPIs never wait for the chart, (b) switching date ranges keeps the old chart visible instead of flashing a skeleton, (c) a chart-query failure doesn't blank the page.

Solution 
function Dashboard() {
  const [range, setRange] = useState('7d');
  const [isPending, startTransition] = useTransition();

  return (
    <>
      <Header />                                    {/* no boundary: instant */}
      <RangePicker
        value={range}
        onChange={(r) => startTransition(() => setRange(r))} />

      <Suspense fallback={<KpiSkeleton />}>         {/* fast: own boundary so it */}
        <KpiStrip range={range} />                  {/* never waits on the chart */}
      </Suspense>

      <ErrorBoundary fallback={<ChartError />}>     {/* (c) failure is contained */}
        <Suspense fallback={<ChartSkeleton />}>     {/* (a) slowest gets its own */}
          <div style={{ opacity: isPending ? 0.6 : 1 }}>
            <MainChart range={range} />             {/* (b) range change is a  */}
          </div>                                    {/* transition → old chart */}
        </Suspense>                                 {/* stays, no fallback     */}
      </ErrorBoundary>

      <Suspense fallback={<FeedSkeleton />}>
        <ActivityFeed />
      </Suspense>
    </>
  );
}

Key points: one boundary per independently-loading visual unit; the skeleton fallbacks fire only on initial load; the transition on range change is what prevents the fallback flash (visible-content-suspends-in-transition rule); the error boundary sits at the same altitude as the Suspense boundary it protects.

Exercise 3 — Fix the broken RSC boundary

This Next.js App Router code fails to build. Identify every boundary violation and fix them using the composition pattern.

// SidebarShell.tsx
'use client';
import { useState } from 'react';
import { UserPanel } from './UserPanel';       // ← ?

export function SidebarShell() {
  const [open, setOpen] = useState(true);
  return (
    <aside>
      <button onClick={() => setOpen(o => !o)}>toggle</button>
      {open && <UserPanel />}
    </aside>
  );
}

// UserPanel.tsx  (no directive)
import { db } from '@/lib/db';
export async function UserPanel() {
  const user = await db.currentUser();
  return <div>{user.name} <button onClick={() => alert('hi')}>wave</button></div>;
}
Solution

Violations: (1) A client module imports UserPanel, which forces it — and @/lib/db — into the client bundle; async components + db access can't run there, so the build fails. (2) UserPanel mixes worlds: server data access and an onClick handler in the same component — server components can't have event handlers.

// SidebarShell.tsx — accept the server content as a SLOT
'use client';
import { useState } from 'react';

export function SidebarShell({ userPanel }) {
  const [open, setOpen] = useState(true);
  return (
    <aside>
      <button onClick={() => setOpen(o => !o)}>toggle</button>
      {open && userPanel}
    </aside>
  );
}

// WaveButton.tsx — extract the interactive leaf
'use client';
export function WaveButton() {
  return <button onClick={() => alert('hi')}>wave</button>;
}

// UserPanel.tsx — pure server component (add `import 'server-only'` to lib/db)
import { db } from '@/lib/db';
import { WaveButton } from './WaveButton';    // server → client import: fine
export async function UserPanel() {
  const user = await db.currentUser();
  return <div>{user.name} <WaveButton /></div>;
}

// page.tsx — a SERVER component wires them together
import { SidebarShell } from './SidebarShell';
import { UserPanel } from './UserPanel';
export default function Page() {
  return <SidebarShell userPanel={<UserPanel />} />;
}

Key points: client → server imports are illegal; server → client imports are the normal direction; server output crosses into client components only as JSX props/children; interactivity is pushed to the smallest possible client leaf. Note the caveat this pattern implies: the server content in a slot is rendered eagerly on the server even if open starts false — hiding it client-side doesn't avoid the server work.

Exercise 4 — Full Action loop: validated mutation with optimistic UI

Build a "add comment" feature with React 19 primitives: a server action that validates (reject comments under 5 chars) and persists; useActionState for validation errors; a reusable <SubmitButton> using useFormStatus; and useOptimistic so the comment appears instantly and disappears if the server rejects it.

Solution 
// actions.ts
'use server';
export async function addComment(prevState, formData) {
  const text = String(formData.get('text') ?? '').trim();
  if (text.length < 5) return { error: 'Comment must be at least 5 characters.' };
  const session = await auth();                 // ALWAYS authorize server-side
  if (!session) return { error: 'Sign in to comment.' };
  await db.comments.insert({ text, userId: session.userId });
  revalidatePath('/post');                      // refresh server-rendered list
  return { error: null };
}

// SubmitButton.tsx — reusable, form-agnostic
'use client';
import { useFormStatus } from 'react-dom';
export function SubmitButton({ children }) {
  const { pending } = useFormStatus();          // reads the ANCESTOR form
  return <button disabled={pending}>{pending ? 'Posting…' : children}</button>;
}

// Comments.tsx
'use client';
import { useActionState, useOptimistic, useRef } from 'react';
import { addComment } from './actions';
import { SubmitButton } from './SubmitButton';

export function Comments({ comments }) {        // server-provided truth
  const formRef = useRef(null);
  const [optimistic, addOptimistic] = useOptimistic(
    comments,
    (cur, text) => [...cur, { id: 'tmp', text, pending: true }]
  );
  const [state, formAction] = useActionState(
    async (prev, formData) => {
      addOptimistic(formData.get('text'));      // instant append
      const result = await addComment(prev, formData);
      if (!result.error) formRef.current?.reset();
      return result;                            // error → optimistic item expires
    },
    { error: null }
  );

  return (
    <>
      <ul>
        {optimistic.map(c =>
          <li key={c.id} style={{ opacity: c.pending ? 0.5 : 1 }}>{c.text}</li>)}
      </ul>
      <form ref={formRef} action={formAction}>
        <input name="text" />
        {state.error && <p role="alert">{state.error}</p>}
        <SubmitButton>Post</SubmitButton>
      </form>
    </>
  );
}

Key points: validation errors are returned (rendered from state.error), not thrown; the optimistic item disappears automatically on failure because it lives only for the action's duration and the source comments never changed; on success the revalidated server data includes the real comment so the handoff is seamless; SubmitButton works in any form because useFormStatus reads the nearest ancestor form.

Exercise 5 — Compiler-readiness audit (whiteboard exercise)

Given the component below, list every Rules-of-React violation that would make the React Compiler skip it, explain why each is unsafe under memoization/concurrency, and rewrite it compliant. Then answer: after the compiler processes the fixed version, which of its manual optimizations are redundant?

let renderCount = 0;
function ProductCard({ product, onAdd }) {
  renderCount++;                                   // A
  product.viewedAt = Date.now();                   // B
  const tags = useMemo(() => product.tags.sort(), [product.tags]); // C
  const handleAdd = useCallback(() => onAdd(product.id), [onAdd, product.id]);
  return <Card tags={tags} onAdd={handleAdd} />;
}
Solution

Violations:

  • A — external mutation during render. Concurrent React may render this 0–N times per commit (interrupted/abandoned/StrictMode renders), so the counter is meaningless and the render isn't pure. Move to an effect if genuinely needed.
  • B — mutating a prop. Breaks memoization everywhere: the object identity doesn't change, so neither the compiler's caches nor memo can detect the change — plus it corrupts data shared with other components. Side-effect it in an event/effect, or derive server-side.
  • C — sort() mutates in place. Even inside useMemo, it mutates product.tags (a prop) — same class of bug as B. Use [...product.tags].sort() (or toSorted()).
function ProductCard({ product, onAdd }) {
  useEffect(() => { trackView(product.id); }, [product.id]);   // A, B → effect
  const tags = [...product.tags].sort();                        // C → copy
  const handleAdd = () => onAdd(product.id);
  return <Card tags={tags} onAdd={handleAdd} />;
}

Redundant after compilation: both useMemo and useCallback — the compiler caches tags keyed on product.tags and handleAdd keyed on (onAdd, product.id) automatically, plus the JSX itself, so Card won't re-render when inputs are unchanged even without wrapping it in memo. The senior nuance: the fix had to come first — with violations present, the compiler skips the component entirely and none of that happens; this is why the compiler lint rules belong in CI.
