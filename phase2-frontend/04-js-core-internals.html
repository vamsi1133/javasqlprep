04 — JS Core Internals

Execution contexts, closures, this, prototypes, and the event loop — the mental models behind every "what does this print?" question.

Why interviewers ask this

Senior frontend loops at FAANG and product startups lean hard on JS internals for one reason: they separate people who use JavaScript from people who can predict it. Seven years of React experience can coexist with fuzzy models of the microtask queue or this binding — frameworks paper over both — and interviewers know it. The dominant format is output prediction: a 5–15 line snippet, "walk me through what prints and why." There is no partial credit for vibes; you either trace the engine correctly or you don't.

The good news: this entire topic is a small set of deterministic rules. Hoisting has exactly three behaviors. this has exactly four binding rules plus one arrow-function exception. The event loop has one drain rule. Once those are airtight, every snippet — no matter how adversarial — becomes mechanical. This tutorial builds each model from the spec's point of view, then drills it with the exact snippet shapes interviewers use.

Signal to send in interviews: narrate your trace in engine terms ("this call pushes a frame… the await suspends and queues a microtask… the task ends, so we drain the microtask queue…"). Interviewers upgrade candidates who show the machine, not just the answer.

1. Execution model: contexts, stack, scope, hoisting & TDZ T1

Execution contexts and the call stack

Every time JS runs code it does so inside an execution context. There are three kinds: the global context (created once per realm), a function context (created per call), and eval (ignore it). Each context carries three things you should be able to name:

  • A Lexical Environment — where let/const/functions live, with a pointer to the outer environment (this pointer chain is the scope chain).
  • A Variable Environment — where var declarations live (function-scoped, hence the classic bugs).
  • A this binding — resolved at call time for normal functions (section 3).

Contexts are managed on the call stack: calling a function pushes a context, returning pops it. Creation happens in two passes per context:

  1. Creation phase — the engine scans declarations: var names are registered and initialized to undefined; function declarations are registered and initialized to the full function object; let/const/class names are registered but left uninitialized (the TDZ).
  2. Execution phase — statements run top to bottom, assignments happen.

Hoisting — precisely

Everything is hoisted, in the sense that the engine knows about every declaration before executing a line. What differs is initialization:

console.log(a);      // undefined  (var hoisted & initialized to undefined)
console.log(f());    // "hi"       (function declaration fully hoisted)
console.log(g);      // undefined  (only the var half of the expression hoisted)
console.log(b);      // ReferenceError: Cannot access 'b' before initialization

Proving the TDZ (let/const ARE hoisted)

A favorite senior follow-up: "if let isn't accessible early, is it even hoisted?" Yes — and here's the proof. If let x were simply "not hoisted," the inner console.log would find the outer x via the scope chain. Instead it throws, because the inner block's x is registered (shadowing the outer one) but sits in its Temporal Dead Zone from the start of the block until its declaration executes:

let x = "outer";
{
  // TDZ for the inner `x` starts at the top of this block
  console.log(x);   // ReferenceError — NOT "outer"!
  let x = "inner";  // TDZ ends here
}

Even typeof x throws inside the TDZ — unlike a truly undeclared identifier, where typeof missing safely returns "undefined". TDZ variables are more protected than nonexistent ones.

Function scope vs block scope

function demo() {
  if (true) {
    var v = 1;     // scoped to the FUNCTION
    let l = 2;     // scoped to the BLOCK
  }
  console.log(v);  // 1
  console.log(l);  // ReferenceError: l is not defined
}

The classic: var + setTimeout in a loop (and 3 fixes)

for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 3, 3, 3

Why: var i creates one function-scoped binding. All three arrow functions close over that single variable. The synchronous loop finishes (leaving i === 3) before any timer callback runs — timers can't run until the call stack is empty (section 5). Each callback then reads the same, final i.

Fix 1 — let (the modern answer): the spec gives let loop headers special treatment — a fresh binding per iteration, copied from the previous iteration's final value. Each closure captures its own i.

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);  // 0, 1, 2
}

Fix 2 — IIFE (the pre-ES6 answer): create a new scope per iteration by passing i as an argument; the parameter is a fresh variable each call.

for (var i = 0; i < 3; i++) {
  (function (j) {
    setTimeout(() => console.log(j), 0);  // 0, 1, 2
  })(i);
}

Fix 3 — setTimeout's extra arguments (the trivia answer): arguments after the delay are passed to the callback, evaluated now.

for (var i = 0; i < 3; i++) {
  setTimeout(j => console.log(j), 0, i);  // 0, 1, 2
}

Mentioning that let works because the spec creates a per-iteration environment record (not because "let is block scoped" in a hand-wavy sense) is a genuine differentiator. A block-scoped variable declared once per loop would still print 3,3,3 — the magic is the fresh copy per iteration.

2. Closures — deep T1

The precise definition

A closure is a function bundled with a reference to the Lexical Environment in which it was defined. When a function is created, it stores an internal [[Environment]] pointer to the current environment record. When called later — from anywhere — its scope chain starts from that stored environment, not from wherever it was called. That's it. Everything else (counters, modules, stale state) falls out of this.

Key consequence: an environment record normally dies when its function returns, but if any surviving function still points at it, the GC keeps it alive. Closures are how local variables outlive their stack frame.

Canonical examples: counter and module

function makeCounter() {
  let count = 0;                // lives in makeCounter's environment record
  return {
    inc: () => ++count,
    dec: () => --count,
    get: () => count,
  };
}
const c = makeCounter();
c.inc(); c.inc();
console.log(c.get());           // 2
console.log(c.count);           // undefined — truly private, not a property

const c2 = makeCounter();
console.log(c2.get());          // 0 — each call = fresh environment

This is the module pattern: private state plus a public API, and the standard way to get privacy before class #fields existed. All three methods share one environment (same count); separate calls to makeCounter create independent environments.

Closures capture VARIABLES, not values

The single most-tested closure fact. A closure holds a live link to the binding; it sees the current value at call time, not a snapshot from creation time:

function make() {
  let x = 10;
  const get = () => x;   // captures the variable x, not the number 10
  x = 20;
  return get;
}
console.log(make()());   // 20 — not 10

This is exactly why the var loop prints 3,3,3 and why React state goes stale (below).

Memory implications

A closure keeps its whole captured environment reachable. Closing over one small value inside a function that also holds a 50 MB parsed payload can retain far more than you intended:

function processData(hugeArray) {
  const summary = hugeArray.length;
  return function report() {
    // only uses `summary`, but naive mental model says hugeArray is retained too
    return `processed ${summary} rows`;
  };
}

Modern engines can free variables the closure provably never references — but the optimization is per-environment: if any closure in the same scope references hugeArray (even via eval or a debugger-visible path), all closures sharing that environment retain it. The safe senior habit: copy out what you need (const summary = ...) and let the big object go out of scope, or explicitly hugeArray = null after use in long-lived closures.

Long-lived closures are a top-3 frontend leak source: an event listener or interval callback closing over a large component scope keeps that scope alive as long as the listener is registered. Removing the listener releases it — forgetting to is the leak (section 8).

Closures in React — why stale state happens

Every render of a function component is a new function call, so every render creates fresh variables and fresh closures over them. An effect or timer created in render #1 closes over render #1's count — forever:

function Timer() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);   // `count` is render #1's count = 0, always
    }, 1000);                // so this sets 1, 1, 1, 1... forever
    return () => clearInterval(id);
  }, []);                    // empty deps = closure never refreshed
}

Fixes: functional update setCount(c => c + 1) (reads latest state from React, not from the closure), add count to deps (re-create the interval each change), or a ref for an escape hatch. The mechanism is pure closure semantics — nothing React-specific is "broken." Tutorial 08 builds on this.

Private state: closures vs #fields

// Pre-#fields: closure privacy (per-instance functions — small memory cost)
function BankAccount(opening) {
  let balance = opening;
  this.deposit = (n) => { balance += n; };
  this.getBalance = () => balance;
}

// Modern: #fields (methods stay on the prototype, shared)
class BankAccount2 {
  #balance;
  constructor(opening) { this.#balance = opening; }
  deposit(n) { this.#balance += n; }
  getBalance() { return this.#balance; }
}

Both give hard privacy (no reflection escape hatch, unlike _underscore conventions). #fields win on memory (shared prototype methods) and on instanceof-style brand checks (#x in obj).

3. this binding — the complete ruleset T1

For a normal function, this is determined entirely by how the function is called — never by where it's defined. Four rules, in precedence order:

#RuleTriggerthis is…
1new bindingnew Fn()The freshly created object
2Explicit bindingfn.call(obj), fn.apply(obj), fn.bind(obj)obj
3Implicit bindingobj.fn() — a receiver at the call siteThe object before the dot
4Default bindingPlain call fn()globalThis (sloppy) / undefined (strict)

Exception that overrides everything: arrow functions have no own this (and no own arguments, super, or new.target). They resolve this lexically — through the scope chain, like any other variable — from the enclosing normal function or module. call/apply/bind cannot change an arrow's this; new on an arrow throws.

Each rule in code

"use strict";
function whoAmI() { return this; }

// 4. Default: plain call
whoAmI();                          // undefined (strict); globalThis in sloppy mode

// 3. Implicit: receiver at the call site
const obj = { name: "obj", whoAmI };
obj.whoAmI();                      // obj — the object before the dot, at THIS call

// 2. Explicit: call/apply/bind beat implicit
const other = { name: "other" };
obj.whoAmI.call(other);            // other

// 1. new beats everything (even a bound this — see below)
function Person(name) { this.name = name; }
const p = new Person("Ada");       // this = the new object

What bind really does — and why it's permanent

function greet(greeting) { return `${greeting}, ${this.name}`; }
const ada = { name: "Ada" };
const bob = { name: "Bob" };

const greetAda = greet.bind(ada);
greetAda("Hi");                    // "Hi, Ada"

// Attempt to re-bind or override: IMPOSSIBLE for `this`
greetAda.call(bob, "Hi");          // "Hi, Ada" — call's thisArg ignored
greetAda.bind(bob)("Hi");          // "Hi, Ada" — re-bind wraps but inner binding wins

// The one thing that pierces bind: `new`
function Point(x) { this.x = x; }
const BoundPoint = Point.bind({ x: 999 });
const pt = new BoundPoint(5);
console.log(pt.x);                 // 5 — new ignores the bound this (partial-applied args DO still apply)

bind returns a new exotic bound function object wrapping the target: the wrapper hard-codes this (and any partially applied arguments), which is why later call/bind attempts can't reach the original. Interviewers love asking you to implement it — exercise 1.

Lost-this classics

class Store {
  items = [];
  add(item) { this.items.push(item); }
}
const store = new Store();

// 1. Extracted method — receiver lost
const add = store.add;
add("x");                            // TypeError: Cannot read properties of undefined (class bodies are strict)

// 2. Passed as a callback — same failure, just deferred
button.addEventListener("click", store.add);   // `this` = the button, not store
[1, 2, 3].forEach(store.add);                  // `this` = undefined

// Fixes:
button.addEventListener("click", store.add.bind(store));   // bind
button.addEventListener("click", (e) => store.add(e));     // arrow wrapper (implicit binding preserved)

The mental model that fixes everything: store.add as an expression is just a function value — the store. part only matters when the property access and the call happen together at the call site (store.add()). Separate them and the receiver is gone.

Class field arrows — the "fix" and its cost

class Store {
  items = [];
  // Field initializers run per-instance with this = the instance,
  // so the arrow lexically captures each instance:
  add = (item) => { this.items.push(item); };
}
button.addEventListener("click", store.add);   // works — this is locked to store

Cost: a normal method lives once on the prototype; a class-field arrow is a fresh function per instance (more memory, worse for many instances), it's an own property (shows up in spreads, shadows prototype methods, awkward to spy/override in tests, and super.add won't find it in subclasses). Fine for a handful of component instances; wrong default for a class instantiated thousands of times.

Getters/setters and event listeners

const user = {
  first: "Ada", last: "Lovelace",
  get fullName() { return `${this.first} ${this.last}`; }  // this = object the getter is invoked THROUGH
};
const clone = Object.create(user);
clone.first = "Grace";
console.log(clone.fullName);   // "Grace Lovelace" — this = clone (receiver), even though getter lives on user

// DOM listeners: normal function → this === e.currentTarget
button.addEventListener("click", function (e) {
  console.log(this === e.currentTarget);   // true
});
// Arrow listener → this is lexical (the enclosing scope), NOT the element
button.addEventListener("click", (e) => {
  console.log(this);                       // whatever `this` is outside — use e.currentTarget instead
});

4. Prototypes & inheritance T1

__proto__ vs prototype — the eternal confusion

Two completely different things with confusingly similar names:

  • [[Prototype]] (exposed as __proto__, properly read via Object.getPrototypeOf(obj)) — an internal slot every object has: the next object in its lookup chain.
  • Fn.prototype — an ordinary property that only functions have. It is not the function's own prototype; it's the object that will be assigned as [[Prototype]] to instances created via new Fn().
function Animal(name) { this.name = name; }
Animal.prototype.speak = function () { return `${this.name} makes a sound`; };

const rex = new Animal("Rex");

┌─────────────┐  .prototype   ┌──────────────────────┐
│  Animal (fn)│──────────────▶│ Animal.prototype     │──[[Prototype]]──▶ Object.prototype ──▶ null
└─────────────┘◀──────────────│  { speak, constructor}│
      ▲          .constructor └──────────────────────┘
      │                                  ▲
      │ [[Prototype]] of Animal itself   │ [[Prototype]] (aka rex.__proto__)
      │ is Function.prototype     ┌──────┴──────┐
      │                           │ rex         │
      │                           │ { name:"Rex"}│
                                  └─────────────┘

Object.getPrototypeOf(rex) === Animal.prototype;   // true
rex.prototype;                                     // undefined — rex isn't a function
Animal.__proto__ === Function.prototype;           // true — a function's OWN chain

Property lookup walk

rex.speak → engine checks rex's own properties → not found → follows [[Prototype]] to Animal.prototype → found. If not found there, continues to Object.prototype, then null, then yields undefined. Writes don't walk the chain: rex.speak = fn creates an own property that shadows the prototype's (exception: inherited setters intercept, and inherited non-writable data properties reject the write).

What new does — the 4 steps

  1. Create a fresh plain object.
  2. Set its [[Prototype]] to Fn.prototype.
  3. Call Fn with this = the new object (and new.target set).
  4. If Fn returns an object, that object is the result; otherwise (primitives/undefined ignored) return the new object from step 1.
function myNew(Fn, ...args) {
  const obj = Object.create(Fn.prototype);     // steps 1 + 2
  const result = Fn.apply(obj, args);          // step 3
  return (typeof result === "object" && result !== null) ? result : obj;  // step 4
}

Object.create and delegation without constructors

const canSpeak = { speak() { return `${this.name} speaks`; } };
const dog = Object.create(canSpeak);   // dog's [[Prototype]] = canSpeak
dog.name = "Rex";
dog.speak();                           // "Rex speaks"
Object.create(null);                   // object with NO prototype — no toString, no proto-pollution surface; great for dictionaries

class is (mostly) sugar — what it actually creates

class Animal {
  constructor(name) { this.name = name; }       // → the Animal function itself
  speak() { return `${this.name} speaks`; }     // → Animal.prototype.speak (NON-enumerable)
  static create(n) { return new Animal(n); }    // → Animal.create (on the constructor itself)
  #secret = 42;                                 // per-instance private slot, brand-checked
  getSecret() { return this.#secret; }
}
class Dog extends Animal {
  constructor(name) {
    super(name);        // calls Animal's constructor with this; MUST run before `this` is touched
  }
  speak() { return super.speak() + " (woof)"; } // super = lookup starting at Animal.prototype, this stays the Dog
}

Exactly what the sugar produces: a constructor function, methods on .prototype, statics on the constructor, and for extends: Dog.prototype.[[Prototype]] = Animal.prototype and Dog.[[Prototype]] = Animal (so statics inherit too). Where it's more than sugar: class bodies are always strict mode; classes throw if called without new; methods are non-enumerable and can't be used as constructors; class declarations have TDZ; #fields and super have no pre-class equivalent.

instanceof mechanics

x instanceof Fn walks x's prototype chain asking "is Fn.prototype anywhere on it?" (customizable via Symbol.hasInstance).

rex instanceof Animal;            // true: Animal.prototype is on rex's chain
rex instanceof Object;            // true: Object.prototype is further up
Object.create(null) instanceof Object;  // false — chain is just null
// Consequence: instanceof is unreliable across iframes/realms (each realm has its own Array, Object…)
Array.isArray(arr);               // realm-safe; prefer over `arr instanceof Array`

hasOwnProperty vs in

"speak" in rex;                        // true  — checks the WHOLE chain
rex.hasOwnProperty("speak");           // false — own properties only
Object.hasOwn(rex, "name");            // true  — modern, safe replacement (works on null-proto objects too)

Prototype pollution — the security angle

If attacker-controlled keys reach a recursive merge/assign, a key of "__proto__" can write onto Object.prototype itself — poisoning every object in the app (e.g., setting isAdmin: true globally, or breaking library checks). Real CVEs hit lodash merge, jQuery extend, and countless query-string parsers.

const payload = JSON.parse('{ "__proto__": { "isAdmin": true } }');
naiveDeepMerge({}, payload);
({}).isAdmin;    // true — every object in the realm now "has" isAdmin

Defenses: block the keys __proto__, prototype, constructor in merges; use Object.create(null) or Map for untrusted-key dictionaries; Object.freeze(Object.prototype) in hardened environments.

5. The event loop — definitive T1

The single most-asked JS interview topic. JS is single-threaded: one call stack, one thing at a time. Concurrency comes from the host (browser/Node) running work elsewhere and queueing callbacks. The moving parts:

  • Call stack — currently executing frames. The loop can't proceed until it's empty.
  • Web APIs / host — timers, fetch/network, DOM events run outside JS; when done, they enqueue callbacks.
  • Task queue(s) (a.k.a. macrotask) — setTimeout/setInterval, I/O completions, UI events, postMessage, script execution itself. One task is taken per loop turn.
  • Microtask queue — promise reactions (.then/.catch/.finally and everything after an await), queueMicrotask, MutationObserver callbacks.

THE rule (memorize verbatim): run one task to completion → drain the ENTIRE microtask queue (including microtasks queued by other microtasks) → maybe render → take the next task. Microtasks always win over waiting tasks; a microtask that queues microtasks forever will starve tasks and rendering completely.

Task (macrotask)Microtask
SourcessetTimeout/setInterval, DOM events, I/O, postMessage, MessageChannel, script tagsPromise callbacks, await continuations, queueMicrotask, MutationObserver

Rendering and requestAnimationFrame

Rendering (style → layout → paint) is not a task; it's an opportunity the browser takes between tasks (at most ~once per frame, ~16.7ms at 60Hz), always after microtasks drain. requestAnimationFrame callbacks run in a special slot just before that render step — which is why rAF is the right place for visual updates and setTimeout(fn, 0) is not (it lands in some future turn with no frame alignment). Order within a turn that renders: task → microtasks → rAF callbacks → style/layout/paint.

async/await desugared

await x means: wrap x in a resolved promise, suspend the function, and register the rest of the function body as a microtask continuation. The function returns a pending promise to its caller immediately. Mental rewrite:

async function f() {
  console.log("before");
  const v = await getValue();
  console.log("after", v);
}
// ≈ equivalent to:
function f() {
  console.log("before");
  return Promise.resolve(getValue()).then(v => {
    console.log("after", v);       // "the rest of the function" = a microtask
  });
}

Everything up to the first await runs synchronously in the caller's turn. Since ES2019, await on an already-resolved native promise (or a plain value) costs exactly one microtask tick; awaiting a foreign thenable costs extra ticks.

Walkthrough 1 — the baseline

console.log("1");
setTimeout(() => console.log("2"), 0);
Promise.resolve().then(() => console.log("3"));
console.log("4");

Output: 1, 4, 3, 2. The script itself is a task. Sync logs 1, registers a timer (task), queues a microtask, logs 4. Script task ends → drain microtasks → 3. Next task → 2. setTimeout(…, 0) can never beat a microtask.

Walkthrough 2 — microtasks spawning microtasks

Promise.resolve().then(() => {
  console.log("A");
  Promise.resolve().then(() => console.log("B"));
});
setTimeout(() => console.log("C"), 0);
Promise.resolve().then(() => console.log("D"));

Output: A, D, B, C. After sync: micro = [A-cb, D-cb], tasks = [C]. A-cb logs A and appends B-cb to the (still-draining) microtask queue → [D-cb, B-cb]. D logs, B logs — the drain continues until empty. Only then the task: C. Newly-queued microtasks jump ahead of every waiting task.

Walkthrough 3 — chains interleave tick by tick

Promise.resolve().then(() => console.log("p1")).then(() => console.log("p2"));
Promise.resolve().then(() => console.log("q1")).then(() => console.log("q2"));

Output: p1, q1, p2, q2. Each .then link is one microtask, queued only when its predecessor settles. Micro = [p1, q1] → p1 runs (queues p2) → q1 runs (queues q2) → p2 → q2. Chains of equal depth interleave like round-robin.

Walkthrough 4 — where await splits the function

async function f() {
  console.log("f1");
  await undefined;
  console.log("f2");
}
console.log("s1");
f();
Promise.resolve().then(() => console.log("m1"));
console.log("s2");

Output: s1, f1, s2, f2, m1. f() runs synchronously to the await: logs f1, then suspends, queueing its continuation (f2) — before the next line queues m1. Micro = [f2-cont, m1]. Sync finishes with s2, then f2, then m1. Common wrong answer: putting f2 after m1 — but f's continuation was enqueued first.

Walkthrough 5 — awaited values vs promises (post-ES2019 they cost the same)

async function one() { await Promise.resolve(); console.log("one"); }
async function two() { await null; console.log("two"); }
one();
two();
Promise.resolve().then(() => console.log("three"));

Output: one, two, three. Both awaits cost exactly one tick (the ES2019 await optimization treats native resolved promises and plain values identically), so continuations queue in call order, then the .then. Pre-2019 engines printed two, three, one (awaiting a promise cost 3 ticks) — worth mentioning if the interviewer looks old-school; today's engines all print one/two/three.

Walkthrough 6 — returning a promise from .then costs ~2 extra ticks

Promise.resolve()
  .then(() => { console.log("a"); return Promise.resolve("x"); })
  .then(v => console.log("b", v));

Promise.resolve()
  .then(() => console.log(1))
  .then(() => console.log(2))
  .then(() => console.log(3))
  .then(() => console.log(4));

Output: a, 1, 2, 3, b x, 4. Resolving a promise with another promise isn't instant: the spec queues (1) a job that calls inner.then(resolveOuter), then (2) the reaction that actually resolves the outer — two extra microtask ticks before b's callback can even be queued. So b lands after 1, 2, 3 (which each cost one tick) but before 4. This exact snippet (as "0 1 2 3 4 5 6") is a famous hard-mode drill; if you can trace this, you own the event loop.

Node.js differences — awareness level

  • Node's loop (libuv) runs phases: timers → pending callbacks → poll (I/O) → check (setImmediate) → close callbacks. Microtasks still drain between every callback.
  • process.nextTick is its own queue that drains before promise microtasks — "microtask VIP lane" (and can starve promises if abused).
  • setImmediate vs setTimeout(fn, 0): inside an I/O callback, setImmediate always fires first (check phase follows poll); at top level the order is nondeterministic (timer clamp race).

Starving the loop — the performance tie-in

Any task (or microtask drain) over ~50ms is a "long task": input events queue behind it and INP (Interaction to Next Paint) tanks. Fixes: chunk work and yield with await scheduler.yield() (or the setTimeout(0)/MessageChannel fallback — crucially these yield as tasks, letting input and paint through, whereas chunking with microtasks yields nothing), move work to a Web Worker, or use isInputPending(). Note the subtle trap: await-ing resolved promises in a loop keeps you in the same microtask drain — the browser never paints.

Interview gold: "microtasks run before rendering, so a microtask loop blocks paint just as badly as sync code; to actually yield to the browser you must yield at task granularity." Very few candidates articulate this.

6. Coercion & equality T2

The == algorithm — the highlights that matter

  1. Same types → behaves like ===.
  2. null == undefined → true, and they equal nothing else (not 0, not "", not false).
  3. Number vs string → string is converted to number.
  4. Boolean vs anything → boolean converted to number first (true→1, false→0).
  5. Object vs primitive → object converted via ToPrimitive (Symbol.toPrimitive, else valueOf, else toString).
[] == false     // true:  false→0; []→""(toString)→0; 0 == 0
[] == ![]       // true:  ![] is false (arrays are truthy!), then same as above
"" == 0         // true:  ""→0
"0" == false    // true:  false→0, "0"→0
null == 0       // false: null only equals undefined

NaN == NaN      // false — NaN equals nothing, even itself

React's bail-out checks (useState setter, deps arrays) use Object.is — which is why setState(NaN) when state is already NaN does not re-render, and why a new object literal always does.

Truthiness — the complete falsy list

Exactly eight falsy values: false, 0, -0, 0n, "", null, undefined, NaN. Everything else is truthy — including "0", "false", [], {}, and new Boolean(false) (it's an object!).

"When would you ever use ==?" — the senior answer

"Almost never — I lint with eqeqeq. The one deliberate exception is x == null, which is the idiomatic check for 'null or undefined' in a single comparison, because the spec guarantees null and undefined coerce-equal only each other. Any other == in a codebase is a bug waiting for a code review." Bonus: in TS codebases x == null correctly narrows T | null | undefined to T.

Property descriptors

Every data property has four attributes: value, writable (can assign?), enumerable (shows in for…in/Object.keys/spread?), configurable (can delete/redefine?). Accessor properties swap value/writable for get/set.

const obj = {};

  value: 7, writable: false, enumerable: false, configurable: false
});

delete obj.id;             // false — non-configurable
// Note: defineProperty defaults are all FALSE; literal properties default all TRUE.
// Class methods are writable+configurable but NOT enumerable.

Computed keys & key ordering

// Own-key ordering rule (Object.keys / for-in / spread / JSON.stringify):
// 1. integer-like string keys, ASCENDING numerically
// 2. remaining string keys, INSERTION order
// 3. symbol keys, insertion order (excluded from Object.keys)

WeakMap / WeakSet — metadata without leaks

A WeakMap holds its keys weakly: if nothing else references a key object, the GC collects it and the entry vanishes. No .size, no iteration (contents are nondeterministic by design). Perfect for attaching data to objects you don't own:

const nodeMeta = new WeakMap();
function track(el) {
  nodeMeta.set(el, { renderedAt: Date.now() });   // annotate a DOM node
}
// When the node is removed from the DOM and dereferenced, BOTH the node
// and its metadata are collected. A regular Map here would pin every node
// forever — a textbook leak.

Other uses: per-instance private data (pre-#fields), memoization caches keyed by object arguments, "have I seen this object" via WeakSet.

Iterators & generators

Protocol: an object is iterable if it has [Symbol.iterator]() returning an iterator — an object with next() returning { value, done }. for…of, spread, destructuring, Array.from, Promise.all all consume it.

class Range {
  constructor(from, to) { this.from = from; this.to = to; }
  *[Symbol.iterator]() {                     // generator = easiest way to implement an iterator
    for (let n = this.from; n <= this.to; n++) yield n;
  }
}
[...new Range(1, 4)];        // [1, 2, 3, 4]

function* take(iterable, n) {                // generators are LAZY — pull-based
  let i = 0;
  for (const v of iterable) {
    if (i++ >= n) return;
    yield v;
  }
}
function* naturals() { let n = 0; while (true) yield n++; }   // infinite is fine…
[...take(naturals(), 3)];    // [0, 1, 2] …because consumers pull only what they need

Generator use cases: lazy/infinite sequences, pagination cursors, custom iteration for data structures, coroutine-style control flow (redux-saga), and async generators for streams (for await…of).

Proxy & Reflect

A Proxy wraps a target and intercepts fundamental operations via traps (get, set, has, deleteProperty, apply, construct…). Reflect mirrors those operations as functions — the correct way to forward to default behavior inside a trap.

const user = { name: "Ada", age: 36 };
const validated = new Proxy(user, {
  set(target, prop, value, receiver) {
    if (prop === "age" && (!Number.isInteger(value) || value < 0)) {
      throw new TypeError("age must be a non-negative integer");
    }
    console.log(`set ${String(prop)} = ${value}`);       // logging trap
    return Reflect.set(target, prop, value, receiver);   // forward correctly (handles getters/receiver)
  }
});
validated.age = 37;      // logs, then sets
validated.age = -1;      // TypeError

Framework relevance: Vue 3's reactivity wraps state in a Proxy — get traps record which effect read which key (dependency tracking), set traps trigger re-runs. MobX 5+ does the same. This is why Vue 3 tracks property additions and array indices that Vue 2's defineProperty approach couldn't. Limits worth knowing: proxies can't intercept private #fields or built-in internal slots (so wrapping a Map naively breaks it), and identity differs (proxy !== target).

8. Memory & garbage collection T2

Reachability & mark-and-sweep

JS GC is based on reachability, not reference counting: starting from roots (global object, current stack, active closures' environments), the collector marks everything transitively reachable and sweeps the rest. Cycles are collected fine (A↔B referencing each other but unreachable from roots → both freed) — "circular reference leak" is an obsolete IE6 answer. The corollary that matters: a leak in JS is always a reachable-but-unwanted reference. You never free memory; you sever paths from roots.

V8 specifics — awareness level

V8 is generational: new objects go to a small "new space" collected frequently by a fast copying Scavenger (most objects die young); survivors get promoted to "old space," collected by mark-sweep-compact, done mostly incrementally and concurrently (Orinoco) to keep pauses off the main thread. Practical takeaways: short-lived garbage is cheap; churning long-lived objects is what hurts; huge allocation spikes trigger major GC pauses visible as frame drops.

The four classic frontend leaks

LeakMechanismFix
Detached DOM nodesJS variable/array/Map still references a node removed from the document — the node (and often its whole subtree) can't be freedNull out refs on teardown; use WeakMap/WeakRef for node metadata
Forgotten listeners & timersaddEventListener on a long-lived target (window, document, store) with a callback closing over component scope; setInterval never clearedSymmetric cleanup (React effect cleanup fn), AbortController signal for listeners, clear intervals
Closures over large dataLong-lived callback retains a big environment (section 2)Extract only needed values; null out big refs
Unbounded cachesModule-level Map/array/memoizer that only growsLRU with a cap, WeakMap keys, TTL eviction

Finding leaks — the DevTools workflow

  1. Suspect: Performance panel memory graph (or performance.memory) sawtooths ever upward across interactions that should be net-zero.
  2. Prove: Memory panel → the 3-snapshot technique: snapshot → do the action + undo it (open/close modal) → snapshot → repeat → snapshot. Compare 2 vs 3; leaked objects grow linearly with repetitions. Filter for "Detached" to find detached DOM trees.
  3. Diagnose: select a leaked object and read its Retainers tree — the reference path from a GC root that keeps it alive. That path names the culprit (e.g., window → myStore → listeners[3] → context → bigComponentScope).

WeakRef & FinalizationRegistry — awareness

const ref = new WeakRef(bigObject);        // doesn't prevent collection
const maybe = ref.deref();                 // the object, or undefined if collected

const registry = new FinalizationRegistry((label) => {
  console.log(`${label} was collected`);   // cleanup callback SOME time after GC
});
registry.register(bigObject, "bigObject");

Legit uses: caches that shouldn't pin entries, mapping to external resources (Wasm memory, workers). The senior caveat: GC timing is nondeterministic and unobservable by spec — never build program correctness on WeakRef/finalizers; TC39's own proposal text says to avoid them if possible.

9. Modules: ESM vs CJS T2

ESM semantics

ES modules are static: imports/exports are declared syntax, resolved at parse time, before any code runs. This enables tree-shaking, cyclic-safe linking, and toolable analysis — and it's why you can't conditionally import x from cond ? "a" : "b" (use dynamic import() for that). Modules are singletons (evaluated once per graph, cached), always strict mode, with module-level this === undefined.

Live bindings — prove it

ESM exports are live read-only views onto the exporting module's bindings — not copies:

// counter.mjs
export let count = 0;
export function inc() { count++; }

// main.mjs
import { count, inc } from "./counter.mjs";
console.log(count);   // 0
inc();
console.log(count);   // 1  ← the import SEES the update (live binding)
count++;              // TypeError: Assignment to constant variable / read-only binding

The equivalent CJS (exports.count = 0 then count++ internally… actually reassigning a primitive on module.exports) hands consumers a value copy at require time — destructured const { count } = require("./counter") stays 0 forever. That difference is the #1 source of "why is my mocked/updated value stale" confusion in mixed codebases.

ESM vs CJS — the comparison table

Exports areLive bindingsValues snapshotted when require returns
Conditional importOnly via import()if (x) require(…) just works
Tree-shakingYes (static structure)Effectively no
Top-level awaitYesNo (require is sync)

Miscimport.meta.url__dirname, __filename, require.cache

Interop pain — the default-export mismatch: CJS has no real "default", so when a transpiler compiles export default fn it emits exports.default = fn plus an __esModule flag; consuming that from real CJS gives you { default: fn }, hence the classic require("lib").default dance and "lib_1.default is not a function" errors. Conversely, Node importing a CJS module from ESM synthesizes module.exports as the default export. Dual-package setups can even load both copies (the "dual package hazard") breaking instanceof across them.

Circular imports

ESM: the graph links all bindings before evaluation, so cycles resolve — but a module that runs early may read a binding of its cyclic partner while it's still in TDZ (ReferenceError) or, for var/hoisted functions, see them fine. Function declarations across a cycle work; top-level const reads at import-evaluation time can throw. CJS: a cycle returns the partial module.exports as it existed when the cycle was hit — silently missing later exports (classic "undefined is not a function" hours into debugging). In both systems the real fix is to break the cycle (extract shared code) or defer access until runtime (call-time, not load-time).

Dynamic import() and top-level await

const module = await import(`./locales/${lang}.js`);  // returns a promise of the namespace object

// Top-level await (ESM only): the whole module — and its importers — wait
const config = await fetch("/config.json").then(r => r.json());
export default config;

import() is the primitive under every code-splitting feature (React.lazy, route-based chunks). Bundler mechanics are covered in tutorial 16.

What strict mode actually changes

  • Assigning to undeclared variables throws (no accidental globals); writes to non-writable/frozen properties throw instead of silently failing.
  • Default this in plain function calls is undefined, not globalThis.
  • delete variable, duplicate parameter names, octal literals like 012 → SyntaxError; with banned.

  2. 1-cb: logs 1 → queues 2-cb. Queue: [J1, 2-cb].
  3. J1: subscribes to P; P is already resolved so its reaction J2 is queued. Queue: [2-cb, J2].
  4. 2-cb: logs 2 → queues 3-cb. Queue: [J2, 3-cb].
  5. J2: outer promise finally resolves with "x" → queues b-cb. Queue: [3-cb, b-cb].
  6. 3-cb: logs 3 → queues 4-cb. Queue: [b-cb, 4-cb].
  7. b-cb: logs "b x". Then 4-cb: logs 4.

Moral: return somePromise inside .then = +2 microtask ticks (thenable-adoption job + reaction). Using return await somePromise... behaves similarly (+ticks). If an interviewer gives you this, narrate the adoption job explicitly — it's the whole question.

Interview Q&A

What is an execution context and what does it contain?

The environment a piece of code runs in: a Lexical Environment (let/const/function bindings + pointer to the outer environment — the scope chain), a Variable Environment (var bindings), and a this binding. Global context is created once; a function context per call. Contexts live on the call stack; creation happens in two passes (declaration registration, then execution), which is what produces hoisting behavior.

Explain hoisting precisely — including let/const.

All declarations are registered during the creation phase, so the engine knows every name before executing a line. The difference is initialization: var → initialized to undefined; function declarations → initialized to the complete function object (callable early); let/const/class → registered but uninitialized until their declaration executes — accessing them in that window (the Temporal Dead Zone) throws a ReferenceError. Proof they're hoisted: an inner let x makes an earlier read in the same block throw rather than resolve to an outer x.

What is a closure and where do you actually use them?

A function plus a reference to the lexical environment where it was defined (its [[Environment]] slot); called later, it resolves free variables through that captured environment, keeping it alive past the enclosing call. Uses: private state (module pattern, memoization caches), factories (once, debounce, curried functions), event/callback state, and every React hook — a component render's state values are visible to effects and handlers purely via closure. Crucial nuance: closures capture variables (live bindings), not values.

Why does the var-in-a-loop setTimeout print the final value, and what are the fixes?

var creates one function-scoped binding shared by all iterations; timers can't run until the sync loop finishes (call stack must empty), so all callbacks read the final value. Fixes: (1) let — spec creates a fresh binding per iteration; (2) an IIFE passing the value as a parameter (fresh scope per iteration); (3) setTimeout's extra arguments, evaluated at scheduling time. Bonus: .forEach with a callback parameter achieves the same as (2).

How is this determined? Give the precedence order.

For normal functions, entirely by the call site, in precedence: (1) new → the fresh instance; (2) explicit — call/apply/bind → the given object; (3) implicit — obj.fn() → the receiver before the dot; (4) default — plain fn() → undefined in strict mode, globalThis in sloppy. Arrow functions opt out entirely: no own this, resolved lexically from the enclosing scope, immune to call/apply/bind and illegal with new. Precedence proof: new even overrides a bound this.

call vs apply vs bind?

call(thisArg, a, b) and apply(thisArg, [a, b]) invoke immediately — identical except argument style (spread syntax has mostly obsoleted apply). bind(thisArg, ...partial) does NOT invoke: it returns a new bound function with this (and optionally leading arguments) permanently fixed — used for callbacks executed later.

Can you re-bind a function that's already bound?

No. bind returns an exotic wrapper hard-wired to the original target and thisArg; a second bind or a call just wraps/invokes the wrapper, which ignores the new thisArg. The single exception: constructing with new — the bound this is discarded in favor of the new instance (partially applied arguments still apply).

Arrow functions vs regular functions — all the differences?

Arrows have no own this, arguments, super, or new.target (all lexical); can't be constructors (no new, no prototype property); can't be generators; and as object-literal methods they don't see the object as this. Practical guidance: arrows for callbacks and anywhere you want lexical this; regular methods on prototypes/classes; class-field arrows only when you need a pre-bound handler per instance and accept the per-instance memory + testing/override costs.

Explain __proto__ vs prototype.

__proto__ (properly Object.getPrototypeOf) is the internal [[Prototype]] link every object has — the next stop in property lookup. Fn.prototype is a regular property only functions have: the object that will become the [[Prototype]] of instances created by new Fn(). So rex.__proto__ === Animal.prototype, while Animal.__proto__ === Function.prototype (the function's own chain). One is a link, the other is a template-to-be-linked.

What exactly does the new operator do?

Four steps: (1) create a fresh object; (2) set its [[Prototype]] to Fn.prototype; (3) invoke Fn with this bound to that object (and new.target set); (4) return the object — unless the function explicitly returned a different object, which wins (returned primitives are ignored). Being able to write this as a myNew(Fn, ...args) helper is a common follow-up.

How does instanceof work, and when is it unreliable?

x instanceof Fn walks x's prototype chain checking each link against the current Fn.prototype (overridable via Symbol.hasInstance). Unreliable when: prototypes are reassigned after construction; across realms/iframes (each has its own Array etc. — use Array.isArray); on null-prototype objects; and with primitives ("a" instanceof String is false). For brand-checking class instances robustly, use a #field in obj check.

Is class just syntactic sugar over prototypes?

~90% sugar: it produces a constructor function, prototype methods, statics on the constructor, and for extends both SubClass.prototype → Super.prototype and SubClass → Super links. Beyond sugar: class bodies are always strict; calling without new throws; methods are non-enumerable and non-constructible; class bindings have TDZ; derived constructors must call super() before touching this; and #private fields plus super method dispatch have no faithful pre-class emulation.

Define microtask vs (macro)task, and state the event loop's core rule.

Tasks: setTimeout/setInterval callbacks, DOM/UI events, I/O completions, postMessage, script execution. Microtasks: promise reactions and await continuations, queueMicrotask, MutationObserver. Core rule: run one task to completion, then drain the entire microtask queue — including microtasks enqueued during the drain — then optionally render (rAF callbacks immediately before paint), then take the next task. Consequences: a microtask always beats a pending setTimeout-0; a self-perpetuating microtask chain starves rendering and input entirely.

What does await actually do under the hood?

An async function runs synchronously until the first await. There, the awaited value is wrapped via Promise.resolve-like semantics, the function suspends, and the remainder of its body is registered as a microtask continuation (with resume-or-throw depending on settlement); the caller immediately receives the function's still-pending promise. Post-ES2019, awaiting a plain value or an already-resolved native promise costs exactly one microtask tick. So await is essentially compiler-generated .then chaining over a state machine — plus try/catch mapping to rejections.

Why doesn't setTimeout(fn, 0) run immediately?

Three reasons: (1) the callback is a task — it can't run until the current task finishes and the call stack is empty; (2) all pending microtasks drain first; (3) timers have clamping — browsers clamp nested timeouts (depth > 5) to a 4ms minimum, and background tabs throttle far more aggressively (≥1s). So "0" means "as soon as the loop gets around to it," never "now."

Node.js: process.nextTick vs setImmediate vs setTimeout(0)?

process.nextTick — its own queue drained before promise microtasks, i.e., soonest of all (and easiest to starve the loop with). Promise microtasks next. setTimeout(0) runs in the timers phase; setImmediate in the check phase after poll — so inside an I/O callback setImmediate reliably beats setTimeout(0), while at top level their order is a nondeterministic race with the 1ms timer clamp. Naming is historically backwards (nextTick is more immediate than setImmediate).

== vs === vs Object.is — and when would you use ==?

===: no coercion; NaN ≠ NaN; +0 === −0. Object.is: like === but NaN equals NaN and ±0 differ (React's bail-out comparison). ==: coercing algorithm — booleans→numbers, string↔number conversion, ToPrimitive on objects, and the special rule that null/undefined equal only each other. Senior answer for using ==: only x == null as the deliberate "null or undefined" check, everything else banned by eqeqeq linting.

When would you choose Map over a plain object?

When keys are dynamic or untrusted (no prototype-pollution/injection surface), non-string (objects, numbers kept as numbers), when you need reliable insertion-order iteration (objects reorder integer-like keys), O(1) .size, or heavy add/delete churn (engines optimize Map for it; objects suffer hidden-class churn). Plain objects remain right for fixed-shape records, JSON round-trips, and destructuring ergonomics. WeakMap when keys are objects whose lifetime you don't control.

What problems do WeakMap/WeakSet solve?

Associating data with objects without extending their lifetime. Keys are held weakly: when the key becomes otherwise unreachable, the entry is collected. Canonical cases: metadata for DOM nodes (a normal Map would pin removed nodes forever — a leak), private per-instance data, caches keyed by object identity, seen-object tracking in graph traversal/cloning. Deliberate limitations — no size, no iteration — exist because contents depend on nondeterministic GC timing.

How does JS garbage collection work, and what leaks do you look for in a frontend app?

Reachability-based mark-and-sweep from roots (globals, stack, live closures); cycles are handled fine. V8 is generational (fast scavenger for the young space, incremental/concurrent mark-sweep-compact for old space). A "leak" is always a reachable-but-unwanted reference: detached DOM nodes still referenced from JS, listeners/intervals on long-lived targets closing over component scopes, long-lived closures over large data, unbounded module-level caches. Diagnosis: DevTools Memory panel, 3-snapshot comparison across a repeatable action, filter "Detached," then read the Retainers path to find what's pinning the object.

Key differences between ESM and CommonJS?

ESM is static (parse-time graph → tree-shaking, no conditional import syntax, async-capable loading, top-level await) with live read-only bindings; CJS is dynamic (sync require anywhere) exporting value snapshots. ESM is always strict with import.meta; CJS has __dirname/require.cache. Interop pain: CJS lacks a true default export, so transpiled defaults become .default properties (__esModule flag), and dual packages risk loading two copies. Circular deps: ESM pre-links bindings (may hit TDZ); CJS returns partial exports silently.

Is Object.freeze deep? How would you deep-freeze?

Shallow — only the object's own properties are locked; nested objects stay mutable. Deep-freeze = recursively freeze children first (post-order), guarding against cycles with a WeakSet, and skipping already-frozen objects. Production note: freezing large graphs costs time and de-optimizes some engine fast paths — in app code, immutability by convention/types (readonly, Immer) usually beats runtime freezing; freeze shared constants and use it in dev to catch accidental mutation.

What are Symbols for? Name well-known symbols.

Guaranteed-unique property keys: collision-free metadata on objects you don't own, hidden from Object.keys/for…in/JSON.stringify (though discoverable via getOwnPropertySymbols — hiding, not security). Well-known symbols are protocol hooks the language invokes: Symbol.iterator/asyncIterator (for-of / for-await), Symbol.toPrimitive (coercion), Symbol.toStringTag (branding), Symbol.hasInstance (instanceof), plus regex hooks like Symbol.replace. Symbol.for(key) uses a global cross-realm registry.

How do frameworks like Vue use Proxy?

Vue 3's reactive() wraps state in a Proxy: the get trap records "current effect depends on this object+key" (dependency tracking), the set/deleteProperty traps look up and re-run subscribed effects (targeted re-render). Versus Vue 2's defineProperty: Proxies catch property additions/deletions and array index writes without Vue.set. Caveats worth volunteering: proxy identity ≠ target identity, built-ins with internal slots (Map/Set) need special-cased traps, and reads are marginally slower than raw access — hence Vue's shallowRef/markRaw escape hatches.

structuredClone vs JSON.parse(JSON.stringify())?

structuredClone (the postMessage/IndexedDB algorithm) correctly clones Dates, Map/Set, RegExp, typed arrays/ArrayBuffers, Blobs, NaN/Infinity/undefined, and — critically — handles circular references. JSON round-trip corrupts all of those (dates→strings, Map→{}, undefined dropped, NaN→null) and throws on cycles. Neither clones functions (JSON drops silently; structuredClone throws) and neither preserves prototypes — class instances come back as plain objects. It's also generally faster than stringify/parse for large object graphs.

Which array methods mutate, and what changed in ES2023?

Mutators: push, pop, shift, unshift, splice, sort, reverse, fill, copyWithin. Non-mutating: map, filter, slice, concat, flat, flatMap (plus readers like reduce/find/includes). ES2023 added immutable twins — toSorted, toReversed, toSpliced, with(index, value) — designed for immutable-state code so you stop writing [...arr].sort(). The trap that motivates them: sort/reverse mutate AND return the same reference, so setState(items.sort(...)) both corrupts previous state and can skip re-render (same reference under Object.is).

Hands-on exercises

Exercise 1 — implement Function.prototype.myBind

Write myBind(thisArg, ...partial) matching native semantics: returns a new function with locked this and partial args; later call/bind can't override; and — the senior part — when the bound function is invoked with new, the bound this is ignored and the instance's prototype chain points at the original function's prototype. Hint: detect construction via this instanceof boundFn or new.target. Verify against native bind with a test that does new BoundPoint(5).

Exercise 2 — event-loop trace by hand, then verify

Without running it, write the exact output of a snippet combining: two setTimeout(0)s, a three-link promise chain, an async function with two awaits, and a queueMicrotask inside one of the .thens. Maintain explicit "task queue / microtask queue / stack" columns on paper for every step. Then run it in the browser console AND in Node — outputs should match. Finally add a process.nextTick (Node) and predict where it lands before running.

Exercise 3 — once and memoize with proper closure hygiene

Implement (a) once(fn) — first call runs and caches; later calls return the cached result — and make it release its reference to fn after first call (why does that matter for memory?). (b) memoize(fn) that uses a Map for primitive single arguments and a WeakMap when the argument is an object — explain what leak the WeakMap variant prevents. Both must preserve this (test as an object method) — decide: arrow or function, and why?

Exercise 4 — mini EventEmitter with leak-safe subscriptions

Build class Emitter with on(event, fn) returning an unsubscribe function, once, off, and emit. Requirements: emit must tolerate a listener unsubscribing itself mid-emit (iterate a copy); once must remove before invoking (re-entrancy); listeners run with a documented this. Then write the leak scenario: a long-lived emitter, a "component" object subscribing with a method that closes over a large array, and show that forgetting unsubscribe pins the array — confirm with a DevTools heap snapshot retainer path.

Exercise 5 — reproduce and fix a detached-DOM leak

Make a page with a button that renders a list of 1,000 items into a container and a "clear" button that does container.innerHTML = "" — but first stashes container.firstChild into a module-level array ("for undo"). Click render/clear 5 times. Take heap snapshots, filter "Detached," and follow the Retainers path to your array. Fix it three ways: cap the undo stack, switch to WeakRefs, and null the array on a timer — compare snapshots after each fix and note which actually returns memory.
