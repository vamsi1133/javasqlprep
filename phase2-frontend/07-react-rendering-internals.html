07 — React Rendering Internals

What actually happens between setState and pixels — elements, reconciliation, Fiber, and the commit pipeline.

Why interviewers ask this

You can ship React apps for years without knowing what a Fiber is. Senior interviews exist to find out whether you're operating on cargo-culted rules ("wrap it in memo", "don't use index keys") or on an accurate mental model you can reason from when the rules run out. Internals questions are the fastest way to tell the difference: someone with the real model can explain why a child re-renders when no props changed, why an effect runs twice in dev, why an input's value jumps to the wrong row — and, crucially, predict behavior in code they've never seen.

At FAANG, these show up as "walk me through what happens when you call setState" and predict-the-output drills. At product startups, they show up as debugging exercises: here's a jank/state-loss/ double-fetch bug, find it. Both are testing the same thing — do you know the machine you drive? This tutorial builds that model bottom-up: elements → renders → diffing → Fiber → commit → the DOM layer. Everything here is accurate for React 18/19 with createRoot; where 17-era behavior differs (batching, event delegation, pooling) it's called out explicitly, because "what changed in 18" is itself a classic question.

1. JSX, elements, components, instances T1

Everything in this tutorial rests on one fact: JSX is not HTML, not a template language, and not magic — it's syntax sugar for function calls that return plain JavaScript objects. If you get this section exactly right, half of React's "surprising" behavior stops being surprising.

What the compiler does with JSX

Babel/SWC/TypeScript transpile JSX before React ever sees it. Since React 17, the default is the automatic runtime: JSX compiles to calls to jsx() / jsxs() imported from react/jsx-runtime (before that, to React.createElement — which still works and is why old code needed import React in every file). 

// What you write:
const el = <button className="primary" onClick={handle}>Save</button>;

// Automatic runtime (React 17+) — what the compiler emits:
import { jsx as _jsx } from "react/jsx-runtime";
const el = _jsx("button", { className: "primary", onClick: handle, children: "Save" });

// Classic runtime — equivalent:
const el = React.createElement("button", { className: "primary", onClick: handle }, "Save");

An element is a plain, frozen object

The return value is a small immutable descriptor — no DOM node, no class instance, no hidden machinery: 

{
  $$typeof: Symbol.for('react.element'), // 'react.transitional.element' in React 19
  type: 'button',          // string for host elements, function/class for components
  key: null,
  props: { className: 'primary', onClick: handle, children: 'Save' },
  ref: null                 // in React 19, ref is just a normal prop inside props
}
$$typeof is a security measure, not bookkeeping. It's a Symbol, and symbols can't survive JSON serialization — so attacker-controlled JSON from a server can never smuggle in a fake "element" that React would render (which could inject dangerouslySetInnerHTML). Knowing this is a nice senior flex.

In development, React Object.freezes elements and their props. That's the mechanical reason you can't mutate props — but the deeper reason is architectural: an element describes what the UI should look like at one point in time. React compares element trees across renders to decide what to do; if you mutated an element in place, React couldn't tell "before" from "after" and the diffing model collapses. Cheap immutable descriptions, recreated every render, compared by React — that's the whole design.

<Button /> does not call Button. It creates { type: Button, props: {} }. React calls Button(props) later, during the render phase, when it's this element's turn in the work loop. This gap — element creation vs component invocation — is exactly why the children optimization works, and why calling a component directly ({Button(props)} instead of <Button />) subtly breaks things: React sees the output of Button inlined into the parent, so Button gets no fiber of its own — its hooks get attributed to the parent, and conditional rendering of it will corrupt hook order.

Referential identity — the foundation for everything later

Because JSX creates a new object every time it runs, two renders of the same JSX produce elements that are ===-unequal even if every prop is identical. React's bailout checks (memo, the same-element bailout) compare references, not deep-equality. So: 

// Every render of Parent creates a NEW element object here:
<Child items={[1, 2, 3]} />   // new element AND new array prop each time

// But an element held in a variable/state keeps its identity across renders:
const [modal] = useState(() => <Modal />);  // same element object forever

Corollary you'll use constantly: if React receives the exact same element object it saw last render, it can skip re-rendering that subtree entirely (the props can't have changed — the object is frozen and identical). This single fact powers the children-composition optimization in section 9 and is the honest answer to "how do I avoid re-renders without memo".

2. The render model — what "rendering" actually means T1

This is the single most important mental model in the file. Most React confusion — and most wrong interview answers — comes from an imprecise definition of "render". Here is the precise one:

A render is React calling your component functions to compute the next element tree. It is a pure computation. Nothing touches the DOM during a render. "Re-render" ≠ "repaint" ≠ "DOM update" — a component can re-render 50 times and produce zero DOM mutations if its output didn't change.

Exactly three things trigger a re-render

  1. A state update in the component (useState/useReducer setter, class setState) — with one caveat below.
  2. Its parent re-rendered — and didn't bail out for this child.
  3. A context it consumes changed value (compared with Object.is on the Provider's value).

That's the whole list. Props are conspicuously absent — see the misconception below. (Force-update mechanisms and hot reload are just state updates under the hood.)

The Object.is bail-out

const [count, setCount] = useState(0);

setCount(0);  // same value per Object.is → React bails out, no re-render
setCount(1);  // different → schedules a render
setCount(1);  // later, again 1 → bails out

// Objects compare by reference:
setUser({ ...user });        // NEW object → re-renders even if fields identical
setUser(user);               // same reference → bails out
The bail-out has a nuance worth volunteering: when you set state to the same value, React may still render that one component (and run its function body) before deciding its children can be skipped — it needs a render to confirm nothing changed in some edge cases (e.g., a previous update in the queue). So "same value → guaranteed zero renders" is slightly too strong; "same value → React bails out and won't render the children, and usually skips the component too" is the accurate phrasing. Never rely on the extra invocation either way — renders must be pure precisely so this is unobservable.

The #1 misconception: "components re-render when their props change"

Backwards. When a component renders, React renders all of its children by default — regardless of whether their props changed. Props changing is not a trigger; it's a consequence of the parent re-rendering (the parent re-ran, so it created new elements with possibly-new props). React does not sit around watching props. There is no dirty-checking of props. The cascade is top-down and unconditional unless something stops it: 

          setState in B
               │
   A           ▼            A   — not rendered (state lives in B; parents never
   └─ B ◄── re-renders            re-render because a child updated)
      ├─ C  ── re-renders   C   — re-renders even though its props are identical
      └─ D  ── re-renders   D   — re-renders even though it takes NO props at all
         └─ E ── re-renders E   — the cascade goes all the way down

Only three things stop the cascade at a subtree:

  1. React.memo — shallow-compares props, skips the render if all are Object.is-equal.
  2. Same-element bailout — the child element object is referentially identical to last render's (typically because it came through children or was hoisted — section 9).
  3. (Class components) shouldComponentUpdate / PureComponent.
A memoized child still re-renders if any prop is a freshly-created reference — inline object/array/function props defeat memo every time. That's the entire reason useCallback/useMemo exist: they are referential-stability tools for crossing memoization boundaries, not general "make it faster" knobs. Saying that sentence in an interview is worth a lot.
Context is the one hole in memo's armor: a context-value change re-renders every consumer, punching straight through memo'd ancestors that don't consume it (they stay skipped; the consumer below them still renders). And because Provider value is compared with Object.is, an inline value={{user, theme}} re-notifies all consumers on every Provider render — the classic context perf bug.

Renders must be pure — and React enforces it

React reserves the right to call your component more than once per committed update (StrictMode dev double-invocation, §7) and to throw a computed tree away entirely (an interrupted concurrent render that got superseded by a higher-priority update — §4). If your render mutates a module-level variable, pushes to an array it didn't create this render, fires analytics, or writes to a ref, those side effects will run a nondeterministic number of times. The contract:

  • Same props + state + context → same JSX. No exceptions.
  • No mutation of anything that existed before this render began (mutating objects you created during this render is fine — "local mutation").
  • Side effects belong in event handlers (preferred) or effects (for synchronization with external systems) — both run only for committed updates.

"Is setState asynchronous?" — the senior answer

The precise answer: setState is synchronous as a function call, but the re-render it requests is batched and deferred — and the state variable in your closure never changes within a render. It's not Promise-async; there's no microtask you can await to get the new value. setCount(1) synchronously enqueues an update and schedules work; your local count still holds the old value because it's a const captured when this render ran. "State behaves like a snapshot per render" is the modern framing; "setState is async" is the 2016 framing that interviewers will probe. 

function handleClick() {
  setCount(count + 1);
  console.log(count);      // old value — not because "async", but because `count`
}                          // is a constant snapshot belonging to THIS render

One more consequence worth stating before we move on: because every state setter merely enqueues an update, several setters called in one event handler produce one render, not several — the queue is drained when React finally renders (§6 covers the queue mechanics). And the render that gets scheduled has to answer a question we've been dancing around: given the new element tree your components return, what should React actually do to the DOM? That's reconciliation.

3. Reconciliation, diffing heuristics, and keys T1

After a render produces a new element tree, React must transform the previous DOM into the new description with the minimum reasonable set of mutations. The theoretically optimal tree-diff is O(n³) — a thousand-element tree would need a billion comparisons. React gets to O(n) by refusing to solve the general problem and betting on two heuristics that hold for real UIs:

  1. Two elements of different type produce different trees. React never tries to morph a <div> subtree into a <span> subtree or a <UserPanel> into an <AdminPanel> — it tears down and rebuilds. And there is no cross-level matching, ever: if a node moves to a different depth in the tree, it's destroyed and recreated at the new location.
  2. The developer can hint stable identity across renders with key. Keys turn list diffing from positional matching into identity matching.

The type comparison — position by position

Reconciliation walks the old fiber tree and the new element tree together, comparing the element at each position with the fiber that occupied that position last render:

Comparison resultWhat React doesState?
Same type, host element (div → div) Keeps the DOM node, diffs the props, patches only changed attributes/styles, recurses into children DOM state (focus, scroll, input value) preserved
Same type, component (Panel → Panel) Keeps the fiber (the instance), updates props, re-renders it, recurses Hook/class state preserved
Different type (div → span, Panel → Modal) Unmounts the entire old subtree (cleanup effects run, DOM removed), mounts the new one from scratch All state in the subtree destroyed
Different key, same position Treated as a different element — unmount + remount, even if type is identical State destroyed — this is the reset tool 
// Same position + same type = SAME instance, state survives:
{isEditing
  ? <TextInput placeholder="Edit…" />
  : <TextInput placeholder="Search…" />}
// Toggling isEditing does NOT remount TextInput. React sees TextInput at the
// same position both renders → keeps the fiber, patches the placeholder prop.
// Whatever the user typed persists across the toggle — often a surprise.

// Different type at the position = teardown, even though a TextInput sits inside both:
{isEditing
  ? <div><TextInput /></div>
  : <section><TextInput /></section>}
// div → section: React destroys the whole subtree. The inner TextInput loses
// its state even though "the same component" appears in both branches.
"State is tied to position in the tree, not to the JSX you wrote" is the sentence to say. Two <Counter />s rendered at the same position across renders share one instance; the same <Counter /> rendered at two different positions is two instances. React has no idea your two JSX branches "mean" the same component — it only sees type + position (+ key).
Defining a component inside another component's body creates a new function identity every render — and a new function is a different type. React unmounts and remounts that child's entire subtree on every parent render: state wiped, effects re-run, DOM rebuilt, focus lost. This is the mechanical reason "never define components during render" is a hard rule, not a style preference. 
function Parent() {
  // ❌ New function identity each render → different type → remount every render
  function Child() { return <input />; }
  return <Child />;   // typing in this input loses focus on every keystroke
}

List diffing — what happens without keys

For a set of siblings, React's default is positional matching: old child 0 vs new child 0, old child 1 vs new child 1, and so on. Appending to the end is cheap — positions 0…n-1 match, one insert at the end. But prepending or removing from the middle is pathological: every element after the change point sits at a shifted position, so React sees "child at position i changed props/content" for the entire rest of the list and patches every row — and any per-row state stays glued to the position, not the data. 

// Old:  [A, B, C]        New: [Z, A, B, C]   (prepend Z, no keys)
// React compares by position:
//   pos 0: A → Z   "props changed" → patch entire row
//   pos 1: B → A   patch
//   pos 2: C → B   patch
//   pos 3: (new) → C   insert
// 3 full patches + 1 insert, instead of the truthful: 1 insert.

// With stable keys [z, a, b, c] vs [a, b, c]:
//   React maps old children by key, finds a/b/c unchanged (same key + type
//   → reuse fiber and DOM node, possibly move it), inserts only z.

Mechanically, keyed reconciliation does a first pass matching old and new children in order until keys stop lining up, then builds a Map of the remaining old children by key and, for each remaining new element, either reuses the matching old fiber (moving the DOM node if needed) or mounts a new one; leftovers in the map are deleted. One caveat worth knowing: React's algorithm only moves nodes forward — a "move the last item to the front" reorder can degrade to moving every other node instead, which is why large animated reorders are sometimes slower than expected. Still O(n), still correct.

Why index-as-key is a bug, concretely

Using the array index as the key re-encodes exactly the positional matching you were trying to escape — the key is the position. Two classes of bugs follow, and interviewers love making you produce them on a whiteboard:

Bug 1 — component state sticks to the position

// Each row has local state: an "expanded" flag.
{comments.map((c, i) => <Comment key={i} data={c} />)}

// User expands row 0 ("Alice"). A new comment "Zed" is prepended.
// key=0 now belongs to Zed's row. React sees key 0 in both renders →
// SAME fiber, same state:  Zed renders expanded, Alice renders collapsed.
// The expansion belongs to slot 0, not to Alice. Deleting a middle row
// shifts every later row's state up by one — same disease, other direction.

Bug 2 — uncontrolled inputs and DOM state

{rows.map((row, i) => (
  <li key={i}>
    {row.label}
    <input defaultValue={row.draft} />   // uncontrolled — value lives in the DOM
  </li>
))}
// Delete row 1 of 3. With positional keys React deletes the LAST li (key 2
// vanished) and patches the labels of keys 0/1. The <input> DOM nodes at
// positions 0 and 1 are reused untouched — so row 2's label now sits next to
// row 1's typed text. Same story for focus, checkboxes, scroll, transitions.
When is index-as-key actually fine? When the list is effectively static: never reordered, never filtered, items only appended or the whole list replaced wholesale, and rows carry no state — no inputs, no animations, no memoization relying on identity. Read-only rendering of a fixed array qualifies. The honest senior answer: "fine under those conditions, but the conditions are fragile — the day someone adds a delete button, the bug appears far from the code that caused it, so I default to real ids." And key={Math.random()} or anything regenerated per render is worse than an index: every render unmounts and remounts every row.

Key as an identity and reset tool

Because a changed key means "different element, remount," key doubles as React's official state-reset API — usable on any element, not just list items: 

// A form that should reset completely when the selected user changes:
<UserEditForm key={selectedUserId} userId={selectedUserId} />
// key changed → unmount + fresh mount: every useState initializer re-runs,
// uncontrolled inputs are cleared, effects torn down and re-created. One line
// replaces an error-prone "reset all fields in a useEffect when userId changes."

Compared to effect-based resets, key-reset is atomic — no intermediate render showing the previous user's data, no synchronization logic to get wrong. The trade-off: it's a full remount (effects re-run, DOM rebuilds), which is exactly right for "different record" and exactly wrong per-keystroke. Also remember the scoping rules: keys only need to be unique among siblings, not globally; and key is consumed by React — it never arrives in the child's props (pass a separate id prop if the child needs the value).

4. Fiber architecture — why and how T2

Fiber is React's ground-up reimplementation of the reconciler, shipped in React 16 (2017), and it's the substrate every modern feature stands on — concurrent rendering, transitions, Suspense, selective hydration. The interview question is rarely "recite the fields of a fiber"; it's "why did React need Fiber?" — a design question with a crisp answer.

Why the stack reconciler had to go

Pre-16 React reconciled recursively: rendering a component immediately and synchronously rendered its children, theirs, and so on — the work lived on the JavaScript call stack. That has a fatal property: you cannot pause a call stack. Once a big update started, the main thread was locked until the entire tree finished — a 200ms reconciliation meant 200ms of dropped frames, frozen inputs, and janked animations. No interrupting for a higher-priority event, no splitting work across frames, no abandoning work that went stale mid-flight.

The fix: reify the call stack as a data structure on the heap. Instead of stack frames, each unit of work becomes a fiber — a plain object React can process one at a time in a loop, pause between any two, resume, reprioritize, or throw away. A fiber is literally a virtual stack frame; the name comes from the OS concept of fibers as cooperatively-scheduled lightweight threads.

Anatomy of a fiber node

{
  // Identity — mirrors the element
  tag: 0,                    // FunctionComponent | HostComponent | HostRoot | …
  type: App,                 // the function/class, or 'div' for host fibers
  key: null,
  stateNode: null,           // the real DOM node (host fibers) / class instance

  // Tree structure — a linked list, NOT a children array
  child: fiber,              // first child
  sibling: fiber,            // next sibling
  return: fiber,             // parent — "where to return to": the stack metaphor

  // State
  pendingProps: {…},         // props from the new element, about to be applied
  memoizedProps: {…},        // props from the last completed render
  memoizedState: hook,       // function components: the LINKED LIST OF HOOKS
  updateQueue: […],          // pending state updates / effects for this fiber

  // Scheduling & bookkeeping
  lanes: 0b0100,             // priority bitmask of pending updates here
  flags: 0b0110,             // side effects to apply at commit: Placement, Update…
  alternate: fiber           // the OTHER version of this fiber — see double buffering
}

Two of these fields explain long-standing "rules" mechanically. memoizedState holds hook state as a linked list in call order — hook #1's node points to hook #2's, with no names attached. That is the entire reason hooks can't be called conditionally: React identifies "which useState is this?" purely by position in the list, so one skipped hook shifts every subsequent hook onto the wrong node's state. And child/sibling/return turn the tree into something traversable with a pointer-following loop — down to child, across to sibling, up via return — which is what makes the work pausable: all "where am I?" state lives in one pointer instead of a call stack.

Double buffering — current and workInProgress

React keeps two fiber trees. The current tree describes what's on screen right now. When an update arrives, React builds the next tree — the workInProgress tree — by walking the current tree and, for each fiber that needs work, creating or reusing its alternate. The two versions of each node point at each other via alternate, and untouched subtrees are shared structurally between the trees rather than copied. 

   current (on screen)          workInProgress (being built)
   ┌──────────┐    alternate    ┌──────────┐
   │ App  v1  │ ◄─────────────► │ App  v2  │
   └────┬─────┘                 └────┬─────┘
        │                            │
   ┌────▼─────┐    alternate    ┌────▼─────┐
   │ List v1  │ ◄─────────────► │ List v2  │
   └──────────┘                 └──────────┘

   Commit = an atomic pointer flip:  root.current = workInProgressTree.
   The old tree becomes the recycled alternate pool for the NEXT update.

This is classic double buffering, exactly as in graphics: never mutate the frame being displayed; compose the next frame off-screen and flip a pointer. It buys three things: the screen can never show a half-rendered state; an interrupted or abandoned workInProgress tree costs nothing to discard (the current tree was never touched); and memory churn stays low because each node's two versions are reused alternately forever.

The work loop and time slicing

// Conceptually (real source: ReactFiberWorkLoop.js)
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    workInProgress = performUnitOfWork(workInProgress);
  }
}
// performUnitOfWork: beginWork (call the component, reconcile its children,
// return the child fiber to work on next) … and when a subtree bottoms out,
// completeWork (prepare host-node updates, bubble effect flags upward), then
// move to the sibling, or return to the parent.

shouldYield() is where concurrency lives: React's scheduler slices render work into ~5ms chunks and yields the main thread between chunks so the browser can handle input and paint. The scheduling itself rides a MessageChannel macrotask loop — not requestIdleCallback (fires too infrequently, patchy support) and not a microtask (microtasks don't let paint through). If a higher-priority update arrives while a low-priority render is mid-flight, React abandons the workInProgress tree, renders the urgent update, then restarts the low-priority render from scratch against the new state. That restart is why render purity is a hard contract rather than a style rule: your components observably run for renders that never commit.

Lanes — priority as a bitmask

Every update is assigned a lane — one bit in a 31-bit mask — encoding its priority class. Conceptually, from most to least urgent:

Lane (conceptual)SourceBehavior
SyncDiscrete events: click, keydown, submitRendered synchronously to completion — a click's result must not be interleaved
InputContinuousContinuous events: scroll, drag, mousemoveHigh priority, coalesced
DefaultNetwork responses, timersNormal concurrent rendering
TransitionstartTransition, useDeferredValueInterruptible background rendering — abandoned and restarted when urgent work lands
Idle / OffscreenHidden or prerendered contentOnly when nothing else is pending

Bitmasks make the bookkeeping cheap: "does this subtree have pending work at this priority?" is one bitwise & against childLanes bubbled up the tree, and multiple compatible lanes batch into a single render. You don't need implementation-level lane detail in interviews — you need the model: updates carry priorities; urgent renders interrupt background ones; interrupted work is discarded and redone. That model is exactly what useTransition exposes as API.

Preempt the common misconception: Fiber did not make individual renders faster — the same components run the same code, and a sliced render actually finishes slightly later in wall-clock terms. It made rendering schedulable: pausable, prioritizable, abandonable. The payoff is responsiveness under load, not throughput — and every headline React feature since 2017 (Suspense, transitions, streaming SSR with selective hydration, <Activity>) is a scheduling feature that's only possible because the reconciler stopped living on the call stack.

5. Render phase vs commit phase — the full timeline T1

Every React update flows through the same two-phase pipeline, and knowing precisely what runs in which phase — and where the browser's paint fits — is the backbone of "walk me through setState to pixels": 

trigger (setState) → schedule (lane assigned, batched)
        │
        ▼
┌─ RENDER PHASE ────────────────────────────────────────────────┐
│  interruptible · pausable · restartable · MUST BE PURE        │
│  beginWork:   call components, reconcile children             │
│  completeWork: compute host-node diffs, bubble effect flags   │
│  output: a finished workInProgress tree + effect list         │
└───────────────────────────────────────────────────────────────┘
        │  (may be thrown away and restarted — nothing visible happened yet)
        ▼
┌─ COMMIT PHASE ── synchronous, never interruptible ────────────┐
│  1. before-mutation:  getSnapshotBeforeUpdate                 │
│  2. mutation:         DOM inserts/updates/deletes,            │
│                       detach refs, run layout-effect CLEANUPS │
│         → root.current = workInProgress  (the tree flip)      │
│  3. layout:           attach refs, useLayoutEffect setups,    │
│                       componentDidMount/DidUpdate             │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
   browser paints the mutated DOM
        │
        ▼
   passive effects flush: useEffect cleanups, then setups

Render phase: interruptible, therefore pure

Everything up to the finished workInProgress tree is computation without consequence — no DOM mutations, no lifecycle side effects. That's a design requirement, not a nicety: because concurrent React can pause this phase, run something more urgent, and restart it, any code that runs here may execute multiple times per committed update, or for updates that never commit at all. This is the phase your component function bodies, useMemo callbacks, useState initializers/updaters, and render-time reconciliation all belong to.

Commit phase: synchronous by necessity

Once React decides to commit, it applies every mutation for the update in one uninterruptible pass. Why can't commit be sliced too? Because a half-mutated DOM would be visible: the browser could paint between slices, showing users an inconsistent frame (old header, new list). The commit's three sub-phases each exist for a specific timing need:

Sub-phaseWhat runsDOM state at that moment
Before-mutation getSnapshotBeforeUpdate (class) Still the old DOM — last chance to read pre-update layout (the canonical use: capture scrollHeight of a chat list so you can preserve scroll position after prepending messages)
Mutation Insert/update/delete host nodes per effect flags; refs of removed/replaced nodes set to null; layout-effect cleanups run; unmounting components' componentWillUnmount Transitioning old → new; at the end, root.current flips to the new tree
Layout Refs attached (ref.current = node); useLayoutEffect setups; componentDidMount/DidUpdate; useImperativeHandle values wired New DOM, fully mutated — but not yet painted

useLayoutEffect vs useEffect — the paint boundary

The layout sub-phase runs after mutation, before the browser paints. That's the entire difference between the two effect hooks:

  • useLayoutEffect runs synchronously inside the commit. It can read the new layout (getBoundingClientRect) and even set state, forcing a synchronous re-render+re-commit before anything is painted — the user never sees the intermediate frame. This is the flicker-free way to position tooltips/popovers: render invisibly (or at 0,0), measure, reposition, all within one frame.
  • useEffect is deferred until after paint (scheduled as a passive effect). The user may see the first frame before the effect runs — which is fine for its actual job: synchronizing with external systems (subscriptions, fetches, logging), none of which need to beat the paint.
// Flicker with useEffect, no flicker with useLayoutEffect:
function Tooltip({ anchorRef, children }) {
  const [top, setTop] = useState(0);
  useLayoutEffect(() => {
    const rect = anchorRef.current.getBoundingClientRect();
    setTop(rect.bottom + 8);   // re-render + re-commit happen BEFORE paint
  }, [anchorRef]);
  return <div style={{ position: 'fixed', top }}>{children}</div>;
}
// With useEffect: frame 1 paints the tooltip at top:0, THEN it jumps into
// place — a visible one-frame flicker on slower devices.
Because useLayoutEffect blocks paint, heavy work inside it is a direct hit to interaction latency — every ms in a layout effect is a ms the user stares at a stale frame. Measure/position work only; move data work to useEffect. Two more timing facts worth volunteering: passive effects are "after paint" as a scheduling default, not an absolute guarantee (React will flush pending passive effects early — e.g. before processing the next discrete event or the next sync render — so sequencing logic must not depend on paint having happened); and on the server, useLayoutEffect triggers a warning because neither effect runs during SSR — there's no layout to measure.

Effect ordering — the full picture for one commit

// For an UPDATE that affects parent P and child C (effects run bottom-up):
//   mutation:  C layout-cleanup → P layout-cleanup → DOM mutated
//   layout:    C useLayoutEffect setup → P useLayoutEffect setup
//   (browser paints)
//   passive:   C useEffect cleanup → P useEffect cleanup →
//              C useEffect setup   → P useEffect setup
// Children before parents — a parent's effect can rely on its children's
// effects having run (same guarantee componentDidMount always gave).

6. Batching and the state-update queue T1

Batching is React collecting multiple state updates and processing them in one render + one commit. It exists because the alternative — render-per-setState — would make ordinary code pathologically slow and would surface half-updated, inconsistent frames (first name updated, last name not yet). The interview relevance is the React 18 change, plus the queue mechanics that explain the classic setCount(count + 1) puzzle.

React 17 vs 18 — the automatic batching change

In React 17 and earlier, batching only happened inside React event handlers — React wrapped your handler in a "batching context" and flushed once at the end. Updates fired from anywhere else — setTimeout, promise callbacks, await continuations, native event listeners — each triggered their own synchronous render. React 18's createRoot made batching automatic everywhere: 

function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
  // 17: one render (inside a React event handler) · 18: one render
}

setTimeout(() => {
  setCount(c => c + 1);
  setFlag(f => !f);
  // 17: TWO renders, one per setter · 18: ONE render — the change
}, 100);

async function save() {
  await api.save(data);
  setSaving(false);
  setToast('Saved');
  // 17: two renders · 18: one render. (Updates on opposite sides of an
  // `await` are separate ticks → separate batches — that part is unchanged.)
}
Mechanically there's no timer or debounce involved: each setter enqueues an update and schedules a render as a microtask-ish continuation; every setter called synchronously in the same tick lands in the same pending batch before React gets control back. Batching also composes with lanes — React groups pending updates of compatible priority into one render, so a sync click update won't wait for a transition to batch with it.

The update queue — why setCount(count + 1) × 3 gives 1

Each hook's fiber node keeps a queue of pending updates. A setter call appends an update object — either a replacement value or an updater function — and during the next render React reduces the queue in order to produce the new memoized state: 

const [count, setCount] = useState(0);   // this render: count === 0

function handleClick() {
  setCount(count + 1);   // enqueue REPLACE with 0+1 = 1  (count is the snapshot: 0)
  setCount(count + 1);   // enqueue REPLACE with 1 — same stale snapshot
  setCount(count + 1);   // enqueue REPLACE with 1
}
// queue: [replace 1, replace 1, replace 1] → next render: count === 1

function handleClickFn() {
  setCount(c => c + 1);  // enqueue UPDATER — receives the queue's running result
  setCount(c => c + 1);
  setCount(c => c + 1);
}
// queue: [0→1, 1→2, 2→3] → next render: count === 3

// They mix — the queue is reduced strictly in order:
setCount(5);           // → 5
setCount(c => c + 1);  // → 6
setCount(42);          // → 42 (replacement ignores the running result)

The rule of thumb that falls out: use the functional form whenever the next state depends on the previous state — it's immune to stale closures (interval callbacks, memoized handlers with [] deps) and immune to multiple-updates-per-batch surprises. Replacement form is fine when the new value is independent of the old.

flushSync — opting out, at a price

import { flushSync } from 'react-dom';

function addTodo(todo) {
  flushSync(() => {
    setTodos(prev => [...prev, todo]);
  });
  // React has rendered AND committed here — the DOM is updated.
  listRef.current.lastElementChild.scrollIntoView();  // safe to measure/scroll
}

flushSync forces React to render and commit the wrapped updates synchronously before returning. Legitimate uses are narrow and all of the shape "the very next line must see the committed DOM": scroll a just-added row into view, focus a just-mounted input, integrate with a non-React library that reads the DOM imperatively, or print (window.print() needs the DOM final now). It is a performance cliff by design — it defeats batching, forces work to complete at the highest priority, and can force-flush pending effects. Treat every usage as a code smell requiring a comment.

"Batched and deferred" also means you can never read the new state synchronously after a setter — not even with flushSync: the fiber's state updates, but your closure's count const is still the old snapshot. If code needs the value being set, compute it first (const next = count + 1), use it, then setCount(next) — or derive in an effect keyed on the state.

7. StrictMode — why everything runs twice in dev T2

<StrictMode> is a development-only correctness fuzzer: it deliberately double-invokes the code that React's programming model says must be safe to run more than once, so that violations crash and burn on your machine instead of intermittently in production. It renders no UI, adds zero behavior in production builds, and is the answer key to two of the most-Googled React mysteries: "why does my component render twice" and "why does my effect run twice on mount."

What exactly runs twice

Double-invoked (dev only)What it's checking

useState/useMemo initializers, updater functions, useReducer reducersThese run during render, so they must be pure too

// Mount sequence WITH StrictMode (dev): render → render →
//                                       effect setup → cleanup → setup

  const ws = new WebSocket(url);        // runs twice on mount in dev

The bugs it actually catches

  • Missing effect cleanup: the classic. An effect that subscribes/connects/sets an interval without a cleanup produces two live subscriptions after the StrictMode remount — duplicate WebSocket messages, double intervals, doubled analytics events. In production the same bug fires whenever the component genuinely remounts (route back-and-forward, key change) or the dependency changes; StrictMode just makes it happen on every mount so you can't miss it.
  • Impure renders: mutating props/state, pushing into a module-level array, incrementing a shared counter, Date.now()-based "unique" ids created during render — the double render makes the duplication observable (two array entries, ids that don't match between renders).
  • Fetch-in-effect without cancellation: the doubled request in dev is loudly annoying — which is the point. The fix (an AbortController or an ignore flag in cleanup) is exactly what protects the production race where a slow first response overwrites a fast second one.
Historical wrinkle worth knowing: effect double-invocation arrived in React 18 (renders were already double-invoked in 17-era StrictMode), and React 18 suppressed console logs from the second render while React 19 shows them, dimmed, in DevTools. So "my effect runs twice but I only see one log" has been a version-dependent experience — another reason candidates half-remember this area.
The "why" that separates senior answers: StrictMode's remount simulation isn't pedantry — it's the compatibility test for reusable state. Features like <Activity> (offscreen rendering), back/forward cache-style navigation, Fast Refresh, and any future "preserve the tree, detach the effects" work all need components whose effects can be torn down and re-established without breaking. StrictMode is React asking, on every mount, "would you survive that?" Production is unaffected — one render, one setup — but code that only works with exactly-once effects is code with a latent bug even today (remounts happen), and it's opting out of where React is going. If the doubled behavior "breaks" your app in dev, the app was broken; fix the effect, don't remove StrictMode.

8. The synthetic event system T2

React does not attach your onClick to the button. It implements its own event layer on top of the DOM's — for cross-browser normalization, for integration with batching and priorities (a click handler's updates get the Sync lane because React controls the dispatch), and for memory efficiency on large trees. Knowing where the listeners actually live explains a whole family of "why doesn't stopPropagation work" surprises.

Delegation — one listener per event type, at the root

React attaches one native listener per event type on your root container (the element you passed to createRoot) — not one per button. When a native event fires and bubbles up to the root, React's listener looks up the target's fiber, walks up the fiber tree collecting every matching onX/onXCapture prop along the path, and dispatches them in the right order (capture props top-down, then bubble props bottom-up) with a synthetic event object. 

<div id="root">            ← React 18/19: native listeners live HERE
  <App>
    <button onClick={…}>   ← NO native click listener on this node; onClick is
                             just a prop React reads during its root dispatch
Until React 16, delegation targeted document. React 17 moved it to the root container — partly to let multiple React versions coexist on one page (each root owns its events; a 17 tree nested in a 16 tree no longer fights over document-level handlers), and partly to make e.stopPropagation() in React behave sanely relative to non-React listeners outside the root: stopping propagation in a React handler now actually prevents the event reaching a listener on document. "Where does React attach event listeners, and what changed in 17?" is a stock trivia question with this exact answer.

The SyntheticEvent wrapper — and the pooling history

Handlers receive a SyntheticEvent: a cross-browser wrapper with a normalized interface (e.target, e.preventDefault(), consistent key/mouse fields) and the real event under e.nativeEvent. Through React 16, synthetic events were pooled: the object was reused across events and all fields nulled after the handler returned, so setTimeout(() => console.log(e.target)) logged null unless you called e.persist(). React 17 removed pooling entirely — modern engines made the allocation savings irrelevant and the confusion wasn't worth it. Events are now plain per-dispatch objects; e.persist() is a no-op kept for compatibility. If an interviewer asks about e.persist(), the correct answer is "that's a ≤16 concern; explain pooling, then say it's gone."

Where stopPropagation surprises people

// Surprise 1: a native listener between the target and the root fires FIRST.
containerEl.addEventListener('click', () => console.log('native container'));
// <div ref={containerEl}><button onClick={() => console.log('react')}/></div>
// Click logs: "native container" → "react".  React's onClick can't run until
// the native event finishes bubbling to the ROOT — every native bubble-phase
// listener below the root beats every React bubble handler, regardless of depth.

// Surprise 2: native stopPropagation SILENCES React.
containerEl.addEventListener('click', e => e.stopPropagation());
// The native event now never reaches the root → React never dispatches →
// every React onClick under that container is dead. Classic culprit: a
// third-party widget or an old-school "close dropdown on outside click"
// helper calling stopPropagation at document/body level.

// Surprise 3 (inverse): React's e.stopPropagation() DOES stop the native event
// too — React calls nativeEvent.stopPropagation() under the hood, so listeners
// above the root (window/document) won't hear it either.

The debugging rule that falls out: React handlers and native handlers live in two different propagation systems that only synchronize at the root. Mixing them (analytics snippets, jQuery-era widgets, portals into non-React DOM) is where events "mysteriously don't fire" — check for a native stopPropagation between the target and the root before anything else.

Passive listeners and other modern details

  • Passive by default for scroll-blocking events: React registers its root-level touchstart/touchmove/wheel listeners as passive: true (matching browser defaults for document-level listeners), which keeps scrolling off the main-thread critical path — but means e.preventDefault() in those React handlers cannot cancel the scroll. Need to actually prevent it (custom pinch-zoom, drawing surfaces)? Attach a native non-passive listener via a ref + effect.
  • Not everything is delegated: events that don't bubble or have special semantics — scroll (since 17, no longer bubbles through React trees), focus/blur (implemented via focusin/focusout internally), media events, load/error — are attached directly to the element or handled specially. The mental model "one root listener per type" has documented exceptions; know they exist.
  • Priorities enter here: the event type determines the update lane — discrete events (click, keydown) produce sync-priority updates; continuous ones (mousemove, scroll, drag) produce input-continuous priority. The event system is the front door of the scheduler.

9. children & composition as a performance mechanism T1

children is not magic syntax — it's an ordinary prop that happens to be filled from JSX nesting, and its value is whatever you put there: an element, an array of elements, a string, a function (render props), or anything else. Two properties make it architecturally powerful: it's opaque to the receiving component (a layout shell renders {children} without knowing or caring what's inside), and it's created by the parent's parent — which is a re-render firewall hiding in plain sight.

The mechanism: elements are created by whoever writes the JSX

function App() {                       // App creates BOTH elements:
  return (
    <Sidebar>                          // { type: Sidebar, props: { children:
      <HeavyNav />                     //     { type: HeavyNav, props: {} } } }
    </Sidebar>
  );
}

function Sidebar({ children }) {
  const [open, setOpen] = useState(true);   // state churn lives here
  return <aside className={open ? 'open' : ''}>{children}</aside>;
}

When setOpen fires, Sidebar re-renders — but it does not re-create the HeavyNav element; App made that object, and App didn't render. Sidebar just returns the same children reference it received. During reconciliation React hits the bailout from §1: the element at this position is referentially identical to last render's element, so its props cannot have changed (elements are immutable), so the entire subtree is skipped — fiber reused, not re-rendered, not diffed. Zero API, zero dependency arrays, and it composes: everything App passes through Sidebar is firewalled from Sidebar's state churn.

The firewall has exactly one requirement: the element must be created outside the re-rendering component. If Sidebar built <HeavyNav /> in its own body, that's a fresh element object per render — no bailout. Same for children re-created via React.Children.map or cloneElement on the way through: cloning mints new element objects and forfeits the identity bailout. (Also note the flip side: when App re-renders, it re-creates the children element, so the bailout disappears for that render — this pattern firewalls against the wrapper's churn, not the owner's.)

Slot patterns — the same trick, multiplied

Nothing is special about the name children; any prop can carry elements. "Slots" — multiple named element props — give layout components their flexibility and keep the same referential-identity benefits: 

function PageShell({ header, sidebar, footer, children }) {
  const [collapsed, setCollapsed] = useState(false);   // shell-local churn
  return (
    <div className={collapsed ? 'rail' : 'full'}>
      <header>{header}</header>
      <nav>{sidebar}</nav>
      <main>{children}</main>
      <footer>{footer}</footer>
    </div>
  );
}

<PageShell header={<TopBar />} sidebar={<NavTree />} footer={<Legal />}>
  <Dashboard />
</PageShell>
// Collapsing the shell re-renders PageShell only — all four slot subtrees
// keep their element identity and bail out.
This is the answer to "how would you reduce re-renders without memo?" — restructure so hot state lives in a leaf-ish wrapper that receives expensive content as children/slots ("lift content up"), or move the state down into its own component ("push state down"). Structural fixes beat memoization: no comparator to get wrong, no dependency array to maintain, and they usually improve the design anyway. memo is for the cases composition can't reach. (Full performance treatment: tutorial 09.)

10. Refs and the DOM — attachment timing and layout reads T2

A ref is a stable box ({ current }) that survives renders without participating in them — mutating it triggers nothing. For DOM refs the interview substance is timing: exactly when does ref.current hold the node, and where can you safely read layout from it?

The attachment timeline

render phase      │ ref.current is NOT set (null on first render; on updates it
                  │ still points at the OLD node) — never read refs during render
commit·mutation   │ refs of deleted/replaced nodes are DETACHED (set to null /
                  │ callback invoked with null / React 19 cleanup fn runs)
commit·layout     │ refs ATTACHED to new nodes — ref.current = domNode —
                  │ BEFORE useLayoutEffect setups run
                  ▼
useLayoutEffect   │ ✅ node available, pre-paint — measure here to avoid flicker
(browser paint)   │
useEffect         │ ✅ node available, post-paint — fine for non-visual setup

The ordering guarantee — refs attach in the layout sub-phase before layout effects fire — is why useLayoutEffect(() => ref.current.getBoundingClientRect(), []) just works: React commits DOM mutations, wires refs bottom-up, then runs your layout effects. It's also why "why is my ref null?" almost always has one of three answers: you read it during render; the element is conditionally not rendered this pass; or it's on a custom component that doesn't forward it to a host element.

Callback refs — and React 19's cleanup

// A callback ref runs at attach (node) and detach (null) — pre-19:
<video ref={(node) => {
  if (node) observer.observe(node);
  else observer.disconnect();          // the awkward null branch
}} />

// React 19: return a cleanup function instead — effect-like symmetry:
<video ref={(node) => {
  observer.observe(node);
  return () => observer.disconnect(); // runs on detach; null is no longer passed
}} />

Callback refs are the tool when you need to react to the node appearing/disappearing (attach observers, measure conditionally-rendered nodes) rather than merely hold it. Two operational details: a callback ref runs during commit, so setting state inside one is legal (it triggers a synchronous follow-up render, like a layout effect would); and an inline callback ref has a new function identity each render — pre-19 React would call it with null then the node again on every render, so hoist it with useCallback if the attach/detach work is non-trivial. React 19 rounds this area out with ref as a regular prop — function components accept ref directly, and forwardRef is on the deprecation path.

Measuring the DOM safely

  • Synchronous, before the user sees it: useLayoutEffect + read (getBoundingClientRect, scrollHeight) + optionally set state to adjust — commits again before paint, so no flicker. This is the tooltip/popover/auto-size-textarea pattern.
  • Reactive to size changes: don't poll — a ResizeObserver wired in an effect (or a callback ref) with cleanup. Same for visibility: IntersectionObserver.
  • Batch reads before writes: interleaving reads and style writes forces synchronous layout each time (layout thrashing). Read everything, then write.
Don't read or write ref.current during render. Reading is unreliable (stale or null, per the timeline); writing is an impurity — StrictMode's double render will do it twice, and a concurrent render that never commits will have mutated your ref anyway. The blessed places to touch refs: event handlers, effects, layout effects, and callback refs. The one tolerated render-time pattern is lazy initialization (if (ref.current === null) ref.current = makeThing()) — idempotent by construction.

11. ReactDOM specifics — roots, portals, the package split T3

Why two packages — react vs react-dom

react is the platform-agnostic core: elements/JSX runtime, hooks, component model, Suspense, context. react-dom is one renderer — the code that knows how to turn the fiber tree's effects into DOM mutations and how to speak browser events. The shared reconciler lives behind the renderer interface (react-reconciler), which is how the same core drives React Native, react-three-fiber (Three.js scenes), Ink (terminal UIs), and PDF/canvas renderers: they implement "create instance / append child / commit update" for their host environment, and everything from hooks to reconciliation comes for free. That's the architecture answer to "why can't hooks live in react-dom" and "how does React Native share code with React."

createRoot vs the legacy API

// Legacy (React ≤17 — REMOVED in React 19):
ReactDOM.render(<App />, document.getElementById('root'));

// Modern (React 18+):
import { createRoot } from 'react-dom/client';
const root = createRoot(document.getElementById('root'));
root.render(<App />);
// later: root.render(<App tab="settings" />)  — update in place
//        root.unmount()                        — tear down

The API change is the switch for concurrent React: apps still on ReactDOM.render in React 18 ran in legacy mode — no automatic batching outside handlers, no transitions, no concurrent features. React 19 removed the legacy entry points entirely. createRoot also changed hydration's spelling: 

// SSR hydration — attach to existing server-rendered markup:
import { hydrateRoot } from 'react-dom/client';
hydrateRoot(document.getElementById('root'), <App />);
// Renders to compare-and-adopt instead of create: React walks the existing DOM,
// attaches fibers + listeners to it, and errors on markup mismatches (React 19
// prints an actual diff). Mismatch classics: Date.now()/locale text, browser-only
// branches like `typeof window !== 'undefined'` changing the tree shape.

Portals — DOM elsewhere, React tree unchanged

import { createPortal } from 'react-dom';

function Modal({ onClose, children }) {
  return createPortal(
    <div className="overlay" onClick={onClose}>{children}</div>,
    document.body            // physically render here…
  );
}
// …but LOGICALLY the modal stays a child of whoever rendered <Modal>:
// context flows in normally, and events BUBBLE THROUGH THE REACT TREE —
// a click inside the modal fires onClick handlers on the modal's React
// ANCESTORS, even though in the DOM the overlay is a sibling of #root.

Portals solve the CSS containment problem — modals, tooltips, dropdowns escaping overflow: hidden, stacking contexts, and transformed ancestors — without breaking the component model: state, context, and error boundaries all behave as if the portal content were inline. The event-bubbling-through-the-React-tree behavior is the classic gotcha in both directions: a "click outside to close" handler on a React ancestor will hear clicks inside the portal (they bubble through the React tree), while a native document-level listener sees the true DOM path instead. Also worth a sentence: portals don't manage focus for you — trap and restore focus yourself, or better, build modals on <dialog>/a11y-audited primitives.

Odds and ends that live in react-dom and show up in screens: flushSync (§6); react-dom/server's streaming SSR (renderToPipeableStream); React 19's resource preloading APIs (preload, preinit) and built-in <title>/<meta> hoisting; and useFormStatus/form actions, which are react-dom hooks because forms are a DOM concept. If asked "what's in react vs react-dom," the sorting rule is: platform-agnostic model in react, anything that knows what a DOM node is in react-dom.

12. Interview Q&A

1. Walk me through everything that happens from a setState call to pixels on screen.

(1) The setter enqueues an update object on the hook's queue on that fiber and schedules a render with a priority lane derived from the context (click handler → sync lane; transition → transition lane). (2) React batches: every other setter called in the same tick joins the same batch. (3) Render phase: starting from the root (skipping clean subtrees via bailouts), React builds a workInProgress fiber tree — calling component functions, reducing state-update queues to new memoizedState, and reconciling each component's returned elements against the old fibers (same type = reuse, different type/key = teardown, keyed list diff). This phase is pure and, for non-sync lanes, interruptible — it can be abandoned and restarted. (4) Commit phase, synchronous: before-mutation (getSnapshotBeforeUpdate), mutation (apply DOM inserts/updates/deletes, detach refs, run layout-effect cleanups, flip root.current), layout (attach refs, run useLayoutEffect, componentDidMount/Update). (5) The browser then paints the mutated DOM. (6) Passive effects (useEffect) flush after paint. Bonus points for noting that steps 3–4 may not happen at all if every update bails out via Object.is.

2. Why did React need Fiber? What could the old reconciler not do?

The stack reconciler rendered recursively — the work lived on the JS call stack, and a call stack can't be paused. Any large update locked the main thread until the whole tree finished: dropped frames, frozen inputs, no way to prioritize a keystroke over a data refresh, no way to abandon work made stale by a newer update. Fiber reifies each unit of work as a heap object (a fiber — a "virtual stack frame") linked via child/sibling/return pointers, processed by an iterative work loop that can yield every ~5ms, resume, restart, or discard work. That's what makes updates prioritizable (lanes), renders interruptible, and features like transitions, Suspense, and selective hydration possible. Key nuance: Fiber didn't make renders faster — it made them schedulable; the win is responsiveness, not throughput.

3. Is setState asynchronous?

Precise answer: the call is synchronous, but the state application is deferred — the setter enqueues an update and schedules a batched render; it doesn't return a promise and there's nothing to await. The reason you "can't read the new value" isn't asynchrony at all: your count is a const captured when this render ran — a per-render snapshot — and no amount of waiting changes a closed-over constant; the next render's closure gets the new value. Follow-ups to expect: multiple setters in one handler → one render (batching, and since React 18 this holds in timeouts/promises too); need next-state-from-previous-state → functional updater; need the committed DOM immediately → flushSync, which forces the commit but still never changes your current closure's variable.

4. How does React "know" which components to re-render?

It doesn't watch anything — there's no dirty-checking of props or observables on state. A state update marks that fiber (and bubbles pending-lane bits up its ancestors so the root knows where work exists). At render time React walks down from the root: subtrees with no pending work and unchanged props/context are skipped via bailouts; the component that owns the update re-renders; and then, by default, everything it returns re-renders too — rendering a child is how React discovers what changed, not the result of detecting a prop change. The cascade is stopped only by memo's shallow compare, the same-element-reference bailout (children/hoisted elements), or shouldComponentUpdate. Context is the extra channel: a provider's value change (Object.is) re-renders all consumers regardless of memo in between.

5. "A component re-renders when its props change." Critique that statement.

It's backwards in both directions. First, props changing isn't a trigger: props only "change" because the parent re-rendered and created a new element — the parent's render is the trigger, and the child re-renders even if its props are identical (or if it takes no props at all). Second, changed props don't guarantee a render is caused by them — a memo'd child with changed props renders, but a non-memo'd child renders regardless. The accurate statement: a component renders when (a) its own state updates, (b) its parent renders and no bailout applies, or (c) a context it consumes changes value. Props are just data flowing along for the ride; React never diffs props to decide whether to render (only memo does, as an opt-in).

6. How does React's diffing get away with O(n) when tree diff is O(n³)? What are the heuristics?

By not solving the general problem. Heuristic 1: elements of different types produce different trees — so React compares position-by-position, and on a type mismatch it tears down the whole old subtree and rebuilds, never attempting cross-type morphing or cross-level moves. Heuristic 2: keys let developers declare identity within a sibling list, turning positional matching into map-lookup matching. Both heuristics are nearly always true for real UIs, so the rare pathological case (a subtree that "moved" across levels gets recreated) is an acceptable price for a single linear pass. Consequences worth naming: same type + position = same instance (state persists across conditional branches that render the same component), different type = state annihilation for the whole subtree, and key changes force remounts — which is also a feature (key-based reset).

7. Why do lists need keys? What does React do without them?

Without keys, sibling diffing is positional: old[0] vs new[0], old[1] vs new[1]. Appends are fine; prepends/removals/reorders shift every subsequent position, so React patches nearly every row (sees them all as "changed") and — worse — per-row state (hook state, uncontrolled input values, focus, animations) stays attached to the position, showing up next to the wrong data. Keys give React an identity map: same key + same type = same fiber and DOM node, reused and moved as needed; new keys mount; missing keys unmount. Result: minimal DOM operations and state that travels with the data. Keys must be stable (survive re-renders), unique among siblings only, and are consumed by React — they don't appear in props.

8. Index as key — when is it actually fine, and what exactly breaks when it isn't?

Fine when the list is static in identity terms: no reorder, no filter, no insert/delete except appends (or full replacement), and rows are stateless — no inputs, no row-local hooks, no animations, no memo depending on identity. Then index and identity coincide and nothing can desynchronize. It breaks the moment positions shift: index keys re-encode positional matching, so hook state and uncontrolled-input DOM values stick to their slot — delete row 1 and row 2's data renders next to row 1's typed text; prepend an item and the expanded/checked state belongs to the wrong record. Two extra points for seniority: key={Math.random()} is strictly worse (full remount of every row every render), and when data truly has no id, synthesize one at data-creation time (crypto.randomUUID when the item enters state), never at render time.

9. You need to reset a form when the selected record changes. Compare key-based reset vs an effect.

<Form key={recordId}/> makes React treat each record's form as a different instance: unmount, remount, every useState initializer re-runs, uncontrolled inputs cleared — atomic, declarative, zero synchronization code, and no frame where the previous record's draft is visible. The effect approach ("watch recordId, setState everything back") renders once with stale state, then re-renders after the effect — a flash of wrong data — and grows a maintenance liability: every new field needs remembering in the reset. Trade-off to name: key-reset is a full remount (effects re-run, DOM rebuilt, focus lost), perfect for "different entity," wasteful for high-frequency changes; and if you need to preserve some state across the switch, that state must be lifted above the keyed boundary.

10. Output drill: what does this log in development with StrictMode, and why?

Setup: a component whose body has console.log('render') and a useEffect(() => { console.log('setup'); return () => console.log('cleanup'); }, []), mounted once. Dev + StrictMode logs: render, render, setup, cleanup, setup — StrictMode double-invokes the function body (purity check) and runs the mount effect's setup → cleanup → setup cycle (remount-survivability check). Production logs: render, setup. Version wrinkle: React 18 suppressed the second render's console output (so people saw one "render"); React 19 shows it, dimmed, in DevTools. The deeper point to volunteer: if the doubled effect breaks the app (double subscription, doubled fetch), the effect is missing a cleanup or cancellation — the same bug that fires in production on any genuine remount or dep change.

11. What's in a fiber node, and what is double buffering?

A fiber holds identity (type, key, stateNode → the DOM node or class instance), structure as a linked list (child, sibling, return — the "virtual stack frame" pointers that make iterative, pausable traversal possible), state (pendingProps/memoizedProps, memoizedState — which for function components is the linked list of hooks in call order, the mechanical reason hook order must be stable), the updateQueue, and scheduling data (lanes bitmask, effect flags, alternate). Double buffering: React maintains two trees — current (on screen) and workInProgress (being built), each fiber paired with its alternate. Rendering mutates only the WIP tree; commit is an atomic root.current pointer flip. Benefits: no half-rendered UI is ever visible, abandoned concurrent renders cost nothing to discard, and node pairs are recycled so allocation churn stays low.

12. Render phase vs commit phase — what runs in each, and where do getSnapshotBeforeUpdate, useLayoutEffect, and useEffect fall relative to paint?

Render phase: component functions, state-queue reduction, reconciliation — pure, no DOM contact, interruptible/restartable in concurrent lanes. Commit phase, synchronous, three sub-phases: before-mutation runs getSnapshotBeforeUpdate against the old DOM (capture scroll positions); mutation applies DOM changes, detaches refs of removed nodes, and runs layout-effect cleanups, ending with the current-tree pointer flip; layout attaches refs then runs useLayoutEffect setups and componentDidMount/Update against the new but unpainted DOM. Then the browser paints. Then useEffect (passive) cleanups + setups flush. So: snapshot = pre-mutation, layout effects = post-mutation pre-paint (can measure and synchronously re-render without flicker, but block paint), passive effects = post-paint by default (with the caveat that React may flush them early before the next sync work).

13. What changed about batching in React 18? Predict the render counts.

React ≤17 batched only inside React event handlers; updates from setTimeout, promise/await continuations, and native listeners each flushed a separate synchronous render. React 18 with createRoot batches automatically everywhere. Drill: a handler calls setA then setB → one render in both versions. The same two setters inside setTimeout → two renders on 17, one on 18. Inside an async function, setA before an await and setB after it → two renders even in 18 (opposite sides of an await are different ticks, hence different batches), but two setters both after the same await → one render. Escape hatch: flushSync forces a synchronous render+commit per wrapped call when you must read the committed DOM between updates. Also note batching respects priority — updates in different lanes don't merge into one render.

14. Output drill: three setCount(count + 1) calls in one handler vs three functional updates — final values and why.

With count === 0: three setCount(count + 1) calls enqueue three replacement updates, each computed from the same stale closure snapshot — the queue is [replace 1, replace 1, replace 1], so the next render shows 1. Three setCount(c => c + 1) calls enqueue updater functions; React reduces the queue in order, feeding each updater the running result: 0→1→2→3, next render shows 3. Mixed forms reduce strictly in order — setCount(5); setCount(c => c + 1); setCount(42) ends at 42. The rule: functional form whenever next state depends on previous state; it's also the fix for stale closures in intervals and useCallback([], …) handlers, and it usually lets you drop the state from dependency arrays.

15. When is flushSync justified, and what does it cost?

Justified when the very next synchronous line must observe the committed DOM: scrolling a just-added list item into view, focusing a just-mounted input, window.print() after a state change, or handing a final DOM to an imperative third-party library. It forces React to render and commit the wrapped updates before returning — breaking batching, running at max priority, potentially force-flushing pending effects, and turning what could have been interruptible work into a main-thread block. So: rare, commented, and never in a loop or render path (React warns if called during rendering). Also worth saying: flushSync updates the committed DOM and fiber state, but your current closure's state variables are still the old snapshot — it is not a way to "read state synchronously."

16. Why does StrictMode double-invoke renders and effects, and what bug classes does it surface?

It stress-tests the two contracts React's model depends on: renders are pure (safe to run twice — and concurrent React genuinely does re-run renders it discards), and effects are symmetric (setup/cleanup pairs that survive an unmount/remount — which real apps hit on navigation, key changes, Fast Refresh, and future features like <Activity> offscreen trees). Bug classes surfaced: missing effect cleanup (double WebSocket/interval/subscription after the simulated remount), impure renders (module-level mutation, render-time id generation, array pushes — duplicated visibly), and uncancelled fetches in effects (doubled requests reveal the response race you already had). Dev-only; production gets single invocation. The senior kicker: deleting StrictMode to "fix" doubled effects just re-hides a real bug — the correct fix is a cleanup/AbortController.

17. Where does React attach event listeners, and what changed in React 17?

React uses delegation: one native listener per event type attached to the root container (the createRoot element) — not per component. Your onClick props are just data on fibers; when a native event bubbles to the root, React walks the target's fiber ancestry, collecting onXCapture handlers (dispatched top-down) and onX handlers (bottom-up), and invokes them with a SyntheticEvent. React 17 moved delegation from document to the root container — enabling multiple React versions on one page (each root owns its events) and making stopPropagation interop with non-React code saner. 17 also removed event pooling (e.persist() became a no-op), switched onFocus/onBlur to focusin/focusout internally, and stopped onScroll bubbling. Exceptions to delegation exist: non-bubbling/special events (scroll, media, load/error) attach directly to elements.

18. A React onClick mysteriously never fires, but only inside one widget. Likely cause?

Prime suspect: a native listener between that widget and the React root calling e.stopPropagation(). React's handlers run only when the native event reaches the root listener; a native stopPropagation below the root starves React of the event entirely — classic offenders are third-party embeds, jQuery-era plugins, or hand-rolled "close on outside click" code using addEventListener. Diagnosis: log a native capture-phase listener on document to confirm the event fires, then bisect down the DOM. Related traps for the same conversation: native listeners on inner nodes always fire before React bubble handlers (React waits for the root); React's e.stopPropagation also halts the native event above the root; and React's root touch/wheel listeners are passive, so preventDefault can't cancel scrolling from React handlers — attach a native non-passive listener when you genuinely need that.

19. Why does passing an expensive subtree as children skip its re-renders? What's the mechanism?

Element referential identity. JSX creates element objects, and they're created by whoever writes the JSX — so in <Wrapper><Heavy/></Wrapper>, the parent of Wrapper creates the Heavy element and passes it in as the children prop. When Wrapper's own state changes and it re-renders, it returns the same children object it received — Wrapper's render didn't re-execute the JSX that created Heavy. Reconciliation sees new element === old element and bails out: elements are immutable, so an identical reference proves the props can't have changed, and the entire subtree is skipped without rendering or diffing. Requirements: the element must be created outside the re-rendering component, and not cloned en route (cloneElement mints a new object). It's memo-grade optimization from pure structure — no comparator, no dependency arrays — and it generalizes to any element-valued "slot" prop.

20. When exactly is ref.current set, and what do callback refs add — including React 19's change?

Timeline: during render, refs are not usable (null on mount; stale on updates). In the commit's mutation sub-phase, refs on removed/replaced nodes are detached; in the layout sub-phase, refs attach — before useLayoutEffect runs. So layout effects and everything after (paint, passive effects, event handlers) can rely on the node. Callback refs run at those same attach/detach moments and are the tool for reacting to a node's appearance — attaching ResizeObserver/IntersectionObserver, measuring conditionally-rendered nodes. Pre-19 the callback was invoked with the node on attach and null on detach (and an inline callback's changing identity caused a null+node re-invocation every render). React 19 lets the callback return a cleanup function — effect-style symmetry, no null branch — and also makes ref an ordinary prop on function components, deprecating forwardRef. And never read/write refs during render: lazy init is the one idempotent exception.

21. A modal rendered through a portal into document.body — where do its click events go?

Through the React tree, not the DOM tree. createPortal changes where the DOM nodes are placed, but the portal's fiber remains a child of the component that rendered it — so synthetic events bubble to the modal's React ancestors: an onClick on the component wrapping <Modal> fires for clicks inside the modal, even though in the DOM the modal is a sibling of the app root. Context, error boundaries, and Suspense behave the same way — logical tree wins. The flip side: native document-level listeners see the true DOM path, so a hand-rolled "close when clicking outside" using contains(e.target) on the trigger's DOM node will treat in-modal clicks as "outside" — the classic portal bug. Why portals exist at all: escaping overflow:hidden, stacking contexts, and transformed ancestors for modals/tooltips/dropdowns without breaking the component model.

22. What's the difference between <Comp /> and calling {Comp()} directly in JSX?

<Comp /> creates an element — { type: Comp, props } — and React later gives Comp its own fiber: its own state slots, its own position in the tree, its own bailout boundary. {Comp()} executes immediately during the parent's render and inlines Comp's output; React never knows Comp existed. Consequences: any hooks inside Comp attach to the parent's hook list — so conditionally calling it (or changing how many times it's called) corrupts hook order and throws the "rendered fewer hooks than expected" error; it can't be memoized, keyed, or shown in DevTools as a component; its "state" merges into the parent's, resetting in surprising ways. The same trap wears a disguise in render-prop and hook-returning-JSX patterns. Rule: components are called by React, always through elements.

23. Can a render happen and never reach the screen? What follows from that?

Yes, two ways: bailouts (Object.is-equal state updates may run one reconfirming render that commits nothing) and, more importantly, concurrent rendering — a transition/deferred render in progress is abandoned wholesale when a higher-priority update arrives, then restarted against fresh state; StrictMode's dev double-render simulates exactly this. What follows: render-phase code must be side-effect free, because it can run 0, 1, or N times per visible update — analytics fired in render double-counts, module-level mutation corrupts, render-generated random ids mismatch. It also creates the tearing problem: an interruptible render reading a mutable external store can see one value early in the pass and another late (an urgent update mutated the store mid-render), rendering an inconsistent tree — which is precisely what useSyncExternalStore solves by snapshotting the store per render and forcing consistency checks. Effects and event handlers, by contrast, run only for committed updates.

24. Why can't hooks be called conditionally? Answer from the data structure.

Hook state lives on the component's fiber as a linked list in call order — memoizedState points at hook node #1, which points at #2, and so on. Hooks carry no names or keys; the only thing binding this useState call to that state node is "it was the third hook called." On every render React walks the list in parallel with your calls: call #1 reads node #1, call #2 reads node #2. Skip a hook behind a condition and every subsequent call reads its neighbor's node — a useState suddenly returns an effect's deps array, or your count becomes your username. React detects list-length mismatches and throws, but same-length misalignment would be silent corruption, hence the lint rule enforcing it statically. Postscript: React 19's use() is the deliberate exception — readable conditionally because it doesn't allocate a slot in the hook list.

Exercise 3 — Build a render-logging harness

Implement useRenderLog(name, props): logs every render of a component, numbers them, and prints which props changed by reference since the last render (a mini why-did-you-render). It must not violate render purity. Where must the logging happen, and why?

Solution

The diffing state lives in refs (survives renders, triggers nothing). The log itself is a side effect, so it belongs in an effect — logging in the render body would double-fire misleadingly under StrictMode and would also fire for renders that never commit under concurrent rendering. Running the effect with no dependency array makes it fire after every committed render — exactly the definition of what we're counting.

function useRenderLog(name, props) {
  const count = useRef(0);
  const prev = useRef(props);

  useEffect(() => {              // no deps → runs after EVERY commit
    count.current += 1;
    const changed = {};
    const keys = new Set([...Object.keys(prev.current), ...Object.keys(props)]);
    for (const k of keys) {
      if (!Object.is(prev.current[k], props[k])) {
        changed[k] = { from: prev.current[k], to: props[k] };
      }
    }
    console.log(
      `[${name}] render #${count.current}`,
      Object.keys(changed).length ? changed : '(no prop changes — parent render or state/context)'
    );
    prev.current = props;
  });
}

// Usage: function Row(props) { useRenderLog('Row', props); … }

Reading the output is the real lesson: "no prop changes" on a storm of renders means the parent is cascading (fix structurally or with memo); the same prop flagged every render with deep-equal values means an inline object/function/array is being re-created — the memo-defeat signature. Bonus discussion: this counts committed renders; to observe render-phase executions (StrictMode's double-invoke), you'd log in the body — accepting that it's an impurity you'd never ship, or use the React DevTools Profiler which reports "why did this render" without any harness.

function Clock({ children }) {

  }, []);

      <ExpensiveTree />   // element created by App — once per APP render

Why it bails: each tick re-runs only Clock's body. Clock returns the same children element object it received — App didn't render, so that object is the very reference from last time. Reconciliation compares new child element vs old at that position, finds ===, and since elements are immutable, identical reference proves unchanged props — React reuses the fiber and skips the whole subtree: no render, no diff. Verify in the Profiler: per-tick commits show only Clock (<1ms); ExpensiveTree stays gray.
