03 — Modern CSS & Architecture

Container queries, :has(), layers, typed custom properties, compositor-friendly animation, and how to argue styling architecture like a senior.

Why interviewers ask this

CSS questions at the senior level are rarely about memorizing property names. They are a currency check and an architecture check:

  • Currency: CSS has changed more between 2021 and 2025 than in the previous decade — container queries, :has(), cascade layers, native nesting, @property, oklch(), view transitions. Interviewers probe whether you kept up or whether your knowledge froze at "flexbox and BEM."
  • Architecture: "How would you style a large React app?" is a classic senior screen. The wrong answer is naming one tool with religious conviction. The right answer names the axes (runtime cost, scoping, SSR/RSC compatibility, design-system maturity, team scale), argues trade-offs, and then commits to a default with reasons.
  • Performance: Animation and rendering questions ("why is transform cheaper than top?") test whether you understand the browser pipeline, which transfers directly to debugging jank in production.
Tier legend: Tier 1 = asked constantly, must be fluent. Tier 2 = frequent at senior level, differentiates candidates. Tier 3 = occasional, awareness is enough.

Container queries Tier 2

The problem media queries cannot solve

Media queries respond to the viewport. But in a component-driven world, the same component renders in wildly different contexts: a <ProductCard> might live in a full-width hero, a 3-column grid, or a 280px sidebar — all on the same viewport width. A media query cannot distinguish those. For years, design systems worked around this with prop-driven variants (<Card layout="compact" />) or JS-based ResizeObserver hacks, both of which push a layout concern into component APIs.

Container queries let a component respond to the size of its own container. The card decides for itself: "if I have less than 400px of inline space, stack vertically."

Establishing a container: container-type

/* 1. Mark an ancestor as a query container */
.card-slot {
  container-type: inline-size;   /* query the inline axis (width in LTR/RTL) */
  container-name: card;          /* optional, lets descendants target it by name */
  /* shorthand: container: card / inline-size; */
}

/* 2. Descendants query the nearest (or named) container */
@container card (width < 400px) {
  .card        { flex-direction: column; }
  .card__image { aspect-ratio: 16 / 9; width: 100%; }
  .card__title { font-size: 1rem; }
}

@container card (width >= 400px) {
  .card        { flex-direction: row; gap: 1.5rem; }
  .card__image { width: 40%; }
}

size vs inline-size — and why containment is required

This is the part interviewers use to separate people who have shipped container queries from people who read a blog post. To query a container's size, the browser must be able to compute that size without looking at the container's contents — otherwise you get circularity (contents change → container resizes → query flips → contents change → …). So container-type applies containment:

ValueWhat you can queryContainment appliedPractical implication
inline-size Inline axis only (width, typically) Layout, style, and inline-size containment Element still grows vertically with content. This is what you want 95% of the time.
size Both axes (width and height) Layout, style, and full size containment Element no longer sizes from its content in either axis — you must give it an explicit height or it collapses. Rarely what you want.
normal Style queries only (@container style(--x: y)) None for size Default; the element is not a size query container.
A common bug: developers slap container-type: inline-size directly on a grid item that was being sized by its content, and layout subtly changes because inline-size containment makes the element's intrinsic inline size behave as if it had no contents (auto-width shrink-to-fit breaks). Rule of thumb: put containment on a dedicated wrapper ("slot") element whose size is determined by the surrounding layout, not by its own children.

Container query length units

Inside (and even outside) @container rules, you can size things relative to the queried container:

.card__title {
  /* 1cqi = 1% of the container's inline size */
  font-size: clamp(1rem, 4cqi, 1.75rem);
}
  • cqw / cqh — 1% of container width / height
  • cqi / cqb — 1% of container inline / block size (logical; prefer these)
  • cqmin / cqmax — smaller / larger of cqi and cqb

These enable container-relative fluid typography: a card headline that scales with the card, not the viewport — huge for design systems.

Style queries (awareness)

Beyond size, @container style(--theme: dark) queries a container's custom property value. Style queries on custom properties shipped in Chromium and have been rolling out elsewhere; treat them as "know it exists" rather than production-default as of 2025/2026.

Browser support & migration strategy

  • Support: size container queries and cq units are supported in all evergreen browsers since early 2023 (Chrome/Edge 105+, Safari 16+, Firefox 110+). By 2025/2026 they are safely production-default for evergreen-browser audiences.
  • Migration strategy: don't rewrite; migrate component by component. Start with design-system leaf components that already have "variant" props purely for sizing (card, media object, stat tile). Keep media queries for page-level layout (how many columns the grid has); use container queries for component-level adaptation (how the card lays out inside its cell). That division of labor is the senior framing: media queries for macro layout, container queries for micro layout.
  • Fallback: @supports (container-type: inline-size) if you must support older browsers; the unqueried styles are your mobile-first baseline.
Interview sound bite: "Container queries fix the coupling between a component's responsive behavior and its page context. Before them, our design system leaked layout context into component APIs as variant props; after them, the card owns its own breakpoints. Media queries didn't go away — they moved up to page-level composition."

:has() — the "parent selector" Tier 2

:has() was the most-requested CSS feature for two decades: select an element based on what it contains (or what follows it). It's a functional pseudo-class taking a relative selector list. Supported in all evergreen browsers since Firefox 121 (Dec 2023); production-safe by 2025.

Canonical real-world uses

/* 1. Form field wrapper reacts to its input's validity */
.field:has(:invalid)          { border-color: var(--color-danger); }
.field:has(:focus-visible)    { outline: 2px solid var(--color-focus); }
.field:has(:user-invalid) .field__error { display: block; }

/* 2. Card adapts depending on whether it contains an image */
.card:has(> img)       { grid-template-rows: 200px 1fr; }
.card:not(:has(> img)) { padding-block-start: 2rem; }

/* 3. Quantity queries: style a list differently once it has 5+ items */
.tab-list:has(> :nth-child(5)) {
  font-size: 0.875rem;
  gap: 0.25rem;
}

/* 4. Previous-sibling styling (impossible before :has) */
label:has(+ input:focus) { color: var(--color-accent); }

/* 5. Page-level state without JS: dim the page when a dialog is open */
html:has(dialog[open]) { overflow: hidden; }

Pattern 1 is worth internalizing: combined with :user-invalid (only matches after user interaction), you get accessible inline validation styling with zero JavaScript state.

Performance considerations

  • Browsers match selectors right-to-left; :has() forces the engine to look down and sideways from candidate elements, which is inherently more work. Engines heavily optimize it (invalidation sets, bloom filters), so typical usage is fine.
  • The expensive pattern is broad subject + deep, unanchored argument: *:has(.foo .bar) or body:has(.deeply .nested :hover) — every DOM mutation anywhere can force re-checking. Keep the subject specific (.field:has(...)) and the inner selector shallow (child combinator > where possible).
  • :has() cannot be nested inside another :has(), and pseudo-elements are not valid inside it.

Composing :has() with :not()

Order matters semantically:

/* "cards that do NOT contain an image" */
.card:not(:has(img)) { ... }

/* "cards that contain something that is not an image"
   — almost always true; almost never what you meant */
.card:has(:not(img)) { ... }
:has(:not(img)) vs :not(:has(img)) is a favorite interviewer gotcha. The first matches a card containing any non-image element (a title counts). The second matches a card containing no image at all. Say it back in English before writing it.
Specificity: like :is(), :has() takes the specificity of the most specific selector in its argument list — it is not zero-cost like :where().

Cascade layers — @layer Tier 2

The problem: specificity wars

In a large codebase, the cascade is fought with escalation: a reset gets overridden by the framework, which gets overridden by component styles, which get overridden by page overrides — each layer of intent enforced by accidental specificity (.btn.btn.btn, #app .card, !important). The intent was always "this category of styles should beat that category," but pre-layers CSS had no way to express priority between categories; only per-selector specificity.

@layer makes that priority explicit: layer order beats specificity. A selector in a later layer wins even if a selector in an earlier layer is more specific.

/* Declare order up front — first line of your entry stylesheet.
   Later layers win. */
@layer reset, tokens, base, components, utilities, overrides;

@layer reset {
  * { margin: 0; box-sizing: border-box; }
}

@layer components {
  .btn { background: var(--color-primary); }
}

@layer utilities {
  /* wins over .some #very .specific .btn selector in `components`,
     despite lower specificity — layer order decides first */
  .bg-transparent { background: transparent; }
}

/* Layers also work with imports: */
@import url("vendor/datepicker.css") layer(vendor);

The rules that get asked

  • Unlayered styles beat all layered styles (for normal declarations). This is the crucial migration property: you can wrap a third-party library in a layer and know your unlayered app styles always win, regardless of the library's selector specificity.
  • !important inverts layer order. Important declarations in earlier layers beat important declarations in later layers (and important layered beats important unlayered). Mnemonic: !important reverses everything — so a reset layer's !important is the strongest thing on the page. This is a deliberate design: it lets low-level layers make guarantees that app code cannot casually stomp.
  • Specificity still applies within a layer — layers settle the fight between categories, then specificity settles it inside one.
  • Layers can nest: @layer components.card { ... }.

revert-layer

@layer overrides {
  .card--plain {
    /* roll this property back to whatever previous layers said,
       as if this layer never declared it */
    box-shadow: revert-layer;
  }
}

Complete the keyword family for interviews: initial (spec default) → inherit (parent's computed value) → unset (inherit if inherited property, else initial) → revert (roll back to user-agent/user styles) → revert-layer (roll back one cascade layer).

How design systems use layers

The emerging convention: the design system ships everything inside layers (@layer ds.reset, ds.tokens, ds.components), consumers import it into a layer, and consumer styles — layered later or unlayered — always win without specificity hacks. Open props, Bootstrap 5.3+, and many internal systems adopted this. It also finally gives a principled home for utility classes: put them in the last layer instead of marking every utility !important (Tailwind v4 organizes its output with @layer theme, base, components, utilities).

Support: all evergreen browsers since March 2022 — one of the fastest-adopted features ever. Fully production-safe.

Senior framing: "Layers turn the cascade from an accident into an architecture. Specificity was always a proxy for 'how authoritative is this style' — layers state that authority directly. The migration killer feature is that unlayered beats layered, so you can layer vendor CSS today with zero risk to existing styles."

Custom properties, deeply Tier 1

Every senior candidate "knows" custom properties. Interviews probe the edges: inheritance semantics, failure modes, typing, and theming architecture.

Inheritance and the * misconception

Custom properties are ordinary inherited properties. Declared on :root, they're available everywhere via inheritance — not because :root is magic, but because inheritance flows down the tree. The classic misconception is defining them with *:

/* BAD: sets the property directly on EVERY element.
   Direct values beat inheritance, so overriding --gap on a parent
   no longer cascades to children — every element has its own copy. */
* { --gap: 1rem; }

/* GOOD: define at root, override at subtree scope */
:root      { --gap: 1rem; }
.compact   { --gap: 0.5rem; }   /* everything inside .compact inherits 0.5rem */

This "override at subtree scope" behavior is the superpower: custom properties are the only CSS mechanism where a parent can parameterize arbitrary descendants — effectively props for CSS.

Fallbacks

.btn {
  color: var(--btn-color, var(--color-text, #222));
  /* fallback only applies when the property is NOT DEFINED (guaranteed-invalid),
     not when it's defined with a junk value */
}

Invalid at computed-value time — why var misuse fails silently and weirdly

This is the deepest interview question in this section. Normal CSS has parse-time error recovery: color: 16px is dropped at parse time, and the previous cascaded value wins. But var() substitution happens at computed-value time, after parsing and after the cascade already picked a winner. If substitution produces garbage:

:root { --size: 16px; }
p {
  color: red;                 /* cascaded, then discarded below */
  color: var(--size);        /* 16px is not a color... */
}
/* Result: color is NOT red. The declaration is "invalid at
   computed-value time": too late to fall back to the previous
   declaration, so the property becomes `unset` —
   → inherited value for inherited properties (color inherits: parent's color)
   → initial value otherwise.
   The red declaration is gone; the cascade already threw it away. */
Symptom in the wild: "my button text turned black/inherited and my earlier fallback declaration didn't apply." Because var() failure resolves to inherit/initial rather than the previous declaration, defensive "old value first, var() second" progressive enhancement does not work for custom properties the way it does for e.g. width: 90%; width: clamp(...). Use the var(--x, fallback) second argument or @property instead.

Custom properties in media queries — no; @property — yes to typing

You cannot use var() in a media query condition (@media (min-width: var(--bp)) is invalid — media queries are evaluated outside any element, and custom properties only have values on elements). The spec-track answer for reusable environment values is Media Query Variables / custom environment variables, still emerging; today you preprocess breakpoints (Sass/PostCSS) or use container queries where they fit.

@property registers a custom property with a type, initial value, and inheritance behavior — which fixes two things: invalid values now fall back to the typed initial value at computed-value time, and the property becomes animatable because the engine knows how to interpolate it:

@property --gradient-angle {
  syntax: "<angle>";
  inherits: false;
  initial-value: 0deg;
}

.fancy-border {
  background: conic-gradient(from var(--gradient-angle), #f0f, #0ff, #f0f);
  transition: --gradient-angle 0.6s ease;   /* impossible for untyped props:
     untyped custom properties interpolate discretely (they just flip) */
}
.fancy-border:hover { --gradient-angle: 180deg; }

Support: Chromium since 2020, Safari 16.4+, Firefox 128+ (mid-2024) — Baseline as of 2024, safe for evergreen audiences.

Theming architecture: semantic tokens over raw values

/* Layer 1 — primitives (never referenced by components) */
:root {
  --blue-600: oklch(0.55 0.18 255);
  --gray-900: oklch(0.25 0.02 255);
}

/* Layer 2 — semantic tokens (what components consume) */
:root {
  --color-bg: white;
  --color-text: var(--gray-900);
  --color-accent: var(--blue-600);
}

/* Dark mode = re-mapping the SEMANTIC layer only */
[data-theme="dark"] {
  --color-bg: var(--gray-900);
  --color-text: oklch(0.95 0.01 255);
}

.card { background: var(--color-bg); color: var(--color-text); }

Because tokens live in the cascade, dark mode requires zero component changes and re-themes subtrees (theme a single modal by putting data-theme on it). This is the single strongest argument for custom-property-based theming over compile-time theming.

Custom properties + React

// Custom props are the clean bridge for dynamic values in React:
// one static CSS rule, per-instance value via inline style.
function ProgressBar({ percent }) {
  return (
    <div
      className="progress"
      style={{ '--progress': `${percent}%` }}
    />
  );
}

/* CSS — stays cacheable and static: */
/* .progress::after { width: var(--progress); } */

Compare with generating a new class per value (CSS-in-JS runtime cost) or setting width inline directly (loses the ability for CSS to mediate — transitions, clamping, pseudo-elements). This pattern is the standard answer to "how do you pass dynamic values into CSS Modules / Tailwind codebases."

Modern sizing & fluid design Tier 2

clamp() fluid typography — and the zoom-safety formula

/* clamp(MIN, PREFERRED, MAX) */
h1 {
  font-size: clamp(1.75rem, 1rem + 2.5vw, 3rem);
}

Why 1rem + 2.5vw and not just 4vw? Accessibility. A purely viewport-based preferred value doesn't respond to the user's font-size preference or browser zoom text scaling — WCAG 1.4.4 requires text to be resizable up to 200%. Mixing a rem component in guarantees the value still scales when the root font size changes. Deriving the numbers: pick font size at two viewport widths (e.g. 28px @ 400px, 48px @ 1200px), the slope is (48−28)/(1200−400) = 0.025 → 2.5vw, and the rem intercept is 28 − 0.025×400 = 18px ≈ 1.125rem. Tools like Utopia automate this "fluid type scale."

min() / max()

.container {
  width: min(90%, 72rem);        /* replaces width:90%; max-width:72rem */
  padding-inline: max(1rem, env(safe-area-inset-left));
}

Mnemonic that interviewers appreciate: min() acts as an upper bound (it picks the smaller), max() acts as a lower bound. clamp(a, x, b) ≡ max(a, min(x, b)).

aspect-ratio

.video-embed { aspect-ratio: 16 / 9; width: 100%; }
.avatar      { aspect-ratio: 1; inline-size: 3rem; }

Kills the old padding-top: 56.25% hack. Note it sets a preferred ratio: if both dimensions are constrained, or content overflows, the ratio yields. Also key for CLS: giving images width/height attributes lets the browser derive an aspect ratio and reserve space before load.

Logical properties — the i18n/RTL benefit

/* Physical (breaks in RTL) */       /* Logical (writing-mode aware) */
margin-left: auto;                     margin-inline-start: auto;
padding: 0 1rem;                      padding-inline: 1rem;
width / height;                        inline-size / block-size;
top / left (insets);                   inset-block-start / inset-inline-start;
border-left;                           border-inline-start;
text-align: left;                      text-align: start;

With logical properties, flipping the entire UI for Arabic/Hebrew is dir="rtl" on <html> — no [dir=rtl] override stylesheets, no RTLCSS build step. Any company shipping internationally treats this as table stakes; using margin-left in a new design system in 2026 is a review comment.

text-wrap: balance / pretty

h1, h2, .card__title { text-wrap: balance; }  /* equalizes line lengths; engines cap
                                                  it around ~6 lines — headings only */
p                     { text-wrap: pretty; }   /* better breaks; avoids single-word
                                                  orphan last lines */

Pure progressive enhancement — unsupported browsers ignore it. balance is Baseline (2023/24); pretty shipped in Chromium and Safari with Firefox following. Note pretty trades a little layout performance for quality; applying it to everything on a huge page is measurable, headings and body copy are fine.

Modern selectors & features — rapid tour Tier 2

:is() vs :where() — the specificity difference

/* Both match the same elements: */
:is(h1, h2, h3) a      { ... }  /* specificity of the MOST specific argument
                                   — here (0,0,1) from hN + (0,0,1) = (0,0,2) */
:where(h1, h2, h3) a   { ... }  /* :where contributes ZERO specificity
                                   → total (0,0,1) — trivially overridable */
  • :is() = grouping sugar; also "forgiving" — one invalid selector in the list doesn't kill the rule (unlike a plain selector list, where one bad selector invalidates the whole rule).
  • :where() = the tool for resets and library defaults: styles that should exist but lose to literally anything the consumer writes. Modern resets (e.g. Andy Bell's) wrap everything in :where() for this reason.

Native CSS nesting

.card {
  padding: 1rem;

  &:hover { box-shadow: var(--shadow-2); }

  .card__title { font-weight: 600; }   /* implicit & — works since the
                                            relaxed syntax (2023/24) */

  @media (width >= 48rem) {              /* at-rules nest too */
    padding: 2rem;
  }
}

Gotchas vs Sass nesting:

  • No string concatenation: &__title does NOT build .card__title. & is a live selector reference, not text. BEM authored via nesting doesn't port.
  • Nested selectors desugar to :is(.card) .card__title-style matching, so the parent part carries :is() specificity semantics (most-specific-argument), which can differ subtly from Sass output when the outer selector is a list.
  • Same maintainability caution as Sass: deep nesting couples CSS to DOM structure and inflates specificity. Two levels is a good review guideline.
  • Baseline since late 2023 (relaxed syntax through 2024); production-safe evergreen. Sass now passes native nesting through.

Color: oklch() and color-mix()

:root {
  /* oklch(Lightness Chroma Hue) — perceptually uniform */
  --accent: oklch(0.65 0.2 250);
  --accent-hover: oklch(from var(--accent) calc(l - 0.1) c h); /* relative color syntax */
  --accent-soft: color-mix(in oklch, var(--accent) 15%, white);
}

Why perceptual uniformity matters for design systems: in HSL, hsl(60 100% 50%) (yellow) and hsl(240 100% 50%) (blue) have the same "lightness" number but wildly different perceived brightness — so a palette generated by holding HSL lightness constant has inconsistent contrast across hues, and contrast ratios break. In OKLCH, equal L is (approximately) equal perceived lightness across all hues, so you can generate a 50–950 scale per hue by varying only L and get consistent contrast behavior — which is why Tailwind v4, Radix Colors, and most new token systems moved to OKLCH. It also unlocks wide-gamut P3 colors that hex/sRGB can't express. color-mix() replaces Sass darken()/lighten()/rgba() tricks at runtime — meaning it works with custom properties, which Sass functions never could.

:user-valid / :user-invalid

:invalid matches immediately on page load (an empty required field is invalid before the user touches it) — the classic red-form-on-arrival bug. :user-invalid/:user-valid only match after significant user interaction (typically after change + blur). Baseline 2023. Combine with :has() on the field wrapper for JS-free validation UI.

@supports feature queries

@supports (container-type: inline-size) { ... }
@supports selector(:has(a))              { ... }
@supports not (animation-timeline: scroll()) { /* fallback */ }

Senior nuance: prefer structuring CSS so fallback is the unqueried default and the enhancement sits inside @supports — never gate core layout on a positive query only old browsers fail.

Animations & rendering performance Tier 1

The rendering pipeline — WHY transform/opacity are cheap

Every frame the browser may run some or all of: Style → Layout → Paint → Composite.

  1. Style: recompute which rules apply and computed values.
  2. Layout (reflow): compute geometry — position and size of every affected box. Expensive, and it cascades: resizing one element can relayout ancestors, siblings, and the whole document.
  3. Paint: rasterize pixels (text, colors, shadows, borders) into layer bitmaps.
  4. Composite: the compositor thread assembles already-painted layers on the GPU — position, scale, rotate, fade them — without the main thread.

The property you animate determines where you enter this pipeline:

Animated propertyPipeline stages per frameCost
width, height, top/left, margin, font-sizeLayout → Paint → CompositeWorst — full pipeline every frame, on the main thread
background-color, color, box-shadow, border-radiusPaint → CompositeMedium — skips layout, still repaints
transform, opacity (and filter, mostly)Composite onlyCheapest — the element gets its own compositor layer; the GPU just re-blends existing bitmaps. Runs even while the main thread is blocked by JS.

That last cell is the key sentence for interviews: compositor-only animations stay smooth even when the main thread is janked, because the compositor is a separate thread.

/* Layout thrash — relayouts the document 60×/second on the main thread: */
.drawer { left: -300px; transition: left 0.3s; }
.drawer.open { left: 0; }

/* Compositor version — same visual, ~free: */
.drawer { transform: translateX(-100%); transition: transform 0.3s; }
.drawer.open { transform: translateX(0); }

Related JS-side thrash: interleaving reads (offsetHeight, getBoundingClientRect) with writes forces synchronous layout ("layout thrashing"). Batch reads, then writes — or use requestAnimationFrame phases.

will-change — sparingly

.drawer:hover, .drawer:focus-within { will-change: transform; }
/* hint just-in-time, remove after — NOT: * { will-change: transform } */

It pre-promotes the element to its own compositor layer so the first animation frame doesn't stall on layer creation. Cost: every layer is a GPU-memory bitmap (roughly width × height × 4 bytes × device-pixel-ratio²); blanket use exhausts GPU memory, especially on mobile, and can cause jank. Treat it like a profiler-driven fix, not a default. Modern engines auto-promote animating elements anyway, so its value is mostly avoiding the promotion hitch at animation start.

FLIP — First, Last, Invert, Play

The technique for animating layout changes (list reorder, grid→detail expand, shared-element moves) using only cheap transforms:

  1. First: measure the element's position (getBoundingClientRect) before the change.
  2. Last: apply the state change (reorder the array, add the class), let layout land, measure again.
  3. Invert: apply a transform that visually moves it back to the old spot: translate(firstX − lastX, firstY − lastY). The user still sees "before."
  4. Play: transition the transform to none. The element glides to its true position — one layout, then compositor-only animation.
// React list-reorder sketch (the classic interview extension):
const positions = new Map();

useLayoutEffect(() => {          // runs after DOM mutation, BEFORE paint
  for (const [id, node] of nodeRefs) {
    const last = node.getBoundingClientRect();
    const first = positions.get(id);
    if (first) {
      const dx = first.left - last.left, dy = first.top - last.top;
      if (dx || dy) {
        node.animate(                              // Web Animations API
          [{ transform: `translate(${dx}px, ${dy}px)` }, { transform: 'none' }],
          { duration: 250, easing: 'ease-out' }
        );
      }
    }
    positions.set(id, last);
  }
}, [items]);

useLayoutEffect matters: it runs before the browser paints the new layout, so the invert is applied without a flash. Libraries: Framer Motion's layout prop is FLIP under the hood; the View Transitions API is effectively browser-native FLIP with snapshots.

prefers-reduced-motion — a requirement, not a nicety

Vestibular disorders make large parallax/zoom/spin motion physically nauseating; this is an accessibility requirement (WCAG 2.3.3) and a code-review blocker at mature orgs.

/* Strategy: motion is the enhancement, opt-in */
@media (prefers-reduced-motion: no-preference) {
  .card { transition: transform 0.3s; }
  .hero { animation: float 6s ease-in-out infinite; }
}

/* Reduced ≠ none: keep small opacity/color feedback,
   kill translation/scale/parallax/autoplay. */

Also respect it in JS: matchMedia('(prefers-reduced-motion: reduce)') before triggering Framer Motion/WAAPI sequences and smooth scrolling.

Web Animations API (WAAPI) — awareness

element.animate(keyframes, options) gives CSS-quality (often compositor-driven) animations with a JS handle: .pause(), .reverse(), .finished promise, dynamic keyframe values (as in FLIP above). It's the primitive under Framer Motion's hood for many paths, and the right tool when keyframe values are only known at runtime.

animation-timeline — awareness

@supports (animation-timeline: scroll()) {
  .reading-progress {
    animation: grow linear;
    animation-timeline: scroll(root);   /* progress bound to page scroll,
                                            off the main thread */
  }
}
@keyframes grow { from { transform: scaleX(0); } to { transform: scaleX(1); } }

Styling architecture at scale Tier 1

This is the senior CSS discussion. The interviewer isn't grading your favorite tool; they're grading whether you can name the axes, argue trade-offs fairly, and commit to a contextual decision. Give every option a fair treatment.

Option 1 — Plain CSS / Sass + BEM

/* Block__Element--Modifier: scoping by naming convention */
.search-form { }
.search-form__input { }
.search-form__button--disabled { }
  • Pros: zero runtime, zero build magic, fully cacheable static CSS, works everywhere (any framework, no framework), the platform never breaks it. Flat specificity (single class) keeps the cascade tame.
  • Cons: scoping is a social contract, not a guarantee — nothing stops a collision or a "quick fix" targeting someone else's block. Naming discipline decays across teams and years; dead CSS accumulates because deletion is scary (who knows what still uses .card--legacy?). Dynamic styling requires class toggling gymnastics.
  • Modern posture: pairs well with cascade layers and :where()-based resets; still an excellent choice for content sites, design-system CSS distribution, and small teams with discipline.

Option 2 — CSS Modules

/* Card.module.css */
.card  { padding: 1rem; }
.title { composes: heading from './typography.module.css'; }

// Card.tsx
import styles from './Card.module.css';
// styles.card → "Card_card__x7f3a" — hashed, collision-proof
<div className={styles.card}>
  • Pros: build-time scoping — collisions are impossible by construction, not convention. Zero runtime cost; output is plain static CSS. Deletion is safe (unused import = dead file, and typed CSS module plugins even flag it). composes gives real reuse. Was the React community default for most of a decade, and works cleanly with Server Components because there's nothing to execute.
  • Cons: dynamic values need the custom-property bridge (style={{'--x': v}}) or class switching; global theming needs a parallel token system; naming within a file is still on you; co-location is two files per component.
  • Verdict axis: the "boring, safe default" — the option you can commit to when nothing else in the context forces a different answer.

Option 3 — Tailwind (utility-first)

<button class="inline-flex items-center gap-2 rounded-lg bg-blue-600
  px-4 py-2 text-sm font-medium text-white hover:bg-blue-700
  focus-visible:ring-2 disabled:opacity-50">
  Save
</button>
  • Pros: consistency via constraints — you compose from a finite design-token scale (spacing 4/8/12…, a fixed palette), so a 40-person team can't invent 40 shades of gray or 17 padding values. Total colocation: style lives with markup, no naming, no file-hopping, fearless deletion (delete the JSX, the styles are gone). JIT compilation means output CSS is only what you use — and crucially it plateaus: utilities are shared, so CSS size grows sub-linearly with app size, unlike per-component approaches. Zero runtime; v4 is CSS-first config with native cascade-layer output and OKLCH palette.
  • Critiques and the standard answers:
    • "Unreadable class soup." Answer: readability is local — everything affecting the element is right there, versus mentally joining 4 CSS files; editor tooling (sorting, hover previews) closes most of the gap. It's a real trade, though — templates are denser.
    • "It's just inline styles." Answer: no — utilities support hover/focus/media/container states, come from a constrained token scale, and are cacheable in a stylesheet; inline styles are none of those.
    • "Duplication everywhere." Answer: in a component framework, the component IS the reuse boundary — you repeat the classes once inside <Button> and render it 500 times. Tailwind without a component layer genuinely does duplicate; that's the honest caveat.
    • @apply: use with real caution — recreating semantic classes via @apply everywhere reintroduces exactly the naming/indirection Tailwind exists to remove, plus a proprietary dialect. Tailwind's own docs discourage broad use; acceptable for a handful of primitives when you can't componentize (CMS content, third-party markup).
  • Where it struggles: heavy dynamic styling (arbitrary runtime values need the CSS-variable bridge or [--x:...] arbitrary properties); design systems needing themable, consumer-overridable CSS output; teams whose designers don't work on the token scale.

Option 4 — Runtime CSS-in-JS (styled-components, Emotion)

const Button = styled.button`
  padding: 0.5rem 1rem;
  background: ${p => p.$variant === 'danger' ? 'crimson' : 'royalblue'};
`;
  • Pros (why it won 2017–2021): true colocation, real JS scoping and refactoring, effortlessly dynamic (styles are functions of props/theme), dead-style elimination for free, theming via context.
  • Runtime cost: on every render, the library interpolates template strings, hashes them, generates class names, and injects <style> rules via CSSOM — main-thread work proportional to component count, on top of shipping the library (~10–20 kB) and your styles as JS (parsed and executed, not just streamed like CSS). SSR requires per-request style extraction and doubles the payload (critical CSS + the same styles again in the JS bundle).
  • The RSC incompatibility — why the ecosystem moved away: runtime CSS-in-JS fundamentally depends on client-side JS execution and, for theming, on React Context. Server Components don't re-render on the client, can't use context, and shouldn't ship style-generating JS. styled-components/Emotion therefore only work inside "use client" boundaries — meaning adopting them forces your styling choice to dictate your component architecture, backwards. Combined with the measured runtime cost (the widely-cited Spot/Emotion case study showed double-digit % render-time overhead), the ecosystem consensus shifted around 2022–2023 to zero-runtime approaches. styled-components was put into maintenance mode in 2025, which interviewers may expect you to know.
  • Fair treatment: for a client-rendered SPA with heavy prop-driven dynamic styling and no RSC plans, it still works and existing large codebases are not "wrong" — the argument is about defaults for new projects and migration pressure, not retroactive shame.

Option 5 — Zero-runtime CSS-in-JS

The "keep the DX, drop the runtime" generation — styles authored in TS, extracted to static CSS at build time:

  • vanilla-extract: .css.ts files, fully type-safe tokens/themes (createTheme), variants via recipes; output is plain CSS, RSC-compatible. Think "CSS Modules with a typed API."
  • Linaria: styled-components-like syntax compiled away; dynamic values become CSS custom properties automatically — the clearest migration path off styled-components.
  • StyleX (Meta): atomic compile-time styles — every declaration becomes a deduplicated atomic class (Tailwind-like output, typed-JS authoring), with deterministic "last-applied-wins" merging across component boundaries, which solves the specificity-of-overrides problem at Meta scale. Awareness-level unless you're interviewing at Meta.
  • Panda CSS also fits here (Chakra's successor direction). Common thread: constraint — styles must be statically analyzable, so arbitrary runtime interpolation is out; dynamic values go through CSS variables.

How to answer "how would you style a large app?" like a senior

Structure the answer in four beats:

  1. Clarify context first (or state assumptions): rendering model (RSC? SPA?), team size, design-system maturity, perf budget, existing codebase. "It depends" alone is junior; "it depends on these four axes" is senior.
  2. Name the axes: scoping guarantee, runtime cost, dynamic-styling story, SSR/RSC compatibility, token/theming integration, deletability, hiring/onboarding familiarity.
  3. Commit to a default: e.g. "For a new Next.js product app with a design token scale and a team of 10+, my default is Tailwind for product surfaces plus CSS custom properties for theming, with a thin set of shared components as the reuse boundary. If the team has strong CSS ownership and dislikes utility HTML, CSS Modules with a token layer is the equally sound alternative — the non-negotiables are zero runtime and build-time scoping, not the syntax."
  4. Name what you'd avoid and why: "I'd avoid runtime CSS-in-JS for anything new because of RSC incompatibility and main-thread cost — while acknowledging existing styled-components codebases migrate via Linaria/vanilla-extract, not rewrites."
Anti-patterns in this answer: religious advocacy ("Tailwind is objectively best"), tool-bashing without numbers, pretending one tool fits libraries and apps equally, or never actually committing. Interviewers deliberately push back on your choice — the test is whether you defend with trade-offs or fold instantly.

Design tokens & theming Tier 2

Token tiers: primitive → semantic → component

:root {
  /* Tier 1 — primitives: raw values, exhaustive, never used directly */
  --blue-500: oklch(0.62 0.19 255);
  --space-4: 1rem;
  --radius-2: 0.5rem;

  /* Tier 2 — semantic: meaning, not appearance. THE theming surface. */
  --color-action: var(--blue-500);
  --color-surface: white;
  --color-text-muted: var(--gray-500);

  /* Tier 3 — component: only when a component needs an override point */
  --button-bg: var(--color-action);
  --button-radius: var(--radius-2);
}

Why tiers matter: components consuming semantic tokens means (a) dark mode remaps one tier, (b) a rebrand swaps primitives without touching components, (c) component tokens give consumers a supported override API (.my-btn { --button-bg: ... }) instead of specificity fights. The W3C Design Tokens Community Group format standardizes the JSON source of truth; tools (Style Dictionary, Terrazzo) compile it to CSS custom props, TS constants, and native platform formats — one source, N targets.

Dark mode strategies

StrategyMechanismTrade-offs

Avoiding dark-mode FOUC (flash of wrong theme)

If theme is applied in a useEffect, SSR/static HTML paints light, then flashes dark. Fix: a tiny render-blocking inline script in <head>, before any content paints:

<script>
  // Inline in <head> — runs before first paint, so no flash.
  try {
    const stored = localStorage.getItem('theme');
    const system = matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
    document.documentElement.dataset.theme =
      (stored === 'light' || stored === 'dark') ? stored : system;
  } catch (e) {}
</script>

Also set color-scheme: light dark on :root so form controls, scrollbars, and the UA canvas match the theme — and know the newer light-dark(lightVal, darkVal) CSS function (Baseline 2024) that picks a value based on the resolved color-scheme, letting one declaration serve both themes.

Density & spacing scales

Same token mechanics generalize beyond color: a spacing scale (--space-1…--space-9, usually a 4px base), type scale, radius scale. Density theming (comfortable vs compact tables for data-heavy screens) is just remapping the spacing tier under [data-density="compact"] — a pattern Material and enterprise design systems formalize. If components consume raw pixel values anywhere, density theming is impossible; that's the review argument for "no magic numbers, tokens only."

Interview Q&A

What problem do container queries solve that media queries can't?

Media queries respond to viewport size; components render in many container contexts on the same viewport (sidebar vs main column vs modal). Container queries let a component adapt to its own container's size, decoupling component responsive behavior from page layout. Before them, design systems leaked this into props (variant="compact") or JS ResizeObserver hacks. Division of labor: media queries for page-level macro layout, container queries for component-level micro layout.

Why does container-type: size require you to set an explicit height, while inline-size usually doesn't?

To avoid circularity (content changes container size → query flips → content changes), a query container's queried axis must be computable without looking at its contents — so container-type applies containment. inline-size only contains the inline axis, and block elements typically get inline size from their parent anyway, so height still grows with content. size contains both axes: the element no longer derives height from content, so with no explicit block size it collapses to zero. That's why inline-size is the default choice.

What are container query units and when would you use cqi over cqw?

cqw/cqh/cqi/cqb/cqmin/cqmax — percentages of the nearest size container's dimensions. cqi is the logical inline equivalent of cqw: identical in horizontal writing modes, but correct in vertical writing modes — same reasoning as preferring logical properties generally. Use case: container-relative fluid type, e.g. font-size: clamp(1rem, 4cqi, 1.75rem) so a card headline scales with the card, not the viewport.

Explain the difference between .card:not(:has(img)) and .card:has(:not(img)).

.card:not(:has(img)): cards containing no image anywhere. .card:has(:not(img)): cards containing at least one element that isn't an image — true for nearly every card (a title matches). The negation scope differs: negate the "contains" test vs negate the descendant type. Translate to English before writing.

Is :has() a performance problem?

Not inherently — engines use invalidation-set optimizations, and typical usage (specific subject, shallow argument, ideally child combinator) is fine. The risky pattern is a broad subject with a deep unanchored argument (body:has(.a .b:hover)): any mutation or hover anywhere can trigger wide re-matching. Also note constraints: no nesting :has() inside :has(), no pseudo-elements inside, and it carries the specificity of its most specific argument (unlike :where()).

How do cascade layers change the cascade algorithm? Where do unlayered styles and !important fit?

After origin/importance, the cascade compares layer order before specificity: a declaration in a later layer beats an earlier layer regardless of selector specificity; specificity only breaks ties within a layer. For normal declarations, unlayered author styles beat all layered ones (great for migration: layer vendor CSS, your styles win). For !important declarations the order inverts: important-in-earlier-layer beats important-in-later-layer, and important-layered beats important-unlayered — so foundational layers can make unstompable guarantees.

Order these and explain: initial, inherit, unset, revert, revert-layer.

initial = spec's initial value, ignoring everything. inherit = parent's computed value, always. unset = inherit for inherited properties, else initial. revert = roll back author styles, leaving user/UA styles (e.g. reverting display on a div gives block, not inline which is the true initial). revert-layer = roll back just the current cascade layer, letting earlier layers apply — the surgical one for layered architectures.

Why does `color: red; color: var(--size)` (where --size is 16px) not render red?

Because var() substitution happens at computed-value time, after the cascade already discarded the red declaration in favor of the later one. When substitution yields an invalid value ("invalid at computed-value time"), the browser can't resurrect earlier declarations; the property behaves as unset — inherited value for inherited properties like color, initial otherwise. Mitigations: var(--x, fallback), or register the property with @property so invalid values fall back to the typed initial-value.

Why is `* { --gap: 1rem }` an anti-pattern?

It sets the property directly on every element, and direct declarations beat inheritance. So .compact { --gap: .5rem } only changes .compact itself — its children still get 1rem from the universal rule, breaking subtree theming. Declare tokens on :root and override at subtree scope; inheritance does the propagation.

What does @property give you over a plain custom property?

Registration with syntax (type), initial-value, and inherits. Consequences: (1) the property becomes animatable — the engine knows how to interpolate an <angle> or <color>, whereas untyped custom properties flip discretely; (2) invalid values fall back to the typed initial value instead of the unset/inherit surprise; (3) you can opt out of inheritance. Classic demo: transitioning a gradient angle or gradient color stops, which is impossible otherwise. Baseline since Firefox 128 (2024).

Can you use custom properties in media queries? What's the alternative?

No — @media (min-width: var(--bp)) is invalid. Custom properties are resolved per-element via the cascade; a media query is evaluated against the environment with no element context. Alternatives: preprocess breakpoints (Sass/PostCSS variables at compile time), use container queries where component-relative logic fits, or await the specced custom environment variables. Distinguish this from using custom properties inside a media query block, which is fine and is exactly how responsive token remapping works.

Write fluid typography with clamp() and explain why the middle value should mix rem with vw.

font-size: clamp(1.75rem, 1.125rem + 2.5vw, 3rem). A pure-vw preferred value doesn't scale when the user zooms text or raises their root font size — failing WCAG 1.4.4 (text resizable to 200%). Adding a rem term keeps the formula responsive to both viewport and user preference. Derivation: slope = Δfont / Δviewport between your two design points; rem intercept = size − slope × viewport at either point.

:is() vs :where() — when do you reach for each?

Both group selectors and are forgiving of invalid list items. :is() takes the specificity of its most specific argument; :where() contributes zero. Use :where() for resets and library base styles that must lose to any consumer selector; use :is() as plain grouping sugar where normal specificity is fine. Bonus: native nesting desugars the parent reference with :is() semantics, which is why nested-rule specificity can differ from Sass output.

What's different between native CSS nesting and Sass nesting?

Key gotchas: (1) no string concatenation — &__title doesn't build .card__title; & is a selector reference, so BEM-by-nesting doesn't port; (2) the implied parent behaves like :is(parent), so with selector lists the specificity can differ from Sass's expanded output; (3) at-rules (@media, @container) nest natively too. Same discipline caveat as Sass: shallow nesting only, or you couple CSS to DOM structure and inflate specificity.

Why did design systems move to OKLCH?

Perceptual uniformity: equal OKLCH lightness ≈ equal perceived brightness across hues, unlike HSL where yellow at L=50% looks far brighter than blue at L=50%. That means palette scales generated by varying L behave consistently for contrast/accessibility across every hue, hue rotations don't shift perceived lightness, and interpolation avoids muddy midpoints. It also expresses wide-gamut P3 colors that sRGB hex can't. Tailwind v4 and Radix rebuilt palettes on it. Pair with color-mix(in oklch, …) and relative color syntax for runtime color math that works with custom properties.

Walk me through why animating transform is cheaper than animating top.

Pipeline: Style → Layout → Paint → Composite. Animating top changes geometry, so every frame runs layout (which can cascade to ancestors/siblings), repaint, composite — all main-thread work that competes with JS. Animating transform (or opacity) changes neither geometry nor pixels of the layer: the element is promoted to its own compositor layer and the compositor thread re-positions/re-blends existing GPU textures. Consequence worth stating: compositor animations stay smooth even while the main thread is blocked. That's also the whole motivation for FLIP.

Explain FLIP and where React's useLayoutEffect fits in.

First-Last-Invert-Play animates layout changes with transforms only: measure old position (First), apply the real change and measure new position (Last), transform the element back by the delta so it appears unmoved (Invert), then transition the transform to none (Play) — one layout, then compositor-only motion. In React, do the Last+Invert inside useLayoutEffect: it fires after DOM mutation but before paint, so the user never sees the un-inverted frame. Framer Motion's layout prop and the View Transitions API are FLIP industrialized.

When is will-change harmful?

Each hinted element gets its own compositor layer — a GPU bitmap costing roughly width × height × 4 bytes × dpr². Blanket will-change exhausts GPU memory (mobile especially), forces extra compositing work, and can create blurry text on transformed layers. Correct use: apply just before an animation is likely (e.g. on hover/focus of the trigger), remove after; treat it as a profiler-driven fix. Modern engines auto-promote animating elements, so its main value is avoiding the promotion hitch on the first frame.

How do you implement prefers-reduced-motion properly?

Treat motion as the enhancement: wrap non-essential animation in @media (prefers-reduced-motion: no-preference) so reduced is the default, not a bolted-on override. "Reduced" ≠ "none": keep small opacity/color feedback, remove translation, scale, parallax, autoplaying video. Mirror it in JS (matchMedia) before triggering WAAPI/Framer sequences and smooth scroll. Frame it as an accessibility requirement (vestibular disorders; WCAG 2.3.3), not polish.

Why did the ecosystem move away from runtime CSS-in-JS?

Two compounding reasons. Runtime cost: styles are computed, hashed, and injected during render on the main thread, ship as JS (parse + execute, not streamable like CSS), and SSR needs per-request extraction plus double-shipping styles. RSC incompatibility: Server Components don't run client JS and can't use React Context, which runtime CSS-in-JS needs for injection and theming — so styled-components/Emotion only work in "use client" trees, letting the styling library dictate component architecture. The consensus shifted to zero-runtime options (CSS Modules, Tailwind, vanilla-extract, Linaria, StyleX); styled-components entered maintenance mode in 2025. Fair caveat: an existing client-only SPA using Emotion isn't broken — this is about defaults for new work.

Tailwind's critics say "unreadable class soup" and "duplication." Steelman both sides.

Critique steelman: long class strings hurt scanability and diff noise; markup and design concerns intermix; without a component layer, duplication is real and drift follows; @apply as an escape hatch recreates the naming problem in a proprietary dialect. Defense steelman: readability is local — no cross-file cascade archaeology; the constrained token scale is an enforced design system (the real win at team scale); components, not classes, are the reuse boundary in React, so duplication lives in one place; JIT output plateaus sub-linearly; deletion is fearless because styles die with markup. Senior close: Tailwind is a bet that consistency-via-constraint beats expressiveness for product UI teams — right for many app teams, wrong for shipped design-system libraries needing consumer overridability.

How do you pass a dynamic runtime value into a statically-extracted styling system (CSS Modules/Tailwind)?

The CSS custom-property bridge: style={{'--progress': `${pct}%`}} on the element, and static CSS consumes var(--progress). The static rule stays cacheable; only the tiny value is inline. This handles the vast majority of "we need CSS-in-JS for dynamic styles" cases, works in RSC (it's just an attribute), and keeps transitions/pseudo-elements in CSS. Zero-runtime libraries (Linaria, vanilla-extract's assignInlineVars) automate exactly this pattern.

Design a dark mode that respects system preference but allows user override, without a flash of wrong theme.

Semantic tokens as the theming surface; default dark mapping under @media (prefers-color-scheme: dark) scoped to "no explicit choice" (or resolve everything to data-theme). User override sets data-theme="light|dark" on <html>, persisted in localStorage with three states: light/dark/system. FOUC fix: a tiny render-blocking inline script in <head> that reads localStorage, falls back to matchMedia, and sets the attribute before first paint — a useEffect is too late by definition. Also set color-scheme: light dark for native controls, and know light-dark() as the newer one-declaration option.

What are the three design-token tiers and why not just use primitives everywhere?

Primitive (raw values: --blue-500), semantic (intent: --color-action), component (override points: --button-bg). If components consume primitives directly, dark mode and rebrands require touching every component, and "why is this blue" has no answer in the code. Semantic tokens make theming a single-tier remap; component tokens give consumers a supported override API instead of specificity wars. Source of truth typically lives in the W3C DTCG JSON format, compiled by Style Dictionary to CSS/TS/native.

Do you still need Sass in 2026?

Mostly no for new projects: custom properties replaced variables (and are more capable — runtime, themable), native nesting replaced nesting, color-mix()/relative color syntax replaced color functions, calc() family replaced math. What's left: parameterized @mixins and loops for code generation — and native CSS @function/@mixin are on the standards track with Chromium prototypes. Posture: don't rewrite existing Sass; stop adding Sass variables for anything themable; keep Sass only where mixins/loops genuinely pay.

What are view transitions, and how would you adopt them safely?

The browser snapshots the old and new DOM states and animates between them (default cross-fade; customizable per element via view-transition-name and ::view-transition-* pseudo-elements). Same-document via document.startViewTransition(cb); cross-document (MPA) via @view-transition { navigation: auto }. Adoption: strictly progressive enhancement — feature-detect (if (!document.startViewTransition) { update(); return; }), respect prefers-reduced-motion, and lean on framework integrations (React Router, Next, Astro). Same-document is broadly available across engines as of 2025; cross-document is newer.

Exercise 1 — Container-query card

Build a .media-card (image + title + description + CTA) that renders horizontal (image left, 40%) when its container is ≥ 420px and stacked when narrower, with a container-relative fluid title. Place the same card in a 3-column grid and a 280px sidebar to prove it adapts with zero media queries or props.

Solution 
.card-slot { container: card / inline-size; }

.media-card {
  display: flex; flex-direction: column; gap: 0.75rem;
}
.media-card__img { width: 100%; aspect-ratio: 16/9; object-fit: cover; }
.media-card__title { font-size: clamp(1rem, 0.8rem + 2cqi, 1.5rem); }

  .media-card { flex-direction: row; gap: 1.25rem; }

}

Notes: containment lives on a dedicated .card-slot wrapper sized by the surrounding layout (grid cell / sidebar), never on the card itself; 2cqi makes the title scale with the card. Mobile-first stacked styles are the unqueried baseline, so older browsers degrade gracefully.

Exercise 2 — JS-free form validation UI with :has() and :user-invalid

Given <div class="field"><label>…</label><input required type="email"><p class="field__error">…</p></div>: (a) don't show errors on page load; (b) after the user interacts and leaves the field invalid, tint the wrapper border, color the label, and reveal the error; (c) show a success tint when user-valid; (d) highlight the wrapper on focus.

Solution 
.field { border: 1px solid var(--color-border); border-radius: 0.5rem; padding: 0.75rem; }
.field__error { display: none; color: var(--color-danger); }

.field:has(:focus-visible) { outline: 2px solid var(--color-focus); outline-offset: 2px; }

/* :user-invalid, not :invalid — empty required fields are :invalid on load */
.field:has(:user-invalid) { border-color: var(--color-danger); }
.field:has(:user-invalid) label { color: var(--color-danger); }
.field:has(:user-invalid) .field__error { display: block; }

.field:has(:user-valid) { border-color: var(--color-success); }

Key insight for the interview: :invalid would flag the pristine form on arrival; :user-invalid waits for real interaction. Still pair with server-side validation and aria-describedby/aria-invalid for screen readers — CSS reveals the message visually but semantics need markup.

Exercise 3 — Layer a legacy vendor stylesheet

A vendor datepicker stylesheet full of #datepicker .day.selected-grade selectors keeps beating your overrides, and your codebase has accumulated !importants to fight it. Restructure with @layer so all vendor CSS loses to your unlayered app styles, define a full layer order for the app, and use revert-layer to strip the vendor's box-shadow in one theme.

Solution 
/* entry.css — order declared first; later wins */
@layer vendor, reset, tokens, components, utilities;

@import url("vendor/datepicker.css") layer(vendor);
@import url("reset.css") layer(reset);

@layer components {
  /* single class now beats #datepicker .day.selected —
     layer order is compared before specificity */
  .day-selected { background: var(--color-action); }
}

@layer components {
  [data-theme="minimal"] .datepicker {
    box-shadow: revert-layer; /* falls back to vendor layer's value…
      or wrap vendor tweaks in a later sub-layer to fully unset */
  }
}

Then delete the !importants incrementally. The migration is safe because unlayered existing app styles automatically beat everything you moved into layers — you can layer the vendor file on day one with zero visual risk, then move app CSS into layers gradually.

Exercise 4 — FLIP a React list reorder

Given a list re-sorted by a button click, animate each item gliding to its new position at 60fps — no layout-animating libraries. Requirements: transforms only, useLayoutEffect, WAAPI, and skip the animation under prefers-reduced-motion.

Solution 
function useFlip(items) {
  const refs = useRef(new Map());   // id → node
  const prev = useRef(new Map());   // id → DOMRect

  useLayoutEffect(() => {           // after mutation, BEFORE paint
    const reduce = matchMedia('(prefers-reduced-motion: reduce)').matches;
    refs.current.forEach((node, id) => {
      const last = node.getBoundingClientRect();
      const first = prev.current.get(id);
      if (first && !reduce) {
        const dx = first.left - last.left;
        const dy = first.top - last.top;
        if (dx || dy) node.animate(
          [{ transform: `translate(${dx}px,${dy}px)` }, { transform: 'none' }],
          { duration: 220, easing: 'cubic-bezier(.2,.8,.2,1)' }
        );
      }
      prev.current.set(id, last);
    });
  }, [items]);

  return (id) => (node) => node ? refs.current.set(id, node) : refs.current.delete(id);
}

Talking points: keys must be stable ids (index keys would remount and kill FLIP); all reads happen before any animation write (no read/write interleaving = no forced synchronous layout); the animation is transform-only so after the single reorder layout, every frame is compositor work.

Exercise 5 — Token architecture + flashless hybrid dark mode

Design the token sheet for a small app: primitive OKLCH grays/brand, semantic tier, dark remap that follows the system by default but honors a stored user override, the head script preventing FOUC, and color-scheme so native controls match.

Solution 
:root {
  color-scheme: light dark;
  /* primitives */
  --gray-100: oklch(0.97 0.005 260);
  --gray-900: oklch(0.22 0.02 260);
  --brand-500: oklch(0.62 0.19 255);
  /* semantic (light defaults) */
  --color-bg: var(--gray-100);
  --color-text: var(--gray-900);
  --color-action: var(--brand-500);
}

/* system dark, only when user hasn't chosen */
@media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) {
    --color-bg: var(--gray-900);
    --color-text: var(--gray-100);
  }
}
/* explicit user choice wins either way */
[data-theme="dark"] {
  --color-bg: var(--gray-900);
  --color-text: var(--gray-100);
}

<script>/* inline in <head>, before first paint */
  try {
    const t = localStorage.getItem('theme');
    if (t === 'light' || t === 'dark')
      document.documentElement.dataset.theme = t;
  } catch (e) {}
</script>

Notes: components only ever touch the semantic tier; the media query is scoped with :not([data-theme="light"]) so "system" remains the default state (nothing stored → no attribute → media query governs); the script only sets the attribute for explicit choices, so it stays tiny and the flash is impossible because it's render-blocking in <head>.
