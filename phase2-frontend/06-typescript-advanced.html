06 — TypeScript Advanced

The type-system mechanics behind the syntax you use daily — generics, narrowing, mapped/conditional types, and typing React the way library authors do

Why interviewers ask this

Every senior frontend candidate claims TypeScript on their resume, so interviewers don't ask "what is an interface." They probe whether you understand how the type system works — because that predicts whether you can design a library API, debug a cryptic 40-line type error, review a junior's as any, and articulate the trade-offs TS deliberately makes (it is intentionally unsound in specific places for ergonomics). The interview signals they're fishing for:

  • Mental model — do you know TS is structural, that types are erased at runtime, and what that implies?
  • Type-level programming — can you implement Pick, ReturnType, or extract route params from a string literal? (FAANG and TS-heavy startups love live type-challenges.)
  • Applied React typing — discriminated-union reducers, generic components, extending native element props. This is where 7 years of experience should show.
  • Judgment — when as is acceptable, why satisfies exists, which strict flags you'd turn on and why.
A recurring theme in strong answers: TS constantly trades soundness for ergonomics (bivariant methods, excess property checks only on fresh literals, any existing at all). Naming the trade-off — not just the rule — is what reads as senior.

1. Type-system mental model T1

Structural typing ("duck typing, but checked")

TypeScript decides compatibility by shape, not by name. If a value has at least the members a type requires, it's assignable — no implements, no inheritance chain needed. This is the opposite of nominal systems (Java, C#, Rust) where the declared name is the identity.

interface Point { x: number; y: number; }

class Vector {          // never mentions Point
  constructor(public x: number, public y: number) {}
}

function len(p: Point) { return Math.hypot(p.x, p.y); }

len(new Vector(3, 4));       // ✅ fine — same shape
len({ x: 3, y: 4 });          // ✅ fine — a plain object works too

Consequence: two types that mean different things but share a shape are interchangeable — UserId and OrderId both being string means you can pass one where the other is expected and TS won't blink. When you genuinely need nominal behavior, the idiom is a branded type:

type UserId  = string & { readonly __brand: 'UserId' };
type OrderId = string & { readonly __brand: 'OrderId' };

// One controlled "blessing" point — the only place `as` appears:
function toUserId(raw: string): UserId {
  // validate here (regex, lookup, ...)
  return raw as UserId;
}

function getUser(id: UserId) { /* ... */ }

declare const orderId: OrderId;
getUser(orderId);            // ❌ error — brands don't match
getUser('123');              // ❌ error — plain string lacks the brand
getUser(toUserId('123'));    // ✅
The brand property never exists at runtime — it's a compile-time-only intersection. That's the whole point: you're simulating nominal typing inside a structural system with zero runtime cost. Libraries like Zod expose z.string().brand<'UserId'>() for exactly this. Mentioning "we branded IDs/currency amounts to stop unit-mixing bugs at compile time" is a great real-world flex.

Inference & widening: const vs let, and as const

TS infers the narrowest useful type, then widens literals depending on mutability. A let can be reassigned, so a literal widens to its base type; a const binding can't, so the literal type is kept:

let   a = 'hello';   // string          (widened — could be reassigned)
const b = 'hello';   // 'hello'         (literal — binding is immutable)

// BUT object properties are mutable even under const:
const req = { url: '/api', method: 'GET' };
// { url: string; method: string }  ← method widened!

fetchIt(req.url, req.method);
// ❌ if fetchIt wants method: 'GET' | 'POST' — string isn't assignable

// `as const` freezes the whole tree — deeply:
const req2 = { url: '/api', method: 'GET', tags: ['a', 'b'] } as const;
// {
//   readonly url: '/api';
//   readonly method: 'GET';
//   readonly tags: readonly ['a', 'b'];   ← tuple, not string[]
// }

as const does three things at once: keeps every literal as a literal type, makes every property readonly, and turns arrays into readonly tuples — recursively. It's the foundation of config-object patterns and of deriving union types from data:

const ROLES = ['admin', 'editor', 'viewer'] as const;
type Role = typeof ROLES[number];   // 'admin' | 'editor' | 'viewer'
// Single source of truth: the runtime array AND the type stay in sync.

Excess property checks — why fresh object literals are special

Structural typing says "extra properties are fine." But TS makes one deliberate exception: when an object literal is assigned directly ("fresh" literal) to a typed target, unknown properties are errors. It's a lint-like typo catcher layered on top of structural rules — and it's the classic "why does this error here but not there?" confusion:

interface Options { title: string; width?: number; }

configure({ title: 'Hi', widht: 100 });
// ❌ 'widht' does not exist in type Options — fresh literal → checked

const opts = { title: 'Hi', widht: 100 };
configure(opts);
// ✅ compiles! opts's type is { title: string; widht: number } —
// structurally a valid Options (has title). Freshness was lost
// the moment it got its own inferred type.
Excess property checking is not a guarantee that objects have no extra keys. Any indirection (intermediate variable, function return, spread through a wider type) bypasses it. If your logic depends on "no extra keys" (e.g., building a DB row), you need runtime validation — the type system will not enforce closed objects.

function handle(payload: unknown) {
  payload.toUpperCase();                     // ❌ must narrow first
  if (typeof payload === 'string') {
    payload.toUpperCase();                   // ✅ narrowed to string
  }
}

// never = "this cannot happen" — the compiler enforces it:
function assertNever(x: never): never {
  throw new Error(`Unexpected: ${JSON.stringify(x)}`);
}
unknown is "I don't know what this is, and I want the compiler to force me to find out." any is "stop checking." A senior answer also notes never appears organically: an exhausted union narrows to never, Exclude<T, T> is never, and a never in a union simply vanishes (string | never = string) — that vanishing is what makes key-filtering mapped types work.

void quirks — why forEach(x => arr.push(x)) works

void in a return-type position of a function type doesn't mean "returns nothing" — it means "the caller promises to ignore the return value." So a function that returns something is assignable to a () => void signature:

type Callback = () => void;
const cb: Callback = () => 42;   // ✅ legal! return value will be ignored

// This rule is WHY this compiles — push returns a number:
const src = [1, 2, 3];
const out: number[] = [];
src.forEach(x => out.push(x));   // forEach cb is (…) => void; push returns number — fine

// But a function you DECLARE with `: void` cannot return a value:
function log(msg: string): void {
  return msg;                    // ❌ error inside a literal void annotation
}
This leniency has a real gotcha: onSelect?: () => void happily accepts an async function async () => {…} (returns Promise<void>). The promise is silently ignored — unhandled rejections disappear. If you care about awaiting or error propagation, type it () => void | Promise<void> and handle both.

2. Interfaces vs type aliases — the real differences T1

The mid-level answer is "they're mostly the same." The senior answer names the actual mechanical differences and gives a policy.

Dimensioninterfacetype alias
Can expressObject shapes, function shapes, and can be implemented by classesEverything: unions, intersections, tuples, primitives, mapped, conditional, template literals
Declaration merging✅ Multiple declarations with the same name merge (this is how declare global augmentation and lib patching work)❌ Duplicate name = error
Extensionextends — checked at declaration; conflicting member types are an immediate error& intersection — conflicts don't error at the definition; a conflicting property silently becomes never and blows up at use sites
Error messages / displayKeeps its name in errors and hovers (Type 'X' is not assignable to 'Props')Often inlined/expanded in errors — walls of structure for complex aliases

// Declaration merging — the superpower AND the footgun:
interface Window { analytics: AnalyticsClient; }   // augments the global Window

// Intersection conflict is silent at definition…
type A = { x: string };
type B = { x: number };
type AB = A & B;        // compiles; AB['x'] is never
const ab: AB = { x: 1 }; // ❌ number not assignable to never — confusing error, far away

// interface extends catches it at the source:
interface C extends A { x: number; }  // ❌ error right here
The policy I'd state in an interview: use interface for object shapes that form a public contract (props of a published component, plugin APIs — merging lets consumers augment them; names survive in errors). Use type for everything an interface literally cannot express: unions, discriminated unions, mapped/conditional results, tuples. Within a team, consistency beats the choice itself — and note that declaration merging is a reason some teams ban interfaces for internal code: a duplicate name silently merges instead of erroring.

3. Generics deep T1

Type parameters are inferred from arguments

A generic function is a function whose type is completed per call site. You almost never pass type arguments explicitly — inference fills them from the values:

function identity<T>(value: T): T {
  return value;
}

const n = identity(42);        // T inferred as 42 (literal in const position)
const s = identity('hi');      // T = 'hi'
const u = identity<string>('hi'); // explicit — only when inference can't see it

// "Return what you're given" — generics preserve the RELATIONSHIP
// between input and output. Compare:
function firstBad(arr: any[]): any { return arr[0]; }      // type info destroyed
function first<T>(arr: T[]): T | undefined { return arr[0]; } // preserved
If a type parameter appears only once in a signature, it's usually pointless — function log<T>(x: T): void gains nothing over unknown. Generics earn their keep by relating two or more positions (arg↔return, arg↔arg).

Constraints, defaults, and the keyof pattern

// Constraint: T must have .length — inside the body you can use it
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}
longest([1, 2], [3]);      // ✅ arrays have length
longest(10, 20);           // ❌ number has no length

// Default type parameters — like default function args:
interface ApiResponse<TData = unknown> {
  status: number;
  data: TData;
}
const r: ApiResponse = { status: 200, data: null }; // TData falls back to unknown

The single most-asked generic signature in interviews — walk through every piece:

function get<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { id: 1, name: 'Ada', admin: true };
const name  = get(user, 'name');   // string  — K = 'name', T[K] = string
const admin = get(user, 'admin');  // boolean
get(user, 'email');                // ❌ '"email"' not assignable to keyof T
  • T is inferred from obj → { id: number; name: string; admin: boolean }.
  • K extends keyof T constrains K to the union 'id' | 'name' | 'admin'; passing a string literal infers K as that exact literal, not string.
  • T[K] is an indexed access type — "the type of property K on T." Because K is the literal 'name', the return type is precisely string, not a union of all value types.

Generic functions vs generic types

// Generic TYPE: parameterized shape — caller supplies T when naming it
type Result<T, E = Error> =
  | { ok: true;  value: T }
  | { ok: false; error: E };

// Generic FUNCTION TYPE: T is bound at CALL time, not at declaration —
// note where the <T> sits:
type Mapper = <T, U>(items: T[], fn: (t: T) => U) => U[];
// vs type MapperOf<T, U> = (items: T[], fn: …) => U[]  — bound when the type is used

Variance — the soundness question T2

"Is Array<Dog> assignable to Array<Animal>?" In TS: yes — and that's deliberately unsound.

interface Animal { name: string }
interface Dog extends Animal { bark(): void }
interface Cat extends Animal { meow(): void }

const dogs: Dog[] = [];
const animals: Animal[] = dogs;   // ✅ allowed (covariant) — reads are safe…
animals.push({ name: 'Felix', meow() {} } as Cat); // …writes are NOT
dogs[0].bark();                    // 💥 runtime error, zero compile errors

Why TS allows it: a mutable Array<T> is read (covariant position) and written (contravariant position), so soundness demands invariance — which would reject tons of perfectly reasonable read-only usage. TS chose ergonomics. Mechanically, this works because methods declared with shorthand syntax are checked bivariantly, even under strictFunctionTypes:

// strictFunctionTypes makes standalone function types contravariant in params:
type Handler = (a: Animal) => void;
const h: Handler = (d: Dog) => d.bark();   // ❌ rejected — correctly!
// (a Dog-handler can't safely receive an arbitrary Animal)

// …but METHOD syntax stays bivariant on purpose:
interface Emitter { on(cb: (a: Animal) => void): void }   // method — bivariant
interface Emitter2 { on: (cb: (a: Animal) => void) => void } // property — strict
The intuition to recite: outputs are covariant, inputs are contravariant. A producer of Dogs is a valid producer of Animals; a consumer of Animals is a valid consumer of Dogs. Arrays are both, so treating them covariantly is a hole — the same hole Java has with arrays. If you want safety, accept readonly T[] in signatures: read-only means covariance actually is sound, and callers can pass mutable arrays into it for free.

4. Narrowing & control-flow analysis T1

TS re-types a variable per code location based on the checks you've performed — control-flow analysis. The built-in guards:

function fmt(v: string | number | Date | null, other: 'a' | 'b') {
  if (v === null) return '—';              // equality narrowing: v is string|number|Date
  if (typeof v === 'string') return v.trim();  // typeof guard
  if (v instanceof Date) return v.toISOString(); // instanceof guard
  return v.toFixed(2);                     // only number remains
}

// `in` guard — for plain object shapes:
type Square = { size: number };
type Rect   = { width: number; height: number };
function area(s: Square | Rect) {
  return 'size' in s ? s.size ** 2 : s.width * s.height;
}

// Truthiness narrowing — beware falsy-but-valid values:
function pad(n?: number) {
  if (!n) return DEFAULT;   // ⚠️ also swallows 0! use n === undefined
}

Discriminated unions — the workhorse of app TypeScript

A union of object types sharing one literal-typed discriminant property. Check the discriminant, and TS narrows to exactly one branch. This is the pattern for async state, reducer actions, API results, and form states:

type RequestState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error';   error: Error };

function render(state: RequestState<User[]>) {
  switch (state.status) {
    case 'idle':    return null;
    case 'loading': return spinner();
    case 'success': return list(state.data);    // data exists ONLY here
    case 'error':   return alert(state.error.message);
    default:        return assertNever(state);   // exhaustiveness — see below
  }
}

Why this beats optional fields. The naive alternative:

// ❌ The "bag of optionals" — every impossible state is representable:
interface BadState<T> {
  loading: boolean;
  data?: T;
  error?: Error;
}
// loading:true AND data set? error AND data both set? data undefined on
// success? The compiler can't rule any of it out, so every consumer
// defensively if-checks everything — or worse, uses data! assertions.

The discriminated union makes illegal states unrepresentable: data cannot exist without status: 'success', the compiler proves you handled every state, and adding a 'refreshing' state produces compile errors at every unhandled switch — a to-do list generated by the compiler.

Custom type predicates & assertion functions

// Type predicate: the return type `arg is T` teaches the narrower.
function isSuccess<T>(s: RequestState<T>): s is { status: 'success'; data: T } {
  return s.status === 'success';
}

// Predicates power .filter() narrowing:
const values = [1, null, 2, undefined, 3];
const nums = values.filter((v): v is number => v != null); // number[]

// Assertion function: narrows the REST of the scope, or throws.
function assertIsUser(v: unknown): asserts v is User {
  if (typeof v !== 'object' || v === null || !('id' in v)) {
    throw new Error('Not a User');
  }
}

function load(raw: unknown) {
  assertIsUser(raw);
  raw.id;              // ✅ raw is User from here down
}
A type predicate is a promise you make, not one the compiler checks. function isString(x: unknown): x is string { return typeof x === 'number' } compiles fine and poisons every call site. Treat predicate bodies with the same suspicion as as — they carry unchecked responsibility. (TS 5.5 does infer predicates for simple boolean-returning functions, which removes the hand-written risk for easy cases.)

Exhaustiveness — never and the satisfies trick

// Classic: a default branch that only type-checks if the union is exhausted.
function assertNever(x: never): never {
  throw new Error(`Unhandled case: ${JSON.stringify(x)}`);
}
// If you add {status:'refreshing'} to the union, `state` in the default
// branch is 'refreshing'-typed, not never → compile error where you forgot it.

// Expression-position alternative (no helper needed):
switch (state.status) {
  /* …all cases with returns… */
  default:
    state satisfies never;   // errors unless every member was handled
    throw new Error('unreachable');
}

Interviewers rarely ask "what does Partial do." They ask "implement Partial" — because each built-in utility is a two-line showcase of one type-system feature. Know the implementation and you know the feature.

Watch distributivity in action — it's why Exclude works at all:

type MyExclude<T, U> = T extends U ? never : T;

type X = MyExclude<'a' | 'b' | 'c', 'a'>;
// The union DISTRIBUTES — the conditional runs per member:
//   ('a' extends 'a' ? never : 'a')   → never
// | ('b' extends 'a' ? never : 'b')   → 'b'
// | ('c' extends 'a' ? never : 'c')   → 'c'
// never | 'b' | 'c'  →  'b' | 'c'   (never vanishes from unions)

String-manipulation intrinsics — implemented inside the compiler, used constantly with template literal types:

type A = Uppercase<'click'>;    // 'CLICK'
type B = Lowercase<'GET'>;      // 'get'
type C = Capitalize<'click'>;   // 'Click'
type D = Uncapitalize<'OnClick'>; // 'onClick'

6. Mapped types & key remapping T2

A mapped type is a for-loop over keys: [K in KeyUnion] introduces a type variable K ranging over each member, and the right side computes the value type — where T[K] gives you the original property type.

type Flags<T> = { [K in keyof T]: boolean };      // same keys, all boolean
type Clone<T> = { [K in keyof T]: T[K] };         // homomorphic copy —
// mapping over `keyof T` preserves ?, readonly, and doc comments per key

// Modifiers: + adds (default), - removes:
type Mutable<T>  = { -readonly [K in keyof T]: T[K] };
type Concrete<T> = { [K in keyof T]-?: T[K] };   // = Required

Key remapping with as — rename, prefix, filter

// The interview favorite — derive a getters API from a data model:
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

interface User { id: number; name: string; }
type UserGetters = Getters<User>;
// { getId: () => number; getName: () => string }
// `string & K` filters out symbol keys — template literals need strings.

// Remapping to `never` DELETES a key — conditional filtering:
type OnlyMethods<T> = {
  [K in keyof T as T[K] extends (...args: any) => any ? K : never]: T[K];
};

// Mapping over an arbitrary union (not keyof) — builds objects from strings:
type Feature = 'darkMode' | 'newNav' | 'betaSearch';
type FeatureFlags = { [F in Feature]: { enabled: boolean; rollout: number } };
A mapped type of the exact form { [K in keyof T] …: X } is called homomorphic — TS knows it's a structure-preserving map over T, so it carries over ?/readonly modifiers and even maps over arrays/tuples sensibly (this is why Readonly<string[]> is readonly string[], not an object with array keys).

7. Conditional types, infer & template literals T2

The basics: type-level if/else

type IsString<T> = T extends string ? true : false;
type A = IsString<'hi'>;   // true
type B = IsString<42>;     // false

Distribution over unions — and the tuple trick to stop it

When the checked type is a naked type parameter, the conditional automatically distributes over each union member. Usually that's what you want (Exclude); sometimes it isn't:

type ToArray<T> = T extends any ? T[] : never;
type R1 = ToArray<string | number>;
// string[] | number[]   ← ran once per member (distributed)

// Wrap both sides in a one-tuple → no longer "naked" → no distribution:
type ToArrayAll<T> = [T] extends [any] ? T[] : never;
type R2 = ToArrayAll<string | number>;
// (string | number)[]   ← evaluated once for the whole union

// Classic bug this explains — IsNever must use the tuple trick:
type IsNeverBad<T> = T extends never ? true : false;
type N1 = IsNeverBad<never>;  // never! (distributing over the empty union = nothing runs)
type IsNever<T> = [T] extends [never] ? true : false;
type N2 = IsNever<never>;     // true ✅

infer — pattern matching for types

infer R declares a type variable inside the extends pattern and lets TS solve for it. It works in any structural position — return type, parameters, array elements, tuple slots, template literal segments:

// Return position:
type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

// Parameter position (infers the whole tuple):
type MyParameters<T> = T extends (...args: infer P) => any ? P : never;

// Element position:
type ElementOf<T> = T extends (infer E)[] ? E : never;

// Tuple slots + rest:
type First<T extends readonly unknown[]> =
  T extends readonly [infer F, ...unknown[]] ? F : never;
type Last<T extends readonly unknown[]> =
  T extends readonly [...unknown[], infer L] ? L : never;

// Recursion — unwrap nested promises (simplified Awaited):
type MyAwaited<T> = T extends PromiseLike<infer U> ? MyAwaited<U> : T;
type W = MyAwaited<Promise<Promise<string>>>;   // string

Template literal types — the showstopper demo

String literal types can be composed and pattern-matched. The demo that lands hardest in interviews: type-safe route params, exactly like typed routers (TanStack Router, tRPC) do it:

type ExtractParams<Path extends string> =
  Path extends `${infer _Pre}:${infer Param}/${infer Rest}`
    ? { [K in Param]: string } & ExtractParams<Rest>
    : Path extends `${infer _Pre}:${infer Param}`
      ? { [K in Param]: string }
      : {};

type P = ExtractParams<'/users/:id/posts/:postId'>;
// { id: string } & { postId: string }

function navigate<Path extends string>(path: Path, params: ExtractParams<Path>) { /* … */ }

navigate('/users/:id/posts/:postId', { id: '1', postId: '9' }); // ✅
navigate('/users/:id/posts/:postId', { id: '1' });               // ❌ postId missing

How the match works on '/users/:id/posts/:postId': TS solves the first pattern with _Pre = '/users/', Param = 'id' (earlier infer holes match as little as possible up to the next literal), Rest = 'posts/:postId'. Recursion on Rest hits the second pattern → { postId: string }. Intersect and done.

// Event-name mapping — the other classic:
type DomEvent<T extends string> = T extends `on${infer E}` ? Lowercase<E> : never;
type E1 = DomEvent<'onClick'>;   // 'click'

// …and the reverse, generating handler props from an event map:
type HandlerProps<T> = {
  [K in keyof T & string as `on${Capitalize<K>}`]: (payload: T[K]) => void;
};
type H = HandlerProps<{ click: MouseEvent; focus: FocusEvent }>;
// { onClick: (payload: MouseEvent) => void; onFocus: (payload: FocusEvent) => void }

8. satisfies, as const, and assertion discipline T2

The problem satisfies solves

Before TS 4.9 you had two bad options for "check this object against a type but keep its precise inferred type":

type RouteConfig = Record<string, string | { path: string; lazy: boolean }>;

// Option 1 — annotation: validated ✅ but WIDENED ❌
const routes1: RouteConfig = {
  home: '/',
  admin: { path: '/admin', lazy: true },
};
routes1.home.toUpperCase();   // ❌ home is string | {…} — TS forgot which
routes1.tpyo;                 // no error — Record<string,…> allows any key

// Option 2 — `as`: narrow ✅ but UNCHECKED ❌ (it's an assertion, not a check)
const routes2 = {
  home: '/',
  admin: { path: '/admin', lazy: 'yes' },   // bug sails through
} as RouteConfig;

// satisfies — validated ✅ AND narrow inference preserved ✅
const routes = {
  home: '/',
  admin: { path: '/admin', lazy: true },
} satisfies RouteConfig;

routes.home.toUpperCase();    // ✅ TS knows home is a string
routes.admin.lazy;            // ✅ boolean
routes.tpyo;                  // ❌ error — the inferred type has exact keys

Rule of thumb: satisfies = "prove this expression is assignable to T, then keep the inferred type, not T." Perfect for config objects, theme tokens, route maps, i18n dictionaries — anywhere you want validation without losing key-level precision. It also pairs with as const:

const theme = {
  primary: '#1a56db',
  spacing: [4, 8, 16] ,
} as const satisfies Theme;
// literal types + readonly (as const) AND validated against Theme (satisfies)

Assertion discipline — as is a lie you tell the compiler

  • as performs no runtime conversion and no check — it overrides the checker's opinion. Every as is a place where the type system is only as correct as you are.
  • TS only permits assertions between overlapping types; to jump between unrelated types you must launder through unknown: value as unknown as Target. The double assertion is a deliberate speed bump — treat needing it as a design smell.
  • Acceptable uses: DOM lookups you can vouch for (document.getElementById('app') as HTMLDivElement — though a null check + instanceof is better), test fixtures, the single blessed constructor of a branded type, working around known lib-type gaps (comment why!).
  • Prefer instead: narrowing, type predicates, satisfies, generics, or runtime validation (Zod et al.) at trust boundaries.
as any in a return position is the worst variant: it silently infects every downstream consumer via inference, and no error will ever point back at it. If you must lie, lie as locally and loudly as possible.

The design-system pattern — your Button should accept everything a real <button> does without you re-declaring 200 attributes:

Typing hooks

// useState — inference from the initial value is usually enough:

useRef and the null story. The two ref use-cases get different shapes on purpose:

// DOM ref: YOU never write .current — React does. Initialize with null:

React 19's types simplified this — useRef now requires an argument and always returns a mutable RefObject<T | null>-style shape; the RefObject/MutableRefObject split is being phased out. Know both stories: legacy codebases show the old one.

useReducer with discriminated-union actions — the canonical senior answer for "type a reducer":

interface CartState {
  items: { id: string; qty: number }[];
  discountCode: string | null;
}

type CartAction =
  | { type: 'add';    id: string }
  | { type: 'remove'; id: string }
  | { type: 'setQty'; id: string; qty: number }
  | { type: 'applyDiscount'; code: string }
  | { type: 'clear' };

function cartReducer(state: CartState, action: CartAction): CartState {
  switch (action.type) {
    case 'add':
      return { ...state, items: [...state.items, { id: action.id, qty: 1 }] };
    case 'remove':
      return { ...state, items: state.items.filter(i => i.id !== action.id) };
    case 'setQty':
      return { ...state, items: state.items.map(i =>
        i.id === action.id ? { ...i, qty: action.qty } : i) };  // qty exists ONLY here
    case 'applyDiscount':
      return { ...state, discountCode: action.code };
    case 'clear':
      return { items: [], discountCode: null };
    default:
      return assertNever(action);   // add an action type → compiler lists gaps
  }
}

const [state, dispatch] = useReducer(cartReducer, { items: [], discountCode: null });
dispatch({ type: 'setQty', id: 'a', qty: 2 }); // ✅ payload checked per action type
dispatch({ type: 'setQty', id: 'a' });          // ❌ qty required for 'setQty'

Generic components — the typed Select<T>

interface SelectProps<T> {
  options: readonly T[];
  value: T | null;
  onChange: (value: T) => void;
  getKey: (option: T) => string;
  renderOption?: (option: T) => React.ReactNode;
}

// A generic FUNCTION component — T binds per usage site:
function Select<T>({ options, value, onChange, getKey, renderOption }: SelectProps<T>) {
  return (
    <ul role="listbox">
      {options.map(opt => (
        <li key={getKey(opt)} onClick={() => onChange(opt)}
            aria-selected={value !== null && getKey(opt) === getKey(value)}>
          {renderOption ? renderOption(opt) : getKey(opt)}
        </li>
      ))}
    </ul>
  );
}

/>

Context — the null-check pattern

interface AuthContextValue {
  user: User | null;
  login: (email: string) => Promise<void>;
  logout: () => void;
}

// No sensible default exists → default to null, and gate access via a hook:

Why this beats createContext<AuthContextValue>({} as AuthContextValue): the assertion version silently hands out a broken empty object when the provider is missing (runtime undefined-method crashes far from the cause); the null-check version fails fast with an actionable message and gives every consumer a non-nullable type.

Declaration files & module declarations

// A .d.ts file contains ONLY types — no runtime output. Sources:
//  1. Generated from your code:  "declaration": true → dist/index.d.ts
//  2. Shipped by libraries ("types" field) or @types/* (DefinitelyTyped)
//  3. Hand-written ambient declarations for untyped things:

// globals.d.ts — ambient (no import/export → global scope):

Module resolution: know that "moduleResolution": "bundler" is the modern setting for bundled apps (matches how Vite/esbuild resolve, supports exports maps without file extensions), while "nodenext" is for code actually run by Node.

Types don't exist at runtime — who compiles what

The dominant modern setup splits responsibilities: esbuild/swc/babel strip types and emit JS (fast, per-file, no type knowledge), while tsc --noEmit runs in CI/editor as a pure type checker. This works because erasing types is a per-file syntactic operation — with two infamous exceptions:

// enum EMITS runtime code — it isn't erasable:
enum Status { Active, Archived }
// → var Status; (function (Status) { … })(Status || (Status = {}));

// const enum is WORSE for transpilers: values are inlined at use sites,
// which requires cross-file type information — exactly what per-file
// transpilers (esbuild/swc/babel, isolatedModules) don't have. Avoid.
const enum Dir { Up, Down }
const d = Dir.Up;   // tsc inlines to `0`; a per-file transpiler may just break

// The erasable alternative — union + const object:
const STATUS = { Active: 'active', Archived: 'archived' } as const;
type Status = typeof STATUS[keyof typeof STATUS]; // 'active' | 'archived'
The ecosystem is converging on erasable-only TypeScript: Node's built-in type stripping runs .ts files by deleting types only (no enums, no namespaces-with-values, no constructor parameter properties), and TS 5.8 added erasableSyntaxOnly to enforce that subset at compile time. Being able to say "we enable isolatedModules + erasableSyntaxOnly so any tool — esbuild, swc, Node itself — can strip our types" is a very current, very senior answer. And the umbrella point: types are erased, so x instanceof MyInterface is impossible, type guards must check runtime evidence, and API-boundary validation needs a runtime schema (Zod/Valibot) — from which you derive the static type (z.infer) so the two can't drift.

11. Advanced trivia — awareness level T3

Function overloads vs unions

// Overloads shine when the RETURN type depends on the ARG type:
function createEl(tag: 'img'): HTMLImageElement;
function createEl(tag: 'a'): HTMLAnchorElement;
function createEl(tag: string): HTMLElement {   // implementation signature —
  return document.createElement(tag);            // NOT callable itself, must be
}                                                 // compatible with all overloads

const img = createEl('img');   // HTMLImageElement — resolved top-down, first match wins

// But prefer a union/generic when args are independent — overloads multiply:
function pad(v: string | number): string { /* one signature, done */ return String(v); }
  • Implementation-signature rule: the body's signature is invisible to callers; only the overload list is public. Order matters — put the most specific first.
  • Conditional return types or generics with extends constraints can often replace overloads and compose better; overloads read better when the arg→return mapping is a small finite table.

this typing

// A fake first parameter that only constrains the call context — erased in JS:
function onClickHandler(this: HTMLButtonElement, ev: MouseEvent) {
  this.disabled = true;   // this is typed!
}
// ThisParameterType<typeof onClickHandler> → HTMLButtonElement
// OmitThisParameter<…> strips it. `this: void` forbids this-usage entirely.

Recursion limits, variadic tuples, decorators

  • Recursion depth: conditional-type instantiation depth is capped (~100 for general recursion; tail-recursive conditional types get elevated limits ~1000). Deeply recursive string parsers hit "Type instantiation is excessively deep" — restructure to be tail-recursive with an accumulator.
  • Variadic tuple types: spreads inside tuple types — type Concat<A extends unknown[], B extends unknown[]> = [...A, ...B], type Shift<T> = T extends [unknown, ...infer Rest] ? Rest : never. They're what make typed pipe()/curry() and Parameters-surgery possible.
  • Decorators status: two incompatible worlds — legacy experimentalDecorators (Angular, old NestJS/TypeORM ecosystems, param decorators, works with emitDecoratorMetadata) vs the TC39 standard decorators shipped in TS 5.0 (no flag needed, different API, no param decorators). Know which one a codebase uses before touching them.

Interview Q&A

Answer out loud before expanding. Each model answer includes the "why" an interviewer is listening for.

Is TypeScript's type system sound? What does your answer imply?

No — deliberately. Soundness means "if it type-checks, no type error can occur at runtime," and TS explicitly trades that for ergonomics and JS interop. Known unsound spots: any, type assertions, covariant mutable arrays (Dog[] → Animal[]), bivariant method parameters, non-fresh excess properties, unchecked indexed access (without the flag), and Function.bind-era holes. Implication: TS is a bug-catching and design tool, not a security boundary — runtime validation is still required at trust boundaries (network, storage, user input). Interviewers listen for whether you frame this as a deliberate trade-off rather than a flaw.

Explain structural typing and one problem it causes in a large app.

Compatibility is by shape, not name: any value with the required members is assignable. Problem: semantically distinct types with identical shapes are interchangeable — UserId, OrderId, and SessionToken all being string means the compiler can't stop you swapping them. Fix: branded types (string & { __brand: 'UserId' }) with one blessed constructor function, giving nominal behavior at zero runtime cost. Bonus: mention that structural typing is also why extra properties are generally fine and why mocking in tests is so easy.

Why does { title: 'x', widht: 1 } error when passed inline, but not via a variable?

Excess property checks apply only to fresh object literals — literals assigned directly to a typed target. It's an intentional lint layered onto structural typing to catch typos and misplaced options. Once the literal is bound to a variable, it gets its own inferred type ({ title: string; widht: number }), freshness is gone, and plain structural compatibility takes over — extra members are allowed. Senior addition: therefore TS objects are never "closed"; if extra keys are actually harmful (spreading into a DB row), enforce at runtime.

unknown vs any — give a concrete rule for when each appears in your code.

unknown is the type-safe top type: anything assigns to it, but you can't use it until you narrow — so it appears wherever data enters the system: JSON.parse results, catch (e), postMessage payloads, generic defaults. any disables checking in both directions and is contagious through inference — every expression it touches degrades. Legitimate any: incremental migration seams and rare type-system dead-ends, ideally confined and commented. Rule: "if I'd have to check it before using it anyway, it's unknown; any means I've decided not to know."

Why does numbers.forEach(n => results.push(n)) compile when the callback should return void?

Because void in a function-type's return position means "the caller ignores the return value," not "must return nothing." A (x) => number is assignable to (x) => void. This is deliberate: without it, every concise-arrow callback returning an expression (push returns the new length) would need braces. Contrast: a function you declare with an explicit : void annotation can't return value in its body. Gotcha worth volunteering: () => void silently accepts async functions, orphaning their promises.

Interface vs type — your actual policy, with the mechanical reasons.

Mechanics: interfaces support declaration merging (needed for global/module augmentation, risky internally), keep their name in error messages, and extends errors immediately on member conflicts; type aliases can express unions, tuples, mapped/conditional/template-literal types — which interfaces can't at all — but & intersections resolve conflicts to never silently. Performance differences (cached named extends vs recomputed intersections) exist but only matter at huge scale. Policy: interfaces for public object contracts and anything classes implement or consumers may augment; type aliases for unions and computed types; team consistency above all.

Walk through function get<T, K extends keyof T>(obj: T, key: K): T[K].

Two related type parameters: T infers from the object argument; K extends keyof T constrains the key to the union of T's literal key names — and because the argument is a string literal, K infers as that exact literal, not string. The return type T[K] is an indexed access: the type of that specific property. Net effect: invalid keys are compile errors, and the return type is exact per call site (get(user, 'name') → string). This signature demonstrates the core value of generics: relating input types to output types instead of erasing the relationship with any.

Is Array<Dog> assignable to Array<Animal>? Should it be?

It is (covariant), and strictly it shouldn't be for mutable arrays: after const animals: Animal[] = dogs, animals.push(cat) corrupts dogs with zero compile errors. Sound behavior would be invariance, but that rejects vast amounts of safe read-only code, so TS chose ergonomics — mechanically enabled by method parameters being bivariant even under strictFunctionTypes (which only makes standalone function types contravariant). Practical mitigation: accept readonly T[] in function signatures — covariance is genuinely sound for reads, and mutable arrays are assignable to readonly ones.

Design the state type for a data-fetching component. Defend it.

A discriminated union: {status:'idle'} | {status:'loading'} | {status:'success'; data:T} | {status:'error'; error:Error}. Defense: (1) illegal states are unrepresentable — data can't coexist with 'error', unlike the {loading, data?, error?} bag where 2^n combinations exist and most are nonsense; (2) narrowing on status gives typed access per branch with no ! assertions; (3) exhaustiveness via a never default means adding a state produces compile errors at every unhandled site. This is "parse, don't validate" thinking applied to UI state — and the same shape TanStack Query exposes.

What responsibility do custom type predicates carry?

arg is T is trusted, not verified: the compiler believes your boolean means what the signature claims. A wrong predicate body silently poisons every call site with incorrect narrowing — it's an assertion wearing a function costume. Mitigations: keep predicate bodies trivial and property-complete, colocate them with the type so shape changes are reviewed together, generate them from runtime schemas (Zod's safeParse), and note TS 5.5 now infers predicates for simple boolean returns, eliminating the hand-written gap for easy cases. Same story for assertion functions (asserts arg is T), which additionally must throw on failure.

How do you make the compiler tell you about unhandled switch cases?

Exhaustiveness via never: in the default branch, pass the discriminated value to assertNever(x: never): never — if all cases are handled, the value has narrowed to never and it compiles; add a union member and the default branch sees the new type, which isn't assignable to never → compile error at every incomplete switch. Modern alternative without a helper: state satisfies never; in the default branch. Also mention noFallthroughCasesInSwitch and that if the function returns a value, the return-type check often catches gaps too — but the never pattern works even for void functions.

Implement Omit from scratch and explain each piece.

type MyOmit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>. keyof T yields the key union; Exclude is a distributive conditional (T extends U ? never : T) that runs per key and drops the excluded ones (returned nevers vanish from the union); Pick maps over what's left: { [P in Keys]: T[P] }. Note K extends keyof any (i.e., string | number | symbol) rather than keyof T — matching the built-in, which permits omitting keys not on T. Mention you could inline it with key remapping: { [P in keyof T as P extends K ? never : P]: T[P] }.

What is a distributive conditional type, and how do you turn distribution off?

When the checked type in T extends U ? X : Y is a naked type parameter and T is instantiated with a union, the conditional maps over each member and unions the results — ToArray<string | number> = string[] | number[]. It's what makes Exclude/Extract/NonNullable one-liners. Disable by de-nakedifying both sides: [T] extends [U] ? … — the tuple wrapper makes it a single structural comparison, giving (string | number)[]. Killer detail: never is the empty union, so a distributive conditional over never evaluates zero branches and returns never — which is why IsNever requires the tuple trick.

What does infer do, and where can it appear?

infer R declares a fresh type variable inside an extends pattern and asks the compiler to solve for it during the match — pattern matching at the type level. It can sit in any structural position: return type (ReturnType), parameter tuple (Parameters), array element, specific tuple slots including alongside rests ([infer First, ...infer Rest]), promise payload (Awaited), and template-literal segments (`on${infer E}`). Since TS 4.7 you can constrain it inline: infer S extends string. It only exists within the true branch of the conditional where it was declared.

satisfies vs a type annotation vs as — what does each do to the type?

Annotation (const x: T = …): checks assignability, then the variable's type is T — you lose the narrower inferred type (which keys exist, literal values). Assertion (as T): no meaningful check — it overrides the compiler, so bugs pass. satisfies T: full assignability check, but the expression keeps its inferred type — validation plus precision. Canonical use: config/theme/route objects where you want "conforms to the schema" and "routes.home is specifically a string" simultaneously. Combine with as const for readonly literal trees that are still schema-validated.

Type a Button that accepts all native button props plus a variant.

type ButtonProps = React.ComponentPropsWithoutRef<'button'> & { variant?: 'primary' | 'ghost' }, destructure variant and spread the rest onto <button>. Why this over ButtonHTMLAttributes<HTMLButtonElement>: nearly equivalent, but ComponentPropsWithoutRef also works uniformly for components (ComponentPropsWithoutRef<typeof Input>), so it's the one idiom to remember. Follow-ups to preempt: use ComponentPropsWithRef/forwardRef (or React 19 ref-prop) if refs must pass through; Omit native props that collide with your custom ones (size, color).

Why useRef<HTMLInputElement>(null) — what's the null story?

The ref starts as null because the DOM node doesn't exist until after commit, and React (not you) writes .current. Historically, the null-initialized overload returned RefObject<T> (with current: T | null, treated as read-only for your code), while useRef<number | undefined>(undefined)-style calls returned MutableRefObject for instance-variable use (timer ids, previous values) that you write yourself. React 19's types simplify this: useRef requires an argument and the MutableRefObject/RefObject split is collapsed — refs are just mutable objects. Either way, the two use-cases (DOM ref vs mutable box) remain the concept to articulate.

Why type reducer actions as a discriminated union instead of {type: string; payload?: any}?

The union {type:'setQty'; id:string; qty:number} | {type:'clear'} | … gives: (1) dispatch rejects unknown action types and wrong/missing payloads at the call site; (2) inside the reducer, switching on type narrows so each branch sees exactly its payload — no payload as casts; (3) exhaustiveness via never means new actions can't be silently unhandled. The loose version erases all three: any string dispatches, payload is untyped everywhere, refactors don't propagate. This is Redux Toolkit's model too — createSlice generates precisely-typed action creators for the same reason.

Why does createContext<T | null>(null) + a throwing custom hook beat createContext({} as T)?

Contexts often have no sensible default — the real value comes from a provider. {} as T lies: with a missing provider, consumers receive an empty object typed as fully-populated, and you crash later on ctx.login is not a function, far from the cause. null default + useAuth() that throws "must be used inside AuthProvider" fails fast with an actionable message at the exact misuse — and the hook's null check narrows, so every consumer gets a clean non-nullable type. Bonus: the custom hook is also the seam for selecting/memoizing and for swapping implementation in tests.

What do strictNullChecks and noUncheckedIndexedAccess each catch? Which isn't in strict?

strictNullChecks: removes null/undefined from every type's domain, so possibly-absent values must be modeled (T | null) and narrowed before use — without it TS provides essentially no null safety. noUncheckedIndexedAccess: makes dynamic indexing (arr[i], record[key]) return T | undefined, reflecting reality — it is not part of strict and must be enabled explicitly, which is a favorite gotcha question. It's noisy on array-heavy code (hence excluded from strict) but catches real out-of-bounds and missing-key bugs; idioms like .at(i), destructuring with defaults, and in-checks keep it livable.

Your build uses esbuild/swc. What role does tsc still play, and what syntax becomes problematic?

esbuild/swc strip types file-by-file with no cross-file type knowledge — fast, but zero checking. tsc --noEmit runs in CI and the editor as the actual type checker; the contract is isolatedModules, which flags anything unsafely transpilable per-file (e.g., re-exporting types without export type). Problematic syntax: const enum (inlining needs cross-file info), enum and namespaces (emit runtime code — types are supposed to erase), emitDecoratorMetadata. Trend: erasableSyntaxOnly (TS 5.8) bans all non-erasable syntax so even Node's native type stripping can run the code. Punchline: types don't exist at runtime, so checking and transpiling are genuinely separable jobs.

When are function overloads better than a union parameter?

When the return type depends on which argument shape was passed, and the mapping is a small finite table: createEl('img') → HTMLImageElement, getConfig(key) → typeof config[key], or different arities (fetchUser(id) vs fetchUser(id, opts)). A single union signature returns the same (union) type regardless of input, forcing callers to narrow. Rules to recite: overloads resolve top-down (most specific first); the implementation signature is invisible to callers and must be compatible with every overload; and for many cases a generic with a conditional return type (or K extends keyof Map lookup) scales better than an overload list.

How would you type a function so getEventName('onClick') returns the literal 'click'?

Generic over the string literal + template-literal conditional: function getEventName<T extends string>(name: T): T extends `on${infer E}` ? Lowercase<E> : never — with the implementation doing the runtime lowercase/slice and, realistically, one localized as cast to the conditional return type (conditional return types generally can't be verified against the body; that's a known TS limitation worth naming). The pattern — literal type parameter in, template-literal pattern match, intrinsic string type out — is the same machinery behind typed document.querySelector, tRPC paths, and route-param extraction.

The compiler says "Type instantiation is excessively deep and possibly infinite." What's happening?

A recursive conditional/mapped type exceeded the instantiation depth cap (~50–100 nested instantiations; tail-recursive conditional types get an elevated ~1000 limit). Common causes: type-level string parsing character-by-character on long inputs, recursive tuple building, or accidental infinite recursion (no terminating branch). Fixes: make the recursion tail-recursive with an accumulator parameter (TS specifically optimizes conditional types whose recursive call is the entire branch result), chunk the work (match multiple characters per step), simplify the type, or accept a coarser type and validate at runtime. Recognizing this proves you've actually pushed the type system, not just read about it.

Where do you still need runtime validation despite full strict-mode TypeScript, and why?

At every trust boundary: HTTP/GraphQL responses, JSON.parse, localStorage, URL/query params, postMessage, env vars, webhooks, user file uploads. Why: types are erased — an as User or a generic fetchJson<User>() is a promise the compiler can't keep, because the data arrives at runtime where no types exist. The modern pattern is schema-first: define a Zod/Valibot schema, parse at the boundary (fail fast with good errors), and derive the static type via z.infer so runtime and compile-time views can never drift. Inside the validated core, the type system's guarantees then actually hold.

Type-challenge drills

Implement each in a scratch file or the TS Playground before expanding the solution. These are the exact style of live type-puzzles TS-heavy interviews use. Verify with the provided test lines — they should type-check (or fail) as annotated.

Drill 1 — MyPick<T, K>

Reimplement Pick: select a subset of keys from T. MyPick<Todo, 'title' | 'done'> → { title: string; done: boolean }. Invalid keys must be a compile error.

Solution & explanation 
type MyPick<T, K extends keyof T> = {
  [P in K]: T[P];
};

// tests
interface Todo { title: string; done: boolean; created: Date }
type T1 = MyPick<Todo, 'title' | 'done'>;   // { title: string; done: boolean }
type T2 = MyPick<Todo, 'nope'>;              // ❌ error, as required

Three features in one line: the constraint K extends keyof T rejects invalid keys at the call site; the mapped type [P in K] iterates the supplied union (not all of keyof T); the indexed access T[P] looks up each original value type. Because the map iterates K directly, only requested keys appear.

Drill 2 — MyReturnType<T>

Reimplement ReturnType without using it. Must handle (x: number) => string and functions with any arg list.

Solution & explanation 
type MyReturnType<T> =
  T extends (...args: any[]) => infer R ? R : never;

// tests
type R1 = MyReturnType<() => number>;                 // number
type R2 = MyReturnType<(s: string) => Promise<boolean>>; // Promise<boolean>
type R3 = MyReturnType<string>;                        // never (not a function)

infer R in the return position asks the compiler to solve for the return type during the match. The params must be ...args: any[]: parameters are contravariant, so unknown[] would fail to match functions with specific parameter types (a (s: string) => … is not assignable to (...args: unknown[]) => …) — any deliberately turns that check off. Volunteering that params-vs-any subtlety is exactly the depth interviewers hope for.

Drill 3 — DeepReadonly<T>

Make every property readonly recursively, leaving functions untouched. Readonly<T> is shallow — nested objects stay mutable.

Solution & explanation 
type DeepReadonly<T> = {
  readonly [K in keyof T]:
    T[K] extends (...args: any[]) => any
      ? T[K]                       // leave functions alone
      : T[K] extends object
        ? DeepReadonly<T[K]>       // recurse into objects (incl. arrays)
        : T[K];                    // primitives as-is
};

// tests
type D = DeepReadonly<{ a: { b: number[] }; f: () => void }>;
declare const d: D;
d.a = { b: [] };      // ❌ readonly
d.a.b[0] = 1;         // ❌ readonly (homomorphic map over the array)
d.f();                // ✅ still callable

Key ideas: the readonly modifier applies at each level while recursion carries it downward; functions must be excluded before the object test (functions are objects — mapping over one would destroy its call signature); and because { [K in keyof T] … } is homomorphic, arrays/tuples map elementwise into readonly arrays instead of degrading into keyed objects. Nuance to mention: naive recursion doesn't special-case Map/Set/Date — production versions add branches for them.

Drill 4 — TupleToUnion<T>

TupleToUnion<['a', 'b', 1]> → 'a' | 'b' | 1. Should also work for as const (readonly) tuples.

Solution & explanation 
type TupleToUnion<T extends readonly unknown[]> = T[number];

// tests
type U1 = TupleToUnion<['a', 'b', 1]>;              // 'a' | 'b' | 1
const arr = ['x', 'y'] as const;
type U2 = TupleToUnion<typeof arr>;               // 'x' | 'y'

One-liner via indexed access with an index-type: T[number] means "the type you get indexing T with any number" — for a tuple, the union of all element types. The readonly unknown[] constraint is what accepts as const tuples (readonly arrays are the wider set; mutable arrays are assignable to them, not vice versa). This is the same idiom as the earlier typeof ROLES[number] single-source-of-truth pattern. An equivalent infer-based recursive solution exists, but reaching for T[number] first shows fluency.

Drill 5 — MyAwaited<T>

Unwrap a promise type — including nested promises and thenables: MyAwaited<Promise<Promise<string>>> → string.

Solution & explanation 
type MyAwaited<T> =
  T extends PromiseLike<infer U>
    ? MyAwaited<U>        // recurse — U might itself be a promise
    : T;

// tests
type A1 = MyAwaited<Promise<string>>;                    // string
type A2 = MyAwaited<Promise<Promise<number>>>;           // number
type A3 = MyAwaited<boolean>;                            // boolean (passthrough)
type A4 = MyAwaited<{ then(cb: (v: 42) => void): void }>; // 42 (thenable!)

Two deliberate choices: PromiseLike, not Promise — it only requires then, so custom thenables (which await genuinely unwraps at runtime) are handled, matching real semantics; and recursion, because await flattens nested promises. The base case is the false branch returning T unchanged. The real built-in Awaited adds edge handling for null | undefined and malformed then, but this captures the mechanism interviewers want.

Drill 6 — First<T> and Last<T>

Get the first and last element types of a tuple. First<[]> must be never.

Solution & explanation 
type First<T extends readonly unknown[]> =
  T extends readonly [infer F, ...infer _Rest] ? F : never;

type Last<T extends readonly unknown[]> =
  T extends readonly [...infer _Init, infer L] ? L : never;

// tests
type F1 = First<['a', 'b', 'c']>;  // 'a'
type F2 = First<[]>;               // never — empty tuple fails the pattern
type L1 = Last<[1, 2, 3]>;         // 3

This is variadic tuple pattern matching: [infer F, ...infer Rest] requires at least one element, so the empty tuple falls to the false branch — the never comes for free rather than needing a special case. Note infer can sit after a rest too ([...Init, infer L]), which is impossible in runtime JS destructuring — the type language is more expressive here. Alternative for First: T extends [] ? never : T[0]; the naive T[0] alone would be undefined-ish/wrong for the empty tuple.

Drill 7 — CamelizeKeys<T> (snake_case → camelCase keys)

The API returns { user_id: number; created_at: string }; your frontend wants { userId: number; createdAt: string }. Write the type that renames the keys (one level deep is fine).

Solution & explanation 
type CamelCase<S extends string> =
  S extends `${infer Head}_${infer Tail}`
    ? `${Head}${Capitalize<CamelCase<Tail>>}`
    : S;

type CamelizeKeys<T> = {
  [K in keyof T as CamelCase<K & string>]: T[K];
};

// tests
type Api = { user_id: number; created_at: string; is_admin_user: boolean };
type Client = CamelizeKeys<Api>;
// { userId: number; createdAt: string; isAdminUser: boolean }

Trace CamelCase<'is_admin_user'>: matches with Head='is', Tail='admin_user' (first infer matches minimally, splitting at the first underscore) → `is${Capitalize<CamelCase<'admin_user'>>}` → inner call yields 'adminUser' → Capitalize → 'AdminUser' → 'isAdminUser'. The mapped type then uses key remapping (as) to apply it per key; K & string filters out symbol/number keys, which template literals can't accept. Follow-up they may ask: make it deep — recurse on T[K] extends object like DeepReadonly.

Drill 8 — ExtractParams<Path> (the typed-router finale)

From '/users/:id/posts/:postId' produce a params object type { id: string; postId: string }. Stretch goal: make :id(number)-style segments unnecessary but support a trailing catch-all *.

Solution & explanation 
type ExtractParams<Path extends string> =
  Path extends `${infer _Pre}:${infer Param}/${infer Rest}`
    ? { [K in Param]: string } & ExtractParams<Rest>
    : Path extends `${infer _Pre}:${infer Param}`
      ? { [K in Param]: string }
      : {};

// prettify the intersection for readable hovers (a pro move):
type Prettify<T> = { [K in keyof T]: T[K] } & {};
type RouteParams<P extends string> = Prettify<ExtractParams<P>>;

// tests
type P1 = RouteParams<'/users/:id/posts/:postId'>; // { id: string; postId: string }
type P2 = RouteParams<'/about'>;                    // {}
type P3 = RouteParams<'/files/:name'>;              // { name: string }

Mechanics: the first pattern captures a param that is followed by more path — earlier infer holes match as little as possible up to the next literal character, so Param stops at the /; recursion continues on Rest. The second pattern catches a trailing param (no / after it). Base case: no : at all → {} (the identity for intersection). { [K in Param]: string } is a mapped type over the single-literal union — a tidy way to build a one-key object from an inferred string. Prettify flattens {id} & {postId} into one object in hovers/errors — knowing that trick marks you as someone who's shipped type-level code others had to read. This is precisely how TanStack Router and tRPC-style libraries type their APIs.
