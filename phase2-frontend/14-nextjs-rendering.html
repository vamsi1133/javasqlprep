14 — Next.js & Rendering Strategies

CSR vs SSR vs SSG vs ISR vs streaming, hydration, the App Router's four caches — and how to argue the trade-offs like an architect.

Why interviewers ask this

"How would you render this page, and why?" is one of the most reliable senior-frontend interview questions in existence. It is a perfect discriminator: a mid-level engineer answers with a framework name ("I'd use Next.js"), while a senior engineer answers with a decision process — who the users are, whether SEO matters, how fresh the data must be, whether the page is personalized, what infra the team can operate, and what each strategy costs in first paint, interactivity, and server dollars. The rendering strategy question has no single right answer, which is exactly why interviewers love it: it exposes how you think.

Next.js matters here for a pragmatic reason: it is the reference implementation interviewers assume. When a panel says "SSR", "ISR", "server components", or "streaming", they are almost always picturing Next.js semantics. Even if you'd choose Remix or a Vite SPA in real life, you need to speak fluent Next — especially the App Router's caching model, which is notoriously confusing and therefore a favorite place to probe depth. FAANG interviews use this topic in system-design rounds ("design a news site / e-commerce PDP / dashboard"); startup interviews use it to check you can make architecture calls without a platform team behind you.

Senior signal: the single strongest move in this topic is per-page thinking. Rendering strategy is not an app-level decision; it's a page-level (even component-level, with streaming and PPR) decision. Saying "the marketing pages are SSG on a CDN, the product pages are ISR, the checkout is SSR because it's personalized, and the account dashboard is effectively CSR behind auth" in one breath is worth more than an hour of framework trivia.

1. The rendering spectrum Tier 1 — must know cold

Before any Next.js API, own the framework-agnostic mental model. Every rendering strategy answers one question: where and when is the HTML for this page produced? At build time, at request time on a server, or at runtime in the browser? Everything else — SEO, first paint, infra cost, data freshness — falls out of that answer.

1.1 CSR — Client-Side Rendering

The server (or CDN) sends a nearly empty HTML shell plus a JavaScript bundle. The browser downloads, parses, and executes the JS; React then fetches data and renders the UI into an empty <div id="root">. This is the classic Create React App / Vite SPA model. 

CSR timeline (user on a mid-range phone, 4G):

t0 ──── request ────► CDN returns ~1KB shell        (TTFB: great, page: blank)
t1 ◄─── shell HTML ── <div id="root"></div>
t2 ──── fetch JS ───► download 300KB bundle          (still blank / spinner)
t3      parse + execute JS                           (still blank)
t4 ──── fetch data ─► API round-trip                 (skeleton at best)
t5      React renders                                FCP + LCP happen HERE, late
t6      interactive                                  TTI ≈ FCP (once painted, it works)

|--blank--|--blank--|--spinner--|== content + interactive ==|
  • First paint story: worst of all strategies — nothing meaningful paints until JS downloads, executes, and data arrives. It's a waterfall: HTML → JS → data → render.
  • TTI story: ironically decent relative to FCP — the moment content appears, it's interactive, because the client rendered it. There's no hydration gap. The problem is everything happens late.
  • SEO: historically poor. Googlebot does execute JS (second-wave indexing), but it's slower, budgeted, and other crawlers (social link unfurlers, some search engines) don't. If organic search or link previews matter, CSR is a liability.
  • When CSR is genuinely fine: authenticated dashboards, admin panels, internal tools, editors — pages behind a login where SEO is irrelevant, users are on repeat visits with warm caches, and app-like interactivity dominates. Do not apologize for CSR in these cases; it's the simplest architecture (static hosting, no server to run).

1.2 SSR — Server-Side Rendering (per request)

The server renders the React tree to an HTML string on every request, sends it down, then the client downloads JS and hydrates (attaches interactivity — section 2). The user sees content fast but can't interact until hydration completes. 

SSR timeline:

t0 ──── request ────► server: fetch data + renderToString   (TTFB: SLOWER —
t1 ◄─── full HTML ──  content visible! FCP/LCP early          you pay data +
t2 ──── fetch JS ───► bundle downloads in background           render per req)
t3      hydration runs                                        TTI happens here
        ── the gap between t1 and t3 is "visible but dead" ──

|--wait TTFB--|===== content visible (dead) =====|== interactive ==|
  • Fast FCP, delayed TTI: the inverse of CSR. Content is visible early; interactivity lags behind JS + hydration. The FCP→TTI gap is the "uncanny valley" (section 2.3).
  • TTFB sensitivity: the whole page waits on the slowest data fetch on the server. One slow downstream API = slow TTFB = blank screen. This is the problem streaming solves.
  • Server cost: every request costs CPU (rendering React on the server is not free) and holds a connection. You need servers, autoscaling, monitoring — real infra, unlike SSG/CSR on a CDN.
  • Personalization: the killer feature. Because HTML is produced per request, it can read cookies, auth, geo, A/B buckets — fully personalized HTML with fast first paint. SSG can't do this; CSR does it late.

1.3 SSG — Static Site Generation (build time)

HTML is rendered once at build time, output as static files, and served from a CDN. Requests never touch your origin. 

SSG timeline:

BUILD TIME (once):   fetch data ─► render ─► write .html files ─► deploy to CDN

REQUEST TIME (every user):
t0 ──── request ────► CDN edge (nearest PoP)      TTFB: near-instant
t1 ◄─── full HTML ──  content visible immediately  FCP: best possible
t2      JS + hydration                             TTI: after hydration

|== content visible ==|== interactive ==|          (no origin involved)
  • Best case for everything except freshness: fastest FCP, effectively zero marginal infra cost, perfect SEO, trivially scalable to any traffic spike (it's just CDN cache hits).
  • The stale-data problem: the page shows the world as it was at the last build. Price changed? Article corrected? You must rebuild and redeploy. This is what ISR fixes.
  • Build-time scaling: 50 marketing pages: fine. 2 million product pages: your build takes hours and rebuilds on every content change. Rule of thumb — SSG all pages when the page count is small and change frequency is low; beyond that you need ISR (build a subset, generate the rest on demand) or SSR.
  • No personalization in the HTML: everyone gets identical bytes. Personalization must layer on client-side (or via edge middleware rewrites).

1.4 ISR — Incremental Static Regeneration

SSG plus a revalidation contract: static pages that regenerate in the background. Semantically it's stale-while-revalidate applied to HTML, which is exactly the phrase to use in an interview. 

ISR with revalidate: 60

t=0     build generates page v1
t=30    request ─► serve v1 (fresh, cache hit)
t=90    request ─► serve v1 (STALE — but served instantly)
                   └─► background: regenerate ─► v2 stored
t=95    request ─► serve v2

Key semantics:
  • The user who triggers regeneration STILL GETS THE STALE PAGE. Nobody waits.
  • revalidate is a MINIMUM staleness window, not a cron — regeneration happens
    lazily, on the first request after expiry.
  • First request to a never-built path (with fallback) can block-render on demand,
    then the result is cached like any other static page.
  • Time-based revalidation: "this page may be up to N seconds stale." Great default for content/commerce catalog pages.
  • On-demand revalidation: your CMS webhook calls revalidatePath('/blog/slug') or revalidateTag('products') — pages are effectively fresh and static. This is the modern best-of-both answer for content sites: "static until the data actually changes."
  • Cost model: CDN economics for reads, occasional server render for regeneration. Handles the "2M product pages" problem: prebuild the top 10k, generate the long tail on first request.

1.5 Streaming SSR

Instead of waiting for all server data before sending any HTML, the server flushes the shell immediately and streams slow sections as they resolve, using Suspense boundaries as the seams. HTTP chunked transfer; React inlines small scripts that swap fallbacks for real content as chunks arrive. 

Classic SSR:        [.... wait for slowest fetch ....][ send everything ]
                    TTFB gated by slowest data

Streaming SSR:      [ send shell + fast parts ]───►  user sees layout now
                        [ stream <Suspense> hole #1 when ready ]───►
                            [ stream hole #2 when ready ]───►
                    TTFB gated by NOTHING slow; slow parts show fallbacks

<Layout>                          ← flushed immediately
  <Header />                      ← flushed immediately
  <Suspense fallback={Skeleton}>
    <SlowRecommendations />       ← streamed at t+800ms
  </Suspense>
</Layout>
Framing for interviews: streaming doesn't make the slow data faster — it decouples TTFB and FCP from the slowest fetch, converting a blank-screen wait into a progressive reveal. It improves perceived performance and real FCP/LCP for the fast parts.

1.7 Per-page strategy thinking

Real apps mix strategies. A typical e-commerce architecture, page by page:

PageStrategyWhy

Pitfall: answering "which rendering strategy?" for the whole app. The moment you say "it depends on the page — let me go page by page", you've differentiated yourself from most candidates.

2. Hydration, deeply Tier 1 — must know cold

2.1 What hydration actually is

Server rendering gives the browser inert HTML — pixels, no behavior. Hydration is React running on the client, rendering the same component tree in memory, and instead of creating new DOM nodes, adopting the existing server-rendered DOM: it walks the HTML, matches each element to its fiber, wires up React's synthetic event system, and initializes state/effects. After hydration, the page behaves exactly as if it had been client-rendered. 

Server:   ReactDOMServer.renderTo(Pipeable)Stream(<App />)  ─►  HTML string
Client:   hydrateRoot(document.getElementById('root'), <App />)
             │
             ├─ re-render <App /> in memory (build the fiber tree)
             ├─ MATCH each fiber to an existing DOM node (adopt, don't create)
             ├─ attach event listeners (React 17+: delegated at the root)
             └─ run effects, initialize state
Key implication: hydration re-executes all your component code on the client anyway. Classic SSR ships the rendering work twice — once as HTML, once as JS that re-derives the same tree. This double-payment is the motivation for Server Components (ship no JS for server-only components) and islands architecture.

2.2 Hydration mismatches — the classic bug class

React hydration assumes server HTML and the client's first render produce identical output. If they differ, React warns ("Hydration failed because the initial UI does not match…"), and in React 18 it discards the server HTML for the affected subtree and client-renders from scratch — you lose the SSR benefit and can get visible flicker.

The canonical causes (know all four):

CauseExampleFix
Time / locale / randomness new Date().toLocaleString(), Math.random(), crypto.randomUUID() in render — server value ≠ client value Compute once on the server and pass as prop; or render in useEffect; suppressHydrationWarning for a lone timestamp <time>
Environment branching in render typeof window !== 'undefined' ? <Widget/> : null — server renders null, client renders widget on first pass Gate on mounted state via useEffect, or next/dynamic with ssr: false
Client-only state read during render Reading localStorage/matchMedia for theme/viewport in the initial render Render a neutral default on both sides; correct after mount (or use a cookie so the server knows too)
Third parties mutating HTML Browser extensions (Grammarly, password managers) injecting attributes before hydration; also invalid HTML nesting (<p><div>) that browsers "fix" Not your bug for extensions — recognize it in triage; fix invalid nesting since the parser-corrected DOM won't match React's tree 
// The standard "client-only after mount" pattern:
function LocalTime({ iso }: { iso: string }) {
  const [mounted, setMounted] = useState(false);
  useEffect(() => setMounted(true), []);
  // Server + first client render agree (both render the ISO string),
  // then we upgrade to the locale-formatted version after mount.
  return <time dateTime={iso}>{mounted ? new Date(iso).toLocaleString() : iso}</time>;
}

// Legitimate suppressHydrationWarning — a single text node you EXPECT to differ:
<time dateTime={iso} suppressHydrationWarning>
  {new Date(iso).toLocaleString()}
</time>

// Skip SSR entirely for a genuinely client-only widget (maps, charts, editors):
const Map = dynamic(() => import('./Map'), { ssr: false });
Pitfall: suppressHydrationWarning is one element deep and only silences text/attribute mismatches — it is not a fix for structural mismatches, and sprinkling it everywhere hides real bugs. Use it for the timestamp case; use effect-gating or ssr: false for structural differences.

2.3 The uncanny valley: visible but dead

Between FCP (HTML painted) and hydration completing, the page looks interactive but isn't: buttons don't respond, inputs may drop focus, clicks vanish. On low-end devices with big bundles this gap can be seconds. Users rage-click. This is the core UX criticism of naive SSR and the reason "just SSR everything" is not a senior answer. 

FCP                                TTI
 │◄──────── uncanny valley ────────►│
 │  page painted, JS still loading, │
 │  handlers not attached — clicks  │
 │  are silently lost               │

2.4 Selective hydration (React 18) — the fix

Pre-18, hydration was all-or-nothing: one synchronous pass over the whole tree; nothing interactive until everything was. React 18 + Suspense changes this:

  • Streaming boundaries: each <Suspense> boundary is an independent unit — its HTML can arrive late (streamed) and hydrate independently, before or after siblings.
  • Hydration doesn't wait for all JS/data: boundaries hydrate as their code and content become available.
  • Interaction-driven prioritization: if the user clicks inside a not-yet-hydrated boundary, React synchronously prioritizes hydrating that boundary first, and replays the event. The comment section the user is touching hydrates before the footer.
  • Interruptible: hydration happens in concurrent chunks, so it doesn't block the main thread in one long task.
Without Suspense:   [========= hydrate entire app (one long task) =========] TTI
With boundaries:    [hydrate nav] [hydrate hero]      user clicks comments!
                                        └─► [hydrate comments NOW] ... [sidebar]
Senior soundbite: "Suspense boundaries are simultaneously streaming seams on the server and hydration units on the client. React 18 turned hydration from a monolithic blocking pass into a prioritized, interruptible, per-boundary schedule — which is what makes SSR's uncanny valley tolerable at scale."

2.5 Islands & resumability (literacy level)

  • Islands (Astro): the page is static HTML by default; only explicitly-marked interactive components ("islands") ship JS and hydrate — <Counter client:visible /> hydrates when scrolled into view. For content-heavy sites this can mean 90%+ less JS than a React SSR app. React Server Components are philosophically similar (server-only components ship zero JS) but inverted defaults: Astro is static-first, RSC is server-rendered-React-first.
  • Resumability (Qwik): skips hydration entirely — the server serializes app state and listener wiring into the HTML, and the client resumes execution lazily on interaction, downloading only the handler code needed. One-liner for interviews: "Qwik's bet is that hydration is unnecessary work; you resume where the server left off instead of replaying the render."

3. The Next.js App Router model Tier 1 — must know cold

3.1 File conventions

app/
├── layout.tsx          // root layout (required): <html><body> — wraps everything
├── page.tsx            // route UI for "/"
├── loading.tsx         // automatic <Suspense> fallback for this segment
├── error.tsx           // error boundary ('use client', receives error + reset)
├── not-found.tsx       // 404 UI for notFound()
├── template.tsx        // like layout but REMOUNTS on every navigation
├── (marketing)/        // route group — organizes files, no URL segment
│   ├── layout.tsx      //   separate layout for marketing pages
│   └── about/page.tsx  //   → /about
├── blog/
│   └── [slug]/page.tsx // dynamic segment → params.slug
├── shop/[...parts]/    // catch-all;  [[...parts]] = optional catch-all
└── api/things/route.ts // route handler (GET/POST/... exports)
FileThe interview-relevant behavior
layout.tsxPersists across navigations — does not re-render or lose state when children change. Nested layouts compose (root → segment → leaf). Layouts do not receive searchParams (they don't re-render on navigation, so the value would go stale).
template.tsxSame nesting position as layout, but a fresh instance per navigation — state resets, effects re-run. Use for enter animations or per-page reset semantics.
loading.tsxSugar for wrapping the segment's page in <Suspense fallback={<Loading/>}>. This is what enables instant navigation + streaming with zero manual wiring.
error.tsxBecomes a React error boundary around the segment. Must be a client component. Receives { error, reset } — reset() re-renders the segment to retry. Does not catch errors in the layout above it (put global-error.tsx at the root for that).
Route groups (name)Group segments under different layouts without affecting URLs — e.g. (marketing) vs (app) with different shells.

3.2 Server Components by default

Every component in app/ is a React Server Component (RSC) unless a module (or an importer above it) declares 'use client'. Recap from tutorial 10, condensed to what this tutorial needs:

  • Server components run only on the server (build time or request time). Their code, their dependencies, and their data-fetching secrets never ship to the browser. They can be async and await data directly.
  • They cannot use state, effects, event handlers, or browser APIs. Anything interactive needs 'use client'.
  • 'use client' marks a boundary, not a single component: everything it imports becomes client-bundle code. Server components can be passed through a client component as children/props (the "slot" pattern) without becoming client code.
  • Placement heuristic: push client boundaries to the leaves. A page is a server component; the like button, the search input, the dropdown are small client islands. Marking a page-level component 'use client' drags the whole subtree into the bundle — the most common RSC design smell.
// app/products/[id]/page.tsx — server component (default)
import { AddToCart } from './add-to-cart';        // client island (leaf)

export default async function ProductPage(
  { params }: { params: Promise<{ id: string }> }  // Next 15: params is async
) {
  const { id } = await params;
  const product = await db.products.find(id);       // direct data access, no API hop
  return (
    <article>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <AddToCart id={product.id} />                 {/* only THIS ships JS */}
    </article>
  );
}
Important mental correction interviewers listen for: client components are still server-rendered to HTML on the initial request (then hydrated). 'use client' means "ships JS and hydrates," not "renders only in the browser."

3.3 Nested layouts & the data pattern

Each layout and page fetches its own data — there's no getServerSideProps-style single provider per route. Layout and page fetches run in parallel (they're siblings in the tree), and request memoization (section 4.1) means the layout and the page can both call getUser() without a duplicate network hit. Don't thread data through props across layout levels; fetch where you need it and let the cache dedupe.

3.4 Parallel & intercepting routes (awareness level)

  • Parallel routes: named slots — app/@feed/page.tsx and app/@modal/page.tsx both render into the parent layout as props ({ feed, modal }). Each slot can have its own loading/error states and navigate independently. default.tsx supplies a slot's fallback when it has no match.
  • Intercepting routes: (.)photo/[id] intercepts navigation to /photo/123 from within the current layout and renders it in place (e.g. into the @modal slot) instead of a full page change.
  • The pattern to cite — modal over feed (Instagram-style): click a photo in the feed → intercepted route renders it as a modal over the feed; the URL is /photo/123, so refresh or direct-link renders the full photo page. Shareable modal URLs without duplicating the photo UI in two places.

3.5 Route handlers

// app/api/products/route.ts
import { NextResponse } from 'next/server';

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const items = await db.products.list({ q: searchParams.get('q') });
  return NextResponse.json(items);
}

export async function POST(request: Request) {
  const body = await request.json();
  const created = await db.products.create(body);
  return NextResponse.json(created, { status: 201 });
}

Web-standard Request/Response, HTTP-verb exports, per-route runtime = 'edge' | 'nodejs'. Use them for webhooks, public APIs, and non-React clients. For your own app's mutations, prefer server actions; and note that server components should call the database/service directly rather than fetching their own route handlers (a pointless self-HTTP hop is a classic code-review catch).

3.6 Middleware

// middleware.ts (project root) — runs BEFORE the route, on the edge runtime
import { NextResponse, type NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const session = request.cookies.get('session');
  if (!session && request.nextUrl.pathname.startsWith('/app')) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
  return NextResponse.next();
}
export const config = { matcher: ['/app/:path*'] };
  • Use for: auth gating/redirects, locale routing, A/B bucket assignment via cookie + rewrite, header manipulation — cheap decisions close to the user, before rendering.
  • Limitations (say these): edge runtime — no Node APIs, no direct DB drivers (TCP), tight CPU/size budgets, and it runs on every matched request, so keep it fast. Check a session cookie's validity here; don't run your full ORM-backed authorization. Treat middleware auth as optimistic UX-level gating, and re-verify in the data layer (there was a real middleware-bypass CVE in 2025 — defense in depth is the lesson).

3.7 Server actions

// app/posts/actions.ts
'use server';
import { revalidatePath, revalidateTag } from 'next/cache';
import { redirect } from 'next/navigation';

export async function createPost(formData: FormData) {
  const user = await requireUser();                    // ALWAYS re-auth inside the action
  const title = String(formData.get('title'));
  const post = await db.posts.create({ title, userId: user.id });
  revalidateTag('posts');                              // purge cached data for this tag
  redirect(`/posts/${post.id}`);
}

// In a server component — progressively enhanced (works before hydration):
<form action={createPost}>
  <input name="title" />
  <button type="submit">Create</button>
</form>
  • A server action compiles to an RPC endpoint the framework wires up; the client posts to it — no hand-written API route, end-to-end types.
  • useActionState / useFormStatus on the client give pending state and returned validation errors.
  • revalidatePath / revalidateTag after a mutation is the write-path half of the caching story: mutate, then purge exactly the cached data that changed, and Next re-renders affected routes with fresh data on the same round trip.
  • Security framing (senior): actions are public HTTP endpoints regardless of where the button lives. Validate input and authorize inside every action, exactly as you would an API route.

4. Data fetching & the four caches Tier 1 — the differentiator

This is the most confusing part of modern Next.js and therefore the highest-leverage thing to be crisp about. There are four distinct caches. Candidates who can name all four, say where each lives, and explain what invalidates each one stand out immediately. 

                 ┌────────────────────────────────────────────────────┐
   ONE RENDER    │ 1. REQUEST MEMOIZATION  (React, per render pass)   │
   PASS          │    same fetch(url) called twice → one network hit  │
                 └────────────────────────────────────────────────────┘
                 ┌────────────────────────────────────────────────────┐
   SERVER,       │ 2. DATA CACHE  (Next, persistent across requests   │
   PERSISTENT    │    and deploy-spanning)                            │
                 │    fetch options: force-cache / no-store /         │
                 │    next: { revalidate, tags }                      │
                 └────────────────────────────────────────────────────┘
                 ┌────────────────────────────────────────────────────┐
   SERVER,       │ 3. FULL ROUTE CACHE  (rendered HTML + RSC payload  │
   PER ROUTE     │    for STATICALLY rendered routes)                 │
                 │    built at build time / ISR; skipped for dynamic  │
                 └────────────────────────────────────────────────────┘
                 ┌────────────────────────────────────────────────────┐
   BROWSER       │ 4. ROUTER CACHE  (client-side, RSC payload per     │
                 │    visited segment — instant back/forward)         │
                 └────────────────────────────────────────────────────┘

4.1 Request memoization

Within a single render pass, identical fetch(url, options) calls are deduped — the layout, the page, and three components can each call getUser() and only one request goes out. This is React-level, lasts only for the render, and applies to GET fetches. For non-fetch data access (direct DB calls), wrap in React's cache() to get the same per-render dedup: 

import { cache } from 'react';
export const getUser = cache(async (id: string) => db.user.find(id));

Why it matters architecturally: it removes the pressure to hoist data fetching to a common ancestor and prop-drill it down. Components can be self-sufficient about their data without duplicate queries.

4.2 The Data Cache

A server-side, persistent cache of fetch responses, keyed by URL + options, surviving across requests and users. Controlled per fetch: 

// Cached indefinitely until invalidated:
fetch(url, { cache: 'force-cache' })

// Never cached — fresh on every request (also opts route into dynamic rendering):
fetch(url, { cache: 'no-store' })

// Time-based revalidation — stale-while-revalidate with a 1h window:
fetch(url, { next: { revalidate: 3600 } })

// Tag-based — the powerful one:
fetch(url, { next: { tags: ['products', `product-${id}`] } })
// ...later, in a server action or webhook route handler:
revalidateTag('products')        // purges every fetch tagged 'products'
revalidatePath('/products')      // purges data + route cache for a path
Next 15 defaults change (know this story): in Next 13/14 App Router, fetch was cached by default (force-cache). This surprised nearly everyone — devs shipped stale data without realizing they'd opted into caching — and it became the loudest community complaint about the App Router. Next 15 flipped the default to uncached (no-store semantics): you now opt in to caching explicitly ('force-cache', revalidate, or route-level export const fetchCache). Similarly, GET route handlers and the client Router Cache for pages became uncached by default. Telling this story — aggressive-by-default caching, backlash, explicit-by-default reversal — is a strong currency signal, and it means "which version?" is a legitimate clarifying question in interviews.

4.3 The Full Route Cache & static vs dynamic rendering

At build time, Next renders every route it can and stores the HTML + RSC payload — that stored result is the Full Route Cache. Whether a route is statically rendered (cached) or dynamically rendered (per request) is determined by what the route uses:

Route stays STATIC when…Route becomes DYNAMIC when it uses…
All data is cached (or there is none); no request-time APIs used. Rendered once at build (or per ISR window), served like SSG. cookies(), headers(), or connection() — request-specific input
searchParams accessed in a page
any fetch(..., { cache: 'no-store' })
explicit export const dynamic = 'force-dynamic' 
// Route segment config — the explicit knobs:
export const dynamic = 'force-static' | 'force-dynamic' | 'auto';
export const revalidate = 60;            // ISR window for this whole route
export const runtime = 'nodejs' | 'edge';
The sentence that shows you get it: "In the App Router, SSG vs SSR isn't an API you pick — it's an outcome the compiler infers. A route is static unless something in it demands request-time information; one cookies() call or one no-store fetch flips the whole route to dynamic rendering." Then add the failure mode: a shared header reading cookies() for auth silently makes every route dynamic — a very common accidental de-optimization.

4.4 Partial Prerendering (PPR)

PPR resolves the static-vs-dynamic all-or-nothing: serve a static shell from the CDN instantly, with dynamic holes (Suspense boundaries around request-time content) streamed into it from the server — one HTTP response, both worlds. 

/product/123 with PPR:

  ┌─ STATIC SHELL (prerendered, CDN) ─────────────────┐
  │  nav, product name, description, images, footer   │
  │  ┌─ dynamic hole (Suspense) ──────────────┐       │
  │  │ <PersonalizedPrice/>  ← reads cookies  │◄─ streamed per request
  │  └────────────────────────────────────────┘       │
  └───────────────────────────────────────────────────┘

export const experimental_ppr = true;   // opt-in while experimental

export default function Page() {
  return (
    <>
      <StaticProductInfo />                       {/* prerendered shell */}
      <Suspense fallback={<PriceSkeleton />}>

Status framing: experimental through Next 14/15 (flag-gated, canary), positioned as the intended default rendering model going forward. Cite it as "where the framework is heading — the static/dynamic decision moves from the route level to the Suspense-boundary level," and flag that you'd check current stability before adopting.

4.5 The Router Cache (client side)

In the browser, Next caches the RSC payloads of visited (and prefetched) route segments in memory. Back/forward navigation restores instantly from it; <Link> prefetching populates it before the click. It's invalidated by router.refresh(), by revalidatePath/Tag from a server action, and by cookie-setting in actions. Historically its aggressive staleness windows caused "why is my data stale after navigating back?" bugs — Next 15 set page segments' default staleTime to 0 (layouts still reused). If asked "user updates their name but the old name shows when they navigate back" — this cache is the answer.

// app/blog/[slug]/page.tsx — SSG for a dynamic route

  const posts = await getTopPosts(1000);              // prebuild the head of the tail

5. Streaming in Next.js Tier 2 — strong to know

5.1 loading.tsx and nested Suspense granularity

loading.tsx gives you a whole-segment Suspense boundary for free — navigation is instant (shell + skeleton paint immediately) while the page's server components fetch. But one segment-level skeleton is coarse: the whole page pops in at once, gated by its slowest fetch. Nested <Suspense> inside the page gives per-section granularity: 

// app/dashboard/page.tsx
export default function Dashboard() {
  return (
    <>
      <Header />                                     {/* fast — in initial flush */}
      <Suspense fallback={<RevenueSkeleton />}>
        <Revenue />                                  {/* awaits warehouse query, ~2s */}
      </Suspense>
      <Suspense fallback={<ActivitySkeleton />}>
        <ActivityFeed />                             {/* awaits feed service, ~400ms */}
      </Suspense>
    </>
  );
}
t=0ms    flush: Header + both skeletons     ← user sees structured page
t=400ms  stream: ActivityFeed replaces its skeleton
t=2000ms stream: Revenue replaces its skeleton
         (vs non-streaming: blank/loading until t=2000, then everything)
Design judgment to voice: boundary placement is a product decision, not just a technical one. Too many boundaries = popcorn UI and layout shift; too few = slowest-fetch gating. Put boundaries around independently-useful, independently-slow regions, and give fallbacks correct dimensions to avoid CLS.

5.2 The promise-as-prop / use() pattern

Sometimes the client component needs the data (e.g., a chart with interactions) but you don't want to await it on the server (blocking the stream) or fetch it client-side (waterfall). Answer: start the promise on the server, don't await it, pass it down, unwrap with use() in the client under Suspense. 

// Server component — kick off the fetch, do NOT await:
export default function Page() {
  const statsPromise = getStats();               // starts immediately, server-side
  return (
    <Suspense fallback={<ChartSkeleton />}>
      <StatsChart statsPromise={statsPromise} />
    </Suspense>
  );
}

// Client component:
'use client';
import { use } from 'react';
export function StatsChart({ statsPromise }: { statsPromise: Promise<Stats> }) {
  const stats = use(statsPromise);               // suspends until resolved, streamed in
  return <InteractiveChart data={stats} />;
}

The fetch starts at server render time (no client waterfall), the shell streams immediately (no server blocking), and the resolved value streams across the RSC wire into the client component.

5.3 TTFB vs perceived performance — the narrative

With buffered SSR, TTFB ≈ slowest fetch, and users stare at white. With streaming, TTFB is near-constant (the shell), and slowness is localized to the component that's actually slow, with honest skeletons in the meantime. Metrics version: streaming improves FCP and usually LCP (if the LCP element is in the shell), keeps TTFB flat as backend latency varies, and pairs with selective hydration so early content is interactive early. The senior caveat: it doesn't fix slow backends — if everything is slow, you've just replaced a blank page with a skeleton page, and the real fix is caching or backend work.

  • Both routers coexist in one app — pages/ and app/ side by side (App Router wins conflicts). The sane strategy is incremental: new features in app/, migrate high-value routes opportunistically, leave stable low-churn pages alone.

9. The decision conversation Tier 1 — rehearse this out loud

"Would you use Next.js for this, or a plain SPA?" is the rendering question's final form. Have a structured, honest answer ready — including Next's costs.

9.1 When Next.js earns its complexity

  • SEO / social sharing matters: content, commerce, marketing — anything acquired through search or shared links needs server-rendered HTML and metadata.
  • First-paint on the open web matters: anonymous users, cold caches, mid-range devices — server HTML beats a JS waterfall.
  • Content pages at scale: SSG/ISR + on-demand revalidation is a genuinely great content platform.
  • Server data proximity: RSC lets components read the DB directly — no API layer to design, version, and secure just for your own frontend; smaller bundles as a bonus.
  • Mixed rendering in one product: marketing + app + docs under one roof, each page picking its own strategy.

9.2 When a Vite SPA is the better call

  • Internal tools / dashboards behind auth: no SEO, repeat visitors with warm caches, app-like interactivity. A Vite SPA is radically simpler: static files on a CDN/S3, no server to run, patch, scale, or wake up for at 3am.
  • The modern SPA stack is excellent: TanStack Router (type-safe routing, search-param state) + TanStack Query (caching, revalidation, optimistic updates) covers most of what people actually wanted from a framework.
  • Deploy is a folder upload; local dev is instant; no RSC mental model, no serialization boundaries, no four-cache debugging.

9.3 Name Next's costs honestly (this is the senior move)

  • Server infra: you now operate a Node service (or pay a platform) — scaling, cold starts, observability, incidents.
  • Framework coupling: caching semantics, routing, and rendering are deeply Vercel-flavored; self-hosting is supported but the best-path gravity toward Vercel is real, and framework churn (Pages→App, caching default flips) is a tax on teams.
  • Caching complexity: four caches with different lifetimes and invalidation rules is a genuine source of production bugs ("why is this stale?" is the App Router's signature incident).
  • RSC learning curve: server/client boundaries, serializability, "why can't I use useState here" — a real onboarding cost for teams.

9.4 Remix / React Router v7 literacy

Interviewers raise Remix as the comparison. Its model: each route exports a loader (server data for GET) and an action (mutations for POST); forms post to actions and loaders automatically revalidate — the framework leans on web fundamentals (forms, HTTP semantics, browser caching via Cache-Control headers) instead of a bespoke cache layer. Nested routes load data in parallel. Remix merged into React Router v7 ("framework mode"), so the loader/action model now lives in React Router itself. Fair one-line comparison: "Remix/RRv7 has a simpler, more explicit data story built on HTTP caching; Next has RSC, a bigger ecosystem, and finer-grained server/client code splitting. Both are good; Remix is easier to reason about, Next is more capable at the bleeding edge."

9.5 When plain SSG (Astro) beats both

For content-dominant sites — docs, blogs, marketing — where interactivity is a search box and a mobile menu, Astro's islands model ships near-zero JS and wins Core Web Vitals almost by default. Running React SSR + hydration for pages with two interactive widgets is paying the full framework tax for none of the benefit. Rule of thumb to say: mostly content → Astro; mostly app with public/SEO surface → Next; entirely app behind auth → Vite SPA.

9.6 A 30-second rehearsable answer

"First I ask: does anyone hit these pages logged-out, via search or shared links? If yes, I need server-rendered HTML — Next.js, rendering per page: SSG or ISR for content, dynamic/streamed SSR for personalized pages, client islands for interactivity. If it's entirely behind auth — an internal tool, a dashboard — I'd default to a Vite SPA with TanStack Router and Query: static hosting, no server fleet, simpler mental model. And if it's almost pure content with a sprinkle of interactivity, Astro ships less JavaScript than either. Next is powerful but not free — you take on server infra, a four-layer cache model, and the RSC learning curve — so I spend that complexity only where server rendering actually buys something."

10. Deployment awareness Tier 3 — good to have

  • Vercel: zero-config; pages/functions map onto their serverless + edge network, ISR and the Data Cache are backed by managed infra, image optimization is built in (and metered). Fastest path; the trade is cost at scale and platform coupling.
  • Self-hosting: output: 'standalone' in next.config.js emits a minimal server + traced node_modules for a small Docker image; run node server.js behind your LB. Multi-instance gotcha: ISR/Data-Cache state is per-instance on disk by default — you need a shared cache handler (cacheHandler config, e.g. Redis-backed) or sticky revalidation behavior gets inconsistent across pods.
  • CDN headers recap: the semantics under ISR are the classic pair — Cache-Control: s-maxage=60, stale-while-revalidate=86400: shared caches serve for 60s, then serve stale while refreshing in the background. Next sets appropriate headers for static/ISR content; if you put your own CDN in front, make sure it respects them and that you have a purge story for on-demand revalidation.
  • Image optimization self-hosted: sharp on your Node server + an on-disk cache; budget CPU for traffic spikes of uncached sizes, or point images.loader at a dedicated image CDN.
  • Monorepo integration: Next apps as workspace packages with shared UI/config libs, remote build caching — covered in tutorial 16.

Interview Q&A

1. Walk me through CSR, SSR, SSG, and ISR. How do you choose?

They differ on where/when HTML is produced. CSR: browser builds the UI after downloading JS — worst first paint, weak SEO, but simplest infra; fine behind auth. SSR: HTML per request on a server — fast FCP, full personalization, live data; costs server compute and is TTFB-sensitive. SSG: HTML at build time on a CDN — best FCP, near-zero cost, but stale until rebuilt and impossible to personalize in the HTML. ISR: SSG plus stale-while-revalidate regeneration (time-based or on-demand) — CDN economics with bounded staleness. Choice is per page, driven by: SEO needed? personalized? how fresh must data be? how many pages? what infra can we operate? Then I'd go page-by-page through the product.

2. What exactly happens during hydration?

The client re-renders the same component tree in memory to build the fiber tree, but instead of creating DOM nodes it adopts the existing server-rendered DOM — matching each fiber to its element — then attaches React's event system and initializes state and effects. It assumes server HTML and the client's first render match exactly. Notably, it re-executes all component render code on the client, which is the "double payment" of classic SSR that Server Components and islands architectures attack.

3. What causes hydration mismatch errors and how do you fix them?

Anything that renders differently on server vs client's first pass: dates/locales/randomness computed in render; typeof window branches; reading localStorage/matchMedia during render; plus invalid HTML nesting the browser "corrects," and browser extensions mutating the DOM pre-hydration. Fixes: make the first client render match the server (compute values server-side and pass as props; render client-only content after a useEffect-set mounted flag); next/dynamic with ssr:false for genuinely client-only widgets; suppressHydrationWarning narrowly for expected text differences like timestamps. In React 18, a mismatch discards server HTML for that subtree and client-renders it — so you also lose the SSR benefit, not just get a warning.

4. What is selective hydration and what problem does it solve?

Pre-React-18, hydration was one synchronous all-or-nothing pass: no part of the page was interactive until all of it hydrated — the "visible but dead" uncanny valley. React 18 uses Suspense boundaries as independent hydration units: they stream in and hydrate independently, hydration is interruptible/concurrent, and if the user interacts with an unhydrated boundary, React prioritizes hydrating that boundary immediately and replays the event. Result: interactivity arrives progressively and follows user attention rather than waiting on the whole tree.

5. Explain the four caches in the Next.js App Router.

(1) Request memoization — React-level, per render pass: identical fetches dedupe so layout and page can both call getUser() for one network hit; React.cache() extends this to non-fetch calls. (2) Data Cache — server-side, persistent across requests: stores fetch responses, controlled by force-cache/no-store/next.revalidate/tags, invalidated by revalidateTag/Path. (3) Full Route Cache — the rendered HTML + RSC payload of statically rendered routes, built at build time or ISR; dynamic routes skip it. (4) Router Cache — in the browser, RSC payloads of visited/prefetched segments for instant back/forward, invalidated by router.refresh() and server-action revalidations. Different layers, different lifetimes, different invalidation — most "stale data" bugs are confusion about which layer is serving you.

6. What makes an App Router route static vs dynamic?

It's inferred, not declared. A route is statically rendered (prerendered into the Full Route Cache) unless it uses request-time information: cookies(), headers(), searchParams in a page, a no-store fetch, or an explicit dynamic = 'force-dynamic'. One such usage flips the entire route to per-request rendering — so a shared header component calling cookies() silently makes every page dynamic, a classic accidental de-optimization. Partial Prerendering is the emerging fix: static shell with dynamic Suspense-bounded holes, moving the decision to boundary granularity.

7. What changed about caching defaults in Next 15, and why?

Next 13/14 cached fetch by default (force-cache), plus GET route handlers and aggressive client Router Cache staleness. Developers constantly shipped unexpectedly stale data — it became the App Router's biggest complaint. Next 15 flipped to uncached-by-default: fetches are dynamic unless you opt in via 'force-cache' or revalidate/tags; GET handlers uncached; Router Cache page staleTime 0. The lesson I'd cite: implicit aggressive caching violated least-surprise, and the ecosystem forced explicit opt-in. It also means caching behavior is version-dependent — worth clarifying which version in any discussion.

8. How does ISR work under the hood? What are its exact semantics?

Stale-while-revalidate for HTML. A page rendered at build (or on first request) is served from cache; after the revalidate window expires, the next request still gets the stale page instantly while regeneration runs in the background; subsequent requests get the fresh version. So revalidate is a lazy minimum-staleness bound, not a scheduled rebuild — no user ever waits on regeneration (except the first-ever render of an unbuilt path with blocking fallback). On-demand revalidation (revalidatePath/Tag, typically from a CMS webhook) purges immediately, giving "static until the data actually changes."

9. Server components vs client components — where do you draw the boundary?

Default everything to server; push 'use client' to the smallest interactive leaves — buttons, inputs, dropdowns — because the directive marks a boundary: everything it imports joins the client bundle. Keep pages and layouts as server components that compose client islands, and pass server components through client wrappers as children so a context provider doesn't drag the app into the bundle. Also correct a common misconception: client components still SSR to HTML on first load — 'use client' means "hydrates and ships JS," not "browser-only."

10. How does streaming SSR improve performance if the data is still slow?

It doesn't make data faster — it decouples TTFB/FCP from the slowest fetch. Buffered SSR gates the entire response on the slowest server-side await; streaming flushes the shell and fast content immediately and streams slow Suspense-bounded sections when ready. Users get layout and usable content in hundreds of milliseconds with honest skeletons for the slow parts; paired with selective hydration, early content is interactive early. Real FCP improves, LCP usually improves if the LCP element is in the shell, and perceived performance improves a lot. Caveat I'd volunteer: if everything is slow you've swapped a blank page for a skeleton page — then the fix is caching or the backend.

11. What does loading.tsx actually do?

It's convention-based sugar: Next wraps the segment's page in a Suspense boundary with your loading component as the fallback. That yields instant navigations — the router paints the shell + fallback immediately while the page's server components fetch and stream in. It's segment-granular; for section-level granularity you nest <Suspense> boundaries inside the page yourself. It also composes: each nested segment's loading.tsx creates its own boundary.

12. Explain the promise-as-prop pattern with use().

When a client component needs server data without blocking the stream or creating a client waterfall: the server component calls the fetch function without awaiting, passes the promise as a prop, and the client component unwraps it with use(promise) inside a Suspense boundary. The fetch starts at server render time, the shell streams immediately, and the resolved data streams into the client component when ready. It's the best of three worlds: server-initiated fetch timing, non-blocking TTFB, and client-side interactivity over the data.

13. What are server actions and what would you watch out for?

Functions marked 'use server' that the framework exposes as RPC endpoints — forms and event handlers invoke them directly, no hand-written API route, with progressive enhancement (a form with action={fn} can submit before hydration). After mutating, you call revalidatePath/Tag so cached data and routes refresh in the same round trip; useActionState/useFormStatus handle pending/error UI. Watch-outs: they're public HTTP endpoints, so authenticate and validate inside every action regardless of where the button renders; they run sequentially (not for parallel-heavy workloads); and keep them thin — delegate to a service layer you can also test directly.

14. getServerSideProps vs getStaticProps vs the App Router equivalents?

Pages Router: getServerSideProps = per-request SSR with full req/res access; getStaticProps = build-time SSG, plus revalidate for ISR; getStaticPaths declares which dynamic paths to prebuild with fallback controlling unknown paths. App Router folds these into rendering semantics: static rendering is the default (≈ getStaticProps), using cookies()/headers()/uncached fetches makes it dynamic (≈ getServerSideProps), generateStaticParams + dynamicParams replace getStaticPaths/fallback, and revalidate exists per-fetch or per-route. The deeper shift: data fetching moves from one per-page function to per-component async server components, with memoization deduping shared calls.

15. How would you migrate a large Pages Router app to the App Router?

Incrementally — both routers coexist in one deployment, so there's no big bang. Order: recreate the shared shell as a root layout; move leaf/content pages first (lowest risk, immediate RSC bundle wins); then data-heavy pages, rethinking one-blob-of-props into per-component fetching; auth and middleware-adjacent flows last. The gotchas are context providers (need client-boundary wrappers), anything reading Node req/res, and caching behavior differences — I'd re-verify freshness semantics per route. And I'd be honest about ROI: stable low-churn pages can stay on the Pages Router indefinitely; migrate where you're already working or where a route materially benefits.

16. What does next/image actually do, and what does it cost?

Four concrete things: on-demand resize/re-encode (WebP/AVIF via content negotiation) through an optimization endpoint with srcset generation from sizes; native lazy-loading below the fold; required dimensions that reserve layout space so images never cause CLS; and priority to eager-load + preload the LCP image. The cost: optimization is real compute — metered pricing on Vercel, sharp CPU + cache storage when self-hosting — so at scale many teams point a custom loader at a dedicated image CDN instead. Biggest practical wins in audits: correct sizes (phones stop downloading desktop images) and priority on the hero.

17. Edge runtime vs Node runtime — when does edge actually win?

Edge = V8 isolates at CDN PoPs: near-zero cold start, global proximity, but web-standard APIs only — no fs, no TCP database drivers, tight CPU/bundle limits. Node = full ecosystem, regional, heavier cold starts. Edge wins when the work needs no distant data: middleware auth-cookie checks, geo routing, A/B assignment, rewrites — decisions in microseconds next to the user. The trap: edge compute calling a single-region database round-trips the ocean per query and can be slower than Node deployed beside the DB. So: middleware and light personalization at the edge; data-heavy rendering in Node near the data, unless the data layer itself is globally distributed.

18. How do you handle SEO for dynamic pages in the App Router?

Async generateMetadata per route — it can fetch the entity (request memoization dedupes with the page's own fetch) and return title/description/OG/canonical, merged down the layout hierarchy. Add app/sitemap.ts and app/robots.ts file conventions, JSON-LD injected server-side as an application/ld+json script for rich results, and opengraph-image.tsx with ImageResponse for per-page dynamic social cards. The foundation is that these pages are server-rendered (SSG/ISR/SSR), so crawlers and unfurlers get complete HTML without JS execution.

19. Next.js vs Remix (React Router v7) — how do you compare them?

Remix's model: per-route loader (server data) and action (mutation); forms post to actions and loaders revalidate automatically; caching leans on standard HTTP Cache-Control rather than a framework cache — fewer novel concepts, very predictable. It merged into React Router v7 framework mode. Next counters with RSC (per-component server code with zero-JS shipping), streaming/PPR at finer granularity, ISR machinery, and a larger ecosystem. My honest framing: Remix/RRv7 is simpler to reason about and debug; Next is more capable at the frontier and better for static/ISR-heavy content. Either is defensible — the choice is usually team familiarity and how much you value web-standards simplicity vs bleeding-edge capability.

20. When would you NOT use Next.js?

Internal tools and authed dashboards — no SEO, warm-cache repeat users — where a Vite SPA with TanStack Router/Query is simpler: static hosting, no server fleet, no RSC model, no four-cache debugging. Content-dominant sites with trivial interactivity — Astro's islands ship near-zero JS and win vitals by default. Teams without appetite for operating servers or absorbing framework churn. And I'd name Next's costs explicitly: server infra, Vercel-flavored coupling, caching complexity as a bug source, RSC onboarding. Choosing Next everywhere by reflex is exactly the non-senior answer this question probes for.

21. A user updates their profile, navigates away and back, and sees stale data. Diagnose.

Layer by layer. Most likely: the client Router Cache serving a cached RSC payload on back-navigation — fix by performing the update via a server action that calls revalidatePath('/profile') (which also busts the router cache), or router.refresh() after a client mutation. Next suspect: the Data Cache — the profile fetch was cached (default in Next 14!) with no tag/revalidate; add tags: ['user-123'] and revalidateTag on write, or no-store for always-fresh. Then: the route was statically rendered when it should be dynamic (it reads no request data, so it froze at build). Also check any CDN in front honoring stale s-maxage. The systematic answer — memoization → data cache → route cache → router cache → CDN — is what's being tested.

22. What is Partial Prerendering and why does it matter?

PPR serves a statically prerendered shell from the edge instantly, with dynamic "holes" — Suspense boundaries around request-time content like cookies()-reading components — streamed into the same response from the server. It dissolves the static-vs-dynamic route-level binary: today one personalized element forces the whole route dynamic; with PPR the decision moves to Suspense-boundary granularity, so a product page gets CDN-fast shell delivery and per-user pricing in one request. Status: experimental through Next 14/15 (flag-gated), positioned as the intended future default — I'd cite it as direction, and verify stability before production use.

23. Explain layout.tsx persistence and when you'd use template.tsx instead.

Layouts don't re-render or remount when navigating between their children — a sidebar keeps scroll position, a provider keeps state, effects don't re-run. That's also why layouts can't read searchParams: they don't re-render on navigation, so the value would be stale. template.tsx occupies the same slot but creates a fresh instance per navigation — state resets, effects re-fire — for enter animations, per-page logging, or forms that must reset between sibling routes. Default to layout; reach for template only when you specifically need remount semantics.

24. How do parallel and intercepting routes enable the modal-over-feed pattern?

Parallel routes give a layout named slots (@feed, @modal) rendered as props, each with independent navigation and loading/error states. An intercepting route like (.)photo/[id] under @modal catches client-side navigation to /photo/123 and renders it into the modal slot over the feed instead of replacing the page — while a hard refresh or shared link hits the real /photo/[id]/page.tsx full page. Result: shareable, deep-linkable modal URLs with correct back-button behavior and no duplicated photo UI — the Instagram pattern. It's awareness-level API, but naming the use case shows you know why it exists.

25. Your Next SSR app has p95 TTFB of 3s. Walk through your investigation.

First, is the route dynamic unnecessarily? — a stray cookies() call or no-store fetch may be defeating static rendering; make it static/ISR if the content allows. If it's legitimately dynamic: profile server data fetches — serial awaits are the classic culprit (parallelize with Promise.all or component-level parallel fetching), then cache slow upstreams in the Data Cache with tags. If a slow dependency is irreducible, wrap it in Suspense and stream so TTFB detaches from it. Then infra: cold starts (bundle size, provisioned concurrency), server region vs data region, connection pooling to the DB. Order matters: eliminate the render-time work before optimizing it, cache before streaming, stream before scaling hardware.

Hands-on exercises

Exercise 1 — Choose the strategy: news publisher

Scenario: A national news site: ~50k articles and growing hourly, breaking-news pages updated minute-by-minute for the first hours, evergreen archive, a personalized "For You" rail on every article page, comments, heavy SEO dependence. Design the rendering strategy per surface and defend it.

Model reasoning:

  • Article pages → ISR with on-demand revalidation. SEO-critical and read at 1000:1 read/write ratios — CDN static serving is mandatory economics. Time-based revalidate: 60 alone is wrong for corrections (a libel fix must not wait); wire the CMS "publish/update" webhook to revalidateTag(`article-${'{'}id{'}'}`) so pages are static until content actually changes. Use generateStaticParams for the most recent/trafficked few thousand; dynamicParams: true renders the archive long tail on first hit then caches — never build 50k pages per deploy.
  • Breaking-news liveblogs → static/ISR shell + client polling or streaming for the update feed. Even 30s staleness is acceptable for the article frame, but the live-update stream should be a client island fetching (or SSE) — don't force the whole page dynamic for the one live region. This is a PPR-shaped problem; say so.
  • "For You" rail → out of the HTML cache path entirely. Either a client-fetched island under Suspense-style skeleton, or (if in-HTML matters) a PPR/streamed dynamic hole. The cardinal sin is letting a per-user component read cookies() at page level and flip every article to dynamic rendering — one candidate mistake interviewers plant deliberately.
  • Comments → lazy client-side island. Below the fold, no SEO value worth the cost (or SSR only the first N for long-tail keywords if product insists), hydrate on visibility.
  • Home page → ISR with a short window (30–60s). Same for everyone, changes constantly but tolerates a minute of staleness, absorbs traffic spikes as pure CDN hits — exactly what ISR is for.

Exercise 2 — Choose the strategy: B2B analytics dashboard

Scenario: A SaaS analytics product: everything behind login, heavy charting and cross-filtering, data freshness matters (users watch live campaign numbers), 5-person frontend team, marketing site already exists separately on Webflow. The CTO asks: "Should this be Next.js?" Argue both sides, then commit.

Model reasoning:

  • Interrogate what Next would buy: SEO — none (authed). First-paint for anonymous cold visits — none (marketing is elsewhere). Server data proximity — real but replaceable by a normal API. Mixed rendering — not needed; every page is personalized and dynamic.
  • The SPA case: a Vite SPA + TanStack Router/Query is the honest default here: static hosting (no server fleet for a 5-person team to operate), Query's cache/refetch/optimistic-update model is better suited to live-updating dashboards than RSC's request-time rendering, and the RSC learning curve buys nothing when ~everything would be 'use client' anyway.
  • The Next case (steelman): server actions + RSC could thin the API layer; if the team later adds public shareable report pages (SEO/unfurl), Next handles them natively; one framework across future surfaces.
  • Commit: Vite SPA. The one Next-shaped requirement on the horizon — public share links with OG previews — can be served by a tiny separate Next app or even an edge function generating OG tags, cheaper than adopting Next's operational and conceptual costs product-wide. State the reversal condition explicitly: "If public, search-indexed content pages become core to the product, I'd revisit." Committing with a named trigger for reversal is the senior close.

Exercise 3 — Debug the cache: the stale price incident

Scenario: Next 14 App Router e-commerce. Ops report: prices updated in the admin tool don't appear on product pages for up to an hour; sometimes a hard refresh shows the new price but in-app navigation shows the old one. app/products/[id]/page.tsx awaits fetch(`${'{'}API{'}'}/products/${'{'}id{'}'}`) with no options and the route has revalidate = 3600. Explain every layer involved and fix it properly.

Model reasoning:

  • Data Cache: in Next 14, an options-less fetch defaults to force-cache; combined with route revalidate = 3600, the price response is legitimately served stale for up to an hour. That's the headline hour-long staleness.
  • Full Route Cache: the route is statically rendered (nothing dynamic in it), so the rendered HTML itself is also cached for the same window — two server-side layers agreeing to be stale.
  • Router Cache: "refresh shows new, navigation shows old" is the client-side RSC payload cache serving a previously visited segment — hard refresh bypasses it, client-side navigation doesn't. Classic Next 14 symptom (30s+ staleness windows).
  • The fix — event-driven invalidation, not shorter timers: tag the fetch (next: {'{'} tags: [`product-${'{'}id{'}'}`] {'}'}); have the admin mutation (server action or webhook → route handler) call revalidateTag(`product-${'{'}id{'}'}`), which purges the Data Cache entry, invalidates the route's cached render, and propagates to router caches on next navigation. Keep revalidate as a long backstop. Dropping to no-store "fixes" it by paying full dynamic rendering on every view — the junior fix; tags keep CDN economics and instant correctness.
  • Bonus depth: if truly live fields (flash-sale countdown, stock) coexist with cacheable content, isolate them into a streamed dynamic hole or client island rather than making the route dynamic — the PPR-shaped decomposition again.

Exercise 4 — Design the streaming plan: travel search results

Scenario: Flight search results page: the search header and filters render from the query string instantly; the results list depends on a fan-out to airline APIs (p50 1.2s, p95 4s); a price-history chart depends on an internal service (200ms); a "price alerts" widget needs the user session. SEO matters for popular route pages (/flights/sfo-to-nyc). Design the rendering + streaming architecture.

Model reasoning:

  • Route classification: results depend on searchParams → dynamic rendering; that's fine and correct here. For canonical route pages like /flights/sfo-to-nyc, create ISR'd landing pages (static content + cached typical prices) that then stream live results — SEO gets stable static HTML, users get live data.
  • Streaming layout: flush the shell (header, filters — derivable from the URL synchronously) immediately. Wrap the results list in its own <Suspense> with a realistic skeleton (fixed row heights → no CLS). The 200ms chart could go in the shell, but tying it to the results boundary couples fast to slow — give it its own boundary and let it appear at ~200ms.
  • The 4s p95 problem: don't hold the results boundary for the full fan-out. Stream partial results: render what airlines returned by ~1.5s and let a client component merge late arrivals (promise-as-prop with use(), or client subscription to a results stream). Perceived performance beats completeness for search.
  • Session widget: reads cookies — quarantine it in its own Suspense boundary (PPR-shaped) or make it a client island, so personalization never blocks or de-optimizes the shared content path.
  • What to say about metrics: TTFB stays flat (shell), FCP ≈ shell paint, LCP is the results list — so invest in the partial-results strategy, and confirm the skeleton isn't accidentally the LCP element. Close with the caveat: streaming is presentation; the durable fix for a 4s p95 fan-out is caching airline responses server-side with short TTLs.
