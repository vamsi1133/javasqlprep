16 — Build Tooling & Modules

Module systems, bundler mechanics, Vite/webpack internals, publishing, and monorepos — the layer that separates "framework user" from "senior engineer".

Why interviewers ask this

Anyone with two years of React can write components. What interviewers use build tooling for is to find out whether you understand the system your code runs through. When a candidate says "the bundle is 2 MB, I don't know why" or "I added a library and tree shaking stopped working, so I gave up" — that's a mid-level signal. When a candidate says "I opened the analyzer, found moment's locales in the vendor chunk, traced the import to a barrel file, and replaced it with dayjs behind an adapter" — that's a senior signal: they can debug the build, not just the app.

Concretely, build-tooling questions probe three things:

  • Mechanics — do you actually know why ESM enables tree shaking, what an exports map does, why Vite's dev server is fast? Or do you repeat blog-post slogans?
  • Debugging under ambiguity — "the bundle doubled last sprint, walk me through finding out why" is a classic FAANG follow-up with no framework crutch.
  • Trade-off judgment — single vendor chunk vs granular splitting, dual-publish vs ESM-only, monorepo vs polyrepo. Senior answers name the trade-offs and pick based on context, not fashion.
How to use this page: Tier 1 sections are asked almost verbatim in interviews — know them cold, including the code. Tier 2 is where senior loops differentiate. Tier 3 is currency: one-line awareness answers that show you follow the ecosystem.

1. Module systems: ESM vs CJS from the build angle Tier 1

1.1 Why the difference matters to a bundler

You've seen the syntax comparison a hundred times. The senior framing is different: CJS and ESM differ in when their structure is knowable, and that single property is why the entire modern toolchain looks the way it does.

  • CommonJS is dynamic. require() is a plain function call that runs at runtime. You can call it conditionally, compute the path, mutate module.exports in a loop. A tool reading the source cannot in general know what a CJS module exports or imports without executing it.
  • ESM is static. import/export are syntax, not functions. They must appear at the top level, specifiers must be string literals, and the full import/export graph is knowable by parsing alone. That static analyzability is why tree shaking exists: a bundler can prove "nothing ever imports formatCurrency" and delete it, because there is no way to sneak in a dynamic access the parser can't see (property access on namespace objects being the tricky edge — see §3).
// CommonJS — structure only knowable at runtime
const name = process.env.IMPL === 'fast' ? './fast' : './slow';
const impl = require(name);          // legal CJS, opaque to static analysis
module.exports[Math.random() > 0.5 ? 'a' : 'b'] = impl;  // exports unknowable

// ESM — structure knowable by parsing
import { format } from './format.js';   // must be top-level, literal specifier
export { format };                       // export names fixed at parse time

1.2 Live bindings vs value copies

CJS exports are value copies at require time (you get whatever module.exports held when require returned). ESM exports are live bindings — the importer holds a read-only view into the exporting module's binding, and sees later reassignments. This matters for circular dependencies (ESM can tolerate cycles via hoisted bindings that are "temporal-dead-zone until initialized"; CJS hands you a partially-filled snapshot) and for understanding why bundler CJS-interop shims are hairy. 

// counter.mjs
export let count = 0;
export function increment() { count++; }

// main.mjs
import { count, increment } from './counter.mjs';
increment();
console.log(count); // 1 — live binding, importer sees the update
// count = 5;       // TypeError: Assignment to constant variable (read-only view)

// CJS equivalent: `const { count } = require('./counter')` copies 0, stays 0.

Two more baseline facts worth stating unprompted: ESM is strict mode by default (no sloppy-mode this === global, no undeclared assignments), and ESM evaluation is deferred/asynchronous-capable (which is what makes top-level await possible), whereas require is fully synchronous.

1.3 Interop pain in practice

The classic bugs a senior candidate should be able to diagnose on sight:

  • The default-export mismatch. CJS has no concept of a "default export" — there is just module.exports. When Babel/TS compile export default to CJS, they emit exports.default = ... plus an exports.__esModule = true marker. Consuming code then checks the marker: if present, import x from 'lib' reads .default; if absent (a genuine hand-written CJS lib), the whole module.exports is treated as the default. Tools disagree at the margins, which is why you've seen lib.default.default or "x is not a function" after changing bundlers. TypeScript's esModuleInterop: true exists to paper over exactly this.
  • The dual-package hazard. A package that ships both CJS and ESM builds can be loaded twice in one process — once via require (by a CJS dependency) and once via import (by your app). Now you have two module instances: two singletons, two instanceof realms, two React copies ("Invalid hook call" is often exactly this), err instanceof MyError failing across the boundary. Mitigations: ship ESM-only; or make one format a thin re-export wrapper of the other so state lives in a single instance; or isolate shared state in a small CJS core that both entry points require.
Pitfall: "Invalid hook call" / "two copies of React" is frequently a resolution problem, not a React problem. Duplicate versions in the lockfile, a linked package resolving its own node_modules/react, or the dual-package hazard. Debug with npm ls react and bundler aliasing (resolve.alias / resolve.dedupe in Vite) before touching app code.

1.4 Node resolution: the "exports" map

Historically packages advertised entry points via "main" (CJS) and the unofficial "module" field (ESM, honored by bundlers only, never by Node). Node 12+ replaced this with "exports" — an explicit, encapsulating map of every path a consumer may import, with conditional resolution: 

{
  "name": "my-lib",
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",       // MUST be first — resolvers take first match
      "browser": "./dist/index.browser.js",
      "import": "./dist/index.js",        // ESM consumers (import / import())
      "require": "./dist/index.cjs",      // CJS consumers (require)
      "default": "./dist/index.js"        // fallback, keep last
    },
    "./utils": {
      "types": "./dist/utils.d.ts",
      "import": "./dist/utils.js",
      "require": "./dist/utils.cjs"
    },
    "./styles.css": "./dist/styles.css",  // non-JS assets must be listed too
    "./package.json": "./package.json"    // tools often need to read this
  },
  "main": "./dist/index.cjs",             // fallback for old tooling only
  "module": "./dist/index.js",            // legacy bundler hint, superseded
  "sideEffects": false
}

Key behaviors to be able to state:

  • Conditions are matched in object order, first match wins — hence "types" first, "default" last. Conditions include import, require, browser, node, development/production, and custom ones (React Server Components use a react-server condition).
  • Exports encapsulate. Once "exports" exists, deep imports of unlisted paths (my-lib/dist/internal.js) throw ERR_PACKAGE_PATH_NOT_EXPORTED. This is a feature (real public API) and a migration foot-gun (it broke countless lodash/get-style deep imports when libraries adopted it).
  • The "types" condition + moduleResolution: "bundler". TypeScript historically ignored exports maps under moduleResolution: "node" (now "node10"). Modern app tsconfig should use "bundler" (resolves exports/conditions like a bundler, without forcing file extensions); published Node libraries typically use "node16"/"nodenext". If types "can't be found" for a modern package, a stale moduleResolution is suspect #1.

1.5 "sideEffects": false

A declaration to bundlers: "no module in this package does observable work at import time — if you don't use its exports, you may drop the whole module without evaluating it." This lets webpack/Rollup skip modules entirely (module-level pruning), which is much stronger than statement-level dead-code elimination. 

// The gotcha: CSS imports ARE side effects.
// If your library does:
import './button.css';    // evaluating this module injects styles

// and package.json says "sideEffects": false,
// bundlers may drop the CSS import when tree-shaking. Fix — allowlist:
{
  "sideEffects": ["*.css", "./src/polyfills.js"]
}
Pitfall: "My library's styles vanished in the consumer's prod build but work in dev" is the canonical sideEffects: false + CSS-import bug. Dev builds often skip tree shaking, so it only bites in production. Same story for polyfills, global registrations (customElements.define), and monkey-patches.

1.6 Import specifiers and dynamic import()

Specifier kindExampleResolved by
Bareimport React from 'react'node_modules walk + exports map (Node/bundlers). Invalid in the browser without an import map.
Relativeimport x from './utils.js'Relative to the importing file. Browsers & Node ESM require the extension; bundlers/TS traditionally infer it.
Absolute / URLimport x from 'https://esm.sh/preact'Browser-native ESM; Node supports node:, file: and (flagged/experimental) network imports.

Dynamic import() is the one dynamic escape hatch that stays analyzable enough: it returns a promise of the module namespace, and — crucially — every bundler treats it as a code-split point. The imported module and its unique dependencies become a separate chunk, loaded on demand. If the specifier contains an expression (import(`./locales/${lang}.js`)), webpack/Vite glob the possible matches at build time and emit a chunk per candidate plus a runtime map — which is clever, but can silently emit dozens of chunks. 

// A split point in every bundler:
const { renderChart } = await import('./chart.js');  // → chart-[hash].js chunk

// React-level sugar over the same mechanism:
const Chart = React.lazy(() => import('./Chart'));

2. What a bundler actually does Tier 1

The single most valuable mental model on this page. Every bundler — webpack, Rollup, esbuild, Rspack, Parcel — is a pipeline: 

entry points
   │
   ▼
1. RESOLVE      turn each specifier into an absolute file
                (node_modules walk, exports map conditions, aliases, extensions)
   │
   ▼
2. LOAD + TRANSFORM   read the file, run it through loaders/plugins
                (TS → JS, JSX → calls, CSS → JS module or extracted asset,
                 SVG → component/URL, down-level syntax for targets)
   │
   ▼
3. BUILD MODULE GRAPH  parse imports/exports of the transformed module,
                enqueue its dependencies; repeat until closure is complete
   │
   ▼
4. TREE-SHAKE   mark used exports from the entries outward; drop the rest
                (needs ESM; respects sideEffects declarations)
   │
   ▼
5. CHUNK        assign modules to output files:
                entry chunks, dynamic-import chunks, shared/vendor chunks
   │
   ▼
6. EMIT         write chunks + assets + source maps, inject a tiny RUNTIME
                (module registry, chunk loader, HMR client in dev)

When any build question stumps you in an interview, map it onto this pipeline out loud: "is this a resolution problem, a transform problem, a graph problem, or a chunking problem?" That structure alone reads as senior.

2.1 Why bundle at all in the HTTP/2+ era?

"HTTP/2 multiplexes, so just ship 2,000 tiny ESM files" is a trap answer. Bundling still matters because:

  • Request waterfalls, not request count, are the killer. Native ESM discovers dependencies by fetching and parsing. A chain main → app → page → widget → util is five sequential round trips before the leaf executes — depth, not width, and multiplexing doesn't help depth. Bundlers flatten the graph; browsers can only partially mitigate with <link rel="modulepreload">.
  • Compression efficiency. gzip/brotli build their dictionaries per response. One 300 KB file compresses meaningfully better than 300 one-KB files, and each response also carries header + framing overhead.
  • Transforms have to happen somewhere. TS, JSX, CSS Modules, vendor CJS→ESM — browsers run none of these. Even "no-bundle" dev servers (Vite) transform per-file and pre-bundle deps.
  • Whole-graph optimizations — tree shaking, scope hoisting/concatenation, minification with cross-module knowledge, chunk-level cache strategy — only exist if a tool sees the graph.

2.2 Source maps

A source map is a JSON file with a mappings field of Base64-VLQ–encoded segments; each segment says "generated line/column X came from original file F, line/column Y (and optionally original identifier name N)". The generated file links to it with //# sourceMappingURL=app.js.map. DevTools and error-reporting services use it to translate minified stack traces back to your TS/JSX. 

// vite.config.ts — production stance most teams should take
export default defineConfig({
  build: {
    sourcemap: 'hidden'   // generate .map files but omit the sourceMappingURL
                          // comment → browsers/users don't fetch them,
                          // but you can upload them to Sentry/Datadog to
                          // symbolicate production errors.
  }
});
// webpack equivalent: devtool: 'hidden-source-map'
Senior take: shipping full source maps publicly leaks source (including comments and sometimes secrets baked in via env define); shipping none makes prod errors unreadable. hidden maps uploaded to the error tracker during CI, then deleted from the artifact, is the standard grown-up answer. In dev, cheap variants (eval-cheap-module-source-map in webpack terms) trade column accuracy for rebuild speed.

3. Tree shaking for real Tier 1

Tree shaking = dead-code elimination performed on the ESM module graph. Because ESM import/export structure is static (§1.1), the bundler can mark which exports are actually referenced starting from the entry points, and drop unreferenced exports — and, with sideEffects info, entire unevaluated modules. The minifier (Terser/esbuild/SWC) then removes the now-unreachable statements.

3.1 What defeats tree shaking

  • CJS anywhere in the chain. A CJS module is an opaque runtime object; the bundler must keep all of it. One CJS dependency of a dependency can anchor a large subgraph.
  • Module-level side effects (or missing sideEffects declarations). If importing a module might do observable work, it must be evaluated even if nothing from it is used.
  • Dynamic property access on a namespace. import * as utils from './utils'; utils[key]() — the bundler can't prove which exports are unused, so it keeps them all.
  • Re-export barrels. The big one in real codebases — next.

3.2 The barrels problem

// src/components/index.ts — the classic "barrel"
export * from './Button';
export * from './DataGrid';      // imports ag-grid, 400 KB
export * from './RichTextEditor'; // imports prosemirror, 300 KB
export * from './Chart';          // imports d3

// consumer
import { Button } from '@/components';   // just wants a button…

In theory a perfect tree shaker prunes the rest. In practice: if any barrel-reachable module has (or can't be proven free of) side effects, or the package lacks sideEffects: false, or a re-exported dep is CJS, the whole barrel graph is retained. Even when shaking eventually succeeds, dev servers like Vite must still load and transform every module in the barrel to serve one import — barrels are also a dev-speed tax. Framework teams now actively de-barrel (Next.js added optimizePackageImports to rewrite barrel imports to direct ones automatically).

The lodash example every interviewer knows:

import { get } from 'lodash';      // ❌ lodash is CJS → entire ~70 KB min+gz? no —
                                   //    ~25 KB min+gz, ALL of it, untreeshakeable
import get from 'lodash/get';      // ✔ deep import: only get + its internals (CJS but scoped)
import { get } from 'lodash-es';   // ✔ ESM build, sideEffects flagged → shakes properly

3.3 /*#__PURE__*/ annotations

Function calls at module top level are conservatively treated as side-effectful. The /*#__PURE__*/ annotation tells minifiers/bundlers "this call has no side effects — if its result is unused, delete the call." Transpilers emit it automatically (e.g. class down-leveling, styled.div`...` wrappers in library builds), and library authors write it by hand: 

export const registry = /*#__PURE__*/ createRegistry();  // droppable if unused
export const Button   = /*#__PURE__*/ styled('button')({ ... });

Awareness-level: you don't hand-write these in app code, but if a library's unused top-level constants keep landing in your bundle, missing PURE annotations (or a minifier not honoring them) is the mechanism to name.

3.4 Debugging walkthrough: "why is moment 300 KB in my bundle?"

This is the archetypal build-debugging interview story. Narrate it as a method, not a fact:

  1. Measure first. Run the analyzer — rollup-plugin-visualizer (Vite) or webpack-bundle-analyzer — on a production build. Treemap shows moment/locale/*: ~160 locale files dominating a ~290 KB (minified) moment blob.
  2. Explain the mechanism. Moment is CJS and its main file effectively drags in every locale (via a require pattern bundlers resolve eagerly); nothing here is tree-shakeable.
  3. Trace who imports it. Search the graph: maybe it's yours, maybe a date-picker dependency. npm ls moment / analyzer's "reasons" view / Vite's vite-bundle-visualizer.
  4. Pick a fix by blast radius:
    • Config-only, zero code change: stub the locales — webpack new webpack.IgnorePlugin({ resourceRegExp: /^\.\/locale$/, contextRegExp: /moment$/ }), or alias the locale dir to an empty module in Vite. Bundle drops to ~70 KB.
    • Better long-term: migrate to date-fns (per-function ESM imports, shakes to a few KB) or dayjs (~2 KB core + opt-in plugins) — moment itself is in maintenance mode and recommends this.
    • If it's a transitive dep, alias moment → dayjs compatibility shims or replace the offending dependency.
  5. Lock the win in: add a size budget (size-limit, bundlesize CI check, webpack performance.maxAssetSize) so the regression can't silently return.
Senior framing: the specific library matters less than the loop — analyzer → identify mass → trace importer → fix at the cheapest effective layer → add a regression guard. Interviewers reward the loop; they've all heard "use dayjs" as a slogan.

4. Code splitting mechanics Tier 1

4.1 Chunk taxonomy

Chunk typeCreated byLoaded
Entry chunkEach configured entry pointVia <script> in the HTML, immediately
Dynamic (async) chunkEvery import() expressionOn demand by the bundler runtime
Shared/vendor chunksplitChunks / manualChunks extraction rulesAs a dependency of whichever chunks reference it
Runtime chunkwebpack optimization.runtimeChunkThe chunk-loading machinery itself, split out so app-chunk hash changes don't invalidate it

4.2 Shared-chunk extraction and the vendor-splitting debate

// webpack: splitChunks
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      framework: {           // react + friends: changes rarely → stable hash
        test: /[\\/]node_modules[\\/](react|react-dom|scheduler)[\\/]/,
        name: 'framework',
        priority: 40,
      },
      vendor: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendors',
        priority: 20,
      },
    },
  },
  runtimeChunk: 'single',
}

// Vite/Rollup: manualChunks
build: {
  rollupOptions: {
    output: {
      manualChunks(id) {
        if (id.includes('node_modules')) {
          if (/react|react-dom|scheduler/.test(id)) return 'framework';
          return 'vendor';
        }
      },
    },
  },
}

The debate you should be able to argue both sides of:

  • Single vendor chunk — fewer requests, simple; but bumping any dependency changes the contenthash and invalidates the whole vendor cache for every returning user.
  • Granular per-package chunks (Next.js's approach: hash-named chunks per npm package above a size threshold) — a lodash patch only invalidates the lodash chunk; framework chunk stays cached for months. Costs: more requests (fine on H2), more complex graph, risk of many tiny chunks hurting compression.
  • The reasoning primitive is cache stability with [contenthash]: filenames derive from content, served with Cache-Control: immutable; splitting strategy is really a question of which bytes change together. Group by change-frequency: framework (glacial) / vendors (monthly) / app (daily).

4.3 Preload / prefetch of chunks

// webpack magic comments
const Settings = lazy(() => import(
  /* webpackPrefetch: true */   // <link rel="prefetch"> — idle-time, low priority,
  './Settings'                  // for "probably needed later" (next route)
));
const Editor = lazy(() => import(
  /* webpackPreload: true */    // <link rel="preload"> — parallel with parent chunk,
  './Editor'                    // for "needed almost certainly, right now"
));

// Vite: build.modulepreload — Vite auto-injects
// <link rel="modulepreload"> for each entry's static ESM deps,
// flattening the discovery waterfall. For route prefetching you typically
// trigger the import() on link hover/viewport (what Next/Remix do).
Pitfall: preloading everything is anti-optimization — preload competes with critical resources for bandwidth. Prefetch is the safe default for routes; reserve preload for assets on the critical path of the current interaction.

4.4 CSS splitting and the runtime

Modern bundlers split CSS per chunk: the CSS imported by modules of an async chunk is emitted as a sibling .css asset, and the runtime loads it before executing the chunk (Vite does this by default; webpack via MiniCssExtractPlugin). Know the failure mode: a flash of unstyled content or an "await on CSS" delay when a chunk's stylesheet is slow. Also be able to say what the runtime is: the few KB of generated code that maps chunk ids to URLs, deduplicates in-flight loads, wires import() to script/CSS insertion, and (in dev) hosts the HMR client.

5. Vite deep dive Tier 1

Vite is the current default; interviewers expect you to explain why it's fast mechanically, not vibes.

5.1 The dev-mode architecture (the core answer)

Dev is unbundled: Vite serves your source as native ESM, transformed on demand, per file. The browser's module loader does the graph traversal; Vite is "just" a smart transform server with a cache. Cold start is near-instant because nothing is bundled up front — cost is proportional to the modules the current page actually imports, not the whole app.

Two esbuild-powered pieces make this viable:

  • Dependency pre-bundling. On first run, Vite scans your imports and bundles each bare dependency once with esbuild into node_modules/.vite/deps. Why: (a) CJS→ESM conversion — the browser can't load CJS, and much of npm still is; (b) waterfall collapse — lodash-es is 600+ internal modules; pre-bundling makes it one request instead of 600 sequential-ish fetches. Pre-bundled deps are cached and served with immutable headers; the cache keys off the lockfile, which is why "weird stale dep behavior" is fixed by vite --force or deleting .vite.
  • Per-file transforms. TS and JSX are handled by esbuild, which strips types — it does no type checking. This is a deliberate trade: transforms stay ~20-30x faster, and type checking moves out-of-band — your editor in dev, and tsc --noEmit (or vue-tsc for Vue SFCs) as a CI gate. If a team has no tsc --noEmit in CI, their Vite app can deploy with type errors; saying that unprompted is a strong senior signal.

5.2 The prod side: Rollup (and Rolldown)

Production builds use Rollup: mature tree shaking, scope hoisting, a deep plugin ecosystem, and fine-grained output control — things esbuild intentionally doesn't prioritize. So Vite is really two tools behind one config and one plugin API (Vite plugins are a superset of Rollup plugins with extra dev hooks like configureServer and handleHotUpdate). The dev/prod engine split is Vite's main criticism — subtle dev/prod inconsistencies — which is exactly what Rolldown (the Rust Rollup-compatible bundler from the Vite team, folding in Oxc for transforms) is built to eliminate: one Rust engine for both modes, and much faster prod builds. Rolldown-powered Vite is the announced direction; know it exists and why.

5.3 Config anatomy

// vite.config.ts
import { defineConfig, loadEnv } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '');
  return {
    plugins: [react()],                    // Rollup-compatible plugin objects
    resolve: {
      alias: { '@': '/src' },             // mirror tsconfig "paths"
      dedupe: ['react', 'react-dom'],     // kill duplicate-React issues
    },
    define: {
      // compile-time constant replacement (like webpack DefinePlugin);
      // values are injected as source text → JSON.stringify strings!
      __APP_VERSION__: JSON.stringify(process.env.npm_package_version),
    },
    server: {
      proxy: {                             // dev-only API proxy: same-origin
        '/api': {                          // requests → no CORS in dev
          target: 'http://localhost:4000',
          changeOrigin: true,
        },
      },
    },
    build: {
      sourcemap: 'hidden',
      target: 'es2020',
    },
  };
});

Env vars: exposed on import.meta.env, statically replaced at build time — and only variables prefixed VITE_ reach client code. The prefix is a security design: it makes leaking a server secret into the bundle an explicit act (you'd have to rename DB_PASSWORD to VITE_DB_PASSWORD) rather than a default. The dual lesson to state: nothing in a client bundle is secret regardless of prefix — the prefix prevents accidents, not extraction.

Library mode (build.lib): entry + formats (es/cjs/umd), with rollupOptions.external: ['react'] so peers aren't bundled — the standard way to build a publishable component library with Vite.

5.4 HMR mechanics and React Fast Refresh

On file change, Vite invalidates the module, then walks up the import graph looking for an accepting boundary — a module that called import.meta.hot.accept(). It pushes the update over WebSocket; the client re-fetches just the changed module(s) (cache-busted URL query) and hands the new module to the boundary's accept callback. If the walk reaches the entry with no accepting boundary → full page reload. 

// what a framework plugin generates conceptually:
if (import.meta.hot) {
  import.meta.hot.accept((newModule) => {
    // decide how to apply newModule without reloading the page
  });
}

React Fast Refresh builds on this: the plugin makes every module whose exports are all React components self-accepting. On update, React re-renders the component with the new implementation but preserves hook state by keeping the fiber and matching hooks positionally. What resets or downgrades the experience:

  • Non-component exports in the same file (a component plus an exported constant/util): the module can't be safely self-accepting → update propagates upward, often becoming a full reload or a state reset. This is the practical reason for "components-only files".
  • Anonymous/default-exported unnamed components (Refresh can't track identity), and edited hook signatures (order/count of hooks changed) → state is intentionally reset.
Interview phrasing: "HMR is a protocol between the dev server and accepting module boundaries; Fast Refresh is React's policy layered on it: component-only modules self-accept and remount is avoided by preserving fibers, so hook state survives edits — unless the file mixes in non-component exports."

6. Webpack literacy Tier 2

You'll meet webpack in legacy codebases, ejected CRA apps, and — most importantly for interviews — Module Federation micro-frontends (see chapter 15). You need literacy, not mastery.

6.1 Core concepts — and the classic distinction question

// webpack.config.js
module.exports = {
  mode: 'production',            // sets sane defaults: minify, DefinePlugin
                                 // NODE_ENV, deterministic ids…
  entry: { app: './src/index.tsx' },
  output: {
    filename: '[name].[contenthash].js',
    path: path.resolve(__dirname, 'dist'),
    publicPath: '/assets/',      // URL prefix chunks are loaded from at runtime
    clean: true,
  },
  module: {
    rules: [
      { test: /\.tsx?$/, use: 'swc-loader', exclude: /node_modules/ },
      { test: /\.css$/, use: [MiniCssExtractPlugin.loader, 'css-loader'] },
      { test: /\.png$/, type: 'asset' },   // asset modules replaced url/file-loader
    ],
  },
  plugins: [
    new HtmlWebpackPlugin({ template: 'src/index.html' }),
    new MiniCssExtractPlugin({ filename: '[name].[contenthash].css' }),
    new webpack.DefinePlugin({
      'process.env.API_URL': JSON.stringify(process.env.API_URL),
    }),
  ],
};

"Loader vs plugin?" — a genuinely common question. Loaders are per-file transforms in the module pipeline: they take one module's source in, return JS (or another loader's input) out — ts → js, css → js module. Plugins hook into the compiler's lifecycle via the tapable event system and can see or mutate everything — the whole compilation, chunk graph, and emitted assets. Rule of thumb: transforms one file = loader; needs whole-build awareness (emit HTML, extract CSS to separate files, define globals, analyze) = plugin.

Key plugins to recognize: DefinePlugin (compile-time constant substitution — enables dead-code elimination of if (process.env.NODE_ENV !== 'production') blocks), MiniCssExtractPlugin (CSS as real files instead of style-injecting JS), HtmlWebpackPlugin (emits HTML with hashed script/link tags injected).

6.2 Why it's slower — and when it's still the right call

  • Slower because: it's JavaScript end-to-end (parse/transform/minify all single-thread-JS unless you swap in swc/esbuild loaders), and its dev model is eager bundling — build the entire graph before serving byte one, and rebundle (incrementally) on change. Vite inverted exactly these two decisions.
  • Still chosen when: Module Federation is a hard requirement at maturity (webpack is its home; Rspack now also ships MF, and Vite's ecosystem plugin has caveats); when a build depends on deep webpack-specific plugin/loader behavior; or when the migration risk of a huge legacy config outweighs dev-speed pain.

6.3 The webpack → Vite migration story

Interviewers love "you're migrating a 5-year-old webpack app to Vite — what breaks?" Checklist answer:

  • require.context() and other webpack-only APIs → rewrite to Vite's import.meta.glob().
  • Env vars: process.env.REACT_APP_X → import.meta.env.VITE_X; sweep the codebase, and shim process.env via define only as a bridge.
  • Worker syntax: loader-based (worker-loader) → standard new Worker(new URL('./w.ts', import.meta.url), { type: 'module' }).
  • CJS in app source (sneaky require() in old files) — fine under webpack, breaks under native-ESM dev.
  • Loader chains for exotic assets (SVGR, raw imports) → equivalent Vite plugins / ?raw/?url suffixes; resolve.alias/tsconfig paths re-wired; index.html moves to project root as the entry.
  • Then validate: dev/prod parity smoke tests, bundle-size diff before/after, and keep the webpack build runnable until the Vite build has shipped to prod for a cycle.

7.1 Babel's targets thinking (still the canonical model)

>0.5%, last 2 versions, not dead, not op_mini all

}]]

The idea generalizes to every tool: declare targets once, let the toolchain compute the minimal set of syntax down-leveling and polyfills. esbuild/SWC/Vite all take a target the same way (Vite defaults to a modern 'baseline-widely-available'-style ES2020+ target).

7.2 What down-leveling costs — and why the baseline moved

  • async/await → generator + regenerator-runtime: historically ~2-3x code size for async-heavy files plus a runtime helper, and worse debuggability/perf. Class fields, optional chaining, spread — each expands similarly.
  • The history: the ES5 era begat differential serving — <script type="module"> (modern) + <script nomodule> (legacy) shipping two builds, because module-supporting browsers were a clean proxy for ES2017 support. Vite's @vitejs/plugin-legacy is the living descendant.
  • Today: every evergreen browser has years of ES2017–ES2020; a modern baseline (ES2020-ish, aligned with Baseline "widely available") is the default, and you down-level further only for a measured legacy audience — a business decision backed by analytics, not a reflex. Saying "we shipped ES2019 because 4% of revenue was on an old embedded WebView, and it cost us 9% bundle size" is exactly the kind of quantified trade-off senior interviews want.

8. Package publishing & package managers Tier 2

8.1 Dual ESM/CJS publishing — or ESM-only

The library-author decision tree:

  • Dual publish: build both formats (tsup/unbuild/Rollup make this one config line), wire the exports map from §1.4 (import → .js, require → .cjs, types per format). Maximum compatibility; costs: double artifacts, the dual-package hazard (§1.3), and per-format type declarations (or "masquerading" type bugs — the class of problem the arethetypeswrong checker exists to catch).
  • ESM-only: increasingly viable and where the ecosystem is heading — Node 20.17+/22 can require() synchronous-graph ESM natively, bundlers consume ESM natively, and major libraries have gone ESM-only. Costs: cuts off old Node CJS consumers and some Jest/ts-node configs; classic pain sign is ERR_REQUIRE_ESM in consumers' CI.

8.2 peerDependencies — the singleton question

{
  "peerDependencies": { "react": ">=18" },
  "devDependencies":  { "react": "^19.0.0" }   // for the lib's own tests/build
}

Why peer and not regular: React (and any singleton-by-contract lib — react-dom, styled-components with its context, form libraries sharing context objects) must exist exactly once in the host app. A regular dependency risks nesting a second copy → two Reacts → "Invalid hook call". peerDependencies says "the host provides this; I just declare the compatible range." Also mark the peer external in your build so it isn't bundled.

8.3 Semver, lockfiles, and npm ci

  • ^1.2.3 = any 1.x ≥ 1.2.3; ~1.2.3 = patch-only. 0.x is special: ^0.2.3 only allows 0.2.x — minors are treated as breaking pre-1.0.
  • What the lockfile guarantees: the exact resolved version, registry URL, and integrity hash (SHA-512) of every package in the tree — reproducible installs and tamper evidence. Ranges live in package.json; truth lives in the lockfile.
  • npm ci vs npm install: ci installs exactly the lockfile (deletes node_modules first, errors if lockfile and package.json disagree, never writes the lockfile). install may re-resolve and rewrite the lockfile. CI pipelines must use ci — "our build changed with no commits" is almost always npm install in CI absorbing a new transitive release.

9. Monorepos Tier 2

9.1 Why (and the honest costs)

  • Shared code without publish cycles: packages/ui is consumed by three apps directly — no version bump, publish, and triple-upgrade dance for a one-line fix.
  • Atomic cross-package changes: change a shared component's API and all its call sites in one PR, one review, one CI run.
  • Single toolchain: one ESLint/TS/build config, one dependency policy, one CI story.
  • When it hurts: genuinely unrelated products with different release cadences (you get coupling without cohesion); CI cost explodes without caching discipline (every PR building everything); tooling/ownership overhead needs a platform-minded owner; VCS scale is a real (if distant) ceiling.

9.2 Workspaces mechanics

// pnpm-workspace.yaml
packages: ['apps/*', 'packages/*']

// apps/web/package.json
{
  "dependencies": {
    "@acme/ui": "workspace:*"   // ALWAYS link the local package; the protocol
  }                             // errors rather than silently fetching from the
}                               // registry, and pnpm rewrites it to a real
                                // semver range on publish.

Workspaces = the package manager symlinks local packages into node_modules so imports resolve to sibling source/dist, and hoists/links shared external deps once at the root.

9.3 Task orchestration: Turborepo / Nx

// turbo.json
{
  "tasks": {
    "build": {
      "dependsOn": ["^build"],            // topological: build my deps first
      "inputs": ["src/**", "package.json", "tsconfig.json"],
      "outputs": ["dist/**"]
    },
    "test": { "dependsOn": ["build"], "inputs": ["src/**", "test/**"] },
    "lint": {}
  }
}

The three mechanisms to explain:

  • Task graph: dependsOn: ["^build"] derives execution order from the package dependency graph — maximal parallelism with correct ordering.
  • Caching = inputs hash → replay outputs. Hash the declared inputs (files, env vars, dep-package hashes, task definition); on hit, skip execution and restore the recorded output files and replay the logs. Corollary interviewers probe: an undeclared input (env var not listed, file outside inputs) means stale cache hits — the classic "works after turbo run --force" bug.
  • Remote caching + affected-only: share the cache across CI runners and laptops (CI builds it, you replay it); and compute the packages affected by a diff (turbo run test --filter=...[origin/main], nx affected -t test) so a docs change doesn't run 40 test suites. Together these are what make monorepo CI bills survivable.

10. Ecosystem currency Tier 3

One-liner awareness answers — enough to show you track the field, no more:

  • Import maps are native in all modern browsers: <script type="importmap">{"imports":{"react":"https://esm.sh/react@19"}}</script> lets bare specifiers resolve without a bundler — the primitive behind buildless setups and some micro-frontend version negotiation.
  • CDN ESM (esm.sh, jsdelivr /+esm): serves npm packages as browser-ready ESM (CJS converted on the fly). Great for prototypes, playgrounds, and islands; production apps still bundle for waterfalls/compression (§2.1).
  • The "JS infra rewritten in Rust/Go" narrative: the CPU-bound tier of the toolchain — parse, transform, minify, bundle, lint — has been systematically ported to native languages: esbuild (Go), SWC, Rspack, Turbopack, Rolldown/Oxc, Lightning CSS (Rust). The JS layer that remains is the plugin/config surface.
  • Bun: runtime + package manager + test runner + bundler in one (Zig, JavaScriptCore). Its install speed and built-in TS execution are real; teams commonly adopt it as a faster npm/test runner before trusting the bundler for prod.
  • Biome: Rust replacement for ESLint + Prettier in one binary — ~10-100x faster format/lint; the trade-off is a smaller rule/plugin ecosystem (notably type-aware rules).
  • TypeScript 7 / the Go port ("tsgo", Project Corsa): Microsoft is porting the TS compiler and language server to Go, with ~10x compile/check speedups demonstrated on large repos. Relevance to this chapter: tsc --noEmit being the slow CI gate (§5.1) stops being true, and editor responsiveness on monorepos changes materially.

Interview Q&A

1. Why does tree shaking require ESM? Why can't bundlers tree-shake CommonJS?

Tree shaking is static dead-export elimination, and it needs the import/export graph to be knowable without executing code. ESM guarantees that: import/export are top-level syntax with literal specifiers, so parsing alone reveals exactly which bindings each module exposes and consumes. CJS gives no such guarantee — require() is a runtime function call with computable arguments, and module.exports can be mutated arbitrarily (conditionally, in loops, with computed keys). A bundler facing CJS must conservatively keep the whole module object. Some tools attempt heuristic CJS "shaking" for simple patterns, but it's best-effort; the reliable contract is ESM plus sideEffects metadata so unevaluated modules can be dropped entirely.

2. Explain the dual-package hazard and how you'd mitigate it as a library author.

If I publish both CJS and ESM builds behind an exports map, one Node process can load both — my app imports the ESM file while a CJS dependency requires the CJS file. Node treats them as two distinct modules, so any module-level state is duplicated: singletons split, instanceof fails across the boundary, React context/hooks break ("Invalid hook call" is often this in disguise). Mitigations: (1) go ESM-only if my consumer base allows it — the hazard vanishes with a single format; (2) make one format a thin wrapper that re-exports the other, so there's a single stateful instance (feasible now that modern Node can require() ESM); (3) isolate mutable state in one small CJS file both entry points share. I'd also document that peer-style singletons (my lib's context objects) must not be nested by consumers.

3. Walk me through a modern package.json "exports" map — what does each condition do, and what changed vs main/module?

"exports" is an explicit map from every importable subpath to files, resolved per-condition: "import" for ESM consumers, "require" for CJS, "types" for TypeScript (must come first — first matching condition wins), "browser"/"node" for environment forks, "default" as the last-resort fallback. Versus the old world: "main" was a single CJS entry, "module" was an unofficial bundler-only ESM hint, and any file in the package was deep-importable. Exports gives (1) per-consumer-type resolution that Node itself honors, (2) encapsulation — unlisted paths throw ERR_PACKAGE_PATH_NOT_EXPORTED, making the public API real, and (3) subpath and condition granularity (e.g. a react-server condition). Consumer-side, TypeScript only respects exports maps under moduleResolution: "bundler"/"node16" — stale "node10" resolution is the usual cause of "types not found" for modern packages.

4. What does "sideEffects": false actually declare, and what's the classic bug it causes?

It's a package-level promise to bundlers: "importing any module here has no observable effects beyond providing exports — if no exports are used, skip evaluating the module entirely." That enables whole-module pruning, far stronger than statement-level DCE. The classic bug: a component library whose modules do import './styles.css' — that import is a side effect (it injects styles), so with sideEffects: false the consumer's production build tree-shakes the CSS away. Symptoms are sneaky: styles present in dev (no shaking), missing in prod. Fix: allowlist — "sideEffects": ["*.css", "./src/polyfill.js"]. Same class of bug for polyfills, customElements.define calls, and global registrations.

5. Why is Vite's dev server so much faster than webpack-dev-server? Be mechanical.

Two inverted decisions. First, no eager bundling: webpack builds the full module graph into a bundle before serving anything, so cold start scales with app size. Vite serves source files as native ESM, transformed on demand — the browser's module loader drives graph traversal, and cold-start cost is proportional to what the first page imports. Second, native-speed transforms: TS/JSX go through esbuild (Go, parallel) rather than JS-based loaders — an order-of-magnitude-plus difference. The load-bearing enabler is dependency pre-bundling: esbuild bundles each npm dep once into .vite/deps, which converts CJS to ESM (browsers can't load CJS) and collapses per-dependency waterfalls (lodash-es's 600 internal modules become one cached request). HMR is similarly bounded: invalidate one module, push over WebSocket, re-fetch just that module — no rebundle. The honest caveat: esbuild strips types without checking them, so tsc --noEmit must run in CI, and prod uses a different engine (Rollup), which is the dev/prod-parity criticism Rolldown targets.

6. Your production bundle doubled since last month. Walk me through finding the cause.

First, bisect the timeline: CI bundle-size history or git bisect with a build+measure script narrows it to a PR range. In parallel, diff the treemaps: run the analyzer (rollup-plugin-visualizer / webpack-bundle-analyzer) against current and last-month builds; the delta usually names the culprit in minutes — a new heavyweight dep, duplicated versions of one lib (check npm ls <pkg> and the lockfile diff), a barrel import pulling a whole library, moment-style locale/data payloads, or accidentally bundling a peer (externals config lost). Then classify the mechanism: new code (product decision — maybe fine, maybe split it behind import()), duplication (dedupe/resolutions/alias), failed tree shaking (CJS dep or sideEffects regression), or config regression (mode not production, minify off, sourcemaps inlined). Fix at the cheapest layer, and finish by adding a size budget (size-limit in CI) so the next regression is a red PR check, not a quarterly surprise.

7. What's the difference between a webpack loader and a plugin?

Loaders are per-module transforms: functions in module.rules that take one file's content and return JavaScript (or input for the next loader in the chain, applied right-to-left) — ts-loader, css-loader. Their scope is a single module in the graph-building phase. Plugins hook the compiler lifecycle via the tapable event system and can observe or mutate anything: the whole compilation, chunk graph, and emitted assets. HtmlWebpackPlugin emits a new asset; MiniCssExtractPlugin moves CSS out of JS into files (note it needs both a plugin and a loader — the loader intercepts modules, the plugin assembles assets); DefinePlugin rewrites source constants build-wide. Rule of thumb: transform one file → loader; need whole-build awareness or new assets → plugin.

8. How does dynamic import() interact with the bundler, and when would you not use it?

Every bundler treats import() as a split point: the target module and its exclusive dependency subgraph become a separate chunk, emitted with a content hash and loaded by the runtime (script injection + a promise for the namespace, plus that chunk's CSS first). Templated specifiers (import(`./locales/${l}.js`)) are handled by globbing candidates at build time — one chunk each. When not to: code needed for first paint (splitting it adds a request-waterfall hop and hurts LCP); tiny modules where per-chunk overhead exceeds savings; and inside hot paths where the async boundary complicates state (a modal's animation stuttering while its chunk loads — mitigate with prefetch-on-hover). Also remember each split point is a new failure mode: chunk-load errors after a deploy (old HTML referencing purged hashed chunks) need a retry/reload strategy.

9. Single vendor chunk vs granular per-package chunks — argue both sides.

The real question is cache stability under change, since contenthashed files are immutable-cached. Single vendor chunk: fewest requests, simplest mental model, best compression per byte — but bumping any dependency changes the hash and re-downloads the entire vendor payload for every returning user; in a weekly-deploy app your "stable" chunk is never stable. Granular (Next.js-style per-package chunks above a size threshold): a patch to one lib invalidates one small chunk; the React framework chunk survives for months — much better returning-user economics. Costs: dozens of requests (fine under HTTP/2 multiplexing but worse compression per response), a more complex graph, and possible over-fragmentation into sub-KB files. My default: three change-frequency bands — framework (glacial), vendor (monthly), app (daily) — plus a split runtime chunk; go granular when analytics show heavy repeat visits, keep it coarse for campaign-traffic sites where every visit is effectively cold.

10. What is React Fast Refresh actually doing on top of HMR, and what causes a full state reset?

Base HMR is a protocol: the dev server pushes changed modules, and the update propagates up the import graph until a module that called import.meta.hot.accept() (or webpack's module.hot.accept) handles it; no acceptor → full reload. Fast Refresh is React's policy layered on that: the babel/SWC plugin registers every component, and modules whose exports are all components are made self-accepting. On update, React swaps the component's implementation on the existing fiber and re-renders — hook state survives because the fiber and its positional hook list are preserved. Resets happen when identity or hook shape can't be preserved: the file exports non-components alongside components (module can't self-accept; update escalates), components are anonymous/unnameable, or the edit changed the hooks' order/count (Refresh deliberately resets to avoid corrupt state). This is the mechanical reason for the "one component per file, components-only exports" convention.

11. Explain the barrels problem and how you'd fix it in a large codebase.

A barrel (index.ts re-exporting a directory) makes every consumer import pull the entire re-export graph into the module graph. Tree shaking should prune it, but fails whenever anything in the graph is CJS, side-effectful, or unflagged by sideEffects — and even when prod shaking works, dev servers must transform every barrel-reachable module to serve one import, so dev is slow regardless. Diagnosis: analyzer treemap plus "why is this module included" traces; in Vite, watching hundreds of requests for one page. Fixes, in order of leverage: (1) codemod consumers to direct file imports (eslint no-restricted-imports on the barrel path to hold the line); (2) for third-party barrels, use per-module entry points (lodash-es, date-fns/format) or the framework's rewriter (Next's optimizePackageImports); (3) if a public barrel must exist (a library entry), ensure the package is pure ESM with sideEffects correctly declared so shaking actually can prune. I'd sell the change with before/after numbers from the analyzer, not ideology.

12. What does npm ci guarantee that npm install doesn't, and why does it matter in CI?

npm ci is a pure function of the lockfile: it removes node_modules, installs exactly the locked resolutions (verifying each integrity hash), fails hard if package.json and the lockfile disagree, and never mutates the lockfile. npm install may re-resolve ranges (new transitive releases within semver), write the lockfile, and thus produce different trees on different days from identical commits. In CI that difference is everything: with install, a dependency's Tuesday patch release can break Wednesday's build of an untouched branch — the "no commits, build broke" mystery — and your deploy artifact stops being reproducible from the git SHA. It's also a supply-chain control: integrity hashes in the lockfile mean a tampered registry response fails the install. Rule: humans run install to change deps; machines run ci.

13. Why is react a peerDependency of component libraries rather than a dependency?

Because React must be a singleton per app: hooks and context rely on module-level state (the current dispatcher, context registry), so two React instances in one tree produce "Invalid hook call" and context values that mysteriously don't cross library boundaries. If my library listed react as a regular dependency, a version mismatch with the host could nest a second copy under my package. peerDependencies: { "react": ">=18" } inverts ownership: the host app provides React, I declare the compatible range, and package managers install/validate accordingly (npm 7+ auto-installs peers; pnpm is stricter about conflicts). Two companion moves: keep react in my devDependencies for local dev/tests, and mark it external in the build config so it's never bundled into my dist. The same logic applies to any lib whose value is shared state: react-dom, react-router, styled-components, form-context libraries.

14. How does Turborepo (or Nx) caching work, and how can it serve a stale result?

Each task run is keyed by a hash of its declared inputs: relevant source files, the task definition, listed env vars, and the hashes of upstream package tasks (via dependsOn: ["^build"]). On execution, declared outputs (files + terminal logs) are stored under that key; on a later run with an identical hash, the tool skips execution and replays — restores output files, reprints logs. Remote caching shares this store across CI runners and developer machines, so CI's build is your laptop's cache hit. Staleness happens exactly when reality diverges from declaration: an env var the build reads but isn't listed in the task's env inputs; a file consumed outside declared inputs (reading from repo root, a generated file); non-hermetic steps (network fetches, timestamps); or outputs written outside declared outputs (so replay restores an incomplete set). The tell is "fails normally, passes with --force" — the fix is always to make the undeclared input declared, not to force-run in CI.

15. What is dependency pre-bundling in Vite and why is it necessary?

On first dev-server run (and whenever the lockfile changes), Vite scans source for bare imports and uses esbuild to bundle each dependency into single ESM files under node_modules/.vite/deps, served with strong cache headers. Two reasons it's not optional: (1) format — the browser only loads ESM, and a large share of npm is still CJS/UMD; pre-bundling converts them and applies the interop shims (named-export detection over CJS) once, up front; (2) waterfalls — a well-factored ESM package like lodash-es is hundreds of internal modules; served raw, that's hundreds of sequential-discovery requests for one import, so pre-bundling collapses each dep to one fetch. Practical corollaries: deps are optimized, your source isn't; discovering a new dep mid-session triggers re-optimization (a full-reload hiccup); and mismatched dep state causes the classic "outdated optimize dep" errors solved by vite --force. Linked monorepo packages are excluded by default so they stay hot-reloadable.

16. What breaks when migrating a legacy webpack app to Vite?

The predictable list: (1) webpack-only APIs — require.context() → import.meta.glob(); module.hot → import.meta.hot. (2) Env vars — process.env.REACT_APP_X doesn't exist in the browser; migrate to import.meta.env.VITE_X, optionally shimming via define during transition. (3) Workers — worker-loader imports → standard new Worker(new URL(...), { type: 'module' }). (4) Stray CJS in app source — old files using require()/module.exports break under native-ESM dev. (5) Loader chains — SVGR, raw/inline imports, sass options → Vite plugins and ?raw/?url suffixes. (6) Structure — index.html becomes the root entry; publicPath → base; aliases re-declared in resolve.alias. (7) Node-globals assumptions (Buffer, process) that webpack 4 auto-polyfilled. Strategy: run both builds in parallel, migrate behind a branch with a dev/prod parity smoke suite and a bundle-size diff, and don't delete the webpack config until Vite prod has shipped for a full release cycle.

17. What are source maps mechanically, and what's your production policy for them?

A source map is JSON alongside a generated file: sources (original files, optionally embedded via sourcesContent), names, and mappings — Base64-VLQ segments mapping generated line/column → original file/line/column/name. The generated file opts in via //# sourceMappingURL=. DevTools use them live; error trackers use uploaded maps to symbolicate minified stack traces. Production policy: hidden source maps — generate them (build.sourcemap: 'hidden' in Vite, devtool: 'hidden-source-map' in webpack), upload to Sentry/Datadog keyed by release during CI, and exclude them from the deployed artifact. That gets readable prod stack traces without publishing your source (public maps expose comments, internal URLs, and occasionally secrets inlined by define/env substitution). Dev policy: fast, cheap variants — rebuild speed matters more than column-perfect mapping.

18. ESM-only vs dual publishing for a new library in 2026 — what would you ship?

Default: ESM-only, unless the consumer analysis says otherwise. The blockers have fallen: modern Node can require() ESM (synchronous graphs), all bundlers consume ESM natively, and the ecosystem's biggest packages have led the migration — while dual publishing carries permanent costs: two artifacts, the dual-package hazard for any stateful lib, and per-format types subtleties (the arethetypeswrong class of bugs). I'd still choose dual if the audience demands it: enterprise consumers pinned on old Node LTS, Jest-without-ESM-config shops, or a library commonly required from CJS server code. If dual, the discipline is: exports map with types-per-condition, both builds from one source via tsup/Rollup, state isolated so the hazard can't bite, and attw + publint in CI to catch packaging errors before consumers do. Either way: files: ["dist"], prepublishOnly build+test, provenance on.

19. What are phantom dependencies, and how does pnpm prevent them?

A phantom dependency is a package your code imports but never declares — it works because npm/yarn's hoisting flattens transitive deps into the root node_modules where Node's resolution finds them. The failure is deferred and out of your control: the intermediate dependency drops or bumps that transitive package, and your build breaks with no change in your repo — or worse, prod resolves a different version than dev. pnpm's layout makes this structurally impossible: packages live once in a global content-addressable store; a project's node_modules contains symlinks only for declared dependencies, with each package's own deps nested in a hidden .pnpm virtual store. Undeclared imports fail immediately at dev time — the latent bug becomes a loud one. Bonus effects worth naming: massive disk/install savings via hard links, and stricter-by-default workspaces. Interop caveat: some legacy tools assume a flat layout, which is what pnpm's shamefully-hoist escape hatch exists for.

20. When would you *not* recommend a monorepo?

When the repo would contain coupling without cohesion: genuinely unrelated products, separate release cadences and on-call rotations, little shared code — you inherit shared CI queues, shared tooling upgrades, and shared breakage for zero atomic-change benefit. When the org won't fund the platform discipline: monorepo CI without remote caching and affected-only execution scales cost linearly with repo size, and someone must own turbo/Nx config, boundary rules, and dependency policy — absent that owner, it decays into a slow polyrepo with extra steps. When external constraints bite: open-sourcing one component from a private monorepo, vendor/partner access to a single project, or compliance boundaries requiring repo-level isolation. And when the driver is fashion: if the actual problem is "publishing shared UI is annoying," weigh lighter fixes (a single shared-packages monorepo consumed by polyrepo apps) before moving everything. My framing: monorepo is a tooling investment that pays off in shared-code density; measure that density first.

Hands-on exercises

Exercise 1 — Analyze a real bundle

Setup: take any real Vite React app you own (or scaffold one and add moment, lodash, and a chart library to make it interesting).

  1. Add the analyzer: npm i -D rollup-plugin-visualizer, then in vite.config.ts plugins: visualizer({ gzipSize: true, brotliSize: true, open: true }). Run vite build.
  2. Answer from the treemap: What are the top 5 modules by gzip size? What fraction is your code vs node_modules? Is any package present twice (two versions)?
  3. Pick the single largest dependency and trace why it's included: find the import chain from an entry to it. Is it statically imported where a dynamic import() would do?
  4. Apply one fix (dynamic-import a route/chart, replace moment, dedupe a duplicate) and record the before/after gzip total.
  5. Add size-limit with a budget 5% above your new total and wire it as an npm script — the regression guard is part of the exercise.

Done when: you can narrate the §3.4 debugging loop over your app with real numbers — that narration is directly reusable in interviews.

Exercise 2 — Break tree shaking, then fix it

Setup: create a tiny Vite lib-consumer pair: src/utils/ with 5 modules (one importing a heavy dep like chart.js), plus src/utils/index.ts barrel re-exporting all of them. In main.ts, import one light util from the barrel.

  1. Build and check output size. Now sabotage: add a top-level side effect to the heavy module (console.log('loaded') at module scope) and rebuild. Explain the size change.
  2. Sabotage #2: convert one util to CJS (module.exports) and import it through the barrel. Observe and explain what the bundle keeps.
  3. Sabotage #3: in main.ts, do import * as utils from './utils' and access utils[name]() with a runtime string. Verify all exports are retained.
  4. Fix each in turn: remove/annotate the side effect (/*#__PURE__*/ where applicable), restore ESM, restore static named imports, and finally replace the barrel import with a direct file import. Measure at each step.

Done when: you have a table of bundle size per sabotage/fix and can state, for each, which analysis guarantee was broken (side-effect provability, static exports, static usage).

Exercise 3 — Write a correct exports map for a dual-format library

Scenario: you maintain @acme/dates, TypeScript source in src/index.ts and src/tz.ts, plus src/styles.css for an included picker component. It must serve: modern bundler users (ESM + types), legacy Node CJS consumers, and it must keep ./tz as a separate subpath so timezone data isn't pulled in by default.

  1. Set up a tsup (or Rollup) build emitting dist/index.js+.cjs, dist/tz.js+.cjs, and both .d.ts/.d.cts declaration sets.
  2. Write the package.json: "type", full "exports" with types/import/require per subpath (types first!), the CSS subpath, "./package.json", correct "sideEffects" (careful — there's a CSS file), "files", legacy "main"/"module" fallbacks, and react as a peer if the picker needs it.
  3. Validate with npx publint and npx @arethetypeswrong/cli --pack — fix every finding and write one sentence per finding explaining the failure mode it would have caused for a consumer.
  4. Prove encapsulation: in a scratch consumer, confirm import '@acme/dates/dist/internal-helpers' fails while @acme/dates/tz resolves in both a require and an import context.

Done when: both checkers pass clean and you can explain why "types" ordering and the CSS sideEffects entry are load-bearing.

Exercise 4 — Monorepo task graph with a cache-poisoning bug

Setup: pnpm workspace with packages/ui (built lib), apps/web (Vite app depending on @acme/ui via workspace:*), Turborepo with build (dependsOn: ["^build"]) and test.

  1. Verify orchestration: change a file in ui, run turbo run build, confirm ui rebuilds then web rebuilds; run again, confirm both are FULL TURBO cache hits.
  2. Plant the bug: make ui's build read an env var (THEME=dark baked into output) without declaring it in turbo.json's env. Build with THEME=dark, then build with THEME=light — observe the stale cache hit serving dark output.
  3. Fix it by declaring the env input; verify the hash now changes with the variable. Then repeat the class of bug with an undeclared file input (build reads a root-level brand.json outside inputs) and fix that too.
  4. Stretch: enable remote caching (Vercel or a self-hosted cache), run CI-then-local, and confirm the local run replays CI's outputs. Add an eslint no-restricted-imports rule forbidding apps/web from importing @acme/ui/src/* internals.

Done when: you can explain "inputs hash → replay outputs" from experience, and you've personally caused and cured a stale-cache incident — the exact story interviewers ask for.
