12 — Component Patterns & Forms

Designing component APIs people can't misuse — compound components, controlled/uncontrolled, react-hook-form, error boundaries, and the pattern history you'll be quizzed on.

Why interviewers ask this

After seven years, nobody doubts you can build a tabs widget. What senior interviews probe is whether you can design its API: the props, the composition surface, the controlled/uncontrolled contract, the accessibility guarantees, the escape hatches. "Design a reusable Modal/Select/Tabs" is the frontend equivalent of a system-design round — there's no single correct answer, and the interviewer is watching which trade-offs you name unprompted. Do you reach for a boolean prop or for children? Do you know why Radix supports both value and defaultValue? Can you explain what an error boundary can't catch without being asked twice?

Machine-coding rounds (very common at product startups, increasingly at FAANG) make this concrete: 45–60 minutes, "build an accessible accordion / star rating / modal with a focus trap", graded on correctness, API design, keyboard support, and code organization — in that order of frequency but reverse order of what actually separates candidates. Forms are the other half of this tutorial because forms are where component patterns meet real product pain: controlled vs uncontrolled is the most misunderstood dichotomy in React, and react-hook-form + zod is the de-facto industry stack you're expected to know cold. This tutorial covers the patterns, the flagship implementations, and the history (HOC → render props → hooks) that still shows up as a knowledge-check question.

1. Component API design principles T1

Props are the contract

A component's props are a public API. Everything you know about API design applies: minimal surface area, sensible defaults, hard to misuse, easy to evolve without breaking callers. The single most useful framing in an interview: every prop you add is a promise you must keep forever (or break with a migration). Good component APIs are small and lean on composition; bad ones accrete a boolean per feature request.

The boolean-props explosion smell

The classic evolution of a Button in a codebase without API discipline: 

// Year 1
<Button label="Save" />

// Year 3 — configuration over composition, the smell in full bloom
<Button
  label="Save"
  icon="check"
  iconPosition="left"
  showSpinner={isSaving}
  spinnerSize="sm"
  truncateLabel
  tooltipText="Saves the record"
  badgeCount={3}
/>

Each prop exists because one caller needed it, and now every caller pays for it — in bundle size, in cognitive load, in impossible prop combinations (icon + showSpinner: which wins?). The fix is composition over configuration: accept children and let the caller own the markup. 

// Composition: the Button knows about button-ness, nothing else
<Button>
  {isSaving ? <Spinner size="sm" /> : <CheckIcon />}
  Save
  <Badge count={3} />
</Button>
Senior signal: articulate the rule of thumb — props for behavior and variants the component must know about (variant="danger", disabled, onClick); children/slots for content the component merely positions. When two boolean props can conflict, that's the API telling you it wants a variant enum or a composition seam instead. A quick heuristic interviewers love: if adding a feature means adding a prop AND an if inside the component, ask whether a slot would let the caller do it with zero component changes.

Controlled vs uncontrolled — the component design version

This is not just about <input>. Any stateful component (Tabs, Accordion, Modal, Select) faces the same choice: who owns the state? The React community converged on a naming convention, and you should follow it letter-for-letter because it's what every library user expects:

ModePropsState ownerUse when
Uncontrolled defaultValue (+ optional onChange for notification) The component (internal useState) Caller doesn't care about the value mid-flight; simplest usage
Controlled value + onChange (required together) The caller Caller needs to read, sync, veto, or derive from the value (URL sync, analytics, cross-component coordination)

A production-grade component supports both, like Radix does internally with useControllableState. The sketch below is a top-tier interview answer to "how would you let callers optionally control your Tabs?": 

function useControllableState({ value, defaultValue, onChange }) {
  const [internal, setInternal] = React.useState(defaultValue);
  const isControlled = value !== undefined;      // presence of `value` decides, once
  const current = isControlled ? value : internal;

  const setValue = React.useCallback(
    (next) => {
      // Support functional updates against the *resolved* current value
      const resolved = typeof next === 'function' ? next(current) : next;
      if (!isControlled) setInternal(resolved);  // uncontrolled: we own it
      onChange?.(resolved);                      // always notify
    },
    [isControlled, current, onChange]
  );

  return [current, setValue];
}

// Usage inside Tabs:
const [activeTab, setActiveTab] = useControllableState({
  value: props.value,
  defaultValue: props.defaultValue ?? firstTabId,
  onChange: props.onValueChange,
});
Pitfall: a component must not switch between controlled and uncontrolled during its lifetime — that's the source of React's famous warning. Decide based on whether value !== undefined on first render and treat a later flip as a caller bug (Radix and MUI both warn in dev when this happens). The most common way callers trip this: value={data?.name} where data starts as undefined — the input mounts uncontrolled, then becomes controlled when data arrives.

Rest props + ComponentPropsWithoutRef — the design-system button

A wrapper around a native element should accept everything the native element accepts — aria-*, data-*, onMouseEnter, form, type — without you enumerating them. TypeScript's ComponentPropsWithoutRef<'button'> gives you the full native prop set; spread the rest onto the element and merge the props you also use yourself: 

import { forwardRef, type ComponentPropsWithoutRef } from 'react';
import { clsx } from 'clsx';

type ButtonProps = ComponentPropsWithoutRef<'button'> & {
  variant?: 'primary' | 'secondary' | 'danger';
  size?: 'sm' | 'md' | 'lg';
  isLoading?: boolean;
};

export const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  function Button(
    { variant = 'primary', size = 'md', isLoading, className, children, disabled, ...rest },
    ref
  ) {
    return (
      <button
        ref={ref}
        type="button"                         // safe default: don't submit forms accidentally
        className={clsx('btn', `btn-${variant}`, `btn-${size}`, className)}
        disabled={disabled || isLoading}
        aria-busy={isLoading || undefined}
        {...rest}                             // caller can still override type, aria-*, handlers
      >
        {isLoading && <Spinner aria-hidden />}
        {children}
      </button>
    );
  }
);

Design decisions worth narrating in an interview:

  • Spread order matters. {...rest} after your own props means callers can override defaults (like type); before means they can't. Pick deliberately and be consistent — Radix spreads last so consumers win.
  • Destructure what you consume (className, disabled) so it doesn't get double-applied by the spread; merge rather than clobber (clsx(..., className)).
  • forwardRef so the button works with focus management, tooltips, and form libs. (In React 19, ref is a regular prop and forwardRef is no longer needed — mention this, it's a current-events point — but you'll maintain forwardRef code for years.)
  • ComponentPropsWithoutRef vs ComponentProps: the former excludes ref from the props type, which you want because forwardRef handles ref separately. ComponentPropsWithRef exists for the rare case you're typing something that passes ref through as a prop.

The polymorphic as prop

Design systems often let one component render as different elements: <Text as="label">, <Button as="a" href="...">. The typed sketch: 

import type { ComponentPropsWithoutRef, ElementType, ReactNode } from 'react';

type PolymorphicProps<E extends ElementType> = {
  as?: E;
  children?: ReactNode;
} & Omit<ComponentPropsWithoutRef<E>, 'as' | 'children'>;

function Text<E extends ElementType = 'span'>({ as, ...rest }: PolymorphicProps<E>) {
  const Component = as ?? 'span';
  return <Component {...rest} />;
}

// TS now knows href is valid here and invalid on the default span:
<Text as="a" href="/docs">Docs</Text>   // ✅
<Text href="/docs">Docs</Text>          // ❌ type error

Trade-offs to volunteer:

  • Fully general polymorphic types (with ref forwarding) get gnarly fast and are notoriously slow for the TS compiler in large design systems — some teams (and Radix, via its asChild pattern instead) moved away from as for exactly this reason.
  • asChild (Radix/Slot): instead of as="a", you pass the element as the child and the component merges its props onto it. Cleaner types, works with custom components like router <Link>, at the cost of a slightly odd mental model.
  • Semantics risk: <Button as="div"> silently loses keyboard and a11y behavior. A good API constrains as to sensible elements or re-adds behavior (role="button", tabIndex, key handlers) when it can't.

className / style escape hatches

Every styled component in a design system eventually meets a layout it didn't anticipate. Accepting className and style and merging them is the pressure-release valve that keeps consumers from wrapping your component in three divs or forking it. The senior nuance: escape hatches are for positioning and one-off spacing, not for re-theming — if consumers routinely override internals, your variants API is missing a case. Some teams intentionally omit className on leaf components to keep the design language closed; know that this is a deliberate spectrum (open like Radix ↔ closed like some internal DS) and name where you'd sit and why.

Interview framing: when asked "design a reusable X", spend the first two minutes on the contract before writing code: What's controlled/uncontrolled? What's composition vs props? What native props pass through? What are the a11y guarantees? Interviewers consistently rate candidates who lay out the API first far above those who start typing JSX.

2. Compound components — the flagship senior pattern T1

Compound components are a set of components that work together sharing implicit state, the way <select> and <option> do in HTML. The parent owns state and shares it via context; children read it. The payoff: the caller owns the markup — they can reorder, wrap, style, and interleave pieces — while the component owns the behavior. This is the single most-requested pattern in senior React interviews ("build Tabs where consumers control the layout").

Full implementation: Tabs / TabList / Tab / TabPanel

import {
  createContext, useContext, useId, useState, useCallback,
  type ReactNode,
} from 'react';

type TabsContextValue = {
  activeValue: string;
  setActiveValue: (v: string) => void;
  baseId: string;            // for wiring aria-controls/aria-labelledby
};

const TabsContext = createContext<TabsContextValue | null>(null);

function useTabsContext(component: string) {
  const ctx = useContext(TabsContext);
  if (!ctx) {
    // Fail loudly and helpfully — this error message is itself API design
    throw new Error(`<${component}> must be rendered inside <Tabs>`);
  }
  return ctx;
}

export function Tabs({
  value, defaultValue, onValueChange, children,
}: {
  value?: string;
  defaultValue?: string;
  onValueChange?: (v: string) => void;
  children: ReactNode;
}) {
  // Controlled + uncontrolled support (see useControllableState in §1)
  const [internal, setInternal] = useState(defaultValue ?? '');
  const isControlled = value !== undefined;
  const activeValue = isControlled ? value : internal;
  const baseId = useId();

  const setActiveValue = useCallback((v: string) => {
    if (!isControlled) setInternal(v);
    onValueChange?.(v);
  }, [isControlled, onValueChange]);

  return (
    <TabsContext.Provider value={{ activeValue, setActiveValue, baseId }}>
      {children}
    </TabsContext.Provider>
  );
}

export function TabList({ children, label }: { children: ReactNode; label: string }) {
  // Roving tabindex keyboard nav: ArrowLeft/Right move focus AND activate (APG "automatic activation")
  const onKeyDown = (e: React.KeyboardEvent<HTMLDivElement>) => {
    const tabs = Array.from(
      e.currentTarget.querySelectorAll<HTMLButtonElement>('[role="tab"]:not([disabled])')
    );
    const i = tabs.indexOf(document.activeElement as HTMLButtonElement);
    if (i === -1) return;
    let next = i;
    if (e.key === 'ArrowRight') next = (i + 1) % tabs.length;
    else if (e.key === 'ArrowLeft') next = (i - 1 + tabs.length) % tabs.length;
    else if (e.key === 'Home') next = 0;
    else if (e.key === 'End') next = tabs.length - 1;
    else return;
    e.preventDefault();
    tabs[next].focus();
    tabs[next].click();       // automatic activation on focus move
  };

  return (
    <div role="tablist" aria-label={label} onKeyDown={onKeyDown}>
      {children}
    </div>
  );
}

export function Tab({ value, children, disabled }: {
  value: string; children: ReactNode; disabled?: boolean;
}) {
  const { activeValue, setActiveValue, baseId } = useTabsContext('Tab');
  const selected = activeValue === value;
  return (
    <button
      role="tab"
      id={`${baseId}-tab-${value}`}
      aria-selected={selected}
      aria-controls={`${baseId}-panel-${value}`}
      tabIndex={selected ? 0 : -1}          // roving tabindex: one tab stop for the whole list
      disabled={disabled}
      onClick={() => setActiveValue(value)}
    >
      {children}
    </button>
  );
}

export function TabPanel({ value, children }: { value: string; children: ReactNode }) {
  const { activeValue, baseId } = useTabsContext('TabPanel');
  const selected = activeValue === value;
  return (
    <div
      role="tabpanel"
      id={`${baseId}-panel-${value}`}
      aria-labelledby={`${baseId}-tab-${value}`}
      hidden={!selected}
      tabIndex={0}                          // panel is focusable so keyboard users can reach content
    >
      {children}
    </div>
  );
}

Senior answer: they're not enemies — mature libraries often ship compound primitives and a convenience config wrapper built on top of them (<SimpleTabs items={...}/> internally renders the compound parts). Offering the low-level composable API plus a high-level ergonomic one is exactly how Radix (primitives) relates to shadcn/ui (prebuilt compositions). Also note: a config API is effectively mandatory when the component must own iteration — e.g. a virtualized tab strip with 10,000 tabs can't accept 10,000 child elements.

Static properties vs named exports

// Style A: static properties (dot notation)
Tabs.List = TabList; Tabs.Tab = Tab; Tabs.Panel = TabPanel;
<Tabs.Tab value="a">A</Tabs.Tab>

// Style B: named exports (what Radix does)
export { Tabs, TabList, Tab, TabPanel };

Dot notation reads nicely and namespaces the parts, but historically hurt tree-shaking (properties on a function object are hard for bundlers to prove unused) and confused React.lazy/HMR edge cases. Named exports tree-shake cleanly and play better with code splitting; most modern libraries chose them. Knowing why the ecosystem shifted is the differentiator, not which one you pick.

Handling non-direct children

A naive compound implementation that walks React.Children + cloneElement to inject props breaks the moment a consumer wraps a Tab in their own component or a fragment: 

<TabList label="…">
  {tabs.map(t => <Tab key={t.id} value={t.id}>{t.label}</Tab>)}  {/* array — fine with context */}
  <Tooltip content="Admins only">                                   {/* wrapper component */}
    <Tab value="admin">Admin</Tab>                              {/* cloneElement-based impls break here */}
  </Tooltip>
</TabList>

This is the argument for context over cloneElement: context reaches any depth, survives wrappers, fragments, and .map(), and doesn't create an implicit "I will secretly add props to you" contract. cloneElement-based compounds only work with direct children of an exact type — a fragility the React docs now explicitly warn about (§10 covers cloneElement's problems in detail).

Finish the compound-components answer with the escape hatch: export the context hook (a public useTabs()) alongside the parts. That converts "the library didn't anticipate my piece" from a fork into a feature — consumers can build their own <TabIndicator> or analytics-reporting tab on the same shared state. Compound components + a public context hook is the complete pattern; the shipped components are just the default consumers of that context.

3. Controlled vs uncontrolled forms T1

The input-level definition — say it precisely

For form elements the terms have an exact meaning. A controlled input has its value driven by React state: value={state} + onChange. The DOM input is a dumb display; React state is the single source of truth, and every keystroke round-trips through React (event → setState → re-render → new value painted). An uncontrolled input keeps its value in the DOM, where inputs have kept their value since 1995; React sets an initial defaultValue and reads the value on demand via a ref or FormData. The interview one-liner: controlled = React owns the value on every render; uncontrolled = the DOM owns it and you sample it. 

// Controlled — state is the source of truth
const [email, setEmail] = useState('');
<input value={email} onChange={(e) => setEmail(e.target.value)} />

// Uncontrolled — the DOM is the source of truth, sample when needed
const emailRef = useRef<HTMLInputElement>(null);
<input defaultValue="a@b.co" ref={emailRef} />
// later: emailRef.current!.value

// Uncontrolled at the FORM level — no refs at all, read everything at submit:
function onSubmit(e: React.FormEvent<HTMLFormElement>) {
  e.preventDefault();
  const data = new FormData(e.currentTarget);     // reads every named field
  api.post('/signup', Object.fromEntries(data));  // { email: '...', plan: '...' }
}
<form onSubmit={onSubmit}>
  <input name="email" type="email" required />   {/* name attributes are the API */}
  <select name="plan" defaultValue="pro">…</select>
</form>

What controlled buys you — and what it costs

You need controlled when the UI must react to the value as it changes: live validation messages, character counters, input masking/formatting (uppercase, phone formatting), enabling a button when a field is valid, filtering a list as the user types, or vetoing input (rejecting non-digits). None of that is possible by sampling at submit. The cost is re-renders: every keystroke re-renders the component that owns the state — and if that component is a 40-field form, every keystroke re-renders all 40 fields plus everything else in the component. On fast machines with small forms nobody notices; on a long form with expensive rows (selects with hundreds of options, rich validation UI) typing latency becomes visible, and interviewers know it because they've all felt it. 

// The classic scale problem:
function BigForm() {
  const [form, setForm] = useState(initial40Fields);
  const update = (k) => (e) => setForm(f => ({ ...f, [k]: e.target.value }));
  // EVERY keystroke in ANY field re-renders all 40 inputs + labels + errors.
  // Mitigations, in escalating order:
  //  1. memo() each field row and pass stable handlers (helps, adds ceremony)
  //  2. split state per field / per section (helps, loses single-object ergonomics)
  //  3. move the value out of React state entirely — uncontrolled — which is
  //     exactly what react-hook-form productizes (§4).
}

When uncontrolled + refs/FormData is the right call

  • Submit-only forms: login, search, contact, "add a comment" — nothing reacts per-keystroke, so paying per-keystroke renders buys nothing. FormData + name attributes + native validation attributes (required, type="email", minLength, pattern) covers a surprising amount for free, including a11y (the browser announces validation errors).
  • File inputs: <input type="file"> is uncontrolled by design — its value can't be set programmatically for security reasons. It's the proof that "everything controlled" was never a rule.
  • Integration with non-React code or progressive enhancement: a form that must also work when JS fails (or before hydration) needs names and a real action anyway — which is exactly where React 19 form actions pick up (below).
  • Performance-critical big forms — via react-hook-form rather than hand-rolled refs.

Mixed strategies — the pragmatic senior answer

The dichotomy is per-input, not per-form. A registration form can keep 18 fields uncontrolled and control the two that need live behavior: the password field (strength meter) and the username (async availability check). Similarly, "uncontrolled with notifications" is a legitimate middle: leave the value in the DOM but attach an onChange purely for side effects (analytics, dirty-tracking) without storing the value in state — onChange does not make an input controlled; providing value does. That distinction — controlled is defined by the value prop, not by having handlers — is a reliable knowledge check.

value without onChange makes a read-only input (React warns; the DOM ignores keystrokes). value={undefined} then value={"x"} later is the uncontrolled-to-controlled warning from §1 — endemic with async-loaded edit forms. Fix by not rendering the form until data exists, keying the form on the record id, or using a form library's reset(data). And never "fix" it with value={data?.name ?? ''} scattered across 30 fields when defaultValues + reset exists.

React 19 form actions — the calculus shifts back toward uncontrolled

React 19 made forms a first-class primitive again. You can pass a function to <form action>; React calls it with the form's FormData, resets the form on success (for uncontrolled inputs), and exposes pending state through hooks: 

function Signup() {
  // useActionState: wraps the action, returns [state, wrappedAction, isPending]
  const [result, submitAction, isPending] = useActionState(
    async (prev, formData: FormData) => {
      const res = await api.signup(Object.fromEntries(formData));
      if (!res.ok) return { error: res.message, values: Object.fromEntries(formData) };
      return { success: true };
    },
    null
  );

  return (
    <form action={submitAction}>
      <input name="email" type="email" required
             defaultValue={result?.values?.email} />   {/* repopulate on error */}
      {result?.error && <p role="alert">{result.error}</p>}
      <SubmitButton />
    </form>
  );
}

function SubmitButton() {
  // useFormStatus: reads the nearest parent <form>'s pending state — no prop drilling.
  // Must be rendered INSIDE the form, in a child component.
  const { pending } = useFormStatus();
  return <button disabled={pending}>{pending ? 'Signing up…' : 'Sign up'}</button>;
}

Points to make in an interview: (1) this model is uncontrolled-first — the action receives FormData, so name attributes matter again and per-keystroke state is unnecessary for plain submit flows; (2) in a server-components framework the same action can be a server function, giving progressive enhancement (the form posts and works before hydration); (3) useOptimistic pairs with actions for optimistic UI; (4) it does not replace form libraries for rich client validation, dependent fields, dirty tracking, or arrays — RHF and actions compose (RHF's handleSubmit inside a transition, or RHF validation before calling the action). The senior framing: form actions move submission and pending/error plumbing into React; field-level interaction state is still yours to manage — with the same controlled/uncontrolled judgment as before.

4. React Hook Form — the de-facto standard T1

react-hook-form (RHF) won the form-library war (Formik is in maintenance mode; Redux Form is a cautionary tale — form state in a global store, every keystroke a dispatched action). You're expected to know RHF's API and its architecture, because "why is RHF fast?" is the real question behind every RHF question.

The core: register = uncontrolled inputs + a subscription system

import { useForm } from 'react-hook-form';

type FormValues = { email: string; password: string };

function Login() {
  const { register, handleSubmit, formState: { errors, isSubmitting } } =
    useForm<FormValues>({ defaultValues: { email: '', password: '' } });

  const onSubmit = async (data: FormValues) => {   // runs only if validation passed
    await api.login(data);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)} noValidate>
      <label htmlFor="email">Email</label>
      <input id="email" type="email"
        aria-invalid={!!errors.email || undefined}
        {...register('email', {
          required: 'Email is required',
          pattern: { value: /^\S+@\S+$/, message: 'Enter a valid email' },
        })}
      />
      {errors.email && <p role="alert">{errors.email.message}</p>}

      <input type="password" {...register('password', { required: true, minLength: 8 })} />

      <button disabled={isSubmitting}>Log in</button>
    </form>
  );
}

What register('email', rules) actually returns: { name, ref, onChange, onBlur }. The ref lets RHF grab the DOM node and read input.value directly; the handlers update RHF's internal, non-React store of values and validation state. Typing does not call setState — the value lives in the DOM, RHF's store mirrors it, and React renders nothing. That is the entire performance story: RHF is fast because keystrokes bypass React rendering entirely. Re-renders happen only when something you subscribed to changes.

The subscription mechanism is the senior detail: formState is a Proxy. When you destructure formState.errors or read formState.isDirty, the proxy records that this component cares about that slice, and RHF re-renders it only when that slice changes. Read nothing → the form component re-renders roughly twice for an entire fill-and-submit session. This is also a footgun: because subscription is read-based, const { formState } = useForm() passed down and read conditionally can miss updates — destructure what you use at the top level.

Controller — the bridge to controlled UI libraries

register needs a real DOM input to attach a ref to. Component libraries (MUI's Select, react-select, date pickers, rich editors) expose value/onChange props instead — they're controlled components. Controller adapts them: it holds that one field's value in RHF's store and feeds it to the child as a controlled prop, isolating the re-render to that field: 

import { Controller, useForm } from 'react-hook-form';

<Controller
  name="country"
  control={control}
  rules={{ required: 'Pick a country' }}
  render={({ field, fieldState }) => (
    <Select
      {...field}                         // value, onChange, onBlur, ref, name
      options={countries}
      aria-invalid={fieldState.invalid || undefined}
    />
  )}
/>
// useController(name, control) is the hook form of the same thing — use it when
// building your own reusable form-field components instead of inlining render props.
The architecture takeaway to say out loud: RHF is uncontrolled by default, controlled per-field where required — the mixed strategy from §3, productized. Fields under Controller re-render on their own changes; registered fields never do. That's why a 60-field RHF form with two Controller-wrapped selects stays fast.

  age: z.coerce.number().int().min(18, 'Must be 18+'),   // coerce: inputs give strings!

  message: 'Passwords must match',
  path: ['confirm'],                       // attach the error to a field, not the root

5. Form architecture beyond RHF T2

Library fluency gets you through the syntax questions; the differentiator is the architecture questions — "design a 5-step onboarding wizard", "how do server errors reach the right field", "make this form accessible". These are small system-design rounds and they're where seven years of experience should show.

Multi-step wizards — the two decisions that matter

Decision 1: where does accumulated data live? Options, in order of preference:

  • One useForm for the whole wizard, steps as views. All fields registered in one form (unmounted steps kept via shouldUnregister: false, the default); the step is just UI state. Simplest data flow, free whole-form validation at the end, dirty tracking across steps. Works until steps are truly independent forms or the wizard is huge.
  • One form per step + an accumulator. Each step is a self-contained useForm + per-step zod schema; on step submit, merge into a parent-owned accumulator (useReducer, or a small Zustand store persisted to sessionStorage so refresh resumes — see tutorial 11's wizard exercise). Cleaner isolation, per-step code splitting, steps testable alone.
  • Server-persisted draft. Each step submit PATCHes a draft resource. Mandatory when steps must survive devices/sessions or when later steps depend on server computation (pricing, eligibility). The wizard becomes a sequence of small forms over server state — often the honest answer for enterprise onboarding.

Decision 2: per-step validation. Validate only the current step's fields before advancing: with one big form, call await trigger(['email', 'password']) (RHF validates just those names) or give each step its own schema; never run the full-form schema on step 1 — you'll show errors for fields the user hasn't seen. Guard deep links: landing on step 3 with no step-1 data redirects to the first incomplete step (derivable from the accumulator — don't store a separate "completed steps" flag that can drift).

Server errors → fields

Client validation is UX; the server revalidates and returns its own errors ("email already taken", "coupon expired"). The pattern: map the server's error payload onto fields with setError, reserving a root error for non-field failures: 

const onSubmit = async (data: FormValues) => {
  const res = await api.signup(data);
  if (!res.ok) {
    // Expected payload: { errors: { email: 'Already registered', ... } }
    for (const [field, message] of Object.entries(res.errors ?? {})) {
      if (field in data) {
        setError(field as keyof FormValues, { type: 'server', message });
      }
    }
    if (res.formError) {
      setError('root.serverError', { message: res.formError });  // not tied to a field
    }
    return;
  }
};

{errors.root?.serverError && <p role="alert">{errors.root.serverError.message}</p>}
Two refinements interviewers reward: (1) treat the server's field names as an API contract — if the backend returns user_email and the form has email, put the mapping in one translation function, not scattered ifs; (2) decide when server errors clear — RHF clears a field's error when its rules re-pass, but a type: 'server' error has no client rule, so clear it manually on change of that field (or rely on reValidateMode + a schema that can't re-validate uniqueness — meaning you clear it optimistically and let the next submit re-check).

Admin builders, CMS-driven surveys, per-tenant fields: the form isn't known at build time. The standard shape — a schema array driving both rendering and validation:

Form accessibility — the checklist that gets graded

  • Every input has a label — <label htmlFor> (or aria-label as a last resort). Placeholder is not a label: it vanishes on input and fails contrast.
  • Errors are linked, not just adjacent: aria-invalid on the input + aria-describedby pointing at the error element's id, so screen readers announce the message when the field is focused.
  • Announce errors when they appear: role="alert" (or aria-live="assertive") on error messages/summary so they're read without moving focus.
  • Focus management on failed submit: move focus to the first invalid field (RHF does this by default for registered inputs — shouldFocusError: true) or to an error-summary box at the top that links to each field (the GOV.UK pattern — the strongest answer for long forms).
  • Group related controls: <fieldset> + <legend> for radio groups and address blocks; the legend is announced with each field inside.
  • Don't disable the submit button as your only invalid-state signal — a disabled button gives no feedback about why. Prefer enabled submit → validate → announce + focus errors.

6. Error boundaries T1

What they are — and why they're still classes

An error boundary is a component that catches errors thrown during rendering, in lifecycle methods, and in constructors of its child tree, and renders a fallback instead of unmounting the whole app (React's default since 16: an uncaught render error unmounts the entire root — a blank page). Boundaries are the only remaining must-write-a-class API: there is still no hook equivalent of getDerivedStateFromError/componentDidCatch as of React 19 (the team has said hooks for this may come; interviewers like hearing that you know it's a known gap, which is why everyone uses the react-error-boundary package instead of hand-rolling). 

class ErrorBoundary extends React.Component<
  { fallback: React.ReactNode; children: React.ReactNode },
  { hasError: boolean }
> {
  state = { hasError: false };

  // Render phase: derive fallback state from the error. Must be PURE — no side effects.
  static getDerivedStateFromError(_error: unknown) {
    return { hasError: true };
  }

  // Commit phase: side effects allowed — log to Sentry/Datadog here.
  componentDidCatch(error: unknown, info: React.ErrorInfo) {
    logger.error(error, { componentStack: info.componentStack });
  }

  render() {
    return this.state.hasError ? this.props.fallback : this.props.children;
  }
}

Why two methods: getDerivedStateFromError runs during render so React can paint the fallback in the same pass (important under concurrent rendering — pure, may be retried); componentDidCatch runs during commit, so it's the safe place for logging side effects and it receives componentStack. Use both, for their respective jobs.

What error boundaries do NOT catch — the guaranteed follow-up

Not caughtWhyWhat to do instead
Event handlers They don't run during render — React has already painted; the error can't corrupt the tree Ordinary try/catch; or showBoundary(err) to route it into a boundary deliberately
Async code (setTimeout, promise rejections, fetch callbacks) The call stack when it throws is not React's render stack Catch and set state; or rethrow inside render (the trick query libraries use — below)
SSR Class lifecycle differs on the server; server render errors are handled by the framework/stream (React 18 streaming SSR can degrade a Suspense boundary's subtree to client rendering) Framework error handling (Next.js error.tsx covers both via its own boundary + server plumbing)
The boundary itself A boundary can't catch its own render errors The error propagates to the next boundary up — keep fallbacks dead simple so they can't throw 
// The async trick: an error boundary CAN catch async errors if you re-throw in render.
function useAsyncError() {
  const [, setState] = useState();
  return useCallback((err: unknown) => {
    setState(() => { throw err; });     // throws during the next render → boundary catches
  }, []);
}

Granularity strategy — where do boundaries go?

"One boundary at the root" and "a boundary around everything" are both wrong. The senior answer is three deliberate levels:

  • Root/app level — last resort, full-page "something went wrong" + reload action. Should almost never be seen.
  • Route/page level — each route wrapped (Next.js error.tsx, or a boundary in your route config). A broken page doesn't take down navigation, header, or other tabs the user has context in.
  • Widget level — around independent, failure-prone islands: third-party embeds, charts over unreliable data, comments sections, anything lazy-loaded. The product question that decides placement: "if this crashes, what's the largest UI area that is genuinely useless?" — that's your boundary. A crashed related-products carousel should not blank a checkout page.

react-error-boundary — the production idioms

import { ErrorBoundary } from 'react-error-boundary';

<ErrorBoundary
  fallbackRender={({ error, resetErrorBoundary }) => (
    <div role="alert">
      <p>Couldn't load the dashboard.</p>
      <pre>{error.message}</pre>
      <button onClick={resetErrorBoundary}>Try again</button>
    </div>
  )}
  onError={(error, info) => logger.error(error, info)}   // centralized reporting
  onReset={() => refetch()}                              // fix the cause before re-render
  resetKeys={[userId, filters]}                          // auto-reset when inputs change
>
  <Dashboard userId={userId} filters={filters} />
</ErrorBoundary>

The two idioms that make retries actually work: onReset runs when resetErrorBoundary() is called — this is where you fix the underlying cause (refetch, clear a cache entry, reset local state); resetting without fixing the cause just re-throws instantly. resetKeys auto-resets the boundary when listed values change — the user navigates to a different userId, so the error state for the old one is no longer relevant. Without it, boundaries "stick": the tree stays in fallback even though the failing input is gone.

Pairing with Suspense — the declarative loading/error grammar

Don't wrap every component "for safety." Each boundary is a place the UI can silently degrade; too many and real bugs hide in half-rendered pages nobody reports. Boundaries are a product decision about failure blast radius, plus mandatory reporting in onError — a fallback without telemetry is a bug muffler.

7. Portals T2

createPortal — render there, behave here

import { createPortal } from 'react-dom';

function Modal({ children, onClose }: { children: React.ReactNode; onClose: () => void }) {
  return createPortal(
    <div className="overlay" onClick={onClose}>
      <div className="dialog" onClick={(e) => e.stopPropagation()}>
        {children}
      </div>
    </div>,
    document.body           // physically mounted here, logically still a child of Modal's parent
  );
}

A portal renders children into a different DOM container while keeping them in the same React tree. Everything React-level flows as if the portal weren't there: context is read from the React parent, state and lifecycles are unaffected, and — the flagship interview fact — events bubble through the React tree, not the DOM tree. React attaches its listeners at the root and dispatches synthetic events along the component hierarchy, so a click inside a portal fires onClick on the portal's React ancestors even though the DOM node lives in document.body. 

<div onClick={() => console.log('React parent sees this')}>   {/* fires! */}
  <Modal><button>Inside portal</button></Modal>
</div>
// But a NATIVE listener on that div's DOM node does NOT fire —
// the DOM ancestor of the button is document.body. React tree ≠ DOM tree.
// Practical bite: "close on outside click" via a document mousedown listener will

// panelDomNode.contains(e.target) against the trigger's subtree. Check containment

Why portals exist: overflow, stacking contexts, z-index

Modals, dropdowns, tooltips, and toasts must visually escape their container. Rendered in place, they get clipped by any ancestor with overflow: hidden/auto, and — the deeper problem — trapped by stacking contexts: an ancestor with transform, filter, opacity < 1, or will-change creates a stacking context, and no z-index inside it can escape above content outside it. z-index: 999999 doesn't help; it's a hierarchy problem, not a number problem. Portaling to document.body exits every ancestor stacking context and clip — that's the whole trick. (Corollary: a transform on body or an app-wide wrapper breaks even portaled position: fixed children — worth mentioning as a war story.)

Focus management + inert — where modal implementations get graded

Portals move DOM, not focus. A production modal must handle:

  • Initial focus into the dialog on open (first focusable element, or the dialog itself with tabIndex={-1} for reading order).
  • Focus trap: Tab/Shift+Tab cycle within the dialog — because the portal sits at the end of body, tabbing past its last element otherwise dumps focus into the browser chrome or the page behind.
  • Return focus to the triggering element on close (store document.activeElement on open).
  • Escape closes, overlay click closes (product decision), and background scroll is locked.
  • Background made inert: aria-hidden alone hides the background from screen readers but not from Tab. The modern answer is the inert attribute on the app root while the modal is open — unfocusable and invisible to assistive tech in one attribute (broad browser support since 2023). This is what libraries polyfilled for years with focus-trap walls.
useEffect(() => {
  if (!open) return;
  const appRoot = document.getElementById('root')!;
  const previouslyFocused = document.activeElement as HTMLElement | null;
  appRoot.inert = true;                        // background: unfocusable + hidden from AT
  document.body.style.overflow = 'hidden';     // scroll lock
  dialogRef.current?.focus();
  return () => {
    appRoot.inert = false;
    document.body.style.overflow = '';
    previouslyFocused?.focus();                // return focus — the step everyone forgets
  };
}, [open]);
Two current-events points: the native <dialog> element with showModal() gives focus trapping, Escape handling, ::backdrop, and top-layer rendering (above all stacking contexts — no portal needed) for free, and the Popover API covers lighter-weight overlays; know they exist and why libraries still wrap them (styling control, animation coordination, older-browser support). And toasts: a single portal container with aria-live="polite" so announcements happen without focus theft — focus-stealing toasts are an a11y bug interviewers love to probe.

8. Legacy pattern literacy: HOC → render props → hooks T3

All three patterns answer the same question — how do we share stateful logic between components? — and the progression is a favorite knowledge-check because it tests whether you understand why the ecosystem moved, not just that it did. You'll also read all three in any codebase older than ~2019.

Higher-order components

// A HOC is a function: Component in → enhanced Component out.
function withAuth<P extends object>(Wrapped: React.ComponentType<P & { user: User }>) {
  return function WithAuth(props: P) {
    const user = useCurrentUser();               // (modern HOCs use hooks inside)
    if (!user) return <LoginRedirect />;
    return <Wrapped {...props} user={user} />;   // inject props
  };
}

const AccountPage = withAuth(AccountPageInner);
// Pre-hooks equivalents: connect(mapState, mapDispatch)(Component), withRouter(Component)

Its problems, each worth naming precisely:

  • Wrapper hell: withAuth(withTheme(withRouter(connect(...)(Page)))) — deep devtools trees, and composition order can matter invisibly.
  • Prop collisions: injected props share one flat namespace. Two HOCs injecting data, or an injected user clobbering a real prop — silent, no type error in the JS era.
  • Indirection: reading the component tells you nothing about where user comes from; you must find the wrapping call.
  • Static hoisting & refs: the wrapper doesn't automatically carry the inner component's statics (hence the hoist-non-react-statics package) and swallows refs (hence forwardRef plumbing in every serious HOC).
  • Don't call inside render: withAuth(Foo) inside a component creates a new type per render → full remount of that subtree every time. HOCs are module-scope decorations.

Render props

// Share logic by passing a FUNCTION that receives the state and returns UI.
class MouseTracker extends React.Component {
  state = { x: 0, y: 0 };
  handleMove = (e) => this.setState({ x: e.clientX, y: e.clientY });
  render() {
    return <div onMouseMove={this.handleMove}>{this.props.children(this.state)}</div>;
  }
}

<MouseTracker>
  {({ x, y }) => <Crosshair x={x} y={y} />}     {/* explicit — no hidden injection */}
</MouseTracker>

Render props fixed HOCs' worst sins — the data flow is explicit (you see exactly what you receive and name it yourself, so no collisions) and composition happens at usage site, dynamically. The new problem: consuming several sources nests into a pyramid of closures ("callback hell, JSX edition"), and everything below the render prop re-renders whenever the provider updates, with awkward memoization ergonomics.

Why hooks won

// The same logic as a hook: flat, explicit, composable, typed.
function useMouse() {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  useEffect(() => {
    const onMove = (e: MouseEvent) => setPos({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', onMove);
    return () => window.removeEventListener('mousemove', onMove);
  }, []);
  return pos;
}

function Page() {
  const { x, y } = useMouse();     // three sources = three lines, not three wrappers
  const user = useAuth();
  const theme = useTheme();
}

Hooks share logic without imposing structure: no extra components in the tree, no namespace to collide (you name the return value), values traceable to an import, composition is sequential lines instead of nesting, and TypeScript inference is natural. The cost — "rules of hooks" (top-level only, consistent order) — exists precisely because hooks rely on call order for state identity, which is the trade the ecosystem accepted.

When you still see HOCs and render props — legitimately

  • React.memo is a HOC. So is forwardRef (pre-19). The shape never died — component-in, component-out is still how you decorate a component wholesale.
  • Cross-cutting wrappers that gate rendering: withAuth/withPermission around route components remain reasonable — a hook can't prevent a component from rendering; a wrapper can. (Error boundaries, similarly, are structural and often applied as withErrorBoundary(Widget).)
  • Render props survive where UI depends on values only the component can provide per-render: Downshift's prop getters, TanStack Virtual's row callbacks, <Formik>/Controller's render — a render prop is just "component-as-prop with arguments" (§10) and is still idiomatic.
  • Legacy migration reading: connect → useSelector/useDispatch; withRouter → useNavigate/useParams; a class + render prop logic provider → a custom hook. In interviews about legacy codebases, propose incremental strangler migration, never a rewrite.

9. The headless UI pattern T2

Logic and accessibility without styles

A headless component ships behavior, state, and accessibility — and no visual opinion. The insight behind the pattern: in a Select or Combobox, the styling is the easy 20% and the hard 80% is invisible — keyboard interaction per the WAI-ARIA Authoring Practices, focus management, ARIA wiring, type-ahead, portal positioning, touch/screen-reader quirks. Headless libraries let you buy the hard 80% and own the 20% that differentiates your product. The major players, on a spectrum of "how much do they render":

LibraryFormNotes
Radix Primitives Unstyled compound components Renders real DOM with ARIA + keyboard done; style via className/data-attributes; asChild for composition. The base of shadcn/ui.
Headless UI Unstyled components (render-prop state) Tailwind Labs; exposes state like { open, active } to className callbacks.
React Aria Hooks (+ newer components) Adobe; maximum control — useButton, useComboBox return prop bags you spread onto your own JSX; deepest a11y/i18n rigor.
TanStack Table Hook, renders nothing Pure table state machine (sorting, filtering, grouping, pagination); you render every <td>. Fully headless — works with divs, canvas, anything.

Why design systems build on headless bases rather than forking a styled kit: a styled library (MUI, Ant) bakes in a design language you then fight with overrides — every override is a fragile coupling to its internal DOM. Headless flips the ownership: your design system owns 100% of the visuals and none of the behavior maintenance. The trade-offs to volunteer: more assembly required (you write the styled layer), bundle scrutiny per primitive, and you inherit the library's compound API shape as your API's foundation.

The state reducer pattern — inversion of control, final form

Prop like onChange lets callers observe state changes; a state reducer lets callers intercept and modify them. The component runs every proposed state transition through a caller-supplied reducer, so consumers can veto or alter behavior the author never anticipated — without new props: 

// Inside the headless component (Downshift-style):
function useToggle({ stateReducer = (s, a) => a.changes } = {}) {
  const [state, dispatch] = useReducer((state, action) => {
    const changes = toggleReducer(state, action);          // 1. default logic proposes
    return stateReducer(state, { ...action, changes });    // 2. caller gets final say
  }, { on: false });

  return {
    on: state.on,
    toggle: () => dispatch({ type: 'toggle' }),
    setOff: () => dispatch({ type: 'setOff' }),
  };
}

// Consumer: "clicking outside should NOT close the menu while an item is being dragged"
// — a requirement the author never imagined, solved with zero library changes:
useSelect({
  stateReducer: (state, action) => {
    if (action.type === 'blur' && isDragging) {
      return { ...action.changes, isOpen: state.isOpen };  // override just that field
    }
    return action.changes;                                  // everything else: default
  },
});
Present the escalation ladder of control APIs: onChange (observe) → controlled props (own one value) → state reducer (intercept every transition). Each step hands the consumer more control and costs more API surface — action types become public API you must version. The state reducer is the "ultimate inversion of control" answer, and citing Downshift as its origin (and noting Kent Dodds designed it after drowning in feature-request props) shows you know the lineage, not just the trick.

Naming which of the three a design calls for — and why — is a very cheap way to sound senior in a machine-coding round.

cloneElement — the problems in detail (the §2 promise)

cloneElement(child, newProps) copies an element with merged props. Legacy compound components used it to inject state into children. The React docs now mark it "uncommon and can lead to fragile code" — here is the precise indictment:

  • Implicit contract. The child receives props nobody passed in the visible JSX. <Tab> works only because a parent secretly injects isActive/onSelect — unreadable at the call site, undiscoverable in types (the injected props must be optional in the child's props type, weakening it for every other consumer).
  • Direct-child fragility. The parent walks React.Children — one level deep. Wrap a Tab in a Tooltip, a fragment, or your own <UpgradeTab /> abstraction and injection hits the wrapper instead of the Tab. Consumers discover this as "my component broke when I refactored my own JSX", the worst kind of API violation. (Also: filtering children by child.type === Tab breaks across module duplication, HMR, and wrapping in memo.)
  • Merge hazards. cloneElement shallow-merges: injected props overwrite same-named child props. Both parent and consumer set onClick or className? One silently wins — correct merging (compose handlers, concat classNames, merge styles) is manual work cloneElement doesn't do.
  • Refs and keys. ref and key aren't ordinary props; preserving a consumer's ref while attaching your own requires deliberate ref-composition. Naive clones drop or clobber refs.

Modern alternatives, by use case:

  • Sharing state with descendants → context (compound components, §2): any depth, explicit consumption via hook, type-safe, survives wrappers.
  • Rendering as a different element while keeping behavior → the asChild/Slot pattern (Radix): the component renders its child as its root and merges props correctly — event handlers composed (both run), classNames concatenated, styles merged, refs composed. Same power as cloneElement, but opt-in, single-child, with defined merge semantics:
// Radix Slot in essence:
import { Slot } from '@radix-ui/react-slot';

function Button({ asChild, ...props }: ButtonProps & { asChild?: boolean }) {
  const Comp = asChild ? Slot : 'button';
  return <Comp {...props} />;      // Slot merges props ONTO the child element
}

// Consumer: Button's classes/handlers land on the router Link — one DOM node, no nesting:
<Button asChild>
  <Link to="/settings">Settings</Link>
</Button>
  • Per-datum rendering → render callbacks, and caller-owned markup → children — both make the data flow visible instead of injected.
Interview sequencing tip: if asked "how did libraries build compound components before context was cheap?", describe cloneElement fairly (it worked, and reading it is still required literacy), then pivot to why context + Slot replaced it. Dismissing legacy patterns without being able to operate them reads as shallow.

Build 1 — Accordion (compound, single or multiple expand, full keyboard)

import { createContext, useContext, useId, useState, type ReactNode } from 'react';

type AccordionCtx = {
  openItems: Set<string>;
  toggle: (v: string) => void;
  baseId: string;
};
const Ctx = createContext<AccordionCtx | null>(null);
const useAccordion = (part: string) => {
  const ctx = useContext(Ctx);
  if (!ctx) throw new Error(`<${part}> must be inside <Accordion>`);
  return ctx;
};

export function Accordion({
  children, multiple = false, defaultOpen = [],
}: { children: ReactNode; multiple?: boolean; defaultOpen?: string[] }) {
  const [openItems, setOpen] = useState(() => new Set(defaultOpen));
  const baseId = useId();

  const toggle = (value: string) =>
    setOpen((prev) => {
      const next = new Set(prev);
      if (next.has(value)) next.delete(value);
      else {
        if (!multiple) next.clear();          // single mode: close others
        next.add(value);
      }
      return next;                            // new Set: state must be a new reference
    });

  return <Ctx.Provider value={{ openItems, toggle, baseId }}>{children}</Ctx.Provider>;
}

export function AccordionItem({ value, title, children }: {
  value: string; title: ReactNode; children: ReactNode;
}) {
  const { openItems, toggle, baseId } = useAccordion('AccordionItem');
  const open = openItems.has(value);
  const headerId = `${baseId}-${value}-header`;
  const panelId  = `${baseId}-${value}-panel`;

  return (
    <div className="accordion-item">
      <h3>                                    {/* heading wrapper: real document outline */}
        <button
          id={headerId}
          aria-expanded={open}
          aria-controls={panelId}
          onClick={() => toggle(value)}
        >
          {title}
          <span aria-hidden="true">{open ? '−' : '+'}</span>
        </button>
      </h3>
      <div id={panelId} role="region" aria-labelledby={headerId} hidden={!open}>
        {children}
      </div>
    </div>
  );
}

Points to narrate while building:

  • A real <button> inside a heading gives Enter/Space activation, focusability, and screen-reader semantics for free — a div onClick is the instant-fail version. aria-expanded on the trigger + aria-controls to the panel is the APG accordion contract; keyboard is just Tab between triggers (roving arrows are optional for accordions, required for tabs).
  • Set unifies single/multiple modes — single mode is "clear before add", not a different state shape. hidden keeps closed panels in the DOM (find-in-page, no remount cost); conditional rendering is the right call instead when panels are heavy — say the trade-off out loud.
  • Extensions to offer unprompted: controlled mode (value/onValueChange via useControllableState), disabled items, animation (grid-rows or max-height trick, since height: auto doesn't transition).

Build 2 — Modal (portal + focus trap + scroll lock)

import { useEffect, useRef, type ReactNode } from 'react';
import { createPortal } from 'react-dom';

const FOCUSABLE =
  'a[href], button:not([disabled]), input:not([disabled]), select, textarea, [tabindex]:not([tabindex="-1"])';

export function Modal({ open, onClose, title, children }: {
  open: boolean; onClose: () => void; title: string; children: ReactNode;
}) {
  const dialogRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    if (!open) return;
    const previouslyFocused = document.activeElement as HTMLElement | null;
    const dialog = dialogRef.current!;

    // Initial focus: first focusable inside, else the dialog itself
    const focusables = dialog.querySelectorAll<HTMLElement>(FOCUSABLE);
    (focusables[0] ?? dialog).focus();

    const onKeyDown = (e: KeyboardEvent) => {
      if (e.key === 'Escape') { e.stopPropagation(); onClose(); return; }
      if (e.key !== 'Tab') return;
      // Focus trap: wrap Tab at the edges
      const list = dialog.querySelectorAll<HTMLElement>(FOCUSABLE);
      if (list.length === 0) return;
      const first = list[0], last = list[list.length - 1];
      if (e.shiftKey && document.activeElement === first) { e.preventDefault(); last.focus(); }
      else if (!e.shiftKey && document.activeElement === last) { e.preventDefault(); first.focus(); }
    };

    document.addEventListener('keydown', onKeyDown);
    document.body.style.overflow = 'hidden';                 // scroll lock
    return () => {
      document.removeEventListener('keydown', onKeyDown);
      document.body.style.overflow = '';
      previouslyFocused?.focus();                            // return focus
    };
  }, [open, onClose]);

  if (!open) return null;

  return createPortal(
    <div className="overlay" onMouseDown={(e) => { if (e.target === e.currentTarget) onClose(); }}>
      <div
        ref={dialogRef}
        role="dialog"
        aria-modal="true"
        aria-labelledby="modal-title"
        tabIndex={-1}
        className="dialog"
      >
        <h2 id="modal-title">{title}</h2>
        {children}
        <button onClick={onClose} aria-label="Close dialog">×</button>
      </div>
    </div>,
    document.body
  );
}
  • The five behaviors that make or break it (recite before coding): initial focus in, focus trapped, focus returned on close, Escape closes, background scroll locked. role="dialog" + aria-modal="true" + a label; add inert on the app root (§7) as the modern upgrade over the Tab-wrap trap.
  • Overlay close uses e.target === e.currentTarget on mousedown — the classic bug is closing when a click starts inside the dialog (text selection) and ends on the overlay; mousedown-based checks avoid it.
  • Mention without building: stacking multiple modals (a manager stack; Escape closes topmost), the native <dialog>.showModal() alternative and why you're hand-rolling (the exercise), and unmount vs hidden for the closed state (unmount resets form state — often desirable).

export function StarRating({

Attack notes — Autocomplete and OTP input

Autocomplete/Typeahead (the hardest of the set — usually a strong-senior filter):

  • State: query, results, isOpen, activeIndex (keyboard highlight). Derived: nothing stored that can be computed.
  • Async discipline is the real test: debounce (~300ms), stale-response guard (AbortController or a request-id check — the "results for 'par' overwrite results for 'paris'" bug is what they're fishing for), empty-query short-circuit, loading/empty/error states.
  • Keyboard: ArrowDown/Up move activeIndex (with wrap), Enter selects, Escape closes, input keeps DOM focus the whole time — highlight is virtual (aria-activedescendant pointing at the active option's id), not real focus movement.
  • ARIA combobox pattern: input has role="combobox", aria-expanded, aria-controls; the list is role="listbox" with role="option" children + aria-selected. Outside-click closes (mousedown listener; remember the portal containment caveat from §7 if the list is portaled).

OTP input (6 boxes, one digit each):

  • State: a single string (or array) of length ≤ 6 — not six useStates. Each box renders one char; one source of truth makes paste and delete tractable.
  • Behaviors graded: typing a digit advances focus; Backspace on an empty box focuses the previous box and clears it; paste of "492817" distributes across boxes (handle onPaste on the container, slice to length, focus the next empty); arrow keys navigate; non-digits rejected (inputMode="numeric", pattern="[0-9]*", filter in the handler).
  • A11y: each input labeled ("Digit 1 of 6"), or — the pragmatic alternative worth mentioning — one visually segmented single input with autocomplete="one-time-code" so SMS autofill works; a hidden combined value for the form either way.
  • Refs: an array of refs (refs.current[i]) for focus control — a natural place to show you know callback refs.
Round management, which is graded even though it isn't on the rubric: restate requirements and get agreement on scope ("controlled or uncontrolled? multiple open at once?"); state your plan and state shape before typing; build in vertical slices that always run (static markup → state → keyboard → a11y polish); test in the browser as you go, don't debug a 200-line dump at minute 40; and when time runs out, spend the last two minutes listing what you'd do next (animation, virtualization, RTL, tests) — a prioritized backlog is senior signal, silence is not.

Interview Q&A

Q1. "What are compound components and when would you choose them over a props-based API?"

A set of components sharing implicit state through context — parent owns state, children consume it — modeled on <select>/<option>. Choose them when consumers need to own the markup: reorder parts, interleave arbitrary elements, style structurally. A configuration API (items prop) wins when rendering is data-driven or the component must own iteration (virtualized lists). They're not exclusive: libraries ship compound primitives plus a convenience wrapper built on them. I'd also mention the invariant trade: config APIs can enforce structure; compounds trade that for freedom, so guard with a context-missing error that names the fix ("Tab must be inside Tabs").

Q2. "Why implement compound components with context instead of React.Children + cloneElement?"

cloneElement injection only reaches direct children of the expected type — wrap a Tab in a Tooltip, a fragment, or the consumer's own abstraction and the injected props hit the wrapper, silently breaking. It's also an implicit contract (props appear that no JSX passed), shallow-merges so same-named props clobber (onClick, className), and mishandles refs/keys. Context reaches any depth, survives wrappers and .map(), is consumed explicitly via a hook, and types cleanly. The React docs themselves now flag cloneElement as fragile and suggest context, render props, or plain data instead.

Q3. "Define controlled vs uncontrolled inputs precisely. What makes an input controlled?"

The value prop — not the presence of handlers. value={state} + onChange makes React state the source of truth: the DOM displays whatever the last render said, and every keystroke round-trips through setState. defaultValue (or nothing) leaves the value in the DOM; you sample it via ref or FormData. An onChange without value is still uncontrolled — handlers are notifications, not control. Corollaries: value without onChange is read-only (React warns), and value flipping between undefined and defined across renders triggers the controlled/uncontrolled switch warning — usually async data feeding value={data?.field}.

Q4. "When is an uncontrolled form the right engineering call?"

When nothing needs to react per-keystroke: login, search, contact forms — read everything at submit with FormData and name attributes, lean on native validation attributes, and pay zero re-renders while typing. File inputs are uncontrolled by necessity. Progressive-enhancement forms (must work pre-hydration) need names and a real action anyway, which is the model React 19 form actions embrace. Controlled earns its cost only for live behavior: masking, character counters, dependent fields, filter-as-you-type, veto logic. And the dichotomy is per-input — a mostly-uncontrolled form with two controlled fields is often the right answer, which is exactly RHF's architecture.

Q5. "Why is react-hook-form fast? What does register actually do?"

register(name, rules) returns { name, ref, onChange, onBlur }. The ref gives RHF the DOM node; values stay in the DOM (uncontrolled), and the handlers update RHF's internal non-React store — no setState per keystroke, so React renders nothing while you type. Re-renders are subscription-based: formState is a Proxy that records which slices you actually read (errors, isDirty…) and re-renders only when those change. Contrast with Formik's model — form values in React state, whole form re-renders per keystroke. The escape valves keep the model honest: Controller makes one field controlled for UI libraries, useWatch scopes reactive reads to a child component, getValues reads without subscribing.

Q6. "watch vs useWatch vs getValues — when does each re-render?"

watch('field') subscribes the component that called useForm — typically the whole form — so every change of that field re-renders everything; fine for small forms, a self-inflicted wound in big ones. useWatch({ control, name }) subscribes only the component calling it, so you isolate the reactive UI (price summary, conditional section) in a small child that re-renders alone. getValues() never subscribes — a point-in-time read for event handlers and submit logic. The pattern: render from useWatch in leaves, decide from getValues in handlers, and reserve watch for genuinely form-wide reactions.

Q7. "Why schema validation with zod over inline rules, and how do types flow?"

A zod schema is a value: one definition yields the validation rules (via zodResolver), the TypeScript type (z.infer<typeof schema> feeding useForm<T>), and a server-side validator for the same payload — client validation is UX, server validation is enforcement, and a shared schema keeps them from drifting. It also centralizes cross-field rules (refine with a path to attach the error to a field) and coercion (z.coerce.number() — inputs emit strings). Inline register rules scatter logic through JSX, can't be reused server-side, and don't produce a type. The one-liner: schema-first means the form's contract is data, not markup.

Q8. "Design a 5-step signup wizard. Where does state live and how does validation work?"

Decide two things. Data location: either one useForm spanning all steps (steps are views; simplest, good for short wizards) or per-step forms merging into an accumulator (useReducer/small store, sessionStorage-persisted so refresh resumes) — and if drafts must survive devices or later steps depend on server computation, a server-persisted draft resource. Validation: per-step — either trigger([...stepFields]) on the single form or a zod schema per step; never full-form validation on step 1. Guard deep links by deriving the furthest-reachable step from accumulated data and clamping. The step indicator goes in the URL; the form data never does. Submission state is a state ('submitting'), not a boolean, so double-submit dies structurally.

Q9. "The server rejects a submitted form. How do errors reach the right fields?"

Map the server's error payload with setError(field, { type: 'server', message }), translated through one function if backend names differ from form names, plus setError('root.serverError') for failures that belong to no field, rendered near the submit button with role="alert". Two subtleties: server errors have no client rule to clear them, so clear on change of that field (or on next submit); and focus management — send focus to the first server-errored field or an error summary, same as client validation. Also validate the same schema server-side regardless: client checks are UX, the server is the enforcement point, and "email taken" can only ever be known there.

Q10. "What do React 19 form actions and useActionState/useFormStatus change?"

<form action={fn}> accepts a function receiving FormData; useActionState wraps it returning [state, wrappedAction, isPending] for error/result plumbing; useFormStatus lets any child of the form read pending state without prop drilling (submit buttons especially). The calculus shift: submission, pending, and error state move into React's primitive, and the model is uncontrolled-first — FormData and name attributes, not per-keystroke state. In RSC frameworks the action can be a server function, giving progressively-enhanced forms that work before hydration. What it doesn't replace: rich client-side field interaction — live validation, dependent fields, arrays, dirty tracking — which stays RHF territory; they compose rather than compete.

Q11. "What can't error boundaries catch, and what do you do about each gap?"

Four gaps. Event handlers — they run after render, so use try/catch, or route into a boundary deliberately with showBoundary. Async code (timeouts, promises, fetch callbacks) — the throw isn't on React's render stack; catch it and either set error state or rethrow during render (the setState(() => { throw err }) trick; TanStack Query's throwOnError does exactly this). Server-side rendering — handled by the framework layer (Next's error.tsx). And the boundary's own render — errors propagate to the next boundary up, so keep fallbacks trivially simple. Bonus points for the mechanism split: getDerivedStateFromError is the pure render-phase state update, componentDidCatch is the commit-phase side-effect slot for logging.

Q12. "How do you decide where error boundaries go in an app?"

By failure blast radius, at three deliberate levels: a root boundary as the never-seen last resort; a boundary per route so a broken page leaves navigation and app shell alive; and widget boundaries around independent failure-prone islands — third-party embeds, charts, comments — where the question is "if this crashes, what's the largest area that's genuinely useless?" Each boundary needs an owner story: reporting in onError (a fallback without telemetry muffles bugs), a reset path wired to fix the cause (onReset refetching, resetKeys on the inputs so navigating away auto-clears), not just a re-render button. Anti-pattern: wrapping everything — you get half-rendered pages that hide real failures.

Q13. "A click inside a portal — which onClick handlers fire?"

Every onClick on the portal's React ancestors, even though the DOM node lives in document.body — React events propagate through the component tree, since React attaches listeners at the root and dispatches synthetically along the React hierarchy. Native DOM listeners on the original ancestors do not fire; the DOM ancestor chain is body-side. Practical consequences: an "outside click" detector using a document listener plus container.contains(e.target) will misclassify clicks inside a portaled dropdown as outside unless you also test containment against the portal node; and a modal that shouldn't trigger its React parent's click handlers needs stopPropagation at the dialog — the React tree carries the event up regardless of the DOM location.

Q14. "Why does a modal need a portal at all? z-index: 9999 exists."

Because z-index competes only within a stacking context. Any ancestor with transform, filter, opacity below 1, or will-change creates one, and nothing inside can render above content outside it, whatever its z-index — it's a hierarchy problem, not a magnitude problem. Ancestor overflow: hidden clips too. Portaling to document.body exits every ancestor stacking context and clip, which is the entire point. Follow-ups worth volunteering: the native <dialog>.showModal() renders in the top layer — above all stacking contexts, no portal needed — plus free focus trapping and Escape; and portals move DOM only, so focus management (initial focus, trap or inert background, return focus) is still on you.

Q15. "Compare HOCs, render props, and hooks for sharing logic. Why did hooks win?"

All three share stateful logic. HOCs decorate a component (component in, enhanced component out) but inject props implicitly — namespace collisions, wrapper hell, static/ref plumbing, and you can't tell where a prop comes from. Render props made the flow explicit (you receive and name values at the call site) but nest into pyramids and put logic in JSX. Hooks share logic with no structural footprint: values are named local variables traceable to an import, composition is sequential lines, no extra tree nodes, natural TypeScript. The trade: rules of hooks, because state identity rides on call order. Still-legitimate HOCs: memo, connect, withAuth-style gates (a wrapper can prevent rendering; a hook can't), error-boundary wrappers. Render props survive as prop getters and per-datum render callbacks.

Q16. "What does 'headless UI' mean and why would a design system build on Radix or React Aria?"

Behavior, state, and accessibility shipped without visual opinion — the consumer owns 100% of styling. The economics: in a combobox, styles are the easy part; the WAI-ARIA keyboard contract, focus management, screen reader wiring, and positioning are person-years of edge cases that aren't your product. Styled kits (MUI) bundle a design language you fight via overrides coupled to their DOM; headless libraries invert ownership. Spectrum to name: Radix (unstyled compound components, asChild), Headless UI, React Aria (hooks returning prop bags — maximum control), TanStack Table (renders literally nothing). Trade-offs: you build and maintain the styled layer, and their compound API becomes your API's foundation. shadcn/ui is the proof of pattern: styled copy-paste components over Radix primitives.

Q17. "Explain the state reducer pattern. What problem does it solve that props can't?"

The component computes each proposed state transition, then passes it through a consumer-supplied reducer that gets the final say: stateReducer(state, { ...action, changes }). That lets consumers modify behaviors the author never anticipated — "don't close the menu on blur while dragging" — without a new prop per feature request, which is exactly the failure mode it was invented to stop (Downshift). It's the top of the control ladder: onChange observes, controlled props own a value, the state reducer intercepts every transition. Cost: your action types become public, versioned API. It's the signature move of headless libraries because inversion of control is their whole thesis.

Q18. "Spell out the value/defaultValue/onChange contract for a reusable component."

value present on first render → controlled: render from the prop only, report intents via onChange with the proposed value, mutate nothing internal — the caller may veto by not updating. Only defaultValue → uncontrolled: seed internal state once; later defaultValue changes are ignored (resets are the caller's job via key). Both → controlled wins, dev warning. Switching modes mid-lifetime → dev warning, keep first-render mode; the classic trigger is value={data?.x} with async data. Implementation is useControllableState: isControlled = value !== undefined, current = controlled ? prop : internal, setter updates internal only when uncontrolled but always notifies. Radix and MUI both ship exactly this.

Q19. "What are prop getters and why do headless hooks return them instead of prop objects?"

Functions that produce prop bags while merging the consumer's props with correct policy: getToggleButtonProps({ onClick: mine }) composes my handler with the internal one (callAll(internal, mine) — both run), concatenates classNames, and keeps ARIA wiring intact. A plain returned object can't do that — spreading consumer props after it clobbers the internal onClick, before it clobbers the consumer's. So the getter owns the merge, the exact problem cloneElement handled badly (silent overwrite). Also worth saying: a getter can accept options that change what it returns (getItemProps({ index })), which a static object can't. Two-line implementation of callAll is a common follow-up — be able to write it.

Q20. "In a machine-coding round for an accessible Tabs/Accordion, what separates a senior submission?"

Sequence and invisible correctness. Seniors clarify the contract first (controlled? multiple open? keyboard spec?), state the minimal state shape aloud (one active value / one Set — everything else derived), then build in always-running slices. The a11y layer is the separator: real buttons, aria-expanded/aria-controls for accordion; role=tablist/tab/tabpanel with aria-selected, roving tabindex, arrow-key navigation for tabs — naming the APG pattern signals you didn't memorize attributes. Then edge cases unprompted: disabled items, zero/one item, rapid toggling, focus after activation. Finally API quality — context-based compound parts with a helpful missing-context error — and a spoken backlog when time expires. Juniors ship divs with onClick and perfect CSS; it fails the round.

Q21. "How do you make form validation errors accessible?"

Four wires. Every input gets a real label (htmlFor; placeholder is not a label). Invalid fields get aria-invalid plus aria-describedby pointing at the error message's id, so the message is announced with the field on focus. Newly appearing errors announce without focus theft via role="alert"/aria-live. On failed submit, move focus to the first invalid field (RHF's shouldFocusError default) or to an error-summary block at the top linking to each field — the GOV.UK pattern, strongest for long forms. Plus judgment calls: don't rely on color alone, don't make a disabled submit button the only invalid signal (no feedback about why), and validate onTouched-ish so users aren't punished mid-typing.

Q22. "Your Autocomplete shows results for a previous query. What happened and how do you fix it?"

A race: responses resolve out of order — the request for "par" returns after the request for "paris" and overwrites newer results. Fixes, strongest first: AbortController — abort the in-flight request whenever a new one starts (and on unmount), so stale responses die at the network layer; or a last-write-wins guard — capture a request id/the query in the closure and ignore responses that aren't the latest. Layer debounce on top so you're not firing per keystroke, and short-circuit empty queries. This generalizes: it's the same stale-closure race as any effect-driven fetch, and the reason query libraries key requests by input — with TanStack Query, queryKey: ['search', term] makes stale responses land in a different cache entry, dissolving the bug structurally.

function useControllableSet({ value, defaultValue = [], onChange }) {

  const isControlled = value !== undefined;

Solution 
const callAll = (...fns) => (...args) => fns.forEach((fn) => fn?.(...args));

function toggleReducer(state, action) {
  switch (action.type) {
    case 'toggle': return { on: !state.on };
    case 'setOn':  return { on: true };
    case 'setOff': return { on: false };
    default: return state;
  }
}

function useToggle({ stateReducer = (s, a) => a.changes } = {}) {
  const [state, dispatch] = useReducer(
    (state, action) => stateReducer(state, { ...action, changes: toggleReducer(state, action) }),
    { on: false }
  );
  const toggle = () => dispatch({ type: 'toggle' });
  const getTogglerProps = ({ onClick, ...props } = {}) => ({
    'aria-pressed': state.on,
    onClick: callAll(onClick, toggle),         // consumer handler AND ours
    ...props,
  });
  return { on: state.on, toggle, getTogglerProps };
}

// (a) max 3 activations — consumer-side only:
const [count, setCount] = useState(0);
const { on, getTogglerProps } = useToggle({
  stateReducer: (state, action) => {
    if (action.type === 'toggle' && !state.on && count >= 3) return state; // veto
    return action.changes;
  },
});
// track activations: <button {...getTogglerProps({ onClick: () => !on && setCount(c => c + 1) })}>

// (b) can't turn off while locked:
useToggle({
  stateReducer: (state, action) =>
