17 — Web Performance & Core Web Vitals

From URL to pixels, LCP/INP/CLS mastery, caching that actually works, and a rehearsable "how I made it fast" story.

Why interviewers ask this

Two questions are near-guaranteed in senior frontend loops: "What happens when you type a URL and press Enter?" and "Tell me about a time you made something fast." The first tests whether you understand the machine you program against — the browser's rendering pipeline, not just React's. The second tests whether you can run a real performance investigation: measure, attribute, fix, verify — with numbers, not vibes.

Performance is also where senior signal lives because it cuts across everything: networking, caching, the JS event loop, CSS internals, fonts, images, build tooling, and product metrics (conversion, bounce). A mid-level engineer says "we lazy-loaded images." A senior engineer says "our p75 LCP was 4.8s in CrUX; sub-part attribution showed 2.1s of resource load delay because the hero image was discovered late via a CSS background — we moved it into the initial HTML with fetchpriority="high" and preloaded it, cutting LCP to 2.1s and lifting conversion 4%." This tutorial builds you toward the second answer.

Currency check (this is tested): INP (Interaction to Next Paint) replaced FID as a Core Web Vital in March 2024. If you say "FID" as a current vital in an interview, that's a red flag. The three Core Web Vitals are LCP, INP, CLS, each evaluated at the 75th percentile of real-user (field) data.

1. Critical Rendering Path — URL to pixels Tier 1

This is the frontend half of the classic "type a URL, what happens?" question. You should be able to narrate it end-to-end without notes. The full pipeline: 

DNS ──> TCP ──> TLS ──> HTTP request ──> TTFB (first byte of HTML)
                                            │
                                            ▼
      ┌─────────────── HTML bytes stream in ───────────────┐
      │                                                    │
      ▼                                                    ▼
  HTML parser ──> DOM            preload scanner (looks ahead,
      │                          kicks off fetches for img/css/js
      │  <link rel=stylesheet>   even while parser is blocked)
      ▼
  CSS parser ──> CSSOM     (render-blocking: no pixels until ready)
      │
      ▼
  DOM + CSSOM ──> Render tree (only visible nodes, computed styles)
      │
      ▼
  Layout (geometry: where and how big everything is)
      │
      ▼
  Paint (rasterize: fill pixels into layers — text, colors, images)
      │
      ▼
  Composite (GPU stitches layers together ──> frame on screen)

HTML → DOM: a streaming parse

The browser does not wait for the whole HTML document. It parses bytes as they arrive (tokenize → nodes → DOM tree) and can start rendering above-the-fold content while the rest of the document is still on the wire. This is why streaming SSR works: flush the <head> and shell early, stream the rest. The parser is incremental and resumable — but it can be blocked, which is where scripts come in.

CSS → CSSOM: why CSS is render-blocking

CSS is render-blocking but not parser-blocking. The parser keeps building the DOM while a stylesheet downloads, but the browser will not paint anything until the CSSOM is complete. Why? Two reasons:

  • Correctness: the last rule wins in the cascade. Painting with half the CSS would render a wrong page, then visibly restyle — a flash of unstyled content (FOUC).
  • Script correctness: a <script> below a stylesheet might call getComputedStyle(). So a pending stylesheet also blocks execution of subsequent scripts, which in turn blocks the parser. This is the sneaky interaction: CSS can indirectly become parser-blocking via JS.
Pitfall: "CSS blocks the parser" is imprecise. Say: "CSS blocks rendering, and it blocks script execution of later scripts (because they might read computed styles), which transitively blocks the parser." That precision is exactly what distinguishes senior answers.

JS and the parser: defer / async / module semantics

A classic <script src> is parser-blocking: the HTML parser halts, the script downloads, executes, and only then does parsing resume. The historical reason: scripts can document.write() into the token stream, so the parser can't safely proceed. Know this table cold:

AttributeDownloadExecutionOrder guaranteeBlocks parser?Use for
<script src> (none)Immediately, parser pausesImmediately on arrivalDocument orderYes (download + execute)Legacy; avoid
deferIn parallel with parsingAfter parsing completes, before DOMContentLoadedDocument order preservedNoApp code that needs the DOM
asyncIn parallel with parsingAs soon as downloaded (parser pauses for execution only)None — first-loaded runs firstOnly during executionIndependent scripts: analytics, ads
type="module"In parallel (plus its import graph)Deferred by default (like defer)Document orderNoModern app entry points

Execution-order gotcha: defer scripts run in document order even if a later one downloads first. async scripts race — never use async for scripts with dependencies between them. Also: defer on an inline script is ignored (nothing to download); modules are the exception — inline type="module" is deferred.

The preload scanner

While the main parser is blocked (on a script, say), a second lightweight parser — the preload scanner — races ahead through the raw HTML and starts fetches for <img src>, <link rel="stylesheet">, <script src>, fonts referenced via preload, etc. This is a huge, invisible optimization — and it's why certain patterns hurt so much:

  • Invisible to the scanner: CSS background-image, images injected by JS, @import in CSS, fonts referenced only in @font-face. These are discovered late.
  • Visible to the scanner: anything declared as real HTML markup in the initial document. This is the core argument for putting your LCP image in the HTML.

Render tree → Layout → Paint → Composite

DOM + CSSOM combine into the render tree (visible nodes only — no display:none, no <head>). Then three costed stages per frame:

StageWhat it doesTriggered by changing…Relative cost
Layout (reflow)Computes geometry — position and size of every boxwidth, height, top/left, margin, font-size, DOM insertionMost expensive — can cascade through the tree
PaintRasterizes pixels into layerscolor, background, box-shadow, border-radius, visibilityMedium
CompositeGPU assembles layers into the final frametransform, opacity, filter (mostly)Cheapest — off main thread

Each stage implies the ones after it: layout → paint → composite; paint → composite. This is why the golden rule of animation is animate only transform and opacity — they can be handled entirely by the compositor thread on the GPU, skipping layout and paint, and they stay smooth even when the main thread is busy running JS. Animating top/left/height forces layout on every frame. 

/* Janky: layout every frame, runs on main thread */
.slide-bad  { transition: left 300ms; left: 200px; }

/* Smooth: compositor-only, survives a busy main thread */
.slide-good { transition: transform 300ms; transform: translateX(200px); }
/* Hint the browser to promote to its own layer (use sparingly): */
.slide-good { will-change: transform; }

Where fonts block: FOIT vs FOUT

Web fonts are discovered late (a font downloads only after CSSOM is built and layout finds text actually using that @font-face). While the font loads, browsers historically chose between:

  • FOIT (Flash of Invisible Text): text is hidden until the font arrives — most browsers hide up to ~3s then fall back. Terrible for FCP/LCP if your LCP element is a headline.
  • FOUT (Flash of Unstyled Text): render immediately in a fallback font, swap when the web font arrives. Better for paint metrics, but the swap can shift layout → CLS. Section 8 covers the modern zero-CLS fix.

You control this with font-display: block (FOIT-ish), swap (FOUT), optional (use it only if cached/instant — zero CLS by definition).

The rehearsable answer: first paint, end to end

Say this in the interview (60–90 seconds):

"The browser first resolves DNS, opens a TCP connection, negotiates TLS — with HTTP/3 the transport is QUIC over UDP, which folds the handshakes together. It sends the request and TTFB marks the first byte of HTML. The HTML parser is streaming: it tokenizes bytes as they arrive and builds the DOM incrementally. When it hits a <link rel="stylesheet"> it fetches it; CSS is render-blocking, so no pixels appear until the CSSOM is ready. A classic script tag blocks the parser — download plus execute — which is why we use defer or modules; meanwhile the preload scanner races ahead of the blocked parser to kick off fetches early. Once DOM and CSSOM exist, the browser builds the render tree, runs layout to compute geometry, paints pixels into layers, and the compositor assembles them on the GPU — that first frame is FCP. When the largest element in the viewport — usually the hero image or headline — finishes rendering, that's LCP. So the levers are: get bytes fast (CDN, caching, TTFB), keep the critical path skinny (minimal blocking CSS, deferred JS), and make the LCP resource discoverable in the initial HTML with high priority."

2. Core Web Vitals, precisely Tier 1

The three Core Web Vitals measure loading (LCP), interactivity (INP), and visual stability (CLS). Google evaluates each at the p75 of field data per origin/page, segmented mobile vs desktop. Thresholds — memorize exactly:

MetricMeasuresGoodNeeds improvementPoor
LCP — Largest Contentful PaintLoading: render time of largest content element in viewport≤ 2.5 s2.5 – 4.0 s> 4.0 s
INP — Interaction to Next PaintResponsiveness: worst interaction latency across page lifetime≤ 200 ms200 – 500 ms> 500 ms
CLS — Cumulative Layout ShiftVisual stability: sum of unexpected layout shift scores≤ 0.10.1 – 0.25> 0.25

LCP: what counts, and its four sub-parts

Elements that can be the LCP candidate: <img> (including inside <picture>), <image> in inline SVG, <video> (poster or first frame), an element with a CSS background-image: url(...), and block-level text elements. The browser reports the largest as it renders; the final LCP is the last candidate before first user input or page unload. Low-content placeholders and full-viewport background washes are heuristically excluded.

The senior move is sub-part attribution — LCP time decomposes into four sequential phases:

|-- TTFB --|-- Resource load delay --|-- Resource load time --|-- Element render delay --|
0                                                                                   LCP paint

TTFB                : server + network time to first HTML byte
Resource load delay : gap between TTFB and LCP resource STARTING to load
                      (late discovery: JS-injected img, CSS background, lazy-load)
Resource load time  : the image download itself (bytes, format, compression)
Element render delay: resource is loaded but can't paint yet
                      (render-blocking CSS, main thread busy, client-side rendering)
Diagnosis heuristics: big load delay → discovery problem (preload / move into HTML / remove loading="lazy"). Big load time → weight problem (AVIF/WebP, srcset, compression, CDN). Big render delay → critical path problem (blocking CSS/fonts, or the framework hasn't hydrated/rendered the element yet — the classic SPA disease where LCP waits on the whole JS bundle). Big TTFB → backend/CDN problem. Being able to name which sub-part dominates is what turns "the image was slow" into a senior answer.

INP: the interactivity vital (replaced FID, March 2024)

INP observes every click, tap, and key press during the page's lifetime and reports (approximately) the worst interaction's latency — for pages with very many interactions, a high percentile of them. Crucially, unlike FID (which measured only input delay of the first interaction), INP measures the full arc of an interaction: 

user clicks
   │
   ├─ 1. INPUT DELAY        main thread busy (long task running) — event can't start
   ├─ 2. PROCESSING TIME    your event handlers run (and everything they trigger:
   │                        React re-render, state updates, synchronous effects)
   └─ 3. PRESENTATION DELAY style/layout/paint/composite for the next frame
                                                          │
                                                          ▼
                                               next frame painted  = INP for
                                                                     this interaction

The connection to long tasks: any main-thread task > 50 ms is a "long task." If a user interacts while one runs, the entire remainder becomes input delay. So INP optimization is largely about (a) never running long tasks, (b) keeping handlers lean, (c) not forcing expensive rendering work before the next paint. Section 4 is the playbook.

Pitfall: "INP is the time until my click handler finishes" — wrong. It's until the next frame paints. A handler that finishes in 10 ms but synchronously triggers a 400 ms React render + layout still yields a ~410 ms INP. Conversely, showing an instant visual acknowledgment (spinner, pressed state) and deferring the heavy work gives a great INP even if the full operation takes a second.

CLS: the layout-shift math

Every unexpected layout shift gets a score:

layout shift score = impact fraction × distance fraction

impact fraction   : union of the areas the unstable element occupied
                    (before ∪ after) ÷ viewport area
distance fraction : greatest distance any unstable element moved ÷
                    viewport's largest dimension

Example: a banner injects at the top, pushing content that covers 60% of
the viewport down by 25% of the viewport height:
score = 0.60 × 0.25 = 0.15   ← already "needs improvement" from ONE shift

CLS is the largest burst of shifts: shifts are grouped into session windows (shifts < 1 s apart, window capped at 5 s), and CLS is the worst window's total. Shifts within 500 ms of a user interaction are excluded — expanding an accordion the user clicked doesn't count.

The usual suspects: images/videos without dimensions, ads and embeds with unknown size, dynamically injected banners (cookie consent!), web-font swaps changing text metrics, and content injected above existing content. Fixes in section 5.

RUM with the web-vitals library

// npm i web-vitals — the attribution build tells you WHY, not just the number
import { onLCP, onINP, onCLS } from 'web-vitals/attribution';

function sendToAnalytics(metric) {
  const body = JSON.stringify({
    name: metric.name,           // 'LCP' | 'INP' | 'CLS'
    value: metric.value,
    rating: metric.rating,       // 'good' | 'needs-improvement' | 'poor'
    id: metric.id,               // unique per page load — dedupe key
    // Attribution: the diagnostic gold
    attribution: {
      // LCP: element selector, url, and the four sub-part timings
      element: metric.attribution.element,
      // INP: interactionTarget, inputDelay, processingDuration, presentationDelay
      // CLS: largestShiftTarget, largestShiftTime
      ...metric.attribution,
    },
  });
  // sendBeacon survives page unload; fetch keepalive is the fallback
  navigator.sendBeacon?.('/rum', body) ||
    fetch('/rum', { body, method: 'POST', keepalive: true });
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);   // reports on visibilitychange/pagehide (worst so far)
onCLS(sendToAnalytics);
Senior point: aggregate RUM at p75 (and look at p95), never the mean — performance distributions are heavily right-skewed, and the mean hides the users who bounce. Slice by device class, connection, country, and page template: "LCP is fine except product pages on Android in India" is an actionable finding; a single global number is not.

3. LCP optimization playbook Tier 1

Organize your answer by the four sub-parts. That structure alone signals seniority.

3.1 Attack TTFB

  • CDN in front of everything, including HTML where possible — serve from an edge near the user.
  • Cache HTML at the edge with s-maxage + stale-while-revalidate when content allows (see section 6).
  • Stream the response: flush <head> before slow data fetches complete (streaming SSR — React renderToPipeableStream, covered in the SSR tutorial). The browser starts fetching CSS/fonts/hero image while your server still queries the DB.
  • Cut server work: cache DB queries, avoid redirect chains (each redirect is a full round trip added to TTFB).

3.2 Kill resource load delay — discover the LCP resource early

The single highest-leverage LCP fix on most sites. The LCP image must be discoverable by the preload scanner in the initial HTML:

<!-- BAD: discovered only after CSS downloads & layout runs -->
<div class="hero"></div>
<style>.hero { background-image: url(hero.jpg); }</style>

<!-- WORSE: discovered only after the JS bundle downloads, parses,
     executes, and React renders -->
{isLoaded && <img src="hero.jpg" />}

<!-- GOOD: in the initial HTML, high priority, eagerly loaded -->
<img src="hero.avif" width="1200" height="630"
     fetchpriority="high" decoding="async"
     alt="…">

<!-- If it MUST be a CSS background or comes late, preload it: -->
<link rel="preload" as="image" href="hero.avif"
      fetchpriority="high"
      imagesrcset="hero-800.avif 800w, hero-1600.avif 1600w"
      imagesizes="100vw">
  • fetchpriority="high" is the modern trick: browsers start images at Low priority and upgrade after layout proves they're in-viewport; this attribute skips the wait and puts the hero in the first wave of requests.
  • Never lazy-load the LCP image. loading="lazy" on the hero delays it behind layout — one of the most common self-inflicted LCP wounds (auditable: it's a Lighthouse flag). Lazy-load below-the-fold images only.

3.3 Shrink resource load time — image weight

<picture>
  <source type="image/avif"
          srcset="hero-800.avif 800w, hero-1200.avif 1200w, hero-1600.avif 1600w"
          sizes="(max-width: 768px) 100vw, 50vw">
  <source type="image/webp"
          srcset="hero-800.webp 800w, hero-1200.webp 1200w, hero-1600.webp 1600w"
          sizes="(max-width: 768px) 100vw, 50vw">
  <img src="hero-1200.jpg" width="1200" height="800"
       fetchpriority="high" alt="…">
</picture>
  • AVIF is typically ~50% smaller than JPEG at equivalent quality; WebP ~25–30% smaller. Serve via <picture> type-fallback or CDN content negotiation (Accept header).
  • srcset/sizes done right: srcset lists candidates with intrinsic widths (800w); sizes tells the browser the layout width so it can pick before layout. The classic bug: omitting sizes defaults it to 100vw, so the browser downloads a full-width image for a half-width slot.
  • Compress aggressively (quality 60–75 is usually indistinguishable), and use an image CDN (Cloudinary/imgix/Next Image) so resizing/format is automatic.

3.4 Cut element render delay

  • Render-blocking CSS diet: ship only critical CSS in the blocking path. Split per-route; remove dead rules (DevTools Coverage tab shows unused CSS). Every KB of blocking CSS delays all paint.
  • Inline critical CSS — the honest take: inlining above-the-fold CSS into <style> eliminates a render-blocking round trip and can be a big FCP/LCP win on cold loads. But: it bloats every HTML response, isn't cacheable across pages, and the tooling to extract "critical" CSS (Critters/Beasties, penthouse) is brittle with dynamic classes. Modern verdict: with HTTP/2 and small, code-split CSS files served from the same connection, the win is smaller than folklore suggests. Do it when your CSS is small and your TTFB is good; measure before adopting the complexity. (Tailwind-style atomic CSS often makes the whole file small enough to inline wholesale.)
  • Font strategy matters when LCP is a text block: font-display: swap (or optional) so text paints in a fallback instead of waiting — full treatment in sections 5 and 8.
  • If LCP render delay is dominated by JS rendering the element (CSR SPA), the fix is architectural: SSR/SSG the shell so the LCP element exists in HTML — pointer to the SSR/RSC tutorial.

4. INP optimization playbook Tier 1

Map every fix to the three phases: input delay, processing, presentation.

4.1 Break up long tasks — yield to the main thread

The main thread runs one task at a time. A 300 ms task means any interaction arriving during it waits up to 300 ms before your handler even starts. Chunk and yield:

// The modern primitive: scheduler.yield() — yields but resumes with
// PRIORITY (front of queue), unlike setTimeout which goes to the back.
async function processItems(items) {
  for (const item of items) {
    processItem(item);                 // each chunk stays well under 50ms
    if (shouldYield()) {
      await scheduler.yield?.() ??     // Chrome 129+
        new Promise(r => setTimeout(r, 0));  // fallback: macrotask break
    }
  }
}

// Yield-on-demand: only pay the cost when a user is actually waiting
function shouldYield(last = performance.now()) {
  return navigator.scheduling?.isInputPending?.() ||
         performance.now() - last > 50;
}
Why setTimeout(0) and not Promise.resolve().then()? Microtasks run before the browser gets control back — awaiting a resolved promise does not yield. You need a macrotask (setTimeout, MessageChannel) or scheduler.yield() for the event loop to process pending input and paint.

React-level yielding: startTransition / useTransition marks a state update as non-urgent; React's concurrent renderer time-slices it and lets input interrupt — the framework-native version of chunk-and-yield. Type-ahead filtering over a big list is the canonical case: the input's controlled value updates urgently, the filtered-list render is a transition.

4.2 Keep input handlers lean; separate feedback from work

button.addEventListener('click', () => {
  button.classList.add('is-busy');     // 1. instant visual feedback…
  requestAnimationFrame(() =>          // 2. let that frame PAINT…
    setTimeout(() => {                 //    (rAF alone runs BEFORE paint —
      doExpensiveWork();               //     rAF + setTimeout lands after it)
      button.classList.remove('is-busy');
    }, 0));
});

INP measures time-to-next-paint, so painting a pressed state / spinner immediately and deferring the expensive work gives users (and the metric) instant response. Measure with the DevTools Performance panel's Interactions track: record, interact, and it shows each interaction as a span broken into input delay / processing / presentation — with the responsible call stack below it.

4.3 Avoid layout thrashing (forced synchronous layout)

// BAD: read → write → read → write forces a synchronous layout PER ITERATION.
// Reading offsetHeight after a style write makes the browser flush layout
// immediately to answer accurately.
for (const el of items) {
  el.style.height = el.offsetHeight + 10 + 'px';   // read+write interleaved
}

// GOOD: batch all reads, then all writes — one layout at most
const heights = items.map(el => el.offsetHeight);  // reads
items.forEach((el, i) => el.style.height = heights[i] + 10 + 'px'); // writes

Layout-forcing reads to know: offsetHeight/Width/Top, clientWidth, scrollTop, getBoundingClientRect(), getComputedStyle() (for layout-dependent props). In the Performance panel this shows up as purple Layout blocks with a red "forced reflow" warning.

4.4 The rest of the arsenal

  • Debounce/throttle expensive reactions: resize handlers, scroll-linked effects, autosave, search-as-you-type network calls. Keep per-keystroke work O(small).
  • Web Workers for CPU-bound work: parsing large JSON/CSV, image manipulation, diffing, crypto, search indexing — anything > ~50 ms that doesn't need the DOM. Comlink makes the postMessage plumbing ergonomic. The main thread should be a UI thread, period.
  • Hydration cost is an INP source: a server-rendered page that looks ready but is running a monolithic hydrate pass gives brutal input delay to early clicks. Mitigations: React 18 selective hydration (Suspense boundaries hydrate independently, and React prioritizes the boundary the user clicked), and RSC (server components ship zero hydration JS) — pointers to the SSR/RSC tutorial.
  • Third-party script containment: tag managers, chat widgets, and video embeds are chronic long-task factories you don't control.
    • Facade pattern: render a static lookalike (a thumbnail with a play button for YouTube — e.g. lite-youtube-embed; a fake chat bubble) and load the real embed only on interaction. Cuts hundreds of KB and the associated main-thread work from initial load.
    • Load tags async/defer, after load, or on first interaction.
    • Partytown (awareness): runs third-party scripts in a Web Worker, proxying DOM access — main thread stays clean. Clever, with real edge cases; know it exists and what it trades.

5. CLS optimization playbook Tier 1

CLS is the most mechanically fixable vital: every shift has a nameable cause and a standard fix.

5.1 Size your media. Always.

<!-- width/height set the intrinsic ASPECT RATIO; CSS can still make it fluid.
     Browsers reserve the correct box BEFORE the image loads. -->
<img src="product.avif" width="800" height="600" alt="…">
<style> img { max-width: 100%; height: auto; } </style>

<!-- When dimensions are unknown but ratio is known (video embeds, charts): -->
.embed-slot { aspect-ratio: 16 / 9; width: 100%; }

This single habit — dimensions or aspect-ratio on every <img>, <video>, and <iframe> — eliminates the biggest CLS class on the web.

5.2 The modern zero-CLS font swap

font-display: swap fixes FOIT but introduces the swap-shift: fallback and web font have different metrics, so lines rewrap and the page jumps. The fix is metric-tuned fallbacks:

@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter-var-subset.woff2') format('woff2');
  font-display: swap;
}

/* A fallback face whose metrics are ADJUSTED to occupy the same space
   as Inter — when the swap happens, nothing moves. */
@font-face {
  font-family: 'Inter-fallback';
  src: local('Arial');
  size-adjust: 107%;        /* scale glyphs so line lengths match */
  ascent-override: 90%;     /* match vertical metrics */
  descent-override: 22%;
  line-gap-override: 0%;
}

body { font-family: 'Inter', 'Inter-fallback', sans-serif; }

Tools like fontaine or Next.js next/font compute these overrides automatically. Combine with subsetted woff2 + preload (section 8) so the swap window is tiny — often the web font wins the race entirely. font-display: optional is the nuclear zero-CLS option: if the font isn't ready within a tiny block period, this page load simply uses the fallback (and the font is cached for next time).

5.3 Reserve space for everything dynamic

  • Ads/embeds: style the slot with a fixed min-height (largest likely creative, or a fixed size). An empty reserved box beats a shift. Same for A/B-test banners.
  • Injected banners (cookie consent, app banners): overlay them (position: fixed) or reserve the slot in the initial layout — never push content down after render.
  • Skeletons must match final size: a skeleton that's 200 px shorter than the loaded content just moves the shift later. Match the real component's dimensions — aspect-ratio boxes and fixed-height rows.
  • Insert below, not above: new content (infinite scroll, "new items" pills) should append below the viewport or behind an explicit user action (clicks give you the 500 ms exclusion window).

5.4 Animate with transform

/* Shifts layout every frame — each frame's movement scores CLS
   (animations via layout properties are NOT excluded): */
.toast-bad  { animation: slide-bad 300ms; }
@keyframes slide-bad  { from { top: -60px; } to { top: 0; } }

/* Compositor-only, zero CLS by definition (transforms don't shift layout): */
.toast-good { animation: slide-good 300ms; }
@keyframes slide-good { from { transform: translateY(-60px); } to { transform: none; } }
Debugging CLS: DevTools Performance panel → record a load → the Layout Shifts track shows each shift; click one to see the moved elements highlighted. Or run the web-vitals attribution build and log largestShiftTarget. CLS bugs that only appear in the field (real ads, real cookie banners, real fonts on cold cache) are exactly why you need RUM.

6. Caching & delivery Tier 1

The fastest request is the one you never make. Caching questions are checked answers — precision required.

6.1 Cache-Control directives, precisely

DirectiveMeaning — exactly
max-age=NResponse is fresh for N seconds; while fresh, the browser serves it with zero network.
s-maxage=NLike max-age but for shared caches only (CDN/proxy); overrides max-age there. Lets you say "CDN: cache 1 h; browser: don't."
no-cacheDoes NOT mean "don't cache." Means: store it, but revalidate with the server before every use (conditional request; 304 if unchanged).
no-storeThe actual "don't cache": never write this response to any cache. For sensitive data. Also breaks bfcache (section 10).
private / publicprivate: browser cache only, never CDN (per-user responses). public: shared caches may store it even with auth.
immutable"This will never change during its freshness" — skip revalidation even on reload. Only safe with hashed filenames.
stale-while-revalidate=NAfter expiry, serve the stale copy instantly for up to N s while revalidating in the background. Users get cache speed, cache stays fresh.
must-revalidateOnce stale, must revalidate — never serve stale (e.g., when origin is unreachable).
The checked answer: no-cache ≠ no-store. no-cache = "cache, but ask first" (a 304 revalidation still costs a round trip but saves the body transfer). no-store = "never cache." Interviewers probe this deliberately.

6.2 Revalidation: ETag / Last-Modified

First response:
  200 OK
  ETag: "abc123"
  Last-Modified: Tue, 01 Jul 2025 10:00:00 GMT
  Cache-Control: no-cache

Later (stale or no-cache) — conditional request:
  GET /page
  If-None-Match: "abc123"                  ← ETag wins if both present
  If-Modified-Since: Tue, 01 Jul 2025 …

Unchanged → 304 Not Modified (headers only, no body — cache entry reused)
Changed   → 200 with new body + new ETag

6.3 THE canonical setup: hashed assets + no-cache HTML

# Build output: content hash in every asset filename
/assets/app.9f8e7d6c.js
/assets/app.4b3a2c1d.css

# Assets — cache "forever": the URL changes when content changes
/assets/*   Cache-Control: public, max-age=31536000, immutable

# HTML — always revalidate: it's the pointer to the current hashes
/index.html Cache-Control: no-cache          # (+ ETag)
# or edge-cached flavor:
#           Cache-Control: max-age=0, s-maxage=300, stale-while-revalidate=60

Why this works: the HTML is the mutable entry point; everything else is immutable-by-URL. Deploys are instant and atomic — new HTML references new hashed URLs; old assets remain cached for users mid-session; repeat visits load all assets with zero requests. This is the pattern every bundler + host (Vite/Next/Netlify/Vercel) converges on. If you say one thing about caching in an interview, say this.

6.4 CDN mechanics

  • Edge caching: PoPs near users terminate TLS and serve cached responses; misses go to origin (or a shielding tier) and populate the edge. Wins on latency (RTT) even for uncached content because the TCP/TLS handshakes happen near the user.
  • Cache key: what the CDN indexes by — URL + host, optionally varied by headers (Vary: Accept-Encoding, device class) or cookies. Over-keying (varying on all cookies) destroys hit rate; under-keying serves wrong content. Normalize query params.
  • Invalidation: purge by URL/tag (surrogate keys) on deploy or content change — or sidestep it entirely with hashed URLs (you never invalidate immutable assets). s-maxage + stale-while-revalidate gives you fast-but-fresh HTML without purge choreography.

6.5 Compression

Brotli (br) beats gzip by ~15–20% on text at comparable speed for static assets (compress once at max level 11 at build/CDN time); gzip remains the universal fallback and is often better for on-the-fly dynamic responses at low levels. Every serious CDN/host enables both by default and negotiates via Accept-Encoding — your job is mostly to verify (check Content-Encoding in DevTools) and to never compress already-compressed formats (images, woff2, video).

6.6 HTTP/2 and HTTP/3 — what each fixed

HTTP/1.1: one in-flight request per connection (head-of-line blocking at the HTTP layer), so browsers opened ~6 parallel connections per origin and we invented hacks: domain sharding, sprite sheets, bundling everything into one file. HTTP/2 multiplexes many streams over one connection with header compression (HPACK) and prioritization — the HTTP-layer HOL problem is gone, and the old hacks became anti-patterns (sharding defeats connection reuse and prioritization).

What HTTP/2 did not fix: TCP-level head-of-line blocking. All streams share one TCP pipe; one lost packet stalls every stream until retransmission, because TCP guarantees in-order delivery of the whole byte stream. HTTP/3 fixes this by replacing TCP with QUIC over UDP: streams are independent at the transport layer (a lost packet stalls only its own stream), and the crypto handshake is folded in (0/1-RTT connection setup, plus seamless connection migration across network changes). One paragraph of this is exactly the expected interview depth.

7. JS delivery strategy & budgets Tier 2

Tutorial 16 covered the mechanics (code splitting, tree shaking, dynamic import); here's the performance framing. JS is the most expensive byte type: unlike an image, every KB must be downloaded, then parsed, compiled, and executed on the main thread — and often re-executed as hydration. 300 KB of gzipped JS can mean over a second of main-thread work on a mid-range Android phone.

  • Split by route + interaction: initial bundle = shell + current route. Modals, editors, charts, below-fold widgets load on import() at interaction or visibility.
  • The budget conversation: a common starting point is ~170 KB compressed JS for the critical path — roughly what a median mobile device can download and execute within interactive-time targets on 4G. The exact number matters less than having a budget and enforcing it, because unmanaged bundles only grow.
  • Enforce in CI:
// package.json — size-limit (or bundlesize) fails the PR on regression
{
  "size-limit": [
    { "path": "dist/assets/index-*.js",  "limit": "170 KB" },
    { "path": "dist/assets/vendor-*.js", "limit": "120 KB" }
  ],
  "scripts": { "size": "size-limit" }
}
  • Measure what users pay: DevTools Coverage tab (Cmd/Ctrl-Shift-P → "Coverage") shows per-file unused bytes at load — 60%+ unused code in the initial bundle is the code-splitting smell. Bundle analyzers (rollup-plugin-visualizer, webpack-bundle-analyzer) show what's in there — the recurring finds: moment + all locales, full lodash, duplicate dependency versions, a chart library on pages without charts.
  • Loading order: app entry as type="module"/defer; analytics and tags async or post-load; heavy third-parties behind facades or (awareness) Partytown's worker-thread execution.

8. Fonts, deep Tier 2

  • woff2 only. All modern browsers. Built-in Brotli compression. Never ship TTF/OTF/woff to the web.
  • Subset ruthlessly: a full multi-script font can be 300 KB+; a Latin subset (pyftsubset/glyphhanger) is often 15–30 KB. Declare coverage with unicode-range so non-Latin subsets download only if the page actually uses those characters:
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter-latin.woff2') format('woff2');
  unicode-range: U+0000-00FF, U+0131, U+2013-2014, U+2018-201A, U+201C-201E;
  font-display: swap;
}
/* Cyrillic face declared separately — fetched ONLY if Cyrillic text renders */
  • Variable fonts: one file carrying the whole weight/width axis. If you use 3+ weights, a variable font is usually smaller than the sum — and unlocks fluid weight animation. Two static weights? Statics may still be smaller. Measure.
  • Preload the 1–2 critical faces (body text + heading). Fonts are doubly late-discovered (CSSOM → layout → fetch), making them the canonical preload use case. Note the crossorigin — font fetches are CORS-mode even same-origin, and without it the preload is wasted (double download):
<link rel="preload" as="font" type="font/woff2"
      href="/fonts/inter-latin.woff2" crossorigin>
  • Self-host. Google Fonts via the CDN costs a preconnect + can't be cached cross-site anymore (partitioned HTTP cache since 2020 — the "everyone has Roboto cached" argument is dead). Self-hosted fonts ride your existing connection and immutable-cache setup.
  • The zero-cost option — system font stack: font-family: system-ui, -apple-system, "Segoe UI", Roboto, sans-serif;. Zero bytes, zero FOUT, zero CLS, native feel. For dashboards and app UIs this is often the right call; mentioning it shows you weigh cost against brand value rather than defaulting to web fonts.
  • Tie-in: pair swap with size-adjust/ascent-override metric-tuned fallbacks (section 5.2) for the complete zero-CLS strategy.

Situation: "Our e-commerce product detail page had p75 LCP of 4.8 s on mobile in CrUX — failing — and search + paid traffic teams were escalating. Lighthouse scored 85, which is exactly why I trust field data: our users were on mid-range Android, far from origin."

bfcache — the instant back button

The back/forward cache keeps the entire page alive in memory (full DOM + JS heap, frozen) on navigation away; back/forward restores it instantly. A huge share of navigations are back/forward, so eligibility is a real performance feature. What breaks it:

  • unload event handlers (the classic — use pagehide/visibilitychange instead; note some analytics SDKs add unload listeners for you);
  • Cache-Control: no-store on the main document;
  • open connections at navigation time — WebSockets, held IndexedDB locks (close on pagehide);
  • window.opener references in some cases.

Audit it: DevTools → Application → Back/forward cache → "Test back/forward cache" lists the exact blockers. Handle restores via pageshow with event.persisted === true (refresh stale data, restart timers).

Interview Q&A

1. Walk me through what happens from typing a URL to pixels on screen.

Use the narrative from section 1: DNS → TCP → TLS (→ QUIC folds these on H3) → request → TTFB. HTML parses streaming into the DOM; <link rel="stylesheet"> starts a render-blocking CSS fetch (CSSOM); classic scripts block the parser (hence defer/async/modules) while the preload scanner races ahead starting fetches. DOM + CSSOM → render tree → layout (geometry) → paint (rasterize layers) → composite (GPU assembles the frame) = FCP; when the largest viewport element renders = LCP. Close with the levers: fast bytes, skinny critical path, early LCP-resource discovery.

2. Why is CSS render-blocking? Is it parser-blocking?

Render-blocking because painting with a partial CSSOM would show a wrong page then visibly restyle (FOUC), and the cascade means later rules can change everything. It is not directly parser-blocking — the DOM keeps building — but a pending stylesheet blocks execution of subsequent scripts (they might call getComputedStyle), and a blocked script blocks the parser. So CSS is indirectly parser-blocking when scripts follow it.

3. defer vs async vs type="module" — exact semantics and ordering?

All three download in parallel with parsing. defer: executes after parsing, before DOMContentLoaded, in document order. async: executes the moment it arrives (pausing the parser for execution only), no order guarantee — races. type="module": deferred by default (plus fetches its whole import graph); add async to run when the graph is ready. Use defer/modules for app code with dependencies; async only for independent scripts like analytics.

4. What is the preload scanner and why should I care?

A secondary lookahead parser that scans raw HTML for fetchable resources (img src, stylesheets, scripts) while the main parser is blocked, starting downloads early. It's why resources declared in initial HTML markup load fast, and why late-discovery patterns hurt: CSS background-image, JS-injected images, @import, and fonts are invisible to it. Practical consequence: put the LCP image in the HTML; preload what must stay hidden in CSS.

5. Which CSS properties trigger layout vs paint vs composite? Why do we animate transform/opacity?

Geometry properties (width/height/top/left/margin/font-size) trigger layout → paint → composite. Visual properties (color/background/shadow) trigger paint → composite. transform and opacity are compositor-only: the GPU re-blends existing layers with no layout or paint, and it runs off the main thread — so those animations stay at 60fps even while JS is busy. That's the whole case for transform-based animation.

6. Name the three Core Web Vitals with exact thresholds and how they're evaluated.

LCP ≤ 2.5 s (loading), INP ≤ 200 ms (responsiveness), CLS ≤ 0.1 (visual stability) — each at the 75th percentile of field data, mobile and desktop assessed separately. "Poor" starts at 4 s / 500 ms / 0.25. INP replaced FID in March 2024.

7. What elements can be the LCP element? What are LCP's sub-parts?

Candidates: <img>, <image> in SVG, <video> poster/first frame, CSS background-image: url(), and block-level text. Sub-parts, in sequence: TTFB → resource load delay (discovery gap) → resource load time (download) → element render delay (loaded but not painted). Each sub-part maps to a different fix family — load delay = discovery, load time = weight, render delay = critical path/CSR.

8. INP vs FID — what changed and why does it matter?

FID measured only the input delay of the first interaction — a page could freeze for a second processing every click and still pass. INP measures the full latency (input delay + processing + presentation delay) of essentially the worst interaction across the whole page lifetime. It became a Core Web Vital in March 2024. Practically it shifted optimization from "load less JS up front" to "never block the main thread, ever."

9. How is a CLS score calculated?

Per shift: impact fraction (union of moved elements' before+after area ÷ viewport) × distance fraction (max move distance ÷ largest viewport dimension). CLS = the worst session window of shifts (shifts < 1 s apart, window ≤ 5 s). Shifts within 500 ms of user input are excluded, as are transform changes — which is why transform animations are CLS-free.

10. Lighthouse says 95 but real users complain. Explain.

Lab vs field. Lighthouse is one synthetic run: a fixed device profile, throttled but consistent network, cold cache, logged-out, no interactions — and it reports TBT, not INP. Real users have slower devices, worse networks, other geographies, logged-in heavy pages, and actual interactions. Check CrUX/RUM p75 sliced by device and geography; treat field as truth and Lighthouse as a reproducible debugger. Also verify you're testing the same page state users actually see.

11. Why is TBT the lab stand-in for INP?

INP requires real user interactions, which a synthetic load-only test doesn't have. TBT sums the over-50 ms portion of every long task between FCP and TTI — measuring how occupied the main thread is. A busy main thread is what creates input delay, so TBT correlates with (but doesn't equal) INP. Low TBT with bad field INP usually means slow handlers or post-load third-party work, not load-time tasks.

12. Your LCP image loads late. Walk me through the fix.

First attribute: if load delay dominates — ensure it's an <img> in the initial HTML (not CSS background or JS-injected), remove any loading="lazy", add fetchpriority="high", and preload with imagesrcset if it must stay late-declared. If load time dominates — AVIF/WebP via <picture>, correct srcset/sizes, compression, image CDN. If render delay — slim the blocking CSS, fix fonts, or SSR the element if it's waiting on client-side rendering.

13. What's wrong with lazy-loading everything?

loading="lazy" defers fetch until layout determines near-viewport position — for the LCP/hero image that adds the entire discovery + layout wait to your LCP. Rule: eager + fetchpriority="high" for above-the-fold, lazy for below. Blanket lazy-loading via a lint rule or CMS default is a classic self-inflicted regression (and a Lighthouse audit flag).

14. A click handler takes 300 ms. How do you get INP under 200 ms?

Profile the interaction in the Performance panel's interactions track to split input delay / processing / presentation. Then: paint feedback immediately and defer the heavy work past the next frame (rAF + setTimeout); chunk the work with scheduler.yield() (setTimeout fallback) so input can interleave; in React, wrap non-urgent state updates in startTransition; move pure computation to a Web Worker; check for forced synchronous layout in the handler; and cut third-party long tasks competing for the thread.

15. What is layout thrashing? Show the classic example.

Interleaving style writes with layout-dependent reads. After a write, layout is dirty; reading offsetHeight/getBoundingClientRect() forces the browser to run layout synchronously right now to answer. In a loop — el.style.height = el.offsetHeight + 10 + 'px' per item — that's a forced reflow per iteration. Fix: batch all reads, then all writes (or use rAF-phased read/write helpers). DevTools shows it as repeated purple layout blocks with "forced reflow" warnings.

16. no-cache vs no-store?

no-cache: the cache may store the response but must revalidate with the origin (conditional request via ETag/If-None-Match) before each reuse — a 304 skips the body transfer. no-store: never write it to any cache at all — for sensitive data. Bonus points: no-store on the document also disqualifies the page from bfcache, hurting back/forward UX.

17. Design the caching headers for a typical SPA/static-site deployment.

Content-hash every built asset (app.9f8e7d.js) and serve /assets/* with public, max-age=31536000, immutable — the URL is the version, so it can never be stale. Serve HTML with no-cache (+ ETag), or max-age=0, s-maxage=300, stale-while-revalidate=60 to let the CDN absorb traffic while browsers always get current pointers. Result: instant atomic deploys, zero-request repeat visits for assets, no purge choreography.

18. What did HTTP/2 fix, and what problem remained for HTTP/3?

HTTP/2 fixed HTTP-layer head-of-line blocking by multiplexing many streams over one connection (plus header compression and priorities), killing the 6-connection limit and the sharding/spriting hacks. It did not fix TCP-layer HOL: one lost packet stalls all streams because TCP delivers bytes in order. HTTP/3 runs on QUIC over UDP — independently-delivered streams (loss only stalls the affected stream), integrated TLS handshake (0/1-RTT), and connection migration.

19. preconnect vs preload vs prefetch — and how does preload go wrong?

preconnect warms an origin (DNS+TCP+TLS) — for 1–3 critical third-party hosts. preload fetches a specific resource at high priority for the current page — for critical late-discovered resources (fonts, CSS background LCP image); needs as, and crossorigin for fonts or the preload double-fetches. prefetch is idle-time, lowest priority, for future navigations. Preload misuse: preloading scanner-discoverable resources just steals bandwidth from the critical path and can regress LCP — the "preloaded but unused" console warning is the tell.

20. How do you make a web-font swap cause zero layout shift?

Stack: subsetted woff2 + preload (shrink the swap window) + font-display: swap + a metric-tuned fallback @font-face using size-adjust, ascent-override, descent-override so the fallback occupies identical space — when the swap happens nothing moves. Tools like fontaine/next-font generate the overrides. Alternatives: font-display: optional (guaranteed zero CLS — fallback used unless font is instant) or system fonts (zero cost entirely).

21. What is the bfcache and what breaks it?

Back/forward cache: the browser freezes the whole page in memory (DOM + JS heap) on navigation and restores it instantly on back/forward — a huge share of real navigations. Breakers: unload handlers (use pagehide; watch for analytics SDKs adding them), Cache-Control: no-store on the document, open WebSockets/held locks at navigation. Test in DevTools → Application → Back/forward cache; handle restores via pageshow + event.persisted.

22. Your SPA's metrics look great but route changes feel slow. Why, and what do you do?

Core metrics (LCP/FCP/TTFB) are defined per hard navigation — soft (client-side) route changes are invisible, so dashboards only reflect the landing page. Instrument route transitions yourself: performance.mark on navigation intent, mark when meaningful content renders, performance.measure between, ship to RUM at p75. Also note INP does keep covering SPA interactions, and Chrome's experimental soft-navigations API aims to close this gap officially.

23. How would you contain a third-party chat widget that's tanking performance?

Quantify first (block it in DevTools request blocking, re-trace — attribute its long tasks and bytes). Then: facade pattern — render a static lookalike button and load the real widget on first interaction or idle; load its script async/post-load rather than sync; sandbox in an iframe where possible; consider Partytown to move its execution to a worker; and negotiate — bring the trace to the vendor/stakeholder with numbers. Add a size/TBT budget so it can't silently regress again.

24. Tell me about a time you made something significantly faster. (The big one)

Deliver the section-9 story shape with your own project: failing field metric with a number → attribution to sub-parts (name the tool: web-vitals attribution, DevTools trace) → 2–4 fixes each mapped to a cause → verified improvement in field data → business impact → CI guardrails so it stuck. The two details that consistently impress: distinguishing lab from field data, and naming which sub-part of the metric each fix targeted.

25. How do you stop performance regressing after you've fixed it?

Three layers: (1) CI gates — Lighthouse CI asserting metric budgets (LCP/TBT/CLS) on key pages per PR, plus size-limit/bundlesize failing builds on bundle growth; (2) RUM dashboards at p75/p95 with alerting, sliced by page template and device; (3) culture — performance review in design/PR templates for new features (especially third-party additions), and a named budget so "we're 20 KB over" is an objective conversation, not a vibe.

Hands-on exercises

Exercise 1 — Full audit of a real page

Pick a public content site (news/e-commerce, not a static blog).

  1. Run it through PageSpeed Insights. Record field p75 (LCP/INP/CLS, mobile) and the lab scores. Do they agree? Write one sentence explaining any gap.
  2. Open DevTools → Performance, set CPU 4× slowdown + Fast 4G, record a reload. Identify: the LCP element (Lighthouse/trace marks it), when its request started (load delay!), all render-blocking resources, and every long task > 100 ms.
  3. Open the Network panel: check Content-Encoding (brotli?), protocol column (h2/h3?), and the Cache-Control headers on the HTML vs hashed assets. Score their caching setup against section 6.3.
  4. Deliverable: a 5-item prioritized fix list, each naming the metric sub-part it targets and an estimated impact.

Exercise 2 — Fix a CLS bug (and prove it)

Build a page that commits all four CLS sins: an unsized <img> (delay it with DevTools throttling), a web font swapping from a very different fallback, a "cookie banner" injected via setTimeout(…, 1500) that pushes content down, and a toast animated with top.

  1. Record a Performance trace; use the Layout Shifts track to attribute each shift. Log scores with a PerformanceObserver({ type: 'layout-shift', buffered: true }) — sum them, excluding hadRecentInput.
  2. Fix each: width/height attributes; size-adjust-tuned fallback font; overlay (or pre-reserved) banner; transform toast.
  3. Prove CLS went from > 0.25 to < 0.05 with before/after observer output.

Exercise 3 — Kill a long task (INP drill)

Create a button whose click handler filters + sorts a 100k-item array and renders the top 500 results synchronously.

  1. Record the interaction in the Performance panel; note the interaction span and its processing time (expect several hundred ms) and the long task under it.
  2. Version A: instant visual feedback + defer work with rAF+setTimeout. Version B: chunk the filter loop with scheduler.yield()/setTimeout batches. Version C: move the filter/sort to a Web Worker. (Bonus D: React version with useTransition.)
  3. Re-record each; compare interaction latency. Write two sentences on which approach you'd ship and why (hint: worker for pure CPU, yielding when DOM access is needed).

Exercise 4 — Build a performance budget into CI

Take any Vite/Next side project.

  1. Add size-limit with a 170 KB limit on the entry chunk; run it, then intentionally import all of lodash and watch it fail; fix with lodash-es named imports or a native rewrite.
  2. Add Lighthouse CI (@lhci/cli) with the section-9 lighthouserc.json: assert LCP < 2500 ms, CLS < 0.1, TBT < 300 ms against the local preview build, 3 runs.
  3. Wire both into a GitHub Action on pull_request. Open a PR that adds a 300 KB dependency and confirm the pipeline blocks it.
  4. Deliverable: the failing PR screenshot — that's your "how I enforce budgets" interview artifact.
