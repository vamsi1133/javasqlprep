01 — HTML & Accessibility

Semantics, the accessibility tree, ARIA done right, focus management, and the platform features that separate senior answers from mid-level ones.

Why interviewers ask this

Accessibility is the single most common area where otherwise-strong senior frontend candidates flunk. Everyone at 7 YOE can build a modal in React; very few can build one that traps focus, restores it on close, announces itself to a screen reader, and degrades gracefully — and even fewer can explain why each of those steps matters in terms of the accessibility tree and the ARIA spec. FAANG interviewers use a11y as a seniority filter precisely because it can't be faked with framework knowledge: it requires understanding what the browser actually does with your markup.

The second reason: HTML is where "knows React" gets separated from "knows the web platform." Questions about <dialog>, the popover API, inert, constraint validation, or Shadow DOM reveal whether you track the platform or only your framework. In practical/UI rounds, interviewers routinely score the DOM you produce — div soup with click handlers on non-interactive elements is an instant signal of a mid-level ceiling.

A senior answer in this space always has three layers: (1) what the correct markup/pattern is, (2) what the browser and assistive technology do with it, (3) what the trade-offs are versus the alternatives. Practice saying all three out loud for every topic on this page.

Semantic HTML at senior depth T1

Why semantics matter (the three-audience argument)

Every element choice serves three consumers simultaneously:

  • The accessibility tree. The browser maps semantic elements to roles, names, and states that assistive technology (AT) consumes. <button> gives you role=button, keyboard activation (Enter and Space), focusability, and :disabled semantics for free. A <div onClick> gives you none of that.
  • Machines that aren't screen readers. Search engine crawlers, reader modes, RSS extractors, and increasingly LLM agents all use element semantics to understand page structure. <article>, <time datetime>, and heading hierarchy are structured data even without JSON-LD.
  • The next engineer. <nav>, <aside>, <figure> encode intent in a way class names can't be trusted to. Semantics are self-documenting architecture.

Landmark elements and how screen readers use them

Screen reader users don't read pages top to bottom — they navigate by structure. In NVDA/JAWS/VoiceOver, users pull up a list of landmarks (VoiceOver rotor, NVDA D key) or headings (H, 1–6 keys) and jump. If your page has no landmarks and a broken heading hierarchy, that entire navigation model is unusable.

ElementImplicit ARIA roleRules & gotchas
<header>banner — only when a direct-ish child of <body>Inside <article>/<section> it's a generic group, not a landmark. One banner per page.
<nav>navigationMultiple navs are fine but must be distinguished with aria-label ("Primary", "Breadcrumb", "Pagination"). Don't wrap every link list in nav — only major navigation blocks.
<main>mainExactly one visible <main> per page. Screen readers offer "jump to main" — it's the target of your skip link.

Too many landmarks is as bad as none. If everything is a region, nothing is. A well-structured page typically has: one banner, one main, one contentinfo, 1–3 labelled navs, and a small number of named regions/asides.

article vs section vs div — the decision procedure

The interview-ready heuristic:

  1. <article> — the content is self-contained and independently distributable. Litmus test: would it make sense in an RSS feed or as a standalone card? Blog post, product card, tweet, comment (yes, comments — nested articles are valid).
  2. <section> — a thematic grouping with a heading. If you can't name it (with a heading or aria-label), it shouldn't be a section. "A section is a chapter; an article is a document."
  3. <div> — pure styling/layout hook with zero semantic claim. This is the correct answer surprisingly often. Using div for a flexbox wrapper is not a failure of semantics; using div for a button is.
<!-- A product listing: article for each independent item -->
<section aria-labelledby="deals-heading">
  <h2 id="deals-heading">Today's deals</h2>
  <article aria-labelledby="p1-name">
    <h3 id="p1-name">Mechanical keyboard</h3>
    <p><data value="79.00">$79</data></p>
  </article>
  <!-- div is fine as a pure grid wrapper -->
  <div class="grid">…</div>
</section>

Headings: the rules that actually apply in 2026

  • One <h1> per page, describing the page's purpose (usually mirrors <title>).
  • Never skip levels downward (h2 → h4 is a WCAG-adjacent failure and breaks heading navigation). Skipping upward when closing a subsection is fine.
  • The "document outline algorithm" is dead. HTML5 originally specified that h1s would re-scope inside sectioning elements (an h1 inside <section> would "become" an h2). No browser or AT ever implemented it, and it was removed from the WHATWG spec in 2022. Heading levels are literal. Saying this in an interview signals you follow the spec, not 2012 blog posts.
  • Headings are for structure, not styling. If you need big text without a structural claim, style a <p>.
In component-driven apps, heading levels are a composition problem: a <Card> component can't hardcode <h3> because it doesn't know its nesting depth. Senior answer: accept a headingLevel/as prop, or use a context-based <Heading> that derives its level from a <Section> provider (this is exactly what React Aria's useHeading-style patterns and Reakit did). Mention that a future native <h> element was proposed but never shipped.

  • <time datetime> — machine-readable dates; crawlers and reader modes parse it.
  • <address> — contact info for the nearest article/body, not postal addresses generally (classic trivia).

The "div soup in React" problem

React encourages wrapper divs: every component that returns multiple nodes tempts you into <div>, layout components stack wrappers, and portals/providers add more. The result is a DOM that's 8 layers of divs deep before any content — which bloats the a11y tree, breaks CSS parent selectors like grid/flex relationships, and hides structure from AT.

// Bad: wrapper div purely to satisfy JSX's single-root rule
function UserMeta({ user }) {
  return (
    <div>                          // semantic noise
      <dt>{user.label}</dt>
      <dd>{user.value}</dd>
    </div>                         // also INVALID: div isn't allowed between dl and dt
  );
}

// Good: Fragment keeps the DOM flat and the dl valid
// (Note: HTML does allow <div> as a dl child wrapper since 2017 —
//  the one place div-in-dl is legal — but Fragment is still cleaner.)
function UserMeta({ user }) {
  return (
    <>
      <dt>{user.label}</dt>
      <dd>{user.value}</dd>
    </>
  );
}

Fragment discipline checklist for code review:

  • Default to <>…</>; only add a real element when it carries semantics or styling.
  • Content-model violations to watch for: only <li> in <ul>/<ol>, only <tr>-family in tables (a stray div makes the browser hoist it out of the table during parsing), no block elements inside <p> (the parser auto-closes the p, so your DOM won't match your JSX — and React hydration will mismatch).
  • Polymorphic as props on design-system primitives (<Stack as="ul">) let layout components stop forcing divs.
Invalid nesting like <p><div></div></p> or a div directly inside <tbody> doesn't error — the HTML parser silently restructures the DOM. In SSR'd React this produces hydration mismatches that look like framework bugs. React 19 even ships specific diagnostics for this. Great interview anecdote material.

The accessibility tree & accessible names T1

What the accessibility tree actually is

The browser builds a parallel tree from the DOM (plus CSS!) and exposes it to the OS accessibility APIs — UIA on Windows, NSAccessibility on macOS, AT-SPI on Linux, and via those to screen readers, voice control, switch devices. Key facts a senior should know:

  • It's derived from DOM + CSS + ARIA. display: none and visibility: hidden remove nodes from the a11y tree; opacity: 0 and clip-path tricks do not. The classic "visually-hidden" utility class exists precisely to hide visually while staying in the tree.
  • Nodes that add nothing (bare divs/spans with no semantics, name, or interactivity) are pruned or flattened — this is why div soup is survivable but still costly.
  • You can inspect it directly: Chrome DevTools → Elements → Accessibility pane, or the full-page a11y tree toggle. Firefox has an excellent Accessibility inspector with issue auditing. Demoing this in a pairing interview is a strong signal.
  • CSS can change semantics: display: contents historically removed elements from the a11y tree in some browsers (mostly fixed, but table display overrides still cause issues); generated content (::before/::after) is exposed as text in the tree unless content: "…" / "" alternative-text syntax says otherwise.

The role / name / state / value model

Every node in the a11y tree is described to AT along four axes. This model is the mental frame for all ARIA work:

AxisWhat it answersExample for a checked checkbox labelled "Subscribe"
RoleWhat kind of thing is it?checkbox
NameWhat is it called?"Subscribe"
State/PropertiesWhat condition is it in? How does it relate to other nodes?checked, focusable, described-by → hint text
ValueCurrent value (sliders, inputs, comboboxes)—

A screen reader stitches these into an utterance: "Subscribe, checkbox, checked." When you build a custom widget, your job is to make all four axes correct — native elements do it for free; ARIA lets you do it manually; and getting only two of four right is what makes half-accessible widgets worse than plain markup.

Accessible name computation — the precedence order

Interviewers love this because most candidates guess. The AccName spec computes a name by walking these sources in order, first match wins:

  1. aria-labelledby — highest priority. Resolves space-separated IDs, concatenates their computed names (can even reference the element itself plus another node, and works with hidden elements).
  2. aria-label — a literal string. Overrides content entirely.
  3. Host-language labelling — <label for> / wrapping label for form controls, alt for images, <caption> for tables, <legend> for fieldsets.
  4. Name from content — the element's subtree text, for roles that allow it (button, link, heading… but not textbox or grouping roles).
  5. title attribute — last resort. Unreliable (no touch/keyboard exposure of the tooltip) — treat it as "better than nothing" only.
  6. placeholder — some browsers use it as a fallback name for inputs. Never rely on it: it disappears on input and fails contrast.
<!-- What is announced? -->
<button aria-label="Close dialog">×</button>
<!-- → "Close dialog, button". aria-label beats content ("×"). -->

<button aria-labelledby="a b" aria-label="ignored">also ignored</button>
<span id="a">Delete</span> <span id="b" hidden>invoice #482</span>
<!-- → "Delete invoice #482" — labelledby wins, and even reads the hidden span. -->
Three name-computation traps:
  • aria-label on non-interactive elements (plain div/span with no role) is ignored by much AT — a name needs a role that supports naming.
  • Label-in-name (WCAG 2.5.3): if visible text says "Search" but aria-label="Find products", voice-control users saying "click Search" fail. The accessible name must contain the visible label. Prefer aria-labelledby pointing at the visible text.
  • Icon buttons with no name announce as "button" — the single most common audit finding in React apps. Every icon-only control needs aria-label or visually-hidden text.
/* The canonical visually-hidden utility — in the a11y tree, not on screen */
.visually-hidden {
  position: absolute;
  width: 1px; height: 1px;
  padding: 0; margin: -1px;
  overflow: hidden;
  clip-path: inset(50%);
  white-space: nowrap;
  border: 0;
}

aria-describedby vs the name

Descriptions are the second string AT reads, after a pause — used for hints, error messages, and extended context. Name = "what is it"; description = "what else should I know." A senior distinguishes these instinctively: put the essential identity in the name, supplementary info in the description. There's also aria-description (string form, now widely supported) when no visible node exists to reference.

ARIA done right T1

The rules of ARIA (the first one is the whole game)

  1. Don't use ARIA if a native element does the job. <button> over role="button", <dialog> over role="dialog", <details> over a hand-rolled disclosure. Native gives you keyboard behavior, focus semantics, and state for free; ARIA gives you only the announcement. "No ARIA is better than bad ARIA" is a direct quote from the APG and worth saying verbatim.
  2. Don't change native semantics unless you truly must (<h2 role="tab"> destroys the heading).
  3. All interactive ARIA widgets must be keyboard-operable — the spec gives you zero behavior.
  4. Never put role="presentation" or aria-hidden="true" on focusable elements.
  5. All interactive elements must have an accessible name.

States & properties you must know cold

  • aria-expanded — goes on the trigger, not the panel. true/false, toggled in JS. Announces "expanded/collapsed."
  • aria-selected vs aria-checked vs aria-pressed: selection within a composite (tabs, options) vs check state (checkbox/radio/menuitemcheckbox) vs toggle button. Using the wrong one produces bizarre announcements.
  • aria-current — "you are here": page for the active nav link, step for wizards, date for calendars. Far better than a bare .active class.
  • aria-hidden="true" — removes the subtree from the a11y tree only (still visible!). Pitfalls: it does not remove focusability, so a hidden-but-focusable element becomes a keyboard black hole; never put it on an ancestor of the focused element (browsers now warn); aria-hidden="false" cannot re-reveal inside a hidden subtree — it's effectively one-way.
  • aria-disabled vs disabled: the native attribute removes from tab order and blocks events; aria-disabled only announces. Senior pattern: for a submit button you want discoverable-but-inactive, aria-disabled="true" + JS guard keeps it focusable so SR users can find it and hear why it's disabled (describedby to the reason) — a deliberate UX trade-off worth articulating.
  • aria-controls — weak support, low value; include it for APG conformance but never rely on it.

Live regions — the toast example

Live regions solve "the DOM changed but focus is elsewhere." Rules that trip people up: the live region container must exist in the DOM before the announcement (AT registers regions at parse/insert time — injecting a div with aria-live and text simultaneously often announces nothing), and you announce by changing its text content.

<!-- Mounted once at app root, empty -->
<div id="toast-region" role="status" aria-live="polite" class="visually-hidden"></div>
<div id="alert-region" role="alert" class="visually-hidden"></div>
// React: a global announcer — the visual toast and the announcement are decoupled
function useAnnounce() {
  return useCallback((msg, assertive = false) => {
    const region = document.getElementById(assertive ? 'alert-region' : 'toast-region');
    region.textContent = '';              // clear so repeated identical messages re-announce
    requestAnimationFrame(() => { region.textContent = msg; });
  }, []);
}

// usage after a save:
announce('Draft saved');                  // polite: waits for SR to finish speaking
announce('Session expired — log in again', true); // assertive: interrupts
  • polite (= role="status"): queued after current speech. Default choice — 95% of cases.
  • assertive (= role="alert"): interrupts immediately. Reserve for time-critical errors; overuse trains users to ignore your app or literally cuts off what they were reading.
  • aria-atomic="true": read the whole region, not just the changed node. role="alert" implies atomic + assertive.
  • Don't put focusable content inside a visually-hidden live region; visual toasts with actions ("Undo") need a separate strategy — e.g., announce "Draft deleted, press Alt+U to undo" and a real keyboard shortcut, since moving focus to a transient toast is hostile.

Keyboard & focus management T1

Tab order and tabindex

Sequential focus order follows DOM order of focusable elements (interactive elements + anything with tabindex ≥ 0). CSS reordering (flex order, grid placement, position) does not change tab order — a WCAG 2.4.3 (Focus Order) failure waiting to happen when visual and DOM order diverge. (CSS reading-flow is emerging to fix this; know it exists.)

ValueEffectWhen
tabindex="0"In tab order, at natural DOM positionMaking a custom widget host focusable; scrollable regions with no interactive children (Chrome now does this automatically for keyboard users)
tabindex="-1"Programmatically focusable (el.focus()), not tabbableFocus targets: headings on route change, error summaries, roving-tabindex "inactive" items, modal containers
tabindex="1+"Jumps ahead of everything, in tabindex orderNever. It creates a shadow ordering that's unmaintainable and hijacks user expectations. The fix for bad tab order is fixing DOM order. Say this categorically.
Adding tabindex="0" to non-interactive content (divs, list items) so keyboard users can "reach" it is an anti-pattern — screen readers already read non-focusable content, and extra tab stops slow everyone down. Focusable ⇢ actionable.

Focus trap for modals — full implementation

Prefer <dialog>.showModal() (next section) which traps for free via making the rest of the page inert. But interviewers still ask for the hand-rolled version to test mechanics:

const FOCUSABLE = [
  'a[href]', 'button:not([disabled])', 'input:not([disabled])',
  'select:not([disabled])', 'textarea:not([disabled])',
  '[tabindex]:not([tabindex="-1"])', '[contenteditable="true"]'
].join(',');

function trapFocus(container) {
  const previouslyFocused = document.activeElement;   // 1. remember opener

  function getTabbables() {
    // re-query each time: modal content may change (async load, validation)
    return [...container.querySelectorAll(FOCUSABLE)]
      .filter(el => el.offsetParent !== null || el === document.activeElement);
  }

  function onKeydown(e) {
    if (e.key === 'Escape') return close();
    if (e.key !== 'Tab') return;
    const tabbables = getTabbables();
    if (tabbables.length === 0) { e.preventDefault(); return; }
    const first = tabbables[0], last = tabbables[tabbables.length - 1];
    // 2. wrap at the edges
    if (e.shiftKey && document.activeElement === first) {
      e.preventDefault(); last.focus();
    } else if (!e.shiftKey && document.activeElement === last) {
      e.preventDefault(); first.focus();
    }
  }

  container.addEventListener('keydown', onKeydown);
  (getTabbables()[0] ?? container).focus();            // 3. move focus in

  function close() {
    container.removeEventListener('keydown', onKeydown);
    previouslyFocused?.focus();                        // 4. RESTORE focus — the step everyone forgets
  }
  return close;
}

Completeness points to mention unprompted: the trap alone doesn't stop screen reader virtual cursor browsing behind the modal — you also need inert on the background (or aria-hidden on siblings, the old library approach); scroll-lock the body; and if the modal contents can change, tabbables must be recomputed per keydown (as above) rather than cached.

Roving tabindex — the composite-widget pattern

// React roving tabindex for a toolbar/tablist: ONE tab stop, arrows inside
function Toolbar({ items }) {
  const [active, setActive] = useState(0);
  const refs = useRef([]);

  function onKeyDown(e) {
    const delta = { ArrowRight: 1, ArrowLeft: -1 }[e.key];
    let next;
    if (delta !== undefined) next = (active + delta + items.length) % items.length;
    else if (e.key === 'Home') next = 0;
    else if (e.key === 'End')  next = items.length - 1;
    else return;
    e.preventDefault();
    setActive(next);
    refs.current[next].focus();     // move REAL focus, not just state
  }

  return (
    <div role="toolbar" aria-label="Formatting" onKeyDown={onKeyDown}>
      {items.map((item, i) => (
        <button key={item.id}
                ref={el => (refs.current[i] = el)}
                tabIndex={i === active ? 0 : -1}>   {/* the "roving" part */}
          {item.label}
        </button>
      ))}
    </div>
  );
}

Why one tab stop? A page with a 20-button toolbar and a 30-tab tablist would need 50 Tab presses to cross. Composites collapse to one stop each; arrows navigate within. This matches native OS widgets, which is exactly what AT users expect.

:focus-visible and focus styling

/* Modern reset: ring for keyboard/AT focus, none for mouse clicks */
:focus { outline: none; }                  /* only safe BECAUSE of the next rule */
:focus-visible {
  outline: 2px solid var(--ring);
  outline-offset: 2px;
}

The heuristic: browsers apply :focus-visible when focus arrived via keyboard (or the element expects keyboard input, like text fields — those match even on click). This ended the decade of designers demanding outline: none and breaking keyboard users. WCAG 2.4.7 requires a visible indicator; WCAG 2.2 added 2.4.11 Focus Not Obscured — sticky headers/cookie banners must not cover the focused element (test by tabbing under a sticky header; fix with scroll-padding-top).

Focus in SPAs — the problem native pages solve for free

A full page load resets focus to the document top and announces the new title. Client-side routing does neither — after "navigation," focus is still on the clicked link (which may no longer exist), and SR users hear nothing. The senior fix:

// Route-change focus management (React Router flavored)
function RouteAnnouncer() {
  const { pathname } = useLocation();
  const headingRef = useRef(null);
  const firstRender = useRef(true);

  useEffect(() => {
    if (firstRender.current) { firstRender.current = false; return; } // don't steal focus on load
    document.title = titleFor(pathname);       // keep <title> in sync — SRs announce it
    headingRef.current?.focus();               // h1 with tabindex="-1"
  }, [pathname]);

  return <h1 tabIndex={-1} ref={headingRef}>{titleFor(pathname)}</h1>;
}

Options and trade-offs to narrate: focus the h1 (best: user hears the new page name and reading position is at the content start) vs focus a wrapper vs only announcing via live region (Next.js's route announcer approach — less disruptive but doesn't reposition the reading cursor) vs focusing the skip link. Also suppress outline on the programmatically-focused heading (:focus-visible handles it) and remember scroll restoration is a separate concern.

Dynamic content: where does focus go?

  • Deleting the focused thing (e.g., row delete button): focus vanishes to <body> — keyboard users are teleported to the page top. Move focus to the next row's equivalent control, or the list container, before/as removal.
  • Async content appearing: usually announce (live region), don't steal focus — unless the user's action explicitly requested it (opening a form → focus first field).
  • Loading states: aria-busy="true" on the region, plus a status announcement for long operations ("Loading results…" / "12 results loaded").

WCAG 2.2 practical T1

POUR + conformance levels in one breath

WCAG organizes success criteria under four principles — Perceivable (can users detect it: alt text, contrast, captions), Operable (can they use it: keyboard, timing, target size), Understandable (predictable, labeled, error-tolerant), Robust (parses correctly for AT: name/role/value). Levels: A = bare minimum (missing = broken for some users), AA = the legal/industry standard (ADA rulings, EU EAA 2025, Section 508 all target AA), AAA = aspirational, not required site-wide (7:1 contrast, sign language). "We target WCAG 2.2 AA" is the correct professional posture; the EAA making AA legally enforceable across the EU since June 2025 is a current-events point worth dropping.

How to audit — the layered strategy

  1. Automated (catches ~30-40%): axe-core (the engine behind most tools), Lighthouse a11y score (uses axe under the hood), browser a11y panes. Automation finds missing names, contrast, invalid ARIA — it cannot find wrong reading order, meaningless alt text, focus traps, or bad announcements.
  2. Keyboard pass (5 min): unplug the mouse. Tab through — can you reach everything, see where you are, escape every widget, activate everything with Enter/Space?
  3. Screen reader smoke test (15 min): VoiceOver (Cmd+F5, free on macOS) or NVDA (free, Windows — and NVDA+Chrome is the most common real-user combo). Navigate by headings and landmarks; operate your core flow. You are not simulating a real user, but you catch the top 20 issues.
  4. Zoom/reflow: 400% zoom, 320px width, forced-colors mode (Windows High Contrast — test with forced-colors: active).

Accessible error messaging — the full pattern

<label for="card">Card number</label>
<input id="card" inputmode="numeric" autocomplete="cc-number"
       aria-invalid="true"
       aria-describedby="card-err card-hint">

  1. aria-invalid="true" — announces "invalid entry" on focus. Only set it after validation fails, never on pristine fields.
  2. aria-describedby to the error (and hint) — the message is read after the label when the field is focused. Order the IDs: error first.
  3. On submit failure: move focus to the first invalid field, or to an error-summary box (tabindex="-1", role="alert" or focus-target heading) with anchor links to each field — the GOV.UK pattern, gold standard for long forms.
  4. Inline errors appearing on blur → a polite live region or the describedby-on-refocus handles announcement; don't make every error role="alert" or typing becomes a scream of interruptions.
  5. Never color-only (WCAG 1.4.1): icon + text + border, not just red.

What showModal() gives you that took hand-rolled modals ~200 lines:

  • Top layer rendering — a browser-managed layer above all stacking contexts. No z-index: 9999 wars, no React portal needed to escape overflow: hidden ancestors. (Top layer is its own concept: dialog, popover, and fullscreen all use it.)
  • The rest of the document becomes inert — unfocusable, unclickable, hidden from AT. This is a true focus trap including the SR virtual cursor, which JS traps can't fully achieve.
  • Esc to close (cancelable via the cancel event), focus restoration to the previously focused element on close, initial focus (first focusable or autofocus).
  • ::backdrop pseudo-element for the scrim: dialog::backdrop { background: rgb(0 0 0 / .5); backdrop-filter: blur(4px); }
  • Implicit role="dialog"; you still owe it an accessible name (aria-labelledby the heading).
Dialog gotchas: (1) show() vs showModal() are different beasts — only showModal gets backdrop/inert/top-layer; (2) an open attribute set manually renders it non-modally; (3) body scroll is NOT locked automatically — add body:has(dialog[open]) { overflow: hidden; } or the newer overscroll-behavior containment; (4) in React, drive it imperatively via ref + useEffect, since modal-ness isn't attribute-reflected. Newer platform sugar: closedby="any" attribute for light-dismiss dialogs, and requestClose() which respects cancel handlers.

Why CSR hurts SEO — the precise version

The mid-level answer is "Google can't see JS." The senior answer: Googlebot does render JavaScript (evergreen Chromium), but rendering is a second wave — pages queue for the render step, so indexing of client-rendered content is delayed and crawl-budget-expensive on large sites; content behind interactions or slow hydration may never be seen; and every other crawler (Bing historically weaker, OG scrapers, many LLM crawlers) is far less capable. Plus CSR's real killers are indirect: empty initial HTML means no meaningful title/meta per route without SSR, soft-404s (JS renders "Not found" with an HTTP 200), and Core Web Vitals — a ranking signal — suffer from client-side data-fetch waterfalls.

Without it, mobile browsers assume a ~980px legacy layout viewport and scale it down. width=device-width matches the layout viewport to the ideal viewport; initial-scale=1 sets 1 CSS px zoom. Never ship user-scalable=no or maximum-scale=1 — blocking pinch-zoom is a WCAG 1.4.4 failure (and iOS Safari ignores it anyway). Modern additions: viewport-fit=cover for notch/safe-area (with env(safe-area-inset-*)), interactive-widget=resizes-content to control virtual-keyboard resize behavior.

data-* attributes

<button data-testid="save-btn" data-analytics-id="cta.save" data-state="loading">Save</button>
btn.dataset.analyticsId          // "cta.save" — camelCased via the dataset DOMStringMap
[data-state="loading"] { opacity: .6; }   /* styling hook — Radix/Headless UI's entire styling API */

Legit uses: test hooks, analytics tags, CSS state hooks (often better than class toggling because the attribute is also inspectable state), delegation payloads. Anti-pattern: using dataset as a client-side database for large data — it serializes everything to strings in the DOM; keep real state in JS. Also: data-* is invisible to AT — never encode meaning there that users need.

contenteditable gotchas

  • Browsers generate wildly inconsistent markup for the same edit (Enter → <div> vs <p> vs <br> across engines/eras) — the reason Slate, Lexical, ProseMirror, Tiptap exist: they treat the DOM as a rendering target, keep a separate document model, and intercept beforeinput.
  • document.execCommand() is deprecated but has no full replacement; the modern primitives are beforeinput events + Input Events spec + Selection/Range APIs.
  • contenteditable="plaintext-only" (now Baseline) kills rich-paste — perfect for inline-rename UI without input-styling gymnastics.
  • Pasting brings hostile HTML — sanitize on paste (intercept, extract text/plain, or run through the emerging Sanitizer API); XSS via paste is a real vector.
  • React fights it: React can't diff user-mutated DOM, hence suppressContentEditableWarning and why editors mount an uncontrolled island.
  • A11y: a bare contenteditable div has no textbox semantics or label — add role="textbox" + aria-multiline="true" + a name, or users hear nothing.

Interview Q&A

Why does semantic HTML matter beyond "it's best practice"?

Three concrete payoffs. First, the accessibility tree: semantic elements carry implicit roles, names, states, and behavior — a <button> is focusable, keyboard-activatable with Enter and Space, and announced as a button with zero JS; recreating that on a div requires role, tabindex, two key handlers, and disabled-state management, and teams get it wrong constantly. Second, machine consumers beyond screen readers — search crawlers, reader modes, and LLM agents parse structure from elements, so semantics are free structured data. Third, maintainability: <nav> and <article> encode intent that class names can't be trusted to. The senior framing: semantics are an interface contract with every non-human consumer of your DOM, and native elements are the cheapest accessibility you'll ever buy — everything else on top of divs is rebuilding the platform, badly.

Walk me through how a screen reader user experiences a well-structured page vs div soup.

Screen reader users navigate by structure, not linearly. On a well-structured page they'll hear the title, jump landmarks (banner → nav → main) with a single key, pull up the headings list to scan the outline like sighted users scan visually, and hear "navigation, Primary" or "list, 8 items" as context. With div soup, every one of those affordances is gone: the landmarks list is empty, the headings list is empty or lies, lists don't announce counts, and the only strategy left is arrowing through every text node linearly. Same content, but one page takes 20 seconds to orient in and the other takes 10 minutes. I'd also mention the flip side: over-marking (15 regions, nav around every link cluster) makes landmark navigation noise. Structure should mirror the visual hierarchy a sighted user perceives.

What's the accessible name of a button, and how is it computed when multiple sources exist?

The accessible name is computed per the AccName spec with strict precedence: aria-labelledby wins over everything (it resolves referenced elements' text, even hidden ones, and concatenates multiple IDs); then aria-label; then host-language labeling (label[for], alt, legend, caption); then name-from-content for roles that allow it (button/link text); then title as a weak fallback; placeholder is a nonstandard last resort in some browsers. So <button aria-label="Close">×</button> announces "Close" — content is overridden. The senior additions: aria-label on a role that doesn't support naming (a plain div) is ignored; and WCAG 2.5.3 Label in Name means the accessible name must contain the visible text, or voice-control users saying "click Submit" hit nothing — which is why I prefer aria-labelledby pointing at visible text over duplicating strings in aria-label, which also drift and miss translation pipelines.

What is the first rule of ARIA and why does it exist?

"Don't use ARIA if a native HTML element or attribute gives you the semantics and behavior you need." It exists because ARIA changes only what's announced — it grants zero behavior. role="button" on a div tells the screen reader "button," so the user presses Space… and the page scrolls, because nothing implemented Space activation. That's worse than no ARIA: it makes a promise and breaks it, and the WebAIM Million surveys consistently find pages using ARIA have more detected errors than pages without. Native elements ship the whole contract — focus, keyboard, states, form participation. The corollaries matter too: never put aria-hidden or role=presentation on focusable elements, and every ARIA widget role you claim obligates you to its full APG keyboard pattern. My rule of thumb in review: every ARIA attribute is a liability that needs justification; the best diff is one that replaces ARIA with a native element.

aria-label vs aria-labelledby vs aria-describedby — when do you use each?

aria-labelledby when visible text already exists — reference it. It wins precedence, stays in sync with the UI, gets translated with the page, and satisfies Label in Name. aria-label when there is genuinely no visible text: icon-only buttons, distinguishing multiple navs ("Primary" vs "Breadcrumb"). Its costs: it's invisible (drifts from the UI silently), and it's frequently missed by translation tooling. aria-describedby is not a name at all — it's the supplementary second sentence AT reads after the name: hints, error messages, consequences ("This permanently deletes the project"). Name answers "what is this," description answers "what else should I know." One more: aria-description for string-form descriptions without a DOM node, now well supported.

Explain aria-live. How would you make toast notifications accessible?

Live regions announce DOM changes to AT without moving focus. aria-live="polite" (equivalent to role="status") queues after current speech; assertive (role="alert") interrupts — reserved for urgent errors because interrupting is hostile. For toasts: mount a persistent, empty, visually-hidden live region at app root at load time — the critical gotcha is AT registers live regions when they enter the DOM, so injecting a region together with its message often announces nothing. To announce, set the region's textContent (clearing first so repeated identical messages re-fire). I decouple the visual toast from the announcement channel. For toasts with actions like Undo, I don't move focus to a transient element — I announce the action's keyboard shortcut instead, and make sure the toast persists long enough or is recallable (WCAG timing). Also aria-atomic="true" so the whole message is read, not the diffed text node.

Design an accessible autocomplete/combobox. What are the moving parts?

Per ARIA 1.2 APG: the input itself gets role="combobox", aria-expanded reflecting popup state, aria-controls pointing at the popup, and aria-autocomplete="list". The popup is role="listbox" with role="option" children. The key architecture decision is focus strategy: DOM focus stays on the input (so the user can keep typing) and aria-activedescendant on the input points at the visually-highlighted option's ID — virtual focus, versus roving tabindex which moves real focus and would leave the text field. Keyboard: ArrowDown opens/moves down, ArrowUp up, Enter commits and closes, Escape closes then clears, Home/End in the text. Then the parts everyone forgets: a polite live region announcing "N results available" as the list filters (sighted users see it shrink; SR users get silence otherwise), aria-selected on the active option, scrollIntoView({block:'nearest'}) since virtual focus doesn't scroll, and — the production trap — with virtualized lists the activedescendant ID must exist in the DOM. Native autocomplete must be off. And I'd say out loud: this is the hardest common widget, so my first move in a real product is reaching for React Aria/Downshift rather than hand-rolling, unless the interview is the point.

When would you use role="menu" — and when is it wrong?

role="menu" means a desktop-application menu of actions — like a right-click menu or a row's "…" overflow with Duplicate/Archive/Delete. Claiming it commits you to the full contract: arrow-key navigation with a single tab stop, Escape closes and returns focus to the trigger (which needs aria-haspopup="menu" + aria-expanded), first-letter navigation, close-on-activate — and screen readers may switch interaction modes when they hear "menu," so unmet expectations actively confuse. It's wrong for the most common case people use it: navigation dropdowns. A nav dropdown should be a disclosure — button with aria-expanded revealing a <nav> with plain links — because links navigate, Tab should work normally, and no app-menu semantics apply. And for "choose one value" it's wrong again — that's listbox/option (selection, aria-selected) or just a native select. The taxonomy: actions → menu, selection → listbox, navigation → disclosure + links.

Explain tabindex values. Why is positive tabindex forbidden?

tabindex="0" inserts an element into the natural tab order at its DOM position — for making custom widget hosts or scrollable regions focusable. tabindex="-1" makes it programmatically focusable via .focus() but not tabbable — for focus targets like the h1 on route change, error summaries, and the inactive items in roving tabindex. Positive values create a separate priority ordering that jumps ahead of every naturally-ordered element on the page. It's forbidden in practice because it's globally fragile — any new tabindex="1" anywhere reorders everything; it violates user expectations that focus follows reading order (WCAG 2.4.3); and it's a band-aid whose real fix is reordering the DOM. In 7 years I've never seen a legitimate use. Related senior point: CSS visual reordering (flex order, grid areas) doesn't change tab order, so big visual/DOM order divergence creates the same class of bug without any tabindex at all.

Implement a modal focus trap. What do people forget?

Mechanics: on open, store document.activeElement, move focus to the first tabbable (or the dialog container with tabindex="-1"), listen for Tab/Shift+Tab and wrap focus at the edges by querying tabbables at keydown time (content changes — cached lists go stale), close on Escape, and on close restore focus to the stored opener. The four commonly-forgotten pieces: (1) focus restoration — without it keyboard users are dumped at body and have to tab back through the whole page; (2) the JS trap doesn't stop the screen reader virtual cursor from wandering into the background — you need inert on the rest of the app (or the old aria-hidden-on-siblings hack); (3) scroll-locking the body; (4) the dialog needs role="dialog", aria-modal, and an accessible name. Then the senior kicker: in 2026 I wouldn't hand-roll this — <dialog>.showModal() gives the trap, true inertness, top-layer rendering, Escape, and focus restoration natively; the hand-rolled version is now an interview exercise and a legacy-codebase reality, not the right first choice.

How do you handle focus on client-side route changes in a React SPA?

The problem: a real navigation resets focus to the document and announces the new title; client routing does neither — after clicking a nav link, focus sits on the old link (or nothing) and screen reader users get silence. My approach: on pathname change (skipping first render so I don't steal focus on load), update document.title, then move focus to the new view's <h1 tabindex="-1"> — the user hears the page name and their reading position is at the content start. Alternatives with trade-offs: focusing a wrapper or skip link (less informative), or announce-only via a live region — Next.js's built-in route announcer does this — which is less disruptive but doesn't move the reading cursor, so it suits soft transitions like tab-like sub-routes. Scroll restoration is a separate axis handled by the router. And for modals-as-routes, focus goes to the dialog instead. The point I'd emphasize: this is app architecture, so it belongs in the router layout once, not per-page.

What's new in WCAG 2.2, and what level do you target and why?

WCAG 2.2 (October 2023) added nine criteria — the ones that change frontend work: 2.5.8 Target Size Minimum (AA, 24×24 CSS px or equivalent spacing), 2.4.11 Focus Not Obscured (AA — sticky headers can't cover the focused element), 3.3.8 Accessible Authentication (AA — no cognitive tests; allow paste and password managers), 3.2.6 Consistent Help, 3.3.7 Redundant Entry (don't make users re-type info within a flow), and 2.5.7 Dragging Movements (single-pointer alternative to drag). It also removed 4.1.1 Parsing, since modern browsers recover from markup errors uniformly. We target AA: A is a floor with unacceptable gaps, AAA is explicitly not intended as a site-wide requirement (7:1 contrast, sign language), and AA is what regulation converges on — ADA title II rules, Section 508, the EN 301 549 standard behind the European Accessibility Act, which became enforceable for consumer products in June 2025. So AA is both the ethical baseline and the legal-risk answer.

How would you build accessibility into a team's process, not just fix audit findings?

Layered, shifting left. Design: annotations for focus order, accessible names, and announcements in specs; contrast checked in the design tool. Build: the highest-leverage move is an accessible-by-default design system — fix the Button/Modal/Combobox once and every product team inherits it; plus eslint-plugin-jsx-a11y at lint time. Test: jest-axe/vitest-axe on components, @axe-core/playwright in E2E against real states (modal open, form errored — static-page scans miss the interesting states), and Testing Library's getByRole queries which make tests fail when semantics break. Gate: CI fails on new violations — a ratchet. Then the honest limits: automation catches maybe 30-40% of issues; it can't judge alt-text quality, focus flow sanity, or announcement UX. So: a 15-minute keyboard + VoiceOver/NVDA smoke test in the definition of done for new flows, periodic expert audits, and ideally paid testing with actual disabled users. The framing I'd give: automated checks prevent regression; humans establish correctness.

Native constraint validation vs a form library's validation — how do you decide?

Native gives you declarative constraints (required, type, minlength, pattern) driving the ValidityState API, localized messages, CSS hooks, and submit-blocking with zero JS — great baseline, and setCustomValidity folds custom rules into the same machinery. But the native error bubbles are unstylable, inconsistent across browsers, show one error at a time, and vanish — product/design teams reject them. Libraries give full control of copy, timing, and cross-field/async rules, but you must rebuild the accessibility natively provided. My production answer is the hybrid: keep the HTML constraint attributes (they still power autofill, AT hints, mobile keyboards, and the validity object), set novalidate on the form to suppress bubbles, validate through the library, and render errors with the full accessible pattern — aria-invalid set only after failure, aria-describedby to the message, focus moved to the first invalid field or a GOV.UK-style error summary on submit. Also worth naming: :user-invalid now gives interaction-aware invalid styling in pure CSS, fixing the old "red form on page load" reason people avoided :invalid.

Why is type="number" wrong for credit cards and ZIP codes?

type="number" models a quantity: it strips leading zeros (killing ZIP codes like 02134), accepts scientific-notation characters (e, +, -, .), silently changes value on scroll-wheel while focused, shows spinners that make no sense for identifiers, hits floating-point representation on long digit strings like card numbers, and its valueAsNumber semantics fight you. The correct pattern is type="text" with inputmode="numeric" (mobile digit keyboard without number semantics) plus the right autocomplete token — cc-number, postal-code, one-time-code — which is where the real UX win lives via autofill. This separates three orthogonal attributes: type = semantics and validation, inputmode = virtual keyboard, autocomplete = autofill mapping. GOV.UK's design system documented this migration publicly, which is a nice authority to cite. Rule: number is for "how many," text+inputmode is for "digits that happen to be an identifier."

<dialog> vs the popover attribute — when do you reach for each?

The real decision is modal vs non-modal. dialog.showModal() is for blocking interactions — confirmations, must-complete flows: it renders in the top layer, makes the rest of the document truly inert (including for the SR virtual cursor), handles Escape, and restores focus on close; it carries role="dialog" implicitly. popover is a behavior, not a semantic: any element gets top-layer rendering, declarative toggling via popovertarget with automatic aria-expanded wiring, and light dismiss (auto flavor) — but no role, no focus trap, focus stays on the trigger. That's exactly right for menus, tooltips, pickers, and toasts (manual flavor), where trapping focus would be wrong; you add the appropriate role yourself. They compose — <dialog popover> is a light-dismissible non-modal dialog. What both kill: z-index wars and portal gymnastics, since top layer sits above all stacking contexts. Remaining gaps I'd flag: body scroll isn't locked for modals automatically, anchor positioning for popovers is only now maturing, and in React you drive dialog imperatively via ref since modality isn't an attribute.

What does the inert attribute do, and what did it replace?

inert makes a subtree completely non-interactive: unfocusable, unclickable, unfindable by find-in-page, and removed from the accessibility tree. It's the primitive behind showModal's background behavior, exposed for custom use. It replaced a genuinely bad stack of hacks: aria-hidden="true" on all siblings of a modal (which hid content from AT but left it clickable and focusable), recursive tabindex="-1" walking (which destroyed elements' original tabindex state), and event-capture blocking — libraries like focus-trap and react-aria maintained hundreds of lines doing this incompletely, because none of it stopped the screen reader virtual cursor. Uses beyond modals: off-screen carousel slides, closed off-canvas drawers, content under loading overlays, disabled multi-step-form sections. Two caveats: it's all-or-nothing for the subtree — no un-inerting a child; and it provides no visual signal, so you pair it with styling. Baseline since 2023, so it's safe to rely on.

Googlebot executes JavaScript — so why does CSR still hurt SEO?

Because "Google renders JS" comes with asterisks that compound. Rendering is a second, queued wave — HTML is crawled immediately, but JS rendering waits for Google's render infrastructure, so fresh or frequently-changing content indexes late, and on large sites crawl/render budget means some pages may not get rendered at all. Content behind interaction or slow hydration is never seen. Every other consumer is worse: OG/Twitter scrapers run zero JS (so all shared links show the generic app-shell card — often the most business-visible symptom), and many other crawlers and LLM agents are HTML-first. Then the indirect damage: CSR makes per-route title/meta/canonical an afterthought, client-side "404s" return HTTP 200 creating soft-404s, client-side redirects are invisible, and Core Web Vitals — a ranking input — suffer from fetch waterfalls after hydration. Fixes in order of preference: SSG/ISR for content pages (CDN-cached, cheap), SSR for dynamic ones, hybrid architectures (SSR the marketing/product surface, keep the logged-in app CSR — usually the right cost/benefit), and dynamic rendering for bots only as a legacy retrofit since Google now discourages it. Plus hygiene that costs nothing: real <a href> links, correct status codes from the edge, sitemaps.

When do Web Components beat React components? Give a real scenario.

Scenario: a company with products in React, Angular, and a legacy server-rendered stack needs one design system. Building it three times triples cost and guarantees drift; building it as custom elements (probably with Lit) gives one implementation every stack consumes, with Shadow DOM guaranteeing the host page's CSS can't break components — that's why Adobe Spectrum, Microsoft FAST, and GitHub's elements went this way. Second scenario: embeds on third-party pages (chat widget, checkout button) where style collision is certain. Third: micro-frontends needing hard isolation between team deployments. Where React wins: single-framework orgs (WC interop tax buys nothing), anything needing rich composition — context, hooks, suspense, render props have no WC equivalent — and SSR-centric stacks, though declarative shadow DOM has closed much of that gap and React 19 finally handles custom element properties correctly. The synthesis: they're different layers — WCs are a distribution/encapsulation format for primitives; frameworks are application architecture. Mature setups ship WC primitives with thin framework wrappers. I'd also volunteer the honest costs: cross-shadow-boundary ID references break aria-labelledby (a real a11y pain the Reference Target proposal is fixing), and form participation needs ElementInternals.

defer vs async vs type="module" — and when does each matter?

A bare script blocks parsing while it fetches and executes — legacy-only. defer fetches in parallel and executes after parsing completes, preserving order among deferred scripts, before DOMContentLoaded — right for app code with dependencies. async fetches in parallel and executes the moment it arrives, interrupting the parser, with no order guarantee — right only for independent scripts like analytics; an async dependency graph is a race condition. type="module" is deferred by default, plus strict mode, module scope, CORS-mode fetching, deduplication, and top-level await; module + async means "execute when the graph is ready" — good for independent modern widgets. Details that signal depth: defer is ignored on inline scripts but module inline scripts do defer; async scripts can land before or after DOMContentLoaded depending on network; and in bundler reality, your framework emits module/defer scripts already, so where this knowledge actually bites is third-party tags — the difference between an analytics snippet being async vs blocking is directly visible in TBT and LCP.

What are the gotchas of aria-hidden="true"?

It removes a subtree from the accessibility tree only — the content remains fully visible and, critically, focusable. The failure modes: (1) hiding something focusable creates ghost tab stops — keyboard focus lands on an element AT can't describe, announcing nothing or "blank" — so any aria-hidden subtree must also be made unfocusable (tabindex="-1"s or better, inert, which does both jobs correctly); (2) applying it to an ancestor of the currently-focused element strands focus — browsers now log warnings and this is the source of the React "Blocked aria-hidden on an element because its descendant retained focus" warning people paste into issues; (3) aria-hidden="false" does not re-expose content inside a hidden ancestor — it's effectively one-way, a spec subtlety most people learn the hard way; (4) it's routinely misused where display:none/hidden (hide from everyone) or visually-hidden CSS (hide from sighted only — the opposite tool) was intended. Legitimate uses are narrow: decorative icons next to text labels, and the legacy modal-background hack that inert has replaced.

How does Shadow DOM style encapsulation actually work — what gets in and out?

Shadow DOM scopes selector matching: page stylesheets can't match nodes inside a shadow root, and inner styles can't match outside — that's the isolation guarantee. But it's not a wall for everything: inheritance still flows through, so color, font-family, and — the key theming channel — CSS custom properties cascade into shadow trees. Deliberate crossing points: ::part(name) lets outside styles target elements the component explicitly exposed via the part attribute (with exportparts to forward through nesting); :host and :host(.selector) style the host from inside; ::slotted() styles light-DOM children placed into slots, top level only — slotted content actually stays in the light DOM and keeps taking page styles, which surprises people. Events bubble out but get retargeted (target becomes the host) and composed:false events don't escape. The sharpest edge for me is accessibility: IDREF associations — aria-labelledby, label for — cannot cross the boundary, which historically broke labeling in composed widgets; ARIA element reflection and the Reference Target proposal are the platform's answers. Trade-off summary: strong isolation for distribution scenarios, at the cost of theming ceremony (design-token custom properties become your public CSS API) and those boundary a11y issues.

A designer asks you to remove focus outlines because they're "ugly." Response?

I'd reframe rather than refuse: the requirement (WCAG 2.4.7, plus 2.4.11 and non-text contrast 3:1 on the indicator) is that keyboard users can always see where focus is — it says nothing about the browser default ring. So the modern fix is :focus-visible: mouse clicks show no ring (which is what actually bothered the designer), while keyboard navigation shows a designed indicator — offset outline in a brand color, whatever fits, as long as it meets 3:1 contrast and isn't hidden under sticky UI. I'd bake it into the design system reset (:focus { outline: none } :focus-visible { outline: 2px solid var(--ring); outline-offset: 2px }) so the conversation never recurs, and hand the designer focus-state specs as a first-class deliverable alongside hover states. The anecdote-level point: blanket outline: none was the single most common a11y regression of the 2010s, and :focus-visible exists precisely to end the designer-vs-keyboard-user standoff — both sides now fully win, which makes this an easy yes-and conversation instead of a fight.

How do you make a "delete row" interaction accessible end-to-end in a React table?

Naming first: each delete button needs a row-specific accessible name — "Delete, button" twenty times is useless — so aria-label={`Delete ${invoice.number}`} or aria-labelledby referencing the row's key cell. Confirmation: <dialog>.showModal() labelled by its heading, with the destructive consequence in the body (describedby); focus lands in the dialog and returns to the trigger on cancel automatically. The step everyone fails: after confirmed deletion the trigger element no longer exists, so native focus restoration lands on body, teleporting keyboard users to page top — I explicitly move focus to the next row's delete button (or previous, or the table container/heading if the list emptied) before React removes the node, typically in the mutation's success handler with a ref map. Then announce "Invoice 482 deleted" through a polite live region, since focus moved but nothing visually obvious happened for AT. If there's an undo, the announcement includes how to reach it. This question is a favorite of mine because it chains four skills — naming, dialog semantics, focus lifecycle on removed nodes, and live regions — and the focus-after-delete step cleanly separates people who've tested with a keyboard from people who haven't.

What is the top layer, and what problem does it solve?

The top layer is a browser-managed rendering layer above the entire document — above every stacking context, unaffected by ancestor z-index, overflow, transform, or filter. Elements enter it via showModal(), showPopover()/popover invocation, and fullscreen; within it, ordering is by promotion time (last opened on top), and ::backdrop renders per-element within it. It solves the two chronic overlay problems: z-index arms races (z-index: 2147483647 wars between your modal, a third-party chat widget, and a cookie banner) and the clipping/containment trap — a tooltip inside an overflow:hidden or transformed ancestor could never escape it, which is the entire reason React portals became mandatory for overlays. With dialog/popover, the element can stay in place in the DOM — keeping logical reading order and event bubbling intact — while rendering above everything, so portals stop being necessary for the rendering reason (you might still portal for other architectural reasons). Nuance worth adding: you can't z-index within the top layer — stacking is purely promotion order — and closing/reopening re-promotes, which is how you "raise" something.

How do you approach alt text — when is empty alt correct?

Alt text is a judgment call about function in context, not a description of pixels. Decision tree: purely decorative (ambience, spacers, an icon duplicating adjacent text) → alt="", which removes it from the a11y tree — and empty alt is load-bearing: omitting the attribute makes some screen readers announce the filename, which is worse. Informative → convey the information, not the medium: for a chart, the insight ("p95 latency dropped from 480ms to 120ms after the cache deploy"), not "a line chart"; complex data gets a short alt plus an adjacent table or long description. Functional (image inside a link/button) → describe the action, "Search", not the icon, "magnifying glass". Never prefix "image of" — the role announces that. In React codebases I enforce the decision with jsx-a11y's alt-text rule (which requires the attribute, forcing the deliberate empty-vs-content choice) and catch quality in review, because automation can verify presence, not meaning. Icon fonts and inline SVGs follow the same logic — aria-hidden when decorative, role="img" + aria-label or a <title> when meaningful.

Exercise 1: Audit and fix a div-soup card

The following renders a clickable product card. List every accessibility and semantic problem, then rewrite it correctly.

<div class="card" onclick="goToProduct(42)">
  <div class="card-title">Ergo Keyboard K3</div>
  <img src="k3.jpg">
  <div class="price">$79</div>
  <div class="btn" onclick="addToCart(42); event.stopPropagation()">Add to cart</div>
</div>
Solution

Problems: clickable div (no role, name, focus, or keyboard support); nested interactive-in-interactive; fake button div; missing alt; "title" div instead of a heading; no article/list semantics; stopPropagation coupling.

<article class="card">
  <h3 class="card-title">
    <a href="/products/42" class="card-link">Ergo Keyboard K3</a>
  </h3>
  <img src="k3.jpg" alt="" width="300" height="200">  <!-- decorative: name is in the link -->
  <p class="price">$79</p>
  <button type="button" onclick="addToCart(42)">
    Add to cart<span class="visually-hidden">: Ergo Keyboard K3</span>
  </button>
</article>
/* Whole-card clickability WITHOUT a div handler: the pseudo-element trick */
.card { position: relative; }
.card-link::after { content: ""; position: absolute; inset: 0; }
.card :is(button) { position: relative; z-index: 1; }  /* button stays above the stretched link */

Key points: navigation = real link (crawlable, open-in-new-tab, focusable); one accessible name per interactive element, made unique per card with hidden text; the stretched-link pattern gives the big hit area without nesting interactives.

Exercise 2: Build an accessible disclosure from scratch, then replace it with zero JS

Implement a "Show filters" toggle with correct ARIA and keyboard behavior in vanilla JS. Then rewrite it with no JavaScript at all and state what you lose/gain.

Solution 
<button id="t" aria-expanded="false" aria-controls="p">Show filters</button>
<div id="p" hidden>…filters…</div>
<script>
  const t = document.getElementById('t'), p = document.getElementById('p');
  t.addEventListener('click', () => {
    const open = t.getAttribute('aria-expanded') === 'true';
    t.setAttribute('aria-expanded', String(!open));
    p.hidden = open;
  });
</script>

<!-- Zero-JS version -->
<details>
  <summary>Show filters</summary>
  <div>…filters…</div>
</details>

Because it's a real <button>, Enter/Space work for free — no key handlers needed; the whole contract is toggling aria-expanded (on the trigger) and hidden (on the panel) together. The details version gains: works without JS, find-in-page auto-expand, exclusive accordions via name. Loses: fine control over the marker/animation, and you can't put the panel elsewhere in the DOM. Both are correct; details is the "first rule of ARIA" answer.

Exercise 4: Roving-tabindex tablist in React

Build an APG-conformant tabs component: correct roles/states, one tab stop, arrow/Home/End navigation, and manual activation (Enter/Space selects). State why you chose manual over automatic activation.

Solution 
function Tabs({ tabs }) {
  const [selected, setSelected] = useState(0);  // which panel shows
  const [focused, setFocused] = useState(0);    // which tab has focus (manual mode: may differ)
  const refs = useRef([]);
  const id = useId();

  function onKeyDown(e) {
    const n = tabs.length;
    const map = { ArrowRight: (focused + 1) % n, ArrowLeft: (focused - 1 + n) % n,
                  Home: 0, End: n - 1 };
    if (e.key in map) {
      e.preventDefault();
      setFocused(map[e.key]);
      refs.current[map[e.key]].focus();
    } else if (e.key === 'Enter' || e.key === ' ') {
      e.preventDefault();
      setSelected(focused);
    }
  }

  return (
    <>
      <div role="tablist" aria-label="Settings" onKeyDown={onKeyDown}>
        {tabs.map((t, i) => (
          <button key={t.id} role="tab"
                  id={`${id}-tab-${i}`}
                  aria-selected={i === selected}
                  aria-controls={`${id}-panel-${i}`}
                  tabIndex={i === focused ? 0 : -1}
                  ref={el => (refs.current[i] = el)}
                  onClick={() => { setSelected(i); setFocused(i); }}>
            {t.label}
          </button>
        ))}
      </div>
      {tabs.map((t, i) => (
        <div key={t.id} role="tabpanel"
             id={`${id}-panel-${i}`}
             aria-labelledby={`${id}-tab-${i}`}
             tabIndex={0}
             hidden={i !== selected}>
          {t.content}
        </div>
      ))}
    </>
  );
}

Manual activation chosen because panel switching may be expensive (lazy content, network) — automatic activation would fire a load on every arrow press as the user browses tab labels. If panels are trivial static content, automatic (select follows focus) is friendlier — one keystroke fewer. Saying that trade-off out loud is the point of the exercise. Note the two pieces of state: in manual mode, focused ≠ selected while browsing.

Exercise 5: Replace a hand-rolled modal with <dialog>

You inherit a 300-line React modal: portal, z-index 9999, manual focus trap, aria-hidden on #root, body scroll lock. Rewrite it on native <dialog>. What code disappears, and what must you still handle?

Solution 
function Modal({ open, onClose, title, children }) {
  const ref = useRef(null);
  const titleId = useId();

  useEffect(() => {
    const dlg = ref.current;
    if (open && !dlg.open) dlg.showModal();   // imperative: modality isn't an attribute
    else if (!open && dlg.open) dlg.close();
  }, [open]);

  return (
    <dialog ref={ref} aria-labelledby={titleId}
            onClose={onClose}                        {/* fires for Esc too — state stays in sync */}
            onClick={(e) => { if (e.target === ref.current) onClose(); }}> {/* backdrop click */}
      <h2 id={titleId}>{title}</h2>
      {children}
    </dialog>
  );
}
/* CSS */
dialog::backdrop { background: rgb(0 0 0 / .5); }
body:has(dialog[open]) { overflow: hidden; }         /* scroll lock — NOT automatic */

Deleted: the portal (top layer escapes all stacking contexts), z-index management, the entire Tab-wrap focus trap, aria-hidden/inert juggling on the app root, Esc handling, focus restoration to the opener (all native). Still yours: scroll lock; accessible name (aria-labelledby); syncing React state via the close event (covers Esc); backdrop-click dismissal if desired (or the newer closedby="any"); initial-focus choice via autofocus when the default (first focusable) is wrong; enter/exit animations (@starting-style + allow-discrete). Roughly 300 lines → 30, and the a11y is better because inert-ness also stops the SR virtual cursor, which the JS trap never did.
