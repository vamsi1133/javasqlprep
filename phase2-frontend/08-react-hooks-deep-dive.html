08 — Hooks Deep Dive

The fiber-level mechanics of state, effects, and refs — plus the design judgment that separates senior answers from junior ones.

Why interviewers ask this

Every React candidate "knows hooks." That is exactly why interviewers dig here: hooks are the one topic where surface familiarity and real understanding look identical until you ask a precise question. A senior candidate is expected to explain why the Rules of Hooks exist (call-order slot resolution on the fiber), exactly when effects and cleanups fire relative to commit and paint, and why a setInterval inside an empty-dep effect logs a stale count forever — tracing the closure, not hand-waving "it's a stale closure."

The second axis is judgment. FAANG interviewers and startup leads both probe for it: When does logic belong in an event handler instead of an effect? When is useReducer worth the ceremony? How do you design the return API of a custom hook so it composes without foot-guns? Wrong-but-common patterns (props-to-state syncing, effect chains, inline-object deps) are deliberately planted in code-review-style interview questions. This tutorial covers the mechanics first, then the judgment, then drills both with Q&A and exercises.

1. How hooks work under the hood TIER 1

Hooks have no magic access to "the component instance" — function components don't have instances. Instead, every component has a fiber (its unit of work in the reconciler, covered in tutorial 07), and the fiber's memoizedState field points at a linked list of hook objects. Each call to useState, useEffect, useRef, etc. during render consumes the next node in that list. There are no names or keys — hooks are resolved purely by call order.

// Conceptual model of what React keeps per component fiber:
fiber.memoizedState = {
  memoizedState: 0,          // hook 1: useState(0) → the state value
  queue: { /* pending updates */ },
  next: {
    memoizedState: 'idle',  // hook 2: useState('idle')
    queue: { ... },
    next: {
      memoizedState: {        // hook 3: useEffect → effect object
        create: fn, deps: [id], destroy: cleanupFn
      },
      next: null
    }
  }
};

On the initial render (mount), each hook call appends a new node to the list. On every subsequent render (update), React walks the same list with a cursor: first hook call reads node 1, second call reads node 2, and so on. The hook's identity is literally "I was the 3rd hook called." This is the entire reason for the Rules of Hooks.

The Rules of Hooks — and what actually breaks

  • Only call hooks at the top level — never inside conditions, loops, early returns, or nested functions. The call sequence must be identical every render.
  • Only call hooks from React function components or custom hooks — because only then is a fiber being rendered and a dispatcher installed (below).

Here is the classic corrupted-state demonstration. Watch what happens to the slot alignment when a hook is called conditionally:

function Profile({ user }) {
  const [name, setName] = useState(user.name);      // hook slot 1
  if (user.isAdmin) {
    const [audit, setAudit] = useState([]);          // hook slot 2 ...sometimes
  }
  const [theme, setTheme] = useState('light');      // slot 3 or slot 2?!
  ...
}
RenderSlot 1Slot 2Slot 3
Render A (isAdmin: true)nameaudittheme
Render B (isAdmin: false)nametheme ← reads the audit node!(list has a leftover node)

On render B, the theme call is the second hook call, so React hands it the stored state of the audit hook — theme is suddenly []. React actually detects the mismatch in the hook count and throws "Rendered fewer hooks than expected" (or warns about a changed hook order), but the conceptual failure is silent state corruption: values shifting into the wrong slots. The same logic explains why an early return before a hook or a hook inside a loop with variable iterations is illegal.

If you need "conditional state", the fix is structural, not conditional hooks: either always call the hook and conditionally use the value, or split the conditional part into a child component that mounts/unmounts — each component owns its own fiber and hook list, so mounting a child is how you "conditionally allocate" hooks.

The dispatcher — why hooks only work during render

useState exported from the react package is a thin wrapper. It reads a module-level mutable reference — the current dispatcher — and delegates:

// Simplified from the React source:
function useState(initialState) {
  const dispatcher = ReactCurrentDispatcher.current; // set by the renderer
  return dispatcher.useState(initialState);
}

Right before react-dom renders your component, it installs the appropriate dispatcher: a mount dispatcher (creates hook nodes) on first render, an update dispatcher (walks existing nodes) on re-renders. When no component is rendering, the dispatcher is set to an implementation that throws. This explains several things at once:

  • "Invalid hook call" when you call a hook outside a component: no valid dispatcher is installed.
  • Two copies of React break hooks: the component renders under renderer A's dispatcher, but imports useState from copy B, whose ReactCurrentDispatcher was never set.
  • Custom hooks need no registration — they are plain functions that happen to call hooks while a dispatcher is installed, i.e. during someone's render.
  • Mount vs update behavior differences (e.g. the initial-state argument being ignored after mount) are literally two different function implementations.
The linked-list-by-call-order design is a deliberate trade-off: it avoids requiring explicit keys for every piece of state (as some frameworks do) at the cost of the Rules of Hooks. Mentioning that eslint-plugin-react-hooks statically enforces the rules — and that the React Compiler depends on those same guarantees to memoize safely — signals you understand the rules as an architectural contract, not folklore.

2. useState deep dive TIER 1

Initial state runs once — the lazy initializer

The argument to useState is only used on mount, but the expression is evaluated on every render — it's a normal function argument. That's the trap:

// ❌ expensiveParse runs on EVERY render; result discarded after mount
const [data, setData] = useState(expensiveParse(rawJson));

// ✅ lazy initializer: React calls the function once, on mount only
const [data, setData] = useState(() => expensiveParse(rawJson));

If you pass a function, React treats it as an initializer and invokes it only when creating the hook node. Same idiom applies to useReducer's third argument. Classic follow-up: "what if the state itself should hold a function?" — then you must wrap it: useState(() => myFn), otherwise React calls myFn as an initializer.

The Object.is bail-out

When you call the setter, React compares the incoming value to the current one with Object.is. If they're identical, React bails out: no state change, and it skips re-rendering the component and its children.

Precise nuance worth stating in an interview: in some cases React may still render that one component once more before bailing out (it needs to run the function to confirm nothing changed when it couldn't decide eagerly), but it will not descend into children and will not commit. So "setting the same value never re-renders" is almost right — the guaranteed part is that it won't propagate or commit. This also implies: with objects, setUser({ ...user }) always re-renders (new reference), while mutating user.name = 'x'; setUser(user) never does (same reference — a top-3 real-world hooks bug).

Updater-function queue semantics — the classic drill

Each setter call doesn't overwrite state; it enqueues an update on the hook's queue. During the next render, React processes the queue in order. There are two kinds of updates: replace (you passed a value) and apply (you passed a function, which receives the latest pending state).

function handleClick() {          // count is 0 in this render's closure
  setCount(count + 1);            // enqueue "replace with 0 + 1 = 1"
  setCount(count + 1);            // enqueue "replace with 0 + 1 = 1"
  setCount(count + 1);            // enqueue "replace with 0 + 1 = 1"
}                                 // next render: count === 1

function handleClickUpdater() {
  setCount(c => c + 1);           // enqueue "apply: pending 0 → 1"
  setCount(c => c + 1);           // enqueue "apply: pending 1 → 2"
  setCount(c => c + 1);           // enqueue "apply: pending 2 → 3"
}                                 // next render: count === 3

Why the difference: count + 1 is computed now, from the snapshot captured by this render's closure — all three calls compute 1. The updater function is stored uncalled and later fed the running pending value as React drains the queue. Mixed forms follow the same rule:

setCount(count + 5);   // pending: 5 (replace)
setCount(c => c + 1);  // pending: 6 (apply to 5)
setCount(42);          // pending: 42 (replace discards the 6)
// next render: 42

State is a snapshot per render

The single most-tested mental model. count is a const local variable in one execution of your function. Calling setCount does not change that variable — it schedules a re-render, in which a new function call will receive the new value:

const [count, setCount] = useState(0);

function handleClick() {
  setCount(count + 1);
  console.log(count);        // logs 0 — NOT 1
  setTimeout(() => {
    console.log(count);      // STILL 0, even after the re-render happened —
  }, 3000);                  // this closure belongs to the old render
}

Why, precisely: (1) setCount only enqueues an update and marks the fiber dirty — it never mutates anything the current closure can see; (2) the re-render creates a new closure with count === 1, but the running handleClick and any timeouts it spawned still close over the old render's bindings. State behaves like a snapshot photograph handed to each render. Every stale-closure bug in section 3 is this fact wearing a different costume.

Batching recap

Since React 18, all state updates are automatically batched — event handlers, promises, setTimeout, native event listeners. Multiple setter calls (even across different hooks and components) in one synchronous block produce one re-render, processed together. Pre-18 only React event handlers batched, which is why legacy code sometimes relied on updates in a timeout flushing immediately. Escape hatch when you truly need synchronous DOM reads between updates: flushSync(() => setX(...)) from react-dom — use rarely, it forces sync render and hurts performance.

Derived state — the anti-pattern and its fixes

Storing anything computable from existing props/state is redundant state that will drift. The default answer is: compute during render.

// ❌ redundant state + effect to "sync" it — extra render, drift risk
const [fullName, setFullName] = useState('');
useEffect(() => { setFullName(first + ' ' + last); }, [first, last]);

// ✅ derive it — cheap computations don't even need useMemo
const fullName = first + ' ' + last;

The props-to-state sync trap. The subtler version: seeding state from a prop, then trying to keep it in sync.

function EmailEditor({ contact }) {
  // ❌ initializes once; when parent passes a different contact,
  // the draft silently keeps the OLD contact's email
  const [draft, setDraft] = useState(contact.email);
  ...
}

Ranked fixes:

  1. Reset with key — tell React it's a different logical instance; the subtree remounts and all state re-initializes. Cleanest when "new contact = start over": <EmailEditor key={contact.id} contact={contact} />
  2. Render-time compare — for resetting part of the state. Storing the previous prop in state and calling the setter during render is legal (for the same component): React discards the render output and re-runs immediately, before touching the DOM or children — cheaper than an effect-based reset, which commits a stale frame first. 
    const [prevContact, setPrevContact] = useState(contact);
const [draft, setDraft] = useState(contact.email);
if (contact !== prevContact) {       // during render, no effect
  setPrevContact(contact);
  setDraft(contact.email);
}
  3. Lift the state up — often the honest fix: make it controlled and let the parent own the draft.
Historical framing interviewers appreciate: class components had componentWillReceiveProps, whose misuse was so pervasive that React 16.3 replaced it with the deliberately awkward static getDerivedStateFromProps — and the accompanying blog post ("You Probably Don't Need Derived State") is the direct ancestor of the modern key-reset and render-time-compare guidance. The hooks-era doctrine is the same: derived state is a last resort, not a pattern.
The effect-based version useEffect(() => setDraft(contact.email), [contact]) is not just slow (paint stale → effect → re-render → paint again, a visible flicker under load) — it also stomps user input whenever the parent re-creates contact with a new reference but identical data. Object identity in deps strikes again (section 3).

3. useEffect: the complete story TIER 1

The mental model: synchronization, not lifecycle

The senior framing: an effect synchronizes your component with an external system (network, subscriptions, timers, browser APIs, non-React widgets). It is not "code that runs on mount" — it's "code that makes the outside world match the current props/state, and undoes it when they change or the component leaves." That said, know the lifecycle vocabulary because interviewers still use it:

Class-era vocabularyHooks equivalentCaveat
componentDidMountuseEffect(fn, [])Runs after paint (didMount ran before paint — useLayoutEffect is the closer match)
componentDidUpdateuseEffect(fn, [deps])No prevProps/prevState — the deps diff replaces manual comparison
componentWillUnmountthe returned cleanupCleanup also runs between re-runs, not only at unmount — the key difference

Execution timing — where effects sit in the pipeline

render phase        → your function runs, React diffs (interruptible in concurrent React)
commit phase        → DOM mutations applied
                    → refs attached, useLayoutEffect cleanups + setups run (SYNC, pre-paint)
browser paints      → user sees the new frame
passive effects     → useEffect cleanups run, then useEffect setups run

useEffect is deliberately deferred until after paint so it never blocks the frame — that's why data fetching, logging, and subscriptions belong there. Two precision points for interviews: (1) React may flush passive effects earlier — e.g. before processing a new discrete event like a click — so you can't assume "always after paint," only "never blocking this paint"; (2) effects flush bottom-up: children's effects run before their parent's.

In StrictMode (development only), React 18+ mounts, runs effects, immediately runs cleanups, then runs effects again. This is a deliberate stress test: if your effect's cleanup fully undoes its setup, the double-invoke is invisible. If your app breaks under StrictMode, you have a real cleanup bug, not a React bug.

Cleanup timing — a subscription trace

function ChatRoom({ roomId }) {
  useEffect(() => {
    const conn = connect(roomId);
    return () => conn.disconnect();   // cleanup closes over THIS run's roomId
  }, [roomId]);
}
EventWhat runs, in order
Mount with roomId='general'render → commit → paint → connect('general')

Re-render with same roomIdrender → commit → deps Object.is-equal → effect skipped entirely

The crucial detail: each effect run pairs with its own cleanup, and the cleanup closes over the same render's values. React always runs the previous cleanup before the next setup. So the model is not "unmount cleanup"; it's "every synchronization gets undone before the next one starts."

Dependency array semantics

  • Deps are compared element-by-element with Object.is against the previous render's array. Any mismatch → cleanup + re-run.
  • [] means "synchronizes with nothing reactive" → setup after mount, cleanup at unmount (plus the StrictMode dev cycle).
  • No array at all → runs after every render. Rarely what you want; usually a smell.
  • The rule is not "list what should trigger the effect" — it's "list every reactive value the effect reads": props, state, and anything derived from them in the component body. Deps are a consistency declaration, not a trigger config.
Suppressing react-hooks/exhaustive-deps doesn't make the effect run less — it makes the effect lie. The closure still captures the omitted value from whatever render last created it, so the effect silently reads stale data forever after. Nine times out of ten the correct move is not suppression but restructuring: use an updater function, move the value into the effect, memoize the dependency, or hoist a non-reactive value out of the component. Legitimate suppressions exist (e.g. an intentional mount-only analytics ping) but should be rare, commented, and defensible in review.

Stale closures — the interval classic, traced exactly

function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log('count is', count);   // logs "count is 0" forever
      setCount(count + 1);               // always computes 0 + 1
    }, 1000);
    return () => clearInterval(id);
  }, []);                                // ⚠️ effect reads `count` but lists nothing

  return <h1>{count}</h1>;
}

Exact behavior: the display shows 0, ticks to 1 after one second, and then never changes again, while the console prints count is 0 every second, forever. Walk the trace:

  1. Render #1 executes with count = 0. The effect function — and the interval callback inside it — are created in render #1's scope. The callback's count is permanently bound to that render's const count = 0.
  2. Deps are [], so this effect runs once and is never replaced. The interval keeps firing the render-#1 callback.
  3. Tick 1: logs 0, calls setCount(0 + 1) → re-render with count = 1. Render #2 creates a new closure — but nobody is using it; the interval still holds render #1's.
  4. Tick 2 onward: logs 0 again, calls setCount(1). Current state is already 1 → Object.is bail-out → no re-render. Steady state: UI frozen at 1, console stuck at 0.

Fixes, ranked:

  1. Functional updater — best when the only need is "next state from previous." The effect no longer reads count, so [] becomes honest: 
    setCount(c => c + 1);   // React supplies the latest pending value
  2. Restart on dep change — add count to deps. Correct, and the right shape for subscriptions that genuinely depend on the value — but here it tears down and recreates the interval every second, and resets the tick cadence. Know it; also know its cost.
  3. Latest-value ref — when you must read fresh state inside a long-lived callback without restarting it (see the useLatest pattern in section 4). More moving parts; reserve for callbacks that are expensive to re-establish.
  4. useEffectEvent awareness — the experimental API extracts non-reactive "event" logic from an effect: it always sees latest props/state and is excluded from deps. Not stable yet; name it as the designed solution to this class of problem, don't ship it.

The fetch race condition — and the full cleanup pattern

Effects re-run when deps change, but network responses don't arrive in dep-change order. Type "ab" quickly: the effect for "a" fires, then the effect for "ab"; if "a"'s response lands last, it overwrites the correct results. Two idiomatic fixes — the ignore flag (universal) and AbortController (also cancels the request):

useEffect(() => {
  const controller = new AbortController();
  let ignore = false;                    // belt AND suspenders shown; either works

  async function run() {
    try {
      const res = await fetch(`/api/search?q=${query}`, {
        signal: controller.signal,
      });
      const json = await res.json();
      if (!ignore) setResults(json);     // stale runs never touch state
    } catch (err) {
      if (err.name === 'AbortError') return; // expected on cleanup — not an error
      if (!ignore) setError(err);
    }
  }
  run();

  return () => {
    ignore = true;
    controller.abort();                  // frees the connection, rejects the fetch
  };
}, [query]);

Why the cleanup solves it: when query changes, the old effect's cleanup flips its own ignore and aborts its own request before the new effect starts. Each in-flight request is owned by exactly one effect run, and only the latest run's flag is still false. Note the AbortError swallow — forgetting it makes every keystroke log a spurious error. (In real apps, reach for React Query/SWR or router loaders — say so, then prove you can hand-roll it.)

What belongs where: render vs event handler vs effect

The "You Might Not Need an Effect" doctrine, condensed to a decision rule:

The effect-chain anti-pattern. One effect sets state, which triggers another effect, which sets more state:

// ❌ 3 cascading render passes to compute what one handler could do
useEffect(() => { if (card !== null) setGoldCount(c => c + 1); }, [card]);
useEffect(() => { if (goldCount > 3) setRound(r => r + 1); }, [goldCount]);
useEffect(() => { if (round > 5) setGameOver(true); }, [round]);

// ✅ compute the entire next state in the event handler that caused it
function handlePlaceCard(next) {
  const nextGold = next.gold ? goldCount + 1 : goldCount;
  setCard(next);
  setGoldCount(nextGold);
  if (nextGold > 3) setRound(round + 1);
  if (round + 1 > 5) setGameOver(true);
}

Effect chains are slow (each link is a full render+commit cycle), fragile (ordering is implicit), and unreadable (the business rule is smeared across the file). Consolidate into the handler — or into a reducer, which is exactly the "multi-field transition" case useReducer exists for (section 7).

Object and function deps — the churn problem

function Map({ lat, lng }) {
  const options = { center: { lat, lng }, zoom: 12 };  // new object EVERY render

  useEffect(() => {
    map.setOptions(options);
  }, [options]);   // ❌ Object.is(prev, next) is false every time → runs every render
}

Worse, if the effect also sets state, you get an infinite loop: render → new object → effect → setState → render → new object… Fixes, in order of preference:

  1. Depend on primitives: [lat, lng] and build the object inside the effect. Primitives compare by value under Object.is; this is the most robust fix.
  2. Move creation into the effect if the object is only used there — it stops being a dependency at all.
  3. useMemo the object / useCallback the function when it must be created outside (e.g. shared with JSX) — stabilizes the reference as long as its deps are stable.
  4. Hoist it out of the component if it doesn't depend on props/state at all — module-level constants are never reactive.
This same churn problem is why custom hooks should be suspicious of option-object parameters: useFetch(url, { headers }) with an inline object re-triggers everything each render unless the hook defends itself (serialize, destructure to primitives, or document the contract). Section 11 shows the defensive patterns.

4. useRef patterns TIER 1

The mutable box

useRef(initial) returns { current: initial } — the same object on every render (it's stored in the hook node). Two defining properties: mutating .current never triggers a re-render, and the value survives re-renders. It is an instance variable for function components — the hooks answer to this.something on a class.

useStateuseRef
Triggers re-render on changeYesNo
Safe to read during renderYes — it's the snapshotNo (except lazy-init idiom)
Mutation semanticsImmutable; setter enqueuesMutable; write .current anytime outside render
Value during a renderFixed snapshot for that renderLive — always the latest write
Use forAnything the UI displays or derives fromTimer ids, DOM nodes, library instances, previous/latest values, "have I run yet" flags

The decision rule: if changing the value should change what the user sees → state. If it's bookkeeping the render output never reads → ref. If you find yourself calling forceUpdate-style hacks after writing a ref, it should have been state.

DOM refs and attachment timing

function SearchBox() {
  const inputRef = useRef(null);
  useEffect(() => {
    inputRef.current.focus();     // safe: refs attach during commit,
  }, []);                         // before ANY effects run
  return <input ref={inputRef} />;
}

Timing precisely: during the commit phase React sets ref.current to the DOM node after mutating the DOM and before running layout effects — so both useLayoutEffect and useEffect can rely on the ref being attached. During render, inputRef.current is null on mount (the node doesn't exist yet) — reading it in render is both a rules violation and simply wrong. On unmount React sets it back to null. Conditional rendering means the ref can be null at any time; guard accordingly, or use a callback ref when you need to react to attach/detach itself.

The latest-value ref (useLatest) — defeating stale closures

function useLatest(value) {
  const ref = useRef(value);
  useEffect(() => { ref.current = value; });  // after every commit
  return ref;
}

function Chat({ onMessage }) {
  const onMessageRef = useLatest(onMessage);
  useEffect(() => {
    const socket = connect();
    socket.on('msg', m => onMessageRef.current(m)); // always the latest handler
    return () => socket.close();
  }, []);   // socket survives re-renders; the ref is stable, its contents fresh
}

The trick: the ref object's identity never changes (safe to omit from deps / include harmlessly), while .current is updated after each commit. Long-lived callbacks read through the ref at call time instead of capturing at creation time. This is the manual predecessor of useEffectEvent. Caveat: because the ref updates in an effect, it lags during the render phase itself — fine for event/timer callbacks, wrong for render logic.

Timers, instances, and the guard pattern

const timerRef = useRef(null);         // setTimeout/setInterval ids
const chartRef = useRef(null);         // third-party class instance

// Lazy-init idiom — the ONE sanctioned render-phase ref access:
function getChart() {
  if (chartRef.current === null) {
    chartRef.current = new ExpensiveChart();   // created once, on demand
  }
  return chartRef.current;
}

Why not read/write refs during render (in general): React's contract is that rendering is pure — same inputs, same JSX. In concurrent rendering, React may render a component twice, abandon a render, or render at a different priority; render-phase ref mutations then execute an unpredictable number of times and can leak between attempts. The lazy-init idiom is exempt because it's idempotent: the first write wins and every subsequent render observes the same value, so purity is preserved in effect.

forwardRef, React 19's ref prop, and useImperativeHandle

ref is not a normal prop in React ≤18 — function components need forwardRef to receive it. React 19 removes the ceremony: ref arrives as a regular prop on function components, and forwardRef is on the deprecation path.

// React ≤18
const Input = forwardRef(function Input(props, ref) {
  return <input ref={ref} {...props} />;
});

// React 19 — just a prop
function Input({ ref, ...props }) {
  return <input ref={ref} {...props} />;
}

useImperativeHandle customizes what a parent's ref receives — the design-system use case is exposing a minimal imperative surface instead of the raw DOM node:

const FancyInput = forwardRef(function FancyInput(props, ref) {
  const inputRef = useRef(null);

  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current.focus(),
    clear: () => { inputRef.current.value = ''; },
    // deliberately NOT exposing the node itself
  }), []);

  return <input ref={inputRef} {...props} />;
});

// Parent: inputHandle.current.focus() — can't reach into styles/DOM internals

Use sparingly — and say why: imperative handles are escape hatches. Every method you expose is an API contract that bypasses the declarative data flow, is invisible to React DevTools' props view, and tends to accumulate ("just add a scrollIntoView…"). Legitimate uses are focus management, scrolling, text selection, media playback, and imperative animation triggers — things that are inherently imperative in the DOM. If a parent needs to imperatively set a child's data, model it as props/state instead.

5. useMemo & useCallback mechanics TIER 1

This section covers what they are; the strategy of when memoization pays off (profiling, React.memo trees, the React Compiler) lives in tutorial 09.

What they actually do

Both store a [value, deps] pair in their hook node. On re-render, React compares the new deps to the stored ones (Object.is, element-wise). Match → return the cached value without calling your function. Mismatch → recompute, store, return. That's the entire mechanism — a per-hook, single-entry cache keyed by the dependency array.

const sorted = useMemo(() => [...items].sort(cmp), [items]);  // caches the RESULT
const onSave = useCallback(handleSave, [draftId]);            // caches the FUNCTION

// These two are exactly equivalent — useCallback is sugar:
useCallback(fn, deps)
useMemo(() => fn, deps)

useMemo calls your function and caches its return value; useCallback caches the function itself. Interviewers love the equivalence question because getting it backwards (useMemo(fn, deps)) means "call fn now and cache what it returns" — a real bug.

Hints, not guarantees

The docs are explicit: you may rely on useMemo as a performance optimization, not a semantic guarantee. React reserves the right to throw away cached values — for example for offscreen/reusable-state features, or when it needs to free memory — and recompute on next render. Consequences:

  • The memoized function must be pure and safe to re-run at any time.
  • Never use useMemo for "run once" semantics (creating a client, starting a subscription) — that's a ref (lazy-init idiom) or an effect.
  • Correctness must never depend on referential stability; it may only be improved by it.

The three legitimate referential-equality use cases

  1. Props to memoized children: React.memo(Child) compares props shallowly — an inline object/lambda defeats it every render. Stabilize what you pass.
  2. Effect dependencies: an object/function dep churns the effect (section 3). useMemo/useCallback stops the loop when the value must live outside the effect.
  3. Context values: value={{user, login}} re-renders every consumer on every provider render; memoize the object (section 8).

Plus the honest fourth: genuinely expensive computations (thousands of items, heavy transforms) — measure before claiming this one.

The cost of overuse

Every memo hook costs: allocation of the deps array each render, the comparison loop, memory for the cached value, and — the real price — reader complexity and a false sense of safety (one unstable dep silently voids the whole chain). Wrapping every value in a component with no memoized consumers is pure overhead. The senior answer: memoize at the boundaries that need it (memo children, effects, context), keep the rest plain, and note that the React Compiler exists precisely to make manual memoization obsolete by auto-memoizing based on the Rules of Hooks guarantees.

6. useLayoutEffect & useInsertionEffect TIER 2

The timing difference — one line, then the consequences

useLayoutEffect runs synchronously after DOM mutations, before the browser paints. useEffect runs after paint. Same signature, same cleanup semantics — only timing differs, and it changes everything about what each is for.

DOM updated → [useLayoutEffect cleanups + setups] → PAINT → [useEffect cleanups + setups]
              ↑ blocks the frame; user never sees          ↑ user has already seen
                the intermediate state                       the committed frame

The measure-then-position tooltip — the canonical example

A tooltip should flip below the anchor when it doesn't fit above. You can't know its height until it's in the DOM — so you must render, measure, and reposition before the user sees anything:

function Tooltip({ anchorRect, children }) {
  const ref = useRef(null);
  const [pos, setPos] = useState(null);   // null = not measured yet

  useLayoutEffect(() => {
    const { height } = ref.current.getBoundingClientRect();
    const fitsAbove = anchorRect.top - height >= 0;
    setPos(fitsAbove ? anchorRect.top - height : anchorRect.bottom);
  }, [anchorRect]);

  return (
    <div ref={ref} style={{ position: 'fixed', top: pos ?? 0,
                            visibility: pos === null ? 'hidden' : 'visible' }}>
      {children}
    </div>
  );
}

Why this works: setting state inside useLayoutEffect forces React to re-render and re-commit synchronously, still before paint. The browser only ever paints the final position. Swap in useEffect and you get the flicker demo: frame 1 paints the tooltip at the wrong position (or at 0,0), then the effect measures and moves it — a visible one-frame jump, obvious on slow devices and in <React.StrictMode> with CPU throttling.

Two costs of useLayoutEffect: (1) it blocks painting — heavy work here directly delays the frame, so keep it to measurement + a quick state set; (2) it does nothing on the server and React warns during SSR, since there's no layout to read. For SSR components, either gate the component to client-only rendering, or suppress the initial mismatch deliberately. Default to useEffect; reach for useLayoutEffect only when the user could see the intermediate state.

useInsertionEffect — awareness level

A niche hook for CSS-in-JS library authors: it fires before layout effects, at a point where injected <style> rules will be in place before any layout effect reads getBoundingClientRect — avoiding the "inject style during layout effect → invalidate layout → force recalc for every subsequent read" performance cliff. Constraints: refs are not attached yet and you can't schedule state updates from it. Interview answer: "I know what it's for — runtime style injection by libraries like styled-components — and that application code should essentially never use it."

7. useReducer TIER 2

When it beats useState

  • Multi-field transitions: one action updates several fields consistently (submit → loading, clear error, keep data). With useState that's three setters that must always travel together.
  • Next state derived from previous across complex rules: the reducer receives the authoritative previous state — stale closures over state largely disappear.
  • Testable logic: a reducer is a pure function — unit test state transitions with zero React imports.
  • Stable dispatch: React guarantees dispatch's identity never changes — pass it deep without useCallback, put it in context without churn.
  • Centralized invariants: impossible states become unrepresentable in one place instead of policed at every call site.

Rule of thumb: independent primitives → useState; a state machine (fields that move together, events with names) → useReducer. "Number of useState calls" alone is not the criterion — coupling is.

Full example: reducer + TypeScript discriminated-union actions

type State =
  | { status: 'idle' }
  | { status: 'loading'; attempt: number }
  | { status: 'success'; data: User[] }
  | { status: 'error'; message: string; attempt: number };

type Action =
  | { type: 'fetch_started' }
  | { type: 'fetch_succeeded'; data: User[] }
  | { type: 'fetch_failed'; message: string }
  | { type: 'retried' };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'fetch_started':
      return { status: 'loading', attempt: 1 };
    case 'fetch_succeeded':
      return { status: 'success', data: action.data };  // `data` narrowed here
    case 'fetch_failed': {
      const attempt = state.status === 'loading' ? state.attempt : 1;
      return { status: 'error', message: action.message, attempt };
    }
    case 'retried':
      return state.status === 'error'
        ? { status: 'loading', attempt: state.attempt + 1 }
        : state;                                          // illegal transition → no-op
    default: {
      const _exhaustive: never = action;  // compile error if a case is missed
      return state;
    }
  }
}

const [state, dispatch] = useReducer(reducer, { status: 'idle' });

Why the discriminated union matters: TypeScript narrows both action (per case) and state (per status check), so "read data while loading" is a compile error, and the never trick makes adding an action type break the build until every reducer handles it. Also note the reducer must be pure — no fetches, no mutation of the incoming state (StrictMode double-invokes reducers in dev to catch exactly this).

The dispatch-in-context pattern

const TasksContext = createContext(null);
const TasksDispatchContext = createContext(null);

function TasksProvider({ children }) {
  const [tasks, dispatch] = useReducer(tasksReducer, []);
  return (
    <TasksContext.Provider value={tasks}>
      <TasksDispatchContext.Provider value={dispatch}>
        {children}
      </TasksDispatchContext.Provider>
    </TasksContext.Provider>
  );
}

Because dispatch is stable, the dispatch context's value never changes — components that only trigger actions (toolbars, buttons) subscribe to it and re-render zero times when the data changes. Only readers of TasksContext re-render. This split is the standard answer to "how do you build a small Redux with hooks," and previews the context-performance discussion next.

8. useContext mechanics TIER 2

How propagation actually works

When a provider re-renders with a new value (compared with Object.is), React walks the subtree and schedules a re-render for every component that reads that context. The critical mechanic: context propagation punches through memoization bail-outs. A React.memo parent with unchanged props will be skipped — but React still traverses past it to reach consumers underneath, and those consumers re-render anyway. Memoizing intermediate components does not protect consumers; conversely, it does still protect non-consumers from a re-rendering provider.

Provider (value changed)
 └── React.memo(Middle)   ← bails out, does NOT re-render itself
      └── Consumer        ← re-renders ANYWAY (reads the context)
      └── PlainChild      ← does not re-render (parent bailed out)

Also: a consumer re-renders based on the nearest provider above it, and there is no built-in selector — a consumer re-renders when the value changes at all, even if it only uses one field. That limitation drives every pattern below.

The classic churn bug: value={{...}}

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  // ❌ new object identity on EVERY provider render —
  // every consumer re-renders even when user/login are unchanged
  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

Any unrelated state change in the provider component (or its parents re-rendering it) creates a fresh object, fails Object.is, and fans out to all consumers. Fixes, in escalating order:

  1. Memoize the value: 
    const login  = useCallback((creds) => { ... }, []);
const logout = useCallback(() => setUser(null), []);
const value  = useMemo(() => ({ user, login, logout }), [user, login, logout]);
    Now consumers re-render only when user actually changes.
  2. Split state and dispatch contexts (previous section): action-only components stop re-rendering entirely.
  3. Split slices into separate contexts: ThemeContext, UserContext, SettingsContext instead of one AppContext — each consumer subscribes to exactly the slice it needs.
  4. Children-as-stable-element: because {children} is created by the parent's render, a provider whose own state changes re-renders itself but passes the same children element — subtree reconciliation bails out except for consumers. Knowing this explains why "provider components with children" are cheap.

Context vs a store

Context is a dependency-injection mechanism, not a state-management system: it has no selectors, no memoized subscriptions, no out-of-render updates. It shines for low-frequency, tree-scoped values — theme, locale, auth identity, the dispatch half of a reducer. For high-frequency or widely-shared data (form keystrokes, live prices, large normalized caches), per-field subscriptions from a store (Zustand/Redux/Jotai — built on useSyncExternalStore, next section) avoid the fan-out. Full decision framework in tutorial 11.

9. useSyncExternalStore TIER 2

The tearing problem

Concurrent rendering means React can pause a render, handle something else, and resume. Suppose components read a mutable external variable (a store outside React). Component A renders and reads price = 10; React yields; the store updates to 11; React resumes and component B reads 11. The committed frame shows 10 and 11 simultaneously — a "torn" UI that React's own state can never produce (state updates are snapshotted and scheduled through React). Tearing is invisible in synchronous rendering and only emerges under concurrent features (startTransition, Suspense) — which is why the hook appeared in React 18.

The API

const snapshot = useSyncExternalStore(
  subscribe,          // (onStoreChange) => unsubscribe — how React learns of changes
  getSnapshot,        // () => current value — MUST be cached/immutable
  getServerSnapshot   // () => value for SSR + hydration (optional but needed for SSR)
);
  • React calls subscribe once (per subscribe identity — keep it stable or hoisted!) and re-subscribes if it changes.
  • On store change, React calls getSnapshot, compares with Object.is, and re-renders subscribers if different. It also re-checks the snapshot during render to detect mid-render changes — if the store changed while rendering, React restarts the render synchronously, guaranteeing every component in the commit sees the same snapshot. That is the anti-tearing mechanism (updates through this hook are always treated as synchronous/non-deferred).
  • getSnapshot must return the same reference until the store actually changes. Returning a fresh object each call (() => ({ w: innerWidth })) makes React think the store changed every render → infinite re-render loop with the famous "getSnapshot should be cached" error.

Build useWindowWidth correctly

// Hoisted: stable identities, no re-subscribe churn
function subscribe(onStoreChange) {
  window.addEventListener('resize', onStoreChange);
  return () => window.removeEventListener('resize', onStoreChange);
}
const getSnapshot = () => window.innerWidth;   // primitive → Object.is-safe
const getServerSnapshot = () => 1024;          // deterministic SSR fallback

export function useWindowWidth() {
  return useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot);
}

Contrast with the naive useState + resize-listener effect: that version can tear under transitions, double-subscribes per component, and needs a typeof window guard for SSR. Here the snapshot is a primitive (identity-safe), subscription is declarative, and SSR is handled by contract. This is also the correct tool for navigator.onLine, media queries, localStorage sync, and router location.

How the libraries use it: Redux's useSelector and Zustand's useStore are wrappers around useSyncExternalStore (via the use-sync-external-store shim, whose useSyncExternalStoreWithSelector adds selector + equality-function support). Their stores are plain emitters; the hook is what bridges them safely into concurrent React. Trade-off worth naming: because uSES updates are handled synchronously, external-store updates can't be deferred as non-blocking transitions the way useState updates can — one argument for keeping fast-changing UI state inside React.

10. useId, useDebugValue, transitions TIER 2

useId — SSR-safe unique ids

function EmailField() {
  const id = useId();   // e.g. ":r1:" — opaque, stable, unique per hook call
  return (
    <>
      <label htmlFor={id}>Email</label>
      <input id={id} aria-describedby={id + '-hint'} />
      <span id={id + '-hint'}>Work email preferred</span>
    </>
  );
}

Why not Math.random() or a module counter: with SSR, the server generates ids into the HTML, then the client re-generates them during hydration — random values won't match, producing hydration mismatches; a counter depends on render order/timing, which differs between server and client (especially with streaming and selective hydration). useId derives the id from the position of the component in the tree, identical on both sides. It is explicitly not for list keys — keys must come from your data. One useId can prefix several related attributes, as above.

useDebugValue — trivia tier

Labels a custom hook's value in React DevTools: useDebugValue(isOnline ? 'Online' : 'Offline'). Second argument is a lazy formatter — useDebugValue(date, d => d.toISOString()) — deferred until DevTools actually inspects, so production and non-inspecting renders pay nothing. Worth one sentence in an interview; using it in shared library hooks is a nice polish signal.

useTransition & useDeferredValue — one paragraph each (full story in tutorial 10)

useTransition returns [isPending, startTransition]; updates inside startTransition(() => ...) are marked low-priority: React keeps the current UI interactive, renders the new tree in the background, can interrupt it if new input arrives, and exposes isPending for spinners. Mental model: "this state update produces a heavy screen — don't block typing/clicking on it." Classic use: tab switches over expensive content, filtering huge lists.

useDeferredValue is the value-side twin: const deferredQuery = useDeferredValue(query) gives a value that "lags behind" during urgent updates — the input re-renders instantly with query, while expensive consumers re-render later with deferredQuery. Use it when you don't control the setState call (e.g. a prop) or want debounce-like behavior driven by rendering cost, adaptively, rather than a fixed timeout.

11. Custom hook design TIER 1

This is where senior judgment is graded. A custom hook is not a code-organization trick — it's a composition unit: a named, reusable piece of stateful behavior with a designed API. The framing to say out loud: "hooks are composition units, not lifecycle hacks" — a custom hook should express what it synchronizes or computes (useOnlineStatus, useChatRoom), never when it runs (useMount, useUpdateEffect are lifecycle nostalgia that hide dependencies and fight the model).

When to extract

  • Reuse: the same stateful logic appears in ≥2 components (form field logic, media queries, data polling).
  • Complexity worth naming: even single-use — if a component drowns in effects/refs, extracting useVideoPlayer(src) turns 60 lines of plumbing into one declarative line. Naming is the point: the component reads as intent.
  • Not: trivial wrappers (useToggle around one useState is fine but adds little), or extracting just to share an effect whose deps differ per caller — sometimes a plain function or a component is the right unit.

Key mechanics fact interviewers check: custom hooks share logic, not state. Two components calling useOnlineStatus() each get their own hook nodes — completely independent state. Shared state requires lifting, context, or a store.

Return API design

ShapeWhenExample
Single valueThe hook answers one questionconst width = useWindowWidth()
Tuple (array)A value + its setter/action pair, ≤2–3 items — callers rename freely, mirrors useStateconst [copied, copy] = useClipboard()

Stability of returned identities is part of your API contract. If your hook returns a function, wrap it in useCallback; if it returns an object callers might put in deps or pass to memo children, useMemo it. Otherwise every consumer inherits your churn and can't fix it from outside:

function useCounter(initial = 0) {
  const [count, setCount] = useState(initial);
  const increment = useCallback(() => setCount(c => c + 1), []);
  const reset = useCallback(() => setCount(initial), [initial]);
  // stable unless a dep truly changed — safe in consumers' deps
  return useMemo(() => ({ count, increment, reset }), [count, increment, reset]);
}

Parameterizing: values vs refs vs option objects

  • Primitives are the friendliest parameters — identity-safe in your internal deps.
  • Objects/functions from callers are hostile by default — assume inline creation each render. Defend: destructure to primitives for deps, serialize where cheap (JSON.stringify(threshold)), or accept-and-useLatest callbacks so your effect doesn't restart when the caller passes a new lambda.
  • Refs as parameters when the hook needs a DOM node (useIntersectionObserver(ref, opts)): the ref object is identity-stable, but ref.current is not reactive — the hook must read it inside effects, and callback-ref designs handle late/conditional attachment better.

SSR safety

A library-grade hook must not explode on the server: no window/document at module scope or in render; initial state must be deterministic (guard with typeof window !== 'undefined' in initializers, do real reads in effects); anything subscribing to a browser API should prefer useSyncExternalStore with a getServerSnapshot — it makes the server value an explicit, hydration-safe part of the design instead of an accident.

Walkthrough 1: useDebounce, properly

export function useDebounce(value, delay = 300) {
  const [debounced, setDebounced] = useState(value);   // seeded — no undefined flash

  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id);   // each change cancels the pending timer —
  }, [value, delay]);                // this cancellation IS the debounce

  return debounced;
}

// Usage: input stays perfectly responsive; search fires 300ms after typing stops
const [query, setQuery] = useState('');
const debouncedQuery = useDebounce(query, 300);
useEffect(() => { /* fetch with debouncedQuery */ }, [debouncedQuery]);

Design notes: (1) the entire mechanism is the effect-cleanup pairing — every value change clears the previous timer, so only the last one in a quiet window survives; (2) seeding useState(value) means consumers never see an empty first render; (3) delay is a dep — changing it mid-flight restarts correctly; (4) if value is an object created inline by the caller, the debounce never settles — either document "pass primitives" or add a custom equality parameter. That caveat, volunteered, is a senior signal.

Walkthrough 2: useIntersectionObserver, with option-identity handling

export function useIntersectionObserver(
  targetRef,
  { root = null, rootMargin = '0px', threshold = 0 } = {}
) {
  const [entry, setEntry] = useState(null);

  // threshold may be a number or array; serialize so an inline [0, 0.5, 1]
  // doesn't recreate the observer every render
  const thresholdKey = JSON.stringify(threshold);

  useEffect(() => {
    const el = targetRef.current;
    if (!el || typeof IntersectionObserver === 'undefined') return;  // SSR / old browser

    const observer = new IntersectionObserver(
      ([e]) => setEntry(e),
      { root, rootMargin, threshold: JSON.parse(thresholdKey) }
    );
    observer.observe(el);
    return () => observer.disconnect();   // no leaked observers on unmount/option change
  }, [targetRef, root, rootMargin, thresholdKey]);

  return entry;   // callers derive: const isVisible = !!entry?.isIntersecting
}

Design decisions to narrate: destructure the options parameter immediately so deps are primitives (an inline {} from the caller costs nothing); serialize threshold because it may be an array; read targetRef.current inside the effect (render-time reads are illegal and the node may not exist yet); cleanup disconnects so option changes swap observers atomically. Remaining honest limitation: if the ref attaches after this effect runs (conditional rendering), the effect won't re-fire — the fix is a callback-ref variant where attachment itself triggers state; mentioning that trade-off unprompted is exactly the depth interviewers want.

13. Interview Q&A

1. Why must hooks be called in the same order on every render?

Hooks are stored as a linked list on the component's fiber (fiber.memoizedState), with no names or keys — each hook call during render consumes the next node in the list. React's only way to match "this useState call" to "that stored state" is call position. If a conditional skips a hook, every subsequent call reads the previous hook's node: state values shift into the wrong variables. React detects the count/order mismatch and throws ("Rendered fewer hooks than expected"), but the underlying failure is slot misalignment. The Rules of Hooks (top level only; only from components/custom hooks) exist purely to keep that sequence deterministic — and tools like the React Compiler build on that guarantee.

2. What is the hooks "dispatcher," and why do hooks only work while rendering?

The useState you import is a shim that delegates to ReactCurrentDispatcher.current — a mutable reference the renderer sets immediately before calling your component: a mount dispatcher on first render (creates hook nodes), an update dispatcher on re-renders (walks existing nodes), and a throwing implementation outside render. So calling a hook outside a component fails because no valid dispatcher is installed — not because of any syntactic rule. Bonus points: this also explains the "two copies of React" version of "Invalid hook call" (the rendering copy set its dispatcher; you imported the hook from the other copy), and why mount and update behave differently (they're literally different implementations).

3. Is setState synchronous? What does it actually do?

The setter never mutates the current render's variable. It (1) enqueues an update object on the hook's queue, (2) schedules a re-render of the fiber. The update is applied when React processes the queue during the next render — so within the current execution, state is unchanged, which is why console.log(count) right after setCount(count + 1) logs the old value. In React 18+, updates from any origin (handlers, timeouts, promises) are batched into a single re-render. The re-render itself is usually async (scheduled), and flushSync is the escape hatch to force it synchronously. Precise phrasing that scores: "state is a snapshot per render; setters request the next snapshot."

4. Three calls to setCount(count + 1) vs three to setCount(c => c + 1) — results and why?

+1 vs +3. count + 1 is evaluated immediately from the closure's snapshot — with count = 0, all three enqueue "replace with 1." Updater functions are stored uncalled; when React drains the queue it feeds each one the running pending result: 0→1→2→3. Mixing follows the same drain order: set(count + 5); set(c => c + 1); set(42) ends at 42 (the final replace discards pending work). Rule of thumb: whenever the next state depends on the previous, use the updater — it's also the cheapest stale-closure fix inside effects and async callbacks.

5. When does calling a setter not cause a re-render?

When the new value is Object.is-equal to the current one, React bails out: no commit, children skipped. Nuance: React may still execute that one component's function once more before deciding to bail (when it can't determine equality eagerly), but it won't descend or commit — so "no visible re-render" is the accurate claim. Practical corollaries: mutating an object and setting the same reference never re-renders (top real-world bug), and spreading into a new-but-identical object always re-renders. The same Object.is comparison governs deps arrays, context values, and getSnapshot results — it's the one equality primitive across the hooks API.

6. What's the difference between useState(expensive()) and useState(() => expensive())?

Both produce the same initial state, but the first calls expensive() on every render — it's an ordinary argument expression — with the result thrown away after mount. The second passes a lazy initializer that React invokes exactly once when creating the hook node. Use it whenever computing initial state is non-trivial (parsing localStorage, building an index). Follow-up trap: to store a function in state you must double-wrap — useState(() => myFn) — or React will call myFn as the initializer. The same lazy pattern exists as useReducer's third argument.

7. Explain exactly when effects and their cleanups run.

Pipeline: render → commit (DOM mutations, then refs attached, then useLayoutEffect cleanups+setups synchronously) → browser paint → passive (useEffect) cleanups, then setups. On a deps change, the previous run's cleanup executes right before the new setup — cleanup is not unmount-only; every effect run pairs with its own cleanup, closing over that run's values. On unmount, only the cleanup runs. Effects flush bottom-up (children before parents). Precision caveat: passive effects usually fire after paint, but React may flush them earlier (e.g., before the next discrete event) — the guarantee is "doesn't block this paint," not "always after paint." And in dev StrictMode: setup → cleanup → setup on mount, deliberately.

8. An interval inside useEffect(..., []) logs count = 0 forever. Trace why and give the fixes.

The effect ran once, after the first render, so the interval callback closes over render #1's scope where const count = 0. Later renders create new closures nobody uses — the interval keeps invoking the original. So it logs 0 every tick; setCount(0 + 1) moves state to 1 once, then every subsequent setCount(1) hits the Object.is bail-out — UI frozen at 1, log stuck at 0. Fixes ranked: (1) setCount(c => c + 1) — effect no longer reads count, deps stay honestly empty; (2) add count to deps — correct but recreates the interval every tick and resets cadence; (3) a latest-value ref read at call time (useLatest) for long-lived callbacks that must see fresh state; (4) know useEffectEvent as the experimental purpose-built answer.

9. How do you prevent race conditions when fetching in an effect?

The bug: effects re-run per dep change but responses arrive out of order, so a stale response can overwrite a newer one. Fix with cleanup-scoped ownership: each effect run creates its own let ignore = false and/or AbortController; the async callback only calls setState if !ignore; the cleanup sets ignore = true and calls controller.abort(). When deps change, the old run is neutralized before the new one starts — only the latest run can commit results. Details that score: swallow AbortError in the catch (it's expected, not an error); the ignore flag works for non-abortable async (dynamic imports, SDK calls); AbortController additionally saves bandwidth by cancelling the request; and in production you'd reach for React Query/SWR or router loaders, which encapsulate exactly this.

10. When does logic belong in an event handler vs an effect vs plain render?

Ask "what causes this code?" Caused by a specific interaction (submit → POST, click → toast) → event handler; putting it in an effect makes it re-fire on unrelated re-renders, double-fire in StrictMode, and detaches cause from code. Computable from current props/state → render (with useMemo only if measured-expensive); an effect that only calls setState from other state is a redundant extra render pass. Caused by "the component is displaying with X and something outside React must match" (subscription, document title, non-React widget, analytics-on-view) → effect. Resetting state on identity change → key. Reading external mutable sources → useSyncExternalStore. Also name the effect-chain anti-pattern: cascading setState-in-effect sequences should collapse into one handler or a reducer.

11. Why does an object or function in a dependency array cause loops, and what are the fixes?

Deps are compared per element with Object.is — reference equality for objects. An inline object/lambda is a new reference each render, so the effect re-runs every render; if it sets state, that's an infinite loop. Fixes in preference order: depend on primitives and rebuild the object inside the effect; move creation into the effect if it's only used there; useMemo/useCallback when the value must exist outside (JSX + effect both use it); hoist to module scope if it's not reactive at all. Senior addendum: the same trap inverted is your responsibility as a custom-hook author — stabilize what you return, defend against unstable option objects you accept.

12. Why is suppressing exhaustive-deps dangerous, and when is it acceptable?

The deps array is a declaration of every reactive value the effect reads; omitting one doesn't stop the effect from using it — it freezes the closure at whatever render last created the effect, so the effect silently operates on stale data with no error. The lint rule is a consistency checker, not a style preference. Most "I need to suppress" cases have structural fixes: updater functions (drop the state dep), move the value inside, memoize the dep, useLatest for callbacks. Acceptable suppressions are rare and intentional — e.g. a strictly-once mount analytics event — and should carry a comment explaining which value is deliberately non-reactive. Bonus: useEffectEvent is the sanctioned future answer for "reads latest values but shouldn't re-trigger."

13. useRef vs useState — how do you choose?

Does changing the value need to change what the user sees? Yes → state (setter schedules a render; value is an immutable per-render snapshot). No — it's bookkeeping the render output never reads (timer ids, DOM nodes, library instances, previous/latest values, "already fired" flags) → ref (stable {current} box; mutation is free and invisible to React). Two contract details: don't read/write refs during render — concurrent React can double-execute or abandon renders, making render-phase mutations non-deterministic — with the one idempotent exception of lazy init (if (ref.current === null) ref.current = make()); and if you catch yourself forcing a render after writing a ref, it was state all along.

14. When exactly is a DOM ref attached, and what changed about refs in React 19?

React sets ref.current during the commit phase — after DOM mutations, before layout effects — so both useLayoutEffect and useEffect can safely use the node, while during render it's still null on mount. On unmount (or when a callback ref changes) React detaches. React 19 changes: ref is now a regular prop on function components (no forwardRef ceremony; forwardRef heads toward deprecation), and callback refs can return a cleanup function that runs on detach — replacing the old "called again with null" idiom and giving ref-based subscriptions effect-style cleanup guarantees.

15. What do useMemo/useCallback actually guarantee? How are they related?

Mechanically: a single-entry cache in the hook node keyed by the deps array; deps Object.is-equal → return cached value without calling the function. useCallback(fn, deps) is exactly useMemo(() => fn, deps) — cache the function itself vs cache its result. Guarantee: none, semantically — the docs reserve the right to discard cached values (memory pressure, offscreen features) and recompute, so treat memoization as a performance hint over pure functions, never as "run once" (that's a ref or effect). Legitimate uses: props to React.memo children, effect deps, context values, measured-expensive computations. Overuse costs allocations, comparisons, and reader complexity — and the React Compiler is the systemic fix.

16. useEffect vs useLayoutEffect — when is the layout version required?

useLayoutEffect runs synchronously after DOM mutation, before paint; useEffect after paint. Required when the user could see an intermediate state: measure-then-position (tooltips, popovers), scroll restoration, synchronous DOM reads that must precede the visible frame. Setting state in a layout effect re-renders synchronously before paint, so only the corrected frame is painted — with useEffect you'd paint the wrong frame first (the flicker). Costs: it blocks painting (keep it to measure + set), and it doesn't run on the server (SSR warning — gate to client or restructure). Rounding it out: useInsertionEffect fires before layout effects, exists for CSS-in-JS libraries to inject styles before any layout reads, and shouldn't appear in app code.

17. When do you reach for useReducer over useState?

When state is a machine rather than independent values: multiple fields that must transition together (loading/error/data), next state derived from previous under non-trivial rules, or invariants worth centralizing ("can't retry unless in error state"). Concrete wins: the reducer is a pure, React-free unit-testable function; TypeScript discriminated unions on both actions and state make illegal states unrepresentable (plus a never default arm for exhaustiveness); and dispatch has a guaranteed stable identity — pass it deep or through context with zero churn, which enables the state-context/dispatch-context split where action-only components never re-render on data changes. Constraint to mention: reducers must be pure (StrictMode double-invokes them in dev to catch mutation).

18. Why do context consumers re-render even under a React.memo parent? How do you tame a context that re-renders everything?

Because context propagation is a separate mechanism from parent-child rendering: when a provider's value fails Object.is against the previous one, React traverses the subtree and schedules every reader of that context — through memo bail-outs. Memoized intermediates skip their own render but don't shield consumers. The classic bug is value={{...}} creating a fresh object every provider render, fanning out to all consumers on unrelated changes. Fixes in order: useMemo the value (with useCallback'd methods); split state and dispatch contexts (dispatch is stable → action-only components never re-render); split slices into separate contexts; and remember providers rendering {children} pass a stable element, so their own state changes don't re-render the subtree except consumers. Beyond that scale — high-frequency or selector-shaped access — move to an external store on useSyncExternalStore.

19. What is tearing, and how does useSyncExternalStore prevent it?

Under concurrent rendering React can pause mid-tree and resume later. If components read a mutable external source directly, the source can change during the pause — component A committed with the old value, component B with the new one: a single frame showing two versions of the truth. React state can't tear (updates are snapshotted through the scheduler); external mutable reads can. useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot) fixes it by re-checking the snapshot during render — if it changed mid-render, React restarts synchronously so every component in the commit sees one consistent snapshot; external-store updates are treated as urgent/synchronous rather than deferrable. Contracts: stable subscribe identity (or it re-subscribes), and getSnapshot must return a cached/same reference until the store truly changes — a fresh object per call means an infinite loop. Redux's useSelector and Zustand sit on this hook (via the shim with selector support).

20. Why useId instead of Math.random() or a global counter for element ids?

Because of SSR + hydration: ids are generated twice — once on the server into the HTML, once on the client during hydration — and they must match. Math.random() obviously differs; a module counter depends on execution order and timing, which streaming SSR and selective hydration make non-deterministic. useId derives the id from the component's structural position in the tree, identical on both sides, so label htmlFor/aria-describedby wiring hydrates cleanly. Two follow-ups: it is not for list keys (keys must come from data — useId is per-hook-call, not per-item), and one useId can prefix multiple related attributes (id, id + '-hint').

21. How do you decide to extract a custom hook, and what makes its API well-designed?

Extract when stateful logic is reused across components, or when it's complex enough that a name improves the component (useVideoPlayer over 60 lines of effects) — and name it after what it synchronizes/computes, never after lifecycle (useMount is a smell). Mechanics: hooks share logic, not state — each caller gets independent hook nodes. API design: single value for one answer; tuple for a rename-friendly pair (the useState convention); object for many/optional fields. Return-identity stability is part of the contract — useCallback returned functions, useMemo returned objects, or consumers inherit churn they can't fix. Defend against hostile inputs (inline option objects → destructure to primitives / serialize; callbacks → latest-ref). And be SSR-safe: deterministic initializers, browser reads in effects, or useSyncExternalStore with a server snapshot.

22. Props-to-state: a component seeds state from a prop, and the UI goes stale when the prop changes. What are the fixes?

Diagnosis: useState(prop) reads its argument only at mount — later prop changes don't touch state, by design. Fixes, best-first: (1) don't copy — if the value is derivable, compute during render; (2) key reset — <Editor key={contact.id} /> when a new identity should mean a fresh instance: the subtree remounts and all state re-initializes; (3) render-time compare for partial resets — store the previous prop in state and call setters during render when it differs; React discards that render and re-runs immediately, before commit, which beats an effect-based reset that paints a stale frame first; (4) lift the state and make it controlled. Historical hook: this is the hooks-era descendant of the componentWillReceiveProps → getDerivedStateFromProps saga and the "You Probably Don't Need Derived State" doctrine — deliberately awkward APIs because the pattern is usually wrong.

23. What does StrictMode's double-invocation actually do to hooks, and why?

In development only, StrictMode: double-invokes component functions, initializers, updaters, and reducers (to surface render-phase impurity — impure code produces visibly wrong results when run twice); and on mount runs effects as setup → cleanup → setup (to verify cleanup fully mirrors setup, simulating the remount behavior future features like state-preserving offscreen/Activity rely on). Consequences you should be able to predict: a mount-only analytics effect fires twice in dev (the fix is idempotence or moving it to the interaction, not deleting StrictMode); a subscription with correct cleanup shows no visible difference — which is precisely the test. Production behavior is completely unaffected. Framing: StrictMode failures are latent bugs being surfaced early, not React noise.

24. Quick React 19 screen: use(), useActionState, useOptimistic, useFormStatus — what are they?

use() reads a promise (suspends until resolved, integrating with Suspense/error boundaries) or a context — and uniquely may be called conditionally, since it doesn't occupy a state slot in the hook list. useActionState wraps an async action (commonly a form action) returning [state, formAction, isPending] — standardized pending/error/result plumbing for mutations. useOptimistic exposes an optimistic view of state you set instantly while the mutation is in flight, auto-reconciled when it settles. useFormStatus (react-dom) lets any child of a form read the form's pending status without prop drilling — e.g. a shared SubmitButton. Together with ref-as-prop and ref cleanup functions, the theme is moving async-mutation and plumbing boilerplate into the framework.

Exercise 1 — Fix the stale interval three different ways

The classic broken counter: the interval logs 0 forever and the UI freezes at 1. Fix it three ways — (a) functional update, (b) latest-value ref, (c) a reusable useInterval custom hook — and state the trade-off of each. (Tracing why it's broken is section 3; here you defend the fixes.)

function Counter() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => {
      console.log(count);          // 0, 0, 0, ...
      setCount(count + 1);         // always setCount(1)
    }, 1000);
    return () => clearInterval(id);
  }, []);
  return <h1>{count}</h1>;
}
Solution

(a) Functional update — the best fix when it fits: the callback no longer reads count at all, so the empty deps array becomes honest and the lint rule is satisfied structurally, not suppressed. Limitation: only works when the update is expressible as prev → next over a single state value; it doesn't help a callback that needs to read several reactive values.

setCount(c => c + 1);   // React feeds the updater the queue's latest value

(b) Latest-value ref — a ref is a stable box, so the once-created interval closure can dereference it at call time instead of capturing a snapshot at creation time:

const countRef = useRef(count);
useEffect(() => { countRef.current = count; });   // re-sync after every commit
useEffect(() => {
  const id = setInterval(() => {
    console.log(countRef.current);                // always fresh
    setCount(countRef.current + 1);
  }, 1000);
  return () => clearInterval(id);
}, []);

Trade-off: it generalizes to arbitrary reads, but it deliberately steps outside React's reactivity — the value is no longer a dependency, so nothing re-runs when it changes. That's the point, and also the danger: use it for "long-lived callback needs fresh values", never as a lint silencer.

(c) useInterval — package the latest-ref pattern once (Dan Abramov's canonical version), and gain pause/resume for free via a nullable delay:

function useInterval(callback, delay) {
  const savedCallback = useRef(callback);
  useEffect(() => { savedCallback.current = callback; });
  useEffect(() => {
    if (delay === null) return;                     // null ⇒ paused
    const id = setInterval(() => savedCallback.current(), delay);
    return () => clearInterval(id);
  }, [delay]);
}

useInterval(() => setCount(count + 1), isRunning ? 1000 : null);

The callback closure is recreated every render (so it's never stale), but the interval only depends on delay — cadence survives re-renders. Trade-off: an abstraction to maintain and explain, but it's lint-clean, reusable, and declarative. Senior close: "adding count to the deps also works, but recreates the timer every tick and drifts the cadence — correct, and the worst of the three."

Exercise 2 — Implement useDebounce and useThrottle

Write useDebounce(value, delay) and useThrottle(value, interval) returning the debounced/throttled value. Requirements: correct cleanup (no timers left after unmount), no lint suppressions, and the throttle must emit a trailing update (the final value always lands). Then explain why the debounced-callback variant is a harder identity problem.

Solution

Debounce falls out of effect semantics almost verbatim — the cleanup of the previous run cancelling the previous timer is the debounce:

function useDebounce(value, delay = 300) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id);   // each new value cancels the pending emit
  }, [value, delay]);
  return debounced;
}

Every value change re-runs the effect; the cleanup kills the not-yet-fired timer, so only a value that survives delay ms of silence commits. Unmount runs the final cleanup — no setState-after-unmount, no leak.

Throttle needs a timestamp ref (bookkeeping the render never reads → ref, not state) plus a trailing timer for the last value in a burst:

function useThrottle(value, interval = 300) {
  const [throttled, setThrottled] = useState(value);
  const lastRun = useRef(Date.now());
  useEffect(() => {
    const elapsed = Date.now() - lastRun.current;
    if (elapsed >= interval) {
      lastRun.current = Date.now();
      setThrottled(value);                 // leading edge: emit immediately
    } else {
      const id = setTimeout(() => {      // trailing edge: last value still lands
        lastRun.current = Date.now();
        setThrottled(value);
      }, interval - elapsed);
      return () => clearTimeout(id);
    }
  }, [value, interval]);
  return throttled;
}

Why the callback variants (useDebouncedCallback) are harder: the returned function must have a stable identity (or consumers' effects and memoized children churn) while still calling the latest callback with the latest closure — exactly the tension Exercise 6 resolves with the latest-ref pattern. Naively wrapping debounce(fn) in useMemo with [fn] deps recreates the debouncer every render (inline fn = new reference), silently resetting its internal timer — the debounce never fires. That failure mode is invisible in a code read and deadly in review questions.

Exercise 3 — usePrevious, and why the timing works

Implement usePrevious(value) returning the value from the previous committed render. The implementation is four lines; the interview substance is explaining the read/write timing that makes it correct, plus one caveat about what "previous" actually means here.

Solution 
function usePrevious(value) {
  const ref = useRef(undefined);
  useEffect(() => {
    ref.current = value;     // runs AFTER commit — too late to affect this render
  });
  return ref.current;        // read during render — still holds last commit's value
}

The timing: during render N, ref.current is read before any effect from render N runs — so it still contains what render N−1's effect wrote. After render N commits, the (deps-less) effect overwrites the box with render N's value, staging it for render N+1. The hook exploits two guarantees: refs mutate without scheduling renders, and effects run strictly after the render output that read them was committed. This is a sanctioned read-during-render because the write happens outside render — it doesn't violate the "no ref mutation during render" contract from section 4.

Caveat: it returns the value as of the previous committed render, not the previous distinct value. If the component re-renders because unrelated state changed, usePrevious(count) now equals count — any prev !== current logic must tolerate that. StrictMode's setup → cleanup → setup just writes the same value twice — harmless, which is precisely what StrictMode is checking. If you need "compare and react during render" (the props-to-state reset from Q&A 22), store the previous value in state and call the setter during render instead — the ref version is one commit late by construction.

Exercise 4 — useLocalStorage: SSR-safe and synced across tabs

Build useLocalStorage(key, initial) with the useState-tuple API. Requirements: no crash during SSR; functional updates supported; a change in one browser tab updates all others. Bonus: identify the hydration flaw the naive version still has, and rewrite on useSyncExternalStore to fix it.

Solution 
function useLocalStorage(key, initial) {
  const [value, setValue] = useState(() => {           // lazy: reads storage once
    if (typeof window === 'undefined') return initial;  // SSR guard
    try {
      const raw = window.localStorage.getItem(key);
      return raw !== null ? JSON.parse(raw) : initial;
    } catch { return initial; }                          // quota / private mode / bad JSON
  });

  const set = useCallback(next => {                      // stable identity — part of the API contract
    setValue(prev => (typeof next === 'function' ? next(prev) : next));
  }, []);

  useEffect(() => {                                      // persist as an effect, not inside the
    window.localStorage.setItem(key, JSON.stringify(value)); // updater — updaters must stay pure
  }, [key, value]);                                      // (StrictMode double-invokes them)

  useEffect(() => {                                      // 'storage' fires in OTHER tabs only
    const onStorage = e => {
      if (e.key === key && e.storageArea === window.localStorage) {
        setValue(e.newValue !== null ? JSON.parse(e.newValue) : initial);
      }
    };
    window.addEventListener('storage', onStorage);
    return () => window.removeEventListener('storage', onStorage);
  }, [key]);

  return [value, set];
}

Design notes worth saying aloud: lazy initializer so storage is read once, not parsed every render; writes moved to an effect because setState updaters must be pure; set is useCallback'd because return-identity stability is part of a custom hook's contract (section 11); the storage event filter checks both key and storageArea (sessionStorage fires the same event type).

The remaining flaw: the SSR guard prevents the crash but not the hydration mismatch — the server rendered initial, the client's first render reads the stored value, and React hydrates against different content. This is exactly the tearing-shaped, "external mutable source" problem useSyncExternalStore exists for:

function useLocalStorage(key, initial) {
  const raw = useSyncExternalStore(
    useCallback(cb => {
      window.addEventListener('storage', cb);
      window.addEventListener('local-storage', cb);   // custom event: same-tab writes
      return () => {
        window.removeEventListener('storage', cb);
        window.removeEventListener('local-storage', cb);
      };
    }, []),
    () => window.localStorage.getItem(key),  // snapshot is the raw STRING — Object.is-stable
    () => null                               // server snapshot: hydrate as "no value"
  );
  const value = useMemo(
    () => (raw !== null ? JSON.parse(raw) : initial),  // parse memoized on the string
    [raw]
  );
  const set = useCallback(next => {
    window.localStorage.setItem(key, JSON.stringify(next));
    window.dispatchEvent(new Event('local-storage')); // notify THIS tab's subscribers
  }, [key]);
  return [value, set];
}

Three contract points from section 9 show up concretely: getSnapshot returns the raw string (a fresh parsed object per call would fail the stability contract and loop); getServerSnapshot makes server and hydration renders agree, then the client corrects itself; and the 'storage' event only fires in other tabs, so same-tab writes need a manual notification — the custom event. Both tabs and all components sharing the key now converge, tear-free.

Exercise 6 — Implement useEventCallback (the latest-ref pattern)

Implement useEventCallback(fn): returns a function whose identity never changes but which always invokes the latest fn with the latest closed-over values. Explain (a) the problem it solves that useCallback cannot, (b) why the ref is synced in useLayoutEffect rather than useEffect, and (c) why the result must not be called during render.

Solution 
function useEventCallback(fn) {
  const ref = useRef(fn);
  useLayoutEffect(() => {
    ref.current = fn;                // re-point at this render's closure, pre-paint
  });
  return useCallback((...args) => ref.current(...args), []);  // stable forever
}

(a) useCallback forces a choice: [] deps → stable identity but a frozen closure (stale reads); honest deps → fresh closure but new identity whenever they change, cascading through React.memo children, effect deps, and subscriptions. useEventCallback decouples the two: identity is a stable trampoline created once; behavior is indirected through a ref that's re-pointed at the newest closure after every render. Stable and fresh — the exact combination a debounced handler, a subscription callback, or an onEvent-style prop into a memoized child needs.

(b) useLayoutEffect syncs the ref synchronously in the commit phase, before paint and before any passive effect. If it were useEffect, there'd be a window after commit in which another component's layout effect — or an event firing between paint and the passive flush — invokes the trampoline and reaches the previous render's closure. Narrow, but exactly the kind of rare-flake bug this hook exists to eliminate, so eliminate the window too.

(c) During render N, ref.current still points at render N−1's closure (the sync hasn't run yet) — so calling it during render can compute from stale props/state; and reading a mutable ref during render is non-deterministic under concurrent rendering anyway (section 4). Hence the name: it's for event-time code paths only.

Name-drop the lineage: this is the userland version of the pattern the React team is standardizing as useEffectEvent — same latest-ref mechanics, same "don't call during render" restriction, plus lint integration that excludes it from deps. If asked "why doesn't React just make all callbacks behave this way?" — because a frozen-per-render closure is the feature that makes renders predictable snapshots; opting out must be explicit.
