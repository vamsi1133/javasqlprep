04 — JVM Internals

Memory, garbage collection, class loading, and JIT — enough depth to sound like you've operated a JVM in production, not just read about it.

Why interviewers ask this. Nobody expects a backend engineer to write a garbage collector. But senior interviews probe JVM internals for two reasons: (1) production incidents — OutOfMemoryErrors, GC pauses, memory leaks — are diagnosed with this knowledge, and (2) it separates "I write Java syntax" from "I understand the platform my code runs on." Coming from Node, you already have a useful mental model: V8 also has a generational GC, a JIT (TurboFan), and heap limits (--max-old-space-size). The JVM is the same ideas with more knobs, more collectors to choose from, and true multi-threading sharing one heap. This is mostly a T2/T3 topic — you need solid, honest answers, not exhaustive ones.

JVM architecture overview T2

The JVM is a spec (implemented most commonly by HotSpot, which ships with OpenJDK). Your .java files compile to .class bytecode — a portable intermediate format — and the JVM loads, verifies, and executes that bytecode. "Write once, run anywhere" is real because the bytecode is platform-neutral; the JVM itself is platform-specific.

The four pieces to be able to name in an interview:

  • Classloader subsystem — finds, loads, verifies, and initializes classes lazily at runtime.
  • Runtime data areas — heap (shared), metaspace (shared), and per-thread stacks + program counters.
  • Execution engine — interprets bytecode initially, then JIT-compiles hot methods to native machine code. The GC also lives here conceptually.
  • Native interface (JNI) — bridge to C/C++ libraries and OS calls (file I/O, sockets, etc.).

V8 comparison: Node has the same broad shape — parse/compile JS, interpret via Ignition, JIT-optimize hot functions via TurboFan, garbage-collect a generational heap. The biggest structural difference: Node runs one JS thread per process sharing the heap with libuv worker threads doing I/O; a JVM process typically runs hundreds of application threads all sharing one heap, which is why per-thread stacks and thread-safe GC matter so much more here.

Runtime memory areas T2

Heap — shared, GC-managed

Every object created with new lives on the heap (with a caveat we'll hit in the JIT section: escape analysis can eliminate some allocations). The heap is shared across all threads and is divided generationally:

HEAP
┌──────────────────────────────────┬───────────────────────────┐
│  YOUNG GENERATION                │   OLD GENERATION          │
│  ┌──────────────┬─────┬─────┐    │   (tenured)               │
│  │    EDEN      │ S0  │ S1  │    │   long-lived objects,     │
│  │ new objects  │ survivor    │  │   promoted from young     │
│  │ born here    │ spaces      │  │                           │
│  └──────────────┴─────┴─────┘    │                           │
└──────────────────────────────────┴───────────────────────────┘
   collected often (minor GC)         collected rarely (major)
  • Eden — where allocations happen. Allocation is extremely cheap: usually a pointer bump in a thread-local allocation buffer (TLAB), so threads don't contend for a lock on every new.
  • Survivor spaces (S0/S1) — two equal spaces; live objects from a minor GC are copied into one of them, ping-ponging between the two on each cycle.
  • Old generation — objects that survive enough minor GCs get promoted here.

Stack — per-thread, frames

Each thread gets its own stack (default ~1MB on most 64-bit platforms, tunable with -Xss). Every method call pushes a frame containing local variables, the operand stack, and a reference to the method's class. Frames pop on return. No GC needed — it's pure LIFO.

Metaspace — class metadata

Class metadata (method bytecode, field layouts, constant pools) lives in metaspace, which is allocated from native memory. Before Java 8 this was PermGen, a fixed-size region inside the heap that caused infamous OutOfMemoryError: PermGen space failures during hot-redeploys. Java 8 replaced it with metaspace, which grows dynamically from native memory (bounded only by -XX:MaxMetaspaceSize if you set it). Interned strings also moved to the regular heap back in Java 7.

Program counter and native memory

Each thread has a program counter holding the address of the current bytecode instruction (undefined while executing native code). Beyond the heap, the JVM process also uses native memory for thread stacks, metaspace, JIT code cache, GC bookkeeping, and direct ByteBuffers — which is why a container needs headroom beyond -Xmx.

What lives where

ThingWhere it lives

Primitive local variables (int, long, …)Stack frame, by value
Primitive fields of an objectInside the object, on the heap
static fieldsWith the class — held via the heap (referenced from class metadata)

String literals (interned strings)String pool, on the heap (since Java 7)

StackOverflowError vs OutOfMemoryError

// StackOverflowError: one thread's stack exhausted — almost always
// runaway recursion. The heap can be nearly empty.
int depth(int n) { return depth(n + 1); }

// OutOfMemoryError: Java heap space — the heap is full and GC
// cannot reclaim enough. All threads are affected.
List<byte[]> hog = new ArrayList<>();
while (true) hog.add(new byte[10_000_000]);

StackOverflowError is per-thread (stack, recursion); OutOfMemoryError is process-wide (heap, metaspace, or native memory exhaustion). Both extend Error, not Exception — catching them is almost always the wrong move, because the JVM may be in an unrecoverable state. The one sane response to OOM is to let the process die and restart (ideally after dumping the heap — see the flags section).

Garbage collection in depth T2

The generational hypothesis

Empirically, most objects die young — request-scoped DTOs, intermediate strings, iterator objects. A small minority live a long time (caches, connection pools, Spring beans). Generational GC exploits this: collect the young generation frequently and cheaply (most of it is garbage, and copying only the few survivors is fast), and touch the old generation rarely.

Reachability and GC roots

Java's GC is a tracing collector (no reference counting — cycles are handled for free). An object is garbage when it's unreachable from any GC root:

  • Local variables and operand stacks of every live thread's frames
  • static fields of loaded classes
  • JNI references held by native code
  • Live Thread objects themselves, and some JVM-internal references

The GC marks everything transitively reachable from roots; everything else is reclaimable. This is why "a static Map that only grows" is a leak: it's permanently reachable from a root.

Minor vs major/full GC, promotion and tenuring

  • Minor GC — Eden fills up → live objects are copied to a survivor space, Eden is wiped. Each survival increments the object's age (stored in its header). When an object's age passes the tenuring threshold (adaptive, up to 15), or the survivor space overflows, it is promoted to the old generation. Minor GCs are frequent and usually take single-digit milliseconds.
  • Major/old-gen GC — collects the old generation. Slower, because the old gen is larger and live-object-dense.
  • Full GC — collects the whole heap (young + old, and unloads classes). In modern collectors a full GC is usually a failure mode (e.g., G1's concurrent cycle couldn't keep up) and shows up as a long stop-the-world pause. Frequent full GCs are the classic symptom of an undersized heap or a leak.

Stop-the-world pauses

To safely examine the object graph, the GC brings all application threads to safepoints and pauses them — a stop-the-world (STW) pause. Every HotSpot collector has STW phases; they differ in how much work happens inside them. Modern collectors do the heavy marking concurrently with the application and keep STW phases short.

G1 in slightly more depth (it's the default — know it best)

G1 ("Garbage First") abandons physically contiguous generations. It divides the heap into ~2048 equal regions (1–32MB each), and each region plays a role — Eden, survivor, old, or humongous (for objects bigger than half a region). Generations are now logical sets of regions.

G1 heap = grid of regions:
┌───┬───┬───┬───┬───┐   E = eden      O = old
│ E │ O │ O │ E │ S │   S = survivor  H = humongous
├───┼───┼───┼───┼───┤   . = free
│ H │ H │ . │ O │ E │
├───┼───┼───┼───┼───┤   G1 evacuates the regions with the
│ O │ . │ E │ O │ . │   most garbage first — hence the name.
└───┴───┴───┴───┴───┘
  • Pause-time goal: -XX:MaxGCPauseMillis=200 (default). G1 sizes the young generation and picks how many regions to evacuate per cycle to try to stay under it. It's a goal, not a guarantee.
  • Concurrent marking: when old-gen occupancy crosses a threshold, G1 marks live objects concurrently with the application.
  • Mixed collections: after marking, young collections also evacuate a handful of the garbage-richest old regions — reclaiming old-gen space incrementally instead of in one huge pause.
  • Evacuation = compaction: live objects are copied to fresh regions, so G1 doesn't fragment the way older concurrent collectors (CMS) did.

Honest V8 comparison: V8 is also generational — a small "new space" collected by a copying Scavenger (semi-space, just like Eden/survivor) and an "old space" collected by a mostly-concurrent mark-sweep-compact (Orinoco). The core ideas — generational hypothesis, copying young collection, promotion, concurrent marking, STW pauses — transfer directly. The differences: the JVM offers multiple selectable collectors with tunable pause goals, its heaps routinely run 10–100× larger, and GC must coordinate safepoints across hundreds of application threads rather than one main JS thread. If you say "V8's Scavenger is basically Eden plus survivor spaces," that's accurate and lands well.

Senior framing for "how would you tune GC?": "First I wouldn't — G1's defaults are good. I'd start by looking at evidence: enable GC logging (-Xlog:gc*), check pause times and full-GC frequency in the logs or via JFR. If pauses are the problem and the heap is large, I'd evaluate ZGC. If the old gen is growing monotonically, that's a leak, not a tuning problem — heap dump and MAT before touching flags." Interviewers want restraint, not a recitation of exotic flags.

Memory leaks in Java T2

"Java has GC, so no memory leaks" is false, and interviewers love this question. A Java leak is an object that's still reachable (so the GC must keep it) but never used again. The GC can't read your intent. The classic patterns:

  • Static collections that only grow. A static Map<UserId, Session> used as an ad-hoc cache with no eviction. Reachable from a GC root forever. Same disease as a module-level Map in Node.
  • Unclosed resources. Streams, JDBC connections, HTTP clients. The Java-side wrapper may eventually be collected, but native file descriptors and pooled connections leak deterministically. Fix: try-with-resources, always.
  • Listener/callback registration without deregistration. Register an observer on a long-lived subject and never unregister — the subject's listener list pins your object (and everything it references). Identical to forgetting removeEventListener in the browser.
  • ThreadLocal in thread pools. Pool threads live for the life of the application. A ThreadLocal value set during a request and never remove()d stays attached to that pool thread forever. Server frameworks like Tomcat warn loudly about this on redeploy. Rule: try { tl.set(x); ... } finally { tl.remove(); }.
  • Caches without eviction. Hand-rolled HashMap caches. Use a real cache (Caffeine) with size/TTL bounds, or soft/weak references where appropriate.
  • Non-static inner classes / lambdas capturing this held by something long-lived — the small holder pins a huge outer object.
// The classic: an unbounded "cache" reachable from a static root
public class UserCache {
    private static final Map<Long, User> CACHE = new HashMap<>();
    public static void put(User u) { CACHE.put(u.id(), u); } // never removed!
}

// The fix: a bounded cache with eviction (Caffeine)
Cache<Long, User> cache = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(Duration.ofMinutes(10))
        .build();

Diagnosing a suspected leak

  1. Confirm the trend: watch old-gen occupancy after full GCs (via jstat -gcutil, GC logs, or your APM). A leak shows a rising floor — heap usage after each full GC keeps climbing.
  2. Capture a heap dump: jmap -dump:live,format=b,file=heap.hprof <pid>, or better, have -XX:+HeapDumpOnOutOfMemoryError already set so the crash produces one.
  3. Analyze in Eclipse MAT (or VisualVM): the "dominator tree" and "Leak Suspects" report show which objects retain the most memory and, crucially, the path to GC roots — i.e., who is holding this alive. That path is your answer.

Know which OutOfMemoryError you're looking at — the message tells you where to look:

OOM messageMeaningUsual cause
Java heap spaceHeap exhaustedLeak, undersized -Xmx, or a genuinely huge allocation
GC overhead limit exceeded>98% of time in GC reclaiming <2% of heapHeap almost-full-but-not-quite; effectively a slow-motion heap-space OOM
MetaspaceClass metadata space exhaustedClassloader leaks (redeploys), runaway dynamic class generation (proxies, bytecode gen)
unable to create native threadOS refused a new threadThread leak / unbounded thread creation, or OS ulimits — not a heap problem

Don't respond to an OOM by just raising -Xmx. If it's a leak, a bigger heap only delays the crash and makes the eventual GC thrashing and heap dump bigger. Diagnose first: is post-full-GC usage flat (undersized heap — raise it) or climbing (leak — find the retainer)?

Reference types: strong, soft, weak, phantom T2

Everything you write normally is a strong reference. java.lang.ref provides weaker grades that let you hold an object without forcing the GC to keep it — the building blocks of caches and cleanup mechanisms.

TypeCollected when…get() behaviorTypical use
Strong (Object o = x)Never, while reachablen/a — it's the objectNormal code
Soft (SoftReference)Only under memory pressure, before an OOM would be thrownObject until cleared, then nullMemory-sensitive caches (though bounded caches like Caffeine are usually better)
Weak (WeakReference)At the next GC once no strong/soft refs remainObject until cleared, then nullCanonical maps — WeakHashMap (weak keys): entry disappears when the key is otherwise unreachable
Phantom (PhantomReference)Enqueued after the object is finalized/unreachable; get() always returns nullAlways nullPost-mortem cleanup of native resources via a ReferenceQueue — the mechanism behind Cleaner, the modern replacement for finalizers
 
// WeakHashMap: entries vanish when the KEY has no other strong refs
Map<CacheKey, Metadata> meta = new WeakHashMap<>();
CacheKey k = new CacheKey("a");
meta.put(k, load(k));
k = null;   // after some future GC, the entry is gone

JS analogue you already know: WeakMap/WeakRef in modern JavaScript are the same concepts (weak keys / weak references). Java just had them for decades and adds the soft and phantom grades.

WeakHashMap has weak keys, not weak values. If the value strongly references its own key, the entry can never be collected — a subtle self-inflicted leak.

Class loading T3

Classes are loaded lazily, at runtime, on first use. The lifecycle has three phases:

LOADING ──► LINKING ─────────────────────────► INITIALIZATION
 find .class      ├─ verify:  bytecode is safe      run static
 bytes, create    ├─ prepare: allocate statics      initializers
 Class object     │           with default values   & static {}
                  └─ resolve: symbolic refs →       blocks
                              actual refs (lazy)
  • Loading — a classloader locates the bytecode (classpath, JAR, network…) and creates the Class object.
  • Linking — verify (bytecode is structurally valid and type-safe — the reason you can't just forge malicious bytecode), prepare (allocate static fields, set to defaults: 0/null/false), resolve (turn symbolic references in the constant pool into direct references; may be deferred).
  • Initialization — run static field initializers and static {} blocks, exactly once, thread-safely (the JVM locks during init — this is what makes the holder-idiom singleton work).

When is a class initialized? Lazily — on first "active use": first instantiation, first static method call, first access to a non-constant static field, or when a subclass is initialized. Merely referencing Foo.class or a static final compile-time constant does not trigger it.

Classloader hierarchy and parent delegation

Bootstrap classloader        (JVM-internal; loads java.base: java.lang.*, java.util.*)
        ▲
Platform classloader         (JDK modules beyond core, e.g. SQL, crypto)
        ▲
Application (system) loader  (your classpath / your app + dependencies)
        ▲
Custom loaders               (app servers, plugins, frameworks)

Parent delegation: when a loader is asked for a class, it first asks its parent, and only tries to load it itself if the parent chain fails. Two reasons why:

  1. Security/consistency: nobody can slip in a fake java.lang.String — the bootstrap loader always wins for core classes.
  2. Uniqueness: a class's identity is (fully-qualified name, defining loader). Delegation ensures shared classes are loaded once by one loader, so instanceof and casts behave sanely.

Where this bites in real life: app servers and plugin systems break delegation deliberately (child-first loading) so each webapp can bundle its own library versions — and that's the root of both classic ClassCastException: X cannot be cast to X bugs (same class name, two loaders) and metaspace leaks on redeploy (an old classloader pinned by one surviving thread or ThreadLocal). You don't need deep expertise here — recognizing the failure signatures is the senior-level bar.

JIT compilation and warm-up T3

HotSpot starts by interpreting bytecode, while counting method invocations and loop iterations. When counters show a method is hot, it's compiled to native code — in tiers:

Interpreter ──hot──► C1 (client compiler)      ──hotter──► C2 (server compiler)
 slow, starts         fast compile, light                   slow compile, heavy
 instantly            optimization, adds                    optimization: aggressive
                      profiling instrumentation             inlining, using C1's profile
  • Tiered compilation (default): interpreter → C1 with profiling → C2. C1 gets you decent speed quickly; the profile it gathers lets C2 make aggressive, speculative optimizations later.
  • Inlining is the most important optimization — copying a small callee's body into the caller — because it unlocks everything else (the optimizer can then see across the call boundary). This is why "many small methods" is not a performance smell in Java.
  • Speculation and deoptimization: C2 optimizes for what the profile has seen (e.g., "this interface call always hits class Foo → inline Foo's method, guarded by a cheap check"). If the assumption later breaks, the JVM deoptimizes back to the interpreter and may recompile. Exactly like V8 optimizing for observed hidden classes and deopting on shape change.
  • Escape analysis: if the JIT proves an object never escapes a method (no field, no return, no other thread can see it), it can avoid a real heap allocation via scalar replacement — breaking the object into local variables. Awareness-level point: "not every new is a heap allocation once C2 is involved."

Why JVM apps "warm up": for the first seconds-to-minutes, hot paths are still interpreted or C1-compiled, classes are still loading, and profiles are still building. This is why (a) benchmarks need warm-up iterations and a real harness (JMH), and (b) first-request latency after deploy is worse. Mitigations to name-drop: keep instances warm behind the LB before shifting traffic, CDS/AppCDS for faster class loading, and — for serverless/CLI-style cold-start problems — GraalVM native image (compiles ahead-of-time; trades peak JIT performance and some dynamic features for instant startup).

V8 mapping, which is accurate and worth saying out loud: Ignition ≈ the interpreter, Sparkplug/Maglev ≈ C1's role, TurboFan ≈ C2, deopt ≈ deopt. The JVM's advantage is decades of profile-guided inlining tuned for large server workloads; the model is the same.

Container gotcha worth telling as a story: heap (-Xmx) is only part of the JVM's memory. Metaspace, thread stacks (1MB × 400 threads = 400MB), JIT code cache, GC overhead, and direct buffers all live outside it. Set -Xmx equal to the container limit and Kubernetes will OOM-kill the pod with no Java exception at all — the process just disappears with exit code 137. That's the diagnostic tell: OutOfMemoryError in logs = heap; silent kill + 137 = container limit.

Interview Q&A

Walk me through the JVM's high-level architecture.

Three main subsystems plus a native bridge. The classloader subsystem loads .class bytecode lazily at runtime — loading, then linking (verify/prepare/resolve), then initialization. The runtime data areas hold state: a shared heap for objects, metaspace for class metadata, and per-thread stacks and program counters. The execution engine interprets bytecode and JIT-compiles hot methods to native code via tiered compilation (C1 then C2); the garbage collector also lives here. Finally, JNI bridges to native libraries and OS calls. Conceptually it's very close to V8/Node — interpreter plus optimizing JIT plus generational GC — but multi-threaded with a shared heap and far more operational tunability.

What lives on the heap vs the stack in Java?

All objects live on the heap — everything created with new, arrays, boxed types. The stack holds per-thread method frames: primitive local variables by value, and references to heap objects (the pointer is on the stack; the object it points at is on the heap). Primitive fields of an object live inside the object on the heap. Class metadata lives in metaspace. One nuance worth adding: JIT escape analysis can scalar-replace objects that provably don't escape a method, so not literally every new results in a heap allocation in optimized code.

StackOverflowError vs OutOfMemoryError — difference and causes?

StackOverflowError is per-thread: one thread's call stack (default ~1MB) is exhausted, almost always by runaway recursion; the heap may be fine. OutOfMemoryError is resource exhaustion at the process level, and the message matters: Java heap space (heap full — leak or undersized -Xmx), Metaspace (class metadata — classloader leaks or dynamic class generation), GC overhead limit exceeded (spending >98% of time in GC for <2% reclaimed — a heap-full death spiral), unable to create native thread (OS thread limits, not memory at all). Both are Errors — you don't catch them; you let the process die, ideally with -XX:+HeapDumpOnOutOfMemoryError set so you have evidence.

Explain the generational hypothesis and why the heap is split into generations.

The observation is that most objects die young — request-scoped objects, temporary strings — while a small minority live a long time. So the heap is split: new objects go to Eden in the young generation; when Eden fills, a minor GC copies the few survivors to a survivor space and wipes Eden wholesale, which is very cheap because copying-collector cost is proportional to live data, not garbage. Objects that survive repeated minor GCs get promoted to the old generation, which is collected far less often. The payoff: frequent, fast, small collections handle the bulk of garbage; expensive whole-heap work is rare. V8 does exactly the same thing — its Scavenger on new space is a semi-space copying collector, morally identical to Eden plus survivor spaces.

What are GC roots, and how does the JVM decide an object is garbage?

Java uses tracing GC, not reference counting. GC roots are the starting points that are alive by definition: local variables on every live thread's stack frames, static fields of loaded classes, JNI references, and live threads themselves. The collector marks everything transitively reachable from roots; anything unmarked is garbage. Because it's reachability-based, reference cycles are collected for free — two objects pointing at each other but unreachable from any root are still garbage. The flip side is that reachable-but-useless objects are leaks: a static map that only grows is permanently reachable, so the GC must keep it.

Minor GC vs major/full GC — what's the difference operationally?

Minor GC collects the young generation: frequent, short (often single-digit milliseconds), triggered when Eden fills; survivors age and eventually promote to old gen. Major GC collects the old generation; with G1 this happens incrementally via concurrent marking plus mixed collections that evacuate the garbage-richest old regions alongside young ones. A full GC — the entire heap, stop-the-world — is in modern collectors mostly a failure mode, e.g. G1 couldn't reclaim space fast enough and fell back. Operationally: minor GCs are business as usual; frequent full GCs in the logs mean an undersized heap or a leak, and that's what I'd investigate.

Compare G1 and ZGC. When would you switch?

G1, the default since Java 9, divides the heap into regions, marks concurrently, and evacuates incrementally to meet a pause-time goal (default 200ms) — good balanced behavior with pauses typically in the tens of milliseconds. ZGC does essentially all its work concurrently using colored pointers and load barriers, delivering typically sub-millisecond pauses nearly independent of heap size, and it's generational since Java 21. The trade: ZGC spends more CPU on barriers and needs more memory headroom, so throughput can be somewhat lower. I'd stay on G1 by default and switch to ZGC when I have measured evidence that GC pauses violate latency SLOs — tail-latency-sensitive services or very large heaps — not preemptively.

Java has GC — how can it still leak memory? Give concrete examples.

GC frees unreachable objects; a leak is an object that stays reachable but will never be used again — the GC can't know your intent. Concrete patterns I'd name: a static collection used as a cache with no eviction; listeners registered on long-lived subjects and never removed (same bug as a forgotten removeEventListener); ThreadLocal values not remove()d on pooled threads, which live forever; unclosed resources leaking native handles; and hand-rolled unbounded caches — the fix being a bounded cache like Caffeine with size and TTL limits. Diagnosis: confirm the rising-floor pattern in old-gen occupancy after full GCs, take a heap dump, and use Eclipse MAT's dominator tree and path-to-GC-roots to find exactly who is retaining the memory.

Why is ThreadLocal dangerous in a thread pool specifically?

A ThreadLocal value is stored in a map attached to the thread. With short-lived threads, the value dies with the thread. But pool threads (Tomcat workers, executor threads) live for the entire application lifetime, so a value set during one request and never removed stays pinned to that thread indefinitely — a leak, and worse, a correctness bug: the next request handled by that thread sees the previous request's stale value. The discipline is set() in a try and remove() in finally. This is also a classic source of classloader/metaspace leaks on webapp redeploy, because a value's class can pin the old app's entire classloader.

Explain the four reference types and a real use for weak references.

Strong is the default — the object lives while reachable. Soft references are cleared only under memory pressure, as a last resort before OOM — usable for memory-sensitive caches, though bounded caches are usually a better tool. Weak references are cleared at the next GC once no stronger references exist — the canonical use is WeakHashMap, where entries disappear when the key is otherwise unreachable, e.g. associating metadata with objects you don't own without prolonging their lifetime. Phantom references never return the object (get() is always null); they're enqueued on a ReferenceQueue after the object is unreachable, enabling post-mortem native-resource cleanup — the mechanism under java.lang.ref.Cleaner, the modern replacement for finalizers. JS's WeakMap/WeakRef map directly onto the weak variant.

What replaced PermGen, and why?

Java 8 replaced PermGen with metaspace. PermGen was a fixed-size region of the heap holding class metadata; because it was fixed and hard to size, apps that loaded many classes — especially app servers doing hot redeploys where old classloaders leaked — died with OutOfMemoryError: PermGen space. Metaspace allocates class metadata from native memory and grows dynamically, bounded only if you set -XX:MaxMetaspaceSize (worth doing in containers, so a classloader leak surfaces as a diagnosable Java OOM rather than the container being killed). Note the leaks themselves didn't go away — a pinned classloader now exhausts metaspace instead — the failure just moved and got more room.

Describe the class loading process and the parent delegation model.

Three phases: loading (find bytecode, create the Class object), linking — verify (bytecode is type-safe), prepare (allocate statics with default values), resolve (link symbolic references, lazily) — and initialization (run static initializers exactly once, thread-safely, on first active use). Loaders form a hierarchy: bootstrap (core java.*), platform (rest of the JDK), application (your classpath), plus custom loaders. Parent delegation means a loader asks its parent before loading anything itself — for security (no one can substitute a fake java.lang.String) and identity (a class's identity is name plus defining loader, so delegating keeps shared classes unique). App servers deliberately break delegation for per-webapp library isolation, which is where "X cannot be cast to X" errors come from — same class name, two loaders.

Why do JVM applications need to "warm up", and how does the JIT work?

HotSpot starts interpreting bytecode while profiling invocation and loop counts. Hot methods get compiled in tiers: C1 compiles quickly with light optimization and inserts profiling; C2 later recompiles the hottest methods with heavy, profile-guided optimization — aggressive inlining, speculative devirtualization, escape analysis. Speculation means C2 optimizes for observed behavior and deoptimizes back to the interpreter if an assumption breaks — the same optimize/deopt cycle as V8's TurboFan. Warm-up is the window before hot paths reach C2: first requests after deploy are slower, and naive microbenchmarks measure the interpreter, which is why JMH exists. Mitigations: warm instances before shifting traffic, CDS for class-loading, or GraalVM native image when cold start dominates — trading peak JIT throughput for instant startup.

How would you size the JVM heap in a Kubernetes container?

Two rules. First, the heap is only part of the process's memory — metaspace, thread stacks, JIT code cache, GC overhead, and direct buffers live outside -Xmx — so the heap should get roughly 50–75% of the container limit, e.g. -XX:MaxRAMPercentage=75, which is container-aware via cgroup limits and adapts if the pod is resized. Second, set -Xms equal to -Xmx (or use the percentage for both): no resize churn and a predictable footprint. The failure mode to articulate: if heap plus native memory exceeds the container limit, Kubernetes OOM-kills the pod — exit code 137, no Java exception in the logs. That silent-kill-vs-OutOfMemoryError distinction is the diagnostic tell for "container limit" vs "heap" problems.

Production service's memory grows until it crashes daily. Walk me through your diagnosis.

First, characterize: is it heap or native? If logs show OutOfMemoryError: Java heap space, it's heap; if the pod dies silently with 137, suspect native/container sizing. For heap: watch old-gen occupancy after full GCs with jstat -gcutil or GC logs — a rising floor confirms a leak versus a merely undersized heap. Then evidence: ensure -XX:+HeapDumpOnOutOfMemoryError is set so the next crash yields an .hprof, or take one live with jmap -dump:live. Open it in Eclipse MAT, use the dominator tree and Leak Suspects report, and follow the path to GC roots to find the retainer — typically a static collection, a cache without eviction, or ThreadLocals on pool threads. Fix the retention, and only then consider whether the heap size also needs adjusting. What I wouldn't do is just raise -Xmx — that delays the crash and enlarges the eventual dump.

The service is completely hung — requests time out, CPU is near zero. What do you do?

Low CPU plus hung points at blocking, not computation — so a thread dump, not a profiler. jstack <pid> (or kill -3 for a dump to stdout). I'd look for: an explicit "Found one Java-level deadlock" section, which jstack detects and prints with the exact lock cycle; or, short of a true deadlock, all pool threads in WAITING/BLOCKED on the same monitor or stuck in socket reads — pool starvation from a slow downstream with no timeouts. Take two or three dumps a few seconds apart: threads stuck at the same frames across dumps are genuinely stuck, not just busy. The fix depends on the finding — consistent lock ordering for deadlock, timeouts and bulkheads for starvation.

How does JVM GC compare to V8's, honestly?

The fundamentals are the same: both are generational tracing collectors based on the weak generational hypothesis. V8's new space uses a semi-space copying Scavenger — directly analogous to Eden plus survivor spaces — and its old space uses concurrent mark-sweep-compact with incremental marking to keep main-thread pauses short. Both promote survivors, both have stop-the-world phases, both moved marking work concurrent/parallel over time. The differences are operational: the JVM offers multiple selectable collectors (Parallel for throughput, G1 balanced, ZGC for sub-millisecond pauses) with explicit tuning like pause-time goals; JVM heaps routinely run tens of GB where Node's default old-space cap is a few GB; and JVM GC must safepoint hundreds of application threads sharing one heap, versus V8's single JS thread per isolate. So: same theory, much bigger and more configurable machine.

What is escape analysis and why should an application developer care?

Escape analysis is a C2 optimization that determines whether an object can "escape" the method that creates it — via being returned, stored in a field, or published to another thread. If it provably can't, the JIT can apply scalar replacement: never allocate the object at all, just keep its fields in registers/locals; it can also elide locks on objects only one thread can see. Practical consequences: short-lived helper objects in hot paths are often free in optimized code, so idiomatic "allocate a small immutable object" style is cheaper than it looks; and it's a reason micro-optimizing allocations by hand before profiling is usually wasted effort. Caveat I'd state: it's best-effort and depends on inlining succeeding first, so I'd verify with a JFR allocation profile rather than assume.

What's the difference between a heap dump and a thread dump, and when do you take each?

A thread dump (jstack) is a cheap, instant snapshot of every thread's stack trace and state — for hangs, deadlocks, pool starvation, "what is this service doing right now." A heap dump (jmap -dump, or automatic on OOM) is a full snapshot of every object in the heap — large, and it pauses the JVM while written, so it's the tool for memory questions: leaks, "what's eating the heap," analyzed offline in MAT. Rule of thumb: hang-shaped problems → thread dump; memory-shaped problems → heap dump; performance-shaped problems → JFR recording, which profiles continuously at ~1% overhead and is safe in production.

When are static initializers run, and is that thread-safe?

Lazily, on first active use of the class: first instantiation, first static method call, first read/write of a non-constant static field, or initialization of a subclass. Merely referencing Foo.class or a static final compile-time constant doesn't trigger it. The JVM guarantees initialization runs exactly once and locks during it, so other threads block until it completes — which is precisely why the initialization-on-demand holder idiom gives you a thread-safe lazy singleton with zero synchronization code of your own. The dark side of that lock: if a static initializer itself blocks (e.g., does I/O) you can get threads piling up or even initialization deadlocks between two classes initializing each other — so keep static initializers trivial.

Hands-on exercises

Exercise 1 — Trigger and observe both memory errors

Write one class with two methods: one that causes a StackOverflowError and one that causes OutOfMemoryError: Java heap space. Run the heap one with -Xmx64m -XX:+HeapDumpOnOutOfMemoryError and confirm an .hprof file is produced. Explain why one error is per-thread and the other process-wide.

Solution 
public class Blowups {
    // java -Xss512k Blowups stack
    static long recurse(long n) { return recurse(n + 1); }

    // java -Xmx64m -XX:+HeapDumpOnOutOfMemoryError Blowups heap
    static void exhaustHeap() {
        var hog = new java.util.ArrayList<byte[]>();
        while (true) hog.add(new byte[1_048_576]); // 1MB chunks
    }

    public static void main(String[] args) {
        if (args[0].equals("stack")) recurse(0); else exhaustHeap();
    }
}

The recursion exhausts one thread's stack — other threads would be unaffected, and the heap is nearly empty when it dies. The heap version keeps every array reachable from the hog local (a GC root via the stack frame), so GC can reclaim nothing; when allocation fails after a last-ditch full GC, the JVM throws OutOfMemoryError: Java heap space and, with the flag set, writes java_pid<pid>.hprof first. Open it in Eclipse MAT — the dominator tree shows the ArrayList retaining ~64MB of byte[], with a path to GC roots ending at main's stack frame.

Exercise 2 — Watch generational GC happen

Write a loop that allocates short-lived garbage (e.g., builds and discards 1KB strings) while keeping a small, growing list of long-lived objects. Run it with -Xlog:gc -Xmx256m and, in a second terminal, jstat -gcutil <pid> 1s. Identify: minor GCs, survivor space ping-ponging, and promotion to old gen.

Solution 
public class GcWatch {
    public static void main(String[] a) throws Exception {
        var longLived = new java.util.ArrayList<byte[]>();
        for (long i = 0; ; i++) {
            byte[] garbage = new byte[1024];      // dies young
            if (i % 100_000 == 0) {
                longLived.add(new byte[64 * 1024]); // survives
                Thread.sleep(50);
            }
        }
    }
}

What to observe: the GC log shows frequent Pause Young (Normal) entries — minor GCs, each just a few ms, reclaiming almost all of Eden. In jstat -gcutil, the E (Eden) column repeatedly climbs to ~100 and drops to ~0; S0/S1 alternate between ~0 and nonzero — the survivor ping-pong; and O (old gen) ratchets slowly upward as the long-lived arrays age past the tenuring threshold and get promoted. YGC counts up fast; FGC should stay near zero. This is the healthy baseline picture — knowing it makes the leak pattern (old gen climbing and never dropping after full GCs) instantly recognizable.

Exercise 3 — Build a leak, then find it with a histogram

Simulate the classic listener leak: a long-lived EventBus with a static list of listeners, and a loop that registers a new listener (each holding a 1MB payload) per "request" without ever unregistering. Use jmap -histo:live <pid> to identify the leaking class, then fix the leak two different ways.

Solution 
class EventBus {
    static final java.util.List<Runnable> LISTENERS = new java.util.ArrayList<>();
    static void register(Runnable r) { LISTENERS.add(r); }
}

class RequestHandler {
    final byte[] payload = new byte[1_048_576];
    void handle() { EventBus.register(() -> use(payload)); } // leak: never removed
    void use(byte[] p) {}
}

Running jmap -histo:live <pid> | head shows byte[] dominating (by bytes) and an ever-growing count of RequestHandler and lambda instances — the chain is: GC root → static LISTENERS → lambda → captured payload. Fixes: (1) deterministic deregistration — register() returns a handle and the handler removes itself in a finally (or the bus is request-scoped, not static); (2) weak listeners — the bus holds WeakReference<Runnable> and prunes cleared refs on dispatch, so a listener whose owner is gone stops pinning memory. Option 1 is preferable when you control the lifecycle — weak listeners can make delivery nondeterministic (a listener may vanish whenever GC runs, and a lambda referenced only by the weak ref can be collected immediately). Option 2 fits frameworks that can't trust callers to unregister.

Exercise 4 — Read a deadlocked thread dump

Write two threads that acquire two locks in opposite order to force a deadlock, then capture a thread dump with jstack <pid> and locate the deadlock report. State the fix.

Solution 
public class Deadlock {
    static final Object A = new Object(), B = new Object();
    public static void main(String[] args) {
        new Thread(() -> { synchronized (A) { pause(); synchronized (B) {} } }, "t1").start();
        new Thread(() -> { synchronized (B) { pause(); synchronized (A) {} } }, "t2").start();
    }
    static void pause() { try { Thread.sleep(100); } catch (InterruptedException e) {} }
}

jstack output ends with an explicit section:

Found one Java-level deadlock:
=============================
"t1":  waiting to lock monitor 0x... (object 0x..., a java.lang.Object),
       which is held by "t2"
"t2":  waiting to lock monitor 0x... (object 0x..., a java.lang.Object),
       which is held by "t1"

Each thread's stack shows - waiting to lock <0x...B> and - locked <0x...A> annotations, letting you reconstruct the cycle even without the summary. The fix is a global lock ordering — both threads acquire A before B, making a cycle impossible; alternatives are collapsing to a single lock, or ReentrantLock.tryLock with timeout and backoff when ordering can't be imposed. Note jstack only auto-detects monitor and java.util.concurrent lock deadlocks — pool-starvation "virtual deadlocks" (all pool threads waiting on work that needs the same pool) require reading the dump yourself.
