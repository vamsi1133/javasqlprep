Controllers & request mapping T1

@RestController vs @Controller

Both are stereotypes (specializations of @Component, so they get picked up by component scanning — see chapter 05). The difference is what happens to a handler method's return value:

  • @Controller — the classic Spring MVC stereotype. By default, the return value (a String, typically) is interpreted as a view name, resolved by a ViewResolver to a server-rendered template (Thymeleaf, JSP). This is the pre-SPA world.
  • @ResponseBody — placed on a method (or class), it says: "do not resolve a view; instead serialize the return value directly into the HTTP response body via an HttpMessageConverter."
  • @RestController — literally just @Controller + @ResponseBody as a meta-annotation. Every method's return value goes to the body. This is what you use for JSON APIs, i.e. everything you've ever consumed from React.
// The two are equivalent:

@Controller
public class UserHtmlController {
    @GetMapping("/users/{id}")
    public String userPage(@PathVariable Long id, Model model) {
        model.addAttribute("user", service.find(id));
        return "user-detail";   // view name → templates/user-detail.html
    }

    @GetMapping("/api/legacy/users/{id}")
    @ResponseBody                     // opt this ONE method into body serialization
    public UserResponse userJson(@PathVariable Long id) {
        return service.find(id);      // serialized to JSON by Jackson
    }
}

@RestController                       // = @Controller + @ResponseBody on the class
@RequestMapping("/api/users")         // class-level prefix, combined with method paths
public class UserController {
    // every method's return value → response body
}

Interview soundbite: "@RestController is a convenience meta-annotation combining @Controller and @ResponseBody. The real mechanism is @ResponseBody, which routes the return value through the HttpMessageConverter chain instead of the view-resolution pipeline." Naming the actual mechanism (message converters) is what separates a senior answer from a memorized one.

Binding pieces of the request: the full menu

Think of each annotation as answering "which part of the HTTP request feeds this parameter?" — the same request you used to assemble on the React side with template strings and JSON.stringify:

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService;
    public OrderController(OrderService orderService) { this.orderService = orderService; }

    // GET /api/orders/42            → path segment → @PathVariable
    @GetMapping("/{id}")
    public OrderResponse one(@PathVariable long id) {
        return orderService.findById(id);
    }

    // GET /api/orders?status=SHIPPED&page=0&size=20   → query string → @RequestParam
    @GetMapping
    public Page<OrderResponse> list(
            @RequestParam(required = false) OrderStatus status,   // optional filter (null if absent)
            @RequestParam(defaultValue = "0") int page,           // default when absent
            @RequestParam(defaultValue = "20") int size) {
        return orderService.search(status, page, size);
    }

    // POST /api/orders  with a JSON body       → @RequestBody (deserialized by Jackson)
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public OrderResponse create(@Valid @RequestBody CreateOrderRequest request) {
        return orderService.create(request);
    }

    // Header extraction — e.g. a tenant id or an idempotency key
    @PostMapping("/{id}/refund")
    public RefundResponse refund(
            @PathVariable long id,
            @RequestHeader("Idempotency-Key") String idempotencyKey,
            @RequestHeader(value = "X-Tenant-Id", required = false) String tenantId) {
        return orderService.refund(id, idempotencyKey, tenantId);
    }
}

Key behaviors interviewers probe:

  • @RequestParam required by default. A missing required param → MissingServletRequestParameterException → 400. Make it optional with required = false (use a wrapper type like Integer, not int, or you'll get an IllegalStateException trying to assign null to a primitive) or with defaultValue (which implies not-required). Optional<String> also works.
  • @PathVariable name matching. If the parameter name matches the template variable ({id} ↔ long id) you can omit the annotation value — but only if the code is compiled with -parameters (Spring Boot's build plugins do this for you; since Spring 6.1 it fails fast with a clear message otherwise).
  • @RequestBody reads the body once. There is exactly one body; you can't have two @RequestBody params. Malformed JSON → HttpMessageNotReadableException → 400.
  • @ModelAttribute (awareness): binds query/form parameters onto a POJO's setters — the form-post world. For JSON APIs you use @RequestBody; but @ModelAttribute is implicitly what happens when you declare a POJO parameter with no annotation on a GET (handy for binding many filter params into one SearchCriteria object). Know it exists and how it differs: @RequestBody = parse the body with Jackson; @ModelAttribute = bind request parameters via setters, with validation errors going to a BindingResult.

Type-conversion failures are 400s, not 500s: GET /api/orders/abc against @PathVariable long id throws MethodArgumentTypeMismatchException. If your global error handler doesn't map it, Boot's default does — but with a generic message. Handle it explicitly so the frontend gets "id must be a number", not a blob.

Mapping from your React mental model: fetch(`/api/orders/${id}?status=${s}`, { method: 'POST', headers: { 'Idempotency-Key': k }, body: JSON.stringify(payload) }) decomposes exactly into @PathVariable (the template literal segment), @RequestParam (the query string), @RequestHeader (the headers object), and @RequestBody (the JSON body). Every annotation is one field of the fetch options object.

Hands-on: build a CRUD REST API end-to-end Hands-on

You have the individual annotations now. This section wires them into one complete, runnable resource — a Task API — so you can see how a request travels down through the layers and back, and reproduce the whole vertical slice from memory. Everything after this section (Jackson internals, the DTO rationale, the exception ladder, validation edge-cases) then dissects a piece you'll have already seen working end-to-end. Think of it as the reference implementation you build once and refer back to; it's the Java equivalent of scaffolding a full CRUD feature — route, handler, service, data layer — the way you'd stand up an Express router with a Prisma model behind it.

We build a single resource, Task (fields: title, description, status, timestamps), with the full stack: Entity → Repository → Service → DTOs → Controller, plus validation, pagination, a ProblemDetail error contract, and one outbound call to an external API. Assume a Spring Boot 3.x project with spring-boot-starter-web, -data-jpa, -validation, and a PostgreSQL driver on the classpath.

1. The layers, one flow

Every request walks the same path down and back up. Each layer has exactly one job — that separation is what keeps the code testable and lets each piece change without breaking the others:

HTTP request  (JSON body / query params)
   │
   ▼
TaskController   @RestController   ← HTTP concerns only: bind & validate input, map DTO⇄domain, choose status codes
   │
   ▼
TaskService      @Service          ← business rules + transactions; knows nothing about HTTP or SQL
   │
   ▼
TaskRepository   @Repository       ← data access; Spring Data generates the SQL from method names
   │
   ▼
PostgreSQL
   │
   ▲   Task (entity) ──> TaskResponse (DTO)   ← mapping happens at the CONTROLLER edge, never deeper
   └───────────────────────────────────────── JSON response

Why the layering earns its keep: the controller can be swapped (REST → GraphQL → a message listener) without touching business logic; the service is unit-testable with a mocked repository and no web server; the repository can be re-pointed at a different query without the service caring. One responsibility per layer, DTO mapping only at the HTTP boundary.

2. The entity

A plain JPA @Entity. Keep it focused on persistence — it is not your API shape (that's the DTO's job, see step 3 and the dedicated DTO section).

@Entity
@Table(name = "tasks")
public class Task {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)   // Postgres identity/serial column assigns the id
    private Long id;

    @Column(nullable = false, length = 200)
    private String title;

    @Column(length = 2000)
    private String description;

    @Enumerated(EnumType.STRING)              // store "TODO", not the ordinal 0 (ordinals break on reorder — see ch.07)
    @Column(nullable = false, length = 20)
    private TaskStatus status = TaskStatus.TODO;

    @CreationTimestamp                        // Hibernate sets this on INSERT
    @Column(updatable = false)
    private Instant createdAt;

    @UpdateTimestamp                          // Hibernate bumps this on every UPDATE
    private Instant updatedAt;

    protected Task() {}                        // JPA requires a no-arg constructor

    // getters + setters omitted — or use Lombok @Getter/@Setter
}

public enum TaskStatus { TODO, IN_PROGRESS, DONE }

3. The DTOs — request and response as records

Two Java records: one for what a client is allowed to send, one for what it's allowed to see. Never bind @RequestBody straight to the entity — that couples your JSON contract to the DB schema, opens you to over-posting (a client sending {"id":1,"createdAt":...} to overwrite server-owned fields), and drags lazy relationships into Jackson's serializer. The DTO section argues all four reasons in full; here we just apply the pattern.

// ---- request DTO: create AND update share this shape; only client-settable fields ----
public record TaskRequest(
        @NotBlank @Size(max = 200) String title,
        @Size(max = 2000) String description,
        @NotNull TaskStatus status          // note: no id, no createdAt — those are server-owned
) {}

// ---- response DTO: exactly what leaves over the wire ----
public record TaskResponse(
        Long id,
        String title,
        String description,
        TaskStatus status,
        Instant createdAt,
        Instant updatedAt
) {
    // static factory keeps the entity→DTO mapping next to the shape it produces
    static TaskResponse from(Task t) {
        return new TaskResponse(t.getId(), t.getTitle(), t.getDescription(),
                                t.getStatus(), t.getCreatedAt(), t.getUpdatedAt());
    }
}

Mapping approach here is a hand-written static factory — zero magic, compile-time safe, and fine for a handful of DTOs. Once you have dozens, reach for MapStruct (compile-time codegen, covered in the DTO section) rather than a runtime reflection mapper.

4. The repository

One interface. Extending JpaRepository<Task, Long> gives you save, findById, findAll(Pageable), existsById, deleteById, count — all implemented by Spring Data at runtime, no code to write. Add a derived query and Spring Data parses the method name into a SQL WHERE clause:

public interface TaskRepository extends JpaRepository<Task, Long> {

    // method name → SELECT * FROM tasks WHERE status = ?  (paged & sorted by the Pageable)
    Page<Task> findByStatus(TaskStatus status, Pageable pageable);
}

5. The service — business logic & transactions

@Service, constructor injection (no field @Autowired — constructor injection is the senior default: it makes dependencies explicit and the class trivially testable). Writes are @Transactional; reads use @Transactional(readOnly = true). Not-found is signalled by throwing a domain exception, which the global handler (step 9) turns into a 404 — the service never touches HTTP types.

@Service
public class TaskService {

    private final TaskRepository repository;

    public TaskService(TaskRepository repository) {   // constructor injection
        this.repository = repository;
    }

    @Transactional
    public TaskResponse create(TaskRequest req) {
        Task task = new Task();
        task.setTitle(req.title());
        task.setDescription(req.description());
        task.setStatus(req.status());
        return TaskResponse.from(repository.save(task));   // save → INSERT, id populated
    }

    @Transactional(readOnly = true)
    public TaskResponse getById(Long id) {
        Task task = repository.findById(id)
                .orElseThrow(() -> new NotFoundException("Task", id));   // → 404 in the advice
        return TaskResponse.from(task);
    }

    @Transactional(readOnly = true)
    public Page<TaskResponse> list(TaskStatus status, Pageable pageable) {
        Page<Task> page = (status == null)
                ? repository.findAll(pageable)
                : repository.findByStatus(status, pageable);
        return page.map(TaskResponse::from);        // Page.map preserves the paging metadata
    }

    @Transactional
    public TaskResponse update(Long id, TaskRequest req) {
        Task task = repository.findById(id)
                .orElseThrow(() -> new NotFoundException("Task", id));
        task.setTitle(req.title());
        task.setDescription(req.description());
        task.setStatus(req.status());
        return TaskResponse.from(task);   // no save() needed: dirty-checking flushes the UPDATE on commit
    }

    @Transactional
    public void delete(Long id) {
        if (!repository.existsById(id)) throw new NotFoundException("Task", id);
        repository.deleteById(id);
    }
}

The update method has no repository.save() and that's deliberate, not a bug. Inside a transaction, a managed entity is dirty-checked: Hibernate compares its state at commit and auto-issues the UPDATE. Being able to explain that — and that readOnly = true skips dirty checking for a small read speedup — is a reliable senior signal. (Full treatment in the persistence chapter.)

6. The controller — all CRUD endpoints

@RestController + @RequestMapping("/api/tasks"). Each method binds its slice of the request, delegates to the service, and returns the right status code. Note the deliberate mix: fixed-status reads return a bare DTO (200 by default), while create/delete use ResponseEntity because their status/headers are the point (201 + Location, 204 empty).

@RestController
@RequestMapping("/api/tasks")
public class TaskController {

    private final TaskService service;
    public TaskController(TaskService service) { this.service = service; }

    // CREATE → 201 Created + Location header pointing at the new resource
    @PostMapping
    public ResponseEntity<TaskResponse> create(@Valid @RequestBody TaskRequest req,
                                              UriComponentsBuilder uri) {
        TaskResponse created = service.create(req);
        URI location = uri.path("/api/tasks/{id}").buildAndExpand(created.id()).toUri();
        return ResponseEntity.created(location).body(created);   // 201, Location: /api/tasks/42
    }

    // READ one → 200, or 404 when the service throws NotFoundException
    @GetMapping("/{id}")
    public TaskResponse one(@PathVariable Long id) {
        return service.getById(id);                              // bare DTO → 200 OK
    }

    // READ many → paged, with an optional ?status= filter
    @GetMapping
    public PageResponse<TaskResponse> list(
            @RequestParam(required = false) TaskStatus status,
            @PageableDefault(size = 20, sort = "createdAt", direction = Sort.Direction.DESC)
            Pageable pageable) {
        return PageResponse.from(service.list(status, pageable));
    }

    // UPDATE (full replace) → 200 with the updated resource
    @PutMapping("/{id}")
    public TaskResponse update(@PathVariable Long id, @Valid @RequestBody TaskRequest req) {
        return service.update(id, req);
    }

    // DELETE → 204 No Content (empty body)
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        service.delete(id);
        return ResponseEntity.noContent().build();               // 204
    }
}

We used PUT for a full replacement (client sends the complete TaskRequest). For partial updates — e.g. flipping only status — you'd add a @PatchMapping; the PUT-vs-PATCH decision and the "absent vs null" merge-patch trap are covered in the REST design section. For a simple, fully-populated DTO like this one, PUT is the honest choice.

7. Validation — the annotations already do the work

The constraints are already on TaskRequest (@NotBlank, @Size, @NotNull). The one line that activates them is @Valid in front of @RequestBody in the controller. With it, a bad payload never reaches your service: Spring throws MethodArgumentNotValidException, which the advice (step 9) turns into a structured 400. Without @Valid, those annotations are inert decoration — the single most common "why isn't my validation firing" bug.

// POST /api/tasks  with  {"title":"","status":null}
//   @NotBlank on title  → violation
//   @NotNull  on status → violation
// → MethodArgumentNotValidException → 400 (see the error body in step 9)

Forgetting @Valid means invalid data flows straight into your service and DB. The annotations on the DTO don't self-execute — @Valid (bodies) / @Validated (path/query params) is the trigger. The full annotation menu and the two-different-exceptions gotcha live in the Bean Validation section.

8. Pagination & sorting

Spring Data binds ?page=0&size=20&sort=createdAt,desc straight into a Pageable — you never parse query params by hand. @PageableDefault supplies sane defaults (and caps you'd otherwise get from unbounded client input). The repository's findAll(pageable)/findByStatus(status, pageable) returns a Page<Task>; the service .map()s it to Page<TaskResponse>, and we wrap it in a stable envelope rather than serializing Page directly (its JSON shape isn't a guaranteed contract — Boot even logs a warning):

// A stable, self-defined page envelope (reused across endpoints)
public record PageResponse<T>(
        List<T> content,
        int page,
        int size,
        long totalElements,
        int totalPages,
        boolean last) {

    public static <T> PageResponse<T> from(Page<T> p) {
        return new PageResponse<>(p.getContent(), p.getNumber(), p.getSize(),
                                  p.getTotalElements(), p.getTotalPages(), p.isLast());
    }
}

N+1 on the list endpoint. The moment Task gains a lazy relationship (say @ManyToOne Project) and your mapper touches t.getProject().getName(), listing 20 tasks fires 1 query for the page + 20 more for the projects. Fix at the repository with a fetch-join or an @EntityGraph, not by making the relationship eager. This is the single most common performance bug in exactly this kind of list-with-mapping code.

9. Error handling — one ProblemDetail contract

The service throws NotFoundException; @Valid throws MethodArgumentNotValidException. A single @RestControllerAdvice maps both to RFC 7807 ProblemDetail bodies so the whole API speaks one error dialect (the full multi-exception ladder is in the exception-handling section):

// domain exception — plain RuntimeException, thrown from the service
public class NotFoundException extends RuntimeException {
    public NotFoundException(String resource, Object id) {
        super(resource + " " + id + " not found");
    }
}

@RestControllerAdvice
public class ApiExceptionHandler {

    // 404 — resource not found
    @ExceptionHandler(NotFoundException.class)
    public ProblemDetail handleNotFound(NotFoundException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Resource not found");
        pd.setProperty("timestamp", Instant.now());
        return pd;   // status is taken from the ProblemDetail
    }

    // 400 — @Valid failure → per-field errors the frontend can map onto inputs
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, "Validation failed");
        pd.setTitle("Invalid request");
        List<Map<String, String>> errors = ex.getBindingResult().getFieldErrors().stream()
                .map(fe -> Map.of("field", fe.getField(),
                                  "message", fe.getDefaultMessage() == null ? "invalid" : fe.getDefaultMessage()))
                .toList();
        pd.setProperty("errors", errors);
        return pd;
    }
}

10. Calling an external API (RestClient)

Real services also consume other APIs — this is the server's fetch(). Suppose creating a task should notify an external webhook service. Use Spring 6's RestClient (the recommended blocking client; WebClient is its reactive/async sibling for high-concurrency fan-out — see the clients section):

@Service
public class NotificationClient {

    private final RestClient http;

    public NotificationClient(RestClient.Builder builder) {   // Boot pre-configures the builder
        this.http = builder.baseUrl("https://hooks.internal").build();
    }

    // deserialize the response straight into a record
    public record Ack(String id, String status) {}

    public Ack notifyCreated(TaskResponse task) {
        return http.post()
                .uri("/v1/notify")
                .contentType(MediaType.APPLICATION_JSON)
                .body(Map.of("event", "task.created", "taskId", task.id()))
                .retrieve()
                .onStatus(HttpStatusCode::is4xxClientError, (req, res) -> {   // basic error handling
                    throw new IllegalStateException("notify rejected: " + res.getStatusCode());
                })
                .body(Ack.class);      // blocking; returns the deserialized DTO
    }
}

Always set connect/read timeouts on outbound clients (the defaults can hang a request thread indefinitely) and don't let a flaky downstream fail the whole create — for a side-effect like a notification, log-and-continue or push it to an async queue rather than blocking the user's 201.

Quick Q&A on this build

Why return 201 with a Location header instead of just 200?

201 Created is the semantically correct status for a successful POST that creates a resource, and the Location header tells the client the canonical URL of the thing it just made (/api/tasks/42) so it can immediately GET or link to it without guessing the id. 200 works mechanically but throws away that information and misrepresents "created" as generic "OK". Spring makes it one call: ResponseEntity.created(location).body(dto). Interviewers specifically check for the Location header — it's a small tell that you build APIs, not just endpoints.

You used PUT for update. When would you reach for PATCH here instead?

PUT is a full replacement: the client sends the complete TaskRequest and the server overwrites all mutable fields — which is exactly what our DTO expresses (title + description + status, all present). Reach for PATCH when clients need to change one field without resending the rest — e.g. a Kanban board flipping only status to DONE. The catch with PATCH is distinguishing "field omitted" from "field explicitly set to null"; a plain DTO can't, so you read the body as a JsonNode/Map and check key presence. For this resource, PUT is simpler and honest; add PATCH only when a real partial-update use case appears.

Why map the entity to a DTO at all — the fields are nearly identical?

Because "nearly identical today" is a trap. The moment the entity grows a field you don't want exposed (an internal ownerId, a soft-delete flag) or a lazy relationship, returning the entity leaks it or triggers serialization bombs (N+1, LazyInitializationException, infinite recursion on bidirectional links). And on input, binding @RequestBody to the entity invites over-posting — a client setting id or createdAt you meant to own server-side. The DTO is a deliberate, stable contract decoupled from the schema, so the two can evolve independently. The tiny mapping cost buys you that boundary; the DTO section lays out all four reasons.

Jackson: how JSON (de)serialization actually happens T1

When your React app did await res.json(), someone on the server had already turned a Java object into those bytes. That someone is Jackson, invoked by Spring through the HttpMessageConverter abstraction.

HttpMessageConverter — the actual mechanism

Spring MVC keeps a list of converters. For each request/response, it picks one by matching the Content-Type (reading) or Accept header (writing) against each converter's supported media types, plus the Java type involved:

  • MappingJackson2HttpMessageConverter — application/json ↔ objects (the one you care about; auto-registered because Jackson is on the classpath via spring-boot-starter-web).
  • StringHttpMessageConverter — text/plain ↔ String.
  • ByteArrayHttpMessageConverter, ResourceHttpMessageConverter — binary payloads, file downloads.

Flow for a POST: body bytes → converter selected by Content-Type: application/json → ObjectMapper.readValue() → your @RequestBody DTO. Flow for the response: your return value → converter selected by content negotiation → ObjectMapper.writeValue() → bytes on the wire.

Classic gotcha: return a plain String from a @RestController method and it goes through StringHttpMessageConverter as text/plain — not JSON-quoted. The frontend's res.json() then throws. Return a DTO (or a Map), or set produces = MediaType.APPLICATION_JSON_VALUE and return an object.

Jackson defaults and the annotations you'll actually use

Jackson serializes public getters and deserializes via a no-arg constructor + setters, or via a constructor whose parameter names it can see (records work out of the box — see the DTO section). The annotations that come up in interviews:

public class UserResponse {

    @JsonProperty("user_name")          // rename: Java field ↔ JSON key
    private String userName;

    @JsonIgnore                          // never serialized, never bound from input
    private String passwordHash;

    @JsonInclude(JsonInclude.Include.NON_NULL)   // omit key entirely when null
    private String middleName;

    @JsonFormat(pattern = "yyyy-MM-dd")  // override the default ISO output
    private LocalDate birthDate;

    private Instant createdAt;           // → "2026-07-02T09:15:30Z" (ISO-8601)
}

Raw Jackson would serialize Instant/LocalDate as ugly numeric timestamps or fail entirely. That is exactly what new Date(iso) and every JS date library expect. Global tweaks go in properties, not code:

spring:
  jackson:
    default-property-inclusion: non_null      # drop nulls everywhere
    deserialization:
      fail-on-unknown-properties: false        # default; set true for strict APIs
    property-naming-strategy: SNAKE_CASE       # if your API contract is snake_case

Senior take on fail-on-unknown-properties: leaving it lenient (Boot's default) makes clients forward-compatible — the server can't be broken by an extra key. But for security-sensitive write endpoints, strict rejection surfaces client bugs early (a typoed field silently ignored is a data-loss bug the frontend team will hunt for days). Many teams keep it lenient globally and rely on Bean Validation to catch missing required fields, which is the failure mode that actually matters.

The DTO pattern: never expose entities T1

This is arguably the single most-asked API-design question in Java interviews: "Why not return your JPA entity from the controller?" You must have four reasons ready, not one.

The four reasons

  1. Coupling / contract stability. Your entity mirrors the database schema. If it's also your JSON contract, every column rename becomes a breaking API change, and every API-shape request from the frontend team pressures your schema. DTOs decouple the wire format from persistence so each can evolve independently — the same reason you don't let a backend response shape dictate your React component props without a mapping layer.
  2. Lazy-loading serialization bombs. Entities have lazy relationships (@ManyToOne(fetch = LAZY), collections). When Jackson walks the object graph it touches those getters and either (a) triggers N+1 queries mid-serialization, (b) throws LazyInitializationException because the session closed, or (c) recurses infinitely on a bidirectional relationship (User → orders → user → orders…) until StackOverflowError. People "fix" this with @JsonIgnore/@JsonManagedReference sprinkled on entities — which is just admitting the entity is doing two jobs.
  3. Over-posting / mass assignment. If @RequestBody User user binds directly to the entity, a malicious client can send {"role":"ADMIN","balance":999999} and Jackson will happily populate those fields. A request DTO that simply does not have a role field is structurally immune. (Same class of bug as Rails' mass-assignment CVEs; interviewers love this one.)
  4. Versioning & shaping. Different endpoints need different projections (list view vs detail view), computed fields, aggregated data from multiple entities, or v1/v2 shapes. Entities can't be all of these at once.

Records as DTOs — the modern default

Java records (16+) are the idiomatic DTO since Boot 3: immutable, concise, value-based equality, and Jackson deserializes them via the canonical constructor with zero annotations:

// Request DTO — only fields the client is ALLOWED to send
public record CreateUserRequest(
        @NotBlank @Size(max = 100) String name,
        @NotBlank @Email String email,
        @Size(min = 12) String password        // note: no 'role', no 'id' — immune to over-posting
) {}

// Response DTO — only fields the client SHOULD see (no passwordHash!)
public record UserResponse(
        Long id,
        String name,
        String email,
        Instant createdAt
) {
    // static factory keeps mapping next to the shape it produces
    public static UserResponse from(User entity) {
        return new UserResponse(entity.getId(), entity.getName(),
                                entity.getEmail(), entity.getCreatedAt());
    }
}

Request DTO vs response DTO — keep them separate. They differ almost always: the request has password but no id; the response has id/createdAt but never password. One shared "UserDto" inevitably grows nullable fields with comments like "only set on create" — the Java equivalent of a TypeScript type where half the fields are | undefined depending on context. Two small records beat one ambiguous one.

Mapping options

1. Manual mapping (static factory or a small mapper class) — zero magic, debuggable, compile-time safe, trivially fast. For most services this is the right answer and saying so in an interview reads as experienced, not lazy.

2. MapStruct — an annotation processor that generates the mapping code at compile time. You write an interface; MapStruct writes the implementation as plain Java:

@Mapper(componentModel = "spring")      // generated impl is a Spring bean → inject it
public interface UserMapper {

    @Mapping(target = "fullName", source = "name")   // rename
    @Mapping(target = "passwordHash", ignore = true)
    UserResponse toResponse(User entity);

    List<UserResponse> toResponses(List<User> entities);  // list version generated free

    @Mapping(target = "id", ignore = true)
    @Mapping(target = "createdAt", expression = "java(java.time.Instant.now())")
    User toEntity(CreateUserRequest request);
}

// Generated at build time — real code you can step through:
// public class UserMapperImpl implements UserMapper {
//     public UserResponse toResponse(User entity) {
//         if (entity == null) return null;
//         return new UserResponse(entity.getId(), entity.getName(), ...);
//     } ... }

3. Reflection-based mappers (ModelMapper, Dozer, Orika) — avoid. They resolve field matches via reflection at runtime, per call. Costs: measurable overhead in hot paths, silent breakage on rename (no compile error, fields just come back null), and opaque debugging. MapStruct gives the same convenience with compile-time errors and hand-written-speed code because it generates plain getters/setters calls. The interview phrasing: "MapStruct is codegen, ModelMapper is runtime reflection — I want mapping mistakes to be compile failures, not production nulls."

Global exception handling & ProblemDetail T1

As a frontend dev you've felt the pain of APIs where 404 returns HTML, validation errors return three different shapes, and 500s leak stack traces. The fix on the backend is one centralized error boundary: @RestControllerAdvice.

The machinery

  • @ExceptionHandler(SomeException.class) — a method that handles that exception (and subclasses) thrown by handlers. Inside a controller it's local to that controller; that doesn't scale.
  • @ControllerAdvice / @RestControllerAdvice — a component whose @ExceptionHandler methods apply globally (scopeable by package/annotation). @RestControllerAdvice = @ControllerAdvice + @ResponseBody, same pattern as before.
  • Resolution order: Spring picks the handler with the most specific exception type — a handler for OrderNotFoundException beats one for RuntimeException beats one for Exception. This lets you write a precise ladder with a catch-all at the bottom.

ProblemDetail — RFC 7807 / RFC 9457

Instead of every team inventing its own {error: ..., msg: ...} envelope, RFC 7807 "Problem Details for HTTP APIs" (revised as RFC 9457) standardizes one: content type application/problem+json with fields type (URI identifying the error category), title, status, detail, instance — plus arbitrary extension members. Spring 6 / Boot 3 ships first-class support: the ProblemDetail class and ErrorResponseException.

A complete, production-shaped error setup

// 1. A domain exception — thrown from the service layer
public class NotFoundException extends RuntimeException {
    public NotFoundException(String resource, Object id) {
        super(resource + " " + id + " not found");
    }
}

// 2. The single global error boundary
@RestControllerAdvice
public class GlobalExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    // --- 404: domain "not found" ---
    @ExceptionHandler(NotFoundException.class)
    public ProblemDetail handleNotFound(NotFoundException ex, HttpServletRequest req) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Resource not found");
        pd.setType(URI.create("https://api.example.com/errors/not-found"));
        pd.setProperty("timestamp", Instant.now());          // extension member
        return pd;                                            // status taken from the ProblemDetail
    }

    // --- 400: @Valid failure on a @RequestBody → structured field errors ---
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
                HttpStatus.BAD_REQUEST, "Validation failed");
        pd.setTitle("Invalid request");
        pd.setType(URI.create("https://api.example.com/errors/validation"));

        List<Map<String, String>> errors = ex.getBindingResult().getFieldErrors().stream()
                .map(fe -> Map.of(
                        "field", fe.getField(),
                        "message", fe.getDefaultMessage() == null ? "invalid" : fe.getDefaultMessage(),
                        "rejectedValue", String.valueOf(fe.getRejectedValue())))
                .toList();
        pd.setProperty("errors", errors);   // frontend can map these onto form fields
        return pd;
    }

    // --- 400: unreadable/malformed JSON body ---
    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ProblemDetail handleUnreadable(HttpMessageNotReadableException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
                HttpStatus.BAD_REQUEST, "Malformed request body");
        pd.setTitle("Invalid request");
        return pd;   // deliberately do NOT echo ex.getMessage() — leaks class names
    }

    // --- 409: optimistic-lock / duplicate conflicts ---
    @ExceptionHandler(DataIntegrityViolationException.class)
    public ProblemDetail handleConflict(DataIntegrityViolationException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
                HttpStatus.CONFLICT, "Request conflicts with existing data");
        pd.setTitle("Conflict");
        return pd;
    }

    // --- 500: the catch-all. NEVER leak internals. ---
    @ExceptionHandler(Exception.class)
    public ProblemDetail handleUnexpected(Exception ex, HttpServletRequest req) {
        String correlationId = MDC.get("correlationId");
        // log the FULL exception server-side with the correlation id…
        log.error("Unhandled exception [{}] on {} {}", correlationId,
                  req.getMethod(), req.getRequestURI(), ex);

        // …but return only a generic body to the client
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
                HttpStatus.INTERNAL_SERVER_ERROR, "An unexpected error occurred");
        pd.setTitle("Internal error");
        pd.setProperty("correlationId", correlationId);   // user can quote it to support
        return pd;
    }
}

Never leak stack traces. A raw stack trace in a 500 response reveals framework versions, internal class names, file paths, sometimes SQL. Log the full exception with a correlation ID server-side; send the client a generic message plus that ID. Also check server.error.include-stacktrace=never (Boot's default since 2.3) so the fallback /error endpoint doesn't leak either.

The senior framing: "The error contract is API surface, exactly like the success contract." Your React self can vouch for this — one consistent ProblemDetail envelope means the frontend writes one error interceptor (an axios interceptor / React Query onError) that maps status + errors[] to toasts and form-field messages, instead of per-endpoint special cases. Also worth mentioning: extending ResponseEntityExceptionHandler gives you Spring's ~15 built-in framework exceptions (unsupported media type, method not allowed…) pre-mapped to ProblemDetail, and you override only what you customize.

public record CreateOrderRequest(
        @NotBlank(message = "customer name is required")
        @Size(max = 200)
        String customerName,

) {}

Most constraints (@Email, @Pattern, @Size, @Past…) treat null as valid — by design, so optionality is expressed separately. If a field is required and shaped, you need both: @NotNull @Email. Forgetting this ships "optional but validated" fields you meant to be mandatory. Also: nested objects need an explicit @Valid on the field or validation does not cascade into them.

@Validated on the class — validating params, not bodies

@Valid works on @RequestBody. To validate simple params (@PathVariable, @RequestParam), put @Validated on the controller class — this enables method-level validation via an AOP proxy:

@RestController
@Validated                              // enables constraint checks on method params
@RequestMapping("/api/products")
public class ProductController {

    @GetMapping
    public Page<ProductResponse> list(
            @RequestParam @Min(0) int page,
            @RequestParam @Min(1) @Max(100) int size) {   // cap page size!
        ...
    }
}
// violation → ConstraintViolationException (NOT MethodArgumentNotValidException)
// → returns 500 by default! Map it to 400 in your advice:
@ExceptionHandler(ConstraintViolationException.class)
public ProblemDetail handleParamViolation(ConstraintViolationException ex) {
    ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, "Invalid parameters");
    pd.setProperty("errors", ex.getConstraintViolations().stream()
            .map(v -> Map.of("param", v.getPropertyPath().toString(),
                             "message", v.getMessage())).toList());
    return pd;
}

Interview trap worth pre-empting: body validation fails with MethodArgumentNotValidException (400 by default), param validation fails with ConstraintViolationException (500 unless you map it). Knowing they are two different exceptions with two different default statuses is a reliable senior signal. (Spring 6.1's built-in method validation narrows this gap, but handle both explicitly and you're safe everywhere.)

Custom validator — full ConstraintValidator example

// 1. The annotation
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = SkuFormatValidator.class)
public @interface ValidSku {
    String message() default "invalid SKU format";
    Class<?>[] groups() default {};                       // boilerplate the spec requires
    Class<? extends Payload>[] payload() default {};
}

// 2. The validator — can be a Spring bean, so you CAN inject dependencies
public class SkuFormatValidator implements ConstraintValidator<ValidSku, String> {

    private static final Pattern SKU = Pattern.compile("^[A-Z]{3}-\\d{4}-[A-Z0-9]{2}$");

    @Override
    public boolean isValid(String value, ConstraintValidatorContext ctx) {
        if (value == null) return true;      // convention: null-validity handled by @NotNull
        return SKU.matcher(value).matches();
    }
}

// 3. Usage
public record OrderLine(@NotNull @ValidSku String sku, @Positive int quantity) {}

Validation groups — awareness level

Groups let one DTO carry different rule sets per operation: @NotNull(groups = OnCreate.class) on password, then @Validated(OnCreate.class) at the binding site. Know the mechanism, and know the senior counter-position: in practice, separate CreateXRequest / UpdateXRequest records are clearer than one DTO with mode-dependent rules — groups are best left for legacy shared-DTO codebases.

REST design at senior level T1

Framework syntax is table stakes; this section is what senior interviews actually grade. You've consumed hundreds of these decisions — now you must be able to defend them.

Resource naming conventions

  • Nouns, plural, lowercase, hyphenated: /api/orders, /api/orders/42/line-items. Not /getOrders, not /api/Order.
  • Hierarchy for ownership: /customers/7/orders when orders are meaningfully scoped to a customer; but don't nest past two levels — /orders/42 beats /customers/7/orders/42 once the order has its own identity.
  • Verbs are the HTTP method, not the path. Exception: genuinely non-CRUD operations get a sub-resource action — POST /orders/42/cancellation or the pragmatic POST /orders/42/cancel. Purists model it as a state resource; pragmatists ship the verb. Be able to say both.
  • Query params for cross-cutting concerns: filtering, sorting, pagination — never for identity.

HTTP verbs, idempotency & safety

Idempotent = performing the request N times has the same effect as once (not: same response body). Safe = no state change at all. This matters concretely: proxies, clients and retry libraries will automatically retry idempotent methods after a timeout.

PUTNoYesFull replace: applying the same full state twice = same result
DELETENoYesDeleting twice leaves the same end state (2nd may 404 — still idempotent in effect)
POSTNoNoTwo POSTs = two resources / two charges. Mitigate with an Idempotency-Key header pattern

Money quote for payment-adjacent interviews: "POST is not idempotent, which is why payment APIs (Stripe et al.) require an Idempotency-Key header — the server stores the key with the first response and replays that response for retries, converting POST into an effectively idempotent operation. That's a design decision the backend must make; the client can't retrofit it."

Pagination conventions

Offset pagination is the default; expose page/size/sort and return an envelope with metadata — precisely what your React tables always wanted:

// GET /api/orders?page=1&size=20&sort=createdAt,desc — Spring Data binds this automatically
@GetMapping
public PageResponse<OrderResponse> list(@ParameterObject Pageable pageable) {
    Page<Order> page = repository.findAll(pageable);
    return PageResponse.from(page.map(OrderResponse::from));
}

// Stable envelope (don't serialize Page<> directly — its JSON shape is not a guaranteed contract;
// Boot even logs a warning suggesting PagedModel/a DTO)
public record PageResponse<T>(List<T> content, int page, int size,
                              long totalElements, int totalPages) {
    public static <T> PageResponse<T> from(Page<T> p) {
        return new PageResponse<>(p.getContent(), p.getNumber(), p.getSize(),
                                  p.getTotalElements(), p.getTotalPages());
    }
}

Know the alternative: cursor (keyset) pagination — ?after=eyJpZCI6NDJ9&size=20 — for infinite scroll and large/fast-moving datasets. Offset pagination does OFFSET 100000 scans and skips/duplicates rows when data shifts between pages; keyset (WHERE (created_at, id) < (?, ?) ORDER BY created_at DESC, id DESC LIMIT 20) is O(log n) and stable. Trade-off: no "jump to page 47". Filtering/sorting: whitelist sortable fields (sort=createdAt,desc) — never pass raw client input into an ORDER BY.

Partial update: PUT vs PATCH

  • PUT = full replacement: client sends the complete resource; omitted fields are semantically nulled. Idempotent by construction.
  • PATCH = partial: client sends only changes. Two standard body formats:
    • JSON Merge Patch (RFC 7386, application/merge-patch+json): send the fields to change; null means "delete this field". Simple, but you cannot distinguish "field absent" from "set to null" with a plain DTO — the classic PATCH bug.
    • JSON Patch (RFC 6902, application/json-patch+json): an array of operations — [{"op":"replace","path":"/status","value":"SHIPPED"}]. Precise (including test/copy/move ops) but heavyweight; rare in typical CRUD services.
// Merge-patch done correctly: read as JsonNode / Map so "absent" ≠ "null"
@PatchMapping(value = "/{id}", consumes = "application/merge-patch+json")
public OrderResponse patch(@PathVariable long id, @RequestBody JsonNode patch) {
    Order order = repository.findById(id).orElseThrow(() -> new NotFoundException("Order", id));
    if (patch.has("status")) {                 // key present at all?
        JsonNode v = patch.get("status");
        if (v.isNull()) throw new UnprocessableException("status cannot be cleared");
        order.setStatus(OrderStatus.valueOf(v.asText()));
    }
    if (patch.has("note")) {
        order.setNote(patch.get("note").isNull() ? null : patch.get("note").asText());
    }
    return OrderResponse.from(repository.save(order));
}

ResponseEntity & content negotiation T2

ResponseEntity — full control over status, headers, body

Returning a bare DTO gives you 200 with default headers. ResponseEntity<T> is the fluent builder for everything else:

@PostMapping
public ResponseEntity<OrderResponse> create(@Valid @RequestBody CreateOrderRequest req,
                                            UriComponentsBuilder uriBuilder) {
    OrderResponse created = orderService.create(req);
    URI location = uriBuilder.path("/api/orders/{id}").buildAndExpand(created.id()).toUri();
    return ResponseEntity.created(location)      // 201 + Location header — do this, interviewers check
                         .body(created);
}

@GetMapping("/{id}")
public ResponseEntity<OrderResponse> one(@PathVariable long id) {
    return orderService.maybeFind(id)             // Optional<OrderResponse>
            .map(order -> ResponseEntity.ok()
                    .eTag("\"" + order.version() + "\"")          // conditional-request support
                    .cacheControl(CacheControl.maxAge(Duration.ofSeconds(60)))
                    .body(order))
            .orElseGet(() -> ResponseEntity.notFound().build());   // 404, no body
}

@DeleteMapping("/{id}")
public ResponseEntity<Void> delete(@PathVariable long id) {
    orderService.delete(id);
    return ResponseEntity.noContent().build();    // 204
}

Style guidance: prefer the plain-DTO return + @ResponseStatus where the status is fixed, and reserve ResponseEntity for genuinely dynamic status/headers. Mixing exceptions-for-errors with ResponseEntity-for-success keeps controllers thin — the advice handles the sad paths.

Content negotiation: Accept, produces, consumes

  • consumes restricts which request Content-Type the mapping accepts. Mismatch → 415 Unsupported Media Type.
  • produces restricts/declares what the handler emits, matched against the client's Accept header. No acceptable match → 406 Not Acceptable.
  • Two handlers can share a path and differ only in consumes — e.g. application/json vs application/merge-patch+json on the same PATCH URL.
// Same URL, two representations — negotiated by Accept header
@GetMapping(value = "/{id}/invoice", produces = MediaType.APPLICATION_JSON_VALUE)
public InvoiceResponse invoiceJson(@PathVariable long id) { ... }

@GetMapping(value = "/{id}/invoice", produces = MediaType.APPLICATION_PDF_VALUE)
public ResponseEntity<Resource> invoicePdf(@PathVariable long id) { ... }

From the fetch() side you set Accept without thinking (axios defaults to application/json, text/plain, */*). Now you're the one reading it: Spring's ContentNegotiationManager resolves the requested media type (header-based by default in Boot) and selects both the handler (via produces) and the message converter with it.

The request pipeline: Filter → DispatcherServlet → Interceptor T2

Interviewers ask "what happens when a request hits your Spring Boot app?" to check you see beyond the controller. The order of the layers:

TCP → Tomcat (connector, worker thread)
  → Servlet Filter chain            ← Servlet spec, pre-Spring-MVC (but filters can be Spring beans)
      (CorsFilter, Spring Security's filter chain, your correlation-ID filter…)
  → DispatcherServlet               ← Spring MVC's front controller: one servlet routes everything
      → HandlerMapping                finds the @RequestMapping method for the URL
      → HandlerInterceptor.preHandle()
      → argument resolvers            @PathVariable/@RequestBody binding, validation
      → your controller method
      → return-value handling         HttpMessageConverter serializes
      → HandlerInterceptor.postHandle()       (before the view/body finalized)
      → HandlerInterceptor.afterCompletion()  (always runs, even on exception — cleanup)
  ← back out through the filter chain (response passes through in reverse)

Filter (jakarta.servlet.Filter): container-level, sees raw ServletRequest/Response, runs for every request including ones Spring never routes (static resources, 404s). Can short-circuit (never call chain.doFilter), wrap/mutate the request/response streams. Interceptor (HandlerInterceptor): Spring MVC-level, runs after handler resolution so it knows which controller method will execute; three hooks (preHandle can veto by returning false; afterCompletion is your finally-block). Exception subtlety: an exception thrown in a filter never reaches your @ControllerAdvice — advice only covers the DispatcherServlet's world. That's why Spring Security errors (401/403 from the filter chain) need their own AuthenticationEntryPoint/AccessDeniedHandler, not an @ExceptionHandler.

Correlation-ID filter — the canonical example

@Component
@Order(Ordered.HIGHEST_PRECEDENCE)          // run first so EVERYTHING downstream is tagged
public class CorrelationIdFilter extends OncePerRequestFilter {
                                            // OncePerRequestFilter: guaranteed single execution per request,
                                            // even with forwards/error dispatches
    public static final String HEADER = "X-Correlation-Id";

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        // honor an inbound id (gateway / calling service), else mint one
        String id = request.getHeader(HEADER);
        if (id == null || id.isBlank()) id = UUID.randomUUID().toString();

        MDC.put("correlationId", id);       // every log line on this thread now carries it
        response.setHeader(HEADER, id);      // echo it so the frontend can log/report it too
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove("correlationId");    // CRITICAL: threads are pooled and reused —
        }                                    // leaked MDC values bleed into other users' requests
    }
}

This one filter closes the debugging loop you know from the frontend: user reports an error toast showing a correlation ID → grep the ID in centralized logs → the exact request's full trail, including the 500's stack trace your ProblemDetail deliberately hid. Mention the MDC-cleanup-in-finally detail; it demonstrates you understand servlet thread pooling.

CORS — from the backend side this time T2

You have lost hours to "blocked by CORS policy". Time to internalize the truth you half-knew: CORS errors are almost always backend configuration issues. The browser enforces; the server declares the policy; the frontend can do nothing but proxy around it in dev.

Preflight mechanics, 60-second recap

  1. Simple requests (GET/HEAD/POST with form-ish content types, no custom headers) are sent directly; the browser checks Access-Control-Allow-Origin on the response and blocks your JS from reading it on mismatch. The request did reach the server — CORS is read-protection, not a firewall.
  2. Everything a JSON API does is non-simple: Content-Type: application/json, an Authorization header, PUT/DELETE — each alone triggers a preflight: an OPTIONS request with Origin, Access-Control-Request-Method, Access-Control-Request-Headers. The server must answer with matching Access-Control-Allow-* headers or the browser never sends the real request.
  3. Access-Control-Max-Age lets the browser cache the preflight verdict and skip repeats.

@CrossOrigin vs global configuration

@CrossOrigin(origins = "https://app.example.com") on a controller/method works but scatters policy across the codebase and is easy to forget on new controllers. Production answer: one global config —

@Configuration
public class CorsConfig {

    @Bean
    public WebMvcConfigurer corsConfigurer(
            @Value("${app.cors.allowed-origins}") String[] allowedOrigins) {  // per-env property!
        return new WebMvcConfigurer() {
            @Override
            public void addCorsMappings(CorsRegistry registry) {
                registry.addMapping("/api/**")
                        .allowedOrigins(allowedOrigins)          // e.g. https://app.example.com
                        .allowedMethods("GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS")
                        .allowedHeaders("Authorization", "Content-Type", "X-Correlation-Id")
                        .exposedHeaders("X-Correlation-Id")     // headers JS may READ from responses
                        .allowCredentials(true)                  // only if using cookies
                        .maxAge(3600);                            // cache preflight for 1h
            }
        };
    }
}

Credentials + wildcard = broken by spec. With allowCredentials(true), the browser rejects Access-Control-Allow-Origin: * — the server must echo a specific origin (and should send Vary: Origin). Spring throws at startup if you combine allowedOrigins("*") with credentials; the escape hatch is allowedOriginPatterns("https://*.example.com"), which echoes the matching origin back. Also remember exposedHeaders: by default JS can read only a handful of "safelisted" response headers — your custom X-Correlation-Id is invisible to response.headers.get() until you expose it. You've probably hit that one from the other side.

With Spring Security in play, configure CORS through Security too (http.cors(...) with a CorsConfigurationSource bean), because the security filter chain runs before MVC and will 401 the anonymous preflight OPTIONS otherwise. "CORS works until we add Spring Security" is a rite of passage — and a great interview anecdote about filter ordering.

Interview Q&A

What's the difference between @Controller and @RestController?

@RestController = @Controller + @ResponseBody as a meta-annotation. With plain @Controller, return values are treated as view names and go through ViewResolver (server-rendered templates). @ResponseBody — per method or inherited from @RestController — routes the return value through the HttpMessageConverter chain instead, serializing it (typically Jackson → JSON) straight into the response body. For a JSON API, use @RestController; the mixed form (@Controller + per-method @ResponseBody) exists for apps serving both HTML and JSON.

How does a Java object become JSON on the wire? Walk through the mechanism.

The controller returns an object; because of @ResponseBody, Spring's return-value handler asks the registered HttpMessageConverters which can write that type for the media type negotiated from the Accept header. MappingJackson2HttpMessageConverter matches for application/json and calls Jackson's ObjectMapper.writeValue(), streaming JSON to the response output. Inbound is symmetric: Content-Type selects the converter, which calls readValue() to build the @RequestBody argument; failures surface as HttpMessageNotReadableException → 400. Boot auto-configures the ObjectMapper (JavaTimeModule, ISO-8601 dates) and honors spring.jackson.* properties.

@RequestParam vs @PathVariable — when do you use which?

@PathVariable binds a URI template segment and should carry identity — which resource: /orders/{id}. @RequestParam binds the query string and should carry modifiers — filtering, sorting, pagination: ?status=SHIPPED&page=2. Behavioral differences: request params support required=false and defaultValue; a missing path variable is usually just a different route (404) rather than a binding error. Rule: if removing it changes which resource you mean, it's a path variable; if it changes how you view a collection, it's a query param.

Why should you never return JPA entities from controllers?

Four reasons. (1) Contract coupling — the entity mirrors the DB schema, so schema changes become breaking API changes. (2) Lazy-loading hazards — Jackson walking lazy relationships triggers LazyInitializationException or N+1 queries, and bidirectional relationships cause infinite recursion. (3) Over-posting / mass assignment — binding a request straight to an entity lets clients set fields like role or id; a request DTO that lacks those fields is structurally immune. (4) Shaping/versioning — different endpoints need different projections and computed fields. Fix: separate request/response DTOs (records), mapped manually or with MapStruct.

What is mass assignment / over-posting and how do DTOs prevent it?

Mass assignment is when a data binder populates any field the client sends. If @RequestBody User binds to the entity, {"name":"x","role":"ADMIN"} silently escalates privileges because Jackson sets every matching property. Defenses: a request DTO containing only client-settable fields (best — the dangerous field can't be expressed), @JsonIgnore on sensitive setters (fragile), or strict allow-lists in the binder. The DTO approach also documents the contract explicitly, which is why it's the standard answer.

@NotNull vs @NotEmpty vs @NotBlank?

@NotNull: only rejects null — "" passes; works on any type. @NotEmpty: rejects null and empty — for String, Collection, Map, arrays; "  " (whitespace) still passes. @NotBlank: String-only; rejects null, "", and whitespace-only. So for a user's name field you want @NotBlank. Bonus point: most other constraints (@Email, @Size, @Pattern) treat null as valid, so "required and well-formed" needs a null-check annotation plus the format annotation.

How do you implement consistent error responses across an API?

One @RestControllerAdvice as the global boundary, returning RFC 7807/9457 ProblemDetail (built into Spring 6) for every failure: specific handlers for domain exceptions (NotFoundException → 404), MethodArgumentNotValidException → 400 with an errors[] extension listing field/message pairs, ConstraintViolationException → 400, conflict-type exceptions → 409, and a catch-all Exception handler → generic 500 that logs the full stack trace with a correlation ID but never returns it. Spring resolves the most specific matching handler. The payoff is the frontend writes exactly one error interceptor. For the framework's own ~15 exceptions, extend ResponseEntityExceptionHandler.

What is ProblemDetail / RFC 7807?

A standardized error body for HTTP APIs (RFC 7807, revised as RFC 9457): media type application/problem+json with members type (URI categorizing the error), title, status, detail, instance, plus arbitrary extension members (e.g. errors, correlationId). Spring 6 provides the ProblemDetail class; returning it from an @ExceptionHandler sets the status and content type automatically. Value: clients and gateways can handle errors generically instead of learning each API's bespoke envelope.

A validation error on a @RequestBody vs on a @RequestParam — what's different?

Different exceptions and different defaults. @Valid @RequestBody failure throws MethodArgumentNotValidException, which Spring maps to 400 out of the box and which carries a BindingResult with structured field errors. Parameter constraints (enabled by @Validated on the class) throw ConstraintValidationException… precisely ConstraintViolationException, which is not mapped by default and surfaces as a 500 unless your advice maps it to 400. Knowing this asymmetry (and that Spring 6.1's method-validation integration improves it) is the senior detail.

Which HTTP methods are idempotent, and why does it matter operationally?

GET/HEAD/OPTIONS (also safe), PUT and DELETE are idempotent; POST is not; PATCH depends on the patch semantics (absolute set = yes, relative op like increment = no). It matters because intermediaries and client libraries will retry idempotent requests after timeouts without fear of double effects — retrying a POST can double-charge. That's why POST-based operations that must be retry-safe adopt an Idempotency-Key header: the server stores the first outcome per key and replays it for duplicates, making the operation effectively idempotent.

401 vs 403 vs 404 — when do you return each?

401 Unauthorized: not authenticated — credentials missing/invalid/expired; the client should (re)authenticate. 403 Forbidden: authenticated, but this principal lacks permission; re-authenticating as the same user won't help. 404 Not Found: resource doesn't exist — and deliberately also used instead of 403 when you don't want to reveal that a resource exists (GitHub returns 404 for private repos you can't see). Note 401/403 typically originate in the security filter chain, before MVC, so they're handled by security's entry point / access-denied handler, not @ControllerAdvice.

PUT vs PATCH — and what is JSON Merge Patch?

PUT replaces the full resource: the client sends the complete representation, omitted fields are removed/nulled, and it's idempotent by construction. PATCH sends only changes. JSON Merge Patch (RFC 7386, application/merge-patch+json) is the common PATCH format: keys present are set, null means delete the field, absent keys are untouched. Implementation gotcha: a plain DTO can't distinguish "absent" from "explicit null" — both deserialize to null — so you read the patch as JsonNode/Map (checking has() vs isNull()) or use a patching library. RFC 6902 JSON Patch (operation array) is the more precise, less-used alternative.

How do you design pagination for a large collection endpoint?

Default: offset pagination with page/size/sort (Spring Data binds a Pageable automatically), a hard cap on size, a whitelist of sortable fields, and a stable response envelope (content + page/size/totalElements/totalPages) instead of serializing Page directly, since its JSON shape isn't a guaranteed contract. For deep or fast-moving datasets, switch to keyset/cursor pagination (?after=<cursor>): the DB seeks on an indexed (sortKey, id) tuple instead of scanning an OFFSET, and results don't skip/duplicate when rows are inserted between pages — the right fit for infinite scroll. Trade-off: no random page access and no cheap total count.

Compare API versioning via URI, header, and media type.

URI (/api/v1/…): explicit, visible in logs/browsers, trivial gateway routing and caching — but the same resource has multiple URIs and migrations touch every client URL. Header (X-API-Version): clean URIs, but invisible in casual debugging, needs Vary for caches, harder to demo. Media type (Accept: application/vnd.acme.v2+json): the most "RESTful" — version is a representation concern — but clunky for consumers and rare outside APIs like GitHub's. Pragmatically most teams choose URI versioning for operability, and — more importantly — version rarely: additive optional fields plus tolerant readers avoid most version bumps.

Filter vs HandlerInterceptor — differences and when to choose each?

A servlet Filter is a container-spec construct running before the DispatcherServlet: it sees every request (including non-MVC ones), works on raw request/response, can wrap streams and short-circuit — right level for correlation IDs, logging, compression, and where Spring Security lives. A HandlerInterceptor runs inside Spring MVC after handler resolution, so it knows exactly which controller method will run (can read its annotations) and offers preHandle/postHandle/afterCompletion — right level for per-endpoint policies like annotation-driven rate limiting. Key consequence: exceptions thrown in filters never reach @ControllerAdvice; MVC exception handling only covers the dispatcher's world.

What happens between Tomcat accepting a connection and your controller method running?

Tomcat parses the request on a worker thread and passes it through the servlet filter chain (security, CORS, correlation-ID filters). It reaches the DispatcherServlet — Spring MVC's front controller — which asks HandlerMappings to resolve the @RequestMapping method, invokes HandlerInterceptor.preHandle, then runs argument resolvers (path/param binding, HttpMessageConverter for @RequestBody, Bean Validation for @Valid), then the controller method. The return value goes through return-value handlers/message converters, postHandle, then afterCompletion (even on exceptions), and the response travels back out through the filters. Exceptions from the handler are routed to HandlerExceptionResolvers, which is where @ExceptionHandler/@ControllerAdvice plug in.

A React dev on your team reports a CORS error calling your API. Walk through your diagnosis.

First: CORS errors are a browser-enforced, server-declared policy problem — fix it on the backend. Check (1) is the failing request the preflight? JSON + Authorization header always preflights; look for the OPTIONS call and whether it returns the Access-Control-Allow-* headers. (2) Is the exact origin (scheme+host+port) in the allowed list — http://localhost:5173 ≠ http://localhost:3000. (3) If cookies are involved: allowCredentials(true) forbids wildcard origin — must echo the specific origin. (4) With Spring Security, is CORS configured in the security chain so the anonymous preflight isn't 401'd? (5) If JS can't read a custom response header, it's missing from exposedHeaders. And the non-CORS red herring: a 500 without CORS headers looks like a CORS failure in the browser console.

Why can't Access-Control-Allow-Origin be * when credentials are allowed?

Spec-level protection: credentialed requests attach the user's cookies, so "any site may read responses made with the victim's cookies" would let a malicious page exfiltrate personal data from any credentialed API — CSRF escalated to full read access. Browsers therefore require an explicit, specific origin echo (plus Access-Control-Allow-Credentials: true) so servers must consciously enumerate trusted origins. Spring enforces this at config time (startup error for allowedOrigins("*") + credentials); allowedOriginPatterns is the mechanism for wildcard subdomain matching that still echoes a concrete origin, and responses should carry Vary: Origin for cache correctness.

RestTemplate vs WebClient vs RestClient — which do you pick in 2026?

RestTemplate is in maintenance mode — fine in legacy code, wrong for new code. WebClient is the reactive client: non-blocking, returns Mono/Flux; the right tool on a WebFlux stack or for streaming/high-fan-out — but on a servlet stack calling .block() everywhere buys complexity without benefit. RestClient (Spring 6.1+) is the modern blocking client with WebClient's fluent API and message-converter integration — the default for typical MVC services. Regardless of client: configure timeouts explicitly, add retry/circuit-breaking where appropriate, and propagate correlation headers. @HttpExchange interfaces can sit on top of either for declarative clients.

How does file download work — why Resource and Content-Disposition?

Return ResponseEntity<Resource> (e.g. InputStreamResource) so the file is streamed via ResourceHttpMessageConverter rather than loaded into heap — critical for large files. Set Content-Type to the real MIME type, Content-Length when known, and Content-Disposition to control browser behavior: attachment; filename="report.pdf" triggers a save dialog, inline renders in-browser. Use Spring's ContentDisposition builder to get RFC 6266/5987 filename encoding right for non-ASCII names. On the frontend this is the endpoint you consumed with res.blob() + object URL.

What is HATEOAS, and why do so few teams use it?

HATEOAS embeds hypermedia links in responses (_links.self, _links.cancel…) so clients navigate the API dynamically by following links, not hardcoding URLs — Richardson Level 3, supported by Spring HATEOAS. In practice adoption is thin because real clients are built against a known contract: OpenAPI specs plus generated typed clients deliver the discoverability value at build time with less runtime payload and complexity, and truly generic hypermedia clients basically don't exist outside browsers. The durable subset: returning permitted actions per resource so UIs render buttons from server-side authorization truth. Being able to define it and explain the pragmatic trade-off is exactly the senior expectation.

How would you add request/response documentation to a Spring API, and how does it help a frontend team?

Add springdoc-openapi: it introspects mappings, DTOs, and jakarta.validation annotations to generate an OpenAPI 3 document at /v3/api-docs with Swagger UI on top; annotate with @Operation/@Schema where inference isn't enough, and document error responses (ProblemDetail schemas) too. For the frontend, the spec is machine-readable contract: generate TypeScript types and clients (openapi-typescript, Orval), so a backend field rename fails the frontend build instead of failing in production. It also enables contract testing and mock servers. Keep the spec generated from code (or code-first with review) so it can't drift from reality.

Where do you put cross-cutting logic: Filter, Interceptor, AOP, or ControllerAdvice?

Decide by the layer the concern belongs to. Must cover all HTTP traffic, needs raw streams, or must run before Spring (auth, correlation ID, logging, compression) → Filter. Needs to know which handler will run, HTTP-only, per-endpoint policy (annotation-driven rate limits, tenant guards) → Interceptor. Not HTTP-specific, applies to service-layer methods (transactions, retries, auditing, timing business operations) → AOP aspect. Exception-to-response mapping and shared binder/model configuration → @ControllerAdvice. Anti-pattern to call out: business logic in filters/aspects where it's invisible and untestable — cross-cutting mechanisms are for cross-cutting concerns only.

How do you validate a request param like ?size= so clients can't ask for a million rows?

Put @Validated on the controller class to enable method-parameter validation, then constrain the param: @RequestParam @Min(1) @Max(100) int size. Violations throw ConstraintViolationException, which you map to 400 in the global advice (unmapped it's a 500). Defense in depth: also clamp in code or configure Spring Data's max-page-size, because validation annotations only protect endpoints where someone remembered to add them. This is a resource-exhaustion guard, not just tidiness — unbounded page sizes are an easy self-DoS.

Your endpoint returns dates that the React app misparses. What controls date JSON format in Spring Boot?

Boot registers Jackson's JavaTimeModule and disables WRITE_DATES_AS_TIMESTAMPS, so java.time types serialize as ISO-8601 strings — Instant → "2026-07-02T09:15:30Z", LocalDate → "2026-07-02" — which JS Date and libraries parse natively. Misparses usually mean: someone used legacy java.util.Date (epoch millis by default), a stray @JsonFormat with a non-ISO pattern, or a LocalDateTime with no zone information being treated as UTC/local inconsistently — prefer Instant/OffsetDateTime for moments in time. Global tuning belongs in spring.jackson.* properties or a Jackson2ObjectMapperBuilderCustomizer, not per-field annotations scattered everywhere.

@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = UniqueEmailValidator.class)
public @interface UniqueEmail {
    String message() default "email is already registered";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// Spring instantiates ConstraintValidators as beans → constructor injection works
public class UniqueEmailValidator implements ConstraintValidator<UniqueEmail, String> {
    private final UserRepository users;
    public UniqueEmailValidator(UserRepository users) { this.users = users; }

    @Override
    public boolean isValid(String email, ConstraintValidatorContext ctx) {
        if (email == null || email.isBlank()) return true;  // @NotBlank's job
        return !users.existsByEmail(email.toLowerCase());
    }
}

public record CreateUserRequest(
        @NotBlank @Email @UniqueEmail String email,
        @NotBlank String name) {}

Why it's insufficient: check-then-act race — two concurrent registrations can both pass existsByEmail before either inserts. The database UNIQUE constraint is the real guarantee; the validator is UX. Validation for fast feedback, constraint for correctness — saying both is the senior answer.

@Component
public class TimingInterceptor implements HandlerInterceptor {
    private static final Logger log = LoggerFactory.getLogger(TimingInterceptor.class);
    private static final String START = "req.start";

    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse res, Object handler) {
        req.setAttribute(START, System.nanoTime());
        return true;                                  // false would veto the request
    }

    @Override
    public void afterCompletion(HttpServletRequest req, HttpServletResponse res,
                                Object handler, Exception ex) {   // runs even on exceptions
        long ms = (System.nanoTime() - (long) req.getAttribute(START)) / 1_000_000;
        log.info("{} {} -> {} ({} ms)", req.getMethod(), req.getRequestURI(),
                 res.getStatus(), ms);                // correlationId auto-included via MDC pattern
    }
}

@Configuration
public class WebConfig implements WebMvcConfigurer {
    private final TimingInterceptor timing;
    public WebConfig(TimingInterceptor timing) { this.timing = timing; }

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(timing).addPathPatterns("/api/**");
    }
}

Why the split: the correlation ID must exist for everything — including security-filter rejections, static resources, and errors that never reach MVC — and must wrap the entire processing including other filters, so it's a filter at highest precedence. Timing per handler only cares about MVC endpoints, benefits from path patterns and handler awareness, and wants the afterCompletion guarantee — a natural interceptor.

Exercise 5 — Merge-patch endpoint that distinguishes absent from null

Add PATCH /api/books/{id} consuming application/merge-patch+json supporting partial updates of title and price, where {"price": null} is rejected with 422 ("price cannot be cleared") but an absent price key leaves it untouched. Prove with three curl calls that title-only, price-only, and explicit-null behave correctly.

Solution 
@PatchMapping(value = "/{id}", consumes = "application/merge-patch+json")
public BookResponse patch(@PathVariable long id, @RequestBody JsonNode patch) {
    Book book = service.getOrThrow(id);

    if (patch.has("title")) {                        // key present (maybe null)
        JsonNode v = patch.get("title");
        if (v.isNull() || v.asText().isBlank())
            throw new UnprocessableException("title cannot be cleared");
        book.setTitle(v.asText());
    }
    if (patch.has("price")) {
        JsonNode v = patch.get("price");
        if (v.isNull())
            throw new UnprocessableException("price cannot be cleared");
        BigDecimal price = v.decimalValue();
        if (price.signum() <= 0)
            throw new UnprocessableException("price must be positive");
        book.setPrice(price);
    }
    return BookResponse.from(service.save(book));
}

// In the advice: UnprocessableException → ProblemDetail with HttpStatus.UNPROCESSABLE_ENTITY (422)

# 1. title only — price untouched
curl -s -X PATCH localhost:8080/api/books/1 \
     -H 'Content-Type: application/merge-patch+json' -d '{"title":"New Title"}'

# 2. price only — title untouched
curl -s -X PATCH localhost:8080/api/books/1 \
     -H 'Content-Type: application/merge-patch+json' -d '{"price": 19.99}'

# 3. explicit null — rejected 422
curl -s -X PATCH localhost:8080/api/books/1 \
     -H 'Content-Type: application/merge-patch+json' -d '{"price": null}'
# → {"status":422,"detail":"price cannot be cleared"}

The load-bearing insight: a typed DTO cannot express the absent-vs-null distinction (both bind to null), which is why the handler reads JsonNode and checks has() before isNull(). That distinction is the merge-patch spec.
