The equals() / hashCode() contract T1

By default, equals() compares object identity — are these the exact same object in memory? Usually you want value equality: two Point(1,2) objects should be "equal." You get that by overriding equals(). The catch — and the reason this is a top interview topic — is that you must override hashCode() at the same time, from the same fields.

Why? Hash-based collections (HashMap, HashSet) do a two-step lookup: hashCode() picks the bucket, then equals() confirms the match inside it. The contract:

  • Equal objects must have equal hash codes.
  • Unequal objects may share a hash code (a "collision" — allowed, just slower).
public class Point {
    private final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point p)) return false;  // pattern matching, Java 16+
        return x == p.x && y == p.y;                // same fields both methods use
    }
    @Override
    public int hashCode() { return Objects.hash(x, y); }  // derived from the SAME fields
}

Override equals but forget hashCode and this happens: map.put(new Point(1,2), "a") then map.get(new Point(1,2)) returns null. The two points are value-equal but have different default hash codes, so they land in different buckets and equals is never even called. A HashSet will happily store "duplicates." Always override both, from the same field set.

Never use a mutable object as a map key. If you mutate a field that feeds hashCode after inserting, the entry sits in the bucket for the old hash while every lookup computes the new hash and searches the wrong bucket — the entry becomes unreachable yet still occupies memory. This is why keys should be immutable: String, boxed numbers, or records.

The good news: you'll rarely write this by hand anymore — records generate both methods correctly (below). But you must be able to explain the contract, because it's the "do they actually understand Java" litmus test.

Immutability — designing a bulletproof object T1

An immutable object can't change after construction. That single property buys you an enormous amount: it's automatically thread-safe (no writes to race on — page 04), safe to share and cache, and safe to use as a map key or set element. String, Integer, and LocalDate are all immutable, and it's why String is such a reliable key.

The recipe for a hand-rolled immutable class:

  1. Make the class final (no subclass can add mutable state).
  2. Make all fields private final.
  3. Set every field in the constructor; provide no setters.
  4. Defensively copy any mutable input or output (arrays, collections, dates) so callers can't reach in and mutate your internals.
public final class Money {
    private final long cents;
    private final List<String> tags;

    public Money(long cents, List<String> tags) {
        this.cents = cents;
        this.tags = List.copyOf(tags);   // defensive copy — caller's later edits can't leak in
    }
    public List<String> tags() { return tags; }  // List.copyOf is already unmodifiable — safe to hand out
    // "mutation" returns a NEW object, like String.toUpperCase()
    public Money plus(long more) { return new Money(cents + more, tags); }
}

Immutability and the equals/hashCode contract are really one topic: immutable objects make safe keys precisely because their hash can never drift. Reach for immutability by default and only add mutability where you truly need it.

Records — value types with zero boilerplate T1

Since Java 16, a record is the one-liner for an immutable data carrier. It auto-generates the constructor, private final fields, accessors, and — crucially — a correct equals(), hashCode(), and toString() from its components:

public record Point(int x, int y) { }   // that's the whole class

Point p = new Point(1, 2);
p.x();                              // accessor (note: x(), not getX())
p.equals(new Point(1, 2));       // true — value equality for free
p;                                  // toString ⇒ Point[x=1, y=2]

// Add validation with a compact constructor
public record Money(long cents, String currency) {
    public Money {                       // runs before the fields are assigned
        if (cents < 0) throw new IllegalArgumentException("negative");
    }
}

Reach for a record for DTOs, API request/response bodies, map keys, coordinates, query results — anything that's "just data." It's the modern default and it makes the equals/hashCode contract a non-issue because the compiler honors it for you. Use a regular class when you need mutability, inheritance, or hidden internal state.

The collections framework: List, Set, Map, Queue

Java's collections are organized under a few interfaces. Program to the interface on the left, pick the implementation on the right:

InterfaceMeaningGo-to implementationJS analog
List<E>Ordered, indexed, allows duplicatesArrayListArray
Set<E>No duplicatesHashSet / LinkedHashSetSet
Map<K,V>Key → valueHashMap / LinkedHashMapMap / object
Queue<E> / Deque<E>FIFO / double-endedArrayDequearray as stack/queue
 
List<String> names = new ArrayList<>();  // declare as the INTERFACE, not ArrayList
names.add("Ada");
for (String n : names) { ... }

Map<String, Integer> scores = new HashMap<>();
scores.put("ada", 10);
scores.getOrDefault("bob", 0);            // no NPE on missing key
scores.merge("ada", 1, Integer::sum);       // increment-or-insert in one call

// computeIfAbsent shines for a map-of-lists (grouping)
Map<String, List<String>> byDept = new HashMap<>();
byDept.computeIfAbsent("eng", k -> new ArrayList<>()).add("Ada");  // create list once, then append

Set<String> seen = new HashSet<>();
boolean isNew = seen.add("x");            // false if already present — handy for dedup

// Immutable literals (Java 9+) — great for constants and test data
List<Integer> primes = List.of(2, 3, 5);   // throws if you try to modify it

Declaring variables by their interface (List, not ArrayList) means you can swap the implementation later without touching callers — the everyday face of "program to interfaces" from page 01. computeIfAbsent, getOrDefault, and merge are the map methods that make code clean; know them cold.

The variants worth knowing

  • LinkedHashMap / LinkedHashSet — same as the hash versions but preserve insertion order when you iterate. Reach for these when order matters.
  • TreeMap / TreeSet — keep keys sorted; O(log n) ops and range queries (headMap, ceilingKey). Use when you need sorted order or "nearest key" lookups.
  • ArrayDeque — the modern stack and queue. Faster than the legacy Stack and LinkedList; use it for both.

Choosing the right collection

You need…Use
An ordered list, index access, iterate a lotArrayList
Uniqueness / fast "have I seen this?"HashSet
Key → value lookupsHashMap
…and insertion order preservedLinkedHashMap / LinkedHashSet
…and sorted order / range queriesTreeMap / TreeSet
Stack or queue (add/remove at ends)ArrayDeque

The classic "ArrayList vs LinkedList?" — almost always ArrayList. Even for the operations LinkedList theoretically wins (insert in the middle), you first have to walk to the position (O(n)), and its scattered nodes destroy CPU cache locality, so it loses in practice. Reach for ArrayDeque, not LinkedList, when you need a queue. Real answer: "I default to ArrayList; I've essentially never needed LinkedList."

Sorting: Comparable vs Comparator T1

Two ways to define order. Comparable is the type's one natural ordering, baked into the class. Comparator is an external, swappable ordering you pass in — and the one you'll use constantly.

// Comparator — the modern, composable way (no need to touch the class)
record Employee(String name, String dept, int salary) { }

staff.sort(Comparator.comparing(Employee::dept)          // by dept asc,
        .thenComparing(Employee::salary, Comparator.reverseOrder())  // then salary desc,
        .thenComparing(Employee::name));                    // then name asc

Comparator.comparing(...).reversed() reverses the entire chain, not just the last key. To reverse only one key, pass Comparator.reverseOrder() to that specific comparing/thenComparing call, as above. Flipping the whole chain by accident is a very common bug.

Java's List.sort uses TimSort, which is stable — equal elements keep their relative order. That lets you sort by successive keys (sort by name, then by dept) and get a correct multi-key result. Interviewers like this because it shows you understand ordering, not just calling .sort().

Generics you can read and write

Generics are Java's version of TypeScript generics: parameterize a type so List<String> only holds Strings and the compiler catches misuse. You already use them every time you write List<X>. Writing your own is straightforward:

// A generic method — <T> before the return type declares the type variable
static <T> T firstOrNull(List<T> list) {
    return list.isEmpty() ? null : list.get(0);
}

// A generic class
class Box<T> {
    private T value;
    public T get() { return value; }
    public void set(T v) { this.value = v; }
}

// Bounded type — "T must be a Number (or subtype)" so you can call number methods
static <T extends Number> double sum(List<T> xs) {
    double total = 0;
    for (T x : xs) total += x.doubleValue();
    return total;
}

Wildcards (?) express flexible relationships: List<? extends Number> is "a list of some subtype of Number — I'll only read from it," and List<? super Integer> is "a list I can write Integers into." The rule of thumb (PECS — "Producer Extends, Consumer Super") and Java's type erasure are worth a deeper look once the basics are solid — see the appendix. For day-to-day work, being able to read <T>, write a bounded method, and understand why List<String> isn't a List<Object> is enough.

Exceptions & try-with-resources

Java splits exceptions into two families:

  • Checked (IOException, SQLException) — the compiler forces you to either catch them or declare throws. Meant for recoverable, expected failures (a file's missing, the network hiccuped).
  • Unchecked (RuntimeException and subclasses: NullPointerException, IllegalArgumentException) — no compiler ceremony. Meant for programming errors.
// try-with-resources — auto-closes anything AutoCloseable, even on exception
try (var reader = Files.newBufferedReader(path)) {
    return reader.readLine();
} catch (IOException e) {
    throw new UncheckedIOException("failed reading " + path, e);  // wrap, keep the cause
}   // reader.close() runs automatically here — no finally block needed

Best practices worth stating: use try-with-resources for anything closeable (files, streams, JDBC connections) — it replaces error-prone finally blocks. Never swallow an exception with an empty catch. When you rethrow, preserve the cause (new XException(msg, e)) so the stack trace stays intact. Throw IllegalArgumentException/IllegalStateException for bad inputs/state. The prevailing modern style favors unchecked exceptions for most application errors — checked exceptions don't compose well with lambdas and streams.

How HashMap works inside T2

A HashMap is an array of "buckets." put(k,v): (1) compute hash = key.hashCode() ^ (h >>> 16) — the XOR spreads high bits into the low bits, because only the low bits pick the bucket. (2) Index with (n-1) & hash — capacity n is a power of two, so this equals modulo but is a single AND instruction. (3) If the bucket is empty, place the node; otherwise walk the chain comparing hash then equals — equal key replaces the value, else append. (4) When size > capacity × 0.75 (the load factor), the table doubles and rehashes. Amortized O(1).

Treeification (Java 8): when a single bucket's chain reaches 8 nodes and capacity ≥ 64, that bucket converts from a linked list to a red-black tree, bounding worst-case lookup at O(log n) instead of O(n). This defends against hash-flooding denial-of-service attacks (an attacker deliberately sending colliding keys). Bins revert to lists at 6 nodes — the gap prevents flip-flopping.

ConcurrentHashMap T2

A plain HashMap is not thread-safe — concurrent writes can corrupt it or (pre-Java 8) spin into an infinite loop during resize. ConcurrentHashMap is the answer: it locks only the individual bucket being written (fine-grained locking / CAS), so reads are lock-free and writes to different buckets proceed in parallel. Use its atomic compound methods — merge, computeIfAbsent — because a get-then-put across two calls is not atomic even on a concurrent map. (Much more on this in page 04.)

Fail-fast iterators T3

Modifying a collection while iterating it with a for-each loop throws ConcurrentModificationException — the iterator tracks a modCount and "fails fast" when it changes underneath. To remove during iteration, use the explicit Iterator.remove() or collection.removeIf(predicate). It's called fail-fast because surfacing the bug immediately beats silent corruption.

PECS & type erasure T3

Erasure: generics exist only at compile time — the JVM sees raw List, the type arguments are erased. Hence you can't write new T[] or check x instanceof List<String>. PECS — "Producer Extends, Consumer Super" — decides wildcard direction: use <? extends T> for a source you only read from (a producer), <? super T> for a sink you only write to (a consumer). Collections.copy(dest, src) is the canonical example: copy(List<? super T> dest, List<? extends T> src).

  • Override equals and hashCode together, from the same fields — or hash collections silently break. Prefer records, which do it for you.
  • Immutable objects are thread-safe, cacheable, and safe as keys — make them your default.
  • Declare variables by their interface (List, Map); default to ArrayList, HashMap, HashSet, ArrayDeque.
  • Know the map power-methods: getOrDefault, computeIfAbsent, merge.
  • Use Comparator.comparing().thenComparing() for multi-key sorts — and reverse a single key, not the whole chain.
  • Use try-with-resources, preserve exception causes, don't swallow.

Interview Q&A

Why must you override hashCode() whenever you override equals()?

Hash collections do a two-phase lookup: hashCode() selects the bucket, equals() confirms the match within it. The contract requires equal objects to have equal hash codes. Override only equals and two value-equal objects keep distinct default hash codes, land in different buckets, and equals is never called — so map.get returns null after a put with an equal key, and a HashSet stores duplicates. Derive both methods from the same fields, or use a record.

What happens if you mutate an object after using it as a HashMap key?

If the mutation changes a field used by hashCode, the entry is stranded: it physically sits in the bucket for the old hash, but every get/remove/containsKey computes the new hash and searches a different bucket. The entry becomes unreachable through the API yet still consumes memory and still shows up in iteration — a leak plus phantom data. That's why keys must be immutable: String, boxed numbers, records.

How do you make a class immutable?

Make the class final, all fields private final, set them only in the constructor, expose no setters, and defensively copy any mutable input or returned collection/array/date so callers can't reach in and mutate internal state. "Mutations" return a new instance instead of changing the current one (like String). Or, for plain data, use a record — it's immutable by construction. Immutable objects are inherently thread-safe and safe as map keys.

ArrayList vs LinkedList — which and why?

ArrayList almost always. It's a backing array: O(1) random access and excellent cache locality, so iteration and access are fast. LinkedList's theoretical win — O(1) insertion — requires you to first walk to the position (O(n)), and its scattered nodes wreck cache performance, so it loses in practice and costs more memory per element. For a queue or stack, use ArrayDeque, not LinkedList. I default to ArrayList and have essentially never needed LinkedList.

When would you use a record instead of a class?

Use a record for immutable data carriers — DTOs, API bodies, value objects, map keys, query results. It auto-generates the constructor, accessors, and a correct equals/hashCode/toString from the components, eliminating both boilerplate and the risk of a broken equality contract. Use a regular class when you need mutability, inheritance, or encapsulated internal state that isn't just its components. Records can't extend another class (they implicitly extend Record) but can implement interfaces.

What's the difference between checked and unchecked exceptions?

Checked exceptions (subclasses of Exception but not RuntimeException, e.g. IOException) must be caught or declared with throws — the compiler enforces it; they model recoverable, expected failures. Unchecked exceptions (RuntimeException and subclasses, e.g. NullPointerException, IllegalArgumentException) need no ceremony and model programming errors. Modern style leans toward unchecked exceptions for application errors because checked ones don't compose with lambdas/streams; the key discipline is preserving the cause when you wrap and rethrow.

Walk me through what HashMap.put() does. T2

Compute hash = key.hashCode() ^ (h >>> 16) to spread high bits down; index the bucket with (n-1) & hash (n is a power of two, so this is modulo via one AND). If the bucket is empty, place a node; otherwise walk the chain comparing hash then equals — replace on an equal key, else append. If a chain hits 8 nodes with capacity ≥ 64 it treeifies to a red-black tree (O(log n) worst case, a hash-flooding defense). When size exceeds capacity × 0.75, the table doubles and rehashes. Amortized O(1).

Hands-on exercises

Exercise 1: Break and fix a HashSet

Write a class User { String email; } that overrides equals to compare emails but forgets hashCode. Add two Users with the same email to a HashSet. How many are in the set, and why? Then fix it.

Solution

The set contains two users. With only equals overridden, the two objects have different default (identity) hash codes, so they hash to different buckets and equals is never invoked to detect the duplicate — the "no duplicates" guarantee is broken. Fix by adding public int hashCode() { return Objects.hash(email); } so equal emails share a bucket, or better, make it record User(String email) {} and delete all the hand-written code.

Exercise 2: Word frequency count

Given List<String> words, build a Map<String,Integer> of counts using one map method per word (no get-then-put). Which method, and what's the one-liner?

Solution 
Map<String, Integer> freq = new HashMap<>();
for (String w : words)
    freq.merge(w, 1, Integer::sum);   // insert 1, or add 1 to the existing count

merge(key, 1, Integer::sum) inserts 1 if the key is absent, otherwise applies sum to the old and new values — the whole increment-or-insert in a single atomic call. This same one-liner is the correct, thread-safe way to count on a ConcurrentHashMap, which is why interviewers like it.

Exercise 3: Multi-key sort with a reverse trap

Sort List<Employee> (a record of name, dept, salary) by dept ascending, then salary descending, then name ascending — and explain why you can't just append .reversed().

Solution 
staff.sort(Comparator.comparing(Employee::dept)
        .thenComparing(Employee::salary, Comparator.reverseOrder())
        .thenComparing(Employee::name));

Appending .reversed() to the whole chain would flip every key — dept and name would go descending too. To reverse only salary, pass Comparator.reverseOrder() as that single key's comparator. Because TimSort is stable, this composed comparator yields a correct, deterministic multi-key ordering.
