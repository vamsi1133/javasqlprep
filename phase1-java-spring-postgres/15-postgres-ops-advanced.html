VACUUM, autovacuum & bloat T2

Why VACUUM exists: the bill for MVCC

Recall from tutorial 14: Postgres implements MVCC by never updating rows in place. An UPDATE writes a new row version and marks the old one as expired; a DELETE just marks the row as expired. Old versions must stick around as long as some transaction might still need to see them. Once no transaction can see a dead row version, it's garbage — but Postgres doesn't clean it up inline during normal query execution. That deferred cleanup job is VACUUM.

So a table that receives heavy updates is constantly generating dead tuples. Without vacuuming:

  • The table file grows even though the logical row count is stable — this is bloat.
  • Sequential scans read pages full of dead rows; everything gets slower for the same data.
  • Indexes bloat too, since each new row version usually needs new index entries.

What VACUUM actually does

Plain VACUUM (the kind autovacuum runs) does three important things:

ActionEffect
Reclaims dead tuples for reuse Marks the space inside pages as free so future INSERTs/UPDATEs can reuse it. Note: it usually does not shrink the file on disk (it can only truncate empty pages at the very end).
Updates the visibility map Marks pages where all tuples are visible to every transaction. This is what makes index-only scans possible — the planner can skip fetching the heap page. A badly vacuumed table quietly kills your index-only scans.
Freezes old transaction IDs Replaces very old xids on rows with a special "frozen" marker so the 32-bit transaction ID counter can safely wrap around (see below). 
-- Manual vacuum of one table (non-blocking for reads/writes)
VACUUM (VERBOSE, ANALYZE) orders;

-- ANALYZE alone refreshes planner statistics (row counts, value distributions)
ANALYZE orders;

VACUUM vs VACUUM FULL

-- Rewrites the entire table into a new file, returns space to the OS.
-- Takes an ACCESS EXCLUSIVE lock: blocks ALL reads and writes for the duration.
VACUUM FULL orders;   -- rarely the right answer in production
VACUUMVACUUM FULL
LockingRuns alongside normal trafficExclusive lock — table is offline
Disk spaceMarks space reusable inside the fileRewrites table, returns space to OS
WhenContinuously, via autovacuumRare recovery from severe bloat, in a maintenance window (or use pg_repack for an online rewrite)

"The table is bloated, just run VACUUM FULL" is a classic junior mistake in prod. It takes an exclusive lock — on a big table that's an outage. The senior answer: fix the cause (autovacuum tuning, long-running transactions), let regular VACUUM contain it, and if you truly must reclaim disk, use pg_repack or schedule VACUUM FULL in a window.

Autovacuum and the scale-factor problem

Autovacuum is a background daemon that vacuums tables when their dead-tuple count crosses a threshold. The default trigger is roughly: 

dead_tuples > autovacuum_vacuum_threshold          -- default 50
             + autovacuum_vacuum_scale_factor      -- default 0.2 (20%!)
               * reltuples (estimated live rows)

The percentage is the problem on big tables. 20% of a 500-million-row table is 100 million dead rows before autovacuum even wakes up — and then it has an enormous job to do. Common fix: lower the scale factor per table so vacuum runs earlier and does smaller increments: 

ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor = 0.01,   -- vacuum at ~1% dead
  autovacuum_analyze_scale_factor = 0.005
);

What happens when vacuum falls behind

  • Bloat compounds: table and index files grow, cache hit rates drop, scans slow down.
  • Index-only scans degrade: the visibility map goes stale, so "index-only" scans start hitting the heap anyway — a query that was fast quietly becomes slow with no plan change.
  • Planner stats go stale (if ANALYZE also lags): bad row estimates → bad plans.
  • Transaction ID wraparound — the scary-sounding one, explained calmly below.

XID wraparound, without the panic

Postgres transaction IDs are 32-bit counters used to decide row visibility ("was this row created before or after my snapshot?"). Because the counter wraps, a row whose xid is more than ~2 billion transactions old would become ambiguous. VACUUM prevents this by freezing old rows — marking them "visible to everyone forever" so their original xid no longer matters. If freezing falls extremely far behind, Postgres gets increasingly loud: warnings in logs, then aggressive anti-wraparound autovacuums you can't cancel, and in the true worst case it stops accepting new transactions until a vacuum completes. In practice, you get weeks of warnings first. The interview takeaway: know it exists, know VACUUM freezing is the fix, and know the usual root cause is autovacuum being starved or blocked — not some random cosmic event.

Monitoring bloat and dead tuples

SELECT relname,
       n_live_tup,
       n_dead_tup,
       round(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
       last_autovacuum,
       last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 10;

The #1 vacuum blocker: long-running transactions

VACUUM can only remove dead rows that no transaction could still see. One forgotten idle in transaction session — a connection that ran BEGIN and went to lunch — pins the "oldest visible" horizon for the entire database. Autovacuum runs, but reclaims nothing. Bloat grows everywhere while your monitoring says "vacuum is running fine." 

-- Find long-running / idle-in-transaction sessions
SELECT pid, state, now() - xact_start AS xact_age, left(query, 60) AS query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start
LIMIT 10;

-- Guardrail: kill transactions idle too long
ALTER SYSTEM SET idle_in_transaction_session_timeout = '5min';

Interview-ready summary: "VACUUM is the garbage collector MVCC requires. Plain VACUUM reclaims dead tuples for reuse, maintains the visibility map (which index-only scans depend on), and freezes old xids to prevent wraparound. VACUUM FULL rewrites the table under an exclusive lock — almost never in prod. On big tables I lower the per-table scale factor so autovacuum runs early and often, and the first thing I check when bloat grows despite vacuuming is long-running or idle-in-transaction sessions holding back the cleanup horizon."

Connection pooling: HikariCP & PgBouncer T2

Why Postgres connections are expensive

Coming from Node, this is a mindset shift. Postgres forks a dedicated OS process per connection — not a thread, not an async task. Each backend process carries several megabytes of memory, and per-connection work_mem can multiply that during sorts/hashes. Beyond memory, more backends means more contention on shared locks and buffers, and more processes for the scheduler to juggle. A Postgres server is happiest with a few dozen to a few hundred active connections — not thousands.

App-side pooling: HikariCP sizing recap

In Spring Boot, HikariCP is the default pool. The counterintuitive truth (well-documented by the HikariCP project): a small pool with a queue outperforms a big pool. Requests waiting briefly in the pool's queue finish sooner overall than requests all hammering the database at once, because the DB isn't drowning in context switches and lock contention. 

# A useful starting-point formula (from the HikariCP wiki):
#   pool_size ≈ (cpu_cores * 2) + effective_spindle_count
# For a typical cloud Postgres with SSD: cores*2 + a small constant.
# An 8-core DB server → pool of ~16-20 TOTAL across all app instances.

# application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10        # per app instance — keep TOTAL in mind
      minimum-idle: 10
      connection-timeout: 3000     # fail fast instead of piling up
      max-lifetime: 1800000        # recycle before infra kills idle conns

The total that matters is pool_size × number_of_app_instances. Ten Kubernetes pods with maximum-pool-size: 50 is 500 potential connections. Teams "fix" the resulting errors by raising max_connections on the server — which makes the database slower (more memory pressure, more contention) while hiding the real problem. Right fix: shrink app pools and/or add PgBouncer.

PgBouncer: pooling in front of the database

When many app instances (or serverless functions) need Postgres, put PgBouncer — a lightweight connection proxy — in front. Thousands of client connections on one side, a small pool of real Postgres connections on the other.

ModeHow it maps clients → server connsTrade-off
Session One server connection per client for the whole session Fully transparent, but pools poorly — barely better than no bouncer for idle-heavy apps
Transaction (the usual choice) Server connection borrowed only for the duration of each transaction Excellent multiplexing, but no session state between transactions
Statement Per statement Most aggressive; breaks multi-statement transactions — niche

Transaction-mode caveats you should be able to name:

  • No session-level state: SET (non-local), session advisory locks, temp tables, LISTEN — all unreliable, because your next transaction may run on a different server connection.
  • Server-side prepared statements historically broke (prepared on connection A, executed on connection B). Modern PgBouncer (1.21+) can track protocol-level prepared statements; otherwise JDBC drivers are configured to avoid server-side prepares, at a small performance cost.
; pgbouncer.ini (sketch)
[databases]
appdb = host=10.0.0.5 port=5432 dbname=appdb

[pgbouncer]
pool_mode = transaction
default_pool_size = 20        ; real PG connections per db/user pair
max_client_conn = 2000        ; clients pgbouncer will accept

Layered answer: "HikariCP pools within one JVM; PgBouncer pools across the whole fleet. I size Hikari small — roughly cores×2 on the DB, divided across instances — and if instance count is large or elastic, I add PgBouncer in transaction mode, being careful about session state and prepared-statement settings. Raising max_connections is the anti-pattern: each connection is a whole process, so more connections past the sweet spot means lower throughput, not higher."

Replication, read replicas & the lag problem T2

Streaming replication in one paragraph

Every change Postgres makes is first written to the WAL (write-ahead log) — the same log that powers crash recovery. Streaming replication ships that WAL, in real time, from the primary to one or more replicas, which replay it continuously. Replicas are byte-identical copies, open for read-only queries (hot standby). All writes go to the primary. 

-- On a replica, confirm role and lag:
SELECT pg_is_in_recovery();                        -- true on a replica
SELECT now() - pg_last_xact_replay_timestamp() AS replay_lag;

-- On the primary, per-replica view:
SELECT client_addr, state, sync_state,
       pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS byte_lag
FROM pg_stat_replication;

Synchronous vs asynchronous

Async (default)Sync
Commit returns when…WAL is durable on the primary…and at least one sync replica has confirmed it
Data loss on primary failurePossible: last few ms/seconds of commits (RPO > 0)None for confirmed commits (RPO ≈ 0)
CostNone on the write pathEvery commit pays a network round-trip; a slow/dead sync replica can stall writes

Most systems run async and accept a tiny RPO. Sync replication is for "losing one committed row is unacceptable" domains (payments ledgers), often with synchronous_commit tuned per-transaction so only the critical writes pay the latency.

Read replicas — and the application problem they create

Replicas scale reads: dashboards, reports, search, feed rendering. But async replication means a replica is always slightly behind — usually milliseconds, occasionally seconds (vacuum storms, big transactions, network blips). That gap creates the classic read-your-own-writes bug:

User saves their new profile photo → UPDATE hits the primary → app redirects to the profile page → that read is routed to a replica that hasn't replayed the update yet → user sees their old photo and files a bug ("save is broken!"). Nothing is broken — you served a stale read after a write.

Mitigations (know at least three):

  1. Read-from-primary-after-write: after a user writes, route that user's reads to the primary for a short window (e.g., a "wrote recently" flag with a TTL in the session/cache). Simple and common.
  2. Sticky sessions / pinning: pin a session to the primary for the rest of the request (or a few seconds after any write within it).
  3. Lag-aware routing: only route to replicas whose measured lag is under a threshold; eject laggy replicas from the read pool.
  4. LSN tokens (causal consistency): capture the primary's WAL position after the write; a replica may serve the read only once it has replayed past that LSN. Most precise, most plumbing.
  5. Design tolerance: for feeds/analytics, just accept staleness — cheapest of all.

In Spring, the routing itself is typically an AbstractRoutingDataSource keyed off @Transactional(readOnly = true), or a JDBC-level splitter. Awareness of the pattern is enough: read-only transactions → replica datasource, everything else → primary, plus one of the mitigations above for read-your-own-writes. 

// Awareness-level sketch: route by transaction read-only flag
public class ReadWriteRoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
                ? "replica" : "primary";
    }
}

Failover

When the primary dies, a replica must be promoted to primary and traffic re-pointed. Doing this safely by hand is stressful (split-brain risk: two nodes both believing they're primary). Production setups use an orchestrator — Patroni is the de facto standard (uses etcd/Consul for leader election, handles promotion and fencing) — or the cloud provider's managed HA (RDS Multi-AZ, Cloud SQL HA). Awareness level: know the term, know why automated fencing matters, don't hand-roll it.

Logical replication vs streaming

Streaming (physical)Logical
ReplicatesWAL bytes — the entire cluster, exactlyRow changes on selected tables (publish/subscribe)
VersionsSame major version requiredCross-version — the standard tool for near-zero-downtime major upgrades
Target writable?No — replica is read-onlyYes — subscriber is a normal DB with its own data

-- Logical replication: publisher side
CREATE PUBLICATION orders_pub FOR TABLE orders, order_items;

-- Subscriber side (another cluster)
CREATE SUBSCRIPTION orders_sub
  CONNECTION 'host=primary dbname=appdb user=repl'
  PUBLICATION orders_pub;

The same logical-decoding machinery powers CDC (change data capture) tools like Debezium, which stream row changes into Kafka. That's the production-grade companion to the outbox pattern from the transactions tutorial: write the event to an outbox table in the same transaction as the business change, and let Debezium tail the WAL and publish it — no dual-write problem, no polling.

Table partitioning T2

What and why

Declarative partitioning splits one logical table into physical child tables by a key — range (dates), list (region, tenant), or hash (even spread). Queries still target the parent; Postgres routes rows and prunes partitions automatically.

The honest reasons to partition (in order of real-world value):

  1. Instant data retention: DROP TABLE orders_2024_01 removes a month in milliseconds. The alternative — DELETE FROM orders WHERE created_at < ... on a huge table — grinds for hours and generates a mountain of dead tuples for VACUUM.
  2. Manageable maintenance: VACUUM, ANALYZE, and reindex work per-partition — small, predictable jobs instead of one monster.
  3. Partition pruning: queries filtered on the partition key scan only relevant partitions, with smaller per-partition indexes.
-- Range partitioning: orders by month
CREATE TABLE orders (
  id          bigint GENERATED ALWAYS AS IDENTITY,
  customer_id bigint NOT NULL,
  status      text   NOT NULL,
  total_cents bigint NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (id, created_at)          -- must include the partition key!
) PARTITION BY RANGE (created_at);

CREATE TABLE orders_2026_06 PARTITION OF orders
  FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');
CREATE TABLE orders_2026_07 PARTITION OF orders
  FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');

-- Catch-all for rows outside any declared range (use with care)
CREATE TABLE orders_default PARTITION OF orders DEFAULT;

-- Indexes on the parent cascade to all partitions
CREATE INDEX ON orders (customer_id, created_at DESC);

Seeing pruning in EXPLAIN

EXPLAIN (COSTS OFF)
SELECT * FROM orders
WHERE created_at >= '2026-07-01' AND created_at < '2026-07-05';

--  Seq Scan on orders_2026_07 orders          <-- ONLY this partition
--    Filter: created_at >= ... AND created_at < ...
--
-- Without a partition-key predicate, you'd see an Append node
-- scanning every partition — pruning only helps queries that
-- actually filter on the key.

When NOT to partition

  • Small/medium tables. Under ~100M rows / tens of GB, good indexing usually wins. Partitioning adds planning overhead and operational complexity for nothing.
  • No natural key in your WHERE clauses. If queries don't filter on the candidate partition key, there's no pruning — every query fans out to all partitions and gets slower.
  • As a performance hail-mary. Partitioning is primarily a data lifecycle and maintenance tool; it's not a substitute for a missing index.

Pitfalls to name in an interview

Unique constraints must include the partition key. Postgres can't enforce global uniqueness across partitions, so PRIMARY KEY (id) alone is rejected — you need (id, created_at). Consequence: no plain unique email column on a partitioned users table, and foreign keys referencing the partitioned table need that composite key too. This is why "partition everything" is not free.

Other gotchas: (1) a default partition catches stray rows but then blocks creating new partitions whose range overlaps rows already sitting in the default — many teams skip it and instead auto-create future partitions with a scheduled job or pg_partman. (2) You can't UPDATE a row's partition key in a way that violates its partition without Postgres physically moving the row (a delete+insert under the hood). (3) Too many partitions (thousands) inflates planning time.

Backups, PITR & sharding awareness T3

# Logical: custom-format dump, parallel restore
pg_dump -Fc -f appdb.dump appdb
pg_restore -d appdb_restored -j 4 appdb.dump

# Physical: base backup (WAL archiving configured separately via archive_command
# or a tool like pgBackRest / WAL-G)
pg_basebackup -D /backups/base -X stream -P

Vocabulary worth using precisely: RPO (recovery point objective — how much data you can afford to lose; WAL archiving gets this near zero) and RTO (recovery time objective — how long until you're back; dominated by restore + replay time). And the old ops wisdom that always lands well: an untested backup is a hope, not a backup — restore drills are part of the backup strategy.

Sharding: the honest awareness answer

Sharding = splitting data across multiple independent Postgres clusters (by tenant, user-id hash, etc.). Options: app-level sharding (application picks the shard; simple concept, but cross-shard queries, transactions, and rebalancing become your problem) or Citus (Postgres extension that distributes tables across worker nodes behind a coordinator).

The senior take interviewers want: "One well-tuned Postgres with read replicas goes further than most people think — a single beefy instance handles multiple TB and tens of thousands of QPS with good schema, indexes, and caching. Sharding costs you cross-shard joins, global transactions, and painful rebalancing, so I'd exhaust vertical scaling, replicas, caching, and partitioning first, and treat sharding as a system-design conversation with real numbers, not a default." Then defer detail to the system-design round.

JSONB in depth — and "why not MongoDB?" T2

json vs jsonb

jsonjsonb
StorageExact input textParsed binary: no duplicate keys, no key order, whitespace normalized
Read/query speedRe-parses on every accessFast field access, rich operators
Indexable (GIN)NoYes
UseOnly if you must preserve exact original text (e.g., signed payloads)Everything else — the default

Operators you must know cold

CREATE TABLE events (
  id      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  type    text NOT NULL,
  payload jsonb NOT NULL,
  at      timestamptz NOT NULL DEFAULT now()
);

INSERT INTO events (type, payload) VALUES
('order.placed', '{"orderId": 42, "customer": {"id": 7, "tier": "gold"},
                   "items": [{"sku": "A1", "qty": 2}], "tags": ["promo"]}');

-- ->  returns jsonb          ->>  returns text  (THE classic gotcha)
SELECT payload -> 'customer'            FROM events;  -- {"id": 7, "tier": "gold"}  (jsonb)
SELECT payload ->> 'type'               FROM events;  -- text or NULL
SELECT payload -> 'customer' ->> 'tier' FROM events;  -- 'gold' as text

-- #>> : extract by path, as text
SELECT payload #>> '{customer,tier}' FROM events;     -- 'gold'

-- @> : containment — "does the left side contain this shape?"
SELECT * FROM events WHERE payload @> '{"customer": {"tier": "gold"}}';

-- ?  : does this top-level key exist?
SELECT * FROM events WHERE payload ? 'tags';

-- Beware type mixing: payload -> 'orderId' is jsonb 42, not integer 42
SELECT * FROM events WHERE (payload ->> 'orderId')::bigint = 42;

-> vs ->> trips everyone: -> yields jsonb (chainable, compares as JSON), ->> yields text (comparable to strings, castable to numbers). payload -> 'tier' = 'gold' fails or misbehaves; you want payload ->> 'tier' = 'gold'.

Updating JSONB

-- jsonb_set(target, path, new_value, create_missing)
UPDATE events
SET payload = jsonb_set(payload, '{customer,tier}', '"platinum"')
WHERE id = 1;

-- Merge / add keys with ||
UPDATE events SET payload = payload || '{"reviewed": true}' WHERE id = 1;

-- Remove a key
UPDATE events SET payload = payload - 'tags' WHERE id = 1;

Remember MVCC: every JSONB "field update" rewrites the whole row (and large payloads live in TOAST storage, compressed out-of-line). Hot, frequently-updated fields belong in real columns; JSONB suits mostly-read documents.

Indexing JSONB

-- GIN with default jsonb_ops: supports @>, ?, ?|, ?&
CREATE INDEX idx_events_payload ON events USING gin (payload);

-- jsonb_path_ops: smaller/faster index, but ONLY supports @> containment
CREATE INDEX idx_events_payload_path ON events USING gin (payload jsonb_path_ops);

-- Best of all for a known hot field: a plain B-tree expression index
CREATE INDEX idx_events_customer_tier ON events ((payload -> 'customer' ->> 'tier'));

-- This now uses the B-tree — ordinary equality, ordinary speed:
SELECT * FROM events WHERE payload -> 'customer' ->> 'tier' = 'gold';

Rule of thumb: ad-hoc shape queries → GIN (jsonb_path_ops if you only ever use @>); known field you filter on constantly → expression B-tree index, which is smaller and supports ranges/sorting. You can even add constraints on extracted fields (CHECK ((payload ->> 'orderId') IS NOT NULL)) — schema-on-write where it matters.

When JSONB is right vs wrong

Right toolWrong tool
Sparse/variable attributes (product specs differing per category) Core relational data you join, aggregate, and constrain (orders, users, money)
External payloads you receive and mostly store (webhooks, API responses) Fields you filter/sort on in every query (promote them to columns)
Event/audit data with evolving shapes Anything needing FKs, uniqueness, or NOT NULL on inner fields
User-defined/custom fields per tenant Frequently-updated hot fields (whole-row rewrite each time)

The tax you pay is schema-on-read: nothing stops a producer writing "customerId" one day and "customer_id" the next; typos become silent NULLs at read time; every consumer re-implements validation. Columns push that enforcement to write time, where errors are loud and immediate. A good hybrid: real columns for identity/joins/hot filters, one jsonb column for the long tail.

The model answer: "Why Postgres JSONB and not MongoDB?"

"Most systems have a relational core — users, orders, payments — that needs joins, foreign keys, unique constraints, and multi-row ACID transactions. Postgres gives me all of that natively, and JSONB gives me document-style flexibility inside the same database for the parts that are genuinely schema-variable, with GIN indexes making those queries fast. Choosing Mongo means either giving up or hand-rolling relational guarantees for the core data (application-level joins, weaker constraint enforcement — Mongo has multi-document transactions now, but they're bolted onto a document model rather than the foundation), and it means a second system to operate, back up, secure, and hire for. One well-understood Postgres with JSONB covers roughly 90% of 'we need a document store' cases. Where Mongo genuinely fits: the data really is document-shaped end-to-end with minimal cross-entity relationships, you want its horizontal scale-out sharding as a first-class built-in, or the team already runs it well. It's a legitimate tool — my default is just to exhaust the one-database option first, because operational simplicity compounds."

Full-text search T3

Why not just ILIKE?

SELECT * FROM articles WHERE body ILIKE '%postgres%';

A leading-wildcard ILIKE can't use a B-tree index — it's a sequential scan of every row, every time. It also has zero language awareness: no stemming ("running" won't match "run"), no ranking, no multi-word logic. Full-text search (FTS) fixes all of this inside Postgres.

tsvector + tsquery

-- to_tsvector: normalize text into stemmed lexemes with positions
SELECT to_tsvector('english', 'The quick brown foxes are jumping');
-- 'brown':3 'fox':4 'jump':6 'quick':2     (stop words dropped, words stemmed)

-- to_tsquery / plainto_tsquery / websearch_to_tsquery build the query side
SELECT websearch_to_tsquery('english', '"connection pool" -bouncer');
-- websearch_to_tsquery is friendliest for raw user input:
-- quotes = phrase, - = NOT, OR supported, and it never throws on messy input

-- The match operator
SELECT to_tsvector('english', 'jumping foxes') @@ plainto_tsquery('english', 'fox jump');
-- true

Production pattern: generated column + GIN + ranking

ALTER TABLE articles
  ADD COLUMN search tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(body,  '')), 'B')
  ) STORED;

CREATE INDEX idx_articles_search ON articles USING gin (search);

SELECT id, title,
       ts_rank(search, query) AS rank
FROM articles,
     websearch_to_tsquery('english', 'postgres replication lag') AS query
WHERE search @@ query
ORDER BY rank DESC
LIMIT 20;

The generated column keeps the vector in sync automatically (no trigger needed on modern Postgres), setweight makes title matches outrank body matches, and the GIN index makes matching fast even on millions of rows.

When to graduate to Elasticsearch/OpenSearch

Postgres FTS comfortably covers "search our articles/products/tickets" for most apps. Reach for a dedicated search engine when you need: typo tolerance and sophisticated fuzzy relevance, faceted navigation at scale, search-specific analyzers per language at depth, or search QPS high enough that you want it off the primary database. The cost is a second system plus an indexing pipeline that is eventually consistent with your source of truth — the same class of sync problem as any cache.

Related tool: the pg_trgm extension indexes trigrams so that ILIKE '%foo%' and fuzzy similarity queries can use a GIN index. FTS is for word/language search; pg_trgm is for substring and typo-ish matching (names, SKUs). They're complementary — know both exist.

CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING gin (name gin_trgm_ops);
-- Now this can use the index:
SELECT * FROM users WHERE name ILIKE '%rodrig%';

Materialized views T3

A regular VIEW is a saved query — it runs fresh every time, always current, costs full query price. A materialized view runs the query once and stores the result like a table: reads are instant, but the data is frozen until you refresh it. 

CREATE MATERIALIZED VIEW daily_revenue AS
SELECT date_trunc('day', created_at) AS day,
       count(*)          AS orders,
       sum(total_cents)  AS revenue_cents
FROM orders
WHERE status = 'COMPLETED'
GROUP BY 1;

-- You can index it like a table
CREATE UNIQUE INDEX idx_daily_revenue_day ON daily_revenue (day);

-- Plain refresh: recomputes, but LOCKS the view against reads while doing so
REFRESH MATERIALIZED VIEW daily_revenue;

-- CONCURRENTLY: readers keep working during the refresh.
-- Requirement: the matview must have a UNIQUE index (it diffs old vs new).
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue;

Classic use case: dashboards and aggregation endpoints. A revenue rollup over 100M orders might take 30 seconds live; as a matview refreshed every 10 minutes by a scheduled job, the API reads it in single-digit milliseconds and "up to 10 minutes stale" is an easy business sell. It's the same trade as caching — freshness for speed — but the "cache" lives in the database, is queryable with SQL, and refreshes atomically. Postgres has no built-in incremental refresh, so a refresh recomputes everything; if the base query itself becomes too slow to recompute, you graduate to incremental summary tables maintained by the application or triggers.

Triggers, functions & LISTEN/NOTIFY T3

The canonical example: auto-touch updated_at

CREATE OR REPLACE FUNCTION touch_updated_at()
RETURNS trigger
LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at := now();
  RETURN NEW;              -- BEFORE triggers return the (possibly modified) row
END;
$$;

CREATE TRIGGER trg_orders_touch
BEFORE UPDATE ON orders
FOR EACH ROW
EXECUTE FUNCTION touch_updated_at();

Anatomy: a function holds the logic; a trigger binds it to table events (BEFORE/AFTER × INSERT/UPDATE/DELETE × ROW/STATEMENT). Inside a row trigger, NEW and OLD are the incoming and previous row versions. An audit-log trigger is the same shape with an AFTER timing and an INSERT INTO audit_log ... body.

The "business logic in the database?" debate

The balanced senior answer: "Data integrity logic belongs in the database — constraints, FKs, CHECKs, and mechanical triggers like updated_at or audit trails, because they must hold no matter which code path writes. Domain logic — pricing rules, workflow transitions, notifications — belongs in the application: it needs unit tests, code review, debuggers, version control, and visibility, and trigger-based domain logic becomes invisible action-at-a-distance that surprises every future developer ('why did this row change? I only updated that one'). Triggers also can't easily be mocked and fire on every write, including migrations and bulk fixes. So: DB enforces what must always be true; app decides what should happen."

LISTEN/NOTIFY: lightweight pub-sub

-- Session A subscribes:
LISTEN cache_invalidation;

-- Session B (often from a trigger) publishes:
NOTIFY cache_invalidation, 'products:42';
-- or inside SQL/functions:
SELECT pg_notify('cache_invalidation', 'products:42');

Every listening connection gets the payload — no polling. Great fits: cache invalidation across app instances, "wake up the worker, there's a new job row" (poll the table on notify, so nothing is lost), lightweight live-update pushes.

LISTEN/NOTIFY is not a message queue. Delivery is at-most-once to currently connected listeners — if your app was restarting, the notification is gone. There's no persistence, no replay, no acknowledgment, payloads are small (8KB), and it needs a real dedicated connection (awkward behind transaction-mode PgBouncer). Use it as a doorbell, keep the truth in a table; for durable messaging use a real queue/outbox+CDC.

Interview Q&A

Why does Postgres need VACUUM at all?

It's the cost of MVCC. Updates and deletes never remove data in place — they create new row versions and mark old ones expired, so readers never block writers. Once no transaction can see an old version it's dead weight. VACUUM reclaims that space for reuse, updates the visibility map (which index-only scans depend on), and freezes old transaction IDs to prevent xid wraparound. No VACUUM → unbounded bloat and, eventually, forced anti-wraparound shutdown.

VACUUM vs VACUUM FULL — when would you use each?

Plain VACUUM runs alongside normal traffic and marks dead-tuple space reusable within the file; it's what autovacuum runs continuously and it's almost always sufficient. VACUUM FULL rewrites the whole table into a compact new file and returns disk to the OS — but under an ACCESS EXCLUSIVE lock that blocks all reads and writes, so on a production table it's effectively an outage. I'd only use it to recover from severe historical bloat in a maintenance window, and I'd prefer pg_repack, which achieves the rewrite online. The real fix is preventing the bloat: tune autovacuum and eliminate long-running transactions.

Autovacuum is running but a table keeps bloating. What do you check?

First, long-running or idle-in-transaction sessions in pg_stat_activity — VACUUM can only remove tuples invisible to all transactions, so one stale transaction pins the cleanup horizon for everything (fix the app; set idle_in_transaction_session_timeout as a guardrail). Also stale replication slots for dead consumers, which hold the horizon the same way. Second, whether autovacuum simply can't keep up: on huge tables the default 20% scale factor means it triggers far too late — lower autovacuum_vacuum_scale_factor per-table, and consider raising autovacuum's cost limits so it works faster. pg_stat_user_tables (n_dead_tup, last_autovacuum) tells the story.

What is transaction ID wraparound and should I be scared of it?

Postgres xids are 32-bit and wrap, so row visibility for very old rows would become ambiguous after ~2 billion transactions. VACUUM prevents this by freezing old rows — marking them permanently visible so their xid no longer matters. If freezing falls billions of transactions behind, Postgres escalates: log warnings, then aggressive anti-wraparound vacuums, and in the extreme it refuses new transactions until vacuum catches up. Scared? No — respectful. You get abundant warning, and the root cause is always the same family: autovacuum starved, disabled, or blocked by something holding the xid horizon. Monitor age(datfrozenxid) and it's a non-event.

Why are Postgres connections expensive, and what's the right pool size?

Each connection is a forked OS process with megabytes of overhead, plus per-connection work_mem during queries — so connections cost memory, context switches, and lock contention, not just a socket. Throughput peaks at a surprisingly small number of active connections; the HikariCP guidance is roughly cores×2 plus effective disks for the total across all app instances. A small pool with requests queueing briefly beats a huge pool hammering the DB — same total work, far less thrashing. If the app fleet is large or elastic, multiplex through PgBouncer rather than raising max_connections.

What does PgBouncer do, and what breaks in transaction pooling mode?

PgBouncer is a lightweight proxy that accepts thousands of client connections and funnels them through a small pool of real Postgres connections. Transaction mode — the mode that actually multiplexes well — hands a server connection to a client only for the duration of each transaction. The catch: anything that assumes a persistent session breaks, because consecutive transactions may run on different server connections — session-level SET, temp tables, session advisory locks, LISTEN, and classically server-side prepared statements (prepared on one connection, executed on another). Mitigations: newer PgBouncer versions track protocol-level prepared statements; otherwise configure the JDBC driver to avoid server-side prepares, and keep session-state tricks out of the app.

Explain streaming replication and the sync vs async trade-off.

Every change goes through the WAL; streaming replication ships that WAL continuously to replicas, which replay it and serve read-only queries. Async (the default): commits return once durable on the primary — zero added write latency, but a primary failure can lose the last moments of commits (small RPO). Sync: the commit also waits for a sync replica to confirm — RPO of zero for confirmed commits, but every commit pays a network round trip and a sick sync replica can stall all writes. Most systems run async; sync is for data where losing one committed transaction is unacceptable, sometimes applied per-transaction via synchronous_commit so only critical writes pay.

A user updates their profile, then immediately sees old data. What happened and how do you fix it?

Classic read-your-own-writes violation from replication lag: the write hit the primary, the follow-up read was routed to an async replica that hadn't replayed it yet. Fixes, cheapest first: route that user's reads to the primary for a short window after any write (a TTL flag per user/session); pin the session to the primary after a write; lag-aware routing that ejects replicas beyond a lag threshold; or, most precisely, capture the primary's WAL LSN after the write and only let replicas serve that user once they've replayed past it. And for genuinely staleness-tolerant reads (feeds, analytics), just accept the lag. In Spring the split is typically an AbstractRoutingDataSource keyed on @Transactional(readOnly=true), with one of these mitigations layered on top.

What's the difference between streaming and logical replication?

Streaming is physical: it ships WAL bytes, replicating the whole cluster exactly — same major version, read-only replicas; it's the HA and read-scaling mechanism. Logical decodes the WAL into row-change events for selected tables via publications/subscriptions — the subscriber is a normal writable database and can be a different major version, which makes it the standard tool for near-zero-downtime upgrades and selective data distribution. The same logical decoding powers CDC tools like Debezium, which stream changes into Kafka — the production-grade transport for the outbox pattern.

When would you partition a table, and when would you refuse to?

I partition when the table is genuinely large (hundreds of millions of rows) AND there's a natural key that appears in queries and drives data lifecycle — typically time. The killer feature is retention: dropping an old month's partition is instant, versus a massive DELETE that runs for hours and floods VACUUM with dead tuples; secondary wins are per-partition maintenance and partition pruning. I'd refuse when the table is modest (indexes are enough), or when queries don't filter on any candidate key — then every query fans out to all partitions and gets slower, and you've paid the complexity tax (composite unique constraints, partition management automation) for negative return. Partitioning is a lifecycle tool, not a substitute for indexing.

Why can't a partitioned table have a simple PRIMARY KEY (id)?

Each partition maintains its own indexes; Postgres has no global cross-partition index, so it can't enforce uniqueness of a column that doesn't include the partition key — two partitions could otherwise each hold id=42. Hence every unique constraint and primary key must include the partition key, e.g. PRIMARY KEY (id, created_at). Knock-on effects: no plain unique email column on a partitioned table, and foreign keys referencing it need the composite key. It's one of the main hidden costs of partitioning and a reason not to partition casually.

pg_dump vs pg_basebackup with WAL archiving — how do you choose?

pg_dump is logical: a portable snapshot of schema and data, great for smaller databases, per-table exports, dev seeding, and cross-version migration — but restore rebuilds indexes (slow at scale) and you can only recover to dump time. Physical backup (pg_basebackup or pgBackRest/WAL-G) plus continuous WAL archiving gives point-in-time recovery: restore the base copy and replay WAL to any moment — 'to 14:02, right before the bad deploy dropped the table.' Production DR means physical + WAL (near-zero RPO; RTO governed by restore/replay time), often with pg_dump alongside for portability. And a backup is only real once you've drilled restoring it.

Why choose Postgres with JSONB over MongoDB? (Give the full answer.)

Most systems have a relational core — users, orders, payments — needing joins, foreign keys, unique constraints, and multi-row ACID transactions, which Postgres provides natively. JSONB then adds document flexibility inside the same database: binary storage, containment and path operators, GIN indexes for ad-hoc shape queries, expression B-tree indexes for hot fields. Choosing Mongo instead means recreating relational guarantees at the application layer for the core data (its transactions exist but are bolted onto a document model), plus a second system to operate, back up, monitor, secure, and hire for. One well-understood Postgres covers most 'we need a document store' cases: the sparse/variable data goes in a JSONB column next to real columns. Mongo genuinely fits when the data is document-shaped end-to-end with few cross-entity relationships, when built-in scale-out sharding is a hard requirement, or when the team already operates it well. My default is exhausting the one-database option first — operational simplicity compounds.

What's the difference between -> and ->> in JSONB, and why does it matter?

-> returns jsonb; ->> returns text. It matters for every comparison: payload -> 'tier' = 'gold' compares jsonb against a string and fails or surprises, while payload ->> 'tier' = 'gold' works. For numbers you extract text and cast: (payload ->> 'qty')::int. Practical pattern: chain -> to navigate objects, end with ->> (or #>> with a path) to extract the final scalar. It also matters for indexing — an expression index must match the exact expression, operators included, that the query uses.

How do you index JSONB queries efficiently?

Two tools. For ad-hoc shape queries — 'find rows containing this sub-document' — a GIN index: default jsonb_ops supports containment (@>) and key-existence (?); jsonb_path_ops is smaller and faster but only supports @>. For a specific field you filter on constantly, a B-tree expression index on the extracted text — e.g. on ((payload -> 'customer' ->> 'tier')) — is smaller, faster, and supports range queries and ordering, which GIN can't. If a field earns an expression index it's usually also a candidate to be promoted to a real column. GIN indexes are costlier to update than B-trees, so on write-heavy tables index deliberately rather than slapping GIN on every jsonb column.

Why is Postgres full-text search better than ILIKE, and when would you move to Elasticsearch?

A leading-wildcard ILIKE is an unindexable sequential scan with zero language awareness. FTS normalizes text into stemmed lexemes (tsvector), matches structured queries (tsquery, with websearch_to_tsquery safely parsing raw user input), ranks with ts_rank, and — the key part — is served by a GIN index, so it stays fast at millions of rows. Production shape: a stored generated tsvector column with setweight for title-vs-body, plus a GIN index. I'd graduate to Elasticsearch when I need serious typo tolerance and relevance tuning, faceted search at scale, or enough search traffic that it should be off the primary — accepting the cost of a second system and an eventually-consistent indexing pipeline. For substring/fuzzy matching on names or SKUs, pg_trgm with a trigram GIN index accelerates ILIKE itself — different tool, complementary.

Materialized view vs regular view vs cache — how do you frame the trade?

A view is a saved query: always fresh, full price on every read. A materialized view stores the result: table-speed indexed reads, but stale until refreshed — and REFRESH CONCURRENTLY (which requires a unique index on the matview) lets you refresh without blocking readers. It's caching with the same fundamental trade — freshness for speed — but living in the database: queryable with SQL, joinable, atomically swapped on refresh, no serialization layer. Perfect for dashboards where 'revenue as of 10 minutes ago' is fine. Limits: no built-in incremental refresh, so each refresh recomputes everything; when that gets too heavy, move to incrementally-maintained summary tables.

Should business logic live in database triggers?

Split it by kind. Integrity and mechanical concerns — constraints, foreign keys, updated_at touching, audit trails — yes, in the database, because they must hold regardless of which code path writes. Domain logic — pricing, workflow, notifications — no: it needs tests, code review, debugging, and visibility, and trigger-hosted logic becomes invisible action-at-a-distance ('why did this row change?'), fires on every write including migrations and bulk fixes, and is hard to mock. My rule: the database enforces what must always be true; the application decides what should happen. When I do use a trigger, it's small, mechanical, and documented next to the schema migrations.

What is LISTEN/NOTIFY good for, and why isn't it a message queue?

It's built-in lightweight pub-sub: sessions LISTEN on a channel, NOTIFY pushes a small payload to all current listeners — no polling, no extra infrastructure. Good uses: cache invalidation across app instances, waking a worker when a job row is inserted. Why it's not a queue: delivery is at-most-once to currently-connected listeners only — a restarting consumer misses everything; no persistence, replay, or acks; small payloads; and it needs a dedicated real connection, which fights transaction-mode PgBouncer. The safe pattern is 'doorbell, not mailbox': the durable truth lives in a table (or an outbox consumed via CDC), NOTIFY just says 'check now.'

Your dashboard queries are slowing the primary. Walk through your options.

In escalating order. First, optimize: EXPLAIN the queries, fix indexes — often that's it. Second, precompute: materialized views refreshed on a schedule turn 30-second aggregations into millisecond reads, and dashboards tolerate staleness by nature. Third, offload: route read-only/reporting traffic to a streaming replica — lag is irrelevant for dashboards, though heavy analytics queries on a replica can conflict with WAL replay (know hot_standby_feedback exists, and its vacuum-horizon cost on the primary). Fourth, if analytics needs genuinely diverge — huge scans, columnar workloads — replicate into a proper warehouse via logical replication/CDC. I'd stop at the earliest step that meets the need.

Hands-on exercises

1. Watch bloat happen, then fix it. In a scratch database, create a table and fill it: CREATE TABLE bloaty (id int PRIMARY KEY, val text); INSERT INTO bloaty SELECT g, repeat('x', 100) FROM generate_series(1, 200000) g; Note its size with SELECT pg_size_pretty(pg_relation_size('bloaty'));. Now update every row three times (UPDATE bloaty SET val = val || 'y'; ×3) and check size and n_dead_tup in pg_stat_user_tables. Run VACUUM VERBOSE bloaty; — does the file shrink? (Mostly no — space is reclaimed for reuse.) Update all rows again and confirm the size stays flat this time. Finally, open a second session, run BEGIN; SELECT 1; and leave it idle, generate more dead tuples, and observe that VACUUM now reports tuples "not yet removable." Commit the idle transaction and vacuum again.

2. Partitioning and pruning. Create the partitioned orders table from this page with three monthly partitions and insert a few thousand rows spread across months (use generate_series with random timestamps). Run EXPLAIN SELECT count(*) FROM orders WHERE created_at >= '2026-07-01' AND created_at < '2026-07-08'; and confirm only one partition is scanned. Then run a query filtered only on customer_id and observe the Append over all partitions. Finish by "expiring" the oldest month with DROP TABLE orders_2026_05; and time it against a DELETE of equivalent rows on an unpartitioned copy.

3. JSONB operators and indexing. Create the events table from this page and insert ~100k rows with varying payloads (mix tiers, random orderIds). Write queries using ->>, #>>, @>, and ?. EXPLAIN ANALYZE the containment query payload @> '{"customer": {"tier": "gold"}}' before and after adding a GIN index (try both jsonb_ops and jsonb_path_ops and compare index sizes with pg_size_pretty(pg_relation_size(...))). Then add the expression B-tree index on the extracted tier and compare the plan for the equality query. Bonus: try WHERE payload -> 'customer' ->> 'tier' = 'gold' ORDER BY id LIMIT 10 with each index.

4. Full-text search end to end. Create an articles(id, title, body) table and load a few hundred rows (paste paragraphs, or generate filler with distinctive keywords sprinkled in). Add the stored generated tsvector column with setweight (title = A, body = B) and a GIN index. Query with websearch_to_tsquery, order by ts_rank, and verify that a keyword appearing in a title outranks the same keyword in a body. Compare EXPLAIN output against ILIKE '%keyword%'. Bonus: install pg_trgm, add a trigram index on title, and confirm ILIKE '%ostgre%' switches from a sequential scan to an index scan.
