05 — Spring Core, DI & Boot Fundamentals

The IoC container, bean lifecycle, and how Spring Boot auto-configuration really works — the mechanics behind the annotations.

Why interviewers ask this

Every Java backend interview touches Spring, but senior interviews probe one level below the annotations. Anyone can write @Autowired; the differentiator is explaining what the container actually does: how beans are discovered, in what order lifecycle callbacks fire, where proxies get woven in, why a constructor-injection cycle fails fast, and — the single most common Boot question — "how does auto-configuration work?"

Coming from React/Angular you already have the mental model: Angular's injector and React context both solve "how does a component get its collaborators without constructing them?" Spring is the same idea, industrial-strength: a container that instantiates your object graph, manages lifecycles, and decorates objects with cross-cutting behavior (transactions, security, caching) via proxies. This page builds that model from first principles and then goes deep enough for a senior bar.

Frontend bridge: Angular DI ≈ Spring DI (tokens ≈ bean types/names, providedIn: 'root' ≈ singleton scope, @Injectable ≈ @Component). React context is a poor man's DI — no lifecycle, no proxying, resolution at render time. Spring resolves the whole graph at startup and fails fast if it can't — a property you'll learn to love.

Hands-on: bootstrapping a real Spring Boot project Hands-on

You know the theory; this section is the muscle memory a React/Node background doesn't give you: how you actually create a Boot app, where code physically goes, how config is layered per environment, and how you run the thing. Nothing here is deep auto-config internals (that's later) — it's the scaffolding every real project starts from.

1. Generate the project with Spring Initializr

There is no npx create-spring-app. The canonical scaffolder is start.spring.io (Spring Initializr) — a web UI (also built into IntelliJ: New Project → Spring Boot) that generates a zip with a build file, wrapper scripts, and a main class. Pick:

  • Project: Maven (most common in interviews; Gradle is the alternative). Maven's pom.xml ≈ package.json, but declarative XML.
  • Language: Java. Spring Boot: latest 3.x (stable). Java: 21 (current LTS).
  • Packaging: Jar. Group: com.example, Artifact: app → base package com.example.app.
  • Dependencies — these are starters: curated dependency bundles, like a meta-package that pulls a whole coherent set of libs at compatible versions (think a well-chosen bundle in package.json, but version-managed for you by the Boot BOM):
Starter you tickArtifactWhat it actually pulls in

2. The main class — your entry point

package com.example.app;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class AppApplication {
    public static void main(String[] args) {
        SpringApplication.run(AppApplication.class, args);
    }
}

@SpringBootApplication is a meta-annotation that bundles three: @Configuration (this class can declare beans), @EnableAutoConfiguration (turn on Boot's auto-config), and @ComponentScan (discover your @Component/@Service/etc. classes). SpringApplication.run(...) boots the container, starts the embedded server, and returns the live ApplicationContext.

Why package placement matters: @ComponentScan here has no explicit base package, so it scans from this class's package downward — com.example.app and everything under it. Keep the main class in the root package and put all your code beneath it, and scanning "just works."

3. Standard package layout

Boot is unopinionated about internal structure, but the near-universal convention is a layered, package-by-role tree under the root. Adopt this on day one:

com.example.app
├── AppApplication.java        ← main class, sits at the root so scanning covers everything
├── controller                 ← HTTP layer: @RestController, request/response mapping only
├── service                    ← business logic, @Service, @Transactional boundaries
├── repository                 ← data access, Spring Data JPA interfaces (@Repository)
├── entity                     ← JPA @Entity classes (the DB-mapped domain model)
├── dto                        ← request/response records — the API's shape, decoupled from entities
├── config                     ← @Configuration classes, beans for third-party libs
└── exception                  ← custom exceptions + @RestControllerAdvice handlers

One line each: controller translates HTTP ↔ method calls; service holds logic and transaction boundaries; repository talks to the database; entity maps tables to objects; dto defines what crosses the wire (never expose entities directly); config wires beans you don't own; exception centralizes error-to-HTTP mapping. Coming from React, this is your components / hooks / api / types discipline, enforced by convention rather than folders of convenience.

4. application.yml essentials

Initializr gives you an empty src/main/resources/application.properties. Rename it to application.yml (YAML nests better for the tree of settings). This file is your .env + config combined — but typed and validated at startup, not stringly-typed at read time.

# src/main/resources/application.yml
server:
  port: 8080              # the embedded Tomcat port

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/appdb
    username: appuser
    password: localpassword
  jpa:
    hibernate:
      ddl-auto: update   # dev-only convenience; see pitfall below
    show-sql: true        # log generated SQL — great while learning
    properties:
      hibernate:
        format_sql: true

logging:
  level:
    org.springframework.web: INFO
    org.hibernate.SQL: DEBUG       # see the actual queries
    com.example.app: DEBUG

.properties vs .yml: identical capabilities, different syntax. server.port=8080 (flat, dot-separated) vs the nested YAML above. YAML wins for deep hierarchies and lists; properties is fine for a handful of keys. Pick one per file — don't split the same concern across both (Boot loads both if present, which just confuses people).

5. Profiles: per-environment config

You never want dev database credentials in production. Boot's answer is profiles — named config layers. Put shared settings in application.yml, then override per environment in application-{profile}.yml:

# application-dev.yml — local Postgres, verbose, auto-schema
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/appdb
    username: appuser
    password: localpassword
  jpa:
    hibernate:
      ddl-auto: update
    show-sql: true
# application-prod.yml — credentials injected from the environment, schema locked
spring:
  datasource:
    url: ${DB_URL}              # resolved from env vars at startup
    username: ${DB_USER}
    password: ${DB_PASSWORD}
  jpa:
    hibernate:
      ddl-auto: validate       # never mutate a prod schema automatically
    show-sql: false

Activate a profile in any of these ways (later ones override earlier):

# baked into application.yml (usually just the default/dev)
spring:
  profiles:
    active: dev

# or at launch — CLI system property:
mvn spring-boot:run -Dspring-boot.run.profiles=dev
java -jar app.jar --spring.profiles.active=prod

# or via env var (how you'd do it in a container / CI):
SPRING_PROFILES_ACTIVE=prod java -jar app.jar

You can also gate beans on a profile with @Profile — e.g. a fake email sender in dev, the real SES client in prod:

@Configuration
public class MailConfig {
    @Bean @Profile("dev")
    MailSender logMailSender()  { return new ConsoleMailSender(); }

    @Bean @Profile("prod")
    MailSender sesMailSender()  { return new SesMailSender(); }
}

6. Externalized config & secrets

Two ways to read config into code. @Value is the quick, one-off injection; @ConfigurationProperties is the typed, grouped, testable approach you should prefer for anything non-trivial.

// @Value — fine for a single scalar
@Service
public class TokenService {
    @Value("${security.jwt.expiry-seconds:3600}")  // :default after the colon
    private long expirySeconds;
}
// @ConfigurationProperties — a typed, validated group. Prefer this.
@ConfigurationProperties(prefix = "security.jwt")
@Validated
public record JwtProperties(
    @NotBlank String secret,          // injected from env, never hardcoded
    @Min(60) long expirySeconds
) {}
// enable with @ConfigurationPropertiesScan on the main class,
// then inject JwtProperties like any other bean.
# application.yml — bind env vars; the secret is NEVER a literal here
security:
  jwt:
    secret: ${JWT_SECRET}        # fails fast at startup if the env var is missing
    expiry-seconds: 3600

Three ways this bites theory-only devs: (1) A @Service in com.example.other — a sibling of com.example.app, not under it — is silently never scanned; you get "required a bean of type X that could not be found" at startup. Keep everything under the main class's package. (2) ddl-auto: update (or worse, create) in production lets Hibernate mutate your live schema on deploy — data loss waiting to happen. Use validate in prod and manage schema with Flyway/Liquibase. (3) Committing password: localpassword or a JWT secret to git. Real secrets come from env vars / a secrets manager and are referenced as ${...}; the committed files hold only non-secret defaults.

7. Running it

# from the project root, using the Maven wrapper Initializr generated:
./mvnw spring-boot:run           # (Windows: mvnw.cmd spring-boot:run)

# or build a self-contained "fat jar" and run it — how it ships to prod:
./mvnw clean package             # produces target/app-0.0.1-SNAPSHOT.jar
java -jar target/app-0.0.1-SNAPSHOT.jar

In an IDE you just click Run on AppApplication.main() — same thing. On startup you'll see a banner and finally a line like Started AppApplication in 2.317 seconds (process running for 2.9). That line means the container built the whole bean graph, ran migrations/init, and the embedded Tomcat is now listening on port 8080. There is no external servlet container to install or deploy a WAR into — the HTTP server ships inside your jar. This is why a Boot app runs identically on your laptop and in a container: java -jar and it's serving.

Add spring-boot-devtools (a dependency) for a faster inner loop: it watches the classpath and auto-restarts the app when you recompile — the Java equivalent of Node's nodemon / a dev-server hot reload. It's automatically disabled in the packaged jar, so it never runs in production.

Startup time and memory are real interview topics for the JVM. A senior habit: keep auto-configuration lean (only the starters you use), because every starter on the classpath means more @Conditional evaluation at boot. For latency-sensitive or serverless deploys you'd reach for spring-boot-starter-actuator's startup endpoint to see which beans cost the most, lazy-init non-critical beans (spring.main.lazy-initialization=true), or — the big lever — compile to a GraalVM native image, trading JIT warmth for sub-100ms cold starts. Don't volunteer all of this unprompted, but knowing the fat jar contains an embedded server (not a WAR on Tomcat) and that scanning happens at startup is the baseline.

What does @SpringBootApplication actually do?

It's a convenience meta-annotation combining three: @Configuration (the class may declare @Bean methods), @EnableAutoConfiguration (activates Boot's conditional auto-configuration — datasource, MVC, Jackson, etc., based on what's on the classpath), and @ComponentScan (discovers your stereotype-annotated classes). Because the bundled @ComponentScan has no explicit base package, it scans from the annotated class's package downward — which is exactly why convention puts the main class in the root package. SpringApplication.run(...) then creates and refreshes the ApplicationContext, starts the embedded server, and returns the live context.

How do you manage configuration across dev, staging, and production?

Layered profiles. Shared, non-secret defaults live in application.yml; per-environment overrides live in application-{profile}.yml and win when that profile is active. I select the profile at deploy time — usually the SPRING_PROFILES_ACTIVE env var in the container, or --spring.profiles.active on the command line — never by editing code. Secrets (DB passwords, API keys, JWT secret) are never committed: the YAML references environment variables via ${DB_PASSWORD}, injected by the platform or a secrets manager, and Boot fails fast at startup if a required one is missing. For anything more than a scalar I bind config into a typed, @Validated @ConfigurationProperties record so bad values are caught at boot, not at first use. And prod uses ddl-auto: validate with Flyway for schema, never update.

IoC & DI from first principles T1

The problem: manual wiring

Start without any framework. A service needs a repository, which needs a data source:

public class OrderService {
    private final OrderRepository repository;

    public OrderService() {
        // The service constructs its OWN dependency — tight coupling
        this.repository = new JdbcOrderRepository(
            new HikariDataSource(loadDbConfigSomehow()));
    }
}

Three problems, and they compound as the app grows:

  • Coupling to concrete classes. OrderService hard-codes JdbcOrderRepository. Swapping in a different implementation (or a different database) means editing the service.
  • Untestable. You cannot unit-test OrderService without a real database, because the constructor insists on building the real repository. There is no seam to insert a fake.
  • Duplicated wiring knowledge. Every class that needs a repository must know how to build one — connection pool config, credentials, all of it, scattered everywhere.

Step 1: Dependency Injection (the pattern, no framework)

Flip it: the class declares what it needs and someone else provides it.

public class OrderService {
    private final OrderRepository repository; // interface, not impl

    public OrderService(OrderRepository repository) { // injected
        this.repository = repository;
    }
}

// Somewhere at the top of the app — the "composition root":
DataSource ds = new HikariDataSource(config);
OrderRepository repo = new JdbcOrderRepository(ds);
OrderService service = new OrderService(repo);

This is DI without Spring — just constructors and interfaces. The service is now testable (new OrderService(fakeRepo)) and decoupled. But the composition root becomes a giant, order-sensitive wiring file that you maintain by hand. In a real app with hundreds of components, plus concerns like "wrap every repository call in a transaction," this manual approach collapses.

Step 2: Inversion of Control — the container

IoC is the broader principle: instead of your code driving object creation and lifecycle, a container does — control is inverted ("don't call us, we'll call you"). DI is the mechanism the container uses to deliver dependencies. Spring's container:

  • Scans/reads bean definitions (recipes: class, scope, dependencies, init/destroy callbacks),
  • Instantiates beans in dependency order (it topologically sorts the graph for you),
  • Injects each bean's dependencies,
  • Runs lifecycle callbacks and wraps beans in proxies where cross-cutting behavior is declared (@Transactional, @Cacheable, security),
  • Hands you a fully-wired, ready object graph — and destroys it cleanly on shutdown.

The container interface you interact with is the ApplicationContext:

@SpringBootApplication
public class ShopApplication {
    public static void main(String[] args) {
        ApplicationContext ctx = SpringApplication.run(ShopApplication.class, args);
        // You almost never touch ctx directly — beans get injected into beans.
        // But it's there: ctx.getBean(OrderService.class)
    }
}

A senior answer defines both terms precisely: "IoC is the principle — the framework controls object creation and lifecycle, not application code. DI is the dominant technique for implementing it — dependencies are supplied from outside rather than constructed internally. Spring's ApplicationContext is the IoC container: it builds the entire bean graph at startup, in dependency order, applying lifecycle callbacks and proxy-based AOP, and fails fast if the graph can't be satisfied." Mentioning fail-fast startup and proxies signals depth.

Beans are just plain Java objects managed by the container. There is nothing special about the class itself — the same class can be a bean in one app and manually constructed in a test. "Bean" describes a relationship with the container, not a kind of object.

Constructor vs setter vs field injection T1

Spring supports three injection styles. This is a guaranteed interview topic because the "wrong" one (@Autowired on fields) is what most tutorials show.

// 1. FIELD injection — avoid
@Service
public class A {
    @Autowired private OrderRepository repo; // reflection magic
}

// 2. SETTER injection — for optional/reconfigurable deps only
@Service
public class B {
    private MetricsCollector metrics;
    @Autowired(required = false)
    public void setMetrics(MetricsCollector metrics) { this.metrics = metrics; }
}

// 3. CONSTRUCTOR injection — the default choice
@Service
public class OrderService {
    private final OrderRepository repo;
    private final PaymentGateway gateway;

    // Since Spring 4.3: no @Autowired needed when there is exactly ONE constructor
    public OrderService(OrderRepository repo, PaymentGateway gateway) {
        this.repo = repo;
        this.gateway = gateway;
    }
}

Field injection pitfalls to name in an interview: (1) you can't make the field final; (2) unit tests must either boot a container or use reflection to poke private fields; (3) the class silently accumulates dependencies because nothing hurts; (4) NullPointerException if someone instantiates the class with new outside Spring; (5) it hides a violation of the Single Responsibility Principle that a 9-parameter constructor would make obvious.

The Lombok idiom

In real codebases you'll rarely see the constructor written by hand. Lombok generates it:

@Service
@RequiredArgsConstructor // generates a constructor for all final fields
public class OrderService {
    private final OrderRepository repo;
    private final PaymentGateway gateway;
    // no constructor code; Lombok writes it at compile time,
    // Spring sees a single constructor and injects into it
}

This works because of two rules composing: Lombok's @RequiredArgsConstructor generates a constructor taking every private final field, and Spring auto-injects into a class's single constructor without needing @Autowired. Declaring a dependency becomes one line: add a final field. This is the de-facto standard idiom in Boot 3.x codebases.

When is setter injection legitimate? For genuinely optional dependencies or ones that can be swapped at runtime (rare). Field injection survives mostly in test classes (@Autowired on a @SpringBootTest field is fine — the container owns the test instance) and legacy code. In production code, default to constructor injection and say so plainly in interviews.

Stereotype annotations & component scanning T1

How does Spring find your classes? Two mechanisms: component scanning (annotate the class, Spring discovers it) and explicit @Bean methods (covered later). Scanning uses stereotype annotations:

AnnotationLayerExtra behavior beyond @Component
@ComponentGenericNone — the base stereotype; all others are meta-annotated with it.
@ServiceBusiness logicNone technically today — pure semantics/documentation (and a hook for future tooling & AOP pointcuts targeting the annotation).
@RepositoryPersistenceException translation: a PersistenceExceptionTranslationPostProcessor proxies the bean so vendor-specific exceptions (JDBC SQLException wrapped, Hibernate exceptions) are translated into Spring's unified, unchecked DataAccessException hierarchy (e.g. DuplicateKeyException, DataIntegrityViolationException).
@ControllerWeb (MVC views)Detected by Spring MVC as a handler; methods return view names by default.
@RestControllerWeb (JSON APIs)= @Controller + @ResponseBody: return values are serialized to the response body (via Jackson) instead of resolved as view names.

The exception-translation point on @Repository is the classic "do you actually know the difference?" trap. Interviewers ask "is @Service different from @Component?" — correct senior answer: "Functionally identical today; it's semantic. @Repository is the one with real behavior: persistence exception translation into the DataAccessException hierarchy. That hierarchy matters because it's unchecked and vendor-neutral — service code can catch DuplicateKeyException without knowing whether JDBC or JPA is underneath."

Component scanning & the classic "bean not found" pitfall

@SpringBootApplication  // includes @ComponentScan with NO basePackages set
public class ShopApplication { ... }

When @ComponentScan has no explicit base package, it scans from the package of the annotated class downward. Boot convention: put the main class in the root package (e.g. com.acme.shop) so everything under com.acme.shop.* is scanned automatically.

com.acme.shop.ShopApplication          ← scan root
com.acme.shop.order.OrderService       ✅ found
com.acme.shop.payment.PaymentGateway   ✅ found
com.acme.billing.InvoiceService        ❌ NOT SCANNED — sibling package!

The #1 beginner-to-Spring bug: a bean in a package that is not under the main class's package silently isn't registered, and you get NoSuchBeanDefinitionException / "required a bean of type X that could not be found" at startup. Fixes: move the class (preferred — keep the convention), or explicitly widen: @SpringBootApplication(scanBasePackages = {"com.acme.shop", "com.acme.billing"}). The same applies to JPA: @EntityScan and @EnableJpaRepositories also default to the main class's package.

Mechanically, scanning does not load every class with a ClassLoader — that would be slow and trigger static initializers. Spring reads the .class files' bytecode metadata with ASM (ClassPathScanningCandidateComponentProvider) to check for stereotype annotations, and only then registers BeanDefinitions. A nice detail to drop when asked "how does scanning work?"

The bean lifecycle, in order T1

Interviewers love "walk me through the bean lifecycle" because the order reveals whether you understand where proxies come from and why @PostConstruct can't see the proxy. Learn it as a sequence:

            CONTAINER STARTUP
┌──────────────────────────────────────────────────────────┐
│ 1. Bean DEFINITIONS loaded (scanning, @Bean methods,      │
│    auto-configuration) → BeanDefinition registry          │
│ 2. BeanFactoryPostProcessors run                          │
│    (edit definitions BEFORE any bean exists —             │
│     e.g. property placeholder resolution)                 │
└──────────────────────────────────────────────────────────┘
            FOR EACH BEAN (dependency order)
┌──────────────────────────────────────────────────────────┐
│ 3. INSTANTIATION        constructor runs                  │
│                         (constructor deps injected here)  │
│ 4. POPULATE             @Autowired fields / setters set   │
│ 5. AWARE callbacks      BeanNameAware, BeanFactoryAware,  │
│                         ApplicationContextAware           │
│ 6. BeanPostProcessor    postProcessBEFOREInitialization   │
│ 7. INIT callbacks       ① @PostConstruct                  │
│                         ② InitializingBean                │
│                            .afterPropertiesSet()          │
│                         ③ @Bean(initMethod = "...")       │
│ 8. BeanPostProcessor    postProcessAFTERInitialization    │
│    ★ PROXIES CREATED HERE (@Transactional, @Async,       │
│      @Cacheable, security) — the container may return     │
│      a DIFFERENT object than your instance                │
│ 9. READY — bean is in the singleton cache, serving calls  │
└──────────────────────────────────────────────────────────┘
            CONTAINER SHUTDOWN (context.close())
┌──────────────────────────────────────────────────────────┐
│ 10. DESTROY callbacks   ① @PreDestroy                     │
│                         ② DisposableBean.destroy()        │
│                         ③ @Bean(destroyMethod = "...")    │
└──────────────────────────────────────────────────────────┘
import jakarta.annotation.PostConstruct;   // Spring 6 / Boot 3: jakarta.*, NOT javax.*
import jakarta.annotation.PreDestroy;

@Service
@RequiredArgsConstructor
public class CacheWarmer {
    private final ProductRepository products;

    @PostConstruct          // step 7①: deps are injected, safe to use them
    void warmUp() {
        products.findTopSellers().forEach(this::preload);
    }

    @PreDestroy             // step 10①: graceful cleanup on shutdown
    void flush() { /* release resources, drain queues */ }
}

Why order matters — two classic bugs:

  • Constructor uses an @Autowired field → NPE. Field/setter population (step 4) happens after the constructor (step 3). If you must run logic after injection, use @PostConstruct — or better, use constructor injection so everything is available in the constructor.
  • Calling a @Transactional method from @PostConstruct on the same bean may run without a transaction. The transactional proxy is created at step 8, after @PostConstruct (step 7) — and even afterwards, self-invocation (this.method()) bypasses the proxy entirely. Proxies only intercept calls that arrive from outside through the proxy reference.

The one-sentence senior summary: "The key insight in the lifecycle is that BeanPostProcessor.postProcessAfterInitialization is where AOP proxies wrap the bean — so the object other beans receive may not be your class at all, but a CGLIB subclass or JDK dynamic proxy delegating to it. That single fact explains @Transactional self-invocation failures, why final methods can't be advised by CGLIB, and why @PostConstruct runs on the raw target." If you can say that fluently, this topic is done.

Preference order for init/destroy hooks in modern code: @PostConstruct/@PreDestroy (JSR-250, portable, no Spring import) > @Bean(initMethod/destroyMethod) (for third-party classes you can't annotate) > InitializingBean/DisposableBean interfaces (couple your class to Spring — avoid). Boot registers a JVM shutdown hook so @PreDestroy fires on normal SIGTERM — this is how graceful shutdown of connection pools works.

Bean scopes T1

ScopeInstancesNotes
singleton (default)One per containerCached in the singleton registry; shared by every injection point. Must be stateless or thread-safe — it's hit by all request threads concurrently.
prototypeNew instance per getBean()/injectionContainer creates & injects, then forgets it: no destruction callbacks (@PreDestroy never fires), caller manages the lifecycle.
requestOne per HTTP requestWeb-only. Needs a scoped proxy to be injected into singletons.
sessionOne per HTTP sessionWeb-only; e.g., a shopping cart. Same proxy caveat.
applicationOne per ServletContextRare; like singleton but scoped to the servlet context.

Singleton = one instance shared across all threads. Mutable instance fields on a singleton @Service are a race condition. This is the Spring equivalent of module-level mutable state in a Node server. Keep singletons stateless; per-request state belongs in method parameters, request-scoped beans, or (sparingly) ThreadLocal. Also: Spring's singleton is per container, not the GoF per-JVM-per-classloader singleton — worth distinguishing out loud.

The singleton-holding-prototype problem

A favorite interview trap. What's wrong here?

@Component
@Scope("prototype")
public class ReportBuilder { /* mutable, stateful */ }

@Service
@RequiredArgsConstructor
public class ReportService {          // singleton
    private final ReportBuilder builder;  // ⚠️ injected ONCE, at startup

    public Report generate() {
        return builder.addHeader().addRows().build(); // same instance every call!
    }
}

Injection happens once, when the singleton is created. The "prototype" is effectively promoted to a singleton — and if it's stateful, requests now share (and corrupt) state. The fix is to ask the container for a fresh instance per use:

@Service
@RequiredArgsConstructor
public class ReportService {
    private final ObjectProvider<ReportBuilder> builders; // lazy handle, not the bean

    public Report generate() {
        ReportBuilder builder = builders.getObject(); // NEW instance each call
        return builder.addHeader().addRows().build();
    }
}

Alternatives: ObjectProvider<T> (preferred — also has getIfAvailable(), stream()), @Lookup method injection (Spring subclasses your class via CGLIB and overrides the method), or a scoped proxy. Honestly, in modern code you'd often skip the prototype bean entirely and just new ReportBuilder() — not everything needs to be a bean; only objects that need injection, lifecycle, or proxying do.

Web scopes need scoped proxies

The same shape of problem: injecting a request-scoped bean into a singleton. At startup there is no request, so there's nothing to inject. The fix is a proxy placeholder:

@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST,
       proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext {
    private String tenantId;   // per-request state, safe here
    // getters/setters
}

@Service
@RequiredArgsConstructor
public class PricingService {              // singleton
    private final RequestContext requestContext; // actually a CGLIB proxy

    public Price priceFor(String sku) {
        // each call: proxy looks up the bean bound to the CURRENT request thread
        return priceFor(sku, requestContext.getTenantId());
    }
}

The singleton holds one stable proxy object; on every method call the proxy resolves the real request-scoped instance from the current request (via a ThreadLocal-backed RequestContextHolder). Conceptually similar to how a React context consumer reads the nearest provider's current value at call time rather than capturing it once.

Component scanning works for your classes. For third-party classes (a Stripe client, an ObjectMapper tweak, a connection pool) you can't add @Component to code you don't own — instead you declare beans with factory methods:

Full vs lite mode: the CGLIB trick

A subtle, senior-level detail. What does this print — one client or two?

@Configuration
public class PaymentConfig {
    @Bean StripeClient stripeClient() { return new StripeClient(); }

    @Bean PaymentGateway gateway()   { return new StripePaymentGateway(stripeClient()); }
    @Bean RefundService  refunds()   { return new RefundService(stripeClient()); }
    // stripeClient() called twice — two instances?
}

One instance. @Configuration classes are subclassed by CGLIB at runtime ("full mode"): the proxy intercepts @Bean method calls and returns the cached container-managed singleton instead of executing the method body again. Inter-bean method calls behave like getBean().

// Opting out: "lite mode"
@Configuration(proxyBeanMethods = false)
public class PaymentConfig {
    // No CGLIB proxy → faster startup, less memory, GraalVM-native-friendly.
    // BUT direct @Bean method calls now execute the body → NEW instance each call.
    // Rule for lite mode: never call one @Bean method from another;
    // take dependencies as method parameters instead (as in the first example).
}

Spring Boot's own auto-configuration classes all use proxyBeanMethods = false for startup speed — and they always pass dependencies as parameters. The parameter style is the safe habit: it works identically in both modes. @Bean methods inside a plain @Component are also "lite" — a source of one-instance-vs-many bugs when people copy code around.

Circular dependencies

A depends on B, B depends on A:

@Service class OrderService   { public OrderService(CustomerService c) {...} }
@Service class CustomerService{ public CustomerService(OrderService o) {...} }

// Startup: BeanCurrentlyInCreationException —
// "The dependencies of some of the beans in the application context form a cycle"
  • Constructor injection: fails fast at startup. Neither constructor can complete without the other's finished instance. This is a feature — the cycle is a design smell (the two services are entangled; there's probably a hidden third concept).
  • Setter/field injection can tolerate cycles via Spring's early-reference mechanism (it exposes a half-built bean from an "early singleton" cache so the other side can grab a reference before initialization finishes). You get two objects wired together, but one of them may observe the other before it's fully initialized — subtle bugs.
  • Boot 2.6+ rejects circular references by default even where technically resolvable (spring.main.allow-circular-references=false is the default). The team's message: cycles are bugs; fix the design.

Fix hierarchy (best to worst):

  1. Redesign. Extract the shared logic into a third bean both depend on; or invert one direction with events (ApplicationEventPublisher — B reacts to A's event instead of A calling B).
  2. @Lazy on one side. public OrderService(@Lazy CustomerService c) injects a proxy that resolves the real bean on first use, breaking the cycle at construction time. Pragmatic, but the cycle still exists conceptually.
  3. Setter injection for one edge (plus re-enabling allow-circular-references). Last resort; hides the smell.

Interview framing: "I treat a circular-dependency failure as the container doing me a favor. Nine times out of ten there's a missing abstraction — I extract it, or I decouple one direction with domain events. @Lazy is my tactical escape hatch when I can't refactor immediately, and I never just flip allow-circular-references=true globally."

Disambiguation: @Qualifier, @Primary, collections, @Conditional, @Profile T2

Spring injects by type. When one interface has multiple implementations, the container can't choose and throws NoUniqueBeanDefinitionException — "expected single matching bean but found 2". Your disambiguation toolbox:

public interface NotificationSender { void send(Notification n); }

@Component("email") class EmailSender implements NotificationSender {...}
@Component("sms")   class SmsSender   implements NotificationSender {...}
@Component("push")  class PushSender  implements NotificationSender {...}

// Option 1: @Primary — the default when nobody specifies
@Primary @Component class EmailSender implements NotificationSender {...}

// Option 2: @Qualifier — the injection point picks by name
public AlertService(@Qualifier("sms") NotificationSender sender) {...}

@Primary sets a global default; @Qualifier is a per-injection-point override (and wins over @Primary). For type safety you can define custom qualifier annotations instead of strings.

Injecting all implementations — the strategy-pattern idiom

Often you don't want to choose — you want all of them, selected at runtime. Spring injects collections of a type, and this is the cleanest strategy-pattern implementation in Java:

public interface PaymentMethod {
    String key();                       // "card", "paypal", "wallet"
    Receipt charge(Order order);
}

@Service
public class CheckoutService {
    private final Map<String, PaymentMethod> methods;

    // List<PaymentMethod> also works (ordered by @Order); with Map,
    // keys are BEAN NAMES — or build your own keyed map:
    public CheckoutService(List<PaymentMethod> all) {
        this.methods = all.stream()
            .collect(Collectors.toMap(PaymentMethod::key, m -> m));
    }

    public Receipt checkout(Order order) {
        PaymentMethod m = methods.get(order.paymentType());
        if (m == null) throw new UnsupportedPaymentException(order.paymentType());
        return m.charge(order);
    }
}

Adding a new payment method = adding one new @Component class. No switch statement to edit, no registration code — the container assembles the registry. This is Open/Closed via DI, and it's a fantastic thing to bring up unprompted when asked "how has DI changed how you design code?"

BeanFactoryPostProcessor vs BeanPostProcessor — how @Autowired itself works

Two extension points, distinguished by what they operate on and when:

  • BeanFactoryPostProcessor runs once, after definitions are loaded but before any bean is instantiated. It mutates metadata (BeanDefinitions). Example: PropertySourcesPlaceholderConfigurer resolving ${...} placeholders in definitions.
  • BeanPostProcessor runs for every bean instance, around initialization (steps 6 and 8 in the lifecycle). It can modify or replace the instance — this is the AOP hook.

The punchline interviewers fish for: Spring's own annotations are implemented as post-processors, not compiler magic.

  • AutowiredAnnotationBeanPostProcessor — scans each bean's class for @Autowired/@Value members and injects them during population.
  • CommonAnnotationBeanPostProcessor — handles @PostConstruct/@PreDestroy (jakarta annotations).
  • AnnotationAwareAspectJAutoProxyCreator — in postProcessAfterInitialization, checks whether any advice (transactions, caching, @Aspects) applies to the bean; if so, returns a proxy wrapping it. This is literally where @Transactional becomes behavior.
// A trivial custom BeanPostProcessor — e.g., timing bean creation
@Component
public class StartupProfiler implements BeanPostProcessor {
    @Override
    public Object postProcessAfterInitialization(Object bean, String name) {
        if (bean.getClass().getPackageName().startsWith("com.acme")) {
            log.debug("initialized bean {} ({})", name, bean.getClass().getSimpleName());
        }
        return bean;  // return a different object here and you've "become" AOP
    }
}

Spring Boot auto-configuration mechanics T1

This is the single most-asked Spring Boot interview question. "How does Spring Boot know to configure a DataSource / Tomcat / Jackson without you writing any config?" A senior answer walks the full chain. Here it is.

1. @SpringBootApplication is three annotations

@SpringBootApplication
// ≡ (plus a couple of meta-details)
@SpringBootConfiguration      // specialization of @Configuration
@EnableAutoConfiguration      // ← the magic switch
@ComponentScan                // scan from this package down (with default filters)

2. Where the candidate list comes from

@EnableAutoConfiguration imports AutoConfigurationImportSelector, which reads a plain-text list of candidate configuration classes from every jar on the classpath:

# In each auto-configuring jar (e.g. spring-boot-autoconfigure.jar):
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports

# contents: one fully-qualified class name per line, e.g.
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
org.springframework.boot.autoconfigure.jackson.JacksonAutoConfiguration
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
...

History for older codebases: before Boot 2.7 this list lived under the org.springframework.boot.autoconfigure.EnableAutoConfiguration key in META-INF/spring.factories. Boot 2.7 introduced the dedicated AutoConfiguration.imports file, and Boot 3.0 removed spring.factories support for auto-configurations entirely. Knowing both signals you've seen real migrations.

3. Candidates are guarded by conditions

Roughly 150 candidate classes are listed, but most evaluate to nothing. Each is a @Configuration class fenced with @Conditional* guards, so it only takes effect when the classpath and existing beans say it should:

// Simplified sketch of the CANONICAL example — DataSourceAutoConfiguration
@AutoConfiguration(before = SqlInitializationAutoConfiguration.class)
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@EnableConfigurationProperties(DataSourceProperties.class)  // binds spring.datasource.*
public class DataSourceAutoConfiguration {

    @Configuration(proxyBeanMethods = false)
    @ConditionalOnClass(HikariDataSource.class)       // Hikari on classpath?
    @ConditionalOnMissingBean(DataSource.class)        // user hasn't defined one?
    @ConditionalOnProperty(name = "spring.datasource.type",
        havingValue = "com.zaxxer.hikari.HikariDataSource", matchIfMissing = true)
    static class Hikari {
        @Bean
        HikariDataSource dataSource(DataSourceProperties properties) {
            // url, username, password, pool size — all from application.yml
            return createDataSource(properties, HikariDataSource.class);
        }
    }
}

Read the guards as a sentence: "IF the JDBC classes are on the classpath (you added spring-boot-starter-data-jpa or -jdbc) AND Hikari is present AND you haven't declared your own DataSource bean, THEN create a Hikari pool configured from spring.datasource.* properties."

4. Backing off: your @Bean beats the default

@Configuration
public class MyDataConfig {
    @Bean
    DataSource dataSource() {
        // Your custom pool. Because of @ConditionalOnMissingBean on the
        // auto-config, Boot's default simply DOES NOT REGISTER. No override
        // ceremony, no config flag — the default "backs off".
        return myExoticDataSource();
    }
}

This is the core design contract of Boot: convention with graceful surrender. Auto-configuration provides opinionated defaults; user beans always win. To make ordering work (user configs must be processed before auto-configs so @ConditionalOnMissingBean sees your beans), auto-configurations are processed last, and order among themselves via @AutoConfiguration(before/after = ...).

5. Debugging: the condition evaluation report

DevTools, runners & startup tuning T3

DevTools

spring-boot-devtools (a runtime-scoped dev dependency) gives you: automatic restart when classes change — fast because it uses two classloaders (a base loader for unchanging jars, a restart loader for your classes; only the second is thrown away), LiveReload triggering for the browser, and dev-friendly property defaults (caches off). Startup tasks: CommandLineRunner vs ApplicationRunner

Both are callbacks that run once, after the context is fully ready (all beans wired, server started) — the right home for seed data, cache warming, or startup validation. The only difference is the argument type:

@Component
@Order(1)   // multiple runners? order them explicitly
@RequiredArgsConstructor
public class DataSeeder implements CommandLineRunner {
    private final UserRepository users;

    @Override
    public void run(String... args) {          // raw args

}

@Component
public class FlagsPrinter implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) { // PARSED args
        if (args.containsOption("reindex")) {      // --reindex=products
            reindex(args.getOptionValues("reindex"));
        }
    }
}

An exception thrown from a runner aborts startup — sometimes exactly what you want (fail fast on bad state), sometimes a paging incident. Also don't confuse runners with @PostConstruct: @PostConstruct runs per-bean mid-wiring; runners run once after everything is up, so they can safely use any bean, transactions and all.

Interview Q&A

What is Inversion of Control, and how does it differ from Dependency Injection?

IoC is the principle: the framework, not application code, controls object creation, wiring, and lifecycle. DI is the dominant technique for implementing IoC — objects declare dependencies (ideally via constructor) and the container supplies them. Spring's ApplicationContext is the IoC container: it loads bean definitions, builds the whole graph in dependency order at startup, applies lifecycle callbacks and proxy-based AOP, and fails fast if the graph can't be satisfied. IoC also shows up beyond DI — template callbacks and lifecycle hooks are IoC too.

Why is constructor injection preferred over field injection?

Constructor injection allows final fields (immutability and safe publication), makes required dependencies explicit in the type's API, guarantees the object is never observable half-initialized, and keeps the class testable with plain new — no container or reflection utilities. It also fails fast on circular dependencies, surfacing design smells, and a bloated constructor creates visible pressure to refactor. Field injection hides dependencies, requires reflection to test, can't be final, and NPEs if the class is instantiated outside Spring. In practice: @RequiredArgsConstructor + final fields; since Spring 4.3 a single constructor needs no @Autowired.

How does @Autowired actually work under the hood?

It's not compiler magic — it's a BeanPostProcessor. AutowiredAnnotationBeanPostProcessor inspects each bean's class during the population phase, finds annotated members/constructors, resolves matching beans from the factory by type (then qualifier/name), and injects them reflectively. That's also why the mechanism is extensible — @PostConstruct is handled by CommonAnnotationBeanPostProcessor, and AOP proxying by another post-processor. Knowing that Spring's annotations are implemented via the container's own extension points is the senior-level framing.

Difference between @Component, @Service, @Repository, and @Controller?

All are meta-annotated with @Component and picked up by component scanning. @Service is purely semantic today. @Repository adds real behavior: persistence exception translation — a post-processor proxies it so vendor-specific exceptions (SQL/Hibernate) become Spring's unchecked, vendor-neutral DataAccessException hierarchy (DuplicateKeyException, etc.). @Controller is detected by MVC as a request handler; @RestController adds @ResponseBody, so return values are written to the response body as JSON instead of resolved as views.

Walk me through the Spring bean lifecycle.

Definitions are loaded first (scanning, @Bean methods, auto-config), then BeanFactoryPostProcessors can modify definitions before anything is instantiated. Per bean, in dependency order: constructor (instantiation, constructor deps injected) → property population (@Autowired fields/setters) → Aware callbacks → BeanPostProcessor.postProcessBeforeInitialization → init callbacks in order @PostConstruct, afterPropertiesSet(), custom init-method → postProcessAfterInitialization, where AOP proxies are created → bean ready. On context close: @PreDestroy, destroy(), custom destroy-method. The key insight is proxy creation at post-init: it explains why @PostConstruct sees the raw object and why self-invocation bypasses @Transactional.

Why must singleton beans be stateless, and what scope alternatives exist?

Singleton (the default) means one instance per container shared by every consumer — in a web app that instance is hit by many request threads concurrently, so mutable instance fields are race conditions. Keep state in method parameters, the database, or appropriately scoped beans. Alternatives: prototype (new instance per retrieval — but the container doesn't manage its destruction), and web scopes request/session/application, which need scoped proxies to be injectable into singletons. I'd also note Spring's singleton is per-container, not the GoF per-JVM singleton.

What happens when you inject a prototype bean into a singleton, and how do you fix it?

Injection happens once, at singleton creation — so the "prototype" is captured and reused forever, effectively becoming a singleton; if it's stateful, requests share state. Fixes: inject ObjectProvider<T> and call getObject() per use (preferred), a @Lookup method (CGLIB overrides it to fetch a fresh bean), or a scoped proxy. Or step back: if the object is just a stateful helper needing no injection, don't make it a bean — new it.

How does a request-scoped bean get injected into a singleton that's created before any request exists?

Via a scoped proxy (proxyMode = ScopedProxyMode.TARGET_CLASS). The singleton receives a stable CGLIB proxy at startup; on each method call the proxy resolves the real instance bound to the current request from RequestContextHolder (ThreadLocal-backed). Without the proxy you'd get a startup failure because there's no request scope active during wiring.

Explain @Configuration full mode vs lite mode.

By default (proxyBeanMethods = true, "full mode") a @Configuration class is subclassed by CGLIB; calls between @Bean methods are intercepted and return the cached container singleton, preserving singleton semantics. With proxyBeanMethods = false ("lite mode") there's no proxy — faster startup, native-image friendly — but a direct @Bean method call executes the body and creates a new instance. Rule: in lite mode (which all of Boot's own auto-configurations use), never call @Bean methods directly; declare dependencies as method parameters. @Bean methods on plain @Component classes are always lite.

How do you handle a circular dependency between two beans?

With constructor injection it fails fast at startup (BeanCurrentlyInCreationException) — which I treat as useful feedback: a cycle usually means a missing abstraction. Best fix is redesign: extract the shared piece into a third bean, or break one direction with application events. Tactical fix: @Lazy on one constructor parameter injects a resolve-on-first-use proxy. Last resort: setter injection for one edge plus spring.main.allow-circular-references=true — which Boot 2.6+ disables by default precisely because early-reference wiring can expose a half-initialized bean.

How does Spring choose when multiple beans implement the same interface?

It can't by type alone — NoUniqueBeanDefinitionException. Resolution: @Primary marks a global default; @Qualifier("name") at the injection point overrides it; a parameter/field name matching a bean name is a fallback tiebreaker. Alternatively inject the whole set — List<T> or Map<String,T> (keyed by bean name) — and select at runtime, which is the idiomatic strategy-pattern implementation: adding an implementation is just adding a class, no dispatch code to edit.

How does Spring Boot auto-configuration work? (the big one)

@SpringBootApplication includes @EnableAutoConfiguration, which imports AutoConfigurationImportSelector. The selector reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports from every jar (pre-2.7: spring.factories; removed for auto-config in Boot 3) to get candidate @AutoConfiguration classes. Each is fenced with conditions: @ConditionalOnClass reacts to what starters put on the classpath, @ConditionalOnMissingBean makes defaults back off when the user defines their own bean, @ConditionalOnProperty gates on config. They're processed after user configuration and ordered via @AutoConfiguration(before/after), binding settings via @ConfigurationProperties. Example: DataSourceAutoConfiguration creates a Hikari pool from spring.datasource.* iff JDBC and Hikari are on the classpath and no user DataSource exists. Debug with debug=true → condition evaluation report.

How do you override a bean that auto-configuration provides?

Just declare your own @Bean of that type. Auto-configurations guard their defaults with @ConditionalOnMissingBean and are evaluated after user configuration, so the default detects your bean and never registers — "backing off." No flags or ceremony. If I want an entire auto-configuration gone (not just one bean), I exclude it: @SpringBootApplication(exclude = ...) or spring.autoconfigure.exclude. And I verify with the debug=true report or the actuator /conditions endpoint.

What is a Spring Boot starter, really?

Essentially a code-less pom: a curated, version-coherent bundle of dependencies for one capability. spring-boot-starter-web pulls in Boot core + Jackson + embedded Tomcat + spring-web/webmvc. Starters work with auto-configuration as a pair: the starter puts classes on the classpath; @ConditionalOnClass in auto-config detects them and configures beans. Versions come from the Boot BOM so the whole matrix is integration-tested — you swap components by exclusion (Tomcat → Jetty) rather than by version wrangling.

Explain Spring Boot's configuration precedence order.

Later/outer sources override earlier/inner ones. The list to give, highest first: command-line args → SPRING_APPLICATION_JSON → Java system properties → OS environment variables → profile-specific config files outside the jar → non-profile files outside → profile-specific inside the jar → application.yml inside → @PropertySource → coded defaults. Two rules of thumb cover 95%: "command line beats env vars beat files," and "profile-specific beats the base file." Plus relaxed binding: SPRING_DATASOURCE_URL binds to spring.datasource.url, making 12-factor env-based config free.

@ConfigurationProperties vs @Value — when do you use each?

@Value for a genuinely one-off scalar. For any coherent group: a @ConfigurationProperties record with a prefix — typed binding of the whole tree (including Durations, lists, nested objects), relaxed binding, IDE metadata, and crucially Bean-Validation via @Validated so bad config fails startup, not a 3 a.m. request. Records give immutable constructor binding in Boot 3, and the properties object is trivially constructible in tests. @Value's one unique power is SpEL expressions.

Why does "bean not found" happen for a class that clearly has @Service on it?

Almost always a component-scan miss: @SpringBootApplication's implied @ComponentScan starts at the main class's package and scans downward only. A @Service in a sibling or parent package is never registered. Fix by moving the class under the root package (preferred convention) or widening with scanBasePackages. Related variants: JPA entities/repositories outside the default packages need @EntityScan/@EnableJpaRepositories; or the bean exists but is fenced off by a @Profile/@Conditional that didn't match — the debug=true report shows which.

Why does calling a @Transactional method from another method of the same class not start a transaction?

Because @Transactional is implemented by a proxy created around the bean at post-initialization. Other beans hold the proxy, so external calls are intercepted; but inside the class, this.method() is a direct call on the raw target — the proxy never sees it, so no transactional advice runs. Same story for @Cacheable and @Async. Fixes: move the transactional method to a separate bean (best — usually a cohesion smell anyway), or self-inject the proxy / use TransactionTemplate programmatically. Root cause in one phrase: Spring AOP is proxy-based, not bytecode weaving.

BeanFactory vs ApplicationContext?

BeanFactory is the core container SPI — definitions, creation, lookup, lazy by default. ApplicationContext extends it and is what applications actually use: it auto-registers BeanPostProcessors, adds event publishing, i18n MessageSource, the Environment/property abstraction, resource loading — and eagerly pre-instantiates singletons so misconfiguration fails at startup rather than first use. Boot picks the concrete type by app flavor (e.g. AnnotationConfigServletWebServerApplicationContext). You'd only touch a bare BeanFactory when building framework-level tooling.

What are Spring application events, and when would you use them over a direct call?

In-process, typed pub/sub: inject ApplicationEventPublisher, publish any object (records work well), and any bean with a matching @EventListener method receives it — dispatched by type. Use them to decouple side effects from core flows (order placed → send email, update analytics) so the publisher doesn't accumulate dependencies on every consumer, and new reactions are added without touching the publisher. Caveats: listeners are synchronous and in the publisher's thread — and transaction — by default; use @Async for off-thread, and @TransactionalEventListener(AFTER_COMMIT) so side effects only fire when the data actually committed. For cross-service communication you'd graduate to a real broker.

How do you secure Actuator endpoints in production?

Default HTTP exposure is health-only; keep the allowlist deliberate via management.endpoints.web.exposure.include — never "*" facing the internet, since /env and /heapdump can leak secrets. Standard layers: expose only health/info/metrics/prometheus; move actuator to a separate management port firewalled to the cluster; require a role via Spring Security (EndpointRequest.toAnyEndpoint() matcher) for anything sensitive; show-details: when_authorized on health. For Kubernetes, use the dedicated liveness/readiness health groups as probe targets.

What actually happens inside SpringApplication.run()?

High level: deduce application type from the classpath (servlet vs reactive vs none) → prepare the Environment (merge property sources, resolve profiles) → create the matching ApplicationContext → register bean definitions from scanning, user configuration, and auto-configuration imports → refresh(): run factory post-processors and condition evaluation, instantiate all singletons through the bean lifecycle, and — for a web app — create and start the embedded server and register the DispatcherServlet as part of refresh → publish ready events and invoke CommandLineRunner/ApplicationRunner beans. The embedded server being a bean created during refresh is the detail worth naming.

CommandLineRunner vs ApplicationRunner vs @PostConstruct for startup work?

@PostConstruct runs per-bean, mid-wiring — fine for initializing that bean's own state, wrong for app-level tasks (other beans may not exist yet; proxies may not be applied). Runners execute once after the entire context is ready and the server is up — the right place for seed data, warmups, and startup validation; they can safely use any bean including transactional ones. The two runner interfaces differ only in argument style: raw String... vs parsed ApplicationArguments (--option=value access). Order multiple runners with @Order; an exception from a runner aborts startup.

What does spring.main.lazy-initialization=true trade away?

It defers all bean creation to first use, cutting startup time — attractive in dev. The costs: you lose fail-fast (a misconfigured bean now explodes on the first request that needs it, in production, instead of in the deploy pipeline), and first-hit latency spikes as beans construct on demand. Sensible middle ground: keep eager init in production, use targeted @Lazy on a few genuinely heavy, rarely-used beans, and attack startup time with excluded auto-configs, CDS, or — where cold start really matters — GraalVM native images via Boot 3 AOT.

Coming from frontend: is Spring DI just Angular DI on the server?

Directionally yes — same core idea (declare dependencies, container resolves them; tokens/types, hierarchical resolution, singleton-by-default). Key differences worth naming: Spring resolves the entire graph eagerly at startup and fails fast, rather than lazily at injection time; Spring beans have a rich managed lifecycle (@PostConstruct/@PreDestroy, post-processors); and — the big one — Spring uses DI as the delivery mechanism for proxy-based AOP: transactions, caching, security are woven around beans by the container. React context is a weaker analogy: value propagation only, no lifecycle, no interception, resolution at render time.

Exercise 1 — DI by hand, then let Spring take over

Without Spring: define interface Notifier { void send(String msg); }, an EmailNotifier implementation, and a SignupService that depends on Notifier via constructor. Wire them manually in main and write a plain JUnit test using a hand-rolled fake. Then convert to Spring Boot: annotate, delete the manual wiring, and add a CommandLineRunner that triggers a signup. Verify the unit test still passes unchanged.

Solution 
// Plain Java — no framework
public interface Notifier { void send(String msg); }

public class SignupService {
    private final Notifier notifier;
    public SignupService(Notifier notifier) { this.notifier = notifier; }
    public void signup(String email) { notifier.send("Welcome " + email); }
}

// Test — needs no Spring, before OR after the conversion
@Test void sendsWelcome() {
    List<String> sent = new ArrayList<>();
    SignupService svc = new SignupService(sent::add);   // lambda fake!
    svc.signup("a@b.com");
    assertEquals(List.of("Welcome a@b.com"), sent);
}

// Spring version: just annotations — the test above is untouched
@Component class EmailNotifier implements Notifier {
    public void send(String msg) { System.out.println("EMAIL: " + msg); }
}
@Service @RequiredArgsConstructor
class SignupService { private final Notifier notifier; /* as before */ }

@Component @RequiredArgsConstructor
class Demo implements CommandLineRunner {
    private final SignupService svc;
    public void run(String... args) { svc.signup("demo@acme.com"); }
}

The takeaway: constructor injection means Spring is an assembly concern only — the classes and their tests are framework-free.

Exercise 2 — Strategy pattern via collection injection

Build a DiscountEngine with a DiscountRule interface (boolean applies(Order o); BigDecimal discount(Order o);) and three implementations (bulk order, first purchase, seasonal — seasonal only active when property discounts.seasonal.enabled=true). The engine must apply all applicable rules and require zero changes when a fourth rule is added.

Solution 
public interface DiscountRule {
    boolean applies(Order o);
    BigDecimal discount(Order o);
}

@Component class BulkOrderRule implements DiscountRule {
    public boolean applies(Order o)  { return o.itemCount() >= 10; }
    public BigDecimal discount(Order o) { return o.total().multiply(new BigDecimal("0.05")); }
}

@Component class FirstPurchaseRule implements DiscountRule { /* ... */ }

@Component
@ConditionalOnProperty(name = "discounts.seasonal.enabled", havingValue = "true")
class SeasonalRule implements DiscountRule { /* ... */ }

@Service
@RequiredArgsConstructor
public class DiscountEngine {
    private final List<DiscountRule> rules;  // Spring injects ALL registered impls

    public BigDecimal totalDiscount(Order o) {
        return rules.stream()
            .filter(r -> r.applies(o))
            .map(r -> r.discount(o))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

Adding rule #4 = one new class. The @ConditionalOnProperty rule simply isn't in the injected list when disabled — the engine never knows.

Exercise 3 — Write your own auto-configuration

Exercise 4 — Typed, validated configuration

Model this YAML as an immutable @ConfigurationProperties record with validation: prefix mail, required non-blank host, port 1–65535, Duration timeout defaulting to 5s, and a nested retry object (maxAttempts 1–10, backoff Duration). Make a wrong port fail at startup, and unit-test the record without Spring.

Solution 
@ConfigurationProperties(prefix = "mail")
@Validated
public record MailProperties(
    @NotBlank String host,
    @Min(1) @Max(65535) int port,
    @DefaultValue("5s") Duration timeout,
    @Valid Retry retry                      // @Valid cascades into nested record
) {
    public record Retry(@Min(1) @Max(10) int maxAttempts, Duration backoff) {}
}
// register: @ConfigurationPropertiesScan on the app class
mail:
  host: smtp.acme.com
  port: 587
  retry:
    max-attempts: 3        # relaxed binding → maxAttempts
    backoff: 200ms
// port: 99999 → startup fails with a BindValidationException listing the
// property, the value, and the violated constraint. Fail fast, not at send time.

@Test void retryConfigIsUsable() {

        new MailProperties.Retry(3, Duration.ofMillis(200)));

}

Exercise 5 — Lifecycle + events + health, end to end

Build a FeatureFlagService that (a) loads flags in @PostConstruct, (b) publishes a FlagsReloadedEvent whenever flags reload, (c) has a listener logging each reload, and (d) exposes a custom HealthIndicator reporting DOWN if flags have never loaded. Predict the startup log order before running.

Solution 
public record FlagsReloadedEvent(int count, Instant at) {}

@Service
@RequiredArgsConstructor
public class FeatureFlagService {
    private final ApplicationEventPublisher events;
    private final AtomicReference<Map<String, Boolean>> flags =
        new AtomicReference<>();   // singleton + concurrent reads → atomic swap, no locks

    @PostConstruct
    void initialLoad() { reload(); }

    public void reload() {
        Map<String, Boolean> loaded = fetchFromSource();
        flags.set(Map.copyOf(loaded));
        events.publishEvent(new FlagsReloadedEvent(loaded.size(), Instant.now()));
    }

    public boolean isLoaded() { return flags.get() != null; }
}

@Component
class FlagAuditListener {
    @EventListener
    void on(FlagsReloadedEvent e) { log.info("{} flags loaded at {}", e.count(), e.at()); }
}

@Component
@RequiredArgsConstructor
class FeatureFlagHealthIndicator implements HealthIndicator {
    private final FeatureFlagService flags;
    @Override public Health health() {
        return flags.isLoaded() ? Health.up().build()
                                : Health.down().withDetail("reason", "flags never loaded").build();
    }
}

Order prediction: the service's constructor → its @PostConstruct (which publishes the event; the listener bean is created on demand if needed and receives it synchronously, so the listener log prints during the service's initialization) → context finishes refresh → health endpoint reports UP. One subtlety worth noticing: the @PostConstruct-time event fires while the context is still starting — listeners must not assume the whole app is ready. If that matters, publish from an ApplicationRunner instead, or listen for ApplicationReadyEvent.
