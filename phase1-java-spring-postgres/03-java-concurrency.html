03 — Concurrency & Multithreading

From single-threaded event loop to true parallelism — the #1 senior Java interview topic

Why interviewers ask this. Concurrency is the senior Java differentiator. Anyone can write a REST controller; a senior engineer knows what happens when 500 requests hit it at once, why a counter silently loses increments, how a thread pool decides to reject work, and how to read a thread dump at 3 a.m. Coming from JavaScript you have a real advantage (you deeply understand async composition — CompletableFuture will feel like Promises) and one big gap: in Java, multiple threads genuinely run your code at the same time on different CPU cores, sharing the same heap memory. There is no event loop protecting you. Every section below builds that mental model, contrasts it with JS where useful, and covers exactly what 7-YOE interviews probe.

The one-sentence mental shift: in Node.js, your code is never interrupted mid-statement — only between callbacks. In Java, another thread can interleave with yours between any two machine instructions, and may not even see your writes to memory unless you use the right tools. Everything in this page follows from that.

Thread basics: lifecycle, Runnable vs Callable, interruption T1

A thread is an independent path of execution scheduled by the OS. All threads in a JVM process share the same heap (objects), but each has its own stack (local variables, call frames). This is the opposite of JS workers, which share nothing by default and communicate via message passing. Java threads communicate through shared mutable memory — which is both the power and the entire source of danger.

Thread lifecycle / states

The Thread.State enum has exactly six states. Interviewers love asking the difference between BLOCKED and WAITING.

                 start()
   NEW ──────────────────────▶ RUNNABLE ◀────────────────────────┐
                               (ready or actually running         │
                                on a CPU core; the JVM does       │
                                not distinguish)                  │
                                 │  │  │                          │
        waiting for a monitor    │  │  │   run() returns /        │
        lock held by another ◀───┘  │  └──▶ uncaught exception    │
        thread                      │            │                │
           │                        │            ▼                │
        BLOCKED                     │       TERMINATED            │
           │                        ▼                             │
           │              wait() / join() / park()                │
           └──────▶        WAITING  ──────────────────────────────┤
   (lock acquired          TIMED_WAITING                          │
    → back to              (sleep(ms), wait(ms),                  │
    RUNNABLE)               join(ms), tryLock(ms))────────────────┘
                            notify()/notifyAll()/timeout/interrupt
                            → back to RUNNABLE (possibly via
                              BLOCKED to re-acquire the lock)
  • NEW — created, start() not yet called.
  • RUNNABLE — eligible to run; may or may not be on a core right now.
  • BLOCKED — waiting to acquire an intrinsic lock (stuck at a synchronized boundary).
  • WAITING — parked indefinitely by wait(), join(), or LockSupport.park(); needs another thread to wake it.
  • TIMED_WAITING — same, but with a timeout (sleep(1000), wait(500)).
  • TERMINATED — run() finished or threw. A terminated thread can never be restarted (start() twice → IllegalThreadStateException).

BLOCKED vs WAITING is a classic screen-out question: BLOCKED means "I want a monitor lock someone else holds"; WAITING means "I voluntarily parked and need an explicit signal." In a thread dump, hundreds of BLOCKED threads on the same monitor = lock contention hotspot; hundreds of WAITING threads in a pool = normal idle workers.

Creating threads: Runnable vs Callable

// Runnable: no return value, cannot throw checked exceptions
Runnable task = () -> System.out.println("Hello from " + Thread.currentThread().getName());

// Callable<V>: returns a value AND may throw checked exceptions
Callable<String> fetch = () -> {
    Thread.sleep(100);            // throws InterruptedException — fine in a Callable
    return "result";
};

// Raw thread (you should almost never do this in production code):
Thread t = new Thread(task, "worker-1");
t.start();                        // start() spawns a new thread that calls run()
t.join();                         // current thread waits for t to finish (like await, but blocking)

Calling t.run() instead of t.start() executes the body on the current thread — no concurrency at all. It compiles, it "works", and it is a famous trick question. start() is what asks the OS for a new thread.

Why raw threads are bad practice → ExecutorService

Creating a platform thread costs ~1 MB of stack and a syscall; creating one per task under load will exhaust memory and thrash the scheduler. Threads created ad-hoc are also unbounded (no backpressure), unnamed (miserable to debug), and their failures vanish silently. The fix — and the expected senior answer — is: never hand-roll new Thread(...) in application code; use an ExecutorService (a managed, bounded, reusable pool — full section below).

Daemon threads

The JVM exits when all non-daemon threads finish. Daemon threads (set via t.setDaemon(true) before start()) are background helpers — GC threads, cache evictors — that get killed abruptly at JVM exit, with no finally blocks run. Never do work with durability requirements (flushing files, committing transactions) on a daemon thread.

Interruption — cooperative cancellation done right

Java has no safe way to kill a thread (Thread.stop() is deprecated because it releases locks mid-mutation, leaving objects corrupted). Instead, cancellation is cooperative: you request a stop with t.interrupt(), which sets a flag; the target thread must notice and exit cleanly. Blocking methods (sleep, wait, join, blocking queue ops) notice for you by throwing InterruptedException — and crucially, throwing it clears the interrupt flag.

// The right pattern: either propagate, or restore the flag.
class Worker implements Runnable {
    private final BlockingQueue<Task> queue;
    Worker(BlockingQueue<Task> queue) { this.queue = queue; }

    @Override public void run() {
        try {
            while (!Thread.currentThread().isInterrupted()) {
                Task task = queue.take();     // blocks; throws InterruptedException on interrupt
                task.process();
            }
        } catch (InterruptedException e) {
            // We were asked to stop. Restore the flag so callers up the
            // stack (and the pool) can also see the interruption:
            Thread.currentThread().interrupt();
        } finally {
            cleanup();                        // release resources deterministically
        }
    }
}

The cardinal sin — swallowing the interrupt:

// BROKEN: interruption is silently eaten; shutdownNow() can no longer stop this thread
try { Thread.sleep(1000); } catch (InterruptedException e) { /* ignore */ }

Rule: either rethrow InterruptedException, or call Thread.currentThread().interrupt() in the catch block. Empty catch = a thread that can never be cancelled, which breaks shutdownNow(), timeouts, and graceful shutdown.

JS contrast worth saying in an interview: AbortController in JS is the same cooperative-cancellation philosophy — you signal, the task checks. Java just predates it by 25 years and uses a per-thread flag plus an exception. Also note join() vs await: both wait for completion, but join() blocks a real OS thread, whereas await frees the event loop. Blocking is fine on worker threads; it is catastrophic on a small shared pool (see the CompletableFuture section).

synchronized, intrinsic locks & wait/notify T1

Every Java object has a hidden intrinsic lock (a.k.a. monitor). synchronized acquires it on entry and releases it on exit (even via exception). Only one thread can hold a given monitor at a time; others attempting entry go BLOCKED. It gives you both of the two guarantees you ever need:

  • Mutual exclusion (atomicity): the critical section runs without interleaving.
  • Visibility (memory): releasing a monitor flushes writes; acquiring the same monitor sees them (a happens-before edge — see the JMM section).

Method vs block, instance vs class lock

class Counter {
    private int count = 0;
    private final Object lock = new Object();

    // 1) synchronized instance method → locks `this`
    public synchronized void increment() { count++; }

    // 2) synchronized block on a private lock object (preferred style —
    //    callers can't accidentally lock `this` and interfere)
    public void incrementFine() {
        synchronized (lock) { count++; }
    }

    // 3) synchronized STATIC method → locks Counter.class (the Class object),
    //    a completely different lock from any instance's `this`!
    public static synchronized void resetGlobal() { /* ... */ }
}

Instance-level and class-level locks are independent. A synchronized instance method and a static synchronized method on the same class can run simultaneously on two threads — they lock different monitors (this vs Counter.class). If they touch the same static state, you have a race despite everything "being synchronized". Likewise, two threads calling a synchronized instance method on two different instances don't block each other.

Reentrancy

Intrinsic locks are reentrant: a thread that holds a monitor can re-enter synchronized blocks on the same monitor (the JVM keeps a hold count). Without this, a synchronized method calling another synchronized method on this — or a subclass calling super's synchronized method — would self-deadlock.

class Reentrant {
    public synchronized void outer() {
        inner();  // same thread re-acquires the same monitor: fine, count 1 → 2
    }
    public synchronized void inner() { /* ... */ }
}

wait / notify / notifyAll

These live on Object and implement the low-level "wait for a condition" protocol. wait() atomically releases the monitor and parks the thread; notify() wakes one waiter, notifyAll() wakes them all (they then compete to re-acquire the lock). Three iron rules:

  1. You must hold the monitor to call any of them (IllegalMonitorStateException otherwise).
  2. Always wait() inside a loop re-checking the condition — never an if — because of spurious wakeups and because another thread may consume the condition between notify and re-acquisition.
  3. Prefer notifyAll() unless you can prove exactly one waiter can proceed.
// Classic bounded buffer with wait/notify (know it; don't ship it)
class BoundedBuffer<T> {
    private final Queue<T> items = new ArrayDeque<>();
    private final int capacity;
    BoundedBuffer(int capacity) { this.capacity = capacity; }

    public synchronized void put(T item) throws InterruptedException {
        while (items.size() == capacity) {   // while, NOT if
            wait();                          // releases lock, parks
        }
        items.add(item);
        notifyAll();                         // wake consumers
    }

    public synchronized T take() throws InterruptedException {
        while (items.isEmpty()) {
            wait();
        }
        T item = items.remove();
        notifyAll();                         // wake producers
        return item;
    }
}

The senior answer to "explain wait/notify" ends with: "…and in real code I would use an ArrayBlockingQueue, a CountDownLatch, or a Condition instead." The java.util.concurrent utilities (2004, Doug Lea) exist precisely because hand-rolled wait/notify is easy to get subtly wrong (missed notify, wrong monitor, if-instead-of-while). Knowing the primitive proves depth; reaching for the library proves judgment.

volatile: visibility vs atomicity T1

This topic doesn't exist in JS (one thread = every write is instantly "visible"), so build the intuition from hardware: each CPU core has its own caches and store buffers, and both the JIT compiler and the CPU aggressively reorder and cache memory operations. Without explicit synchronization, a write made by core 1 may sit in its store buffer indefinitely as far as core 2 is concerned. Legally, this loop may never terminate:

// BROKEN: `running` may be hoisted into a register / never re-read
class Poller {
    private boolean running = true;              // ← not volatile

    void loop() {
        while (running) { doWork(); }            // JIT may compile to while(true)
    }
    void stop() { running = false; }             // other thread may never see this
}

volatile fixes exactly this: every read goes to the coherent view of memory, every write is published immediately, and reordering across the volatile access is restricted. Formally: a write to a volatile field happens-before every subsequent read of that field.

class Poller {
    private volatile boolean running = true;     // FIXED: visibility guaranteed
    void loop()  { while (running) doWork(); }
    void stop()  { running = false; }
}

Why volatile counter++ is still broken

volatile gives visibility, not atomicity. count++ is not one operation — it is a read-modify-write sequence of three steps, and two threads can interleave between them:

  volatile int count = 0;              count++ compiles to:
                                         1. read  count  (→ temp)
  Thread A                Thread B      2. add   1      (temp+1)
  ─────────               ─────────    3. write count  (← temp)
  read count → 0
                          read count → 0     ← B reads BEFORE A writes
  add 1      → 1
  write count = 1
                          add 1      → 1
                          write count = 1    ← A's increment is LOST
                                              final value: 1, expected 2

Both reads and both writes were perfectly "visible" — volatile did its job. The problem is the gap between read and write. Fix with synchronized, AtomicInteger (CAS), or LongAdder.

When volatile alone suffices

  • Status / shutdown flags — writes don't depend on the previous value (running = false).
  • One-writer, many-readers publication — e.g., a config object rebuilt periodically by one thread and read by many: volatile Config current; then current = newImmutableConfig;.
  • The volatile in double-checked locking (see JMM section) — prevents readers from seeing a partially constructed object.

Litmus test for "is volatile enough?": Does any write depend on a previously read value, or must multiple variables change together? If yes to either, volatile is insufficient — you need atomics or a lock. If the variable is an independent flag or a reference to an immutable snapshot with a single writer, volatile is the cheapest correct tool.

ExecutorService & thread pools — deep dive T1

This is the most reliably asked practical topic at senior level. ExecutorService decouples task submission from thread management: you submit Runnable/Callable tasks; a managed pool of reused worker threads executes them; a queue buffers the overflow; a rejection policy handles the flood. (JS analogy: the closest thing is a worker pool library like piscina — Node gives you a hidden libuv pool for I/O, but you never manage it. In Java, you own the pool, its size, and its failure mode.)

The factory methods and — crucially — their queue choices

The senior take: every factory method has an unbounded dimension — either the queue (fixed, single) or the thread count (cached). Under sustained overload each will take down the JVM. Production services construct ThreadPoolExecutor directly with a bounded queue and an explicit rejection policy, so overload produces fast, controlled failure (backpressure) instead of slow death. Alibaba's Java style guide outright bans the Executors factories for this reason — great line to drop in an interview.

ThreadPoolExecutor pool = new ThreadPoolExecutor(

    new ArrayBlockingQueue<>(100),        // workQueue       – BOUNDED buffer for waiting tasks
    new ThreadFactory() {                 // threadFactory   – name your threads! (debuggability)
        private final AtomicInteger n = new AtomicInteger(1);
        @Override public Thread newThread(Runnable r) {
            Thread t = new Thread(r, "order-svc-" + n.getAndIncrement());
            t.setDaemon(false);

                log.error("Uncaught in {}", th.getName(), ex));
            return t;
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy()   // rejectedExecutionHandler
);

Task flow — the classic interview trap

When you submit a task, the pool decides in this order (many candidates get step 2 vs 3 backwards):

  submit(task)
      │
      ▼
  ┌───────────────────────────────┐   yes
  │ running threads < corePoolSize │ ─────▶ create NEW core thread, run task
  └───────────────┬───────────────┘
                  │ no
                  ▼
  ┌───────────────────────────────┐   yes
  │   queue has free capacity?     │ ─────▶ ENQUEUE task (wait for a worker)
  └───────────────┬───────────────┘
                  │ no (queue FULL)
                  ▼
  ┌───────────────────────────────┐   yes
  │ running threads < maxPoolSize  │ ─────▶ create EXTRA thread, run task
  └───────────────┬───────────────┘
                  │ no (at max AND queue full)
                  ▼
        RejectedExecutionHandler fires

The trap: extra threads beyond corePoolSize are only created when the queue is completely full — the queue is preferred over new threads. Consequence #1: with newFixedThreadPool's unbounded queue, maximumPoolSize is irrelevant — it can never be reached. Consequence #2: a "burst-absorbing" config like core=4, max=32, queue=10 000 will happily let latency climb through a 10 000-deep queue before ever adding a 5th thread. If you want threads to scale before queueing, use a small bounded queue or a SynchronousQueue.

Sizing a pool

  • CPU-bound (parsing, compression, in-memory computation): ≈ number of cores (Runtime.getRuntime().availableProcessors()). More threads just add context-switch overhead.
  • I/O-bound (DB calls, HTTP calls): ≈ cores × (1 + wait time / compute time). Example: 8 cores, each task computes 5 ms and waits 95 ms on the network → 8 × (1 + 95/5) = 160 threads to keep the CPUs busy.
  • These are starting points — validate with load tests, and remember downstream limits (a 200-thread pool hammering a 20-connection DB pool just moves the queue).

Scheduled pools — fixed rate vs fixed delay

ScheduledExecutorService sched = Executors.newScheduledThreadPool(2);

// fixedRate: aims for a tick every 10s measured start-to-start.
// If a run takes longer than the period, the next run starts immediately after
// (executions never overlap — they queue up; there is no concurrent double-fire).
sched.scheduleAtFixedRate(this::pollUpstream, 0, 10, TimeUnit.SECONDS);

// fixedDelay: waits 10s AFTER each run finishes — self-pacing, can't pile up.
sched.scheduleWithFixedDelay(this::compactCache, 10, 10, TimeUnit.SECONDS);

If a periodic task throws, that task is silently cancelled — no log, no further runs, and the pool keeps humming. One of the sneakiest production bugs in Java ("the cleanup job just stopped three weeks ago"). Always wrap the body of scheduled tasks in a try/catch that logs.

Shutdown — the three-step idiom

pool.shutdown();                          // stop accepting NEW tasks; running + queued tasks finish
try {
    if (!pool.awaitTermination(30, TimeUnit.SECONDS)) {
        List<Runnable> neverRan = pool.shutdownNow();  // interrupt workers, drain queue
        log.warn("{} tasks never started", neverRan.size());
        if (!pool.awaitTermination(10, TimeUnit.SECONDS)) {
            log.error("Pool did not terminate — check for swallowed interrupts");
        }
    }
} catch (InterruptedException e) {
    pool.shutdownNow();
    Thread.currentThread().interrupt();
}
  • shutdown() — graceful: no new tasks, in-flight and queued work completes.
  • shutdownNow() — best-effort forceful: interrupts running workers (only works if tasks respond to interruption — see the swallowed-interrupt pitfall above) and returns tasks that never started.
  • awaitTermination(t, u) — blocks the caller until termination or timeout; neither shutdown method waits by itself.

submit vs execute — the swallowed exception pitfall

// execute(): exception hits the thread's UncaughtExceptionHandler → stack trace in logs
pool.execute(() -> { throw new IllegalStateException("boom"); });   // you'll SEE this

// submit(): exception is captured INSIDE the returned Future — nothing is printed!
Future<?> f = pool.submit(() -> { throw new IllegalStateException("boom"); });
// ...silence... until someone calls:
try {
    f.get();                              // NOW it surfaces, wrapped:
} catch (ExecutionException e) {
    Throwable actual = e.getCause();      // the IllegalStateException
}

"We submit tasks and never look at the Futures" is one of the most common real-world bugs: every failure disappears. If you fire-and-forget, use execute() (so failures reach the uncaught handler / logs), wrap the task body in try/catch, or attach handling via CompletableFuture.runAsync(...).exceptionally(...).

Race conditions, deadlock, livelock, starvation T1

Race conditions — the two canonical shapes

A race condition is when correctness depends on the (uncontrolled) timing of thread interleaving. Nearly all of them reduce to two patterns:

1. Read-modify-write — covered above with count++. The read and write are separate; an interleaved thread makes one update vanish.

2. Check-then-act — the state you checked changes before you act on it:

// BROKEN: classic check-then-act. Two threads can both pass the check.
class NaiveCache {
    private final Map<String, Connection> cache = new HashMap<>();

    public Connection get(String key) {
        if (!cache.containsKey(key)) {        // CHECK  ── thread B can slip in here
            cache.put(key, expensiveOpen(key)); // ACT   ── both threads open a connection;
        }                                       //         one is leaked; HashMap itself may corrupt
        return cache.get(key);
    }
}

// FIXED: make check-and-act one atomic operation
class SafeCache {
    private final ConcurrentHashMap<String, Connection> cache = new ConcurrentHashMap<>();
    public Connection get(String key) {
        return cache.computeIfAbsent(key, k -> expensiveOpen(k));  // atomic per key
    }
}

Using a thread-safe collection does not make compound operations safe: if (!map.containsKey(k)) map.put(k, v); is still a race even on ConcurrentHashMap — each call is atomic, the pair isn't. Use the atomic compound methods: putIfAbsent, computeIfAbsent, merge, replace. Same trap: list.isEmpty() then list.remove(0) on a synchronized list.

Deadlock

Two or more threads each hold a lock the other needs, forever. All four Coffman conditions must hold simultaneously — break any one and deadlock is impossible:

  1. Mutual exclusion — resources can only be held by one thread at a time.
  2. Hold and wait — a thread holds one resource while waiting for another.
  3. No preemption — locks can't be forcibly taken away.
  4. Circular wait — a cycle: A waits for B's lock, B waits for A's.
// BROKEN: guaranteed-to-eventually-deadlock money transfer
class Account {
    private long balance;
    void withdraw(long amt) { balance -= amt; }
    void deposit(long amt)  { balance += amt; }
}

void transfer(Account from, Account to, long amt) {
    synchronized (from) {                 // T1: transfer(a, b) locks a
        synchronized (to) {               // T2: transfer(b, a) locks b — now each waits forever
            from.withdraw(amt);
            to.deposit(amt);
        }
    }
}

Prevention 1 — global lock ordering (breaks circular wait): always acquire locks in a canonical order, e.g. by a unique id:

void transfer(Account from, Account to, long amt) {
    Account first  = from.id() < to.id() ? from : to;
    Account second = from.id() < to.id() ? to : from;
    synchronized (first) {
        synchronized (second) {
            from.withdraw(amt);
            to.deposit(amt);
        }
    }
}

Prevention 2 — tryLock with timeout (breaks hold-and-wait / adds preemption-ish behavior), requires ReentrantLock:

boolean transfer(Account from, Account to, long amt) throws InterruptedException {
    while (true) {
        if (from.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
            try {
                if (to.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
                    try {
                        from.withdraw(amt);
                        to.deposit(amt);
                        return true;
                    } finally { to.lock.unlock(); }
                }
            } finally { from.lock.unlock(); }
        }
        // couldn't get both — back off with jitter to avoid livelock, retry
        Thread.sleep(ThreadLocalRandom.current().nextInt(1, 10));
    }
}

Detection is easy; the fix is always design — ordering, scope reduction, or replacing shared locks with queues and immutability. Livelock & starvation (know the one-liners)

  • Livelock: threads are not blocked — they're busily responding to each other and making no progress (two people side-stepping the same way in a corridor; two transactions that each detect conflict, roll back, and retry in lockstep). Fix: randomized backoff.
  • Starvation: a thread is perpetually denied CPU or a lock — e.g., a non-fair lock under heavy contention always won by "barging" threads, or readers starving a writer in a read-mostly ReadWriteLock. Fix: fair locks (at a throughput cost), bounded queues, priority review.

CompletableFuture — async composition for Promise veterans T2

This is where your JS experience pays off directly. CompletableFuture<T> (CF) is Java's Promise: a container for a value that will exist later, with chainable transformation and error-handling stages. Two fundamental differences from JS, though:

  1. CF stages run on real threads in parallel. A JS Promise chain interleaves callbacks on one thread; CF callbacks can execute simultaneously on multiple cores — so the code inside your stages still needs the thread-safety rules from the rest of this page.
  2. You choose (and should choose) the executor. JS has one implicit scheduler; CF defaults to the shared ForkJoinPool.commonPool(), which is a trap for blocking work (below).

Creation and the Rosetta stone

// JS: const p = (async () => await fetchUser(id))();
CompletableFuture<User> userF =

thenApply vs thenCompose — the map/flatMap question

Identical to the JS distinction between a .then callback returning a value vs returning another Promise — except JS flattens automatically and Java makes you pick the method. Use thenApply when the function returns a plain value; use thenCompose when it returns another CompletableFuture (otherwise you get a nested CF<CF<T>>).

Exception handling

Like a rejected Promise, an exception skips forward past thenApply/thenCompose stages until a recovery stage catches it. Exceptions surface wrapped in CompletionException inside callbacks.

The async variants and the common-pool pitfall

Every stage method has three forms: thenApply(f) (may run on the thread that completed the previous stage — or the calling thread if already complete), thenApplyAsync(f) (re-dispatches to the default pool), thenApplyAsync(f, executor) (re-dispatches to your pool).

The default pool for all *Async methods is ForkJoinPool.commonPool() — sized to cores − 1 and shared by the whole JVM (parallel streams use it too). Run blocking I/O (JDBC, HTTP) on it and a handful of slow calls starves every other user of the pool — a notorious production incident pattern. Rule: blocking work always gets an explicit, bounded, named executor; the common pool is only for short CPU-bound work. Also remember non-async thenApply may run your callback on the completing thread — e.g., a Netty event-loop thread — so heavy work in a non-async stage can block an I/O framework.

anyOf and racing a fallback

// Promise.race: first completion wins — value OR exception, whichever is first

    supplyAsync(() -> primaryDc.fetch(key), ioPool),
    supplyAsync(() -> replicaDc.fetch(key), ioPool));

CompletableFuture<String> primary = supplyAsync(() -> primaryDc.fetch(key), ioPool);
CompletableFuture<String> hedge =
    supplyAsync(() -> replicaDc.fetch(key),
                CompletableFuture.delayedExecutor(200, TimeUnit.MILLISECONDS, ioPool));
String result = (String) CompletableFuture.anyOf(primary, hedge).join();

Two anyOf gotchas: it completes with the first result even if that first result is an exception (there is no "first success" built in — compose exceptionally per branch if you want that), and the losing branch keeps running — completion doesn't cancel it. If the loser holds resources, chain a cancellation/cleanup explicitly. Same is true in JS (Promise.race doesn't abort losers) — the fix there is AbortController; here it's cooperative interruption or resource-scoped cleanup.

Atomics & CAS: lock-free counters, ABA, LongAdder T2

java.util.concurrent.atomic classes provide lock-free thread safety for single variables via CAS (compare-and-set) — a CPU instruction (e.g., x86 lock cmpxchg) that atomically says: "if the value is still what I expect, replace it; otherwise tell me I lost." No thread ever blocks; losers just retry.

AtomicInteger counter = new AtomicInteger();
counter.incrementAndGet();                // atomic ++ — fixes the volatile counter

// What incrementAndGet does under the hood — the canonical CAS retry loop:
int current;
do {
    current = counter.get();              // 1. read current value (volatile read)
} while (!counter.compareAndSet(current, current + 1));  // 2. swap iff unchanged; else retry

// The same loop generalizes to any pure function — atomic max:
AtomicInteger high = new AtomicInteger(Integer.MIN_VALUE);
void recordSample(int v) {
    high.accumulateAndGet(v, Math::max);  // JDK runs the CAS loop for you
}

// AtomicReference for immutable snapshots:
AtomicReference<Config> config = new AtomicReference<>(Config.defaults());
config.updateAndGet(c -> c.withTimeout(500));   // function must be side-effect-free (may re-run!)

Atomic classes give both guarantees for a single variable: atomicity (CAS) and visibility (the value is stored with volatile semantics). Under low-to-moderate contention they beat locks (no blocking, no context switch). Under extreme contention many threads spin retrying — that's where LongAdder comes in.

The ABA problem

CAS checks "is the value still A?" — but the value may have gone A → B → A while you weren't looking. For counters that's harmless; for pointer-based structures (lock-free stacks where a node was popped, freed, and a recycled node with the same address pushed) it corrupts the structure: the CAS succeeds against a value that is "equal" but semantically different. Fix: pair the value with a version stamp so A→B→A becomes (A,0)→(B,1)→(A,2), and the stale CAS fails.

AtomicStampedReference<Node> top =
    new AtomicStampedReference<>(initialNode, 0);

boolean tryReplace(Node expected, Node update) {
    int[] stampHolder = new int[1];
    Node current = top.get(stampHolder);          // read value + stamp together
    return current == expected
        && top.compareAndSet(expected, update,
                             stampHolder[0], stampHolder[0] + 1);  // both must match
}
// (AtomicMarkableReference is the boolean-flag variant.)

LongAdder — for write-heavy contention

Under heavy contention, every AtomicLong increment fights over one memory location (cache-line ping-pong between cores). LongAdder splits the total across multiple internal cells — threads increment different cells with near-zero contention — and sum() adds them up on read. Trade-off: sum() is not an atomic snapshot and costs more; there's no compareAndSet.

LongAdder requests = new LongAdder();
requests.increment();                     // hot path: contention-free-ish
long approxTotal = requests.sum();        // read path: aggregate the cells

Rule of thumb: need the exact value on every operation or CAS semantics → AtomicLong. High-frequency writes, occasional reads (metrics, hit counters) → LongAdder. (Sibling: LongAccumulator for max/min-style reductions.)

Multiple Conditions — the killer feature over wait/notify. A bounded buffer can wake only producers or only consumers instead of notifyAll()-ing everyone:

class BoundedBuffer<T> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Object[] items;
    private int count, putIdx, takeIdx;
    BoundedBuffer(int cap) { items = new Object[cap]; }

    public void put(T x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) notFull.await();   // still while-loop!
            items[putIdx] = x;
            putIdx = (putIdx + 1) % items.length;
            count++;
            notEmpty.signal();            // wake exactly the waiters that can proceed
        } finally { lock.unlock(); }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) notEmpty.await();
            T x = (T) items[takeIdx];
            items[takeIdx] = null;
            takeIdx = (takeIdx + 1) % items.length;
            count--;
            notFull.signal();
            return x;
        } finally { lock.unlock(); }
    }
}

Default answer for "which do you use?": synchronized, unless I need a ReentrantLock feature. synchronized can't be leaked (auto-release), is well optimized by modern JVMs (biased/thin locks historically, lock elision), and signals intent simply. Reach for ReentrantLock for tryLock deadlock-avoidance, timed/interruptible acquisition, fairness, or multiple conditions. One virtual-threads-era footnote: before JDK 24, blocking inside synchronized could "pin" a virtual thread to its carrier (ReentrantLock didn't) — JEP 491 in JDK 24 fixed this, so it's no longer a reason to migrate.

ReadWriteLock

ReentrantReadWriteLock splits into a shared read lock (many concurrent readers) and an exclusive write lock. Wins when reads vastly outnumber writes and the critical section is non-trivial; under write-heavy load it's often slower than a plain lock (bookkeeping + writer starvation risks).

private final ReentrantReadWriteLock rw = new ReentrantReadWriteLock();
private final Map<String, Widget> cache = new HashMap<>();

public Widget read(String k) {
    rw.readLock().lock();
    try { return cache.get(k); }
    finally { rw.readLock().unlock(); }
}
public void write(String k, Widget w) {
    rw.writeLock().lock();
    try { cache.put(k, w); }
    finally { rw.writeLock().unlock(); }
}
// Note: upgrading read → write is NOT supported (deadlocks);
// downgrading write → read is (acquire read, release write).

StampedLock (awareness level)

StampedLock (Java 8) adds optimistic reading: tryOptimisticRead() takes no lock at all — you read, then validate(stamp) checks whether a writer intervened; if so, fall back to a real read lock. Excellent for read-dominated hot paths. Caveats worth stating: not reentrant, no conditions, and misuse (forgetting to validate, or reading inconsistent multi-field state before validation) is easy — so it's a specialist tool, not a default.

Java Memory Model & happens-before T2

The JMM answers one question: when is a write made by thread A guaranteed to be seen by thread B? The uncomfortable default answer is "possibly never, and possibly out of order," because:

  • CPU caches & store buffers — each core works on local copies; writes drain to shared memory lazily.
  • Compiler/JIT optimizations — hoisting reads out of loops, reordering independent stores, keeping values in registers.
  • CPU out-of-order execution — hardware itself reorders memory operations for speed.

All of this is invisible in single-threaded code (the JIT preserves as-if-serial semantics per thread) — which is exactly why JS never needed to teach you this. It becomes visible the instant a second thread reads the same memory.

Happens-before: the rules you cite in interviews

If action X happens-before action Y, then X's writes are visible to Y and X is ordered before Y. The rules that matter:

  1. Program order: within one thread, each action happens-before the next (per that thread's own view).
  2. Monitor lock: unlocking a monitor happens-before every subsequent lock of that same monitor. (Why synchronized gives visibility, not just exclusion.)
  3. Volatile: a write to a volatile field happens-before every subsequent read of it. Bonus: everything written before the volatile write becomes visible to code after the volatile read (piggybacking).
  4. Thread start: t.start() happens-before anything inside t.run() — a thread sees everything set up before it was started.
  5. Thread join: everything in t.run() happens-before t.join() returning — after join you see all the thread's writes.
  6. Transitivity: X hb Y and Y hb Z ⇒ X hb Z. (Executors/CF build on this: submitting happens-before execution; completion happens-before dependent stages / get() returning.)

Double-checked locking — why it needs volatile

// BROKEN without volatile — the most famous JMM bug
class Singleton {
    private static Singleton instance;              // ← must be volatile!

    static Singleton getInstance() {
        if (instance == null) {                     // 1st check: no lock (fast path)
            synchronized (Singleton.class) {
                if (instance == null) {             // 2nd check: under lock
                    instance = new Singleton();     // ← the dangerous line
                }
            }
        }
        return instance;
    }
}

instance = new Singleton() is really three steps: (1) allocate memory, (2) run the constructor, (3) store the reference. The JIT/CPU may legally reorder (2) and (3). A second thread on the fast path (no lock, so the monitor rule doesn't protect it) can then observe a non-null instance whose fields are still default-initialized — a partially constructed object. Declaring the field volatile forbids that reordering: the reference store becomes a volatile write that can't float above the constructor.

// FIX 1: volatile
private static volatile Singleton instance;

// FIX 2 (preferred): static holder idiom — class-init locking does the work, zero volatile
class Singleton {
    private Singleton() {}
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();  // initialized on first access,
    }                                                        // guaranteed safe by the JLS
    static Singleton getInstance() { return Holder.INSTANCE; }
}

// FIX 3: enum singleton — also serialization- and reflection-proof
enum Config { INSTANCE; }

Safe publication idioms

"Publishing" = making an object visible to other threads. Safe ways, in preference order:

  1. Immutability with final fields — the JMM gives final fields a special guarantee: if the object isn't leaked from its constructor, every thread sees the final fields fully initialized, even without synchronization. This is why immutable objects (and records) are the ultimate concurrency tool: no locks needed, ever.
  2. Static initializer / holder class — class initialization is internally locked by the JVM.
  3. volatile / AtomicReference field — publish a fully built object through a volatile write.
  4. Under a lock / via a concurrent collection — e.g., putting into a ConcurrentHashMap or a BlockingQueue safely publishes (these establish their own happens-before edges).

Never let this escape the constructor (registering listeners, starting a thread with an inner-class Runnable inside the constructor). Another thread can call back into a half-built object — the final-field guarantee is void once this leaks early.

ThreadLocal — per-thread state

A ThreadLocal<T> gives each thread its own independent copy — thread confinement instead of sharing. Classic uses:

  • Non-thread-safe reusable objects: the textbook example is SimpleDateFormat (mutable, not thread-safe) — one instance per thread. (Modern answer: just use immutable DateTimeFormatter.)
  • Request context propagation: auth principal, tenant id, trace/correlation id — set at the start of a request, readable anywhere down the call stack without threading parameters through every signature. This is how Spring Security's SecurityContextHolder, MDC logging, and (historically) transaction managers work.
public final class RequestContext {
    private static final ThreadLocal<String> TENANT = new ThreadLocal<>();

    public static void set(String tenantId) { TENANT.set(tenantId); }
    public static String get()              { return TENANT.get(); }
    public static void clear()              { TENANT.remove(); }
}

// In a servlet filter / interceptor:
try {
    RequestContext.set(extractTenant(request));
    chain.doFilter(request, response);
} finally {
    RequestContext.clear();               // ← NON-NEGOTIABLE, see pitfall
}

The thread-pool leak / bleed: pool threads never die, so a ThreadLocal value set during request A survives on that worker thread and is still there when request B — a different user! — is served by the same thread. Consequences: memory leaks (values are never GC-able while the thread lives and classloader leaks in app servers) and, far worse, data bleeding between users (request B silently runs as request A's tenant). Rule: every set() on a pooled thread is paired with remove() in a finally block.

InheritableThreadLocal copies the parent's value into a newly created child thread. It breaks with pools for the same structural reason: pool threads were created once, long ago, at pool startup — they inherit whatever the creating thread had then, not the submitting request's context now. Real systems propagate context explicitly instead: capture at submit time and restore in the task (wrapping executors — Spring's TaskDecorator, Micrometer's context-propagation library), or use JDK 21+ ScopedValue with structured concurrency.

Bridge: virtual threads change the calculus T2

Virtual threads (JDK 21, covered in tutorial 02) don't change any correctness rule on this page — races, the JMM, locks, and ThreadLocal semantics all apply unchanged — but they overturn the economics. Platform threads are expensive (~1 MB stack, OS-scheduled), which is the entire reason thread pools, careful sizing formulas, and callback-style CompletableFuture chains exist. Virtual threads cost roughly a few hundred bytes to a few KB and are scheduled by the JVM, which unmounts them from their carrier thread whenever they block on I/O. So for I/O-bound services, "thread-per-request" becomes viable at 100k+ concurrent requests with plain, readable, blocking code — Executors.newVirtualThreadPerTaskExecutor() instead of a sized pool, and simple sequential calls instead of deep CF pipelines. What you keep from this page: pools are still right for CPU-bound work (parallelism is still bounded by cores), Semaphore replaces pool size as the way to limit concurrency against fragile downstreams, ThreadLocal gets costly at millions of threads (prefer ScopedValue), and pre-JDK-24 you'd watch for pinning inside synchronized.

Interview Q&A

What's the difference between a process and a thread?

A process is an OS-level unit with its own isolated memory space; a thread is a unit of execution within a process. Threads in the same JVM process share the heap (objects, statics) but each has a private stack and program counter. Sharing memory makes thread communication cheap (just read/write objects) but is the root of all concurrency hazards — versus processes, which are isolated and must use IPC. JS framing: Node worker_threads are closer to processes in spirit (isolated by default, message passing); Java threads are true shared-memory siblings.

Walk me through the thread states. What's the difference between BLOCKED and WAITING?

Six states: NEW (created, not started), RUNNABLE (eligible/running), BLOCKED, WAITING, TIMED_WAITING, TERMINATED. BLOCKED means the thread is trying to enter a synchronized section and another thread holds that monitor — it will proceed the moment the lock frees. WAITING means the thread parked itself via wait(), join(), or park() and needs an explicit signal (notify, thread death, unpark). TIMED_WAITING is the same with a deadline (sleep, timed wait/join/tryLock). Diagnostic value: masses of BLOCKED threads on one monitor in a thread dump = contention hotspot; WAITING pool workers = normal idle state.

Runnable vs Callable vs Future?

Runnable.run() returns void and can't throw checked exceptions — fire-and-forget work. Callable<V>.call() returns V and may throw — work with a result. Submitting a Callable to an ExecutorService returns a Future<V>: a handle to the eventual result with get() (blocking, throws ExecutionException wrapping the task's failure), cancel(), and status checks. Follow-up I'd volunteer: plain Future is impoverished — no composition, no callbacks — so in practice we use CompletableFuture, which is Future plus a Promise-style combinator API.

Why is creating raw threads considered bad practice? What do you use instead?

Unbounded resource usage (each platform thread ≈ 1 MB stack plus scheduler load; thread-per-task under load = OOM), no reuse (creation cost per task), no backpressure, unnamed threads (debugging misery), and silently vanishing failures. Instead: ExecutorService — bounded, reusable workers, an explicit queue, an explicit rejection policy, lifecycle management (shutdown), and named threads via a ThreadFactory. In modern code, virtual threads restore "thread per task" for I/O-bound work, but through Executors.newVirtualThreadPerTaskExecutor(), still not hand-rolled new Thread().

Explain how ThreadPoolExecutor decides when to create threads. Any surprises?

The order is: (1) below corePoolSize → create a new core thread, even if others are idle; (2) at core size → enqueue the task; (3) only if the queue is full → create threads up to maximumPoolSize; (4) queue full and at max → the RejectedExecutionHandler fires. The surprise most people miss: the queue is preferred over new threads, so max threads only ever appear when the queue is completely full. Corollary: with an unbounded queue (what newFixedThreadPool uses), maximumPoolSize is dead configuration, and a large queue silently absorbs latency before scaling. If you want scale-then-queue behavior you need a small bounded queue or SynchronousQueue.

How do you size a thread pool?

Depends on the workload. CPU-bound: about the number of cores — more threads only add context switching. I/O-bound: cores × (1 + wait/compute); e.g., 8 cores with tasks that wait 90 ms per 10 ms of CPU → 8 × 10 = 80 threads to keep cores busy. Then the senior caveats: these are starting points to validate under load; the pool must respect downstream limits (DB connection pool size, rate limits) or you just relocate the queue; the work queue must be bounded with a deliberate rejection policy so overload fails fast instead of OOMing; and for pure I/O concurrency on JDK 21+, virtual threads sidestep the sizing question entirely — you bound concurrency with a Semaphore instead.

What's the difference between submit() and execute()?

execute(Runnable) is fire-and-forget: no return, and an uncaught exception propagates to the thread's UncaughtExceptionHandler, so it appears in logs. submit() accepts Runnable or Callable and returns a Future — but any exception is captured inside that Future and only surfaces when someone calls get(). The classic production bug: submit tasks, drop the Futures, and every failure disappears silently. Rules: use submit only if you'll consume the Future; for fire-and-forget use execute, catch inside the task, or use CompletableFuture with exceptionally.

shutdown() vs shutdownNow() — and how do you shut a pool down properly?

shutdown() stops accepting new tasks but lets running and queued tasks complete. shutdownNow() additionally interrupts workers and returns the list of never-started tasks — "best effort" because it relies on tasks honoring interruption; a task that swallows InterruptedException will keep running. Neither blocks, so the standard idiom is: shutdown(), then awaitTermination(timeout), then if not terminated shutdownNow() and await again, logging leftovers; catch InterruptedException by calling shutdownNow and re-interrupting. This is also why swallowed interrupts inside tasks are a serious bug — they break graceful shutdown.

What does synchronized actually guarantee?

Two independent guarantees. Atomicity/mutual exclusion: only one thread executes code guarded by a given monitor at a time. Visibility: releasing a monitor happens-before a subsequent acquisition of the same monitor, so the next thread sees all writes made under the lock — synchronization is a memory operation, not just a turnstile. Details worth adding: an instance method locks this, a static method locks the Class object (independent locks!), intrinsic locks are reentrant, and both threads must synchronize on the same monitor for either guarantee to apply — one thread writing under a lock and another reading without it is still a data race.

volatile vs synchronized?

volatile provides visibility and ordering for a single field but no mutual exclusion — concurrent read-modify-write sequences still interleave. synchronized provides both exclusion and visibility over an arbitrary block. So volatile boolean running for a stop flag is perfect; volatile int count with count++ is broken because ++ is read-add-write and two threads can both read the same old value. volatile is cheaper (no blocking, no lock acquisition) — the right tool when writes don't depend on prior state and only one variable is involved; otherwise synchronized/Atomic/Lock.

Why exactly is volatile counter++ broken if volatile guarantees visibility?

Because visibility is not atomicity. counter++ is three operations: read the current value, add one, write back. volatile guarantees each individual read sees the latest value and each write publishes immediately — but it cannot stop thread B from reading between A's read and A's write. Both read 5, both write 6, one increment is lost. Nothing was "stale"; the interleaving in the gap is the bug. Fixes: AtomicInteger.incrementAndGet() (CAS retries on interference), synchronized (excludes the interleaving), or LongAdder under heavy write contention.

What is a race condition? Give the two canonical patterns.

Correctness depending on uncontrolled thread interleaving. Pattern 1 — read-modify-write: count++, balance -= amt; the value changes between your read and your write. Pattern 2 — check-then-act: if (!map.containsKey(k)) map.put(k, expensive()), lazy init, "check balance then withdraw"; the condition you checked is stale by the time you act. The fix in both cases is making check-and-act / read-and-write one atomic unit: locks, CAS loops, or atomic compound methods (computeIfAbsent, putIfAbsent). Key trap to mention: a thread-safe collection doesn't make a sequence of its calls atomic.

What is deadlock? Conditions, prevention, and how do you find one in production?

Threads permanently blocked, each holding a lock another needs. Requires all four Coffman conditions: mutual exclusion, hold-and-wait, no preemption, circular wait — break any one to prevent it. Practical prevention: (1) a global lock-ordering discipline (e.g., always lock accounts by ascending id) breaks circular wait; (2) tryLock with timeout plus release-and-retry with randomized backoff breaks hold-and-wait; (3) shrink lock scope / prefer immutable data and message passing so fewer locks exist. Detection: thread dump via jstack <pid> or jcmd Thread.print — the JVM prints "Found one Java-level deadlock" with the exact cycle; programmatically ThreadMXBean.findDeadlockedThreads(). I'd add that the dump tells you where, but the fix is a design change, not a patch.

Explain thenApply vs thenCompose on CompletableFuture.

It's map vs flatMap. thenApply(fn) takes a function returning a plain value: CF<T> → CF<U>. thenCompose(fn) takes a function returning another CompletableFuture and flattens: without it you'd get CF<CF<U>>. Direct JS mapping: .then(x => x + 1) vs .then(x => fetch(x)) — except JS auto-flattens returned promises inside .then, while Java makes the distinction explicit in the method name. Use thenCompose whenever the next step is itself async (a dependent service call); thenApply for pure transformations.

How does CompletableFuture differ from a JavaScript Promise?

API-wise they're cousins (thenApply/thenCompose ≈ then; exceptionally ≈ catch; whenComplete ≈ finally; allOf/anyOf ≈ all/race). Execution model differs fundamentally: JS callbacks all run on one thread — concurrency without parallelism, so callback bodies need no synchronization. CF stages run on thread pools with true multi-core parallelism — real speedups, but stage bodies touching shared state need locks/atomics/immutability. Second difference: executor control — JS has one implicit scheduler; CF defaults *Async methods to ForkJoinPool.commonPool(), and choosing the right executor (never blocking I/O on the common pool) is your job. Third: join()/get() block a real thread, unlike await which yields. Also CF is completable from outside (complete(value)) — closer to a deferred/resolvable promise.

What's dangerous about the ForkJoinPool common pool?

It's a single JVM-wide shared pool sized to cores−1, used by default by parallel streams and all CompletableFuture *Async methods when no executor is given. Two dangers: (1) blocking I/O tasks occupy its few threads, starving every other user of the pool — unrelated features slow down mysteriously; (2) with cores−1 threads, on small containers it can be tiny (even effectively single-threaded). Rule: common pool only for short CPU-bound work; any blocking or long-running work gets an explicit, bounded, named executor passed to the Async variants.

What is CAS and how does AtomicInteger work?

Compare-and-set is an atomic CPU instruction: "update this memory location to new only if it still equals expected; return whether it worked." AtomicInteger's operations run a retry loop: read current value, compute the new value, CAS; if another thread changed it in between, the CAS fails and the loop retries with the fresh value. It's optimistic and lock-free — no thread ever blocks or is descheduled holding a lock; contention costs retries instead of context switches, which wins at low/moderate contention. Trade-offs to mention: heavy contention causes spin-retry churn (use LongAdder for hot counters), CAS covers only one variable (multi-variable invariants need locks), and plain CAS is exposed to ABA.

Explain the ABA problem and its fix.

CAS checks value equality, not history. If a value goes A → B → A between your read and your CAS, the CAS succeeds even though the world changed. Harmless for a counter; corrupting for pointer-based lock-free structures — e.g., a lock-free stack where the node you saw at top was popped and a recycled node at the same reference was pushed back: your CAS "succeeds" against a structurally different stack. Fix: version the reference — AtomicStampedReference pairs the value with an int stamp incremented on every change, so A returns as (A, stamp+2) and the stale CAS fails. AtomicMarkableReference is the boolean variant. Garbage collection makes ABA rarer in Java than in C++ (addresses aren't recycled while referenced) but designed object reuse still triggers it.

When would you choose ReentrantLock over synchronized?

Default is synchronized — automatic release, simple intent, well optimized. ReentrantLock when I need what the keyword can't do: tryLock() to avoid deadlock or fail fast, timed acquisition (tryLock(t, unit)) for bounded waiting, lockInterruptibly() so a blocked thread can be cancelled, fairness for starvation-sensitive scenarios (accepting lower throughput), or multiple Condition objects for precise signaling (wake only consumers vs only producers). Cost: manual unlock() in a finally block — a leaked lock is a latent deadlock. Note that performance is no longer a differentiator on modern JVMs, and since JDK 24 the virtual-thread pinning argument for ReentrantLock is gone too.

What is happens-before? Give the main rules.

The JMM's ordering relation: if X happens-before Y, X's memory effects are visible to Y. Without an hb edge between a write and a read on different threads, the read may see a stale value or reorderings — that's a data race. Key rules: program order (within a thread); monitor rule (unlock hb subsequent lock of the same monitor — why synchronized gives visibility); volatile rule (write hb subsequent read of that field — and it fences surrounding writes too); Thread.start() hb the thread's first action; a thread's last action hb join() returning; and transitivity, which lets the library extend these — task submission hb execution, CF completion hb dependent stages. Senior framing: happens-before is the vocabulary for explaining exactly why double-checked locking needs volatile and why unsynchronized publication is broken.

Why does double-checked locking need volatile?

Because instance = new Singleton() decomposes into allocate → construct → publish reference, and the JIT/CPU may reorder construct and publish. The second thread's first check reads the field without any lock, so the monitor rule doesn't protect it — it can see a non-null reference to an object whose constructor hasn't run, and use garbage field values. Making the field volatile forbids the reordering (the reference store becomes a release write; the racy read becomes an acquire read), so a non-null reference implies a fully constructed object. Better answers I'd offer: the static holder idiom (JVM class-init locking gives the same laziness with no volatile and no double-check) or an enum singleton.

CountDownLatch vs CyclicBarrier?

Latch: one-shot event counter — waiters block until countDown() has been called N times; the counters don't have to wait, and it can never be reset. Use for "don't proceed until N startup tasks / N responses complete." Barrier: a meeting point for a fixed party of threads — each calls await() and blocks until all N arrive, then it resets automatically for the next cycle, optionally running a barrier action on each trip. Use for phased parallel algorithms where the same workers repeatedly synchronize between iterations. Mnemonic: latch counts events downward once; barrier gathers threads repeatedly.

What are the use cases and risks of ThreadLocal?

Use cases: per-thread copies of non-thread-safe reusable objects (the classic SimpleDateFormat example), and implicit request context — tenant id, auth principal, trace id — set once per request and readable anywhere without parameter plumbing (Spring SecurityContextHolder, MDC). Risks, all rooted in thread pools where threads never die: values persist across requests, causing memory/classloader leaks and — worse — context bleeding, where user B's request runs with user A's leftover tenant. Discipline: pair every set() with remove() in a finally block, typically in a filter/interceptor. Also: InheritableThreadLocal doesn't help with pools (workers were created at pool startup, not per request) — real propagation wraps tasks to capture/restore context, and JDK 21+ ScopedValue is the modern, safer replacement, especially with virtual threads.

wait() vs sleep()? And why must wait() be called in a loop inside synchronized?

Differences: wait() is on Object, must be called while holding that object's monitor, and releases the lock while parked, waking on notify/notifyAll/interrupt; sleep() is a static Thread method, needs no lock, and holds any locks it has for the whole duration (which is why sleeping inside synchronized is an anti-pattern). Inside synchronized: wait/notify manipulate the monitor's wait-set, so calling without the lock is an IllegalMonitorStateException — and the lock is what makes "check condition, then wait" atomic against a missed notification. In a loop: (1) spurious wakeups are permitted by the spec; (2) between being notified and re-acquiring the lock, another thread may consume the condition; (3) with notifyAll, all waiters wake but perhaps only one can proceed. So: while (!condition) wait(); — never if.

What are livelock and starvation?

Livelock: threads aren't blocked — they're actively running — but make no progress because they keep reacting to one another: two transactions that each detect a conflict, abort, retry simultaneously, and collide again forever; or two threads politely releasing and re-acquiring locks in lockstep. Fix: randomized/exponential backoff to break the symmetry. Starvation: a runnable thread is perpetually denied resources — e.g., under a non-fair lock, freshly arriving "barging" threads keep winning over one that's been queued; or writers never getting a ReadWriteLock under constant reader traffic. Fixes: fair locks (throughput trade-off), bounded FIFO queues, avoiding thread priorities, or restructuring the read/write balance. Both matter because they pass a naive "no deadlock" check while the system still fails.

How do virtual threads change how you design concurrent services?

They change economics, not correctness. Platform threads are expensive, which forced pooling, sizing formulas, and async/callback style (CompletableFuture chains, reactive) to survive high I/O concurrency. Virtual threads are cheap JVM-scheduled threads that unmount from their carrier while blocked on I/O, so thread-per-request with plain blocking code scales to hundreds of thousands of concurrent requests — I'd use newVirtualThreadPerTaskExecutor() and write sequential code instead of deep CF pipelines for I/O-bound paths. What survives: all JMM/locking rules; pools sized to cores for CPU-bound work (parallelism is still core-limited); Semaphores replace pool size as the concurrency limiter for fragile downstreams; ThreadLocal becomes costly at huge thread counts (prefer ScopedValue); and don't pool virtual threads — they're disposable by design.

Hands-on exercises

Exercise 1 — Prove the race, then fix it three ways

Write a program where 10 threads each increment a shared plain int counter 100 000 times. Print the final value (it will be less than 1 000 000). Then fix it three different ways — synchronized, AtomicInteger, LongAdder — and state when you'd choose each.

Solution 
public class RaceDemo {
    static int plain = 0;                                    // broken
    static final Object LOCK = new Object();
    static int guarded = 0;                                  // fix 1
    static final AtomicInteger atomic = new AtomicInteger(); // fix 2
    static final LongAdder adder = new LongAdder();          // fix 3

    public static void main(String[] args) throws InterruptedException {
        int threads = 10, perThread = 100_000;
        try (ExecutorService pool = Executors.newFixedThreadPool(threads)) {
            CountDownLatch done = new CountDownLatch(threads);
            for (int t = 0; t < threads; t++) {
                pool.execute(() -> {
                    for (int i = 0; i < perThread; i++) {
                        plain++;                                   // lost updates here
                        synchronized (LOCK) { guarded++; }
                        atomic.incrementAndGet();
                        adder.increment();
                    }
                    done.countDown();
                });
            }
            done.await();
        }
        System.out.println("plain   = " + plain);        // e.g. 738 412 — nondeterministic
        System.out.println("guarded = " + guarded);      // 1 000 000
        System.out.println("atomic  = " + atomic.get()); // 1 000 000
        System.out.println("adder   = " + adder.sum());  // 1 000 000
    }
}

Choosing: synchronized when the increment is part of a larger multi-field invariant; AtomicInteger for a standalone counter whose exact value you read often or CAS against; LongAdder for write-hot, read-rare counters (metrics) under heavy contention. Note the latch: without done.await() (or pool shutdown+awaitTermination) main might print before workers finish — itself a happens-before lesson: countDown() hb await() returning, so main sees all the writes.

Exercise 2 — Build a ThreadPoolExecutor and watch the task-flow rules fire

Construct a pool with core=2, max=4, queue capacity=2, AbortPolicy. Submit 10 slow tasks (sleep 2s) in a loop, logging each submission and each rejection. Predict, before running: how many run immediately, how many queue, how many trigger extra threads, how many are rejected? Then verify.

Solution 
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2, 4, 30, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(2),
    r -> new Thread(r, "demo-" + System.nanoTime() % 1000),
    new ThreadPoolExecutor.AbortPolicy());

for (int i = 1; i <= 10; i++) {
    final int id = i;
    try {
        pool.execute(() -> {
            System.out.printf("task %d on %s%n", id, Thread.currentThread().getName());
            try { Thread.sleep(2000); } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
        System.out.printf("submitted %d (pool=%d queue=%d)%n",
            id, pool.getPoolSize(), pool.getQueue().size());
    } catch (RejectedExecutionException e) {
        System.out.println("REJECTED " + id);
    }
}
pool.shutdown();

Prediction (the interview answer): tasks 1–2 start immediately on new core threads; tasks 3–4 go to the queue (core is saturated, queue has room); tasks 5–6 cause thread 3 and thread 4 to be created (queue is now full, pool below max) and run before tasks 3–4, which are still queued — a subtle consequence worth calling out; tasks 7–10 are rejected (queue full, at max). So: 4 running (1,2,5,6), 2 queued (3,4), 4 rejected (7–10). Swap in CallerRunsPolicy and re-run: tasks 7+ execute on main, which visibly slows submission — backpressure in action.

Exercise 3 — Create a deadlock, capture it with jstack, then fix it

Write two threads locking two resources in opposite order with a sleep in between so the deadlock is deterministic. Run it, capture a thread dump (jstack <pid> or jcmd <pid> Thread.print), and find the "Found one Java-level deadlock" section. Then fix it twice: with lock ordering, and with tryLock+backoff.

Solution 
public class DeadlockDemo {
    static final Object A = new Object();
    static final Object B = new Object();

    public static void main(String[] args) {
        new Thread(() -> {
            synchronized (A) {
                sleep(100);                       // guarantee the interleaving
                synchronized (B) { System.out.println("t1 got both"); }
            }
        }, "t1").start();

        new Thread(() -> {
            synchronized (B) {                    // OPPOSITE order → circular wait
                sleep(100);
                synchronized (A) { System.out.println("t2 got both"); }
            }
        }, "t2").start();
    }
    static void sleep(long ms) {
        try { Thread.sleep(ms); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

Run jps to find the pid, then jstack <pid>. The dump ends with a section like:

Found one Java-level deadlock:
=============================
"t1": waiting to lock monitor 0x...(object 0x..., a java.lang.Object),
      which is held by "t2"
"t2": waiting to lock monitor 0x...(object 0x..., a java.lang.Object),
      which is held by "t1"

Fix 1 — ordering: both threads acquire in a canonical order, e.g. by System.identityHashCode (with a tie-breaker lock if hashes collide):

Object first  = System.identityHashCode(A) < System.identityHashCode(B) ? A : B;
Object second = first == A ? B : A;
synchronized (first) { synchronized (second) { /* work */ } }

Fix 2 — tryLock: replace monitors with two ReentrantLocks; loop: tryLock the first, then tryLock(50ms) the second; on failure unlock everything, sleep a random 1–10 ms (jitter prevents livelock), retry. See the transfer example in the deadlock section for the full pattern.

Exercise 4 — CompletableFuture fan-out with timeout and fallback

A product page needs: details (200 ms), price (150 ms), and reviews (flaky, sometimes 2 s). Fetch all three in parallel on a dedicated executor; combine details+price (both required); give reviews a 500 ms budget with an empty-list fallback; the page should render in ~max(200, 150, 500) ms, never the sum. Handle a price failure by failing the whole page with a clear error.

Solution 
ExecutorService io = Executors.newFixedThreadPool(8,
    r -> new Thread(r, "page-io"));

record Page(Details d, Price p, List<Review> reviews) {}

CompletableFuture<Details> detailsF =
    CompletableFuture.supplyAsync(() -> catalog.details(id), io);

CompletableFuture<Price> priceF =
    CompletableFuture.supplyAsync(() -> pricing.price(id), io);
    // no fallback: price is mandatory — let failure propagate

CompletableFuture<List<Review>> reviewsF =
    CompletableFuture.supplyAsync(() -> reviews.fetch(id), io)
        .completeOnTimeout(List.of(), 500, TimeUnit.MILLISECONDS)  // degrade gracefully
        .exceptionally(ex -> List.of());                           // flaky service → empty

CompletableFuture<Page> pageF =
    detailsF.thenCombine(priceF, (d, p) -> new AbstractMap.SimpleEntry<>(d, p))
            .thenCombine(reviewsF, (dp, rv) -> new Page(dp.getKey(), dp.getValue(), rv))
            .orTimeout(1, TimeUnit.SECONDS)          // overall budget
            .whenComplete((page, ex) -> metrics.recordPageBuild(ex));

Page page = pageF.join();   // only block at the very edge (controller boundary)

Points to verify you got: all three supplyAsync calls start immediately in parallel (creating the CFs starts the work — unlike lazy alternatives); reviews degrade instead of failing; a price exception skips both thenCombines and surfaces from join() as a CompletionException wrapping the cause; the explicit io pool keeps blocking calls off the common pool. Total latency ≈ 500 ms worst case, not 200+150+2000.

Exercise 5 — Semaphore bulkhead around a fragile downstream

You call a legacy service that collapses above 5 concurrent requests. Build a wrapper that (a) allows at most 5 in-flight calls, (b) makes callers wait at most 300 ms for a slot then fails fast, (c) never leaks a permit even when the call throws. Simulate with 20 threads and print how many succeeded / were shed.

Solution 
class Bulkhead {
    private final Semaphore slots = new Semaphore(5, true);  // fair: FIFO under contention
    private final AtomicInteger ok = new AtomicInteger();
    private final AtomicInteger shed = new AtomicInteger();

    String call(int id) throws InterruptedException {
        if (!slots.tryAcquire(300, TimeUnit.MILLISECONDS)) {
            shed.incrementAndGet();
            throw new IllegalStateException("bulkhead full — shed " + id);
        }
        try {
            Thread.sleep(400);                     // simulate the slow legacy call
            ok.incrementAndGet();
            return "resp-" + id;
        } finally {
            slots.release();                       // ALWAYS — even if the call threw
        }
    }
}

// Driver:
Bulkhead bh = new Bulkhead();
try (ExecutorService pool = Executors.newFixedThreadPool(20)) {
    for (int i = 0; i < 20; i++) {
        final int id = i;
        pool.execute(() -> {
            try { System.out.println(bh.call(id)); }
            catch (Exception e) { System.out.println(e.getMessage()); }
        });
    }
}
System.out.printf("ok=%d shed=%d%n", bh.ok.get(), bh.shed.get());

Expected: the first 5 run immediately; waiters that get a slot within 300 ms proceed (roughly the next 0–5 depending on timing); the rest are shed fast instead of piling up — with 400 ms calls and a 300 ms wait budget, expect ~10 ok / ~10 shed. This is exactly how a virtual-threads service limits downstream concurrency (a Semaphore replaces pool sizing), and it's the core of Resilience4j's bulkhead.

Exercise 6 — Demonstrate the ThreadLocal bleed, then fix it

Using a 2-thread pool, "handle" 6 requests where only some set a ThreadLocal user, and each request prints the user it sees. Show that requests which never set a user still see one (leaked from a previous request on the same pool thread). Fix with remove() in finally, and sketch how you'd propagate context to a child task correctly.

Solution 
static final ThreadLocal<String> USER = new ThreadLocal<>();

static void handle(int reqId, String user, boolean cleanup) {
    try {
        if (user != null) USER.set(user);
        // deep in the call stack:
        System.out.printf("req %d on %s sees user=%s%n",
            reqId, Thread.currentThread().getName(), USER.get());
    } finally {
        if (cleanup) USER.remove();
    }
}

public static void main(String[] args) throws Exception {
    ExecutorService pool = Executors.newFixedThreadPool(2);
    // PHASE 1 — no cleanup: watch anonymous requests inherit stale users
    pool.submit(() -> handle(1, "alice", false));
    pool.submit(() -> handle(2, "bob", false));
    Thread.sleep(100);
    pool.submit(() -> handle(3, null, false));   // prints alice or bob — BLEED!
    pool.submit(() -> handle(4, null, false));   // prints the other one — BLEED!
    Thread.sleep(100);
    // PHASE 2 — with cleanup=true on every request, 5 and 6 correctly see null
    pool.submit(() -> handle(5, "carol", true));
    pool.submit(() -> handle(6, null, true));    // null — as it should be
    pool.shutdown();
    pool.awaitTermination(5, TimeUnit.SECONDS);
}

Propagating to child tasks (InheritableThreadLocal won't work — pool threads already exist): capture at submit time, restore inside the task, always clean up:

static Runnable withContext(Runnable task) {
    String captured = USER.get();               // on the submitting thread, NOW
    return () -> {
        USER.set(captured);                     // on the worker thread, at run time
        try { task.run(); }
        finally { USER.remove(); }
    };
}
childPool.execute(withContext(() -> audit()));

This capture/restore wrapper is precisely what Spring's TaskDecorator and Micrometer context-propagation automate. On JDK 21+, ScopedValue + structured concurrency solve the same problem with scoping enforced by the language instead of by developer discipline.
