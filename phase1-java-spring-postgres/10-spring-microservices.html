10 — Microservices & Ecosystem

Architecture trade-offs, resilience patterns, Kafka, sagas & the outbox — the distributed-systems layer of a senior Java interview

At 7 years of experience, interviewers stop asking "what is a microservice?" and start asking "when would you not use one?" Even teams running a monolith ask these questions, because they test whether you understand distributed systems: partial failure, consistency trade-offs, and operational cost. You've spent years consuming these APIs from React — you already know what a flaky backend feels like from the browser. This page gives you the other side: why that spinner hung, why the retry made it worse, and what a senior engineer does about it. It's also the on-ramp to system-design rounds: every pattern here (circuit breakers, sagas, outbox, consumer groups) is a building block you'll be expected to reach for by name.

Monolith vs Microservices — arguing both ways T1

The single biggest signal you can send in this topic is refusing to treat microservices as "the modern default." A senior answer weighs both architectures against a specific team and product, and knows that most systems should start as a well-structured monolith.

What microservices actually buy you

BenefitWhat it really meansWhen it matters
Independent deployabilityTeam A ships the payments service Friday afternoon without coordinating with Team B's checkout release. Deploys become small and low-risk.Many teams sharing one release train; deploy contention is the #1 practical driver.
Independent scalingThe image-resizing service runs 40 pods; the admin service runs 2. You scale (and pay for) only the hot path.Wildly asymmetric load profiles (e.g., search vs. account settings).
Team autonomyEach team owns its service end-to-end: code, schema, deploy, on-call. Fewer cross-team merge conflicts and meetings.Org grows past ~3–4 teams touching the same codebase.
Fault isolationA memory leak in recommendations OOM-kills that service, not the checkout flow. Smaller blast radius — if you also add resilience patterns (see below).Parts of the system have very different availability requirements.
Tech freedomThe ML team uses Python; the core uses Java 21. Upgrade Spring Boot one service at a time.Genuinely different workloads. Often overrated — polyglot sprawl is its own cost.

What microservices cost you

CostConcrete pain
Distributed-system complexityEvery in-process method call becomes a network call that can fail, be slow, or succeed without you knowing (response lost). You now design for partial failure everywhere.
Network failure modesTimeouts, retries, retry storms, cascading failures, thundering herds. A monolith has none of these between its modules.
Data consistencyNo cross-service ACID transactions. You trade @Transactional for sagas, outboxes, and eventual consistency — a huge jump in design effort.
Observability burdenA single user request now spans 6 services. Without correlation IDs, distributed tracing, and centralized logs, debugging is archaeology.
Operational overheadN services = N CI pipelines, N dashboards, N on-call surfaces, N sets of secrets, contract versioning between them, local-dev environments that need half the company running in Docker.
LatencyAn in-process call is ~nanoseconds; a service hop is ~1–10 ms plus serialization. Chatty designs (10 hops per request) get slow fast.

The distributed monolith. The worst outcome: services split along the wrong boundaries so every feature change touches three repos, services call each other synchronously in deep chains, and they must be deployed together anyway. You pay all the microservice costs and get none of the benefits. Interviewers love hearing you name this failure mode — it shows you've seen (or can foresee) real damage.

The modular monolith: the sane default

A modular monolith is one deployable with strictly enforced internal module boundaries: each module has a public API (Java interfaces / a facade package), private internals, and ideally its own schema or table namespace. Spring Modulith and ArchUnit tests can enforce the boundaries at build time.

Modular monolith (one JVM, one deploy)          Microservices (N deploys)
┌───────────────────────────────────┐        ┌─────────┐  ┌─────────┐
│  orders/   payments/  inventory/  │        │ orders  │─▶│payments │
│  ├─api     ├─api      ├─api       │  vs    └────┬────┘  └─────────┘
│  └─internal└─internal └─internal  │             │ HTTP/Kafka
│        one database (schemas)     │        ┌────▼────┐   3 DBs, 3 pipelines,
└───────────────────────────────────┘        │inventory│   3 on-call rotations
  in-process calls, ACID txns                └─────────┘

The killer argument: module boundaries are cheap to change; service boundaries are expensive to change. Early in a product's life you don't yet know the right boundaries (they follow the domain, which is still being discovered). Get them wrong inside a monolith → refactor. Get them wrong across services → data migrations, API version dances, cross-team coordination. So: find the seams in a monolith first, extract services later along proven seams.

"When would you split?" — a decision framework

  1. Team size / cognitive load. Roughly: one codebase per 2–4 teams max. If 8 teams queue behind one release train and step on each other's code, split along team ownership lines. (Amazon's "two-pizza team owns a service" heuristic.)
  2. Deploy contention. If releases are blocked because unrelated changes share a pipeline, and trunk is red half the time because of someone else's module — that's the most concrete trigger.
  3. Independent scaling need. One workload needs 50× the resources of the rest (e.g., a PDF-rendering path), or needs different hardware (GPU).
  4. Divergent non-functionals. One part needs 99.99% availability and another is a batch back-office tool; one part must be PCI-scoped and you want the audit boundary small.
  5. A genuinely separate bounded context with a narrow, stable interface — the DDD test. If the candidate service would need to join the monolith's tables, it's not a real boundary yet.

If none of these pressures exist, the split is résumé-driven architecture. Say that (politely) in an interview.

Conway's law

"Organizations design systems that mirror their own communication structure." Practical reading: your architecture will end up matching your org chart whether you like it or not, so do the inverse Conway maneuver — shape teams around the service boundaries you want. Splitting a system across teams that must constantly coordinate just moves the coupling from code review to meetings. When an interviewer asks "how do you decide service boundaries?", mentioning team ownership and Conway's law alongside bounded contexts is a strong senior signal.

A crisp 30-second answer: "I default to a modular monolith with enforced boundaries. I extract a service when there's a concrete forcing function — deploy contention across teams, an asymmetric scaling need, or an isolation/compliance requirement — and I extract along a boundary the monolith has already proven, using strangler-fig style incremental extraction rather than a big-bang rewrite. Microservices are an organizational scaling tool more than a technical one."

Inter-service communication: sync vs async T2

Two fundamental styles, and the choice shapes everything downstream — failure modes, consistency, and debuggability.

Synchronous request/response (REST, gRPC)

Order Service ──HTTP GET /users/42──▶ User Service
      ◀────────── 200 {…} ───────────
(caller blocks; both must be up at the same moment = temporal coupling)
  • Pros: simple mental model, immediate answer, easy to debug (one request, one response), fits naturally when the caller needs the result to proceed (price check before charging).
  • Cons: temporal coupling — both services must be healthy simultaneously. Latencies add up across chains. Worst of all: cascading failure — if User Service hangs, Order Service's threads pile up waiting, its own callers pile up waiting on it, and one slow leaf takes down the tree.

Asynchronous messaging (Kafka, RabbitMQ, SQS)

Order Service ──publish OrderPlaced──▶ [ broker ] ──▶ Inventory Service
                                                  └──▶ Email Service
(producer doesn't know or care who consumes; consumers can be down and catch up later)
  • Pros: temporal decoupling (broker buffers while a consumer is down or slow), natural load-leveling, fan-out to new consumers without touching the producer, resilience by default.
  • Cons: eventual consistency (the caller gets "accepted", not "done"), harder to reason about ("who consumes this event?"), duplicate delivery to design for, and operating a broker is real work.
Sync REST/gRPCAsync messaging
CouplingTemporal + locationOnly to the message contract
Failure modeCascading failures, timeoutsGrowing consumer lag, duplicates
ConsistencyRead-your-call immediatelyEventual
Best forQueries; commands needing an immediate answerNotifications, workflows, integration events, load spikes

Rule of thumb worth saying out loud: queries tend to be sync, state-change notifications tend to be async. "Get me the user's profile" → REST. "An order was placed, whoever cares should react" → event. And prefer shallow sync call graphs: if A→B→C→D synchronously, the availability of the whole chain is the product of each hop's availability (0.99⁴ ≈ 0.96) and the latency is the sum.

OpenFeign: declarative REST clients

Spring Cloud OpenFeign turns an interface into an HTTP client — it will feel like a typed fetch wrapper generated from an interface:

@FeignClient(name = "user-service", url = "${clients.user-service.url}")
public interface UserClient {

    @GetMapping("/api/users/{id}")
    UserDto getUser(@PathVariable("id") long id);
}

// usage — Spring generates the implementation
@Service
public class OrderService {
    private final UserClient userClient;

    public OrderDetails enrich(Order order) {
        UserDto user = userClient.getUser(order.userId()); // looks local, is a network call!
        return new OrderDetails(order, user);
    }
}

# application.yml — NEVER ship a client without explicit timeouts
spring:
  cloud:
    openfeign:
      client:
        config:
          user-service:
            connectTimeout: 1000    # ms to establish the TCP connection
            readTimeout: 2000       # ms to wait for the response

Timeouts are non-negotiable on every remote call. Default timeouts are often infinite or absurdly long (e.g., 60s+). One hung downstream with no timeout = your worker threads exhaust = your service is now down too, for all endpoints, even ones that never touch that dependency. Pick timeouts from the callee's p99 latency plus headroom (e.g., p99 is 300 ms → timeout 1s), not from vibes. Connect timeout short (~1s), read timeout based on the operation. This is the single most common real-world resilience miss.

Feign's elegance is also its danger: the call looks like a local method. Treat every Feign interface as a shouted reminder that failure, latency, and retries all apply. That's why the next section exists.

Resilience patterns with Resilience4j T2

Resilience4j is the standard library for these patterns in Spring (Hystrix is dead/maintenance-mode — mention that if asked). Each pattern answers a specific failure question.

Circuit breaker — fail fast, let downstream recover

Problem: a dependency is down. Every call still burns a thread for the full timeout, and your callers pile up. Worse, your traffic keeps hammering the struggling dependency so it can never recover.

Solution: track the failure rate. Past a threshold, "open the circuit": reject calls instantly (fail fast) for a cool-down period, then probe cautiously.

            failure rate ≥ threshold
   ┌────────┐  (over sliding window)   ┌────────┐
   │ CLOSED │ ────────────────────────▶│  OPEN  │  all calls fail fast with
   │ normal │                          │        │  CallNotPermittedException
   └────────┘◀───────┐                 └───┬────┘
        ▲            │                     │ waitDurationInOpenState elapses
        │   probes   │                     ▼
        │   succeed  │                ┌───────────┐
        └────────────┴────────────────│ HALF_OPEN │ lets N trial calls through
                       probes fail ──▶│  probing  │ (permittedNumberOfCalls
                       (back to OPEN) └───────────┘  InHalfOpenState)
  • CLOSED: normal operation; outcomes recorded in a sliding window (count-based, e.g., last 100 calls, or time-based).
  • OPEN: failure rate (or slow-call rate!) crossed the threshold. Calls are rejected immediately — no thread burned, no load on the sick dependency. This is the point: fail fast for you, breathing room for them.
  • HALF_OPEN: after the wait duration, a handful of real calls are let through as probes. All good → CLOSED. Failures → back to OPEN.
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      userService:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 20              # evaluate over last 20 calls
        minimumNumberOfCalls: 10            # don't judge on tiny samples
        failureRateThreshold: 50            # % failures to trip
        slowCallDurationThreshold: 2s       # a "slow" call counts against you too
        slowCallRateThreshold: 80
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3

@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
public UserDto getUser(long id) {
    return userClient.getUser(id);
}

// same signature + trailing exception parameter
private UserDto getUserFallback(long id, Throwable t) {
    log.warn("user-service unavailable, serving placeholder for {}", id, t);
    return UserDto.anonymous(id);   // degraded but useful: cached copy, default, or partial page
}

A good fallback is a product decision, not just a try/catch: show cached recommendations, a default shipping estimate, or hide the widget — like an error boundary in React that renders something useful instead of a white screen. If there's no sensible degraded answer (e.g., payment authorization), the right fallback is a fast, clear error — never a fake success.

Retry — exponential backoff + jitter, idempotent only

Problem: transient blips (connection reset, brief 503, leader election). Solution: try again — carefully.

resilience4j:
  retry:
    instances:
      userService:
        maxAttempts: 3
        waitDuration: 200ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2       # 200ms → 400ms → 800ms
        enableRandomizedWait: true            # jitter — see pitfall
        retryExceptions:
          - java.io.IOException
          - feign.RetryableException
        ignoreExceptions:
          - com.example.BusinessException     # a 400/422 will never succeed on retry
  • Only retry transient failures: timeouts, 502/503/504, connection errors. Never a 400/401/404/422 — the request is wrong, not unlucky.
  • Only retry idempotent operations. Retrying GET is safe. Retrying POST /payments after a timeout may charge the customer twice — the timeout doesn't tell you whether the first attempt succeeded. Make writes idempotent first (idempotency keys, later section), then retry.
  • Jitter matters: if 5,000 clients fail at the same instant and all retry exactly 200 ms later, you've synchronized a retry storm — a self-inflicted DDoS wave hitting a recovering service (thundering herd). Randomizing the wait spreads the wave.

Retry amplification across layers. If the gateway retries 3×, the service retries 3×, and the client retries 3×, one user click becomes 27 requests to the struggling dependency. Retry near the edge or deep in the stack — not at every layer. Circuit breakers cap the damage.

Timeout — TimeLimiter

resilience4j:
  timelimiter:
    instances:
      userService:
        timeoutDuration: 2s
        cancelRunningFuture: true

// TimeLimiter needs an async return type
@TimeLimiter(name = "userService")
public CompletableFuture<UserDto> getUserAsync(long id) {
    return CompletableFuture.supplyAsync(() -> userClient.getUser(id));
}

In practice most timeouts live on the HTTP client itself (Feign/WebClient config, shown earlier) — TimeLimiter adds an overall budget on top, useful when a single logical operation makes several calls.

Bulkhead — isolate so one slow dependency can't drown everything

Named after ship compartments: one flooded compartment shouldn't sink the ship. Without it, a slow recommendations API can consume all of Tomcat's worker threads, taking down checkout too — even though checkout never calls recommendations.

Without bulkhead:                      With bulkhead:
[ 200 shared threads ]                 [ checkout: 150 ] [ recs: 25 ] [ other: 25 ]
slow recs call eats all 200            recs pool saturates → recs calls rejected fast,
→ WHOLE service unresponsive           checkout unaffected

resilience4j:
  bulkhead:                    # semaphore-based: caps concurrent calls
    instances:
      recommendations:
        maxConcurrentCalls: 25
        maxWaitDuration: 0     # don't queue — reject immediately
  thread-pool-bulkhead:        # dedicated pool: full isolation for async calls
    instances:
      recommendations:
        coreThreadPoolSize: 10
        maxThreadPoolSize: 25
        queueCapacity: 50

Rate limiter — protect yourself and others

resilience4j:
  ratelimiter:
    instances:
      partnerApi:
        limitForPeriod: 50          # 50 calls…
        limitRefreshPeriod: 1s      # …per second
        timeoutDuration: 100ms      # wait up to 100ms for a permit, else fail

Use it outbound to honor a partner's quota, or inbound (usually at the gateway) to protect yourself from abusive or buggy clients.

Combining patterns — order matters

Stack the annotations and Resilience4j applies them in a fixed default order (outermost first):

Retry ( CircuitBreaker ( RateLimiter ( TimeLimiter ( Bulkhead ( yourCall ) ) ) ) )

@Retry(name = "userService")
@CircuitBreaker(name = "userService", fallbackMethod = "fallback")
@TimeLimiter(name = "userService")
@Bulkhead(name = "userService")
public CompletableFuture<UserDto> getUser(long id) { … }

Why this order is sensible: the timeout bounds each individual attempt; each timed-out attempt is a failure recorded by the circuit breaker; the retry, outermost, decides whether to attempt again — and if the breaker has opened meanwhile, the retry hits CallNotPermittedException instantly instead of burning another timeout. If you put retry inside the breaker, one logical call could register as three failures and trip the breaker prematurely — a subtlety worth mentioning in an interview.

Frontend analogy that lands well: a circuit breaker is like your React app noticing the API is down and switching the whole widget to a cached/offline state instead of firing a failing fetch on every render tick.

API Gateway, service discovery, central config T2

API Gateway — the single front door

                        ┌──────────────────────┐
  Browser/Mobile ─────▶ │      API GATEWAY     │────▶ order-service
  (one origin,          │ • routing            │────▶ user-service
   one TLS cert,        │ • authN (JWT check)  │────▶ catalog-service
   one auth story)      │ • rate limiting      │
                        │ • CORS termination   │
                        └──────────────────────┘

Responsibilities you should rattle off: routing (path/host → service), authentication (validate the JWT once at the edge; services then trust the gateway or re-verify a lightweight internal token), rate limiting, CORS termination (one place sets the headers — as a frontend dev you know the pain of per-service CORS drift), plus TLS termination, request logging, and sometimes response aggregation (BFF pattern). Spring Cloud Gateway config:

# Spring Cloud Gateway
spring:
  cloud:
    gateway:
      routes:
        - id: orders
          uri: lb://order-service          # lb:// = resolve via service discovery
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
        - id: users
          uri: lb://user-service
          predicates:
            - Path=/api/users/**

Don't let the gateway grow business logic. Once it starts transforming payloads and enforcing domain rules, it becomes a shared bottleneck every team must change — a mini distributed monolith at the front door. Cross-cutting concerns only.

Service discovery

Instances come and go (autoscaling, deploys, crashes), so hardcoded hosts don't work. Two flavors:

  • Client-side (Eureka): services register with a registry and heartbeat; callers fetch the instance list and load-balance client-side (Spring Cloud LoadBalancer). Classic Netflix stack.
  • Platform-provided (Kubernetes DNS): in K8s, a Service object gives a stable DNS name (http://user-service) with built-in load balancing across healthy pods. This is why Eureka is fading — the platform does discovery. Say this: "on Kubernetes I'd just use Service DNS; Eureka matters mostly on VMs or non-K8s setups."

Centralized configuration

Awareness level: config shouldn't be baked into the jar (12-factor). Options: Spring Cloud Config Server (config in a Git repo, served to services, refreshable via @RefreshScope + actuator), or on K8s, ConfigMaps/Secrets mounted as env vars or files. Secrets belong in a secret manager (Vault, AWS Secrets Manager), never in Git plaintext.

Kafka fundamentals T2

Log, not queue — the mental model

A traditional queue (RabbitMQ classic queues) removes a message once consumed. Kafka is a distributed, append-only, durable log: messages are written to the end, retained for a configured time (e.g., 7 days) regardless of consumption, and each consumer just remembers its own position (offset). Consequences:

  • Multiple independent consumer groups read the same data without copies (fan-out for free).
  • You can replay: reset offsets and reprocess history (rebuild a projection, backfill a new service, recover from a consumer bug).
  • Consuming is cheap sequential reads; Kafka sustains enormous throughput.

Topics, partitions, ordering

topic: orders (3 partitions)
partition 0: [m0][m3][m6]…   ← strictly ordered WITHIN the partition
partition 1: [m1][m4][m7]…
partition 2: [m2][m5][m8]…

producer:  partition = hash(key) % numPartitions
           key = orderId  →  all events for one order land in ONE partition, in order
           key = null     →  round-robin/sticky, no ordering guarantee at all
  • A topic is split into partitions — the unit of parallelism and the unit of ordering. Kafka guarantees ordering only within a single partition, never across the topic.
  • The message key decides the partition. Choosing the key is a design decision: key by orderId and all events for an order are totally ordered; key by userId and one hyperactive user (or a null-key celebrity account keyed by userId) can create a hot partition that lags while its siblings idle.
  • Partition count is your max consumer parallelism, and it's painful to change (repartitioning breaks key→partition mapping), so size it with headroom.

Consumer groups

topic orders (3 partitions)      consumer group "inventory-svc"
   p0 ──────────────────────────▶ consumer A
   p1 ──────────────────────────▶ consumer B
   p2 ──────────────────────────▶ consumer B      (a partition has exactly ONE
                                                   consumer within a group)
4th consumer in this group? → sits IDLE. consumers ≤ partitions for useful scaling.

group "email-svc" reads the SAME topic independently with its own offsets.

Each partition is assigned to exactly one consumer within a group; different groups are fully independent. To scale processing, add consumers up to the partition count.

Offsets & commit strategies — where duplicates come from

  • Auto-commit (enable.auto.commit=true): offsets committed on a timer (default every 5s), decoupled from whether your processing succeeded. Crash after commit but before processing → message lost. Crash after processing but before commit → reprocessed. Fine for metrics; wrong for money.
  • Manual commit (Spring Kafka AckMode.MANUAL / RECORD): process, then acknowledge. Crash before the ack → the record is redelivered → at-least-once: you never lose a message, but you will see duplicates.

At-least-once is the practical default, therefore consumer idempotency is mandatory, not optional. Design every consumer to survive seeing the same event twice: check a processed-event-ID table before acting, use DB upserts/unique constraints, or make the operation naturally idempotent ("set status = SHIPPED" is safe to repeat; "balance += 10" is not). Interviewers probe this constantly.

Producer side: acks and idempotence

acksMeaningTrade-off
0Fire and forget — don't wait for the brokerFastest; silent loss on any failure. Metrics/telemetry only.
1Leader wrote itLost if the leader dies before replication.
allAll in-sync replicas have it (pair with min.insync.replicas=2)Durable; slightly higher latency. The default for anything that matters.

Idempotent producer (enable.idempotence=true, on by default in modern clients): the broker deduplicates producer retries using a producer ID + sequence number, so a retried send can't create a duplicate in the log. Note the scope: it fixes producer-retry duplicates only — it does nothing about your consumer processing a message twice.

Exactly-once semantics — the honest answer

Kafka offers transactions: a consumer-transform-produce pipeline can commit consumed offsets and produced messages atomically, with downstream consumers reading isolation.level=read_committed. That is real exactly-once — but only within Kafka (topic → app → topic, e.g., Kafka Streams). The moment a side effect leaves Kafka — writing Postgres, calling Stripe, sending an email — no broker setting can make it exactly-once. There's no distributed transaction spanning Kafka and your DB. The engineering answer is: at-least-once delivery + idempotent effects = effectively-once outcomes. Saying exactly that sentence in an interview is worth more than reciting transaction configs.

Rebalancing

When a consumer joins/leaves/crashes (or is presumed crashed because it didn't poll within max.poll.interval.ms), the group rebalances: partitions are reassigned. During a classic (eager) rebalance all consumers stop — a processing pause. Modern clients use the cooperative sticky assignor which moves only the partitions that must move. Classic incident: a slow consumer takes too long processing a batch → coordinator evicts it → rebalance → its records get redelivered elsewhere → more load → more slowness. Fixes: smaller max.poll.records, faster processing, or move heavy work off the poll thread.

Dead-letter queue (DLQ)

A poison message (bad payload, bug in the handler) will fail forever; endless retries block the partition behind it (head-of-line blocking). Pattern: retry a few times, then publish the failing record to orders.DLT with error metadata, commit, move on. Alert on DLQ depth; build a redrive path for after the fix.

// Spring Kafka: listener + retry-then-DLQ error handling
@Component
public class OrderEventsListener {

    @KafkaListener(topics = "orders", groupId = "inventory-svc")
    public void onOrderPlaced(ConsumerRecord<String, OrderPlaced> record,
                              Acknowledgment ack) {
        OrderPlaced event = record.value();
        if (processedEvents.alreadyHandled(event.eventId())) { // idempotency guard
            ack.acknowledge();
            return;
        }
        inventoryService.reserve(event.orderId(), event.lines());
        processedEvents.markHandled(event.eventId());   // same DB tx as reserve()
        ack.acknowledge();                              // manual commit AFTER success
    }
}

@Bean
DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
    var recoverer = new DeadLetterPublishingRecoverer(template); // → orders.DLT
    return new DefaultErrorHandler(recoverer,
            new ExponentialBackOff(500L, 2.0));  // retry with backoff, then DLQ
}

Kafka vs RabbitMQ — decision table

One-liner: "Kafka when the data is a stream of facts multiple systems care about (and replay matters); RabbitMQ when it's jobs to be done once with rich routing. And I wouldn't introduce Kafka for one queue's worth of traffic — it's an operational commitment."

Distributed data: sagas, outbox, idempotency T2

Database-per-service and its consequence

Each service owns its database exclusively; nobody else touches its tables. This is what makes services independently deployable (schema changes don't ripple) and independently scalable. The price is brutal and you must own it out loud:

  • No cross-service joins. "Orders with customer names" now means API composition (call both, join in memory), data duplication (orders keeps a copy of the customer name, updated via events), or a CQRS read model fed by events.
  • No cross-service ACID transactions. You cannot atomically debit the wallet in one service and reserve stock in another.

Why not 2PC (two-phase commit)?

2PC gives distributed atomicity via a coordinator (prepare → all vote → commit). It's avoided in microservices because it's a blocking protocol: while a participant is "prepared," it holds locks waiting for the coordinator's verdict. If the coordinator dies at the wrong moment, participants sit locked, in doubt, potentially for a long time. Availability of the whole transaction is the product of every participant's availability, latency is the sum, and it needs XA support everywhere (Kafka and most modern stores don't play). It optimizes consistency by sacrificing exactly the availability and autonomy microservices exist to provide. Hence: sagas.

SAGA pattern — in depth

A saga replaces one distributed ACID transaction with a sequence of local transactions, each committed in one service. If step N fails, you don't roll back (can't — earlier steps already committed); you execute compensating transactions that semantically undo the completed steps (refund the payment, release the reservation). Consistency is eventual; there's a window where the system is mid-saga.

     │                         │                       ├── StockReserved ──▶ Order → CONFIRMED
     │                         │                       │
     │                 FAILURE PATH (out of stock):    │
     │                         │◀── StockRejected ─────┤
     │                         │ refund (compensate)   │
     │◀── PaymentRefunded ─────┤                       │
     │ order → CANCELLED (compensate)                  │

Pros: no central component, services stay loosely coupled, each service only knows "which events do I react to." Cons: the workflow exists nowhere — it's an emergent property of event subscriptions. With 6+ steps, answering "what's the flow when payment succeeds but inventory times out?" requires archaeology across repos. Cyclic event dependencies can sneak in. Hard to see, hard to test end-to-end.

Orchestration — a central coordinator

            ┌──────────────────────────────┐
            │     Order Saga Orchestrator  │  (a state machine per order)
            └──┬───────────┬───────────┬───┘
   1. ChargePayment   2. ReserveStock   on step-2 failure:
       │  ▲                │  ▲          3. RefundPayment (compensation)
       ▼  │ reply          ▼  │ reply    4. CancelOrder
   Payment Svc         Inventory Svc     (commands via queues or REST)

// sketch of an orchestrator's state machine (conceptual)
switch (saga.state()) {
    case STARTED          -> send(new ChargePayment(orderId, amount));
    case PAYMENT_DONE     -> send(new ReserveStock(orderId, lines));
    case STOCK_RESERVED   -> complete(orderId);                      // happy end
    case STOCK_REJECTED   -> send(new RefundPayment(orderId));       // compensate
    case PAYMENT_REFUNDED -> cancel(orderId);                        // compensate
}

Pros: the workflow is explicit, in one place — readable, testable, easy to add timeouts/monitoring ("all sagas stuck > 1h" is a query). Cons: the orchestrator couples to every participant's API, and can drift toward a god-service if business logic leaks into it (it should sequence, not decide domain rules). Tools: Temporal, Camunda, Axon, or a hand-rolled state table.

Compensations are semantic undo, not rollback — and some actions can't be fully undone (the email was sent; the money left via a card network with fees). Design steps so risky/irreversible ones happen last ("pivot transaction"), and accept that compensation itself can fail, which needs retries + alerting + occasionally a human.

Outbox pattern — the interview classic

The question: "Your service saves an order to Postgres and publishes OrderPlaced to Kafka. How do you keep them consistent?" The naive code has a fatal window either way:

// BROKEN — pick your failure:
orderRepository.save(order);          // tx commits…
kafkaTemplate.send("orders", event); // …then app crashes here → DB updated, event NEVER sent

// swap the order? send first, then save → crash → event sent for an order that DOESN'T EXIST
// send inside @Transactional? Kafka doesn't join the DB transaction — same problem, hidden.

The fix: write the event into the same database, in the same local transaction, and let a separate process deliver it.

┌──────────────── one ACID transaction ────────────────┐
│  INSERT INTO orders (…)                              │
│  INSERT INTO outbox (id, aggregate_id, type, payload)│   ← atomic: both or neither
└──────────────────────────────────────────────────────┘
            │
            ▼  afterwards, asynchronously
   ┌─────────────────┐         ┌────────┐
   │  Relay          │ ──────▶ │ Kafka  │   two delivery options:
   └─────────────────┘         └────────┘
   (a) polling publisher: SELECT unsent rows → publish → mark sent
   (b) CDC (Debezium): tails the Postgres WAL, streams outbox inserts
       to Kafka with no polling and low latency

@Transactional
public void placeOrder(PlaceOrderCommand cmd) {
    Order order = orderRepository.save(Order.from(cmd));
    outboxRepository.save(OutboxEvent.of(          // SAME transaction → atomic
        "OrderPlaced", order.getId(), toJson(order)));
}   // relay/Debezium publishes later; if publish fails it retries → at-least-once

Delivery becomes at-least-once (the relay may crash after publishing but before marking sent) — which is fine, because your consumers are idempotent. Right? That's the closing line interviewers want.

Idempotency keys — for consumers and APIs

  • Consumers: every event carries a unique eventId; the consumer records handled IDs in a processed_events table in the same transaction as its business write. A duplicate hits the unique constraint → skip. (This is what the @KafkaListener example above does.)
  • APIs: for non-idempotent operations (POST /payments), the client generates an Idempotency-Key header (UUID) per logical operation; the server stores key → response and replays the stored response on retry. Exactly Stripe's model — your React app retrying a failed checkout fetch can't double-charge.
POST /payments

{ "orderId": 42, "amount": 99.00 }

Eventual consistency & the frontend T2

This is where your React years become an interview asset. Eventual consistency isn't just a backend property — it leaks into UX, and a senior full-stack engineer designs both sides together.

What to tell the frontend

  • Return 202-style semantics honestly. When an operation kicks off a saga, respond 202 Accepted with a status URL or resource in PENDING state — not a fake 200 "done". The contract says "we've durably recorded your intent," which (thanks to the outbox) is exactly true.
  • Optimistic UI: you've done this with React Query / SWR mutations — update the cache immediately, reconcile on settle. Same idea: show the order as "Placing…" instantly, upgrade to "Confirmed" when the saga completes (poll the status endpoint, or push via WebSocket/SSE), and roll back the UI with an explanation if the saga compensates ("Payment declined — order cancelled").
  • Design explicit pending states. PENDING → CONFIRMED | CANCELLED should be a first-class status in the API and the UI, not something the frontend infers from timeouts. Amazon does this: "Thanks for your order" is not "your card was charged" — that email comes later.

Read-your-own-writes

Classic bug: user submits an edit, the write goes to service A, the next page load reads from a lagging replica or an event-fed read model in service B — and their change "disappears." Users forgive other people's data being seconds stale; they do not forgive their own write vanishing. Mitigations, cheapest first:

  1. Return the written state in the write response and seed the client cache from it (React Query's setQueryData after mutation) — often you never need the immediate re-read at all.
  2. Pin the reader: for N seconds after a user's write, route their reads to the primary / the owning service (session stickiness or a "recently wrote" flag).
  3. Version-aware reads: the write returns a version/offset; the read waits until the replica or read model has caught up to at least that version.

Strong framing for interviews: "Consistency is a per-use-case budget, not a global switch. The user's own writes need read-your-writes; the product catalog can be minutes stale; inventory counts on a listing page can be approximate as long as checkout re-validates. I push for the weakest consistency each screen can tolerate, because that's what buys availability and latency."

Do not wire external dependencies (DB, Kafka) into the liveness probe. If the database blips, every pod's liveness fails, K8s restart-loops your entire healthy fleet, and a 30-second DB hiccup becomes a full outage. Dependencies belong in readiness only — pods step out of rotation and step back in when the DB returns.

Docker for Java — layered jars

Docker images are stacked layers; a layer rebuilds (and re-uploads) only if its contents changed. A naive fat-jar image puts your 5-line code change and 80 MB of unchanged dependencies in one layer — every build ships 80 MB. Spring Boot's layered-jar mode splits the jar so dependencies (rarely change) sit below your classes (change every commit):

# multi-stage Dockerfile with Spring Boot layertools
FROM eclipse-temurin:21-jre AS builder
WORKDIR /app
COPY target/app.jar app.jar
RUN java -Djarmode=tools -jar app.jar extract --layers --destination extracted

FROM eclipse-temurin:21-jre
WORKDIR /app                                     # least → most volatile:
COPY --from=builder /app/extracted/dependencies/          ./   # cached across builds
COPY --from=builder /app/extracted/spring-boot-loader/    ./
COPY --from=builder /app/extracted/snapshot-dependencies/ ./
COPY --from=builder /app/extracted/application/           ./   # only this rebuilds
ENTRYPOINT ["java", "-jar", "app.jar"]

Also mention: run as a non-root user; modern JVMs are container-aware (heap sized from cgroup limits — set -XX:MaxRAMPercentage, don't hardcode -Xmx blindly).

Interview Q&A

When would you choose a monolith over microservices?

Almost always at the start. A modular monolith gives fast iteration, ACID transactions, trivial debugging, and one pipeline — while domain boundaries are still being discovered. I'd choose microservices only under concrete pressure: multiple teams blocked by deploy contention, a component needing independent scaling, or isolation/compliance boundaries. Microservices trade development simplicity for organizational scalability; if you don't have the organizational problem, you're paying the distributed-systems tax for nothing.

What is a "distributed monolith" and how does it happen?

Services that are physically separate but logically coupled: every feature touches several of them, they call each other in deep synchronous chains, share databases or fragile contracts, and must be released together. You get all microservice costs (network failures, ops overhead, tracing pain) with none of the benefits (independent deploy/scale). It happens when systems are split by technical layer or by org-chart accident instead of along bounded contexts, or when teams split before boundaries were proven in the monolith.

Explain Conway's law and its practical implication for architecture.

Systems mirror the communication structure of the organizations that build them. Practically: service boundaries that don't match team boundaries generate constant cross-team coordination, and the architecture will erode toward the org chart anyway. The inverse Conway maneuver applies it deliberately — restructure teams to match the architecture you want, giving each team end-to-end ownership of its services.

How do you decide where service boundaries go?

Along bounded contexts from the domain: a service should own a cohesive set of data and behavior with a narrow, stable interface. Litmus tests: can it be described without mentioning another service's internals? Would it need to join another service's tables (bad sign)? Does one team fully own it? Do most changes stay within one boundary? I'd validate boundaries as modules inside a monolith first — module boundaries are cheap to move, service boundaries are not.

Sync REST vs async messaging — how do you choose?

Queries and commands that need an immediate answer are sync (get user profile, validate a coupon at checkout). State-change notifications and workflows tolerant of delay are async (order placed → email, analytics, inventory). Sync is simpler but temporally couples services and creates cascading-failure risk in deep chains; async decouples availability and levels load, at the cost of eventual consistency and duplicate handling. I keep sync call graphs shallow and push cross-service side effects to events.

Why are timeouts on every remote call non-negotiable?

Without a timeout, a hung dependency holds your worker thread indefinitely; under load the pool exhausts and your whole service stops responding — including endpoints unrelated to that dependency — and the failure cascades to your callers. Defaults are often effectively infinite. I set connect timeouts ~1s and read timeouts from the callee's p99 plus headroom, and pair them with a circuit breaker so persistent slowness turns into fast failure.

Walk me through circuit breaker states.

CLOSED: calls flow normally; outcomes recorded in a sliding window. When the failure rate (or slow-call rate) over a minimum number of calls crosses the threshold, it transitions to OPEN: calls are rejected instantly with CallNotPermittedException — fail fast for the caller, breathing room for the sick downstream. After a wait duration it goes HALF_OPEN: a limited number of trial calls pass through; if they succeed the breaker closes, if they fail it reopens. Key configs: sliding window size, minimumNumberOfCalls (avoid judging tiny samples), failureRateThreshold, waitDurationInOpenState.

Why does a circuit breaker help the failing service, not just the caller?

A struggling service usually needs reduced load to recover — it's saturated on CPU, connections, or GC. If every caller keeps retrying at full rate, it never gets that window; breakers collectively shed load. That's also why retries without breakers are dangerous: they increase load exactly when the target can least handle it.

When is retrying a request actively harmful?

Three cases. Non-idempotent operations: retrying a timed-out payment POST may execute it twice, since a timeout doesn't reveal whether the first attempt succeeded — fix with idempotency keys first. Non-transient errors: a 400/422 will fail identically every time; retrying wastes capacity. Overload: if the downstream is saturated, synchronized retries create a retry storm that deepens the outage — hence exponential backoff with jitter, retry budgets, and retrying at one layer only, capped by a circuit breaker.

What is the bulkhead pattern and what incident does it prevent?

Partitioning resources (threads, connections, concurrency permits) per dependency so one slow dependency can't consume everything. Classic incident: a slow third-party recommendations API gradually occupies all 200 Tomcat worker threads because each request waits on it; now checkout — which never touches recommendations — is down too. With a 25-permit bulkhead around that client, the 26th concurrent call fails fast (or gets a fallback) and the rest of the service keeps working. Semaphore bulkheads cap concurrency in-thread; thread-pool bulkheads give full isolation for async calls.

In what order should retry, circuit breaker, and timeout compose, and why?

Timeout innermost (bounds each attempt), circuit breaker around that (each timed-out attempt is recorded as a failure), retry outermost (decides whether to attempt again; if the breaker opened mid-sequence, the retry fails instantly instead of burning another full timeout). Resilience4j's default matches this: Retry(CircuitBreaker(RateLimiter(TimeLimiter(Bulkhead(call))))). Putting retry inside the breaker makes one logical call count as multiple failures, tripping the breaker prematurely.

What belongs in an API gateway — and what doesn't?

Cross-cutting edge concerns: routing, TLS termination, authentication (validate the JWT once), rate limiting, CORS, request logging, sometimes response aggregation for specific frontends (BFF). What doesn't: business logic and payload transformation tied to domain rules — that turns the gateway into a shared bottleneck every team must modify, coupling all services through it. The gateway should be boring and thin.

Why is ordering in Kafka only guaranteed per partition, and how do you exploit that?

A topic's partitions are independent logs consumed in parallel by different consumers — there's no global sequence across them by design; that independence is where the throughput comes from. Order exists only within one partition. You exploit it via the message key: hash(key) picks the partition, so keying by orderId puts all events for one order in one partition, in order. The trade-offs: key skew creates hot partitions, and repartitioning changes key→partition mapping, so choose keys and partition counts deliberately.

You have 12 partitions and add a 13th consumer to the group. What happens?

It sits idle. Within a consumer group, each partition is assigned to at most one consumer, so useful parallelism is capped at the partition count. The 13th instance is only a warm standby that takes over on a rebalance. If I need more parallelism I need more partitions (planned ahead — resizing breaks key ordering assumptions) or faster per-record processing.

Compare Kafka's acks=0, 1, and all.

acks=0: producer doesn't wait — fastest, silently loses data on any broker issue; only for disposable telemetry. acks=1: leader acknowledged, but data is lost if the leader dies before followers replicate. acks=all with min.insync.replicas=2: acknowledged only when the in-sync replica set has it — survives a broker loss; the default choice for business data, at modest latency cost. I'd also enable the idempotent producer so broker-side retries can't duplicate records in the log.

Is exactly-once delivery possible? Be precise.

Within Kafka, yes: transactions atomically commit consumed offsets plus produced records, and read_committed consumers see only committed data — real exactly-once for Kafka-to-Kafka pipelines (Kafka Streams uses this). Across system boundaries, no: there's no atomic commit spanning Kafka and your database or a payment provider. The honest engineering answer is at-least-once delivery plus idempotent processing, which yields exactly-once effects. Any claim of universal exactly-once delivery should make you suspicious.

What causes a Kafka consumer rebalance, and why can rebalances cascade?

Membership changes: a consumer joins, leaves, crashes, or is evicted because it didn't poll within max.poll.interval.ms or missed heartbeats. During eager rebalances consumption pauses group-wide (cooperative sticky assignment reduces this to only the moved partitions). The cascade: a consumer processing a slow batch gets evicted → rebalance → its partitions and unacked records shift to survivors → they slow down too → more evictions. Fixes: lower max.poll.records, keep the poll loop fast, move heavy work off the polling thread.

What's a dead-letter queue and when does a message go there?

A side topic for messages that keep failing — a poison message (malformed payload, handler bug) would otherwise be retried forever, blocking every record behind it in the partition (head-of-line blocking). Pattern: bounded retries with backoff, then publish to orders.DLT with error metadata, commit the offset, continue. Operationally: alert on DLQ depth, and have a redrive process to replay messages after deploying the fix.

Why is 2PC avoided in microservice architectures?

It's a blocking protocol: after voting "prepared," participants hold locks awaiting the coordinator's decision; if the coordinator fails at that moment they're stuck in-doubt, holding locks. The transaction's availability is the product of all participants' availability, latency is additive, and it requires XA support that Kafka and most modern datastores don't offer. It sacrifices availability and autonomy — the two things microservices exist to maximize. Sagas invert the trade: local commits plus compensations, accepting eventual consistency.

Explain the saga pattern with a concrete failure scenario.

Order placement: (1) order service saves order PENDING, (2) payment service charges the card, (3) inventory reserves stock. Each step is a local transaction. If step 3 fails — out of stock — we can't roll back the committed payment, so we compensate: refund the payment, then cancel the order. Choreography does this via events (PaymentCompleted → StockRejected → PaymentRefunded), simple but the flow is invisible; orchestration uses a central state machine issuing commands, explicit and monitorable but coupled to all participants. I'd choreograph short flows and orchestrate anything with 4+ steps or timeout/SLA requirements. Also: sequence irreversible steps last, and remember compensations themselves can fail and need retries plus alerting.

How do you atomically update the database AND publish an event? (The outbox question.)

You can't do it directly — Kafka doesn't participate in the DB transaction, so save-then-publish loses events on a crash between the two, and publish-then-save emits events for state that never committed. The outbox pattern fixes it: in the same ACID transaction as the business write, insert the event into an outbox table — atomic by construction. A separate relay delivers it: either a poller that selects unsent rows and publishes, or Debezium CDC tailing the WAL for low latency. Delivery becomes at-least-once (relay can crash after publish, before marking sent), which is fine because consumers are idempotent via event-ID dedup.

How do idempotency keys work for a payment API?

The client generates a unique key (UUID) per logical operation and sends it as an Idempotency-Key header, reusing the same key on retries. The server inserts the key into a table with a unique constraint alongside the operation; a retry hits the constraint, and the server returns the stored response instead of re-executing. This makes a non-idempotent POST safely retryable end-to-end — a timed-out checkout retry can't double-charge. Stripe's API is the canonical example. Keys need a TTL/cleanup policy and should be scoped per operation type.

The UI shows stale data right after the user's own update. What's happening and how do you fix it?

A read-your-own-writes violation: the write hit the primary/owning service, but the follow-up read hit a lagging replica or an event-fed read model that hasn't caught up. Fixes in order of cheapness: return the full written state in the write response and seed the client cache from it (often eliminates the re-read); pin that user's reads to the primary for a short window after a write; or use version-aware reads where the write returns a version and reads wait for the replica to reach it. Users tolerate others' data being stale, never their own write vanishing — so I budget consistency per use case.

What are the three pillars of observability and what does each answer?

Logs — discrete events with context: "what exactly happened?"; structured JSON, centralized, joined across services by trace/correlation ID (via MDC). Metrics — cheap numeric aggregates over time: "is the system healthy, and is it trending worse?"; Micrometer → Prometheus, RED (rate/errors/duration) per endpoint, always percentiles not averages. Traces — the tree of timed spans one request produces across services: "where did the latency go / which hop failed?"; propagated via the W3C traceparent header, visualized in Jaeger/Tempo/Zipkin. They compose: an alert fires on metrics, the trace localizes the slow hop, the logs on that trace ID explain why.

Difference between liveness and readiness probes — and the classic mistake?

Liveness asks "is this process irrecoverably broken?" — failure triggers a container restart. Readiness asks "can it serve traffic right now?" — failure just removes the pod from load balancing. The classic mistake is checking the database in the liveness probe: a brief DB outage then fails liveness on every pod, Kubernetes restart-loops the whole healthy fleet, and a small blip becomes a self-inflicted total outage. External dependencies belong in readiness only.

Why layered jars for Java Docker images?

Image layers are cached and only rebuilt/re-pushed when their content changes. A fat jar puts ~80MB of rarely-changing dependencies and your constantly-changing classes in one layer, so every commit rebuilds and ships everything. Spring Boot's layertools extracts the jar into layers ordered by volatility — dependencies, loader, snapshot deps, application — so a typical build only rebuilds the tiny application layer: faster CI, faster pulls, less registry storage.

Hands-on exercises

Exercise 1 — Argue the split (design review)

Your company runs a Spring Boot monolith: orders, payments, catalog, notifications. 5 backend teams, ~40 engineers. Symptoms: releases happen weekly because QA must regression-test everything; the catalog search feature needs 10× the compute of everything else during sales; a notifications bug once OOM-killed the whole app during Black Friday. The CTO says "let's do microservices." Write your recommendation: what would you extract (if anything), in what order, and what would you explicitly NOT split? Name the prerequisites you'd demand before any extraction.

Model answer

Diagnosis first: three real forcing functions exist — deploy contention (weekly train, 5 teams), asymmetric scaling (catalog search), and blast radius (notifications OOM took down checkout). That justifies targeted extraction, not a full decomposition.

Order of extraction: (1) Notifications first — it's naturally async (consume OrderPlaced-style events), has no synchronous callers, minimal data coupling, and it caused the worst incident; lowest risk, immediate blast-radius win. (2) Catalog search second — read-heavy, scales independently, can be fed by CDC/events from the catalog data with eventual consistency being acceptable for search results. (3) Payments only if PCI scope reduction or team ownership demands it — it's the scariest to move because of transactional coupling with orders.

Explicitly not split: orders + core checkout stays in the (now modular) monolith — it's the transactional heart; splitting it forces sagas immediately for little gain. Also don't split by layer (no "backend-for-entities service").

Prerequisites before any extraction: module boundaries enforced inside the monolith first (prove the seam); CI/CD able to deploy services independently; observability baseline (centralized structured logs, tracing with traceparent propagation, RED dashboards); an event backbone (Kafka) with outbox in the monolith so extracted services can subscribe; on-call/ownership model agreed per team (Conway alignment). Method: strangler fig — route traffic gradually, keep the old path until parity is proven.

Exercise 2 — Design the order saga

Design order placement across order-service, payment-service, and inventory-service. Requirements: no order may be confirmed without successful payment AND stock reservation; a customer must never be charged for an unconfirmed order for more than a few minutes; the flow must be observable ("show me all stuck orders"). Choose choreography or orchestration and justify; enumerate every failure point and its handling; specify the events/commands and compensations.

Model answer

Choice: orchestration. Three-plus steps, a hard business rule about compensation timing ("never charged more than a few minutes"), and an explicit observability requirement — a central saga state machine gives me a queryable table of every saga's state (SELECT * FROM sagas WHERE state != 'DONE' AND updated_at < now() - interval '5 min' = "stuck orders" for free). Choreography would scatter that state across three services.

Happy path: order-service saves order PENDING + saga row (same tx, outbox for the first command) → orchestrator sends ChargePayment → payment-service replies PaymentCompleted → orchestrator sends ReserveStock → inventory replies StockReserved → orchestrator marks order CONFIRMED. Ordering choice: reserve stock could go first (releasing a reservation is a gentler compensation than refunding money) — either is defensible; charging last means the irreversible-ish step is the pivot.

Failure points: (a) payment declined → saga → order CANCELLED(payment_declined), no compensation needed. (b) stock unavailable → send RefundPayment, await PaymentRefunded, order CANCELLED(out_of_stock). (c) payment-service times out — ambiguous! Don't blindly retry charge; query payment status by idempotency key, or retry ChargePayment safely because it carries an idempotency key. (d) refund fails → retry with backoff; after N attempts, park saga in COMPENSATION_FAILED and page a human — money movement gets an operator, not silent retries forever. (e) orchestrator crashes mid-saga → state is in the DB; on restart a recovery poller resumes any saga not in a terminal state. (f) duplicate replies (at-least-once messaging) → saga state machine ignores replies that don't match the current state; participants dedupe commands by command ID.

The "few minutes" rule: a timeout watchdog — any saga in PAYMENT_DONE longer than T triggers automatic compensation (refund) rather than waiting indefinitely for inventory.

Transport: commands/replies over Kafka (per-saga key = orderId for ordering), all writes via outbox, all consumers idempotent.

Exercise 3 — Make this endpoint survive a flaky dependency

Given: GET /api/products/{id} composes core product data (own DB) with live pricing from pricing-service (recent p99: 180 ms; had two 20-minute outages last month) and recommendations from recs-service (p99: 900 ms, frequently slow). Currently both are plain Feign calls with defaults, and product pages hang when either misbehaves. Specify the full resilience config (timeouts, breakers, bulkheads, retries, fallbacks) per dependency, with numbers and reasoning.

Model answer

Pricing (critical-ish, fast, occasionally down): connect timeout 500 ms, read timeout 600 ms (p99 180 ms × ~3 headroom). Retry: 2 attempts total, 100 ms backoff with jitter, on IO/timeout only — the call is a GET (idempotent) and fast enough that one retry fits the budget. Circuit breaker: count-based window 20, minimumNumberOfCalls 10, failureRateThreshold 50%, slowCallDurationThreshold 500 ms, waitDurationInOpenState 15 s — during those 20-minute outages the breaker stays open and probes every 15 s instead of hammering. Fallback: last-known price from a short-TTL cache with a "price may have changed, confirmed at checkout" flag; if no cached price, show the product without a price and disable buy-now (product decision — never show a guessed price). Bulkhead: 30 concurrent permits.

Recommendations (optional, slow): this is decoration, so budget it tightly: read timeout 400 ms — deliberately below its p99; we'd rather drop recs on slow tail requests than delay the page. No retries (optional data, retrying a slow service adds load for marginal value). Breaker: slowCallRateThreshold 80% at slowCallDurationThreshold 300 ms so chronic slowness opens it, not just errors. Bulkhead: small — 10 permits — so recs can never eat the thread pool (this was the original hang). Fallback: empty list / hide the carousel; the frontend already renders fine without it (like a failed non-critical widget behind an error boundary).

Composition: fetch pricing and recs in parallel (CompletableFuture), overall endpoint budget ~800 ms. Emit metrics per dependency (breaker state, fallback rate) and alert on fallback rate > 5% — resilience without observability just hides outages.

Exercise 4 — Kafka topic design for order events

Design the Kafka setup for order events consumed by inventory-service (must process each order's events in order, ~500 events/s peak), analytics (can lag, wants full history for 30 days), and a future fraud service. Specify: topic(s), partition count, message key, producer settings, consumer group layout, commit strategy, and how you handle a poison message. Then answer: what breaks if someone publishes with a null key "for performance"?

Model answer

Topic: one topic orders.events (all lifecycle events: placed, paid, shipped…) — per-order ordering across event types requires them in one topic/partition. Retention 30 days (analytics requirement). Partitions: 12 — 500/s is modest, but partitions are hard to add without breaking key mapping, so provision several times current peak; 12 gives good divisor options for consumer counts. Key: orderId — guarantees all events for one order hit one partition in order; cardinality is high and uniform, so no hot-partition risk (keying by customerId would risk skew from big B2B customers).

Producer: acks=all, min.insync.replicas=2, idempotence enabled, and events written via the outbox pattern from order-service (never dual-write). Each event carries a unique eventId.

Consumers: three independent groups — inventory-svc, analytics, fraud-svc (added later with zero producer changes; it can even replay 30 days of history by starting from earliest — the log model's payoff). Inventory: up to 12 instances, manual ack after processing, idempotency via processed_events(event_id) unique insert in the same DB tx as the stock mutation — at-least-once delivery, exactly-once effect. Analytics: auto-commit is acceptable (tolerates loss/dupes in aggregates) or batch commits for throughput.

Poison messages: DefaultErrorHandler with ExponentialBackOff (e.g., 5 attempts) then DeadLetterPublishingRecoverer → orders.events.DLT; alert on DLT depth; redrive tool replays after a fix. This prevents head-of-line blocking of the partition.

Null key: events for the same order round-robin across partitions, so inventory can process "shipped" before "paid" — ordering silently breaks, likely as rare, unreproducible state-machine bugs under load. Also, per-order replay/compaction options die. The "performance" gain is negligible here; the correctness loss is severe. This is exactly why key choice is a design review item, not a config detail.
