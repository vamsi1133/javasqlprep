13 — Indexing & Query Performance

B-tree mechanics, EXPLAIN ANALYZE fluency, and the slow-query diagnosis story every senior interview wants to hear

Why interviewers ask this

Every senior full-stack interview eventually arrives at some version of: "A page in production got slow. The team says it's the database. Walk me through what you do." This is not trivia — it's a probe for whether you have a diagnosis-first mindset (measure, hypothesize, verify) or whether you'll cargo-cult "add an index" and hope. Indexing is the highest-yield database topic because it connects everything: data structures (B-trees), the query planner, statistics, write amplification, and app-level behavior like N+1 queries and pagination.

As a frontend developer you already have the right instincts from performance work: you'd never "optimize" a React app without the Profiler flamegraph. EXPLAIN ANALYZE is the database's flamegraph. This page teaches you to read it fluently, predict what the planner will do, and — critically — tell the "we had a slow query" story with real numbers. That story is a guaranteed interview moment; by the end of this page you'll have a complete, realistic one you can rehearse.

Everything here is PostgreSQL-specific where it matters (heap tables, visibility map, HOT updates), but the B-tree mechanics and plan-reading skills transfer to MySQL, SQL Server, and Oracle almost unchanged. If the interviewer uses MySQL, say so and note the one big difference: InnoDB clusters the table on the primary key; Postgres heaps are unordered and every index — including the PK — points at heap tuples.

B-tree mechanics: how an index actually works T1

A B-tree index is a balanced, sorted, block-oriented tree. Postgres stores it in 8 KB pages, exactly like the table (the "heap"). There are three kinds of pages:

  • Root page — one page; the entry point. Holds sorted separator keys and child-page pointers.
  • Internal pages — more separator keys and pointers, fanning out. With ~8 KB pages, a single page can hold hundreds of entries, so the fan-out is enormous.
  • Leaf pages — sorted (key, ctid) pairs, where the ctid is the physical address of the row in the heap: (block_number, item_offset). Leaf pages are linked left-to-right (and right-to-left) into a doubly-linked list, which is what makes range scans and ORDER BY traversal cheap.

Why lookups are O(log n)

Each step down the tree eliminates all but one child's worth of keys. Because fan-out is in the hundreds, the tree stays very shallow: a 1-million-row index on a bigint is typically 3 levels; a billion rows is 4–5. So an equality lookup touches a handful of pages regardless of table size — that's the O(log n), and in practice "log" with base ~300, not base 2.

Equality vs. range traversal

  • Equality (WHERE id = 4242): binary-search the root, descend to an internal page, descend to the leaf, binary-search inside the leaf, emit matching (key, ctid) entries.
  • Range (WHERE created_at BETWEEN x AND y): descend once to the leaf containing the lower bound, then walk the leaf-page linked list rightward until the upper bound. One descent + a sequential leaf walk. This is also how an index satisfies ORDER BY created_at without sorting.

The heap hop: why an index scan isn't free

The index gives you a ctid, not the row. Postgres must then fetch the heap page and read the tuple (to get the other columns and to check row visibility under MVCC — the index alone doesn't know whether a tuple is visible to your transaction). If you fetch 5,000 rows via an index and those rows are scattered across 5,000 different heap pages, that's 5,000 random page reads. On a cold cache with spinning disks this is catastrophic; even on SSDs it's real work. This is exactly why the planner will sometimes refuse your index: past a certain selectivity, reading the whole table sequentially is cheaper than millions of random hops.

The word to drop in interviews is correlation. pg_stats.correlation tells the planner how well the physical heap order matches the index order. An index on created_at for an append-only table has correlation ≈ 1.0 — range scans hit consecutive heap pages, so the planner prices index scans cheaply. An index on email has correlation ≈ 0 — every fetch is a random hop. Same index type, wildly different real cost. Mentioning correlation signals you understand cost, not just structure.

Write cost: every index is a tax

On INSERT, Postgres must add an entry to every index on the table — descend each tree, insert into the correct leaf, and split the leaf page if it's full (splits ripple upward occasionally). On UPDATE, Postgres writes a whole new row version (MVCC), so by default every index needs a new entry pointing at the new version — even indexes on columns you didn't change (the HOT optimization can avoid this; see When indexes hurt). DELETE doesn't remove index entries immediately; vacuum cleans them later, which is where index bloat comes from.

"Indexes make reads faster" is the junior framing. The senior framing: an index is a copy of part of your table, maintained synchronously on every write, that lets you trade write throughput and storage for read latency. Every index must earn its keep.

What CAN'T use a B-tree index — the trap list T1

Interviewers love these because each one looks like it "should" use the index. The unifying principle: a B-tree can only help when the predicate can be turned into "seek to a point in sorted order, then scan contiguously". Anything that breaks sorted-order seekability breaks the index.

1. Leading-wildcard LIKE

-- CAN use a B-tree (prefix = a contiguous sorted range):
SELECT * FROM users WHERE email LIKE 'ana%';

-- CANNOT (matching suffixes are scattered all over sorted order):
SELECT * FROM users WHERE email LIKE '%@gmail.com';
EXPLAIN ANALYZE SELECT * FROM users WHERE email LIKE '%@gmail.com';

 Seq Scan on users  (cost=0.00..24346.00 rows=312400 width=72)
                    (actual time=0.014..214.882 rows=309911 loops=1)
   Filter: (email ~~ '%@gmail.com'::text)
   Rows Removed by Filter: 690089
 Planning Time: 0.101 ms
 Execution Time: 231.507 ms

Two footnotes worth knowing: (a) for 'ana%' to use the index in a non-C locale you need an operator class like text_pattern_ops, or a C-collation column; (b) genuine substring search is a job for a trigram GIN index (pg_trgm), which can serve LIKE '%foo%'. Saying "leading wildcard → seq scan, unless we add a trigram GIN index" is a strong answer.

2. Functions on the indexed column

-- Index on users(email). This does NOT use it:
SELECT * FROM users WHERE lower(email) = 'ana@example.com';
-- The index stores 'Ana@Example.com'; lower(email) values aren't in the tree.

-- Fix: an expression index — the tree stores the *result* of the expression:
CREATE INDEX idx_users_email_lower ON users (lower(email));
EXPLAIN ANALYZE SELECT * FROM users WHERE lower(email) = 'ana@example.com';

 Index Scan using idx_users_email_lower on users
     (cost=0.42..8.44 rows=1 width=72)
     (actual time=0.043..0.045 rows=1 loops=1)
   Index Cond: (lower(email) = 'ana@example.com'::text)
 Planning Time: 0.144 ms
 Execution Time: 0.061 ms

The same trap wears many costumes: date(created_at) = '2026-07-01' (rewrite as a half-open range: created_at >= '2026-07-01' AND created_at < '2026-07-02'), total + tax > 100, substring(sku, 1, 3) = 'ABC'. Either rewrite the predicate to leave the column "bare", or index the expression.

3. Type mismatches and implicit casts

-- column: user_id bigint, index on user_id.
-- A JDBC/ORM layer binds the parameter as text:
SELECT * FROM events WHERE user_id = '12345'::text;   -- forces user_id::text = ...
-- Postgres casts the COLUMN to match, wrapping it in a function -> no index.

In Postgres, comparing bigint to an integer-family literal is fine (cross-type operators exist), but comparisons that force a cast of the column — text vs numeric, varchar id column compared to an integer parameter, timestamp vs timestamptz in some setups — degrade to a seq scan silently. This is a classic ORM bug: the query looks identical in the log; only the bound parameter type differs. When a query is fast in psql but slow from the app, suspect parameter types.

4. Negations and inequality (a selectivity problem, not a structural one)

SELECT * FROM orders WHERE status <> 'completed';
SELECT * FROM orders WHERE status NOT IN ('completed', 'cancelled');

Structurally a B-tree could serve <> as two ranges, but the planner usually estimates it matches most of the table, so a seq scan wins. The senior move: if you only care about a small set of "live" statuses, flip the predicate to a positive IN ('pending','processing'), or use a partial index (WHERE status NOT IN (...)) that materializes exactly the interesting rows.

5. OR conditions — and when bitmap OR saves you

-- OR across two different columns can't be one B-tree traversal.
SELECT * FROM users WHERE email = $1 OR phone = $2;

-- If BOTH columns are individually indexed, the planner can do a BitmapOr:
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'a@x.com' OR phone = '555-0142';

 Bitmap Heap Scan on users  (cost=8.87..16.55 rows=2 width=72)
                            (actual time=0.052..0.056 rows=2 loops=1)
   Recheck Cond: ((email = 'a@x.com'::text) OR (phone = '555-0142'::text))
   Heap Blocks: exact=2
   ->  BitmapOr  (cost=8.87..8.87 rows=2 width=0) (actual time=0.044..0.045 rows=0 loops=1)
         ->  Bitmap Index Scan on idx_users_email  (cost=0.00..4.43 rows=1 width=0)
               Index Cond: (email = 'a@x.com'::text)
         ->  Bitmap Index Scan on idx_users_phone  (cost=0.00..4.43 rows=1 width=0)
               Index Cond: (phone = '555-0142'::text)

If either side of the OR has no index, the whole thing degrades to a seq scan. A common rewrite when bitmap OR isn't available is UNION ALL of two indexed queries (dedupe if needed).

6. The leading-column rule for composite indexes

CREATE INDEX idx_orders_cust_status ON orders (customer_id, status);

-- USES the index (leading column constrained):
WHERE customer_id = 42
WHERE customer_id = 42 AND status = 'pending'

-- CANNOT SEEK (leading column unconstrained):
WHERE status = 'pending'

The index is sorted by customer_id first, then status within each customer. Rows with status = 'pending' are sprinkled throughout — there is no single contiguous region to seek to. Think of a phone book sorted by (last_name, first_name): finding everyone named "Maria" requires reading the whole book.

Nuance for bonus points: Postgres can still choose the composite index for a status-only query as a full index scan with a filter (reading every leaf page) — occasionally cheaper than a seq scan because the index is narrower than the table — and it can consider a "skip scan"-style plan in very recent versions when the leading column has few distinct values. But it cannot seek. Say "can't seek on a non-leading column" rather than "can't use the index at all".

Composite index design: column order, ORDER BY, covering indexes T1

Column-order strategy

The working rule, in priority order:

  1. Equality columns first, then the range/sort column. Once you hit a range condition in the key, everything after it can only be filtered, not sought. WHERE customer_id = $1 AND created_at > $2 wants (customer_id, created_at) — all of customer 42's rows are contiguous, and within them, sorted by time. The reverse order (created_at, customer_id) would scan every customer's recent rows and filter.
  2. Among equality columns, lead with the one that's most useful standing alone — because the leading column doubles as a single-column index (leftmost-prefix rule). If some queries filter only on customer_id and none filter only on status, lead with customer_id.
  3. Selectivity is a tiebreaker, not the headline. "Most selective first" is the folklore answer; the sharper answer is that for pure multi-equality lookups the order barely changes the traversal cost — order matters mostly for (a) which prefixes are useful to other queries and (b) keeping range columns last. Saying that nuance out loud is a senior signal.
-- The canonical shape:
CREATE INDEX idx_orders_cust_created
    ON orders (customer_id, created_at);          -- equality, then range/sort

-- Serves all of these with a single seek + contiguous walk:
WHERE customer_id = $1
WHERE customer_id = $1 AND created_at >= $2
WHERE customer_id = $1 ORDER BY created_at DESC LIMIT 20

Covering the ORDER BY: killing the Sort node

Because leaf pages are sorted and linked, an index whose order matches the ORDER BY lets Postgres stream rows already sorted — no Sort node, and crucially, LIMIT can stop early. This turns "sort 50,000 rows to return 20" into "read 20 index entries." 

-- WITHOUT a matching index: fetch all, sort, take 20
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 4242
ORDER BY created_at DESC LIMIT 20;

        (actual time=18.343..18.348 rows=20 loops=1)
   ->  Sort  (cost=9483.11..9495.53 rows=4967 width=41)
             (actual time=18.341..18.344 rows=20 loops=1)
         Sort Key: created_at DESC
         Sort Method: top-N heapsort  Memory: 27kB
         ->  Bitmap Heap Scan on orders  (cost=58.92..9350.94 rows=4967 width=41)
                                          (actual time=1.204..16.902 rows=5121 loops=1)
               Recheck Cond: (customer_id = 4242)
               Heap Blocks: exact=4893
               ->  Bitmap Index Scan on idx_orders_customer  (cost=0.00..57.68 rows=4967 width=0)
                     Index Cond: (customer_id = 4242)

-- WITH (customer_id, created_at): stream in order, stop after 20
 Limit  (cost=0.43..24.15 rows=20 width=41)
        (actual time=0.038..0.164 rows=20 loops=1)
   ->  Index Scan Backward using idx_orders_cust_created on orders
           (cost=0.43..5891.40 rows=4967 width=41)
           (actual time=0.036..0.158 rows=20 loops=1)
         Index Cond: (customer_id = 4242)
 Execution Time: 0.189 ms

Index Scan Backward = walking the leaf chain right-to-left to serve DESC; a plain ascending index handles it. You only need (customer_id, created_at DESC) explicitly when mixing directions, e.g. ORDER BY a ASC, b DESC.

Covering indexes and INCLUDE

-- Query: SELECT status, total FROM orders WHERE customer_id = $1;
CREATE INDEX idx_orders_cust_covering
    ON orders (customer_id) INCLUDE (status, total);

INCLUDE columns are stored as payload in leaf pages only — they're not part of the sort key, don't bloat internal pages, and can be types with no B-tree ordering. If the index carries every column the query needs, Postgres can answer from the index alone: an Index Only Scan, skipping the heap hop entirely. You can also get a covering effect by simply putting the column in the key ((customer_id, status, total)); INCLUDE is the cleaner statement of intent and works with unique indexes (UNIQUE (email) INCLUDE (name) stays unique on email alone).

Index-only scans and the visibility map catch

 Index Only Scan using idx_orders_cust_covering on orders
     (cost=0.42..134.81 rows=4967 width=14)
     (actual time=0.029..1.611 rows=5121 loops=1)
   Index Cond: (customer_id = 4242)
   Heap Fetches: 0                      <-- the number that matters
 Execution Time: 2.011 ms

Here's the MVCC catch: index entries carry no visibility information, so an "index only" scan must still verify each tuple is visible. Postgres keeps a visibility map — one bit per heap page saying "every tuple on this page is visible to all transactions." If the bit is set, no heap visit needed. The bit is set by VACUUM and cleared by any write to the page. Consequence: on a heavily updated table, Heap Fetches climbs and your "index only" scan quietly degrades into a regular index scan until vacuum catches up. If you see Heap Fetches: 48,213, the fix is often VACUUM (or more aggressive autovacuum), not a new index.

Interview-ready summary: "Index-only scans are a contract between the index (has the columns) and the visibility map (proves the pages are all-visible). Both must hold. I check Heap Fetches in the plan; a big number means vacuum isn't keeping up, and that's a vacuum-tuning problem wearing an indexing costume."

Don't create one mega-index per query. Indexes on (a), (a,b), and (a,b,c) together are almost always waste — (a,b,c) serves the other two's seeks via the leftmost-prefix rule. Audit for redundant prefixes before adding; drop the narrow ones.

NULLs and indexes — three facts interviewers check

  • B-trees DO index NULLs (a common myth says otherwise — that's Oracle's single-column behavior, not Postgres). WHERE col IS NULL can use an index; NULLs sort last by default (NULLS LAST for ASC).
  • Unique indexes allow multiple NULLs by default, because NULL = NULL is unknown, not true. Postgres 15+ gives you UNIQUE NULLS NOT DISTINCT to change that; before 15, use a partial unique index or a coalescing expression index.
  • If a nullable column is mostly NULL and you only query the non-NULL rows, a partial index WHERE col IS NOT NULL can be a fraction of the size.
-- "at most one default address per user", where is_default is nullable:
CREATE UNIQUE INDEX one_default_addr
    ON addresses (user_id)
    WHERE is_default;        -- boolean partial: NULL and false rows excluded

EXPLAIN / EXPLAIN ANALYZE deep-dive T1

EXPLAIN shows the plan the optimizer chose with cost estimates. EXPLAIN ANALYZE actually runs the query and adds real timings and row counts. Always add BUFFERS: 

EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
-- Careful: ANALYZE really executes. Wrap DML in a transaction:
BEGIN; EXPLAIN ANALYZE UPDATE ...; ROLLBACK;

Read the tree bottom-up, inside-out

Indentation = child nodes. Leaves (scans) execute first and feed rows upward into joins, sorts, and aggregates. Your eyes should go: deepest node first, then follow rows up. When hunting a problem, find the node where actual time explodes or where estimated rows and actual rows diverge wildly.

Cost numbers decoded

Seq Scan on orders  (cost=0.00..20834.00 rows=4967 width=41)
                     ^^^^^^^^^^^^^^^^^^  ^^^^^^^^^  ^^^^^^^^
                     startup..total cost  est. rows  avg row bytes
  • Cost units are abstract, anchored to seq_page_cost = 1.0 ("cost of reading one page sequentially"). random_page_cost defaults to 4.0; cpu_tuple_cost 0.01. Costs are for comparing plans, not predicting milliseconds.
  • Startup cost = cost before the first row can be emitted. A Sort has high startup (must consume all input first); an Index Scan has near-zero. This is why LIMIT favors low-startup plans.
  • rows is the planner's estimate from statistics. Comparing it against ANALYZE's actual rows is the single highest-value diagnostic act.

Rows estimate vs actual — the root of most bad plans

->  Index Scan using idx_ev_type on events
        (cost=0.43..8.45 rows=1 width=64)
        (actual time=0.055..912.331 rows=184220 loops=1)
                                    ^^^^^^^^^^^
    Planner expected 1 row; got 184,220.

A 5-orders-of-magnitude misestimate like this makes the planner pick strategies that only work for tiny inputs (typically a Nested Loop), which then run 184,220 times. Causes: stale stats (ANALYZE the table), correlated columns (planner multiplies independent selectivities — fix with CREATE STATISTICS ... (dependencies)), skewed data beyond the MCV list (ALTER TABLE ... ALTER COLUMN ... SET STATISTICS 1000), or non-immutable expressions the planner can't see through.

loops — per-loop actuals multiply!

->  Index Scan using orders_pkey on orders
        (cost=0.43..8.45 rows=1 width=41)
        (actual time=0.021..0.022 rows=1 loops=48213)

actual time and rows are per loop. Total row count = 1 × 48,213; total time ≈ 0.022 ms × 48,213 ≈ 1.06 s. A node that looks innocent ("0.022 ms!") can be your entire runtime. Always multiply by loops. This is also what an app-level N+1 looks like when it happens inside one query via a nested loop join.

BUFFERS — cache visibility

   Buffers: shared hit=2112 read=85342

hit = pages found in shared_buffers (RAM); read = pages fetched from the OS/disk. 85k reads × 8 KB ≈ 667 MB of I/O for one query — that's your smoking gun for "why is it slow only sometimes" (cold cache) and for spotting queries that churn the cache for everyone else. A query can be "fast in staging, slow in prod" purely because of hit ratios.

Hash Join: equality joins over big row sets. The build side must fit in work_mem or it spills (Batches: 8 = 8 passes over temp files). The planner builds on the smaller side. 

-- MERGE JOIN: both inputs sorted on the join key; zipper them together.
 Merge Join  (cost=0.85..61240.90 rows=1000000 width=105)
   Merge Cond: (o.customer_id = c.id)
   ->  Index Scan using idx_orders_customer on orders o   (pre-sorted, free)
   ->  Index Scan using customers_pkey on customers c

Merge Join shines when both sides arrive pre-sorted (via indexes) — huge joins with no memory pressure. If a side needs an explicit Sort first, the sort cost usually hands the win back to Hash Join.

Sort, incremental sort, and work_mem spills

 Sort  (cost=131824.29..134324.29 rows=1000000 width=41)
       (actual time=812.406..1094.221 rows=1000000 loops=1)
   Sort Key: created_at DESC
   Sort Method: external merge  Disk: 48720kB     <-- spilled to temp files
   Buffers: shared hit=12448, temp read=6090 written=6103

quicksort Memory: ...kB = fit in work_mem, fine. external merge Disk: = didn't fit; Postgres wrote sorted runs to disk and merged them. Fixes, in order of preference: make an index serve the ordering (best — no sort at all), sort fewer rows/columns, or raise work_mem (session-level for the reporting query, not globally — it's per sort node per connection, so a global bump can blow out memory under load). Incremental Sort appears when input is already sorted by a prefix of the sort key (index on (a), query orders by a, b): it sorts small per-group batches — low memory, low startup cost.

Why the planner ignores your index T1

"I added the index and it still does a seq scan" — the checklist, roughly in order of likelihood:

  1. Stale statistics. The planner prices plans using stats from the last ANALYZE. After bulk loads or big deletes, estimates can be fiction. First move, always: ANALYZE tablename; and re-explain. (Autovacuum usually handles this, but it lags after mass changes.)
  2. Low selectivity — the seq scan is CORRECT. If the predicate matches ~30–50%+ of rows, per-row index random I/O costs more than one sequential pass. The planner is doing its job. Rough mental model: index scans win below a few percent of the table; bitmap scans hold the middle; seq scans win the rest. Don't fight it; fix the query shape (or accept it).
  3. Small table. A 50-page table is cheaper to read whole than to traverse a tree for. Indexes on tiny lookup tables exist for constraints, not speed.
  4. Cost settings lying about your hardware. Default random_page_cost = 4.0 models spinning disks. On SSD/NVMe (and cloud storage), random reads are nearly as cheap as sequential; common practice is 1.1. Left at 4.0, the planner over-penalizes index scans and clings to seq scans and hash joins. Also check effective_cache_size (should reflect total RAM available for caching — it makes index scans look cheaper by modeling cache hits).
  5. Wrapped column / expression mismatch. Anything from the trap list: function on the column, implicit cast, collation mismatch.
  6. Wrong column order in a composite index — the leading-column rule again.
  7. The index can't produce what the query needs — e.g., ordering direction mismatch with mixed ASC/DESC, or a partial index whose WHERE clause doesn't provably cover the query's predicate.

Verifying: the enable_seqscan trick (diagnosis ONLY)

SET enable_seqscan = off;   -- session-local; makes seq scans astronomically expensive
EXPLAIN ANALYZE SELECT ... ;
RESET enable_seqscan;

Two outcomes, both informative:

  • Planner still seq-scans → the index is unusable for this query (wrapped column, wrong order, cast). Structural problem.
  • Planner uses the index and it's slower (or barely faster) → the cost model was right; selectivity is the issue.
  • Planner uses the index and it's much faster → costing problem: stale stats, random_page_cost, correlation. Fix the inputs to the cost model.

Never ship enable_seqscan = off (or friends) to production, and say so explicitly in the interview: "these are diagnostic scalpels, not tuning knobs." Interviewers actively probe whether you know these settings are global sledgehammers.

When indexes HURT: write amplification, HOT updates, bloat T1

Write amplification

Every INSERT writes the heap tuple plus one entry per index — each a tree descent, a leaf insert, possible page splits, and WAL for all of it. A table with 9 indexes does ~10x the write work of an unindexed one. On an ingest-heavy table, that's the difference between keeping up and falling behind. Bulk-load pattern: drop (or create after) indexes, load, then CREATE INDEX — building an index in one pass is far cheaper than a million incremental inserts.

HOT updates — and how one index defeats them

An UPDATE in Postgres writes a whole new row version. Normally every index needs a new entry pointing at it. The HOT (Heap-Only Tuple) optimization skips all index writes when two conditions hold: (1) no indexed column changed, and (2) the new version fits on the same heap page (helped by a fillfactor below 100 leaving room). The old version just points to the new one within the page; indexes keep pointing at the chain head.

The trap: someone adds an index on last_seen_at or updated_at — a column touched by nearly every UPDATE. Condition (1) now fails for every update, so every index on the table gets written on every update, not just the new one. One "harmless" index can multiply the write load of the whole table and inflate bloat everywhere. Check impact with pg_stat_user_tables.n_tup_hot_upd vs n_tup_upd before and after — a collapsing HOT ratio is the tell.

Bloat

Dead tuples (from UPDATE/DELETE) leave dead index entries too. Vacuum reclaims space within pages, but B-tree pages that became sparse don't merge back gracefully — indexes ratchet up in size under churny workloads. Symptoms: index physically larger than expected, degraded cache hit ratio, slower scans. Fix: REINDEX INDEX CONCURRENTLY idx_name; (production-safe rebuild, Postgres 12+). Monitor with pgstattuple/pgstatindex (avg_leaf_density well under ~70% on a stable-size index suggests a rebuild).

Production index operations: CONCURRENTLY or downtime

A plain CREATE INDEX takes a lock that blocks all writes to the table for the entire build — minutes to hours on a big table. In production, always: 

CREATE INDEX CONCURRENTLY idx_orders_cust_created
    ON orders (customer_id, created_at);

DROP INDEX CONCURRENTLY idx_orders_old;
REINDEX INDEX CONCURRENTLY idx_orders_bloated;   -- PG 12+

Know the fine print — this is exactly where interviewers poke:

  • CONCURRENTLY can't run inside a transaction block — it manages its own (so it doesn't mix with transactional migration tools; Flyway needs the migration marked non-transactional).
  • It does two table scans plus waits for concurrent transactions, so it's slower and needs care on very busy tables (long-running transactions stall it).
  • If it fails or is cancelled midway, it leaves an INVALID index behind — still maintained on writes (pure cost, never used for reads). Find and drop them: SELECT indexrelid::regclass FROM pg_index WHERE NOT indisvalid;
  • Pair migrations with lock_timeout (e.g. SET lock_timeout = '3s') so any DDL that does need a lock fails fast instead of queueing behind a long query while blocking everyone behind it.

Caveats before you drop: stats reset on pg_stat_reset() and (pre-PG16) on failover to a replica — make sure the observation window is long enough to include monthly/quarterly jobs; check replicas too (their index usage is tracked separately). Safe path: note the definition, DROP INDEX CONCURRENTLY, and be ready to recreate. The symptom of over-indexing at the system level: write latency creeping up, autovacuum permanently busy, and indexes collectively larger than the table.

Specialized indexes: partial, expression, GIN, GiST, BRIN, hash T2

Partial indexes — index only the rows you query

-- 98% of orders are 'completed'; the app only ever queries live ones.
CREATE INDEX idx_orders_live
    ON orders (customer_id, created_at)
    WHERE status IN ('pending', 'processing');

The index contains only matching rows: ~2% of the size, hotter in cache, cheaper to maintain (writes to completed orders never touch it). The planner uses it only when the query's WHERE clause provably implies the index's predicate — write the same condition (or a stricter one) in the query. Killer bonus use: conditional uniqueness, which a plain unique constraint can't express: 

-- "one active subscription per user" — cancelled ones don't count
CREATE UNIQUE INDEX one_active_sub_per_user
    ON subscriptions (user_id)
    WHERE status = 'active';

-- soft-delete-safe unique emails
CREATE UNIQUE INDEX users_email_unique
    ON users (lower(email))
    WHERE deleted_at IS NULL;

Expression indexes

CREATE INDEX idx_users_email_lower ON users (lower(email));
-- Query must use the SAME expression to match:
SELECT * FROM users WHERE lower(email) = lower($1);

The tree stores computed results. The expression must be immutable. Combines with partial (see above). Run ANALYZE after creating one — expression indexes get their own statistics.

GIN — inverted index for "contains" questions

GIN maps each element/key/lexeme → list of rows containing it — perfect for JSONB containment (@>), array overlap (&&), full-text search (tsvector @@ tsquery), and (with pg_trgm) LIKE '%foo%'. Costs: notably slower writes than B-tree (many entries per row; mitigated by fastupdate pending lists, which trade read latency spikes), large size, and no ordering — GIN can't serve ORDER BY. jsonb_path_ops is smaller/faster but only supports @>.

A B-tree on a jsonb column is nearly useless (it indexes whole-document comparison). Either GIN the column, or B-tree an extracted expression: CREATE INDEX ON events ((payload->>'type')); — the right choice when you always query that one key with equality and want ordering/statistics.

GiST — overlaps, nearest-neighbor, exclusion constraints

-- Prevent double-booking a room: no two reservations may overlap.
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE reservations ADD CONSTRAINT no_overlap
    EXCLUDE USING gist (room_id WITH =, during WITH &&);

GiST is a generalized tree for data with "containment/overlap" rather than total order: ranges, geometry (PostGIS), and k-nearest-neighbor (ORDER BY location <-> point). The exclusion-constraint example is a spectacular interview answer for booking systems — it's a uniqueness constraint over overlap, enforced at the database level, race-free.

BRIN — tiny index for huge, naturally-ordered tables

CREATE INDEX idx_metrics_time_brin ON metrics USING brin (recorded_at);

-- 2 TB append-only time-series table:
--   B-tree on recorded_at:  ~45 GB
--   BRIN on recorded_at:    ~2 MB

BRIN stores only the min/max of the column per block range (default 128 pages). A time filter skips every block range whose min/max can't match. It only works when physical order correlates with the column — append-only time-series is the canonical case. Wrong tool for scattered data (correlation ≈ 0 → it skips nothing). Trade: much coarser than B-tree (always a lossy bitmap scan with rechecks), but the maintenance and memory cost is near zero.

Hash indexes

Equality only — no ranges, no ordering, no multi-column. WAL-logged and crash-safe since Postgres 10, so they're legitimate now, and can beat B-trees slightly for equality on very long keys (URLs). But a B-tree handles equality fine and does everything else too, so the honest interview answer is: "rarely; B-tree is the default for a reason. For long text keys I'd sooner B-tree a hash expression like (digest(url,'sha256')) and get uniqueness support."

When the interviewer says "production got slow," do NOT jump to "add an index." Walk the runbook. The structure itself — symptom → locate → measure → hypothesize → fix → verify — is what they're grading.

App-level interplay: N+1, SELECT *, pagination, COUNT(*), pools T2

N+1 — the database is innocent

-- What pg_stat_statements shows:
--   calls=991042  mean=2.3ms   SELECT * FROM order_items WHERE order_id = $1
-- Each call is FAST. The workload is the problem: 1 query for orders,
-- then 1 per order for items. Latency = N × (round trip + 2.3ms).

Indexing can't fix N+1 — every individual query already uses an index. Fix at the app layer: a JOIN, a batched WHERE order_id = ANY($1), or in JPA a JOIN FETCH / @EntityGraph (see tutorial 07). The interview tell that it's N+1 and not a slow query: huge calls, tiny mean_exec_time, page latency scaling with result-set size. Your frontend analogy lands well here: it's a waterfall of sequential fetches instead of one batched request.

SELECT * — three separate costs

  • Defeats index-only scans: the moment you ask for a column not in the index, every row needs a heap fetch.
  • Network and serialization: dragging a 2 KB jsonb or text blob across the wire for a list view that shows three fields — multiplied by rows × requests. (Wide values live in TOAST storage, fetched and decompressed only when actually selected — SELECT * forces that work too.)
  • Fragility: plans and ORM mappings silently change when columns are added.

Deep pagination: LIMIT/OFFSET is O(offset)

EXPLAIN ANALYZE SELECT id, title, created_at FROM posts
ORDER BY created_at DESC LIMIT 20 OFFSET 200000;

 Limit  (cost=6841.32..6841.99 rows=20 width=52)
        (actual time=312.441..312.688 rows=20 loops=1)
   ->  Index Scan Backward using idx_posts_created on posts
           (cost=0.43..68413.20 rows=2000000 width=52)
           (actual time=0.038..305.912 rows=200020 loops=1)
 Execution Time: 312.821 ms

Postgres must produce and discard 200,000 rows to return 20 — rows=200020 on the scan says it plainly. Page 10,000 is 10,000x the work of page 1, and rows shift between pages under concurrent writes. The fix is keyset (cursor) pagination: remember where the last page ended and seek there: 

-- Cursor = (created_at, id) of the last row of the previous page.
-- id breaks ties so pagination is total-ordered and stable.
SELECT id, title, created_at FROM posts
WHERE (created_at, id) < ($1, $2)          -- row-value comparison: one index seek
ORDER BY created_at DESC, id DESC
LIMIT 20;

-- Needs: CREATE INDEX ON posts (created_at DESC, id DESC);

 Limit  (cost=0.43..7.28 rows=20 width=52)
        (actual time=0.041..0.152 rows=20 loops=1)
   ->  Index Scan using idx_posts_created_id on posts
           (cost=0.43..684530.11 rows=1998112 width=52)
           (actual time=0.039..0.146 rows=20 loops=1)
         Index Cond: (ROW(created_at, id) < ROW('2026-06-30 08:14:02+00', 1837421))
 Execution Time: 0.178 ms

Every page costs the same as page 1. The API returns an opaque cursor instead of a page number — the trade-off is no "jump to page 47" (usually fine: infinite scroll, feeds, exports). This exact rewrite — with the composite row-value comparison — is a fantastic whiteboard answer.

COUNT(*) on big tables

COUNT(*) must visit rows to check MVCC visibility — no shortcut exists; on a 100M-row table it's a multi-second parallel scan (an index-only scan helps but still touches every entry). Alternatives, by use case: 

-- UI badge / dashboard: planner's estimate is instant and usually plenty
SELECT reltuples::bigint AS approx_rows
FROM pg_class WHERE relname = 'events';       -- accurate to last ANALYZE

-- "10,000+ results" UX: count with a cap
SELECT count(*) FROM (
  SELECT 1 FROM events WHERE tenant_id = $1 LIMIT 10001
) t;

-- Exact + fast + frequent: maintain a counter (trigger or app-side, accept the write cost)

"The DB is slow" that isn't: connection pool exhaustion

Symptom: p99 explodes, dashboards blame the database — but pg_stat_statements shows normal mean_exec_time and the DB is idle. The time is spent in the app, queued waiting for a connection from HikariCP (default pool: 10). Common causes: a pool sized too small for a traffic spike, one slow endpoint (or that N+1) holding connections 10x longer than normal, or transactions held open across HTTP calls. Diagnose with pool metrics (Hikari's pending/awaiting gauges, connection acquisition time) and pg_stat_activity (lots of idle in transaction = app holding transactions open). Fix the hog first; more connections is often the wrong fix — Postgres degrades past a few hundred backends, which is why PgBouncer exists.

Guardrails: statement timeout

-- Per role/app: fail fast instead of letting a runaway query pile up
ALTER ROLE app_user SET statement_timeout = '5s';
ALTER ROLE app_user SET idle_in_transaction_session_timeout = '30s';
-- Reporting jobs get their own role with a bigger budget.
ALTER ROLE reporting SET statement_timeout = '5min';

A missing index plus no timeout = one bad deploy saturates I/O, every other query slows, pools exhaust, the whole site is down. With a 5 s timeout the blast radius is one failing endpoint and a clean error in the logs. Timeouts convert "mystery outage" into "specific query to fix" — mention them as the systemic guardrail at the end of your slow-query story.

Interview Q&A

Walk me through what happens inside Postgres when you run WHERE email = 'x' with a B-tree index on email.

The planner first decides an index scan is worth it based on statistics. Execution: binary-search the root page for the key, descend through internal pages to the leaf (2–4 page reads even for huge tables since fan-out is in the hundreds), binary-search within the leaf for matching (key, ctid) entries. Each ctid is a physical heap address — fetch that heap page, read the tuple, and check MVCC visibility, since the index doesn't know if the row version is visible to my snapshot. So the real cost is tree descent plus one heap hop per matching row — which is why index scans win at high selectivity and lose when they'd trigger millions of random heap reads.

Why is a B-tree lookup O(log n), and why is that even faster than it sounds?

Each level of the tree partitions keys by sorted separator values, so a descent eliminates all but one child per step — that's the log. But because Postgres pages are 8 KB and hold hundreds of keys, the branching factor is ~several hundred, not 2: a million rows is a 3-level tree, a billion is 4–5. In practice "O(log n)" means "a handful of page reads regardless of table size," and the upper levels are almost always cached.

You have an index on (customer_id, status). Which queries can use it to seek?

WHERE customer_id = $1 and WHERE customer_id = $1 AND status = $2 can seek — the index is sorted by customer_id first, so both define a contiguous leaf range. WHERE status = $1 alone cannot seek: matching entries are scattered across every customer's section. The planner might still walk the whole index with a filter if that beats a heap seq scan, but there's no seek. If status-only queries matter, I'd add a separate index on status — probably partial, since status columns are usually low-cardinality with a hot subset.

How do you order columns in a composite index?

Equality-filtered columns first, then the range or sort column last — once the key hits a range condition, later columns can't be sought. Among the equality columns, I lead with whichever is useful standing alone, since the leading column doubles as a single-column index. Selectivity ordering is a tiebreaker, not the main rule — for pure multi-equality lookups, order barely changes traversal cost; it mostly determines which other queries can reuse prefixes. So for WHERE customer_id = ? AND status = ? ORDER BY created_at DESC: (customer_id, status, created_at), or make status a partial-index predicate instead of a key column.

What's an index-only scan and when does it unexpectedly hit the heap anyway?

If the index contains every column the query reads (key columns or INCLUDE payload), Postgres can answer from the index without heap fetches. But index entries carry no MVCC visibility info, so it consults the visibility map — a bitmap of "all tuples on this heap page are visible to everyone." Pages recently written have that bit cleared, forcing a heap fetch to check visibility. After heavy writes, Heap Fetches in the plan climbs and the scan silently degrades to a normal index scan until VACUUM re-marks pages all-visible. So a slow "index only" scan is often a vacuum problem, not an index problem.

Why doesn't LIKE '%gmail.com' use a B-tree index, while LIKE 'ana%' can?

A B-tree serves predicates that map to a contiguous range in sorted order. A prefix defines such a range — everything from 'ana' up to 'anb' is adjacent in the tree (given a suitable collation or text_pattern_ops opclass). A suffix match has no relationship to sort order; matching values are scattered everywhere, so the only B-tree option is scanning everything. For genuine substring search I'd use a trigram GIN index from pg_trgm, which decomposes strings into 3-grams and can serve %foo% patterns.

Your query is fast in psql but slow from the Java app. Same SQL. What do you check?

First hypothesis: parameter type mismatch — the JDBC driver binding a String against a numeric/uuid column (or vice versa) forces a cast on the column and kills the index; psql with a literal doesn't. I'd capture the actual production plan with auto_explain to compare. Other candidates: the app fetching far more rows (missing LIMIT), a generic prepared-statement plan gone bad after repeated executions (custom vs generic plans), N+1 so no single query is slow but the workload is, connection pool wait time being blamed on the query, or different session settings (work_mem, search_path). The unifying move: measure the app's execution specifically, don't assume psql reproduces it.

Explain a Bitmap Heap Scan and why the planner picks it over a plain Index Scan.

It's the middle ground for medium selectivity. A plain index scan does one random heap hop per row in index order — great for a few rows, terrible for 100k. A bitmap scan first collects all matching ctids from the index into an in-memory bitmap, sorts by physical page, then visits each heap page exactly once, sequentially. It trades index ordering (so ORDER BY needs a Sort afterward) for vastly better I/O patterns, and enables BitmapAnd/BitmapOr to combine multiple indexes. The Recheck Cond exists because if the bitmap exceeds work_mem it degrades to lossy per-page bits, and every tuple on those pages must be rechecked against the predicate — lots of lossy blocks in the plan is a hint to raise work_mem.

When is a Nested Loop join the best plan, and when is it a disaster?

Best when the outer side is small and the inner side has an index on the join key: k cheap index probes, near-zero startup cost — ideal for "this customer's 12 orders and their items," and the only join that can exploit LIMIT well. Disaster when the outer side is much larger than estimated (planner thought 5 rows, actual 200k) or the inner side has no index, turning it into 200k inner scans. In plans I check loops= on the inner node and multiply per-loop time by loops — that product is often the whole runtime. Root cause is usually a row misestimate, so the fix is stats/extended statistics before anything else.

What do the two cost numbers in EXPLAIN mean, and what unit are they in?

cost=startup..total: startup is the estimated cost before emitting the first row (a Sort must consume all input first, so high startup; an index scan is near zero), total is for producing all rows. Units are abstract, anchored to seq_page_cost = 1.0 — one sequential page read — with random_page_cost, cpu_tuple_cost etc. as multiples. They're for comparing candidate plans, not predicting milliseconds. The startup/total split matters for LIMIT: it charges roughly startup plus a prorated fraction of total, which is why LIMIT favors streaming plans over sort-everything plans.

EXPLAIN ANALYZE shows rows=1 estimated but rows=180000 actual. What happens next?

That misestimate is the root cause of almost any bad plan built on top of it — the planner will have chosen strategies that only work for one row, like a nested loop that now executes 180k inner probes. Diagnosis order: run ANALYZE on the table (stale stats after a bulk load is the most common cause); if it persists, look for correlated predicates — the planner multiplies column selectivities as if independent, so city = 'SF' AND state = 'CA' is underestimated; fix with CREATE STATISTICS (dependencies) ON city, state FROM t. For skewed columns, raise the statistics target so the MCV list captures the hot values. If it's an expression, add an expression index or extended statistics so it gets stats at all.

What does Buffers: shared hit=200 read=90000 tell you?

The query touched ~90,200 8 KB pages and only 200 came from shared_buffers — the rest were read from the OS or disk: ~700 MB of I/O for one execution. That explains "slow sometimes": with a warm cache it's fast, cold it's awful, and it also evicts everyone else's pages, so this one query degrades unrelated queries. The fix is reducing pages touched — better index, tighter predicate, fewer columns — not just more RAM. I always run EXPLAIN with the BUFFERS option for exactly this; timings tell you it's slow, buffers tell you why.

What does "Sort Method: external merge Disk: 48720kB" mean and how do you fix it?

The sort didn't fit in work_mem, so Postgres wrote sorted runs to temp files on disk and merge them — the slowest sort path. Fixes in order of preference: eliminate the sort entirely with an index matching the ORDER BY (best — also lets LIMIT stop early); sort less data (project fewer columns, filter earlier); or raise work_mem for that session or role — not globally, because work_mem is per sort/hash node per query, so a global increase can multiply into memory exhaustion under concurrency. I'd also check whether an Incremental Sort is available if an index already provides a prefix of the sort key.

You added an index but EXPLAIN still shows a Seq Scan. Give your checklist.

(1) ANALYZE the table — stale stats after bulk changes is the #1 cause. (2) Check selectivity: if the predicate matches 30%+ of rows, the seq scan is correct and I should say so, not fight it. (3) Small table — seq scan is optimal. (4) Is the column wrapped — function, cast, collation mismatch — making the index structurally unusable? (5) Composite index with the wrong leading column. (6) Cost settings: random_page_cost=4 on NVMe over-penalizes index scans; ~1.1 is realistic. To distinguish structural from costing issues, I'll SET enable_seqscan = off in a session as a diagnostic only: if it still seq-scans, the index is unusable for this query; if it uses the index and is much faster, the cost model inputs are wrong; if it uses it and it's slower, the planner was right all along.

Why might adding an index on updated_at slow down the whole table's writes disproportionately?

Because it defeats HOT updates. Normally an UPDATE that doesn't change any indexed column and fits on the same heap page skips all index maintenance — the new tuple version is reached via an in-page chain. updated_at changes on virtually every UPDATE, so with it indexed, condition one fails for every update: now every index on the table gets a new entry per update, not just this one. Write amplification across all indexes, more WAL, faster bloat. I'd check pg_stat_user_tables' n_tup_hot_upd/n_tup_upd ratio before and after — if the HOT ratio collapses, that index needs a very good justification, or the query it serves should be rethought (e.g., BRIN if it's for time-range maintenance jobs).

How do you find indexes you should drop, and what are the gotchas?

Query pg_stat_user_indexes for idx_scan = 0 (or near-zero relative to table writes), excluding unique/constraint-backing indexes which exist for correctness, and sort by size to prioritize. Gotchas: the stats window must be long enough to cover periodic jobs — a month-end report's index looks unused for 29 days; counters reset on pg_stat_reset and, before PG16, on replica promotion; and replicas track usage separately, so check them if reads are offloaded. Also look for redundant prefixes — (a) alongside (a,b) — where the wider index makes the narrow one droppable. Process: record the definition, DROP INDEX CONCURRENTLY, monitor, ready to recreate.

What's a partial index and what's the killer use case beyond size?

An index with a WHERE clause — it only contains rows matching the predicate. Benefits: a fraction of the size (hot subset stays cached), cheaper writes (rows outside the predicate never touch it), and the planner uses it when the query provably implies the predicate. The killer feature is conditional uniqueness, which a plain constraint can't express: CREATE UNIQUE INDEX ... ON subscriptions (user_id) WHERE status = 'active' enforces "one active subscription per user" at the database level, race-free — while allowing unlimited cancelled rows. Same trick makes unique emails coexist with soft deletes: unique on lower(email) WHERE deleted_at IS NULL.

When would you reach for GIN vs a B-tree expression index on a JSONB column?

Depends on query shape. If queries are arbitrary containment — "documents containing {plan: pro}" on varied keys — GIN (often jsonb_path_ops for @>-only workloads: smaller, faster) is built for it. If the app always queries one specific key with equality or ranges — payload->>'type' — a B-tree expression index on that extraction is smaller, cheaper to maintain, supports ordering, and gets real per-value statistics. GIN's costs: significantly slower writes (many index entries per row), large size, no ORDER BY support. And honestly, a key queried that way constantly is a candidate for promotion to a real column — schema design beats index heroics.

What is BRIN and when is it the right answer?

Block Range INdex: instead of indexing rows, it stores the min/max of a column per block range (default 128 pages, ~1 MB). A range predicate skips block ranges that can't match. On a 2 TB append-only time-series table, a BRIN on the timestamp can be a couple of MB where a B-tree would be tens of GB — trivial to cache and nearly free to maintain. The requirement: physical row order must correlate with the column, which append-only insertion gives naturally. On scattered data it prunes nothing. It's coarser than a B-tree — always a lossy bitmap scan with rechecks — so it suits "scan last week of metrics" analytics, not single-row point lookups.

Tell me about a slow query you fixed. (The guaranteed question — model story.)

Structure it symptom → locate → measure → fix → verify: "p99 latency alert on the orders page. pg_stat_statements, sorted by total_exec_time, pointed at one query: 2.5 s mean over 7k calls. EXPLAIN (ANALYZE, BUFFERS) with production parameters showed a seq scan over 10M rows with 'Rows Removed by Filter: 9.99M' and 85k buffer reads — 667 MB of I/O per execution. Row estimates were sane so it wasn't statistics; the table simply had no index matching the shape: customer_id equality + status equality + ORDER BY created_at DESC LIMIT 20. I created a composite partial index — (customer_id, created_at DESC) WHERE status = 'pending' — with CREATE INDEX CONCURRENTLY to avoid blocking writes. Result: 0.24 ms, 25 buffer hits, and the Sort node vanished because the index feeds LIMIT pre-sorted. Verified the HOT-update ratio and insert latency didn't regress, and added auto_explain + statement_timeout so the next incident comes with a plan attached and a bounded blast radius." Numbers + trade-off awareness + guardrails is what makes it senior.

Why is OFFSET 500000 slow, and what's the fix?

OFFSET doesn't skip — Postgres must generate and discard all 500k preceding rows in order before returning yours; the plan shows rows=500020 on the scan feeding a LIMIT of 20. Cost grows linearly with page depth, and concurrent inserts shift rows between pages. Fix: keyset pagination — return a cursor of the last row's sort key, e.g. (created_at, id) with id as tiebreaker, and fetch the next page with WHERE (created_at, id) < ($1, $2) ORDER BY created_at DESC, id DESC LIMIT 20 backed by an index on (created_at DESC, id DESC). Row-value comparison compiles to a single index seek, so every page costs the same as page one. Trade-off: no random page jumps — fine for feeds and infinite scroll, which is most real UX anyway.

Why is COUNT(*) slow on a 100M-row table and what do you do about it?

MVCC: row visibility is per-snapshot, so Postgres can't keep one true counter — COUNT(*) must visit tuples (or index entries plus visibility checks) to decide what your transaction can see. Parallel workers and index-only scans help but it's still O(rows). Alternatives by requirement: approximate is usually fine for UI — pg_class.reltuples is instant and accurate to the last ANALYZE; for search UX, cap it: count a subquery with LIMIT 10001 and display "10,000+"; for exact-and-frequent, maintain a counter table via trigger or app logic and accept the write cost and contention. The senior move is asking what the count is for before choosing.

The team says "the database is slow" but pg_stat_statements shows normal query times. What's going on?

Classic connection-pool exhaustion masquerading as DB slowness. Requests queue in the app waiting for a connection (HikariCP defaults to 10); the wall-clock time is spent waiting, not querying, so DB-side metrics look healthy. I'd check pool metrics — connection acquisition time, pending threads — and pg_stat_activity for symptoms like many 'idle in transaction' sessions (app holding transactions across external calls). Root causes: one slow endpoint or an N+1 hogging connections, transactions spanning HTTP calls, or genuine undersizing. Fix the hog first — just raising pool size often makes it worse, since Postgres itself degrades with too many backends; that's what PgBouncer is for. Also lock waits: fast plans + slow wall-clock inside the DB shows up as wait_event of type Lock, which is a different diagnosis branch.

What guardrails do you set so one bad query can't take down production?

Per-role statement_timeout (e.g. 5s for the web app role, minutes for reporting roles) so a runaway query fails fast instead of saturating I/O and cascading into pool exhaustion; idle_in_transaction_session_timeout so abandoned transactions can't hold locks and block vacuum; lock_timeout for migrations so an ALTER TABLE queues briefly instead of blocking all traffic behind its lock request; log_min_duration_statement + auto_explain (with ANALYZE and BUFFERS) so slow production executions arrive with plans attached; and pg_stat_statements as the always-on profiler. On the app side: pool metrics with alerts, and CREATE INDEX CONCURRENTLY as a hard rule for production migrations since plain CREATE INDEX blocks writes for the duration.

What's the difference between EXPLAIN and EXPLAIN ANALYZE, and what's the risk with the latter?

EXPLAIN plans without executing — estimates only. EXPLAIN ANALYZE actually runs the query and reports per-node actual times, row counts, loops, and (with BUFFERS) I/O. The risk: it really executes — an EXPLAIN ANALYZE on an UPDATE/DELETE modifies data, so wrap it in BEGIN ... ROLLBACK. Also worth knowing: per-node timing instrumentation adds overhead (heavy on some systems), the numbers reflect current cache state so a second run can look very different, and for parameterized queries you should test with realistic, skewed values — the plan for the hottest customer can differ from the plan for an average one. When I can't reproduce prod behavior, auto_explain captures the plan of the actual slow execution.

How does an index satisfy ORDER BY ... LIMIT without sorting, and why is that a big deal?

B-tree leaf pages are sorted and doubly linked, so an Index Scan emits rows already in index order — ascending or, walking the chain backward, descending. If that order matches the ORDER BY, the planner drops the Sort node entirely. The big deal is the interaction with LIMIT: a Sort must consume its entire input before emitting anything (high startup cost), so ORDER BY + LIMIT 20 over 5M matching rows sorts 5M rows to return 20. An index scan streams: it reads exactly ~20 entries and stops. That's the difference between 2 seconds and 0.2 ms, and it's why my composite indexes for list endpoints almost always end with the sort column — equality keys first, sort key last.

Exercise 1 — The 1M-row lab: watch plans change as you add indexes

Build a realistic table and observe every major plan shape. Run each EXPLAIN before and after each index.

CREATE TABLE orders (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  customer_id bigint      NOT NULL,
  status      text        NOT NULL,
  total       numeric(12,2) NOT NULL,
  created_at  timestamptz NOT NULL
);

INSERT INTO orders (customer_id, status, total, created_at)
SELECT (random() * 99999)::bigint + 1,
       (ARRAY['completed','completed','completed','completed',
              'pending','processing','cancelled'])[1 + (random()*6)::int],
       (random() * 500)::numeric(12,2),
       now() - (random() * interval '730 days')
FROM generate_series(1, 1000000);

ANALYZE orders;
  1. EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE customer_id = 42; — note the Seq Scan, Rows Removed by Filter, and buffer reads.
  2. CREATE INDEX idx_orders_customer ON orders (customer_id); then re-run. Did you get Index Scan or Bitmap Heap Scan? Explain why (about 10 rows per customer — what if it were 10,000?).
  3. Run the ORDER BY query: ... WHERE customer_id = 42 ORDER BY created_at DESC LIMIT 5; — find the Sort node. Create idx_orders_cust_created ON orders (customer_id, created_at), re-run, and confirm the Sort node disappears (look for Index Scan Backward).
  4. Query WHERE status = 'pending' — why does it seq-scan even though you could index status? Check the fraction of matching rows. Then try WHERE status = 'cancelled' after creating an index on status — compare.
  5. Record index sizes with \di+ as you go, and time INSERT ... SELECT ... generate_series(1, 100000) with 0, 1, and 3 indexes present. Write down the write-cost ratio you observe.
