Write PostgreSQL live, on the spot: joins, window functions, NULL traps, and the classic problems every interviewer asks.

Why interviewers ask this

As a senior full-stack candidate you will almost certainly face a live SQL round: a shared editor, a small schema, and questions like "top 3 salaries per department" or "month-over-month growth". Interviewers use SQL because it is impossible to fake — either you can produce a correct LEFT JOIN with the filter in the right place, or you can't. Unlike system design, there is a right answer, and they can watch you get there.

Coming from 7 years of frontend, your risk is not concepts — it's fluency. You know what a join is; the question is whether you can type one in 30 seconds without hesitating over ON vs WHERE. This page is built for exactly that: every concept comes with a runnable PostgreSQL query against one consistent sample schema, the expected result, and the trap interviewers are secretly testing. Work through the "Classic problems" section with your hands on a real psql session (setup at the bottom) — reading SQL and writing SQL are different skills.

Senior-level signal in a SQL round is not exotic syntax. It's: (1) narrating your plan before typing, (2) knowing the logical evaluation order (FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT), (3) reaching for window functions instead of self-join gymnastics, and (4) spotting NULL traps before the interviewer points them out.

Hands-on: psql and everyday CRUD Hands-on

Everything above and below assumes you can actually reach a Postgres and type SQL into it. Coming from 7 years of React, that's the one gap theory never fills: you know what a LEFT JOIN is, but you've never opened psql, never run CREATE TABLE, never watched an UPDATE report UPDATE 3. This section is the on-ramp — get a database running, connect, and run the CRUD you'll write every single day. Do it with your hands, not your eyes; the rest of the page lands far harder once these are muscle memory.

Getting Postgres running & connecting

You need a running server and a client. The fastest server for local practice is Docker — one command, no system install, throw it away when done:

# Docker: a throwaway Postgres on the standard port 5432
docker run --name pg -e POSTGRES_PASSWORD=secret -p 5432:5432 -d postgres

# ...or a native install, then it's already running as a service:
#   macOS:   brew install postgresql@16 && brew services start postgresql@16
#   Ubuntu:  sudo apt install postgresql && sudo systemctl start postgresql

The default superuser is postgres and the default database is also postgres. Connect with the psql command-line client. If you used the Docker image above and don't have psql installed locally, docker exec -it pg psql -U postgres gets you a shell inside the container. 

# Connect: -h host, -U user, -d database (it'll prompt for the password)
psql -h localhost -U postgres -d appdb

# Or pass everything as a connection URL — the same string apps use:
psql postgresql://postgres:secret@localhost:5432/appdb

That URL shape — postgresql://user:pass@host:5432/db — is worth memorizing: it's exactly what you'll drop into an app's config, a DATABASE_URL env var, or a Docker Compose file. Once connected, your prompt becomes appdb=# (the # means superuser; a regular user gets =>).

Prefer a GUI? DBeaver (free, all databases) and pgAdmin (Postgres-specific) give you a table browser, a query editor, and visual EXPLAIN. They're great for exploring. But interviews and production incidents happen in psql over SSH — learn the terminal client first, treat the GUI as a convenience, not a crutch.

Create a database, schema & table

Let's build a tiny scratch database from nothing so every statement below has something to run against. This is a throwaway learning DB — separate from the sample schema this page uses elsewhere. From the default postgres connection: 

-- Run once, from any database, to create a fresh one:
CREATE DATABASE appdb;
-- then in psql: \c appdb   (CREATE DATABASE can't run inside a transaction block)

Everything lives in a schema (a namespace); if you don't create one, tables land in the default schema public. Now the tables — a small, coherent domain of customers and their orders: 

CREATE TABLE customers (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,  -- auto-incrementing PK (SQL-standard)
  email       text        NOT NULL UNIQUE,                      -- TEXT: unbounded, no length games
  full_name   varchar(120) NOT NULL,
  is_active   boolean     NOT NULL DEFAULT true,
  created_at  timestamptz NOT NULL DEFAULT now()                -- store timestamps WITH time zone
);

CREATE TABLE orders (
  id           bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  customer_id  bigint       NOT NULL REFERENCES customers(id),  -- FOREIGN KEY → customers
  amount       numeric(10,2) NOT NULL,                          -- NUMERIC for money — never float!
  status       varchar(20)  NOT NULL DEFAULT 'pending',
  created_at   timestamptz  NOT NULL DEFAULT now()
);

Type choices, decoded — this is the everyday palette:

  • bigint GENERATED ALWAYS AS IDENTITY — the modern auto-increment PK. BIGSERIAL is the older shorthand and still everywhere; both give you an auto-filled 64-bit id.
  • text / varchar(n) — text. numeric(10,2) — exact decimals for money. timestamptz — a point in time. boolean — true/false.
  • NOT NULL forbids missing values; DEFAULT fills them in; PRIMARY KEY is unique + not-null + the row's identity; REFERENCES makes a foreign key so an order can't point at a customer that doesn't exist.

Two Postgres-specific choices worth defending out loud. (1) timestamptz over timestamp, always. Despite the name, timestamptz doesn't store a zone — it stores an absolute instant (UTC) and converts on the way in/out per the session's zone. Plain timestamp stores a wall-clock reading with no zone, so the same value means different instants for different users — a bug factory. (2) text vs varchar(n): in Postgres they're the same performance; a length cap buys you nothing but a truncation error, so reach for text unless a real business rule needs the limit.

INSERT — putting rows in

-- Single row. We DON'T supply id/created_at — IDENTITY and DEFAULT fill them.
INSERT INTO customers (email, full_name)
VALUES ('ada@example.com', 'Ada Lovelace');

-- Multi-row: one statement, many tuples — cheaper than N round-trips.
INSERT INTO customers (email, full_name, is_active) VALUES
  ('grace@example.com', 'Grace Hopper', true),
  ('alan@example.com',  'Alan Turing',  false),
  ('edsger@example.com','Edsger Dijkstra', DEFAULT);   -- DEFAULT = use the column default (true)

-- RETURNING: get the generated id back in the SAME statement — no follow-up SELECT.
INSERT INTO orders (customer_id, amount, status)
VALUES (1, 250.00, 'paid')
RETURNING id, created_at;

RETURNING is the one every ORM leans on: after an insert you almost always need the new id, and this hands it back without a second query. (You'll see the same clause on UPDATE and DELETE below.)

SELECT — the everyday patterns

The bread and butter. WHERE filters rows, ORDER BY sorts, LIMIT caps:

-- Common predicates you'll type constantly
SELECT id, email, full_name FROM customers WHERE is_active = true;
SELECT * FROM orders   WHERE status IN ('paid', 'refunded');       -- set membership
SELECT * FROM customers WHERE email LIKE '%@example.com';          -- case-sensitive pattern
SELECT * FROM customers WHERE full_name ILIKE 'a%';                -- ILIKE = case-INSENSITIVE (a Postgres extra)
SELECT * FROM orders   WHERE amount BETWEEN 100 AND 500;           -- inclusive range
SELECT * FROM orders   WHERE status IS NULL;                       -- IS NULL, never = NULL (see pitfall)

-- Sort, then page: newest first, 10 per page, second page
SELECT id, amount, created_at
FROM orders
ORDER BY created_at DESC
LIMIT 10 OFFSET 10;

-- DISTINCT: unique values of a column
SELECT DISTINCT status FROM orders;

A light touch of aggregation (the deep GROUP BY material is later on this page):

SELECT status,
       COUNT(*)      AS n,
       SUM(amount)   AS total,
       ROUND(AVG(amount), 2) AS avg_amount
FROM orders
GROUP BY status;

-- A simple 2-table join: each order with its customer's name
SELECT o.id, c.full_name, o.amount, o.status
FROM orders o
JOIN customers c ON c.id = o.customer_id
ORDER BY o.created_at DESC;

UPDATE and DELETE — change rows carefully

-- UPDATE: SET the new values, WHERE picks the rows. RETURNING shows what changed.
UPDATE orders
SET status = 'refunded'
WHERE id = 1
RETURNING id, status;

-- DELETE: WHERE picks what to remove.
DELETE FROM orders
WHERE status = 'cancelled'
RETURNING *;

An UPDATE or DELETE without a WHERE clause hits every row in the table — UPDATE orders SET status='paid' silently repaints all of them; DELETE FROM orders empties the table. There is no undo outside a transaction. Two habits that save you: (1) write the WHERE first, then go back and add SET; (2) run a SELECT with the same WHERE first to see exactly which rows you're about to touch. In psql you can also wrap risky work in BEGIN; ... ROLLBACK; to preview, and swap to COMMIT; only when the row count looks right.

UPSERT — the idempotent write

"Insert this row, or if it already exists, update it instead." You'll write this constantly — syncing external data, handling retries, maintaining counters. Postgres does it atomically with ON CONFLICT (the conflict target must have a unique constraint — here, email): 

INSERT INTO customers (email, full_name)
VALUES ('ada@example.com', 'Ada King')
ON CONFLICT (email) DO UPDATE
SET full_name = EXCLUDED.full_name;   -- EXCLUDED = the row we tried to insert

Run it once and it inserts; run it again and it updates full_name instead of erroring on the duplicate email. That "same statement, safe to repeat" property is what idempotent means. Use DO NOTHING instead of DO UPDATE when a re-run should simply be a no-op. (There's a deeper dive on ON CONFLICT and EXCLUDED in the UPSERT section below.)

Pagination in practice

LIMIT n OFFSET m is the pagination you'll reach for first, and it's fine for small offsets and admin screens. Its catch: OFFSET 100000 still makes Postgres generate and discard 100,000 rows to reach your page, so deep pages get slower and slower, and rows shifting under you can duplicate or skip entries between pages. 

-- Offset pagination: simple, but degrades on deep pages
SELECT id, amount FROM orders ORDER BY id LIMIT 20 OFFSET 40;   -- page 3

The scalable alternative is keyset (seek) pagination: instead of an offset, remember the last row you saw and ask for rows after it — WHERE id > :last_id ORDER BY id LIMIT 20. It stays fast at any depth because an index jumps straight to the cursor. It's the right default for infinite-scroll feeds and public APIs; there's a fuller treatment elsewhere, but knowing the trade-off is enough to say the right thing in an interview.

Wiring it up from Spring

This connects straight to the Java side of the course. The exact same server, the exact same SQL — Spring just points a JDBC URL at it. In application.properties: 

spring.datasource.url=jdbc:postgresql://localhost:5432/appdb
spring.datasource.username=postgres
spring.datasource.password=secret

Note the jdbc: prefix on the same host:5432/appdb shape you connected to with psql. Whether you use JdbcTemplate, Spring Data JPA repositories, or Hibernate, the SQL that ultimately hits the database is the SELECT/INSERT/UPDATE you just practised by hand — an ORM writes it for you, it doesn't replace it. Being able to open psql and reproduce (or debug) the query your app runs is exactly the fluency interviewers are probing for.

How do you do an upsert in PostgreSQL?

INSERT ... ON CONFLICT (col) DO UPDATE SET .... The conflict target column(s) must be covered by a unique constraint or unique index — that's how Postgres detects the collision race-free. Inside DO UPDATE, EXCLUDED.col refers to the row you tried to insert and the table name refers to the existing row, so you can merge them (e.g. SET views = orders.views + EXCLUDED.views to accumulate). Use DO NOTHING when a duplicate should be silently ignored. It's a single atomic statement, which is why it's the go-to pattern for idempotent writes and retry-safe imports — far better than a read-then-write in application code, which has a race between the check and the insert.

LIMIT/OFFSET vs keyset pagination — when would you switch?

LIMIT/OFFSET is simple and fine for shallow pages, but Postgres must scan and throw away every row before the offset, so page 5,000 is slow, and concurrent inserts/deletes can shift the window and cause rows to repeat or vanish across pages. Keyset (seek) pagination uses the last-seen value as a cursor — WHERE (created_at, id) < (:last_ts, :last_id) ORDER BY created_at DESC, id DESC LIMIT 20 — so an index seeks directly to the next page in constant time regardless of depth, and it's stable under concurrent writes. I'd switch as soon as pages go deep or the data is high-churn: infinite-scroll feeds, public list APIs, exports. The trade-off is you can't jump to an arbitrary page number, only next/previous, and multi-column ordering needs a matching composite index.

The five joins in one mental model

A join matches rows from two tables using a predicate. The join type only decides what happens to rows that don't match:

  • LEFT JOIN — keep every left row; right-side columns become NULL when there's no match.
  • RIGHT JOIN — mirror of LEFT. In practice, seniors rewrite it as a LEFT JOIN with tables swapped (easier to read top-to-bottom).

  • CROSS JOIN — no predicate at all: every left row paired with every right row (Cartesian product).

THE trap: ON vs WHERE on a LEFT JOIN

This is the single most common join mistake in live interviews. Task: "For every department, count employees hired since 2021 — departments with zero must still appear." 

-- WRONG: filter on the right table in WHERE
SELECT, COUNT( AS cnt
FROM departments d
LEFT JOIN employees e ON e.dept_id =
WHERE e.hired_at >= DATE '2021-01-01'    -- ← kills the NULL-padded rows!
GROUP BY;

Result (wrong): Engineering 1 (Dev), Sales 1 (Ravi), HR 1 (Lena). Marketing has vanished. Why: the LEFT JOIN dutifully produced a (Marketing, NULL) row, but then WHERE e.hired_at >= '2021-01-01' evaluated to NULL >= '2021-01-01' → UNKNOWN → row filtered out. A WHERE condition on right-table columns turns a LEFT JOIN into an INNER JOIN.

-- RIGHT: put right-table filters in the ON clause
SELECT, COUNT( AS cnt
FROM departments d
LEFT JOIN employees e
       ON e.dept_id =
      AND e.hired_at >= DATE '2021-01-01'   -- filter BEFORE the outer join pads
GROUP BY
ORDER BY;

Result (right): Engineering 1, HR 1, Marketing 0, Sales 1. The rule:

Filter targetWhere it goesWhy
Left (preserved) tableWHEREYou genuinely want to exclude those left rows
Right (nullable) tableONFilter match candidates, keep unmatched left rows
Right table, and you want INNER semanticsWHERE — but then just write JOINBe honest about intent

Also note COUNT(, not COUNT(*). COUNT(*) counts the NULL-padded row too, so Marketing would show 1 instead of 0. Counting a right-table column on a LEFT JOIN is the idiomatic "count matches, zero allowed" pattern.

Anti-joins: "rows in A with no match in B"

Two idiomatic patterns; know both, prefer NOT EXISTS when in doubt:

-- Pattern 1: LEFT JOIN ... IS NULL
SELECT
FROM departments d
LEFT JOIN employees e ON e.dept_id =
WHERE IS NULL;                         -- probe a NOT NULL column of the right table

-- Pattern 2: NOT EXISTS (usually the same plan in Postgres, and NULL-safe)
SELECT
FROM departments d
WHERE NOT EXISTS (
  SELECT 1 FROM employees e WHERE e.dept_id =
);

Result of both: Marketing. The IS NULL test must be on a column that can only be NULL because the join failed — the right table's primary key is the safe choice. Never write NOT IN (SELECT dept_id ...) here: dept_id contains a NULL (Omar) and the query returns zero rows — full explanation in the NULL section.

The mirror pattern, a semi-join ("rows in A that have at least one match in B"), is EXISTS — better than JOIN + DISTINCT because it never duplicates left rows:

SELECT FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = AND o.status = 'paid');
-- Acme, Globex, Initech, Umbrella (Soylent excluded)

Aggregation: GROUP BY, HAVING, FILTER T1

GROUP BY mechanics

GROUP BY collapses rows into one row per distinct group key. After grouping, the only things you may SELECT are (a) the grouping columns and (b) aggregates of everything else. This is the #1 beginner error interviewers watch for: 

-- ERROR: column "" must appear in the GROUP BY clause or be used in an aggregate
SELECT dept_id, name, AVG(salary) FROM employees GROUP BY dept_id;

Postgres refuses because after collapsing Engineering's 4 rows into 1, "the name" is ambiguous — whose name? (If you actually want "a name per group", that's the top-1-per-group problem — window functions or DISTINCT ON, later.) The correct aggregate query:

SELECT AS department,
       COUNT(*)              AS headcount,
       ROUND(AVG(e.salary))  AS avg_salary,
       MAX(e.salary)         AS top_salary
FROM employees e
JOIN departments d ON =
GROUP BY,
ORDER BY avg_salary DESC;

WHERE vs HAVING

WHERE filters rows before grouping; HAVING filters groups after aggregation. Mnemonic: WHERE can't see aggregates because they don't exist yet.

-- Departments whose average PAID... no — whose avg salary among post-2019 hires exceeds 100k
SELECT dept_id, ROUND(AVG(salary)) AS avg_salary
FROM employees
WHERE hired_at >= DATE '2019-01-01'   -- row filter (cheap, can use indexes)
GROUP BY dept_id
HAVING AVG(salary) > 100000;          -- group filter (needs the aggregate)

Result: dept 1 → 125000 (Arjun, Maya, Dev — Priya's 180k excluded by WHERE), dept 2 → 125000. HR's 70000 group is computed, then discarded by HAVING; Omar's NULL dept group (70000) likewise.

Putting a plain row condition in HAVING (HAVING hired_at > ...) is both an error (not grouped, not aggregated) and a smell even where legal: filter rows as early as possible. In interviews, misplacing the filter is read as "doesn't understand evaluation order".

COUNT(*) vs COUNT(col) vs COUNT(DISTINCT col)

SELECT COUNT(*)                 AS all_rows,        -- 8: counts rows, period
       COUNT(dept_id)           AS with_dept,       -- 7: skips NULLs (Omar)
       COUNT(DISTINCT dept_id)  AS distinct_depts   -- 3: distinct non-NULL values
FROM employees;

All aggregates except COUNT(*) ignore NULLs. That's why COUNT(right_table.pk) gives a true zero on LEFT JOINs, and why AVG can silently exclude rows you thought you were averaging.

NULL gotchas & three-valued logic T1

Three-valued logic in 60 seconds

NULL means "unknown", not "empty". Any comparison with NULL yields UNKNOWN — not true, not false. WHERE keeps only rows where the predicate is true, so UNKNOWN rows are dropped, silently. 

SELECT NULL = NULL;      -- NULL   (not true!)
SELECT NULL <> 5;        -- NULL
SELECT NULL AND false;   -- false  (false wins regardless of unknown)
SELECT NULL OR true;     -- true   (true wins regardless of unknown)
SELECT NULL IS NULL;     -- true   (IS is the only safe NULL test)
SELECT NULL IS DISTINCT FROM 5;   -- true: NULL-safe "not equal"

Consequence: WHERE dept_id = 1 plus WHERE dept_id <> 1 does not cover all employees — Omar (NULL dept) is in neither. To partition data completely you need dept_id <> 1 OR dept_id IS NULL, or the NULL-safe operator dept_id IS DISTINCT FROM 1.

The NOT IN trap (the most famous NULL bug in SQL)

-- "Departments with no employees" — looks right, returns NOTHING:
SELECT name FROM departments
WHERE id NOT IN (SELECT dept_id FROM employees);
-- 0 rows. Expected: Marketing.

Why: the subquery returns {1, 1, 1, 1, 2, 2, 3, NULL} (Omar's NULL). 4 NOT IN (1,2,3,NULL) expands to 4<>1 AND 4<>2 AND 4<>3 AND 4<>NULL = true AND true AND true AND UNKNOWN = UNKNOWN → filtered out. One NULL in the list poisons the entire NOT IN. Fixes, best first:

-- 1. NOT EXISTS: NULL-safe, and the planner turns it into an efficient anti-join
SELECT name FROM departments d

COALESCE and NULLIF

-- COALESCE: first non-NULL argument (like JS ?? but variadic)
SELECT name, COALESCE(dept_id::text, 'unassigned') AS dept FROM employees;

-- NULLIF(a, b): NULL if a = b, else a. Classic use: dodge division by zero
SELECT paid / NULLIF(total, 0) AS paid_ratio FROM (...) t;   -- NULL, not an error, when total = 0

-- They compose: default the ratio to 0
SELECT COALESCE(paid / NULLIF(total, 0), 0) FROM (...) t;

NULLs in ORDER BY

Postgres treats NULL as larger than everything for sorting: ORDER BY x ASC puts NULLs last, ORDER BY x DESC puts NULLs first — which surprises people building "top N" lists on a nullable column:

SELECT name, dept_id FROM employees ORDER BY dept_id DESC;
-- Omar (NULL) comes FIRST. Usually not what you want:
SELECT name, dept_id FROM employees ORDER BY dept_id DESC NULLS LAST;

NULLs and aggregates

  • SUM/AVG/MIN/MAX/COUNT(col) all skip NULLs. AVG divides by the count of non-NULL values, not the row count.
  • Aggregating an empty set: COUNT → 0, everything else → NULL. Wrap in COALESCE(SUM(x), 0) for reports.
  • GROUP BY and DISTINCT treat NULLs as equal to each other (unlike =): all NULL dept_ids form one group. Same for PARTITION BY and UNION dedup.
  • string_agg, array_agg skip NULL inputs too.

Window functions T1

Window functions are the modern-SQL differentiator in interviews. They compute an aggregate-like value per row over a "window" of related rows — without collapsing the rows like GROUP BY does. If you learn one section on this page cold, make it this one.

OVER(): the anatomy

function(args) OVER (
  PARTITION BY ...   -- restart the calculation per group (optional)
  ORDER BY ...       -- ordering within the partition (needed for rank/lag/running)
  ROWS BETWEEN ...   -- frame: which neighbors to include (optional)

Running totals

SELECT id, customer_id, created_at, amount,
       SUM(amount) OVER (ORDER BY created_at, id) AS running_total
FROM orders;

Adding ORDER BY inside OVER turns SUM into a cumulative sum: each row gets the sum of everything up to and including itself. First rows: 250.00 → 370.00 → 670.00 → 1120.00 … Add PARTITION BY customer_id for a per-customer running balance (classic problem 10). The , id tiebreaker makes the total deterministic when two orders share a date.

Moving averages — the frame clause

SELECT id, created_at, amount,
       ROUND(AVG(amount) OVER (
         ORDER BY created_at, id
         ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
       ), 2) AS moving_avg_3
FROM orders;

ROWS BETWEEN 2 PRECEDING AND CURRENT ROW = "this row and the two before it": a 3-order moving average. Other useful frames: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (explicit running total), RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW (time-based window, PG11+).

Subtlety worth mentioning if probed: with ORDER BY and no explicit frame, the default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — RANGE, not ROWS — so ties in the ORDER BY key are included together. For running totals over a non-unique column, spell out ROWS or add a unique tiebreaker.

Subqueries & CTEs T2

The three shapes of subquery

-- Scalar subquery: returns exactly one value; usable anywhere a value goes
SELECT name, salary,
       salary - (SELECT AVG(salary) FROM employees) AS vs_company_avg
FROM employees;

-- Row/column subquery: feeds IN / comparison with a set
SELECT name FROM employees
WHERE dept_id IN (SELECT id FROM departments WHERE name IN ('Sales','HR'));

-- Table subquery (derived table): a query in FROM — must be aliased
SELECT dept_id, max_sal
FROM (SELECT dept_id, MAX(salary) AS max_sal
      FROM employees GROUP BY dept_id) t
WHERE max_sal > 100000;

A scalar subquery that returns more than one row throws ERROR: more than one row returned by a subquery used as an expression — at runtime, only when the data misbehaves. If multiple rows are possible, force one with LIMIT 1, an aggregate, or rewrite as a join.

Correlated subqueries

A correlated subquery references columns of the outer query, so conceptually it re-runs per outer row:

-- Employees earning above their own department's average
SELECT name, dept_id, salary
FROM employees e
WHERE salary > (SELECT AVG(salary)
                FROM employees e2
                WHERE e2.dept_id = e.dept_id);   -- ← correlation
-- Priya, Arjun, Maya (Eng avg 138750... Priya yes; 140000 > 138750 yes), Ravi (130k > 125k)

Performance note: Postgres often can't decorrelate an arbitrary correlated subquery — think O(outer × inner) unless an index saves you. EXISTS-style correlation is fine (planned as a semi-join); correlated aggregates like the above are cleaner and usually faster as a window function:

SELECT name, dept_id, salary
FROM (SELECT *, AVG(salary) OVER (PARTITION BY dept_id) AS dept_avg
      FROM employees) t
WHERE salary > dept_avg;   -- one scan instead of N subquery executions

CTEs (WITH) — structuring a query like functions

CTEs name intermediate result sets. In a live interview they are your best friend: build the answer in named steps, testing each step as you go — exactly like extracting helper functions:

-- "Which customers' paid revenue is above the average customer's paid revenue?"
WITH per_customer AS (
  SELECT customer_id, SUM(amount) AS revenue
  FROM orders
  WHERE status = 'paid'
  GROUP BY customer_id
),
avg_revenue AS (
  SELECT AVG(revenue) AS avg_rev FROM per_customer
)
SELECT, pc.revenue
FROM per_customer pc
JOIN customers c ON = pc.customer_id
CROSS JOIN avg_revenue a
WHERE pc.revenue > a.avg_rev;
-- avg is (690+500+650+75)/4 = 478.75 → Acme (690), Initech (650), Globex (500)

MATERIALIZED / NOT MATERIALIZED (PG12+)

Before PostgreSQL 12, every CTE was an optimization fence: computed once into a temp result, predicates from outside could not be pushed in. Since PG12, single-use CTEs are inlined into the main query by default (so indexes and filters apply through them). You can override: 

WITH big AS MATERIALIZED (          -- force compute-once (fence): dedupe expensive work,
  SELECT ...                        -- or pin a plan the optimizer keeps getting wrong
)
SELECT ... ;

WITH big AS NOT MATERIALIZED (      -- force inlining even if referenced twice
  SELECT ...
)
SELECT ... ;

Interview soundbite: "CTEs are for readability; since PG12 they're inlined by default so there's no performance penalty for single-use CTEs. If a CTE is referenced multiple times or I explicitly want it computed once, I mark it MATERIALIZED." That one sentence covers 95% of CTE follow-up questions.

UPSERT & RETURNING T2

INSERT ... ON CONFLICT

Postgres's atomic upsert. Requirement: the conflict target must be covered by a unique index or constraint (PK, UNIQUE, or a unique partial index) — that's how Postgres detects the collision race-free.

-- DO NOTHING: idempotent insert (retries, imports, at-least-once consumers)
INSERT INTO customers (id, name, country)
VALUES (2, 'Globex', 'DE')
ON CONFLICT (id) DO NOTHING;         -- INSERT 0 0 — row existed, no error, no change

-- DO UPDATE: insert-or-update. EXCLUDED = the row you TRIED to insert.
INSERT INTO customers (id, name, country)
VALUES (2, 'Globex GmbH', 'DE')
ON CONFLICT (id) DO UPDATE
SET name    =,       -- take the incoming value
    country =
WHERE IS DISTINCT FROM;  -- optional: skip no-op updates (avoids dead tuples + triggers)

Mental model for the two row aliases inside DO UPDATE: customers.col (or the table alias) = the existing row; EXCLUDED.col = the incoming row that conflicted. Mixing them lets you do merge logic:

-- Accumulate instead of overwrite (e.g. a counters table)
INSERT INTO page_views (page, views) VALUES ('/pricing', 1)
ON CONFLICT (page) DO UPDATE SET views = page_views.views + EXCLUDED.views;

ON CONFLICT without a target (ON CONFLICT DO NOTHING) matches any unique violation; DO UPDATE always requires an explicit target — Postgres must know which constraint's collision you're handling. And no unique constraint = there is no unique or exclusion constraint matching the ON CONFLICT specification error. Interviewers love asking "what does ON CONFLICT need to work?"

RETURNING — skip the follow-up SELECT

INSERT/UPDATE/DELETE can hand back the affected rows in the same statement:

INSERT INTO orders (id, customer_id, amount, status, created_at)
VALUES (111, 5, 42.00, 'pending', CURRENT_DATE)
RETURNING id, created_at;            -- get generated ids/defaults without a second round-trip

UPDATE orders SET status = 'paid'
WHERE status = 'pending' AND created_at < DATE '2025-02-01'
RETURNING id, customer_id;           -- exactly which rows changed

DELETE FROM orders WHERE status = 'cancelled'
RETURNING *;                         -- audit what was removed

Connection to your Java track: when JPA/Hibernate saves an entity with GenerationType.IDENTITY, the generated key comes back via exactly this mechanism — the driver runs INSERT ... RETURNING id under the hood. Knowing that links your ORM knowledge to raw SQL nicely in an interview.

Recursive CTEs & set operations T3

Recursive CTE: walking the org chart

Hierarchies (org charts, category trees, comment threads) live in adjacency lists like our manager_id. A recursive CTE traverses them:

WITH RECURSIVE org AS (
  -- 1. Base case: the root(s) — runs once
  SELECT id, name, manager_id,
         1 AS depth,
         name AS path
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- 2. Recursive step: join NEW rows from last iteration ("org") to their reports
  SELECT,, e.manager_id,
         o.depth + 1,
         o.path || ' > ' ||
  FROM employees e
  JOIN org o ON e.manager_id =
)
SELECT depth, path FROM org ORDER BY path;

Line by line: the base case seeds the working set with Priya (depth 1). Each iteration joins employees against only the rows produced by the previous iteration — so round 2 finds Priya's directs (Arjun, Maya, Sana, Lena), round 3 finds their reports (Dev under Arjun; Ravi and Omar under Sana), round 4 finds nothing and recursion stops. UNION vs UNION ALL, INTERSECT, EXCEPT

SELECT country FROM customers          -- US, DE, US, UK, US
UNION
SELECT 'US';                           -- UNION dedups: {US, DE, UK} — 3 rows

SELECT country FROM customers
UNION ALL
SELECT 'US';                           -- keeps everything: 6 rows
  • UNION ALL: concatenate, cheap, streams. UNION: concatenate + de-duplicate — a hash/sort over the whole result. If the branches can't overlap (or duplicates are fine), UNION ALL; saying "UNION ALL, because dedup costs a sort and I know these sets are disjoint" is free senior credit.
  • INTERSECT: rows in both. EXCEPT: rows in the first minus the second. Both dedup (there are ALL variants). Handy: they compare whole rows and treat NULLs as equal — a quick NULL-safe diff of two tables: (TABLE a EXCEPT TABLE b) UNION ALL (TABLE b EXCEPT TABLE a).
  • All branches must have the same number of columns with compatible types; ORDER BY/LIMIT apply to the combined result (parenthesize a branch to sort it alone).
-- Customers who ordered in BOTH January and February (also see classic problem 9)
SELECT customer_id FROM orders WHERE created_at >= '2025-01-01' AND created_at < '2025-02-01'
INTERSECT
SELECT customer_id FROM orders WHERE created_at >= '2025-02-01' AND created_at < '2025-03-01';
-- → 1 (Acme), 2 (Globex)

Classic interview problems — solve before revealing

These are the questions that actually get asked, ordered roughly by frequency. Protocol: read the problem, write your answer in psql against the sample schema, then open the solution. If you couldn't produce it in ~3 minutes, redo it from scratch tomorrow.

1. Find the second highest salary (be ready to show 3 ways). 
-- Way 1: DISTINCT + OFFSET (shortest)
SELECT DISTINCT salary FROM employees
ORDER BY salary DESC
OFFSET 1 LIMIT 1;                                   -- → 140000

-- Way 2: max below max (returns NULL, not empty, if no 2nd — often the asked-for behavior)
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- Way 3: DENSE_RANK (generalizes to "Nth highest" — say this!)
SELECT DISTINCT salary
FROM (SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
      FROM employees) t
WHERE rnk = 2;

Common wrong answer: ORDER BY salary DESC OFFSET 1 LIMIT 1 without DISTINCT. If the top salary is duplicated (imagine two people at 180000), row #2 is still 180000 — you return the highest salary, not the second highest. On our data it happens to work, which is exactly how this bug ships. Also state the tie question up front: "second highest value or second person?" — DENSE_RANK vs ROW_NUMBER.

7. Find users with a streak of 3+ consecutive login days (gaps-and-islands). 
-- Setup for this one problem:
CREATE TABLE logins (user_id int, login_date date);
INSERT INTO logins VALUES
  (1,'2025-03-01'),(1,'2025-03-02'),(1,'2025-03-03'),(1,'2025-03-05'),
  (2,'2025-03-02'),(2,'2025-03-04'),(1,'2025-03-02');  -- note a duplicate login

WITH days AS (
  SELECT DISTINCT user_id, login_date FROM logins        -- dedupe multiple logins per day!
),
grouped AS (
  SELECT user_id, login_date,
         login_date - ROW_NUMBER() OVER (PARTITION BY user_id
                                         ORDER BY login_date)::int AS grp
  FROM days
)
SELECT user_id,
       MIN(login_date) AS streak_start,
       MAX(login_date) AS streak_end,
       COUNT(*)        AS streak_len
FROM grouped
GROUP BY user_id, grp
HAVING COUNT(*) >= 3;
-- → user 1: 2025-03-01 .. 2025-03-03, len 3

The trick: for consecutive dates, date − row_number is constant (03-01−1 = 03-02−2 = 03-03−3 = Feb 28); the moment a day is skipped, the difference jumps to a new constant. That constant (grp) identifies each "island", and a GROUP BY over it measures streaks. The alternative spelling uses LAG: flag rows where login_date - LAG(login_date) > 1 as streak-starts, then a running SUM of flags builds the same group id.

Common wrong answers: forgetting the DISTINCT — duplicate same-day logins shift ROW_NUMBER and break every streak after them; and trying nested self-joins (t1.date =) — it detects exactly-3 windows but doesn't generalize and explodes combinatorially.

8. Month-over-month revenue growth in % (paid orders only). 
WITH monthly AS (
  SELECT date_trunc('month', created_at)::date AS month,
         SUM(amount) AS revenue
  FROM orders
  WHERE status = 'paid'
  GROUP BY 1
)
SELECT month, revenue,
       ROUND(
         100.0 * (revenue - LAG(revenue) OVER (ORDER BY month))
               / NULLIF(LAG(revenue) OVER (ORDER BY month), 0)
       , 1) AS growth_pct
FROM monthly
ORDER BY month;

9. Customers who placed orders in BOTH January and February 2025. 
-- Way 1: GROUP BY + HAVING over distinct months (scales to "all N months")

              AND o.created_at >= '2025-01-01' AND o.created_at < '2025-02-01')
  AND EXISTS (SELECT 1 FROM orders o WHERE o.customer_id =
              AND o.created_at >= '2025-02-01' AND o.created_at < '2025-03-01');

Common wrong answer (the classic): WHERE month = 'Jan' AND month = 'Feb' — always empty; a single row has one month, AND applies per-row. "Both" across rows requires grouping (HAVING over distinct months), set intersection, or two correlated existence checks. This misread is precisely what the question screens for.

10. Running total of order amounts per customer, ordered by date. 
SELECT customer_id, created_at, id AS order_id, amount,
       SUM(amount) OVER (
         PARTITION BY customer_id
         ORDER BY created_at, id
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total
FROM orders
ORDER BY customer_id, created_at, id;

For Acme (customer 1): 250.00 → 370.00 → 460.00 → 780.00. PARTITION restarts the sum per customer; ORDER BY inside OVER makes it cumulative; id breaks same-day ties deterministically. The frame clause is technically the default behavior here spelled with ROWS — writing it shows you know the RANGE-default subtlety (ties in the ORDER BY key would otherwise be summed together).

Common wrong answers: plain GROUP BY customer_id with SUM — gives the final total only, losing the per-order progression; putting the ORDER BY only at statement level (outside OVER) — that orders the display, not the accumulation, so you get the full total on every row.

11. Median salary (and per-department median). 
SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM employees;
-- 8 salaries: 70,70,95,120,130,140,140,180 → (120+130)/2 = 125000

SELECT dept_id,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM employees
GROUP BY dept_id;
-- dept 1 → 140000, dept 2 → 125000, dept 3 → 70000, NULL → 70000

PERCENTILE_CONT is an ordered-set aggregate — the WITHIN GROUP (ORDER BY ...) clause supplies the ordering the percentile is computed over. _CONT interpolates between the two middle values on even counts; PERCENTILE_DISC(0.5) returns an actual value from the data (here 120000).

Common wrong answers: AVG (that's the mean — the interviewer chose "median" deliberately); hand-rolling with ROW_NUMBER and COUNT to pick middle rows — legal but 10 lines of fiddly off-by-one risk, and it signals you don't know Postgres has percentiles built in.

13. Each employee's salary rank within their department, and the % of dept payroll they represent. 
SELECT name, dept_id, salary,
       DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC)     AS dept_rank,
       ROUND(100.0 * salary / SUM(salary) OVER (PARTITION BY dept_id), 1) AS pct_of_payroll
FROM employees

Two windows over the same partition, no self-joins, rows preserved. This is the "show me you actually use window functions" composite — rank + share-of-total in one pass.

Common wrong answer: computing dept totals in a GROUP BY subquery and joining back. It works, but it's the pre-window-function idiom; produce the window version and mention the join-back as the legacy shape.

14. For each customer show their first and most recent order amount in one row. 
-- Way 1: FIRST_VALUE / LAST_VALUE (mind the frame!) + DISTINCT
SELECT DISTINCT customer_id,
       FIRST_VALUE(amount) OVER w AS first_amount,
       LAST_VALUE(amount)  OVER w AS latest_amount
FROM orders
WINDOW w AS (PARTITION BY customer_id ORDER BY created_at, id
             ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING);

-- Way 2 (often cleaner): two DISTINCT ON queries joined, or conditional agg:
SELECT customer_id,
       (array_agg(amount ORDER BY created_at, id))[1]                    AS first_amount,
       (array_agg(amount ORDER BY created_at DESC, id DESC))[1]          AS latest_amount
FROM orders
GROUP BY customer_id;
-- customer 1 → first 250.00, latest 320.00

The trap this question exists for: LAST_VALUE with the default frame (... AND CURRENT ROW) returns the current row's value — the frame ends at the row being computed, so "last" is "so far". You must extend the frame to UNBOUNDED FOLLOWING. The named WINDOW w clause avoids repeating the definition. The array_agg(...)[1] trick is a handy Postgres idiom for "first value by some order" inside a plain GROUP BY.

Interview Q&A

Q1: What's the logical order of evaluation of a SELECT statement, and why does it matter?

FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT (expressions, window functions) → DISTINCT → ORDER BY → LIMIT/OFFSET. It explains most SQL "why can't I" questions: WHERE can't use aggregates (not computed yet) or SELECT aliases (SELECT runs later); HAVING can use aggregates (post-GROUP BY); ORDER BY can use aliases (it runs after SELECT); window functions can't be filtered in the same query level (they're part of SELECT). Note it's the logical order — the planner physically reorders freely as long as results match.

Q2: WHERE vs HAVING — when is each evaluated and which is cheaper?

WHERE filters individual rows before grouping; HAVING filters groups after aggregation. WHERE is preferred whenever the condition doesn't involve an aggregate: it shrinks the data before the (expensive) grouping and can use indexes. HAVING exists precisely for conditions on aggregate results (HAVING COUNT(*) > 1). A row-level condition placed in HAVING is at best wasted work, at worst an error.

Q3: GROUP BY vs PARTITION BY — what's the fundamental difference?

GROUP BY collapses: one output row per group, and you lose row-level detail. PARTITION BY (in a window function) annotates: every input row survives, carrying a value computed over its partition. Use GROUP BY when the answer is per-group ("average salary per dept"); use a window when the answer is per-row with group context ("each employee vs their dept average", ranks, running totals). Windows also allow ordering and frames within the partition, which GROUP BY has no concept of.

Q4: Why can a WHERE clause silently turn a LEFT JOIN into an INNER JOIN?

The LEFT JOIN NULL-pads right-side columns for unmatched left rows. Any ordinary WHERE predicate on a right-table column then evaluates to UNKNOWN for those rows (NULL >= x is UNKNOWN) and WHERE discards them — exactly the rows the LEFT JOIN existed to preserve. Fix: move right-table conditions into the ON clause (filter match candidates before padding), or make the predicate NULL-tolerant (... OR right_col IS NULL). One exception is legitimate in WHERE: right_pk IS NULL — that's the anti-join pattern, which deliberately keeps only unmatched rows.

Q5: Explain the NOT IN + NULL problem and your default alternative.

x NOT IN (a, b, NULL) expands to x<>a AND x<>b AND x<>NULL; the last term is UNKNOWN, so the whole conjunction is at best UNKNOWN, and WHERE drops it — one NULL in the subquery result makes NOT IN return zero rows, with no error. Default alternative: NOT EXISTS with a correlated predicate — it asks "is there a matching row?" rather than comparing values, so NULLs can't poison it, and Postgres plans it as an efficient anti-join.

Q6: COUNT(*) vs COUNT(column) vs COUNT(DISTINCT column)?

COUNT(*) counts rows. COUNT(col) counts rows where col IS NOT NULL. COUNT(DISTINCT col) counts distinct non-NULL values. Practical corollary: on a LEFT JOIN, COUNT(right.pk) yields 0 for unmatched left rows while COUNT(*) yields 1 — the classic off-by-one in "count children per parent" reports. Performance-wise COUNT(*) is not slower in Postgres (it doesn't fetch "all columns"; it just counts tuples).

Q7: ROW_NUMBER vs RANK vs DENSE_RANK — and which for "top N per group"?

On ties: ROW_NUMBER numbers arbitrarily but uniquely (1,2,3,4); RANK shares then skips (1,2,2,4); DENSE_RANK shares without gaps (1,2,2,3). For top-N-per-group: ROW_NUMBER when you want exactly N rows per group (add a unique tiebreaker for determinism); DENSE_RANK when you want everyone within the top N values (ties included, possibly more than N rows). The senior move is asking the interviewer which tie behavior they want before choosing.

Q8: What is a correlated subquery, and what's the performance concern?

A subquery that references the outer query's columns, making it logically dependent on — and re-evaluated for — each outer row. Concern: if the planner can't decorrelate it, you get nested-loop behavior, O(N×M) without a supporting index. EXISTS/IN correlations are routinely flattened into semi-joins (fine); correlated aggregate subqueries ("salary > dept average") often aren't — rewrite those as window functions or a pre-aggregated CTE joined back. Verify with EXPLAIN: a SubPlan node executed per-row is the red flag.

Q9: CTE vs subquery vs view — how do you choose, and what changed in PG12?

Functionally interchangeable for a single query; choose for readability — CTEs read top-down like pipeline steps and can be referenced multiple times; a view is a persisted, shared query. Pre-PG12, CTEs were optimization fences (always materialized, no predicate pushdown) — old advice to avoid them for performance dates from then. Since PG12, non-recursive, single-referenced, side-effect-free CTEs are inlined by default; MATERIALIZED forces the old compute-once behavior (useful when referenced multiple times or to pin a plan), NOT MATERIALIZED forces inlining.

Q10: How does INSERT ... ON CONFLICT work, and what does it require?

It attempts the insert; if a unique violation occurs on the specified conflict target, it runs the alternative action atomically — DO NOTHING or DO UPDATE, where EXCLUDED exposes the proposed row and the table name/alias the existing one. Requirements: the target must match a unique index/constraint (or constraint name via ON CONFLICT ON CONSTRAINT ...); DO UPDATE requires an explicit target. It's race-free under concurrency, unlike SELECT-then-INSERT, which needs retry loops. Since PG15 there's also standard MERGE for multi-branch logic, but ON CONFLICT remains the idiomatic upsert.

Q11: What does RETURNING give you, and where have you seen it matter in an app stack?

Any INSERT/UPDATE/DELETE can return the affected rows' columns — generated ids, defaults, computed values — in the same statement, eliminating a second round-trip and a race window. App-stack relevance: ORMs retrieve IDENTITY-generated keys this way; "claim a job" queues use UPDATE ... SET claimed_by=? WHERE id = (SELECT ... FOR UPDATE SKIP LOCKED LIMIT 1) RETURNING *; soft-delete endpoints return the deleted entity from the DELETE itself.

Q12: UNION vs UNION ALL — semantics and cost?

UNION ALL concatenates results; UNION additionally removes duplicate rows (across and within branches), which costs a hash or sort over the entire combined set and blocks streaming. Default to UNION ALL and opt into UNION only when duplicates are possible and unwanted. Bonus nuance: UNION's dedup treats NULLs as equal (IS NOT DISTINCT FROM semantics), unlike =.

Q13: How do NULLs behave in ORDER BY, GROUP BY, DISTINCT, and unique indexes in Postgres?

ORDER BY: NULLs sort as largest — last in ASC, first in DESC; override with NULLS FIRST/LAST. GROUP BY / DISTINCT / PARTITION BY: NULLs are treated as equal, forming one group. Unique indexes: NULLs are treated as distinct, so many rows with NULL in a unique column are allowed by default (PG15+ can change this with UNIQUE NULLS NOT DISTINCT). The inconsistency — equal for grouping, distinct for uniqueness — is a favorite trivia follow-up.

Q14: EXISTS vs IN vs JOIN for "customers that have orders" — trade-offs?

All express a semi-join. JOIN + DISTINCT is the trap: the join first multiplies customer rows by their order count, then dedups — wasted work and a bug if you forget DISTINCT. EXISTS states the intent exactly ("at least one match"), stops probing at the first hit, and its negation is NULL-safe. IN reads well for single-column membership and plans equivalently in Postgres, but NOT IN inherits the NULL trap. Preference order for the positive case: EXISTS ≈ IN > JOIN+DISTINCT; for the negative: NOT EXISTS, full stop.

Q15: What is ctid, and when is it legitimately useful?

ctid is a system column giving the tuple's physical address (page, offset) in the heap. It uniquely identifies a row right now, but changes on any UPDATE (MVCC writes a new tuple version) and on VACUUM FULL/CLUSTER — never store or expose it. Legitimate uses: deduplicating a table that lacks any unique key (DELETE ... USING ... WHERE a.ctid > b.ctid), and low-level debugging (finding which page a row lives on, spotting HOT-update behavior). Mentioning why it's unstable (MVCC) is the senior half of the answer.

E1. For each country, the number of customers and their total paid revenue — countries with customers but no paid orders must still show 0. 
SELECT,
       COUNT(DISTINCT                    AS customers,
       COALESCE(SUM(o.amount), 0)             AS paid_revenue
FROM customers c
LEFT JOIN orders o
       ON o.customer_id =
      AND o.status = 'paid'          -- right-table filter in ON, not WHERE!
GROUP BY
ORDER BY paid_revenue DESC;
-- US 3 / 1340.00 (Acme 690 + Initech 650 + Soylent 0), DE 1 / 500.00, UK 1 / 75.00

Tests the ON-vs-WHERE rule plus COUNT(DISTINCT) — a customer with 3 paid orders must count once. WHERE o.status='paid' would drop Soylent from the US count entirely.

E2. Each order with the gap in days since that customer's previous order. 
SELECT customer_id, id, created_at,
       created_at - LAG(created_at) OVER (
         PARTITION BY customer_id ORDER BY created_at, id
       ) AS days_since_prev
FROM orders
ORDER BY customer_id, created_at;
-- Acme: NULL, 15, 21, 36. (date - date is an integer in Postgres.)

First order per customer has no predecessor → NULL, which is correct (not 0). If asked for average gap per customer, wrap this in a CTE and AVG(days_since_prev) — the NULL first rows are skipped by AVG, which is exactly right.

E3. Management chain above Dev, top-down (Priya > Arjun > Dev). 
WITH RECURSIVE chain AS (
  SELECT id, name, manager_id, 0 AS steps_up
  FROM employees WHERE name = 'Dev'
  UNION ALL
  SELECT,, m.manager_id, c.steps_up + 1
  FROM employees m
  JOIN chain c ON = c.manager_id       -- walk UP: parent of the previous row
)
SELECT name FROM chain ORDER BY steps_up DESC;
-- Priya, Arjun, Dev

Same recursive machinery as the org chart, direction reversed: seed at the leaf, join each round to its manager. ORDER BY steps_up DESC flips bottom-up traversal into a top-down display.

E4. Weekly paid revenue for Jan–Mar 2025 with zero-weeks included, plus a 0/1 flag for weeks above 300. 
SELECT gs.week::date AS week_start,
       COALESCE(SUM(o.amount), 0) AS revenue,
       CASE WHEN COALESCE(SUM(o.amount), 0) > 300 THEN 1 ELSE 0 END AS good_week
FROM generate_series(date_trunc('week', DATE '2025-01-01'),
                     DATE '2025-03-31', INTERVAL '1 week') AS gs(week)
LEFT JOIN orders o
       ON o.created_at >= gs.week
      AND o.created_at <  gs.week + INTERVAL '1 week'
      AND o.status = 'paid'
GROUP BY gs.week
ORDER BY gs.week;

Combines four tools from this page: generate_series for the calendar spine, date_trunc('week', ...) to align the series to Mondays, the ON-clause filter to keep empty weeks, and CASE for the flag. Most weeks in the range show 0 — which is the point: order data alone could never produce those rows.

Last habit to build before the interview: for every query you write, say the answer's grain out loud first — "one row per customer", "one row per month". Grain determines whether you need GROUP BY, a window, or neither, and stating it up front is the clearest fluency signal you can send in the first ten seconds.
