02 — Modern Java (8 → 21)

Lambdas, Streams, Optional, records, sealed types, pattern matching, and virtual threads — the Java that interviews actually test.

Why interviewers ask this. "Modern Java" is the fastest way to separate someone who writes Java from someone who learned it in 2010 and stopped. Java 8 (2014) rebooted the language around functional pipelines; Java 17 and 21 rebooted it again around data-oriented programming (records, sealed types, pattern matching) and lightweight concurrency (virtual threads). A senior candidate is expected to know not just the syntax but the machinery: why streams are lazy, why orElse can be a performance bug, why parallel streams can deadlock a web server, and why virtual threads change the reactive-vs-blocking calculus. Coming from 7 years of JavaScript, you already have the functional instincts — this page maps them onto Java's type system and runtime, and flags where the analogy breaks.

Java version cheat sheet T1

"What's new in Java 17?" is one of the most common warm-up questions in Java interviews — it tests whether you keep up with the platform. LTS (Long-Term Support) releases are 8, 11, 17, 21 (and 25); everything in between is a six-month feature release that most enterprises skip. Memorize this table.

VersionYearHeadline features (what to say in an interview)
8 (LTS)2014Lambdas, Stream API, Optional, default/static methods in interfaces, method references, new Date/Time API (java.time), CompletableFuture, Metaspace replaces PermGen.
9–102017–18Module system (JPMS), jshell, factory methods List.of()/Map.of(), Stream.takeWhile/dropWhile, private interface methods; 10: var local type inference, G1 parallel full GC.

A senior answer connects features to themes: Java 8 = "functional programming arrives"; 11 = "cleanup + HTTP client, first post-module LTS"; 17 = "data-oriented programming: records + sealed types model data as data"; 21 = "concurrency revolution: virtual threads make blocking code scale like async code." Interviewers love hearing the arc, not just the bullet list.

Lambdas & functional interfaces T1

The core idea: a lambda needs a target type

In JavaScript, functions are first-class values with no declared type — you pass x => x * 2 anywhere. Java is nominally typed: a lambda expression has no type of its own; it is compiled against a functional interface — any interface with exactly one abstract method (SAM = Single Abstract Method). The lambda becomes an implementation of that method. That's the whole trick.

@FunctionalInterface          // optional but recommended: compiler enforces "exactly one abstract method"
interface Discount {
    double apply(double price);          // the single abstract method (SAM)
    default Discount then(Discount next) {   // default methods don't count against SAM
        return p -> next.apply(this.apply(p));
    }
}

Discount tenPercent = price -> price * 0.9;      // lambda "targets" Discount
Discount holiday    = tenPercent.then(p -> p - 5);
System.out.println(holiday.apply(100));           // 85.0

Because the type comes from context, the same lambda text can implement different interfaces: Runnable r = () -> log(); and Callable<Void> c = () -> { log(); return null; };. Under the hood a lambda is not an anonymous inner class — it's compiled via invokedynamic, which lets the JVM decide the implementation strategy (usually a lightweight generated class, often cached for stateless lambdas). That's a good senior-level detail to drop.

You rarely define your own functional interfaces because java.util.function ships the common shapes. Learn these six cold — Stream, Optional, and CompletableFuture APIs are built out of them:

Effectively final: Java's closure rule (this WILL surprise you)

JS closures capture variables — live bindings you can mutate (let count = 0; arr.forEach(() => count++) works fine). Java lambdas capture values: a local variable used inside a lambda must be final or effectively final (never reassigned after initialization). Java copies the value into the lambda; allowing reassignment would create the illusion of shared mutable state across threads without any synchronization, so the language forbids it.

int count = 0;
list.forEach(item -> count++);   // COMPILE ERROR: variable used in lambda must be effectively final

// Idiomatic fixes:
long count2 = list.stream().filter(Objects::nonNull).count();   // 1) restructure as a stream (best)
AtomicInteger counter = new AtomicInteger();                       // 2) mutable *object* holder (the reference is final)
list.forEach(item -> counter.incrementAndGet());

Note the loophole: only the variable binding must be effectively final — the object it points to can still be mutated (final List<String> acc = new ArrayList<>(); stream.forEach(acc::add) compiles). That's legal but dangerous in parallel streams: parallelStream().forEach(acc::add) on an ArrayList is a data race. If you're accumulating, use collect(...), never side effects.

Method references — the four kinds

A method reference is shorthand for a lambda that just calls one method. Interviewers ask you to name the four kinds:

// 1. Static method:            ClassName::staticMethod
Function<String, Integer> parse = Integer::parseInt;        // s -> Integer.parseInt(s)

// 2. Instance method of a PARTICULAR object:  instance::method
PrintStream out = System.out;
Consumer<String> log = out::println;                        // s -> out.println(s)

// 3. Instance method of an ARBITRARY object of a type: ClassName::instanceMethod
//    The FIRST lambda parameter becomes the receiver — the classic gotcha.
Function<String, Integer> len = String::length;             // s -> s.length()
BiPredicate<String, String> starts = String::startsWith;    // (s, prefix) -> s.startsWith(prefix)

// 4. Constructor reference:    ClassName::new
Supplier<ArrayList<String>> listFactory = ArrayList::new;   // () -> new ArrayList<>()
Function<String, StringBuilder> sb = StringBuilder::new;    // s -> new StringBuilder(s) — picks ctor by target type

Kinds 2 and 3 look identical syntactically when the receiver is a class name — the difference is whether the receiver is bound (captured at creation) or unbound (supplied as the first argument at call time). String::length as a Function<String,Integer> is unbound; myString::length as a Supplier<Integer> is bound.

Stream API — deep dive T1

Pipeline anatomy: source → intermediate ops → terminal op

If you know arr.filter(...).map(...).reduce(...) from JS, streams feel familiar — but the execution model is fundamentally different. A JS Array.map eagerly materializes a new array at each step. A Java Stream builds a description of a pipeline and executes nothing until a terminal operation pulls data through — one element at a time, through all stages, with no intermediate collections.

List<String> names = orders.stream()                 // SOURCE (collection, array, generator, IO...)
    .filter(o -> o.total() > 100)                     // intermediate — lazy, returns a new Stream
    .map(Order::customerName)                          // intermediate — lazy
    .distinct()                                        // intermediate — stateful (needs a seen-set)
    .sorted()                                          // intermediate — stateful (must buffer everything)
    .limit(10)                                         // intermediate — short-circuiting
    .toList();                                         // TERMINAL — triggers execution

Distinguish stateless intermediate ops (map, filter, peek — process each element independently) from stateful ones (sorted, distinct — must remember prior elements; sorted buffers the entire stream). Stateful ops are where laziness and parallelism get expensive.

Proving laziness with peek

Stream<Integer> pipeline = Stream.of(1, 2, 3, 4, 5)
    .peek(n -> System.out.println("filter sees " + n))
    .filter(n -> n % 2 == 0)
    .peek(n -> System.out.println("map sees " + n))
    .map(n -> n * 10);

System.out.println("Nothing has printed yet!");   // true — pipeline is just a description

Optional<Integer> first = pipeline.findFirst();   // terminal op — NOW it runs
// Output:
// filter sees 1
// filter sees 2
// map sees 2
// => findFirst() got 20. Elements 3, 4, 5 were NEVER touched.

Two lessons: (1) execution is vertical — each element flows through the whole pipeline before the next element starts (not "filter everything, then map everything" like JS); (2) short-circuiting terminals (findFirst, anyMatch, limit) can skip work entirely. This is why Stream can represent infinite sequences.

The workhorse intermediates

record Order(String customerName, String city, double total, List<String> items) {}

// flatMap: one element -> a stream of elements, flattened (JS Array.flatMap)
List<String> allItems = orders.stream()
    .flatMap(o -> o.items().stream())      // Stream<Order> -> Stream<String>
    .distinct()
    .sorted(Comparator.naturalOrder())
    .toList();

// sorted with a Comparator built from key extractors — know this fluent style:
List<Order> ranked = orders.stream()
    .sorted(Comparator.comparingDouble(Order::total).reversed()
                       .thenComparing(Order::customerName))
    .skip(10)          // pagination: skip page*size ...
    .limit(10)         // ... take size
    .toList();

reduce — the three forms

// Form 1: reduce(BinaryOperator) -> Optional<T> (no identity — empty stream = empty Optional)
Optional<Double> max = orders.stream().map(Order::total).reduce(Double::max);

// Form 2: reduce(identity, BinaryOperator) -> T (like JS reduce with an initial value)
double grandTotal = orders.stream().map(Order::total).reduce(0.0, Double::sum);

// Form 3: reduce(identity, accumulator, combiner) -> U
// The 3-arg form exists for PARALLEL streams: accumulator folds an element into a partial
// result; combiner merges partial results from different threads. It also lets the result
// type differ from the element type without a prior map().
int totalChars = words.parallelStream()
    .reduce(0,
            (partial, word) -> partial + word.length(),   // (U, T) -> U
            Integer::sum);                                 // (U, U) -> U

The identity must be a true identity for the combiner (combine(identity, x) == x) and the operation must be associative. reduce(0, (a,b) -> a - b) gives different answers sequentially vs in parallel because subtraction isn't associative — a classic interview trap. Also: never reduce into a mutable container (reduce(new ArrayList<>(), ...)) — that's what collect is for; reduce assumes immutable partial results.

collect and the Collectors zoo

collect is mutable reduction: it creates a container (Supplier), folds elements in (accumulator), and merges containers (combiner). The Collectors class packages the common recipes:

import static java.util.stream.Collectors.*;

List<String>  cities   = orders.stream().map(Order::city).collect(toList());
Set<String>   citySet  = orders.stream().map(Order::city).collect(toSet());
String csv = orders.stream().map(Order::customerName)
                   .collect(joining(", ", "[", "]"));    // delimiter, prefix, suffix — JS Array.join++

// toMap — THE duplicate-key pitfall. This throws IllegalStateException if two
// orders share a customerName, because toMap has no idea how to merge them:
Map<String, Double> naive = orders.stream()
    .collect(toMap(Order::customerName, Order::total));           // 💥 on duplicates

// Fix: supply a merge function (BinaryOperator on VALUES):
Map<String, Double> totalByCustomer = orders.stream()
    .collect(toMap(Order::customerName, Order::total, Double::sum));

// Also: toMap throws NullPointerException on null VALUES (HashMap tolerates them, toMap doesn't).
// Need ordering or a specific map? 4-arg form: toMap(k, v, merge, LinkedHashMap::new)

// groupingBy — SQL GROUP BY in memory. Master the downstream-collector pattern:

Map<String, List<Order>> byCity = orders.stream()
    .collect(groupingBy(Order::city));                             // default downstream: toList()

Map<String, Long> countByCity = orders.stream()
    .collect(groupingBy(Order::city, counting()));                 // SELECT city, COUNT(*) GROUP BY city

Map<String, Double> avgTotalByCity = orders.stream()
    .collect(groupingBy(Order::city, averagingDouble(Order::total)));

Map<String, List<String>> customersByCity = orders.stream()
    .collect(groupingBy(Order::city,
             mapping(Order::customerName, toList())));             // transform inside each group

// Nesting goes arbitrarily deep — city -> customer -> count:
Map<String, Map<String, Long>> nested = orders.stream()
    .collect(groupingBy(Order::city, groupingBy(Order::customerName, counting())));

// partitioningBy — special-case groupingBy for a boolean predicate; ALWAYS has both keys:
Map<Boolean, List<Order>> bigVsSmall = orders.stream()
    .collect(partitioningBy(o -> o.total() > 100));
bigVsSmall.get(true);   // big orders (empty list if none — never null, unlike groupingBy)

Primitive streams: IntStream, LongStream, DoubleStream

Java generics don't work over primitives, so Stream<Integer> boxes every element — a real cost for numeric crunching. The primitive specializations avoid boxing and add numeric terminals (sum, average, max, summaryStatistics). JS has no equivalent because JS numbers are all doubles anyway.

int totalLen = words.stream().mapToInt(String::length).sum();          // no boxing

IntSummaryStatistics stats = IntStream.rangeClosed(1, 100)             // 1..100 inclusive; range() excludes end
    .summaryStatistics();
stats.getMin(); stats.getMax(); stats.getAverage(); stats.getSum();     // one pass, all stats

OptionalDouble avg = orders.stream().mapToDouble(Order::total).average();  // empty stream => empty, not NaN

// Crossing back: boxed() -> Stream<Integer>, mapToObj(...) -> Stream<T>
List<Integer> squares = IntStream.range(0, 10).map(n -> n * n).boxed().toList();

Infinite streams: iterate and generate

// iterate: seed + successor function — like an unfolding generator
List<Long> powersOfTwo = Stream.iterate(1L, n -> n * 2)
    .limit(10)                                    // REQUIRED: without a short-circuit, this never ends
    .toList();                                    // [1, 2, 4, ... 512]

// generate: stateless supplier
List<Double> samples = Stream.generate(Math::random).limit(5).toList();

// Java 9+: bounded iterate — a for-loop as a stream (seed, hasNext, next)
Stream.iterate(1, n -> n <= 1000, n -> n * 2).forEach(System.out::println);

Streams are single-use

Stream<String> s = names.stream();
s.count();
s.findFirst();   // 💥 IllegalStateException: stream has already been operated upon or closed

A stream is a one-shot iterator over a pipeline, not a reusable collection. Need to traverse twice? Re-call .stream() on the source, or collect once and stream the result. (Contrast: a JS array is data you can map over repeatedly; a Java stream is closer to a consumed generator.)

When NOT to use streams

  • Side-effect-heavy logic. If the body mutates external state, throws checked exceptions, or needs break/continue/early return across iterations, a plain loop is clearer. Checked exceptions inside lambdas are notoriously ugly (you must wrap them).
  • Hot inner loops over primitives. For tiny, performance-critical loops the stream overhead (object allocation, megamorphic call sites) can lose to an indexed for loop. Measure with JMH before claiming either way.
  • Simple 2-line iterations. for (var o : orders) total += o.total(); is fine. Streams shine when you'd otherwise write nested loops + temp collections.
  • Debugging-critical paths. Stack traces through stream frames are noisier; stepping through a loop is easier. (Mitigation: extract lambdas into named methods.)

Interview framing: "Streams are about what, loops are about how. I default to streams for transform/aggregate pipelines because they compose and parallelize, and drop to loops for stateful, effectful, or exception-heavy logic." That answer plus one concrete example (checked exceptions in lambdas) reads as senior.

Optional — the right way T1

Optional<T> is Java's answer to the billion-dollar mistake — a container that makes "may be absent" explicit in the type, so callers can't forget to handle it. Think of it as a poor man's TS T | undefined with combinators, or a one-element stream. Crucially, its designers intended it for return types only — it's an API signal, not a general-purpose null wrapper.

Creation and transformation

Optional<String> a = Optional.of("value");        // throws NPE if arg is null — asserts non-null
Optional<String> b = Optional.ofNullable(maybeNull); // null -> empty — the bridge from null-world
Optional<String> c = Optional.empty();

// map / flatMap / filter — like ?. chaining in JS/TS, but explicit:
Optional<String> cityUpper = findUser(id)                // Optional<User>
    .map(User::getAddress)                                // Optional<Address> (null result -> empty)
    .map(Address::getCity)                                // Optional<String>   ~ user?.address?.city
    .filter(city -> !city.isBlank())
    .map(String::toUpperCase);

// flatMap: when the mapper ITSELF returns Optional (avoid Optional<Optional<T>>):
Optional<Account> account = findUser(id).flatMap(User::primaryAccount);  // primaryAccount(): Optional<Account>

orElse vs orElseGet vs orElseThrow — the classic question

MethodArgumentEvaluationUse when
orElse(T)a valueALWAYS evaluated, even if the Optional is presentfallback is a cheap constant (orElse("N/A"), orElse(List.of()))
orElseGet(Supplier<T>)a supplierlazy — only invoked when emptyfallback is expensive: DB call, object construction, logging
orElseThrow(Supplier<X>)exception supplierlazyabsence is an error: orElseThrow(() -> new UserNotFoundException(id))
 
// The bug interviewers fish for: this queries the DB on EVERY call, even on cache hit,
// because Java eagerly evaluates method arguments:
User u = cache.find(id).orElse(loadFromDatabase(id));      // ❌ always hits DB
User u = cache.find(id).orElseGet(() -> loadFromDatabase(id)); // ✅ DB only on miss

// Terminal side effects (Java 9): both branches in one call
findUser(id).ifPresentOrElse(
    user -> audit.log("seen " + user.getName()),
    ()   -> audit.log("user missing: " + id));

// or(Supplier<Optional<T>>) (Java 9): chain fallback SOURCES, staying in Optional-land
Optional<User> user = cache.find(id).or(() -> repository.findById(id));

Anti-patterns (name these unprompted for senior points)

1. get() without checking. optional.get() on empty throws NoSuchElementException — you've reinvented the NPE with more steps. Modern code should basically never call bare get(); use orElseThrow() (same behavior, honest name) or a combinator.

2. isPresent() + get(). if (opt.isPresent()) { use(opt.get()); } is an imperative null-check in a fancier coat. Rewrite with map/ifPresent/orElse*. If you find yourself doing this, the Optional isn't buying you anything.

3. Optional as a field or method parameter. Fields: Optional isn't Serializable, adds a wrapper allocation per field, and can itself be null (now you have three states). Parameters: forces every caller to wrap (service.find(Optional.of(x))) — overload the method or accept null with @Nullable instead. Also don't use it in collections (List<Optional<T>> — just filter the empties out).

JPA/Spring Data returns Optional<Entity> from findById precisely because "row not found" is an expected, type-worthy outcome — you'll write repo.findById(id).orElseThrow(() -> new ResourceNotFoundException(...)) constantly in Spring controllers/services. That one-liner is the idiom to remember.

Records (Java 16) T2

A record is a nominal tuple: a transparent, immutable carrier of data where the state description is the API. If you've written type Point = { readonly x: number; readonly y: number } in TS, a record is that — plus generated equality, plus real runtime identity.

public record Point(int x, int y) {}

// The compiler generates, for free:
//  - private final fields x, y
//  - a canonical constructor  Point(int x, int y)
//  - accessors  x() and y()   (note: NOT getX() — no 'get' prefix)
//  - equals() and hashCode() based on ALL components (value semantics)
//  - toString():  "Point[x=1, y=2]"

var p1 = new Point(1, 2);
var p2 = new Point(1, 2);
p1.equals(p2);        // true — structural equality out of the box (unlike JS object literals!)
p1.x();               // 1

Coming from JS this is a bigger deal than it looks: {x:1} === {x:1} is false in JS and you reach for lodash isEqual or serialize-and-compare. Records give you correct equals/hashCode generated from the components — which means records Just Work as HashMap keys and in Sets, one of their killer use cases (e.g., a composite cache key record CacheKey(String tenant, long userId) {}).

Compact constructors: validation and normalization

public record Money(BigDecimal amount, String currency) {
    public Money {                                    // compact ctor: NO parameter list, no braces around assignments
        Objects.requireNonNull(amount, "amount");
        if (amount.signum() < 0) throw new IllegalArgumentException("negative: " + amount);
        currency = currency.toUpperCase();            // reassigning the PARAMETER normalizes it;
    }                                                 // field assignment happens automatically at the end

    // Records can have static factories, extra methods, and implement interfaces:
    public static Money usd(String amt) { return new Money(new BigDecimal(amt), "USD"); }
    public Money plus(Money other) {
        if (!currency.equals(other.currency)) throw new IllegalArgumentException("currency mismatch");
        return new Money(amount.add(other.amount), currency);   // immutable: return a new record
    }
}

Records vs Lombok

Interview take: new immutable DTOs/value objects should be records; Lombok survives for mutable JPA entities (JPA needs a no-arg constructor and mutable fields — entities cannot be records), builders for many-argument objects, and @Slf4j.

Records as DTOs

// A Spring REST request/response DTO in one line each — Jackson handles records natively:
public record CreateUserRequest(@NotBlank String name, @Email String email) {}
public record UserResponse(long id, String name, String email, Instant createdAt) {
    public static UserResponse from(User entity) {
        return new UserResponse(entity.getId(), entity.getName(), entity.getEmail(), entity.getCreatedAt());
    }
}
// Also perfect for Spring Data projections and multi-value returns from private methods.

Limitations to name in interviews

  • No inheritance: a record implicitly extends java.lang.Record, so it can't extend anything else, and records are implicitly final. They can implement interfaces — pairing beautifully with sealed interfaces (next section).
  • Shallow immutability: the references are final, not the objects. record Team(List<String> members) — callers can still team.members().add("intruder"). Defend in the compact constructor: members = List.copyOf(members);.
  • No instance fields beyond the components (static fields are fine); no setters, ever.
  • Not a fit for JPA entities (proxying, dirty tracking, no-arg ctor requirements).

Sealed classes & interfaces (Java 17) T2

Ordinary Java types are open — anyone can implement your interface. sealed closes the hierarchy: only the types in the permits clause may extend/implement it. That gives you what TS calls a discriminated union (type Shape = Circle | Rectangle | Triangle), but enforced by the compiler at definition site.

public sealed interface Shape permits Circle, Rectangle, Triangle {}

public record Circle(double radius)                 implements Shape {}
public record Rectangle(double w, double h)         implements Shape {}
public record Triangle(double base, double height)  implements Shape {}

// Each permitted subtype must itself be: final (records are), sealed (continue the tree),
// or non-sealed (deliberately reopen that branch). Same-file subtypes can omit 'permits'.

Sealed + records = algebraic data types

The payoff is exhaustive switch: because the compiler knows every possible subtype, a switch over a sealed type needs no default — and if you later add a new subtype, every switch that doesn't handle it becomes a compile error. That's the same safety you get in TS from exhaustive checks with never, but built in.

double area(Shape s) {
    return switch (s) {
        case Circle c        -> Math.PI * c.radius() * c.radius();
        case Rectangle r     -> r.w() * r.h();
        case Triangle t      -> 0.5 * t.base() * t.height();
        // no default needed — add 'record Pentagon' to permits and THIS LINE fails to compile
    };
}

Real-world modeling example — an API result that can't be misused (compare to a TS Result union):

public sealed interface PaymentResult permits PaymentResult.Success, PaymentResult.Declined, PaymentResult.Error {
    record Success(String txId, Money charged)      implements PaymentResult {}
    record Declined(String reason)                   implements PaymentResult {}
    record Error(Exception cause, boolean retryable) implements PaymentResult {}
}

String message = switch (gateway.charge(order)) {
    case PaymentResult.Success(String txId, Money m) -> "Charged " + m + " (tx " + txId + ")";
    case PaymentResult.Declined(String reason)       -> "Declined: " + reason;
    case PaymentResult.Error(var cause, var retry)    -> retry ? "Retrying..." : "Failed: " + cause.getMessage();
};

This is "data-oriented programming": model the closed set of cases as sealed + records, and put behavior in exhaustive switches instead of scattering it across polymorphic methods. Mention the trade-off: polymorphism (visitor/virtual methods) is better when the operations are stable and cases grow; sealed + switch is better when the cases are stable and operations grow — the classic "expression problem" both ways.

Pattern matching, switch expressions, text blocks, var T2

Switch expressions (Java 14)

// Old statement switch: fall-through bugs, no value, verbose breaks.
// New EXPRESSION switch: produces a value, arrow cases don't fall through, must be exhaustive.
int businessDays = switch (month) {
    case JANUARY, MARCH, MAY -> 22;                 // multiple labels per case
    case FEBRUARY -> 20;
    default -> {                                    // block bodies use 'yield' to return the value
        int d = computeDays(month);
        yield d;                                    // 'yield', not 'return' — returns from the switch
    }
};

Pattern matching for instanceof (Java 16)

// Before: test, then cast — repetitive and error-prone
if (obj instanceof String) { String s = (String) obj; use(s.length()); }

// After: test-and-bind in one step; 's' is in scope only where the match is provably true
if (obj instanceof String s && s.length() > 3) use(s.length());   // flow scoping — like TS narrowing
if (!(obj instanceof String s)) return;   // early-return style: s is in scope AFTER this line
use(s);

Pattern matching for switch + guards (Java 21)

String describe(Object obj) {
    return switch (obj) {
        case null                          -> "null";          // explicit null case (else switch on null throws NPE)
        case Integer i when i > 0          -> "positive int";  // guard: pattern + boolean condition
        case Integer i                     -> "non-positive int";
        case String s when s.isBlank()     -> "blank string";
        case String s                      -> "string of length " + s.length();
        default                            -> "something else";
    };
}
// Order matters: cases are checked top-down and the compiler rejects dominated cases
// (putting 'case Integer i' before 'case Integer i when i > 0' is a compile error).

Record deconstruction patterns (Java 21)

record Point(int x, int y) {}

var is compile-time inference like TS/Kotlin — not dynamic typing like JS var/let. The type is fixed at the initializer. Parallel streams T2

Flipping .stream() to .parallelStream() (or inserting .parallel()) splits the source with a Spliterator and processes chunks on the shared ForkJoinPool.commonPool() — a JVM-wide pool sized to availableProcessors() - 1 threads (plus the calling thread, which also participates). That one sentence contains both the power and every pitfall.

long primes = LongStream.rangeClosed(2, 5_000_000)
    .parallel()
    .filter(MyMath::isPrime)      // CPU-bound, stateless, no shared mutation — the ideal case
    .count();

When parallel helps vs hurts

HelpsHurts

The web-app trap. In a Spring Boot service, calling a parallel stream that does blocking IO (HTTP calls, JDBC) parks common-pool threads. The common pool is shared by the entire JVM — other requests' parallel streams, CompletableFuture async stages without an explicit executor, even some framework internals. A handful of slow endpoints can silently throttle the whole application. Fix: don't do IO in parallel streams; use a dedicated executor (or, on Java 21, virtual threads). The old workaround of running the stream inside a custom ForkJoinPool.submit(...) works but is undocumented behavior — mention it only as a known hack.

Senior answer template: "Parallel streams are for in-memory, CPU-bound, associative reductions over splittable sources. I benchmark before enabling them, never do IO inside them, and never mutate shared state — accumulate with collectors. For IO concurrency I reach for virtual threads or an explicit executor instead."

Virtual threads (Java 21, Project Loom) T2

The problem they solve

Classic Java servers use platform threads — thin wrappers over OS threads. Each costs ~1 MB of stack reservation and a kernel scheduling slot, so a Tomcat pool tops out around a few hundred. In thread-per-request servers, threads spend most of their life blocked on IO (waiting for the DB, downstream HTTP), so throughput is capped by thread count, not CPU. The historical escape hatch was reactive programming (WebFlux/Reactor) — high scalability at the price of infectious APIs, hard debugging, and a steep team-wide learning curve. Virtual threads deliver reactive-class scalability while keeping plain blocking code.

How they work: carriers, mounting, unmounting

A virtual thread is a Thread whose stack lives on the Java heap (starting at a few hundred bytes, growing as needed) and which is scheduled by the JVM, not the OS. The JVM maintains a small ForkJoinPool of carrier threads (≈ one per core). To run, a virtual thread mounts a carrier. The magic moment: when it hits a blocking operation (socket read, sleep, lock wait), the JDK's retrofitted blocking calls don't block the carrier — the virtual thread's stack is copied to the heap and it unmounts, freeing the carrier to run another virtual thread. When the IO completes, it's rescheduled onto any available carrier.

// Millions of concurrent "threads" — try this with platform threads and the OS gives up:
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 1_000_000).forEach(i ->
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));   // blocks the VIRTUAL thread; carrier is freed
            return i;
        }));
}   // try-with-resources: executor waits for all tasks on close()

// Other creation styles:
Thread vt = Thread.ofVirtual().name("worker-", 0).start(() -> fetchAndProcess());
Thread.startVirtualThread(() -> doWork());
vt.join();

Do not pool virtual threads. Pooling exists because platform threads are expensive; virtual threads are cheap and disposable — create one per task, always. A fixed-size pool of virtual threads reintroduces exactly the bottleneck Loom removes. If you need to limit concurrency to a fragile downstream (say, max 50 concurrent DB calls), use a Semaphore, not a pool.

The mental model vs JS async/await

You already know this model from Node: JS achieves massive IO concurrency with a tiny number of OS threads by parking suspended work (promise continuations) on the heap and multiplexing on an event loop. Virtual threads are the same idea with the syntax inverted: instead of you marking suspension points with await and living with colored functions (async infects every caller), the JVM suspends implicitly at blocking calls. Plain sequential code — loops, try/catch, JDBC — scales like async code. No function coloring, real stack traces, debuggers and profilers just work. Structured concurrency (preview) is the Promise.all analogue with automatic cancellation of siblings on failure.

Spring Boot integration

# application.properties — Spring Boot 3.2+ on Java 21:
spring.threads.virtual.enabled=true

With that flag, Tomcat handles each HTTP request on a virtual thread and @Async/task executors use virtual threads. Your controller code doesn't change — a blocking restClient.get() or JPA call now parks a virtual thread instead of hogging one of ~200 Tomcat workers, so concurrent-request capacity is no longer bounded by the thread pool. Watch the real bottlenecks that remain: the JDBC connection pool (Hikari's ~10 default connections becomes the new ceiling) and downstream rate limits.

When virtual threads DON'T help

  • CPU-bound work. Suspension only happens at blocking points; a number-crunching task never unmounts. You still have ~N cores of compute — use platform threads / parallel streams and cap concurrency at core count.
  • Latency of a single request. They improve throughput under concurrency, not the speed of one call.
  • In-memory work with no IO. Nothing to unmount for; overhead without benefit.

The pinning problem (interview favorite)

A virtual thread is pinned when it blocks but cannot unmount, holding its carrier hostage. On Java 21 the two causes: blocking inside a synchronized block/method, and blocking inside native code (JNI). If many virtual threads pin simultaneously, carriers run out and throughput collapses — the failure mode looks like a mysterious stall.

// Java 21: PINS the carrier — synchronized + blocking IO inside
synchronized (lock) {
    var data = jdbcTemplate.query(...);   // carrier stuck until IO completes
}

// Fix: ReentrantLock unmounts cleanly while waiting
private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try   { var data = jdbcTemplate.query(...); }
finally { lock.unlock(); }

// Detect pinning:  java -Djdk.tracePinnedThreads=full ...  (or JFR jdk.VirtualThreadPinned events)

Java 24 (JEP 491) fixed the synchronized case — virtual threads can now unmount inside synchronized blocks — but interviewers still ask about pinning constantly, both because many shops run 21 and because it tests whether you understand the mounting mechanism. Answer with: what pinning is, the two 21-era causes, ReentrantLock as the fix, the tracing flag, and "improved in Java 24."

Interview Q&A

What is a functional interface, and what does @FunctionalInterface actually do?

An interface with exactly one abstract method (SAM); default, static, and public Object-signature methods don't count. It's the target type a lambda or method reference compiles against. @FunctionalInterface is purely a compile-time contract check — the interface works as a lambda target without it, but the annotation makes the compiler fail if someone later adds a second abstract method, protecting API consumers. Examples: Runnable, Comparator (its second "abstract-looking" method equals comes from Object, so it still qualifies), and the java.util.function family.

Why must variables captured by a lambda be effectively final? How is this different from JavaScript closures?

Java lambdas capture values, not variable bindings: the local's value is copied into the lambda object because locals live on the stack frame, which may be gone (or on another thread) when the lambda runs. Allowing reassignment would fake shared mutable state with no synchronization, so the compiler requires the variable to be final or effectively final (never reassigned). JS closures capture the live binding — mutating let count inside a callback works because the variable is hoisted into a heap-allocated environment record. Java workarounds: restructure as a stream reduction (preferred), or capture a final reference to a mutable holder (AtomicInteger, single-element array) — noting that the holder approach is unsafe in parallel contexts unless the holder is thread-safe.

Name the four kinds of method references with an example of each.

(1) Static: Integer::parseInt ≡ s -> Integer.parseInt(s). (2) Bound instance (particular object): System.out::println ≡ s -> System.out.println(s). (3) Unbound instance (arbitrary object of a type): String::length as Function<String,Integer> — the first parameter becomes the receiver, so String::compareToIgnoreCase as a Comparator<String> means (a,b) -> a.compareToIgnoreCase(b). (4) Constructor: ArrayList::new; with a Function<Integer,ArrayList> target it picks the capacity constructor — the target type selects the overload. The subtle one is 3: syntactically identical to 1 but dispatches on the first argument.

Explain stream laziness. Why does it matter?

Intermediate operations (map, filter, sorted…) build a pipeline description and do zero work; only a terminal operation (collect, forEach, count, findFirst…) pulls elements through. Elements flow vertically — each element traverses the whole pipeline before the next starts — rather than each stage materializing a full intermediate collection like JS Array.map. Consequences: (1) short-circuiting — filter(...).findFirst() may examine two elements out of a million; (2) infinite streams are representable (Stream.iterate + limit); (3) no intermediate allocations; (4) a fused pipeline can be parallelized as a unit. Provable with peek: build a pipeline with peeks, observe nothing prints until the terminal op runs, and that skipped elements never print.

map vs flatMap — on Stream and on Optional?

map is one-to-one: each element becomes exactly one output element. flatMap is one-to-many-then-flatten: the mapper returns a Stream, and the results are concatenated — orders.stream().flatMap(o -> o.items().stream()) turns Stream<Order> into Stream<String> of all items, exactly like JS Array.flatMap. On Optional, use flatMap when the mapping function itself returns an Optional, to avoid Optional<Optional<T>>: findUser(id).flatMap(User::primaryAccount). Rule of thumb: mapper returns a wrapped/plural thing → flatMap; returns a plain value → map.

What are the three forms of reduce, and when do you need the third?

(1) reduce(BinaryOperator<T>) → Optional<T>, no identity, empty stream yields empty. (2) reduce(T identity, BinaryOperator<T>) → T, like JS reduce with an initial value. (3) reduce(U identity, BiFunction<U,T,U> accumulator, BinaryOperator<U> combiner) → U: needed when the result type differs from the element type and for parallel execution, where per-thread partial results (built by the accumulator) must be merged by the combiner. Constraints interviewers probe: the operation must be associative, the identity must be a true identity, and the functions must be stateless/non-interfering — reduce(0, (a,b) -> a - b) breaks in parallel because subtraction isn't associative. For mutable containers use collect, not reduce.

What goes wrong with Collectors.toMap and how do you fix it?

Two traps. (1) Duplicate keys: the 2-arg toMap(keyFn, valueFn) throws IllegalStateException when two elements map to the same key — supply a merge function: toMap(Order::customer, Order::total, Double::sum) (the merge function is a BinaryOperator over values). (2) Null values: toMap throws NPE if the value function returns null (its internal Map.merge rejects nulls), even though a plain HashMap accepts null values — filter nulls first or collect differently. Bonus: the 4-arg overload lets you choose the map implementation, e.g. LinkedHashMap::new to preserve encounter order or TreeMap::new for sorted keys.

Explain groupingBy with downstream collectors. How would you compute average order total per city?

groupingBy(classifier) buckets elements into a Map<K, List<T>>. The second argument is a downstream collector applied within each bucket, replacing the default toList(): groupingBy(Order::city, averagingDouble(Order::total)) gives Map<String, Double> — the in-memory SELECT city, AVG(total) ... GROUP BY city. Other downstreams: counting(), summingDouble, mapping(fn, toList()) to transform bucket elements, maxBy(comparator), and nested groupingBy for two-level grouping. Related: partitioningBy(predicate) for boolean splits — it always contains both true and false keys (possibly empty lists), whereas groupingBy omits absent keys.

Why do IntStream/LongStream/DoubleStream exist? What do mapToInt and boxed do?

Java generics only work over reference types, so Stream<Integer> boxes every element — an object allocation and pointer chase per number, plus GC pressure. The primitive specializations store raw primitives and add numeric terminals: sum(), average() (returning OptionalDouble), max(), summaryStatistics() (min/max/sum/count/avg in one pass), and factories like range/rangeClosed. mapToInt(String::length) crosses from object stream to IntStream; boxed() or mapToObj crosses back. In an interview, computing "total length of all names" with mapToInt(...).sum() instead of map(...).reduce(0, Integer::sum) signals you know the boxing cost.

Difference between orElse and orElseGet — why does it matter in practice?

orElse(value) takes an already-computed value; because Java evaluates method arguments eagerly, the fallback expression runs even when the Optional is present. orElseGet(supplier) takes a Supplier invoked only on empty. With constants it's stylistic; with expensive fallbacks it's a real bug: cache.find(id).orElse(loadFromDb(id)) hits the database on every single call, cache hit or not. Rule: constant → orElse; anything involving a method call, allocation, or side effect → orElseGet. And when absence is exceptional, orElseThrow(() -> new NotFoundException(id)) documents intent — the standard Spring service-layer idiom.

What are the main Optional anti-patterns?

(1) Bare get() — throws NoSuchElementException on empty; you've recreated the NPE. Prefer orElseThrow() or combinators. (2) isPresent() + get() — an imperative null check wearing a costume; rewrite with map/filter/ifPresent/orElse*. (3) Optional fields — not serializable, extra allocation per instance, and the field itself can be null (three states). (4) Optional method parameters — forces callers to wrap; use overloads or @Nullable. (5) Optional in collections — filter instead. (6) Wrapping just to unwrap: Optional.ofNullable(x).orElse(y) is a verbose ternary. The design intent (per its authors) is narrow: a return type expressing "no result is a legitimate outcome."

What exactly does the compiler generate for a record? What are records' limitations?

For record Point(int x, int y): private final fields; a canonical constructor; component accessors named x()/y() (no get prefix); equals/hashCode over all components (value semantics); and toString. Records implicitly extend java.lang.Record and are final. You can add static members, instance methods, compact constructors for validation/normalization, and implement interfaces. Limitations: no superclass, no inheritance between records, no extra instance fields, no setters, and immutability is shallow — a List component is still mutable unless you defensively copy (members = List.copyOf(members) in the compact constructor). Not suitable for JPA entities (no no-arg ctor, no mutability for dirty tracking, proxying issues) — records are for DTOs, value objects, map keys, and multi-value returns.

Records vs Lombok — would you migrate?

Records are a language feature: guaranteed semantics, understood by javac, IDEs, reflection (getRecordComponents()), and frameworks — Jackson and Spring bind records without annotations. Lombok is an annotation processor hooking into compiler internals: powerful but an extra dependency with occasional JDK-upgrade breakage. Migration policy I'd state: all new immutable carriers become records; keep Lombok where records can't go — mutable JPA entities (@Getter/@Setter), @Builder for constructors with many optional arguments (records lack named/default arguments), and @Slf4j. Also note record accessors are name() not getName(), which matters for reflective tools hardcoding Bean conventions.

What problem do sealed interfaces solve? How do they interact with switch?

They close a type hierarchy: sealed interface Shape permits Circle, Rectangle means the compiler knows the complete set of subtypes (each of which must be final, sealed, or explicitly non-sealed). Combined with records you get algebraic data types — TS discriminated unions with compile-time teeth. The payoff is exhaustiveness: a switch pattern-matching over a sealed type needs no default, and adding a new permitted subtype turns every non-exhaustive switch into a compile error, so the compiler finds all the code that needs updating. Design guidance: use sealed+switch when the set of cases is stable and operations grow (results, events, AST nodes); use plain polymorphism when operations are stable and cases grow — the two sides of the expression problem.

Walk through the evolution of pattern matching in Java (16 → 21).

Java 16: pattern matching for instanceof — if (o instanceof String s) tests and binds in one step, with flow scoping (like TS narrowing). Java 14 (prerequisite): switch expressions with arrow cases, multi-label cases, and yield for block bodies. Java 21 (final): patterns in switch — case String s ->, guards via when (case Integer i when i > 0 ->), an explicit case null (otherwise switching on null throws NPE), dominance checking (unreachable cases are compile errors), and exhaustiveness over sealed hierarchies. Also 21: record deconstruction patterns — case Line(Point(var x1, var y1), Point(var x2, var y2)) destructures nested components with static type checking. Together: data-oriented programming — sealed types define the shape of data, switches consume it exhaustively.

How do parallel streams execute under the hood, and when would you avoid them?

A parallel stream splits its source recursively via its Spliterator into subtasks executed on the shared ForkJoinPool.commonPool() (sized ~cores−1, plus the invoking thread), then combines partial results. Use only when: data is large or per-element work is expensive (N×Q heuristic), functions are stateless and side-effect-free, the operation is associative, and the source splits well (ArrayList, arrays, ranges — not LinkedList or iterator-based sources). Avoid when: small data (overhead dominates); IO-bound work — a blocked common-pool thread starves every other parallel stream and default async CompletableFuture in the JVM, which in a web app is a global slowdown; ordered short-circuit ops (limit, findFirst) that force coordination; or any shared mutable accumulation (forEach(list::add) is a data race — use collect). And always benchmark; parallel is frequently slower.

What is a virtual thread and how does it differ from a platform thread?

A platform thread wraps an OS thread 1:1 — ~1 MB reserved stack, kernel-scheduled, realistically capped at a few thousand per JVM. A virtual thread is a JVM-managed Thread whose resizable stack lives on the heap (hundreds of bytes to start) and which is multiplexed over a small pool of carrier platform threads (~one per core). When a virtual thread performs a blocking operation, the JDK parks it: its stack is saved to the heap, it unmounts, and the carrier immediately runs another virtual thread; on IO completion it remounts on any carrier. Result: millions of concurrent threads, so thread-per-request scales like async without changing the programming model. Same Thread API, ~same per-task semantics; differences to note: virtual threads are always daemons, have no meaningful priority, and should never be pooled — create one per task via Executors.newVirtualThreadPerTaskExecutor().

What is virtual-thread pinning? How do you detect and fix it?

Pinning is when a blocked virtual thread cannot unmount, so it holds its carrier hostage — enough simultaneous pinning exhausts the carrier pool and the app stalls. On Java 21 the causes are blocking while inside a synchronized block/method (the JVM couldn't safely unmount while holding a monitor) and blocking in native/JNI frames. Fixes: replace synchronized around blocking calls with java.util.concurrent.locks.ReentrantLock (which parks cooperatively), or narrow synchronized sections to exclude IO; audit libraries (older connection pools and drivers synchronized internally — modern versions have largely fixed this). Detection: -Djdk.tracePinnedThreads=full prints stack traces on pin events; JFR emits jdk.VirtualThreadPinned. Bonus point: JEP 491 in Java 24 made synchronized non-pinning, but you should still know the mechanism and the ReentrantLock story for 21-era codebases.

Virtual threads vs reactive WebFlux — how do you choose in 2026?

Both solve "don't waste a scarce OS thread while waiting on IO." Reactive solves it in user code: non-blocking operators, callbacks composed as Mono/Flux — scalable, with first-class backpressure and streaming, but infectious (every layer must be reactive, one blocking call poisons the event loop), hard to debug, and expensive for teams. Virtual threads solve it in the runtime: blocking calls suspend cheap threads, so plain MVC + JDBC/JPA code scales comparably for request/response workloads with zero model change (spring.threads.virtual.enabled=true). My default: MVC + virtual threads for typical microservices; reactive where its unique strengths matter — true streaming responses, backpressure-controlled pipelines, event-stream composition. Also mention what virtual threads don't give: they don't help CPU-bound work, don't provide backpressure (bound concurrency with a Semaphore), and the DB connection pool becomes the next bottleneck.

How do virtual threads compare to async/await in JavaScript?

Same destination, opposite syntax. Node multiplexes thousands of concurrent IO operations over one event-loop thread; suspended work lives on the heap as promise continuations, and you mark suspension points explicitly with await — leading to "colored functions": async infects every caller, and you maintain two worlds (sync/async). Virtual threads keep the sequential model: the JVM detects blocking calls and suspends the thread implicitly — the "await" is invisible and everywhere. Advantages over async/await: no function coloring, real stack traces, ordinary try/catch and loops, existing blocking libraries (JDBC) just work, debuggers/profilers see normal threads. And unlike Node's single-threaded loop, carriers run virtual threads on all cores in parallel, so you also get CPU parallelism. CompletableFuture remains Java's Promise for explicit composition, but with Loom you need it far less.

Why can't you reuse a Stream? How would you "reuse" one?

A stream is a one-shot traversal of a pipeline bound to a source — closer to a consumed iterator/generator than to a collection. After a terminal operation it's closed; a second terminal op throws IllegalStateException("stream has already been operated upon or closed"). This is by design: streams may wrap non-replayable sources (file lines, socket data, infinite generators), so replay can't be guaranteed. To traverse twice: call .stream() on the underlying collection again (cheap — streams are lightweight views), collect intermediate results into a List and stream that, or hold a Supplier<Stream<T>> that builds a fresh pipeline per use.

What's new in Java 17? In Java 21? (the rapid-fire version question)

17 (LTS, 2021): sealed classes/interfaces (final); pattern matching for switch as preview; strong encapsulation of JDK internals (illegal-access to internals now fails); enhanced pseudo-random generators; plus everything accumulated since 11 that people associate with it — records (16), instanceof patterns (16), text blocks (15), switch expressions (14), helpful NullPointerException messages (14). 21 (LTS, 2023): virtual threads final; pattern matching for switch final with when guards and case null; record deconstruction patterns; sequenced collections (SequencedCollection, getFirst()/getLast()/reversed()); generational ZGC; previews: string templates, structured concurrency, scoped values. Frame it thematically: 17 = data-oriented programming matures; 21 = lightweight concurrency arrives.

Where is var allowed, and what's your style policy for it?

Allowed: local variables with initializers, for/for-each loop variables, try-with-resources. Not allowed: fields, method parameters, return types, var x = null (no type to infer), and var f = () -> ... (a lambda needs an explicit target interface). It's compile-time inference — the type is fixed at the initializer; nothing dynamic. Style (aligned with OpenJDK's guidelines): use it when the type is evident from the RHS (var users = new ArrayList<User>(), var response = client.send(...)) or when it removes generic noise (Map.Entry iteration); avoid when the RHS is opaque (var result = svc.process(x)). One gotcha: var list = new ArrayList<String>() infers the concrete ArrayList, not the List interface, which slightly weakens program-to-interface style.

How does Stream.toList() differ from collect(Collectors.toList())?

Stream.toList() (Java 16) returns an unmodifiable list (mutators throw UnsupportedOperationException; it does permit null elements, unlike List.of). collect(Collectors.toList()) returns a list with no documented mutability guarantee — in practice an ArrayList, and lots of code (incorrectly, per the spec) relies on mutating it. So mechanically swapping one for the other can break code that adds to the result later. Prefer toList() for new code — shorter and immutability-by-default; use collect(toCollection(ArrayList::new)) when you genuinely need a mutable result. This "safe refactor that isn't" question is a favorite senior filter.

When would you choose a plain for-loop over a stream?

(1) Checked exceptions: lambdas can't throw them past the functional interface, forcing ugly wrap/rethrow boilerplate. (2) Complex control flow: early exit with a value, break/continue across nested levels, mutating multiple accumulators. (3) Performance-critical primitive loops: stream setup and (potentially megamorphic) lambda dispatch can lose to an indexed loop in hot paths — though JIT often closes the gap; measure with JMH. (4) Deep debuggability requirements. (5) Trivial iterations where a stream adds ceremony without clarity. The senior framing: streams are declarative data-transformation pipelines — use them when the code reads as "what," not as a contorted "how"; readability and correctness of the team's code beat dogma in either direction.

Hands-on exercises

Exercise 1: Order analytics with collectors

Given record Order(String customer, String city, double total, List<String> items) and a List<Order> orders, compute in a single class: (a) total revenue per city, sorted by revenue descending; (b) the distinct sorted list of all items across all orders; (c) each customer's order count, but only customers with 2+ orders; (d) a partition of orders into ≥ $100 and < $100.

Solution 
import static java.util.stream.Collectors.*;

// (a) revenue per city, sorted desc — group, then re-stream the entrySet to sort:
Map<String, Double> revenueByCity = orders.stream()
    .collect(groupingBy(Order::city, summingDouble(Order::total)))
    .entrySet().stream()
    .sorted(Map.Entry.<String, Double>comparingByValue().reversed())
    .collect(toMap(Map.Entry::getKey, Map.Entry::getValue,
                   (a, b) -> a, LinkedHashMap::new));   // LinkedHashMap preserves sort order

// (b) distinct sorted items — flatMap flattens the nested lists:
List<String> allItems = orders.stream()
    .flatMap(o -> o.items().stream())
    .distinct().sorted().toList();

// (c) repeat customers — group/count, then filter the map's entries:
Map<String, Long> repeatCustomers = orders.stream()
    .collect(groupingBy(Order::customer, counting()))
    .entrySet().stream()
    .filter(e -> e.getValue() >= 2)
    .collect(toMap(Map.Entry::getKey, Map.Entry::getValue));

// (d) partition:
Map<Boolean, List<Order>> byValue = orders.stream()
    .collect(partitioningBy(o -> o.total() >= 100));

Key takeaways: downstream collectors do the aggregation; sorting a map means streaming its entries into a LinkedHashMap; the 4-arg toMap merge function is required even when duplicates are impossible, because the overload demands it.

Exercise 2: Prove laziness and short-circuiting

Using Stream.iterate and peek, build an infinite stream of integers, and find the first number greater than 1000 that is divisible by 37. Print how many elements were actually examined. Explain why this terminates.

Solution 
AtomicInteger examined = new AtomicInteger();

int answer = Stream.iterate(1001, n -> n + 1)          // infinite source
    .peek(n -> examined.incrementAndGet())
    .filter(n -> n % 37 == 0)
    .findFirst()                                          // short-circuiting terminal
    .orElseThrow();

System.out.println(answer + " after examining " + examined.get());  // 1036 after examining 36

It terminates because streams are pull-based: findFirst requests elements one at a time, and stops requesting the moment the filter passes one through. Only 36 elements ever exist. The same pipeline with a JS-style eager model would hang forever materializing the source. (The AtomicInteger is the effectively-final-capture workaround; in production, peek counting is for demos/debugging only.)

Exercise 3: Refactor null-riddled code with Optional

Refactor this to a single Optional chain with no if-statements: given a userId, find the user; if the user has an address and the address has a non-blank city, return the city uppercased; otherwise return "UNKNOWN". findUser returns Optional<User>; getAddress() may return null.

Solution 
String city = findUser(userId)
    .map(User::getAddress)                 // null address -> empty (map wraps with ofNullable semantics)
    .map(Address::getCity)
    .filter(c -> !c.isBlank())
    .map(String::toUpperCase)
    .orElse("UNKNOWN");                    // constant fallback: orElse is fine here — no laziness needed

Discussion points: Optional.map auto-wraps a null mapper result into empty, which is what makes the chain null-safe; if getAddress() returned Optional<Address> you'd use flatMap instead; and if the fallback were computed (say, a geo-IP lookup) you'd switch to orElseGet.

Exercise 4: Model a domain with sealed + records + exhaustive switch

Model a payment event stream: an event is either Authorized(String txId, Money amount), Captured(String txId), Refunded(String txId, Money amount, String reason), or Failed(String txId, String error). Write a method String summarize(PaymentEvent e) using record deconstruction and no default branch, and explain what happens at compile time when a Disputed event is added later.

Solution 
public sealed interface PaymentEvent
        permits Authorized, Captured, Refunded, Failed {}

public record Authorized(String txId, Money amount)                implements PaymentEvent {}
public record Captured(String txId)                                implements PaymentEvent {}
public record Refunded(String txId, Money amount, String reason)  implements PaymentEvent {}
public record Failed(String txId, String error)                    implements PaymentEvent {}

String summarize(PaymentEvent e) {
    return switch (e) {
        case Authorized(var tx, var amt)          -> "AUTH " + tx + " for " + amt;
        case Captured(var tx)                     -> "CAPTURED " + tx;
        case Refunded(var tx, var amt, var why)  -> "REFUND " + amt + " on " + tx + " (" + why + ")";
        case Failed(var tx, var err)              -> "FAILED " + tx + ": " + err;
    };  // exhaustive over the sealed hierarchy — no default
}

Adding record Disputed(...) implements PaymentEvent (and to permits) makes this switch stop compiling — "the switch expression does not cover all possible input values." Every consumer is forced to handle the new case; with a default branch the new event would have been silently mis-handled at runtime. That "make the compiler find all affected code" story is exactly what the interviewer wants.

Exercise 5: Virtual threads vs platform pool benchmark

Write a program that runs 10,000 tasks, each simulating a 100 ms blocking IO call (Thread.sleep), twice: once on Executors.newFixedThreadPool(200) and once on Executors.newVirtualThreadPerTaskExecutor(). Time both, predict the results before running, and explain them in terms of carriers and unmounting.

Solution 
long time(ExecutorService executor) throws Exception {
    long start = System.nanoTime();
    try (executor) {                              // ExecutorService is AutoCloseable since 19
        var futures = IntStream.range(0, 10_000)
            .mapToObj(i -> executor.submit(() -> {
                Thread.sleep(100);                // simulated blocking IO
                return i;
            }))
            .toList();
        for (var f : futures) f.get();
    }
    return (System.nanoTime() - start) / 1_000_000;
}

System.out.println("fixed(200): " + time(Executors.newFixedThreadPool(200)) + " ms");
System.out.println("virtual:    " + time(Executors.newVirtualThreadPerTaskExecutor()) + " ms");

Expected: fixed pool ≈ 10,000 / 200 waves × 100 ms ≈ 5,000 ms — throughput is capped by thread count because each sleeping platform thread occupies an OS thread doing nothing. Virtual: ≈ 100–300 ms — all 10,000 virtual threads start immediately; each sleep unmounts its carrier, so a handful of carriers (~core count) shepherd all 10,000 concurrent waits. The experiment demonstrates that with virtual threads, blocked-time no longer consumes a scarce resource. Follow-up to discuss: replace sleep with a CPU-bound loop and virtual threads lose their advantage entirely — nothing unmounts.

Exercise 6: FizzBuzz, three ways (idiom fluency drill)

Implement FizzBuzz for 1–30 three ways: (a) an IntStream with mapToObj; (b) using a switch expression on the pair of remainders with record deconstruction; (c) explain which you'd write in production and why.

Solution 
// (a) Stream idiom:
IntStream.rangeClosed(1, 30)
    .mapToObj(i -> i % 15 == 0 ? "FizzBuzz" : i % 3 == 0 ? "Fizz" : i % 5 == 0 ? "Buzz" : String.valueOf(i))
    .forEach(System.out::println);

// (b) Pattern-matching idiom — deconstruct a record of remainders:
record Mods(int three, int five) {}

IntStream.rangeClosed(1, 30).forEach(i -> {
    String out = switch (new Mods(i % 3, i % 5)) {
        case Mods(var t, var f) when t == 0 && f == 0 -> "FizzBuzz";
        case Mods(var t, var f) when t == 0           -> "Fizz";
        case Mods(var t, var f) when f == 0           -> "Buzz";
        case Mods m                                     -> String.valueOf(i);
    };
    System.out.println(out);
});

(c) In production: version (a) — it's the shortest honest expression of the logic. Version (b) exists to show mechanics (guards, dominance ordering — the guarded cases must precede the unguarded case Mods m, and the unguarded total pattern makes the switch exhaustive without default). Being able to say "I know the fancy way and I know when it's overkill" is itself the senior signal.
