The object model, equality, immutability, generics — and how HashMap actually works under the hood

The equals()/hashCode() contract T1

Every Java object inherits equals(Object) and hashCode() from java.lang.Object. The default equals is reference identity (the same as JavaScript's === on objects), and the default hashCode is an identity-based number the JVM generates. The moment you want value equality — two Money(10, "USD") objects being "the same" — you must override both. Not one. Both.

The contract, straight from the Javadoc, is what interviewers want you to recite and explain:

  • equals must be reflexive (x.equals(x)), symmetric (x.equals(y) ⇔ y.equals(x)), transitive, consistent (repeated calls agree if nothing mutated), and x.equals(null) is false.
  • hashCode must be consistent across calls, and — the critical clause — if a.equals(b) then a.hashCode() == b.hashCode(). The reverse is not required: unequal objects may share a hash (that's just a collision).

Why they must move together: HashMap's two-phase lookup

HashMap.get(key) works in two phases: (1) use hashCode() to jump to a bucket, (2) walk the bucket comparing with equals(). If you override equals but not hashCode, two "equal" objects land in different buckets, so the map never even reaches the equals comparison:

class Point {          // BROKEN: equals overridden, hashCode not
    final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override public boolean equals(Object o) {
        return o instanceof Point p && p.x == x && p.y == y;
    }
    // hashCode() inherited from Object → identity-based!
}

Map<Point, String> map = new HashMap<>();
map.put(new Point(1, 2), "home");
map.get(new Point(1, 2));   // null! Different identity hash → wrong bucket.

The symmetric failure — overriding hashCode but not equals — means equal-valued objects collide into the same bucket but equals (identity) says they're different, so a HashSet happily stores "duplicates".

The canonical correct implementation

import java.util.Objects;

public final class Money {
    private final long amountMinor;   // cents — never double for money
    private final String currency;

    public Money(long amountMinor, String currency) {
        this.amountMinor = amountMinor;
        this.currency = Objects.requireNonNull(currency);
    }

    @Override public boolean equals(Object o) {
        if (this == o) return true;                 // fast path: identity
        if (!(o instanceof Money m)) return false; // pattern matching (16+); also rejects null
        return amountMinor == m.amountMinor && currency.equals(m.currency);
    }

    @Override public int hashCode() {
        return Objects.hash(amountMinor, currency);
    }
}

Objects.hash(a, b, c) is the readable one-liner: it boxes the args into an array and computes 31 * result + element repeatedly (31 because it's an odd prime and 31 * x compiles to (x << 5) - x). For hot paths, hand-roll the same formula without the varargs array allocation. Records (record Money(long amountMinor, String currency) {}) generate all of this for you — the modern default for value carriers, similar in spirit to comparing plain data objects structurally in TS.

Mutable fields in equals/hashCode + used as a HashMap key = data loss. If a key's hash-relevant field mutates after insertion, the entry still sits in the bucket computed from the old hash. get with the updated key computes the new hash, looks in a different bucket, finds nothing. Even iterating and comparing won't rescue you through the map API. Rule: keys must be immutable, or at minimum their equals/hashCode fields must be.

Two senior-level flourishes: (1) instanceof vs getClass() in equals — instanceof permits subclass equality but makes it nearly impossible to keep symmetry if a subclass adds state (the classic Bloch Effective Java Item 10 discussion; composition beats inheritance for value types). (2) compareTo should be consistent with equals, otherwise a TreeMap (which uses only compareTo) and a HashMap (which uses equals/hashCode) will disagree about whether two keys are duplicates — BigDecimal famously has this problem: new BigDecimal("1.0") and new BigDecimal("1.00") are compareTo-equal but not equals-equal.

String immutability, the pool, and builders T1

String is a final, immutable class. Every "mutation" (concat, replace, toUpperCase) returns a new object. Why the JDK committed to this:

  • Security & sanity: strings are used as class names, file paths, JDBC URLs, map keys. If callers could mutate them after validation, every security check would be a TOCTOU bug.
  • Hash caching: String caches its hashCode in a field after first computation — legal only because the content can't change. This is why strings are the best-behaved HashMap keys.
  • Thread safety for free: immutable objects can be shared across threads with no synchronization.
  • The string pool exists because of immutability: the JVM can safely share one instance among all code that uses the same literal.

The string pool and == vs equals

String a = "hello";                 // literal → interned in the string pool
String b = "hello";                 // same pooled instance
String c = new String("hello");    // forces a fresh heap object
String d = c.intern();              // returns the pooled instance

a == b;        // true  — same pool object
a == c;        // false — different objects
a.equals(c);   // true  — same content
a == d;        // true  — intern() returned the pooled one

Coming from JS this is the big trap: JS strings are primitives, so ==/=== compares content. In Java, == on any reference type compares references. Always use equals() for strings (or Objects.equals(a, b) when either might be null). The pool (held in the heap since Java 7, a native hashtable of interned strings) is an implementation detail that makes == sometimes work — which is worse than never working, because tests pass with literals and production breaks with runtime-built strings.

Compile-time constant expressions are folded and pooled: "he" + "llo" == "hello" is true. But s1 + s2 with variables is computed at runtime → new object → == is false.

Both define types you can't instantiate; the interview question is really "when do you reach for which, and what changed in Java 8?"

Reach for an abstract class when subclasses genuinely share state and protected helper machinery — e.g. Spring's template classes, or the JDK's AbstractList skeleton implementations.

default and static methods

Java 8 added default methods so interfaces could evolve without breaking every implementor on the planet — the motivating case was adding forEach, stream(), and sort to collections interfaces. TS analogy: it's like an interface that ships a mixin implementation.

public interface Auditable {
    Instant createdAt();                              // abstract

    default boolean olderThan(Duration d) {           // inherited behavior
        return createdAt().isBefore(Instant.now().minus(d));
    }

    static Auditable of(Instant t) {                  // NOT inherited; call as Auditable.of(...)
        return () -> t;
    }
}

The diamond problem — and how Java resolves it

Multiple inheritance of behavior re-introduces the diamond. Java's resolution rules, in priority order:

  1. Class always wins. A concrete/abstract method in the class hierarchy beats any interface default.
  2. More specific interface wins. If B extends A and both have the default, B's is chosen.
  3. Otherwise: compile error. Two unrelated interfaces with the same default force you to override and disambiguate.
interface Swimmer { default String move() { return "swim"; } }
interface Flyer   { default String move() { return "fly"; } }

class Duck implements Swimmer, Flyer {
    // Won't compile without this override:
    @Override public String move() {
        return Swimmer.super.move();   // explicit delegation syntax
    }
}

Interfaces still can't hold instance state, so Java never has the C++ "which copy of the base field?" diamond — only the behavioral one, which the compiler refuses to guess about. That's the one-line answer interviewers want.

final, static, access modifiers & pass-by-value T1

final

  • final variable: assign exactly once. For references it means the reference can't be reseated — the object it points to can still mutate (final List can still add). Exactly like JS const.
  • final method: can't be overridden.
  • final class: can't be extended (String, Integer, all records).

static

static members belong to the class, not instances — one copy per classloader (a subtlety worth dropping: two webapps in one container can each have their own "singleton"). Static methods can't touch instance state and aren't polymorphic — they're resolved at compile time by the declared type, which is why "overriding" a static method is actually hiding.

Java is pass-by-value. Always. No exceptions.

This is the most persistently botched interview question. The confusion: "but when I pass an object and mutate it, the caller sees the change!" Correct — because what got copied was the reference. Java passes a copy of the variable: for primitives, a copy of the value; for objects, a copy of the reference (a pointer value). You can follow the copied pointer and mutate the shared object, but you can never reseat the caller's variable. This is precisely JavaScript's semantics too ("call by sharing") — so your instinct is right; just say "pass-by-value where the value may be a reference".

static void mutate(List<String> list) {
    list.add("added");            // follows the copied reference → caller SEES this
}
static void reassign(List<String> list) {
    list = new ArrayList<>();     // reseats the LOCAL copy → caller sees NOTHING
    list.add("lost");
}
static void increment(int n) {
    n++;                          // local copy of the primitive → caller sees NOTHING
}

List<String> mine = new ArrayList<>();
mutate(mine);     // mine = ["added"]
reassign(mine);   // mine = ["added"]  — unchanged
// Mental picture (stack holds variables; heap holds objects):
//
//  caller's frame          heap                 mutate()'s frame
//  ┌────────────┐      ┌─────────────┐         ┌────────────┐
//  │ mine ──────┼─────▶│ ArrayList   │◀────────┼── list     │  (a COPY of the ref)
//  └────────────┘      │ ["added"]   │         └────────────┘
//                      └─────────────┘
//  reassign() pointed ITS copy elsewhere; caller's arrow never moved.

The classic proof question: "write swap(a, b) for two objects in Java." You can't — there is no way to reseat the caller's variables. If a candidate claims Java is pass-by-reference, this is the counterexample. (In a true pass-by-reference language like C++ with &, swap works.)

Designing an immutable class T2

An immutable object's observable state never changes after construction. The recipe (Bloch, Item 17):

  1. final class — or all constructors private + static factories — so a subclass can't add mutable behavior and masquerade as your type.
  2. All fields private final.
  3. No setters, no methods that mutate state; "withers" return new instances.
  4. Defensive copies IN for mutable constructor args (collections, Date, arrays).
  5. Defensive copies OUT (or unmodifiable views / immutable collections) so getters can't leak a mutable handle.
public final class Order {
    private final String id;
    private final List<String> items;   // mutable type — must be guarded
    private final Instant placedAt;      // java.time types are already immutable

    public Order(String id, List<String> items, Instant placedAt) {
        this.id = Objects.requireNonNull(id);
        this.items = List.copyOf(items);   // defensive copy IN (truly immutable, null-hostile)
        this.placedAt = placedAt;
    }

    public List<String> items() { return items; }  // safe: List.copyOf result is immutable

    // "wither" instead of setter — like an immutable-update spread in JS
    public Order withItem(String item) {
        var next = new ArrayList<>(items);
        next.add(item);
        return new Order(id, next, placedAt);
    }
}

Collections.unmodifiableList(list) is only a read-only view — if the caller kept the original list, they can still mutate through it and your "immutable" object changes underneath you. List.copyOf (Java 10+) actually copies. Also note final List alone protects nothing: it pins the reference, not the contents.

Why immutability matters for concurrency: data races require a write. No writes after construction → no races → immutable objects can be shared freely between threads with zero locks. Deeper JMM point that scores senior credit: final fields get special treatment — the JVM inserts a freeze barrier at constructor end guaranteeing any thread that sees the object sees fully initialized final fields, even without synchronization on the handoff (as long as this doesn't escape the constructor). Non-final fields have no such guarantee. Bonus wins: immutable objects are safe HashMap keys, trivially cacheable, and failure-atomic (no half-mutated state after an exception).

Records (Java 16+) give you the shallow version for free: record Order(String id, List<String> items) {} generates final fields, accessors, equals/hashCode/toString. But records don't deep-copy — add a compact constructor Order { items = List.copyOf(items); } to make them properly immutable.

Generics: erasure, bounds, wildcards, PECS T2

Type erasure — compile-time-only types

Java generics are enforced by the compiler and then erased: at runtime a List<String> and a List<Integer> are both just List. T becomes Object (or its bound), and the compiler inserts casts at use sites. This was a deliberate 2004 choice for binary compatibility with pre-generics code — unlike C#'s reified generics. TS developers feel at home: TypeScript types are also erased, except Java keeps raw runtime classes around, so the seams show differently.

List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
a.getClass() == b.getClass();          // true — one runtime class: ArrayList

// Consequences of erasure — none of these compile:
class Box<T> {
    T make()            { return new T(); }        // ✘ which constructor? T is Object at runtime
    T[] arr()           { return new T[10]; }      // ✘ arrays need a runtime type
    boolean is(Object o){ return o instanceof T; }  // ✘ nothing to check against
    static T shared;                                 // ✘ statics are per-class, T is per-instance
}

// The standard workaround: pass the Class token explicitly
static <T> T create(Class<T> type) throws Exception {
    return type.getDeclaredConstructor().newInstance();
}

Overload clash from erasure: void f(List<String> x) and void f(List<Integer> x) can't coexist — same erased signature. And generic type info does survive in one place: class/field/method signatures (reflection can read List<String> from a field declaration — that's how Jackson deserializes generics via TypeReference, the "super type token" trick).

Bounded types and wildcards

// Bounded type parameter: T must be Comparable to itself (or a supertype — the classic recursive bound)
static <T extends Comparable<? super T>> T max(List<? extends T> list) { ... }

List<? extends Number> nums;   // SOME unknown subtype of Number — could be List<Integer>
List<? super Integer> sink;    // SOME unknown supertype — List<Integer|Number|Object>
List<?> anything;               // unknown; read-only as Object, can add only null

Key insight: generics are invariant. List<Integer> is not a List<Number> (if it were, you could add a Double into an Integer list). Wildcards restore controlled variance at use sites — where TS has structural bivariance-ish behavior and readonly tricks, Java makes you declare it per parameter.

PECS: Producer Extends, Consumer Super

  • You only read T out of the collection → it's a producer → ? extends T. (Reading is safe: whatever the element is, it's at least a T. Writing is forbidden — the compiler doesn't know which subtype the list actually holds.)
  • You only write T in → it's a consumer → ? super T. (Writing a T is safe: any supertype-of-T list can hold a T. Reads only give you Object.)
  • Read and write → exact type, no wildcard.
// The JDK's own signature — the canonical PECS exhibit:
public static <T> void copy(List<? super T> dest,      // consumes T (we write into it)
                            List<? extends T> src) {  // produces T (we read from it)
    for (int i = 0; i < src.size(); i++) dest.set(i, src.get(i));
}

List<Number> dest = new ArrayList<>(List.of(0, 0));
List<Integer> src = List.of(1, 2);
Collections.copy(dest, src);    // works: Number super Integer, Integer extends Integer

List<? extends Number> producer = List.of(1, 2.5);
Number n = producer.get(0);     // ✔ reading is fine
// producer.add(3);             // ✘ compile error — could be a List<Double>

Drop this in interviews: Comparator<? super T> appears all over the JDK (e.g. sort) so a Comparator<Object> or Comparator<Number> can sort a List<Integer> — the comparator consumes elements. Arrays, by contrast, are covariant (Integer[] IS-A Number[]) and pay for it with runtime ArrayStoreException — generics' invariance moved that failure to compile time, which is the whole point.

Exceptions: hierarchy, try-with-resources, best practices T2

The hierarchy

Throwable
├── Error                      // JVM-level disasters: OutOfMemoryError, StackOverflowError.
│                              // Don't catch; nothing sensible to do.
└── Exception
    ├── RuntimeException       // UNCHECKED: NullPointerException, IllegalArgumentException,
    │                          // IllegalStateException, IndexOutOfBounds, ClassCast...
    └── (everything else)      // CHECKED: IOException, SQLException, InterruptedException —
                               // must be caught or declared in `throws`

Checked vs unchecked — the philosophy: checked exceptions represent recoverable conditions the caller should plan for (file missing, network down); unchecked represent programming bugs (null where non-null promised, bad argument). JS/TS has no checked exceptions at all, so this is new muscle. Know the modern consensus too: checked exceptions compose terribly with lambdas/streams (Function can't throw them) and leak implementation details, so contemporary frameworks lean unchecked — Spring famously translates JDBC's checked SQLException into its unchecked DataAccessException hierarchy.

try-with-resources and AutoCloseable

Anything implementing AutoCloseable can go in the resource clause; close() is called automatically in reverse declaration order, even on exception. This replaced the horrific finally-block dance and fixed a real bug class: if both the body and close() throw, the body's exception propagates and the close failure is attached as a suppressed exception (readable via getSuppressed()) instead of silently swallowing the primary failure like the old finally idiom did.

try (var conn = dataSource.getConnection();
     var stmt = conn.prepareStatement("SELECT * FROM users WHERE id = ?")) {
    stmt.setLong(1, id);
    try (var rs = stmt.executeQuery()) {
        return rs.next() ? mapUser(rs) : null;
    }
} // closes rs, then stmt, then conn — guaranteed, in reverse order

Custom exceptions and translation

// Unchecked domain exception carrying context — the common modern shape
public class PaymentFailedException extends RuntimeException {
    private final String orderId;

    public PaymentFailedException(String orderId, Throwable cause) {
        super("Payment failed for order " + orderId, cause);  // ALWAYS chain the cause
        this.orderId = orderId;
    }
    public String orderId() { return orderId; }
}

// Exception translation at a layer boundary: low-level detail → domain meaning
try {
    gateway.charge(order);
} catch (HttpTimeoutException e) {
    throw new PaymentFailedException(order.id(), e);   // wrap-and-rethrow, cause preserved
}

Best practices a senior is expected to recite with reasons:

  • Never swallow: an empty catch block destroys the evidence. At minimum log with the exception object (log.error("...", e) — passing e, not e.getMessage(), keeps the stack trace).
  • Log or rethrow — not both: double-logging the same failure at every layer makes incident triage miserable.
  • Wrap, don't discard: when translating, always pass the original as cause so the full chain survives.
  • Catch the most specific type; never catch (Exception e) as a lazy default, and never catch Throwable/Error.
  • Exceptions are for exceptional flow — not control flow; construction captures a stack trace, which is expensive.
  • InterruptedException: either rethrow or call Thread.currentThread().interrupt() to restore the flag — swallowing it breaks cooperative cancellation.

A return inside finally silently discards any in-flight exception (and any body return). Compilers warn; interviewers ask. Never return or throw from finally.

The memory-leak implication: every non-static inner (and anonymous) class instance carries a hidden Outer.this reference. Hand one to a long-lived registry — a listener list, an executor queue, a static cache — and the entire outer object (plus everything it references) is pinned from garbage collection. public class Cache {

    class LeakyTask implements Runnable {          // pins the 50MB Cache alive
        public void run() { System.out.println("tick"); }
    }
    static class SafeTask implements Runnable {   // no outer reference
        public void run() { System.out.println("tick"); }
    }
}

Enums with fields, methods, and per-constant behavior

A Java enum is a real class: fixed set of instances, can have fields, constructors (implicitly private), methods, and even per-constant method bodies — a built-in, type-safe strategy pattern. Far beyond TS's enum (which is basically a number/string map).

public enum Operation {
    PLUS("+")  { public double apply(double a, double b) { return a + b; } },
    MINUS("-") { public double apply(double a, double b) { return a - b; } },
    TIMES("*") { public double apply(double a, double b) { return a * b; } };

    private final String symbol;
    Operation(String symbol) { this.symbol = symbol; }
    public String symbol() { return symbol; }
    public abstract double apply(double a, double b);
}

EnumMap / EnumSet — when the key domain is an enum, don't use HashMap/HashSet. EnumMap is a plain array indexed by ordinal(): no hashing, no collisions, iteration in declaration order, dramatically less memory. EnumSet is a bit vector (a single long for ≤64 constants) — set operations become single bitwise instructions.

Map<DayOfWeek, List<Shift>> rota = new EnumMap<>(DayOfWeek.class);
Set<DayOfWeek> weekend = EnumSet.of(DayOfWeek.SATURDAY, DayOfWeek.SUNDAY);

Enums are also the bulletproof singleton (serialization- and reflection-proof: the JVM guarantees one instance per constant per classloader) — Bloch's recommended pattern before DI containers made hand-rolled singletons mostly obsolete.

HashMap internals, in depth T1

The single most-asked internals question in Java interviews. Be able to narrate a put and a get end-to-end.

The structure

A HashMap is an array of buckets (Node<K,V>[] table). Each Node holds { int hash; K key; V value; Node next; }. Colliding entries chain via next — a singly linked list, upgraded to a red-black tree when a bucket gets pathological (Java 8+).

table (capacity n = 16)
[ 0] → null
[ 1] → (h, "apple", 3) → (h', "grape", 7) → null      // collision chain
[ 2] → (h, "kiwi", 1) → null
 ...
[15] → TreeNode… (red-black tree once a chain gets long)

Step 1 — hash spreading

// HashMap.hash(): XOR the high 16 bits into the low 16
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

Why? The bucket index only uses the low bits (next step). Many real hashCodes vary mostly in the high bits (e.g. Float keys, aligned pointers); without spreading, they'd all collide in a handful of buckets. XOR-folding the top half down mixes high-bit entropy into the index cheaply. (Also note the null branch: HashMap permits one null key, parked in bucket 0.)

Step 2 — index = (n − 1) & hash, and why capacity is a power of two

With capacity n a power of two, n − 1 is an all-ones bitmask (16 → 0b1111), so (n - 1) & hash equals hash % n but as a single AND instruction — no division, and no negative-modulo edge case. This is why HashMap forces capacities to powers of two (constructor arguments are rounded up via tableSizeFor). The trade-off: only the low bits pick the bucket, which is exactly why step 1's spreading exists. Bonus elegance: on resize, each entry either stays at index i or moves to i + oldCap — decided by a single bit of the hash — so rehashing splits each bucket into exactly two lists without recomputing anything.

Step 3 — put/get walkthrough

// get(key), conceptually:
int h = hash(key);
Node e = table[(table.length - 1) & h];
while (e != null) {
    if (e.hash == h && (e.key == key || key.equals(e.key)))  // hash short-circuit first,
        return e.value;                                       // then == fast path, then equals
    e = e.next;
}
return null;

put follows the same walk: found an equal key → replace value; hit the end → append a node, bump modCount and size, and resize if size > threshold.

Treeification (Java 8)

Before Java 8, an attacker who could choose your keys (HTTP param names!) could force every key into one bucket, degrading lookups to O(n) — a real DoS class (hash-flooding). Java 8's fix: when a single bucket's chain reaches 8 nodes and the table capacity is at least 64, the chain converts to a red-black tree of TreeNodes → worst case becomes O(log n). If capacity is under 64, HashMap resizes instead (growing usually disperses the chain more cheaply than treeifying). On removal/resize a tree bucket shrinking to 6 nodes converts back to a list — the 8/6 gap is hysteresis, preventing flip-flopping at the boundary. Why 8? Under a decent hash function chain lengths follow a Poisson distribution; the probability of a chain reaching 8 is about 6 in 100 million — so trees are a defense mechanism, not the expected mode.

Load factor and resize

Default load factor 0.75: resize when size > capacity × 0.75 (threshold 12 for the default capacity 16). It's the empirical sweet spot between wasted slots (low LF) and collision pileups (high LF). A resize allocates a table of double the capacity and redistributes every entry — O(n) — so if you know you'll store ~10,000 entries, pre-size: new HashMap<>(16_384) or HashMap.newHashMap(10_000) (Java 19+, does the load-factor math for you) to avoid ~10 progressive rehashes.

Mutable keys, again, with feeling: mutate a key's hash-relevant state after put, and the entry is stranded in the old bucket — unreachable by get, un-removable by remove, yet still holding memory and still visible when iterating entrySet(). This is the practical payoff of the whole equals/hashCode section, and the reason String, boxed primitives, and records are the canonical key types.

Complexity summary you should say precisely: get/put are amortized O(1) assuming good hash distribution; worst case O(log n) since Java 8 (was O(n)). Iteration is O(capacity + size) — another reason not to oversize wildly. Also: HashMap is unsynchronized, and concurrent writes without external synchronization can lose updates (and in pre-8 versions could even create a resize cycle causing an infinite loop — a famous production war story; nowadays you "just" get lost data or ConcurrentModificationException).

Why cache locality decides it

An ArrayList is a contiguous Object[]: iterating it walks sequential memory, so the CPU prefetcher streams cache lines ahead of you and each 64-byte line serves several element references. A LinkedList is a chain of individually allocated Node objects scattered across the heap: every next is a dependent pointer chase and a likely cache miss (~100ns each vs ~1ns for cached access — two orders of magnitude). Big-O counts operations; it doesn't price memory access. Benchmarks routinely show ArrayList beating LinkedList even at "insert at random positions" workloads because arraycopy over warm cache lines outruns pointer-chasing to the insertion point.

HashSet: a HashMap wearing a costume

// HashSet's actual implementation:
private transient HashMap<E, Object> map;

Your elements are the keys of an internal HashMap; every value is one shared sentinel. Consequence: everything from the HashMap section — hashing, spreading, treeification, load factor, immutable-element requirement — applies verbatim to HashSet. add returns false for duplicates because put returned a previous value.

Same bucket machinery as HashMap, but every entry is also stitched into a doubly linked list, giving deterministic iteration order — insertion order by default, or access order if you ask. Access order + one overridden hook = a production-grade LRU cache in ten lines, a genuinely common interview exercise:

public class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxEntries;

    public LruCache(int maxEntries) {
        super(16, 0.75f, true);        // true = ACCESS order: get() moves entry to tail
        this.maxEntries = maxEntries;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxEntries;    // called after each put; true → evict head (LRU)
    }
}

TreeMap: sorted keys via a red-black tree

TreeMap keeps keys sorted (natural order or a supplied Comparator) in a self-balancing red-black tree: get/put/remove are O(log n) — you trade HashMap's O(1) for ordering and range queries. Note it uses compareTo/compare exclusively — never hashCode, and equals only implicitly via compare == 0 — and rejects null keys (can't compare null).

The killer feature is the NavigableMap API — nearest-neighbor and range operations no hash structure can offer:

TreeMap<Integer, String> tiers = new TreeMap<>(Map.of(
    0, "bronze", 1_000, "silver", 10_000, "gold"));

tiers.floorEntry(4_500);      // 1000=silver  → greatest key ≤ 4500 (tier lookup!)
tiers.ceilingKey(1_001);      // 10000        → smallest key ≥ 1001
tiers.firstKey();             // 0
tiers.headMap(10_000);        // {0=bronze, 1000=silver} — live view, not a copy
tiers.subMap(500, 20_000);    // range query in O(log n) + output size

Pattern-match these use cases in interviews: price tiers, time-bucketed events ("latest reading at or before t" = floorEntry), consistent-hashing rings (ceilingKey wrapping to firstKey), leaderboards. If the question smells like "nearest key" or "range", the answer is TreeMap. TreeSet is the same tree with sentinel values, same trick as HashSet.

ConcurrentHashMap internals T2

The default answer to "thread-safe map" — but the interview points come from knowing how it avoids a global lock, and how Java 8 changed the design.

Evolution: segments → CAS + per-bin locks

  • Java 7: the table was split into 16 Segments (each a mini-hashtable extending ReentrantLock) → at most 16 concurrent writers. You may still hear "segment locking"; know it's the old design.
  • Java 8+: segments are gone. One table, like HashMap, with per-bucket coordination:
    • Empty bucket → CAS (compare-and-swap) the new node in — lock-free insert on the common path.
    • Occupied bucket → synchronized on the first node of that bin only. Contention is per-bucket, so effective write parallelism scales with the table size, not a fixed 16.
    • Reads take no lock at all: Node.val and next are volatile, so readers see writes via happens-before ordering, not mutual exclusion. get is as fast as HashMap's.
// putVal, conceptually (Java 8+):
for (;;) {
    if (bucket is empty) {
        if (casTabAt(tab, i, null, newNode)) break;   // lock-free; retry loop on CAS failure
    } else {
        synchronized (firstNodeOfBucket) {            // lock ONLY this bin
            // walk chain / tree, insert or replace
        }
    }
}

size() via CounterCells: a single volatile size field would make every write contend on one cache line. Instead CHM uses a baseCount plus a striped array of CounterCells (the LongAdder design): threads that lose a CAS on baseCount update their own probed cell; size() sums base + all cells. Fast concurrent increments, at the price of size() being a non-atomic estimate under concurrent writes — which is fine, because any size you read is stale a nanosecond later anyway. Same story for treeification (bins treeify at 8 like HashMap) and cooperative multi-threaded resize (helpers see a ForwardingNode and pitch in transferring buckets).

Why does CHM ban nulls when HashMap allows them? Ambiguity under concurrency: if get(k) returns null you couldn't tell "absent" from "mapped to null", and the containsKey-then-get two-step isn't atomic — another thread can intervene. Disallowing null removes the ambiguity.

compute/merge atomicity — the real superpower

Thread-safe methods don't make sequences of calls atomic. Check-then-act on a CHM is still a race. The atomic per-key methods fix this and are what you should reach for:

// BROKEN even on ConcurrentHashMap — two threads both see null, both put, one increment lost:
if (!map.containsKey(word)) map.put(word, 0);
map.put(word, map.get(word) + 1);

// Atomic — the remapping runs under that key's bin lock:
map.merge(word, 1, Integer::sum);

// Atomic get-or-create (memoized cache); factory runs at most once per key:
map.computeIfAbsent(userId, id -> loadUser(id));

Keep compute*/merge functions short and non-reentrant: they execute while holding the bin lock. A slow remote call inside computeIfAbsent blocks every other key hashing to that bin, and touching the same map from inside the function can deadlock or throw IllegalStateException ("recursive update").

Fail-fast vs fail-safe iterators T2

Fail-fast (ArrayList, HashMap, HashSet…): the collection keeps a modCount, incremented by every structural modification (add/remove — not set). An iterator snapshots it as expectedModCount at creation and re-checks on every next(); a mismatch throws ConcurrentModificationException immediately. It's a bug detector, not a safety mechanism — best-effort (no synchronization on modCount), and it fires for single-threaded misuse too:

for (String s : names) {
    if (s.isBlank()) names.remove(s);   // CME on the NEXT iteration — modCount drifted
}

Four correct ways to remove while iterating:

// 1. Iterator.remove() — the only mutation the iterator itself sanctions
for (var it = names.iterator(); it.hasNext(); ) {
    if (it.next().isBlank()) it.remove();     // updates expectedModCount in step
}

// 2. removeIf — same thing, bulk, and O(n) on ArrayList (single compaction pass
//    vs O(n²) repeated shifting if you remove one-by-one by index)
names.removeIf(String::isBlank);

// 3. Maps: iterate entrySet's iterator, or entrySet().removeIf(...)
users.entrySet().removeIf(e -> e.getValue().inactive());

// 4. Collect survivors into a new collection (streams filter + toList)

"Fail-safe" (the informal term) iterators never throw CME, by iterating over data that can't shift underneath them:

  • CopyOnWriteArrayList — the iterator holds the array snapshot from creation time; later writes are invisible to it.
  • ConcurrentHashMap — weakly consistent iteration: no snapshot, no exception; reflects some-but-maybe-not-all concurrent updates, each element seen at most once.

Precision scores points: "fail-safe" isn't a JDK term — the Javadoc says snapshot or weakly consistent. Also, fail-fast is explicitly best-effort: absence of a CME never proves correctness, so don't write code that depends on catching it.

CopyOnWriteArrayList, BlockingQueue, ArrayDeque T3

CopyOnWriteArrayList

Every mutation copies the entire backing array under a lock, then swaps the (volatile) reference; reads and iteration touch the immutable current array with zero locking. Writes are O(n) with allocation — so this only makes sense when reads outnumber writes by orders of magnitude. The canonical use: listener/observer lists — registered once at startup, iterated on every event, and safe to iterate while another thread (un)subscribes. Iterators are snapshots: no CME ever, but iterator.remove() is unsupported.

private final List<EventListener> listeners = new CopyOnWriteArrayList<>();

public void publish(Event e) {
    for (var l : listeners) l.onEvent(e);   // lock-free; concurrent register() is fine
}

queue.offer(root); Node n = queue.poll();

Caveats: no null elements (null is the "empty" sentinel for poll/peek), no capacity bound, not thread-safe (that's what BlockingQueue is for), and no indexed access. In coding rounds, reaching for ArrayDeque instead of Stack/LinkedList is itself a quiet signal of Java fluency.

Interview Q&A

Why must you override hashCode() whenever you override equals()?

Because hash-based collections do a two-phase lookup: hashCode() selects the bucket, equals() confirms the match within it. The contract requires equal objects to have equal hash codes. If I override only equals, two value-equal objects keep distinct identity hash codes, land in different buckets, and the map never even calls equals — so map.get(new Point(1,2)) returns null after put(new Point(1,2), v), and a HashSet stores "duplicates". The reverse breakage (hashCode without equals) causes collisions that equals then wrongly reports as distinct. In practice I use records for value types, or Objects.hash over exactly the fields equals uses — the two methods must be derived from the same field set.

What happens if you mutate an object after using it as a HashMap key?

If the mutation changes any field involved in hashCode, the entry becomes stranded: it physically sits in the bucket computed from the old hash, but every subsequent get/remove/containsKey computes the new hash and searches the wrong bucket. The entry is unreachable through the map API yet still consumes memory and still appears during entrySet() iteration — a memory leak plus phantom data. That's why keys should be immutable (String, boxed primitives, records), and why HashMap and immutability are really one interview topic wearing two hats.

Walk me through what HashMap.put() does internally.

(1) Compute hash = key.hashCode() ^ (h >>> 16) — spreading high bits into low ones because only low bits pick the bucket. (2) Index the table with (n-1) & hash — capacity is a power of two so this equals modulo but is one AND instruction. (3) If the bucket is empty, place a new Node. If not, walk the chain comparing stored hash first (cheap) then ==/equals; an equal key means replace-value, otherwise append. (4) If the chain reaches 8 and the table is ≥64 capacity, treeify to a red-black tree (else resize instead). (5) Increment modCount and size; if size > capacity × 0.75, double the table and redistribute — each entry either stays at index i or moves to i+oldCap, decided by one hash bit. Amortized O(1); worst case O(log n) since Java 8.

Why is HashMap's capacity always a power of two?

So bucket selection can be (n-1) & hash: for power-of-two n, n−1 is an all-ones mask, making the AND mathematically equal to hash % n (for non-negative values) but far cheaper than division and immune to Java's negative-modulo quirk. It also makes resizing elegant — doubling capacity adds one mask bit, so each entry either stays put or moves exactly oldCapacity forward, splitting every bucket into two lists in one pass. The cost is that only low bits matter for indexing, which is precisely why hash() XORs the high 16 bits down first.

What changed in HashMap in Java 8 and why?

Treeification. Pre-8, a bucket was always a linked list, so adversarial keys (hash-flooding: attacker sends thousands of colliding HTTP parameter names) degraded operations to O(n) — a denial-of-service vector. Java 8 converts a bucket to a red-black tree when its chain hits 8 nodes and capacity is at least 64 (below that it resizes instead), giving O(log n) worst case; tree bins revert to lists at 6 nodes (hysteresis gap avoids flip-flopping). Threshold 8 was chosen because with a decent hash function, chain length follows a Poisson distribution and P(length ≥ 8) ≈ 6×10⁻⁸ — trees are an attack/pathology defense, not the normal mode. Java 8 also switched insertion from head to tail, which eliminated the infamous multi-threaded resize infinite-loop (though concurrent use is still a bug — use ConcurrentHashMap).

Is Java pass-by-value or pass-by-reference? Prove it.

Strictly pass-by-value. For primitives, the value is copied; for objects, the reference (a pointer value) is copied. You can dereference the copy and mutate the shared object — the caller sees that — but you can never reseat the caller's variable. Proof: it's impossible to write a working swap(a, b) for two object variables in Java; assignment to a parameter changes only the local copy. In pass-by-reference languages (C++ with &) swap works. JavaScript has identical semantics ("call by sharing"), so the precise phrasing is: Java passes references by value.

Why is String immutable, and what does the string pool have to do with it?

Immutability serves four goals: security (strings carry class names, paths, URLs — mutation after validation would be a TOCTOU vulnerability), thread safety without synchronization, safe use as HashMap keys, and hashCode caching (String memoizes its hash — legal only if content is frozen). The string pool is the payoff: because strings can't change, the JVM safely shares one instance for every occurrence of the same literal, saving heap. Which produces the classic trap — == sometimes returns true for equal strings (both from the pool) and false otherwise (new String, runtime concatenation); content comparison must always be equals(). Since Java 7 the pool lives in the heap; intern() adds/fetches the canonical instance.

When would you still use StringBuffer, and what should you use instead of loop concatenation?

Practically never — StringBuffer's method-level synchronization protects individual calls but not compound sequences (interleaved appends from two threads still produce garbage ordering), so it pays the lock cost without delivering meaningful safety. For loop accumulation use StringBuilder (or better, String.join / Collectors.joining) — loop += is O(n²) because every iteration copies the accumulated prefix. Single-expression concatenation is fine: since JDK 9 it compiles to an invokedynamic StringConcatFactory call that's as efficient as a builder.

Interface with default methods vs abstract class — how do you choose?

Interfaces define capabilities and leave the single-inheritance slot free — a class can implement many. Since Java 8 they can ship behavior via default methods, and since 9 private helpers, so the old "abstract class = has code" distinction is gone. What interfaces still can't have: instance state, constructors (so no enforced invariants at construction), and non-public members. My rule: default to an interface; use an abstract class when implementations genuinely share mutable state or protected template machinery (JDK's AbstractList, Spring's template classes). Also worth saying: default methods exist primarily for API evolution — retrofitting forEach/stream onto Collection without breaking every implementor.

Two interfaces provide the same default method. What happens?

Resolution rules in order: a method from the class hierarchy always beats interface defaults; a more specific sub-interface's default beats its parent's; if neither rule disambiguates (two unrelated interfaces), it's a compile error and I must override the method myself, optionally delegating with the special syntax InterfaceName.super.method(). Java never guesses. And because interfaces hold no instance fields, Java only ever has the behavioral diamond, not the C++ state diamond.

What does type erasure actually mean at runtime, and what can't you do because of it?

Generic type arguments exist for the compiler only; after compilation, List<String> is just List, type variables become their bound (usually Object), and the compiler inserts casts at read sites. Chosen in Java 5 for binary/source compatibility with pre-generics code — unlike C#'s reified generics. Consequences: no new T() (no constructor to call at runtime — pass a Class<T> token instead), no new T[], no instanceof T, no overloads differing only in type arguments (same erased signature), and unavoidable "unchecked" warnings at generic/raw boundaries. Caveat worth adding: erasure applies to instances; generic types in class, method, and field signatures survive in the class file and are reflectively readable — that's how Jackson's TypeReference super-type-token trick works.

Explain PECS with a concrete example.

Producer Extends, Consumer Super — choose the wildcard by what the method does with the collection. If it only reads Ts out, the collection is a producer: take List<? extends T> (reading is safe — every element is at least a T; writing is forbidden since the actual list could be of any subtype). If it only writes Ts in, it's a consumer: take List<? super T> (a list of any supertype can hold a T; reads only yield Object). The canonical exhibit is Collections.copy(List<? super T> dest, List<? extends T> src). The underlying reason PECS exists: generics are invariant (List<Integer> is not a List<Number>, else you could insert a Double), and wildcards restore variance selectively at use sites.

Checked vs unchecked exceptions — what's your policy in a real codebase?

The hierarchy: Throwable splits into Error (JVM failures — never catch) and Exception; RuntimeException and its subclasses are unchecked, everything else checked. Design intent: checked = recoverable environmental conditions the caller must plan for; unchecked = programming bugs. My working policy, aligned with modern practice: throw unchecked domain exceptions at service boundaries, translate low-level checked exceptions at the layer where they occur (always chaining the cause), and reserve checked exceptions for cases where every caller genuinely has a recovery path. Rationale: checked exceptions don't compose with lambdas/streams and leak implementation details through signatures — which is why Spring translates checked SQLException into its unchecked DataAccessException hierarchy, a decision I'd cite as precedent.

How does try-with-resources improve on try/finally?

Any AutoCloseable declared in the resource clause is closed automatically in reverse declaration order, whether the body completes or throws — no null checks, no forgotten closes, no nesting pyramid for multiple resources. The subtle correctness win: with old try/finally, if the body threw A and close() then threw B, B replaced A and you lost the root cause. With try-with-resources, A propagates and B is attached as a suppressed exception, retrievable via getSuppressed() and printed in stack traces. Effectively-final variables declared before the try can also be listed directly since Java 9.

Why can a non-static inner class cause a memory leak?

Every instance of a non-static inner (or anonymous) class holds an implicit reference to its enclosing instance (Outer.this). If that inner instance escapes into anything long-lived — a static cache, an executor's queue, a listener registry — the GC cannot collect the outer object or anything it references, even though nothing else uses it. Classic real-world case: Android Handler/AsyncTask leaking an Activity. Fixes: declare nested classes static by default, or use a lambda — lambdas capture only the variables they actually use and capture no outer this unless the body references instance members.

ArrayList vs LinkedList — when is LinkedList actually faster?

Almost never, and I'd want profiler evidence before choosing it. The Big-O table flatters LinkedList (O(1) insert/remove at a known node), but positioning costs O(n) traversal, and each traversal step is a dependent pointer chase into scattered heap nodes — likely cache misses at ~100ns each. ArrayList's O(n) "penalty" is a System.arraycopy over contiguous, prefetched memory, which in benchmarks beats LinkedList even at insert-heavy workloads. Add 3–6× per-element memory overhead (24+ bytes of Node vs one array slot) and more GC pressure. The one defensible niche: many removals through a long-held ListIterator at the cursor. As a deque it's also outclassed by ArrayDeque. Default: ArrayList, pre-sized when n is known (growth is 1.5×, each grow copies the array).

How would you implement an LRU cache in Java using only the JDK?

Extend LinkedHashMap with the three-arg constructor's accessOrder=true — every get then moves the entry to the tail of the internal doubly linked list, so the head is always least-recently-used — and override removeEldestEntry to return size() > capacity, which makes each put evict the head when over budget. All operations O(1). For concurrent access I'd wrap it in Collections.synchronizedMap for low traffic, and reach for Caffeine in production (striped, TinyLFU admission, async loading) — knowing when to stop hand-rolling is part of the answer.

When do you pick TreeMap over HashMap?

Whenever I need order or range/nearest-key queries: sorted iteration, firstKey/lastKey, floorKey/ceilingKey ("greatest key ≤ x" — tier lookups, latest-event-before-t), and subMap/headMap/tailMap live range views. The price is O(log n) per operation (red-black tree) versus HashMap's amortized O(1). Two behavioral differences to flag: TreeMap uses compareTo/Comparator exclusively — never hashCode, and equality means compare()==0, so a comparator inconsistent with equals makes TreeMap and HashMap disagree about duplicates (BigDecimal's 1.0 vs 1.00 is the classic) — and TreeMap rejects null keys since null can't be compared.

How does ConcurrentHashMap achieve thread safety without a global lock (Java 8+)?

Three mechanisms replacing Java 7's 16 segments: (1) inserts into an empty bucket use CAS — no lock at all on that common path; (2) updates to an occupied bucket synchronize on that bin's first node only, so contention is per-bucket and scales with table size; (3) reads take no lock ever — node value/next fields are volatile, giving visibility through happens-before rather than mutual exclusion, so get costs the same as HashMap's. Size is maintained LongAdder-style via baseCount plus striped CounterCells to avoid a single hot cache line, which is why size() is an estimate under concurrency. Resizing is cooperative — threads encountering a ForwardingNode help transfer buckets. Bins treeify at 8 like HashMap.

ConcurrentHashMap is thread-safe — so why can this counter code still lose updates: map.put(k, map.get(k) + 1)?

Thread safety guarantees each individual method call is atomic and visible — not that a sequence of calls is. get-then-put is a check-then-act race: two threads both read 5, both write 6, one increment vanishes. The fix is the per-key atomic methods, which run their function under the bin's lock: map.merge(k, 1, Integer::sum) for counters, computeIfAbsent(k, factory) for get-or-create (factory runs at most once per key). Caveats I'd mention: keep those functions fast and side-effect-free — they hold the bin lock, so a remote call inside computeIfAbsent stalls other keys in that bin, and re-entering the same map from within the function risks deadlock or IllegalStateException.

Why doesn't ConcurrentHashMap allow null keys or values when HashMap does?

Ambiguity that can't be resolved atomically under concurrency. In HashMap, if get(k) returns null you can disambiguate "absent" vs "mapped to null" with containsKey — acceptable in single-threaded code because nothing changes between the two calls. In a concurrent map, another thread can insert or remove between them, so the two-step check is meaningless; a null return must have exactly one interpretation. Doug Lea banned nulls so get returning null always means "absent" — which also makes methods like putIfAbsent and compute (where null means "remove/none") semantically clean.

What causes ConcurrentModificationException and how do you remove elements while iterating correctly?

Fail-fast collections track a modCount bumped by structural modifications; each iterator snapshots it and re-checks on next(), throwing CME on mismatch. It fires in single-threaded code too — the classic is calling list.remove(x) inside a for-each. Correct approaches: (1) the iterator's own remove(), which updates its expected count in step; (2) removeIf(predicate) — also algorithmically better on ArrayList (one O(n) compaction pass vs O(n²) repeated shifts); (3) for maps, entrySet().removeIf(...); (4) build a filtered copy with streams. Two caveats: fail-fast is best-effort, not a guarantee — never rely on catching CME — and concurrent collections avoid it differently (COW snapshots; CHM's weakly consistent iterators).

Comparable vs Comparator — and what does sort stability buy you?

Comparable.compareTo is the type's single natural order, baked into the class (String alphabetical, numbers ascending) and used by default in sorts and TreeMap/TreeSet. Comparator is an external strategy — define unlimited orderings, even for classes you don't own, composed fluently: comparing(E::dept).thenComparing(E::salary, reverseOrder()).thenComparing(E::name). Stability: Java's object sorts use TimSort, which preserves the relative order of equal elements — so sorting by one key then another leaves the first key's order intact within ties, enabling multi-pass sorting and predictable UI tables. Primitive Arrays.sort is dual-pivot quicksort — unstable, but indistinguishable for primitives. Pitfalls to name-drop: a - b comparator overflow (use Integer.compare), TimSort throwing "Comparison method violates its general contract" for non-transitive comparators, and keeping compareTo consistent with equals for sorted collections.

Design an immutable class holding a List — walk through every guard.

(1) final class so no subclass can add mutability behind my type. (2) All fields private final. (3) No setters; "withers" return modified copies. (4) Defensive copy inbound: constructor does this.items = List.copyOf(items) — a genuine copy, unlike Collections.unmodifiableList, which is only a view the caller can still mutate through the original reference. (5) Outbound, return the immutable copy directly (or copy again if using a mutable field type). Concurrency payoff: no post-construction writes means no data races, so instances are freely shareable without locks — and the JMM gives final fields an initialization-safety guarantee (any thread seeing the object sees fully-built final fields, provided this doesn't escape the constructor). In modern Java I'd write a record with a compact constructor doing the List.copyOf.

When is CopyOnWriteArrayList the right tool, and what's the cost?

When reads massively outnumber writes and iteration must be safe under concurrent mutation — the archetype is a listener/subscriber list: registered a handful of times at startup, iterated on every event by multiple threads. Reads and iteration are entirely lock-free against an immutable array snapshot; iterators never throw CME (they also never see updates made after creation, and don't support remove()). The cost: every write copies the whole array under a lock — O(n) time plus allocation and GC pressure — so with frequent writes or large lists it's a pathological choice; there I'd use a synchronized/concurrent alternative or rethink the design.

Why prefer ArrayDeque over Stack and over LinkedList for stacks and queues?

Stack is a 1.0 relic: it extends Vector, so every operation takes a monitor lock you rarely need, and it inherits indexed List access — anyone can insert into the middle of your "stack", violating its contract. LinkedList works but pays a heap-allocated Node per element with prev/next pointers and poor locality. ArrayDeque is a circular growable array: head/tail indices wrap, both ends are amortized O(1), zero per-element allocation, cache-friendly iteration. The Javadoc itself says it's faster than Stack as a stack and LinkedList as a queue. Limits: no nulls (null is the poll/peek "empty" sentinel), not thread-safe, no indexed access. For cross-thread queues, that's BlockingQueue's job.

Your service's in-memory queue caused an OutOfMemoryError under load. What likely happened and how do you fix it?

Almost certainly an effectively unbounded queue in front of a consumer slower than the producer — e.g. new LinkedBlockingQueue<>() defaults to Integer.MAX_VALUE capacity, so under sustained overload it buffers until the heap dies. Fix: bound the queue (ArrayBlockingQueue, or LinkedBlockingQueue with an explicit capacity) so overload becomes backpressure — producers block on put, time out on offer(e, timeout, unit), or shed load on offer returning false — and pair it with a deliberate rejection policy. Same principle applies to ThreadPoolExecutor configuration: its work queue must be bounded and its RejectedExecutionHandler chosen consciously. Unbounded buffering doesn't fix overload; it converts it into a delayed heap dump.

Exercise 1: Break a HashMap, then explain the autopsy

Write a mutable class Ticket { String code; } with equals/hashCode over code. Put one in a HashMap<Ticket, String>, mutate code, then demonstrate: (a) get with the same object fails, (b) get with a fresh equal-to-old-value key fails, (c) the entry still appears when iterating. Explain each in terms of buckets.

Solution 
class Ticket {
    String code;
    Ticket(String code) { this.code = code; }
    @Override public boolean equals(Object o) {
        return o instanceof Ticket t && t.code.equals(code);
    }
    @Override public int hashCode() { return code.hashCode(); }
}

var map = new HashMap<Ticket, String>();
var t = new Ticket("A-100");
map.put(t, "row 4");

t.code = "B-200";                                // mutate hash-relevant state

map.get(t);                                       // (a) null — computes hash("B-200"),
                                                  //     searches the WRONG bucket
map.get(new Ticket("A-100"));                   // (b) null — RIGHT bucket (old hash),
                                                  //     but stored node's equals now says code is "B-200" ≠ "A-100"
map.size();                                       // 1 — (c) the node still occupies its bucket;
for (var e : map.entrySet())                     //     iteration walks buckets directly,
    System.out.println(e.getKey().code);          //     prints B-200 — a stranded, unreachable entry

The entry is now un-gettable and un-removable via the API: no key value can simultaneously hash to the old bucket and equal the new state. Only clear(), iterator removal, or GC of the whole map reclaims it. Moral: HashMap keys must be immutable in their hash-relevant fields.

Exercise 2: Immutable Money with defensive copies

Design an immutable Invoice: String id, List<LineItem> items, Instant issuedAt, plus a total() method and a withItem(LineItem) wither. Prove immutability: show that mutating the list passed to the constructor, and the list returned from the getter, cannot affect the invoice. Then rewrite as a record.

Solution 
public final class Invoice {
    private final String id;
    private final List<LineItem> items;
    private final Instant issuedAt;

    public Invoice(String id, List<LineItem> items, Instant issuedAt) {
        this.id = Objects.requireNonNull(id);
        this.items = List.copyOf(items);      // copy IN: caller's later list.add() is invisible to us
        this.issuedAt = Objects.requireNonNull(issuedAt);
    }

    public List<LineItem> items() { return items; }  // safe OUT: List.copyOf is immutable —
                                                       // caller's add() throws UnsupportedOperationException
    public long total() {
        return items.stream().mapToLong(LineItem::priceMinor).sum();
    }

    public Invoice withItem(LineItem item) {
        var next = new ArrayList<>(items);
        next.add(item);
        return new Invoice(id, next, issuedAt);   // constructor re-copies — invariant holds everywhere
    }
}

// Record version — compact constructor supplies the defensive copy:
public record Invoice(String id, List<LineItem> items, Instant issuedAt) {
    public Invoice {                            // runs before fields are assigned
        Objects.requireNonNull(id);
        items = List.copyOf(items);
    }
    public long total() {
        return items.stream().mapToLong(LineItem::priceMinor).sum();
    }
}

Proof sketch: after construction, the invoice's list reference points at a private immutable copy — the caller's original list and the invoice share no mutable structure in either direction. LineItem should itself be a record for the immutability to be deep.

Exercise 3: LRU cache with eviction callback

Build an LRU cache over LinkedHashMap with capacity N and an onEvict(K, V) hook (e.g. to close a pooled resource). Demonstrate access-order behavior: insert 1,2,3 with capacity 3, get(1), insert 4 — which key was evicted and why?

Solution 
public class EvictingLru<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;
    private final BiConsumer<K, V> onEvict;

    public EvictingLru(int capacity, BiConsumer<K, V> onEvict) {
        super(16, 0.75f, true);                       // accessOrder = true
        this.capacity = capacity;
        this.onEvict = onEvict;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        if (size() > capacity) {
            onEvict.accept(eldest.getKey(), eldest.getValue());
            return true;                              // LinkedHashMap removes the head for us
        }
        return false;
    }
}

var cache = new EvictingLru<Integer, String>(3,
        (k, v) -> System.out.println("evicted " + k));
cache.put(1, "a"); cache.put(2, "b"); cache.put(3, "c");
cache.get(1);            // access moves 1 to the most-recent end → order now 2, 3, 1
cache.put(4, "d");       // prints "evicted 2" — 2 is now least-recently-used, not 1
cache.keySet();          // [3, 1, 4] in LRU→MRU order

Without accessOrder=true, the map tracks insertion order and would have evicted 1 (a FIFO cache). The get(1) relinking is the entire difference between FIFO and LRU — one constructor flag.

Exercise 4: Word frequency, three ways — spot the race

Count word frequencies from multiple threads. Implement (a) a broken version on ConcurrentHashMap using get-then-put, (b) a correct version with merge, (c) a single-threaded stream version with groupingBy. Explain exactly where (a) loses updates.

Solution 
// (a) BROKEN — two threads read the same old value, both write oldValue+1
void countBroken(ConcurrentHashMap<String, Integer> m, String w) {
    Integer old = m.get(w);              // T1 reads 5; T2 reads 5
    m.put(w, old == null ? 1 : old + 1); // T1 writes 6; T2 writes 6 → one increment lost
}
// Each CALL is thread-safe; the get→put SEQUENCE is not. Classic check-then-act race.

// (b) CORRECT — merge runs atomically under the key's bin lock
void count(ConcurrentHashMap<String, Integer> m, String w) {
    m.merge(w, 1, Integer::sum);
}
// Alternative for extreme write contention: ConcurrentHashMap<String, LongAdder> with
// m.computeIfAbsent(w, k -> new LongAdder()).increment(); — striped counters, no CAS retry storms.

// (c) Single-threaded / batch — streams collector
Map<String, Long> freq = words.stream()
    .collect(Collectors.groupingBy(w -> w, Collectors.counting()));

Verification idea: run (a) and (b) with 8 threads × 100,000 increments of the same word; (a) totals below 800,000, (b) is exact. The lesson generalizes: thread-safe collection ≠ thread-safe compound logic — atomicity must cover the whole read-modify-write.

Exercise 5: Multi-key sort with a stability proof

Given record Employee(String name, String dept, int salary), sort by dept ascending, then salary descending, then name. Then demonstrate TimSort's stability: sort the full list by name first, then re-sort by dept only, and show names remain ordered within each dept.

Solution 
var staff = new ArrayList<>(List.of(
    new Employee("Cara", "eng", 120),
    new Employee("Ana",  "eng", 120),
    new Employee("Bob",  "ops", 90),
    new Employee("Dan",  "eng", 150)));

// Part 1: composed comparator
staff.sort(Comparator.comparing(Employee::dept)
        .thenComparing(Employee::salary, Comparator.reverseOrder())
        .thenComparing(Employee::name));
// eng/150/Dan, eng/120/Ana, eng/120/Cara, ops/90/Bob
// Note: reverseOrder() on the salary KEY only. Calling .reversed() at the end
// would instead flip the ENTIRE chain (dept descending too) — common trap.

// Part 2: stability — two successive single-key sorts
staff.sort(Comparator.comparing(Employee::name));   // Ana, Bob, Cara, Dan
staff.sort(Comparator.comparing(Employee::dept));   // TimSort is stable ⇒
// eng: Ana, Cara, Dan (name order preserved within equal dept!), then ops: Bob

Stability means equal elements (same dept) keep their pre-sort relative order — so a name-sort followed by a dept-sort equals a dept-then-name sort. This multi-pass technique only works because List.sort uses TimSort; with an unstable algorithm (like primitive dual-pivot quicksort) the within-group order would be arbitrary.

Exercise 6: Bounded producer/consumer with clean shutdown

Wire one producer and two consumers through a bounded BlockingQueue<String>. Requirements: capacity 100 (backpressure), consumers exit cleanly when work is done (poison-pill pattern), and interruption is handled correctly.

Solution 
static final String POISON = "__POISON__";   // sentinel; one per consumer

BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);
int consumers = 2;

Runnable producer = () -> {
    try {
        for (String job : loadJobs())
            queue.put(job);                    // blocks when full → backpressure, bounded memory
        for (int i = 0; i < consumers; i++)
            queue.put(POISON);                 // one pill per consumer so ALL of them stop
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();    // restore the flag — never swallow it
    }
};

Runnable consumer = () -> {
    try {
        while (true) {
            String job = queue.take();         // blocks when empty — no busy-waiting
            if (job.equals(POISON)) break;     // clean, deterministic shutdown
            process(job);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
};

try (var pool = Executors.newFixedThreadPool(3)) {  // AutoCloseable since Java 19
    pool.submit(producer);
    pool.submit(consumer);
    pool.submit(consumer);
}   // close() waits for tasks to finish

Design points worth articulating: the bound (100) converts producer overrun into blocking instead of heap growth; poison pills give consumers a definitive end-of-stream signal (an alternative is poll(timeout) plus a volatile done-flag); and every InterruptedException handler restores the interrupt flag so pool shutdown-now cancellation still propagates. Choosing ArrayBlockingQueue over an unbounded default is the difference between backpressure and an eventual OutOfMemoryError.
