The version map (8 / 11 / 17 / 21)

You'll be asked "what's new in recent Java." Keep this cheat sheet:

VersionLandmark features
8 (2014)Lambdas, Stream API, Optional, java.time, default methods — the big one
11 LTSvar in lambdas, new String/Files methods, standardized HttpClient
17 LTSRecords, sealed classes, switch pattern matching (preview), text blocks
21 LTSVirtual threads, pattern matching for switch (final), record patterns, sequenced collections

Target 17 or 21. The single most important line: "Java 8 introduced lambdas and streams; Java 21 introduced virtual threads." Those two bookend the modern era.

Lambdas & functional interfaces T1

A lambda is an anonymous function — an arrow function with types. Under the hood it's an instance of a functional interface: any interface with exactly one abstract method. That's the whole trick — a "function you pass around" is really an object implementing a one-method interface.

// These are equivalent — lambda is the concise form
Runnable r = () -> System.out.println("hi");
Comparator<String> byLen = (a, b) -> a.length() - b.length();

// Method reference — shorthand when the lambda just calls one method
list.forEach(System.out::println);   // == x -> System.out.println(x)
list.sort(String::compareToIgnoreCase);

The standard functional interfaces from java.util.function cover almost everything, and their names describe their shape:

InterfaceShapeUsed by
Function<T,R>T -> Rstream.map(...)
Predicate<T>T -> booleanstream.filter(...)
Consumer<T>T -> voidstream.forEach(...)
Supplier<T>() -> Tlazy values, orElseGet

Lambdas can only reference local variables that are effectively final (assigned once). It's the same closure-capture rule you know from JS — Java just enforces that you don't reassign a captured variable, which sidesteps a whole class of confusing bugs.

Streams — the workhorse T1

A stream is a pipeline of operations over a sequence — Java's .map().filter().reduce(). You build it from a collection, chain intermediate operations (lazy, return a new stream), and finish with one terminal operation (eager, produces a result). Nothing runs until the terminal op.

List<String> activeNames = users.stream()          // source
    .filter(u -> u.active())                          // intermediate (lazy)
    .map(User::name)                              // intermediate (lazy)
    .sorted()                                         // intermediate (lazy)
    .toList();                                        // terminal → List (Java 16+)

The operations you'll reach for constantly:

OperationDoes
filter(pred)Keep elements matching a predicate
map(fn) / mapToIntTransform each element
flatMap(fn)Flatten a stream of collections into one stream
sorted() / distinct() / limit(n)Order, dedup, truncate
collect(...) / toList()Gather into a collection
reduce(...) / count() / sum()Fold to a single value
anyMatch / allMatch / findFirstShort-circuiting tests & search

Collectors is the toolbox for the collect terminal — grouping is the one you must know:

// Group employees by department → Map<String, List<Employee>>
Map<String, List<Employee>> byDept =
    staff.stream().collect(Collectors.groupingBy(Employee::dept));

// Group and count → Map<String, Long>  (a downstream collector)
Map<String, Long> headcount =
    staff.stream().collect(Collectors.groupingBy(Employee::dept, Collectors.counting()));

// Sum with primitive stream — avoids boxing
int payroll = staff.stream().mapToInt(Employee::salary).sum();

Streams are single-use — once you call a terminal operation, the stream is consumed; reusing it throws IllegalStateException. And a stream with no terminal operation runs nothing (the intermediate ops are lazy). If your map "isn't executing," you forgot the terminal op.

Use streams for transformation pipelines where they read clearly. Don't force them — a plain for loop is fine (often better) for side effects, early exit with complex state, or when you'd otherwise fight the API. "Readable first" is the senior instinct; a stream that needs a comment to parse should have been a loop.

Optional — the disciplined null T1

Optional<T> is a box that either holds a value or is empty — a type-level signal that "this might be absent," so the caller can't forget to handle it. It's the antidote to the NullPointerException.

Optional<User> found = repo.findByEmail(email);   // method signals "maybe nothing"

String name = found
    .map(User::name)                 // transform IF present
    .filter(n -> !n.isBlank())
    .orElse("anonymous");            // fallback if empty

found.ifPresent(u -> audit(u));       // run only if present
User u = found.orElseThrow(() -> new NotFoundException(email));  // or blow up meaningfully

Anti-patterns interviewers watch for: calling .get() without checking (just a fancier NPE); using Optional for fields or method parameters (it's designed for return values); and opt.isPresent() followed by opt.get() instead of map/orElse. Note orElse(x) always evaluates x; use orElseGet(() -> x) when building the fallback is expensive.

switch expressions & pattern matching T2

Modern switch is an expression (it returns a value), uses arrow syntax with no fall-through, and the compiler checks it's exhaustive:

String label = switch (status) {
    case PAID, SHIPPED -> "done";      // multiple labels, no break needed
    case PENDING       -> "waiting";
    case CANCELLED     -> "void";
};   // exhaustive over the enum ⇒ no default required

Pattern matching removes cast boilerplate. instanceof can bind a variable, and switch can branch on type:

// instanceof pattern — test and cast in one step (Java 16+)
if (obj instanceof String s && s.length() > 3) { use(s); }

// switch on type + record deconstruction (Java 21)
String describe(Shape shape) {
    return switch (shape) {
        case Circle(var r)          -> "circle r=" + r;   // destructure the record
        case Rectangle(var w, var h) -> "rect " + w + "x" + h;
    };
}

Sealed types & exhaustive modeling T2

A sealed interface or class restricts which types may extend it — a closed, known set. Combined with pattern-matching switch, the compiler can then prove your handling is exhaustive (add a new subtype and every non-exhaustive switch fails to compile — a compile-time to-do list). This is the classic "algebraic data type" / discriminated-union pattern, done in Java:

public sealed interface Shape permits Circle, Rectangle { }
public record Circle(double r) implements Shape { }
public record Rectangle(double w, double h) implements Shape { }
// Any switch over Shape must cover Circle and Rectangle — enforced at compile time.

Sealed types + records + switch patterns together give you TypeScript-style tagged unions with full compiler checking — a genuinely modern way to model domains.

var, text blocks & other niceties

  • var (Java 10) — infers a local variable's type from the right-hand side: var users = new ArrayList<User>();. Local variables only; the type is still static, just not written. Use it when the type is obvious from the RHS.
  • Text blocks (Java 15) — multi-line string literals, like template-literal backticks without interpolation: 
    String json = """
    { "id": 1, "name": "Ada" }
    """;
  • Enhanced String — isBlank(), strip(), lines(), repeat(n), formatted(...).
  • java.time — LocalDate, LocalDateTime, Instant, Duration. Immutable and sane; never touch the legacy Date/Calendar.

Level-Up appendix: stream mechanics

The deeper stream behavior that shows up in senior rounds. Get the everyday pipeline above fluent first.

Laziness & short-circuiting T2

Intermediate operations don't run when you call them — they run when the terminal op pulls elements through, one element at a time, all the way down the pipeline. That's why stream.filter(...).findFirst() can stop after the first match instead of filtering the whole list: short-circuiting terminals (findFirst, anyMatch, limit) end the pull early. It also means a map with a side effect only fires for elements the pipeline actually consumes — another reason to keep stream operations pure.

reduce and custom collectors T3

// reduce: fold to one value with an identity + combiner
int total = nums.stream().reduce(0, Integer::sum);

// Collectors compose: group, then map+join each group's values
Map<String, String> namesByDept = staff.stream().collect(
    Collectors.groupingBy(Employee::dept,
        Collectors.mapping(Employee::name, Collectors.joining(", "))));

Parallel streams — the honest take T2

stream.parallel() splits the work across the common ForkJoinPool. It sounds free but rarely helps: it only pays off for large datasets with CPU-bound, independent, stateless work and a cheaply-splittable source (arrays, ArrayList). The overhead of splitting and merging usually swamps the benefit, and a non-thread-safe collector or shared mutable state makes it outright wrong. Default answer: don't reach for parallel streams; measure first, and never use them for I/O (that's what virtual threads on page 04 are for).

  • A lambda is an instance of a one-method (functional) interface; prefer method references when they're clearer.
  • Streams = build → lazy intermediates → one terminal. They're single-use and do nothing without a terminal op.
  • Know filter, map, collect, and Collectors.groupingBy cold — they cover most real pipelines.
  • Optional is for return values; use map/orElse/orElseThrow, never bare .get().
  • switch expressions + pattern matching + sealed + records give you exhaustive, compiler-checked domain modeling.
  • Parallel streams are rarely the answer — measure first, never for I/O.

Interview Q&A

What's a functional interface, and how does it relate to a lambda?

A functional interface has exactly one abstract method (e.g. Runnable, Comparator, Function); the @FunctionalInterface annotation documents and enforces that. A lambda is just concise syntax for an instance of one — the compiler infers which interface from the target type and treats the lambda body as that single method's implementation. So "passing a function" in Java is really passing an object that implements a one-method interface.

Explain lazy evaluation in the Stream API.

Intermediate operations (filter, map, sorted) don't execute when called — they just build the pipeline and return a new stream. Nothing runs until a terminal operation (collect, toList, count, findFirst) pulls elements through. Elements flow one at a time down the whole chain, which enables short-circuiting: findFirst or anyMatch can stop early without processing the rest. A stream with no terminal op does no work at all.

When should you NOT use a stream?

When a plain loop is clearer: pure side effects, early exit with complex mutable state, or when the stream needs awkward workarounds (e.g. checked exceptions, index tracking). Streams also can't be reused after a terminal op. The senior stance is "readable first" — streams shine for transformation pipelines, but forcing imperative logic into them hurts clarity. Choose the form a reviewer parses fastest.

What problem does Optional solve, and how should you use it?

It makes "value may be absent" part of the type, so callers can't silently forget the null case. Use it as a return type and consume it with map, filter, orElse/orElseGet, orElseThrow, or ifPresent. Avoid it for fields and method parameters, and avoid .get() without a guard (that's just a disguised NPE). Prefer orElseGet over orElse when the fallback is expensive to build, since orElse's argument is always evaluated.

Are parallel streams a good way to speed things up?

Usually not. They help only for large, CPU-bound, independent, stateless workloads over a cheaply-splittable source; the split/merge overhead and shared common pool often make them slower or, with mutable shared state, incorrect. Never use them for I/O-bound work — that's what virtual threads are for. The right instinct is to measure a real bottleneck before reaching for .parallel(), not to sprinkle it on hopefully.

What do sealed types plus pattern matching give you?

A closed set of subtypes (sealed ... permits ...) lets the compiler verify a switch covers every case — exhaustiveness checking. Combined with records (for the data) and record patterns (to destructure), you get TypeScript-style discriminated unions with full compile-time safety: add a new permitted subtype and every non-exhaustive switch stops compiling, handing you a to-do list instead of a runtime surprise.

Hands-on exercises

Exercise 1: Rewrite a loop as a stream

Convert this to a single stream pipeline: iterate List<Order>, keep orders over $100, extract each customer's email, dedup, and collect to a sorted list.

Solution 
List<String> emails = orders.stream()
    .filter(o -> o.total() > 100)
    .map(Order::customerEmail)
    .distinct()
    .sorted()
    .toList();

Each stage maps to one intent — filter (keep), map (extract), distinct (dedup), sorted (order), toList (materialize). This reads top-to-bottom as the requirement itself, which is exactly when a stream beats a loop.

Exercise 2: Group and summarize

From List<Employee> (record of name, dept, salary), produce a Map<String, Double> of average salary per department.

Solution 
Map<String, Double> avgByDept = staff.stream()
    .collect(Collectors.groupingBy(
        Employee::dept,
        Collectors.averagingInt(Employee::salary)));

The pattern to remember: groupingBy(classifier, downstream). The classifier chooses the key; the downstream collector (counting, averagingInt, summingInt, mapping(...)) summarizes each group. This two-argument form answers most "group X and compute Y" questions.

Exercise 3: Safe lookup with Optional

Map<String,User> users. Return the uppercased name of the user with a given id, or "UNKNOWN" if there's no such user or the name is blank — with no explicit null check.

Solution 
String result = Optional.ofNullable(users.get(id))  // wrap possibly-null map hit
    .map(User::name)
    .filter(n -> !n.isBlank())
    .map(String::toUpperCase)
    .orElse("UNKNOWN");

Optional.ofNullable turns a nullable value into an Optional so the whole chain is null-safe; each map/filter only runs if a value is still present, and orElse supplies the single fallback. No if (x != null) ladder, and impossible to forget a case.
