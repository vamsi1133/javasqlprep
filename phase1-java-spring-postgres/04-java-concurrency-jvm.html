04 — Concurrency & the JVM (Level Up)

Doing many things at once, and understanding how the machine runs your code — the senior differentiators

Level: expert · read last, after pages 01–03 are solid

This is the top of the climb. Concurrency (running code on multiple threads) and the JVM (how your bytecode actually executes, and how memory is managed) are what separate a mid-level Java developer from a senior one in interviews. You don't write raw threads every day — frameworks like Spring handle most of it — but you're expected to reason about thread-safety, explain a memory leak, and know what garbage collection does. Learn the essentials in the main sections; the deepest theory (the memory model, GC internals, JIT) is quarantined in the ⭐ Level-Up appendix for when you want to go from "can discuss" to "can go deep."

Triage advice. For most interviews, the essentials below — thread pools, CompletableFuture, the synchronized/volatile/atomic distinction, heap vs stack, and what GC does — are enough to be strong. The appendix (Java Memory Model, CAS, ZGC, class loading) is for FAANG systems rounds and "go deeper" follow-ups. Don't let it intimidate you into skipping the essentials.

Threads: what and why

A thread is an independent path of execution within your program. JavaScript is single-threaded (one event loop); Java is genuinely multi-threaded — many threads can run at the same time on different CPU cores. That buys real parallelism (use all your cores) and responsiveness (don't block while waiting on I/O), at the cost of a hard new problem: when two threads touch the same data, order and visibility stop being obvious.

Thread t = new Thread(() -> doWork());  // a Runnable is the body
t.start();                              // runs concurrently; start(), never run()
t.join();                               // wait for it to finish

You almost never create Thread objects by hand in real code — raw threads are expensive to create and easy to leak. Use a thread pool (next section). Creating threads directly is mostly a teaching device and an interview talking point.

Thread pools & ExecutorService T1

A thread pool keeps a set of reusable worker threads and feeds them tasks from a queue — you submit work, it manages the threads. This caps resource use (a request spike can't spawn 10,000 threads) and avoids the cost of creating a thread per task.

try (ExecutorService pool = Executors.newFixedThreadPool(8)) {  // AutoCloseable, Java 19+
    Future<Integer> f = pool.submit(() -> compute());   // returns a handle to the result
    // ... submit more ...
    int result = f.get();                             // blocks until this task is done
}   // close() waits for submitted tasks, then shuts the pool down
  • submit(Callable) returns a Future — a handle you later get() (blocking) for the result.
  • newFixedThreadPool(n) for CPU-bound work (size ≈ core count); newVirtualThreadPerTaskExecutor() for I/O-bound work on Java 21 (below).
  • Always shut the pool down (try-with-resources, or shutdown()) or the JVM won't exit.

CompletableFuture — async composition T2

CompletableFuture is Java's Promise: it represents a value that will arrive later, and you compose callbacks instead of blocking. If you know .then()/async-await, this is the same model.

.then(fn) (transform).thenApply(fn)
.then(fn→promise) (chain async).thenCompose(fn)

CompletableFuture.supplyAsync(() -> fetchUser(id))   // run async, off the calling thread
    .thenApply(User::name)                          // transform when it arrives
    .thenCompose(name -> lookupProfileAsync(name))    // chain another async call
    .exceptionally(ex -> "fallback")                  // handle failure
    .thenAccept(System.out::println);              // consume the result

Use it to run independent I/O calls concurrently and combine the results — the natural fit for a service that fans out to several downstream APIs.

Thread-safety: synchronized, volatile, atomics T1

When multiple threads read and write shared mutable state, two things can go wrong: race conditions (interleaved read-modify-write produces lost updates) and visibility (one thread's write is never seen by another because of caching). Three tools, three jobs:

synchronized — mutual exclusion

Only one thread at a time can hold an object's lock, so a synchronized block/method serializes access to shared state. It fixes both race conditions and visibility.

private final Object lock = new Object();
private int count;
void increment() {
    synchronized (lock) { count++; }   // count++ is read-modify-write — must be atomic
}

volatile — visibility only

volatile guarantees every read sees the latest write (no thread-local caching) but does not make compound operations atomic. Perfect for a flag; wrong for a counter.

private volatile boolean running = true;   // one thread sets false, others see it promptly
// volatile int count; count++  is STILL a race — read/modify/write isn't atomic

Atomics — lock-free single-variable updates

AtomicInteger, AtomicLong, AtomicReference give atomic read-modify-write on one variable without a lock, using a CPU compare-and-swap instruction (see appendix). Faster than synchronized for simple counters.

private final AtomicInteger count = new AtomicInteger();
count.incrementAndGet();   // atomic ++, no lock

The senior-level framing: the best concurrency is no shared mutable state. Prefer immutable objects (page 02), confine state to one thread, or use the thread-safe java.util.concurrent collections (ConcurrentHashMap) instead of hand-rolled locking. Reach for synchronized only when you must protect a multi-step invariant, and keep the critical section tiny to avoid contention and deadlock.

Deadlock: two threads each hold a lock the other needs, and both wait forever. The standard prevention is lock ordering — always acquire multiple locks in the same global order — plus keeping critical sections short and, where possible, using tryLock with a timeout.

Virtual threads (Java 21) — the game-changer T2

The headline modern feature. A traditional ("platform") thread maps 1:1 to an OS thread — expensive, so you can only have a few thousand, which is why blocking I/O forced you into complex async code. A virtual thread is a lightweight thread the JVM schedules onto a small pool of OS threads; when it blocks on I/O, the JVM parks it and reuses the OS thread for other work. You can run millions.

// One virtual thread per request — write simple blocking code, get async scalability
try (var pool = Executors.newVirtualThreadPerTaskExecutor()) {
    for (var req : requests)
        pool.submit(() -> handleBlocking(req));   // blocking is now cheap
}

The payoff: you write straightforward blocking, sequential code — no CompletableFuture gymnastics — and still handle enormous I/O concurrency. It largely dissolves the "async vs simple" tradeoff for I/O-bound server workloads. The interview soundbite: "Virtual threads make blocking cheap, so simple thread-per-request code scales to millions of concurrent I/O-bound tasks."

JVM memory: heap vs stack T1

The JVM divides runtime memory into two regions you must be able to describe:

  • Stack — one per thread. Holds method call frames: local variables, parameters, and references to objects. Fast, automatically freed when a method returns. Too-deep recursion overflows it → StackOverflowError.
  • Heap — shared by all threads. Holds every object (everything you new). Managed by the garbage collector. Runs out → OutOfMemoryError.
Account a = new Account(...);
// the variable `a` (a reference) lives on the STACK
// the Account object it points to lives on the HEAP

This is exactly why "objects are references" (page 01) matters at the machine level: the reference sits on the stack, the object on the heap, and pass-by-value copies the reference. Primitives declared as locals live directly on the stack — no heap object, no GC.

Metaspace holds class metadata (the definitions of your classes), separate from the object heap. Leaking classes (e.g. repeated dynamic class loading) causes OutOfMemoryError: Metaspace — a distinct failure from a heap OOM.

Garbage collection in one pass T1

Java has no free() — the garbage collector automatically reclaims heap objects that are no longer reachable from any live reference (a "GC root": stack variables, static fields, etc.). If nothing can reach an object, it's garbage.

The key insight GC is built on is the weak generational hypothesis: most objects die young. So the heap is split into a young generation (new objects; collected often and cheaply — a "minor GC") and an old generation (objects that survived long enough; collected rarely — a "major GC"). This makes the common case fast.

You mostly just need the concepts and the default collector: G1GC (the default since Java 9) targets low pause times and works well out of the box; ZGC is for very large heaps needing sub-millisecond pauses. The one thing to internalize: GC frees you from manual memory management, but it does not free you from leaks — you can still hold references you forgot about.

Memory leaks & OutOfMemoryError T1

A Java "leak" isn't unreachable memory (GC handles that) — it's reachable memory you no longer need but still reference, so the GC can't collect it. The heap fills and you get OutOfMemoryError: Java heap space. The classic causes:

  • An ever-growing static collection — a static Map cache you add to but never evict. The most common real-world leak.
  • Unremoved listeners/callbacks — registered but never deregistered, keeping their owner alive.
  • Mutated map keys — the stranded-entry leak from page 02.
  • Unbounded queues/buffers — a producer outpacing a consumer with no backpressure.

How to talk about diagnosing one: reproduce, take a heap dump (jmap, or auto-dump on OOM with -XX:+HeapDumpOnOutOfMemoryError), open it in a tool like Eclipse MAT, and look at the "dominator tree" for the object retaining the most memory and the reference chain keeping it alive. The fix is almost always "stop holding the reference" — bound the cache (use an eviction policy / WeakHashMap), deregister the listener, add backpressure.

The Java Memory Model & happens-before T2

Why does volatile even matter? Because compilers and CPUs freely reorder and cache operations for speed — so without rules, one thread's writes may become visible to another in a different order, or not at all. The Java Memory Model (JMM) defines happens-before: a guarantee that if action A happens-before action B, A's effects are visible to B. Key edges: releasing a lock happens-before acquiring it; a volatile write happens-before every subsequent read of that field; Thread.start() happens-before the thread's code; the thread's end happens-before join() returning. Correct concurrent code is really just "establish a happens-before edge between every write and the reads that must see it." That's the single mental model underneath synchronized, volatile, and the atomics.

CAS, the ABA problem & contention T2

Atomics are lock-free, built on a CPU compare-and-swap (CAS): "atomically set this variable to N, but only if it still equals the E I last read; otherwise tell me it changed and I'll retry." No lock, no blocking — but under heavy contention, many threads spin retrying, which can waste CPU (that's why LongAdder exists — it stripes a counter across cells to cut contention). The ABA problem: a value goes A→B→A between your read and your CAS, so CAS succeeds though the world changed underneath; AtomicStampedReference adds a version stamp to defeat it. This is the machinery ConcurrentHashMap and the whole java.util.concurrent package are built on.

Locks beyond synchronized T3

ReentrantLock is synchronized with extras: tryLock(timeout) (deadlock avoidance), interruptibility, and fairness. ReadWriteLock lets many readers share access while writers get exclusivity — good for read-heavy data. StampedLock adds an optimistic-read mode that's faster still when writes are rare. Reach for these only when profiling shows synchronized is the bottleneck.

GC internals: G1 & ZGC T2

G1 ("Garbage First," the default) divides the heap into equal-size regions and collects the ones with the most garbage first, doing most work concurrently with your app to hit a pause-time target (default ~200ms). ZGC and Shenandoah are concurrent collectors that do nearly all work — including compaction — while the app runs, achieving sub-millisecond pauses even on multi-terabyte heaps, at some throughput cost. The old Parallel collector maximizes throughput but "stops the world" longer. Rule of thumb: keep G1 unless you have a measured pause-time problem on a large heap, then evaluate ZGC.

JIT compilation & class loading T3

JIT: the JVM starts by interpreting bytecode, profiles which methods run hot, then just-in-time compiles those to optimized native code (tiered: C1 for a quick win, C2 for aggressive optimization). This is why the JVM "warms up" — peak performance arrives after the hot paths compile, which matters for benchmarks (measure steady state, not startup). Class loading is lazy and hierarchical: classes load on first use through a parent-first chain of class loaders (bootstrap → platform → application), each delegating upward before trying itself — the mechanism frameworks and app servers use to isolate code.

  • Use thread pools (ExecutorService), not raw threads; CompletableFuture is Java's Promise for composing async work.
  • volatile = visibility only; atomics = lock-free single-variable updates; synchronized = mutual exclusion for multi-step invariants. Best of all: no shared mutable state.
  • Virtual threads (21) make blocking cheap — simple thread-per-request code scales to millions of I/O-bound tasks.
  • Stack = per-thread frames & references; heap = all objects, managed by GC. Deep recursion → StackOverflow; full heap → OutOfMemory.
  • GC reclaims unreachable objects (young/old generations, G1 by default) — but can't help a leak, which is memory you still reference.
  • Appendix soundbite: correct concurrency = establishing happens-before edges; atomics ride on CAS.

Interview Q&A

What's the difference between volatile and synchronized?

volatile guarantees visibility — reads always see the latest write, no caching — but does nothing for atomicity, so volatile count++ is still a race (it's read-modify-write). synchronized provides mutual exclusion and visibility: only one thread holds the lock at a time, and it establishes happens-before edges so writes are seen. Use volatile for a simple flag written by one thread; use synchronized (or an atomic for a single variable) when you need a compound operation to be atomic.

Why use a thread pool instead of creating threads directly?

Threads are expensive to create and each consumes memory, so a task-per-thread model lets a load spike exhaust resources or OOM. A pool reuses a bounded set of workers fed by a queue: it caps concurrency, amortizes thread-creation cost, and gives lifecycle control (graceful shutdown, rejection policies). You submit tasks and get Futures back. On Java 21, a virtual-thread-per-task executor removes the "bounded" tradeoff for I/O-bound work because virtual threads are cheap.

What are virtual threads and what problem do they solve?

Platform threads map 1:1 to OS threads — expensive, capped at a few thousand — which forced async/reactive code to scale I/O. A virtual thread is a lightweight JVM-scheduled thread multiplexed onto a small pool of OS threads; when it blocks, the JVM parks it and frees the carrier thread. You can run millions, so you write simple blocking thread-per-request code and still get massive I/O concurrency. It mostly dissolves the "simple vs scalable" tradeoff for I/O-bound servers. They don't add CPU parallelism — cores still bound CPU-heavy work.

Explain the difference between the stack and the heap.

The stack is per-thread and holds method frames: local variables, parameters, and references to objects; it's freed automatically as methods return, and overflowing it (deep recursion) throws StackOverflowError. The heap is shared across threads and holds all objects you allocate with new; it's managed by the garbage collector, and exhausting it throws OutOfMemoryError. A local reference lives on the stack while the object it points to lives on the heap — which is exactly why Java is "pass reference by value."

What causes a memory leak in Java if there's a garbage collector?

GC only reclaims unreachable objects. A leak is memory that's still reachable — you keep a reference you no longer need — so GC can't collect it and the heap grows to OutOfMemoryError. Classic causes: an unbounded static cache/collection, listeners/callbacks never deregistered, objects mutated after use as map keys, and unbounded queues. You diagnose with a heap dump (auto-dump on OOM) analyzed in a tool like Eclipse MAT, find the retaining reference chain, and fix by dropping the reference — bound the cache, deregister, add backpressure.

How would you make a class thread-safe? T2

Preferably by design: make it immutable (page 02) so there's nothing to race on, or confine its state to a single thread. If it must hold shared mutable state, guard every access — use an atomic for a single counter, synchronized/ReentrantLock for multi-step invariants (keeping critical sections small), or delegate to a concurrent collection like ConcurrentHashMap and its atomic methods (merge, computeIfAbsent). The key is that a "thread-safe collection" doesn't make your compound logic atomic — the guard must cover the whole read-modify-write.

What is happens-before? T2

It's the Java Memory Model's ordering/visibility guarantee: if action A happens-before B, then A's memory effects are visible to B and can't be reordered past it. Because CPUs and compilers reorder and cache freely, you need these edges for correct concurrency. The main ones: unlock happens-before a later lock on the same monitor; a volatile write happens-before subsequent reads of it; Thread.start() happens-before the thread's work; a thread's completion happens-before join() returns. Correct concurrent code establishes a happens-before edge between each write and the reads that must observe it.

Hands-on exercises

Exercise 1: Spot the concurrency bug

Two threads each call counter++ 100,000 times on a shared volatile int counter. The final value is less than 200,000. Why, and what are two correct fixes?

Solution

counter++ is three steps — read, add one, write — and two threads can read the same value before either writes, so increments are lost. volatile fixes visibility but not this atomicity gap, so it doesn't help. Fixes: (1) use AtomicInteger with incrementAndGet() — lock-free atomic RMW; or (2) wrap counter++ in a synchronized block. Both make the whole read-modify-write indivisible; the atomic is preferable for a single counter.

Exercise 2: Fan out and combine

You must call three independent, slow HTTP services and combine their results. Sketch it with CompletableFuture so the calls run concurrently rather than one after another.

Solution 
var a = CompletableFuture.supplyAsync(() -> serviceA());
var b = CompletableFuture.supplyAsync(() -> serviceB());
var c = CompletableFuture.supplyAsync(() -> serviceC());

Result combined = CompletableFuture.allOf(a, b, c)  // wait for all three
    .thenApply(v -> merge(a.join(), b.join(), c.join()))  // join() is non-blocking now — all done
    .join();

Starting all three supplyAsync calls first launches them concurrently; allOf completes when the slowest finishes, so total time ≈ the slowest call, not the sum. On Java 21 you'd get the same concurrency from three virtual threads with plain blocking calls — often simpler to read.

Exercise 3: Explain a leak from a heap dump

Production OOMs after days of uptime. A heap dump shows a single static HashMap retaining 90% of the heap, holding millions of session objects. What's the diagnosis and the fix?

Solution

Diagnosis: a classic static-collection leak — sessions are added to the map but never removed, so they stay reachable (GC can't reclaim them) and the map grows without bound until the heap is exhausted. The "days of uptime" and single dominant retainer in the dominator tree are the tells. Fix: bound the cache with an eviction policy (size/TTL — e.g. Caffeine, or Guava's CacheBuilder), remove sessions on logout/expiry, or use a WeakHashMap/soft references so entries can be collected under pressure. The root cause is always "still holding a reference you no longer need."
