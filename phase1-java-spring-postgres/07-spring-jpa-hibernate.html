07 — Spring Data JPA & Hibernate

Entities, relationships, the N+1 problem, transactions, and the persistence context — the topics that decide senior Java interviews.

Why interviewers ask this

Every Java backend interview eventually lands on JPA, and this is where the interviewer separates the seniors from the annotation-users. Anyone can put @Entity on a class and call repository.save(). What interviewers probe is whether you understand the machinery underneath: what SQL Hibernate actually generates, why a lazy collection blows up outside a transaction, why an update happened even though you never called save(), and why the service went from 3 queries to 3,000 in production.

Coming from the JS world: think of JPA/Hibernate as roughly the role Prisma plays, but with a fundamentally different design. Prisma is a query builder — you ask for data, you get plain objects back, the story ends. Hibernate is a stateful ORM with a unit-of-work: it tracks every entity you load, detects your changes automatically, and decides when to write them back. That statefulness is the source of both its power (dirty checking, batching, caching) and nearly every interview trap in this file. Internalize the persistence context and the rest follows.

Terminology map: JPA (Jakarta Persistence API) is the specification — annotations and interfaces like @Entity, EntityManager. Hibernate is the dominant implementation. Spring Data JPA is a Spring layer on top of JPA that generates repository implementations for you. In interviews, say "Spring Data JPA generates the repository, JPA defines the contract, Hibernate executes the SQL" and you have already scored a point.

Entity mapping & ID generation T1

An entity is a Java class mapped to a table. The minimum viable entity:

@Entity
@Table(name = "customers")   // otherwise table name defaults to the class name
public class Customer {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "customer_seq")
    @SequenceGenerator(name = "customer_seq", sequenceName = "customer_seq", allocationSize = 50)
    private Long id;

    @Column(name = "email", nullable = false, unique = true, length = 320)
    private String email;

    @Column(nullable = false)
    private String name;

    @Enumerated(EnumType.STRING)   // NEVER ORDINAL — reordering the enum corrupts data
    private CustomerStatus status;

    protected Customer() {}          // JPA requires a no-arg constructor (can be protected)

    public Customer(String email, String name) {
        this.email = email;
        this.name = name;
        this.status = CustomerStatus.ACTIVE;
    }
    // getters, setters (or accessors) omitted
}

Key rules that trip people up:

  • JPA needs a no-arg constructor (at least protected) because Hibernate instantiates entities reflectively and populates fields afterward.
  • Entities cannot be Java records (records are final, no no-arg constructor, no mutable fields for dirty checking). Records are perfect for DTO projections, not entities.
  • Avoid Lombok's @Data/@EqualsAndHashCode/@ToString on entities — they touch lazy fields and generate broken equals (see below).

ID generation strategies — IDENTITY vs SEQUENCE (this matters on Postgres)

StrategyHow it worksBatching?Verdict for Postgres
IDENTITYDB assigns the id during INSERT (Postgres bigserial / GENERATED ... AS IDENTITY). Hibernate must execute the INSERT immediately at persist() to learn the id.No — JDBC batch inserts are disabledWorks, but forfeits batching
SEQUENCEHibernate calls select nextval('customer_seq') before insert; with allocationSize=50 one DB round-trip yields 50 ids (pooled optimizer). INSERTs can be deferred and batched at flush.YesPreferred
UUIDGenerated in the application (JPA 3.1: GenerationType.UUID). No DB round-trip at all; id known before insert.YesGood for distributed id creation; watch index locality
AUTO"Let the provider choose." On Postgres with Hibernate 6 this picks SEQUENCE, but the choice can shift between dialects/versions.DependsBe explicit instead

Why IDENTITY kills batching, mechanically: the persistence context must know every managed entity's id (it's the key in the first-level cache). With IDENTITY, the only way to get the id is to run the INSERT. So persist() can't just queue the insert for flush time — it fires it right away, one statement at a time, and Hibernate silently ignores hibernate.jdbc.batch_size for that entity.

With SEQUENCE + allocationSize=50, saving 50 entities in one transaction looks like this in the SQL log:

-- one round trip fetches an id block (pooled optimizer)
select nextval('customer_seq')          -- returns e.g. 100; Hibernate hands out 51..100 locally

-- at flush, with spring.jpa.properties.hibernate.jdbc.batch_size=50:
insert into customers (email, name, status, id) values (?, ?, ?, ?)
-- ...sent as ONE JDBC batch of 50 parameter sets, not 50 statements

allocationSize must match the sequence's INCREMENT BY in the database. If the DDL says INCREMENT BY 1 but Hibernate assumes 50, you get duplicate-key errors or huge id gaps depending on the optimizer. When Flyway owns the schema, create the sequence yourself: CREATE SEQUENCE customer_seq START WITH 1 INCREMENT BY 50;

UUID keys

@Id
@GeneratedValue(strategy = GenerationType.UUID)
private UUID id;   // maps to Postgres uuid column

Random UUIDv4 keys scatter writes across a B-tree index (poor locality, index bloat under heavy insert). If you need UUIDs at scale, prefer time-ordered UUIDv7 (Hibernate 6.5+: @UuidGenerator(style = UuidGenerator.Style.TIME) or a custom generator). Mentioning v7 in an interview signals you've operated Postgres at volume.

equals() and hashCode() — the id-is-null-before-persist problem

A classic senior question. The trap: a generated id is null until the entity is persisted. If equals/hashCode are based on the id, then:

  1. You add a transient entity to a HashSet — hashCode computed from id = null.
  2. You persist it — id gets assigned, hashCode changes.
  3. set.contains(entity) now returns false for an object that is in the set. The set is corrupted.

Two accepted solutions:

// Option A: natural/business key (best when one exists, e.g. email, ISBN)
@Override public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Customer other)) return false;
    return email.equals(other.getEmail());
}
@Override public int hashCode() { return email.hashCode(); }

// Option B: id-based, null-safe, with a CONSTANT hashCode (Vlad Mihalcea's pattern)
@Override public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Customer other)) return false;  // instanceof, NOT getClass() —
    return getId() != null && getId().equals(other.getId()); // Hibernate proxies are subclasses!
}
@Override public int hashCode() { return getClass().hashCode(); } // constant: never changes across lifecycle

Details worth saying out loud in an interview: use instanceof (or Hibernate.getClass()) rather than getClass() == because a lazy proxy is a runtime subclass of your entity; and call other.getId() through the getter, because on a proxy the field is null — only the getter triggers initialization/returns the identifier. Two transient entities (both ids null) compare unequal under Option B, which is the safe behavior. The constant hashCode means all entities of a type land in one hash bucket — fine, because entity sets are small (a parent's children), and correctness beats bucket distribution.

Relationships: owning side, cascades, orphans T1

@ManyToOne — the workhorse

@Entity
@Table(name = "orders")
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "order_seq")
    @SequenceGenerator(name = "order_seq", allocationSize = 50)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)  // default is EAGER — always override!
    @JoinColumn(name = "customer_id", nullable = false)
    private Customer customer;

    private BigDecimal total;
    private Instant createdAt;
}

The orders table carries the customer_id foreign key. The FK always lives on the many side — that fact drives everything about owning vs inverse sides below.

@ManyToOne and @OneToOne default to EAGER fetching (JPA spec decision, widely regarded as a mistake). Every senior codebase overrides them to LAZY. EAGER isn't just "one extra join" — it's unremovable at query time: even a query that only needs orders will drag customers along, and EAGER on the wrong path cascades into loading half the object graph.

@OneToMany(mappedBy) — owning side vs inverse side

@Entity
public class Customer {
    // ... id, fields ...

    @OneToMany(mappedBy = "customer", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Order> orders = new ArrayList<>();  // initialize inline — never null
}

mappedBy = "customer" means: "the Order.customer field owns this relationship; I'm just the mirror." Hibernate writes the FK column based only on the owning side. This is the classic trap:

// TRAP: modifying only the inverse side — nothing is persisted to the FK column
customer.getOrders().add(order);      // order.customer is still null
orderRepository.save(order);          // INSERT ... customer_id = NULL (or constraint violation)

// CORRECT: set the owning side (the FK column follows the @ManyToOne field)
order.setCustomer(customer);

To keep the in-memory graph consistent (so both sides agree within the same persistence context), use helper methods that sync both sides — the idiomatic pattern interviewers expect:

// inside Customer
public void addOrder(Order order) {
    orders.add(order);
    order.setCustomer(this);
}
public void removeOrder(Order order) {
    orders.remove(order);
    order.setCustomer(null);   // with orphanRemoval=true this DELETEs the order at flush
}

A genuinely senior take: you often don't need the @OneToMany at all. A unidirectional @ManyToOne plus a repository query (findByCustomerId(Long id, Pageable p)) is simpler, avoids loading unbounded collections into memory, and paginates properly. Model bidirectional associations only when you traverse them together and the collection is naturally bounded (order → order lines: yes; customer → all orders ever: no).

@ManyToMany — and why you'll usually replace it

@Entity
public class Student {
    @ManyToMany
    @JoinTable(name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id"))
    private Set<Course> courses = new HashSet<>();
}

This works until the day product asks: "when did the student enroll?" A join table managed by @ManyToMany cannot hold extra columns. The standard evolution is to promote the join table to a real entity:

@Entity
@Table(name = "enrollments")
public class Enrollment {
    @Id @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "enrollment_seq")
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY) private Student student;
    @ManyToOne(fetch = FetchType.LAZY) private Course course;

    private Instant enrolledAt;
    private Grade grade;
}

Now it's just two @ManyToOnes — easier to query, paginate, and extend. In interviews: "@ManyToMany is fine for a pure link with no attributes, but I reach for a join entity by default because join tables almost always grow columns."

@OneToOne — and its lazy-loading limitation

@Entity
public class User {
    @OneToOne(mappedBy = "user", fetch = FetchType.LAZY)  // LAZY here is a LIE — see note
    private UserProfile profile;   // non-owning side (no FK column in users)
}

On the non-owning (mappedBy) side of a @OneToOne, LAZY generally cannot be honored without bytecode enhancement. Reason: Hibernate must decide whether the field is null or a proxy — and since the FK lives in the other table, the only way to know is to query it. So it queries eagerly anyway. Fixes: map only the owning side (put the FK where you query from), share the primary key with @MapsId, or just query the child by user_id via its repository.

Cascade types & orphanRemoval

CascadeEffectSane usage
PERSISTpersist(parent) also persists new childrenParent-child aggregates (order → lines)
MERGEmerge(parent) also merges childrenSame aggregates
REMOVEremove(parent) deletes childrenOnly when children can't exist alone
ALLAll of the above (+ refresh, detach)True composition only

Never put CascadeType.REMOVE (or ALL) on @ManyToMany. Deleting a Course would cascade-delete its Students — entities that other courses still reference. Cascade semantics assume exclusive ownership; many-to-many is by definition shared. Cascades belong on composition relationships (Order owns its OrderLines), not associations between independent aggregates (never cascade from Order to Customer).

orphanRemoval vs cascade = REMOVE — a favorite distinction question:

  • cascade = REMOVE: when the parent is deleted, delete the children too.
  • orphanRemoval = true: when a child is disconnected from the parent (removed from the collection, or the reference nulled), delete that child — even though the parent lives on. "No parent → no right to exist."
order.getLines().remove(0);   // with orphanRemoval=true, at flush Hibernate runs:
-- delete from order_lines where id=?
// with only cascade=REMOVE: nothing is deleted; the line is merely disassociated

FetchType & the N+1 problem T1

If you remember one section from this whole course, make it this one. N+1 is the single most-asked JPA interview topic and the single most common production performance bug in Spring apps.

The fetch defaults (memorize this table)

AssociationJPA defaultWhat seniors do
@ManyToOneEAGERSet LAZY explicitly
@OneToOneEAGERSet LAZY (owning side; see 1:1 caveat)
@OneToManyLAZYKeep LAZY
@ManyToManyLAZYKeep LAZY

Rule of thumb: everything LAZY, always — then fetch eagerly per query where the use case needs it. EAGER is a global, irreversible decision; LAZY is a per-query choice. LAZY works by giving you a proxy (a runtime subclass for @ManyToOne, a special PersistentBag/PersistentSet for collections) that loads on first access — if a persistence context is still open.

N+1 demonstrated — with the actual SQL

// The innocent-looking code (Order.customer is @ManyToOne LAZY)
@Transactional(readOnly = true)
public List<OrderSummary> recentOrders() {
    return orderRepository.findTop100ByOrderByCreatedAtDesc().stream()
        .map(o -> new OrderSummary(o.getId(), o.getTotal(),
                    o.getCustomer().getName()))   // <-- proxy access, one query EACH
        .toList();
}

The SQL log (this is what you must be able to narrate in an interview):

-- 1 query for the orders:
select o1_0.id, o1_0.customer_id, o1_0.total, o1_0.created_at
from orders o1_0 order by o1_0.created_at desc fetch first 100 rows only

-- ...then N queries, one per DISTINCT customer touched:
select c1_0.id, c1_0.email, c1_0.name, c1_0.status from customers c1_0 where c1_0.id=?
select c1_0.id, c1_0.email, c1_0.name, c1_0.status from customers c1_0 where c1_0.id=?
select c1_0.id, c1_0.email, c1_0.name, c1_0.status from customers c1_0 where c1_0.id=?
-- ... up to 100 more times. 1 + N round trips.

At 1ms per round trip that's ~100ms of pure latency for a page that needed one join. With EAGER it's no better — EAGER on @ManyToOne in a list query also produces N+1 (Hibernate runs the main query, then eagerly initializes each reference with a secondary select). EAGER doesn't fix N+1; it just makes it mandatory.

LazyInitializationException — the flip side of the same coin

org.hibernate.LazyInitializationException:
  could not initialize proxy [com.shop.Customer#42] - no Session

Why: a lazy proxy needs a live persistence context (Hibernate Session) to load itself. The session is bound to the transaction. If you return an entity from a @Transactional service method and Jackson serializes it in the controller — after the transaction closed — touching a lazy field throws. The entity is detached; the proxy is a dangling pointer to a closed session.

Open Session In View — the band-aid and the debate

Spring Boot enables OSIV by default (spring.jpa.open-in-view=true, with a startup warning). OSIV keeps the persistence context open for the entire HTTP request, including view rendering / JSON serialization — so lazy loads from the controller "just work."

Why seniors typically disable it:

  • A DB connection is held for the whole request — including while you call downstream HTTP services. Under load this exhausts the Hikari pool.
  • It hides N+1: serialization silently triggers dozens of lazy loads that no one wrote or reviewed. Queries issued during rendering run outside your service transaction (auto-commit or separate tx), so you can even read inconsistent state.
  • It blurs the architecture: the web layer can do I/O the service layer never sanctioned.
# application.yml — the senior default
spring:
  jpa:
    open-in-view: false

Disabling OSIV forces the discipline that fixes N+1 for real: fetch everything the response needs inside the transaction, ideally as a DTO. The honest interview answer acknowledges the trade-off: OSIV is convenient for small CRUD apps; at scale connection-holding and hidden queries make it a liability.

Fix #1 — JOIN FETCH (JPQL)

public interface OrderRepository extends JpaRepository<Order, Long> {

    @Query("""
        select o from Order o
        join fetch o.customer
        where o.createdAt >= :since
        order by o.createdAt desc
        """)
    List<Order> findRecentWithCustomer(@Param("since") Instant since);
}

-- ONE query, customers hydrated in the same result set:
select o1_0.id, o1_0.created_at, c1_0.id, c1_0.email, c1_0.name, c1_0.status, o1_0.total
from orders o1_0
join customers c1_0 on c1_0.id=o1_0.customer_id
where o1_0.created_at>=?
order by o1_0.created_at desc

Fetch-joining a collection + pagination = in-memory pagination. join fetch o.lines multiplies rows (one row per order line), so SQL-level LIMIT would cut orders mid-collection. Hibernate therefore fetches every matching row and paginates in the JVM, logging: HHH90003004: firstResult/maxResults specified with collection fetch; applying in memory. On a large table this is an OOM waiting to happen. Fixes: paginate the parent query first and fetch collections in a second query (or with @BatchSize), or use two queries keyed by the page of ids. Also: you can't fetch-join two different collections in one query (MultipleBagFetchException with Lists).

On Hibernate 5, select distinct o ... join fetch o.lines needed distinct to deduplicate parent entities (plus hibernate.query.passdistinctthrough=false to keep it out of the SQL). Hibernate 6 deduplicates entity results automatically, so distinct is no longer required for this — a nice "I know what changed in Hibernate 6" flex.

Fix #2 — @EntityGraph

Declarative fetch plan on top of derived or existing queries — no JPQL needed:

@EntityGraph(attributePaths = {"customer"})
List<Order> findByStatus(OrderStatus status);

@EntityGraph(attributePaths = {"customer", "lines.product"})  // nested paths work
Optional<Order> findWithDetailsById(Long id);

Generates the same LEFT JOIN as a fetch join. Nice because one derived method can exist in graph and non-graph variants. Same collection-pagination caveat applies.

Fix #3 — @BatchSize / default_batch_fetch_size

When you genuinely need lazy loading but want it in chunks instead of one-by-one:

# global (recommended) — application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 50

// or per association:
@OneToMany(mappedBy = "customer")
@BatchSize(size = 50)
private List<Order> orders;

-- N+1 becomes 1 + ceil(N/50): lazy loads are grouped into IN queries
select c1_0.id, c1_0.email, c1_0.name, c1_0.status
from customers c1_0
where c1_0.id in (?, ?, ?, ?, ?, ?, ?, ?, ... ) -- up to 50 ids per round trip

This is a superb safety net: it turns catastrophic N+1s into mild ones globally, without touching queries. Many teams set default_batch_fetch_size as a baseline and use fetch joins/DTOs for hot paths.

Fix #4 — DTO projections (often the best fix)

The senior insight: most read endpoints don't need entities at all. If you're not going to modify the data, skip the persistence context entirely and select exactly the columns you need.

// (a) Constructor expression with a record
public record OrderSummary(Long id, BigDecimal total, String customerName) {}

@Query("""
    select new com.shop.dto.OrderSummary(o.id, o.total, c.name)
    from Order o join o.customer c
    where o.createdAt >= :since
    """)
List<OrderSummary> findSummaries(@Param("since") Instant since);

// (b) Interface projection — Spring Data generates the implementation
public interface OrderView {
    Long getId();
    BigDecimal getTotal();
    String getCustomerName();   // maps to alias customerName / nested path
}
List<OrderView> findByStatus(OrderStatus status);  // works on derived queries too

-- Only the needed columns; a plain join; no entities, no dirty-check overhead:
select o1_0.id, o1_0.total, c1_0.name
from orders o1_0 join customers c1_0 on c1_0.id=o1_0.customer_id
where o1_0.created_at>=?

Benefits: no N+1 possible (no proxies exist), no LazyInitializationException possible, less data over the wire, no accidental dirty-checking flush, pagination works normally. Cost: one query per read shape — which is honestly how SQL wants to be used. Prefer closed (getter-only) interface projections; nested/open projections can secretly load whole entities.

Detecting N+1 — properly

# NOT this — show-sql prints to stdout, unformatted, no params, can't be toggled per-env cleanly:
spring.jpa.show-sql: true       # fine for a scratch demo only

# THIS — real logging through your log framework:
logging:
  level:
    org.hibernate.SQL: DEBUG                          # the statements
    org.hibernate.orm.jdbc.bind: TRACE                # bound parameters (Hibernate 6)
spring:
  jpa:
    properties:
      hibernate:
        format_sql: true
  • Hibernate statistics: hibernate.generate_statistics=true logs per-transaction query counts and times — a sudden "executed 147 JDBC statements" is your smoking gun.
  • Hypersistence Utils (Vlad Mihalcea): assert query counts in tests — e.g. wrap datasource with a query-count listener and fail the build if an endpoint exceeds N statements. Saying "we made N+1 a failing test, not a code-review hope" is a strong senior line.
  • APM traces (Datadog/New Relic) showing dozens of identical selects per request are the production tell.

@Transactional, deeply T1

Where it belongs and how it actually works

@Transactional belongs on the service layer — the unit of business work. Not on controllers (the web layer shouldn't own DB semantics; keep connections short) and not usually on repositories (Spring Data methods already run in their own short transactions; you want one transaction spanning the whole use case, not five tiny ones with no atomicity between them).

Mechanics: Spring wraps your bean in an AOP proxy. A call from outside hits the proxy, which opens (or joins) a transaction, obtains a connection, binds the JPA EntityManager to the thread, invokes your method, then commits (flushing the persistence context) or rolls back, and returns the connection to the pool.

// Conceptually, the proxy does:
TransactionStatus tx = txManager.getTransaction(definition);
try {
    Object result = target.actualMethod(args);   // your code
    txManager.commit(tx);                        // flush + COMMIT here
    return result;
} catch (RuntimeException | Error e) {
    txManager.rollback(tx);                      // only unchecked, by default
    throw e;
}

The self-invocation trap (proxy mechanics)

@Service
public class OrderService {

    public void importOrders(List<OrderDto> dtos) {
        for (OrderDto dto : dtos) {
            this.saveOne(dto);   // TRAP: 'this' is the raw object, NOT the proxy
        }                        // @Transactional on saveOne is SILENTLY ignored
    }

    @Transactional
    public void saveOne(OrderDto dto) { ... }
}

Why: the transaction logic lives in the proxy that wraps your bean. Internal calls via this bypass the proxy entirely, so no transaction starts — no error, no warning, just missing atomicity discovered in an incident. Same root cause explains why @Transactional on private methods does nothing: the JDK/CGLIB proxy can't intercept (or even see) them, and Spring ignores the annotation.

Fixes, in order of preference: (1) restructure — move the transactional method to another bean so the call crosses a proxy boundary; (2) inject the bean into itself (ObjectProvider<OrderService> or ApplicationContext.getBean) and call through it; (3) TransactionTemplate for programmatic control. (AspectJ weaving also solves it, but almost nobody deploys that.)

Propagation levels — with scenarios

PropagationBehaviorScenario
REQUIRED (default)Join the caller's tx; start one if none exists99% of service methods
REQUIRES_NEWSuspend the caller's tx; run in a brand-new one that commits/rolls back independentlyAudit log that must survive business rollback
NESTEDJDBC savepoint within the same tx; inner rollback rewinds to the savepoint, outer commit commits all"Try this sub-step; on failure keep the rest"
SUPPORTSJoin if present, else run non-transactionallyRead helpers callable from anywhere
NOT_SUPPORTEDSuspend any tx; run without oneLong external call mid-flow (release the connection)
MANDATORYThrow if no tx existsEnforce "must be called inside a tx"
NEVERThrow if a tx existsGuard genuinely non-transactional code
 
// The canonical REQUIRES_NEW example: the audit row must persist even if the order fails
@Service
public class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void recordAttempt(String action, String detail) {
        auditRepository.save(new AuditEntry(action, detail, Instant.now()));
    }   // commits HERE, immediately, regardless of the caller's fate
}

@Service
public class OrderService {
    @Transactional
    public void placeOrder(PlaceOrderCommand cmd) {
        auditService.recordAttempt("PLACE_ORDER", cmd.toString()); // separate bean → proxy works
        // ... business logic that may throw → outer tx rolls back, audit stays
    }
}

REQUIRES_NEW takes a second connection from the pool while the outer one sits suspended. With maximumPoolSize=10 and 10 concurrent requests each needing an inner tx, everyone holds one connection and waits for a second — a pool-exhaustion deadlock. Use REQUIRES_NEW sparingly and know your pool math. Also note NESTED works with JDBC savepoints via DataSourceTransactionManager; JpaTransactionManager does not support it for JPA — another reason REQUIRES_NEW gets (over)used.

Rollback rules — checked vs unchecked

Default: rollback on RuntimeException and Error only. Checked exceptions do NOT roll back — a historic EJB-inherited default ("checked = anticipated business condition, maybe you handled it"). This surprises everyone once:

@Transactional
public void process() throws IOException {
    repo.save(entity);
    throw new IOException("boom");   // tx COMMITS — the save is persisted!
}

// Fix: be explicit
@Transactional(rollbackFor = Exception.class)
public void process() throws IOException { ... }

Related trap: catching an exception inside the method doesn't always save the commit. If an inner REQUIRED method threw a RuntimeException through its proxy, the shared transaction is already marked rollback-only; when the outer method tries to commit you get UnexpectedRollbackException. You can't "catch and continue" a poisoned transaction.

readOnly = true — what it actually does

@Transactional(readOnly = true)
public CustomerDto getCustomer(Long id) { ... }
  • Sets Hibernate's flush mode to MANUAL: no dirty-check flush at commit — big win when you loaded hundreds of entities (no snapshot comparison walk).
  • Hibernate can skip keeping detached-state snapshots for loaded entities, reducing memory.
  • Hints the JDBC driver/DB (Connection.setReadOnly(true)); on Postgres the tx runs SET TRANSACTION READ ONLY-equivalent, and read-only transactions can be routed to replicas by proxies/drivers configured for it.
  • It is an optimization + intent marker, not a security boundary: it doesn't hard-prevent every write path, so don't present it as one.

Isolation

@Transactional(isolation = Isolation.REPEATABLE_READ) simply asks the database for that isolation level on the transaction's connection — Spring implements nothing itself. Postgres defaults to READ_COMMITTED; its REPEATABLE_READ is snapshot-based and stronger than the ANSI minimum. Full treatment (anomalies, SSI serializable) lives in tutorial 14 — in a JPA interview it's enough to say isolation delegates to the DB and that you mostly combine READ_COMMITTED with optimistic locking.

The @Async pitfall

@Transactional
public void placeOrder(...) {
    orderRepository.save(order);
    emailService.sendConfirmationAsync(order);   // @Async method
}

Transaction context is bound to the thread (a ThreadLocal). An @Async method runs on an executor thread: it does not join the caller's transaction; annotating it @Transactional starts a new, independent one. Worse timing bug: the async task may run before the caller commits — it queries for the order and finds nothing (READ_COMMITTED can't see uncommitted rows). Fixes: fire the async work after commit via @TransactionalEventListener(phase = AFTER_COMMIT) or TransactionSynchronization.afterCommit, or use the transactional-outbox pattern for reliability.

The persistence context & entity lifecycle T1

The persistence context (Hibernate's Session, JPA's EntityManager scope) is a unit-of-work: an identity map of every entity loaded or persisted in the current transaction, plus a snapshot of each one's original state. Nothing in JPA makes sense until this clicks.

First-level cache: find twice = one query

@Transactional
public void demo(Long id) {
    Customer a = customerRepository.findById(id).orElseThrow(); // SELECT ... where id=?
    Customer b = customerRepository.findById(id).orElseThrow(); // NO SQL — served from the context
    assert a == b;   // same Java object — the identity map guarantees it
}

Within one persistence context there is exactly one object per (entity type, id). This is per-transaction and never shared between requests — don't confuse it with the second-level cache. (JPQL/derived queries always hit the DB, but returned rows resolve to the already-managed instances.)

Dirty checking — the "update without save()" surprise

@Transactional
public void rename(Long id, String newName) {
    Customer c = customerRepository.findById(id).orElseThrow();
    c.setName(newName);
    // NO save() call. And yet, at commit:
}
-- update customers set email=?, name=?, status=? where id=?

At flush, Hibernate compares each managed entity against its load-time snapshot and emits UPDATEs for anything that changed. This is the single biggest mental-model shift from Prisma: a setter on a managed entity is a write. Consequences interviewers probe:

  • save() on an already-managed entity is a no-op ritual (it just returns the same managed instance) — harmless but reveals the author doesn't know the model.
  • The dangerous inverse: mutating a managed entity "temporarily" (say, for display math) inside a transaction persists that mutation. Never mutate managed entities unless you mean it; map to DTOs for presentation.
  • By default Hibernate updates all columns (as shown above — precompiled statement caching); @DynamicUpdate switches to changed-columns-only, useful for wide rows/lock contention, at the cost of statement-cache reuse.

Flush timing

Flush = write pending changes as SQL (still inside the tx; not commit). With FlushModeType.AUTO, flushing happens:

  1. Before a query whose result could be affected by pending changes (Hibernate 6 compares the query's tables against dirty entities' tables) — so a JPQL select sees your in-transaction inserts;
  2. At commit;
  3. On explicit em.flush() / saveAndFlush() — mostly useful to force constraint violations to surface at a known line, or before native queries Hibernate can't analyze.

persist vs merge, and the detached-entity error

em.persist(entity);   // entity itself becomes managed; INSERT scheduled; entity must be NEW
Customer managed = em.merge(entity);  // COPIES state onto a managed instance (loading it if needed)
                                       // and returns THAT; the argument stays detached!

Two classic failures: (1) persist() on a detached entity (id already set from an earlier tx or a client payload) → EntityExistsException / Hibernate's "detached entity passed to persist", surfaced by Spring as InvalidDataAccessApiUsageException. Use merge — or better, load-then-mutate. (2) Continuing to use the argument of merge() instead of its return value — your subsequent setters silently go nowhere.

Spring Data's save() hides this switch: SimpleJpaRepository.save calls persist if the entity is new (id null / version null / Persistable.isNew()), else merge. That's why saving a client-supplied object with an id triggers a surprise SELECT before the UPDATE — merge must load current state to copy onto. The cleaner update pattern in a service: findById → mutate → let dirty checking flush.

Spring Data repositories T2

The hierarchy

Repository<T, ID>                    // marker interface — no methods
  └── CrudRepository<T, ID>          // save, saveAll, findById, findAll, delete, count...
        └── PagingAndSortingRepository<T, ID>   // findAll(Pageable), findAll(Sort)
              └── JpaRepository<T, ID>          // JPA extras: flush, saveAndFlush,
                                                // deleteAllInBatch, getReferenceById, List returns

You write the interface; at startup Spring Data creates a proxy backed by SimpleJpaRepository. Extend JpaRepository in app code; extending a narrower interface (or annotating methods onto Repository directly) deliberately restricts the API — e.g. a repository that can't delete.

getReferenceById vs findById — proxy!

Customer c1 = customerRepository.findById(id).orElseThrow();
// SELECT immediately; returns the real entity or empty Optional

// SQL fires only if you touch a non-id field; if the row doesn't exist,
// EntityNotFoundException throws at first ACCESS, not at the call site.

The killer use case — setting an FK without a pointless SELECT:

order.setCustomer(customerRepository.getReferenceById(customerId));

-- generated SQL: just the INSERT into orders with customer_id=? — zero customer SELECTs

Mechanics: at bootstrap, Spring Data parses the method name into a predicate tree (property references validated against the entity model — typos fail at startup, which is a feature) and compiles it to JPQL. When they stop being a good idea: findByStatusInAndCreatedAtBetweenAndCustomerRegionAndTotalGreaterThanOrderByCreatedAtDesc(...) — unreadable, and each new filter combination is a new method. Rule of thumb: 1–3 conditions → derived; more → @Query; dynamic combinations → Specifications.

@Query: JPQL vs native

// JPQL — portable, entity-based, named parameters
@Query("select o from Order o where o.customer.email = :email and o.total > :min")
List<Order> findBigOrdersFor(@Param("email") String email, @Param("min") BigDecimal min);

// Native — when you need SQL that JPQL can't express
@Query(value = """
    select customer_id, total, rank() over (partition by customer_id order by total desc) rnk
    from orders
    """, nativeQuery = true)
List<TopOrderView> rankOrdersPerCustomer();

Reach for nativeQuery = true when you need: window functions (rank/lag/lead), CTEs/recursive queries, Postgres-specific features (jsonb operators, ON CONFLICT, arrays, full-text search), or hand-tuned performance SQL. Costs: no entity-model validation, dialect lock-in, and pagination requires Spring to derive/accept a count query. Always prefer named parameters (:email) over positional (?1) — refactoring-safe and self-documenting.

@Modifying — bulk update/delete bypasses the persistence context

@Modifying(clearAutomatically = true)   // evict stale managed entities after the bulk op
@Query("update Order o set o.status = :to where o.status = :from and o.createdAt < :cutoff")
int expireStale(@Param("from") OrderStatus from, @Param("to") OrderStatus to,
                @Param("cutoff") Instant cutoff);

-- ONE statement, no matter how many rows:
update orders set status=? where status=? and created_at<?

Bulk JPQL goes straight to the database, skipping the persistence context: managed entities already loaded still hold the OLD values, no @Version increments, no lifecycle callbacks, no cascades. If the same transaction reads those entities afterward it sees stale state. clearAutomatically = true clears the context after the update (detaching everything — unsaved changes to other entities are lost, so consider flushAutomatically = true too). Bulk ops need @Modifying and a transaction; forgetting @Modifying throws at runtime ("Update/delete queries cannot be typed").

Keyset (seek) pagination — for deep pages

OFFSET 199980 forces Postgres to produce and discard 199,980 rows before returning 20 — page latency grows linearly with depth. Keyset pagination seeks directly using the last-seen sort key:

-- instead of OFFSET, remember the last row's (created_at, id) and:
select * from orders
where status = ?
  and (created_at, id) < (?, ?)          -- Postgres row-value comparison; id breaks ties
order by created_at desc, id desc
fetch first 20 rows only                   -- constant time at any depth (index on (status, created_at, id))

Trade-offs: no jump-to-page-N, needs a deterministic unique sort key. Spring Data supports this natively since 3.1 via ScrollPosition.keyset() and Window<T> — worth name-dropping.

Sorting on API-supplied fields: whitelist them. Sort.by(request.getParam("sort")) on arbitrary properties invites errors and (with native queries via string concat) injection. Map allowed sort keys explicitly.

Optimistic & pessimistic locking T2

Optimistic — @Version

@Entity
public class Product {
    @Id private Long id;
    private int stock;

    @Version
    private long version;   // Hibernate manages it — never set it yourself
}

-- every UPDATE carries the version in the WHERE clause and increments it:
update products set stock=?, version=? where id=? and version=?
-- if another tx committed first, version no longer matches → 0 rows updated →
-- Hibernate throws OptimisticLockException, Spring wraps it as
-- ObjectOptimisticLockingFailureException → the whole tx rolls back

Handling: catch outside the transactional method and retry the whole use case (re-read, re-apply, re-commit):

@Retryable(retryFor = ObjectOptimisticLockingFailureException.class,
           maxAttempts = 3, backoff = @Backoff(delay = 50))   // spring-retry; or a manual loop
public void decrementStock(Long productId, int qty) {
    txTemplate.executeWithoutResult(tx -> {
        Product p = productRepository.findById(productId).orElseThrow();
        p.decrement(qty);      // dirty checking flushes with the version check
    });
}

Pessimistic — @Lock

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select p from Product p where p.id = :id")
Optional<Product> findByIdForUpdate(@Param("id") Long id);

-- Postgres SQL:
select p1_0.id, p1_0.stock, p1_0.version from products p1_0 where p1_0.id=? for update
-- PESSIMISTIC_READ would generate FOR SHARE;
-- other writers block on this row until our tx commits or rolls back

The row lock lives for the rest of the transaction. Add a timeout so contention fails fast instead of piling up: @QueryHints(@QueryHint(name = "jakarta.persistence.lock.timeout", value = "3000")) (Postgres also supports FOR UPDATE NOWAIT / SKIP LOCKED via native SQL — SKIP LOCKED is the classic job-queue trick).

Strong interview answer: "Default to optimistic — it costs nothing until a conflict happens, and conflicts should be rare in a well-partitioned domain. Go pessimistic for hot rows where retry storms would be worse than short blocking — inventory counters, balances. And keep pessimistic transactions tiny, because that row lock also pins a pool connection." Bonus point: for pure counters, a single atomic UPDATE ... SET stock = stock - ? HikariCP connection pooling T2

Opening a Postgres connection is expensive: TCP handshake, TLS, authentication, and a forked backend process per connection on the server — several milliseconds plus real server memory. A pool keeps warm connections and leases them out. HikariCP is Spring Boot's default and effectively the industry standard.

# application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10        # default 10 — usually about right; see formula
      minimum-idle: 10             # fixed-size pool: simplest, most predictable
      connection-timeout: 30000    # ms to wait for a lease before throwing
      max-lifetime: 1800000        # retire connections before infra (LB/DB) kills them
      leak-detection-threshold: 60000  # log a stack trace if a lease outlives 60s

Small is beautiful

Counter-intuitive but well-documented (HikariCP's own wiki): a smaller pool usually delivers higher throughput. Postgres executes one task per backend at a time per core; hundreds of connections mean context-switching, lock thrash, and memory pressure. Starting formula:

pool_size ≈ (cpu_cores × 2) + effective_spindle_count
-- e.g. a 4-core DB server → ~9–10 connections. Then measure.
-- And it's PER INSTANCE: 10 app pods × pool 10 = 100 server connections.

Pool exhaustion — symptoms and the classic anti-pattern

java.sql.SQLTransientConnectionException:
  HikariPool-1 - Connection is not available, request timed out after 30000ms

Symptoms: healthy CPU but requests stall exactly connectionTimeout ms then 500; hikaricp.connections.pending climbing in metrics; thread dumps full of threads parked in HikariPool.getConnection. Root cause is almost always connections held too long, and the number-one offender is:

// ANTI-PATTERN: slow remote I/O inside a transaction
@Transactional
public void placeOrder(PlaceOrderCommand cmd) {
    Order order = createOrder(cmd);                 // fast DB work
    paymentClient.charge(cmd.card(), order.total()); // 2–8s HTTP call — CONNECTION HELD THE WHOLE TIME
    order.markPaid();
}   // 10 concurrent checkouts with pool=10 → entire app is frozen for everything

Fix: keep transactions short and DB-only. Split into tx #1 (create PENDING order, commit — connection returned), then the HTTP call with no transaction, then tx #2 (mark paid / compensate). Related: OSIV (above) is this same disease applied to every request; and remember each REQUIRES_NEW doubles a request's connection footprint.

Second-level cache awareness T3

The L2 cache is a shared, cross-transaction cache of entity data by id (backed by Ehcache/Infinispan/etc.), sitting between the per-transaction first-level cache and the DB. Enable per entity with @Cacheable @org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE); find() then checks L1 → L2 → DB.

Why it's rarely the right tool today: it only helps id-based lookups (queries need the separate, invalidation-happy query cache); in a multi-node deployment it's stale unless you run a distributed cache, at which point most teams prefer an explicit Redis/Caffeine cache at the service layer (@Cacheable from Spring Cache) with visible TTLs and keys; and any non-Hibernate writer (bulk JPQL, native SQL, another service) silently invalidates nothing. Legit niche: small, hot, read-mostly reference data (countries, currencies, feature flags). Interview line: "I know it exists and how it's layered; I've almost always preferred service-level caching because the invalidation story is explicit."

Dynamic queries: Specifications T3

For search screens with optional filters, derived methods explode combinatorially. Specification<T> wraps the JPA Criteria API into composable predicates:

public interface OrderRepository extends JpaRepository<Order, Long>,
                                         JpaSpecificationExecutor<Order> {}

public final class OrderSpecs {
    public static Specification<Order> hasStatus(OrderStatus s) {
        return (root, query, cb) -> s == null ? null : cb.equal(root.get("status"), s);
    }   // returning null = "no constraint" — makes optional filters trivial
    public static Specification<Order> minTotal(BigDecimal min) {
        return (root, query, cb) -> min == null ? null : cb.ge(root.get("total"), min);
    }
}

// service — compose only what the request supplied:
Page<Order> results = orderRepository.findAll(
    OrderSpecs.hasStatus(filter.status()).and(OrderSpecs.minTotal(filter.minTotal())),
    pageable);
-- SQL contains only the non-null predicates:
-- select ... from orders o1_0 where o1_0.status=? and o1_0.total>=? order by ... offset ...

Alternatives worth naming: QueryDSL (typesafe generated metamodel, nicer syntax, extra build step) and hand-built JPQL with Query by Example for trivial cases.

Schema migrations with Flyway T3

Coming from Prisma Migrate, Flyway will feel familiar: versioned SQL scripts, applied in order, tracked in a history table (flyway_schema_history) with checksums.

src/main/resources/db/migration/
  V1__create_customers.sql
  V2__create_orders.sql
  V3__add_orders_status_index.sql
-- naming: V<version>__<description>.sql (double underscore). Boot runs pending ones at startup.

-- V2__create_orders.sql
create sequence order_seq start with 1 increment by 50;   -- matches allocationSize!

create table orders (
    id          bigint primary key,
    customer_id bigint not null references customers(id),
    status      varchar(32) not null,
    total       numeric(12,2) not null,
    version     bigint not null default 0,
    created_at  timestamptz not null default now()
);
create index idx_orders_customer on orders(customer_id);  -- FKs are NOT auto-indexed in PG

Iron rules:

  • Never edit an applied migration. Flyway checksums every script; editing one that already ran fails validation on every environment that ran the original. Fix mistakes with a new migration (V4 alters what V3 got wrong).
  • spring.jpa.hibernate.ddl-auto=validate in production — never update. update lets Hibernate mutate a production schema on boot: it can't rename (it adds a new column and strands the old), never drops, ignores your index strategy, and couples deploys to uncontrolled DDL. validate checks that entities match the Flyway-owned schema and fails fast on drift. (create-drop is for throwaway tests only.)
  • Migrations are forward-only in the free tier — design them to be safe (additive first; destructive changes in a later release after code stops reading the old column — the expand/contract pattern).

Liquibase is the main alternative (XML/YAML changelogs, rollback support). Interviews rarely care which; they care that the schema is versioned in the repo and the ORM does not own it.

Interview Q&A

What's the difference between JPA, Hibernate, and Spring Data JPA?

JPA is the specification — annotations (@Entity, @Id), interfaces (EntityManager), and JPQL semantics, but no code that runs. Hibernate is the most common implementation: it does the actual SQL generation, dirty checking, caching, and proxying. Spring Data JPA sits on top of JPA: it generates repository implementations from interfaces, adds derived queries, pagination, auditing, and translates JPA exceptions into Spring's DataAccessException hierarchy. So a findById call goes Spring Data proxy → EntityManager (JPA API) → Hibernate → JDBC → Postgres.

Why is SEQUENCE preferred over IDENTITY for ID generation on Postgres?

With IDENTITY, the id is assigned by the database during the INSERT, so Hibernate must execute each INSERT immediately at persist() to learn the id it needs for the persistence context — which makes JDBC batch inserts impossible for that entity. With SEQUENCE, Hibernate fetches ids up front via select nextval(...), and with allocationSize=50 (pooled optimizer) one round trip yields 50 ids; INSERTs are deferred to flush and sent as JDBC batches. For bulk-insert workloads that's an order-of-magnitude difference. Caveat: allocationSize must match the sequence's INCREMENT BY in the DDL.

How should you implement equals() and hashCode() on a JPA entity, and why is it tricky?

The trap is that a generated id is null before persist, and Hibernate proxies are runtime subclasses. If you hash on the id, the hashCode changes when the id is assigned, corrupting any HashSet the entity already sits in. Two accepted patterns: (1) use an immutable natural key (email, ISBN) if one exists; (2) id-based equals using instanceof (not getClass()==, because of proxies), returning true only when both ids are non-null and equal, paired with a constant hashCode (e.g. getClass().hashCode()) so it never changes across the lifecycle. Also access the other object's id through its getter, since a proxy's fields are null. And don't use Lombok's @EqualsAndHashCode on entities.

Explain owning side vs inverse side in a bidirectional @OneToMany.

The foreign key column lives on the many side, so the @ManyToOne field is the owning side — Hibernate writes the FK based only on it. The @OneToMany(mappedBy=...) side is the inverse/mirror side; changes to it alone are ignored at flush. The classic bug: parent.getChildren().add(child) without child.setParent(parent) — the FK column stays null. The fix is helper methods (addChild/removeChild) that keep both sides in sync. I'd add that a unidirectional @ManyToOne plus a repository query is often the better model — bidirectional collections should exist only when bounded and traversed together.

What is the N+1 problem? Walk me through the SQL.

You run one query returning N rows, then accessing a lazy (or list-EAGER) association on each row triggers one additional query per row: 1 + N round trips. Example: select ... from orders (1 query), then o.getCustomer().getName() in a loop fires select ... from customers where id=? up to N times. At 1ms per round trip, a 500-row page costs 500ms of pure latency. It's invisible in dev (small data, local DB) and devastating in prod. Detection: hibernate SQL logging, generate_statistics statement counts, query-count assertions in tests (Hypersistence Utils), APM traces showing repeated identical selects.

What fixes exist for N+1, and when do you use which?

(1) join fetch in JPQL / @EntityGraph — one query with a join, best for a specific use case that needs the association; beware fetch-joining collections with pagination (in-memory paging) and multiple bag fetches. (2) @BatchSize/default_batch_fetch_size — turns N lazy loads into ceil(N/size) IN-queries; excellent as a global safety net. (3) DTO projections (constructor expressions or interface projections) — select exactly the needed columns; no proxies exist, so N+1 and LazyInitializationException are structurally impossible; usually the best fix for read endpoints. (4) For to-one paths, sometimes just getReferenceById/redesign. My default: DTOs for reads, fetch joins for aggregate loads, batch size as insurance.

Why does LazyInitializationException happen, and what's your position on Open Session In View?

A lazy proxy needs a live persistence context to initialize; the context closes with the transaction. If serialization (Jackson) touches a lazy field after the service method returned, there's no session — exception. OSIV "fixes" it by keeping the context open for the whole HTTP request, and Boot enables it by default. I disable it in serious services: it holds a pooled connection across the entire request (including downstream HTTP calls — pool exhaustion under load), it hides unplanned queries triggered by serialization, and those queries run outside the service transaction. Disabling it forces you to load everything the response needs inside the transaction, preferably as DTOs — which is where you want to end up anyway. Fair counterpoint: for a small internal CRUD app OSIV is a pragmatic convenience.

Why does @Transactional on a private method, or a method called via this, do nothing?

Spring implements @Transactional with an AOP proxy that wraps the bean. Transaction begin/commit only happens when a call passes through the proxy. Self-invocation (this.method()) is a plain Java call on the target object — no proxy, no transaction, silently. Private methods can't be overridden/intercepted by a CGLIB subclass proxy at all, so the annotation is ignored. Fixes: move the method to another bean (best — usually a design smell anyway), self-inject the proxy, or use TransactionTemplate programmatically.

Explain REQUIRED vs REQUIRES_NEW vs NESTED with a concrete scenario.

REQUIRED (default): join the caller's transaction — one atomic unit; right for normal service composition. REQUIRES_NEW: suspend the caller and run an independent transaction — the audit-log case, where the audit row must commit even if the business operation rolls back; cost: a second pool connection held while the outer sits suspended, so it's a pool-exhaustion risk. NESTED: a savepoint inside the same transaction — an inner failure rolls back to the savepoint without dooming the outer; but it needs JDBC savepoint support and doesn't work with JpaTransactionManager for JPA, so in Hibernate apps you rarely see it. Also worth naming MANDATORY (assert a tx exists) and NOT_SUPPORTED (suspend around non-DB work).

A @Transactional method throws a checked exception. What happens?

The transaction commits. Spring's default rollback rule covers only RuntimeException and Error — checked exceptions are treated as anticipated business outcomes (an EJB-era default). So work done before the throw is persisted, which is almost never what the author intended. Fix: @Transactional(rollbackFor = Exception.class), or design services to throw unchecked exceptions. Related: if an inner REQUIRED method already threw a RuntimeException through its proxy, the shared tx is marked rollback-only, and the outer method's catch-and-commit attempt throws UnexpectedRollbackException.

What does @Transactional(readOnly = true) actually do?

Three things: (1) sets Hibernate's flush mode to MANUAL, so no dirty-checking flush occurs at commit — a real CPU/memory win when many entities are loaded, since Hibernate can also skip retaining detachment snapshots; (2) calls Connection.setReadOnly(true), a driver/DB hint — Postgres can enforce read-only mode and some infrastructures route read-only transactions to replicas; (3) documents intent. What it's not: a security boundary or a guarantee — it's an optimization and a contract marker.

Explain the persistence context and the first-level cache.

It's a unit-of-work bound to the transaction: an identity map from (entity class, id) to the single managed instance, plus load-time snapshots for dirty checking. Consequences: two findById calls for the same id in one transaction issue one SELECT and return the same object reference; changes to managed entities are auto-detected and flushed as UPDATEs; JPQL queries still hit the DB but resolve rows to the already-managed instances. It's per-transaction and private — not to be confused with the shared second-level cache.

Why did an UPDATE happen when the code never calls save()?

Dirty checking. Any entity loaded inside a transaction is managed; at flush Hibernate diffs each managed entity against its load-time snapshot and generates UPDATEs for changes. So findById(...).setName(...) inside @Transactional persists without save() — save() on a managed entity is redundant. The flip side is the real danger: mutating a managed entity "just for display" writes to the database. The discipline is: only mutate managed entities when you intend a write, and map to DTOs for everything else.

persist() vs merge() — and what does Spring Data's save() do?

persist() makes the passed instance itself managed and schedules an INSERT; it fails on detached entities ("detached entity passed to persist"). merge() copies the state of a (possibly detached) instance onto a managed instance — loading it with a SELECT if it isn't in the context — and returns that managed copy; the argument stays detached, so you must keep using the return value. SimpleJpaRepository.save() picks between them: persist if "new" (null id/version, or Persistable.isNew()), else merge — which is why saving a client-supplied object with an id costs a hidden SELECT before the UPDATE. Cleaner update pattern: load, mutate, let dirty checking flush.

When exactly does Hibernate flush?

With AUTO flush mode: before executing a query whose result could be affected by pending in-memory changes (Hibernate 6 compares the query's table spaces with dirty entities' tables), at transaction commit, and on explicit flush(). Flush writes SQL but does not commit — the changes are still invisible to other transactions and still rollbackable. Explicit flush()/saveAndFlush() is useful to surface constraint violations at a known point or before native queries whose table usage Hibernate can't infer. Bulk @Modifying queries do not pass through flush semantics — they hit the DB directly.

What's the difference between findById and getReferenceById?

findById executes a SELECT immediately and returns Optional with the real entity. getReferenceById returns a lazy proxy carrying only the id, with no SQL at all; the SELECT happens on first access to a non-id field, and a missing row throws EntityNotFoundException at that access point, not at the call. The canonical use is setting foreign keys without a wasted round trip: order.setCustomer(repo.getReferenceById(id)) generates only the INSERT with customer_id bound. Use findById when you need the data or need to validate existence now.

When do you switch from derived query methods to @Query, and to native SQL?

Derived methods for 1–3 simple conditions — they're validated against the entity model at startup, which catches typos early. Beyond that, method names become unreadable (findByAAndBOrCInOrderBy...) and each filter combination is a new method — switch to @Query JPQL with named parameters. Go native when the SQL exceeds JPQL: window functions, CTEs/recursive queries, Postgres jsonb/array operators, ON CONFLICT upserts, FOR UPDATE SKIP LOCKED, or hand-optimized queries. Costs of native: no model validation, dialect coupling, and count queries for pagination need care. For dynamic filter combinations, Specifications or QueryDSL instead of a method explosion.

What are the pitfalls of @Modifying bulk updates?

A bulk JPQL UPDATE/DELETE compiles to one SQL statement that goes straight to the database, bypassing the persistence context entirely: already-loaded managed entities keep stale values, @Version columns don't increment, lifecycle callbacks and auditing listeners don't fire, cascades and orphanRemoval don't apply. If the same transaction reads affected entities afterward it sees the old state from the L1 cache. Mitigations: @Modifying(clearAutomatically = true) to evict the context after (plus flushAutomatically = true so pending changes aren't lost), or run bulk ops in dedicated transactions. The trade is worth it: one statement for a million rows instead of loading them all.

Page vs Slice — what's the cost difference?

Page runs two statements: the data query with LIMIT/OFFSET and a count(*) with the same predicate — on large tables the count can cost more than the page itself. Slice skips the count by fetching size + 1 rows and exposing only hasNext() — ideal for infinite scroll. I'd also volunteer offset pagination's deep-page problem: OFFSET 200000 makes Postgres produce and discard 200k rows, so latency grows with depth; keyset/seek pagination (WHERE (created_at, id) < (?, ?) ORDER BY ... LIMIT 20 on an index) is constant-time at any depth, and Spring Data 3.1+ supports it via ScrollPosition.keyset()/Window.

How does optimistic locking with @Version work mechanically?

Hibernate adds the version to every UPDATE's WHERE clause and increments it: update ... set ..., version=? where id=? and version=?. If a concurrent transaction committed first, the version no longer matches, zero rows are affected, and Hibernate throws OptimisticLockException (Spring: ObjectOptimisticLockingFailureException), rolling back the transaction. Handling: retry the entire use case — re-read, re-apply logic, re-commit — from outside the transactional boundary (spring-retry or a loop), or surface HTTP 409 to the client for user-mediated conflicts. The version even survives detachment, which is how you protect web edit forms and ETag-based APIs against lost updates.

When do you choose pessimistic locking, and what SQL does it generate on Postgres?

@Lock(PESSIMISTIC_WRITE) on a repository query appends FOR UPDATE to the SELECT (PESSIMISTIC_READ → FOR SHARE), taking a row lock held until commit — concurrent writers block. Choose it for hot rows where conflicts are the norm and optimistic retry storms would waste more than brief blocking: inventory decrements, account balances, seat reservations. Rules of use: keep the transaction tiny (the lock also pins a pool connection), set jakarta.persistence.lock.timeout so contention fails fast, and lock in consistent order to avoid deadlocks. For queue-like workloads, native FOR UPDATE SKIP LOCKED lets N workers claim disjoint rows. For simple counters, a single atomic conditional UPDATE beats locking entirely.

How do you size a HikariCP pool, and what are the symptoms of exhaustion?

Small is beautiful: Postgres runs one backend process per connection, so throughput peaks around (cores × 2) + spindles connections on the DB server — roughly 10 for a small instance — then degrades with more due to context switching and contention. Remember it's per app instance: 10 pods × 10 = 100 server connections. Exhaustion symptoms: requests stall for exactly connectionTimeout then throw SQLTransientConnectionException: connection is not available; pending-connection metrics climb; thread dumps park in getConnection. Root cause is nearly always long-held connections: slow HTTP calls inside @Transactional, OSIV holding connections through rendering, REQUIRES_NEW doubling footprints. leakDetectionThreshold logs the stack of any lease held too long — the fastest way to find the offender.

Why is putting a remote HTTP call inside a @Transactional method an anti-pattern?

The transaction holds a pooled DB connection for its whole duration. A 5-second payment API call inside the transaction means 5 seconds of a scarce connection doing nothing; with pool size 10, ten concurrent checkouts freeze the entire application — every other endpoint times out waiting for connections. It also stretches lock hold times and makes the DB work (locks, possible bloat from long-running transactions) hostage to third-party latency. Fix: shorten transactions to DB-only work — commit a PENDING state, make the remote call outside any transaction, then a second transaction records the outcome, with compensation/outbox patterns for reliability.

Why should ddl-auto=update never run in production, and what replaces it?

update hands schema evolution to Hibernate's diff heuristics at boot time: it can't rename anything (a renamed field becomes a new column, stranding the old data), never drops columns or constraints, knows nothing about your indexing strategy or data backfills, and applies DDL outside any review or rollback process. The replacement is versioned migrations (Flyway/Liquibase) owned by the team and reviewed like code, with ddl-auto=validate so the app fails fast if entities drift from the real schema. Corollary rules: never edit an applied migration (checksum validation breaks everywhere it already ran — write a new one), and make changes expand/contract-safe for zero-downtime deploys.

Why doesn't @Transactional propagate into an @Async method, and how do you run work "after the transaction commits"?

Spring binds the transaction (and the EntityManager) to the current thread via ThreadLocals. @Async executes on a different executor thread that has none of that context — it won't join the caller's transaction, and @Transactional on it starts a brand-new one. There's also a race: the async task may run before the caller commits, so its reads can't see the caller's uncommitted data. The clean pattern is to publish an application event and handle it with @TransactionalEventListener(phase = AFTER_COMMIT) (optionally @Async on the listener), so side effects fire only for committed state; for at-least-once delivery guarantees, a transactional outbox table polled by a relay is the production-grade answer.

You fetch-join a collection and add a Pageable. What happens?

Hibernate cannot apply SQL LIMIT/OFFSET, because the join multiplies rows (one per child) and a row-level limit would truncate a parent's collection mid-way. So it fetches the entire result set and paginates in memory, logging HHH90003004: firstResult/maxResults specified with collection fetch; applying in memory. On a big table that's a latency and OOM hazard. Solutions: paginate the parent ids first (query 1: ids with LIMIT; query 2: fetch-join those ids), rely on @BatchSize for children after a paginated parent query, or project to DTOs and assemble. Related: fetch-joining two List collections at once throws MultipleBagFetchException; and on Hibernate 5 you needed distinct to dedupe parents, which Hibernate 6 does automatically.

Hands-on exercises

Exercise 1 — Model the aggregate, prove the owning side

Create Customer (id: SEQUENCE with allocationSize 50, unique email) and Order (LAZY @ManyToOne to customer, status enum as STRING, total). Make the association bidirectional with sync helper methods. Then write a test that adds an order only to customer.getOrders() (without setting order.customer), saves, clears the context, and reloads — observe the FK is null (or the insert fails on the not-null constraint). Fix it with the helper method.

Solution 
@Entity @Table(name = "customers")
public class Customer {
    @Id @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "customer_seq")
    @SequenceGenerator(name = "customer_seq", allocationSize = 50)
    private Long id;

    @Column(nullable = false, unique = true) private String email;

    @OneToMany(mappedBy = "customer", cascade = {CascadeType.PERSIST, CascadeType.MERGE})
    private List<Order> orders = new ArrayList<>();

    public void addOrder(Order o) { orders.add(o); o.setCustomer(this); }
}

// Test sketch (@DataJpaTest):
Customer c = repo.save(new Customer("a@b.com"));
c.getOrders().add(order);                // inverse side only
em.flush(); em.clear();
// order.customer_id was never written → constraint violation (nullable=false)
// or, if nullable, reload shows customer == null. Replacing with c.addOrder(order) fixes it,
// because addOrder sets the OWNING side (order.customer), which is what Hibernate persists.

Exercise 2 — Manufacture an N+1, then fix it three ways

Seed 50 customers × 4 orders. Enable org.hibernate.SQL: DEBUG. Write a service returning "order id + customer name" for all 200 orders and count the queries (expect 1 + up to 50). Fix it with (a) join fetch, (b) @EntityGraph, (c) a record-based DTO projection — verifying one query each time. State which you'd ship and why.

Solution 
// Naive: findAll() then o.getCustomer().getName() → 1 + N SELECTs in the log.

// (a) fetch join
@Query("select o from Order o join fetch o.customer")
List<Order> findAllWithCustomer();

// (b) entity graph on a derived method
@EntityGraph(attributePaths = "customer")
List<Order> findAllByOrderByIdAsc();

// (c) DTO — the one to ship for a read endpoint
public record Row(Long orderId, String customerName) {}
@Query("select new com.shop.Row(o.id, c.name) from Order o join o.customer c")
List<Row> findRows();
// SQL: select o1_0.id, c1_0.name from orders o1_0 join customers c1_0 ... — one query,
// two columns, no managed entities, no possible LazyInitializationException downstream.

Ship (c) for pure reads: minimal data, no persistence-context overhead, immune to N+1 by construction. Use (a)/(b) when you need the managed entity to mutate it.

Exercise 3 — Break @Transactional two ways, then repair it

Write ReportService.generateAll() that loops and calls this.generateOne(id) where generateOne is @Transactional(REQUIRES_NEW). Prove (log TransactionSynchronizationManager.isActualTransactionActive() / tx name) that no new transaction starts. Then make generateOne throw a checked exception after a save and show the save committed. Repair both.

Solution 
// Bug 1: self-invocation — this.generateOne() bypasses the proxy; REQUIRES_NEW never happens.
// Inside generateOne: TransactionSynchronizationManager.getCurrentTransactionName()
// shows the OUTER method's tx (or none) instead of a new one.
// Fix: move generateOne to ReportItemService and inject it — the call now crosses a proxy:
@Service class ReportItemService {
    @Transactional(propagation = Propagation.REQUIRES_NEW,
                   rollbackFor = Exception.class)          // fixes Bug 2 as well
    public void generateOne(Long id) throws ReportException { ... }
}

// Bug 2: checked exception → default rule doesn't roll back → the save committed.
// rollbackFor = Exception.class makes checked exceptions roll back too.
// Bonus observation: with per-item REQUIRES_NEW, each iteration briefly needs its own
// connection — fine sequentially, dangerous if the loop is parallelized against a small pool.

Exercise 4 — Dirty checking & flush timing, observed

In one @Transactional test method: load a customer, change its name (no save), run a JPQL query select c from Customer c where c.name = :newName, and read the SQL log. Explain the order of statements. Then repeat with readOnly = true and explain the difference.

Solution 
-- Log order with a read-write tx:
select ... from customers where id=?          -- the find
update customers set email=?, name=?, ... where id=?   -- AUTO flush BEFORE the query!
select ... from customers c1_0 where c1_0.name=?       -- the JPQL query — and it FINDS the row

Hibernate flushes before the query because the query reads the customers table, which has pending in-memory changes — otherwise the query would miss your own update (the "auto flush before overlapping query" rule). The UPDATE is still uncommitted; commit happens at method end. With readOnly = true the flush mode is MANUAL: no UPDATE is emitted before the query or at commit — the JPQL query does not find the renamed row, and the rename is silently discarded. Moral: readOnly isn't just an optimization flag; it changes visible semantics if you (wrongly) mutate entities inside it.

Exercise 5 — Optimistic locking under concurrency

Add @Version to Product. Write a test with two threads (or two TransactionTemplate blocks) that both load product #1 (stock 10), both decrement by 1, and commit one after the other. Assert the second commit throws ObjectOptimisticLockingFailureException. Then add a retry wrapper so both decrements eventually apply and final stock is 8.

Solution 
// Tx A and Tx B both read (stock=10, version=0).
// A commits: update products set stock=9, version=1 where id=1 and version=0  → 1 row.
// B commits: update products set stock=9, version=1 where id=1 and version=0  → 0 rows
//            → ObjectOptimisticLockingFailureException, B rolls back.

public void decrementWithRetry(Long id) {
    for (int attempt = 1; ; attempt++) {
        try {
            txTemplate.executeWithoutResult(s -> {          // fresh tx per attempt
                Product p = repo.findById(id).orElseThrow(); // RE-READ fresh state + version
                p.setStock(p.getStock() - 1);
            });
            return;
        } catch (ObjectOptimisticLockingFailureException e) {
            if (attempt == 3) throw e;   // give up → surface as 409/error
        }
    }
}
// Key detail: the retry must wrap the WHOLE transaction (re-read included).
// Retrying just the save with the stale entity would fail forever.

Exercise 6 — Flyway + validate: catch drift on purpose

Set spring.jpa.hibernate.ddl-auto=validate and let Flyway own the schema. Write V1__customers.sql and V2__orders.sql (with the sequence at INCREMENT BY 50, FK + index). Boot the app — it should start clean. Now add a new field String nickname to the entity without a migration and boot again. Observe the failure, then write V3 to fix it.

Solution 
-- V1__customers.sql
create sequence customer_seq start with 1 increment by 50;
create table customers (
  id bigint primary key,
  email varchar(320) not null unique,
  name varchar(200) not null,
  status varchar(32) not null
);

-- Boot after adding the unmapped field fails fast with Hibernate's schema validator:
-- SchemaManagementException: Schema-validation: missing column [nickname]
-- in table [customers]

-- V3__add_customer_nickname.sql (a NEW file — never edit V1, its checksum is recorded)
alter table customers add column nickname varchar(100);

This is exactly the safety net you want in prod: entity/schema drift becomes a deploy-time failure instead of a runtime surprise, and every schema change exists as a reviewed, versioned file.
