12 — Schema Design

Designing PostgreSQL schemas at a whiteboard: normalization, keys, types, patterns — and how to narrate your trade-offs like a senior.

Live schema design is one of the most common senior full-stack interview exercises: "design the tables for an e-commerce checkout" or "model a booking system." Interviewers don't grade the final DDL nearly as much as they grade the conversation — do you ask clarifying questions, do you name trade-offs out loud, do you know why you picked a surrogate key or a NUMERIC column, and can you defend a denormalization when pushed? As a frontend developer you already model state shapes and API contracts; schema design is the same discipline with stricter invariants, because the database outlives every frontend rewrite. This page teaches the concepts, then walks two full whiteboard exercises end to end exactly the way you'd narrate them in an interview.

Normalization through one evolving example T1

Normalization is a sequence of increasingly strict rules that remove redundancy — the root cause of update anomalies, insert anomalies, and delete anomalies. Instead of memorizing definitions, walk through one example the way you would at a whiteboard. Start from the "spreadsheet" a small shop might actually keep:

order_idcustomercustomer_emailproductsprices
1001Asha Raoasha@example.comKeyboard, Mouse79.00, 25.00
1002Ben Coleben@example.comMonitor299.00
1003Asha Raoasha@example.comMouse25.00

1NF — atomic values, no repeating groups

Rule: every column holds a single atomic value; no comma-separated lists, no product1/product2/product3 column families. The products column above violates 1NF — you can't ask "how many mice did we sell?" without string parsing, and you can't index into it.

Fix: one row per order line.

-- 1NF: explode the repeating group into rows
CREATE TABLE order_flat (
    order_id        bigint      NOT NULL,
    customer        text        NOT NULL,
    customer_email  text        NOT NULL,
    product_name    text        NOT NULL,
    price           numeric(10,2) NOT NULL,
    PRIMARY KEY (order_id, product_name)   -- composite key for now
);

2NF — no partial dependency on a composite key

Rule: every non-key column must depend on the whole key, not just part of it. Our key is (order_id, product_name), but customer and customer_email depend only on order_id — a partial dependency. Consequence: Asha's email is duplicated on every line of every order she places. Change it in one row and forget another, and the database now disagrees with itself (an update anomaly).

Fix: split the order header out from the lines.

-- 2NF: order-level facts move to an orders table
CREATE TABLE orders_2nf (
    order_id        bigint PRIMARY KEY,
    customer        text NOT NULL,
    customer_email  text NOT NULL
);

CREATE TABLE order_lines_2nf (
    order_id        bigint NOT NULL REFERENCES orders_2nf(order_id),
    product_name    text NOT NULL,
    price           numeric(10,2) NOT NULL,
    PRIMARY KEY (order_id, product_name)
);

3NF — no transitive dependency

Rule: non-key columns must depend on the key directly, not via another non-key column. In orders_2nf, customer_email depends on customer (the person), which depends on order_id — a transitive dependency. Asha's email is still repeated once per order. Also: if Asha has never ordered, we have nowhere to store her at all (an insert anomaly), and deleting her only order deletes her contact info (a delete anomaly).

Fix: customers become their own entity. Same logic pulls products out of the line table.

-- 3NF: each fact lives in exactly one place
CREATE TABLE customers (
    id      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name    text NOT NULL,
    email   text NOT NULL UNIQUE
);

CREATE TABLE products (
    id      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name    text NOT NULL,
    price   numeric(10,2) NOT NULL CHECK (price >= 0)
);

CREATE TABLE orders (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id bigint NOT NULL REFERENCES customers(id),
    created_at  timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE order_lines (
    order_id    bigint NOT NULL REFERENCES orders(id),
    product_id  bigint NOT NULL REFERENCES products(id),
    quantity    int NOT NULL CHECK (quantity > 0),
    unit_price  numeric(10,2) NOT NULL,   -- see the note below!
    PRIMARY KEY (order_id, product_id)
);

The one-line summaries interviewers love: "The key (1NF), the whole key (2NF), and nothing but the key (3NF)." BCNF is the slightly stricter cousin of 3NF: every determinant (anything that functionally determines another column) must be a candidate key. In practice, if you're in 3NF with single-column surrogate keys, you're almost always in BCNF too — mention it, don't dwell on it.

Notice unit_price in order_lines even though products.price exists. That is not a normalization violation and not denormalization — it's a different fact: "the price at the moment of purchase" vs "the current catalog price." Prices change; historical orders must not. Interviewers frequently probe this exact point, and confusing it with denormalization is a classic mid-level mistake. Say it out loud: "I'm capturing temporal state, not caching a lookup."

Denormalization as a deliberate choice T1

Normalization optimizes for write correctness: one fact, one place, no anomalies. But every fact you split out is a JOIN you pay at read time. Senior engineers denormalize on purpose, with a stated maintenance plan — never by accident. The interview framing: "I'll design normalized first, then denormalize the specific reads that measurement shows are hot."

When to denormalize

  • Read-heavy aggregates: "orders count per customer" or "total revenue per product" computed on every page load. Recomputing SUM() over millions of rows per request is wasteful; a maintained order_count column or summary table is not.
  • Avoiding hot joins: a 5-table join on the most-hit endpoint in the system. Copying, say, customer_name onto a reporting table trades storage and consistency risk for latency.
  • Feed/list screens that need one row per item with everything pre-joined (the "read model" in CQRS terms).

How to keep denormalized data consistent — three options

MechanismConsistencyProsCons
Database triggerSynchronous, transactionalImpossible to bypass; correct even for ad-hoc SQLHidden logic; adds write latency; harder to test/version
Application logic (same transaction)Synchronous if disciplinedVisible in code review; testableEvery write path must remember; ad-hoc SQL bypasses it
Periodic rebuild (cron / materialized view)Eventually consistentSimple; zero write-path cost; self-healingStale between refreshes — must be acceptable to the product
 
-- Option 1: trigger-maintained counter
ALTER TABLE customers ADD COLUMN order_count int NOT NULL DEFAULT 0;

CREATE OR REPLACE FUNCTION bump_order_count() RETURNS trigger AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE customers SET order_count = order_count + 1 WHERE id = NEW.customer_id;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE customers SET order_count = order_count - 1 WHERE id = OLD.customer_id;
    END IF;
    RETURN NULL;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_count
AFTER INSERT OR DELETE ON orders
FOR EACH ROW EXECUTE FUNCTION bump_order_count();

-- Option 3: periodic rebuild via materialized view
CREATE MATERIALIZED VIEW customer_stats AS
SELECT c.id, c.name, count(o.id) AS order_count, coalesce(sum(l.unit_price * l.quantity), 0) AS lifetime_value
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
LEFT JOIN order_lines l ON l.order_id = o.id
GROUP BY c.id, c.name;

REFRESH MATERIALIZED VIEW CONCURRENTLY customer_stats;  -- needs a unique index on the MV

Counter columns maintained by triggers become lock hot-spots: every order insert for the same customer serializes on that customer's row. For high-write counters, prefer periodic aggregation, or an append-only tally table that's rolled up asynchronously. Say this trade-off out loud in the interview — it's exactly the kind of second-order thinking being graded.

Keys & constraints T1

Primary keys: natural vs surrogate

A natural key is a real-world attribute (email, SSN, ISBN). A surrogate key is a meaningless generated identifier. Surrogates usually win, and the canonical argument is: emails change. If email is your PK, changing it means cascading updates through every FK in the system, and every historical reference is rewritten. Natural keys also tend to be wide (bad for index size — every secondary index in Postgres stores the row's heap pointer, but FKs replicate the PK into child tables) and occasionally turn out not to be unique after all (two employees genuinely can share a name; SSNs get recycled and mistyped).

The senior nuance: use a surrogate PK and declare the natural key UNIQUE. You get stable joins and still enforce the business rule.

CREATE TABLE users (
    id      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,  -- surrogate: stable forever
    email   text NOT NULL UNIQUE,                             -- natural key: enforced, but mutable
    name    text NOT NULL
);

Foreign keys: choosing the ON DELETE action

ActionBehaviorUse when…
RESTRICT / NO ACTION (default)Parent delete fails if children existChildren are independently valuable: never silently delete orders because a product was removed. Safe default.
CASCADEDeleting the parent deletes childrenChildren are meaningless without the parent: order_lines when the order goes, tag mappings, session rows.
SET NULLChild FK becomes NULLThe relationship is optional and the child survives: tickets keep existing when their assignee's account is deleted.
 
CREATE TABLE order_lines (
    order_id   bigint NOT NULL REFERENCES orders(id) ON DELETE CASCADE,   -- lines die with the order
    product_id bigint NOT NULL REFERENCES products(id) ON DELETE RESTRICT, -- can't drop a purchased product
    ...
);

CREATE TABLE tickets (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    assignee_id bigint REFERENCES users(id) ON DELETE SET NULL  -- note: column must be nullable
);

Postgres does NOT automatically index foreign key columns. It indexes the referenced side (the PK) but not the referencing column. Without an index on order_lines.order_id, every DELETE FROM orders sequential-scans order_lines to check the constraint, and every "lines for this order" query is slow. Rule of thumb: index every FK column unless you have a measured reason not to. This is one of the highest-signal facts you can drop in an interview.

CREATE INDEX idx_order_lines_order   ON order_lines (order_id);
CREATE INDEX idx_order_lines_product ON order_lines (product_id);

UNIQUE — and the NULL subtlety

By default, a UNIQUE constraint allows multiple NULLs, because in SQL NULL = NULL is unknown, not true — two NULLs are not considered duplicates. This matters for optional-but-unique columns like external_id. Postgres 15 added an explicit switch:

-- default: many rows may have external_id IS NULL
ALTER TABLE users ADD CONSTRAINT uq_ext UNIQUE (external_id);

-- PG 15+: treat NULLs as equal, so at most ONE NULL allowed
ALTER TABLE users ADD CONSTRAINT uq_ext UNIQUE NULLS NOT DISTINCT (external_id);

CHECK constraints — guard enums and ranges

Every column with a bounded domain deserves a CHECK. It costs microseconds on write and prevents entire classes of bad data forever — including bugs from services and scripts that bypass your application layer.

CREATE TABLE orders (
    id       bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    status   text NOT NULL DEFAULT 'pending'
             CHECK (status IN ('pending','paid','shipped','delivered','cancelled')),
    total    numeric(12,2) NOT NULL CHECK (total >= 0),
    placed_at   timestamptz NOT NULL DEFAULT now(),
    shipped_at  timestamptz,
    CHECK (shipped_at IS NULL OR shipped_at >= placed_at)   -- cross-column rule
);

NOT NULL by default mindset

Treat every column as NOT NULL unless you can articulate what NULL means for it ("not yet shipped", "no assignee"). Nullable-by-default schemas push null-handling into every consumer forever — the SQL equivalent of a TypeScript codebase without strictNullChecks. That analogy lands well in interviews.

EXCLUSION constraints — the booking superpower

UNIQUE says "no two rows with the same value." EXCLUSION generalizes it: "no two rows for which these operators all return true." The killer use case is preventing overlapping time ranges — double-booked rooms — enforced by the database itself, race-free, no application locking required:

CREATE EXTENSION IF NOT EXISTS btree_gist;  -- lets GiST index plain equality too

CREATE TABLE reservations (
    id       bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    room_id  bigint NOT NULL REFERENCES rooms(id),
    during   tstzrange NOT NULL,
    EXCLUDE USING gist (room_id WITH =, during WITH &&)  -- same room AND overlapping range: rejected
);

Two concurrent transactions trying to book the same room for overlapping windows: one commits, the other gets a constraint violation. Compare with the app-level alternative — SELECT ... FOR UPDATE then check then insert — which is more code and easier to get wrong. Knowing this exists is a genuine differentiator; we use it in the booking walkthrough below.

UUID vs BIGINT identity — the deep dive T1

"What do you use for primary keys?" is asked in nearly every schema interview. The strong answer names concrete mechanics on both sides, then picks based on context.

bigint IDENTITY / BIGSERIALUUID (v4 random)UUIDv7 (time-ordered)
Size8 bytes16 bytes16 bytes
GenerationDB sequence (central coordination)Anywhere — client, app, any shard, offlineAnywhere
B-tree insert patternRightmost page — cache-friendly, dense pagesRandom pages — page splits, fragmentation, poor cache hit rateRightmost-ish — nearly as good as bigint
Safe to expose in URLs?No — sequential IDs are guessable, enable enumeration attacks and leak business volume ("order #1,000,001")YesMostly — leaks creation time, rarely a problem
Sortable by creation?Yes (roughly)NoYes (by design)
Merging datasets / multi-regionCollisions — needs offset schemes or ID serviceTrivially safeTrivially safe

Why the B-tree point matters: with sequential bigints, every insert lands on the current rightmost leaf page — one hot page in cache, pages fill completely. With random UUIDv4s, inserts scatter across the whole index: the working set is the entire index, pages split at ~50% fill, the index bloats, and write throughput drops measurably at scale. The 16-byte width also compounds — the PK value is duplicated into every FK column and every index on those FKs.

UUIDv7 (48-bit millisecond timestamp prefix + randomness) is the modern middle ground: globally generatable and non-guessable-enough, but index-friendly because new values sort last. Postgres 18 ships uuidv7() natively; before that, generate in the application (most languages have libraries) or via an extension.

-- bigint identity (the default choice for internal tables)
CREATE TABLE line_items (
    id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- UUID PK, DB-generated v4
CREATE TABLE api_keys (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid()   -- built-in since PG 13
);

-- Pragmatic hybrid: internal bigint PK + external UUID for the API surface
CREATE TABLE orders (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,  -- joins, FKs: compact + fast
    public_id   uuid NOT NULL UNIQUE DEFAULT gen_random_uuid(),   -- what URLs and clients see
    ...
);

The hybrid pattern is a great interview answer because it acknowledges both concerns: "I keep joins on 8-byte sequential keys for index locality, and expose an unguessable UUID at the API boundary. If the team standard is UUIDs everywhere, I push for UUIDv7 to avoid random-write index fragmentation." That sentence demonstrates you understand storage, security, and pragmatism.

Postgres data types done right T2

text vs varchar(n)

In Postgres they have identical performance and storage — varchar(n) is just text with a length check bolted on. Unlike some databases, there is no padding or speed benefit. The idiomatic choice is text, plus a CHECK when a length limit is a genuine business rule — because changing a CHECK later is easier and better-documented than an opaque varchar(255) someone copied from a tutorial.

-- Idiomatic Postgres
username text NOT NULL CHECK (char_length(username) BETWEEN 3 AND 30)

-- Works, but the 255 is almost always cargo-cult
username varchar(255)

Money: NUMERIC, never float

real/double precision are binary floating point — they cannot represent most decimal fractions exactly, and errors accumulate:

SELECT 0.1::float8 + 0.2::float8;        -- 0.30000000000000004
SELECT (0.1 + 0.2)::numeric = 0.3;       -- true

-- Sum a million 10-cent charges:
SELECT sum(x) FROM generate_series(1, 1000000) g, LATERAL (SELECT 0.10::float8 AS x) t;
-- 100000.00000133288  ← you just invented or lost money

price numeric(12,2) NOT NULL CHECK (price >= 0)   -- exact decimal: correct answer, always

NUMERIC(precision, scale) is exact decimal arithmetic. It's slower than float, and nobody cares — correctness wins for money. (Avoid the legacy money type: it's locale-dependent and awkward. For multi-currency, store amount numeric + currency char(3), or amounts in minor units as bigint.)

timestamptz vs timestamp — always timestamptz

Despite the name, timestamptz does not store a time zone. Both types are 8 bytes; timestamptz stores an absolute instant in UTC and converts to/from the session's TimeZone on input/output. Plain timestamp stores a wall-clock reading with no idea which wall — the moment your app server, DB server, and users disagree on zones (they will), plain timestamps become ambiguous garbage. Default to timestamptz; use date for calendar dates ("birthday", "check-in date") and plain timestamp only for genuinely zone-less concepts (rare).

created_at  timestamptz NOT NULL DEFAULT now(),   -- an instant: always timestamptz
check_in    date NOT NULL,                        -- a calendar date, no time component
duration    interval NOT NULL DEFAULT '30 minutes'  -- spans of time get their own type

boolean, arrays

boolean is a real type (1 byte, accepts true/false) — use it instead of char(1) flags. Arrays (text[], int[]) are legitimate for small, atomic, order-matters lists that you never join through — tags on a row, RGB channels. The moment you need to query "all posts with tag X" efficiently with referential integrity, or attach metadata to the relationship, you want a join table instead. Arrays can't have FKs on their elements — say that sentence in the interview.

-- OK: incidental list, queried via GIN index if needed
tags text[] NOT NULL DEFAULT '{}'
-- CREATE INDEX ON posts USING gin (tags);

-- Better when tags are entities: join table (see modeling patterns)

Enums: native ENUM vs lookup table vs text + CHECK

ApproachProsCons
CREATE TYPE ... AS ENUM4 bytes, self-documenting, typo-proof, has a defined sort orderALTER pain: you can add values, but you cannot remove or reorder them without recreating the type; values are global to the DB, not per-table
Lookup table + FKValues are data: add/retire/rename with DML; can carry extra columns (label, sort order, active flag); admin-editableAn extra join for display; slightly more ceremony
text + CHECKSimple, readable in every query, constraint change is one fast ALTERValues are strings (more bytes); the "list of valid values" lives in DDL only
 
-- Fixed-forever domain: native enum is fine
CREATE TYPE order_status AS ENUM ('pending','paid','shipped','delivered','cancelled');

-- Product-managed domain: lookup table
CREATE TABLE ticket_categories (
    id     int GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    code   text NOT NULL UNIQUE,
    label  text NOT NULL,
    active boolean NOT NULL DEFAULT true
);

-- Pragmatic default: text + CHECK (easy to evolve, no join)
status text NOT NULL CHECK (status IN ('pending','paid','shipped','delivered','cancelled'))

My default interview answer: text + CHECK for small stable domains, lookup table when values need metadata or non-developer editing, native enum only when I'm confident the set is fixed (and I mention the ALTER pain unprompted).

Identity columns and JSONB

Prefer GENERATED ALWAYS AS IDENTITY over the older SERIAL: it's SQL-standard, the sequence is properly owned by the column, and ALWAYS blocks accidental manual inserts into the ID (use BY DEFAULT if you need to bulk-load explicit IDs). SERIAL is just legacy syntax sugar with looser semantics.

JSONB deserves one sentence here: it's the right tool for genuinely schemaless attributes (user-defined fields, webhook payloads) and the wrong tool for data you filter, join, and constrain relationally. Full treatment — operators, GIN indexing, when it beats EAV — is in tutorial 15.

Modeling patterns T2

One-to-many

The FK lives on the many side. Customer has many orders → orders.customer_id. Nothing on the parent. (Frontend intuition check: it's the opposite of nesting — you don't put an array of order IDs on the customer.)

Many-to-many: the join table

Students↔courses, posts↔tags, users↔roles. A third table holds one row per relationship. The design decision interviewers probe: composite PK or surrogate PK + unique constraint?

-- Composite PK: lean, duplicate-proof by construction
CREATE TABLE post_tags (
    post_id bigint NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
    tag_id  bigint NOT NULL REFERENCES tags(id)  ON DELETE CASCADE,
    PRIMARY KEY (post_id, tag_id)
);
CREATE INDEX idx_post_tags_tag ON post_tags (tag_id);  -- reverse-direction lookups

-- Surrogate + UNIQUE: use when the relationship itself carries data
-- or other tables reference it
CREATE TABLE enrollments (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    student_id  bigint NOT NULL REFERENCES students(id),
    course_id   bigint NOT NULL REFERENCES courses(id),
    enrolled_at timestamptz NOT NULL DEFAULT now(),
    grade       text,
    UNIQUE (student_id, course_id)
);

Rule of thumb: pure link table → composite PK. Relationship-with-attributes (an associative entity like an enrollment) or anything other rows point at → surrogate PK, and keep the UNIQUE pair so duplicates stay impossible. Note the composite PK (post_id, tag_id) index serves post→tags lookups but not tag→posts — hence the extra index led by tag_id.

Self-referencing: trees and hierarchies

-- Adjacency list: simplest, the right default
CREATE TABLE employees (
    id         bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name       text NOT NULL,
    manager_id bigint REFERENCES employees(id) ON DELETE SET NULL  -- NULL = the CEO
);
CREATE INDEX idx_employees_manager ON employees (manager_id);

-- Whole subtree via recursive CTE:
WITH RECURSIVE reports AS (
    SELECT id, name, manager_id, 1 AS depth FROM employees WHERE id = 42
    UNION ALL
    SELECT e.id, e.name, e.manager_id, r.depth + 1
    FROM employees e JOIN reports r ON e.manager_id = r.id
)
SELECT * FROM reports;

Alternatives to know by name: a closure table (separate table storing every ancestor→descendant pair; subtree queries become a single join, at the cost of O(depth) rows per node on insert and heavier moves) and ltree/materialized-path. Interview answer: "Adjacency list plus recursive CTEs handles org-chart-scale trees fine; I'd reach for a closure table only if deep-subtree reads dominated and profiling justified the write overhead."

Polymorphic references — comments on posts OR photos

A classic trap question, because the naive approach breaks referential integrity. Three approaches:

ApproachHowTrade-off
1. type + id, no FK (Rails-style)commentable_type text, commentable_id bigintOne table, flexible — but zero referential integrity: orphaned comments, no CASCADE, no join safety. Name this weakness explicitly.
2. Exclusive-arc FKsOne nullable FK per target + CHECK that exactly one is setFull FK integrity and CASCADEs; each new target type is a migration (new column). Usually the best answer for a handful of known types.
3. Separate tablespost_comments, photo_commentsCleanest constraints per table; duplicated structure, and "all comments by user X" becomes a UNION.
 
-- Approach 2: exclusive arc
CREATE TABLE comments (
    id        bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    author_id bigint NOT NULL REFERENCES users(id),
    body      text NOT NULL,
    post_id   bigint REFERENCES posts(id)  ON DELETE CASCADE,
    photo_id  bigint REFERENCES photos(id) ON DELETE CASCADE,
    CHECK (num_nonnulls(post_id, photo_id) = 1),   -- exactly one target
    created_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX ON comments (post_id)  WHERE post_id  IS NOT NULL;
CREATE INDEX ON comments (photo_id) WHERE photo_id IS NOT NULL;

Soft delete vs hard delete

ALTER TABLE users ADD COLUMN deleted_at timestamptz;  -- NULL = alive

-- THE GOTCHA: plain UNIQUE(email) blocks a new signup with a
-- soft-deleted user's email forever. Scope uniqueness to the living:
CREATE UNIQUE INDEX uq_users_email_alive
    ON users (email) WHERE deleted_at IS NULL;

Soft delete quietly infects every query in the system with WHERE deleted_at IS NULL — miss it once (in a report, a join, a uniqueness check) and deleted data leaks back. Mitigations: a view exposing only live rows, ORM default filters (Hibernate @SQLRestriction), or row-level security. The alternative — hard delete + copying the row into an audit/history table (or relying on CDC) — keeps the main tables clean and is often the better answer when "undelete" isn't a real product requirement. Compliance regimes like GDPR erasure can also forbid keeping soft-deleted personal data.

Timestamps and optimistic locking conventions

CREATE TABLE documents (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    title       text NOT NULL,
    body        text NOT NULL,
    version     int  NOT NULL DEFAULT 0,               -- optimistic lock counter
    created_at  timestamptz NOT NULL DEFAULT now(),
    updated_at  timestamptz NOT NULL DEFAULT now()      -- maintained by trigger or app/ORM
);

-- Optimistic locking: the update carries the version you read
UPDATE documents
SET body = $1, version = version + 1, updated_at = now()
WHERE id = $2 AND version = $3;   -- 0 rows updated ⇒ somebody else won; retry or surface a conflict

Put created_at/updated_at on effectively every table — it costs 16 bytes and saves debugging sessions. The version column is exactly what JPA's @Version manages for you (tutorial 08): the schema-level view of the same mechanism. Note Postgres does not auto-update updated_at like MySQL's ON UPDATE CURRENT_TIMESTAMP — you need a trigger or ORM callbacks.

Multi-tenancy strategies T2

"You're building a B2B SaaS — how do you isolate customers' data?" Three strategies, one decision table:

Shared tables + tenant_idSchema per tenantDatabase per tenant
IsolationLogical only — one bad WHERE clause away from a breachStrong-ish (same instance, separate namespaces)Strongest (separate instance possible)
Scales to N tenantsMillionsHundreds–low thousands (catalog bloat, migration fan-out)Dozens–hundreds
MigrationsRun onceRun × N schemasRun × N databases
Per-tenant backup/restoreHard (row extraction)Moderate (pg_dump -n)Trivial
Noisy-neighbor controlWeakWeakStrong
Cost / ops overheadLowestMediumHighest
Typical fitSelf-serve SaaS, many small tenantsMid-market, moderate tenant countEnterprise/regulated tenants demanding isolation
 
-- Shared-table pattern done right
CREATE TABLE projects (
    id         bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    tenant_id  bigint NOT NULL REFERENCES tenants(id),
    name       text NOT NULL,
    UNIQUE (tenant_id, name)              -- uniqueness is PER TENANT, not global
);
-- Composite indexes LED BY tenant_id: every query filters on it first
CREATE INDEX idx_projects_tenant_created ON projects (tenant_id, created_at DESC);

-- Defense in depth: Row-Level Security makes the DB enforce tenancy
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON projects
    USING (tenant_id = current_setting('app.tenant_id')::bigint);
-- App sets: SET app.tenant_id = '42';  (per connection/transaction)

The senior framing: "Default to shared tables with tenant_id — it's the only one that scales operationally — and add RLS so a forgotten WHERE clause fails closed instead of leaking data. I'd reserve database-per-tenant for contractual isolation requirements, and I'd resist schema-per-tenant beyond a few hundred tenants because migrations and connection pooling become the bottleneck." Also note: many real products are hybrids — shared tables for the masses, dedicated DBs for two enterprise whales.

Live design walkthroughs

These two exercises are the most common live-design prompts. Each follows the exact sequence you should use at the whiteboard: clarify requirements → list entities → draw relationships → write DDL → narrate decisions. Practice saying the commentary out loud — the "why" is what's graded.

Walkthrough 1: E-commerce orders system

Prompt: "Design the schema for the checkout/orders part of an online store."

Step 1 — Clarify (ask these, don't assume): Multiple addresses per customer? (Yes.) Do product prices change over time? (Yes — so lines must snapshot price.) Partial shipments/refunds? (Out of scope today, but I'll leave room.) Guest checkout? (No — customers have accounts.) One payment per order? (Start there; model as 1-to-many to allow retries.)

Step 2 — Entities: customers, addresses, products, orders, order_lines, payments.

Step 3 — Relationships: customer 1—N addresses; customer 1—N orders; order 1—N order_lines; product 1—N order_lines (M:N between orders and products, materialized as the line table — which is also the associative entity carrying quantity and snapshot price); order 1—N payments.

Step 4 — Full DDL:

CREATE TABLE customers (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    public_id   uuid NOT NULL UNIQUE DEFAULT gen_random_uuid(),
    email       text NOT NULL UNIQUE CHECK (email ~ '^[^@\s]+@[^@\s]+$'),

Step 5 — Narrate the decisions (this is the graded part):

  • Hybrid keys: internal bigint PKs keep joins and FK indexes compact; public_id uuid is what URLs and emails expose, so order volume isn't leaked and IDs aren't enumerable.
  • unit_price on lines is temporal correctness, not denormalization — the catalog price will change; the order must be a faithful record of what was agreed. I do not store an order total column initially: it's derivable (SUM(quantity * unit_price)), and derived columns are drift risks. If invoice reads get hot, I'd add it maintained in the same transaction that inserts lines — a deliberate denormalization with a named owner.
  • Delete policies differ per FK, on purpose: lines CASCADE with their order (meaningless alone); orders RESTRICT on customer delete (financial records must survive — realistically customers get soft-deleted/anonymized instead); products RESTRICT while referenced (deactivate via active = false, don't delete history).
  • Addresses are referenced, not embedded — but there's a subtlety I'd flag: if a customer edits an address, historical orders would show the new one. Production systems either make address rows immutable (edits create new rows) or snapshot the address text onto the order. I'd raise this and let the interviewer pick the depth.
  • Every FK got an index, and idx_orders_customer is composite with placed_at DESC because "this customer's recent orders" is the hot query.
  • Status is text + CHECK, not a native enum — order-status vocabularies grow ('refunded', 'partially_shipped'), and altering a CHECK is painless. A status-transition history table would be the next evolution if "when did it ship?" matters.
  • Payments UNIQUE (provider, provider_ref) makes webhook processing idempotent at the schema level — retries become no-ops instead of double-charges in your books.
Interviewer follow-up: "Where does inventory/stock reservation fit? What about overselling?"

products.stock_qty with CHECK (stock_qty >= 0) is the last line of defense: a decrementing UPDATE products SET stock_qty = stock_qty - $1 WHERE id = $2 AND stock_qty >= $1 is atomic — if it updates zero rows, the item is out of stock, and two concurrent checkouts can't both take the last unit because row-level locking serializes the updates. For carts that reserve stock before payment, add a reservations table with an expires_at, count active reservations against stock, and sweep expired rows. I'd also flag that at very high scale the product row becomes a lock hot-spot, which pushes you toward per-warehouse stock rows or queue-based allocation — but I wouldn't build that before measuring.

Interviewer follow-up: "The business now wants order history to show the product name even if the product is renamed or removed. Change the design?"

Same reasoning as unit_price: the line should snapshot every fact that must be historically stable — add product_name text NOT NULL (and maybe sku) to order_lines, copied at insert time. Then I can even relax the product FK question: keep product_id as a nullable ON DELETE SET NULL link for analytics, while the line remains self-contained for display. The general principle I'd state: documents (orders, invoices) snapshot; catalogs reference. Confusing the two is the root of most "history changed under us" bugs.

Interviewer follow-up: "Orders is 200M rows and the dashboard's 'revenue per day' query is slow. Options?"

In order of increasing invasiveness: (1) a covering composite index on (placed_at, ...) — probably already insufficient for full scans over months; (2) a materialized view of daily revenue refreshed on a schedule — minutes of staleness is fine for a dashboard; (3) an incremental summary table (daily_revenue(day, total)) updated by the payment-success path or a rollup job — real-time-ish; (4) declarative partitioning of orders by month on placed_at, which also helps archival and vacuum; (5) ship analytics to a columnar store and stop asking OLAP questions of the OLTP database. I'd start at (2), it's an afternoon of work and reversible.

Walkthrough 2: Booking / reservation system

Prompt: "Design a schema for a meeting-room booking system: multiple offices, rooms with capacity, employees book time slots, no double-booking."

Step 1 — Clarify: Continuous time ranges or fixed slots? (Continuous — 9:17 to 9:44 is legal.) Recurring bookings? (Note it, park it.) Cancellations? (Yes — keep the history.) Can a booking span midnight? (Yes, ranges handle it naturally.) Time zones? (Offices are in different zones — timestamptz everywhere, display conversion is the frontend's job.)

Step 2 — Entities: offices, rooms, employees, bookings (+ optional booking_attendees M:N).

Step 3 — Relationships: office 1—N rooms; employee 1—N bookings (organizer); room 1—N bookings; bookings M—N employees as attendees. The interesting constraint isn't a relationship at all — it's "no two bookings for one room overlap in time," and I want the database to enforce it.

Step 4 — Full DDL:

CREATE EXTENSION IF NOT EXISTS btree_gist;   -- equality operators inside GiST indexes

CREATE TABLE offices (
    id        bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name      text NOT NULL UNIQUE,
    time_zone text NOT NULL DEFAULT 'UTC'    -- IANA name, for DISPLAY defaults only
);

CREATE TABLE rooms (
    id        bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    office_id bigint NOT NULL REFERENCES offices(id) ON DELETE CASCADE,
    name      text NOT NULL,
    capacity  int NOT NULL CHECK (capacity > 0),
    UNIQUE (office_id, name)                 -- "Aspen" can exist in every office
);
CREATE INDEX idx_rooms_office ON rooms (office_id);

CREATE TABLE employees (
    id    bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    email text NOT NULL UNIQUE,
    name  text NOT NULL
);

CREATE TABLE bookings (
    id           bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    room_id      bigint NOT NULL REFERENCES rooms(id)     ON DELETE CASCADE,
    organizer_id bigint NOT NULL REFERENCES employees(id) ON DELETE RESTRICT,
    title        text NOT NULL,
    during       tstzrange NOT NULL,
    status       text NOT NULL DEFAULT 'confirmed'
                 CHECK (status IN ('confirmed','cancelled')),
    created_at   timestamptz NOT NULL DEFAULT now(),
    CHECK (NOT isempty(during)),                      -- no zero-length bookings
    CHECK (upper(during) - lower(during) <= interval '12 hours'),
    -- THE core rule: same room + overlapping range + still confirmed ⇒ rejected
    CONSTRAINT no_double_booking
        EXCLUDE USING gist (room_id WITH =, during WITH &&)
        WHERE (status = 'confirmed')
);
CREATE INDEX idx_bookings_organizer ON bookings (organizer_id);
CREATE INDEX idx_bookings_room_time ON bookings USING gist (room_id, during);

CREATE TABLE booking_attendees (
    booking_id  bigint NOT NULL REFERENCES bookings(id)  ON DELETE CASCADE,
    employee_id bigint NOT NULL REFERENCES employees(id) ON DELETE CASCADE,
    response    text NOT NULL DEFAULT 'invited'
                CHECK (response IN ('invited','accepted','declined')),
    PRIMARY KEY (booking_id, employee_id)
);
CREATE INDEX idx_attendees_employee ON booking_attendees (employee_id);

-- Booking insert: no SELECT-then-INSERT race, the constraint IS the check
INSERT INTO bookings (room_id, organizer_id, title, during)
VALUES (7, 3, 'Design review', tstzrange('2026-07-06 14:00+00','2026-07-06 15:00+00'));
-- overlapping concurrent insert ⇒ ERROR: conflicting key value violates
-- exclusion constraint "no_double_booking"

-- "Find free rooms in office 2 for a window" — range operators shine:
SELECT r.*
FROM rooms r
WHERE r.office_id = 2
  AND NOT EXISTS (
      SELECT 1 FROM bookings b
      WHERE b.room_id = r.id
        AND b.status = 'confirmed'
        AND b.during && tstzrange('2026-07-06 14:00+00','2026-07-06 15:00+00')
  );

Step 5 — Narrate the decisions:

  • The exclusion constraint is the star. The naive design — starts_at/ends_at columns plus an application-level "check availability then insert" — has a race: two users can both pass the check and both insert. Fixing that in app code means SELECT ... FOR UPDATE on the room row or SERIALIZABLE isolation. The exclusion constraint moves the invariant into the database where it's enforced atomically for every writer, including the batch script someone writes next year.
  • tstzrange over two columns: one value that carries inclusive/exclusive semantics — ['14:00','15:00') half-open by default, so back-to-back meetings at 15:00 don't "overlap." With separate columns, that boundary logic is re-implemented (and off-by-one'd) in every query.
  • The partial exclusion (WHERE status = 'confirmed') means cancelled bookings stay as history but free the slot — no deletes needed, and no separate archive dance.
  • timestamptz everywhere; office time_zone is display metadata. Instants are stored in UTC; "2 PM in the Berlin office" is a rendering concern. Storing local wall-clock times is how systems double-book across DST transitions.
  • Recurring bookings (parked earlier): I'd store the recurrence rule (RRULE-style) on a parent row and materialize occurrences as ordinary booking rows a window ahead — so the exclusion constraint keeps protecting every occurrence. Expanding rules at query time can't be constrained by the DB.
  • Organizer is RESTRICT, attendees CASCADE: a booking without its organizer is a real record we shouldn't silently lose (deactivate employees instead); an attendee row without its booking is noise.
Interviewer follow-up: "Enforce 'attendee count must not exceed room capacity' — where?"

This is a cross-table invariant, and I'd be honest that Postgres can't declare it as a simple constraint (CHECKs can't reference other tables). Options: (1) enforce in the service layer inside the same transaction that inserts attendees, reading the room capacity with the booking row locked — fine for one write path; (2) a trigger on booking_attendees that counts and raises — enforced for all writers, at the cost of hidden logic and a count query per insert; (3) accept eventual enforcement (warn in UI, reconcile async) since an over-invited meeting isn't a data-corruption disaster. I'd pick (1) with (2) as hardening if multiple services write this table. The meta-point interviewers want: knowing which invariants the database can own (overlap: yes, via exclusion) and which it can't (cross-table counts: not declaratively).

Interviewer follow-up: "Scale this to 100k rooms and years of history — what worries you?"

The GiST index backing the exclusion constraint grows with all confirmed bookings ever, but overlap checks only ever contend with current and future rows — old rows are dead weight in the index. Mitigations: partition bookings by time range (note: exclusion constraints must include the partition key, which during can, and each partition gets its own constraint — cross-partition overlap needs the partition boundaries to align with booking limits, which our 12-hour CHECK helps with); or periodically move ancient bookings to an archive table. Also, availability searches ("free rooms this week across the campus") become the hot query — the (room_id, during) GiST index covers it, and I'd verify with EXPLAIN ANALYZE before inventing anything cleverer.

Interviewer follow-up: "Why not just book fixed 30-minute slots — one row per slot, UNIQUE(room_id, slot_start)?"

It's a legitimate simplification and I'd say so: a plain UNIQUE constraint replaces the exclusion constraint, every B-tree behavior is well-trodden, and 'find free slots' is a simple anti-join. Costs: bookings spanning multiple slots become multiple rows (or a parent + slot rows), you can't represent a 9:17–9:44 booking, changing slot granularity later is a migration nightmare, and multi-slot booking inserts must be atomic so you don't grab 2 of 3 needed slots. If the product genuinely thinks in fixed slots (doctor's appointments), slots win on simplicity. If time is continuous (room bookings, rentals), ranges + exclusion is less code and more correct. Stating the fork and choosing based on the product requirement is exactly the answer being graded.

Interview Q&A

Explain 1NF, 2NF, and 3NF in under a minute.

1NF: every column atomic, no repeating groups — no comma-separated lists. 2NF: no non-key column depends on just part of a composite key — order-level facts don't live on order-line rows. 3NF: no non-key column depends on another non-key column — customer email lives on the customer, not on the order. Mnemonic: the key, the whole key, and nothing but the key. Each level removes a class of redundancy, and redundancy is what causes update, insert, and delete anomalies. BCNF tightens 3NF: every determinant must be a candidate key — with surrogate single-column keys you're usually there already.

When would you deliberately denormalize, and how do you keep it safe?

When a measured hot read path can't be served affordably from normalized data — read-heavy aggregates (counts, totals rendered on every page) or a hot multi-way join. "Safe" means naming an owner for consistency: a trigger (transactional, can't be bypassed, but hidden and adds write latency), application logic in the same transaction (visible and testable, but every write path must cooperate), or a periodic rebuild/materialized view (simplest, but eventually consistent — the product must tolerate staleness). I always add: storing order-line price at purchase time is not denormalization — it's a different, temporal fact.

Natural key or surrogate key for a users table?

Surrogate PK (bigint identity or UUID), with the natural key (email) as a UNIQUE constraint. Emails change — and a changed PK cascades through every FK and every historical reference. Natural keys are also wide (bloating every child table's FK and its indexes) and their "guaranteed uniqueness" often turns out to be a lie discovered in production. The surrogate gives immutable identity; the UNIQUE preserves the business rule.

UUID or BIGINT for primary keys? Defend your choice.

Context-dependent, and I'd name the mechanics: bigint is 8 bytes and appends to the rightmost B-tree page — compact indexes, cache-friendly inserts — but values are guessable (enumeration attacks) and leak volume, and generation needs the DB sequence. UUIDv4 generates anywhere with no coordination and is safe to expose, but it's 16 bytes and random inserts scatter across the whole index, causing page splits and cache misses at scale. UUIDv7 is the modern middle ground: time-ordered prefix restores append-mostly insert locality while staying globally unique. My default: bigint internally with a UUID public_id for the API surface, or UUIDv7 everywhere if the team wants one key type.

Does Postgres index foreign key columns automatically?

No — and this is a top-tier gotcha. The referenced side is indexed (it must be a PK or UNIQUE), but the referencing column is not. Consequences of missing FK indexes: parent deletes/updates trigger sequential scans of the child table for the integrity check (and hold locks while doing it), and every "children of this parent" query scans. My habit: create an index on every FK column at table-creation time, and drop it later only with evidence it's unused.

What does ON DELETE CASCADE do, and when is it dangerous?

Deleting a parent automatically deletes referencing children (transitively — cascades chain). Right for composition relationships: order lines, attendee rows, tag links — children meaningless without the parent. Dangerous when children have independent value: CASCADE from customers to orders means deleting an account silently destroys financial history. Also operationally risky: one innocent-looking DELETE can fan out to millions of rows, holding locks throughout. Default to RESTRICT and opt into CASCADE per relationship with justification; use SET NULL when the link is optional and the child outlives it.

Why should money never be stored as float? What instead?

Floats are binary fractions; most decimal values (0.1, 0.2) have no exact binary representation, so 0.1 + 0.2 = 0.30000000000000004 and summing a million small charges drifts by real cents — unacceptable for anything auditable. Use NUMERIC(precision, scale): exact decimal arithmetic, e.g. numeric(12,2) for currency amounts. Alternative: integer minor units (bigint cents), common in payments code. Avoid Postgres's legacy money type — locale-dependent formatting, awkward casts. For multi-currency, pair the amount with an ISO currency code column.

timestamp vs timestamptz — which and why?

Always timestamptz for instants. The name misleads: it stores no zone — both types are 8-byte UTC-ish microsecond counts. The difference is semantics: timestamptz is an absolute instant, converted to/from the session's TimeZone at the wire; plain timestamp is a zone-less wall-clock reading, ambiguous the moment any two components disagree on zones (or DST shifts). Use date for calendar dates like check-in day, interval for durations. Rendering "2 PM Berlin time" is the frontend's job from the UTC instant.

varchar(255) vs text in Postgres?

No performance or storage difference whatsoever in Postgres — varchar(n) is text plus a length check. The idiom is text, adding CHECK (char_length(col) <= n) only when the limit is a real business rule, because a named CHECK documents intent and is trivially alterable. varchar(255) specifically is cargo cult from MySQL's old index-prefix limits. Bonus point: changing varchar(50) to varchar(100) is a cheap catalog-only ALTER in modern Postgres, but starting with text avoids the ceremony entirely.

How do you model an enum-ish column: native ENUM, lookup table, or text + CHECK?

Three options, chosen by who changes the values and how often. Native ENUM: compact (4 bytes) and typo-proof, but you can add values only (no remove/reorder without recreating the type) and the type is database-global — painful in migrations. Lookup table + FK: values become data — renameable, retireable, can carry labels and sort order, editable by admins — at the cost of a join. Text + CHECK: dead simple, values readable in every query, and evolving the set is one ALTER of the constraint. My default is text + CHECK for small developer-owned domains, lookup table when values need metadata or non-engineers manage them; I reach for native enums rarely and mention the ALTER pain unprompted.

Design a many-to-many relationship. Composite PK or surrogate id on the join table?

Join table with two FKs. Pure link (post_tags): composite PK (post_id, tag_id) — duplicates impossible by construction, no useless extra column; add a second index led by the other column for reverse lookups, since the PK index only serves one direction. When the relationship carries attributes (enrollment with grade and date) or other tables reference it, it's really an entity: give it a surrogate PK and keep UNIQUE (a_id, b_id) so the no-duplicates rule survives. ORMs (JPA) are also happier mapping a surrogate-keyed entity than a composite key class.

How would you model comments that can attach to either a post or a photo?

Polymorphic association — three approaches. (1) commentable_type + commentable_id with no FK (Rails-style): flexible, one table, but zero referential integrity — orphans accumulate and no CASCADE. (2) Exclusive arc: one nullable FK per target plus CHECK (num_nonnulls(post_id, photo_id) = 1) — full FK integrity and cascades, each new target type is one added column; my default for a handful of known types. (3) Separate post_comments/photo_comments tables: cleanest constraints, but duplicated structure and cross-type queries become UNIONs. The differentiator is explicitly naming the integrity loss in option 1 — most candidates don't.

Soft delete: how, and what's the classic gotcha?

deleted_at timestamptz, NULL meaning alive (a timestamp beats a boolean — it's audit info for free). Gotcha #1: UNIQUE(email) now blocks re-registration with a soft-deleted user's email — scope uniqueness with a partial unique index: CREATE UNIQUE INDEX ... ON users(email) WHERE deleted_at IS NULL. Gotcha #2: every query in the system must now filter deleted_at IS NULL; one miss leaks "deleted" data — mitigate with a live-rows view or ORM default filters. I'd also challenge the requirement: if undelete isn't a real feature, hard delete + an audit/history table keeps mainline queries clean, and GDPR erasure may forbid retaining the data anyway.

What is an exclusion constraint and when have you'd use one?

A generalized UNIQUE: "no two rows may satisfy this set of operator comparisons simultaneously," enforced through a GiST index. The canonical use is overlap prevention: EXCLUDE USING gist (room_id WITH =, during WITH &&) guarantees no two confirmed bookings for the same room overlap in time — atomically, for every concurrent writer, with no app-level locking. Needs the btree_gist extension to mix equality columns into the GiST index. It turns the hardest concurrency bug in booking systems (check-then-insert races) into a plain constraint violation you handle like any unique conflict.

Shared tenant_id vs schema-per-tenant vs database-per-tenant?

Shared tables + tenant_id: cheapest, migrations run once, scales to millions of tenants — but isolation is only as good as your WHERE clauses, so add Row-Level Security as a fail-closed backstop and lead every composite index with tenant_id. Schema-per-tenant: stronger separation and per-tenant dump/restore, but migrations fan out ×N and the catalog bloats past a few hundred tenants. Database-per-tenant: strongest isolation, trivial per-tenant backup, per-tenant tuning — and the highest ops cost; reserved for enterprise/regulated customers. Real systems hybridize: shared for the long tail, dedicated for the whales. Default answer: shared + RLS unless a contract says otherwise.

Why is SERIAL discouraged in favor of GENERATED AS IDENTITY?

SERIAL is pre-standard syntax sugar: it creates a sequence, sets a default, and loosely ties ownership — but permissions and copying behavior (CREATE TABLE ... LIKE) have sharp edges, and nothing stops an INSERT from supplying its own value, silently desynchronizing from the sequence (hello, duplicate-key errors after a data import). GENERATED ALWAYS AS IDENTITY is the SQL standard, manages the sequence as an integral part of the column, and rejects manual values unless you say OVERRIDING SYSTEM VALUE — use BY DEFAULT when bulk loads must specify IDs. Same runtime behavior, tighter semantics.

A UNIQUE constraint on a nullable column — what's the behavior, and how did PG 15 change the options?

By default NULLs are considered distinct from each other (SQL's three-valued logic: NULL = NULL is unknown), so a UNIQUE column can hold unlimited NULLs — usually what you want for optional-but-unique fields like external_id. Postgres 15 added UNIQUE NULLS NOT DISTINCT, which treats NULLs as equal and therefore permits at most one NULL row. Before 15, you'd emulate that with a partial unique index or a unique index on COALESCE(col, sentinel). Knowing both the default and the PG15 switch signals you actually work with the database rather than through an ORM only.

Walk me through your first five minutes in a live schema-design interview.

(1) Clarify requirements — read/write patterns, scale, what history must be preserved, what's out of scope; interviewers plant ambiguity to see if you ask. (2) List entities as nouns from the requirements, and identify which are documents (snapshot) vs catalogs (reference). (3) Sketch relationships with cardinalities, calling out M:N tables and any associative entities with attributes. (4) Write DDL for the core tables only, narrating each decision: key strategy, NOT NULL by default, CHECK for domains, FK delete policies, indexes on FKs and hot query paths. (5) Volunteer the trade-offs — where I'd denormalize under load, which invariants the DB can't own, what I'd revisit at 100× scale. The narration is the deliverable; silent perfect DDL scores worse than imperfect DDL with reasoning aloud.

Exercise 1: URL shortener

Design tables for a URL shortener: users create short links to long URLs, links can expire, and you must count clicks (with timestamp and referrer) without slowing redirects. Consider: what's the key for the short code? How do you serve the redirect fast? Where do click counts live?

Model solution + commentary
CREATE TABLE users (
    id    bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    email text NOT NULL UNIQUE
);

CREATE TABLE links (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    short_code  text NOT NULL UNIQUE CHECK (short_code ~ '^[A-Za-z0-9_-]{4,12}$'),
    long_url    text NOT NULL CHECK (char_length(long_url) <= 8192),
    owner_id    bigint REFERENCES users(id) ON DELETE SET NULL,  -- links outlive accounts
    expires_at  timestamptz,                                     -- NULL = never
    created_at  timestamptz NOT NULL DEFAULT now(),
    click_count bigint NOT NULL DEFAULT 0                        -- denormalized, eventually consistent
);
CREATE INDEX idx_links_owner ON links (owner_id);

CREATE TABLE clicks (
    id         bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    link_id    bigint NOT NULL REFERENCES links(id) ON DELETE CASCADE,
    clicked_at timestamptz NOT NULL DEFAULT now(),
    referrer   text,
    user_agent text
);
CREATE INDEX idx_clicks_link_time ON clicks (link_id, clicked_at DESC);

Decisions to narrate: The short code is the natural key of the system but I still use a surrogate id for FKs — clicks referencing an 8-byte int beats duplicating a text code into billions of rows. The redirect path is SELECT long_url, expires_at FROM links WHERE short_code = $1 — one unique-index lookup (and realistically a cache in front). Clicks are an append-only event table; the redirect inserts a click row and does not synchronously update click_count — a periodic rollup (or async increment) maintains it, because a trigger would make the link row a write hot-spot for viral links. That's a deliberate eventual-consistency call: dashboards can lag seconds; redirects can't. Expiry is checked at read time (expires_at > now() or NULL) rather than deleted by a cron — the row remains as history. At real scale I'd flag: clicks partitioned by month, and short-code generation (random base62, retry on unique violation) happens in the app.

CREATE TABLE projects (
    id     bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    org_id bigint NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    name   text NOT NULL,
    UNIQUE (org_id, name)                        -- unique PER TENANT
);
CREATE INDEX idx_projects_org ON projects (org_id);

CREATE TABLE tasks (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    org_id      bigint NOT NULL REFERENCES organizations(id),  -- denormalized on purpose (see note)
    project_id  bigint NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    assignee_id bigint REFERENCES users(id) ON DELETE SET NULL,
    title       text NOT NULL,
    status      text NOT NULL DEFAULT 'todo'
                CHECK (status IN ('todo','in_progress','done')),
    created_at  timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_tasks_org_status ON tasks (org_id, status);
CREATE INDEX idx_tasks_project    ON tasks (project_id);
CREATE INDEX idx_tasks_assignee   ON tasks (assignee_id);

-- Fail-closed isolation with RLS
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
CREATE POLICY org_isolation ON tasks
    USING (org_id = current_setting('app.org_id')::bigint);
-- (repeat per tenant-scoped table; app sets app.org_id per transaction)

Decisions to narrate: Shared tables + tenant_id, because a self-serve tracker expects thousands of orgs and per-tenant schemas/databases don't scale operationally at that count; RLS is the fail-closed backstop for a missed WHERE clause. Carrying org_id on tasks — reachable via project_id — is a small deliberate denormalization: it lets RLS and composite indexes work directly on the hottest table without a join, and it's safe because a task's org never changes. (Hardening option: UNIQUE (id, org_id) on projects plus a composite FK (project_id, org_id) from tasks makes cross-tenant mismatches impossible, not just unlikely.) Uniqueness rules are tenant-scoped — UNIQUE (org_id, name) — never global. Membership is an associative entity (role, joined_at) with a composite PK; users are global identities so one human spans orgs, which also means "user deleted" must CASCADE memberships but only SET NULL task assignments — history survives people leaving.

Exercise 3: Social follows

Model "user A follows user B" for a social app: no self-follows, no duplicate follows, and both directions must be fast — "who do I follow?" and "who follows me?" Then: where would follower counts come from at Instagram scale?

Model solution + commentary
CREATE TABLE follows (
    follower_id bigint NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    followee_id bigint NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    created_at  timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (follower_id, followee_id),          -- no duplicate follows, by construction
    CHECK (follower_id <> followee_id)               -- no self-follows
);
-- PK index serves "who do I follow?" (leading column follower_id).
-- Reverse direction needs its own index:
CREATE INDEX idx_follows_followee ON follows (followee_id, created_at DESC);

Decisions to narrate: This is a self-referencing many-to-many — one users table, a join table with both FKs pointing at it. Composite PK (follower_id, followee_id) encodes "at most one follow per ordered pair"; the CHECK bans self-follows at the schema level (a one-line constraint most candidates forget). Directionality matters: follows are ordered pairs, unlike a "friendship" which is unordered — for friendships you'd store one row with CHECK (user_a_id < user_b_id) to prevent the (A,B)/(B,A) duplicate. The PK's B-tree covers follower→followees; the second index (with created_at DESC for "recent followers" pagination) covers followee→followers — without it, "who follows me" scans. CASCADE on both FKs: follow edges are meaningless once either endpoint is gone. Counts at scale: count(*) per profile view dies at celebrity fan-in; store follower_count on a stats row — but a synchronous trigger makes the celebrity's row a lock hot-spot for millions of concurrent follows, so increment asynchronously (queue/batched rollup) and accept seconds of drift, with a periodic reconciliation job recomputing from truth. That answer — trigger rejected because of lock contention, eventual consistency chosen with a reconciliation story — is the senior version.
