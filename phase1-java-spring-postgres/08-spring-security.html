08 — Spring Security

Filter chains, JWT end-to-end, OAuth2/OIDC — the backend half of the token story you already know from React.

Security is one of the highest-signal interview topics for a senior full-stack role. Interviewers ask it because it exposes whether you understand systems, not just syntax: how a request physically travels through the server, where identity gets attached to it, what "stateless" actually buys you at scale, and what it costs you (revocation). As someone who has spent seven years on the React side arguing about localStorage vs cookies, you already have half the picture — the client half. This page gives you the server half, so in an interview you can narrate the entire lifecycle of a token: minted on the backend, stored on the frontend, validated on every request, rotated, and revoked. Very few candidates can do that end-to-end; it is a genuine differentiator.

Everything on this page is Spring Boot 3.x / Spring Security 6 style: a SecurityFilterChain @Bean with the lambda DSL. The old WebSecurityConfigurerAdapter (extending a class and overriding configure()) is deprecated and removed — you may still see it in legacy codebases and older tutorials, and it's worth being able to say "that's the pre-5.7 style; today you declare a SecurityFilterChain bean instead."

Architecture: how a request flows through Spring Security T1

Before any annotation or config makes sense, you need the mental model: Spring Security is a chain of servlet filters that runs before your controller. Nothing magical — the same Servlet API Filter interface that has existed since the early 2000s. Spring Security's cleverness is in how it plugs Spring-managed beans into that non-Spring servlet world, and in the specific ordered set of filters it installs.

The bridge: DelegatingFilterProxy → FilterChainProxy → SecurityFilterChain

The servlet container (Tomcat) knows nothing about Spring beans. So the wiring goes in three hops:

  1. DelegatingFilterProxy — a plain servlet filter registered with Tomcat. Its only job is to look up a Spring bean (named springSecurityFilterChain) and delegate to it. It's the bridge from the servlet world into the Spring ApplicationContext.
  2. FilterChainProxy — the bean that DelegatingFilterProxy delegates to. It holds one or more SecurityFilterChain instances and picks the first one whose request matcher matches the incoming URL. This is why you can have separate chains for /api/** (stateless JWT) and /admin/** (form login) in one app. It's also the single best place to put a debugger breakpoint.
  3. SecurityFilterChain — an ordered list of security filters plus a request matcher. This is what your @Bean method builds and returns.

Two filters at the end deserve special attention because interviewers probe them:

FilterResponsibilityInterview angle
ExceptionTranslationFilter Sits before AuthorizationFilter and catches its exceptions. Translates AuthenticationException → invoke the AuthenticationEntryPoint (401 territory) and AccessDeniedException → invoke the AccessDeniedHandler (403 territory). "Why do I get a redirect to /login instead of a 401 on my API?" — because the default entry point for form login redirects. You override the entry point (shown in the JWT section).
AuthorizationFilter The last security filter. Evaluates your authorizeHttpRequests rules against the current Authentication. Replaced the older FilterSecurityInterceptor. Authentication filters run earlier; authorization is decided last, once identity is established.

SecurityContextHolder: where "who is logged in" lives

Once a filter authenticates the request, it stores the result in the SecurityContextHolder — by default a ThreadLocal. Every piece of code downstream on the same thread (controllers, services, @PreAuthorize expressions) reads the current user from there without you passing it around as a parameter. 

// Anywhere downstream of the filter chain, on the request thread:
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();

// Or more idiomatically in a controller:
@GetMapping("/me")
public UserDto me(Authentication authentication) {   // injected by Spring MVC
    return userService.findByUsername(authentication.getName());
}

ThreadLocal and async code. Because the context is thread-bound, it does not automatically follow you into @Async methods, CompletableFuture.supplyAsync(...), or a manually created thread — the new thread sees an empty context and your ownership checks silently see an anonymous user. Fixes: set the holder strategy to MODE_INHERITABLETHREADLOCAL (fragile with thread pools, since pooled threads are reused and inherit stale contexts), or better, wrap executors with DelegatingSecurityContextExecutor / use Spring's @Async with a security-context-propagating task decorator. This is a classic senior-level probe.

The Authentication object

Authentication is the central token of identity, with three parts:

PartMeaningTypical content after login
getPrincipal()Who the user isA UserDetails object (or a JWT subject string)
getCredentials()Proof of identityThe password — erased after authentication so it doesn't linger in memory
getAuthorities()What they may doCollection of GrantedAuthority, e.g. ROLE_ADMIN, invoice:write

The same class plays two roles: before authentication it's a request ("here's a username and password, please verify"), and after it's a verified fact ("this is Alice with these authorities", isAuthenticated() == true).

AuthenticationManager → ProviderManager → AuthenticationProvider → UserDetailsService

How does an unverified Authentication become a verified one? Through this delegation chain:

UsernamePasswordAuthenticationFilter (or your login endpoint)
        │  authenticate(unverifiedToken)
        ▼
AuthenticationManager  (interface — "verify this for me")
        │
        ▼
ProviderManager        (the standard implementation: a list of providers)
        │  asks each: "supports() this token type?"
        ├──▶ DaoAuthenticationProvider ──▶ UserDetailsService.loadUserByUsername(name)
        │         │                              │ returns UserDetails (hash, authorities, flags)
        │         └── PasswordEncoder.matches(raw, storedHash)
        │                 │
        │                 ▼  success
        │         fully-populated Authentication (credentials erased)
        ├──▶ JwtAuthenticationProvider   (if using oauth2ResourceServer)
        └──▶ ... other providers (LDAP, OTP, remember-me)

The layering exists so authentication mechanisms are pluggable: the same ProviderManager can try a DAO provider, an LDAP provider, and a JWT provider, and the first that supports the token type and succeeds wins. In interviews, being able to sketch this chain from memory — filter → manager → provider → UserDetailsService + PasswordEncoder — is a strong T1 answer.

A senior framing: "Spring Security is two orthogonal machines. Machine one is authentication — a pipeline that turns credentials into a verified Authentication and parks it in a ThreadLocal. Machine two is authorization — declarative rules (AuthorizationFilter for URLs, @PreAuthorize for methods) evaluated against that object. Almost every customization is either a new way to populate the context (a filter/provider) or a new rule that reads it."

JWT end-to-end: a complete stateless auth setup T1

This is the centerpiece. In most interviews for a Spring backend role you will be asked to explain — or whiteboard — a JWT setup. Below is the whole thing: token anatomy, the trade-offs, and a complete, compilable implementation (config, login endpoint, validation filter, refresh tokens), then the frontend storage debate you already know from the other side.

Anatomy: header.payload.signature

A JWT is three base64url-encoded segments joined by dots:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9        ◀ header    {"alg":"HS256","typ":"JWT"}
.eyJzdWIiOiJhbGljZSIsImlhdCI6MTcyMDAwMDAw   ◀ payload   {"sub":"alice","iat":1720000000,
MCwiZXhwIjoxNzIwMDAwOTAwLCJyb2xlcyI6WyJVU0   "exp":1720000900,"roles":["USER"]}
VSIl19
.4kX9v2mZ...                                ◀ signature  HMAC-SHA256(base64(header)+"."+base64(payload), secret)
ClaimMeaningNotes
subSubject — the user identifierPrefer a stable ID over an email that can change
iatIssued-at (epoch seconds)Useful for "invalidate everything issued before X" strategies
expExpiryValidation MUST reject expired tokens; libraries do this for you
iss / audIssuer / audienceValidate both in multi-service setups — a token for service A shouldn't work on service B
custome.g. roles, tenantIdKeep it small (it rides on every request) and never put secrets in it

The payload is encoded, not encrypted. Anyone — including your React DevTools — can base64-decode it. The signature guarantees integrity (nobody tampered with it) and authenticity (your server issued it), not confidentiality. Saying "the JWT is encrypted so it's safe to put data in it" is an instant red flag in an interview. (Encrypted JWTs — JWE — exist but are rare in typical app auth.)

Signing: HS256 vs RS256

HS256 (symmetric, HMAC)RS256 / ES256 (asymmetric)
KeysOne shared secret signs and verifiesPrivate key signs; public key verifies
Best forSingle service that both issues and validates its own tokensMicroservices / third-party IdP: only the auth server holds the private key; every resource service verifies with the public key (published via JWKS)
Risk profileEvery service holding the secret can mint valid tokens — a compromise anywhere forges identity everywhereResource services can only verify; compromising one doesn't let attackers forge tokens
Key rotationRedistribute the secret to all partiesRotate at the auth server, publish new public key via JWKS endpoint; validators pick it up automatically

Rule of thumb for interviews: one service → HS256 is fine; multiple validators or an external IdP → asymmetric, because the signing capability must stay in exactly one place.

Why stateless — and the honest cost

Classic session auth stores state server-side: login creates a session record, the client gets an opaque JSESSIONID cookie, and every request triggers a session lookup. That means either sticky sessions or a shared session store (Redis) once you scale past one instance.

A JWT flips this: the token is the state. Validation is a local signature check — no database hit, no shared store, any instance behind the load balancer can serve any request. That's the "horizontal scaling" win.

The senior answer always names the cost: you gave up instant revocation. A signed token is valid until exp no matter what — you can't "log someone out" server-side, and a stolen access token works until it expires. Mitigations, in order of common use: (1) keep access tokens short-lived (5–15 min) and pair them with revocable refresh tokens — this is the standard answer; (2) keep a denylist of revoked token IDs (jti) in Redis — but note you've reintroduced shared state, which is exactly the honest trade-off interviewers want you to acknowledge; (3) key rotation as a nuclear option. If someone claims JWTs are strictly better than sessions, they haven't been paged for a "disable this compromised account NOW" incident.

The complete implementation

Dependencies: spring-boot-starter-security plus a JWT library — here io.jsonwebtoken:jjwt-api/impl/jackson (0.12.x API).

1. The SecurityFilterChain bean — stateless, /auth/** open, everything else authenticated, JSON 401s:

@Configuration
@EnableWebSecurity
@EnableMethodSecurity   // enables @PreAuthorize et al. — see Authorization section
public class SecurityConfig {

    private final JwtAuthFilter jwtAuthFilter;

    public SecurityConfig(JwtAuthFilter jwtAuthFilter) {
        this.jwtAuthFilter = jwtAuthFilter;
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // Stateless token API: no session cookie, so CSRF's ambient-credential
            // problem doesn't apply (see the CSRF section for the full nuance).
            .csrf(csrf -> csrf.disable())
            .cors(Customizer.withDefaults())          // picks up the CorsConfigurationSource bean
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**").permitAll()
                .requestMatchers("/actuator/health").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())
            // APIs must answer 401/403 as JSON — never a redirect to a login page.
            .exceptionHandling(ex -> ex
                .authenticationEntryPoint((request, response, authEx) -> {
                    response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
                    response.setContentType("application/json");
                    response.getWriter().write("{\"error\":\"unauthorized\"}");
                })
                .accessDeniedHandler((request, response, deniedEx) -> {
                    response.setStatus(HttpServletResponse.SC_FORBIDDEN);
                    response.setContentType("application/json");
                    response.getWriter().write("{\"error\":\"forbidden\"}");
                }))
            // Our JWT filter runs where the username/password filter would sit.
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return PasswordEncoderFactories.createDelegatingPasswordEncoder(); // bcrypt by default
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration config)
            throws Exception {
        return config.getAuthenticationManager();  // exposes the auto-configured manager
    }
}

2. The token service — issuing and parsing:

@Service
public class JwtService {

    private final SecretKey key;
    private final Duration accessTtl = Duration.ofMinutes(15);

    public JwtService(@Value("${app.jwt.secret}") String secret) {
        // >= 256-bit secret for HS256; injected from env / vault, never hardcoded
        this.key = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }

    public String generateAccessToken(UserDetails user) {
        Instant now = Instant.now();
        return Jwts.builder()
            .subject(user.getUsername())
            .issuedAt(Date.from(now))
            .expiration(Date.from(now.plus(accessTtl)))
            .claim("roles", user.getAuthorities().stream()
                .map(GrantedAuthority::getAuthority).toList())
            .signWith(key)                       // HS256 inferred from key type
            .compact();
    }

    /** Parses AND validates: signature + exp. Throws JwtException if invalid/expired. */
    public Claims parse(String token) {
        return Jwts.parser()
            .verifyWith(key)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }
}

3. The login endpoint — authenticates credentials via the AuthenticationManager (which walks the provider chain from the architecture section), then mints tokens:

@RestController
@RequestMapping("/auth")
public class AuthController {

    private final AuthenticationManager authenticationManager;
    private final JwtService jwtService;
    private final RefreshTokenService refreshTokenService;

    // constructor omitted for brevity

    @PostMapping("/login")
    public ResponseEntity<TokenResponse> login(@RequestBody LoginRequest req) {
        // Delegates to ProviderManager → DaoAuthenticationProvider →
        // UserDetailsService + PasswordEncoder. Throws BadCredentialsException → 401.
        Authentication auth = authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(req.username(), req.password()));

        UserDetails user = (UserDetails) auth.getPrincipal();
        String accessToken  = jwtService.generateAccessToken(user);
        String refreshToken = refreshTokenService.issue(user.getUsername());

        // Refresh token as httpOnly cookie; access token in the JSON body
        // (see "Frontend storage" below for why this split).
        ResponseCookie cookie = ResponseCookie.from("refresh_token", refreshToken)
            .httpOnly(true)
            .secure(true)
            .sameSite("Strict")
            .path("/auth/refresh")          // only sent to the refresh endpoint
            .maxAge(Duration.ofDays(14))
            .build();

        return ResponseEntity.ok()
            .header(HttpHeaders.SET_COOKIE, cookie.toString())
            .body(new TokenResponse(accessToken, 900)); // expires_in seconds
    }

    public record LoginRequest(String username, String password) {}
    public record TokenResponse(String accessToken, long expiresIn) {}
}

4. The validation filter — the piece that runs on every request. It extends OncePerRequestFilter (guaranteed once per request even through forwards/error dispatches). The four-step rhythm to memorize: extract → validate → build Authentication → set context.

@Component
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtService jwtService;

    public JwtAuthFilter(JwtService jwtService) {
        this.jwtService = jwtService;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        // 1. Extract — no token? Not our problem; continue. AuthorizationFilter
        //    will 401 protected routes later via the entry point.
        String header = request.getHeader(HttpHeaders.AUTHORIZATION);
        if (header == null || !header.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }
        String token = header.substring(7);

        try {
            // 2. Validate — signature + expiry checked inside parse().
            Claims claims = jwtService.parse(token);

            // 3. Build a verified Authentication from the claims.
            @SuppressWarnings("unchecked")
            List<String> roles = claims.get("roles", List.class);
            List<SimpleGrantedAuthority> authorities = roles.stream()
                .map(SimpleGrantedAuthority::new)
                .toList();

            var authentication = new UsernamePasswordAuthenticationToken(
                claims.getSubject(),   // principal
                null,                  // credentials — never carry the token forward
                authorities);
            authentication.setDetails(
                new WebAuthenticationDetailsSource().buildDetails(request));

            // 4. Park it in the ThreadLocal so the rest of the request sees it.
            SecurityContextHolder.getContext().setAuthentication(authentication);

        } catch (ExpiredJwtException e) {
            // Expired: fall through unauthenticated. The React client sees 401,
            // its interceptor calls /auth/refresh, retries the original request.
        } catch (JwtException e) {
            // Malformed / bad signature: also fall through unauthenticated.
        }

        filterChain.doFilter(request, response);
    }
}

Don't write the 401 inside the filter for a missing token. A common junior mistake is to reject every request without a token inside the JWT filter — which breaks permitAll() routes like /auth/login. The filter's job is only to populate the context when a valid token exists; enforcement belongs to AuthorizationFilter + your JSON AuthenticationEntryPoint. Also note the filter is a @Component registered via addFilterBefore — if you additionally let Spring Boot auto-register it as a plain servlet filter, it runs twice; suppress that with a FilterRegistrationBean set to disabled, or don't mark it @Component and construct it in the config class instead.

Access + refresh token pattern

The standard resolution of the stateless-vs-revocation tension:

  • Access token: short-lived (5–15 min), stateless, validated by signature only. Sent as Authorization: Bearer ... on every API call. If stolen, the blast radius is minutes.
  • Refresh token: long-lived (days/weeks), stored server-side (a DB/Redis row: token hash, user, expiry, revoked flag). Used only against /auth/refresh to mint a new access token. Because the server keeps a record, it is revocable — delete the row and the session dies at the next refresh. Logout = delete the row.
  • Rotation: every refresh invalidates the old refresh token and issues a new one. If a rotated (already-used) token is presented again, treat it as theft and revoke the whole token family — this is OAuth-standard "refresh token rotation with reuse detection".
@Entity
public class RefreshToken {
    @Id @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    private String tokenHash;     // store a SHA-256 hash, never the raw token
    private String username;
    private Instant expiresAt;
    private boolean revoked;
    private UUID familyId;        // for reuse-detection: revoke the whole family
    // getters/setters omitted
}

@PostMapping("/refresh")
public ResponseEntity<TokenResponse> refresh(
        @CookieValue("refresh_token") String refreshToken) {
    // Validates against the store, checks revoked/expiry, ROTATES:
    // old row marked revoked, new row inserted, new cookie set.
    RotationResult result = refreshTokenService.rotate(refreshToken); // throws → 401
    return ResponseEntity.ok()
        .header(HttpHeaders.SET_COOKIE, result.newCookie().toString())
        .body(new TokenResponse(result.newAccessToken(), 900));
}

Frontend storage: the debate you already know — now with the full picture

You've had the localStorage-vs-cookies argument from the React side. Here is the honest trade-off table an interviewer wants, including what each choice demands of the backend:

StrategyXSS exposureCSRF exposureBackend implicationsVerdict
localStorage High — any injected script reads it and exfiltrates; token usable from the attacker's machine until expiry None (JS must attach it explicitly, so no ambient sending) Simplest: pure Bearer header, CSRF stays off Convenient; hard to defend for long-lived tokens in front of a security-minded interviewer
httpOnly cookie Script cannot read it (exfiltration blocked) — but note XSS can still use it by firing requests from the victim's browser Yes — the browser attaches it automatically, so you need SameSite=Lax/Strict and/or CSRF tokens again Backend must set/clear cookies, configure SameSite, possibly re-enable CSRF protection; CORS needs allowCredentials(true) Strongest at-rest protection; the classic choice for the refresh token
In-memory (JS variable / React context) + silent refresh Lower — nothing persisted to steal at rest; gone on tab close None for the access token itself Needs a refresh endpoint the app calls on boot/401 to re-obtain an access token Best for the short-lived access token

Recommended default (and a great interview answer): refresh token in an httpOnly, Secure, SameSite=Strict cookie scoped to /auth/refresh; access token kept in memory only, re-obtained via silent refresh on page load and on 401. Rationale: the long-lived credential is unreadable by script and only ever travels to one endpoint; the short-lived one is never persisted, so a page-refresh "logout" costs one silent refresh call, not UX pain. Then close with the honest caveat: none of this survives XSS running in your page — an attacker's script can still call your API as the user even if it can't read the cookie — so token storage hygiene complements, never replaces, XSS prevention (output encoding, CSP, dependency audits). Saying that last sentence is what separates senior answers from cargo-cult ones.

Authentication vs authorization, UserDetailsService, PasswordEncoder T2

Returning 403 for a missing/expired token (or 401 for an insufficient role) is a small thing that interviewers — and frontend interceptors — notice. Your React 401-interceptor triggers a token refresh; if the backend sends 403 for expired tokens, the refresh flow never fires. Get the semantics right on the backend and the frontend logic writes itself.

UserDetailsService + UserDetails backed by JPA

UserDetailsService is the single-method port between Spring Security and your user storage: "given a username, return a UserDetails". DaoAuthenticationProvider calls it during login, then checks the password. A production-typical JPA-backed implementation: 

@Entity
@Table(name = "users")
public class UserAccount {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, nullable = false)
    private String username;

    @Column(nullable = false)
    private String passwordHash;   // bcrypt output, e.g. {bcrypt}$2a$10$...

    private boolean enabled = true;

    @ElementCollection(fetch = FetchType.EAGER)   // small set; avoids LazyInit outside session
    @CollectionTable(name = "user_roles", joinColumns = @JoinColumn(name = "user_id"))
    @Column(name = "role")
    private Set<String> roles = new HashSet<>();   // "ADMIN", "USER" — no prefix stored
    // getters/setters omitted
}

@Service
public class JpaUserDetailsService implements UserDetailsService {

    private final UserAccountRepository repo;

    public JpaUserDetailsService(UserAccountRepository repo) { this.repo = repo; }

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        UserAccount u = repo.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("No user: " + username));

        return org.springframework.security.core.userdetails.User.builder()
            .username(u.getUsername())
            .password(u.getPasswordHash())
            .disabled(!u.isEnabled())
            .roles(u.getRoles().toArray(String[]::new))  // .roles() ADDS the ROLE_ prefix
            .build();
    }
}

Declaring this bean is all it takes — Spring Boot wires it into the auto-configured DaoAuthenticationProvider. UserDetails itself is just a contract: username, password hash, authorities, and four boolean status flags (enabled, non-locked, non-expired account, non-expired credentials) that the provider checks and converts into specific exceptions (DisabledException, LockedException...).

PasswordEncoder: why BCrypt and not SHA-256

Never store plaintext, and never store a fast general-purpose hash either. SHA-256 was engineered to be fast — a GPU rig computes billions per second, so a leaked table of SHA-256(password) falls to brute force and rainbow tables quickly. Password hashing needs the opposite property: deliberately slow, memory- or compute-hard functions.

  • BCrypt — the Spring default. Two key properties:
    • Work factor (cost): new BCryptPasswordEncoder(12) means 212 internal rounds. Each +1 doubles the cost — a tunable knob so hashing stays slow (~100–500ms target) as hardware improves. Old hashes verify at their embedded cost; upgrade on next successful login.
    • Per-password random salt, generated automatically and stored inside the output string ($2a$12$<salt><hash>). Consequence one: two users with the same password get different hashes — rainbow tables are useless. Consequence two: encode() is non-deterministic, so you can never compare hashes with equals(); you must call encoder.matches(raw, stored), which extracts the salt and re-hashes.
  • Argon2 — the modern memory-hard alternative (mention it to score points: memory-hardness resists GPU/ASIC attacks better than BCrypt's CPU-hardness).

DelegatingPasswordEncoder — what PasswordEncoderFactories.createDelegatingPasswordEncoder() gives you — prefixes every stored hash with its algorithm id:

{bcrypt}$2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG
{argon2}$argon2id$v=19$m=16384,t=2,p=1$...
{noop}plaintext-for-tests-only

At matches() time it reads the prefix and routes to the right encoder. The payoff is algorithm migration without a big-bang rehash: old {bcrypt} hashes keep verifying while new/changed passwords get {argon2}. You can't rehash proactively — you don't have the plaintext — so migration happens lazily at login. That "why can't you just rehash everyone's password?" follow-up is a favorite.

The infamous startup error There is no PasswordEncoder mapped for the id "null" means a stored hash has no {id} prefix while a delegating encoder is in play — typically hand-inserted seed data. Fix the data (add {bcrypt}) rather than downgrading to a raw BCryptPasswordEncoder bean.

Authorization: URL rules and method security T2

URL-level: authorizeHttpRequests

Rules are evaluated top-down, first match wins — so order from most to least specific:

.authorizeHttpRequests(auth -> auth
    .requestMatchers("/auth/**", "/public/**").permitAll()
    .requestMatchers(HttpMethod.GET, "/api/products/**").permitAll()   // method-scoped
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/reports/**").hasAnyRole("ADMIN", "ANALYST")
    .requestMatchers("/api/billing/**").hasAuthority("billing:write")
    .anyRequest().authenticated())                                       // catch-all LAST

If anyRequest().authenticated() comes first, every later rule is dead code (Security 6 actually rejects rules after anyRequest() at startup — but the ordering trap between two overlapping requestMatchers patterns remains yours to get right).

hasRole vs hasAuthority — the ROLE_ prefix trap

Internally there are only GrantedAuthority strings. A "role" is merely an authority that starts with ROLE_, and the convenience methods handle the prefix differently:

CallActually checks for authority
hasRole("ADMIN")ROLE_ADMIN — the prefix is added for you (passing "ROLE_ADMIN" here throws)
hasAuthority("ROLE_ADMIN")ROLE_ADMIN — exact string, no magic
hasAuthority("invoice:write")invoice:write — fine-grained permissions, no prefix convention

The trap in practice: your JWT filter builds authorities straight from a claim. If the claim says "ADMIN" and you construct new SimpleGrantedAuthority("ADMIN"), then hasRole("ADMIN") fails forever because it's looking for ROLE_ADMIN. Symmetrically, the builder's .roles("ADMIN") adds the prefix while .authorities("ADMIN") does not. When you see mysterious 403s for a user who "definitely has the role", check the prefix first.

Method security: @EnableMethodSecurity + @PreAuthorize

URL rules are coarse. Real business rules — "a user may only cancel their own order" — live at the service layer with @PreAuthorize, enabled by @EnableMethodSecurity on a config class (the Security 6 replacement for the legacy @EnableGlobalMethodSecurity). It works via AOP proxies, which foreshadows the next chapter: self-invocation (this.method()) bypasses the proxy and therefore the check. 

@Service
public class OrderService {

    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(Long orderId) { ... }

    // SpEL can reference method params (#param) and the Authentication object.
    // The canonical ownership check:
    @PreAuthorize("#userId == authentication.name")
    public List<Order> ordersForUser(String userId) { ... }

    // Combine: admins see anyone; users see themselves.
    @PreAuthorize("hasRole('ADMIN') or #userId == authentication.name")
    public Profile profile(String userId) { ... }

    // Delegate complex logic to a bean — keeps SpEL short and testable:
    @PreAuthorize("@orderGuard.isOwner(#orderId, authentication)")
    public void cancelOrder(Long orderId) { ... }
}

@Component("orderGuard")
public class OrderGuard {
    private final OrderRepository orders;
    public OrderGuard(OrderRepository orders) { this.orders = orders; }

    public boolean isOwner(Long orderId, Authentication auth) {
        return orders.findById(orderId)
            .map(o -> o.getOwnerUsername().equals(auth.getName()))
            .orElse(false);   // unknown order → deny; 404-vs-403 leakage is a design choice
    }
}

@PostAuthorize runs the check after the method, with returnObject available — e.g. @PostAuthorize("returnObject.owner == authentication.name") when you can only decide after loading the entity. Know it exists and its caveat: the method has already executed, so it must be side-effect-free (never @PostAuthorize a delete). There are also @PreFilter/@PostFilter for collection filtering — mention-level awareness is enough.

Role hierarchy awareness

Without configuration, ROLE_ADMIN does not imply ROLE_USER — an admin fails hasRole('USER') checks, which surprises people. Instead of littering every rule with hasAnyRole('USER','ADMIN'), declare a hierarchy bean: 

@Bean
public RoleHierarchy roleHierarchy() {
    return RoleHierarchyImpl.withDefaultRolePrefix()
        .role("ADMIN").implies("MANAGER")
        .role("MANAGER").implies("USER")
        .build();
}

URL rules pick this up automatically in Security 6; for method security, also expose a MethodSecurityExpressionHandler bean carrying the hierarchy.

CSRF and CORS: the nuances interviewers love T2

What CSRF actually is

Cross-Site Request Forgery exploits the browser's habit of attaching cookies automatically. The user is logged into bank.com (session cookie present). They visit evil.com, which contains: 

<form action="https://bank.com/api/transfer" method="POST">
  <input type="hidden" name="to" value="attacker">
  <input type="hidden" name="amount" value="10000">
</form>
<script>document.forms[0].submit()</script>

The browser dutifully attaches the bank.com session cookie to that cross-site POST. The server sees a valid session and executes the transfer. The attacker never read the cookie — they just rode along on it. The key phrase: CSRF abuses an ambient credential, something the browser sends without the page's JavaScript doing anything. Defenses: a per-session CSRF token that legitimate pages must echo back (evil.com can't read it, thanks to the same-origin policy), and/or SameSite cookies that simply aren't sent on cross-site requests.

Why it's disabled for stateless token APIs — and the nuance

If your API authenticates only via an Authorization: Bearer header, there is no ambient credential: evil.com's auto-submitted form cannot set that header, and its JS cannot read your token (different origin, and it's not in a cookie anyway). No ambient credential → no CSRF vector → the protection is dead weight that breaks every POST from clients that don't send the token. Hence the standard line in a stateless config: 

.csrf(csrf -> csrf.disable())   // safe IF AND ONLY IF auth is header-based, not cookie-based

The nuance interviewers love: "we use JWT, therefore CSRF off" is only half-true. The moment you put the JWT (or the refresh token) in a cookie — which the previous section just recommended for refresh tokens! — you have recreated an ambient credential, and CSRF is back on the table for whatever endpoints that cookie reaches. Mitigations: SameSite=Strict (or Lax, which still blocks cross-site POSTs), a narrow cookie path (ours only travels to /auth/refresh), and — belt and braces — CSRF tokens on the cookie-authenticated endpoints. The decision key is "how does authentication arrive?", not "do we use JWTs?". Delivering that distinction unprompted is a strong senior signal.

CORS + Security interplay

You know CORS from the browser side: it's the browser enforcing which origins may read responses, driven by server headers. Two backend-specific points:

  1. CORS must be handled inside the security chain. A preflight OPTIONS request carries no Authorization header (the browser sends it before your JS gets a say). If Spring Security runs first and 401s the preflight, the real request never happens — you'll see a CORS error in the React console that is actually an auth error. .cors(Customizer.withDefaults()) puts a CorsFilter early in the security chain so preflights are answered before authentication.
  2. Credentials tighten the rules. If the frontend sends cookies (fetch(..., { credentials: 'include' }) for your refresh cookie), the server must respond with Access-Control-Allow-Credentials: true and a specific origin — the wildcard * is forbidden with credentials by the spec.
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://app.example.com")); // no "*" with credentials
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type"));
    config.setAllowCredentials(true);            // needed for the refresh cookie
    config.setMaxAge(3600L);                     // cache preflight for an hour

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}

CORS is not a security control for your API — it protects browser users, not servers. curl, Postman, and any non-browser client ignore it entirely. If an interviewer asks "does CORS stop attackers from calling my API?", the answer is no — authentication does. Confusing CORS with access control is a common misconception worth explicitly disclaiming.

OAuth2 & OIDC: the flows behind "Login with Google" T2

The four roles

RoleDefinitionIn a "React SPA + Spring API + Google" setup
Resource ownerThe human who owns the dataYour user
ClientThe app requesting access on the user's behalfYour React SPA (a "public client" — it can't keep a secret)
Authorization serverAuthenticates the user, issues tokensGoogle (or Keycloak/Auth0/Okta/Cognito)
Resource serverThe API that accepts and validates access tokensYour Spring Boot API

OAuth2's core insight: the user never gives their password to the client. They authenticate at the authorization server, which hands the client a scoped, expiring token. OAuth2 is a delegated authorization protocol — "authorization" is in the name; identity comes later with OIDC.

Authorization Code + PKCE — the flow your SPA uses

PKCE ("pixie", Proof Key for Code Exchange) exists because a SPA can't hold a client secret — anything shipped to the browser is public. PKCE replaces the static secret with a per-login dynamic proof: 

 React SPA                        Auth server (Google)                 Spring API
    │                                     │                                │
 1. │ generate code_verifier (random)     │                                │
    │ code_challenge = SHA256(verifier)   │                                │
    │                                     │                                │
 2. │── redirect: /authorize?response_type=code                            │
    │        &client_id=...&redirect_uri=...&scope=openid profile          │
    │        &code_challenge=XYZ&code_challenge_method=S256 ──▶            │
    │                                     │                                │
 3. │        user logs in + consents      │  (auth server stores XYZ)      │
    │                                     │                                │
 4. │◀── redirect back: https://app.example.com/callback?code=abc123 ──    │
    │                                     │                                │
 5. │── POST /token { code=abc123, code_verifier=<original random> } ──▶   │
    │                                     │ verifies SHA256(verifier)==XYZ │
 6. │◀── { access_token, id_token, refresh_token } ──                      │
    │                                     │                                │
 7. │── GET /api/orders  Authorization: Bearer <access_token> ────────────▶│
    │                                     │       validates signature via  │
    │                                     │       Google's JWKS, checks    │
    │                                     │       iss/aud/exp ─▶ 200       │

Why PKCE defeats code interception: an attacker who steals the authorization code at step 4 (say, via a malicious redirect handler) can't redeem it at step 5 without the code_verifier, which never left the legitimate app and can't be derived from the challenge (SHA-256 is one-way).

Why the implicit flow is dead: it returned the access token directly in the redirect URL fragment (#access_token=...) — exposed in browser history, referrer leaks, and interceptable by any script, with no proof-of-possession step at all. It existed only because pre-CORS browsers couldn't make the step-5 POST cross-origin. CORS fixed that, PKCE fixed the public-client problem, and OAuth 2.1 removes implicit entirely. Today: authorization code + PKCE for every client type, confidential clients simply add their secret too.

OIDC: the identity layer

OAuth2 by itself answers "may this app access this resource?" — not "who is the user?". OpenID Connect is a thin layer on top that adds identity: the openid scope, a /userinfo endpoint, discovery documents (/.well-known/openid-configuration), and crucially the ID token:

ID tokenAccess token
AudienceThe client (your SPA)The resource server (your API)
PurposeProves who logged in (claims: sub, email, name)Grants access to APIs (claims: scope, aud)
FormatAlways a JWT (per OIDC spec)Opaque or JWT — provider's choice
Common mistakeSending the ID token to your API as if it were an access token — wrong audience, wrong purpose; APIs must reject it via the aud check

Spring as resource server vs client

For the typical SPA + API architecture, your Spring app is a resource server: it never redirects anyone to a login page; it just validates bearer tokens. The configuration is remarkably small: 

// build.gradle: spring-boot-starter-oauth2-resource-server

@Bean
public SecurityFilterChain resourceServerChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable())
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/actuator/health").permitAll()
            .anyRequest().authenticated())
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(Customizer.withDefaults()));   // installs BearerTokenAuthenticationFilter
    return http.build();
}
# application.yml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://accounts.google.com
          # On startup Spring fetches {issuer}/.well-known/openid-configuration,
          # discovers the JWKS URI, and validates every token's signature,
          # exp, and iss claim automatically. Add audience validation yourself.

Notice what you did not write: no JWT parsing code, no key management, no custom filter — the issuer-uri drives discovery, JWKS fetching, key rotation, and validation. This replaces the entire hand-rolled JwtService/JwtAuthFilter from the JWT section when an external IdP issues your tokens. (Hand-rolling remains worth knowing: interviews ask for it, and self-issued tokens are common.) To map token scopes/roles into authorities your hasRole checks understand, plug in a JwtAuthenticationConverter.

The client role (spring-boot-starter-oauth2-client + oauth2Login()) is for server-rendered apps or a BFF: Spring itself performs the redirect dance, holds the tokens server-side, and manages the session. "Login with Google" in a Spring MVC app is ~10 lines of YAML (client-id, client-secret, provider google) plus .oauth2Login(Customizer.withDefaults()). High-level wiring: browser hits a protected page → Spring redirects to Google's authorize endpoint → user consents → Google redirects back to /login/oauth2/code/google → Spring exchanges the code (with its client secret), fetches the ID token/userinfo, and populates an OAuth2AuthenticationToken in the session.

A pattern worth naming in system-design rounds: BFF (Backend-for-Frontend). Instead of the SPA holding tokens at all, a thin server-side layer acts as the OAuth client, keeps tokens server-side, and gives the browser only an httpOnly session cookie. Trade-off: best token security in the browser (there are no tokens in the browser) versus reintroducing server-side session state and another hop. Increasingly recommended by the IETF browser-app BCP; acknowledging it shows you're current.

Rate limiting: where it belongs

Spring Security has no built-in rate limiter — placement is an architecture question. Options, outermost first: edge/CDN (Cloudflare, WAF — cheapest place to shed volumetric abuse), API gateway (Spring Cloud Gateway's RequestRateLimiter with Redis — right place for per-client quotas across service instances; note in-memory buckets don't work behind a load balancer), and in-app filter (e.g. Bucket4j in a OncePerRequestFilter) for endpoint-specific rules that need app context — classically, aggressive limits on /auth/login to blunt credential stuffing, keyed by IP + username, returning 429 with Retry-After. Senior answer: defense in depth — coarse limits at the edge, precise business-aware limits near the endpoint.

Secrets management

  • Never commit secrets — no JWT signing keys, DB passwords, or OAuth client secrets in application.yml in git. A leaked signing key means anyone can mint valid tokens for your API; git history is forever, so a committed secret is a rotate-now incident, not a "deleted it in the next commit" fix.
  • Environment variables as the baseline: app.jwt.secret=${JWT_SECRET}, injected by the platform (Kubernetes Secret, ECS task definition, Heroku config vars).
  • Vault-class systems (HashiCorp Vault via Spring Cloud Vault, AWS Secrets Manager/Parameter Store) for the mature setup: central storage, audit trails, automatic rotation, short-lived dynamic DB credentials. Spring Cloud Vault makes vault entries look like ordinary properties.
  • Plan rotation from day one: for JWT keys, that's exactly the JWKS + kid (key-id header) mechanism asymmetric signing gives you — validators accept tokens from both old and new keys during the overlap window.

Interview Q&A

Walk me through what happens when an HTTP request hits a Spring Security-protected endpoint.

Tomcat runs its servlet filter chain; one of those filters is DelegatingFilterProxy, which bridges into the Spring context by delegating to the FilterChainProxy bean. FilterChainProxy matches the URL against its registered SecurityFilterChains and runs the first match's ordered filters: context restoration (SecurityContextHolderFilter), CORS, CSRF, then authentication filters (form login's UsernamePasswordAuthenticationFilter, or a JWT filter that parses the Bearer token and populates the SecurityContextHolder), then ExceptionTranslationFilter, and finally AuthorizationFilter, which evaluates the authorizeHttpRequests rules against the Authentication. If authorization passes, the request proceeds to DispatcherServlet and the controller; if it fails, ExceptionTranslationFilter routes to the AuthenticationEntryPoint (401) or AccessDeniedHandler (403).

What is the SecurityContextHolder and what's the gotcha with async code?

It's the holder for the current SecurityContext (which wraps the Authentication), backed by a ThreadLocal by default, so any code on the request thread can ask "who's the current user" without parameter-passing. The gotcha: ThreadLocals don't cross thread boundaries, so @Async methods, parallel streams, or CompletableFuture work on pool threads see an empty context — checks silently fail or see anonymous. Fixes: DelegatingSecurityContextExecutor-style wrappers that capture and propagate the context, or MODE_INHERITABLETHREADLOCAL — with the caveat that inherited contexts go stale/wrong with reused pool threads, so the delegating executor is the safer answer.

Explain the AuthenticationManager / ProviderManager / AuthenticationProvider relationship.

AuthenticationManager is the interface — one method, authenticate(), taking an unverified Authentication and returning a verified one or throwing. ProviderManager is the standard implementation: it holds a list of AuthenticationProviders and asks each whether it supports() the token type, delegating to the first that does. Each provider implements one mechanism — DaoAuthenticationProvider loads a UserDetails via UserDetailsService and checks the password with the PasswordEncoder; others handle JWT, LDAP, etc. The design means adding an auth mechanism is adding a provider, not rewriting the pipeline.

What's the difference between the deprecated WebSecurityConfigurerAdapter and the current approach?

Pre-5.7 you extended WebSecurityConfigurerAdapter and overrode configure(HttpSecurity) — inheritance-based configuration. Since 5.7 (removed in Security 6/Boot 3) you declare a SecurityFilterChain @Bean that takes HttpSecurity, configures it with the lambda DSL, and returns http.build(). Component-based beats inheritance: multiple independent chains are natural (one bean each with its own securityMatcher), config is testable and composable, and there's no fragile super-class contract. You'll still see the adapter in legacy codebases — recognize it, and know the migration is mostly mechanical.

What are the three parts of a JWT and what does the signature guarantee?

Header (algorithm + type), payload (claims: sub, iat, exp, iss, aud, custom), signature — each base64url-encoded, dot-joined. The signature is computed over header+payload with a secret (HMAC) or private key (RSA/EC); it guarantees integrity (any tampering breaks verification) and authenticity (only a key holder could have produced it). It does not provide confidentiality — the payload is readable by anyone, so no secrets in claims. Validation must check signature, exp, and in multi-service setups iss and aud.

HS256 vs RS256 — when do you choose which?

HS256 is symmetric: one shared secret both signs and verifies — fine when a single service issues and consumes its own tokens. RS256 is asymmetric: the auth server signs with a private key, everyone verifies with the public key, typically fetched from a JWKS endpoint. Choose asymmetric as soon as more than one party validates tokens: with HS256 every validator holds the signing secret and could forge tokens, so a compromise of any service forges identity everywhere; with RS256 the minting capability stays only at the auth server, and rotation is just publishing a new public key. External IdPs (Google, Auth0, Keycloak) are asymmetric for exactly this reason.

JWTs vs server-side sessions — give the honest trade-off.

Sessions: state lives server-side, client holds an opaque cookie; revocation is trivial (delete the session) but scaling needs sticky sessions or a shared store like Redis. JWTs: state lives in the signed token; validation is a local crypto check, so any instance serves any request with no shared store — great for horizontal scaling and cross-service auth. The cost is revocation: a signed token is valid until exp regardless of what happens server-side. Standard resolution: short-lived access tokens (minutes) plus server-side, revocable refresh tokens — which is a deliberate hybrid, reintroducing a little state where revocation matters most. A denylist of jtis in Redis is the other mitigation, at the price of the shared lookup you were avoiding. Anyone presenting either option as strictly superior is missing the trade-off.

Sketch the JWT validation filter — what are its steps and where does it sit?

An OncePerRequestFilter registered via addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class). Steps: (1) extract the token from the Authorization: Bearer header — if absent, just continue the chain (enforcement is AuthorizationFilter's job, not the filter's, or you'd break permitAll routes); (2) parse and validate — signature and expiry — via the JWT library; (3) build an authenticated UsernamePasswordAuthenticationToken with principal and authorities from the claims; (4) set it on SecurityContextHolder; then continue. Invalid or expired tokens fall through unauthenticated so the entry point returns a clean JSON 401, which the frontend interceptor uses to trigger a refresh.

Explain the access + refresh token pattern, including rotation.

Split the credential by lifetime and revocability. Access token: 5–15 minutes, stateless, validated by signature on every API call — stolen ones expire fast. Refresh token: days to weeks, stored server-side (hashed) so it's revocable, used only against /auth/refresh to mint new access tokens; logout or compromise = delete the row. Rotation: each refresh invalidates the used refresh token and issues a new one; if an already-rotated token is presented again, that's reuse — evidence of theft — so you revoke the entire token family and force re-login. This gets you stateless performance on the hot path (every API call) while confining stateful lookups to the rare refresh call.

Where should a React SPA store tokens? Defend a recommendation.

Three options: localStorage — persistent and simple, but readable by any injected script, so XSS means token exfiltration; httpOnly cookie — script can't read it, but it becomes an ambient credential so CSRF returns (need SameSite, narrow path, possibly CSRF tokens), and XSS can still use it by making requests from the victim's browser; in-memory — nothing persisted, lost on refresh, so it needs silent refresh. My default: refresh token in an httpOnly, Secure, SameSite=Strict cookie scoped to the refresh path; access token in memory only, re-obtained on load and on 401. The long-lived credential is unreadable and travels to one endpoint; the short-lived one is never at rest. And the honest caveat: no storage choice survives XSS running in your page — CSP, output encoding, and dependency hygiene are the real token protection; storage strategy just limits blast radius.

Why BCrypt over SHA-256 for passwords?

SHA-256 is designed to be fast; password hashing must be deliberately slow, because the threat is offline brute force against a leaked table. BCrypt provides a tunable work factor (cost 12 = 2^12 rounds; each increment doubles the cost, so you can keep pace with hardware) and an automatic per-password random salt embedded in the output, defeating rainbow tables and making identical passwords hash differently. That embedded salt is also why encode() is non-deterministic and verification must use matches(), never string equality. For extra credit: Argon2 is the modern memory-hard choice — GPU farms parallelize CPU-hard BCrypt better than they parallelize memory-hard functions.

What does DelegatingPasswordEncoder do and why does it matter?

It prefixes each stored hash with an algorithm id — {bcrypt}$2a$... — and routes verification to the matching encoder based on that prefix. The payoff is zero-downtime algorithm migration: you can't proactively rehash (you don't have plaintexts), so old hashes must keep verifying under the old algorithm while new passwords use the new one; migration completes lazily as users log in. It's Spring's default via PasswordEncoderFactories.createDelegatingPasswordEncoder(). The related error — "no PasswordEncoder mapped for the id null" — means a stored hash lacks the prefix, usually hand-seeded data.

hasRole("ADMIN") vs hasAuthority("ADMIN") — what's the trap?

Internally everything is a GrantedAuthority string; roles are just authorities with a ROLE_ prefix. hasRole("ADMIN") prepends the prefix and checks for ROLE_ADMIN; hasAuthority("ADMIN") checks the literal string ADMIN. The trap: a JWT filter building SimpleGrantedAuthority("ADMIN") from a claim will never satisfy hasRole("ADMIN") — mysterious 403s for users who "have the role". Same asymmetry in the User builder: .roles() adds the prefix, .authorities() doesn't. Convention: ROLE_-prefixed strings for coarse roles, unprefixed permission strings like invoice:write with hasAuthority for fine-grained checks.

How do you implement "users can only access their own resources"?

URL patterns can't express ownership, so use method security: @EnableMethodSecurity plus @PreAuthorize with SpEL that references parameters and the principal — @PreAuthorize("#userId == authentication.name"), or combined: "hasRole('ADMIN') or #userId == authentication.name". When ownership requires a database lookup (order → owner), delegate to a bean: @PreAuthorize("@orderGuard.isOwner(#orderId, authentication)") — keeps SpEL short and the logic unit-testable. Two caveats worth volunteering: method security uses AOP proxies, so self-invocation bypasses the check; and @PostAuthorize (checking returnObject) runs after execution, so never guard side-effecting methods with it.

What is CSRF, and why do stateless JWT APIs usually disable the protection?

CSRF tricks a logged-in user's browser into sending a state-changing request to your site from an attacker's page — the browser attaches session cookies automatically, so the forged request arrives authenticated. It fundamentally depends on an ambient credential: something sent without the page's JS acting. If your API authenticates only via an Authorization header, there's no ambient credential — a cross-site form can't set headers — so CSRF protection adds friction without protection, and you disable it. The senior nuance: the rule is about the transport, not the token format. Put the JWT or refresh token in a cookie and you've recreated the ambient credential — you need SameSite, a narrow cookie path, and possibly CSRF tokens again for those endpoints.

Your React app gets a CORS error on an authenticated endpoint but Postman works. What's going on?

Classic preflight-vs-security clash. The browser sends an OPTIONS preflight without the Authorization header; if Spring Security evaluates it as an ordinary request it returns 401, the preflight fails, and the browser reports it as a CORS error. Postman skips CORS entirely (it's browser-enforced), so it "works". Fix: .cors(Customizer.withDefaults()) in the chain with a CorsConfigurationSource bean — that places a CorsFilter early so preflights are answered before authentication. If cookies are involved, also allowCredentials(true) plus an explicit origin, since the wildcard is illegal with credentials. And the conceptual point: CORS protects browser users, not the API — it's not access control.

Walk through Authorization Code + PKCE and explain why implicit flow is dead.

The SPA generates a random code_verifier and sends its SHA-256 (code_challenge) with the redirect to the authorization server. The user authenticates there; the server redirects back with a one-time authorization code. The SPA POSTs the code plus the original verifier to the token endpoint; the server checks SHA256(verifier) equals the stored challenge and only then issues tokens. A stolen code is useless without the verifier, which never left the app — that's the proof-of-possession a public client needs since it can't hold a static secret. Implicit flow returned the access token directly in the URL fragment: exposed in history and referrers, no possession proof — it only existed because pre-CORS browsers couldn't call the token endpoint cross-origin. CORS removed that constraint, PKCE solved the public-client problem, and OAuth 2.1 drops implicit entirely.

ID token vs access token — and what does OIDC add to OAuth2?

OAuth2 is delegated authorization — it never standardized "who is the user". OIDC layers identity on top: the openid scope, discovery documents, the userinfo endpoint, and the ID token. The ID token is a JWT whose audience is the client; it asserts who authenticated (sub, email, name) and is consumed by the app for login. The access token's audience is the resource server; it grants API access and may be opaque or a JWT. The classic mistake is sending the ID token to your API as a bearer credential — wrong audience, wrong purpose, and a correctly configured resource server rejects it on the aud claim.

What does oauth2ResourceServer().jwt() with an issuer-uri actually do for you?

It installs BearerTokenAuthenticationFilter and a JwtDecoder built from the issuer. On startup Spring fetches {issuer}/.well-known/openid-configuration, discovers the JWKS URI, and thereafter validates each incoming token's signature against the provider's published public keys (with automatic key rotation via the kid header), plus exp and iss. You add audience validation and, typically, a JwtAuthenticationConverter to map scopes or custom role claims to GrantedAuthoritys. Net effect: the entire hand-rolled JwtService-plus-filter apparatus becomes a YAML property when an external IdP issues the tokens — Spring is acting as a resource server, versus oauth2Login() where it acts as the OAuth client doing the redirect dance and holding tokens in a session.

How would you rate-limit a login endpoint, and where does rate limiting belong architecturally?

Defense in depth across layers: coarse volumetric limits at the edge/CDN or WAF (cheapest place to shed abuse), per-client quotas at the API gateway (Spring Cloud Gateway's Redis-backed RequestRateLimiter — it must be shared-state because in-memory counters are per-instance behind a load balancer), and precise, business-aware limits in-app for endpoints like /auth/login — e.g. a Bucket4j-backed OncePerRequestFilter keyed by IP plus attempted username, returning 429 with Retry-After, to blunt credential stuffing. Complements: account lockout with care (it's a DoS vector against known usernames), constant-ish response behavior to avoid user-enumeration timing, and alerting on spike patterns. Spring Security itself ships no limiter — the placement discussion is the actual answer.

Hands-on exercises

Exercise 1 — Wire up a minimal stateless JWT API

Create a Spring Boot 3 app with two endpoints: POST /auth/login (accepts username/password JSON, returns an access token) and GET /api/hello (returns a greeting including the caller's name; requires a valid token). Use an in-memory user (user / password, role USER). Requirements: stateless session policy, CSRF disabled, a JSON-returning 401 entry point, and a working OncePerRequestFilter. Verify with curl: login, capture the token, call /api/hello with and without it.

Solution

Reuse SecurityConfig, JwtService, JwtAuthFilter, and AuthController from the JWT section (drop the refresh-token pieces). For the in-memory user:

@Bean
public UserDetailsService userDetailsService(PasswordEncoder encoder) {
    return new InMemoryUserDetailsManager(
        User.builder()
            .username("user")
            .password(encoder.encode("password"))
            .roles("USER")
            .build());
}

@RestController
class HelloController {
    @GetMapping("/api/hello")
    String hello(Authentication auth) { return "Hello, " + auth.getName(); }
}
# 1. No token → expect 401 JSON body
curl -i http://localhost:8080/api/hello

# 2. Login
TOKEN=$(curl -s -X POST http://localhost:8080/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"user","password":"password"}' | jq -r .accessToken)

# 3. With token → 200 "Hello, user"
curl -i http://localhost:8080/api/hello -H "Authorization: Bearer $TOKEN"

Checks worth doing: tamper with one character of the token (expect 401 — signature), set the TTL to 5 seconds and retry after expiry (expect 401 — exp), and confirm /auth/login works with no token (proves the filter doesn't over-enforce).

Exercise 2 — Ownership checks with method security

Add a Note entity (id, ownerUsername, text) with JPA and a NoteService exposing getNote(Long id) and deleteNote(Long id). Enforce: any authenticated user can create notes; only the owner can read a note; only the owner or an ADMIN can delete. Prove that a second user gets 403 on someone else's note. Bonus: explain why getNote is awkward with @PreAuthorize alone.

Solution 
@Component("noteGuard")
public class NoteGuard {
    private final NoteRepository repo;
    public NoteGuard(NoteRepository repo) { this.repo = repo; }
    public boolean isOwner(Long noteId, Authentication auth) {
        return repo.findById(noteId)
            .map(n -> n.getOwnerUsername().equals(auth.getName()))
            .orElse(false);
    }
}

@Service
public class NoteService {

    @PreAuthorize("@noteGuard.isOwner(#id, authentication)")
    public Note getNote(Long id) { ... }

    @PreAuthorize("hasRole('ADMIN') or @noteGuard.isOwner(#id, authentication)")
    public void deleteNote(Long id) { ... }
}

Don't forget @EnableMethodSecurity on a configuration class. The bonus point: with @PreAuthorize, the guard loads the note once for the check and the service loads it again — two queries. Alternatives: @PostAuthorize("returnObject.ownerUsername == authentication.name") (one load, fine because reading is side-effect-free — but never for the delete), or fold ownership into the query itself (findByIdAndOwnerUsername) and treat "absent" as 404, which also avoids leaking that the note exists. Discussing the 403-vs-404 information-leak trade-off is exactly the level interviewers want.

Exercise 3 — Refresh tokens with rotation and reuse detection

Extend Exercise 1: shorten the access token TTL to 1 minute; on login also issue an opaque refresh token (random 256-bit value), store its SHA-256 hash in a table with username, expiry, revoked flag, and a familyId; deliver it as an httpOnly cookie scoped to /auth/refresh. Implement POST /auth/refresh with rotation, and revoke the entire family when a rotated token is replayed. Implement POST /auth/logout as family revocation.

Solution

Core rotation logic — the interesting part is the reuse branch:

@Transactional
public RotationResult rotate(String rawToken) {
    String hash = sha256(rawToken);
    RefreshToken current = repo.findByTokenHash(hash)
        .orElseThrow(() -> new BadCredentialsException("unknown refresh token"));

    if (current.isRevoked()) {
        // Reuse of a rotated token = likely theft. Kill the whole family.
        repo.revokeFamily(current.getFamilyId());
        throw new BadCredentialsException("refresh token reuse detected");
    }
    if (current.getExpiresAt().isBefore(Instant.now())) {
        throw new CredentialsExpiredException("refresh token expired");
    }

    current.setRevoked(true);                       // rotate out the old one
    RefreshToken next = newToken(current.getUsername(), current.getFamilyId());
    repo.save(next);

    String newAccess = jwtService.generateAccessToken(
        userDetailsService.loadUserByUsername(current.getUsername()));
    return new RotationResult(newAccess, cookieFor(next.rawValue()));
}

Test script: login; call refresh once (works, new cookie); call refresh again with the old cookie value (expect 401 and — the crucial assertion — the token issued by the first refresh is now dead too, because the family was revoked). Notes: store only hashes (a DB leak must not yield usable tokens); the whole rotate must be transactional; scoping the cookie path means the refresh token never rides along on ordinary API calls, shrinking its CSRF surface, and SameSite=Strict shrinks it further.

Exercise 4 — Switch to an external IdP as a resource server

Stand up Keycloak locally (Docker), create a realm, a client, and a user with a realm_access.roles role of admin. Replace your hand-rolled JWT filter with oauth2ResourceServer().jwt() + issuer-uri. Map Keycloak's role claim so that hasRole('ADMIN') works on /api/admin/**. Reflect: what code did you delete?

Solution 
docker run -p 8081:8080 -e KEYCLOAK_ADMIN=admin -e KEYCLOAK_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:24.0 start-dev
# application.yml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:8081/realms/myrealm

Keycloak puts realm roles in realm_access.roles without a ROLE_ prefix, so map them:

@Bean
public JwtAuthenticationConverter jwtAuthenticationConverter() {
    JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
    converter.setJwtGrantedAuthoritiesConverter(jwt -> {
        Map<String, Object> realmAccess = jwt.getClaimAsMap("realm_access");
        if (realmAccess == null) return List.of();
        @SuppressWarnings("unchecked")
        List<String> roles = (List<String>) realmAccess.get("roles");
        return roles.stream()
            .map(r -> new SimpleGrantedAuthority("ROLE_" + r.toUpperCase()))
            .map(GrantedAuthority.class::cast)
            .toList();
    });
    return converter;
}

// in the chain:
.oauth2ResourceServer(oauth2 -> oauth2
    .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtAuthenticationConverter())))

Get a token via the password grant (dev only) or the Keycloak account console, then hit /api/admin/ping. What you deleted: JwtService, JwtAuthFilter, the login endpoint, the secret in config — signing, key rotation (JWKS + kid), and validation are now the IdP's job and Spring's auto-configuration. What stayed yours: authorization rules, the authorities mapping, and the JSON error handling. That division of labor — IdP owns authentication, your service owns authorization — is the takeaway sentence for interviews.
