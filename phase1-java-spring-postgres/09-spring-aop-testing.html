09 — AOP, Proxies & Testing

How Spring's proxy magic actually works, why @Transactional sometimes silently does nothing, and how seniors test Spring apps without lying to themselves.

Why interviewers ask this. AOP and proxies are the machinery behind almost every Spring annotation you have used so far — @Transactional, @Cacheable, @Async, @PreAuthorize. Interviewers probe it because it separates people who use Spring from people who understand it: the self-invocation bug is the single most common "why is my annotation not working?" production incident. Testing questions, meanwhile, are where seniority shows most clearly. Anyone can write a test; a senior knows which layer to test with which tool, why @SpringBootTest everywhere is a smell, and why H2 gives you green builds and red production. Coming from 7 years of Jest and React Testing Library, you already have the instincts — this page maps them to the JVM equivalents (Mockito ≈ jest.mock, MockMvc ≈ supertest, Testcontainers ≈ running real Postgres in Docker for integration tests) and adds the Spring-specific judgment calls.

Why AOP: cross-cutting concerns T2

Some concerns cut across your whole codebase rather than living in one module: logging, timing/metrics, transactions, security checks, caching, retries. Without AOP you would sprinkle the same boilerplate into every service method:

// Without AOP: every method repeats the same ceremony
public Order placeOrder(OrderRequest req) {
    long start = System.nanoTime();                     // timing
    log.info("placeOrder called");                       // logging
    TransactionStatus tx = txManager.getTransaction(...); // transaction
    try {
        // ...the 5 lines of ACTUAL business logic...
        txManager.commit(tx);
    } catch (RuntimeException e) {
        txManager.rollback(tx);
        throw e;
    } finally {
        metrics.record(System.nanoTime() - start);
    }
}

Aspect-Oriented Programming extracts that ceremony into reusable modules ("aspects") that are woven around your business code at runtime. Your method becomes just the business logic plus declarative annotations:

@Transactional
@LogExecutionTime   // our own custom annotation — we'll build it below
public Order placeOrder(OrderRequest req) {
    // only business logic remains
}

If you've used higher-order components or decorators in React/TS (withAuth(Component), @Injectable() in Angular/NestJS), AOP is the same idea: wrap behavior around code without editing the code.

The terminology (interviewers expect exact words)

TermMeaningMental model
AspectA module encapsulating one cross-cutting concern (e.g. "timing").The decorator class itself.
Join pointA point in program execution where an aspect could apply. In Spring AOP this is always a method execution.Every method call is a potential hook.
PointcutA predicate that selects which join points the advice applies to.A CSS selector for methods.
AdviceThe code that runs at a selected join point (before / after / around it).The wrapper function body.
WeavingLinking aspects to target code. Spring weaves at runtime via proxies; AspectJ can weave at compile time.How the wrapper gets applied.
Target / Advised objectThe real bean being wrapped by the proxy.Your original component.

Spring AOP is proxy-based and only intercepts public method executions on Spring beans. Full AspectJ can intercept field access, constructors, private methods, and objects Spring doesn't manage — but almost nobody needs that, and "Spring AOP is a pragmatic subset of AspectJ" is a great interview line.

How Spring proxies work — the mechanics interviewers probe T2

This is the section that explains rules you were just told to memorize in earlier tutorials ("@Transactional only works on public methods", "don't call transactional methods from the same class"). When Spring sees a bean that needs AOP behavior (it has @Transactional, matches a pointcut, etc.), it does not hand you your object. It hands you a proxy — a stand-in that wraps your object:

// What you think you injected:        // What Spring actually injected:
//                                     
//  ┌──────────────┐                   ┌──────────────────────────┐
//  │ OrderService │                   │ OrderService$$SpringCGLIB │  ← the proxy
//  └──────────────┘                   │  before-advice (open tx)  │
//                                     │   ┌──────────────┐        │
//                                     │   │ OrderService │ target │
//                                     │   └──────────────┘        │
//                                     │  after-advice (commit)    │
//                                     └──────────────────────────┘

Two proxy strategies

Mechanismjava.lang.reflect.Proxy generates a class implementing your bean's interfaces at runtime.Generates a subclass of your concrete class at runtime, overriding its methods.

Spring Boot defaultBoot defaults to CGLIB for everything (proxyTargetClass=true), even for beans with interfaces — so injecting by concrete class always works.

The self-invocation problem — the #1 proxy gotcha

The proxy only intercepts calls that arrive from outside, through the proxy reference. When a method calls another method on the same object (this.otherMethod()), the call goes straight to the target — the proxy never sees it, so its advice never runs. Result: @Transactional, @Cacheable, @Async, @PreAuthorize on the inner method silently no-op.

@Service
public class ReportService {

    public void generateAll() {
        for (Long id : reportIds) {
            // BROKEN: this is this.generateOne(id) — a plain Java call
            // on the target object. It bypasses the proxy, so NO new
            // transaction is opened per report. @Transactional no-ops.
            generateOne(id);
        }
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void generateOne(Long id) {
        // you THINK each report gets its own transaction... it doesn't
    }
}

No error, no warning, no log line. The code runs, tests may even pass (if they don't assert transactional behavior), and you discover it in production when one failing report rolls back all the others — or when @Cacheable never caches and your DB melts. Interviewers love asking "my @Transactional isn't working, what do you check first?" — self-invocation is answer #1.

Fix 1 (preferred): extract to another bean

@Service
@RequiredArgsConstructor
public class ReportOrchestrator {
    private final ReportWorker worker;   // injected = proxied reference

    public void generateAll() {
        for (Long id : reportIds) {
            worker.generateOne(id);       // crosses the proxy ✔ — advice runs
        }
    }
}

@Service
public class ReportWorker {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void generateOne(Long id) { /* now truly transactional */ }
}

This is preferred because it usually reveals a hidden design seam anyway — the orchestration and the unit of work are different responsibilities.

Fix 2: self-injection via ObjectProvider

@Service
public class ReportService {
    private final ObjectProvider<ReportService> self;

    public ReportService(ObjectProvider<ReportService> self) {
        this.self = self;   // lazy lookup avoids the circular-dependency error
    }

    public void generateAll() {
        for (Long id : reportIds) {
            self.getObject().generateOne(id);  // goes through the proxy ✔
        }
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void generateOne(Long id) { ... }
}

Mention both fixes but state a preference for extraction: self-injection works but smells — a class calling its own proxied version is a sign two responsibilities are trapped in one class. Also mention the anti-fix: AopContext.currentProxy() exists but requires exposeProxy=true and couples business code to Spring AOP internals — flag it as something you'd reject in code review.

Other proxy consequences (this explains earlier tutorials' rules)

  • final methods/classes can't be advised. CGLIB subclasses your class and overrides methods; Java forbids overriding final. A final class fails at startup; a final method is silently skipped (newer Spring versions log a warning). This is also why Kotlin Spring projects need the kotlin-spring compiler plugin to open classes.
  • private methods can't be advised — can't be overridden, and JDK proxies only see interface methods. @Transactional on a private method does nothing.
  • Field access isn't intercepted. The proxy delegates to the target, whose fields are its own; only method entry is wrapped.
  • Constructors run twice-ish surprises: CGLIB instantiates the proxy subclass; don't put logic in constructors that assumes it runs once per logical bean.

Since Spring Framework 6 / Boot 3, CGLIB is repackaged inside spring-core (org.springframework.cglib), so you never add it as a dependency — it's just there.

Writing your own aspects T2

You rarely write aspects day-to-day (mostly you consume Spring's), but interviewers use "write a timing aspect" as a litmus test, and it's the cleanest way to add cross-cutting logging/metrics. Setup: spring-boot-starter-aop on the classpath (auto-enables @EnableAspectJAutoProxy in Boot).

Advice types

AnnotationRunsCan it...
@BeforeBefore the methodRead args; veto only by throwing.
@AfterReturningAfter normal returnRead (not replace) the return value.
@AfterThrowingAfter an exception escapesInspect the exception; can't swallow it.
@AfterAfter return or throw (finally)Cleanup only.
@AroundWraps the whole callEverything: skip the call, change args, replace return value, catch/translate exceptions. Uses ProceedingJoinPoint.

Pointcut expressions — the two you must know

// 1. execution() — match by method signature pattern
//    execution(modifiers? return-type declaring-type?.name(params))
execution(public * com.demo.service.*.*(..))
//        │      │  └ any class in service pkg . any method (any args)
//        │      └ any return type
//        └ public only (Spring AOP can only see public anyway)

execution(* com.demo..*Service.find*(..))   // ".." = this pkg + subpackages

// 2. @annotation() — match methods carrying a specific annotation
@annotation(com.demo.aop.LogExecutionTime)

// Combinators: &&  ||  !
execution(* com.demo..*(..)) && @annotation(org.springframework.transaction.annotation.Transactional)

Complete example: a custom @LogExecutionTime annotation + timing aspect

// 1. The marker annotation
package com.demo.aop;

import java.lang.annotation.*;

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)   // must survive to runtime for the proxy to see it
public @interface LogExecutionTime {
    long warnAboveMillis() default 500;
}
// 2. The aspect — @Aspect makes it an aspect, @Component makes it a bean
package com.demo.aop;

import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.reflect.MethodSignature;

@Aspect
@Component
public class ExecutionTimeAspect {

    private static final Logger log = LoggerFactory.getLogger(ExecutionTimeAspect.class);

    @Around("@annotation(logExecutionTime)")  // binds the annotation instance as a param
    public Object time(ProceedingJoinPoint pjp, LogExecutionTime logExecutionTime)
            throws Throwable {

        MethodSignature sig = (MethodSignature) pjp.getSignature();
        String method = sig.getDeclaringType().getSimpleName() + "." + sig.getName();

        long start = System.nanoTime();
        try {
            return pjp.proceed();   // ← calls the real method. Forget this = method never runs!
        } finally {
            long ms = (System.nanoTime() - start) / 1_000_000;
            if (ms > logExecutionTime.warnAboveMillis()) {
                log.warn("{} took {} ms (threshold {})", method, ms,
                         logExecutionTime.warnAboveMillis());
            } else {
                log.debug("{} took {} ms", method, ms);
            }
        }
    }
}
// 3. Usage — any Spring bean's public method
@Service
public class SearchService {

    @LogExecutionTime(warnAboveMillis = 200)
    public List<Product> search(String query) { ... }
}

Two classic @Around bugs: (1) forgetting to return pjp.proceed() — for non-void methods every caller suddenly receives null; (2) swallowing the exception from proceed() — the caller sees success while the operation failed. @Around is powerful precisely because it can do these things, so use the least powerful advice type that works (@Before/@AfterReturning can't break the call).

Ordering: when multiple aspects apply to one method (yours + @Transactional + security), order matters. Implement Ordered or add @Order(n) on the aspect — lower value = outermost. E.g. a timing aspect should usually be outside the transaction advice if you want to measure commit time too.

Spring's own annotations that are AOP under the hood T3

Everything below is "just" an aspect + proxy — which is why all of them share the same failure modes: self-invocation, non-public methods, final, and non-Spring-managed objects (new MyService() gets no proxy, no magic).

@Transactional

Covered in depth in the JPA tutorial — the advice opens a transaction before the method, commits on return, rolls back on unchecked exceptions. Now you know why the rules exist: they're proxy rules, not transaction rules.

@Cacheable / @CacheEvict / @CachePut — 60-second caching intro

@Configuration
@EnableCaching          // switches on the caching aspect infrastructure
public class CacheConfig { }

@Service
public class ProductService {

    @Cacheable(cacheNames = "products", key = "#id")
    public Product findById(Long id) {
        // Around-advice: check cache first; only runs this body on a miss,
        // then stores the return value under key `id`.
        return repository.findById(id).orElseThrow();
    }

    @CacheEvict(cacheNames = "products", key = "#product.id")
    public Product update(Product product) {   // evict stale entry on write
        return repository.save(product);
    }
}

Default store is a simple in-memory ConcurrentHashMap; add Caffeine or Redis via starters for real deployments. The key attribute is a SpEL expression over method parameters.

@Cacheable on a method called via this. never caches — same self-invocation trap. Also: cached objects are shared references with the default in-memory store; mutating a returned entity mutates the cache.

@Async

@Configuration
@EnableAsync            // REQUIRED — without it, @Async is silently ignored
public class AsyncConfig { }

@Service
public class NotificationService {

    @Async
    public CompletableFuture<Void> sendWelcomeEmail(String to) {
        // runs on a task-executor thread; the caller returns immediately
        mailClient.send(to, "Welcome!");
        return CompletableFuture.completedFuture(null);
    }
}
  • Return CompletableFuture<T> (or void for fire-and-forget). Think async functions returning a Promise.
  • Exception gotcha: an exception in a void @Async method never reaches the caller — it vanishes into an AsyncUncaughtExceptionHandler (default: just logs). If you return a CompletableFuture, the exception lands in the future and surfaces only when someone calls .join()/.get(). Unhandled promise rejection, JVM edition.
  • Configure a proper ThreadPoolTaskExecutor bean; the default executor settings are rarely what you want in production.

@Retryable (awareness level)

// spring-retry + @EnableRetry
@Retryable(retryFor = TransientDataAccessException.class,
           maxAttempts = 3,
           backoff = @Backoff(delay = 200, multiplier = 2))
public Payment charge(Card card) { ... }

@Recover   // fallback after retries exhausted
public Payment recover(TransientDataAccessException e, Card card) { ... }

Same proxy mechanics; know it exists and that Resilience4j is the common alternative for retry/circuit-breaking in microservices.

JUnit 5 & AssertJ T1

JUnit 5 (a.k.a. Jupiter) is the test runner — your Jest. It's in spring-boot-starter-test along with AssertJ, Mockito, MockMvc and more, so a Boot project needs zero extra test dependencies.

Lifecycle & structure

class PriceCalculatorTest {            // no annotation needed on the class itself

    private PriceCalculator calculator;

    @BeforeAll                            // ≈ beforeAll — static by default
    static void initExpensiveSharedResource() { }

    @BeforeEach                           // ≈ beforeEach — fresh state per test
    void setUp() {
        calculator = new PriceCalculator();
    }

    @AfterEach  void tearDown() { }       // ≈ afterEach
    @AfterAll   static void cleanup() { }  // ≈ afterAll

    @Test
    @DisplayName("applies 10% discount for gold members")  // ≈ it('applies...')
    void appliesGoldDiscount() {
        BigDecimal price = calculator.price(new BigDecimal("100"), Tier.GOLD);
        assertEquals(new BigDecimal("90.00"), price);
    }

    @Test
    @Disabled("flaky until PRICE-123 is fixed")   // ≈ it.skip — always give a reason
    void skipped() { }
}

Key difference from Jest: JUnit creates a new instance of the test class for every test method by default, so instance fields are naturally isolated — you often don't need @BeforeEach at all, just field initializers.

Core assertions

import static org.junit.jupiter.api.Assertions.*;

assertEquals(expected, actual);                   // note the order: expected FIRST
assertTrue(order.isPaid());
assertNotNull(result);

// assertThrows ≈ expect(() => fn()).toThrow() — and returns the exception
IllegalArgumentException ex = assertThrows(
        IllegalArgumentException.class,
        () -> calculator.price(null, Tier.GOLD));
assertEquals("price must not be null", ex.getMessage());

// assertAll: run every assertion, report ALL failures (soft assertions)
assertAll(
    () -> assertEquals("Ada", user.firstName()),
    () -> assertEquals("Lovelace", user.lastName()),
    () -> assertTrue(user.active())
);

AssertJ — the fluent style seniors actually use

import static org.assertj.core.api.Assertions.*;

assertThat(price).isEqualByComparingTo("90.00");      // BigDecimal-aware compare
assertThat(user.name()).isEqualTo("Ada").startsWith("A");

assertThat(orders)                                     // collection assertions shine
        .hasSize(3)
        .extracting(Order::status)
        .containsExactly(Status.NEW, Status.PAID, Status.SHIPPED);

assertThatThrownBy(() -> service.cancel(999L))
        .isInstanceOf(OrderNotFoundException.class)
        .hasMessageContaining("999");

Prefer AssertJ over raw JUnit assertions: it reads left-to-right (actual first, like expect(actual).toBe(expected) — familiar!), gives dramatically better failure messages, and autocompletes. Most Spring codebases standardize on it.

Mockito: mocks, stubs, verification & judgment T1

Mockito ≈ jest.mock + jest.fn() + expect(mock).toHaveBeenCalledWith(...) rolled into one library. The big mindset shift: in Jest you mock modules; in Java you mock the collaborator objects injected via constructor — which is exactly why constructor injection (tutorial 05) matters: it makes classes testable without Spring.

Setup: @Mock / @InjectMocks + MockitoExtension

@ExtendWith(MockitoExtension.class)     // activates the annotations — NO Spring here
class OrderServiceTest {

    @Mock    private OrderRepository orderRepository;   // ≈ jest.mocked dependency
    @Mock    private PaymentClient paymentClient;
    @InjectMocks private OrderService service;         // real class, mocks injected

    @Test
    void placesOrderAndCharges() {
        // stub: when X is called, return Y  ≈ mockFn.mockReturnValue(y)
        when(orderRepository.save(any(Order.class)))
                .thenAnswer(inv -> inv.getArgument(0));      // echo back the arg
        when(paymentClient.charge(any())).thenReturn(PaymentResult.ok("tx-1"));

        Order order = service.placeOrder(new OrderRequest("sku-1", 2));

        assertThat(order.status()).isEqualTo(Status.PAID);
        verify(paymentClient).charge(any());                // ≈ toHaveBeenCalled
    }

    @Test
    void failsWhenPaymentDeclined() {
        when(paymentClient.charge(any()))
                .thenThrow(new PaymentDeclinedException("insufficient funds"));

        assertThatThrownBy(() -> service.placeOrder(new OrderRequest("sku-1", 2)))
                .isInstanceOf(OrderFailedException.class);

        verify(orderRepository, never()).save(any());       // nothing persisted ✔
    }
}

BDDMockito — same engine, given/when/then vocabulary

import static org.mockito.BDDMockito.*;

given(paymentClient.charge(any())).willReturn(PaymentResult.ok("tx-1"));
given(paymentClient.charge(any())).willThrow(new PaymentDeclinedException("no"));

// verification counterpart:
then(paymentClient).should(times(1)).charge(any());
then(orderRepository).should(never()).save(any());

Purely stylistic — some teams mandate it because when(...) in the given section of a given/when/then test reads oddly. Know both spellings.

Verification: times, never, order

verify(mailer).send(any());                 // exactly once (default)
verify(mailer, times(2)).send(any());      // ≈ toHaveBeenCalledTimes(2)
verify(mailer, never()).send(any());        // ≈ not.toHaveBeenCalled()
verify(mailer, atLeastOnce()).send(any());
verifyNoInteractions(auditLog);             // nothing at all touched this mock

InOrder inOrder = inOrder(repo, mailer);    // assert call ordering when it matters
inOrder.verify(repo).save(any());
inOrder.verify(mailer).send(any());

ArgumentCaptor — inspect what was actually passed

Equivalent of mockFn.mock.calls[0][0], but typed. Use it when the interesting behavior is what your service constructed and handed to a collaborator:

@ExtendWith(MockitoExtension.class)
class RegistrationServiceTest {

    @Mock        private UserRepository userRepository;
    @InjectMocks private RegistrationService service;

    @Captor      private ArgumentCaptor<User> userCaptor;   // or ArgumentCaptor.forClass(User.class)

    @Test
    void persistsUserWithNormalizedEmailAndHashedPassword() {
        service.register("  Ada@Example.COM ", "s3cret!");

        verify(userRepository).save(userCaptor.capture());   // grab the actual arg
        User saved = userCaptor.getValue();

        assertThat(saved.email()).isEqualTo("ada@example.com");  // trimmed + lowered
        assertThat(saved.passwordHash()).isNotEqualTo("s3cret!") // never plaintext
                                        .startsWith("$2a$");      // bcrypt prefix
    }
}

Argument matchers — and THE mixing rule

when(repo.findByTierAndActive(any(Tier.class), anyBoolean())).thenReturn(users);

// ❌ COMPILE-TIME OK, RUNTIME ERROR: "InvalidUseOfMatchersException"
when(repo.findByTierAndActive(Tier.GOLD, anyBoolean())).thenReturn(users);

// ✔ RULE: if ANY argument uses a matcher, ALL must — wrap literals in eq()
when(repo.findByTierAndActive(eq(Tier.GOLD), anyBoolean())).thenReturn(users);

The matcher-mixing rule trips up everyone once. Mockito matchers work via a thread-local side-effect stack, so raw values and matchers can't be interleaved. Jest has no equivalent because expect.any() objects are just values.

@Spy vs @Mock

@Mock@Spy
Default behaviorAll methods stubbed: return null/0/empty.Real object; real methods run unless stubbed.
Jest analogyjest.fn() for everythingjest.spyOn(obj, 'method') — partial mock
Stubbing syntaxwhen(mock.m()).thenReturn(x)doReturn(x).when(spy).m() — the when() form would execute the real method while stubbing!
Senior takeDefault choice for collaborators.Usually a design smell: needing a partial mock often means the class does two things. Legit for legacy code you can't split yet.

When NOT to mock — the senior judgment part

  • Value objects & entities. Never mock(Money.class) — just new Money("9.99", EUR). Mocking data is pure noise. Same instinct as never mocking a plain JS object.
  • The class under test. If you're stubbing methods of the thing you're testing (spy on itself), you're testing your stubs.
  • Types you don't own (e.g. RestClient, JPA EntityManager) — mocking them couples tests to a vendor API's call sequence. Wrap them in your own thin adapter interface and mock that, or use WireMock/Testcontainers to test against the real protocol.
  • The over-mocked test smell: if a test sets up 8 stubs and then verifies those same 8 interactions, it asserts "the code calls what the code calls" — a mirror of the implementation that breaks on every refactor and catches no bugs. Prefer state-based assertions (what came back / what was persisted) over interaction verification, and reserve verify() for genuinely important side effects (email sent, event published, nothing saved on failure).

Default choice: services with plain Mockito, no Spring at all

Because you used constructor injection, your service is a plain Java class. Test it exactly as in the Mockito section above — no @SpringBootTest, no context, sub-millisecond tests. This should be the bulk of your suite. Reaching for Spring to test business logic is the #1 junior tell.

@WebMvcTest + MockMvc — the supertest of Spring

@WebMvcTest boots only the web layer: your controller, Jackson, validation, exception handlers, filters. Services are replaced with mocks via @MockitoBean.

Boot 3.4 naming change: @MockBean/@SpyBean are deprecated (removed in later versions) in favor of @MockitoBean/@MockitoSpyBean from org.springframework.test.context.bean.override.mockito. Say @MockitoBean in interviews and mention the rename — it signals you're current.

@WebMvcTest(OrderController.class)        // only this controller's slice
class OrderControllerTest {

    @Autowired   private MockMvc mockMvc;          // ≈ supertest(app)
    @Autowired   private ObjectMapper objectMapper;
    @MockitoBean private OrderService orderService; // mock replaces real bean in context

    @Test
    void createsOrderAndReturns201() throws Exception {
        given(orderService.placeOrder(any()))
                .willReturn(new OrderResponse(42L, "sku-1", 2, Status.PAID));

        mockMvc.perform(post("/api/orders")
                    .contentType(MediaType.APPLICATION_JSON)
                    .content(objectMapper.writeValueAsString(
                            new OrderRequest("sku-1", 2))))
               .andExpect(status().isCreated())
               .andExpect(header().string("Location", "/api/orders/42"))
               .andExpect(jsonPath("$.id").value(42))
               .andExpect(jsonPath("$.status").value("PAID"));
    }

    @Test
    void returns400WithFieldErrorsWhenValidationFails() throws Exception {
        // quantity = 0 violates @Positive on the request DTO —
        // note: the service mock is never touched; validation rejects first
        mockMvc.perform(post("/api/orders")
                    .contentType(MediaType.APPLICATION_JSON)
                    .content("""
                        {"sku": "", "quantity": 0}
                        """))
               .andExpect(status().isBadRequest())
               .andExpect(jsonPath("$.title").value("Validation failed"))
               .andExpect(jsonPath("$.errors[?(@.field=='sku')].message")
                       .value("must not be blank"))
               .andExpect(jsonPath("$.errors[?(@.field=='quantity')].message")
                       .value("must be greater than 0"));

        then(orderService).shouldHaveNoInteractions();   // prove it never got through
    }

    @Test
    void returns404WhenOrderMissing() throws Exception {
        given(orderService.findById(99L))
                .willThrow(new OrderNotFoundException(99L));

        mockMvc.perform(get("/api/orders/99"))
               .andExpect(status().isNotFound())          // exercises @RestControllerAdvice
               .andExpect(jsonPath("$.detail").value("Order 99 not found"));
    }
}

What @WebMvcTest earns you that a unit test can't: HTTP status codes, request/response JSON shape, Bean Validation wiring, your @RestControllerAdvice error mapping, content negotiation, and (with Spring Security on the classpath) security rules via @WithMockUser.

@DataJpaTest — persistence slice

@DataJpaTest       // boots JPA + repositories only; each test runs in a rolled-back tx
class OrderRepositoryTest {

    @Autowired private OrderRepository repository;
    @Autowired private TestEntityManager em;   // arrange/flush helper, bypasses the repo

    @Test
    void findsPaidOrdersNewerThan() {
        em.persist(order("sku-1", Status.PAID,    daysAgo(1)));
        em.persist(order("sku-2", Status.PAID,    daysAgo(10)));
        em.persist(order("sku-3", Status.PENDING, daysAgo(1)));
        em.flush();
        em.clear();   // force the query to hit the DB, not the persistence context

        List<Order> result = repository.findByStatusAndCreatedAtAfter(
                Status.PAID, daysAgo(3));

        assertThat(result).extracting(Order::getSku).containsExactly("sku-1");
    }
}

By default @DataJpaTest swaps your datasource for in-memory H2. That means you are not testing against Postgres — see the Testcontainers section for why that's a lie and how to fix it with two annotations. Also note em.flush(); em.clear();: without them, JPA's first-level cache can return the objects you just persisted without ever executing your query's SQL, making broken queries pass.

@SpringBootTest — when it's justified

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderE2ETest {

    @Autowired private TestRestTemplate rest;   // real HTTP over a real port

    @Test
    void fullOrderLifecycle() {
        ResponseEntity<OrderResponse> created = rest.postForEntity(
                "/api/orders", new OrderRequest("sku-1", 2), OrderResponse.class);

        assertThat(created.getStatusCode()).isEqualTo(HttpStatus.CREATED);

        OrderResponse fetched = rest.getForObject(
                "/api/orders/" + created.getBody().id(), OrderResponse.class);
        assertThat(fetched.status()).isEqualTo(Status.PAID);
    }
}

Justified for: verifying the whole app wires up (one smoke test that the context loads is genuinely valuable), a handful of critical end-to-end flows, and configuration-heavy behavior (security filter chains + real serialization + real DB together). Not justified for: business-logic permutations — that's what the fast layers are for.

Why it's slow and how Spring mitigates it: booting the full context takes seconds. Spring caches contexts across test classes keyed by configuration — so 50 @SpringBootTest classes with identical config boot one context total. Every @MockitoBean, @ActiveProfiles, or @TestPropertySource variation forks a new cache entry, and @DirtiesContext throws the context away. Senior move: standardize test configuration so the cache actually hits.

The test pyramid, senior edition

        /  E2E (@SpringBootTest + Testcontainers)  \      few — critical flows only
       /   slice tests (@WebMvcTest, @DataJpaTest)  \     some — per controller/repo
      /    unit tests (JUnit + Mockito, no Spring)   \    many — all business logic

The senior answer isn't "pyramid good" — it's cost/confidence trade-off per layer: push each behavior to the cheapest layer that can falsify it. Validation rules → @WebMvcTest. Discount math → pure unit. Custom JPQL → @DataJpaTest on real Postgres. "Does the app even start with prod-ish config" → one @SpringBootTest. If a bug class can't be caught below E2E (e.g. security filter ordering), that's what E2E budget is for. Also say the quiet part: a slow suite is a broken suite, because people stop running it.

Testcontainers: real Postgres in your tests T2

This is the modern differentiator topic — teams that adopted it can't imagine going back, and mentioning it fluently marks you as current.

Why H2 lies to you

H2 is a Java in-memory database with a "Postgres compatibility mode" that is compatible right up until it isn't:

  • Postgres-specific SQL & types: JSONB columns and operators (->>, @>), arrays, ON CONFLICT ... DO UPDATE upserts, RETURNING, full-text search — H2 either rejects them or fakes them differently.
  • Window functions & CTE edge cases behave subtly differently or are missing.
  • Different transaction/locking semantics, sequence behavior, case-sensitivity of identifiers, and error codes.
  • Your Flyway migrations are written for Postgres; on H2 you either can't run them or maintain a parallel dialect — so the schema you test isn't the schema you ship.

Net effect: green H2 tests, production incident on real Postgres. The frontend analogy: testing in jsdom and shipping to Safari — fine until you hit anything real.

Testcontainers: throwaway Docker containers per test run

Testcontainers starts a real Postgres in Docker for your tests and tears it down after. Boot 3.1+ made wiring it one annotation: @ServiceConnection reads the container's host/port/credentials and configures the datasource automatically — no URL property gymnastics.

// build.gradle: testImplementation "org.springframework.boot:spring-boot-testcontainers"
//               testImplementation "org.testcontainers:junit-jupiter"
//               testImplementation "org.testcontainers:postgresql"

@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE) // keep OUR datasource, don't swap in H2
@Testcontainers                       // JUnit extension managing container lifecycle
class OrderRepositoryPostgresTest {

    @Container
    @ServiceConnection                // Boot 3.1+: auto-wires spring.datasource.* from the container
    static PostgreSQLContainer<?> postgres =
            new PostgreSQLContainer<>("postgres:16-alpine");

    @Autowired private OrderRepository repository;

    @Test
    void upsertUsesRealOnConflictClause() {
        // This @Query uses ON CONFLICT (sku) DO UPDATE — impossible to
        // verify on H2, trivially verified here on genuine Postgres 16.
        repository.upsert("sku-1", 5);
        repository.upsert("sku-1", 9);

        assertThat(repository.findBySku("sku-1").getQuantity()).isEqualTo(9);
    }

    @Test
    void jsonbQueryFindsByAttribute() {
        repository.save(orderWithMetadata("{\"channel\": \"mobile\"}"));

        List<Order> result = repository.findByMetadataChannel("mobile");  // metadata->>'channel'

        assertThat(result).hasSize(1);
    }
}

The static on the container matters: JUnit starts a static @Container once per test class (instead of per test method). Flyway runs your real production migrations against it, so schema drift is caught too.

Singleton container pattern — once per JVM, not per class

Even once-per-class adds up across 30 repository test classes. The classic pattern shares one container for the whole test run:

// A base class; the container is started once, in a static initializer,
// and deliberately never stopped — Testcontainers' Ryuk sidecar reaps it
// when the JVM exits.
public abstract class AbstractPostgresTest {

    static final PostgreSQLContainer<?> POSTGRES =
            new PostgreSQLContainer<>("postgres:16-alpine").withReuse(true);

    static { POSTGRES.start(); }

    @DynamicPropertySource       // pre-@ServiceConnection way to inject the coordinates
    static void props(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url",      POSTGRES::getJdbcUrl);
        registry.add("spring.datasource.username", POSTGRES::getUsername);
        registry.add("spring.datasource.password", POSTGRES::getPassword);
    }
}

class OrderRepositoryTest extends AbstractPostgresTest { ... }
class CustomerRepositoryTest extends AbstractPostgresTest { ... }  // same container ✔

Alternatively in Boot 3.1+: declare the @ServiceConnection container as a @Bean in a @TestConfiguration(proxyBeanMethods = false) class and @Import it — the context cache then shares it naturally. With .withReuse(true) plus testcontainers.reuse.enable=true in ~/.testcontainers.properties, the container even survives between test runs on your dev machine — near-instant local feedback.

Awareness: the same pattern covers the rest of your infrastructure — KafkaContainer, Redis via GenericContainer("redis:7"), LocalStackContainer for AWS APIs, even a browser for Selenium. One line each with @ServiceConnection support for Kafka/Redis in recent Boot versions. CI needs a Docker daemon (standard on GitHub Actions runners).

Trade-off framing for interviews: Testcontainers tests are integration tests — seconds, not milliseconds. You don't replace unit tests with them; you replace the lying H2 layer with them, keep the count moderate, and use the singleton pattern + context caching so the whole suite still runs in a couple of minutes. Cleanup between tests matters too: @DataJpaTest's per-test rollback still applies; for @SpringBootTest flows, truncate tables in @BeforeEach or use SQL scripts via @Sql.

Contract testing, WireMock & test-quality awareness T3

Contract testing — why microservices need it

Problem: service A's tests mock service B; service B's tests mock service A. Both suites are green while the real API drifted — B renamed a field, A still expects the old one, and nobody finds out until staging (or production). Mocks encode assumptions, and nothing verifies the assumptions.

Consumer-driven contracts fix this: the consumer records its expectations of the provider as a machine-readable contract; the provider's CI replays those contracts against the real provider code. Drift breaks the provider's build, not production.

  • Spring Cloud Contract: contracts as Groovy/YAML in the provider repo; generates provider-side verification tests and consumer-side stubs.
  • Pact: polyglot (you may know pact-js from the frontend world — same tool, JVM flavor); consumer tests generate pact files, shared via a Pact Broker, provider verifies them.

Awareness depth is enough: know the problem (API drift between independently deployed services), the mechanism (consumer expectations verified against the real provider), and the two tool names.

WireMock — stubbing external HTTP

When your service calls a third-party API (payment provider, geocoder), you can't mock RestClient meaningfully and you can't call the real thing in CI. WireMock runs a real local HTTP server that returns canned responses — msw/nock for the JVM, exercising your actual HTTP client, serialization, timeouts and error handling:

// testImplementation "org.wiremock:wiremock-standalone"
@SpringBootTest
class PaymentClientTest {

    static WireMockServer wireMock = new WireMockServer(0);  // random port

    @BeforeAll  static void start() { wireMock.start(); }
    @AfterAll   static void stop()  { wireMock.stop(); }
    @AfterEach  void reset()         { wireMock.resetAll(); }

    @DynamicPropertySource
    static void props(DynamicPropertyRegistry r) {
        r.add("payment.api.base-url", wireMock::baseUrl);  // point client at the stub
    }

    @Autowired PaymentClient client;

    @Test
    void mapsDeclinedResponseToDomainException() {
        wireMock.stubFor(post(urlEqualTo("/v1/charges"))
                .willReturn(aResponse()
                        .withStatus(402)
                        .withHeader("Content-Type", "application/json")
                        .withBody("""
                            {"error": "card_declined", "code": "insufficient_funds"}
                            """)));

        assertThatThrownBy(() -> client.charge(testCard()))
                .isInstanceOf(PaymentDeclinedException.class)
                .hasMessageContaining("insufficient_funds");
    }

    @Test
    void timesOutSlowProvider() {
        wireMock.stubFor(post(urlEqualTo("/v1/charges"))
                .willReturn(aResponse().withFixedDelay(5000)));  // slower than client timeout

        assertThatThrownBy(() -> client.charge(testCard()))
                .isInstanceOf(PaymentUnavailableException.class);
    }
}

Test data builders — taming Java's arrange phase

Java test setup gets verbose fast. The builder pattern with sensible defaults keeps tests focused on what varies (think a typed factory function / fishery in TS):

public class OrderTestBuilder {
    private String sku = "sku-default";         // every field has a valid default
    private int quantity = 1;
    private Status status = Status.NEW;
    private Instant createdAt = Instant.now();

    public static OrderTestBuilder anOrder() { return new OrderTestBuilder(); }

    public OrderTestBuilder paid()             { this.status = Status.PAID; return this; }
    public OrderTestBuilder withSku(String s)  { this.sku = s; return this; }
    public OrderTestBuilder createdDaysAgo(int d) {
        this.createdAt = Instant.now().minus(Duration.ofDays(d)); return this;
    }
    public Order build() { return new Order(sku, quantity, status, createdAt); }
}

// In tests — reads like the requirement:
Order stale = anOrder().paid().createdDaysAgo(10).build();

Coverage honesty & mutation testing (awareness)

Line coverage (JaCoCo on the JVM, ≈ Istanbul) measures what executed, not what was verified — a test with zero assertions can produce 100% coverage. Mutation testing (PIT/pitest for Java, ≈ Stryker in JS) measures test quality directly: it mutates your code (flip a > to >=, delete a call) and checks that some test fails. Surviving mutants = code your suite executes but doesn't actually check. Senior stance: coverage thresholds as a tripwire, never a target (Goodhart's law); run mutation testing occasionally on critical packages rather than every build (it's expensive).

Interview Q&A

What is a cross-cutting concern, and how does Spring AOP address it?

A concern that spans many modules regardless of their business purpose — logging, timing, transactions, security, caching, retries. Without AOP each method repeats the same wrapper boilerplate. Spring AOP extracts that into an aspect: advice code plus a pointcut selecting which method executions (join points) it wraps. Spring implements this with runtime proxies — the container hands you a wrapper object instead of your bean, and the wrapper runs advice around delegated calls. Result: business methods contain only business logic, and the concern is defined once.

Explain the difference between JDK dynamic proxies and CGLIB proxies. Which does Spring Boot use?

JDK dynamic proxies use java.lang.reflect.Proxy to generate, at runtime, a class implementing the target's interfaces — so they require an interface and you must inject by the interface type. CGLIB generates a subclass of the concrete class, overriding its public methods — no interface needed, but the class and advised methods can't be final. Spring Framework's classic default was JDK proxies when an interface exists; Spring Boot defaults to CGLIB for everything (proxyTargetClass=true) so that injecting by concrete class always works. Bonus point: since Spring 6, CGLIB is repackaged inside spring-core, not a separate dependency.

My @Transactional method doesn't open a transaction. Walk me through your debugging checklist.

In order of likelihood: (1) Self-invocation — is the method called via this.method() from another method in the same class? That bypasses the proxy entirely. Fix by extracting to another bean or self-injecting via ObjectProvider. (2) Method visibility — is it public? Proxies can't advise private methods; historically only public was supported. (3) Is the object a Spring bean at all? If it was created with new, there's no proxy. (4) Is the method or class final? CGLIB can't override it. (5) Is the exception checked? Default rollback rules only cover unchecked exceptions — the transaction exists but commits anyway. (6) Rarely: wrong @Transactional import (jakarta vs Spring — both work in Boot but with different attribute sets), or no transaction manager configured. I'd verify by logging TransactionSynchronizationManager.isActualTransactionActive() or enabling org.springframework.transaction DEBUG logs.

What exactly happens on self-invocation, and what are the fixes?

The proxy wraps the target object; advice runs only when a call enters through the proxy reference. Inside the target, this is the raw target, so this.otherMethod() is a plain Java call that never touches the proxy — any @Transactional/@Cacheable/@Async on otherMethod silently no-ops. Fixes: (1) preferred — extract the advised method into a separate bean and inject it, which usually improves the design anyway; (2) self-inject the bean via ObjectProvider<Self> (lazy, avoids circular-dependency issues) and call self.getObject().otherMethod(); (3) AopContext.currentProxy() with exposeProxy=true — works but couples business code to AOP internals, I'd push back on it in review. AspectJ compile-time weaving also solves it but is a heavy hammer.

Name the five advice types and when you'd choose @Around vs the others.

@Before (pre-call), @AfterReturning (successful return, can read the value), @AfterThrowing (exception escaped), @After (finally-style, either way), and @Around, which receives a ProceedingJoinPoint and fully controls the invocation — it can modify arguments, skip the call, replace the return value, catch/translate exceptions, and measure time across the whole call. Rule of thumb: use the least powerful advice that works, because @Around's power is also its risk — forgetting proceed() silently prevents the target from running, and swallowing its exception hides failures. Timing and caching genuinely need @Around; simple logging usually only needs @Before/@AfterThrowing.

What are the limitations of Spring AOP compared to full AspectJ?

Spring AOP is proxy-based and runtime-only, so it can only intercept public method executions on Spring-managed beans, called from outside the object. It cannot advise: self-invocations, private/protected methods, final methods/classes, static methods, constructors, field access, or objects created with new. AspectJ weaves bytecode (compile-time or load-time) and can do all of those, at the cost of build/agent complexity. In practice Spring AOP covers ~95% of real needs, and its limitations are the same ones that explain @Transactional's rules — which is why understanding proxies pays off across the whole framework.

How does @Cacheable work, and what are two gotchas?

With @EnableCaching, Spring wraps the bean in a proxy whose around-advice computes a cache key (from parameters, customizable via SpEL in key), checks the configured CacheManager, and only invokes the method body on a miss, storing the result afterwards. @CacheEvict/@CachePut handle invalidation and forced refresh. Gotchas: (1) self-invocation — an internal call never hits the caching advice, so it recomputes every time; (2) with the default in-memory store, cached values are shared object references — mutating a returned entity corrupts the cache; (3) bonus: null handling and cache stampedes on hot keys need explicit thought (sync=true helps with stampedes).

What are the gotchas of @Async?

(1) It requires @EnableAsync — without it, the annotation is silently ignored. (2) Same proxy rules: self-invoked @Async methods run synchronously. (3) Exception swallowing: for a void method, exceptions never reach the caller — they go to an AsyncUncaughtExceptionHandler that by default just logs; returning CompletableFuture puts the exception in the future, but it only surfaces if someone actually joins it — the JVM's unhandled promise rejection. (4) The default executor should be replaced with a tuned ThreadPoolTaskExecutor; unbounded queues or too-small pools cause invisible latency. (5) Thread-locals (security context, MDC, transaction) don't propagate to the async thread by default.

How do JUnit 5 and Jest differ in test isolation?

Jest isolates via module registry resets and fresh closures per test file; state inside a describe persists across its unless reset. JUnit 5 creates a new instance of the test class for every test method (default PER_METHOD lifecycle), so instance fields are automatically fresh — @BeforeEach is for setup logic, not for resetting fields. Static state (and @BeforeAll, which must be static by default) is shared. You can opt into @TestInstance(PER_CLASS) to share one instance, which is occasionally useful for expensive non-static setup. Structurally, @Nested classes ≈ describe blocks, @DisplayName ≈ the test string, @ParameterizedTest ≈ it.each.

Why prefer AssertJ over JUnit's built-in assertions?

Three reasons: (1) readability/discoverability — assertThat(actual) puts the actual value first (like Jest's expect) and the fluent API autocompletes to type-appropriate assertions; JUnit's assertEquals(expected, actual) argument order is a chronic source of confusing failure messages. (2) Failure messages — AssertJ diffs collections, shows which elements were missing/unexpected, etc. (3) Rich collection/exception support — extracting(), containsExactlyInAnyOrder(), assertThatThrownBy().hasMessageContaining() replace loops and try/catch in tests. It ships in spring-boot-starter-test, so there's no cost to adopting it.

Explain @Mock vs @Spy vs @InjectMocks vs @MockitoBean.

@Mock creates a full stub — every method returns default values until stubbed (≈ jest.fn() for the whole object). @Spy wraps a real object — real methods execute unless explicitly stubbed with doReturn().when() (≈ jest.spyOn); needing one is often a design smell. @InjectMocks instantiates the real class under test and injects the @Mocks into its constructor — all plain Mockito, no Spring context, activated by @ExtendWith(MockitoExtension.class). @MockitoBean is different in kind: it's a Spring test annotation that replaces a bean inside the Spring application context with a mock — used in slice tests like @WebMvcTest. It replaced @MockBean, which is deprecated since Boot 3.4. Rule: no context → @Mock; context → @MockitoBean.

What's the argument-matcher mixing rule in Mockito and why does it exist?

If any argument in a stub/verify uses a matcher (any(), anyLong(), argThat()), then all arguments must be matchers — wrap literals with eq(): verify(repo).find(eq(Tier.GOLD), anyBoolean()). Mixing raw values and matchers throws InvalidUseOfMatchersException at runtime. The reason is mechanical: matchers don't pass values, they push markers onto a thread-local stack as a side effect, and Mockito pops them positionally when the mocked call is recorded — a raw value in the middle desynchronizes the stack. Jest has no analogue because expect.any() returns real objects compared structurally.

When would you use an ArgumentCaptor, and what's the alternative?

Use it when the behavior under test is what your code constructed and passed to a collaborator — e.g. the service builds a User with normalized email and hashed password and hands it to repository.save(). verify(repo).save(captor.capture()) then rich AssertJ assertions on captor.getValue() beats trying to build the exact expected object for an equality match. Alternative: verify(repo).save(argThat(u -> u.email().equals("ada@example.com"))) — fine for one condition, but captor + assertThat gives far better failure messages for multi-property checks. Anti-pattern to mention: capturing arguments merely to re-verify things already asserted elsewhere — captors are for genuinely important outputs.

What does @WebMvcTest actually load, and what belongs in that layer's tests?

It boots only the MVC slice: the specified controller(s), Jackson converters, Bean Validation, @ControllerAdvice exception handlers, filters, and (if present) the Spring Security filter chain — no services, no repositories, no datasource. Collaborators are provided as @MockitoBeans. Tests drive it through MockMvc (mocked servlet infrastructure — no real port; ≈ supertest against the Express app). What belongs here: HTTP semantics — routing, status codes, request/response JSON shape via jsonPath, validation failures producing correct error bodies, exception-to-status mapping, headers, and security rules with @WithMockUser. What doesn't: business logic permutations — those are stubbed here and unit-tested in the service.

Why is @SpringBootTest slow, and how does Spring keep suites tolerable?

It bootstraps the entire application context — component scan, all beans, auto-configuration, connection pools — which costs seconds per distinct context. Spring's mitigation is the context cache: contexts are cached across test classes within a JVM keyed by their full configuration (classes, profiles, properties, bean overrides). Fifty test classes with identical configuration boot one context. What breaks the cache: each distinct combination of @MockitoBeans, @ActiveProfiles, @TestPropertySource, or @DirtiesContext forces a new (or discarded) context. Senior practice: standardize a small number of test configurations (e.g. one shared base class for integration tests), watch for accidental cache-key drift, and keep @SpringBootTest count low by pushing checks down to slices and unit tests.

Why can H2 pass tests that fail on Postgres? Give concrete examples.

H2 is a different database engine with a Postgres compatibility mode, not Postgres. Concrete divergences: JSONB columns and operators (->>, @>) aren't really supported; ON CONFLICT ... DO UPDATE upserts and RETURNING differ; window functions and CTE edge cases behave differently; identifier case-sensitivity, sequence allocation, locking semantics, and error codes all differ. Worse, your Flyway migrations are written in Postgres dialect — on H2 you either can't run them or maintain a parallel set, so the tested schema isn't the shipped schema. Any native query or Postgres-specific feature is therefore untested until production. The fix is Testcontainers: run the real Postgres image in Docker during tests.

Walk me through setting up Testcontainers Postgres in Boot 3.x. What does @ServiceConnection do?

Dependencies: spring-boot-testcontainers, testcontainers:junit-jupiter, testcontainers:postgresql. On the test class: @Testcontainers (JUnit extension managing lifecycle), a static field @Container @ServiceConnection PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine"), and for @DataJpaTest also @AutoConfigureTestDatabase(replace = NONE) so Boot doesn't swap in H2. @ServiceConnection (Boot 3.1+) inspects the container type and auto-populates the matching connection properties — URL, username, password for a datasource; bootstrap servers for Kafka — replacing the older @DynamicPropertySource boilerplate. static means once per class; for suite-wide sharing use the singleton pattern (static initializer in a base class, or a @TestConfiguration bean shared via context caching) plus optionally withReuse(true) for cross-run reuse locally.

You inherit a suite that's 400 @SpringBootTest classes taking 40 minutes. Plan?

(1) Measure first: how many distinct contexts are being created (Spring logs context creation; the cache size is the multiplier), which tests are slowest. (2) Fix cache churn — consolidate @MockitoBean/profile/property variations into a handful of shared configurations; hunt down @DirtiesContext. That alone often halves runtime with zero test rewrites. (3) Demote tests down the pyramid: business-logic tests hiding inside @SpringBootTest become plain Mockito unit tests; controller-shape tests become @WebMvcTest; query tests become @DataJpaTest on a singleton Testcontainer. (4) Parallelize what remains (JUnit parallel execution / Gradle forks — mind shared DB state). (5) Ratchet: fail CI if context count grows. Frame it as an incremental migration with the fast-feedback win stated in minutes, not ideology.

What is consumer-driven contract testing and when is it worth adopting?

It solves mock drift between independently deployed services: consumer tests stub the provider, provider tests stub the consumer, both green while the real API diverged. In CDC, the consumer publishes machine-readable expectations (a contract/pact: "POST /v1/charges with this shape returns 201 with that shape"); the provider's CI replays those contracts against real provider code, so a breaking change fails the provider's build before deployment — and consumers get auto-generated stubs guaranteed to match. Tools: Spring Cloud Contract (JVM-centric, contracts in the provider repo) and Pact (polyglot, broker-based — the same Pact known from pact-js). Worth it when: multiple teams deploy services independently and integration bugs are escaping to staging. Overkill for: a monolith, or two services owned by one team who can just run real integration tests together.

A service has 92% line coverage. Is it well-tested? How would you actually find out?

Unknown — coverage measures execution, not verification. A test that calls code and asserts nothing produces coverage; assertions that check the wrong thing produce coverage; the missing 8% might contain the only genuinely risky branches. To assess quality: (1) read a sample of tests — do they assert observable behavior or mirror implementation via stub-then-verify-the-stubs? (2) run mutation testing (PIT/pitest, ≈ Stryker in JS): it mutates the code (invert conditionals, remove calls) and reports mutants no test kills — surviving mutants in critical code are directly actionable; (3) check the layer mix — is Postgres-specific SQL tested on Postgres or on H2? (4) check what coverage excludes (generated code, config). Senior stance: coverage thresholds as a tripwire against regressions in test discipline, never as a target — Goodhart's law applies immediately.

Hands-on exercises

Exercise 1 — Build an audit aspect

Create a custom @Audited(action = "...") annotation and an aspect that, after a successful method return, records who did what: the annotation's action, the method name, and the current username (assume a CurrentUserProvider bean). On exception, record a failed audit entry instead. Use the appropriate advice types (not everything needs @Around).

Solution 
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Audited {
    String action();
}

@Aspect
@Component
@RequiredArgsConstructor
public class AuditAspect {

    private final AuditLog auditLog;
    private final CurrentUserProvider currentUser;

    @AfterReturning("@annotation(audited)")
    public void onSuccess(JoinPoint jp, Audited audited) {
        auditLog.record(AuditEntry.success(
                audited.action(),
                jp.getSignature().toShortString(),
                currentUser.username()));
    }

    @AfterThrowing(pointcut = "@annotation(audited)", throwing = "ex")
    public void onFailure(JoinPoint jp, Audited audited, Exception ex) {
        auditLog.record(AuditEntry.failure(
                audited.action(),
                jp.getSignature().toShortString(),
                currentUser.username(),
                ex.getClass().getSimpleName()));
    }
}

// Usage:
@Audited(action = "ORDER_CANCELLED")
public void cancel(Long orderId) { ... }

Note the deliberate choice of @AfterReturning/@AfterThrowing over @Around: this aspect cannot accidentally swallow the return value or the exception. Binding audited in the pointcut expression gives type-safe access to the annotation's attributes.

Exercise 2 — Diagnose and fix the silent no-op

The following class is buggy: the cache never populates and each import doesn't get its own transaction. Explain exactly why, then fix it two different ways.

@Service
public class CatalogService {

    public void refreshAll(List<String> skus) {
        for (String sku : skus) {
            importOne(sku);
            Product p = findCached(sku);   // warm the cache?
        }
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void importOne(String sku) { ... }

    @Cacheable("products")
    public Product findCached(String sku) { ... }
}
Solution

Diagnosis: refreshAll calls importOne and findCached through this — plain Java calls on the target object that never pass through the CGLIB proxy. The transactional advice and the caching advice therefore never run: no REQUIRES_NEW transaction per import (everything joins whatever transaction the outer caller has, or none), and every findCached executes the body, never touching the cache.

Fix A — extract a worker bean (preferred):

@Service
@RequiredArgsConstructor
public class CatalogRefresher {
    private final CatalogService catalog;   // injected → proxied

    public void refreshAll(List<String> skus) {
        for (String sku : skus) {
            catalog.importOne(sku);      // crosses the proxy ✔
            catalog.findCached(sku);     // caching advice runs ✔
        }
    }
}

Fix B — self-injection:

@Service
public class CatalogService {
    private final ObjectProvider<CatalogService> self;

    public CatalogService(ObjectProvider<CatalogService> self) { this.self = self; }

    public void refreshAll(List<String> skus) {
        for (String sku : skus) {
            self.getObject().importOne(sku);
            self.getObject().findCached(sku);
        }
    }
    // annotated methods unchanged
}

Prefer Fix A: orchestration and per-item work are separate responsibilities, and the resulting classes are easier to test independently.

Exercise 3 — Unit-test a service with Mockito + ArgumentCaptor

Given TransferService.transfer(fromId, toId, amount) which loads both accounts from AccountRepository, debits/credits them, saves both, and publishes a TransferCompleted event via EventPublisher — write a plain Mockito test (no Spring) proving: both accounts saved with correct balances (use a captor), the event carries the right amount, and that an insufficient balance throws and publishes nothing.

Solution 
@ExtendWith(MockitoExtension.class)
class TransferServiceTest {

    @Mock        AccountRepository accounts;
    @Mock        EventPublisher events;
    @InjectMocks TransferService service;

    @Captor ArgumentCaptor<Account> accountCaptor;

    @Test
    void transfersAndPublishesEvent() {
        given(accounts.findById(1L)).willReturn(Optional.of(account(1L, "100.00")));
        given(accounts.findById(2L)).willReturn(Optional.of(account(2L, "50.00")));

        service.transfer(1L, 2L, new BigDecimal("30.00"));

        verify(accounts, times(2)).save(accountCaptor.capture());
        assertThat(accountCaptor.getAllValues())
                .extracting(Account::getId, Account::getBalance)
                .containsExactlyInAnyOrder(
                        tuple(1L, new BigDecimal("70.00")),
                        tuple(2L, new BigDecimal("80.00")));

        verify(events).publish(argThat((TransferCompleted e) ->
                e.amount().compareTo(new BigDecimal("30.00")) == 0));
    }

    @Test
    void insufficientBalanceThrowsAndPublishesNothing() {
        given(accounts.findById(1L)).willReturn(Optional.of(account(1L, "10.00")));
        given(accounts.findById(2L)).willReturn(Optional.of(account(2L, "50.00")));

        assertThatThrownBy(() -> service.transfer(1L, 2L, new BigDecimal("30.00")))
                .isInstanceOf(InsufficientBalanceException.class);

        verify(accounts, never()).save(any());
        verifyNoInteractions(events);
    }

    private static Account account(long id, String balance) {
        return new Account(id, new BigDecimal(balance));
    }
}

Notes: getAllValues() collects every captured argument across the two save calls; AssertJ's tuple keeps the multi-property assertion readable. The failure test asserts both the exception and the absence of side effects — the latter is the business-critical part.

Exercise 4 — Controller slice test with validation errors

Write a @WebMvcTest for POST /api/customers accepting {"name": "...", "email": "..."} where name is @NotBlank and email is @Email. Cover: 201 + Location header on success; 400 with both field errors when both fields are invalid (assert the error body and that the service was never called).

Solution 
@WebMvcTest(CustomerController.class)
class CustomerControllerTest {

    @Autowired   MockMvc mockMvc;
    @MockitoBean CustomerService customerService;

    @Test
    void createsCustomer() throws Exception {
        given(customerService.create(any()))
                .willReturn(new CustomerResponse(7L, "Ada", "ada@example.com"));

        mockMvc.perform(post("/api/customers")
                    .contentType(MediaType.APPLICATION_JSON)
                    .content("""
                        {"name": "Ada", "email": "ada@example.com"}
                        """))
               .andExpect(status().isCreated())
               .andExpect(header().string("Location", "/api/customers/7"))
               .andExpect(jsonPath("$.name").value("Ada"));
    }

    @Test
    void rejectsInvalidBodyWithFieldErrors() throws Exception {
        mockMvc.perform(post("/api/customers")
                    .contentType(MediaType.APPLICATION_JSON)
                    .content("""
                        {"name": "", "email": "not-an-email"}
                        """))
               .andExpect(status().isBadRequest())
               .andExpect(jsonPath("$.errors.length()").value(2))
               .andExpect(jsonPath("$.errors[?(@.field=='name')]").exists())
               .andExpect(jsonPath("$.errors[?(@.field=='email')]").exists());

        then(customerService).shouldHaveNoInteractions();
    }
}

The second test exercises three things you'd otherwise never cover in unit tests: Bean Validation wiring on the DTO, your @RestControllerAdvice's MethodArgumentNotValidException handler producing the error body shape, and the guarantee that invalid input never reaches the service.

Exercise 5 — Repository test on real Postgres with Testcontainers

You have a native query using Postgres's ON CONFLICT: InventoryRepository.upsertStock(sku, qty). Write a @DataJpaTest that runs against a real postgres:16-alpine container using @ServiceConnection, verifies the upsert behavior (second call updates, doesn't duplicate), and explain the two extra annotations needed beyond a plain @DataJpaTest.

Solution 
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Testcontainers
class InventoryRepositoryTest {

    @Container
    @ServiceConnection
    static PostgreSQLContainer<?> postgres =
            new PostgreSQLContainer<>("postgres:16-alpine");

    @Autowired InventoryRepository repository;

    @Test
    void upsertInsertsThenUpdatesWithoutDuplicating() {
        repository.upsertStock("sku-1", 5);
        repository.upsertStock("sku-1", 12);   // ON CONFLICT (sku) DO UPDATE

        List<Inventory> all = repository.findAll();
        assertThat(all).hasSize(1);
        assertThat(all.get(0).getQuantity()).isEqualTo(12);
    }
}

/* The repository method under test:
@Modifying
@Query(value = """
    INSERT INTO inventory (sku, quantity) VALUES (:sku, :qty)
    ON CONFLICT (sku) DO UPDATE SET quantity = EXCLUDED.quantity
    """, nativeQuery = true)
void upsertStock(@Param("sku") String sku, @Param("qty") int qty);
*/

The two extra annotations: (1) @AutoConfigureTestDatabase(replace = NONE) — without it, @DataJpaTest silently replaces your datasource with embedded H2, which would reject the ON CONFLICT syntax (or, worse in other cases, accept a subtly different behavior); (2) @Testcontainers + the static @Container @ServiceConnection field — @Testcontainers manages start/stop via the JUnit extension, static shares one container across the class's tests, and @ServiceConnection (Boot 3.1+) feeds the container's JDBC URL/credentials into Spring's datasource properties automatically. For a large suite, promote the container to the singleton pattern in a shared base class.
