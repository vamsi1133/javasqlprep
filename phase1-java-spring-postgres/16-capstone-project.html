16 — Capstone: Order-Management API

One project that exercises every must-know topic from tutorials 1–15 — and becomes your best interview story.

Why build this

You have now read about dependency injection, JPA relationship mapping, transaction isolation, composite indexes, and JWT filters. Reading is how a senior engineer ramps fast — but interviews reward recall under pressure, and recall comes from having typed the thing, broken it, and fixed it. This capstone converts fifteen tutorials of passive knowledge into one coherent artifact you can talk about for forty-five minutes without slides.

Three concrete payoffs:

  • Muscle memory. You will hand-write an entity mapping, a @Transactional service, a security filter chain, and a Flyway migration. The second time you do any of these (in the interview, or on the job) takes a tenth of the time.
  • A "tell me about something you built" answer. Instead of describing seven years of React work to a backend-heavy panel, you can say: "I recently built an order-management API in Spring Boot and Postgres — let me walk you through how I handled a concurrent-inventory race and a 300× query speedup." Sections below turn each milestone into a rehearsed story.
  • A place to feel the failure modes. N+1 queries, oversold stock, sequential-scan pagination — you will deliberately cause each one, observe it in logs and EXPLAIN output, and fix it. Interviewers can instantly tell the difference between someone who read about N+1 and someone who has watched 101 queries scroll past for a single HTTP request.
As a senior frontend dev you already know how to design APIs from the consumer side — you have been on the receiving end of bad pagination and vague 500s for years. Lean into that: this project is your chance to build the backend you always wished your React apps were talking to. That "I've consumed hundreds of APIs, here's what I made sure mine did differently" framing is a genuinely strong interview angle for a frontend-to-fullstack move.

Budget: roughly 12–20 focused hours across a week or two. Every milestone below has a checkpoint — a command to run and the output you should see — so you always know whether you are on track.

Project spec

Product: a REST API for a small e-commerce back office. Four domain concepts: customers, products, orders, order_items.

Functional requirements

  • Place an order — POST /api/orders with a list of product lines. Validates that every product exists and has sufficient stock, decrements inventory, and creates the order with its items — all in one transaction. Under concurrent load, stock must never go negative (no overselling).
  • Order lifecycle — status moves PENDING → PAID → SHIPPED, with CANCELLED reachable from PENDING or PAID. Illegal transitions (e.g. SHIPPED → PENDING) are rejected with a clear error. Enforced in the domain model, backed by a database CHECK constraint.
  • List orders — GET /api/orders?status=PAID&page=0&size=20 with pagination and filtering by status and customer.
  • Order details — GET /api/orders/{id} with items and product info.
  • Security — JWT bearer tokens. A CUSTOMER sees and places only their own orders; an ADMIN sees everything and drives status transitions.
  • Auditability — every row carries created_at / updated_at timestamps (timestamptz).

Tech stack (fixed — don't bikeshed this)

  • Java 21, Spring Boot 3.x, Maven
  • PostgreSQL 16 in Docker (docker-compose)
  • Flyway for migrations — ddl-auto stays on validate, always
  • Spring Data JPA / Hibernate, Spring Security (OAuth2 resource server with an HMAC-signed JWT)
  • Testcontainers + JUnit 5 + Mockito + AssertJ for the test pyramid
Base package for everything: com.prep.orders. All code below is written against that package and against consistent entity names (Customer, Product, Order, OrderItem) so the milestones assemble into one runnable project if you type them out in order.

Milestone 2 — Flyway V1: the schema

Goal: the full DDL, written by hand, with every design decision defensible.

src/main/resources/db/migration/V1__init_schema.sql:

-- Sequences: INCREMENT BY 50 matches Hibernate's allocationSize (Milestone 3),
-- so JPA grabs id blocks in one round-trip instead of one nextval per insert.
CREATE SEQUENCE customer_seq   START WITH 1 INCREMENT BY 50;
CREATE SEQUENCE product_seq    START WITH 1 INCREMENT BY 50;
CREATE SEQUENCE order_seq      START WITH 1 INCREMENT BY 50;
CREATE SEQUENCE order_item_seq START WITH 1 INCREMENT BY 50;

CREATE TABLE customers (
    id            BIGINT PRIMARY KEY DEFAULT nextval('customer_seq'),
    email         TEXT        NOT NULL UNIQUE,
    full_name     TEXT        NOT NULL,
    password_hash TEXT        NOT NULL,
    role          TEXT        NOT NULL DEFAULT 'CUSTOMER'
                  CHECK (role IN ('CUSTOMER', 'ADMIN')),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE products (
    id         BIGINT PRIMARY KEY DEFAULT nextval('product_seq'),
    sku        TEXT           NOT NULL UNIQUE,
    name       TEXT           NOT NULL,
    price      NUMERIC(12,2)  NOT NULL CHECK (price >= 0),
    stock      INTEGER        NOT NULL CHECK (stock >= 0),  -- the oversell backstop
    created_at TIMESTAMPTZ    NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ    NOT NULL DEFAULT now()
);

CREATE TABLE orders (
    id           BIGINT PRIMARY KEY DEFAULT nextval('order_seq'),
    customer_id  BIGINT         NOT NULL REFERENCES customers(id),
    status       TEXT           NOT NULL DEFAULT 'PENDING'
                 CHECK (status IN ('PENDING', 'PAID', 'SHIPPED', 'CANCELLED')),
    total_amount NUMERIC(12,2)  NOT NULL DEFAULT 0 CHECK (total_amount >= 0),
    version      BIGINT         NOT NULL DEFAULT 0,   -- optimistic locking (Milestone 3)
    created_at   TIMESTAMPTZ    NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ    NOT NULL DEFAULT now()
);

-- Postgres does NOT auto-index FK columns. Without this, every
-- "orders for customer X" query and every DELETE on customers seq-scans orders.
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

CREATE TABLE order_items (
    id         BIGINT PRIMARY KEY DEFAULT nextval('order_item_seq'),
    order_id   BIGINT        NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
    product_id BIGINT        NOT NULL REFERENCES products(id),
    quantity   INTEGER       NOT NULL CHECK (quantity > 0),
    unit_price NUMERIC(12,2) NOT NULL CHECK (unit_price >= 0),
    UNIQUE (order_id, product_id)   -- also serves as the FK index on order_id
);

CREATE INDEX idx_order_items_product_id ON order_items (product_id);

Decisions worth being able to defend out loud — these are straight from Tutorial 12 (schema design):

  • NUMERIC(12,2) for money, never float/double — binary floating point cannot represent 0.10 exactly; totals drift by cents at scale.
  • TIMESTAMPTZ, not TIMESTAMP — stores an instant (UTC internally); plain timestamp is a wall-clock reading with no timezone and is a classic data bug.
  • CHECK on status and stock >= 0 — the database is the last line of defense. Even if a bug slips past the Java state machine, the DB refuses corrupt data.
  • unit_price denormalized onto order_items — a deliberate, correct denormalization: the order must record the price at purchase time; joining to products.price would rewrite history whenever the catalog price changes.
  • Explicit FK indexes — Postgres indexes the referenced side (the PK) automatically, never the referencing side. Note how UNIQUE (order_id, product_id) doubles as the order_id FK index because order_id is its leading column.
Do not add every index you can imagine "while you're here". Each index taxes every write. The composite index for the list query is deliberately deferred to Milestone 9, where you will justify it with EXPLAIN ANALYZE evidence — that before/after is the whole point.
Checkpoint: restart the app. Flyway logs Migrating schema "public" to version "1 - init schema". Then docker exec -it orders-db psql -U orders -c "\d orders" shows the table with its CHECK constraints.
What you're practicing: Tutorial 12 (schema design: types, constraints, normalization calls) and Tutorial 7 (why migrations, not ddl-auto, own the schema in any real system).

Milestone 3 — JPA entities and relationships, done right

Goal: entities that map the schema cleanly, with every relationship LAZY and the object graph kept consistent by helper methods.

The status enum owns the legal-transition rules — the state machine lives in the domain, not scattered through services:

// com/prep/orders/order/OrderStatus.java
package com.prep.orders.order;

import java.util.EnumSet;
import java.util.Map;
import java.util.Set;

public enum OrderStatus {
    PENDING, PAID, SHIPPED, CANCELLED;

    private static final Map<OrderStatus, Set<OrderStatus>> LEGAL = Map.of(
        PENDING,   EnumSet.of(PAID, CANCELLED),
        PAID,      EnumSet.of(SHIPPED, CANCELLED),
        SHIPPED,   EnumSet.noneOf(OrderStatus.class),
        CANCELLED, EnumSet.noneOf(OrderStatus.class)
    );

    public boolean canTransitionTo(OrderStatus target) {
        return LEGAL.get(this).contains(target);
    }
}

The aggregate root:

// com/prep/orders/order/Order.java
package com.prep.orders.order;

import com.prep.orders.customer.Customer;
import jakarta.persistence.*;
import org.hibernate.annotations.CreationTimestamp;
import org.hibernate.annotations.UpdateTimestamp;

import java.math.BigDecimal;
import java.time.Instant;
import java.util.ArrayList;
import java.util.List;

@Entity
@Table(name = "orders")   // "order" is a reserved word in SQL — table is plural for a reason
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "order_seq")
    @SequenceGenerator(name = "order_seq", sequenceName = "order_seq", allocationSize = 50)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "customer_id", nullable = false)
    private Customer customer;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private OrderStatus status = OrderStatus.PENDING;

    @Column(name = "total_amount", nullable = false)
    private BigDecimal totalAmount = BigDecimal.ZERO;

    @Version                       // optimistic lock: concurrent status updates collide safely
    private Long version;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();

    @CreationTimestamp @Column(name = "created_at", updatable = false)
    private Instant createdAt;

    @UpdateTimestamp @Column(name = "updated_at")
    private Instant updatedAt;

    protected Order() {}          // JPA needs it; protected keeps callers honest

    public Order(Customer customer) {
        this.customer = customer;
    }

    // Helper method: keeps BOTH sides of the bidirectional link in sync.
    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);
    }

    public void recalculateTotal() {
        this.totalAmount = items.stream()
            .map(i -> i.getUnitPrice().multiply(BigDecimal.valueOf(i.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    // State machine enforcement lives on the aggregate, not in the controller.
    public void transitionTo(OrderStatus target) {
        if (!status.canTransitionTo(target)) {
            throw new IllegalOrderStateException(id, status, target);
        }
        this.status = target;
    }

    public Long getId() { return id; }
    public Customer getCustomer() { return customer; }
    public OrderStatus getStatus() { return status; }
    public BigDecimal getTotalAmount() { return totalAmount; }
    public List<OrderItem> getItems() { return items; }
    public Instant getCreatedAt() { return createdAt; }
}
// com/prep/orders/order/OrderItem.java
package com.prep.orders.order;

import com.prep.orders.product.Product;
import jakarta.persistence.*;
import java.math.BigDecimal;

@Entity
@Table(name = "order_items")
public class OrderItem {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "order_item_seq")
    @SequenceGenerator(name = "order_item_seq", sequenceName = "order_item_seq", allocationSize = 50)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "order_id", nullable = false)
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "product_id", nullable = false)
    private Product product;

    @Column(nullable = false)
    private int quantity;

    @Column(name = "unit_price", nullable = false)
    private BigDecimal unitPrice;   // price frozen at purchase time (see Milestone 2)

    protected OrderItem() {}

    public OrderItem(Product product, int quantity, BigDecimal unitPrice) {
        this.product = product;
        this.quantity = quantity;
        this.unitPrice = unitPrice;
    }

    void setOrder(Order order) { this.order = order; }  // package-private: only Order.addItem calls it
    public Product getProduct() { return product; }
    public int getQuantity() { return quantity; }
    public BigDecimal getUnitPrice() { return unitPrice; }
}

Customer and Product follow the same pattern (sequence id via customer_seq / product_seq, timestamps, no relationships pointing outward — keep the graph navigable in one direction where you can). Give Product a convenience constructor Product(String sku, String name, BigDecimal price, int stock); the tests in Milestone 10 use it.

@ManyToOne defaults to EAGER. If you forget fetch = FetchType.LAZY on OrderItem.product and Order.customer, every order load drags customer and product rows along whether you need them or not — and Milestone 8's N+1 hunt gets noisier and harder to reason about. Rule: LAZY everywhere, then fetch explicitly per query.
Checkpoint: restart the app with ddl-auto: validate still on. A clean start means Hibernate agrees your mappings match the Flyway schema exactly — column names, types, nullability. A SchemaManagementException here is the checkpoint doing its job: fix the mapping, not the DDL.
What you're practicing: Tutorial 7 (JPA entity design: LAZY fetching, bidirectional sync helpers, @Version optimistic locking) plus Tutorial 2 (enums with behavior, immutability instincts) and Tutorial 12 (mapping decisions echoing schema decisions — allocationSize matching INCREMENT BY 50 is the kind of cross-layer coherence interviewers probe for).

Milestone 4 — Repositories, service layer, and the transactional place-order flow

Goal: the core business flow — atomic, race-safe order placement.

// com/prep/orders/product/ProductRepository.java
package com.prep.orders.product;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Modifying;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;

public interface ProductRepository extends JpaRepository<Product, Long> {

    /**
     * Atomic stock decrement. The WHERE clause is the guard: if stock is
     * insufficient, 0 rows update and we translate that into a business error.
     * Row-level locking inside UPDATE makes this safe under concurrency.
     */
    @Modifying
    @Query("""
        UPDATE Product p
           SET p.stock = p.stock - :qty
         WHERE p.id = :id AND p.stock >= :qty
        """)
    int decrementStock(@Param("id") Long id, @Param("qty") int qty);
}
// com/prep/orders/order/OrderService.java
package com.prep.orders.order;

import com.prep.orders.customer.Customer;
import com.prep.orders.customer.CustomerRepository;
import com.prep.orders.product.Product;
import com.prep.orders.product.ProductRepository;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final ProductRepository productRepository;
    private final CustomerRepository customerRepository;

    // Constructor injection — no @Autowired needed on a single constructor.
    public OrderService(OrderRepository orderRepository,
                        ProductRepository productRepository,
                        CustomerRepository customerRepository) {
        this.orderRepository = orderRepository;
        this.productRepository = productRepository;
        this.customerRepository = customerRepository;
    }

    public record OrderLine(Long productId, int quantity) {}

    @Transactional
    public Order placeOrder(Long customerId, List<OrderLine> lines) {
        Customer customer = customerRepository.findById(customerId)
            .orElseThrow(() -> new NotFoundException("customer", customerId));

        Order order = new Order(customer);

        for (OrderLine line : lines) {
            Product product = productRepository.findById(line.productId())
                .orElseThrow(() -> new NotFoundException("product", line.productId()));

            int updated = productRepository.decrementStock(line.productId(), line.quantity());
            if (updated == 0) {
                // Throwing from a @Transactional method rolls back EVERYTHING —
                // including decrements already applied for earlier lines.
                throw new InsufficientStockException(product.getSku(), line.quantity());
            }
            order.addItem(new OrderItem(product, line.quantity(), product.getPrice()));
        }

        order.recalculateTotal();
        return orderRepository.save(order);
    }

    @Transactional
    public Order changeStatus(Long orderId, OrderStatus target) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new NotFoundException("order", orderId));
        order.transitionTo(target);      // dirty checking flushes the change; no save() needed
        return order;
    }
}

The race, spelled out. The naive implementation reads stock, checks it in Java, then writes:

// BROKEN — read-check-write race (do not ship this):
Product p = productRepository.findById(id).orElseThrow();
if (p.getStock() < qty) throw new InsufficientStockException(...);
p.setStock(p.getStock() - qty);   // two requests both read stock=1, both pass the check → stock = -1

Two correct fixes — know both, and know why you picked one:

  • Atomic conditional UPDATE (used above). The check and the write are one statement; Postgres takes a row lock during the UPDATE, so concurrent decrements serialize on the row and the WHERE stock >= :qty guard makes the loser update 0 rows. Simple, no explicit locking, minimal lock hold time.
  • Pessimistic lock: SELECT … FOR UPDATE (in JPA: @Lock(LockModeType.PESSIMISTIC_WRITE) on a repository findById variant). Locks the row first, then you check-and-modify the entity in Java. Choose this when the decision logic is too rich for one SQL statement (tiered pricing, reservations, multi-row invariants). Cost: locks held for the rest of the transaction, and you must think about lock ordering to avoid deadlocks when orders touch multiple products.
And there is a third layer regardless: the CHECK (stock >= 0) constraint from Milestone 2. Defense in depth — the constraint turns any future concurrency bug into a loud constraint violation instead of silent oversell. In an interview, walking through naive → atomic UPDATE → FOR UPDATE → DB constraint, with the trade-off at each step, is a complete senior answer to "how do you handle concurrent updates?" (Tutorial 14 material.)
@Transactional only works through the Spring proxy. A self-invocation (this.placeOrder(...) from another method in the same class) bypasses the proxy and runs with no transaction. Also: it rolls back on unchecked exceptions by default — if you ever throw a checked exception and expect rollback, you need rollbackFor. Both are top-five interview questions.
Checkpoint: no HTTP layer yet, so prove it with a quick @SpringBootTest or a CommandLineRunner: seed one product with stock 5, place an order for 3, assert stock is 2 and the order row exists with the right total_amount. Then place an order for 99 and verify the whole thing rolls back (stock still 2, no orphan order row).
What you're practicing: Tutorial 5 (constructor injection, service layering), Tutorial 7 (repositories, dirty checking, @Modifying), Tutorial 14 (transactions, atomicity, row locking, lost-update anatomy).

package com.prep.orders.order.dto;

import com.prep.orders.order.Order;
import java.math.BigDecimal;
import java.time.Instant;
import java.util.List;

public record OrderResponse(
        Long id,
        String status,
        BigDecimal totalAmount,
        Instant createdAt,
        List<ItemResponse> items) {

    public record ItemResponse(Long productId, String productName, int quantity, BigDecimal unitPrice) {}

    public static OrderResponse from(Order order) {
        return new OrderResponse(
            order.getId(),
            order.getStatus().name(),
            order.getTotalAmount(),
            order.getCreatedAt(),
            order.getItems().stream()
                .map(i -> new ItemResponse(i.getProduct().getId(), i.getProduct().getName(),
                                           i.getQuantity(), i.getUnitPrice()))
                .toList());
    }
}
// com/prep/orders/order/OrderController.java
package com.prep.orders.order;

import com.prep.orders.order.dto.OrderResponse;
import com.prep.orders.order.dto.PlaceOrderRequest;
import jakarta.validation.Valid;
import org.springframework.http.HttpStatus;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public OrderResponse place(@Valid @RequestBody PlaceOrderRequest request,
                               @RequestParam Long customerId) {   // TEMPORARY — replaced by the JWT claim in Milestone 7
        var lines = request.lines().stream()
            .map(l -> new OrderService.OrderLine(l.productId(), l.quantity()))
            .toList();
        return OrderResponse.from(orderService.placeOrder(customerId, lines));
    }

    @GetMapping("/{id}")
    public OrderResponse get(@PathVariable Long id) {
        return OrderResponse.from(orderService.getOrder(id));
    }

    @PostMapping("/{id}/status")
    public OrderResponse changeStatus(@PathVariable Long id, @RequestParam OrderStatus target) {
        return OrderResponse.from(orderService.changeStatus(id, target));
    }
}

Centralized error handling with RFC 7807 ProblemDetail (built into Spring 6 — no custom error envelope needed):

// com/prep/orders/common/GlobalExceptionHandler.java
package com.prep.orders.common;

import com.prep.orders.order.IllegalOrderStateException;
import com.prep.orders.order.InsufficientStockException;
import com.prep.orders.order.NotFoundException;
import org.springframework.http.HttpStatus;
import org.springframework.http.ProblemDetail;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    ProblemDetail onValidation(MethodArgumentNotValidException ex) {
        var pd = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, "Request validation failed");
        pd.setProperty("errors", ex.getBindingResult().getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .toList());
        return pd;
    }

    @ExceptionHandler(NotFoundException.class)
    ProblemDetail onNotFound(NotFoundException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    }

    @ExceptionHandler(InsufficientStockException.class)
    ProblemDetail onStock(InsufficientStockException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
    }

    @ExceptionHandler(IllegalOrderStateException.class)
    ProblemDetail onIllegalTransition(IllegalOrderStateException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
    }
}

The three business exceptions are trivial RuntimeException subclasses in their feature packages (e.g. InsufficientStockException(String sku, int requested) with a message like "Insufficient stock for SKU-123 (requested 5)").

Why records for DTOs? Immutable, value-based equality, zero boilerplate — and they make the boundary explicit: the API contract can now evolve independently of the persistence model. Returning entities from controllers couples your JSON to your table layout, invites lazy-loading serialization crashes, and leaks fields (imagine password_hash serializing). This maps directly to what you already know from TypeScript: DTOs are your API types; entities are your database types; never export the latter.
Checkpoint: with security still permitting requests (or temporarily permitAll on /api/** until Milestone 7): POST /api/orders?customerId=1 with {"lines":[{"productId":1,"quantity":2}]} returns 201 and the order JSON. Send {"lines":[]} and expect a 400 ProblemDetail with an errors array. Try an illegal transition (POST /api/orders/1/status?target=SHIPPED on a PENDING order) and expect 409.

// com/prep/orders/order/OrderRepository.java
package com.prep.orders.order;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.JpaSpecificationExecutor;
import org.springframework.data.jpa.repository.Query;

import java.util.List;

public interface OrderRepository
        extends JpaRepository<Order, Long>, JpaSpecificationExecutor<Order> {

    boolean existsByIdAndCustomerId(Long orderId, Long customerId);   // Milestone 7 ownership check

    /** Each customer's most recent order — native, using DISTINCT ON (Postgres idiom). */
    @Query(value = """
        SELECT o.customer_id   AS "customerId",
               c.email         AS "email",
               o.id            AS "orderId",
               o.status        AS "status",
               o.total_amount  AS "totalAmount",
               o.created_at    AS "createdAt"
          FROM (
                SELECT DISTINCT ON (customer_id) *
                  FROM orders
                 ORDER BY customer_id, created_at DESC
               ) o
          JOIN customers c ON c.id = o.customer_id
        """, nativeQuery = true)
    List<LatestOrderRow> findLatestOrderPerCustomer();
}

// com/prep/orders/order/OrderSpecifications.java
package com.prep.orders.order;

import jakarta.persistence.criteria.Predicate;
import org.springframework.data.jpa.domain.Specification;
import java.util.ArrayList;

public final class OrderSpecifications {
    private OrderSpecifications() {}

    public static Specification<Order> withFilters(OrderStatus status, Long customerId) {
        return (root, query, cb) -> {
            var predicates = new ArrayList<Predicate>();
            if (status != null)     predicates.add(cb.equal(root.get("status"), status));
            if (customerId != null) predicates.add(cb.equal(root.get("customer").get("id"), customerId));
            return cb.and(predicates.toArray(new Predicate[0]));
        };
    }
}

package com.prep.orders.security;

import com.prep.orders.customer.Customer;
import org.springframework.security.oauth2.jose.jws.MacAlgorithm;
import org.springframework.security.oauth2.jwt.*;
import org.springframework.stereotype.Service;

import java.time.Instant;
import java.time.temporal.ChronoUnit;

@Service
public class TokenService {

    private final JwtEncoder encoder;

    public TokenService(JwtEncoder encoder) {
        this.encoder = encoder;
    }

    public String issue(Customer customer) {
        Instant now = Instant.now();
        JwtClaimsSet claims = JwtClaimsSet.builder()
            .issuer("orders-api")
            .issuedAt(now)
            .expiresAt(now.plus(1, ChronoUnit.HOURS))
            .subject(customer.getEmail())
            .claim("customerId", customer.getId())
            .claim("role", customer.getRole())          // "CUSTOMER" or "ADMIN"
            .build();
        JwsHeader header = JwsHeader.with(MacAlgorithm.HS256).build();
        return encoder.encode(JwtEncoderParameters.from(header, claims)).getTokenValue();
    }
}

Milestone 8 — The N+1 hunt

Goal: deliberately cause the most famous ORM performance bug, observe it with your own eyes, fix it three ways, and measure.

Step 1 — make it visible. In application.yml (dev only):

spring:
  jpa:
    properties:
      hibernate:
        generate_statistics: true   # logs a per-session summary incl. JDBC statement count

logging:
  level:
    org.hibernate.SQL: DEBUG
    org.hibernate.stat: DEBUG

Step 2 — trigger it. Temporarily change the list endpoint to return the full OrderResponse (with items) instead of OrderSummaryResponse, seed ~50 orders with 2–3 items each, and hit GET /api/orders?size=50. Watch the log:

select o1_0.id, o1_0.customer_id, ... from orders o1_0 ...      -- 1 query for the page
select oi1_0.order_id, oi1_0.id, ... where oi1_0.order_id=?     -- × 50 (items per order)
select p1_0.id, p1_0.name, ...        where p1_0.id=?           -- × ~120 (product per item)

... 171 JDBC statements executed for one HTTP request.

This happens because every relationship is LAZY (correctly!) and the DTO mapper walks order.getItems() then item.getProduct() per row — each first touch fires its own SELECT.

Because you set open-in-view: false in Milestone 1, this lazy walking only works while the transaction is open — do the DTO mapping inside the @Transactional service method. If you had left open-in-view on, the lazy loads would silently run during JSON serialization, outside the service, holding a connection through the whole response write — the default that hides the problem instead of surfacing it.

Step 3 — fix it. Three techniques, in order of preference for this case:

  • Right-size the payload (best here): the list endpoint should return OrderSummaryResponse without items at all — you already built that in Milestone 6. Most N+1s die when you stop over-fetching. 1 query.
  • Fetch join for the detail endpoint, where you genuinely need the graph: 
    @Query("""
    SELECT o FROM Order o
      JOIN FETCH o.items i
      JOIN FETCH i.product
     WHERE o.id = :id
    """)
Optional<Order> findByIdWithItems(@Param("id") Long id);
    One query, one order, whole graph. Equivalent declarative form: @EntityGraph(attributePaths = {"items", "items.product"}) on a repository method.
  • Batch fetching as a safety net: spring.jpa.properties.hibernate.default_batch_fetch_size: 25 turns N single-row selects into ⌈N/25⌉ IN (...) selects app-wide. Good insurance; not a substitute for thinking.
Do not combine a collection fetch join with Pageable. Hibernate cannot apply SQL LIMIT to a query whose row count is multiplied by the joined collection, so it fetches every matching row and paginates in memory — look for the HHH90003004: firstResult/maxResults specified with collection fetch; applying in memory warning. That "fix" can be slower than the N+1 it replaced. For paginated lists: summary DTOs, or a two-step select-ids-then-fetch-graph pattern.
Checkpoint: Hibernate statistics for the list request show the JDBC statement count dropping from 171 to 1 (summary DTO), and the detail request from 3+ to 1 (fetch join). Write those two numbers down — they are the spine of your performance interview story.
What you're practicing: Tutorial 7 (fetching strategies — this is the single most-asked JPA interview topic) and Tutorial 13 (thinking in queries-per-request as a first-class metric). The senior framing: N+1 is not a Hibernate bug, it is the default consequence of lazy loading meeting per-row navigation — and the fix is always "decide the fetch plan per use case."

import com.prep.orders.customer.Customer;
import com.prep.orders.customer.CustomerRepository;

import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.testcontainers.service.connection.ServiceConnection;
import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

import java.math.BigDecimal;
import java.util.List;
import java.util.concurrent.*;

import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest
@Testcontainers
class PlaceOrderIntegrationTest {

    @Container
    @ServiceConnection   // Boot 3.1+: wires datasource props from the container automatically
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine");

    @Autowired OrderService orderService;
    @Autowired ProductRepository productRepository;
    @Autowired CustomerRepository customerRepository;

    @Test
    void placeOrderDecrementsStockAndComputesTotal() {
        var customer = customerRepository.save(Customer.of("it@test.dev", "IT User", "x"));
        var product  = productRepository.save(new Product("SKU-IT", "Widget", new BigDecimal("10.00"), 5));

        var order = orderService.placeOrder(customer.getId(),
                List.of(new OrderService.OrderLine(product.getId(), 3)));

        assertThat(order.getTotalAmount()).isEqualByComparingTo("30.00");
        assertThat(productRepository.findById(product.getId()).orElseThrow().getStock()).isEqualTo(2);
    }

    @Test
    void concurrentOrdersForLastUnitDoNotOversell() throws Exception {
        var customer = customerRepository.save(Customer.of("race@test.dev", "Racer", "x"));
        var product  = productRepository.save(new Product("SKU-RACE", "Last unit", new BigDecimal("9.99"), 1));

        var startGate = new CountDownLatch(1);
        Callable<String> attempt = () -> {
            startGate.await();               // both threads released at the same instant
            try {
                orderService.placeOrder(customer.getId(),
                        List.of(new OrderService.OrderLine(product.getId(), 1)));
                return "OK";
            } catch (InsufficientStockException e) {
                return "REJECTED";
            }
        };

        try (var pool = Executors.newFixedThreadPool(2)) {
            Future<String> first  = pool.submit(attempt);
            Future<String> second = pool.submit(attempt);
            startGate.countDown();

            assertThat(List.of(first.get(), second.get()))
                .containsExactlyInAnyOrder("OK", "REJECTED");   // exactly ONE wins
        }

        assertThat(productRepository.findById(product.getId()).orElseThrow().getStock())
            .isZero();                                          // never -1
    }
}
Do not put @Transactional on the race test class. Test-level transactions would wrap each test in a rolled-back transaction on the test thread — but the two worker threads open their own transactions anyway, and test-managed rollback semantics around multi-threaded code are a reliable source of confusing flakes. Let this test commit for real and assert on real committed state; clean up with @AfterEach deletes or unique SKUs per test.
Checkpoint: ./mvnw verify — everything green. Temporarily revert decrementStock to the naive read-check-write version and run the race test ten times: it should fail at least once (stock −1 or two OKs). Put the fix back. Now you have seen the race, not just read about it.
What you're practicing: Tutorial 9 (test pyramid, Mockito, slices, Testcontainers) and Tutorial 14 (proving isolation behavior empirically). The layered strategy is itself the interview answer: fast unit tests for business rules, slice tests for the HTTP contract, a handful of full-stack tests against real Postgres for things only a real database can verify — locking, constraints, SQL dialect.

Correlation-ID filter — every log line for a request shares an ID, and the ID round-trips to the client:

// com/prep/orders/common/CorrelationIdFilter.java
package com.prep.orders.common;

import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.slf4j.MDC;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;
import java.util.UUID;

@Component
public class CorrelationIdFilter extends OncePerRequestFilter {

    public static final String HEADER = "X-Correlation-Id";

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        String id = request.getHeader(HEADER);
        if (id == null || id.isBlank()) {
            id = UUID.randomUUID().toString();
        }
        MDC.put("correlationId", id);
        response.setHeader(HEADER, id);
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove("correlationId");   // servlet threads are pooled — always clean the MDC
        }
    }
}
# application.yml — put the id into every log line
logging:
  pattern:
    level: "%5p [cid:%X{correlationId:-}]"

Milestone 12 (optional stretch) — Idempotency, caching, outbox

Only after 1–11 are solid. Each of these is a strong senior-differentiator topic:

  • Idempotency keys on order placement. Client sends an Idempotency-Key header; store it in an idempotency_keys table with a UNIQUE constraint and the serialized response. A retry (mobile client, gateway timeout) replays the stored response instead of double-charging. The interesting part: the unique constraint, not application logic, is what makes it race-proof — the same lesson as Milestone 4, and exactly why "how do you make POST safe to retry?" is such a common system-design probe.
  • Redis cache for the product catalog. spring-boot-starter-data-redis + @EnableCaching + @Cacheable("products") on catalog reads, @CacheEvict on writes. Be ready for the follow-up: what staleness is acceptable, and why you would never cache stock this way (it is the one field where staleness = oversell).
  • Outbox pattern on order-paid. When an order transitions to PAID, insert an event row into an outbox table in the same transaction; a @Scheduled poller publishes and marks rows sent. This solves the dual-write problem (DB commit + message publish cannot be made atomic) and is the canonical Tutorial 10 microservices answer to "how do services communicate reliably?" — you can build the whole thing here without a broker by logging the "published" event.

Turning it into interview stories

Behavioral-technical questions reward specifics: numbers, trade-offs, and a decision you personally made. Each entry below is a model narrative grounded in this project — rehearse them out loud, in your own words, with your measured numbers substituted in.

"Tell me about a performance problem you found and fixed."

Situation: "In an order-management API I built with Spring Boot and Postgres, the order-list endpoint felt fine in dev but I'd enabled Hibernate statement statistics, and one page of 50 orders was issuing 171 SQL statements — a classic N+1: one query for the page, then a lazy load per order for its items and per item for its product."

Task/Action: "I fixed it at two levels. For the list view, the real bug was over-fetching — the UI only needed summaries, so I switched to a slim DTO projection: one query. For the detail view, where the full graph is genuinely needed, I wrote a fetch join. I deliberately did not combine the fetch join with pagination, because Hibernate then paginates in memory after fetching everything — that 'fix' is worse than the disease."

Result: "171 statements down to 1 for the list, 3 down to 1 for the detail. Separately, at 100k rows the list query was seq-scanning and sorting — EXPLAIN ANALYZE showed ~39ms; a composite index on (status, created_at DESC) eliminated the sort entirely and brought it to ~0.14ms. The habit I took away: queries-per-request and the query plan are metrics you check before users complain."

"How do you handle concurrent updates?"

"The sharpest example was inventory: two customers buying the last unit simultaneously. Read-check-write in Java is a lost-update race — both threads read stock=1, both pass the check, stock goes to −1. I fixed it with an atomic conditional UPDATE: SET stock = stock − ? WHERE id = ? AND stock >= ?, and treated zero rows updated as a business rejection. Postgres's row lock inside UPDATE serializes the writers, so exactly one wins."

"I considered SELECT … FOR UPDATE — the right tool when the decision logic is too rich for one statement — but it holds the lock for the rest of the transaction and forces you to think about lock ordering across products, so I preferred the shorter critical section. I also kept two more layers: a CHECK (stock >= 0) constraint as a backstop so any future bug fails loudly instead of overselling, and @Version optimistic locking on orders for the low-contention status updates. And crucially, I proved it: a Testcontainers test fires two threads through a CountDownLatch at one unit of stock and asserts exactly one success and stock ending at zero — when I temporarily reverted to the naive code, that test failed, which is exactly what a regression test for a race should do."

"Walk me through your API design."

"Resource-oriented: /api/orders as the collection, POST to place (201 with the created representation), GET /{id} for details, and status transitions modeled as an explicit sub-action rather than a raw PATCH of a status field — because transitions have rules (PENDING→PAID→SHIPPED, CANCELLED only from PENDING or PAID), and the server enforces the state machine, returning 409 on an illegal transition. Errors are RFC 7807 ProblemDetail across the board, so clients get one machine-readable error shape with a field-level errors array for validation failures."

"The boundary is strict: request/response DTOs as Java records, entities never serialized — that keeps the API contract decoupled from the schema and prevents leaking fields. Lists are paginated with filtering by status and customer; and having spent seven years consuming APIs from React, I designed for the client I used to be: predictable envelopes, correlation IDs echoed in headers for support tickets, and a 409-vs-400 distinction the frontend can actually branch on. Security is a stateless JWT resource server; a customer role is scoped to its own orders via a method-security ownership check, and I never trust identifiers from the query string for authorization — the customer id comes from the token."

"How did you test it?"

"As a pyramid, with each layer answering a different question. Business rules — the order state machine, the insufficient-stock path — are plain Mockito unit tests, milliseconds each, verifying things like 'no order row is saved if any line fails'. The HTTP contract — validation, status codes, the ProblemDetail shape — is @WebMvcTest slices with the service mocked, so I test the web layer without booting a database. Then the things only a real database can verify — row locking, CHECK constraints, the actual SQL dialect — run against real Postgres 16 in Testcontainers, wired with @ServiceConnection so config is automatic."

"The test I'm proudest of is the concurrency test: two threads released by a latch racing for the last unit of stock, asserting exactly one success and no negative inventory. I validated the test itself by reverting the fix and watching it fail. The principle: an H2 test would have passed with subtly different locking behavior — integration tests should run against the engine you deploy on, and Testcontainers makes that cheap."

"Tell me about a transaction design decision you made."

"Order placement had to be all-or-nothing across multiple tables: decrement stock for each line, insert the order, insert its items. I drew the transaction boundary at the service method with @Transactional — not the controller, not the repository — because that method is the business invariant. Any line failing throws an unchecked exception, which rolls back everything including stock decrements already applied for earlier lines; I verified with a test that a failing second line leaves the first product's stock untouched."

"Two deliberate choices inside that boundary: default READ COMMITTED isolation was sufficient because the one genuine race — stock — is handled by an atomic guarded UPDATE rather than by escalating isolation to SERIALIZABLE, which would have meant retry loops for a problem I could solve with one statement. And I kept the transaction short: no HTTP calls, no token verification, nothing slow inside it, because a transaction holds a connection and its locks for its whole lifetime. I also know the proxy caveat — @Transactional is proxy-based, so self-invocation silently skips the transaction; that's a bug class I now check for in review."

"Why should we trust a frontend developer with backend work?" (the subtext question)

"Fair question — my answer is this project. It's not a tutorial follow-along: it has a concurrency test that catches overselling, EXPLAIN-verified indexing with before/after numbers, Flyway-owned schema with CHECK constraints, and a documented decision log. Seven years of consuming APIs taught me what good ones feel like from the client side; now I've built one end-to-end and can defend every layer of it. The fundamentals transferred faster than expected — TypeScript made Java's type system familiar, React's data-fetching pain made N+1 instantly intuitive — and where backend is genuinely different, like transactions and locking, I went deep enough to prove behavior with tests rather than assume it. I'd rather be judged on that artifact than on a keyword list."

Common mistakes

The classic ways capstone projects go sideways — each one is also a negative signal an experienced interviewer will spot in your repo within minutes.

EAGER fetching everywhere (or worse: leaving the @ManyToOne default). It "fixes" LazyInitializationException by loading the world on every query, and it makes fetch behavior invisible and unfixable per-use-case. Everything LAZY; choose the fetch plan per query with fetch joins, entity graphs, or projections. If you're getting LazyInitializationException, the answer is a wider transaction boundary or a DTO — never a fetch-type change.
Entities crossing the HTTP boundary. Returning Order from a controller couples your JSON to your schema, risks serializing lazy proxies (Jackson happily triggers N+1 or explodes), and leaks fields — today it's version, next refactor it's passwordHash. DTO records in, DTO records out, always. Same discipline as never exporting your database row types to your React components.
ddl-auto: update instead of Flyway. It cannot rename or drop anything, it drifts silently between environments, there is no history and no review, and it advertises "I have never run this in production." Migrations are code: versioned, reviewed, immutable once applied — and validate keeps Hibernate honest about matching them.
Testing only happy paths. Ten tests that all place a valid order prove almost nothing. The tests that carry interview weight are the unhappy ones: insufficient stock rolls back the whole order, illegal transitions 409, empty lines 400 with a field-level error, foreign customer's order 403 — and above all the concurrency test. One race test outweighs twenty green happy paths.
Giant service methods. When placeOrder grows to 150 lines doing validation, pricing, stock, persistence, and notifications, you can no longer name what it does or test paths in isolation. Keep domain rules on the domain (Order.transitionTo, Order.recalculateTotal), keep the service as orchestration, and let package-by-feature stop the "one OrderService to rule them all" gravity.
Skipping the measurement steps. The N+1 hunt and the EXPLAIN session are the two milestones people skip "to save time" — and they are precisely the ones that generate numbers for your stories. A claim ("I optimized the query") is filler; a measurement ("171 statements to 1; 39ms to 0.14ms") is a hire signal. Do not ship this project without docs/explain-before-after.txt.
