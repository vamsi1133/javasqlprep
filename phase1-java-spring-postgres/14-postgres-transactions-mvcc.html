14 — Transactions, Isolation & MVCC

How PostgreSQL runs many things at once without lying to any of them — and what happens when it can't.

If tutorial 13 was about making single queries fast, this one is about making concurrent queries correct. This is the single most senior-signal topic in the entire Postgres track. Any developer can write a SELECT; interviewers use transactions and isolation to find out whether you understand what the database actually promises you — and where those promises quietly end. Questions like "two requests decrement the same stock counter at the same time — what happens?", "how would you build a job queue on Postgres?", or "why did this migration take the site down?" are all really questions about the material on this page. As a React developer you have already fought race conditions (stale closures, double-submits, optimistic UI rollbacks); this is the same class of problem, except the database gives you real, well-defined tools to solve it — if you know they exist.

ACID, concretely, in Postgres T1

Everyone can recite the acronym. Seniors can explain what each letter does for them in a specific scenario, and what mechanism in Postgres delivers it. Use the classic money transfer as your running example — it exercises all four letters. 

CREATE TABLE accounts (
  id      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  owner   text   NOT NULL,
  balance numeric(12,2) NOT NULL,
  CONSTRAINT balance_non_negative CHECK (balance >= 0)
);

INSERT INTO accounts (owner, balance) VALUES ('alice', 1000), ('bob', 500);

Atomicity — all or nothing

A transfer is two writes that must succeed or fail together. Without a transaction, a crash between the two UPDATEs destroys money. Inside a transaction, either both changes become visible at COMMIT, or neither ever existed: 

BEGIN;
UPDATE accounts SET balance = balance - 200 WHERE owner = 'alice';
UPDATE accounts SET balance = balance + 200 WHERE owner = 'bob';
COMMIT;   -- both visible atomically

-- If anything errors between BEGIN and COMMIT (or the server crashes),
-- Postgres rolls back: neither UPDATE happened, as far as anyone can tell.

Mechanically, atomicity in Postgres is cheap because of MVCC (covered below): the new row versions written by an uncommitted transaction are simply invisible to everyone else. Rollback doesn't have to undo anything in place — it just marks the transaction aborted, and its row versions are ignored and eventually cleaned up.

Consistency — constraints enforce invariants

Consistency means every committed transaction moves the database from one valid state to another. The database can only enforce invariants you have declared: the CHECK (balance >= 0) above, NOT NULL, UNIQUE, foreign keys. Try to overdraw and Postgres aborts the whole transaction: 

BEGIN;
UPDATE accounts SET balance = balance - 5000 WHERE owner = 'alice';
-- ERROR: new row for relation "accounts" violates check constraint "balance_non_negative"
ROLLBACK;  -- the transaction is doomed; only ROLLBACK is accepted now

A cross-row invariant like "the SUM of all balances is constant" or "at least one doctor is on call" can not be expressed as a per-row constraint — and that's exactly where isolation levels matter, because such invariants can be broken by two individually-correct concurrent transactions (see write skew, below). Interviewers love this connection: C in ACID is only as strong as your declared constraints plus your isolation level.

Isolation — a preview

Isolation answers: "while my transaction runs, how much of other transactions' work can I see?" Postgres offers three effective levels — READ COMMITTED (default), REPEATABLE READ, and SERIALIZABLE — each with precise, testable semantics. The next four sections are entirely about this letter, because it's where 80% of interview questions live.

Durability — WAL and fsync at commit

When COMMIT returns success, the change survives a power cut. The mechanism is the Write-Ahead Log (WAL): before any data page is modified, a compact log record describing the change is appended to a sequential log. At commit time, Postgres fsyncs the WAL to disk (with synchronous_commit = on, the default) and only then acknowledges the commit. The actual data pages can be written lazily later (by checkpoints), because after a crash, Postgres replays WAL from the last checkpoint to reconstruct every acknowledged commit. 

-- The durability dial (per session or per transaction):
SET synchronous_commit = off;  -- commit returns before WAL hits disk
                               -- risk: lose the last few hundred ms of commits on crash
                               -- (never corruption, just "the commit evaporated")
-- Useful for high-volume, low-value writes (analytics events), never for payments.

Keep a one-line WAL mental model handy: "an ordered, append-only stream of every change, fsync'd at commit." It pays off three times: durability (crash recovery replays it), replication (replicas replay the same stream — tutorial 15), and performance (sequential WAL writes are why random-write workloads are still fast).

The concurrency anomalies (learn these first) T1

Isolation levels are defined by which anomalies they permit. Learn the five anomalies as concrete two-session movies, not as definitions to recite. Convention for this whole page: Session A and Session B are two psql terminals connected to the same database; time flows downward.

1. Dirty read — seeing uncommitted data

Session ASession B
BEGIN;
UPDATE accounts SET balance = 0 WHERE owner='alice';
SELECT balance FROM accounts WHERE owner='alice';
Dirty read would return 0 — data that may never commit.
ROLLBACK; — the 0 never existed…but B already acted on it.

PostgreSQL never permits dirty reads at any isolation level. Even if you ask for READ UNCOMMITTED, Postgres accepts the syntax but silently gives you READ COMMITTED. This is a classic interview checkpoint: Postgres has four names but three behaviors. In the demo above, B (at any level) sees 1000, not 0.

2. Non-repeatable read — same row, two answers

Session A (READ COMMITTED)Session B
BEGIN;
SELECT balance FROM accounts WHERE owner='alice'; → 1000
UPDATE accounts SET balance = 900 WHERE owner='alice'; (autocommit)
SELECT balance FROM accounts WHERE owner='alice'; → 900 — same transaction, different answer
COMMIT;

Allowed at READ COMMITTED (each statement takes a fresh snapshot); prevented at REPEATABLE READ and above.

3. Phantom read — same predicate, new rows appear

Session A (READ COMMITTED)Session B
BEGIN;
SELECT count(*) FROM accounts WHERE balance > 400; → 2
INSERT INTO accounts (owner, balance) VALUES ('carol', 700);
SELECT count(*) FROM accounts WHERE balance > 400; → 3 — a phantom row appeared in my result set

The distinction from a non-repeatable read: phantoms are about the set of rows matching a predicate changing (new/removed rows), not an existing row's values changing. The SQL standard lets REPEATABLE READ permit phantoms — but Postgres's REPEATABLE READ prevents them (more on this nuance below).

4. Lost update — two read-modify-writes, one survives

Session ASession B
Read balance → 1000. App computes 1000 − 100 = 900.Read balance → 1000. App computes 1000 − 250 = 750.
UPDATE ... SET balance = 900 ...; COMMIT;
UPDATE ... SET balance = 750 ...; COMMIT;
Final balance: 750. A's −100 vanished. Correct answer was 650.

This is the anomaly you will personally cause in production if you learn only one thing wrong. The full demo and all three fixes are in the READ COMMITTED section below.

5. Write skew — two transactions each read what the other writes

The subtlest one, and the flagship reason SERIALIZABLE exists. Two transactions each read a shared invariant, each write to different rows, and neither sees the other. Both were individually fine; together they break the invariant. Canonical example: a hospital requires at least one doctor on call.

Session A (Dr. Alice going off call)Session B (Dr. Bob going off call)
SELECT count(*) FROM doctors WHERE on_call; → 2 ≥ 2, safe to leaveSELECT count(*) FROM doctors WHERE on_call; → 2 ≥ 2, safe to leave
UPDATE doctors SET on_call=false WHERE name='alice';
COMMIT;		UPDATE doctors SET on_call=false WHERE name='bob';
COMMIT;
Both commit. On-call count: 0. No row was written by both — no lock conflict, no lost update — yet the invariant is dead.

Write skew is not prevented by READ COMMITTED or REPEATABLE READ, and FOR UPDATE doesn't naturally catch it either (the rows each transaction writes are different, and the rows it read as "evidence" weren't locked). Only SERIALIZABLE — or manually locking/materializing the invariant — fixes it. Saying this unprompted in an interview is a strong senior signal.

READ COMMITTED — the default, and its famous trap T1

READ COMMITTED is what every connection gets unless you say otherwise. Its contract: each individual statement sees a snapshot of data as of the moment that statement began — it sees everything committed before the statement started, and nothing uncommitted. Two statements in the same transaction can therefore see different data (non-repeatable reads and phantoms are both allowed).

What happens when an UPDATE hits a row someone else is changing

This is the behavior that makes the default level workable. If your UPDATE/DELETE targets a row that a concurrent uncommitted transaction has already modified, your statement blocks until that transaction ends. If it rolled back — proceed on the original row. If it committed — Postgres re-fetches the new (latest committed) version of the row, re-evaluates your WHERE clause against it, and proceeds only if it still matches. (Internally this per-row recheck is called EvalPlanQual.)

The lost-update demo, step by step

The broken pattern: read the value into the application, compute, write the computed value back.

Session ASession B
BEGIN;
SELECT balance FROM accounts WHERE id=1; → 1000		BEGIN;
SELECT balance FROM accounts WHERE id=1; → 1000
app computes 1000−100=900
UPDATE accounts SET balance=900 WHERE id=1;		app computes 1000−250=750
COMMIT;UPDATE accounts SET balance=750 WHERE id=1;
(blocked while A held the row lock; A committed, WHERE id=1 still matches the new version → proceeds, writes 750)
COMMIT; — final balance 750, the −100 is lost

Note that Postgres did nothing wrong: every statement obeyed its contract. The application smuggled a stale value across statements. Three correct fixes, in order of preference:

Fix 1 — atomic UPDATE (best when possible)

-- Don't read-modify-write in the app. Push the arithmetic into the statement:
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance - 250 WHERE id = 1;
-- Under READ COMMITTED the second UPDATE blocks, then re-reads the committed
-- row (900) and computes 900-250 = 650. Correct, no explicit locking needed.

This works precisely because of the re-check behavior described above: balance - 250 is evaluated against the latest committed version, not the snapshot. Add the CHECK constraint and you also get overdraft protection for free.

Fix 2 — pessimistic: SELECT ... FOR UPDATE

BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- takes the row lock NOW
-- Any concurrent FOR UPDATE / UPDATE on this row blocks until we commit,
-- so the value we read cannot change under us. Compute in app, then:
UPDATE accounts SET balance = 900 WHERE id = 1;
COMMIT;

Use when the app genuinely must inspect the value before deciding (e.g., business rules too complex for one statement).

Fix 3 — optimistic: version column

UPDATE accounts SET balance = 900, version = version + 1
WHERE id = 1 AND version = 7;   -- the version we read
-- 0 rows updated => somebody got there first => re-read and retry (or fail the request)

Best for low-contention, user-facing edits (two admins editing the same record). This is exactly what JPA's @Version generates — details in the API design section.

The frontend-dev analogy that will burn you: setCount(count + 1) vs setCount(c => c + 1). Reading state, computing in your closure, and writing the result back is the same bug in React and in SQL. SET balance = balance - 100 is the functional-updater form. If your code ever does repo.findById() → mutate in Java → save() without @Version or a lock, you have written the lost-update bug — it just hasn't fired yet.

REPEATABLE READ — one snapshot, and a Postgres twist T1

REPEATABLE READ upgrades the contract: one snapshot for the whole transaction, taken at the first query (not at BEGIN). Every statement sees the database frozen at that instant, no matter what commits around you. Non-repeatable reads: gone. And here is the interview nuance:

Postgres REPEATABLE READ also prevents phantom reads — stricter than the SQL standard requires, which allows phantoms at this level. Because the implementation is Snapshot Isolation (one frozen snapshot), rows inserted by others simply aren't in your snapshot, so they can't appear mid-transaction. Interviewers ask "which anomalies does REPEATABLE READ allow?" hoping you'll parrot the standard's table; the senior answer is "in the standard, phantoms are allowed; in Postgres they can't happen — Postgres RR is Snapshot Isolation. What Postgres RR still allows is write skew."

Two-session proof of the frozen snapshot

Session ASession B
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT count(*) FROM accounts; → 2 (snapshot taken here)
INSERT INTO accounts (owner,balance) VALUES ('dave', 50);
UPDATE accounts SET balance = 1 WHERE owner='alice';
SELECT count(*) FROM accounts; → still 2 (no phantom)
SELECT balance FROM accounts WHERE owner='alice'; → still the old value
COMMIT; then re-run the count → 3. The world catches up only after the snapshot ends.

The price: serialization failures (SQLSTATE 40001)

Reads are free, but writes have a rule: a REPEATABLE READ transaction may not modify a row that another transaction changed after your snapshot was taken. Instead of silently re-checking like READ COMMITTED does, Postgres aborts you:

Session A (REPEATABLE READ)Session B
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id=1; → 1000
UPDATE accounts SET balance = 900 WHERE id=1; (autocommits)
UPDATE accounts SET balance = balance - 50 WHERE id=1;
ERROR: could not serialize access due to concurrent update (SQLSTATE 40001)
ROLLBACK; — and the application must retry the whole transaction from the top

This means REPEATABLE READ (and SERIALIZABLE) transactions are not fire-and-forget: your application must have retry logic for 40001, or users will see random 500s under load. A retry is safe because the failed transaction wrote nothing. If your team can't or won't add retries, that is itself an argument for staying on READ COMMITTED and using explicit locks. Interview phrasing: "first occurrence of 40001 in logs is not a bug — missing retry handling for it is."

Note the nice interaction: at REPEATABLE READ, the lost-update demo from the previous section cannot lose an update even with the naive read-then-write pattern — the second writer gets 40001 instead. RR turns silent lost updates into loud retryable errors. Write skew, however, still sails straight through (neither doctor's transaction writes a row the other touched).

SERIALIZABLE — true serializability via SSI T1

SERIALIZABLE promises: the outcome of any set of concurrent transactions is equal to some one-at-a-time ordering of them. If no serial order could have produced the result, one transaction is aborted with 40001. Postgres implements this (since 9.1) with Serializable Snapshot Isolation (SSI): it runs transactions optimistically on snapshots — no extra blocking — while tracking read/write dependencies with predicate locks (visible in pg_locks as SIReadLock). When it detects a dangerous dependency cycle, it cancels a transaction rather than let a non-serializable result commit.

Write skew, caught

CREATE TABLE doctors (name text PRIMARY KEY, on_call boolean NOT NULL);
INSERT INTO doctors VALUES ('alice', true), ('bob', true);
Session A (SERIALIZABLE)Session B (SERIALIZABLE)
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT count(*) FROM doctors WHERE on_call; → 2		BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT count(*) FROM doctors WHERE on_call; → 2
UPDATE doctors SET on_call=false WHERE name='alice';UPDATE doctors SET on_call=false WHERE name='bob';
COMMIT; → succeedsCOMMIT; →
ERROR: could not serialize access due to read/write dependencies among transactions (40001)
B retries, re-reads count → 1, refuses to go off call. Invariant saved. Under RR, both commits succeed and count hits 0.

Costs and operational rules

  • CPU + memory overhead for dependency tracking; more aborts (including some false positives — SSI is conservative, especially when predicate locks escalate to page/relation granularity under memory pressure).
  • Every transaction that touches the data must be SERIALIZABLE for the guarantee to hold — one READ COMMITTED writer sneaking in can break invariants that the serializable ones were protecting.
  • Retry loops are mandatory, and transactions should be short and touch few rows to keep abort rates down.
  • Pure read-only work can use BEGIN ISOLATION LEVEL SERIALIZABLE READ ONLY DEFERRABLE; — waits for a safe snapshot, then runs with zero abort risk and zero SSI overhead (great for long reports).

From Spring

  // Isolation is set on the JDBC connection when the tx begins
  @Transactional(isolation = Isolation.SERIALIZABLE)

  // Default maps to the database default => READ COMMITTED on Postgres
  @Transactional
  public void transfer(long from, long to, BigDecimal amount) { ... }
}

Rule-of-thumb answer for "which level do you use?": READ COMMITTED plus deliberate techniques (atomic updates, FOR UPDATE, unique constraints, @Version) for OLTP endpoints; REPEATABLE READ for multi-statement reads that must be mutually consistent (reports, exports, pg_dump uses it); SERIALIZABLE when a cross-row invariant must hold and you can afford retries. Naming the techniques, not just the level, is what makes the answer senior.

MVCC internals — xmin, xmax, snapshots, bloat T1

Everything above is implemented by one idea: Multi-Version Concurrency Control. Postgres never updates a row in place. Every row version (a tuple) carries two hidden system columns:

  • xmin — the transaction ID (XID) that created this version;
  • xmax — the XID that deleted or superseded it (0 while the version is live).

An UPDATE is physically "insert a new version + stamp the old version's xmax". A DELETE just stamps xmax. A snapshot is essentially "the set of XIDs whose effects I'm allowed to see"; each query checks every tuple's xmin/xmax against its snapshot to decide visibility. Different transactions can therefore be looking at different versions of the same logical row at the same moment — that's the whole trick.

See it yourself

SELECT xmin, xmax, ctid, id, balance FROM accounts WHERE id = 1;
--  xmin | xmax | ctid  | id | balance
--  912  |  0   | (0,1) |  1 | 1000        <- created by tx 912, live

BEGIN;
UPDATE accounts SET balance = 950 WHERE id = 1;
SELECT xmin, xmax, ctid, id, balance FROM accounts WHERE id = 1;
--  xmin | xmax | ctid  | id | balance
--  931  |  0   | (0,3) |  1 | 950         <- I see MY new version (created by my XID 931)
COMMIT;

-- Meanwhile, before that COMMIT, another session running the same SELECT saw:
--  xmin | xmax | ctid  | id | balance
--  912  | 931  | (0,1) |  1 | 1000        <- old version, xmax=931 says "tx 931 is
--                                            replacing me"; 931 not committed yet, so
--                                            this version is still the visible one

(ctid is the physical location — note it changed, proving the update created a new tuple. Also try SELECT txid_current(); to see your own XID.)

The three consequences that make this a T1 topic

  1. Readers never block writers, writers never block readers. A plain SELECT takes no row locks and is never stuck behind an UPDATE; an UPDATE never waits for readers. The only row-level blocking is writer-vs-writer on the same row (or writer vs explicit FOR UPDATE-style locks). This is Postgres's answer to "why is it good at mixed read/write load" and the reason you rarely need the dirty-read workarounds other databases taught people.
  2. Dead tuples → bloat → VACUUM. Old versions ("dead tuples") stay on disk after they're invisible to everyone. Something must reclaim them: VACUUM / autovacuum (the deep dive is tutorial 15). Until then they inflate tables and indexes — a heavily-updated table can be 90% corpses. Quick look: SELECT n_live_tup, n_dead_tup FROM pg_stat_user_tables WHERE relname='accounts';
  3. Long-running transactions are toxic. VACUUM may only remove versions invisible to every current transaction. One idle-in-transaction connection (or forgotten REPEATABLE READ report) holds the cleanup horizon (xmin horizon) back for the whole database — dead tuples pile up everywhere, queries slow down, and in extreme cases you march toward XID wraparound. Find offenders: 
    SELECT pid, state, xact_start, now() - xact_start AS age, query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start
LIMIT 5;
    and set idle_in_transaction_session_timeout so leaked transactions get killed automatically.

Interview one-liner that ties the room together: "Postgres trades space for concurrency — updates copy rows so reads never block, and VACUUM pays the bill later. Most operational Postgres pain (bloat, wraparound, replication conflicts) is downstream of that one design decision."

Explicit locking — FOR UPDATE and friends T2

MVCC handles reads; when you need to serialize writes or decisions, you take row locks explicitly with a locking clause on SELECT. The lock lives until the transaction ends (commit or rollback) — there is no "unlock" statement.

The four row-lock strengths

ClauseMeaningConflicts withTypical use
FOR UPDATE"I will modify/delete this row (possibly its keys)"all other row locks + UPDATE/DELETEread-modify-write, queue claiming
FOR NO KEY UPDATE"I will modify non-key columns"like FOR UPDATE, but not with FOR KEY SHAREwhat a plain UPDATE (of non-key cols) takes implicitly
FOR SHARE"Nobody may change this row while I look"UPDATE/DELETE, FOR UPDATE / FOR NO KEY UPDATE; other FOR SHARE okprotect a parent row you're referencing
FOR KEY SHARE"Nobody may change this row's key or delete it"only FOR UPDATE and key-changing writeswhat FK checks take on the referenced row

Why FOR NO KEY UPDATE exists: foreign-key constraint checks take FOR KEY SHARE on the parent row. If you hold a full FOR UPDATE on accounts row 1, a concurrent INSERT INTO payments (account_id) VALUES (1) blocks on the FK check. Hold FOR NO KEY UPDATE instead (or just do a plain UPDATE of non-key columns) and child inserts proceed. Locking rows with FOR UPDATE "to be safe" on a hot parent table is a real-world throughput killer — knowing this distinction is a genuinely senior detail.

Two-session blocking demo

Session ASession B
BEGIN;
SELECT * FROM accounts WHERE id=1 FOR UPDATE; → returns row, holds lock
SELECT * FROM accounts WHERE id=1 FOR UPDATE; → hangs…
SELECT * FROM accounts WHERE id=1; (from a 3rd session) → returns instantly — plain reads don't care
COMMIT;…B's locked select returns now.

NOWAIT — fail instead of queueing

SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;
-- ERROR: could not obtain lock on row in relation "accounts"  (SQLSTATE 55P03)
-- Use when waiting is worse than failing: "seat already being purchased, try another".

FOR UPDATE SKIP LOCKED — the job-queue pattern T2

SKIP LOCKED changes FOR UPDATE from "wait for locked rows" to "pretend locked rows don't exist". That one clause turns a plain table into a correct multi-worker job queue — no Redis, no SQS — and it is the most-asked practical Postgres design question. Build it completely: 

CREATE TABLE jobs (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  status      text NOT NULL DEFAULT 'pending',       -- pending | running | done | failed
  payload     jsonb NOT NULL,
  attempts    int  NOT NULL DEFAULT 0,
  run_after   timestamptz NOT NULL DEFAULT now(),    -- supports retries with backoff
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);

-- Partial index: workers only ever scan claimable rows, index stays tiny
CREATE INDEX jobs_claimable_idx ON jobs (run_after)
  WHERE status = 'pending';

The worker's claim query

BEGIN;

UPDATE jobs
SET    status = 'running',
       attempts = attempts + 1,
       updated_at = now()
WHERE  id IN (
        SELECT id
        FROM   jobs
        WHERE  status = 'pending'
          AND  run_after <= now()
        ORDER  BY run_after, id
        LIMIT  10
        FOR UPDATE SKIP LOCKED          -- the magic: locked rows are invisible
       )
RETURNING *;                            -- worker receives its batch atomically

-- ... process the 10 jobs ...

UPDATE jobs SET status = 'done', updated_at = now() WHERE id = ANY($claimed_ids);
COMMIT;

Why every piece is there:

  • Inner SELECT ... FOR UPDATE SKIP LOCKED LIMIT 10 — each worker locks up to 10 claimable rows; rows already locked by other workers are skipped, not waited on. Ten workers polling simultaneously get ten disjoint batches with zero blocking and zero double-delivery.
  • Wrap in UPDATE ... RETURNING — claiming (status flip) and reading the payload is one atomic statement; no window where a claimed job looks pending.
  • Without SKIP LOCKED: workers convoy — all ten block behind the first one's locks, and your "parallel" queue processes serially. This is the exact wrong behavior interviewers want you to name.
  • Crash safety: if a worker dies mid-transaction, its transaction rolls back — status snaps back to pending automatically. If it dies after committing running (i.e., you claim and process in separate transactions), you need a reaper: UPDATE jobs SET status='pending', run_after = now() WHERE status='running' AND updated_at < now() - interval '10 minutes';
  • Retries with backoff: on failure, SET status='pending', run_after = now() + (interval '30 seconds' * attempts); after N attempts, status='failed' (a dead-letter state).

Two production caveats to volunteer: (1) keep the processing transaction short — if handling a job means calling a slow external API, claim in one transaction, process outside, complete in another, and rely on the reaper for crashes; holding a transaction open for a 30-second HTTP call is the long-transaction poison from the MVCC section. (2) high-churn queue tables bloat fast — make sure autovacuum is keeping up, and delete or partition done rows.

Table-level locks — why migrations block traffic T2

Every statement also takes a table-level lock, mostly invisible: SELECT takes ACCESS SHARE, writes take ROW EXCLUSIVE — these all coexist happily. The trouble is DDL. Many ALTER TABLE forms take ACCESS EXCLUSIVE — the nuclear lock that conflicts with everything, including plain SELECTs.

The classic outage, as a two-session movie

Session A (long report)Session B (migration)Session C (normal traffic)
BEGIN; SELECT ... FROM accounts ...; (runs for minutes, holds ACCESS SHARE)
ALTER TABLE accounts ADD COLUMN note text; → wants ACCESS EXCLUSIVE, queues behind A
SELECT * FROM accounts WHERE id=1; → queues behind B (lock queue is fair — nobody jumps the queue). Site is now down.

The ALTER itself would have taken 1 ms (adding a nullable column is metadata-only in modern Postgres, even with a constant default since v11). The outage is the waiting, which dams up everyone behind it. The guard: 

-- In every migration session:
SET lock_timeout = '3s';           -- give up fast instead of damming traffic
ALTER TABLE accounts ADD COLUMN note text;
-- ERROR: canceling statement due to lock timeout  => retry later; users never noticed

-- Related good habits:
CREATE INDEX CONCURRENTLY ...;     -- doesn't block writes (can't run inside a tx)
-- Split "add constraint" into: ADD CONSTRAINT ... NOT VALID; then VALIDATE CONSTRAINT
-- (VALIDATE takes only a weak lock while it scans).

Seeing locks: pg_locks

-- Who is waiting, and on whom?
SELECT a.pid,
       a.state,
       a.wait_event_type,
       l.locktype, l.mode, l.granted,
       left(a.query, 60) AS query
FROM   pg_locks l
JOIN   pg_stat_activity a USING (pid)
WHERE  NOT l.granted OR l.mode = 'AccessExclusiveLock'
ORDER  BY l.granted;

-- Direct blocker lookup for one stuck pid:
SELECT pg_blocking_pids(12345);

Deadlocks — reproduce, detect, prevent T2

A deadlock is two transactions each holding a lock the other needs. Postgres cannot prevent them (they're a property of your access pattern), but it detects them: after a transaction has waited deadlock_timeout (default 1s), Postgres runs cycle detection and kills one victim with error 40P01. The other transaction proceeds normally.

A recipe that deadlocks every time

Session A (transfer 1 → 2)Session B (transfer 2 → 1)
BEGIN;
UPDATE accounts SET balance=balance-100 WHERE id=1; ✓ locks row 1		BEGIN;
UPDATE accounts SET balance=balance-100 WHERE id=2; ✓ locks row 2
UPDATE accounts SET balance=balance+100 WHERE id=2; → blocks (B holds row 2)UPDATE accounts SET balance=balance+100 WHERE id=1; → blocks (A holds row 1) — cycle complete
~1 second later, one session gets:
ERROR: deadlock detected
DETAIL: Process 111 waits for ShareLock on transaction 931; blocked by process 222. Process 222 waits for ShareLock on transaction 930; blocked by process 111.
The victim's transaction is rolled back; the survivor's blocked UPDATE completes.

Prevention 1 — consistent lock ordering

The deadlock exists because A locks (1, 2) and B locks (2, 1). Make every transaction acquire in the same global order — e.g., ascending id — and cycles become impossible:

-- Fixed transfer: lock BOTH rows first, lower id first, then do the math
CREATE OR REPLACE FUNCTION transfer(p_from bigint, p_to bigint, p_amount numeric)
RETURNS void LANGUAGE plpgsql AS $$
BEGIN
  -- Deterministic lock order regardless of transfer direction:
  PERFORM 1 FROM accounts
   WHERE id IN (p_from, p_to)
   ORDER BY id                -- <- the fix: always ascending id
   FOR UPDATE;                -- (FOR NO KEY UPDATE is even friendlier to FKs)

  UPDATE accounts SET balance = balance - p_amount WHERE id = p_from;
  UPDATE accounts SET balance = balance + p_amount WHERE id = p_to;
END $$;
-- Now transfer(1,2,...) and transfer(2,1,...) both lock row 1 first;
-- the second caller simply waits — no cycle, no deadlock.

Prevention 2 & 3 — short transactions, ordered batches

  • Keep transactions short. Deadlock probability scales with (locks held) × (time held). Don't interleave user think-time, external calls, or heavy computation between lock acquisitions.
  • Order your batch updates. UPDATE ... WHERE id = ANY(ids) from two workers with overlapping, differently-ordered id lists is a deadlock machine. Sort ids in the application (or lock via a subquery with ORDER BY id FOR UPDATE) before updating.
  • Treat 40P01 like 40001: it's retryable — the victim wrote nothing. A generic "retry on serialization/deadlock failure" wrapper covers both.

deadlock_timeout is not "how long until a deadlock errors" — it's "how long a lock wait lasts before Postgres bothers to check for a cycle". Non-deadlocked waiters keep waiting forever (unless lock_timeout is set). Also: log_lock_waits = on logs any wait exceeding this timeout — cheap and invaluable in production.

Transactions in API design T2

This is where interviewers check that you can go from database semantics to endpoint design. Four patterns cover almost everything.

1. Idempotency keys — surviving client retries

Networks retry: the client's POST times out, it retries, and you charge the card twice — unless the operation is idempotent. The database-backed pattern: the client sends a unique Idempotency-Key header; you make it a UNIQUE column and let the constraint do the deduplication atomically: 

CREATE TABLE payments (
  id              bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  idempotency_key text NOT NULL UNIQUE,
  account_id      bigint NOT NULL REFERENCES accounts(id),
  amount          numeric(12,2) NOT NULL,
  status          text NOT NULL DEFAULT 'created'
);

-- In the endpoint's transaction:
INSERT INTO payments (idempotency_key, account_id, amount)
VALUES ($1, $2, $3)
ON CONFLICT (idempotency_key) DO NOTHING
RETURNING id;
-- Row returned  => first time: proceed to charge, all inside this tx.
-- No row        => duplicate: SELECT the existing payment by key and return
--                  its stored result with the same status code as the original.

The unique index makes the race-proof guarantee: two concurrent identical requests cannot both insert; one wins, the other sees the conflict. No advisory locks, no "check then insert" race.

2. Optimistic locking — the version column (JPA @Version in plain SQL)

-- Schema: ALTER TABLE accounts ADD COLUMN version int NOT NULL DEFAULT 0;

-- What Hibernate emits for an @Version entity save:
UPDATE accounts
SET    balance = $1, version = version + 1
WHERE  id = $2 AND version = $3;      -- $3 = version we read

-- rowcount = 1  => success
-- rowcount = 0  => someone updated since we read => Hibernate throws
--                  OptimisticLockException => API returns 409 Conflict
--                  (or re-reads and retries, depending on the use case)

Choose optimistic when contention is rare and losers can be told "record changed, refresh" (CMS edits, settings pages). Choose pessimistic (FOR UPDATE) when contention is common or losing is unacceptable (inventory decrement, balance transfer) — retry storms under high contention make optimistic worse than just queuing on a lock.

3. Retry-on-40001 — the missing piece for RR/SERIALIZABLE

// Spring sketch. CRITICAL: the retry boundary must be OUTSIDE the transaction
// boundary — retrying inside a doomed tx just fails again.
@Service
public class OnCallService {

  @Retryable(
    retryFor = { CannotSerializeTransactionException.class,   // 40001
                 DeadlockLoserDataAccessException.class },     // 40P01
    maxAttempts = 3,
    backoff = @Backoff(delay = 50, multiplier = 2, random = true)) // jitter!
  @Transactional(isolation = Isolation.SERIALIZABLE)
  public void goOffCall(String doctor) { ... }
}
// Gotcha: both annotations are proxy-based. On the same method this works only
// because @Retryable's interceptor is ordered before the tx interceptor — each
// retry gets a FRESH transaction. Safer and more explicit: put @Retryable on a
// caller bean and @Transactional on the callee.

4. One request = one short transaction

  • One HTTP request, one transaction — a single commit point makes the endpoint atomic; partial state can't leak between multiple commits if the request dies halfway.
  • Kept short — every ms a transaction is open it holds locks (blocking other writers) and pins the MVCC snapshot horizon (blocking VACUUM). Transactions are also connections: with a pool of 20, twenty 3-second transactions = total stall.
  • Never hold a transaction across an external call. Calling Stripe (p99: seconds, sometimes 30s timeouts) inside @Transactional means slow-third-party outages become your database outages: pool exhausted, vacuum horizon pinned, lock queues growing. Pattern instead: tx1 = record intent (status pending) + commit → call Stripe with no tx open → tx2 = record outcome. Combine with the idempotency key so the intent row doubles as the dedup record, and a reaper/reconciler handles "we died between the call and tx2".

The Spring-shaped version of this mistake: a @Transactional service method that calls a RestTemplate/WebClient-blocking method three layers down. Nothing in the code looks wrong — you find it as idle in transaction sessions in pg_stat_activity and mysterious bloat. This exact story ("how would you debug connection pool exhaustion?") is a common senior interview scenario.

Advisory locks, SAVEPOINTs, and 2PC T3

Advisory locks — application-defined mutexes

Advisory locks let you lock an arbitrary number (a key you invent) rather than a row. Postgres enforces mutual exclusion on the key but attaches no meaning to it — perfect when the thing you're serializing isn't a row: a cron job, a tenant-wide operation, a distributed singleton. 

-- Session-level (held until unlock or disconnect):
SELECT pg_advisory_lock(42);        -- BLOCKS until acquired
SELECT pg_try_advisory_lock(42);    -- returns true/false immediately — no waiting
SELECT pg_advisory_unlock(42);

-- Transaction-level (auto-released at COMMIT/ROLLBACK — usually what you want):
SELECT pg_try_advisory_xact_lock(42);

-- Two-arg form to namespace keys: (app-area, entity-id)
SELECT pg_try_advisory_xact_lock(hashtext('billing-cron'), 2026);

Leader election / dedup-cron example — five app instances all fire the same scheduled job; exactly one should run it:

-- Each instance, at the scheduled time:
SELECT pg_try_advisory_lock(hashtext('nightly-invoice-run'));
--  true  => I am the leader: run the batch, then pg_advisory_unlock(...)
--  false => someone else is running it: exit quietly
-- No leader-election infra, no ZooKeeper — the database is already a
-- consistent shared service every instance can reach.

Prefer the try variants in app code (a stuck blocking acquire is invisible), prefer xact variants so crashes can't leak locks, and remember session-level advisory locks survive COMMIT — with connection pooling, a leaked session lock rides the pooled connection into some other request's session. This is also how Flyway/Liquibase and ShedLock-style schedulers serialize themselves.

SAVEPOINTs — partial rollback inside a transaction

BEGIN;
INSERT INTO orders ...;                 -- must survive

SAVEPOINT bonus;
INSERT INTO loyalty_points ...;         -- nice-to-have; may violate a constraint
-- ERROR: duplicate key ...             -- tx would normally be doomed, but:
ROLLBACK TO SAVEPOINT bonus;            -- undo ONLY the loyalty insert; tx is healthy again

COMMIT;                                 -- order committed, bonus skipped

Without the savepoint, any error poisons the whole transaction (Postgres refuses further statements until rollback). Spring tie-in: Propagation.NESTED is implemented with exactly this — a JDBC Connection.setSavepoint() when the nested method starts; if it throws, Spring rolls back to the savepoint and the outer transaction can continue. (Contrast with REQUIRES_NEW, which suspends the outer tx and opens a genuinely separate one on a second connection — different durability, different deadlock potential.) One caution: every savepoint has bookkeeping cost; per-row savepoints in a bulk load will crawl.

Prepared transactions / two-phase commit — exists, avoid

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
PREPARE TRANSACTION 'xfer-9231';   -- phase 1: durably parked, survives restart
-- ... an external coordinator later decides:
COMMIT PREPARED 'xfer-9231';       -- or ROLLBACK PREPARED 'xfer-9231';

This is the database half of XA/two-phase commit across multiple resources. Know three things: (1) it's disabled by default (max_prepared_transactions = 0) — a hint from the developers; (2) a prepared transaction holds its locks and pins the vacuum horizon indefinitely, even across server restarts — an orphaned one (coordinator died) will quietly bloat your database for weeks (check pg_prepared_xacts); (3) the modern answer to "atomic across my DB and Kafka/another service" is not 2PC but the outbox pattern — write the business row and an outbox event in one ordinary local transaction, and let a relay publish it — plus sagas/compensation for multi-service workflows.

Interview Q&A

Walk me through what each ACID property means, with the mechanism Postgres uses for it.

Atomicity: all statements in a transaction commit or none do; Postgres gets this nearly free from MVCC — uncommitted row versions are invisible, so rollback just marks the transaction aborted. Consistency: committed transactions preserve declared invariants — CHECK, NOT NULL, UNIQUE, FK constraints abort violating transactions; cross-row invariants additionally need the right isolation level. Isolation: concurrent transactions don't see each other's in-flight work — implemented with MVCC snapshots at three levels (READ COMMITTED, REPEATABLE READ, SERIALIZABLE). Durability: once COMMIT returns, the change survives a crash — the write-ahead log is fsync'd at commit and replayed during recovery; data pages themselves can be written lazily.

What is PostgreSQL's default isolation level, and what exactly does it guarantee?

READ COMMITTED. Guarantee: each statement sees a snapshot taken when that statement began — all data committed before the statement, nothing uncommitted. So no dirty reads, ever; but two statements in one transaction can see different data (non-repeatable reads and phantoms are possible). One more behavior worth naming: when an UPDATE/DELETE hits a row a concurrent transaction modified, it waits, then re-evaluates its WHERE clause against the newly committed version and proceeds — which is why atomic updates like SET balance = balance - 100 are safe at this level while read-then-write from the app is not.

Explain the lost update problem and three ways to prevent it.

Two transactions read the same value, compute in the application, and write results back; the second write silently overwrites the first, losing its effect — e.g., two concurrent −100 and −250 on a 1000 balance ending at 750 instead of 650. Fixes: (1) atomic update — push arithmetic into SQL, SET balance = balance - 100, correct under READ COMMITTED thanks to the re-check; (2) pessimistic — SELECT ... FOR UPDATE before reading, so the value can't change under you; (3) optimistic — version column, UPDATE ... WHERE id=? AND version=?, retry or 409 on zero rows. Also: at REPEATABLE READ the second writer gets serialization error 40001 instead of silently losing — loud beats silent.

What's the difference between a non-repeatable read and a phantom read?

Non-repeatable read: re-reading the same row gives a different value because another transaction updated it and committed. Phantom read: re-running the same predicate query returns a different set of rows because another transaction inserted or deleted matching rows. Both are permitted at READ COMMITTED. Per the SQL standard, REPEATABLE READ must prevent the first but may allow phantoms; in Postgres, REPEATABLE READ prevents both, because it's implemented as snapshot isolation — the transaction literally cannot see rows that weren't in its snapshot.

How does Postgres's REPEATABLE READ differ from the SQL standard's? Why?

The standard defines levels by which anomalies they may permit, and allows phantoms at REPEATABLE READ. Postgres implements RR as Snapshot Isolation: one snapshot for the whole transaction, taken at the first query. Since all reads come from that frozen snapshot, phantoms are impossible — Postgres RR is stricter than required. The remaining anomaly is write skew (two transactions read overlapping data, write disjoint rows, and jointly break an invariant), which snapshot isolation cannot detect; that's what SERIALIZABLE adds. Also worth noting: Postgres accepts READ UNCOMMITTED syntax but gives READ COMMITTED behavior — four names, three behaviors.

What is write skew? Give a concrete example and say which isolation level stops it.

Write skew: two concurrent transactions each read a shared condition, each write different rows based on it, and the combination violates an invariant neither would have violated alone. Classic example: rule "≥1 doctor on call", two on call; Alice's transaction and Bob's transaction each count 2, each set only their own row to off-call, both commit — count is now 0. No row was written by both, so no lock conflict and no lost update; READ COMMITTED and REPEATABLE READ both allow it. Only SERIALIZABLE (Postgres's SSI) detects the read/write dependency cycle and aborts one with 40001. Manual alternatives: FOR UPDATE on all rows read as evidence, or materialize the invariant into a single row/constraint that both must touch.

How does SERIALIZABLE work in Postgres, and what does it cost?

Serializable Snapshot Isolation (SSI): transactions run optimistically on snapshots like REPEATABLE READ — no extra blocking — while Postgres tracks read/write dependencies using predicate locks (SIReadLocks). When it detects a dependency pattern that could produce a non-serializable outcome, it aborts one transaction with 40001. Costs: CPU/memory for tracking, higher abort rates including false positives (predicate locks can escalate to page or relation granularity), and the operational requirements that all transactions on the data run SERIALIZABLE and that the application retries 40001. Bonus point: read-only transactions can use SERIALIZABLE READ ONLY DEFERRABLE to wait for a safe snapshot and then run with no abort risk.

What is SQLSTATE 40001 and how should an application handle it?

"Serialization failure" — Postgres aborted the transaction because letting it continue could violate the isolation guarantee: at REPEATABLE READ, "could not serialize access due to concurrent update"; at SERIALIZABLE, "…due to read/write dependencies". It is not a bug — it's the contract of those levels. Handling: retry the entire transaction from the top (safe: the failed attempt wrote nothing), with capped attempts and jittered backoff. In Spring, the retry boundary must sit outside the transaction boundary — e.g., @Retryable on a caller, @Transactional on the callee — so each attempt gets a fresh transaction. Treat deadlock 40P01 the same way.

Explain MVCC. What are xmin and xmax?

Multi-Version Concurrency Control: Postgres never overwrites a row in place. Each row version (tuple) carries hidden columns — xmin, the transaction ID that created it, and xmax, the transaction ID that deleted/superseded it (0 while live). UPDATE writes a new version and stamps the old one's xmax; DELETE just stamps xmax. Every query carries a snapshot (essentially, which XIDs count as committed for me) and decides per-tuple visibility from xmin/xmax against it. So different transactions can simultaneously see different versions of the same logical row — which is how readers never block writers and writers never block readers. You can watch it: SELECT xmin, xmax, ctid, * FROM t; before and after an update.

"Readers never block writers" — what's the fine print?

Plain SELECTs take no row locks and read from their snapshot, so they neither wait for nor delay writers; the only row-level blocking is writer-vs-writer on the same row, or explicit locking clauses (FOR UPDATE/SHARE), which are writes for this purpose. Fine print: (1) table-level locks trump it — an ALTER TABLE wanting ACCESS EXCLUSIVE queues behind readers and then blocks new readers behind it; (2) readers do impose an indirect cost — a long-running reader pins the snapshot horizon so VACUUM can't clean dead tuples anywhere; (3) FK checks take KEY SHARE row locks, so they can wait on a FOR UPDATE holder. The pure statement is 90% true and the exceptions are exactly the interesting production incidents.

Why are long-running transactions harmful in Postgres even if they only read?

VACUUM may only reclaim row versions invisible to every active transaction. A transaction open for an hour — even idle-in-transaction, even read-only at REPEATABLE READ — holds the xmin horizon back for the whole cluster: dead tuples from every table accumulate, tables and indexes bloat, scans slow down, and in the extreme you accelerate toward XID wraparound pressure. Secondary harms: any locks it holds are held the whole time, and it occupies a pooled connection. Mitigations: keep transactions short, never hold one across external calls or user think-time, set idle_in_transaction_session_timeout, and monitor pg_stat_activity ordered by xact_start.

When would you use SELECT ... FOR UPDATE vs an optimistic version column?

FOR UPDATE (pessimistic): take the row lock before reading, concurrent claimants queue. Choose when contention is common or losing is unacceptable — inventory decrements, balance transfers, queue claiming — because under contention everyone eventually succeeds in turn, with no wasted work. Version column (optimistic): no lock while thinking; the UPDATE carries WHERE version = ?, and zero rows updated means retry or surface 409. Choose when contention is rare and the "loser" flow is acceptable UX — admin edits, settings, CMS content — and when state must round-trip through a stateless HTTP edit form where holding a lock is impossible. Under high contention optimistic degrades into retry storms; under low contention pessimistic adds latency for nothing. Also mention FOR NO KEY UPDATE when locking parent rows that children reference, to avoid blocking FK checks.

Design a job queue on Postgres. What makes it safe with many workers?

A jobs table (status, run_after, attempts, payload) with a partial index on claimable rows; workers claim with one atomic statement: UPDATE jobs SET status='running', attempts=attempts+1 WHERE id IN (SELECT id FROM jobs WHERE status='pending' AND run_after <= now() ORDER BY run_after LIMIT 10 FOR UPDATE SKIP LOCKED) RETURNING *;. FOR UPDATE gives exclusive claim; SKIP LOCKED makes competing workers skip each other's locked rows instead of queueing behind them — disjoint batches, no double-delivery, no convoy. Crash safety: claim-and-process in one transaction rolls back to pending on crash; or claim/complete in separate short transactions with a reaper that re-queues stale 'running' rows. Retries via run_after backoff and a failed/DLQ state. Caveats to volunteer: keep transactions short (no external calls inside), and watch bloat/autovacuum on high-churn queue tables.

Why can a "trivial" ALTER TABLE take down production, and how do you run migrations safely?

Most ALTER TABLE forms need ACCESS EXCLUSIVE, which conflicts with everything including SELECTs. The lock request queues behind any current holder — say a minutes-long report holding ACCESS SHARE — and because the lock queue is fair, all new queries then queue behind the waiting ALTER. The 1 ms DDL becomes a site-wide stall. Safe practice: SET lock_timeout = '3s' in every migration so it fails fast instead of damming traffic, and retry; use CREATE INDEX CONCURRENTLY; split constraint addition into ADD CONSTRAINT ... NOT VALID followed by VALIDATE CONSTRAINT; know which ALTERs are metadata-only (adding a nullable column, or one with a constant default since PG 11) versus table-rewriting (changing a column's type, most of the time).

How does Postgres handle deadlocks, and how do you prevent them?

Postgres detects rather than prevents: when a transaction has waited deadlock_timeout (default 1s), it runs cycle detection on the waits-for graph; if a cycle exists, one participant is aborted with 40P01 "deadlock detected" and the others proceed. Prevention: (1) consistent lock ordering — e.g., a transfer function that locks both account rows with SELECT ... WHERE id IN (a,b) ORDER BY id FOR UPDATE before updating, so opposite- direction transfers can't form a cycle; (2) short transactions — probability scales with locks held × duration; (3) sort ids before batch updates so overlapping batches lock in the same order. And treat 40P01 as retryable with the same wrapper as 40001. Bonus: log_lock_waits = on for visibility.

How do you make a payment endpoint safe against client retries?

Idempotency keys backed by a unique constraint. Client sends Idempotency-Key; the endpoint does INSERT INTO payments (idempotency_key, ...) VALUES (...) ON CONFLICT (idempotency_key) DO NOTHING RETURNING id inside its transaction. Row returned → first execution, proceed; no row → duplicate, load the original payment and replay its stored response. The unique index makes it race-proof: two concurrent identical requests can't both insert. Combine with the external-call rule: tx1 records intent + commits, the provider call happens with no transaction open (passing the same key to the provider — Stripe supports idempotency keys too), tx2 records the outcome; a reconciler covers a crash between call and tx2.

Why should you never call an external API inside a database transaction?

The transaction stays open for the full call duration — seconds to a 30s timeout — and while open it (1) holds every lock it acquired, blocking other writers; (2) pins the MVCC snapshot horizon, stalling VACUUM cluster- wide; (3) occupies a pooled connection, so a slow third party exhausts your pool and takes down unrelated endpoints. The failure mode is nasty because the code looks innocent — you diagnose it via 'idle in transaction' sessions in pg_stat_activity. Fix: record intent and commit; call the API with no transaction open; record the outcome in a second transaction; make the whole thing idempotent so crashes between steps are recoverable.

What are advisory locks and when are they the right tool?

Locks on application-chosen 64-bit keys rather than rows — Postgres enforces mutual exclusion but attaches no meaning to the key. Right tool when the resource you're serializing isn't a row: ensuring one instance runs a cron job (pg_try_advisory_lock(hashtext('nightly-run')) — true means "I'm the leader", false means exit), serializing migrations (Flyway does this), per-tenant maintenance. Variants matter: try_* returns immediately instead of blocking; xact-level locks release at commit and can't leak, session-level survive commit and — with connection pooling — can ride a pooled connection into someone else's session, so prefer pg_try_advisory_xact_lock in app code.

What does a SAVEPOINT do, and how does Spring's NESTED propagation relate?

A savepoint marks a point inside a transaction you can roll back to — undoing later work while keeping earlier work and keeping the transaction alive. Essential in Postgres because any error otherwise dooms the whole transaction (it rejects all further statements until rollback). Spring's Propagation.NESTED is implemented on savepoints: entering the nested method calls Connection.setSavepoint(); if it throws, Spring rolls back to the savepoint and the outer transaction continues; commit is still single, at the outer boundary. Contrast REQUIRES_NEW: a genuinely separate transaction on a separate connection that commits independently — different guarantees, and a possible self-deadlock if the inner tx waits on the outer's locks. Caution: heavy per-row savepoint use (e.g., some ORM batch error-handling) has real overhead.

Does Postgres support two-phase commit? Would you use it?

Yes: PREPARE TRANSACTION parks a transaction durably (survives restart), and COMMIT/ROLLBACK PREPARED finishes it — the database half of XA. But it's off by default (max_prepared_transactions = 0), and a prepared transaction holds its locks and pins the vacuum horizon indefinitely, even across restarts, so an orphaned one (dead coordinator) silently degrades the database for weeks — pg_prepared_xacts is where you look. I'd avoid it in application design: for "DB + message broker atomically", use the transactional outbox pattern (business row + event row in one local transaction, relay publishes); for multi-service workflows, sagas with compensation. I'd reserve prepared transactions for infrastructure that genuinely needs XA and has a real transaction manager babysitting it.

Hands-on exercises

Exercise 1 — Reproduce the lost update, then fix it three ways.

Open two psql terminals (Session A and B) against the same database. Create the accounts table from the top of this page with alice at 1000.

  1. Break it: in both sessions run BEGIN; then SELECT balance FROM accounts WHERE id=1; (both see 1000). In A: UPDATE accounts SET balance=900 WHERE id=1; COMMIT;. In B: UPDATE accounts SET balance=750 WHERE id=1; — watch it block until A commits, then proceed — COMMIT;. Confirm the final balance is 750, not 650: the −100 is lost.
  2. Fix with atomic update: reset to 1000, repeat, but use SET balance = balance - 100 / - 250. Verify 650. Explain to yourself why the blocked UPDATE computed against 900, not 1000 (statement re-check under READ COMMITTED).
  3. Fix with FOR UPDATE: reset; both sessions BEGIN; then SELECT balance FROM accounts WHERE id=1 FOR UPDATE; — B hangs at the SELECT this time. Finish A (compute 900 in your head, UPDATE, COMMIT), watch B unblock seeing 900, finish B. Verify 650.
  4. Fix with REPEATABLE READ: reset; repeat step 1 but both sessions start with BEGIN ISOLATION LEVEL REPEATABLE READ;. B's UPDATE should fail with could not serialize access due to concurrent update (40001). Confirm B wrote nothing and that re-running B's whole transaction from scratch yields 650.

Exercise 2 — Watch MVCC breathe: xmin, xmax, and the frozen snapshot.

  1. In Session A: SELECT txid_current(); (note it), then SELECT xmin, xmax, ctid, * FROM accounts WHERE id=1;. Record xmin, xmax, ctid.
  2. In A: BEGIN; UPDATE accounts SET balance = balance + 1 WHERE id=1; then re-run the xmin/xmax SELECT in both sessions before A commits. A should see a new tuple (xmin = A's XID, new ctid); B should still see the old tuple — check whether B's view shows a non-zero xmax (the old version knows it's being replaced). Commit A and re-check B.
  3. Snapshot demo: in B run BEGIN ISOLATION LEVEL REPEATABLE READ; SELECT count(*) FROM accounts;. In A, INSERT two rows and UPDATE one existing row (autocommit). Show B's count and B's view of the updated row are frozen; then COMMIT; in B and show the world catching up.
  4. Bloat teaser: run UPDATE accounts SET balance = balance + 0 WHERE id=1; twenty times, then SELECT n_live_tup, n_dead_tup FROM pg_stat_user_tables WHERE relname='accounts';. Explain where the dead tuples came from and what will eventually remove them.

Exercise 3 — Manufacture a deadlock, read the error, then make it impossible.

  1. With accounts 1 and 2 in place, run the deadlock recipe from the deadlocks section: A updates row 1 then row 2; B updates row 2 then row 1, interleaved. Time how long until ERROR: deadlock detected (about 1s — that's deadlock_timeout). Read the DETAIL lines and identify which pid waited on which.
  2. Confirm the survivor completed successfully and the victim's transaction rolled back cleanly (check the balances).
  3. Create the transfer(p_from, p_to, p_amount) function from this page with its ORDER BY id FOR UPDATE pre-lock. From the two sessions, run BEGIN; SELECT transfer(1,2,100); and BEGIN; SELECT transfer(2,1,50); interleaved. Observe: the second caller blocks (briefly) instead of deadlocking. Commit both; verify both transfers applied.
  4. Stretch: set log_lock_waits = on (SET log_lock_waits = on; needs superuser for the server log to matter — alternatively just reason about it) and re-run step 3 with a sleep (SELECT pg_sleep(2);) inside A's transaction to see what a logged lock wait would report.

Exercise 4 — Build the SKIP LOCKED job queue and prove it with two workers.

  1. Create the jobs table and partial index from the SKIP LOCKED section. Seed it: INSERT INTO jobs (payload) SELECT jsonb_build_object('n', g) FROM generate_series(1, 20) g;
  2. In Session A (worker 1): BEGIN; then the full claim query with LIMIT 5. Note which ids it claimed. Do not commit yet.
  3. In Session B (worker 2): run the same claim query. Verify it returns a disjoint set of 5 ids instantly — no blocking. Then, as a control, edit B's query to remove SKIP LOCKED, rollback, and run again while A still holds its locks: watch it hang. Ctrl-C, restore SKIP LOCKED.
  4. Crash simulation: kill Session A's terminal (or ROLLBACK;). Confirm its 5 jobs are pending again and claimable by B.
  5. Complete the loop: have B mark its batch done and commit. Write the reaper query (re-queue running rows with stale updated_at) and the backoff-retry UPDATE for a failed job, and test each by hand-crafting a stuck row.
  6. Stretch: run the claim query under EXPLAIN (ANALYZE, BUFFERS) and confirm it uses jobs_claimable_idx. Then think: what happens to this plan when 1M done rows accumulate, and why does the partial index save you?
