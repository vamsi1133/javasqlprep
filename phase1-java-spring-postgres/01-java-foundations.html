The mental shift: JavaScript → Java

Five differences explain 90% of the friction when you cross over. Internalize these and the rest of the language reads naturally.

1. Static types, checked at compile time

In JS a variable holds whatever you put in it and mistakes surface at runtime. In Java, every variable, parameter, and return value has a declared type, and the compiler rejects the program before it ever runs if the types don't line up. That's not bureaucracy — it's a free, exhaustive test suite for "am I passing the right shape of thing," which is precisely the class of bug TypeScript was invented to catch. If you're comfortable with TS, you already have the mindset; Java just enforces it with no escape hatch (any doesn't exist).

// JavaScript — legal, blows up at runtime
let total = "5";
total = total + 3;        // "53" — oops

// Java — won't even compile
int total = 5;
total = total + "3";   // compile error: incompatible types

2. Compiled, then run — two steps

JS is handed to an engine (V8) that runs it directly. Java source (.java) is first compiled by javac into bytecode (.class), and that bytecode runs on the JVM (Java Virtual Machine). "Write once, run anywhere" means the same bytecode runs on any OS that has a JVM. You'll rarely run javac by hand — a build tool does it — but knowing the two phases explains why there's a compile step at all.

3. Primitives vs objects

Java has eight primitive types — int, long, double, boolean, char, byte, short, float — that are not objects. They hold a raw value directly, have no methods, and can never be null. Everything else (String, arrays, your classes, collections) is an object, accessed through a reference. Each primitive has a "boxed" object twin (int→Integer, double→Double) used when you need an object — for example, collections can only hold objects, so a list of ints is really List<Integer>.

Comparing boxed objects with == compares references, not values. Integer a = 1000, b = 1000; a == b is false (two different objects) — use a.equals(b) or compare the primitive int. This is the #1 beginner trap, covered fully on the next page.

4. Classes and interfaces, not prototypes

There is no prototype chain and no free-floating function. Every piece of code lives inside a class. Behavior is shared through class inheritance and interfaces, not by wiring up __proto__. A "function" you pass around is really an object implementing a single-method interface — which is exactly what a lambda is (page 03). If you've used TS interface and class, this maps over cleanly; Java just makes classes mandatory rather than optional.

5. Explicit over implicit

Java favors saying things out loud: types are written, not inferred everywhere; there's no truthiness (only a real boolean goes in an if); no == coercion; visibility (public/private) is declared. It's more verbose, and that verbosity is the point — an interviewer can read your intent off the code. Modern Java (var, records, streams — page 03) trims a lot of the ceremony without giving up the safety.

Hands-on: JDK, Maven & running your first app Hands-on

You know the language on paper but have probably never run mvn. The mental map is short: the JDK is your Node runtime + compiler, and Maven is your npm + build script rolled into one.

JDK vs JRE vs JVM

  • JVM — the runtime engine that executes bytecode. Analogous to V8.
  • JRE — JVM + the standard library. Enough to run Java, not compile it.
  • JDK — JRE + developer tools (javac compiler, java launcher, jshell). This is what you install. Like installing Node gives you the runtime and npm.
$ java -version     # the runtime — what runs your app
openjdk version "21.0.3" 2024-04-16 LTS
$ javac -version    # the compiler — should match
javac 21.0.3

Which version? Pick an LTS (Long-Term Support) release — 17 or 21. They get years of patches and are what companies run. SDKMAN! (sdk install java 21-tem) is the nvm of Java; Eclipse Temurin is the standard free build to download.

Maven in one minute

Real projects have dozens of dependencies, tests, and a packaging step — a build tool automates all of it. Maven does two jobs: dependency management (declare a library, it downloads that library and everything it needs — exactly like npm install) and a build lifecycle (compile → test → package, standardized so every Java project builds the same way). Its config file is pom.xml. Gradle is the alternative (Groovy/Kotlin config, common in Android); learn Maven first — it's everywhere in interview material.

The standard project layout — memorize this tree

my-app/
├── pom.xml                        # the build descriptor (≈ package.json)
├── src/
│   ├── main/java/com/acme/app/    # your production .java files
│   │   └── App.java
│   └── main/resources/            # config, SQL, etc. on the classpath
│   └── test/java/com/acme/app/    # JUnit tests, never shipped
└── target/                        # build output (.class, the .jar) — git-ignored

The folders under src/main/java must mirror the package com.acme.app; line at the top of each file — the directory path is the package name.

Your first program, start to finish

// src/main/java/com/acme/app/App.java
package com.acme.app;

public class App {
    // the entry point — the JVM calls exactly this signature to start
    public static void main(String[] args) {
        System.out.println("Hello, Java");
    }
}

Every runnable Java app starts at a public static void main(String[] args) method — the equivalent of your index.js entry point. Build and run:

$ mvn compile            # javac all sources into target/classes
$ mvn package            # compile + run tests + build target/my-app.jar
$ java -jar target/my-app.jar
Hello, Java

# Even faster for experiments — a REPL, like the node shell:
$ jshell
jshell> "abc".toUpperCase()
$1 ==> "ABC"

Don't just read this — install Temurin 21, run mvn -version, generate a starter with mvn archetype:generate (or use start.spring.io when you reach Spring), and get "Hello, Java" printing. Ten minutes now saves confusion for the whole track.

Everyday syntax, as deltas from JS

The 20% of syntax you'll touch constantly. It's mostly familiar — here's only what differs.

// Variables — type on the LEFT; `var` infers it (local variables only, Java 10+)
String name = "Ada";
int count = 42;
final double PI = 3.14159;      // `final` = `const` (can't reassign)
var items = new ArrayList<String>();  // inferred as ArrayList<String>

// Conditions need a real boolean — no truthiness
if (name != null && !name.isEmpty()) { ... }   // NOT `if (name)`

// Loops — classic and for-each (the `for...of` equivalent)
for (int i = 0; i < items.size(); i++) { ... }
for (String item : items) { ... }        // for-each — use this by default

// Strings — immutable; compare with .equals, NEVER ==
String a = "hi", b = "hi";
a.equals(b);        // true  — value equality (what you want)
a == b;             // compares references — a trap, avoid
String greeting = "Hello, " + name;     // + concatenates
String msg = "Count is %d".formatted(count); // like template literals

// Arrays — fixed length; know these but prefer List (next page)
int[] nums = {1, 2, 3};
nums.length;        // a field, not a method (String uses .length())

System.out.println(...) is console.log. It's verbose because out is a field on the System class and println is its method — in Java there are no globals, so even printing is reached through a class.

Classes, fields, constructors, methods

A class bundles state (fields) with behavior (methods) and a constructor that builds an instance. Nothing surprising versus a TS class — the shape is just mandatory and typed.

public class Account {
    private final String owner;   // field: set once, never reassigned
    private long balance;          // field: mutable state, hidden from outside

    // Constructor — same name as the class, no return type
    public Account(String owner, long opening) {
        this.owner = owner;         // `this` disambiguates field vs parameter
        this.balance = opening;
    }

    // Methods — behavior, with explicit return types
    public void deposit(long amount) {
        if (amount <= 0) throw new IllegalArgumentException("must be positive");
        this.balance += amount;
    }

    public long getBalance() { return balance; }   // a "getter"
}

// Using it — `new` builds an instance
Account acct = new Account("Ada", 100);
acct.deposit(50);
acct.getBalance();   // 150

Two habits Java expects: fields are usually private (callers go through methods — encapsulation), and anything that shouldn't change after construction is final. You can overload constructors and methods — several with the same name but different parameter lists — and the compiler picks by argument types. (Modern Java replaces most of this boilerplate for pure data holders with records, page 02.)

Pass-by-value (the one that trips everyone) T1

A guaranteed interview question, and easy to get wrong. Java is always pass-by-value. The subtlety is what the value is:

  • For a primitive, the value is the number itself — the method gets a copy, and changes don't escape.
  • For an object, the value is the reference (the address), copied. So the method points at the same object and can mutate it — but reassigning the parameter only repoints the local copy, not the caller's variable.
static void tryReassign(Account a) { a = new Account("X", 0); }  // repoints local copy only
static void mutate(Account a)      { a.deposit(10); }                // mutates the shared object

Account acct = new Account("Ada", 100);
tryReassign(acct);   // acct is UNCHANGED — still Ada/100
mutate(acct);        // acct.balance is now 110 — same object mutated

If you know JS, you already live this: primitives are copied, objects are shared references, reassigning a parameter doesn't affect the caller. Java is identical — the phrase "pass-by-value" just refers to the fact that the reference is what's copied. Say it exactly that way in an interview.

Interfaces vs abstract classes T1

Both define a contract subclasses must fulfill; the "which one?" question is a staple.

// Interface — a pure contract. A type can implement MANY.
public interface Payment {
    Receipt charge(long cents);        // abstract by default — no body
    default boolean supportsRefund() { return false; }  // default method (Java 8+)
}

// Abstract class — a partial implementation. A class extends ONE.
public abstract class BasePayment implements Payment {
    protected final String merchantId;   // abstract classes CAN hold state
    protected BasePayment(String m) { this.merchantId = m; }
    protected void audit(String ev) { ... }  // shared concrete helper
    // charge() left abstract — each subclass supplies it
}
InterfaceAbstract class
How manyImplement manyExtend one
State (fields)No instance fields (constants only)Yes — instance fields, constructors
Answers"Can do X" (a capability)"Is a kind of X" (shared base + state)
Reach for it whenUnrelated types share a capability; you want to keep options openRelated types share real code and data

Default methods (Java 8) let interfaces ship behavior, so the gap narrowed — but interfaces still can't hold mutable instance state, and that's the deciding line. Default to an interface (a type can implement several, keeping the design flexible); reach for an abstract class only when subclasses must share fields or constructor logic. "Prefer composition; program to interfaces" is the sentence to have ready.

Inheritance vs composition

Inheritance (extends) says "B is an A" and inherits its guts. It's powerful and overused: it welds the subclass to its parent's internals, so a change to the base can silently break every child (the "fragile base class"). Composition — holding another object as a field and delegating to it — is looser and usually better.

// Inheritance: tightly bound to Engine's internals
class TurboEngine extends Engine { ... }

// Composition: Car HAS-A engine; swap it freely, test with a fake
class Car {
    private final Engine engine;               // held, not inherited
    Car(Engine engine) { this.engine = engine; }  // injected
    void start() { engine.ignite(); }         // delegate
}

The guideline — "favor composition over inheritance" — is one of the most-quoted lines in OO design, and the whole of Spring's dependency injection (page 05) is composition taken to its logical end: objects receive their collaborators instead of building or inheriting them. Use inheritance for genuine "is-a" relationships with a stable base; use composition for everything else.

Access modifiers, static, final

Access modifiers control visibility — Java enforces encapsulation the language, not the linter:

ModifierVisible to
publicEveryone
protectedSame package + subclasses
(none) — "package-private"Same package only (the default)
privateThe declaring class only

static means "belongs to the class, not an instance" — one copy shared by all. A static method (like main, or Math.max) is called on the class itself; a static field is shared state. Roughly your module-level function/constant.

final means "can't change after it's set": a final variable can't be reassigned (const), a final method can't be overridden, a final class can't be subclassed (like String). Marking fields final is a cornerstone of writing safe, immutable objects — the very next page.

Enums

An enum is a type with a fixed, named set of instances — safer than passing around magic strings or ints, and a favorite for status fields.

public enum Status { PENDING, PAID, SHIPPED, CANCELLED }

Status s = Status.PAID;
switch (s) {                       // exhaustive, type-checked
    case PAID    -> ship();
    case PENDING -> waitForPayment();
    default     -> { }
}

Because each constant is a real object, enums can carry fields and methods too — enum Planet { EARTH(5.97e24), MARS(6.42e23); ... } with a constructor and a gravity() method. That's a common "I didn't know enums could do that" interview beat; the everyday use is simply a closed set of valid values the compiler polices for you.

Packages & how code is organized

A package is a namespace and a folder — com.acme.app.billing maps to com/acme/app/billing/. The convention is reverse-domain-name so packages are globally unique. import pulls a type from another package into scope, exactly like an ES module import (Java just imports by fully-qualified name rather than file path):

package com.acme.app.billing;      // this file's home

import java.util.List;              // bring in a standard-library type
import com.acme.app.model.Account;  // bring in one of your own

Types in java.lang (String, Integer, System, Math…) are imported automatically. Package-private visibility (the no-modifier default above) is scoped to exactly this namespace — a lightweight way to keep helper classes internal to a module.

  • Java is statically typed and compiled — the compiler is your safety net; there's no any.
  • Primitives hold raw values and can't be null; objects are references. Compare objects with .equals(), never ==.
  • Everything lives in a class; every app starts at public static void main.
  • Java is pass-by-value — the reference is what's copied, so you can mutate a shared object but not reassign the caller's variable.
  • Prefer interfaces to abstract classes, and composition to inheritance — the design principles Spring is built on.
  • Lock down with private and final by default; open up deliberately.

Interview Q&A

Is Java pass-by-value or pass-by-reference?

Always pass-by-value. For primitives the value is the number, copied. For objects the value is the reference, copied — so the method and caller point at the same object and the method can mutate it, but reassigning the parameter only repoints the local copy and the caller sees nothing. The precise phrasing that scores: "Java passes references by value." It behaves exactly like JavaScript's parameter passing.

What's the difference between a primitive and its wrapper type?

A primitive (int) is a raw value — no methods, never null, stored directly. Its wrapper (Integer) is a full object holding that value, which is what you need when an API only accepts objects — notably generics and collections, so a list of numbers is List<Integer>. Autoboxing converts between them automatically. Two gotchas: wrappers can be null (unboxing a null throws NullPointerException), and == on wrappers compares references, not values — use .equals() or compare the unboxed int.

When would you choose an interface over an abstract class?

Default to an interface: a type can implement many, so it keeps the design flexible and models "can-do" capabilities that cut across unrelated types. Choose an abstract class only when subclasses genuinely share state (instance fields) or constructor logic — the one thing interfaces can't provide, even with default methods. In short: shared contract → interface; shared contract plus shared implementation and data → abstract class.

Why is "favor composition over inheritance" good advice?

Inheritance binds a subclass to the parent's internal implementation, so a change in the base can silently break subclasses (the fragile base class problem), and Java only lets you extend one class, spending that budget early. Composition — holding a collaborator as a field and delegating — is looser: you can swap the collaborator, inject a fake for testing, and combine several behaviors without a rigid hierarchy. Reserve inheritance for true, stable "is-a" relationships; use composition for everything else. Spring's dependency injection is composition at scale.

What does static mean, and how is it different from an instance member?

static members belong to the class itself, not to any instance — there's exactly one copy, and you call it on the class (Math.max(a,b), App.main(...)) without creating an object. Instance members belong to each object and can differ per instance. Use static for stateless utilities and true class-level constants; be cautious with mutable static state because it's shared across the whole program and becomes a concurrency and testing hazard.

What is the JVM, and what does "write once, run anywhere" mean?

The JVM is the runtime that executes Java bytecode — the intermediate form javac produces from your source. Because the same bytecode runs on any platform that has a JVM (Windows, Linux, macOS, ARM, x86), you compile once and run anywhere, without recompiling per OS. It also means the JVM, not your source, is where JIT compilation, memory management, and garbage collection happen — the subject of page 04.

Hands-on exercises

Exercise 1: Predict pass-by-value

Given void swap(int[] pair) { int t = pair[0]; pair[0] = pair[1]; pair[1] = t; } and separately void reset(int x) { x = 0; }, what does the caller see after swap(arr) and after reset(n)? Explain why in one sentence each.

Solution

swap mutates the shared array through the copied reference, so the caller's array is swapped. reset receives a copy of the primitive n; assigning x = 0 changes only the local copy, so the caller's n is unchanged. The rule: you can mutate what a reference points to, but reassigning any parameter (primitive or reference) never affects the caller.

Exercise 2: Model it with the right tool

You're modeling notification channels: Email, SMS, and Push. All must send(message); Email and Push additionally support attach(file) but SMS does not. Sketch the types — where do you use an interface, an abstract class, and composition?

Solution 
interface Channel { void send(String msg); }          // capability all three share
interface Attachable { void attach(File f); }        // capability only some share

abstract class BaseChannel implements Channel {
    protected final RateLimiter limiter;               // shared state ⇒ abstract class earns its place
    BaseChannel(RateLimiter l) { this.limiter = l; }   // composition: limiter is injected, not inherited
}
class Sms   extends BaseChannel { ... }                  // send only
class Email extends BaseChannel implements Attachable { ... }  // send + attach
class Push  extends BaseChannel implements Attachable { ... }

Interfaces model capabilities (so attach lives only where it applies — SMS never gets a method it can't honor); the abstract class exists solely because there's real shared state (the rate limiter); and that limiter is composed in rather than inherited, keeping each channel testable with a fake.

Exercise 3: Build and run from scratch

Without an IDE: create the standard Maven layout, write an App whose main prints the sum of the integer command-line args (args), then mvn package and run the jar with java -jar target/app.jar 3 4 5 expecting 12.

Solution 
public class App {
    public static void main(String[] args) {
        int sum = 0;
        for (String a : args) sum += Integer.parseInt(a);  // args are Strings — parse them
        System.out.println(sum);
    }
}

The takeaways worth internalizing: args arrives as String[] (you parse to int yourself), Integer.parseInt is the standard string→int conversion, and mvn package is the one command that compiles, tests, and produces the runnable jar. Get this loop working once and the toolchain stops being mysterious.
